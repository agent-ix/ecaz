bulk-load escape hatch when the coordinator-routed cost matters.

## Findings

### P3 — Per-row cost decomposition is the right operator
framing

Listing the four operations explicitly (remote begin, remote
prepare, local placement write, remote commit-prepared) lets
operators reason about latency budgets without reading the
code. Worth one line about typical latency per operation in
the runbook (e.g., "≈ 1ms per RTT under normal local-network
conditions") so capacity planning has numbers to work with.
Could land in the Stage F harness packet.

### P3 — Bulk-load read-eligibility caution

The doc warns operators to "keep readers away from partially
registered bulk-load datasets or accept temporary omissions."
That's accurate but the operational shape is unclear: how does
an operator detect a partial registration window? Worth
documenting that
`register_placement_batch` is transactional within the calling
session, so partial visibility only matters across the boundary
between bulk-load tool runs (not within them).

### P3 — Cross-link to existing primitives

Worth confirming the docs cross-reference
`ec_spire_classify_centroid` and
`ec_spire_register_placement_batch` by name so operators can
find the bulk-load primitives directly. Cosmetic.

Accept. Closes 30896 P5 documentation. The tracker now reflects
