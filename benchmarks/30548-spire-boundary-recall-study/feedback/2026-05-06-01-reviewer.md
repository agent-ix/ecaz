## Cross-cutting Phase 5 observations

### Three parallel "rank centroids by IP" implementations

The same predicate now exists in three places:

- `scan/routing.rs::route_routing_object_to_child_pids` (used by scan
  and by `aminsert` via `collect_snapshot_routed_probe_leaf_rows`)
- `build/routing_plan.rs::route_boundary_assignment_for_vector` (used by
  populated single-level builds and recursive boundary builds)
- `update/materialization.rs::route_split_replacement_boundary_indexes`
  (split replacement)

All three agree today on the tie-break order (higher IP, lower centroid
ordinal, lower child PID), but they are independently maintained. This
is the load-bearing primary-placement decision for the entire
boundary-replication contract; future drift between any two of them
would silently corrupt either build/insert symmetry or scan/build
symmetry.

Suggest a follow-up checkpoint that lifts a single `rank_centroids_by_ip`
helper into `routing_plan.rs` (or a sibling `routing.rs`) and rewrites
all three sites in terms of it. Not blocking; file it now while the
shapes are still small.

### Test coverage gaps to record

These are gaps relative to a phase-closeout bar, not blockers for
landing:

1. **Delete-delta + replica suppression at SQL level.** Unit tests
   verify delete deltas reject the `BOUNDARY_REPLICA` flag. There is no
   PG18 SQL test that DELETEing a replicated row removes both the
   primary and the replica candidates from a subsequent scan. This is
   the load-bearing vacuum-interaction invariant for replicated
   indexes.
2. **Split → boundary → scan dedupe at SQL level.** Split replacement
   boundary fanout has unit coverage in
   `update/tests/materialization.rs` but no end-to-end maintenance-run
   SQL test exercising split → publish → scan dedupe with replicas.
3. **Multi-store (Phase 4) × replicas.** No fixture exercises replicas
   spread across multiple local stores. The recall study used a
   single-store config. The design says replicas inherit the existing
   PID placement planner; an explicit test would prove that.
4. **`boundary_replica_count` ≥ `nlists`.** The cap relies on
   `route_boundary_assignment_for_vector`'s
   `selected_pids.contains(&entry.pid)` dedupe and the implicit
   `min(scored_entries.len())` cap. An explicit test pinning behavior
   when `boundary_replica_count` saturates the leaf count would lock
   the bound.

Item (1) is the most valuable to add before declaring Phase 5 done.
The others are reasonable Phase 5 follow-ups.
