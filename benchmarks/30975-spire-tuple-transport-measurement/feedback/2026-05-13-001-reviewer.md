# Review: SPIRE Tuple Transport Measurement (30975)

## Summary

Closes Phase 12.2 tuple-heavy read throughput measurement
row. Local PG18 loopback with separate remote/coordinator
`ec_spire` indexes, `rabitq` storage, 20 KNN queries,
`nprobe=8`, `k=10`, `id,title,body` projection (200 rows /
31,510 bytes per run).

Warm-pass results:

- **JSON fallback**: avg 29.342ms, p50 28.988, p95 30.459,
  p99 35.103, 34.081 q/s.
- **Typed `pg_binary_attr_v1`**: avg 34.293ms, p50 33.892,
  p95 35.179, p99 41.313, 29.160 q/s.

Conclusion: **typed is not faster on this small scalar
loopback fixture**. Does not retire JSON, does not claim
typed speedup.

## Findings

### P1 — Honest "no speedup" reporting

This is the right way to close a measurement row. The
hypothesis (from 30896 P1 / 30913 design / 30963 cutover)
was that per-attribute binary I/O would beat JSON for
tuple-heavy workloads. The measurement says: **not on this
fixture**. Reporting that honestly — rather than burying it,
massaging the fixture to find a typed win, or quietly
deferring the row — is exactly the discipline this phase
needs.

The numbers are also internally consistent with what'd be
expected on a **small scalar loopback**:

- 200 rows × ~157 bytes/row = 31KB total payload per run;
- per-attribute binary serialization has fixed per-call
  overhead that amortizes only over much larger payloads;
- loopback eliminates wire transit, which is where binary
  vs JSON wire-format size differences would matter most.

The result is "no speedup AT THIS SCALE on THIS PATH" —
not a refutation of the broader hypothesis, but a real
data point that bounds when the cutover pays off.

### P1 — Scoped conclusion

The packet correctly does NOT claim:

- typed is slower in general;
- typed should be removed;
- JSON is preferred.

It claims only "no measured speedup on this fixture."
That's the right scope. Larger payloads, network-bound
remote dispatches, or arrays/composites (which JSON
serializes much more verbosely) could still show typed
wins. This fixture doesn't measure those.

The Phase 12.2 throughput row closure is justified by
**the measurement existing**, not by the result favoring
either path. Honest closure.

### P2 — Driver-shape note

The disclosure that the `ecaz bench spire-pipeline` Rust
client tripped the v1 DML frontdoor fail-closed guard for
this remote-placement KNN shape under `tokio-postgres`,
and the packet uses `ecaz dev sql` + psql instead, is
important context.

Two reads:

1. The fail-closed guard is doing its job — an
   unsupported shape gets rejected. That's the design
   contract.
2. The bench client can't currently exercise this
   measurement shape, which means future automation of
   this measurement isn't quite plug-and-play yet.

Worth a follow-up tracker note that the
`tokio-postgres`-via-bench-client KNN path needs widening
(or a documented bypass for measurement runs) before this
measurement can be re-run automatically. For now,
documenting the workaround keeps the evidence reproducible
even if the bench client path stays gated.

### P2 — Same-driver discipline across runs

Critical detail: JSON and typed runs use the **same SQL
driver shape**, differing only by
`ec_spire.remote_tuple_transport`. This eliminates "driver
overhead" as a confounder. Without this, the comparison
would be ambiguous. With it, the delta is attributable to
the transport selector. Right experimental design.

### P2 — Endpoint identity verification

The header records `pg_binary_attr_v1 ready
{pg_binary_attr_v1}`, confirming the typed run actually
went through typed transport (not silently fell back to
JSON because the capability was absent). This is the
auditable proof that the typed run measured what it
claimed to measure. Good — without this, a "no speedup"
result could be "we accidentally measured JSON twice."

### P3 — Remaining open Phase 12 rows

The packet correctly identifies the still-open rows:

- **JSON retirement** (Phase 12.2): unblocked by this
  measurement only in the sense that "no typed speedup
  observed" doesn't mandate retirement. Retirement is a
  separate decision based on compatibility-window expiry
  and operator opt-in, not on speedup.
- **Cost calibration** (Phase 12.3): 30976 addresses.
- **Full artifact capture** (Phase 12.9): live readiness
  bundle.
- **Final readiness bundle** (Phase 12.9): assembly.

Honest open-row list.

### P3 — JSON retirement implication

This is worth thinking about: if typed shows no speedup on
small payloads, is the JSON-retirement gate appropriate?
The original cutover argument (30896 P1) was about
performance. If performance isn't a deciding factor, the
remaining argument for retirement is **code surface
reduction** (one path vs two, no `serde_json` dep) and
**type-system correctness** (typed preserves binary
fidelity better than JSON for some types).

Worth a tracker note that the JSON retirement decision now
hinges on:

- compat window closure (calendar);
- operator-side migration (existing remotes upgraded);
- the code-surface/correctness arguments rather than a
  perf argument that this measurement didn't support.

That's a different decision shape than "typed is faster, so
retire JSON." Worth pinning so the retirement packet
doesn't accidentally cite this measurement as supporting
evidence when it isn't.

### P3 — Future measurement scenarios

If a future Phase 13 (AWS) measurement wants to test the
broader hypothesis, the scenarios that'd most likely show
typed wins are:

- **larger per-row payloads** (KB-scale, where JSON
  overhead becomes meaningful);
- **array/composite-heavy projections** (where JSON
  text-encoding inflates much more than binary);
- **non-loopback transport** (where wire size matters).

This packet's loopback small-scalar result correctly
bounds the negative finding to its conditions.

## Reviewer recommendation

Accept. **Honest closure of the measurement row** with a
clearly scoped negative finding. Right experimental design
(same-driver, capability-verified, predeclared columns),
right reporting discipline (no overclaim either direction),
right open-row identification. Sets up the JSON retirement
decision to be made on the right basis (compat + code
surface), not on perf alone.
