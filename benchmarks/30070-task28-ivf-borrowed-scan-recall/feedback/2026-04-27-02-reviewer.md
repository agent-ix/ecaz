### `647abd1` insert validate without normalize + `302ee78` assign without normalize — mathematically correct

This pair is subtle and worth spelling out because it *looks* wrong:

- `647abd1` removed `training::normalize_vector(...)` from
  `validate_insert_tuple` (`insert.rs:267`).
- `302ee78` rewrote `assign_vector_to_centroid` to call
  `nearest_centroid_for_source` (raw IP) instead of
  `nearest_centroid_for_normalized` (normalize-then-IP).

On its face this changes the metric used for centroid assignment,
since the centroids were trained on normalized vectors via spherical
k-means. **It does not.** Spherical k-means produces unit-norm
centroids. For any source vector `s` with `‖s‖ > 0`:

    argmax_c <s, c>  =  argmax_c (‖s‖ · <s/‖s‖, c>)
                     =  argmax_c <s/‖s‖, c>

because `‖s‖` is a positive constant that does not depend on `c`, so
it factors out of the argmax. The guard at `training.rs:230–236`
(`validate_assignable_vector`) confirms `‖s‖² > f64::EPSILON` before
assignment, so the multiplier is strictly positive. The argmax is
preserved; the per-insert `O(d)` normalization is saved. **Correct,
not a bug.**

The test at `training.rs:340–354` only covers signed-direction
argmax, not magnitude variation, but the math is sound regardless. A
dedicated test asserting argmax stability under magnitude scaling
would strengthen the guard against future refactors that drop the
unit-norm centroid invariant.

### `43563e5` directory-tag-preserving traversal — correct

`page.rs:657–668` previously assumed the *next physical tuple* after a
directory entry is itself a directory entry. The fix at lines 1467–
1545 introduces `find_next_tuple_with_tag` which scans forward, taking
a shared buffer lock per block, and returns the first tuple whose
first byte matches `IVF_LIST_DIRECTORY_TAG`. Correct discipline:

- `BUFFER_LOCK_SHARE` for read.
- `UnlockReleaseBuffer` even on the result-bearing path.
- Bounds-checks `tuple_offset + tuple_len > page_size`.
- Errors with descriptive messages on overflow / out-of-bounds.

This unblocks postings being interleaved with directory tuples on the
same block sequence — necessary once the layout supports growth.

### `ab0ea8a` + `077aae1` planner cost — concerning, not blocking

`cost.rs` constants moved as follows across the two commits:

| constant | before | after `ab0ea8a` | after `077aae1` |
|---|---:|---:|---:|
| `IVF_CENTROID_SCORING_DIMENSION_SCALE` | 0.75 | 0.75 | **0.03** |
| `IVF_POSTING_SCORING_DIMENSION_SCALE` | (same as centroid) | 0.03 | 0.01 |
| page-cost basis | `random_page_cost` | `random_page_cost` | `seq_page_cost × 0.25` |

The split into separate centroid vs posting scaling (`ab0ea8a`) is
defensible: posting-scoring uses TurboQuant SIMD kernels that are an
order of magnitude cheaper per dimension than the f32 centroid IP, so
a separate constant is appropriate.

The further drop in `077aae1` is more concerning. Centroid-scoring
went from 0.75 → 0.03 (25× reduction) **without a corresponding
implementation change** in centroid scoring code. The page-cost basis
shift to `seq_page_cost × 0.25` is also aggressive — claims IVF index
pages cost 25% of a sequential heap page, which is hard to justify on
a cold cache.

**This reads as cost-model tuning to make the planner pick IVF**,
not as a faithful model of runtime cost. The packet trail (30053
"nlists128 planner blocker", 30058 "cost model posting scale") makes
the motivation visible. Two follow-up actions:

1. **Verify** centroid scoring has been quantized somewhere I didn't
   read. If yes, document the constant. If no, the 25× drop is a
   tuning artifact and should be replaced with an honest model.
2. **Cross-test** the cost model against alternative paths (seq scan,
   bitmap, ec_hnsw on the same column) to confirm the new constants
   don't cause IVF to be picked when a faster path exists.

Not a merge blocker — the recall numbers stand on their own — but the
cost model will silently mis-cost mixed workloads if this is left as
hand-tuning.

### `vacuum.rs` — basic shape correct, scale concerns flagged

Read end-to-end. `ec_ivf_ambulkdelete` walks the directory in list-id
order, calling `bulkdelete_list_postings` per list:

- `read_ivf_postings_for_list_blocks_with_tids` loads **all** postings
  for the list into memory before filtering. `Vec<(tid, posting)>` per
  list. **For a 1M-row corpus with `nlists=64`, that's ≈16k postings
  in memory at once per list — fine. For pathological wide-list
  cases (nlists=8 on 10M rows ≈ 1.25M postings per list), this is a
  scaling cliff** and should be replaced with a streaming page-by-page
  delete loop.
- Empty postings are marked `deleted = true` rather than reclaimed —
  tombstones stay on disk forever. **No physical compaction.** Same
  pattern as pgvector ivfflat; acceptable for v1, must be addressed
  before any "vacuum reclaims space" claim.
- Live head/tail block tracking per list, then directory rewrite via
  `rewrite_ivf_list_directory`. Metadata totals updated via
  `initialize_metadata_page`. WAL discipline relies on the underlying
  page rewrites being individually WAL-logged; not audited here.
- Empty-directory short-circuit at line 96: if `directory_head ==
  INVALID && total_live_tuples != 0`, errors. Sane invariant.

**Item 6 from packet 30047 — vacuum surface — closed for the basic
correctness shape.** Scale concerns above need their own packet before
ec_ivf claims a vacuum story for production.

### `training.rs` — sampling confirmed

`build.rs:234` calls `resolve_training_sample_count(training_sample_rows,
heap_tuples.len())`. Default is 10k rows when the reloption is 0
(`build.rs:666`: `resolve_training_sample_count(0, 12_000) == 10_000`).
The reloption is exposed at `options.rs:24, 126` as `training_sample_rows`.

`train_spherical_kmeans` then operates on the sampled subset, not the
full corpus. **Item 7 from packet 30047 — k-means sampling — closed.**
The 1M+ training cliff is structurally avoided.

One observation worth a future packet: `training_sample_rows` is a
fixed integer; the FAISS-canonical heuristic is `min(n,
256 × nlists)`. At `nlists=128` the canonical sample is 32,768 rows;
the current default of 10,000 may under-sample at high `nlists`. Worth
sweeping recall vs sample-size at `nlists ∈ {64, 128, 256}` before
declaring the default tuned.
