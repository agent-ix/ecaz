## Findings

### 1. Task target (<5s seeding) not met — and packet acknowledges this

This is the item that needs a conscious call from the reviewer, not a
code fix.

Task L19 sets the target at "under 5 seconds on the scratch cluster."
Measured on this machine: corpus seed moved from `8.245s` (per-row
baseline) to `8.649s` (batched) — within noise, directionally slightly
*slower*. The packet correctly identifies the real bottleneck:
`encode_to_tqvector` inside the `VALUES` expression at ~16ms/call × 500
rows = ~8s.

The structural anti-pattern is fixed (6 per-row `Spi::run` loops → 2
batched statements), which was the *stated* reason for the task in review
218 item 9. The numerical target is unmet because the originating
review's ~163s claim did not hold on this machine — the real cost was
never SPI round-trips.

**Recommendation:** accept the structural fix and land this branch.
Separately, file a thin follow-up task on `encode_to_tqvector` per-call
cost in the seeding path, since the packet's investigation already
identified the hotspot. That follow-up is where the <5s target actually
lives. Do not block this branch on re-tasking the seeding path.

### 2. Direction of the change is slightly negative on this hardware

`corpus_seed: 8.245s → 8.649s` (+5%, within noise). The ~9 MB SQL string
allocation in Rust plus the server-side parse of a 9 MB VALUES clause is
slightly slower than 500 short statements on this specific machine. On
slower hardware (CI on the scratch cluster, or hosts with
synchronous_commit forced per-row) the SPI savings should swamp this.

Not a blocker, but worth flagging: on this branch's host, the batched
path is not a win. Land it for the anti-pattern removal and the future
hardware benefit, not for current numbers.

### 3. SQL injection surface — safe but worth a comment

`corpus_values` is interpolated from `format_recall_vector_sql_literal`
output, where the vectors are deterministic from `RECALL_SEED = 42`. No
external input, no injection risk. A one-line comment at the
`Spi::run(&format!(...))` call site ("inputs are deterministic and
controlled; no injection surface") would make the safety argument
explicit for future readers.

### 4. No follow-up signal for the remaining wall clock

The packet correctly notes that `~98s` of the remaining ~125s smoke
runtime lives in the probes and `CREATE INDEX` phases, which are review
218 item 4's territory. That is accurate but there is no explicit tracked
follow-up anywhere I could find. If item 4's successor task has not been
filed, please file it now while the measurements are fresh — otherwise
the investigation the packet did will need to be redone when the next
person looks at smoke runtime.
