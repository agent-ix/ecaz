# Feedback: A4 ann-benchmarks Reference Anchor

**Reviewer:** Claude (Opus 4.6)
**Date:** 2026-04-10
**Commit reviewed:** `2f529c0`

## Scope match

The anchor choice is a real public source with a version-pinned URL
(Qdrant `vector-db-benchmark` published results JSON). The converter
profile is additive (`tqhnsw_real_10k` / `tqhnsw_real_50k` unchanged).
The SQL probe reuses `probe_graph_scan_recall_external_summary_for_relation`
cleanly — no duplication of graph-scan logic. Constants live near
`RECALL_GATE_CONFIGS` with a source-URL comment. The `#[ignore]`d
integration test drives convert + load + probe end-to-end. Cross-link
from `docs/RECALL_REAL_CORPUS.md` is in place. Packet and docs correctly
frame a failing anchor as "investigate converter/loader/build, do not
retune the constant."

## Findings

### 1. Query-set skew risk is under-documented

Qdrant's published `recall@10 = 0.96082` is measured against whatever
query set their benchmark ships. This branch synthesizes queries by
taking the last 10k sorted-id rows from the same 1M parquet (packet §1).
The packet claims the 0.02 tolerance "absorbs that gap," but the gap
size is not empirically known until the probe actually runs.

If the first measured delta is >0.02, the review procedure says "land
red, investigate converter/loader/build" — but if the true cause is
query-distribution drift, that investigation will chase a phantom. This
needs to be called out explicitly in
`docs/RECALL_ANN_BENCHMARKS_ANCHOR.md` under the "caveats" section so the
first red result isn't misinterpreted as a code bug. Add one paragraph:
"query-set is synthesized from the last 10k sorted-id rows; if the
measured delta exceeds tolerance by a small margin, rule out
query-distribution drift before touching converter/loader/build."

### 2. `ef_search` parameter is unbound at the SQL surface

`probe_graph_scan_recall_ann_benchmarks_reference_for_relation` takes an
`ef_search` parameter, but the published number is pinned to
`hnsw_ef=128`. Nothing in the probe enforces or warns if a caller passes
`ef_search=64` — the row will report a measured-vs-published delta that
is meaningless because it compares measurements at different operating
points.

Please add a guard at the SQL surface entry: either
- a hard assert (`pgrx::error!`) that rejects any `ef_search != 128`, or
- a `pgrx::warning!` that names the published operating point and asks
  the caller to confirm they understand the constant is pinned to 128.

The hard assert is preferable — this is an oracle, not a dial.

### 3. Constant duplication between `src/lib.rs` and `recall_integration.rs`

`PUBLISHED_RECALL_AT_10 = 0.96082` and `TOLERANCE = 0.02` are declared in
both `src/lib.rs` (under `#[cfg(any(test, feature = "pg_test"))]`) and
`tests/recall_integration.rs`, with a "keep this in sync" comment. The
comment acknowledges the trade-off but this is a maintenance trap — a
future edit that touches one site and not the other will silently diverge
the published constant from the integration-test assertion.

Consider moving both constants to a non-gated module (or a tiny new
`src/recall_constants.rs`) that both the pg_test surface and the
integration test can depend on. Low-urgency refactor, but worth doing
while the constants are fresh.

### 4. Naming: `within_two_percent` is misleading

The `within_two_percent: bool` column is computed as `|delta| <= 0.02`,
which is "within 0.02 absolute," not "within 2% relative to the published
value." For a published value of 0.96082, "2%" would naturally read as
~0.0192 — close but not identical. More accurate names:
`within_tolerance` or `within_two_percentage_points`. Rename before a
future reader misreads the column.

### 5. Manual run deferred, consistent with 10054

The first measured anchor run is deferred to the operator that has the
staged corpus. Same pattern as packet 223. Defensible given scratch
cluster preconditions.

## Verdict

**Approve with follow-ups.** None of the findings block landing the
branch. Items 1 (doc caveat), 2 (ef_search guard), and 4 (column rename)
should all be addressed before the probe is first invoked against the
real corpus, so the first operator who runs it is not misled. Item 3
(constant deduplication) is low-priority polish.
