# Feedback: SPIRE Algorithmic and Efficiency Review

**Reviewer:** outside-review
**Date:** 2026-05-15
**Verdict:** No new local TLS blocker found, but AWS testing should treat the
first results as a profiling pass until the remote-read hot path is
instrumented. Several correctness-safe layers add repeated planning,
connection, fingerprint, and materialization costs that can dominate
cross-node latency.

## Scope

I reviewed the Phase 13c packet with an algorithmic/efficiency focus across the
production CustomScan read path, remote transport phases, read schema-drift
guard, candidate/heap merge logic, payload materialization, and residual test
coverage. This was a static review only; I did not run another smoke or
benchmark.

## Findings

1. **P1 - The production read path recomputes planning state and performs
   diagnostics by executing the expensive path.**

   `remote_search_production_scan_heap_resolution_result_stream` builds the
   local snapshot/object-store view, counts routable leaves, and selects PIDs
   (`src/am/ec_spire/coordinator/remote_candidates/scan_output.rs:430-461`),
   then calls `remote_search_execution_summary_row` with cloned query/PID state
   (`scan_output.rs:464-473`), then separately calls the local heap path
   (`scan_output.rs:503-516`), then calls
   `remote_search_libpq_dispatch_plan_rows` with cloned query/PID state again
   (`scan_output.rs:525-534`), and only then builds the production executor
   and runs candidate plus heap receive (`scan_output.rs:537-554`).

   The operator diagnostic row also calls
   `remote_search_production_scan_heap_resolution_result_stream`
   (`scan_output.rs:744-746`), so a diagnostic read can run the full production
   heap-resolution path instead of reporting a cheap snapshot of readiness and
   blockers.

   This is correctness-friendly, but it means AWS latency will include repeated
   manifest/snapshot/request-plan derivation and possibly full remote execution
   from diagnostic probes. I would expect this to distort small `top_k` reads
   and make the first benchmark harder to interpret.

   **Recommendation:** introduce a per-query production read context that
   carries the selected leaves, target/readiness rows, descriptors, local heap
   rows, dispatch rows, and derived limits through the whole read. Keep
   diagnostics split into a cheap readiness snapshot and an explicit
   "execute-and-explain" surface when full execution is intended.

2. **P1 - Candidate receive and heap receive multiply remote connection,
   identity, and secret-lookup work per node.**

   Candidate receive request construction resolves the conninfo secret and
   clones query/PID state for each planned dispatch
   (`src/am/ec_spire/coordinator/remote_candidates/production_transport.rs:732-769`).
   Heap receive construction repeats the secret lookup and clones the same
   query/PID state again (`production_transport.rs:787-835`). The adapter then
   builds a fresh current-thread Tokio runtime for candidate receive
   (`src/am/ec_spire/coordinator/remote_candidates/dispatch.rs:394-413`) and
   another for heap receive (`dispatch.rs:425-443`).

   Each phase opens its own async remote connection (`dispatch.rs:656-677` and
   `dispatch.rs:893-914`) and repeats `to_regclass` plus endpoint identity
   validation (`dispatch.rs:695-726` and `dispatch.rs:932-963`). With TLS
   enabled, this becomes two TCP/TLS handshakes and two remote identity probe
   sequences per participating node for a tuple-payload CustomScan read.

   This is likely the largest AWS-read bottleneck after network latency itself.
   It is especially visible for small and medium `top_k` where session setup
   dominates actual vector/heap work.

   **Recommendation:** either combine candidate and heap payload resolution into
   one remote endpoint for the production CustomScan read, or keep a
   per-dispatch session alive through both phases and carry the validated
   `remote_index_oid` and endpoint identity forward. At minimum, add counters
   for secret lookups, connection opens, TLS handshakes, `to_regclass` probes,
   and endpoint identity probes to the AWS artifacts.

3. **P2 - The read schema-drift guard adds synchronous per-node fingerprint
   probes before the async dispatch path.**

   Connection plan construction loads descriptors for selected nodes
   (`src/am/ec_spire/coordinator/remote_candidates/libpq_plan.rs:343-347`) and
   calls `remote_search_libpq_read_shape_status` for each request row
   (`libpq_plan.rs:349-397`). If a descriptor and secret exist, that function
   calls `validate_read_shape_fingerprints_before_remote_dispatch`
   (`libpq_plan.rs:400-429`).

   The validator recomputes the current coordinator fingerprint and opens a
   blocking remote connection to compute the current remote fingerprint
   (`src/am/ec_spire/coordinator/remote_candidates/write_payload.rs:437-492`).
   The remote side of that check opens a separate blocking libpq connection and
   runs `ec_spire_remote_index_shape_fingerprint(to_regclass($1))`
   (`write_payload.rs:339-371`).

   The guard is doing the right correctness work, but it sits in front of the
   async production transport and adds another live remote connection/probe per
   selected remote before candidate/heap receive opens its own connections.
   In degraded mode it can also turn a slow or failing fingerprint probe into a
   skipped dispatch before the normal transport path is reached.

   **Recommendation:** measure fingerprint probe count and time separately in
   Phase 13b. For the next optimization slice, cache the current coordinator
   fingerprint per query and fold remote fingerprint validation into the same
   remote session used for candidate/heap receive, or rely on descriptor
   fingerprints until a descriptor TTL/epoch invalidation requires a refresh.

4. **P2 - Candidate and heap merges fully materialize, hash, and sort instead
   of maintaining a bounded top-k structure.**

   Compact candidate merge dedupes with `HashMap<Vec<u8>, Candidate>`, then
   collects all deduped rows into a vector, sorts the whole vector, and only
   then truncates to the requested limit
   (`src/am/ec_spire/coordinator/remote_candidates/pipeline.rs:378-425`).
   Heap merge does the same pattern for local plus remote heap rows
   (`src/am/ec_spire/coordinator/hierarchy_snapshots.rs:938-963`), and the
   production scan materializes local rows plus remote rows before that merge
   (`scan_output.rs:563-568`).

   The current payload caps keep the common case small, but the algorithmic
   shape is still `O(N log N)` plus allocation of dedupe keys for all candidate
   rows. With high fanout, high `nprobe`, many duplicates, or a future larger
   rows-per-batch setting, this becomes avoidable CPU and memory pressure.

   **Recommendation:** move toward a streaming/bounded merge: dedupe by stable
   vec-id key, keep only the best row per key, and maintain a bounded heap or
   selection structure sized near `top_k` plus a duplicate allowance. Add a
   stress fixture with many remote nodes and duplicate vec_ids to measure merge
   CPU/memory separately from network time.

5. **P2 - The default governance posture is still unbounded for AWS unless the
   runbook sets explicit GUCs.**

   Remote search node/PID/concurrency limits default to `0`/unset
   (`src/am/ec_spire/options/mod.rs:36-40` and `options/mod.rs:63-76`). Payload
   rows default to 64 and payload bytes per row default to 1024
   (`options/mod.rs:41-44`, `options/mod.rs:77-80`), while payload limit
   enforcement fails a batch once rows or decoded bytes exceed the current
   settings
   (`src/am/ec_spire/coordinator/remote_candidates/payload_limits.rs:9-49`).

   That means the AWS runbook must choose explicit fanout/concurrency/timeouts
   before any serious benchmark. Otherwise one high-`nprobe` query can create a
   large number of remote dispatches with no default timeout, while real tuple
   projections can fail at the 1024-byte per-row default unless the test schema
   stays intentionally narrow.

   **Recommendation:** make the AWS packet record the active values for
   `max_nodes`, `max_pids`, `max_pids_per_node`, both concurrency GUCs,
   connect/statement timeout, and payload caps. Treat benchmark results without
   those GUCs as smoke-only.

6. **P3 - Typed tuple transport is binary at the PostgreSQL receive boundary,
   but still hex-encodes payload bytes over the coordinator/remote connection.**

   The production tuple payload SQL encodes every bytea payload value to hex
   text before returning it over libpq
   (`src/am/ec_spire/coordinator/remote_candidates/libpq_plan.rs:5-14`). The
   coordinator validates decoded byte size from hex length and decodes each
   value into a `Vec<u8>`
   (`src/am/ec_spire/coordinator/remote_candidates/payload.rs:200-214`,
   `payload_limits.rs:51-66`). CustomScan then copies each binary value again
   into a new `Vec<u8>` before calling the PostgreSQL receive function
   (`src/am/ec_spire/custom_scan/tuple_payload.rs:261-290`).

   This is much safer than JSON for type fidelity, but it roughly doubles wire
   bytes for payload values and adds coordinator CPU for hex decode plus the
   final copy. The payload byte limit counts decoded bytes, not wire bytes, so
   observed network traffic can be materially higher than the cap suggests.

   **Recommendation:** keep this for AWS smoke, but include payload width and
   projected-column-count sweeps in Phase 13b. Longer term, decode bytea arrays
   without hex text or expose a row-wise binary payload shape that avoids the
   hex expansion.

7. **P3 - DML/PK SELECT remains on the older JSON/text conversion path and has
   narrower expression support.**

   DML UPDATE accepts only constant or parameter SET values in v1
   (`src/am/ec_spire/custom_scan/dml.rs:580-604`). PK SELECT calls
   `ec_spire_forward_coordinator_select_tuple_payload` and stores
   `tuple_payload_json` (`custom_scan/dml.rs:657-699`), while the JSON payload
   materializer parses the full JSON object and runs type input conversion per
   attribute (`src/am/ec_spire/custom_scan/tuple_payload.rs:35-97`).

   This is not a Phase 13c regression, but it is a remaining edge area: the
   vector CustomScan tuple path is typed/binary, while PK SELECT and parts of
   DML still depend on text output/input behavior. Arrays, composites, domains,
   collations, bytea-like values, and nontrivial UPDATE expressions should not
   be assumed equivalent to the typed read path until covered.

   **Recommendation:** keep AWS read testing focused on the typed CustomScan
   path, and track DML/PK SELECT typed payload parity as a separate follow-up
   before treating coordinator-routed DML as equally hardened.

## Untested / Measurement Gaps

- Full SPIRE candidate/heap descriptor execution over TLS is still not covered
  locally. The Phase 13c TLS fixture proves the shared helpers and async
  transport probe, but the packet explicitly notes it does not run the full
  descriptor-backed candidate/heap path over TLS.
- Direct `cargo pgrx test` execution remains blocked by the local
  `BufferBlocks` loader issue, so PK SELECT drift coverage comes from the
  installed-extension SQL smoke rather than the direct pg_test harness.
- There is no high-fanout benchmark yet that separates connection setup,
  fingerprint probes, endpoint identity probes, remote search time, heap
  payload time, and merge time.
- Cancellation over TLS has not been proven to leave no remote backend work
  behind. On local cancel, `run_query_with_optional_local_cancel` sends cancel
  and returns immediately (`dispatch.rs:1280-1299`); it does not wait for the
  remote query future to observe cancellation before the caller aborts the
  connection task.
- Payload stress coverage should include wider projected rows, rows above the
  1024-byte decoded default, rows above the 64-row batch default, and mixed
  local/remote duplicate vec_ids under high fanout.

## Suggested Next Step

Proceed to AWS only as an instrumented profiling pass. Before drawing
performance conclusions, add artifact fields or log lines for per-query counts
and timings of planning, descriptor/fingerprint probes, secret lookups,
connection opens, TLS mode, endpoint identity probes, candidate receive, heap
receive, payload decode/materialization, and merge.
