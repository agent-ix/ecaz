## Scope match

All three SQL surfaces (histogram / ef_sweep / failure_breakdown) match the
task spec signatures and column shapes. Each builds the recall context once
per call via `build_external_recall_context`, consistent with the task's
"thin wrapper" design note. `ExternalRecallContext` itself is untouched.

Row-index vs corpus-id math is consistent: recall overlap runs on row
indices on both sides; `missed_ids` maps back to `corpus_ids[idx]` for
human output. `missed_ids = truth_top10 \ (graph_top10 ∪ exact_quantized_top10)`
matches both prompt and packet wording. Deterministic `query_index` ordering
in breakdown and always-11-row histogram are as specified. `SET LOCAL
tqhnsw.ef_search = {value}` is interpolated with a validated integer, safe
against injection.

## Findings

### 1. Task spec has a stale downstream reference

Task prompt L148 says "Latency benchmarking (covered by task 10057)."
Packet L248 says "task 10056". Task 10056 is
`nfr-001-latency-real-corpus` — the packet is correct; the **task spec
has a stale reference** that should be fixed in a follow-up. Not a landing
blocker.

### 2. Scan path differs from the summary probe, contrary to the prompt

Task L55-57 says "run the graph scan once (same path as the existing
summary probe)." The histogram and breakdown surfaces actually call
`am::debug_gettuple_scan_heap_tids` directly, not
`probe_graph_scan_recall_external_summary_for_context`. The packet
justifies this ("no scores needed") but it is a deviation.

The concern: the recall number the histogram reports could in principle
diverge from the sweep/summary number if the debug scan path and the
production scan path are not byte-identical. Please confirm in writing
(packet update or follow-up feedback) that `debug_gettuple_scan_heap_tids`
runs the exact same graph traversal as the production scan path, or switch
the histogram and breakdown to delegate to the summary probe for
consistency.

### 3. No pg_test for the new surfaces themselves

The packet leans on the existing `pg_test_tqhnsw_graph_scan_recall_gate`
to exercise SQL entity installation. That only proves the new surfaces
compile and register — it does not catch bucket off-by-ones in the
histogram, sort-order drift in the breakdown, or threshold filter errors.
A minimal synthetic-fixture test for each of the three surfaces would be
cheap insurance and would match the rigour applied to the encode-path
tasks (10061 / 10062) in this same batch.

### 4. Manual smoke recording deferred, contrary to the task prompt

Task L177-178 explicitly says "Record the observed outputs in the review
packet so coder-1 can use them against the 50k fixture without
rediscovering the invocation." The packet defers this to coder-1 instead.
Defensible given the scratch-cluster preconditions, but it is a scoped-down
delivery vs. the task ask. Please acknowledge this deferral explicitly
when closing the task.

### 5. Minor: unused `m` parameter in two of three surfaces

`let _ = m;` in histogram and failure_breakdown preserves signature parity
across all four recall surfaces. Packet documents this; acceptable as-is.

## Verdict

**Approve with follow-ups.** Items 1 and 2 should be addressed before
closing the task — #1 is a one-line spec edit, #2 is a one-paragraph
confirmation (or a small refactor). Items 3 and 4 can land as separate
follow-ups; they do not block this branch.
