## Findings

### 1. **BLOCKER — `SET LOCAL tqhnsw.ef_search` may be a no-op**

This is the only item I am not confident is correct, and it silently
invalidates every measurement the script will ever produce.

The per-query loop does:

```bash
while IFS= read -r query_line; do
  "$PSQL_BIN" -X -A -t -q <<SQL >> "$results_file"
SET LOCAL tqhnsw.ef_search = ${ef};
EXPLAIN (ANALYZE, TIMING, FORMAT JSON)
SELECT id FROM ${corpus_table} ORDER BY embedding <#> '${query_line}'::real[]
LIMIT ${k};
SQL
done < "$queries_tsv"
```

`SET LOCAL` only has effect inside an open transaction. psql sends
statements from a heredoc one at a time over the simple-query protocol;
each statement runs in its own implicit transaction and auto-commits.
Depending on exactly how psql batches the heredoc, one of two things
happens:

- Each statement is its own simple-query: `SET LOCAL` runs in tx1 and
  auto-commits (a no-op, possibly with a warning), then `EXPLAIN` runs in
  tx2 with the default `ef_search`.
- psql bundles both into one simple-query message: they share an implicit
  transaction and `SET LOCAL` takes effect for the `EXPLAIN`.

I am not confident which path psql takes for multi-statement heredoc
input. Either way, the current script does not provide an explicit
`BEGIN`, so the behavior is implementation-defined.

**Impact if the first path is what actually happens:** every
`(m, ef_search)` cell runs with the server's default ef_search, every cell
reports near-identical latency, and the entire sweep is silently wrong.

**Fix (pick one, whichever you prefer):**
1. Use session-level `SET` instead of `SET LOCAL` — just drop `LOCAL`.
   The setting persists until the next `SET` at the top of the next cell,
   which is exactly what we want. Simplest fix.
2. Wrap in an explicit transaction: `BEGIN; SET LOCAL ...; EXPLAIN ...;
   COMMIT;` Keeps `LOCAL` scoping intent but adds three lines.

**Empirical check before landing any measurements:** run two cells with
wildly different ef_search values (say 40 and 500) against the same
fixture and confirm p50 differs materially. If p50 is identical, the bug
is real.

### 2. Per-query psql spawn dominates the `qps` metric

Each of the ~10k queries × 2 m values × 6 ef_search values = 120k queries
launches a fresh `psql` process. The measured `wall_seconds / n` is
dominated by process spawn overhead, not Postgres throughput. The per-query
`Execution Time` pulled from EXPLAIN JSON is fine — it's server-side and
unaffected. But the reported `qps` column in the output line is
misleading: it says "this is how fast the server can answer," when it
actually says "this is how fast we can launch psql processes."

Fix: either drop the `qps` column from real-corpus mode, or replace it
with a computed `server_qps = n / sum(Execution Time)` derived from the
per-query JSON timings (which is the real per-server rate). Document the
caveat in `docs/RECALL_REAL_CORPUS.md` under the NFR-001 section.

A lower-overhead approach (single psql session reading queries from a
prepared-statement loop) is out of scope per the "minimal diff" design
note, but worth noting as a future optimization.

### 3. Query-line SQL interpolation is safe today, fragile tomorrow

`'${query_line}'::real[]` relies on Postgres's `real[]` unaligned
tuples-only output format containing no single quotes. That holds for the
current `source` column type. If anyone ever changes the column type or
the psql output format, this breaks silently. A defensive check at the
read-query-tsv step ("no line contains a single quote") would catch the
regression at bench-time rather than corrupting SQL.

Low priority — flag as a latent fragility.

### 4. Environment details missing from script output

Task L76-77 asks for GUC snapshot + host CPU/RAM in the recorded output
(`SHOW shared_buffers`, `SHOW max_parallel_workers_per_gather`, host CPU
model, physical RAM, warm/cold cache). The packet defers capturing these
to coder-1 operator discipline, but the script itself emits nothing — the
header banner stops at database name and corpus tables.

A small `print_env_banner` block at the top of `run_real_corpus_bench`
would make the recorded output self-describing. Commands to include:
```bash
psql -c "SHOW shared_buffers"
psql -c "SHOW work_mem"
psql -c "SHOW max_parallel_workers_per_gather"
uname -a
grep "model name" /proc/cpuinfo | head -1
grep MemTotal /proc/meminfo
```

Nice-to-have but saves coder-1 from remembering.

### 5. `--help` example in the docstring doesn't mention `--output`

Minor. The usage banner in `print_help` mentions `--output`, but the
file-header block comment only shows the basic invocation. Update the
header comment to include the full `--output /tmp/nfr1_real_10k.txt`
pattern so operators don't have to re-derive the canonical command.
