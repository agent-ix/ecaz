# FR-005: Inner Product Distance Functions

## Requirement

The extension SHALL provide functions for computing the approximate inner product between `tqvector` values, using two scoring paths optimized for different use cases.

### Functions

#### `tqvector_inner_product(a tqvector, b tqvector) → float4`

- SHALL return the approximate inner product estimate between `a` and `b`
- SHALL reject mismatched `dim` or `bits` with ERROR
- SHALL be `IMMUTABLE`, `STRICT`, `PARALLEL SAFE`
- SHALL use the symmetric code-to-code scoring path defined below

#### `tqvector_negative_inner_product(a tqvector, b tqvector) → float4`

- SHALL return `-1 * tqvector_inner_product(a, b)`
- Exists for the `<#>` operator (ORDER BY ASC = highest similarity first)

### Scoring Path 1: Prepared-Query LUT (decoded-query to code)

Used by the HNSW scan implementation (FR-009) and MAY be used by future executor-integrated sequential-scan support. It is an internal prepared-query path, not the contract of the public SQL function.

#### LUT Structure

The lookup table has shape `[dim] × [num_centroids]` where `num_centroids = 2^(bits-1)`:

```
lut[i][c] = centroid[c] * decoded_query_rotated[i]
```

- `dim` = original persisted dimension
- `num_centroids` = number of MSE codebook centroids (e.g., 8 for 4-bit)
- Each entry is f32
- Memory footprint: `dim × num_centroids × 4` bytes (e.g., 1536 × 8 × 4 = 48 KB for 1536-dim 4-bit)

The prepared query state SHALL also retain:
- the query QJL bits
- the decoded query residual-norm estimate or equivalent scalar(s) required by the chosen estimator

#### LUT Scoring Algorithm

For each candidate code:
1. Unpack the candidate's MSE indices (bit-packed → array of CodeIndex)
2. For each dimension `i`, look up `lut[i][mse_index[i]]` — one table lookup per dimension
3. Sum all lookups → MSE inner product estimate
4. Apply QJL correction using the stored query-side correction state, the candidate's packed QJL bits, and the exact estimator defined in FR-015
5. Return `mse_ip + qjl_correction`

**O(dim) per candidate, zero heap allocation per scoring call.** The LUT is constructed once per query and reused for all candidates.

### Scoring Path 2: Code-to-Code

Used by the public SQL API and during HNSW runtime insert/maintenance where both sides are stored compressed codes.

#### Algorithm

`score_ip_encoded_lite` operates directly on packed bytes:
1. Unpack MSE indices from both codes
2. For each dimension, look up `centroid[idx_a] * centroid[idx_b]` from a pre-computed centroid product table
3. Sum → MSE inner product estimate
4. Apply QJL correction from both codes' packed QJL state using the exact estimator defined in FR-015
5. Return `mse_ip + qjl_correction`

Lower fidelity than the prepared-query path (centroid product quantization introduces additional discretization error and the query must remain compressed) but avoids query preparation and works symmetrically.

### Sequential Scan Usage

When PostgreSQL executes `ORDER BY tqvector_inner_product(...)` or the `<#>` operator without an index, evaluation is row-by-row through the SQL function contract above. This path is approximate and throughput-bound. The extension SHALL NOT rely on hidden scan-local state inside the immutable SQL function. Any future prepared-query reuse for sequential scan SHALL be introduced through an explicit executor integration or separate API.

## Acceptance Criteria

### FR-005-AC-1: Known-vector accuracy
Given two known 1536-dim vectors encoded at b=4, the inner product estimate SHALL achieve the error bounds defined for the chosen estimator in FR-015 and SHALL be benchmarked against true fp32 inner product. This acceptance criterion does not imply per-pair exactness.

### FR-005-AC-2: Dimension mismatch error
`tqvector_inner_product(v1536, v768)` SHALL raise ERROR containing "mismatch".

### FR-005-AC-3: Symmetry
`tqvector_inner_product(a, b)` SHALL equal `tqvector_inner_product(b, a)` for all valid inputs.

### FR-005-AC-4: Zero allocation in hot path
After prepared-query setup, `score_ip_encoded` SHALL not allocate heap memory (verified by benchmarking, no per-call Vec or Box).

### FR-005-AC-5: LUT memory footprint
The LUT for 1536-dim 4-bit SHALL occupy ≤ 48 KB.
