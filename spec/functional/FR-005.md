# FR-005: Inner Product Distance Function

## Requirement

The extension SHALL provide functions for computing the approximate inner product between two `tqvector` values.

### Functions

#### `tqvector_inner_product(a tqvector, b tqvector) → float4`

- SHALL return the TurboQuant asymmetric inner product estimate between `a` and `b`
- SHALL call the `turbo-quant` crate's scoring function
- SHALL reject mismatched `dim` or `bits` with ERROR
- SHALL be `IMMUTABLE`, `STRICT`, `PARALLEL SAFE`

#### `tqvector_negative_inner_product(a tqvector, b tqvector) → float4`

- SHALL return `-1 * tqvector_inner_product(a, b)`
- Exists for the `<#>` operator (ORDER BY ASC = highest similarity first)

### Code-to-Code Scoring (Critical Path)

This function is called on **every edge traversal** during HNSW graph search. Performance is critical.

**Preferred path**: `turbo-quant` exposes `inner_product_estimate(&[u8], &[u8])` — both sides encoded, no decode step. Use this directly.

**Fallback path**: if the crate only supports `inner_product_estimate(codes: &[u8], query: &[f32])`, one side must be decoded via `decode_approximate`. This adds latency to every scoring call.

**Decision**: validate the crate API before implementing FR-008 (HNSW AM). If code-to-code scoring is missing, contribute it upstream or fork.

### Sequential Scan Usage

For agents with < 500K memories, this function is also used as a plain ORDER BY expression (no index). The query planner evaluates it for every row in the table. Performance matters here too, but less critically than in the HNSW inner loop.

## Acceptance Criteria

### FR-005-AC-1: Known-vector accuracy
Given two known 4-dim vectors encoded at b=4, the inner product estimate SHALL be within 10% of the true fp32 inner product.

### FR-005-AC-2: Dimension mismatch error
`tqvector_inner_product(v1536, v768)` SHALL raise ERROR containing "mismatch".

### FR-005-AC-3: Symmetry
`tqvector_inner_product(a, b)` SHALL equal `tqvector_inner_product(b, a)` for all valid inputs.
