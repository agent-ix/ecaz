# FR-009: HNSW Index Access Method — Scan (Query)

## Requirement

The extension SHALL implement the scan callbacks for the `tqhnsw` access method: `ambeginscan`, `amrescan`, `amgettuple`, `amendscan`.

### `ambeginscan`

Initialize scan state:
- Allocate `TqScanState` containing a `BinaryHeap` for candidates and a `first` flag

### `amrescan`

Execute the HNSW search:
1. Extract the query `tqvector` from the `orderbys` array
2. Read the graph entry point from metadata page
3. Execute HNSW beam search using `hnsw_rs` with `ef_search` candidates
4. Score each candidate using `tqvector_inner_product` (FR-005)
5. Load scored candidates into `TqScanState.candidates` BinaryHeap

### `amgettuple`

Return the next result:
1. Pop the next-best candidate from the BinaryHeap
2. Set `scan->xs_heaptid` to the candidate's heap ctid
3. Return `true` if a candidate was returned, `false` when exhausted

### `amendscan`

Free `TqScanState` and all allocated memory.

### `amcostestimate`

Inform the planner:
- startup_cost = 0
- per-tuple cost based on ef_search (proportional to number of distance calculations)
- This enables the planner to choose between index scan and sequential scan

### Scan Parameter

| Parameter | Default | Description |
|---|---|---|
| `ef_search` | 40 | Query-time beam width. Higher = better recall, more distance calculations |

Settable via `SET tqhnsw.ef_search = 200;` (GUC).

## Acceptance Criteria

### FR-009-AC-1: Top-k results returned
`SELECT id FROM t ORDER BY col <#> $q LIMIT 10` SHALL return exactly 10 rows (given >= 10 rows in the table).

### FR-009-AC-2: Results ordered by distance
Results SHALL be ordered by ascending negative inner product (highest similarity first).

### FR-009-AC-3: Index scan used
EXPLAIN SHALL confirm an Index Scan using `tqhnsw`, not a sequential scan, when the index exists and ef_search is set.

### FR-009-AC-4: ef_search affects recall
Higher ef_search values SHALL produce higher recall at the cost of increased latency.
