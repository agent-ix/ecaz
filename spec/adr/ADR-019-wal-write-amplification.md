# ADR-019: WAL Write Amplification Mitigation Strategy

## Context

HNSW index inserts cause WAL (Write-Ahead Log) write amplification because each insert modifies multiple pages: the new node's element and neighbor tuples, plus the neighbor lists of m existing nodes (bidirectional edges). PostgreSQL records each page modification in the WAL for crash recovery and replication.

### Amplification Mechanics

For tqvector with m=8:

| Operation | Pages Modified | WAL per Page (worst case) | Total WAL |
|---|---|---|---|
| New element tuple | 1 | 8KB (full-page image) | 8KB |
| New neighbor tuple | 1 | 8KB | 8KB |
| Update m neighbor lists | up to 8 | 8KB each | 64KB |
| **Total worst case** | **10** | | **~80KB** |
| **Total best case** (warm pages) | 10 | ~200B diffs | **~2KB** |

The worst case occurs when pages haven't been modified since the last checkpoint (PostgreSQL writes a full-page image for the first modification after each checkpoint). The best case occurs when the modified pages have already been dirtied in the current checkpoint cycle.

**VectorChord's measurement**: 2KB vector → 20KB WAL (10x amplification). For tqvector's 783-byte vectors, the amplification ratio is higher (~25-100x) but absolute WAL bytes are comparable because the vector payload is smaller.

### Scale Thresholds — pgvector Empirical Data

| Scale (per index) | Insert Rate (pgvector, 1536-dim fp32) | WAL Concern |
|---|---|---|
| <100K rows | 300-500 inserts/sec | None |
| 100K-1M rows | 50-100 inserts/sec | Moderate — graph traversal during insert hits more pages |
| 1M-10M rows | 3-20 inserts/sec | **Severe** — random I/O dominates, WAL volume impacts replication |
| >10M rows | Build from scratch is faster | WAL + vacuum are the bottleneck |

**Important caveat**: these pgvector numbers are for 6KB fp32 vectors. tqvector's 783-byte vectors mean ~8x more elements per page, which improves locality and reduces the number of distinct pages touched per insert.

### tqvector's Partitioned Architecture

The agent memory table is partitioned by `HASH(agent_id)` into 16 partitions (StR-003). Each partition has its own HNSW index. The relevant scale is **per-partition**, not total:

| Scenario | Agents/Partition | Memories/Agent | Vectors/Partition | Insert Concern |
|---|---|---|---|---|
| Light | 6,250 | 100 | **625K** | No issue |
| Typical | 6,250 | 1K | **6.25M** | Moderate — low insert rate per partition is fine |
| Heavy | 6,250 | 10K | **62.5M** | Significant — bulk load pattern required |

The typical workload is low-rate continuous inserts (agents accumulating memories), not bulk ingestion. A system-wide rate of 100 inserts/sec distributes to ~6 inserts/sec per partition — well within HNSW's comfort zone even at 6.25M vectors.

### IVF Comparison

IVF eliminates write amplification because inserts only append to one posting list:

| Metric | HNSW | IVF |
|---|---|---|
| Pages modified per insert | ~10 (element + neighbors) | **1** (posting list) |
| WAL per insert (worst case) | ~80KB | **~8KB** |
| WAL per insert (best case) | ~2KB | **~200B** |
| Insert CPU cost | O(m log n) graph search | O(k) centroid comparison |
| Incremental quality | Graph adapts per-vector | Centroids go stale |

However, IVF's centroid staleness is a worse problem for tqvector's heterogeneous data shapes (ADR-017). The WAL amplification tradeoff is acceptable.

## Decision

**Accept HNSW's WAL write amplification** as a known cost. Mitigate via PostgreSQL configuration and operational patterns rather than changing the index structure.

### Mitigation Strategies (ordered by impact)

| Strategy | Effect | When to Apply |
|---|---|---|
| `wal_compression = zstd` (PG15+) | Compresses full-page WAL images, ~3-5x reduction in WAL volume | Always — zero-downside configuration |
| Bulk load: drop index → COPY → CREATE INDEX | Eliminates per-row WAL amplification entirely | Initial loads, large migrations |
| `checkpoint_timeout` increase (e.g., 15min → 30min) | Reduces frequency of full-page images | High-insert-rate workloads |
| More hash partitions (32 or 64) | Keeps per-partition indexes smaller, better page locality | Scaling beyond typical scenario |
| `full_page_writes = off` on replicas | Replicas don't need full-page images (they replay from primary) | Replication bandwidth constrained |
| `wal_level = minimal` for bulk loads | Skips WAL entirely during non-replicated bulk operations | Offline bulk loads only |

### Monitoring

Track WAL amplification via:
```sql
-- WAL bytes generated by the tqhnsw index vs heap
SELECT schemaname, relname, 
       pg_stat_get_wal_bytes(relid) as wal_bytes
FROM pg_stat_user_tables 
WHERE relname LIKE '%memories%';
```

Alert if WAL generation rate exceeds expected thresholds based on insert rate.

## Consequences

### Positive
- No architectural changes needed — configuration-only mitigations
- `wal_compression = zstd` provides immediate 3-5x WAL reduction with no code changes
- Partitioned architecture naturally bounds per-index WAL amplification
- Bulk load pattern (FR-021 parallel build) avoids the problem entirely for large loads

### Negative
- Replication bandwidth is 10-25x higher than the raw vector insert volume
- High-insert-rate workloads (>100/sec per partition) will see elevated WAL I/O
- `checkpoint_timeout` increase delays crash recovery window

### Neutral
- WAL amplification is inherent to any graph-based index in PostgreSQL — pgvector has the same issue
- Future PostgreSQL versions may optimize WAL for index updates (e.g., logical WAL for index changes)
