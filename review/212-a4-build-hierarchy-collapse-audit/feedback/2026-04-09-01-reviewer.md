# Feedback: A4 Build Hierarchy Collapse Audit

Request:
- `review/212-a4-build-hierarchy-collapse-audit/request.md`

**Reviewer:** Claude (Opus)
**Date:** 2026-04-09

## Verdict: This Is Likely The Dominant Remaining A4 Bug

`top_level_node_count = 10000` is decisive. The persisted graph had no upper hierarchy at all.
Every node was at level 0. The HNSW structure was operating as a single-layer navigable small
world graph, not a multi-layer hierarchy.

### The build bug

The fix is mechanically correct. `get_layer_iterator(0)` in hnsw_rs iterates only nodes whose
layer-0 neighborhood is populated — which is all nodes, but it only exposes them as layer-0
participants. The actual level assignment (from hnsw_rs's exponential distribution) is available
via `get_point_id().0` on each point, but the layer-0 iterator doesn't surface higher-layer
neighborhood data.

By switching to iterating the full point indexation, the build now sees each point at its
assigned level and can extract neighborhoods for all layers, not just layer 0.

### What this explains retroactively

This reframes most of the Phase 3 evidence:

1. **Review 211 oracle-K**: "Score all nodes at `metadata.max_level`" was scoring ALL 10,000
   nodes because `max_level` was effectively 0. The oracle wasn't finding good *upper-layer
   entry points* — it was doing a brute-force scan of the entire index and picking the best
   seeds. That's why k=10 fully recovered exact quantized recall: 10 good seeds from a
   brute-force scan of 10k nodes is trivially achievable.

2. **Review 211 multi-seed carrydown failure**: The "upper-layer result window" was a layer-0
   beam search from the stored entry point — there were no upper layers to search. Widening
   the beam at layer 0 from a single entry point doesn't help much because layer-0 beam search
   is exactly what the live path already does. This failure was expected on a collapsed hierarchy
   but was misinterpreted as evidence that upper-layer search from the entry point is a dead end.

3. **Review 200 graph-at-91%-of-ceiling**: The graph was at 91% of a low ceiling. With a real
   hierarchy, the ceiling is higher (84.8% exact quantized after 4+0) AND the graph should
   track it better because greedy descent through upper layers provides much better layer-0 seed
   quality than starting from a single arbitrary entry point in a flat graph.

4. **The 25.8% live graph recall**: Makes complete sense for a flat single-layer graph with
   m=8 and ef_search=128 on 10k nodes. Without upper-layer navigation, the beam search starts
   from a single entry point and can only explore a small neighborhood. A proper HNSW hierarchy
   would descend through sparse upper layers to find a good starting region before layer-0
   search begins.

### What this does NOT explain

The Phase 2 quantizer findings remain valid and independent:

- Transform-tail truncation (review 201/203) — fixed by tiled FWHT
- QJL bit reallocation (review 207) — fixed by 4+0
- Codebook dimension sweeps (reviews 205/209) — cb1536 confirmed optimal

These findings raised the exact quantized ceiling from ~43% to ~84.8%. The hierarchy collapse
is why the live graph path couldn't track that ceiling.

### Impact on my review 211 feedback

My feedback on review 211 recommended three next steps:

1. **Multiple stored entry points** — likely unnecessary if the hierarchy fix works. A proper
   HNSW hierarchy with greedy descent should reach good layer-0 seeds without needing multiple
   entry points. Re-evaluate only if the hierarchy fix doesn't close the gap.

2. **Upper-layer reachability audit** — still useful as a validation tool on the fixed hierarchy.
   The coverage probe already built for 212 serves this purpose. Run it on the rebuilt graph to
   confirm the hierarchy is sparse and well-connected.

3. **Check whether 4+0 degraded graph construction** — still worth checking on the fixed
   hierarchy. The build distance function hasn't changed, but now it's building a real
   multi-layer graph instead of a flat one.

### Validation expectations for the rebuild

When the deterministic 10k fixture finishes rebuilding on the hierarchy-fix tree:

**Strong positive signals:**
- `top_level_node_count` should be much smaller than 10,000 (typical for m=8 on 10k: ~20-50
  nodes at the highest level, following the exponential distribution)
- `metadata.max_level` should be > 0 (typically 3-5 for m=8 on 10k)
- Live graph Recall@10 should jump substantially from 25.8%

**If live recall approaches 75-85%** (tracking the 84.8% exact quantized ceiling at 91%+), the
hierarchy collapse was the dominant remaining bug and A4 may be within range of the 89% gate
with parameter tuning (higher m or ef_search).

**If live recall improves but stays below 60%**, there's a secondary traversal issue on top of
the hierarchy fix, and the upper-layer connectivity / multi-entry work from review 211 feedback
becomes relevant again — but now on a correctly built hierarchy.

### Risk: the fix may expose new issues

With a real hierarchy:
- `choose_entry_point` will now select from actual high-level nodes instead of all 10k nodes
- `greedy_descend_from_entry` will now actually traverse upper layers instead of being a no-op
- `search_upper_layer_seed_candidates` will now search real upper-layer neighborhoods
- The `max_level_that_fits` cap in page.rs may truncate some levels if neighbor tuples don't
  fit on 8KB pages at high levels with m=8

These code paths existed but were effectively dead code on the collapsed hierarchy. They may
have latent bugs. The ordered-result regression test
(`test_tqhnsw_graph_first_scan_emits_distance_sorted_scores`) passing is a good sign, but it
uses a tiny corpus (8 rows, 4-dim) that may not exercise upper layers.

### On the draft status

This is correctly marked as a draft progress review. The fix is simple and the diagnosis is
convincing, but the evidence depends on the 10k rebuild completing and the recall numbers
landing. Do not merge the hierarchy fix without:

1. Full `cargo test`
2. Full `cargo pgrx test pg17`
3. `cargo clippy`
4. The 10k recall gate report showing the expected hierarchy structure
5. Confirmation that live recall improved materially

## Disposition

Strongly favorable. This is the most significant single finding since the layer-blind neighbor
reads in review 196. Both were build-time serialization bugs that collapsed the HNSW structure.
Awaiting rebuild results to confirm the expected recall improvement.
