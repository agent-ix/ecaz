# ADR-046 review answers for implementation

I reviewed the ADR-046 draft plus the later gap list on the
`adr034-diskann-access-method` branch. Two issues need to be fixed in
the ADR text before coder-2 should treat it as implementation-ready:

1. **Cold rerank payload handling is currently self-contradictory.**
   ADR-045 says `rerank_tid` is reserved and stays `INVALID` until
   ADR-044 C1 reopens. ADR-046 currently reads as if a cold rerank
   chain already exists on the insert path. For Phase 7, pick one.
   My recommendation is: **follow ADR-045**. V0 `ecdiskann` live insert
   does **not** write an index-side cold rerank payload. Keep
   `rerank_tid = INVALID` and leave hot/cold ordering for a future ADR
   when ADR-044 C1 is actually adopted.
2. **The write-window scoring rule must stay page-local.**
   `RobustPrune` under the page lock is acceptable only if all scoring
   inputs were already materialized during the read-only planning pass
   or live on the target tuple itself (ADR-045 Decision 3). Do not
   fetch remote candidate payloads while holding the target page's
   `EXCLUSIVE` lock.

## Answers

### G1 — overflow heaptid chain on insert

ADR-046 **does** govern this path. Treat duplicate-vector binding to an
existing node as an ADR-046 data-page mutation, not as a separate ADR.

- If the existing node can absorb the new heap TID without growing an
  overflow chain, this is a normal **step-5 ordered rewrite** of the
  existing tuple's page.
- If the chain must grow, use the same split ADR-046 already uses for a
  new graph node:
  1. append the fresh overflow tuple under one isolated page
     `EXCLUSIVE` lock and release it immediately
  2. patch the head / predecessor tuple in the later ordered page pass
     from low block to high block

That keeps the lock story coherent: append-like allocation first,
ordered rewrites second, metadata last.

### G2 — cold rerank payload chain

For the implementation coder-2 should write now: **out of scope**.

- `rerank_tid` remains `INVALID`
- `PAYLOAD_FLAG_COLD_RERANK_PAYLOAD` should not drive any insert-time
  page writes in Phase 7
- ADR-046 should say explicitly that live insert mutates only the hot
  graph pages plus metadata in V0

If ADR-044 C1 lands later, extend ADR-046 at that time with a hot-then-
cold append rule for the same logical node.

### G3 — stale-target retry cap

Use the same cap as ADR-026's landed HNSW insert path:

- **`MAX_BACKLINK_REPLAN_PASSES = 3` total ordered write passes per
  insert**

Do not invent a Vamana-specific number until measurements show the need.
The important property is not the exact integer; it is that retry is
bounded and the bound is named.

### G4 — `needs_medoid_refresh` ownership

**Insert should not set `needs_medoid_refresh`.**

Live insert owns:

- appending a new node or binding a duplicate heap TID
- backlink repair
- `inserted_since_rebuild += 1` for true new-node inserts

Live insert does **not** own medoid migration or medoid-refresh
decisions. The flag should be owned by:

- vacuum, when the current medoid becomes dead/tombstoned
- a future rebuild / maintenance decision that interprets
  `inserted_since_rebuild`

That keeps the flag monotonic and avoids two independent writers to the
same metadata decision bit.

### G5 — insert/vacuum concurrency invariant

State this explicitly:

> `ecdiskann` insert and vacuum are allowed to run concurrently. They do
> not serialize through a global AM lock; they tolerate each other's
> page-local drift through bounded stale-target retry between ordered
> write passes.

That sentence belongs in ADR-046 and ADR-047 both.

### G6 — forward reference to ADR-045

Add it. Step 6 should cite ADR-045 Decision 3 directly for the
"tuple is self-sufficient for prune scoring" claim.

## Additional implementation answers

These were open in the original ADR-046 packet and should be frozen now:

- **Read-only planning scorer:** keep this below ADR level. The ADR
  should constrain lock behavior, not bind the exact prefilter/rerank
  stack. One sentence is enough: planning may use any scorer that
  acquires only read-side locks and materializes all write-window
  inputs before the `EXCLUSIVE` phase.
- **Retry payload shape:** `target_element_tid` plus the new-node
  backlink payload is sufficient. Do **not** carry a neighbor-slice
  snapshot out of the write phase.
- **Deadlock proof target:** the proof obligation is "ascending physical
  page order, one data-page `EXCLUSIVE` at a time, metadata last" even
  when insert races vacuum. No extra generation counter is needed if
  stale detection compares the reopened tuple contents against the
  read-only plan.

## Recommendation

I would not flip ADR-046 to `ACCEPTED` until the ADR text incorporates:

1. explicit overflow-chain treatment under the ordered-page-pass rule
2. explicit statement that V0 insert has **no** cold rerank page writes
3. a named retry cap of `3`
4. explicit ownership: insert does not set `needs_medoid_refresh`
5. an explicit insert/vacuum concurrency sentence
