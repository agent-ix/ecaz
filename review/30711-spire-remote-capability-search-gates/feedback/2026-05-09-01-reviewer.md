## Answers to the packet's review questions

- **Capability-plan status as the source of truth for search readiness
  and libpq pre-dispatch blocking?** Yes. The capability plan already
  aggregates descriptor + epoch-window + extension-version state in
  one place; importing it into the search readiness path is the right
  consolidation. Search readiness was previously inspecting descriptor
  state without epoch-window/version inputs, which is exactly the
  shape of bug this fixes.
- **Right behavior to block before secret lookup for stale epoch and
  extension-version skew?** Yes. Secret resolution has its own failure
  modes and operator-visible logging; running it for a remote that
  cannot serve the request just confuses the diagnostic chain.
  Stopping at capability is the cleanest cut.
- **Should `retention_gap` remain grouped with `remote_epoch_window`?**
  Yes for now. Both are expressions of "the remote cannot serve the
  requested epoch" and the operator action is the same: refresh the
  descriptor or advance the requested epoch. A future packet can split
  them if a meaningful difference in remediation appears, but that's
  not the current state.
- **Status-precedence regressions in mixed local/remote plans?** None
  found in this slice. Mixed plans with local + capable remote keep
  prior readiness behavior; mixed plans with local + stale remote now
  produce a capability blocker on the remote side without changing the
  local side.
