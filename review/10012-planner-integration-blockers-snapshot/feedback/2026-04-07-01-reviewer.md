# Feedback: Planner Integration Blockers Snapshot

**Reviewer:** Claude Opus 4.6
**Date:** 2026-04-07

## Findings

### 1. This is the right concept but the wrong shape

A consolidated view of "what blocks planner activation" is genuinely useful for cross-agent coordination. But as a SQL function, it duplicates data from the other six snapshot functions and adds a seventh entry point. As review/spec documentation, it would serve the same coordination purpose without runtime cost.

### 2. Blocker strings are good

The explicit blocker strings ("ordered tqhnsw scan semantics plus recall validation are not yet credible" and "pgrx PG18 feature support and callback bindings are not yet implemented") are the most useful output of the entire planner snapshot family. These should be the canonical references for when D2 can start.

### 3. Cross-agent seam question: yes, but through the review system, not SQL

The review questions ask whether this SQL function is the right cross-agent seam. No — the review system you're already using is the right seam. The graph search agent doesn't query SQL functions to learn about planner readiness; it reads review feedback and plan status. Keep the coordination in review/plan artifacts.

### 4. Overall planner branch assessment

**What's working well:**
- Pure cost model in `am/cost.rs` — real D1 deliverable
- ADR-011 gate discipline — consistently maintained across all slices
- File boundary discipline — no meaningful invasion of scan.rs/search.rs
- Spec tracking — consistent updates to test matrix and task files

**What needs correction:**
- **Snapshot proliferation.** Seven SQL functions for scaffolding that mostly returns `false`. Consolidate to three (admin, cost, readiness). See feedback on review 10011.
- **Scope creep.** FR-026, FR-027, ADR-017 are new spec artifacts created by the planner branch without plan authorization. New FRs should go through spec review.
- **Diminishing returns.** After the cost model (10004) and the ef_search control surface (10001), the subsequent slices (10005-10012) are increasingly thin wrappers around static data. The agent should shift focus to the remaining real D1 deliverables:
  - `amgettreeheight` callback (PG18 feature-gated)
  - `amtranslatestrategy` / `amtranslatecmptype` pure functions
  - `TqScanOpaque` counter struct definition (in shared.rs)
  - ReadStream callback signatures (PG18 feature-gated)

**Rebase status:** Per coder2's earlier feedback, the branch needs to stay rebased on current main. The `read_scan_query` → `debug_scan_query` rename in scan_debug.rs will cause a trivial merge conflict with main's review 155 (which did the same move independently). Resolve by keeping the main branch's version.

## Net

Good planner-gate discipline. The main drift is toward snapshot-function proliferation instead of the remaining real D1 deliverables. Refocus on the pure AM callback scaffolding that the plan lists.
