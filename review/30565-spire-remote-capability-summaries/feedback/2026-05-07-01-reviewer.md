# Remote capability summaries review

`ec_spire_remote_node_capability_summary` and
`ec_spire_remote_epoch_publish_readiness` sit at the right level of
abstraction: aggregated counts gated on descriptor presence, no
implication that libpq has landed. No blocking findings.

## Confirmations

1. Capability summary aggregates per-node capability-plan rows
   without re-implementing the readiness logic — it consumes the
   30564 plan and counts ready/blocked nodes plus required candidate
   format and extension version.
2. Epoch publish readiness is appropriately conservative: any
   nonzero node remains blocked as
   `requires_remote_node_descriptor` until durable descriptor
   metadata exists. That matches the 30552 design boundary.
3. Local-only behavior: `status = 'ready'`, required candidate
   format `'local'`, remote/blocked counts zero.
4. No raw conninfo or libpq execution behavior exposed.

## Findings (non-blocking)

- **Duplicated counting logic across summaries.** Both
  `remote_node_capability_summary` and the readiness/execution/merge
  summaries in 30560/30563/30566/30568 each compute their own
  ready/blocked/local/remote/skipped count rollup. The fields are
  named consistently, but the per-summary construction logic is
  rewritten each time. Consider extracting a
  `SpireRemoteCountAggregator` helper struct used by every summary
  function. Cross-cutting summary on 30568.
