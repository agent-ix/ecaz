# Review: Task 42 qemu cross-build fix (9056)

## Summary

Follow-up to the first pushed qemu lane run from 9055.
`make endian-qemu` now sets target-specific s390x rustflags so
the cross build doesn't inherit host `target-cpu=native`. CI
job installs PGDG PostgreSQL 18 headers and sets
`PGRX_PG_CONFIG_PATH` for pgrx build-script discovery. Still
runs the same big-endian `tests/on_disk_fixtures.rs` suite
through `qemu-s390x`.

## Confirmations

- **`target-cpu=native` was the right thing to override.**
  Inheriting `RUSTFLAGS='-C target-cpu=native'` from a x86_64
  host into an s390x cross-compile would generate invalid
  s390x machine code. Setting target-specific rustflags
  (rather than relying on inherited environment) is the
  standard cross-compile hygiene. ✔
- **pgrx build script needs PG headers at cross-compile time.**
  PGDG s390x package + `PGRX_PG_CONFIG_PATH` is the standard
  pgrx pattern for cross-compilation. Right fix. ✔
- **Same test suite.** The fix is build infrastructure — the
  test surface (the `on_disk_fixtures.rs` suite) doesn't
  change, validating that the issue was pure cross-build
  tooling, not a test-code endian bug.

## Findings

### Expected first-run iteration

Setting up a qemu cross-arch CI lane almost always needs
build-tooling tuning on the first attempt — host environment
leaks, missing target headers, target-cpu mismatches. This
packet is exactly that iteration. Worth flagging in the qemu
lane doc that the s390x cross-build has these two requirements
(target rustflags + PGDG headers) so a future re-tune doesn't
repeat the discovery.
