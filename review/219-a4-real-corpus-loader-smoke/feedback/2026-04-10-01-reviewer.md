## What Was Good In 219

### The `TQV_PSQL_BIN` escape hatch is the right design

Pure env var, falls back to `shutil.which("psql")`, errors clearly if
neither is present. No `PATH` mangling required, and the fallback order
is least-surprising (explicit override > PATH discovery > hard failure).

### The stale-extension footgun is worth documenting

Section C of the 219 packet ("Stale same-version `pg_test` install in
the scratch DB") is the kind of trap that burns a half-hour per agent
if undocumented. The finding — reinstalling a same-version pg_test build
updates the SQL file on disk but does **not** refresh objects already
created in the database, forcing a
`DROP EXTENSION ... CASCADE; CREATE EXTENSION ...` — should be promoted
out of the review packet and into `docs/RECALL_REAL_CORPUS.md` as a
"Troubleshooting" section. Otherwise the next agent that hits this will
rediscover it from scratch.

### Transparency about failed attempts

The 219 packet's sections A / B / C honestly record three failed setup
attempts before the working run. That is exactly the right way to
document setup discovery — future agents can pattern-match against
those failures rather than spending time re-deriving them.

### Clippy drive-bys in `src/am/scan_debug.rs`

Three `redundant_closure` cleanups
(`.map(|h| debug_item_pointer_coords(h))` → `.map(debug_item_pointer_coords)`).
Unrelated to the A4 lane, but semantically no-op and silences real
clippy warnings. Acceptable as drive-by.

## Critical: The Smoke Output Reveals A New Concern

The 219 packet reports, for a 24-row / 6-query smoke fixture at
`m=8 ef=128`:

```
corpus_rows | query_count | graph_recall@10 | exact_quantized_recall@10 | ndcg@10    | mean_abs_score_error
24          | 6           | 0.93333334      | 0.93333334                | 0.99119234 | 8.020205
```

**`mean_abs_score_error = 8.02` is impossible if the inputs are
unit-normalized.** On unit vectors, inner products are bounded in
`[-1, 1]`, so any per-pair absolute difference is at most `2.0`, and a
mean of 10 such diffs cannot exceed `2.0`. An MAE of 8.02 means one of
two things:

- **(a) The smoke TSV files in `/tmp/tqhnsw_loader_smoke_{corpus,queries}.tsv`
  are not unit-normalized.** The file format allows this —
  `docs/RECALL_REAL_CORPUS.md:25` says "inner product / cosine on
  unit-normalized vectors" but nothing in the loader or probe enforces
  `‖v‖ ≈ 1` on ingest. For the Qdrant ada-002 corpus this happens to be
  fine because ada-002 emits pre-normalized vectors, but the moment
  anyone loads a non-ada embedding (including subsampled-and-perturbed
  test fixtures), the whole "IP = cosine" interpretation silently breaks
  and every downstream metric — recall, NDCG, MAE — is scoring against
  a different metric than the doc claims.
- **(b) The MAE computation is producing values outside its claimed
  range.** Which circles directly back to 218 feedback item 2: the
  column named `mean_abs_score_error` should be bounded, and this output
  proves it isn't.

Either way, this is empirical evidence strengthening 218 item 2, and it
opens a new finding:

### New finding (A): The loader should enforce or document normalization

The dataset contract promise in `RECALL_REAL_CORPUS.md:25` ("Distance:
inner product / cosine on unit-normalized vectors") is currently
unenforced. The 219 smoke packet just documented a case where it was
violated without anyone noticing. Choose one of:

- **Assert on ingest:** compute `‖v‖` per row in the loader and refuse
  to load if the mean norm diverges from 1 by more than a small
  tolerance. Hardest on the user but gives the strongest contract.
- **Optional `--normalize` flag:** loader L2-normalizes on the fly when
  requested, otherwise passes vectors through as-is. Good for
  flexibility, but requires the user to know which mode they need.
- **Warning-only:** log a warning when the mean norm diverges from 1,
  but still load. Weakest guarantee, but matches the current
  "dataset-agnostic file contract" framing.

At minimum the loader should *emit the mean norm at load time* so the
operator can tell at a glance whether the ingested data matches the
metric assumption. This is a ~5-line change.

The fact that the 24-row smoke shows 0.93 recall and 0.99 NDCG alongside
an 8.02 MAE is noteworthy by itself: the first two say "the path works"
and the third says "the metric math does not agree with its name". Both
readings of the 219 packet are compatible with that, and neither is
good.

## Portability / Consistency Nits

### Hardcoded path in `scripts/load_real_corpus_scratch.sh:10`

```bash
export TQV_PSQL_BIN="${TQV_PSQL_BIN:-/home/peter/.pgrx/17.9/pgrx-install/bin/psql}"
```

The default `TQV_PSQL_BIN` is `/home/peter/.pgrx/...` — that is a
specific user's home directory, not a portable default. It will work for
the current environment and any agent running in it, but will not work
for a reviewer on a different machine or a CI runner.

The existing sibling helper `scripts/pg17_scratch_psql.sh` (referenced
in the 219 packet as the precedent) likely has a similar shape — worth
checking whether it uses the same hardcoded path (in which case this is
consistent) or whether it resolves from `$PGRX_HOME` / a
`pg_config --bindir` call (in which case this helper should follow
suit).

Minimum fix: derive from
`${PGRX_HOME:-$HOME/.pgrx}/17.9/pgrx-install/bin/psql` so it respects
the standard pgrx home resolution. Not blocking on this commit, but it
is a latent maintenance cost.

### Database name inconsistency

`load_real_corpus_scratch.sh:9` defaults `PGDATABASE=postgres`, but
`docs/RECALL_REAL_CORPUS.md:130` shows the canonical example as
`PGDATABASE=tqvector_bench`. A reader following both will end up with
two different target databases. Pick one or explain why they differ
(e.g. "scratch uses `postgres` because the pgrx scratch cluster does
not provision `tqvector_bench`; production runs use `tqvector_bench`").

## Still Critical From 218: Items 1 And 2

The 219 smoke **does not** validate the credibility surface of the
gate, which is the whole point of review 217 → 218. Specifically:

- The 24-row smoke is not DBpedia. A gate report row of
  `0.93 Recall@10` on 24 random 1536-D vectors tells you nothing about
  whether the lane will meet NFR-003 on real embeddings.
- The subset selection rule is still undocumented. Whoever runs the
  first "official" DBpedia gate report will pick a 50k subset by
  whatever method is convenient at the time, and that number will
  become the de facto gate baseline even though it is not reproducible
  across humans.
- The MAE semantics question now has empirical evidence (8.02 on the
  smoke) that something is wrong with either the computation or the
  column name, but the code is unchanged.

These two items should land **before** anyone runs the official DBpedia
gate. Running the real gate now would burn the opportunity to pin a
reproducible baseline number.

## Still High From 218: Items 3 And 4 (Perf Refactor)

The 24-row smoke does not hit these — 4 redundant corpus reloads of 24
rows is imperceptible. But the real 50k gate will pay the full cost on
every run:

- ~77 GFLOPs of brute-force dot products recomputed 4 times per gate
  report (item 4)
- ~50M redundant HashMap inserts per gate run from the per-query
  `id_to_row_index_lookup` rebuild (item 3)

These should land before anyone runs the real gate, or the first run
will be slow enough that people stop re-running it for confirmation.
The refactor is a single mechanical change outlined in 218 feedback
item 4 (`ExternalRecallContext` struct, build once, reuse across the
four gate configs).

## Net Verdict

mode (psql discovery) and documents three real setup footguns
(sandboxed psql, blocked socket, stale extension). Addresses one of the
ten 218 items directly.

Does **not** advance the critical path on recording a reproducible
first DBpedia number. The lane's credibility-surface status is
unchanged from the end of review 218. The smoke output additionally
