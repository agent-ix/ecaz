# Standards Compliance Review

Reviewed current HEAD after the `31143` follow-up, using the local spec review,
matrix, and analysis-skill checklists. This is a compliance review only; no spec
edits were made.

## Findings

### P1 - Completion labels still conflict with the declared AC-level traceability gap

The master spec requires acceptance criteria to be traceable at AC granularity;
grouped ranges are allowed only as summaries when the section also records
individual evidence rows or an explicit gap ([spec/spec.md:322](../../../spec/spec.md#L322)).
The follow-up added `GAP-018`, which correctly says strict per-AC evidence is
missing and should be split out before claiming standards-complete 29148
traceability ([spec/tests.md:290](../../../spec/tests.md#L290)). However, the
same matrix still marks many grouped rows as `Complete` or `Implemented`, for
example `US-001`, `US-002`, `US-004`, `US-007`, `US-009`, and `US-010` while
also appending "strict per-AC evidence inventory remains GAP-018"
([spec/tests.md:65](../../../spec/tests.md#L65),
[spec/tests.md:74](../../../spec/tests.md#L74)). The FR table does the same at
range or requirement granularity: `FR-001..FR-006`, `FR-020`, `FR-022`,
`FR-023`, `FR-024`, `FR-026`, `FR-027`, `FR-028..FR-038`, and `FR-048` carry
`Complete` language without individual AC-to-evidence rows
([spec/tests.md:92](../../../spec/tests.md#L92),
[spec/tests.md:115](../../../spec/tests.md#L115)).

This makes the matrix internally inconsistent: it both claims completion and
admits that the required evidence inventory does not yet exist. For standards
alignment, keep grouped rows as summary/status rows, but downgrade completion
language to `Partial pending GAP-018` until each `*-AC-*` has either a concrete
TC/evidence row or an explicit per-AC gap.

### P1 - The architecture trace graph is still not dependable enough for the 42010/828 target

The master spec says structured `relationships:` frontmatter is required for
new or substantially rewritten artifacts, and legacy `traces:` artifacts are not
fully migrated until their dependency and verification links are represented
through structured relationships or the matrix ([spec/spec.md:316](../../../spec/spec.md#L316)).
Current HEAD still has broad active areas outside that structured graph. A local
inventory found 27 legacy `FR-001..FR-027` files missing both `artifact_type`
and `relationships`, plus `NFR-001..NFR-003`, `NFR-005`, `NFR-006`, and five
stakeholder artifacts missing structured relationships. Representative examples:
`FR-001` still uses `traces:` only ([spec/functional/FR-001-tqvector-data-type-registration.md:1](../../../spec/functional/FR-001-tqvector-data-type-registration.md#L1)),
and `StR-001` still uses `derived_usecases:` without `artifact_type` or
`relationships:` ([spec/stakeholder/StR-001-native-compressed-vector-storage.md:1](../../../spec/stakeholder/StR-001-native-compressed-vector-storage.md#L1)).

That is acceptable as legacy history, but not as a dependable ISO 42010
architecture/relationship graph or IEEE 828 configuration-control graph for the
whole current spec. Either make the standards matrix explicitly say the legacy
HNSW/core artifacts remain outside the fully migrated trace graph, or migrate
the active legacy artifacts before treating the whole spec as standards-aligned.

### P2 - Benchmark docs still contain non-conforming rows under an implemented reporting-standard claim

`NFR-015` says every candidate comparison row must identify access method,
opclass, storage/quantizer format, format-specific options, AM reloptions,
session GUC overrides, rerank mode, environment, dataset, command, artifact
packet, claim class, iteration count, cache state, and latency/quality/storage
fields as applicable ([spec/non-functional/NFR-015-benchmark-reporting-standard.md:24](../../../spec/non-functional/NFR-015-benchmark-reporting-standard.md#L24)).
The matrix marks `TC-033` implemented for the standard and `NFR-007` complete
for current docs/review-packet citations ([spec/tests.md:190](../../../spec/tests.md#L190),
[spec/tests.md:143](../../../spec/tests.md#L143)).

The actual benchmark tables still have legacy rows that do not carry enough
per-row identity/provenance to satisfy that schema. The HNSW baseline gives only
corpus/configuration/recall and no packet or command per row
([docs/benchmarks.md:19](../../../docs/benchmarks.md#L19)). The IVF 10K/25K
table omits opclass, GUC overrides, command, claim class, cache state, and
row-level artifact identity, then lists several source packets in aggregate
below the table ([docs/benchmarks.md:53](../../../docs/benchmarks.md#L53),
[docs/benchmarks.md:80](../../../docs/benchmarks.md#L80)). This is useful
engineering documentation, but it should not be described as complete under
NFR-015/15939 without either adding the missing fields or explicitly labeling
those rows as legacy/local summaries outside standards-complete comparisons.

### P2 - User-story acceptance criteria are ID-normalized but not scenario-normalized

The user-story skill expects informative story text plus Given/When/Then
acceptance scenarios. The follow-up normalized IDs, which fixes the largest
traceability problem, but the criteria are still mostly declarative statements
rather than scenarios. For example, `US-001-AC-1..4` are direct behavior
statements ([spec/usecase/US-001-store-compressed-embedding.md:20](../../../spec/usecase/US-001-store-compressed-embedding.md#L20)),
and `US-012-AC-1..3` are also direct behavior statements
([spec/usecase/US-012-store-and-query-ecvector.md:26](../../../spec/usecase/US-012-store-and-query-ecvector.md#L26)).

These can still be testable, so this is not a correctness blocker. It is a
standards/process gap: if the repo wants to claim local `spec-write-us`
conformance for the whole use-case set, the US criteria need scenario form or
the master spec should explicitly accept declarative US ACs as the Ecaz-local
variant.

### P3 - SPIRE typed transport still leaves some faithful-reimplementation fields underspecified

The storage formats are much stronger after the hardening pass, especially
Leaf V2 and routing/delta/top-graph byte layouts. The typed remote transport is
still looser than the storage specs. `FR-056` allows the query to be `real[]` or
"equivalent vector bytes" without defining the byte form
([spec/functional/spire/distributed/FR-056-spire-remote-endpoint-typed-transport.md:31](../../../spec/functional/spire/distributed/FR-056-spire-remote-endpoint-typed-transport.md#L31)),
allows `request_id` to be either `uuid` or opaque bytes
([spec/functional/spire/distributed/FR-056-spire-remote-endpoint-typed-transport.md:36](../../../spec/functional/spire/distributed/FR-056-spire-remote-endpoint-typed-transport.md#L36)),
and requires "stable" receive/fault/typed-transport status labels without
enumerating the label set or ownership of additions
([spec/functional/spire/distributed/FR-056-spire-remote-endpoint-typed-transport.md:51](../../../spec/functional/spire/distributed/FR-056-spire-remote-endpoint-typed-transport.md#L51),
[spec/functional/spire/distributed/FR-056-spire-remote-endpoint-typed-transport.md:69](../../../spec/functional/spire/distributed/FR-056-spire-remote-endpoint-typed-transport.md#L69)).

That is enough for an implementation family, but not quite enough for a
faithful independent reimplementation or durable wire compatibility. Before
format freeze, pin the accepted query encoding(s), request identity encoding,
and status vocabulary or add an explicit compatibility gap.

## Residual Risks

- The Task 34 hardening evidence language is materially better after `31143`;
  the remaining risk is that readers may still infer broader local hardening
  completion from `TC-034` unless per-lane rows are split.
- SPIRE storage byte layouts are close to reimplementation-ready, but the remote
  transport contract still has compatibility choices that could diverge across
  implementations.
- Legacy core/HNSW artifacts can continue to hide broken trace edges if future
  reviews rely on structured graph queries rather than reading `traces:` by
  hand.

## Open Questions

- Should `spec/tests.md` reserve `Complete` for rows with strict AC-level
  evidence, using `Implemented` or `Summary only` for grouped behavior rows?
- Should legacy benchmark tables be converted to the NFR-015 schema, or marked
  as historical/local summaries outside standards-complete comparisons?
