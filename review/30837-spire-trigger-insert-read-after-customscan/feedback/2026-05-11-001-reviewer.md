# Review: SPIRE Trigger Insert Read-After-CustomScan (30837)

## Summary

Closes the read-after-insert lane for the **trigger** front door
that 30835 added, now that 30836 made descriptor refresh
automatic. The multicluster script gains an `--insert-mode
helper|trigger` flag; trigger mode runs an actual SQL `INSERT INTO
coordinator_table` and asserts:
- coordinator heap row suppressed (count = 0);
- remote heap row committed and visible;
- placement-directory row staged with the trigger's canonical
  bigint PK bytes;
- descriptor refreshed to post-insert identity (no manual
  registration);
- `Custom Scan (EcSpireDistributedScan)` returns the inserted
  remote row.

Both `--insert-mode helper` and `--insert-mode trigger` artifacts
attached, proving the script refactor preserves the helper-mode
coverage. `bash -n` syntax check passes.

The three review-focus questions:
- One script with explicit insert modes vs duplication: yes —
  cleaner; the per-mode artifact dirs keep the evidence cleanly
  separated.
- Trigger mode asserts what matters: coordinator suppression,
  remote visibility after commit, placement row, descriptor
  refresh, CustomScan read-back — confirmed.
- Source-identity column added to the fixture schema doesn't
  weaken helper-mode coverage: yes — the helper test still
  exercises the same helper signature; helper-mode artifact passes.

This is the milestone packet that proves end-to-end transparent
INSERT through the v1 trigger path lands a row visible via
CustomScan.

## Findings

### P3 — Fixture asserts `placement_row=2,3,2`

The placement row format `node_id,centroid_id,served_epoch` =
`2,3,2`. Worth annotating in the script comments what each numeral
means (since the trigger mode also passes the canonical bigint PK
which a future reader may want to spot-check).

### P3 — Helper mode still re-registers descriptor?

Now that 30836 made descriptor refresh automatic, the helper-mode
path should also work without manual `ec_spire_register_remote_node_descriptor`.
The packet artifacts confirm helper-mode passes. Worth confirming
the helper-mode script branch no longer carries the manual re-
registration call from 30834 (the 30836 packet says it removed it).
If both branches now use automatic descriptor refresh, the script
is clean.

### P3 — `--remote-port 39244 --coord-port 39245` and `--remote-port 39246
--coord-port 39247`

Different port pairs per insert-mode run prevents collision when
running both in sequence. Good. Combined with the per-mode
artifact-dir, the same script invocation can produce dual-mode
evidence in one CI shard. Worth pinning that the port selection
follows the 30820 P2 finding's preference for unique ports per run.

### P3 — `Custom Scan (EcSpireDistributedScan)` assertion uses the
explicit relation suffix

The plan log shows `Custom Scan (EcSpireDistributedScan) on
ec_spire_insert_read_coord_sql`. Tighter than the 30820 fixture
which only asserted the `Custom Scan` substring. Good evidence.

## Reviewer recommendation

Accept. This is the final v1 read-after-INSERT proof — application
runs `INSERT INTO coordinator_table (...)` and a follow-up `SELECT
... ORDER BY embedding ... LIMIT k` returns the row through the
CustomScan path with no application-level shard awareness. The
ADR-069 transparent INSERT goal is met for the v1 type-shape
constraints.
