## Findings

### 1. `swap_heap_entries` reverse-membership update is not atomic with forward swap

**Severity:** Medium (deferred; single-writer today)
**File:** `src/am/common/parallel.rs:791–802`

The swap sequence is: read lhs/rhs forward entries → `std::ptr::swap`
of forward entries → two separate `Release` stores of reverse
heap_index on each slot. Concurrent readers chasing
`slot.heap_index` to look up a slot's heap position can transiently
see forward/reverse disagreement (forward entry already swapped,
reverse index not yet updated, or vice versa). Today's readers don't
dereference `slot.heap_index`, so this is latent. Cumulative item with
472/478/479 for the `amcanparallel = true` serialization work.

When the LWLock lands, wrap the forward-swap + both reverse-stores in
the same critical section so the invariant
`heap_entries[slot.heap_index] == slot_index` holds across the swap.

### 2. Upsert's re-fetch after sift_up is required, not merely defensive

**Severity:** Informational (positive)
**File:** `src/am/common/parallel.rs:960–965`

```rust
unsafe { sift_up_heap_entry(attachment, heap_index) }?;
heap_index = (unsafe { slot_heap_index(attachment, slot_index) })?
    .expect("slot should stay present after heap update");
let live_entry_count = heap_state.live_entry_count.load(Ordering::Acquire);
unsafe { sift_down_heap_entry(attachment, heap_index, live_entry_count) }?;
```

The re-fetch is load-bearing: sift_up may have moved the slot, so
subsequent sift_down must start at the new position. The `expect` is
justified because reverse membership is always updated atomically
with any swap (subject to finding #1). Good.

### 3. Detach's re-fetch pattern is symmetric and correct

**Severity:** Informational (positive)
**File:** `src/am/common/parallel.rs:933–942`

After promoting the tail into the detached position, `detach` does
sift_up at that position and then sift_down at the replacement's
*re-fetched* position — matching `upsert`'s pattern. This covers
both directions of heap repair; a pure sift-down (what a naive
textbook detach would do) would miss the case where the promoted
entry orders before its new parent.

### 4. `reap_parallel_scan_dead_root_slots_with_attachment` only reaps *roots*

**Severity:** Informational (intentional)
**File:** `src/am/common/parallel.rs:981–1007`

This matches the name and the packet's stated narrowing: the fast
path only needs the root to be live, so reaping stops as soon as a
live root is found. Dead non-root slots are reaped by the full
`reap_dead_parallel_scan_result_slots_with_attachment` on full
refresh. The split is principled: callers that only need "what's the
best live staged result?" pay only for root reap; callers rebuilding
the heap pay for full sweep. Good.

### 5. Version bump discipline followed

**Severity:** Informational (positive)
**File:** `src/am/common/parallel.rs:8`

`EC_PARALLEL_SCAN_STATE_VERSION: u16 = 5`. Adding a field to
`EcParallelCoordinatorResultSlot` changes the DSM layout, so bumping
is required. Magic stays `ECPR`. Correct.

### 6. `rebuild_parallel_scan_heap_with_attachment` now preclears reverse membership

**Severity:** Informational (positive)
**File:** `src/am/common/parallel.rs:860–864`

The full rebuild first clears every slot's reverse `heap_index` to
`INVALID`, then rebuilds forward entries and updates reverse
membership as entries are pushed. Keeps the invariant intact across
the rebuild path too — important because rebuild remains the fallback
from drifted takes / reap-then-rebuild refresh.

## Answers to the packet's review questions

- **Q1** (reverse membership right staged boundary?) — Yes. It's the
  minimum state needed to make in-place detach/upsert O(log n) and
  it's the same state the eventual lock-guarded admission path needs.
- **Q2** (lazy dead-root reap vs full-heap rebuild on publish/clear?)
  — Yes. The root-only fast-path reap matches the fast-path selection
  surface; full reap only fires when the heap itself is rebuilt.
- **Q3** (detach/update coherence under republish/clear/take?) — Yes,
  under the single-writer invariant. The forward/reverse swap
  non-atomicity (finding #1) is the only carry-forward concern.
