# Review: SPIRE Coordinator DELETE Forwarding Primitive (30839)

## Summary

Two-part packet: lands the v1 DELETE forwarding primitive and
closes the 30838 P1 by routing `node_id = 0` UPDATE placements
through the coordinator heap.

DELETE primitive:
- `ec_spire_remote_delete_tuple_payload(index_oid, pk_column,
  pk_value)` — remote endpoint validates `pk_column` against the
  indexed heap, matches by `int8send(pk)::bytea`, and returns
  `deleted_count`.
- `ec_spire_prepare_coordinator_delete_tuple_payload(...)` —
  coordinator helper looks up `(node_id, served_epoch)` from
  `ec_spire_placement`, opens a remote transaction, runs the
  remote DELETE, `PREPARE TRANSACTION`s, removes the local
  placement-directory row, and registers Commit/Abort xact callbacks
  to resolve the prepared remote.
- Fails closed unless `remote_deleted_count == 1`.

UPDATE local-node fix:
- `ec_spire_forward_coordinator_update_tuple_payload(...)` now
  branches on `node_id == 0` and runs the UPDATE directly against
  the coordinator heap via the new
  `ec_spire_update_tuple_payload_on_heap` helper. Returns
  `local_update_applied` / `done`.
- Strict negative-`node_id` rejection retained.

The four review-focus questions:
- DELETE 2PC ordering correct: yes — remote prepared DELETE first,
  local placement removal in the same coordinator transaction,
  then xact callbacks resolve the remote.
- `remote_deleted_count == 1` fail-closed: right shape; surfaces
  silent zero-row drift loudly.
- 30838 P1 resolved: yes — local-node UPDATE now uses the
  coordinator heap; this is the cleaner of the two options I
  suggested.
- Transparent DELETE integration left open: confirmed (closed by
  30843 + the hook design question; see 30843 feedback).

## Findings

### P2 — `remote_deleted_count != 1` fail-closed includes "row already gone"

If the row was DELETEd between the placement lookup and the remote
DELETE (e.g., concurrent DELETE through some other path), the
remote returns `deleted_count = 0` and this primitive errors. The
local placement-directory row also doesn't get removed, so the
placement directory keeps pointing at a row that no longer exists.

That's safer than silent success but means a concurrent DELETE
race surfaces as a hard error rather than idempotent success.
Worth either:
- Documenting the at-most-once contract and recommending callers
  retry on `remote_deleted_count = 0` (which would then cleanly
  remove the placement directory entry on the second pass — except
  the second pass also gets `deleted_count = 0` and errors again).
- Treating `remote_deleted_count = 0` as "already gone, finish the
  cleanup" and removing the placement-directory row anyway.

The second is more idempotent and arguably matches the SQL
DELETE-WHERE semantics. The first is stricter. ADR-069 should
pin the choice.

### P2 — `remote_deleted_count > 1` would be a DELETE-key drift

The fail-closed check rejects `count != 1`. If somehow the remote
table has multiple rows matching the bigint PK (impossible if PK
is enforced, but possible if the remote schema drifted or the PK
constraint is missing), the primitive errors before the local
placement is removed. Good defensive shape; worth a one-line
comment that this is also the multi-row-delete guard, not just
the no-row guard.

### P3 — DELETE primitive uses 2PC; UPDATE doesn't

Per ADR-069, DELETE needs 2PC because the placement directory
*is* coordinator-side state that has to stay consistent with the
remote heap. UPDATE non-embedding doesn't change placement, so
no 2PC. Documented correctly.

### P3 — Local-node UPDATE branch shape

The `node_id == 0` branch:
- Closes index relation early (good — no SPI-during-index-open
  pattern).
- Calls `ec_spire_update_tuple_payload_on_heap` against the heap
  relation OID directly.
- Returns `local_update_applied` / `done` status.
- Does not consult the placement-directory lookup, since the row
  is already in the coordinator heap.

Cleanly mirrors what the SELECT helper does for `node_id = 0`. Good.

### P3 — `node_id < 0` still errors

Defensive: the catalog allows `node_id >= 0`, so negative is a
data-integrity violation. The primitive errors loudly. Good.

## Reviewer recommendation

Accept the DELETE primitive and the UPDATE local-node fix. The P2
on the strict `count == 1` policy is the operational decision worth
pinning in ADR-069 before the transparent DELETE hook depends on
this primitive — operators will hit the concurrent-DELETE case in
real workloads.
