# 30658 - SPIRE architecture review

Review pass on `661d5008`, focused on graph structure, composition across
hierarchy levels, multi-NVMe / multi-instance behavior, the query path, and
the current custom libpq surfaces. Paper cross-check was against
`/home/peter/dev_bak/papers/2512.17264v1.pdf`, which describes SPIRE as a
hierarchical index with top/root graph routing, level-by-level top-m descent,
boundary replication, disaggregated index storage, and stateless query engines
that fetch relevant index data from the store.

I treated RaBitQ as the supported first quantized format. PQ/PQFastScan is
therefore not reviewed as required functionality, except where old option
names can confuse operators.

## Findings

| ID | Severity | Area | Summary |
|----|----------|------|---------|
| A1 | high | top graph / extensibility | The top graph is root-fanout-bound and single-tuple stored, so it cannot become the paper's high-cardinality top-level graph without a redesign. |
| A2 | high | recursive graph routing | Recursive descent has per-parent probes but no global beam or route budget, so deeper hierarchies can multiply leaf routes. |
| A3 | high | multi-instance correctness | Remote merge dedupes only by raw `vec_id`, while the code documents that current writers allocate node-local IDs. |
| A4 | high | libpq / query system | The libpq executor is synchronous and serial per dispatch, despite the pipeline-mode contract surfaces. |
| A5 | medium | scan composability | The AM scan path eagerly materializes all prepared candidates in `amrescan`; it is not a streaming or LIMIT-aware query engine yet. |
| A6 | medium | multi-NVMe performance | Store grouping and prefetch are well-shaped, but reads remain sequential and delta objects are decoded twice per leaf. |
| A7 | medium | top graph hot path | Top-graph routing copies adjacency and scans all root centroids every query, which is acceptable only while the graph is tiny. |
| A8 | low | operator clarity | RaBitQ works, but `auto` still maps to TurboQuant and `pq_fastscan` remains accepted as a reloption before failing later. |

## A1 - Top graph is not yet the SPIRE top graph

`build_recursive_routing_hierarchy_draft` collapses children until
`current_children.len() <= target_fanout`, then creates a root over that small
set (`src/am/ec_spire/build/recursive.rs:21`, `src/am/ec_spire/build/recursive.rs:73`).
The top graph builder then takes only `routing_object.children()` from that
root (`src/am/ec_spire/build/top_graph.rs:47`), and scan validation requires
`top_graph.nodes_len() == root_object.child_count()`
(`src/am/ec_spire/scan/routing.rs:620`). Finally, relation storage rejects any
top graph that does not fit in one tuple
(`src/am/ec_spire/storage/relation_store.rs:225`).

That means a top graph enabled above a recursive hierarchy searches at most
the root fanout, not a large routing frontier. If we later make the graph large
enough to match the paper's root/top in-memory graph, the storage format and
the scan invariant both become blockers.

Recommended direction: separate the top-graph frontier from root child count.
For example, build the graph over an explicit routing level/frontier, store it
segmented or load it into a cached in-memory graph, and validate graph nodes
against that frontier rather than `root_object.child_count()`.

## A2 - Recursive routing can grow multiplicatively

Recursive scan routing applies `nprobe_for_parent_level(parent.header.level)`
to every current parent independently, pushes all selected children into the
next frontier, and repeats until level 1
(`src/am/ec_spire/scan/routing.rs:444`). The policy only controls local
per-parent fanout, defaulting to one child for unconfigured upper levels
(`src/am/ec_spire/options.rs:251`).

This is structurally simple and works for local functionality, but it has no
global beam, no global leaf-route cap, and no dedupe before leaf I/O. With 4-6
levels, route count can grow as the product of configured per-level probes.
Boundary replication and deltas compound the resulting object reads. The SPIRE
paper's balanced granularity discussion is about keeping the operating point
before fidelity loss forces excessive partition probes; this implementation
needs a hard query-time budget to preserve that property as levels grow.

Recommended direction: carry scored frontier entries between levels and keep
the globally best `beam_width` / `max_leaf_routes` rather than taking top-N per
parent in isolation. Also dedupe leaf routes before storage reads and report
diagnostics when routing is truncated by a budget.

## A3 - Multi-node candidate merge needs a global identity

The remote merge code explicitly documents the precondition:
`merge_remote_search_candidates` only works when all candidates share a
coordinator-scoped `vec_id` namespace, but current writers allocate node-local
vec-id bytes (`src/am/ec_spire/root/remote_candidates.rs:3516`). The
implementation then uses `HashMap<Vec<u8>, ...>` keyed only by raw `vec_id`
(`src/am/ec_spire/root/remote_candidates.rs:3522`).

If two remote nodes generate the same local vector sequence, the coordinator
can silently collapse unrelated candidates. This is especially risky with
boundary replication, because cross-node dedupe is necessary but raw node-local
IDs are not a stable original-vector key.

Recommended direction: land a global `SpireVecId` format before using the
remote merge path for multi-instance search, or change the interim dedupe key
to include origin identity. For replicated vectors, the durable key should be
the original vector identity, not the serving node's local row identity.

## A4 - libpq executor does not match the pipeline contract yet

The contract rows say the remote search executor opens per-query connections,
enters libpq pipeline mode, sends requests, receives batches, and merges them
(`src/am/ec_spire/root/remote_candidates.rs:163`,
`src/am/ec_spire/root/remote_candidates.rs:2269`). The actual executor uses
`postgres::Client::connect`, resolves the remote index with a separate
`query_one("SELECT to_regclass($1)::oid")`, then runs synchronous
`client.query(...)` (`src/am/ec_spire/root/remote_candidates.rs:2611`).
Dispatch rows are then processed in a plain loop
(`src/am/ec_spire/root/remote_candidates.rs:2819`), with the heap-candidate
variant following the same serial shape and sorting/truncating after the loop
(`src/am/ec_spire/root/remote_candidates.rs:2858`).

As a diagnostic executor this is fine. As the product distributed query path,
tail latency will be the sum of remote nodes rather than the slowest remote,
and connection setup will occur once per ready remote node per query. There is
also no visible pooling, cancellation, fanout concurrency, or timeout policy
except whatever the conninfo string happens to provide.

Recommended direction: either mark the current SQL/libpq executor surfaces as
diagnostic-only, or implement the real executor with concurrent dispatch,
pipeline or async receive, bounded connection fanout, statement/connect
timeouts, cancel propagation, and cached validated remote index identity.

## A5 - AM scan prepares all candidates up front

`ec_spire_amrescan` loads manifests, opens the object-store set, calls
`prepare_single_level_relation_snapshot_scan_candidates`, and stores the full
candidate vector in the opaque state (`src/am/ec_spire/scan/callbacks.rs:64`).
`amgettuple` only drains that vector (`src/am/ec_spire/scan/callbacks.rs:92`),
and `SpireScanOpaque` / `SpireScanCandidateCursor` own a
`Vec<SpireScoredScanCandidate>` (`src/am/ec_spire/scan/types.rs:293`,
`src/am/ec_spire/scan/types.rs:333`).

The bounded ranking helper is good when a limit exists
(`src/am/ec_spire/scan/candidates.rs:778`), but `candidate_limit` is `None`
whenever `rerank_width == 0` (`src/am/ec_spire/options.rs:583`), so that path
sorts every routed candidate. Exact rerank also fetches heap rows
candidate-by-candidate (`src/am/ec_spire/scan/relation.rs:183`).

Recommended direction: make the candidate budget explicit and hard even when
rerank is disabled, and consider a streaming scorer that keeps a bounded heap
while scanning leaf/delta rows. If exact rerank remains, batch or prefetch heap
TIDs before fetching source vectors.

## A6 - Multi-NVMe shape is good, but execution is still serial

The placement grouping is the strongest part of the current storage design:
leaf and delta reads are grouped by `(node_id, local_store_id)`
(`src/am/ec_spire/scan/snapshot.rs:406`), and relation-backed prefetch groups
blocks by local store before issuing PG18 read-stream prefetches
(`src/am/ec_spire/storage/relation_store.rs:1382`). This is a clean hook for
multi-NVMe placement and future remote-local placement boundaries.

The current execution still iterates route groups and leaf routes serially
(`src/am/ec_spire/scan/snapshot.rs:315`). Store lookup is linear in small
vectors (`src/am/ec_spire/storage/local_store_set.rs:60`,
`src/am/ec_spire/storage/relation_store.rs:1431`), which is fine for a handful
of stores but not a long-term many-store abstraction. Delta handling also
decodes the same delta object once to collect deletes and again to append
insert candidates (`src/am/ec_spire/scan/candidates.rs:551`,
`src/am/ec_spire/scan/candidates.rs:609`).

Recommended direction: keep the grouping API, but add per-store route/candidate
diagnostics and a path to overlap reads across stores. For deltas, decode once
per query/leaf and split deletes/inserts in memory, or cache the decoded delta
for the leaf pass.

## A7 - Top-graph routing has hidden O(graph) setup per query

`route_top_graph_view_to_routes` rebuilds a `VamanaGraph` by copying every
neighbor list on each query (`src/am/ec_spire/scan/routing.rs:307`). It also
computes a query distance offset by scanning all root child centroids before
greedy search (`src/am/ec_spire/scan/routing.rs:314`,
`src/am/ec_spire/scan/routing.rs:677`).

This cost is hidden today because A1 keeps the graph tiny. If the graph is
made large enough to be useful, the per-query copy and full-centroid offset
will dominate before the graph search starts.

Recommended direction: make `greedy_search` accept a borrowed graph view or
store a cached adjacency structure. Also avoid a full offset scan if the
comparator can route on monotonic `-inner_product` scores directly.

## A8 - RaBitQ-first direction should be reflected in options

RaBitQ encode/score paths are present and enforce the expected zero-gamma
contract (`src/am/ec_spire/quantizer.rs:133`, `src/am/ec_spire/quantizer.rs:235`).
PQFastScan still returns a clear unsupported-model error
(`src/am/ec_spire/quantizer.rs:246`), which is acceptable given the current
RaBitQ-first scope.

The operator surface is still slightly misleading: `storage_format=auto` maps
to TurboQuant, and `pq_fastscan` is accepted by reloption parsing before later
failing in encoding (`src/am/ec_spire/options.rs:70`,
`src/am/ec_spire/options.rs:91`).

Recommended direction: make `auto` select the intended production default
format for this phase, or document it as compatibility. If PQFastScan is not
supported, reject it earlier or label it as reserved/unsupported in the
diagnostic surfaces.

## Positive structure

The hierarchy has a coherent object model: root/internal/leaf/delta/top-graph
objects all move through manifest and placement validation, and the relation
store already has large-object support for routing plus segmented leaf V2
storage. The multi-store placement key is also pointed in the right direction:
`(node_id, local_store_id)` appears consistently in scan grouping, making the
multi-NVMe and future remote split more natural than a single flat relation
reader would have been.

The per-level nprobe policy is a useful first control surface. It should not be
the only control surface for deep graphs, but it is a good building block once
global beam/leaf-route budgets are added.

2. Is the current libpq executor intended to become the production AM remote

3. What is the planned global vector identity format for multi-instance search
   and boundary replication?
4. What query-time hard limits should define the SPIRE operating point:
