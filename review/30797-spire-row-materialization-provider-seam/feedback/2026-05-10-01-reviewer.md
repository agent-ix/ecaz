# 30797 — SPIRE row materialization provider seam review

Validation pass on `064e6dd7` + `e6395d82`. **Accepts the
first Stage D materialization implementation slice.** This
is the seam where the catalog-backed mapping provider will
plug in.

## What I checked

- New owner constant
  `SPIRE_REMOTE_MATERIALIZED_HEAP_RESOLUTION =
  "coordinator_materialized_heap"` at
  `src/am/ec_spire/root/remote_candidates.rs:111`. Sits
  in the same constant family as
  `coordinator_local_heap` (packet 30757) — distinguishes
  outputs that came from the materialization provider
  from outputs that were always local.
- AM result-stream finalization now has a provider
  boundary that **can** convert a remote-origin output
  into a coordinator-visible heap TID, but **only when** a
  pre-existing mapping validates seven elements:
  - requested epoch,
  - served epoch,
  - origin node ID,
  - vector identity bytes,
  - opaque origin row locator bytes,
  - scan heap relation OID,
  - scan-snapshot visibility.
  This is the **full 5-element mapping identity from
  packet 30765 plus the 2-element AM-correctness
  constraints (scan heap relation + visibility) from
  packet 30761**. The seam enforces the strictest
  composition of both contracts.
- **Default provider is empty** — remote-origin rows
  remain blocked with `remote_row_materialization` until
  the catalog-backed mapping surface lands. ADR-064 is
  intact: no scan-time heap writes from `amrescan` /
  `amgettuple`; the provider is a lookup, not a writer.
- New tests at `src/am/ec_spire/root/tests.rs`:
  - `production_scan_row_materialization_provider_converts_remote_origin_to_coordinator_tid`
    (line 814) — proves the *positive* case: when a valid
    mapping exists, the AM cursor receives a
    coordinator-local TID via the
    `coordinator_materialized_heap` owner.
  - `production_scan_row_materialization_provider_missing_mapping_keeps_remote_blocker`
    (line 856) — proves the *negative* case: when no
    mapping exists, the AM cursor blocks with
    `remote_row_materialization` (current default
    behavior).
  - `production_scan_row_materialization_provider_rejects_identity_mismatches`
    (line 890) — proves each of the five identity
    elements (epoch×2, node, vec_id, locator) must match
    exactly; partial matches reject.
  - `production_scan_row_materialization_provider_rejects_wrong_relation_or_invisible_tid`
    (line 941) — proves the AM-correctness constraints:
    a mapped TID in a different relation, or a
    snapshot-invisible TID, both reject.
- Together, the four tests exercise the full identity
  validation matrix at the seam — including the
  `xs_heaptid` correctness constraints that PostgreSQL's
  index AM contract requires.

## Answers to the packet's review questions

- **Is the provider seam the right Stage D boundary
  before catalog-backed materialized mapping storage
  lands?** Yes — strongly. Three reasons:
  1. **Empty default is safe**: no behavior change for
     current users; existing fixtures and the
     `requires_remote_row_materialization` blocker
     continue to work.
  2. **Strict validation pre-locked**: when the
     catalog-backed provider lands, it only needs to
     supply the *lookup* — the validation is already in
     the seam, so the catalog provider can't accidentally
     bypass the identity contracts.
  3. **Testable in isolation**: the four positive/negative
     tests exercise the seam with mock providers; the
     catalog provider can be tested independently against
     the same seam.
- **`coordinator_materialized_heap` acceptable as
  AM-deliverable owner while preserving origin node ID
  and opaque row locator in the output row?** Yes.
  Preserving the origin node ID + opaque locator on the
  *output row* (even when the AM delivers a
  coordinator TID) means operator diagnostics can still
  trace materialized rows back to their origin without
  the AM cursor needing to decode the locator. ADR-059
  locator opacity is preserved through the AM boundary.
- **Missing mappings remain a normal classified
  blocker, while malformed mappings fail before
  `xs_heaptid` delivery?** Yes. Missing mappings →
  normal `requires_remote_row_materialization` block
  (handled by the existing classifier from packet
  30757); malformed mappings (identity mismatch, wrong
  relation, invisible TID) reject before AM delivery
  with their own classification. Two distinct failure
  modes correctly separated.

## P2 — close before catalog-backed provider lands

1. **Catalog-backed mapping storage design.** The seam
   accepts a provider; the next implementation slice
   needs to design the catalog storage. Per ADR-064:
   - same-relation pre-existing heap row (so TID lives
     in the right relation),
   - epoch/MVCC-scoped lifetime,
   - cleanup outside the scan cursor.
   The catalog needs to model (origin_node_id, vec_id,
   row_locator, served_epoch) → (local_heap_TID,
   coordinator_index_oid) with epoch retirement
   semantics. Recommend an ADR before the storage code
   lands — same discipline as ADR-064 pinning the
   lifecycle before the seam.
