# Review: SPIRE DML CustomScan Expression Handoff (30872)

## Summary

`SpireDmlFrontdoorPrimitivePlanExpr` bundles the typed primitive
plan with the **raw matched PK value expression pointer** so the
upcoming CustomPath builder can copy it into `custom_exprs`
(where the executor will later evaluate it via `ExecInitExpr` for
parameter resolution).

`dml_frontdoor_primitive_plan_expr_catalog_row(...)` returns the
bundle. Refactors private PK predicate extraction to retain the
matched expression while preserving classifier and replacement-
decision outputs.

PG18 coverage: UPDATE / DELETE / PK SELECT all produce a typed
primitive plan plus non-null const PK expression handoff.

The three review-focus questions:
- Right planner handoff for upcoming CustomPath builder: yes —
  the expression pointer is what the planner needs to copy into
  `custom_exprs` so the executor can evaluate it at execute time
  (especially for `Param` predicates).
- Retaining the raw matched PK value expression doesn't change
  classifier behavior: yes — the classifier consumes only the
  derived value-kind/operator facts, not the raw expr pointer.
- Stays a non-dispatch scaffold packet: confirmed.

## Findings

### P2 — Raw expression pointer lifetime

`SpireDmlFrontdoorPrimitivePlanExpr` carries a raw
`*mut pg_sys::Expr`. That pointer is owned by the original
`Query` tree from the planner. The CustomPath builder must
`copyObjectImpl` it before it can outlive the planner's memory
context. Worth a `# Safety` comment on the struct that the
expression pointer is borrowed and must be deep-copied before
it survives the planner call.

(Looking at the 30873 rewrite code, `pg_sys::copyObjectImpl(pk_value_expr.cast())`
is exactly what it does — confirmed correct usage downstream.)

### P3 — Bundle returned by `_catalog_row` helper

The `_catalog_row` naming convention suggests "this loads
relation context from the catalog" but in this case the helper
also runs PK predicate extraction. Worth either renaming to
`_catalog_decision` (mirrors 30859's `replacement_decision_catalog_row`)
or splitting into two helpers (one for the decision, one for the
expression bundle).

## Reviewer recommendation

Accept. The lifetime comment (P2) is the only thing that would
help a future maintainer avoid a use-after-free trap; the rewrite
slice already handles it correctly.
