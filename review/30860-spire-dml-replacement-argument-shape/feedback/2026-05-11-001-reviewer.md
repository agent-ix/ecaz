# Review: SPIRE DML Replacement Argument Shape (30860)

## Summary

Extends `SpireDmlFrontdoorReplacementDecisionRow` with the executor
argument shape: `pk_column`, `updated_columns`, `projected_columns`.
This gives the future DML CustomScan node everything it needs to
construct the primitive call — PK column name, list of columns to
SET (for UPDATE), list of columns to project (for PK SELECT).

Refactors the relation-backed classification into a detail helper
so the classifier and replacement-decision paths use the same
extracted target-list facts (no risk of drift). The replacement
SQL diagnostic now returns the argument-shape fields too.

PG18 fixtures assert:
- PK SELECT replacement reports `projected_columns = id,title`.
- Non-embedding UPDATE reports `pk_column = id`,
  `updated_columns = title`.

The three review-focus questions:
- Argument-shape carries everything the executor slice needs: yes
  — combined with `target_relation_oid`, `index_oid`,
  `custom_scan_mode`, and the predicate value (extracted from
  the query at exec time), the CustomScan can build the primitive
  invocation directly.
- Refactor keeps classifier and decision aligned via shared
  extraction: yes.
- SQL diagnostic return-shape change acceptable before plan
  rewriting is enabled: yes — the diagnostic is the only consumer
  so far, and operators get a richer shape view.

## Findings

### P2 — PK predicate value still extracted separately at exec time

The replacement decision row carries the *names* needed but not
the *runtime values*. The PK predicate value (the const/param)
isn't part of the decision row — it's extracted later from the
plan tree at execute time. That's the right boundary (decision is
plan-time, value is execute-time, especially for parameterized
queries), but worth a one-line note in the executor slice's
follow-up packet that the plan stores the predicate expression
rather than baking the const value into custom_private.

### P3 — `pk_column` carried on every decision row

Even unsupported shapes carry the pk_column. Useful for the SQL
diagnostic's "what column would you use" view. Harmless on
unsupported decisions because the executor short-circuits before
needing it.

### P3 — `updated_columns` empty for non-UPDATE operations

DELETE / PK SELECT decisions return empty `updated_columns`. The
executor will branch on operation kind anyway, so empty is fine.
Worth a comment that the field is operation-conditional.

### P3 — `projected_columns` empty for non-SELECT operations

Same pattern; operation-conditional. The executor branches on
kind.

## Reviewer recommendation

Accept. The decision row now carries everything the executor
needs at plan time. The next packet should land the actual plan
rewrite — install a custom path / replace the plan tree — that
calls the existing primitives via the decision row.
