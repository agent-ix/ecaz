# Review: SPIRE Typed Transport Feedback Notes (30922)

## Summary

Pure docs response folding the P3s from 30917, 30918, and 30919
into the typed transport design and Phase 12 tracker:

- Empty projections explicitly named as a supported
  `pg_binary_attr_v1` shape (closes 30918 P3).
- v1 composite scope pinned to **named heap-column composites**;
  anonymous computed `record` projections deferred to a future
  projection surface (closes 30917 P3 about `SELECT (id, title)
  FROM tbl`).
- Negotiation section says transport capability advertisement is
  protocol-level; per-column binary I/O gaps still surface as
  `unsupported_type_binary_io` (closes 30919 P3).
- `tuple_transport_default` documented as hardcoded for v1, not
  a per-descriptor operator override (closes 30919 P3).
- Phase 12.2 tracker note that endpoint-level typed coverage now
  spans the v1 scalar / array / composite / domain / NULL gaps
  that blocked the JSON bridge.

The three review-focus questions:
- Anonymous-composite v1 deferral matches table-column
  projection contract: yes — the typed endpoint exists to
  return heap row columns, not arbitrary expression results.
  Anonymous computed records belong to a future expression-
  pushdown surface.
- Capability-vs-column-type wording handles 30919 P3 without
  weakening strict typed validation: yes — the design now
  distinguishes "I support the protocol" (capability) from
  "this specific column type works" (per-column status). Both
  pieces gate strict mode; capability alone isn't enough.
- Tracker note doesn't overclaim CustomScan receive or full
  negotiation: confirmed — receive switch + selection rules
  remain explicitly open.
