# Review: SPIRE DML Relation Context Cache (30941)

## Summary

Closes 30896 P4. Adds a backend-local cache around the
catalog/relcache DML frontdoor relation context loader used by
planner-hook observation and CustomScan handoff:

- Cache hits avoid the `table_open` + `RelationGetIndexList` +
  per-index `index_open` chain on every planner call against a
  SPIRE-eligible relation.
- **Relcache invalidation callback registered**: cached
  contexts evict when the heap relid OR any index relid
  observed while building the context is invalidated.
- Diagnostic surface
  `ec_spire_dml_frontdoor_relation_context_cache()` exposes
  hit/miss counts, invalidation count, entry count, callback
  registration state.
- SPI-backed diagnostic loader unchanged (the catalog-safe
  loader is separate).

The three review-focus questions:
- Cache cannot reuse stale relation context after heap/index
  relcache invalidation: yes — the callback evicts on
  invalidation of any watched relid.
- Watching all index relids from `RelationGetIndexList` is
  appropriate: yes — covers both the `ec_spire` index and the
  PK index dependencies. If either changes (rebuild, drop), the
  cache evicts.
- Diagnostic function narrow enough for Phase 12.3, doesn't
  imply cross-backend or shared-cache behavior: yes — explicit
  backend-local scope.

## Findings

### P2 — Backend-local cache footprint under high QPS

The cache is per-backend. For a coordinator running many
parallel sessions against many SPIRE-eligible relations, total
memory is `N_backends * N_relations * context_size`. For
typical PG18 deployments (≤200 backends, ≤100 relations) this
is bounded; worth pinning an entry-count cap or LRU eviction
policy if that ratio grows. Phase 12 scope is "make it
bounded" (which is done by relcache eviction); a hard cap is
Stage F polish.

### P2 — Per-snapshot vs per-backend cache lifetime

The cache lives for the backend lifetime (with relcache
invalidation eviction). Worth confirming this matches PG's
relcache semantics — relcache itself is per-backend and
invalidated via shared-invalidation messages, so a cache
hanging off relcache callbacks naturally inherits the same
liveness model. Looks correct.

### P3 — Hit/miss/invalidation counters in diagnostics

Operator-visible cache stats are useful for capacity planning
and for catching invalidation-storm regressions. Worth pinning
in the Stage G runbook that elevated invalidation rate is a
canary for DDL-on-the-coordinator workloads.

### P3 — `RelationGetIndexList` returns dependent indexes

Worth confirming the watched-index list includes the PK index
(needed for placement-row PK lookups) AND any
`ec_spire_placement_by_index_oid`-style auxiliary indexes if
they're ever read during context construction. The current
shape (watch all RelationGetIndexList outputs) covers this
correctly — any future auxiliary index added to the heap is
auto-watched.

## Reviewer recommendation

Accept. Closes P4. Cache eviction tied to relcache invalidation
is the correct PG-native pattern.
