# Review: SPIRE DML PK Byte Buffers (30942)

## Summary

Closes 30896 P8. Replaces `Vec<u8>` allocations for bigint PK
encoding with fixed `[u8; 8]` buffers. Carries the fixed-size
buffer through:

- Primitive plans
- Runtime parameter evaluation
- Invocation metadata
- `EcSpireDistributedScan` executor state

Public SQL/SPI bytea boundaries keep `Vec<u8>` conversions where
pgrx requires PG datum ownership — those `to_vec()` calls are
the only remaining allocations.

The three review-focus questions:
- Fixed-size buffer appropriate for v1 bigint-PK-only DML
  contract: yes — bigint PKs are the only supported v1 shape,
  so the 8-byte invariant matches the contract exactly.
- Remaining `to_vec()` calls constrained to PG bytea conversion
  boundaries: yes — only at the SPI/SQL surface where pgrx
  needs an owned datum.
- Removing the old empty-vector checks doesn't drop a reachable
  error path: yes — a `[u8; 8]` is always exactly 8 bytes by
  construction, so the empty-check was unreachable in the new
  representation.

## Findings

### P3 — Future PK type changes

If v2 ever supports UUID or composite PKs, this would need to
move from `[u8; 8]` to a const-generic `[u8; N]` or back to
`Vec<u8>` (since composites have variable size). Worth a
one-line comment on the buffer type that explicitly names the
v1 bigint constraint, so a future contributor adding UUID
support sees the constraint immediately.

### P3 — `to_vec()` at the boundary

The SPI/bytea conversion still allocates per call, but that's
the irreducible cost of crossing into PG datum-ownership land.
Trying to thread `&[u8]` through pgrx's typed datum surface
would fight the bindings. Right tradeoff.

## Reviewer recommendation

Accept. Closes P8.
