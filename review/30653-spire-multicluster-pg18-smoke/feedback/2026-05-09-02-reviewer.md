# Addendum — Phase 9 scoping for routing-quality improvements

Not a packet review. This addendum captures a roadmap
decision the user surfaced after another reviewer flagged
that graph routing could be better. Recorded here so the
coder sees it on the inbox loop and so the next planning
slice doesn't re-litigate it.

## Context

The current routing implementation is an intentional slimmer
subset of the SPIRE paper (2022 vintage). There is a real
improvement ladder:

- **Layer 1 — catch up to the paper.** FR-040 / ADR-049
  already plan most of this; deferred to Phase 3+.
- **Layer 2 — cheap improvements past the paper, no
  training required.**
- **Layer 3 — learned routing.** Research-grade, real risk.

## Decision — Phase 8 vs Phase 9 vs ADR-defer

Phase 8 is closeout (scheduler cleanup, scale measurement,
docs) and is in the home stretch. Folding recall-quality
work into Phase 8 conflates "ship-ready" with "ship-better"
and slips the cutover. **Open Phase 9 for routing-quality
improvements.**

Per-item disposition:

| Layer | Item | Disposition | Rationale |
|---|---|---|---|
| L1 | Recursion (4–6 levels) | Phase 9 | Already planned (FR-040 / ADR-049). Real engineering, not research. |
| L1 | Per-level nprobe | **Phase 8 pull-forward** | TODO already in `scan/types.rs`. Cheap. Scale-measurement run wants the knob to characterize the cost model honestly. |
| L1 | Boundary replication | Phase 9 | Designed in `spire-boundary-replication.md`; execution work, no new design. |
| L1 | Top-level routing graph (HNSW/DiskANN over centroids) | Phase 9 | Mechanical against existing graph libs. The "graph routing could be better" finding lands here. |
| L2 | IMI (Inverted Multi-Index) reshape | Phase 9 | Centroid-table reshape; A/B-able against current storage; no training. |
| L2 | Multi-probe centroid scoring | **ADR-defer (skip)** | Likely subsumed by anisotropic centroid scoring. Record the *why* so it doesn't keep resurfacing. |
| L2 | Adaptive nprobe | Phase 9 | Low-risk runtime change; wins on easy queries, neutral on hard. |
| L2 | Anisotropic centroid scoring (ScaNN-style) | **Phase 9 headline** | Single biggest known win past vanilla IVF/SPIRE. ~1.5–2× recall at same QPS on dense embeddings. |
| L3 | Query difficulty estimator | Phase 9 stretch | Closest L3 cousin to adaptive nprobe — same shape, slightly learned. Cheap to prototype if Phase 9 has bandwidth. |
| L3 | NN-routing classifier | **ADR-defer** | Drift / retraining cadence / eval harness all need decisions before code. |
| L3 | Routing reranker | **ADR-defer** | Same shape as NN-routing classifier — research bet, not a checkpoint. |

## ADRs to open (separate from Phase 9 task list)

1. `ADR-NNN — Multi-probe centroid scoring deferred (subsumed by anisotropic)`
2. `ADR-NNN — Learned NN-routing classifier — deferred research track`
3. `ADR-NNN — Routing reranker — deferred research track`

ADRs should record the *open questions* that gate any future
adoption (drift cadence, retraining infra, eval harness, cost
of being wrong) so the option stays visible without
committing Phase 9 to it.

## Phase 9 scope summary

In: L1 recursion, L1 boundary replication, L1 top-level
graph, L2 IMI, L2 adaptive nprobe, L2 anisotropic (headline),
L3 query difficulty estimator (stretch).

Pulled forward to Phase 8: L1 per-level nprobe.

ADR-deferred: L2 multi-probe, L3 NN-routing classifier,
L3 routing reranker.

## Instruction to coder / planner

- **Do not start any of this in Phase 7 or Phase 8.** The
  remaining Task 30 closeout (heap row resolution polish,
  multi-cluster smoke `make` target, scale measurement run)
  takes priority.
- **When opening Phase 9**, use the table above as the
  starting task decomposition. The headline is anisotropic
  centroid scoring; the highest-leverage L1 catch-up is the
  top-level routing graph.
- **Per-level nprobe is the one Phase 8 pull-forward.**
  Land it before the scale measurement run so the run can
  exercise per-level tuning.
- **Open the three ADRs** when the Phase 9 plan lands,
  not now. They are pointers, not active work.

No rush. This is roadmap scoping, not a slice request.
