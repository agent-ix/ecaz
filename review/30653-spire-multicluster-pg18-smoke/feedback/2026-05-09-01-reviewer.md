## 30650 — manifest apply (`1d8499b9`) local notes

1. **Plan item E step 2 lands cleanly.** The dispatch SQL
   string in `ec_spire_remote_epoch_manifest_libpq_request_plan`
   and the actual libpq call site in
   `ec_spire_remote_epoch_manifest_libpq_executor_results`
   both move from `validate_…` to `apply_…` in the same
   commit. The "dispatch plan vs network call divergence" risk
   I flagged in 30649 feedback does not occur. The test
   `assert!(libpq_request_sql.contains("ec_spire_apply_remote_epoch_manifest_payload"))`
   pins the contract.
2. **F2 closed.** `raw_conninfo_exposed: bool` is replaced
   throughout the manifest executor's eight result construction
   sites with `conninfo_lookup_kind: &'static str`, taking
   `"secret_provider"` when a lookup was attempted and
   `"not_attempted"` when blocked before connection. Diagnostic
   only; never returns the conninfo string itself.
3. **F3 closed.** All six early-return sites and the success
   site in
   `ec_spire_remote_epoch_manifest_libpq_executor_results`
   now construct `SpireManifestExecutorResultRow` and run
   through a single `into_tuple()` at the bottom. Column order
   changes are now a one-site edit.
4. **Apply persistence shape is sound.** Header table
   `ec_spire_remote_epoch_manifest_applied` keyed by
   `(remote_index_oid, active_epoch)` with `ON CONFLICT ...
   DO UPDATE`; entries are `DELETE` then `INSERT` inside the
   same `Spi::connect_mut` block. Since the apply runs as a
   single libpq `SELECT * FROM ec_spire_apply_…` request, the
   delete+insert pair is one transaction on the remote
   session. Idempotent re-apply of the same manifest is well
   defined.
5. **Validation-vs-apply error categories stay distinct.**
   Validation-style failures (`invalid_manifest_payload`,
   `manifest_epoch_mismatch`, `empty_manifest_entries`,
   `invalid_manifest_entry_action`,
   `manifest_entry_not_ready`, etc.) early-return *before* any
   row is inserted. Apply failures (the SPI `update` calls)
   bubble up through `result.unwrap_or_else(pgrx::error!)` and
   surface to the libpq caller as a query error. Good
   separation; no risk of partial header write with no entries.
6. **TOCTOU on the index OID is acceptable.** The function
   opens and immediately closes the index relation just to
   confirm "this OID is an `ec_spire` index" — it does not
   hold the lock through apply. Fine for a payload-validation
   gate; nothing structural depends on the index relation
   being unchanged through the SPI inserts.

Non-blocking carry: `validated_entry_count` is now
"validated AND applied entry count". The request-md notes
this kept the column name for executor-result contract
compatibility. Worth adding a one-line doc-comment on the
column itself in the contract function so the meaning is
discoverable.

## 30651 — remote heap libpq candidates (`6d004304` + `efd0a5cb` + `226d7e42`) local notes

1. **Heap-row resolution wiring is in.** This is plan item 3
   from the 30649 feedback's revised plan. New module
   function `remote_search_libpq_executor_heap_candidates_for_dispatch`
   issues a second libpq query against the existing
   `ec_spire_remote_search_local_heap_candidates(...)`
   endpoint, decodes `(heap_block, heap_offset)`, normalizes
   the remote `node_id`, runs the candidate batch through
   `validate_remote_search_candidate_batch`, and surfaces rows
   marked `heap_lookup_owner = 'origin_node_row_locator'`.
2. **Node-id normalization is correct for loopback and
   non-collapsing topologies.** When the remote returns
   `SPIRE_LOCAL_NODE_ID` (the remote's *own* local-id
   sentinel), it is rewritten to the coordinator's expected
   `node_id`; otherwise the remote-supplied node id passes
   through. This is the right shape for loopback and for
   real multi-cluster where remotes always know themselves
   as the local sentinel.
3. **Served-epoch plumbing is strict.**
   `validate_remote_search_candidate_batch` requires
   `served_epoch == requested_epoch`. That's the correct
   first-cut policy and matches the rest of the SPIRE search
   path. When degradation paths need to surface a *served*
   epoch lower than *requested*, this becomes a touch point
   — flagging as a forward item, not a blocker.
4. **Result-source contract is contract-visible.**
   `remote_heap_candidates` is now in
   `ec_spire_remote_search_coordinator_result_contract()`
   (commit `226d7e42`), and the summary helper
   `ec_spire_remote_search_libpq_executor_heap_candidate_summary`
   exposes `result_source / status / returned_candidate_count`
   without consuming every row. Operator-friendly.
5. **Result composition is *not* yet wired into the merged
   coordinator result summary.** Per the request notes, this
   exposes resolved heap rows as executor output but does not
   yet feed them into
   `ec_spire_remote_search_coordinator_result_summary(...)`.
   That's the natural next slice and is clearly labeled in
   request.md.

## 30652 — DROP INDEX catalog cleanup (`0d96d6a6`) local notes

1. **Atomicity follows from the event trigger model.** A
   `sql_drop` event trigger runs in the same transaction as
   the `DROP INDEX` statement, so partial cleanup is
   impossible — either the drop and the descriptor/manifest
   row removal both commit, or both roll back. Good.
2. **Manual cleanup escape hatch retained.** Both
   `ec_spire_remote_catalog_index_cleanup(oid)` and
   `ec_spire_remote_catalog_orphan_cleanup()` remain callable.
   The 30652 lifecycle contract update keeps them as "explicit
   / restore-era" tools, which is the right framing.
3. **F4 (new, non-blocking) — applied-side catalogs are not
   cleaned.** `ec_spire_remote_catalog_index_cleanup` only
   removes from `ec_spire_remote_node_descriptor`,
   `ec_spire_remote_epoch_manifest`, and
   `ec_spire_remote_epoch_manifest_entry` (all keyed by
   `coordinator_index_oid`). The 30650-introduced
   `ec_spire_remote_epoch_manifest_applied` and
   `ec_spire_remote_epoch_manifest_applied_entry` are keyed by
   `remote_index_oid` and live on the *remote* node, and the
   cleanup helper does not touch them. Two consequences:
   - **On the coordinator**, dropping the coordinator index
     leaves applied-table rows untouched on the remote — but
     those rows were keyed by *remote_index_oid*, not the
     coordinator OID, so this is structurally correct (the
     remote index still exists). No issue here.
   - **On the remote node**, dropping the remote-side
     `ec_spire` index that is the apply target fires the same
     event trigger but
     `ec_spire_remote_catalog_index_cleanup(remote_oid)`
     looks for `coordinator_index_oid = remote_oid`, which is
     never going to match. The applied tables' rows for that
     `remote_index_oid` are therefore retained as orphans
     after the remote index is dropped.
   Recommend extending either the existing helper (with an
   additional `WHERE remote_index_oid = $1` cleanup pass that
   no-ops when nothing matches) or introducing a sibling
   `ec_spire_remote_catalog_remote_index_cleanup(oid)` that
   the same event trigger calls in the same loop. Non-blocking
   because the rows are inert and orphan cleanup can mop up
   manually, but this is the kind of thing that becomes a
   support ticket six months later. Bundle into the next
   slice.

## 30653 — multicluster PG18 smoke (`2e61fddebb20`) local notes

1. **Real two-cluster, not loopback.** The harness `initdb`s
   two separate data directories (`$RUN_DIR/remote`,
   `$RUN_DIR/coord`), starts two postmasters on two ports
   (`39218`, `39219`) listening only over Unix sockets in a
   private `$SOCKET_DIR`, and does all coordinator-side work
   against `$COORD_PORT` while the remote cluster sees only
   apply-side traffic over libpq. This is the genuine
   multi-cluster shape, not loopback-with-different-databases,
   and it closes plan item 4 from the 30649 revised plan.
2. **Remote-node identity is exercised end-to-end.** The
   smoke writes `node_id = 2` into a coordinator placement via
   the new pg-test-only `tests.ec_spire_test_rewrite_placement_node`,
   registers descriptor `node_id = 2`, and asserts that the
   returned heap row has `node_id = 2` after the
   `SPIRE_LOCAL_NODE_ID → expected_node_id` normalization runs
   on the remote-returned candidate. Good coverage of the
   normalization path.
3. **Manifest apply is verified across the boundary.** Smoke
   asserts
   `remote_manifest_applied = 1,1` and
   `remote_manifest_entries = 1,1` by reading the applied
   tables on the *remote* cluster with a separate `psql`
   client. This is the assertion that proves the dispatch
   string and the actual libpq endpoint are consistent in a
   real cross-cluster context — much stronger evidence than
   the loopback fixture.
4. **Test helper exposure is appropriately fenced.**
   `ec_spire_test_rewrite_placement_node` lives inside `mod
   tests` and is only registered when the `pg_test` feature
   is enabled. The harness's `cargo pgrx install --features
   "pg18 pg_test" --no-default-features` makes the gate
   explicit. The helper is NOT exposed in production builds.
   Good.
5. **No `make` integration yet.** The script is invoked by
   absolute path (`scripts/run_spire_multicluster_pg18_smoke.sh`)
   with `RUN_ID` and `RUN_DIR` overridable via env. There is
   no `make spire-multicluster-smoke` target and no CI hook,
   so this presently runs on demand only. That's acceptable
   for a Phase 7 acceptance gate but worth at least a Make
   target before Phase 7 closeout so re-running it is a one-
   liner. Non-blocking; flagging.
6. **Cleanup on failure is best-effort but workable.** `trap
   cleanup EXIT` issues `pg_ctl ... -m fast stop` for both
   clusters and does not attempt to remove `$RUN_DIR` (it
   actively *fails* if `$RUN_DIR` already exists). That's the
   right tradeoff — failed runs leave artifacts on disk for
   inspection. Operators get a fresh `$RUN_DIR` per
   `RUN_ID`. Good.
7. **Packet directory is currently untracked.** The smoke
   *commit* (`2e61fdde`) is pushed, but
   `review/30653-spire-multicluster-pg18-smoke/` (with
   `request.md` and `artifacts/`) is not yet `git add`-ed.
   Add and commit this packet so the artifact log
   (`multicluster-smoke-success.log`) is part of the audit
   trail. Same shape as prior packets — process gap, not a
   code issue.
