# Review: SPIRE Review Follow-Ups 2 (30944)

## Summary

Tiny docs/comment slice closing two accepted P3 notes:

- 30942 P3 — `[u8; 8]` PK buffer documented as ADR-069 v1
  bigint-only, must widen for UUID/composite PKs.
- 30943 P3 — ADR-069 future-ADR list now names the
  prepared-xact recovery helper and the durable GID + PK +
  outcome metadata it requires.

## Findings

### P3 — Comment scope

The `[u8; 8]` comment is exactly the right depth: names the
ADR, the v1 boundary, and the widening trigger. Future-coder
opening that struct will see "this is bigint-only because of
ADR-069 v1 DML" before they try to wedge a 16-byte UUID into
the same buffer.

### P3 — Future-ADR pin completeness

The future-ADR note captures the three things a recovery
helper needs to be safe:

- **GID**: tells the helper which prepared transaction is
  ours.
- **PK**: tells the helper which placement row to consult
  (current state lacks this — `pg_prepared_xacts` doesn't
  carry payload).
- **Outcome**: tells the helper whether to `COMMIT PREPARED`
  or `ROLLBACK PREPARED` without re-deriving from coordinator
  state.

That's the minimal contract. A future ADR could add more
(e.g., served_epoch for descriptor-version invariance) but
the three above are the load-bearing ones.

### P3 — Wording suggestion (non-blocking)

If the future-ADR bullet doesn't already, worth naming the
durable metadata table candidate (e.g.,
`ec_spire_inflight_2pc_intent`) so the next reviewer/coder
has a concrete hook to grep for. Optional.

## Reviewer recommendation

Accept. Both deferred items are now pinned in the right
places (Rust comment at the constraint site, ADR future-list
for the helper). Closes the two P3 follow-ups cleanly.
