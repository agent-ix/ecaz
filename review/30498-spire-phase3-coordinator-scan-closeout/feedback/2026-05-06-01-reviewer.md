# 30498 SPIRE Phase 3 Coordinator and Scan Closeout — feedback

This is a consolidated review across the Phase 3 recursion arc
(packets 30469–30498). Read with `plan/design/spire-recursive-hierarchy.md`
and `plan/tasks/30-spire-ivf-foundation.md` (Phase 3 section, lines 1003–1123).

## What landed

- Design checkpoint: `plan/design/spire-recursive-hierarchy.md` defines level
  numbering, PID invariants, routing-object body shape, recursive build,
  scan descent, and explicitly defers per-level metadata, replication,
  graph routing, and recursive update propagation.
- Pure recursive routing draft helper that materializes routing objects and
  centroid records bottom-up from leaf centroids
  (`build_recursive_routing_hierarchy_draft`, build.rs).
- Level-local primitive `route_routing_object_to_child_pids` plus a
  recursive coordinator that descends through preloaded internal/root
  routing objects without relation I/O (scan.rs).
- Snapshot preload helper that loads root + active internals through the
  existing `SpireObjectReader` boundary
  (`load_snapshot_routing_hierarchy`).
- Recursive route plumbed through routed leaf rows, quantized candidates,
  and recursive leaf-count helper, all preserving root-PID reporting and
  strict/degraded placement semantics.
- Conservative `SpireRecursiveNprobePolicy`: configured `nprobe` at level 1,
  hardcoded `1` above.
- Local recursive routing epoch materializer + relation-backed writer seam
  + recursive epoch publish bundle + relation publish bridge that reuses
  the single-level publish coordinator path; allocator cursors are
  consumed explicitly and only committed on success.
- `recursive_fanout` reloption (0 = single-level, ≥2 = opt-in; 1 rejected at
  accessor time), `ambuild` activation when `recursive_fanout >= 2`.
- SQL diagnostics: `recursive_routing_supported` flag, persisted per-edge
  centroids exposed through `ec_spire_index_routing_centroid_snapshot`,
  and `recursive_fanout` / `recursive_build_enabled` plus recursive-aware
  `active_leaf_count` in `ec_spire_index_options_snapshot`.
- Flat-vs-recursive parity proven both pure (4-leaf hierarchy, identical
  TurboQuant payloads, identical top candidate) and through SQL smoke
  (relation-backed flat and `recursive_fanout=2` builds return the same
  nearest row).

## Architecture — what is genuinely strong

- **Single-level is a degenerate case, not a parallel path.** The
  `root_at_level()` constructor delegates to `root()`,
  `load_snapshot_root_routing_object()` is a thin wrapper over the
  hierarchy loader, and the publish bridge reuses the same
  `encode_manifest_bundle_for_publish` / root-control path as the
  single-level builder. There is one publish coordinator, one manifest
  contract, one placement directory. This is the right backbone.
- **Object-store writer seam.** `SpireRecursiveRoutingEpochObjectStore`
  unifies local and relation writes by delegating to the existing
  `insert_routing_object` / `insert_leaf_object_v2_from_rows` methods.
  The generic `build_recursive_routing_epoch_draft_with_store` is the
  shared body; both call sites get the same validation. No duplicated
  implementations to drift.
- **Allocator-cursor ownership is explicit.** Both PID and
  `local_vec_id` cursors are cloned, advanced on a copy during draft
  build, and only committed on the success path. Publish bundle helpers
  take `next_local_vec_seq` as an explicit parameter rather than owning
  it inside the epoch draft. This was a real risk and you handled it
  cleanly.
- **Level numbering invariants are enforced at the right boundaries.**
  `validate_recursive_hierarchy_shape` rejects duplicate PIDs, multiple
  roots, level/kind mismatches between parent and child, missing
  manifest entries, and duplicate children within one parent — fail
  closed before descent.
- **Parity proof is real.** `recursive_quantized_candidates_match_flat...`
  builds two structurally different hierarchies over the same encoded
  payloads and asserts the same top PID/TID/candidate. That is a much
  stronger contract than "both pipelines run."

## Architecture — things to improve

1. **Validation layering is good but undocumented.** Three barriers exist:
   `validate_recursive_routing_build_draft` (in-memory shape),
   `validate_recursive_epoch_leaf_placements` (post-write placement
   coverage), and `validate_recursive_hierarchy_shape` (snapshot-time
   descent guard). They are correct, but a reader has to discover the
   division. Add a short comment block above the first one naming all
   three and what each is responsible for, so future contributors do not
   add a fourth or move checks across the seams.

2. **Some validation is duplicated across entry points.**
   `recursive_routing_leaf_parent_pids` is invoked from both the
   leaf-input entry point and the assignment-row entry point. That is
   defensible (each entry point must validate independently), but the
   result is that "what counts as a valid recursive draft" is implicit
   in the union of those call sites. Consider a single
   `assert_recursive_draft_invariants(draft)` that both paths call so
   the invariant set lives in one place.

3. **Validation is scattered between hierarchy load and descent.**
   `load_snapshot_routing_hierarchy` filters by kind only; the
   level/parent_pid drift checks live inside
   `require_recursive_internal_child` at descent time. That works today
   because internal objects are not level-discriminated in storage, but
   if a future schema adds a typed-by-level internal it will be
   load-silent. Either document this explicitly or push the
   level-coherence check into the loader.

4. **`SpireRecursiveNprobePolicy` is named for its inputs, not its
   contract.** The struct holds one `leaf_level_nprobe` and a hardcoded
   `1` for everything above. Rename to
   `SpireConservativeRecursiveNprobePolicy` (or similar) and add a
   `// TODO: replace when per-level nprobe metadata lands` next to the
   hardcoded `1` at scan.rs:152. Right now the policy looks like a
   placeholder that has been promoted to a struct; the name should
   advertise that it is the conservative-default policy, not the
   permanent one.

5. **`recursive_fanout = 0` sentinel is clever but rough at the edges.**
   `0`/`>=2` is a fine encoding, but the rejection of `1` only fires at
   accessor time — meaning a relation can be created with
   `recursive_fanout=1` and break later. The reloption help text also
   does not call out the `>=2` requirement. Either validate at parse
   time or expand the help text. Long-term, consider splitting into
   `recursive = bool` + `fanout = N` so the intent is explicit; the
   current encoding will read awkwardly once a per-level surface exists.

6. **Centroid records are cloned but not persisted separately.** The
   draft carries materialized centroid records, but the relation publish
   bridge writes only routing objects + placements; centroid records
   are reconstructed from routing objects by diagnostics. The design
   doc explicitly defers durable centroid records, so this is fine, but
   leave a `TODO` in `SpireRecursiveEpochDraft` next to the centroid
   field saying "not persisted; rebuild via `centroid_records_from_routing`".

## Strategy — things to improve

1. **Layering is principled but the packet count is high.** Phase 3 has
   ~30 packets, several of which add a single field or a single
   validation. The walk-by-walk progression is auditable and made
   review tractable, but as the foundation matures, cluster smaller
   refinements (e.g., the eight "selected-…-input-validators"-style
   packets in phase 2 update mechanics) under one packet with
   sub-bullets. Each packet costs a request.md, a commit, and a context
   switch for review; small cosmetic packets dilute the signal.

2. **Tests are strong on pure paths, light on multi-level depth.** The
   deepest tested hierarchy is two routing levels (root→internal→leaf).
   The conservative nprobe policy means a 3-level index would probe 1
   internal, 1 internal, then `nprobe` leaves — *correct as designed*
   but not under test. Add at least one synthetic 3-level test that
   asserts the level-2 routing actually selects a child rather than
   vacuously passing through a single-child internal.

3. **No degraded-placement test for recursive descent.** Strict-mode
   parity is well-tested; degraded mode (where an internal routing
   object is unavailable) is not exercised. The skip-placement code
   path is reused, so it likely works, but a regression here would not
   be caught.

4. **SQL parity test is a single-row smoke.** `flat_recursive_same_candidate`
   compares one query at one `LIMIT 1`. That is a sanity check, not a
   parity proof. Extend with at least: (a) several query vectors,
   (b) `LIMIT k > 1` with set-equality of returned IDs, (c) one query
   that crosses an internal-routing boundary so the comparison
   exercises non-trivial routing.

5. **Per-level `nprobe` story needs to be communicated.** The plan
   leaves this open and the implementation is conservative
   (`1` per level above 1). For a 3-level index this means total leaves
   probed = `nprobe`, not `nprobe * fanout`. That is *not* what most
   IVF users will assume. Surface the effective per-level fanout in
   `options_snapshot` (e.g., `effective_nprobe_per_level` array) so the
   limitation is observable, and put a one-paragraph note in the user
   guide / release notes when this is exposed externally.

6. **Update mechanics interaction is documented but untested.** The
   design doc rules out recursive update propagation in Phase 3; the
   build path looks like it leaves room for it (centroids stored
   per-edge, replacement-PID allocator already in place). Good. Before
   Phase 4 lands anything that *uses* the recursive build, write a
   short failure-mode note: what happens if a leaf split is attempted
   on a recursive index today? Does the scheduler refuse it, or does
   it rewrite only the level-1 parent and silently leave higher levels
   stale? An explicit guard with a clear error is better than a quiet
   inconsistency.

## Correctness — risks worth tracking

- **Centroid ordinal density.** Materialization enumerates children, so
  ordinals are dense `0..child_count` by construction. Tests rely on
  this. If anyone ever filters routing children mid-draft, ordinals
  will gap and the route heap's tie-break (lower centroid ordinal) will
  silently change winners. Add an assertion in
  `validate_recursive_routing_build_draft` that
  `centroid_ordinal == position` for every entry.

- **`source_count = 1` for first-level centroid records.** Deliberate
  semantic ("this child represents one trained centroid"), not a row
  count. A future reader will misinterpret it. Add a comment.

- **`recursive_fanout = 1` only rejected at accessor.** If anything
  else reads the raw reloption before the accessor, it gets a
  meaningless value. Validate at parse time.
