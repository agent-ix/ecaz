# Review: SPIRE Insert Schema Drift Guard (30933)

## Summary

Closes 30896 H6 — the schema-drift detection at INSERT time
that I flagged as a real production safety gap. The guard fires
**before** any remote libpq work opens, so coordinator-only DDL
can no longer leave orphaned prepared transactions.

What landed (per `369c50d1` `Add SPIRE insert schema drift
guard`):

- `coordinator_insert_shape_fingerprint` column added to
  `ec_spire_remote_node_descriptor`, bootstrap SQL, both
  upgrade migrations.
- SQL helpers hash the coordinator heap column shape from a
  table or SPIRE index. Per the request, the fingerprint
  inputs are: physical column order, name, type OID, typmod,
  collation, NOT NULL flag.
- `ec_spire_register_remote_node_descriptor` binds the
  fingerprint at registration.
- Coordinator-routed INSERT validates the descriptor-bound
  fingerprint **before** opening remote libpq work; on
  mismatch, fails closed with a schema-drift message and the
  pause/apply/refresh/retry remediation hint from 30931.
- Descriptor contract rows extended so the new field is
  operator-visible.
- PG18 fixture `test_ec_spire_schema_drift_fails_before_dispatch_sql`
  alters only the coordinator table, attempts INSERT, and
  verifies no remote row + no SPIRE prepared transaction left
  behind.

The four review-focus questions:

- **Fingerprint inputs are right v1 column-shape boundary**:
  yes — physical order + name + type OID + typmod + collation
  + NOT NULL is the canonical "what `to_jsonb(NEW)` produces +
  what `jsonb_populate_record` requires" matching set. Any
  drift in those properties changes either the JSON keyset
  (rename) or the type semantics (type OID, typmod, NOT NULL
  on receive). Worth confirming `attisdropped` is excluded —
  it should be, since dropped columns don't appear in
  `to_jsonb(NEW)`.

- **Guard runs early enough to prevent orphaned prepared xacts**:
  yes — the guard fires before `coordinator_insert_prepare_remote_*`
  opens the remote transaction. The fixture's `pg_prepared_xacts`
  empty-after-failure assertion is the load-bearing proof.

- **Binding fingerprint through descriptor registration is right
  refresh point**: yes — registration is the operator-controlled
  point where the coordinator declares "I think the remote and I
  agree on shape." Pairing with the 30931 DDL ordering contract
  (refresh descriptors as step 4) makes the operator workflow
  explicit and fail-loud.

- **Migration default/backfill behavior acceptable**: see
  findings — depends on what existing descriptors get for
  `coordinator_insert_shape_fingerprint`.

## Findings

### P2 — Migration backfill for existing descriptors

The 0.1.0→0.1.1 and 0.1.1→0.1.2 migrations add the column.
Existing descriptors will have either NULL or a sentinel for
`coordinator_insert_shape_fingerprint`. Two acceptable shapes:

- **NULL means "unknown, skip the guard"**: backwards-
  compatible, but defeats the guard for existing descriptors
  until they're re-registered. Operator must run
  `ec_spire_register_remote_node_descriptor` once per
  descriptor post-upgrade to enable the guard.
- **NULL means "guard fails closed"**: safer but requires
  every existing descriptor to be re-registered before any
  INSERT works. Operationally disruptive on upgrade.

Per the migration scripts being shown but their behavior not
detailed in the request, worth confirming which option the
code chose. If option 1, document that the guard is opt-in
post-upgrade until descriptors are re-registered.

### P2 — Guard scope: INSERT only?

The packet says the guard fires "before coordinator-routed
INSERT opens remote libpq work." Worth confirming whether
UPDATE / DELETE paths also call this guard. UPDATE and DELETE
forward similar payloads through the JSON bridge — schema
drift would cause the same wrong-results failure mode there.

If the guard is INSERT-only in this packet, that's the right
v1 scope (INSERT is the primary write path), but pin
follow-up coverage for UPDATE/DELETE in the tracker.

### P3 — Fingerprint hash function

Worth confirming the hash function is deterministic across PG
versions and architectures. SHA-256 over a canonical
representation is the safe choice. If the hash uses a
non-determinic input format (e.g., hash dependency on column
internal OIDs), upgrades that change internal OID assignment
could spuriously trigger drift.

Suggest a one-line note in the design that the fingerprint
input is canonicalized before hashing.

### P3 — Pre-dispatch validation cost

The guard adds one fingerprint-comparison per INSERT. Cheap
(a single descriptor-row lookup + hash compare), but worth
confirming this happens after the placement-directory write so
a drift-rejected INSERT doesn't leave a stale placement entry.
From the description it sounds like it fires before any
remote work, which means before the placement-directory write
too — correct.

### P3 — Request.md not yet on origin

The request.md and artifacts are present in my local working
tree but not yet on origin/task-30-spire (only the code commit
369c50d1 is upstream). I'm landing this feedback as a
pre-checkpoint review per `review/REVIEWER.md`. When the

## Reviewer recommendation

Accept. **Closes 30896 H6 — the last big production safety
gap before AWS.** Combined with H1 (descriptor refresh
SQLSTATE) and H10 (DDL ordering docs), schema-drift handling
is now end-to-end:

- Operators have a documented DDL workflow (H10).
- DDL violations are caught by fingerprint comparison before
  any remote work (H6).
- Concurrent descriptor refreshes have a stable retry signal
  (H1).

After the migration-backfill and UPDATE/DELETE-coverage P2
items, this lane is closed.
