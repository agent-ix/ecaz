## Response to Review Focus

### Is extending the bootstrap boundary to 64 elements a reasonable complexity tradeoff?

**Yes.** The `fwht64_avx2_block` helper (`hadamard.rs:288-321`) follows the same recursive compose pattern as the existing `fwht32_avx2_block`: it calls `fwht32` on both halves, then applies the width-32 butterfly with `add_ps`/`sub_ps` across 4 register pairs. This is the natural next step in the doubling chain — no new algorithmic ideas, just one more level. The 8-register input/output tuple is getting large but is still manageable since AVX2 has 16 YMM registers.

The 2.8-3.9% improvement on this host is modest but real, and folding one more outer-stage pass into register work is the right direction for cache locality.

### Is the helper-level coverage sufficient?

**Yes.** `fwht64_avx2_block_matches_scalar_when_available` (`hadamard.rs:489-526`) directly tests the 64-lane helper against scalar output on a deterministic input. Combined with the existing `fwht_runtime_path_matches_scalar_on_random_inputs` fuzzing (1000 random inputs at sizes up to 512), this is adequate.

### Does the packet overstate BC-008?

**No.** The packet explicitly says "still not stable enough on this machine to claim BC-008 closed for all representative sizes." Honest.
