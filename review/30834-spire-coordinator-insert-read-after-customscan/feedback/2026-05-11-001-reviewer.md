# Review: SPIRE Coordinator Insert Read-After-CustomScan (30834)

## Summary

End-to-end PG18 multicluster proof of the helper-based coordinator
INSERT path. The new
`run_spire_multicluster_insert_read_after_customscan_pg18.sh`:
- Provisions separate coordinator + remote PG18 clusters.
- Runs `ec_spire_prepare_coordinator_insert_tuple_payload(...)`,
  commits the coordinator txn, and verifies the remote heap row
  landed (303, 'remote inserted via coordinator').
- Asserts the placement-directory row staged with the coordinator's
  search epoch.
- Manually re-registers the descriptor identity (this slice's
  remaining caveat — closed by 30836).
- Runs `SELECT … ORDER BY embedding <#> … LIMIT 1` and asserts the
  plan is `Custom Scan (EcSpireDistributedScan)` returning the
  inserted remote row.

Closes the 30833 P2 status-naming finding directly: the helper now
returns `status = "remote_insert_prepared_pending_local_commit"` and
`next_step = "await_local_commit"`, distinguishing prepared/pending
from committed/durable. The unit test is updated to assert the new
strings, locking the contract.

The four review-focus questions:
- Multicluster script is a useful read-after-insert proof for the
  helper path: yes — covers the entire prepare → commit → CustomScan
  read path with separate coordinator/remote clusters.
- Epoch alignment legitimate: yes — coordinator stages at epoch 2,
  remote INSERT advances remote index to epoch 2, then the
  descriptor identity is refreshed so the coordinator's CustomScan
  fanout reaches the right epoch.
- Post-staging status names distinguish prepared/pending from
  committed/durable: yes (closes 30833 P2).
- Tracker leaves transparent INSERT hook open: confirmed — the
  trigger packet (30835) addresses that next.

## Findings

### P3 — Manual descriptor re-registration is a known caveat

The script still calls `ec_spire_register_remote_node_descriptor` to
refresh the descriptor identity after the remote INSERT advances
the remote index epoch. This is acknowledged as the next packet's
work (30836). For this fixture's purpose — proving the **helper**
end-to-end — that's fine. Worth confirming the script comments call
out the re-registration as a temporary scaffold so it isn't copied
into a real INSERT runbook.

### P3 — Status strings now load-bearing

The helper's two new constants (`POST_STAGING_STATUS`,
`POST_STAGING_NEXT_STEP`) are referenced by the unit test and the
multicluster fixture. Worth a one-line ADR-069 note that these
strings are part of the SQL surface contract — operators may read
them and write monitoring against them, so future renames need a
deprecation path.

### P3 — `read_row=303,remote inserted via coordinator` evidence

The fixture proves both rows in the multicluster artifacts:
descriptor row, placement row, and the CustomScan-returned tuple.
Strong evidence pattern, consistent with the 30820 multicluster
read fixture.

## Reviewer recommendation

Accept. This is the v1 read-after-insert proof for the helper path.
The trigger-front-door + descriptor-refresh slices follow on
cleanly.
