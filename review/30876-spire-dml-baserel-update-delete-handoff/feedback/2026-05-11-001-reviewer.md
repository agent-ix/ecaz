# Review: SPIRE DML Baserel UPDATE/DELETE Handoff (30876)

## Summary

Generalizes the baserel primitive-plan extraction so UPDATE,
DELETE, and PK SELECT share one path. The PK SELECT-only helper
becomes a thin wrapper that asserts the primitive plan mode is
`CoordinatorPkSelectTuplePayload`; UPDATE and DELETE will get
analogous wrappers in their own rewrite slices.

Key changes:
- `dml_frontdoor_pk_select_primitive_plan_expr_from_baserel`
  renamed to operation-neutral
  `dml_frontdoor_primitive_plan_expr_from_baserel` and accepts
  any DML operation; the PK SELECT wrapper retains the existing
  hook surface.
- New `dml_frontdoor_baserel_target_rtindex(query, rel,
  operation)` returns:
  - `Ok(Some(rtindex))` for `PkSelect` (every base relation in a
    SELECT is a candidate).
  - `Ok(Some(rtindex))` for UPDATE/DELETE only when
    `query.resultRelation == rel.relid` (the target relation).
  - `Ok(None)` for UPDATE/DELETE non-target baserels (e.g., a
    `FROM` table in `UPDATE ... FROM other`) so they pass
    through cleanly.
  - `Err(...)` for relid overflow.
- `dml_frontdoor_query_detail_from_baserel` parameterized by
  operation; populates `updated_columns` for UPDATE,
  `projected_columns` for PK SELECT, both empty for DELETE.
- Predicate extraction (RestrictInfo-first then jointree.quals
  fallback) is shared across all three operations.
- New unit test `baserel_handoff_uses_only_target_rel_for_dml`.

The three review-focus questions:
- UPDATE/DELETE baserel handoff skips non-target baserels
  instead of raising candidate errors for joined DML shapes:
  yes — `Ok(None)` from `dml_frontdoor_baserel_target_rtindex`
  for non-result relations.
- PK SELECT wrapper preserves the currently wired CustomScan
  mode boundary: yes — the wrapper asserts mode equals
  `CoordinatorPkSelectTuplePayload`, otherwise returns
  `Some(Err)` so the existing PK SELECT hook fail-closes if a
  different mode somehow leaks in.
- Shared query-detail extraction carries UPDATE updated columns
  and PK SELECT projected columns without changing DELETE
  payload semantics: yes — DELETE branches through the same
  helper but both column-list outputs stay empty, matching the
  existing primitive-plan validation that DELETE must not carry
  column payloads.

## Findings

### P2 — UPDATE/DELETE non-target rel: silent pass-through OR fail-closed?

For `UPDATE distributed_tbl SET ... FROM other_tbl WHERE ...`,
the planner hook will see two baserels: the result relation
(distributed_tbl) and `other_tbl`. The new code returns
`Ok(None)` for `other_tbl`, so it passes through. That's correct
for the "FROM target is unrelated" case, but if `other_tbl` is
itself a SPIRE-distributed table, the planner would silently
continue the standard plan against it (which would then miss its
remote rows).

Two acceptable shapes:
- Document in ADR-069 that `UPDATE target FROM other` against a
  distributed `other` table is unsupported v1 (and that the
  shape classifier already rejects joined UPDATE shapes via
  `has_join`).
- Add a focused test that confirms `UPDATE distributed_target
  FROM distributed_other` rejects via the existing `has_join`
  check (since the FROM clause makes it a join).

If the existing `has_join` check already catches this, no
behavior change needed — just confirm in a test fixture.

### P2 — `dml_frontdoor_query_has_join_shape` operation parameter

The classifier now takes the operation as a parameter and the
`has_join` helper is also operation-parameterized. Worth
confirming the helper actually branches on operation —
otherwise the parameter is dead. From the diff, it looks like
the function uses the operation for slightly different join
detection rules (PkSelect treats `FROM other` as a join,
UPDATE/DELETE treat `resultRelation != rel.relid` as a join via
the rtindex check, etc.). Worth a one-line comment naming the
per-operation join semantics.

### P3 — `Some(Err)` in PK SELECT wrapper for unexpected mode

The PK SELECT wrapper rejects with `Some(Err(...))` if the
extracted plan's mode isn't `CoordinatorPkSelectTuplePayload`.
That should never happen in practice (PK SELECT operation always
maps to PK SELECT mode), but defending against future drift is
cheap. Good.

### P3 — Test naming

`baserel_handoff_uses_only_target_rel_for_dml` is
self-describing. Worth siblings:
- `baserel_handoff_returns_none_for_pk_select_non_distributed`
- `baserel_handoff_propagates_relid_overflow_as_error`

So all three return-type variants (`None`, `Some(Ok)`,
`Some(Err)`) get explicit regression coverage.

### P3 — Acknowledged scope

Request explicitly notes "still a scaffold packet." Correct —
UPDATE/DELETE wrappers + executor wiring remain.

## Reviewer recommendation

Accept. The generalization is the right shape. UPDATE and DELETE
rewrite slices can now follow the PK SELECT pattern: add a
wrapper that asserts the expected mode, then wire the
CustomScan executor branch that calls the corresponding
primitive (`ec_spire_forward_coordinator_update_tuple_payload`
or `ec_spire_prepare_coordinator_delete_tuple_payload`).

The P2 `UPDATE ... FROM distributed_other` concern is the only
operationally-load-bearing item — confirm the existing
`has_join` rejection covers it before the UPDATE rewrite slice
lands, otherwise add explicit handling.
