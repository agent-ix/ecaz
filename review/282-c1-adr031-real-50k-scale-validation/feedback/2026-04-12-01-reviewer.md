# Review: Packet 282 — C1 ADR-031 Real 50K Scale Validation

Reviewer: agent-ix (via code-review)
Date: 2026-04-12

## Assessment: Clean scale validation — NFR-001 confirmed at 50K

### What landed

Ran the verified warm latency harness on the real 50K corpus with the cached
ADR-031 runtime path at `m=8`, `ef_search=40`. First a 200-query gate, then
the full 1000-query canonical table.

### On the results

Canonical 1000-query read:

- `p50=4.633ms` (target: <5ms)
- `p99=7.661ms` (target: <15ms)
- `mean=4.727ms`

This clears NFR-001 with room to spare on p99 and tight margin on p50.

### Scaling behavior

10K mean=2.83ms to 50K mean=4.73ms is a 1.67x slowdown for a 5x corpus growth.
This is sub-linear scaling, which is expected for HNSW — the graph traversal
visits O(ef_search * log(N)) elements, not O(N). The scaling factor suggests
the ADR-031 architecture is healthy and not hiding a linear-scan regression.

### The two-stage validation approach is good

Running 200 queries first as a gate before committing to 1000 is a cheap way
to catch catastrophic regressions. The 200-query gate (p50=4.557ms,
mean=4.655ms) tracked the canonical result closely, confirming it wasn't a
lucky subset.

### One observation for future runs

The fixture loading step (`load_real_corpus_scratch.sh`) was needed because the
scratch database didn't have the 50K queries yet. This is a one-time setup
cost, but it means the benchmark isn't fully self-contained — it depends on
having the fixture pre-loaded. Not a problem now, but worth noting if the
benchmark matrix expands.

### Verdict

Clean NFR-001 confirmation at the normative 50K scale. Correct keep/pivot
decision (quality validation is the right next question, not more latency work).
