# Review: SPIRE DML CustomScan DELETE Executor (30887)

## Summary

**MILESTONE — first transparent ADR-069 DELETE executes.**
`DELETE FROM distributed_tbl WHERE id = $1` now actually removes
the row through the existing primitive and reports the right
rowcount.

Smaller slice than the UPDATE executor because DELETE doesn't
carry SET-value expressions or row payload — just PK column +
PK value:

- DELETE access branch dispatches to
  `custom_scan_dml_delete_access`.
- Builds `SpireDmlFrontdoorPrimitiveInvocation`, asserts mode is
  `CoordinatorDeleteTuplePayload`.
- Calls `ec_spire_prepare_coordinator_delete_tuple_payload(...)`
  via SPI.
- Increments `estate->es_processed` by `remote_deleted_count`.
- Returns NULL (cleared tuple slot).

PG18 fixture: `DELETE FROM tbl WHERE id = 5` reports `ROW_COUNT
= 1`, the local heap row is gone, the placement row is gone.

The three review-focus questions:
- DELETE execution maps DML CustomScan runtime state to the
  coordinator delete primitive: yes.
- `remote_deleted_count` for `es_processed` is correct for both
  local and future remote prepared paths: yes — the primitive
  returns the same field in both branches, so the executor
  consumer doesn't need to know which path resolved.
- Fixture covers the user-visible DELETE contract for the
  current local-placement path without expanding into remote
  2PC visibility: yes — the local-placement test confirms
  rowcount + heap removal + placement removal in one local txn.

## Findings

### P3 — Symmetric structure to UPDATE executor

The branch shape mirrors `custom_scan_dml_update_access` —
build invocation, call primitive, increment `es_processed`,
return cleared slot. Good consistency; refactoring the two into
a shared "execute primitive then return rowcount" helper would
collapse some duplication, but the branches are different
enough (UPDATE needs row-payload conversion) that the current
shape is readable.

### P3 — Remote DELETE 2PC visibility

The fixture only covers local-placement (`node_id = 0`) DELETE.
Remote-placement DELETE goes through 2PC (the primitive's
existing path), so the test would need a multicluster setup to
exercise prepared-DELETE visibility. 30889 (next packet) adds
exactly that fixture.

### P3 — `dml_payload_emitted` flag controls one-shot semantics

Both UPDATE and DELETE use `dml_payload_emitted` to ensure the
primitive runs exactly once even if `ExecCustomScan` is called
multiple times by the executor loop. Sound pattern; matches the
PK SELECT one-tuple-per-scan shape.

## Reviewer recommendation

**Accept the milestone.** DELETE executor is wired; combined
with the UPDATE executor (30886), all three transparent DML
operations (UPDATE, DELETE, PK SELECT) now route through
EcSpireDistributedScan. ADR-069 v1 transparent write contract
is functionally complete — the documentation and multicluster
fixtures (30888 + 30889) close out the rest.
