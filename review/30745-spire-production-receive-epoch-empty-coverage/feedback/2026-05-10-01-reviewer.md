## Answers to the packet's review questions

- **`served_epoch_mismatch` through the production validation
  failure category the right operator taxonomy?** Yes for now. A
  typed enum would add a parallel surface to
  `failure_category: String` that the diagnostic API and
  diagnostic SQL surfaces don't share. Keeping the category as a
  named string, with `served_epoch_mismatch` joining
  `endpoint_identity_mismatch` and the C2 fault categories
  (`remote_statement_timeout`/`remote_backend_terminated`/
  `remote_query_cancelled`) in the same vocabulary, is consistent
  with the rest of the receive-side taxonomy. C5 can move to a
  typed enum if the AM scan path needs structured matching, but
  it's not a current prerequisite.
- **`top_k = 0` as a ready empty batch the right handoff contract
  for compact merge and Stage D heap resolution?** Yes. The
  alternative — failing zero-top-k receive — would force compact
  merge to special-case empty inputs, which is exactly the kind
  of branching merge should avoid. A ready empty batch composes
  naturally into merge (just doesn't contribute rows) and Stage
  D heap resolution (no candidates to resolve heap visibility
  for).
- **Do the C5 task/design additions capture the
  reviewer-requested production gates?** Yes. The PostgreSQL
  cancellation bridge (30742 P2) and the per-query
  strict/degraded threading + fault matrix (30744 P2) are both
  named in the docs with concrete acceptance criteria. C5 can
  open against a named gate set rather than re-litigating the
  contract.
