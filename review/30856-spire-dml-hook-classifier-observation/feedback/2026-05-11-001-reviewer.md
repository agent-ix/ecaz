# Review: SPIRE DML Hook Classifier Observation (30856)

## Summary

The planner hook now invokes the shape classifier in pass-through
mode, using the catalog-backed relation context (30855) instead of
SPI. Records backend-local last-classification facts and exposes
them via `ec_spire_dml_frontdoor_hook_status()`:
`query_shape_classifier_invoked_by_hook`, `last_classification_supported`,
`last_classification_kind`, `last_classification_status`. Plan
rewriting remains disabled.

PG18 fixture proves an actual `SELECT id FROM ... WHERE id = 5`
executes through the planner with the hook recording
`pk_select_by_pk` / `supported_v1_shape`.

The three review-focus questions:
- Invoking the classifier from the planner hook is safe while plan
  rewriting is disabled: yes — observation only, no plan tree
  mutation, no SPI, no relcache invalidation.
- Hook does not use SPI and only depends on the catalog/relcache
  loader: confirmed.
- SQL-visible hook status fields sufficient for the next executor
  replacement slice: yes — operators can see "the hook saw N
  classifications, last was X" via SQL.

## Findings

### P3 — Backend-local last-classification statics are best-effort

`LAST_HOOK_CLASSIFICATION_*` are `static mut` per backend. Useful
for "did my session just classify a query?" diagnostics but won't
help cross-session debugging. Worth a one-line comment that this
is intentionally per-backend (not shared), so a future
multi-session test doesn't expect cross-process visibility.

### P3 — Observation overhead per planner call

The hook now opens the heap relation (via the catalog-backed
loader) on every planner invocation that targets a
front-door-eligible relation. For a high-throughput workload,
that's one `table_open(AccessShareLock)` + `RelationGetIndexList`
per plan. Acceptable for v1 — these are cheap relcache hits — but
worth a note that a future optimization could skip the catalog
walk when the relation isn't marked as ec_spire (the existing
`get_index_am_oid("ec_spire", true)` lookup is a quick gate).

## Reviewer recommendation

Accept. This is the right minimal step before fail-closed guarding
(30861) and plan replacement (next packet).
