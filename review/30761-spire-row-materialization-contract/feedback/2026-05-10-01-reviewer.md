# 30761 — SPIRE row materialization contract review

Validation pass on `bdc63497` (`Expose SPIRE remote row
materialization contract`). Accepts the SQL-visible contract
surface for remote-origin row delivery.

## What I checked

## Answers to the packet's review questions

- **Is the same-indexed-heap shadow/proxy row requirement
  the right AM-compatible contract for remote-origin result
  delivery?** Yes. PostgreSQL's index AM contract requires
  `xs_heaptid` to point at a heap row in the *currently
  scanned heap relation*. There are exactly two ways to
  satisfy that for remote-origin candidates:
  1. **shadow/proxy row** — coordinator inserts a local heap
     row that proxies the remote, with TID in the local
     relation. The AM returns that TID; downstream tuple
     fetch reads the proxy. This is the v1 path the contract
     pins.
  2. **non-AM integration** — exit the AM contract entirely
     (FDW, custom scan, or a separate executor surface) and
     deliver tuples through a different mechanism.
  Mixing them inside one AM scan is exactly the failure mode
  packets 30757/30758 are guarding against; the contract
  pinning shadow/proxy as the v1 AM path is the right
  decision.
- **Contract explicit enough that origin-node heap
  coordinates cannot be accidentally treated as coordinator
  `xs_heaptid` values?** Yes. Three places now enforce this:
  - the executor classifier (30757) refuses to mark
    remote-origin rows as `coordinator_local_heap`;
  - the AM loader gate (30758/30760) rejects manifest
    directories with remote placements before reaching the
    legacy local-heap cursor;
  - this contract surface (30761) makes the rule
    operator-visible and machine-readable so future code
    cannot drift.
  Three independent enforcement points is appropriate
  belt-and-suspenders for an ADR-059 invariant.
- **FDW/custom executor delivery kept sufficiently separate
  from the v1 index AM path?** Yes. The contract names FDW/
  custom executor as a *future* surface, not a v1 fallback.
  This avoids the trap of building a half-FDW path that
  inherits AM constraints without AM benefits.
- **Blocker/status names consistent with packets 30757,
  30758, 30760?** Yes. All four use
  `remote_row_materialization` (step) and
  `requires_remote_row_materialization` (status) from the
  same constant pool. Verified by grep across the four
  files.

## P2 — close before remote materialization implementation

1. **Shadow/proxy row lifecycle.** The contract pins
   *what* a v1 implementation must produce but not *when*.
   Worth a follow-up that pins:
   - lifetime: per-query / per-cursor / per-transaction?
     (probably per-cursor — proxy rows shouldn't outlive the
     scan that produced them);
   - storage: temp table / unlogged scratch relation /
     in-memory tuple-store? (probably the existing local
     heap relation extended with a transient namespace, but
     this needs explicit design);
   - cleanup: when does the coordinator release proxy rows
     that downstream operators have finished reading?
   These design questions should land as an ADR or design
   doc before the implementation slice opens, otherwise the
   first implementation will lock decisions that are easier
   to revise on paper.
