# Feedback: SPIRE Scan Leaf Route Store Grouping (30519)

Scope: code review of `2224411c` (`Group SPIRE scan leaf routes by local
store`) plus the follow-on `3cc56716` (`Group SPIRE scan leaf and delta reads
by store`) on `task30-spire-partition-object-spec`. Files inspected:
`src/am/ec_spire/scan.rs`, `src/am/ec_spire/meta.rs`.

Cross-cutting Phase 4 / Task 30 cadence and architecture feedback is filed
separately under `review/30509-spire-phase4-local-placement-design/feedback/`.

## 1. The grouping primitive is a no-op as currently wired

In `collect_quantized_routed_probe_candidates` (scan.rs:670–714) the new
grouping is consumed as:

```rust
for route_group in group_leaf_routes_by_local_store(snapshot, leaf_routes)? {
    for route in route_group.routes {
        let leaf_pid = route.leaf_pid;
        ...
        append_quantized_leaf_candidates_for_pid(...)?;
        append_quantized_delta_candidates_for_base_pid(...)?;
    }
}
```

The grouping changes iteration order, but the inner work is unchanged: each
PID is fetched and scored independently exactly as before. There is no
`read_objects_for_store_group(group)` consumer, no batch I/O surface, no
parallel fetch entry point. The store boundary is *placed* but does not yet
*do* anything that depends on grouping.

This matches the larger Phase 4 cadence pattern (see 30509 feedback): the
seam landed before any consumer of the seam exists. Recommendation:

- Do not extend `group_leaf_routes_by_local_store` further (boundary
  shape, additional fields, more callers) until at least one fetch
  surface actually batches reads at the group grain.
- Treat the current shape as provisional. The first real consumer
  (parallel local fetch) is likely to want a different decomposition
  than "iterate groups, iterate routes, do the same per-PID work."

## 2. Two near-identical group structs

`group_leaf_routes_by_local_store` (scan.rs:716) is a thin wrapper around
`group_leaf_and_delta_reads_by_local_store` that calls it with an empty
`delta_routes` and remaps the result from `SpireStoreObjectReadGroup` into a
separate `SpireStoreLeafRouteGroup`. Two struct flavors carrying the same
`(node_id, local_store_id, routes)` payload is the helper-matrix smell at
the type level.

Recommendation: collapse to one type. `SpireStoreObjectReadGroup` with
optional empty `delta_routes` already covers the leaf-only callsite — drop
`SpireStoreLeafRouteGroup` and the wrapper. The current shape commits the
codebase to maintaining two parallel grouping APIs that diverge later for no
real reason.

## 3. `BTreeMap` keying is fine; verify the determinism claim

`reads_by_store: BTreeMap<(u32, u32), ...>` keyed on
`(node_id, local_store_id)` does give deterministic iteration. The packet
description claims "deterministic store-key order plus per-store
route-order preservation" — confirmed by inspection (the input
`leaf_routes` `Vec` order is preserved within each entry by `push`). The
existing tests
(`group_leaf_routes_by_local_store_orders_stores_and_preserves_route_order`
at scan.rs:3571) cover this.

One follow-up: when delta routes also flow through the combined function,
the order between leaf-only stores and delta-only stores is "stores keyed
by `(node_id, local_store_id)`," not "all leaf-bearing stores first." That
is fine, but if a future scan stage wants to issue leaf reads ahead of
delta reads to overlap latency, the current shape will need a second
traversal axis. Worth a comment in the helper noting that the group order
is store-keyed, not phase-keyed.

## 4. Selected-leaf filtering of delta routes

`group_leaf_and_delta_reads_by_local_store` (scan.rs:732–777) filters
delta routes whose `parent_leaf_pid` is not in the selected leaf set. That
is the right behavior for routed scan, but the filter is silent — there is
no diagnostic, no count, no observer event for "delta routes dropped
because parent leaf was not selected." If a delta is unexpectedly orphaned
(directory bug, replacement-epoch race) the scan will silently produce
fewer candidates instead of surfacing the discrepancy.

Recommendation: count dropped delta routes and surface through the scan
observer or an existing diagnostics counter. This is cheap and makes a
real-world regression detectable without re-running with a debugger.

## 5. Validation lane

The packet validates with `cargo test group_leaf_routes_by_local_store
--lib` and `cargo test collect_quantized_routed_probe_candidates --lib`.
Both are pure unit tests against the helper and its first consumer. There
is no end-to-end test where two store relations actually exist and a scan
groups across them — because no two-store fixture exists yet. That is
honest scoping for this packet, but it should be flagged in the Phase 4
tracker as a load-bearing gap before anyone claims the grouping primitive
is "validated."

## Net

The structural placement of the seam is reasonable. The seam is currently
inert. Do not grow it further until a real consumer requires it; collapse
the duplicate group struct; add diagnostics for the silent delta-route
drop. Cross-cutting Phase 4 cadence concerns are in the 30509 feedback
file.
