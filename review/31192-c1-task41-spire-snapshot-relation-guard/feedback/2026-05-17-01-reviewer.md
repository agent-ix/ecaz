# Review: Task 41 SPIRE snapshot relation guards (31192)

## Summary

Migrates SPIRE coordinator aux-store snapshot and cleanup loops
in `src/am/ec_spire/coordinator/snapshots.rs` from manual
`relation_open` / `relation_close` pairs to the
`RelationGuard` from 31191. Introduces a helper
`open_storage_relation_or_index` that returns the root index
relation directly when the storage relid equals the index relid,
and otherwise opens an aux-store via `RelationGuard::try_open`,
returning `(pg_sys::Relation, Option<RelationGuard>)`. Baseline
4,251 → 4,245 (−6).

The helper is the interesting part of this packet: it
encapsulates the "open if different, borrow if same" decision
that the manual code previously re-implemented at every call
site. Clean abstraction.

## Confirmations

- **Helper returns owned guard when it opens.** When
  `storage_relid != index_relid`, the helper returns
  `(guard.as_ptr(), Some(guard))`. The caller binds the guard
  to `_storage_relation_guard` (underscore-prefixed because it's
  not read directly), which keeps it alive for the loop
  iteration. ✔
- **Helper returns `None` guard for the borrow case.** When
  `storage_relid == index_relid`, returns `(index_relation,
  None)`. Caller binds `None` to the same `_storage_relation_guard`
  slot — no Drop runs for that arm; the root index relation
  stays owned by the outer scope (correct, the caller never
  owned it). ✔
- **Per-iteration guard scope.** Each `for storage_relid in ...`
  iteration calls the helper, scans, and the guard drops at
  iteration end. No accumulation of aux-store locks across the
  loop. Matches the prior explicit close at the iteration tail.
  ✔
- **Lockmode preserved.** First call site
  (`index_relation_storage_snapshot`) uses AccessShareLock;
  second (`collect_physical_cleanup_candidates`) uses
  RowExclusiveLock. Both are passed verbatim to the helper. ✔
- **Error-path Drop.** The `scan_result?` after the loop body
  propagates errors; if propagation happens before the guard
  drops naturally, Drop runs as the error unwinds. Same
  Drop-on-unwind invariant as the rest of Task 41.
- **Baseline arithmetic.** −6 across two loops with two
  open/close pairs each plus a few intermediate SAFETY-annotated
  reads. Consistent.

## Findings

No blockers.

### 1. Helper signature note: raw pointer + Option<Guard> pattern

The helper signature is:

```rust
fn open_storage_relation_or_index(...)
    -> Result<(pg_sys::Relation, Option<RelationGuard>), String>
```

Returning a raw pointer alongside the owning guard is the right
practical shape — it avoids dispatching on `Option` at every
read site — but it means the borrow checker can't verify that
the raw pointer outlives the guard. The convention is
load-bearing: callers must drop the `Option<RelationGuard>`
**after** they're done with the raw pointer.

In this packet both call sites bind the guard with `let
(storage_relation, _storage_relation_guard) = ...?;` and the
guard outlives the raw use, so it's correct. But the helper
could be safer with a small wrapper type:

```rust
enum StorageRelation<'a> {
    Borrowed(pg_sys::Relation),  // root index, owned by caller
    Owned(RelationGuard),
}
impl StorageRelation<'_> {
    fn as_ptr(&self) -> pg_sys::Relation { ... }
}
```

That gives a single accessor and forces callers to keep the
enum alive until last use. Stylistic — current code works; flag
as an option if a third call site lands and a similar pattern
appears elsewhere.

### 2. Aux-store walks are now a clean idiom

The "open, scan, close per iteration" pattern in
`index_relation_storage_snapshot` and
`collect_physical_cleanup_candidates` is exactly the shape that
will appear in more aux-store iterators as SPIRE work continues.
Worth promoting this helper (`open_storage_relation_or_index`)
into `src/storage/relation_guard.rs` if a third caller appears,
since it's not coordinator-specific.
