# Review: Task 41 HNSW heap rerank RAII state (31221)

## Summary

HNSW analog of 31220. Replaces the raw grouped-heap-rerank
fields on `TqScanOpaque` (`Relation`, owned booleans,
`Snapshot`, raw slot pointer, source attnum, source kind) with
**one boxed `GroupedHeapRerankState`** owning the same shape
of typed guards.

## Confirmations

- **Same composite-RAII-state pattern as 31220.** Consistency
  between IVF and HNSW heap-rerank state encodings will help
  future audits.
- **`source_kind` preserved.** Unlike IVF, HNSW grouped rerank
  carries a `source_kind` discriminant — the boxed state
  preserves it alongside the guards.
- **Production scan path.** Every HNSW scan that uses grouped
  heap-f32 rerank goes through this state.

## Findings

### 1. (carries from 31220) Verify field declaration order

Same drop-order concern as 31220 finding #1. The slot must
drop before snapshot/relation; for Rust's reverse-declaration
drop semantics to enforce that, slot must be declared **last**
in the struct. Worth verifying field order and adding a one-line
comment anchoring the intent.

### 2. (carries from 31220) Composite RAII state idiom worth
strategy-noting

This is the second packet to land the "compose guards into a
state struct with a nullable raw pointer from the C-side opaque"
pattern. Worth a single-line note in the 31161 strategy doc as
the preferred shape for scan / vacuum / build lifecycles that
own multiple PG resources.
