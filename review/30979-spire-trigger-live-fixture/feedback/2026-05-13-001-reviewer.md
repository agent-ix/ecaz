# Review: SPIRE Trigger Live Fixture (30979)

## Summary

Resolves blocker 1 from 30978. The
`insert-read-after-customscan-pg18 --insert-mode trigger`
fixture's `coordinator_row_count=1` failure was a **fixture
bug, not a product bug**:

- assertion used `WHERE id = 303`, which is a distributed
  PK-select shape that reads through CustomScan, observing
  the remote row;
- now uses `WHERE id + 0 = 303`, which is heap-only;
- normal readback still uses CustomScan KNN path.

Key passing lines:
- `insert_result=trigger_insert_committed`
- `coordinator_row_count=0` (heap suppression confirmed)
- `remote_row=303,remote inserted via coordinator`
- `plan=Limit -> Custom Scan (EcSpireDistributedScan)`
- `read_row=303,remote inserted via coordinator`

## Findings

### P1 — Root cause attribution is correct

The original `WHERE id = 303` was matching one of the SPIRE-
fronted PK-select shapes that the planner hook routes to
CustomScan. So the "heap row count" assertion was actually
asking CustomScan to fetch the remote row — which it
correctly did — making it look like local heap had a row
when local heap was empty.

This is a **subtle fixture bug** that the BEFORE-trigger
heap-suppression product path is not responsible for.
30973's BEFORE ROW + AFTER STATEMENT trigger pair correctly
suppresses local heap rows; the original fixture's
assertion just couldn't see that because its query was
routed away from heap.

Right diagnosis. The PG18 unit fixture passing alongside
this live failure was the load-bearing signal that the
product path was fine and the fixture assertion was
suspect.

### P1 — `WHERE id + 0 = 303` is the right heap-only escape

The trick: `id + 0` produces an expression that doesn't
match any of the SPIRE-fronted PK-select classifier
patterns (which expect bare `WHERE id = constant`). The
planner falls back to a regular heap seqscan, which is
what the assertion needs to actually count heap rows.

Three reasons this is acceptable:

- It's a **fixture** assertion, not production code.
  Production reads should go through CustomScan; the
  fixture specifically wants to bypass that to count
  heap.
- The expression form is robust against the classifier:
  any `+ 0` form fails the bare-equality match.
- It's clearly intentional — a future maintainer reading
  `id + 0 = 303` in a fixture will recognize it as "force
  heap path" rather than as a typo.

Worth a one-line comment in the fixture explaining this is
deliberate (`-- forces heap path; bare 'id = 303' would
route through CustomScan`). Otherwise a future "cleanup"
pass might "simplify" it back to the broken form.

### P2 — Normal readback path unchanged

The normal readback assertion still uses the CustomScan
KNN path and reports `read_row=303` and `plan=Limit ->
Custom Scan (EcSpireDistributedScan)`. So the fixture
exercises both:

- heap suppression (via heap-only assertion);
- distributed read (via CustomScan KNN).

These are two different invariants of the same trigger-
mode INSERT, both load-bearing. Right coverage.

### P2 — Blocker 1 from 30978 closed

This packet resolves the trigger-mode blocker honestly:

- diagnosed it as a fixture-side issue;
- fixed the fixture;
- re-ran with passing evidence;
- preserved both heap suppression AND distributed read
  proof.

30978's blocker manifest can drop blocker 1 from the
"needs investigation" list. The remaining 30978 blockers

### P3 — Alternative heap-only patterns

For comparison, other heap-only escape patterns the fixture
could use:

- `WHERE id IS NOT DISTINCT FROM 303` — likely also
  routes to CustomScan after PK extraction;
- `WHERE id::text = '303'` — coercion would change the
  shape; might or might not bypass classifier;
- `WHERE (SELECT id FROM phase12_table WHERE id = 303
  LIMIT 1) = 303` — definitely heap-only via subquery.

`id + 0` is the cleanest. The arithmetic-on-PK pattern is
a recognized "force heap path" idiom in PostgreSQL test
suites for similar reasons (forcing seqscan past an
indexed-equality predicate). Idiomatic enough that a
PostgreSQL-aware reader will recognize it.
