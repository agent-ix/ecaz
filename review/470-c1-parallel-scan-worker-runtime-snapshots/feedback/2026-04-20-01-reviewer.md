# Feedback: Parallel Scan Worker Runtime Snapshots

**Reviewer:** Claude (Opus 4.7)
**Date:** 2026-04-20
**Head reviewed:** `0a8da4e` (landed; reviewed retrospectively)

## TL;DR for the agent

**Merged correctly.** Publishing at scan lifecycle boundaries
(bind/reset/rescan-setup/linear-fallback/exhaustion) is the right
tradeoff. Hot-path per-event publishing would impose contention
without a clear consumer today.

## Verification performed

- `publish_parallel_scan_worker_slot_runtime_snapshot`
  (`src/am/common/parallel.rs:1280–1325`) guards on both
  `observed_rescan_epoch == rescan_epoch` and `flags == CLAIMED` before
  writing. Correct — a released or stale slot cannot be published into.
- Each field is an independent `AtomicU32` with `Release` store. Reader
  side uses `Acquire` loads (`load_worker_slot_snapshot`,
  `parallel.rs:622`). A concurrent reader can observe a torn snapshot
  across fields (e.g. new `visited_count` but old `emitted_count`) —
  this is fine for **diagnostic** rollup (EXPLAIN, pgstat) but would be
  insufficient for any consumer that needs a consistent
  multi-field view. Flag for later, not a current bug.

## Findings

### 1. Lifecycle-boundary publish cadence is the right staged choice

**Severity:** Informational (positive)

The alternative (hot-path updates on every beam step) would require
either a per-step atomic store or a thread-local staging buffer. Given
no consumer reads these yet, the current 5-point cadence is a clean
shared contract. If later EXPLAIN rollups need per-event cadence, the
switch is local to this file.

### 2. Field-level torn reads

**Severity:** Low (not exploitable today)
**File:** `src/am/common/parallel.rs:622–640, 1296–1324`

`load_worker_slot_snapshot` reads nine AtomicU32s sequentially without
a version/seqlock. A reader interleaved with a publish can see a
field-mixed view (new `visited_count` + old `emitted_count`). Today
this is harmless because (a) no consumer exists and (b) publishes only
happen at lifecycle boundaries, not hot path. If a real consumer is
wired up later, consider a 2-word seqlock (`publish_generation`) around
the runtime snapshot group, or accept the torn-read semantics
explicitly in consumer code.

### 3. `active_result_has_current: u32` vs `bool`

**Severity:** Trivial
**File:** `src/am/common/parallel.rs:81, 186, 1320–1323`

The atomic is `AtomicU32` carrying a 0/1 flag. Fine — AtomicBool would
be cleaner but requires the same backing width on target platforms.
Not worth changing.
