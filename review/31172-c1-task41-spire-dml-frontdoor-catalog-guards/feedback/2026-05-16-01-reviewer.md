# Review: Task 41 SPIRE DML frontdoor catalog guards (31172)

## Summary

Wraps the manual `index_open`/`index_close` pairs in
`src/am/ec_spire/dml_frontdoor/mod.rs` (the catalog-walk and
key-column extraction paths). Introduces a **module-local**
`AccessShareIndexRelation` struct with the same name as the one in
`src/lib.rs`, but a different `open` signature returning
`Option<Self>` instead of erroring on null (because the catalog
walk needs to `continue` past null index opens, not abort).
Baseline 4,351 → 4,347 (−4); module-local `dml_frontdoor/mod.rs`
baseline 164 → 160.

Migration itself is correct. I have a substantive concern about
the duplication shape, captured in finding #1 below.

## Confirmations

- **Catalog walk guard scope is correct.** In
  `dml_frontdoor_catalog_index_and_pk`, each iteration opens its
  own guard via `let Some(index_relation) = AccessShareIndexRelation::open(...) else { continue; };`.
  The guard drops at end of loop iteration — before the next
  `index_open` call. Borrowed `rd_index` / `rd_rel` references via
  `(*index_relation.as_ptr()).rd_index.as_ref()` are scoped within
  the iteration body and don't escape. ✔
- **Key-column extraction returns owned strings.**
  `dml_frontdoor_index_key_column_names_from_rel` builds `Vec<String>`
  inside the unsafe block with guard alive, then drops guard at
  function return. The `Vec<String>` is owned, no borrow on the
  relation. ✔
- **Null-open behavior preserved.** Previously the function
  returned `Err(...)` on null open; the new code preserves that via
  `let Some(...) = AccessShareIndexRelation::open(...) else { return Err(...) }`.
  Same semantics.
- **Loop-body early-`continue` correctness.** Previously the loop
  body did `if index_relation.is_null() { continue; }` before the
  manual close. The new code's `let Some(...) else { continue; }`
  preserves this exactly — no relation opened, no close needed for
  the skipped iteration.
- **Baseline arithmetic.** −4 is plausible: two callsites
  previously held ~2 SAFETY-annotated lines each
  (`index_open` + `index_close`).

## Findings

### 1. (substantive) Three `AccessShareIndexRelation` types in three files

With this packet plus 31173 (`DebugIndexRelation`), there are now
**three near-identical guard structs** living in three files:

| File | Type | Lockmode | Null behavior |
| --- | --- | --- | --- |
| `src/lib.rs:222` | `AccessShareIndexRelation` | Fixed AccessShareLock | `pgrx::error!` |
| `src/am/ec_spire/dml_frontdoor/mod.rs:197` (new) | `AccessShareIndexRelation` | Fixed AccessShareLock | Returns `None` |
| `src/am/ec_spire/coordinator/debug.rs:5` (new in 31173) | `DebugIndexRelation` | Variable | `pgrx::error!` |

Each is ~25 lines of nearly identical body. The two named
`AccessShareIndexRelation` will be a particular source of confusion
in stack traces, IDE jump-to-def, and future refactors — they
shadow each other only because they live in different modules.

Suggest a small shared module (e.g.,
`src/internal/relation_guard.rs` or extending `src/storage/`)
exposing **one** parametric guard with two open variants:

```rust
pub struct IndexRelationGuard { relation: pg_sys::Relation, lockmode: pg_sys::LOCKMODE }

impl IndexRelationGuard {
    /// Opens; errors via pgrx::error! on null.
    pub fn open(oid, lockmode) -> Self { ... }
    /// Opens; returns None on null (for catalog walks that `continue`).
    pub fn try_open(oid, lockmode) -> Option<Self> { ... }
    pub fn as_ptr(&self) -> pg_sys::Relation { ... }
}
```

That consolidates all three call shapes (AccessShareLock-only with
`error!`, AccessShareLock-only with `Option`, variable lockmode
with `error!`) behind one type. The existing
`open_valid_ec_*_index_guard` helpers in `src/lib.rs` keep their
AM-validation logic but build atop this primitive.

I don't think this is a 31172 blocker — the migration is correct
— but the proliferation is real and will accelerate as the buffer/
WAL track lands its own RAII wrappers next. Worth flagging in the
31161 strategy doc as an early-Task-41 consolidation item.

### 2. SAFETY note duplication

The module-local guard's `Drop` impl includes the additional
"pgrx ERROR paths must unwind Rust frames so Drop runs" SAFETY
note (line ~225 of the diff) that I called out as missing on the
`src/lib.rs` version in my 31151 feedback. Good — the contract is
documented here. But if the consolidation in finding #1 happens,
that note moves to the central location. Net win.

### 3. Loop-iteration drop ordering note

The catalog walk loop opens a new guard per iteration. Drop
runs at end of each iteration body, then the next iteration opens
the next guard. There's no overlap, no double-hold, and no risk
of accumulating relation pins across the loop. Worth noting for
future similar callers — the per-iteration scope is what keeps
the catalog walk bounded.
