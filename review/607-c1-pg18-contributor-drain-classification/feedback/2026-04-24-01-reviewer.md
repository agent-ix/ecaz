## What This Does

Classifies each contributor poll-limit exit into one of five reasons by
inspecting shared coordinator state at exit time:

- `MissingHidden` — contributor has no hidden row staged when it times out
- `DuplicateActive` — hidden row is already emitted
- `HandoffReady` — hidden row was in position to drain (not observed)
- `OrderedAfterVisible` — hidden row loses to the visible owner in score order (not observed)
- `NoVisibleOwner` — no selected/admitted visible owner row exists at classification time

All 4 poll-limit exits classify as **`No Visible Owner`**.

---

## Key Finding

The blocking reason is not score-ordering (`OrderedAfterVisible` = 0). The
emitter's visible owner slot is already gone by the time contributors try to
drain. The emitter finishes its 16-row limit and releases its coordinator slot
before contributors can stage useful handoffs.

This is a **timing gap**, not an ordering predicate problem. The current
drain-check only fires while the emitter holds an active visible owner slot.
With a 512-row fixture and 17 bootstrap expansions, the emitter is done in ~14ms;
contributors are still warming up when the window closes.

## Implications for Next Step

The `OrderedAfterVisible` = 0 result means the ordering predicate in
`pending_output_orders_before` is not the active blocker. The problem is earlier:
contributors publish hidden rows after the drain window has already closed.

The fix must either:

1. **Keep the emitter's drain window open longer** — the emitter should check for
   staged hidden contributor rows after each tuple return, including after the
   last row, rather than relying on contributors to beat it to publication.
2. **Contributors publish earlier in their traversal** — if contributors can
   stage useful rows during bootstrap rather than after, they might land in the
   window before the emitter closes.
3. **Emitter actively drains contributor rows inline during amgettuple** — per-row
   drain check on each `amgettuple` call, not just at scan boundary.

Option 3 is the most tractable: it doesn't require changing contributor timing,
just making the emitter check during its own per-tuple loop rather than only
at scan end.
