## Findings

### 1. Cost model recalibration is directionally correct тАФ sanity-check the dead inputs

The old formula charged a graph phase of `tree_height * m + ef_search *
2 * m` pages and a full-index `tuple_estimate * dimensions *
cpu_operator_cost` linear CPU sweep on top. That over-counted on both
axes (graph_pages was ~4├Ч what an HNSW walk actually touches, and the
linear-CPU term double-charged graph-covered pages). The new formula:

```rust
graph_pages = tree_height + ef_search;
graph_cpu = graph_pages * cpu_operator_cost * dimensions;
linear_pages = (index_pages - graph_pages).max(0.0);
linear_fraction = linear_pages / index_pages;
linear_cpu = tuple_estimate * cpu_operator_cost * dimensions * linear_fraction;
```

That is closer to the physical access pattern: one page per upper-layer
greedy descent step + `ef_search` candidate pages at layer 0, plus a
fallback that charges the *uncovered* page fraction of CPU work.
Spot-checked mentally at three points:

- 10 K rows ├Ч 1536 dim ├Ч `ef_search = 40`, `tree_height = 4`: graph тЙИ
  `44 * random_page_cost + 44 * 0.0025 * 1536 тЙИ 345`. Seqscan CPU тЙИ
  `10000 * 1536 * 0.0025 тЙИ 38400`. Index wins by two orders of
  magnitude тАФ matches AC-1 assertion.
- 50 rows ├Ч 4 dim: graph тЙИ `44 * 4 + 44 * 0.0025 * 4 тЙИ 176`. Seqscan
  тЙИ `~1.5`. Seqscan wins by 100├Ч тАФ matches AC-2 assertion.
- 50 rows ├Ч 1536 dim: graph тЙИ `345`, seqscan тЙИ `192`. Seqscan still
  wins but only by ~1.8├Ч. Tight but correct-direction.

**Important observation:** the new formula drops `m` from the graph
input surface entirely. `PlannerCostInputs.m` is still plumbed through
from reloptions into `estimate_planner_cost`, but the function body no
longer reads it after the recalibration. The compiler caught it (`let m
= f64::from(...)` was removed). That is *correct* тАФ the HNSW access
pattern depends on `ef_search`, not `m` тАФ but it means `m` is dead
input. Either remove it from `PlannerCostInputs` (and the call sites
that plumb it through), or keep it with a doc comment saying "reserved
for future calibration, not currently used." Today it looks like an
input that matters but doesn't.

Note on the 10K regression: because the cost crossover is driven by
`dimensions * cpu_operator_cost * tuple_estimate` for seqscan and
`graph_pages * cpu_operator_cost * dimensions` for tqhnsw, the test's
seed-specific `dimensions = 64` is load-bearing. A future regression
that accidentally lowers the test to 4 dim would flip the crossover
(seqscan at 4 dim ├Ч 10K rows would outcompete the graph phase's ~44 ├Ч
4 CPU units). Add a comment in the test saying "64 dimensions is
required for the cost crossover at 10K rows; lowering will silently
flip the plan to seqscan." One line, prevents a future shallow
"optimization" from defeating the AC.

### 2. `NoLock` open/close in `tqhnsw_amcostestimate` is correct but narrow

The packet's Review Focus #1 asks whether `pg_sys::index_open(...,
NoLock)` / `index_close(..., NoLock)` is the right shape. Verified
against the standard PostgreSQL pattern: the planner calls
`amcostestimate` from inside `cost_index`, which is called after
`get_relation_info` has already taken `AccessShareLock` on the index.
Re-taking any lock in `amcostestimate` would be redundant and is
actively wrong in `pgvector`'s `hnswcostestimate` тАФ they use
`NoLock` too. So: correct.

One subtle hazard worth noting but not blocking: `index_open` with
`NoLock` still increments the relcache refcount, so `index_close` with
`NoLock` is *required* for refcount cleanup. If a future refactor
returns early from `compute_amcostestimate` on a panic or error path
that bypasses the `index_close`, the relcache will leak. Today the
single `pgrx::pgrx_extern_c_guard` wrapper catches panics, but
`compute_amcostestimate` can also fail inside `read_metadata_page` (if
the page is corrupt) via `pgrx::error!`, which longjmps past the
`index_close` call. Under pgrx's error handling this is ultimately
fine because the transaction abort path cleans the relcache, but a
defensive `PG_TRY / PG_CATCH` equivalent would be stricter. Low
priority тАФ match pgvector's practice, which is what this code does.

### 3. `contains_cached` TOCTOU is cosmetic тАФ the split is OK but suboptimal

Review Focus #3 asks about the double-lock pattern:

```rust
let cache_hit = ProdQuantizer::contains_cached(dim, bits, seed);  // lock #1
let quantizer = ProdQuantizer::cached(dim, bits, seed);            // lock #2
// ...
if cache_hit { opaque.explain_counters.record_quantizer_cache_hit(); }
```

The window between the two locks is a miss that becomes a hit тАФ another
scan on another connection could race in an insert. That only affects
the *counter* (observability), not correctness: `cached()` always
returns the authoritative Arc, hit or miss.

But there is a cleaner alternative that removes the TOCTOU window and
the double lock acquisition: change `ProdQuantizer::cached` to return
`(Arc<Self>, bool)` where the `bool` is "was present before this call."
That is a single mutex acquisition, exact hit/miss signal, and the call
site is marginally simpler. Worth doing in a cleanup slice because
`cached()` is called from exactly one place today тАФ no cascading churn
тАФ and it eliminates a real "false positive cache hit" signal under
concurrent scans.

**Recommended follow-up:** replace `contains_cached` + `cached` with
`cached_with_presence() -> (Arc<Self>, bool)`. One call site to update
in `store_scan_prepared_query`. Not a blocker.

### 4. Explain counters are write-only тАФ add a test-only observability seam

The packet is explicit that counters are not yet surfaced to PG EXPLAIN
(that lives on the PG18 `explain_per_node_hook` registration), and I
agree that's out of D2 scope. But they are also not observable to
tests, which means every counter increment site can silently regress
without catching anything.

Concrete scenario: someone refactors
`materialize_graph_result_candidate` and accidentally drops the
`record_element_scored()` call. Nothing fails. The test suite stays
green. The counter then reports `0` forever тАФ an issue that is invisible
until the PG18 hook registration lands and someone notices EXPLAIN
shows `Elements scored: 0`.

**Recommended follow-up:** add a test-only debug entry point (gated on
`#[cfg(any(test, feature = "pg_test"))]`) that reads
`TqExplainCounters` out of the scan opaque after an `amgettuple` drive.
Write one SQL/SPI test that runs a scan end-to-end and asserts at
least one of `bootstrap_expansion`, `element_scored`, `heap_tid_returned`
is non-zero per scan. That converts counter-increment correctness from
"trust the grep" into "one regression test." Five-line debug helper,
one 20-line test. File on D2's follow-up slice, not a blocker here.

### 5. `LinearPrefetchState` is functionally ephemeral today тАФ forward-compatible, not yet PG18

Review Focus #2 asks about the resume semantics. Tracing:

- `amrescan` calls `reset_linear_prefetch_state(opaque)` once per
  rescan, seeding `[FIRST_DATA_BLOCK_NUMBER, scan_block_count - 1]`.
- Inside `select_next_linear_scan_result`, the state is *re-reset* at
  the top of every call from `[opaque.next_block_number,
  scan_block_count - 1]`, *then* iterated via `next_block()`.

So the carrier is reset on entry to every `select_next_linear_scan_result`
call тАФ it's functionally ephemeral within one call. The authoritative
cursor across calls is still `opaque.next_block_number`, which is
updated as the scan advances (presumably inside `emit_materialized_output`
or equivalent). The `LinearPrefetchState` is currently a (per-call)
iterator over a pre-computed block range.

That is the *right* intermediate shape for PG17: it compiles against
the real API, exercises the struct, and is correct. But the comment
in the code is worth sharpening: the carrier does not own the cursor,
`next_block_number` does, and the carrier will become the cursor only
once PG18 ReadStream lands. A one-line comment on `next_block_number`
saying "this is the authoritative cursor until PG18 ReadStream flips
ownership to `LinearPrefetchState`" is enough to prevent a future
author from deleting `next_block_number` under the misapprehension
that the carrier replaced it. Low priority.

### 6. Graph prefetch state lifecycle is correct

`free_graph_prefetch_state` mirrors `free_bootstrap_expansion`:
`Box::from_raw` + drop + null. `reset_graph_prefetch_state` lazily
allocates on first rescan and `reset(Vec::new())` on subsequent. Paired
with `tqhnsw_amendscan`'s `free_graph_prefetch_state` call, the
allocation is balanced.

The one thing worth flagging: on the rescan path, if the prior rescan
left `graph_prefetch_state` non-null, the new `reset()` will re-use
the existing `Box` rather than drop and re-allocate. That's the
intended optimization тАФ `GraphPrefetchState::reset()` reuses the
`Vec<u32>` allocation. Good hygiene. But `reset(Vec::new())` passes a
*new* empty `Vec` rather than clearing the existing one in place;
depending on `GraphPrefetchState::reset`'s implementation, that may
swap the old allocation for the empty one and immediately drop the
old. If that's the case, the reuse isn't actually happening. Not
investigated further because `GraphPrefetchState` is in `src/am/stream.rs`
and outside the D2 scope; worth a drive-by `Vec::clear` vs `*self =
Vec::new()` check by whoever owns `stream.rs`.

### 7. Counter semantics: `record_element_scored` fires *after* the filter gate

Tracing the graph path, `record_element_scored` fires at the site where
"this element survived all skip filters and is about to be emitted," not
at the site where a score was actually computed. Specifically: the
`score_scan_element_result` call in `select_next_linear_scan_result` is
the only place a real score is computed in the linear path, and the
counter fires immediately *before* the score computation
(`opaque.explain_counters.record_element_scored(); let score =
score_scan_element_result(...)`). Correct there.

In the graph path, `materialize_graph_result_candidate` uses the
candidate's pre-existing `score` (computed during beam search, not in
this function). So `record_element_scored` in the graph path fires on
"scored candidate survives emission filters," not on "scored in this
call." Semantically that's fine for a counter labeled "scored
elements" because the element *was* scored, but the counter name
implies per-element work in this function тАФ which the graph path
doesn't do. If the counter is ever used to infer scan CPU cost, it
will overcount the graph path and undercount the linear path's filter
work.

**Recommendation:** rename `record_element_scored` to something like
`record_element_emitted_for_scoring`, or document in
`TqExplainCounters` that it's a "results returned to the executor"
counter rather than a "scores computed" counter. Again, low priority
because counters aren't surfaced yet. File on the PG18 EXPLAIN
registration slice where the semantics start mattering to end users.

### 8. Snapshot string divergence is intentional and clear

Review Focus #5 asks whether the two strings should consolidate. I
think keeping them distinct is the right call: `index_explain_snapshot`
says "planner scan selection is live," which is the *user-facing*
statement, while `index_cost_snapshot` says "planner cost activation is
live," which is the *internal model* statement. A user reading
`EXPLAIN` cares about the first; an implementer reading
`tqhnsw_index_cost_snapshot(...)` cares about the second. Keep both.

## Verdict

**Approve with follow-ups.** Both Codex findings are correctly
addressed, the cost model recalibration is directionally sound and
spot-checks correctly at three points spanning the AC range, the
lifecycle management of the new TqScanOpaque fields is correct, and
the regression evidence closes FR-020-AC-1 and AC-2 end-to-end. ADR-011
is properly retired and the snapshot surfaces are coherent.

Follow-ups, none blocking D2 merge:

1. **Item 1 (dead `m` input):** either remove `m` from
   `PlannerCostInputs` or comment it as reserved. Today it's plumbed
   through to a function that doesn't read it.
2. **Item 1 (10K test guard):** add a comment on
   `test_fr020_ac1_planner_chooses_index_scan_for_large_table` noting
   that `dimensions = 64` is required for the cost crossover to favor
   the index at 10 K rows.
3. **Item 3 (TOCTOU cleanup):** replace
   `contains_cached` + `cached` with `cached_with_presence() ->
   (Arc<Self>, bool)`. One call site.
4. **Item 4 (counter observability):** add a test-only debug entry
   point that reads `TqExplainCounters` from the opaque post-scan, plus
   one SPI test asserting non-zero counters after a real scan.
5. **Item 5 (carrier comment):** one-line note on
   `opaque.next_block_number` stating it's the authoritative cursor
   until PG18 ReadStream flips ownership to `LinearPrefetchState`.
6. **Item 6 (graph prefetch reuse):** drive-by check in
   `GraphPrefetchState::reset` to confirm the Vec allocation is
   actually reused rather than swapped and dropped.
7. **Item 7 (counter naming):** rename `record_element_scored` to
   `record_element_emitted_for_scoring`, or document the semantics on
   the struct definition. Matters once PG18 EXPLAIN surfaces them.
8. **Item 9 (merge-time string update):** once this branch lands on a
   main that already has A5 closed, update `next_runtime_blocker` to
   drop the "A5" mention.

The load-bearing win here is that after two Codex findings + this
review, FR-020 now has *three* independent empirical assertions: the
pure unit-test crossover at 10 K rows / 1536 dim, the 10 K SPI natural-EXPLAIN
test, and the 50-row seqscan-preference test. That triangulates the
cost-crossover correctness across both directions, which is exactly
what you want before flipping a planner gate that was previously
`f64::MAX`-locked.
