# Review: SPIRE Prepared Capacity Registration Warning (30923)

## Summary

Closes 30896 H9 by adding the descriptor-registration preflight
check 30921 left open. `ec_spire_register_remote_node_descriptor(...)`
now runs an opportunistic remote `SHOW max_prepared_transactions`
preflight:

- Conninfo secret missing/empty → warn (skipped preflight),
  registration succeeds.
- Secret resolves but connection fails → warn, registration
  succeeds.
- Query fails or returns zero → warn, registration succeeds.
- Registration is **always nonblocking** — preflight is advisory.

Registration contract gains
`preflight_prepared_transaction_capacity` row with validator
`warn_if_remote_max_prepared_transactions_unavailable_or_zero`.

ADR-069 + `docs/SPIRE_DIAGNOSTICS.md` document the nonblocking
warning semantics and the write-readiness implication.

The three review-focus questions:
- Preflight should remain nonblocking rather than rejecting
  registration: yes — registration also gates the read path
  (which doesn't need 2PC). A blocking check would prevent
  read-only deployments. Warning + documenting the
  write-readiness implication is the right shape.
- Warning on unresolved conninfo secret is acceptable: yes —
  same reason; secret resolution failures are an operator
  configuration problem that's separable from the descriptor
  shape.
- Contract row + docs accurately describe write-readiness
  gating, not read descriptor eligibility: yes.

## Findings

### P3 — Nonblocking preflight is the right v1 default

A blocking preflight would conflict with the existing pattern of
"register the descriptor, then resolve the secret later" that
some deployment scripts use. Warning gives operators a clean
signal without forcing ordering. Worth pinning in the runbook
that the warning shows up in `pg_log` / client message and is
the intended operator-facing surface.

### P3 — Connection failures during preflight emit warnings

A registration that warns on every connection-failure case might
become noisy in deployments where conninfo secrets are populated
asynchronously (e.g., k8s secret rotation). Worth confirming
the warning is at NOTICE level rather than WARNING, so it
doesn't pollute psql output for deployments that expect the
preflight to skip.

### P3 — Contract row added with stable validator name

`warn_if_remote_max_prepared_transactions_unavailable_or_zero`
is descriptive and stable. Operator monitoring can grep the
contract surface for this name.

### P3 — Tracker properly closes H9

Both halves of H9 (error-hint wrapping in 30921 + registration
preflight in 30923) now landed. Operators get warned at
registration time AND get a remediation hint when an actual
PREPARE TRANSACTION fails.
