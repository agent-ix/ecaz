# Review: SPIRE Multicluster CLI Smoke/Insert Paths (30968)

## Summary

Phase 12.9 CLI surface extension. Adds two
`ecaz dev spire-multicluster` subcommands:

- `smoke-pg18` — baseline PG18
  one-coordinator/one-remote smoke fixture;
- `insert-read-after-customscan-pg18` — repeated INSERT
  followed by CustomScan read fixture, with
  `--insert-mode helper|trigger`.

Carries through artifact-dir, run/log/smoke-log paths,
ports, run id, pgbin, PGRX home, skip-install. Parser-only
test coverage.

## Findings

### P2 — Operator-facing names

The two names align with the existing
`transport-overlap-pg18` and 30962
`customscan-read-pg18` siblings:

- `smoke-pg18` — minimal correctness oracle.
- `transport-overlap-pg18` — two-remote setup/teardown.
- `customscan-read-pg18` — repeated read.
- `insert-read-after-customscan-pg18` — write-then-read.

The four-command set covers the common operator workflows
under one parent. Naming pattern is consistent. Future
operators picking from this list have enough hint in the
name to choose without reading help text.

### P2 — Shared fixture controls carried through

Carrying artifact-dir, run/log/smoke-log paths, ports, run
id, pgbin, PGRX home, skip-install through both new
commands is the right discipline — operators with one
fixture command in muscle memory can drive any of the four
with the same control set. Loss of any of these would
force them back to direct script invocation, which is the
failure mode the CLI wrapper is supposed to prevent.

### P2 — `--insert-mode helper|trigger` enum

Adding an enum for `helper|trigger` modes is the right
shape vs a free-form string. Enum prevents typos from
silently selecting the wrong path. Worth confirming the
parser test asserts both values parse and that an invalid
value rejects with a clear error.

The two modes correspond to:

- **helper** — direct call to the placement-registration
  helper (the underlying primitive).
- **trigger** — INSERT through the BEFORE INSERT trigger
  (the user-visible front door).

Both are needed: the helper path is what packet-level tests
exercise; the trigger path is what production INSERT looks
like. Keeping both as fixture modes preserves coverage of
both layers under one CLI command.

### P2 — Parse-only test scope

Right scope for this slice. Same reasoning as 30962 — live
fixture execution drags in sandbox socket-bind issues that
are orthogonal to "the CLI surface compiles and parses."
Live execution happens via the actual fixture runs (e.g.,
30961's Stage E rerun pattern), not via this parse test.

### P3 — Help text differentiation

Worth confirming each of the four `spire-multicluster`
subcommands has help text that names what's *different*
from its siblings. Operators reading `ecaz dev
spire-multicluster --help` should be able to pick by:

- smoke = minimal correctness;
- transport-overlap = two-remote setup;
- customscan-read = repeated read;
- insert-read-after-customscan = write-then-read.

If help text is generic ("runs PG18 fixture"), operators
will have to read source. Differentiated help text is the
ergonomic delta that makes the CLI surface useful.

### P3 — Tracker scope

Phase 12.9 tracker evidence row for "ecaz dev fixture
wrappers" — correctly scoped to CLI surface, not to live
execution or final readiness bundle. Honest.

12.9 lane has:
