## TL;DR for the agent

**Merged correctly as a staged slice.** The one-entry-per-worker
min-heap is the right explicit frontier surface. The concurrency
concern I flagged in 472 is now concretely visible here — the heap
entries array is mutated via non-atomic `ptr::swap` during sift-up.

## Verification performed

- `EcParallelScanState` grew `heap_bytes`, `heap_entry_bytes`,
  `heap_entry_count` fields; descriptor sizing in
  `ec_parallel_scan_descriptor_size_for`
  (`src/am/common/parallel.rs:315–356`) routes the new span between
  coordinator and result slots. Overflow-checked throughout.
- `reset_parallel_scan_layout` (`parallel.rs:458+`) stamps every heap
  entry to `EC_PARALLEL_HEAP_ENTRY_INVALID` and clears heap state
  counters. Correct.
- `load_coordinator_heap_snapshot` (`parallel.rs:674–692`) reads
  `live_entry_count`, `heap_generation`, and if non-empty, the root
  slot index from entry 0. That is a direct, lock-free read path —
  fine for readers while there is a single writer.
- `refresh_coordinator_selection_snapshot` (`parallel.rs:774–876`)
  calls reap, clears heap entries to invalid, walks live result
  slots, pushes each into the heap with `sift_up_heap_entry`, then
  stores `live_entry_count` and bumps `heap_generation`. The
  heap rebuild is O(n log n) — fine at n = worker_slot_count ≤ 9.
- `select_best_parallel_scan_coordinator_result_slot_with_attachment`
  (`parallel.rs:723–736`) now reads the heap root (generation/count
  already reflect reap), avoiding the earlier rescan of all slots.

## Findings

### 1. Sift-up uses non-atomic pointer swap — blocker for
`amcanparallel = true`

**Severity:** Medium (deferred blocker; no-op today)
**File:** `src/am/common/parallel.rs:797–814`

```rust
unsafe { std::ptr::swap(child_entry, parent_entry) };
```

This is `std::ptr::swap` on raw `*mut u32` entries. Two concurrent
callers of `refresh_coordinator_selection_snapshot` racing over
sift-up would corrupt the heap arbitrarily, and readers racing with
a sift-up can observe torn-swap states (entry[i] and entry[parent]
neither old nor new simultaneously). `amcanparallel = false` rules
this out today, but the whole point of the chain is to flip that bit.

Same recommendation as 472: take a coordinator-owned LWLock around
the refresh, OR restructure to CAS-per-entry sift with reader-side
generation-based retry. LWLock is simpler and the readers already
tolerate a stale generation (the two-pass retry in
`read_parallel_scan_selected_result_slot_snapshot` handles it).

Readers also read `live_entry_count` then `heap_entries[0]`
(`parallel.rs:678–684`) as separate operations — under concurrent
rebuild, a reader could see `live_entry_count = 3` from generation N
and `heap_entries[0] = INVALID` from generation N+1 mid-rebuild
(between the clear loop and the push loop). Today's rebuild starts
by clearing all entries to INVALID (line 817–822) **after** reaping
— so `live_entry_count` briefly overstates reality. When serialization
lands, hold the lock around the entire clear+rebuild sequence, not
just sift-up.

### 2. `load_coordinator_heap_snapshot` reads entry 0 directly, not
through `attachment.heap_entry(0)`

**Severity:** Low
**File:** `src/am/common/parallel.rs:682`

```rust
let root = unsafe { *attachment.heap_entries };
```

This bypasses the bounds-checked `attachment.heap_entry(0)` helper.
The bypass is safe because `live_entry_count != 0` implies a
heap_entry exists, but it's inconsistent with the rest of the module
which always goes through the helper. Either dereference through
`attachment.heap_entry(0)?` (propagating the error) or add a
debug_assert that heap_entry_count > 0.

### 3. Heap capacity == worker_slot_count is the right staged size

**Severity:** Informational (positive)
**File:** `src/am/common/parallel.rs:363–365`

`heap_entry_capacity_for = worker_slot_count` — exactly one entry
per worker slot, matching the "one staged result per worker" rule.
When the real top-K lands, the heap may need to grow (to K entries,
not one-per-worker), which is another descriptor version bump. Keep
the magic/version discipline.

### 4. Packet text implies this is the frontier layout for the
eventual top-K

**Severity:** Informational
**File:** `review/478-c1-parallel-scan-shared-heap-frontier/request.md:59–61`

"The right explicit frontier surface for the next real coordinator
merge/admission slice." Correct for the one-per-worker frontier — but
the real top-K (where K is the user's query LIMIT, not the worker
count) will need a larger heap. Worth one line in Task 18 notes
distinguishing "worker-frontier heap" (this slice) from "top-K heap"
(future slice). Avoids confusing the two.

## Answers to the packet's review questions

- **Q1** (one-entry-per-worker right frontier?) — Yes, for this stage.
  Not sufficient for the final top-K; separate sizing then.
- **Q2** (rebuild-on-refresh narrow enough?) — Yes today; becomes
  insufficient when lock-guarded mutation lands. Plan the
  serialization *before* that slice, not during it.
- **Q3** (PG17/PG18 coherence?) — Yes. The descriptor layout is
  Postgres-version-agnostic; only `parallel_scan_state_ptr` branches
  on `cfg(feature = "pg18")`. That isolation is still clean.
