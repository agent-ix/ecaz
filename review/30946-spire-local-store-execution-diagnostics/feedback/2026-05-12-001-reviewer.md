# Review: SPIRE Local Store Execution Diagnostics (30946)

## Summary

Phase 12.8 diagnostic surface that names the current
sequential-backend local-store execution mode, distinguishes
PG18 ReadStream read-ahead from real concurrent store-group
execution, and pins the future primitive name.

## Findings

### P2 — Narrow snapshot vs widening existing

Adding `ec_spire_index_scan_local_store_execution_snapshot`
as a separate function is **the right call** over widening
`ec_spire_index_scan_placement_snapshot`. The reasoning:

- pgrx table-row column-count limits are real and would
  force an ABI break if the existing snapshot hit them later
  for an unrelated field.
- Execution-mode columns are a **different conceptual axis**
  from placement (where), prefetch (read-ahead), and routes
  (what was scanned). Forcing them onto the same row pulls
  three concerns through one function signature.
- Operators can correlate by `index_oid` + `query` — the
  composite key is stable across both snapshots.

Endorsed.

### P2 — Label precision

The three labels are the right vocabulary:

- **`local_store_execution_mode = 'sequential_backend'`**:
  honest. Names the limitation, not aspirational behavior.
- **`local_store_read_ahead_primitive = 'pg18_read_stream'`**:
  precise. Read-ahead inside one backend is what
  ReadStream provides; it does NOT give true overlap of
  store-group work.
- **`local_store_parallelism_next_step =
  'async_or_parallel_store_group_executor'`**: correctly
  scoped as a *next step*, not a current capability.

The naming makes it easy for an operator to misread the
diagnostic and think "PG18 ReadStream means I have parallel
local stores now" — but the label structure (mode vs
read-ahead vs next-step) prevents that confusion. Good.

### P3 — Documentation echo

Worth confirming the diagnostic doc and the multi-store
design note both call out the **performance implication**:
multiple local stores today serialize on one backend; PG18
ReadStream amortizes I/O latency but not CPU work.
Operators sizing `local_store_count > 1` for throughput
should read the next-step label as "not yet" rather than
"about to ship."

If the docs already say this, ignore. If they describe the
diagnostic without the perf implication, add one sentence.

### P3 — Future-test hook

When the async/parallel store-group executor lands, the
label change in
`local_store_execution_mode` → e.g., `'parallel_backend'`
will be a load-bearing test signal. Worth pinning a
follow-up note that the label-string transition is itself
operator-visible behavior (i.e., the label is a public
contract, not internal).

## Reviewer recommendation

Accept. Narrow diagnostic with precise labels is the right
shape. The sequential-execution diagnostic row close is
honest about the limitation, not aspirational.
