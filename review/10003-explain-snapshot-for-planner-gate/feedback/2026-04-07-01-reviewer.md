# Feedback: Explain Snapshot For Planner Gate

**Reviewer:** Claude Opus 4.6
**Date:** 2026-04-07

## Findings

### 1. The separate explain snapshot is premature surface area

A dedicated `tqhnsw_index_explain_snapshot(regclass)` function makes sense eventually, but right now all the information it reports (planner gate state, ordered_scan_ready=false, ef_search source) is already available from the admin snapshot. Adding a second SQL function that returns substantially overlapping data creates maintenance burden with limited incremental value.

**Recommendation:** Fold the explain-facing fields into the admin snapshot for now. Create the separate explain snapshot when PG18 EXPLAIN hooks actually exist and the surface needs to carry EXPLAIN-specific data (per-node counters, plan-level stats) that doesn't belong in admin.

### 2. Shared validation helper is the right shape

Parameterizing the validation helper by caller function name is good — it avoids the confusing error text issue and keeps the code DRY. This should survive into the final shape.

### 3. Spec/task updates are appropriately scoped

The ADR-011, FR-009, and test matrix updates correctly record this as scaffolding, not activation. Good discipline.

## Net

The slice is safe and correctly gated. The concern is surface-area creep: the planner branch is accumulating many read-only snapshot functions (admin, explain, cost, stats, pg18 upgrade, pg18 diagnostics, planner integration) that all report `false`/`not ready` in different ways. Consider consolidating — see cross-cutting feedback in the planner-integration-blockers review.
