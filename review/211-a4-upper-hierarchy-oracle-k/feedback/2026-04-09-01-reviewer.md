## Response to Review 211

### The oracle-K result is the cleanest evidence in the lane

The k-curve (k=3 49.8%, k=5 63.2%, k=8 79.6%, k=10 83.4% = exact quantized) provides a
direct, actionable calibration. It proves three things simultaneously:

1. The built graph is structurally sound (layer-0 search recovers exact quantized with correct seeds)
2. The good seeds exist at the top level
3. The live path's failure is specifically in navigating from the stored entry point to those seeds

The failed multi-seed upper-layer bootstrap (25.4%, regressed) is equally valuable as negative
evidence: more exploration FROM the stored entry point is a dead end.

### The live graph path degraded relative to its ceiling

In review 200, graph was at 91% of exact quantized ceiling (39.1% vs 43.1% on 10k). After the
quantizer improvements (tiled FWHT + 4+0 reallocation), graph is now at 31% of ceiling (25.8%
vs 83.4%).

This is mechanically expected: stronger codes mean more truly distinguishable neighbors exist, so
the graph needs to find more of them. The upper-layer navigation from a single entry point was
already marginal — it cannot scale to the harder retrieval task created by the improved quantizer.

## Algorithm Comparison Against Reference HNSW

Reviewed `graph.rs`, `search.rs`, and `scan.rs` against four reference implementations. Three
key differences from reference Malkov & Yashunin 2018:

### 1. Upper layers use full ef_search beam instead of greedy ef=1

`search_upper_layer_seed_candidates` (graph.rs) runs a full beam search with `ef_search`
candidates at every upper layer. Reference Algorithm 5 uses ef=1 greedy descent through upper
layers.

**Assessment**: This is intentionally recall-positive for quantized search (noisy distances make
greedy decisions unreliable). Review 211's failed multi-seed experiment confirms this is NOT the
bottleneck — widening the beam further from the stored entry point doesn't help.

### 2. BeamSearch::run has no early termination

`BeamSearch::run` (search.rs) uses a fixed expansion budget (ef_search iterations). Reference
Algorithm 2 terminates early when the best unexpanded candidate is worse than the worst result.

**Assessment**: The fixed budget ensures consistent exploration, which is appropriate for quantized
distances where early termination heuristics can be fooled by noisy scores. This is a throughput
tradeoff, not a recall problem.

### 3. VisibleFrontier uses O(n) operations

`best_candidate_by_score` and `remove_node` are linear scans over an unsorted Vec.

**Assessment**: Performance concern only. Does not affect recall.

**Conclusion: None of the algorithm differences are part of the recall problem.** The recall
problem is structural — the single stored entry point's upper-layer neighborhood doesn't connect
to the best top-level seeds for arbitrary queries.

## Recommended Next Steps (Ranked)

### 1. Multiple stored entry points at build time (strongest signal, cheapest)

The oracle-K curve directly supports this. At build time:

- Select k diverse top-level nodes instead of 1 in `choose_entry_point`
- Store them in the metadata page (10 ItemPointers = 60 bytes)
- Seed layer-0 search from all k descended entry points
- Search cost increase is small: 10x initial upper-layer descent, but layer-0 beam search
  dominates

**Required pre-check**: Across all 50 queries, count how many unique nodes appear in the
oracle's per-query top-10 sets. If 10-20 nodes cover most queries, fixed build-time entry
points will work. If every query needs entirely different seeds, they won't — and the problem
is more fundamental.

**Risk**: The oracle selects per-query optimal seeds. Fixed entry points are query-independent.
The pre-check determines whether this distinction matters.

### 2. Audit upper-layer reachability from the stored entry point

Before implementing multiple entry points, understand the graph topology:

- From the stored entry point, BFS at max_level to enumerate ALL reachable nodes
- Check what fraction of the oracle's best k=10 nodes per query are in the reachable set
- If most oracle seeds are reachable but not found by beam search: search strategy issue
- If most oracle seeds are NOT reachable: graph connectivity issue, multiple entry points is
  the right fix

### 3. Check whether 4+0 degraded graph construction quality

The 4+0 reallocation changed the codes, which changed `BuildCodeDistance::eval` outputs, which
changed which neighbors hnsw_rs selected. The graph may be worse even though the codes are better
for scoring.

Test: compare graph quality metrics (average neighbor recall against true NNs) on the old 3+1
graph vs the new 4+0 graph. If 4+0 produces a worse graph, the build distance needs attention.

### 4. Increase m (heavy, last resort)

If upper-layer connectivity is fundamentally too sparse at m=8 for the 4+0 distance landscape,
increasing m helps mechanically. But it increases storage, build time, and search time. Only
after exhausting options 1-3.
