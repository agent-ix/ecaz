# Feedback: Linear Scan Rescan After Exhaustion

**Reviewer:** outside-review  
**Date:** 2026-04-05  
**Verdict:** Approved with notes

## Answers to review questions

1. **Rescan after exhaustion with duplicate-coalesced final element**: The current test coverage appears to verify that rescan-after-exhaustion restarts from the beginning. However, there is a subtle gap: if the final returned tuple before exhaustion was mid-way through draining a duplicate-coalesced element, the `pending_heaptid_count` and `pending_heaptid_index` must be reset to zero. Inspecting `reset_scan_position` (scan.rs:223-231), this does happen correctly since both fields are zeroed. No bug here, but a targeted regression for "exhaust while mid-drain, then rescan" would strengthen the contract.

2. **Stale-state risk after exhaustion plus rescan**: `reset_scan_position` correctly resets `next_block_number` to `FIRST_DATA_BLOCK_NUMBER`, `next_offset_number` to 1, `scan_exhausted` to false, pending heap TID state, candidate state, and current result state. This looks thorough. One observation: `scan_block_count` is *not* refreshed during rescan — it was cached in the original `amrescan`. If the relation grew between the original scan and a rescan (e.g., concurrent inserts), the rescanned pass would use the stale block count and miss newly appended pages. This is acceptable for the bootstrap phase but should be called out as a known limitation that future ordered scan work needs to address.

3. **ADR documentation**: Calling this out in the existing ADR-008 or ADR-009 would be appropriate — a one-line note about "rescan restarts the bootstrap linear walk from block 1 with the originally cached block count" would be sufficient.

## Additional notes

- The implementation is sound for this stage. No correctness defect found.
- The type alias cleanup is fine as a hygiene pass.
