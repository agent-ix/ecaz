# Feedback: Scan Metadata Cache

**Reviewer:** outside-review  
**Date:** 2026-04-05  
**Verdict:** Approved with note

## Answers to review questions

1. **Stale-state risk across repeated rescans**: None. The cached fields (`dimensions`, `bits`, `seed`, `code_len`) are all immutable properties of the index — they are set at build time and never change. Re-reading them from the metadata page on every `amgettuple` call was pure waste. Caching them in `amrescan` and reusing across the scan is correct.

2. **Additional metadata to cache now**: `scan_block_count` is also cached in `amrescan` (scan.rs:95-98), which is the right minimum. The entry point TID could also be cached, but it's only needed once at scan start and is already consumed by `initialize_scan_entry_candidate`. No additional caching needed at this stage.

3. **Missing regression around empty-to-nonempty transitions**: If the same scan descriptor were rescanned against a different index with different metadata, that would be a problem — but PostgreSQL doesn't allow this. A scan descriptor is bound to one index relation for its lifetime. The existing coverage (empty index rescan vs. non-empty index rescan) is sufficient.

## Additional notes

- The `scan_seed` field is cached but I notice it's used in `score_scan_element_result` (scan.rs:466) to reconstruct the quantizer via `ProdQuantizer::cached()`. Since the quantizer is also keyed on `(dimensions, bits, seed)`, caching the seed here makes sense as groundwork for eventually caching the quantizer reference itself.
