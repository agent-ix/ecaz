# Feedback: Relation-Cache Reloptions

**Reviewer:** outside-review  
**Date:** 2026-04-05  
**Verdict:** Approved with finding

## Answers to review questions

1. **String reloption decoding for `build_source_column`**: The stored string offset approach from `TqHnswReloptions` is the standard pgrx/PostgreSQL pattern for string reloptions. The offset points into the reloptions struct itself, and the string is null-terminated. This is correct.

2. **Lifetime risk reading `rd_options`**: `rd_options` is populated by the relation cache and stays valid for the lifetime of the relation descriptor. Since `aminsert` receives the relation descriptor from the executor, which holds at least `RowExclusiveLock`, the pointer is stable. No lifetime risk.

3. **Dedicated regression for `build_source_column`**: The existing build and insert paths already exercise `build_source_column` presence and rejection (aminsert rejects it). No additional helper needed.

## Finding: SPI removal is correct but check NULL rd_options

**Severity:** Low  
When `rd_options` is NULL (no WITH clause specified), the code should fall back to defaults. Verify that the current `relation_options` implementation handles `(*rel).rd_options.is_null()` by returning default `TqHnswOptions`. If the reloptions struct is only populated when WITH parameters are explicitly set, a NULL `rd_options` must not dereference the pointer. This is likely already handled, but worth confirming given the path was previously routed through SPI which always returns a result.
