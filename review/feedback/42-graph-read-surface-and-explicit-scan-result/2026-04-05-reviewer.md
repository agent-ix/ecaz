# Feedback: Graph Read Surface And Explicit Scan Result

**Reviewer:** outside-review  
**Date:** 2026-04-05  
**Verdict:** Approved

## Answers to review questions

1. **`graph.rs` as shared read boundary**: Yes. The three functions (`load_graph_element`, `load_graph_neighbors`, `load_graph_adjacency`) provide a clean abstraction over raw page reads. Both scan traversal and future graph-aware insert/vacuum can use these same primitives. The module correctly depends only on `page` types and `pg_sys` buffer operations. No behavior-specific logic leaks in.

2. **Explicit current-result struct as minimal shape**: The `CurrentScanResult` with `(element_tid, heap_tid, score, score_valid)` is the right minimum. A smaller representation (e.g., just element_tid) would lose the score, which is needed for ordered result emission. A larger one (e.g., including the element's code bytes or neighbor list) would add per-tuple copying overhead that belongs in traversal state, not result state.

3. **Missing invariants before candidate heaps land**: The test adjustment to assert actual storage invariants (not assuming non-empty adjacency) is the right call. One invariant worth adding: after `load_graph_adjacency`, assert that the neighbor tuple's TID matches the element's `neighbortid` field. This round-trip check would catch any mismatch between element and neighbor tuple linkage. Not blocking.

## Additional notes

- Clean extraction. The graph module is well-scoped.
