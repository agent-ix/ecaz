# Overall Scan and Graph Architecture Review

**Reviewer:** outside-review  
**Date:** 2026-04-05  
**Scope:** Cross-cutting review of scan execution, graph read surface, candidate frontier, and scoring across all scan/traversal review requests (13-46)

## Summary

The scan and graph implementation follows a disciplined incremental approach: each slice adds one narrow capability, validates it with regression tests, and commits. The bootstrap linear scan is a sound temporary scaffold that exercises the full scan lifecycle (descriptor allocation, query validation, metadata caching, tuple production, duplicate draining, teardown) without depending on graph traversal. The candidate frontier groundwork (entry seed, successor seed, two-slot container) correctly establishes the structural preconditions for HNSW search.

No correctness defects were found in the current scan implementation. The findings below are ordered by priority for the next phase of work.

## Critical path items for ordered traversal

### 1. Persist gamma in TqElementTuple (HIGH priority)

This is the single most impactful change for enabling efficient ordered traversal. Currently, scoring a candidate requires a heap fetch to recover gamma (via `heap_tqvector_gamma`). This is O(1) heap I/O per candidate scored. For ordered traversal with ef_search=40, each `amgettuple` call could explore hundreds of candidates, making this the dominant cost.

**Recommendation:** Add a `gamma: f32` field (4 bytes) to `TqElementTuple`. This is a page-layout change, so do it now while the format is pre-v1. Benefits:
- Eliminates heap fetches during scan scoring
- Eliminates heap fetches during duplicate detection in `find_duplicate_element_tid`
- Removes the fragile representative-heap-row dependency
- Enables the `score_ip_from_parts(prepared, gamma, code_bytes)` optimization (no Vec allocation per candidate)

This change affects: page.rs (element tuple encoding/decoding), mod.rs (aminsert, duplicate detection), scan.rs (scoring), build.rs (tuple construction), graph.rs (GraphElement struct).

### 2. Eliminate per-candidate Vec allocation in scoring (MEDIUM priority)

`score_scan_element_result` (scan.rs:471-474) builds `[gamma][code_bytes]` in a fresh `Vec<u8>` for every scored element. Once gamma is in the element tuple, add `ProdQuantizer::score_ip_from_parts(prepared: &PreparedQuery, gamma: f32, code_bytes: &[u8]) -> f32` that reads the parts in-place without concatenation. This removes the hot-path allocation entirely.

### 3. Visited set for traversal (MEDIUM priority)

The current frontier has no visited-set tracking. When real graph traversal starts, a `HashSet<ItemPointer>` (or a bitset if element TIDs are dense enough) is needed to avoid re-scoring previously visited nodes. This should be allocated in scan-owned state during `amrescan` and cleared on rescan.

### 4. Layer-aware neighbor access (MEDIUM priority)

The flat neighbor TID array needs a layer interpretation for greedy descent. The formula is documented in FR-007: layer 0 gets `2M` slots, higher layers get `M` each. `load_graph_neighbors` should eventually support a `layer` parameter that returns only the relevant slice. This is not needed for the first ordered-scan slice if the initial implementation only does layer-0 beam search, but greedy descent through upper layers requires it.

## Design observations

### Scan state size
`TqScanOpaque` is `repr(C)` with fixed-size arrays. At ~200 bytes it's compact and cache-friendly for the current design. When the candidate heap and visited set are added, these will be heap-allocated (pointer in opaque, data on Rust heap). This is the right approach — don't try to inline a dynamic-size heap into the fixed opaque struct.

### scan_block_count staleness
`scan_block_count` is cached in `amrescan` and not refreshed on rescan or during the scan. If concurrent inserts add pages after the scan starts, the linear scan will miss them. This is acceptable for the bootstrap and for MVCC-correct ordered traversal (which shouldn't see uncommitted inserts anyway), but should be documented.

### Debug helper proliferation
The `#[cfg(any(test, feature = "pg_test"))]` debug helpers in scan.rs now number ~15 functions spanning 500+ lines. These are valuable for regression coverage but are approaching the point where they should be extracted into a `scan/debug.rs` or `scan/test_helpers.rs` module per ADR-012's suggested future split.

### Scoring sign convention
The negation in `score_scan_element_result` (`-quantizer.score_ip_encoded(...)`) produces the `<#>` operator convention (lower = more similar). This is correct and consistent. Ensure this convention is maintained when candidate comparison enters the picture — the frontier's `score <=` comparison (scan.rs:333) assumes lower is better, which matches.

### Buffer pin discipline
All page reads correctly use ReadBuffer + BUFFER_LOCK_SHARE + UnlockReleaseBuffer with no held pins across function boundaries. The graph.rs helpers copy tuple bytes into Vec<u8> before releasing the buffer, which is clean. For traversal, consider whether a "read many neighbors from one page" batch optimization would reduce lock acquire/release overhead — but this is premature before profiling shows it matters.

## ADR recommendations

### ADR-013: Persist gamma in element tuples
Document the decision to add gamma to the on-disk element tuple layout. Rationale: eliminates heap fetches during scan scoring and duplicate detection. Trade-off: 4 bytes per element tuple, page-layout break (acceptable before v1). This is the most important structural decision for the next phase.

### ADR-014: Traversal state budget
Before implementing the full ordered traversal, document the expected memory budget per scan: visited set size, candidate heap size, result buffer size. With ef_search=40 and typical graph fan-out, the visited set could grow to thousands of entries. Setting an explicit budget and spill strategy prevents unbounded memory growth in concurrent scan scenarios.
