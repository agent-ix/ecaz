# Feedback: Scan Entry Candidate State

**Reviewer:** outside-review  
**Date:** 2026-04-05  
**Verdict:** Approved with finding

## Answers to review questions

1. **`element_tid + score` as minimal shape**: Correct. For the entry point seed, you need to know *which* element and *how good it is*. The code bytes are accessible via `load_graph_element` when needed. Level is irrelevant for a fixed two-slot frontier (no layer distinction yet). This is the right minimum.

2. **Seeding during `amrescan` as ownership boundary**: Correct. `amrescan` is when query state is established, so scoring the entry point against the query is naturally done here. The seed lives in `TqScanOpaque` which is per-scan-descriptor and correctly scoped.

3. **Should exhaustion clear the entry candidate?**: 

**Finding: Clearing on exhaustion is premature for future traversal**

**Severity:** Low (design note, not bug)  
Currently, `clear_scan_candidate_state` is called from `next_linear_scan_heap_tid` on exhaustion (scan.rs:441). For the bootstrap linear scan, this is fine — the candidates are unused decoration. But for future ordered traversal, the entry candidate represents the *starting point of the graph search*, not a consumable result. Clearing it on scan exhaustion would mean a subsequent rescan needs to re-read the metadata page to re-seed. 

The current behavior is acceptable because `amrescan` already re-seeds the entry candidate. But the design intent should be clarified: entry candidates are *search initialization state*, not *scan progress state*. When the transition to ordered traversal happens, consider removing the exhaustion-clears-candidates behavior and only resetting candidates on rescan.

## Additional notes

- The `initialize_scan_entry_candidate` function (scan.rs:271-321) correctly handles empty indexes and deleted entry points by early-returning without populating the frontier.
