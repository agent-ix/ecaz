# Feedback: CI Workflow Hardening

**Reviewer:** outside-review  
**Date:** 2026-04-05  
**Verdict:** Approved with note

## Answers to review questions

1. **Concurrency group safety**: `ci-${{ github.workflow }}-${{ github.ref }}` with `cancel-in-progress: true` is safe. For push events, `github.ref` is the branch ref, so force-pushes to the same branch cancel stale runs. For PR events, `github.ref` is `refs/pull/<N>/merge`, so each PR gets its own group. No cross-PR interference.

2. **PGRX_HOME scope**: Job-level `PGRX_HOME=/tmp/tqvector_pgrx_home` on the pgrx job only is correct. The rust job (fmt, clippy, unit tests) doesn't invoke pgrx commands, so setting it there would be dead configuration. Workflow-wide would be harmless but misleading.

3. **Workflow minimality**: The gates (fmt, clippy, test, unsafe audit, pgrx test, license audit) are the right set for this project stage. No unnecessary steps.

## Note

The `PGRX_HOME` path uses `/tmp/` which is cleaned between CI runs. This means pgrx re-initializes from scratch on every run. If CI times are a concern later, consider caching the pgrx home directory alongside cargo artifacts. Not blocking.
