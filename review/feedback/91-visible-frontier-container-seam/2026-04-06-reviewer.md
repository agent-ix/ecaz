**Reviewer:** outside-review

## Answers to review questions

1. **Is this container seam the right narrow boundary before deeper ownership transfer?** Yes. `VisibleCandidateFrontierState` (scan.rs:322-384) encapsulates all Vec mutation behind named methods. No scan path directly mutates `Vec<BeamCandidate>` except through this type. The container can now be replaced (different backing store, moved behind `search.rs`) without changing the mutation surface.

2. **Are the chosen methods the right minimum surface?** Yes. Verified each method maps to a clear runtime need:
   - `clear` → rescan reset
   - `push` / `extend` → seeding via `seed_discovered_candidates`
   - `remove_node` → consume via `take_candidate_frontier_node`
   - `len` / `is_empty` → capacity check in `top_up_bootstrap_frontier`
   - `contains_node` → beam visibility check in `scheduler_best_frontier_candidate`
   - `best_candidate_by_score` → Vec fallback in `current_candidate_frontier_head`
   - `iter` / `slot` → debug snapshot and positional access

   No dead weight, no missing operations.

3. **Do any remaining paths manipulate the frontier too directly?** No. All runtime mutation goes through `visible_frontier_mut(opaque)` (scan.rs:398-404). All reads go through `visible_frontier_ref(opaque)` (scan.rs:390-396).

## Additional notes

- The `EMPTY_VISIBLE_FRONTIER_STATE` static (scan.rs:386-388) is a good pattern — avoids allocation for the read path on null pointers. Clean.
