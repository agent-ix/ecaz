# Review: Parallel Scan Shared Emitted History

**Verdict: Approved**

## What was read

- `git show d3357bd` diff (parallel.rs, scan.rs) — key lines filtered

## Summary

Adds coordinator-owned emitted heap-TID ring (capacity 32, matching the per-worker ring). Ring slots are written via `remember_coordinator_emitted_heap_tid` when admitted results are consumed. The ring is checked in `admit_pending_output_locked` and `pending_output_would_admit_locked` before admitting a pending output whose heap TID was already emitted.

## Findings

**Cursor uses AcqRel fetch_add under coordinator lock.** `remember_coordinator_emitted_heap_tid` uses `coordinator.emitted_heap_tid_cursor.fetch_add(1, Ordering::AcqRel)`. Since this function is always called under the coordinator lock, `Relaxed` would be sufficient. Not a bug — AcqRel is more conservative and correct — but it's inconsistent with `Relaxed` elsewhere in the struct that's serialized by the lock.

**Public read surface.** `remember_parallel_scan_emitted_heap_tid` and `parallel_scan_heap_tid_was_emitted` are both public and acquire the coordinator lock internally. They are not called from any current call site visible in this diff, but they're needed for the external-emit path that records emitted TIDs from `consume_parallel_scan_admitted_result`. The function names are clear.

**Reset path.** The coordinator ring fields are initialized in `reset_parallel_scan_layout` to `INVALID` block/offset numbers. The cursor is reset to 0. `reset_parallel_scan_state` calls `reset_parallel_scan_layout` after incrementing the epoch, so the ring is cleared on rescan. Correct.

**DSM version bump.** 14 → 15. Correct.
