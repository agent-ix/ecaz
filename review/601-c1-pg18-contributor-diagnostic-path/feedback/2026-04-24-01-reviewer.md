# Review: PG18 Contributor Diagnostic Path

**Verdict: Approved**

## What was read

- `git show 753efb1` full diff (scan.rs, shared.rs, lib.rs)
- `contribute_non_emitting_parallel_backend`, `stage_non_emitting_parallel_contributor_output`, `non_emitting_parallel_backend_should_contribute`

## Summary

The contributor mode (gated by `TQVECTOR_PG18_PARALLEL_CONTRIBUTOR_DIAGNOSTIC=1`, test/pg_test builds only) prevents non-elected workers from immediately exhausting. Instead they stage their current result into the hidden coordinator slot and poll for the elected emitter to drain it. Production builds always take the existing exhaustion path.

## Findings

**`contribute_non_emitting_parallel_backend` loop structure.** The loop has two distinct states: `parallel_local_only_output_active=true` (contributor is waiting for the emitter to drain a published row) and `false` (contributor is advancing through the graph to find the next row to publish). In the waiting state, the loop polls via `reconcile_parallel_hidden_owner_progress_from_shared_slot` and sleeps 100μs up to `PARALLEL_CONTRIBUTOR_DRAIN_POLL_LIMIT=100` times (10ms total) before returning `false`. In the advancing state, it calls `ensure_prefetched_output` to extend the graph cursor. The `PARALLEL_CONTRIBUTOR_MAX_OUTPUTS=64` cap limits work per contributor invocation before yielding back to the PG executor. This is a correct cooperative scheduling design.

**Interrupt check in `poll_parallel_contributor_interrupts_and_sleep`.** Checks `pg_sys::InterruptPending` before sleeping. This ensures PG cancellation (`SIGINT`, query timeout) is honored during the drain-poll loop. Correct.

**`non_emitting_parallel_backend_should_contribute` prevents double-election.** Requires `!parallel_scan_backend_may_emit_tuples(opaque)` after the contributor diagnostic env check — so a worker that won the emitter election does not enter contributor mode. Correct.

**`non_emitting_parallel_backend_is_exhausted` now returns `false` when contributor diagnostic is enabled.** The `if parallel_scan_contributor_diagnostic_enabled() { return false; }` guard inside the exhaustion function means non-elected workers in contributor mode do not mark themselves exhausted at amrescan time — they proceed to `contribute_non_emitting_parallel_backend` from `produce_next_scan_heap_tid` instead. Clean.

**Contributor pattern in `lib.rs`.** The `PARALLEL_CONTRIBUTOR_DIAGNOSTIC_ENV` constant and helper are added to shared.rs with the same `#[cfg(any(test, feature = "pg_test"))]` gating pattern used by the multi-emitter diagnostic. Consistent.

**No useful foreign handoffs in current fixture.** The diagnostic log confirms `Parallel Handoffs: Foreign Selected: 0` — non-elected workers publish duplicate initial graph cursors because all workers bootstrap from the same starting point. The correctness contract holds (serial equivalence), but performance improvement requires rank-aware distinct contribution. This is correctly documented as the next step.

## No blockers.
