# Feedback: A5 Overflow Backlink Pruning

**Reviewer:** Claude (Opus 4.6)
**Date:** 2026-04-11
**Commit reviewed:** `975e2d9`

## Scope match

Matches the packet. `BacklinkMutation` now carries a
`BacklinkMutationKind::{InsertIfFree, RewriteFullSlice {expected_slice,
replacement_slice}}`. Planning is separated from writing:
`plan_backlink_mutation` reads the target element + its neighbors, scores
existing slice + new node against the target's persisted code, and emits a
`RewriteFullSlice` only when the new node actually makes the top-`N` cut.
The write phase applies the planned rewrite under the same per-page lock
protocol, with an `expected_slice != live_slice` drift check that skips the
rewrite instead of overwriting a concurrently-changed full slice.

Locking protocol (ADR-026) is unchanged. Tests cover a real full-layer-0
rewrite event.

## Findings

### 1. Plan/write split is the right shape for optimistic concurrency

Splitting "what to write" (done in the planning phase before taking any page
lock) from "actually write it" (done under the page `EXCLUSIVE` lock) is the
piece that makes the per-page WAL transaction short and predictable. The only
work done inside the lock is:

1. decode neighbor tuple
2. for each planned mutation on this tuple, apply in-place
3. re-encode and copy back

No graph traversals, no second page reads, no scoring — all of that happened
during planning. That keeps the lock hold time proportional to the neighbor
tuple size, not to the graph depth. Correct call.

### 2. `expected_slice` drift check handles the interesting case correctly

`apply_backlink_mutation` for `RewriteFullSlice` performs four checks in
order:

1. **`layer_slice.contains(&new_element_tid)`** → someone else already added
   us, skip. Correct.
2. **`insert_backlink_if_free(layer_slice, new_element_tid)`** → a concurrent
   operation freed a slot in the target slice; use the free slot instead of
   the planned eviction. This is a drift-downgrade from `RewriteFullSlice` to
   `InsertIfFree`, which is strictly more conservative (we preserve all prior
   neighbors). Correct.
3. **`layer_slice != expected_slice.as_slice()`** → the slice changed since
   planning; our planned replacement is based on stale data, so skip. This
   is the load-bearing drift check from the packet description.
4. **`layer_slice == replacement_slice.as_slice()`** → already applied (idempotent
   retry case), skip.
5. Otherwise → `copy_from_slice(replacement_slice)`.

The order matters: steps 1–2 accept the drift optimistically (use whatever
free capacity appeared), step 3 rejects the drift pessimistically (refuse to
apply a stale plan). Tracing through, every reachable state for
`layer_slice` gets handled correctly. In particular:

- If `layer_slice == expected_slice` (no drift), step 3 passes, step 4
  passes, step 5 applies. ✓
- If `layer_slice` gained an `INVALID` slot (concurrent delete), step 2
  applies and returns early without touching the planned rewrite. ✓
- If `layer_slice` is a *different* full slice (concurrent pruning by
  another insert), steps 1–2 fail and step 3 catches it. ✓

**One subtle issue worth calling out but not a defect here:** since this
slice does not yet support concurrent live inserts (next slice), the drift
check is guarding against a future that doesn't exist yet. Today the check
is dead defensive code — it costs an allocation (the `expected_slice` Vec)
and a comparison per `RewriteFullSlice` mutation. That cost is acceptable
for the insurance it will buy in slice 235. Keep it.

### 3. Simple top-M pruning is acceptable for A5 but has a recall story

`select_backlink_rewrite_slice` picks the `existing_slice.len()` closest
candidates to the *target element's* code (not the query's, not the new
node's), tie-breaking by `(existing before new, physical TID)`. This is the
"keep my nearest neighbors" contract.

Two things to flag:

a. **This differs from Algorithm 4 (heuristic selection)** used in standard
   HNSW. Algorithm 4 chooses *diverse* neighbors by repeatedly accepting
   candidates only if they're closer to the target than they are to any
   already-accepted candidate. Simple top-M is known to cluster the
   neighbor set near the mode of the data distribution and can leave the
   graph under-connected in low-density regions. The packet acknowledges
   this explicitly ("intentionally the simple selection contract from the
   roadmap, not the full heuristic/Algorithm-4 selector").

   **Required check before A5 closeout:** run the recall gate
   (`a4-recall-gate` / whichever lane is load-bearing for FR-003) on a
   corpus that has *live inserts*, not just a freshly built index. The gate
   passed on a build-only index in the A4 closeout. If live-insert pruning
   hurts recall measurably, the A5 closeout packet needs to land either
   Algorithm 4 or a documented recall delta versus build-only. This is not
   a blocker on this slice, but it *is* a blocker on declaring A5 done.

b. **Insert-time and build-time neighbor selection now diverge.** Build
   uses the external `hnsw_rs` crate's selection (which is Algorithm 4 or a
   variant). Insert now uses simple top-M. So a graph built offline and
   then grown by live inserts will have different neighbor-selection
   heuristics applied to different subsets of its nodes. Same recall concern
   as (a) with a different root cause.

   **Follow-up:** document this divergence in the A5 closeout or in an ADR
   alongside ADR-026. Future maintainers will be surprised by it.

### 4. Score function and tie-breaking are correct

`score_backlink_candidate` returns `-score_code_inner_product(...)`
(lower-is-better, matching the BeamCandidate convention used everywhere else
in insert). Sort is:

```rust
left.score.total_cmp(&right.score)      // lower first = best
    .then_with(|| left.is_new.cmp(&right.is_new))  // false (0) < true (1) = existing first on tie
    .then_with(|| compare_item_pointers(&left.tid, &right.tid))  // stable across reruns
```

`.take(existing_slice.len())` takes the prefix (the best). Tie break orders
existing neighbors before the new candidate, so a new node that's score-tied
with an existing one gets rejected — that matches the packet's "do not churn
the graph on ties" policy. Good.

The physical-TID tie break is the right third key because it's the only
tie-break that produces the same slice across retries of the same planning
call (e.g., if the planner re-runs after a drift abort). Without it, a
score-and-is_new tie could produce two different replacement vectors on two
planning passes over the same data, which would make the `expected_slice`
check unreliable on retry.

### 5. Planning loads both element and neighbors per target

The new `plan_backlink_mutation` signature takes `&target_element,
&target_neighbors` as decoded values. Inside
`add_backlinks_to_forward_neighbors`, per selection:

```rust
let element = load_graph_element(...);
let neighbors = load_graph_neighbors(index_relation, element.neighbortid);
plan_backlink_mutation(..., &element, &neighbors, layer)
```

Two page reads per selection. Note that
`graph::load_graph_adjacency(index_relation, element_tid, code_len)` already
exists at `src/am/graph.rs:76-84` and does exactly this work as one helper
call. Minor cleanup, not a defect — `load_graph_adjacency` is the canonical
"give me both" entry point and the insert planner should use it for
consistency with the scan path.

Cost concern for prod-m (m=16): up to 16 selections × 2 page reads = 32
reads per insert, plus another N reads for candidate scoring inside
`select_backlink_rewrite_slice` when a slice is full. At upper layers this
is smaller (≤ m per layer × tree_height layers). Not a hot-path crisis but
worth monitoring on the latency gate; the insert path's page-read budget
just grew by a meaningful factor and we haven't re-measured.

**Recommended follow-up:** re-run the NFR-001 latency lane (from
`225-a4-nfr-001-latency-real-corpus`) with live inserts in the workload,
not just query-only. Insert latency wasn't the gate before; it might be
now.

### 6. `BacklinkMutation` no longer `Copy` — dedup semantics still correct

`BacklinkMutation` stopped being `Copy` because `RewriteFullSlice` carries
owned `Vec<ItemPointer>` fields. The sort + dedup logic still works because
`Vec` implements `Eq` and `dedup()` compares adjacent elements. But:

- `sort_unstable_by` uses custom comparator on `(neighbor_tid, layer)` only,
  ignoring `kind`. That's correct: we want all mutations against the same
  (neighbor_tid, layer) grouped so we can dedup them. If two mutations had
  the same `(neighbor_tid, layer)` but different `kind`, the dedup would
  keep the first one after sort (which was stable-ordered only on
  neighbor_tid+layer). That shouldn't happen in practice — each layer
  selection produces one mutation — but **add a `debug_assert!` that the
  comparator tie-break on `kind` is a no-op**, so a future change to
  selection logic can't silently produce two conflicting plans for the same
  slice without catching it in tests.

- **Dedup correctness:** `mutations.dedup()` uses `PartialEq::eq` on
  `BacklinkMutation`, which now compares the full struct including `kind`.
  If the same selection is somehow emitted twice with identical
  `expected_slice` / `replacement_slice`, dedup removes it. If the two
  produce *different* plans (e.g., because scoring re-ran against different
  state), dedup keeps both — and the subsequent write phase sees two
  mutations for the same tuple with identical layers. `apply_backlink_mutation`
  would process both; the first wins, the second's `expected_slice` check
  fails because the slice now matches `replacement_slice`, so it skips.
  Correct by accident, but fragile.

  **Required follow-up:** assert at plan time that
  `mutations.iter().map(|m| (m.neighbor_tid, m.layer)).collect::<HashSet<_>>()`
  has the same length as `mutations`, i.e., each `(tuple, layer)` pair
  appears at most once. `debug_assert!` is fine.

### 7. Test `test_tqhnsw_insert_rewrites_full_layer0_backlink_slice` is
   correctly constructed

Building a 12-row fixture at `m=2` (layer-0 width = 4), then walking live
inserts 13..=40 looking for one where:

1. A forward target's layer-0 slice was full (no INVALID) *before* the
   insert
2. The new element appears in the target's layer-0 slice *after* the
   insert

When both conditions hold:

- Assert the full `2M` capacity is preserved (`all tids != INVALID`) — catches
  a rewrite that accidentally inserted an `INVALID` slot.
- Assert `after_layer0 != before_layer0` — catches an idempotent "do nothing"
  path that would pass the containment check only because the slice
  happened to already have the element.

The test calls `return` on the first matching target and panics if none
found in 28 inserts. That bound is appropriate for the deterministic level
formula at the test's seed, but worth a comment saying "if the level
formula or seed changes, this bound may need to grow." Minor.

**Gap closed in a follow-up slice:** the test doesn't assert *which*
element got evicted (it only asserts "something changed"). A tighter test
would score the original slice + new element, compute the expected top-N,
and assert exact set equality. That test is write-once cost but will catch
a regression in the sort or tie-break logic. File it as a 10-minute
enhancement in the A5 closeout.

### 8. The layer-0 test case is the only one exercised; upper layers are
   still only "InsertIfFree"

`test_tqhnsw_insert_rewrites_full_layer0_backlink_slice` exercises the
layer-0 rewrite path. There is no equivalent test for
`test_tqhnsw_insert_rewrites_full_layer1_backlink_slice` (or deeper). Upper
layers use the same code path and should behave identically, but the
`backlink_slot_bounds` math for upper layers (`start = 2m + (layer-1)*m`)
has different arithmetic and a bug there would only show up on upper-layer
rewrites.

**Recommended follow-up (234 or closeout):** add a layer-1 variant of the
test. The fixture needs to be wider (to force upper-layer population) and
pick m/ef_construction that produce a full layer-1 slice. This is
essentially the same test at a different layer parameter and should reuse
`layer_neighbor_slice`.

## Verdict

**Approve.** The planning/write split is the right architecture for this
slice, the drift check handles all the states I can reason about, and the
test exercises the real mutation path. Items 3(a), 5, and 8 are the three
things worth tracking on A5 closeout; none of them block landing 233.

Recommended follow-ups:

1. **Item 3a (recall gate vs live-insert corpus):** run FR-003 recall
   validation on an index built partially by live insert, not just
   build-only. Blocker on declaring A5 done, not on landing 233.
2. **Item 3b (divergence from build):** document the insert-vs-build
   neighbor selection divergence in the A5 closeout or ADR-027. One
   paragraph.
3. **Item 5 (latency re-measure):** re-run NFR-001 with a workload that
   includes live inserts; insert latency just grew meaningfully.
4. **Item 5 (cleanup):** use `graph::load_graph_adjacency` instead of
   separate `load_graph_element` + `load_graph_neighbors` calls in
   `add_backlinks_to_forward_neighbors`.
5. **Item 6 (plan uniqueness):** `debug_assert!` that no two mutations
   share the same `(neighbor_tid, layer)` after plan.
6. **Item 7 (evict exact set):** tighten the test to assert the specific
   expected replacement set, not just "something changed."
7. **Item 8 (upper-layer test):** add a layer-1 analogue of the full-slice
   rewrite test.

The load-bearing win here is that the drift check lets planning happen
without any page locks and the commit happens under a single per-page
`EXCLUSIVE` lock. That architectural shape is what slice 235 will need to
be able to hold against concurrent live inserts. Landing it now with
single-session semantics is the right sequencing.
