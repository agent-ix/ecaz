# 30691 — SPIRE Phase 11 plan gap review

Reviewer-initiated planning review of
`plan/tasks/task30-phase11-spire-distributed-production-parity.md`.

## What is solid

- Nine sub-phases (11.1 paper parity → 11.9 AWS gate) with concrete
  checkboxes per subphase.
- Entry rules cite the Phase 9 and Phase 10 detailed task files; the
  Phase 8 scale-packet gate is explicitly preserved.
- Non-goals fence AWS scale, billion-scale, and PQ/PQFastScan out of v1.
- Exit criteria match the goal statement and are measurable
  (cross-node dedupe, two remote nodes, strict/degraded modes,
  bounded/cancellable fanout).
- Validation rules continue the packet-local raw-log convention.

The plan is detailed enough to start slicing; the gaps below are
production-readiness items that should land in plan text *before* the first
implementation packet opens, so reviewer pushback during slices is on
implementation rather than scope.

## P1 — close before first implementation slice

1. **Quantizer/index identity binding to served epoch.** Phase 11.3 says
   the remote endpoint returns "served epoch, node identity, vector
   identity" in compact candidate rows. Quantizer parameters (RaBitQ
   profile, code length, training stats) must also be bound — otherwise
   coordinator merge across nodes silently mixes incompatible scoring.
   *How to apply:* extend 11.3 contract to include
   quantizer-profile fingerprint or fold it into served epoch with a
   strict-mode mismatch rejection rule.

2. **Schema/extension version skew.** No subphase covers the case where
   coordinator and remotes run different `ec_spire` extension versions or
   different opclass binaries during a rolling upgrade. Strict-mode
   behavior should be defined: reject, fallback to local, or run with a
   compat shim.
   *How to apply:* add 11.3 sub-bullet for protocol/extension version
   negotiation; add a corresponding test in 11.6 for "two remotes at
   different versions".

## P2 — close before exit criteria are claimed

3. **Online lifecycle: DROP / REINDEX / CREATE INDEX CONCURRENTLY across
   nodes.** 11.6 covers epoch consistency for steady-state queries but
   not what happens when one remote DROPs or reindexes a SPIRE index
   mid-fanout. The diagnostic libpq path already handles applied-manifest
   cleanup on remote DROP INDEX (per Phase 7 closeout); the production
   path needs an explicit subphase or sub-bullet specifying coordinator
   behavior under those operations.

4. **Boundary-replica freshness across nodes.** 11.2 ensures replicas
   carry the same global vec ID, but no subphase defines who pushes
   manifest updates to remote nodes, how staleness is detected, or how
   degraded mode reports replica drift. ADR-055 covers identity, not
   freshness.
   *How to apply:* add 11.6 sub-bullet for "replica manifest freshness
   diagnostics" and a degraded-mode failure path.

5. **Failure injection coverage.** 11.4 mentions a "slow-remote fixture"
   for tail-latency proof, but production parity should test: connection
   reset mid-batch, remote `pg_terminate_backend`, `statement_timeout`
   firing remotely vs locally, network partition, version skew, and
   remote OOM. Without these, strict/degraded behavior will be
   under-specified.
   *How to apply:* add 11.6 fault-matrix sub-bullet enumerating each
   failure mode and the strict + degraded expected behavior.

6. **Resource governance and backpressure.** 11.4 bounds fanout per
   query; production needs *global* concurrency control across queries
   (per-coordinator and per-remote-node), per-tenant budgets if
   applicable, and protection against thundering herd on remote nodes.
   *How to apply:* add 11.4 sub-bullet for global/per-remote concurrency
   limits and overload-shedding diagnostics.

## P3 — informational, can defer with a note

7. **Coordinator HA / co-located coordinator.** Plan implicitly assumes a
   single coordinator process. State whether multi-coordinator or
   coordinator-as-SPIRE-node is in scope or explicitly deferred (an ADR
   note is fine).

8. **Distributed insert and read-after-write semantics.** Phase 11
   focuses on read fanout. State explicitly whether distributed inserts
   are in scope here or deferred to a future phase, and what
   read-after-write looks like for a distributed-write workflow.

9. **Capacity-planning thresholds in the runbook.** 11.8 lists what to
    capture; for the runbook to be useful, it should also set numeric
    targets (max concurrent queries, max remotes, max PIDs/node) that
    Phase 11 commits to supporting, even if those numbers are revised
    against AWS evidence later.

10. **Paper-section traceability artifact.**

11. **TLS / transport security — explicitly deferred.** PostgreSQL/libpq
    already supports TLS via `sslmode` in the conninfo string, and ADR-058
    forbids logging raw conninfo. Rather than open a TLS subphase now,
    Phase 11 should record an explicit deferral: the operator owns
    `sslmode` in the conninfo secret, and a future post-Phase-11 packet
    will define strict-mode behavior on cert verification failure plus
    credential-rotation runbook guidance. Pulling it in earlier risks
    blocking testing on a credential-store can of worms.
    *How to apply:* one short ADR or a Phase 11 "Deferred" subsection
    naming TLS, credential rotation, and audit logging as out-of-scope
    until distributed correctness work is testable. 11.1 calls for a paper
    parity checklist; consider committing to a packet-local artifact
    format (table mapping paper section → status → evidence packet) so
    later reviewers can audit coverage at a glance.

## Suggested sequencing

Land 11.1 (parity checklist + production gate definition) and the P1
plan-text additions above as a single planning packet before the first
implementation slice. Then 11.2 (writer-side global IDs) is the right
first implementation slice because 11.3, 11.5, and 11.6 strict-mode
correctness all build on it.
