## Response to Review Focus

### Is the experiment trail a fair reading of what the evidence says?

**Yes.** The 1k and 10k numbers tell a clear and consistent story:

| Dataset | ef=800 graph | Exact quantized | Graph as % of ceiling |
|---------|-------------|-----------------|----------------------|
| 1k | 51.7% | 52.3% | **99%** |
| 10k | 39.1% | 43.1% | **91%** |

The graph search is near-optimal relative to its ceiling. The ceiling itself is the problem.

The structural fixes (layer-aware neighbor slicing, greedy descent, staged ordered results) were
clearly the right calls — they brought 1k from ~1.7% to ~50% and restored the graph-vs-exact
relationship to healthy territory.

### Are the recall summary/probe surfaces the right durable seam?

**Yes.** `tqhnsw_graph_scan_recall_probe` returns three orderings side-by-side (fp32 truth, graph
predicted, exact quantized), which isolates the failure layer cleanly. The SQL-callable format makes
re-running cheap. Keep these.

One addition worth considering: a version that accepts an external corpus (not just internally
generated random vectors) so the same probe can be run against structured data.

### Is the recommendation correct?

**Mostly yes, with one amendment.** The recommendation to treat the next step as a project decision
rather than more scan refactoring is correct. However, the framing should distinguish two sub-paths
under "change the measurement path":

1. **Change the test corpus** — measure on structured data (real embeddings) where the quantizer
   ceiling is expected to be much higher. This is cheap and answers whether the A4 gate is achievable
   at all.
2. **Change the gate thresholds** — lower the recall target to something the current quantizer can
   reach. This is a product decision, not a technical one.

Path (1) should come before path (2). If the quantizer achieves 85%+ recall on structured data, the
gate is fine and the corpus was the problem. If it still can't, then the quantizer design itself needs
work.

### The three hypotheses (H1/H2/H3)

Well-framed. Updated assessment after cross-referencing reviews 195-198 and 201:

- **H1 (graph-runtime gap)**: Real but secondary. Graph is at 91-99% of ceiling. Verified the search
  algorithm (`search_layer0_result_candidates_with_successors`) against 4 reference Rust HNSW
  implementations — it correctly implements Algorithm 5 from the HNSW paper. Score convention (negated
  IP, min-heap) is consistent throughout. No traversal bug remains.
- **H2 (quantized-objective mismatch)**: Dominant. Confirmed by review 201 scorer ablation — all
  scoring variants (exact, gamma_zero, code_proxy, decoded) land within 2% of each other at ~49%.
- **H3 (quantized-path implementation defect)**: Possible but less likely. The decoded-approximate
  variant (reconstruct floats, exact dot product) gets the same 49%, which means the codes themselves
  are the ceiling. Any defect would need to be in encoding, not scoring.

## Additional Findings

Verified against reference implementations in `~/dev_bak/hnsw-references/`:

1. **Layer indexing**: hnswlib-rs `get_neighborhood_id()` iterates `0..nb_layer` — layer 0 = index 0.
   Matches `pack_point_neighbor_slots`. No reversal bug.
2. **`search_layer0_result_candidates_with_successors`**: Correct two-heap beam search. Min-heap
   (`BinaryHeap<Reverse<>>`) for candidate expansion, max-heap for bounded result set. Termination,
   filtering, and pruning all match the paper and all four reference implementations.
3. **Greedy descent**: Standard per-layer best-neighbor descent. Correctly loads per-layer neighbors
   via `load_neighbor_tids_for_layer`.

## Disposition

The structural graph fixes are solid. Graph search work should pause until the quantized ceiling moves.
Next step: test on structured data to determine whether H2 is corpus-specific or fundamental to the
quantizer design.
