# Review: Task 42 upgrade matrix smoke (9053)

## Summary

**Major Task 42 milestone:** introduces the `(AM, format_version,
can_read, can_write)` matrix that the task doc named. Adds
`fixtures/upgrade/matrix.csv` recording current supported
format-version combinations, `tests/upgrade_matrix.rs` enforcing
matrix invariants, `make upgrade-smoke` runner, and
documentation in `docs/on-disk-format.md`.

This closes one of the three remaining Task 42 deliverables I
listed in my 9049 feedback (the other two — qemu cross-arch
lane and `pg_upgrade` smoke — are queued; the qemu lane lands
in 9055).

## Confirmations

- **Matrix-as-CSV is the right shape.** Compact, human-readable,
  diff-friendly. Each row is `(am, format_version, can_read,
  can_write)` — exactly the contract the task doc names. A new
  format-version means a new row, not a code edit.
- **Smoke test enforces real invariants:**
  - Unique matrix rows (no duplicate `(am, version)` claims)
  - Readable writable formats (any `can_write` row must also
    `can_read` — write-only formats would be a bug)
  - Existing fixture references (matrix can't claim
    `can_read = true` for a version with no on-disk fixture)
  - Current writable set is recorded (the test asserts which
    formats the running code can write, which forces an update
    when a format-version bumps)
- **Documentation.** `docs/on-disk-format.md` now records the
  matrix and target. Reviewer can find the contract in one
  place.

## Findings

### 1. Right deliverable for the right time

Adding the matrix before there's any actual cross-version
support work is the right ordering — the matrix becomes the
forcing function for future version bumps. When ECAZ adds
format v4, the matrix update is the place the upgrade story
gets pinned down.

### 2. Future addition: `pg_upgrade` integration

The task doc's "`pg_upgrade` smoke" is still queued. That
requires standing up a PG cluster, running an upgrade, and
verifying ECAZ indexes survive. Bigger lift than this matrix —
makes sense to land matrix first as scaffolding.

### 3. Pairing with qemu lane (9055)

The matrix asserts byte stability across versions; qemu (9055)
asserts byte stability across architectures. Together they
cover the cross-everything-but-pg story. With both landing in
the same packet sequence, Task 42 closeout is in sight.
