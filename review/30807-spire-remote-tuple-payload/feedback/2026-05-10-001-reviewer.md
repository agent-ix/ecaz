# Review: SPIRE Remote Tuple Payload Side Channel (30807)

## Summary

Right shape: sibling endpoint (`ec_spire_remote_search_tuple_payload`)
that leaves the 18-column envelope alone, returns a JSON payload keyed
by `(node_id, vec_id)`, and reuses the existing local-heap candidate
resolver so payload rows only emit after origin-node visibility
succeeds. The requested-column validator excludes system columns and
dropped columns correctly via `attnum > 0 AND NOT attisdropped`.

The three review-focus questions:
- Sibling endpoint + stable envelope + JSON side-channel: yes for the
  diagnostic seam, but JSON is not the production-path return shape
  (CustomScan needs to populate tuple slots directly, not decode JSON).
- Acceptable until CustomScan slots are wired? Acceptable as a *proof*
  surface; not as the production transport. Pin the migration path.
- Schema-drift / system-column concerns: validator handles
  dropped/system columns; one regclass-injection nit below.

## Findings

### P2 — Missing-row payload silently returns empty JSON

`ec_spire_remote_search_tuple_payload_for_ctid` wraps the heap fetch in
`coalesce(jsonb_object_agg(...), '{}'::jsonb)`. If the heap row the
local resolver pointed at has been removed between heap-resolution and
projection (vacuum, concurrent DELETE, snapshot-skew), the function
returns a candidate row whose `tuple_payload = '{}'::jsonb` with no
status indication that the projection failed. The candidate count
matches the resolver count, so a downstream consumer cannot tell a
truly-empty projection from a missing-heap-row failure.

This is the exact gap I flagged in the ADR-068 review (P2 missing-row
contract). Recommend one of:
- Fail strict mode with a new status string
  `remote_tuple_payload_missing` (and skip in degraded mode); or
- Add an explicit `tuple_payload_missing bool` column so the
  coordinator-side merge can apply the same fail-closed/skip policy as
  Stage E heap resolution.

The current behavior would let the CustomScan eventually return rows
with NULL projected columns, which is silently wrong.

### P2 — N+1 SPI query per candidate row

For each candidate row from `spire_remote_search_local_heap_candidate_rows`,
the function opens a fresh `Spi::connect(...).select(...)` to fetch its
JSONified payload by CTID. This means N candidates ⇒ N SPI roundtrips
plus N planner/parse cycles for what is essentially the same query
shape with different CTID parameter values.

Suggestion: build one SQL that joins `unnest($1::tid[])` (the CTID
list) with the heap relation on `ctid = ANY(...)`, returning per-CTID
JSON payloads in one round-trip. Same SPI machinery, one parse, one
plan, one execute. Important even for a diagnostic surface because the
CustomScan executor will inherit this pattern.

If the next slice is moving to typed tuple slots anyway, this becomes
moot — but if any other consumer is going to depend on this endpoint,
batch it before that happens.

### P3 — `regclass::text` interpolated into SQL

`ec_spire_remote_search_tuple_payload_for_ctid` uses
`format!("...FROM {heap_relation_regclass} AS heap_row...")`. The
value comes from `$1::oid::regclass::text`, which Postgres formats
with `quote_ident` for ambiguous identifiers and schema-qualifies on
demand, so this is safe in practice. But the safety property is
"trust the regclass output formatter" rather than a local invariant
of this function.

Lower-risk pattern: look up `pg_class.relname` and `pg_namespace.nspname`
and call `quote_ident` explicitly, or pass the relation OID into a
prepared statement that does the table reference itself (e.g. via
`format(%I.%I, nsp, rel)` inside the SPI). Cosmetic; not blocking.

### P3 — `consistency_mode` parameter is inert in the new code path

The function signature accepts `consistency_mode: String` and
forwards it to `spire_remote_search_local_heap_candidate_rows`, but
the new payload-fetch step does not consult it. Combined with the
P2 missing-row finding above, this means strict and degraded behave
identically when a payload row's heap target vanished. Once the
missing-row policy is pinned, surface it through this parameter.

### P3 — Move to typed tuple slots before adding more callers

The packet asks whether returning JSON is acceptable. For a diagnostic
proof surface: yes. For anything the CustomScan executor consumes:
no — JSON forces a parse-and-decode on every projected row that the
executor would otherwise get for free from a tuple slot. Suggest
treating this packet as the end of the JSON-return shape; the next
slice (CustomScan tuple-payload slot delivery) should switch to
returning a tuple-shaped record (one column per requested heap column,
typed) and replace this endpoint rather than adding consumers.

### P3 — Column ordering inside payload

`jsonb_object_agg(... ORDER BY requested.ordinality)` preserves
caller-requested order in the JSON keyset. Good — and worth
documenting in the request packet so downstream consumers can rely
on it.

## Reviewer recommendation

Accept as a diagnostic seam. Before wiring CustomScan tuple delivery
to consume this endpoint, pin the missing-row contract (P2) and batch
the heap fetch (P2). Treat the JSON return shape as scaffolding to be
replaced when tuple slots land.
