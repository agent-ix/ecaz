## 30630 — local

Three new contract surfaces:
- `ec_spire_remote_operator_entrypoint_contract()` — names the
  compact operator-facing subset (8 entrypoints) of the larger
  ~80-function diagnostic surface.
- `ec_spire_remote_libpq_connection_lifecycle_contract()` —
  per-query, no pooling v1, executor-owned secret resolution,
  never-expose-raw-conninfo, fail-closed-no-implicit-retry,
  one connection per ready remote node per query (or per
  publication).
- `ec_spire_remote_conninfo_secret_resolution_contract()` —
  three candidate provider policies with `selected_v1`,
  `not_selected_v1`, and `rejected_v1` markers. The selected
  policy is `external_executor_secret_provider`. The rejected
  policy explicitly forbids storing raw conninfo in
  extension-owned catalogs.

**This packet directly addresses three concerns from the prior
comprehensive architecture review:**

1. ✅ **Operator entry-point subset** (concern #1):
   `entrypoint_contract` names the operator-facing 8 functions.
   No more "which of the 80 surfaces is the operator
   entrypoint?" ambiguity.
2. ✅ **Connection lifecycle policy** (concern #4):
   `lifecycle_contract` makes the policy explicit and
   enforceable. Per-query, no pooling, fail-closed.
3. ✅ **Secret resolution mechanism** (concern #6, the
   "biggest open architectural decision"): the
   `secret_resolution_contract` has three rows naming the
   selected/not-selected/rejected providers with explicit
   `raw_conninfo_allowed = false` and named validators.

Per packets in 30618-30629 these contract claims are paired
with PG18 invariants that fail if the implementation drifts.

### Forward-looking findings

1. **Cost model scale constants are placeholders.**
   `SPIRE_ROUTING_SCORING_DIMENSION_SCALE` etc. are magic
   numbers in `cost.rs:11-14`. Need empirical calibration
   against the scale packet (30629 → measurement run). After
   measurement, either document the calibration source or
   pull into reloptions for operator tuning.

2. **`ec_spire_amcostestimate` reads active snapshot
   diagnostics every call** (30620 finding). For hot OLTP
   workloads with frequent re-planning, this is a per-query
   manifest read. Worth measuring with the benchmark harness
   once the scale packet runs.

3. **Operator docs slightly stale** (30627 finding) — the
   docs predate 30628's physical cleanup. The cleanup status
   taxonomy in docs still says `blocked_not_implemented`;
   should be updated to the new four-state taxonomy and to
   point at `ec_spire_index_epoch_cleanup_run`. Cosmetic.

4. **Cleanup runs hold publish lock for full duration**
   (30628 finding). Acceptable for v1 operator-driven cron;
   worth a docs note that cleanup creates a pause window for
   concurrent inserts.

5. **`removed_tuple_bytes` measures payload bytes only**
   (30628 finding) — line pointers remain after no-compact
   delete. Probably the right thing for capacity planning,
   but the field name could mislead operators. Cosmetic.

6. **Failure-injection test for cleanup with active queries
   missing** (30628 finding). Happy path is covered; an
   active-query simulation that asserts cleanup refuses to
   reclaim that epoch's tuples would harden the contract.
