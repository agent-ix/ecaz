# Feedback: Linear Scan Exhaustion and Direction Guards

Request:
- `review/17-linear-scan-exhaustion-and-direction-guards.md`

**Reviewer:** Claude (Opus)
**Date:** 2026-04-05

## Answers to Review Questions

### Is there any missing regression around rescanning after exhaustion?

**Covered.** The `test_tqhnsw_gettuple_rescan_after_exhaustion_restarts_scan` test (line 2139) explicitly:
1. Exhausts the scan by draining all tuples
2. Calls `amrescan` again
3. Drains all tuples a second time
4. Verifies both passes return identical heap TID sets

This covers the key invariant: after exhaustion, `amrescan` resets the cursor and `scan_exhausted` flag, allowing a fresh forward scan.

**One minor gap:** There is no test for rescanning after exhaustion with a *different* query vector. The current test reuses the same query datum. Since the linear scan doesn't actually use the query for scoring/ordering (it returns all tuples in page order), the query content doesn't affect the result set. But testing with a different query would verify that `store_scan_query` correctly frees the old payload and stores the new one during rescan-after-exhaustion. Low priority — the `store_scan_query` cleanup is tested via the repeated-rescan-overwrites-dimensions test.

### Should exhausted-scan behavior assert anything about retained `xs_heaptid` state?

**No — `false`-only coverage is correct for this stage.** After exhaustion, `amgettuple` returns `false` and does not modify `xs_heaptid`. The last valid `xs_heaptid` from before exhaustion remains in the scan descriptor. This is consistent with PostgreSQL's convention: `xs_heaptid` is only meaningful when `amgettuple` returns `true`. The executor never reads `xs_heaptid` after a `false` return.

Asserting on `xs_heaptid` after exhaustion would be testing a PostgreSQL implementation detail that could change, so **the current approach of testing only the return value is the right boundary.**

### Is strict backward-direction rejection the correct current contract?

**Yes, absolutely.** The backward-direction error (line 443-444) is the correct contract for several reasons:

1. **`amcanbackward = false`** (line 72) tells the planner this AM doesn't support backward scans. PostgreSQL should never request one. If it does, something is wrong.

2. **Silent no-op would hide bugs.** If backward scan returned `false` silently (no tuples), a planner or executor bug that sends `BackwardScanDirection` would go undetected. The error makes the contract explicit.

3. **Forward-only is the permanent contract for ORDER BY index scans.** Even when graph traversal is implemented, the index produces results in scored order (best-first), not in a reversible sequence. Backward scan has no meaningful semantics for ANN indexes. pgvector (ivfflat, hnsw) also rejects backward scans.

**Do not soften this before ordered traversal exists.** Keep the strict error.

## Additional Findings

### The debug helpers are well-structured

`debug_gettuple_exhaustion_state` (line 2443) collects all heap TIDs, then calls `amgettuple` two more times to verify exhaustion stability. This is the right pattern — it tests both the drain and the post-drain behavior in a single helper, reducing test boilerplate.

`debug_gettuple_backward_after_rescan` (line 2472) triggers the expected panic directly. The `#[should_panic]` annotation at line 2205 captures it cleanly.

### Exhaustion + concurrent insert edge case (documentation only)

If a concurrent transaction inserts new tuples into the index after the scan has started but before it completes, the linear scan's `block_count` (fetched at line 553) would see the new blocks. This means the scan might see some but not all concurrent inserts, depending on timing. This is acceptable for the current bootstrap (not MVCC-compliant anyway — the linear scan doesn't check tuple visibility). When graph traversal is implemented with proper snapshot-based visibility, this edge case resolves naturally.

No code change needed — just worth noting in the ADR that the linear scan bootstrap is not snapshot-consistent by design.

### `scan_exhausted` state persists across page I/O correctly

After exhaustion, `scan_exhausted = true` means `next_linear_scan_heap_tid` returns `None` at line 549 without any I/O. Even if new pages are added concurrently, the exhausted flag prevents stale cursor positions from being used. Only `amrescan` can clear this flag. This is the right safety boundary.
