# Review: SPIRE Pipeline Payload Projection Metrics (30974)

## Summary

Phase 12.2 throughput-measurement harness slice. Existing
`ecaz bench spire-pipeline` query-metrics timed id-only
coordinator KNN — `--remote-tuple-transport` could select
JSON vs typed without necessarily forcing payload slot
delivery. This slice adds:

- `--query-metric-projection-columns` (comma/repeatable
  list of extra corpus columns);
- identifier validation;
- SQL gen keeps `id` first for recall, appends unique
  projected payload columns;
- benchmark header records active projection shape.

Phase 12.2 throughput-measurement row stays open until
packet-local JSON vs `pg_binary_attr_v1` artifacts exist.

## Findings

### P1 — Closes the measurement-validity gap

This is the **right fix for a measurement-validity bug**
that 30964 + 30965 had inadvertently introduced. Without
projected payload columns, an id-only KNN query returns
just the PK, and the remote endpoint may serve that without
exercising payload tuple delivery — meaning the
JSON-vs-typed selector could show no difference because
neither path was actually moving payload bytes.

Forcing payload columns into the projection ensures the
benchmark exercises the path under measurement. Without
this, throughput numbers from 30964/30965 would be
misleading — "JSON and typed are equivalent" could be a
real finding OR could be "we measured nothing." This slice
makes the difference unambiguous.

### P2 — `id` first for recall accounting

Keeping `id` as the first selected column is the right
constraint. The recall calculation from 30965 looks up
exact local truth by PK, so the SQL must return PK in a
known position. First-column convention is the simplest
stable contract — recall code reads column 0 unconditionally.

If column ordering ever became flexible (e.g., named-
column recall lookup), this constraint could relax. For
now, position-based is the right primitive.

### P2 — Identifier validation

Validating column names in the CLI rather than passing
through to SQL is the right defensive primitive. SQL
injection through bench parameters would be a real issue
if validation lived only at the SQL layer. Identifier-
character allow-list at the CLI catches it at the safest
boundary. Confirm the validation rejects:

- quoted identifiers (`"name"`) — these would need different
  escaping;
- schema-qualified names (`schema.col`) — orthogonal axis
  not in scope here;
- empty strings;
- duplicate column names (the request notes "unique
  projected payload columns" so probably handled).

### P2 — Benchmark header records projection shape

Right reproducibility primitive. An artifact looking at
`projection: id, embedding, metadata` tells the analyst
what was measured. Without this, comparing two artifact
runs would require correlating to the CLI invocation log,
which is a fragile cross-reference.

The `--remote-tuple-transport` mode (from 30964) already
lands in the header. Combining it with the projection shape
makes a single artifact header sufficient for "JSON
id-only," "JSON id+payload," "typed id-only," "typed
id+payload" comparisons.

### P3 — Tracker scope discipline

Correctly leaves the actual throughput measurement row
open. This is harness work, not measurement evidence. The
measurement row closes when a packet captures and reports
JSON-vs-typed deltas with the projection columns
representative of production payload shapes.

Honest sub-bullet placement.

### P3 — Default projection?

Worth confirming the default behavior when
`--query-metric-projection-columns` is absent. If id-only
remains the default, that's the right backwards-compatible
choice for existing callers but may produce misleading
results for the throughput row specifically. Worth one
sentence in the bench docs noting that "for typed-vs-JSON
throughput, run with payload projection columns; id-only
runs do not exercise payload tuple delivery."

### P3 — Throughput row remaining work

To close the throughput row, the next packet needs to:

1. Identify representative payload columns from the local
   capacity profile (30952);
2. Run JSON and typed sweeps with matching projection;
3. Capture latency + throughput (rows/sec or bytes/sec)
   per `nprobe` for both transport modes;
4. Report the delta (typed should be faster for tuple-
   heavy workloads per the 30896 P1 hypothesis).

Pin a tracker note that those four steps are the closure
path for the row.

## Reviewer recommendation

Accept. **Right fix for the measurement-validity gap** —
without payload projection, the JSON-vs-typed selector
couldn't measure what it was supposed to. Identifier
validation at CLI is correct safety boundary, header
reproducibility is right primitive, tracker scope honest.
