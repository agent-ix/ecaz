# Remote descriptor catalog review

First durable Phase 7 persistence: `ec_spire_remote_node_descriptor`
table + `ec_spire_register_remote_node_descriptor` upsert. Major
landing — and one **real correctness finding** plus one contract
gap.

## Findings

### 1. Generation-monotonicity not enforced at upsert (BLOCKING-LIKE)

**Severity:** real correctness gap; contract claim broken at the
persistence boundary.

`ec_spire_register_remote_node_descriptor`
(`src/lib.rs:1352-1454`) validates `descriptor_generation >= 0`
but does not enforce that it must *advance* on update. The
registration contract step 9 (30574/30585) declares
`validator: must_advance_descriptor_generation`,
`failure_status: stale_descriptor_generation` — but the upsert
unconditionally overwrites every field including
`descriptor_generation` regardless of its relationship to the
existing row's generation.

A caller that re-upserts the same row with an unchanged or lower
generation will silently succeed. This breaks two design goals:

- **Concurrency safety.** Two coordinators racing to register
  the same `(coordinator_index_oid, node_id)` cannot use the
  generation as the conflict-resolution key, because the
  later-arriving row always wins regardless of whose generation
  is newer.
- **Contract fidelity.** The 30585 SQL contract row claims a
  `must_advance_descriptor_generation` validator. SQL clients
  reading that contract will believe re-registration with stale
  generation fails closed. Today it does not.

**Suggested fix:** add `WHERE EXCLUDED.descriptor_generation >
ec_spire_remote_node_descriptor.descriptor_generation` to the
`ON CONFLICT DO UPDATE`, or split the upsert into a check + insert
that errors on non-advancing generation. Either way, add a PG18
test that re-registers the same node with the same generation
and asserts the second call fails.

### 2. Connection plan ignores descriptor_state (BLOCKING-LIKE)

**Severity:** real correctness gap; bypasses fail-closed contract
for `disabled` and `failed` descriptors.

`load_remote_libpq_connection_descriptors`
(`root/remote_candidates.rs:1090-1167`) loads descriptor rows
filtered only by `coordinator_index_oid` and `node_id` — it does
**not** filter on `descriptor_state`. Then in
`remote_search_libpq_connection_plan_rows` (line 1220-1228), the
readiness check is just `descriptor.is_some()`, with no further
gate on the descriptor's state.

Result: a descriptor row with
`descriptor_state IN ('disabled', 'failed')` is loaded as if it
were `active`, and the connection plan reports:

- `conninfo_resolution = SPIRE_REMOTE_CONNINFO_READY`
- `pipeline_mode = SPIRE_REMOTE_TRANSPORT_LIBPQ_PIPELINE`

This violates:

- The Phase 7 design (30552), which states only `active` and
  `draining` are eligible for reads.
- The descriptor contract (30574), validator
  `must_be_active_or_draining_for_reads`.
- The registration contract step 8 (30585),
  `failure_status: disabled_remote_node`.

Today the bug is invisible because no caller registers `disabled`
or `failed` descriptors yet. But the moment an operator marks a
node `failed` to take it out of rotation, the libpq connection
plan will continue to report it as ready to dispatch, and a
future libpq executor reading from this surface will dispatch to
a dead node.

**Suggested fix:** filter the SQL in
`load_remote_libpq_connection_descriptors` by
`descriptor_state IN ('active', 'draining')`, OR keep the full
result but propagate `descriptor_state` into
`SpireRemoteLibpqConnectionDescriptorRow` and gate the readiness
check on it. Add a PG18 test: register a `failed` descriptor and
assert the connection plan reports
`requires_remote_node_descriptor` (or a new
`disabled_remote_node` status) rather than `ready`.

### 3. Descriptor lifecycle string literals at the boundary

The check constraint (`sql/bootstrap.sql:144`) and the validator
(`lib.rs:1390`) hardcode `'active' | 'draining' | 'disabled' |
'failed'`. The 30586 registry pattern should extend to these
states. Now that 30595 introduces a real persistence boundary,
the literal-vs-constant drift matters more — the database CHECK
constraint and the Rust validator must agree exactly.

## Confirmations

1. **Conninfo opacity preserved.** The table stores only
   `conninfo_secret_name text`; raw conninfo never enters the
   catalog. PG18 test exercises the registration upsert end to
   end.
2. **Coordinator-index validation.** Registration calls
   `open_valid_ec_spire_index(...)` before mutating, so a
   descriptor cannot be attached to a non-`ec_spire` index OID.
3. **Snapshot integration.**
   `load_remote_node_descriptor_rows` does parse and propagate
   `descriptor_state`, so the regular
   `ec_spire_remote_node_snapshot` reports the right status —
   the bug is specifically in the libpq connection plan path.
4. **`ON CONFLICT (coordinator_index_oid, node_id) DO UPDATE`** is
   the right primary key + conflict resolution shape for
   re-registration; the only issue is the missing generation
   guard.
5. **PK + CHECK constraints** (state allowed values, nonempty
   secret/identity/regclass, non-negative epochs) match the
   contract and the Rust validators.
