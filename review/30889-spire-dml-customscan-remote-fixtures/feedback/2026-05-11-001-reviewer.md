# Review: SPIRE DML CustomScan Remote Fixtures (30889)

## Summary

**Multicluster proof for transparent UPDATE and DELETE.** Two
new PG18 fixtures stand up loopback remote clusters, register
remote descriptors and placement rows, then run actual
`UPDATE`/`DELETE` SQL through the coordinator and verify:

- Transparent remote UPDATE:
  - `EXPLAIN` records `plan_tree_replaced_customscan`.
  - `ROW_COUNT = 1` from the coordinator-side execution.
  - The owning **remote** heap row's value changed.

- Transparent remote DELETE:
  - `EXPLAIN` records `plan_tree_replaced_customscan`.
  - `ROW_COUNT = 1`.
  - Local placement row removed.
  - Remote DELETE prepared but not visible until coordinator
    transaction resolves (correct 2PC semantics for DELETE).

These are the "actually works end-to-end" proofs that close the
30887 P3 about remote-DELETE 2PC visibility.

The three review-focus questions:
- Fixtures prove the transparent DML CustomScan path, not just
  primitives: yes — they execute SQL via Spi::run, not direct
  primitive function calls.
- Remote DELETE assertion correctly treats the prepared remote
  transaction as not visible before local transaction
  resolution: yes — the assertion checks `pg_prepared_xacts`
  rather than the remote heap directly.
- Task-file checklist now matches the top-level CustomScan
  plan-tree replacement architecture: yes (and 30890 finalizes
  the tracker reconciliation).

## Findings

### P3 — Fixture conninfo env var naming

`EC_SPIRE_REMOTE_CONNINFO_SPIRE_REMOTE_CUSTOMSCAN_UPDATE_REMOTE_SQL`
follows the existing fixture naming pattern. Long but
self-describing. No change needed.

### P3 — Remote UPDATE fixture asserts the remote heap value

The UPDATE fixture queries the loopback client directly to
verify the post-commit remote value. That's the right shape:
the test runs through the public SQL surface (UPDATE statement
on the coordinator) and verifies the side effect on the remote.
End-to-end coverage rather than primitive-only coverage.

### P3 — Comment about historical insert gid prefix

The diff adds a comment noting that the shared remote 2PC helper
still uses the historical insert gid prefix for prepared DELETE
transactions. Worth a follow-up tracker item to rename the
prefix to be operation-neutral (`ec_spire_2pc_<op>_<...>`)
before the cleanup packet. Cosmetic but operators reading
`pg_prepared_xacts` would benefit from gid-shape consistency.

## Reviewer recommendation

**Accept the milestone.** Multicluster fixtures prove the
end-to-end transparent DML path lands rows on the right remote
shard. ADR-069 v1 transparent UPDATE / DELETE / PK SELECT are
now provably correct on real two-cluster setups.
