## Answers to Review Questions

### Is `src/am/scan.rs` the right place, or should it move to the graph-read surface?

**scan.rs is the right home for now.** The `collect_successor_candidates` helper (scan.rs:813) is a scan-specific policy function — it decides which neighbors to promote into candidates, including the INVALID-skip guard. The graph-read surface (`graph.rs`) is a structural decode layer that returns raw data. Filtering INVALID TIDs is a traversal-policy concern, not a graph-decode concern.

That said, if future graph consumers (e.g., graph-aware insert, vacuum pruning) also need to iterate neighbors and skip INVALIDs, extracting a `graph::valid_neighbor_tids(neighbors) -> impl Iterator<ItemPointer>` filter would be appropriate. But centralizing now would be premature — the scan path is currently the only consumer of neighbor iteration.

### Does the helper capture the current policy cleanly?

**Yes.** The closure-based design (`candidate_for_tid: FnMut → Option<ScanCandidate>`) cleanly separates the INVALID-skip guard from the element-load-and-score policy. The caller in `refill_candidate_frontier_from_source` (scan.rs:670) adds visited-set checking and scoring on top. This separation will scale when the scoring or visited-set logic changes without touching the INVALID-skip guard.

### Other invalid-ref edges in the scan bootstrap path?

No additional unguarded paths found. The only neighbor-iteration site is `refill_candidate_frontier_from_source`, which flows through `collect_successor_candidates`. The linear scan path (`next_linear_scan_heap_tid`, scan.rs:842) iterates page tuples by offset, not neighbor TIDs, so INVALID neighbor refs are irrelevant there.

The unit test (`collect_successor_candidates_skips_invalid_and_collects_multiple`, scan.rs:1370) verifies that INVALID TIDs are skipped and never passed to the candidate closure. Coverage is sufficient.
