## Note: Code Has Evolved

The `Vec<ScanCandidate>` is now maintained in parallel with a `BeamSearch<ItemPointer>` scheduler (scan.rs:1062). `consume_candidate_frontier_head` (scan.rs:661) now consults the beam scheduler's `peek_best()` first and calls `forget_queued` to keep the heap in sync on consume. The Vec remains the source of truth for candidate data; the beam scheduler provides ordering.

## Answers to Review Questions

### Is a heap-owned `Vec<ScanCandidate>` the right next representation?

**Yes — and it has proven durable.** The Vec is now part of a dual-container architecture alongside the `BeamSearch` scheduler. It holds the actual `ScanCandidate` data while the beam's `BinaryHeap` provides O(1) best-candidate access.

The ownership model (`Box::into_raw` / `Box::from_raw` on the Vec itself, stored as `*mut Vec<ScanCandidate>` in scan.rs:1061) is consistent with visited set and prepared query ownership. The accessor helpers `candidate_frontier_ref` / `candidate_frontier_mut` (scan.rs:328-336) centralize the null-check and dereference, which is clean.

### Lifecycle or compaction edge cases around head removal?

`consume_candidate_frontier_head` (scan.rs:661) uses `Vec::remove(head)` which shifts elements left, then calls `forget_queued` on the beam scheduler and recomputes the head. This is O(n) for the Vec shift plus O(n) for `forget_queued`'s heap rebuild, which is fine for the current `MAX_BOOTSTRAP_FRONTIER_CANDIDATES = 3`.

One edge worth noting: `Vec::remove` on the head index is correct because `recompute_candidate_frontier_head` is called immediately after, which rescans and updates the index. There's no window where a stale index could be consumed.

`clear_scan_candidate_state` (scan.rs:295-306) correctly handles the Vec lifecycle — if null, allocates; otherwise clears. This is called from both rescan and exhaustion paths. No lifecycle gap.

### Should the frontier-head representation change before traversal expansion?

**The `Option<usize>` is now secondary.** The beam scheduler's `peek_best()` is the primary source for head selection via `recompute_candidate_frontier_head` (scan.rs:633). The `Option<usize>` serves as a cached index into the Vec for fast access. When the Vec is eventually replaced entirely by the beam, the cached index will become unnecessary.
