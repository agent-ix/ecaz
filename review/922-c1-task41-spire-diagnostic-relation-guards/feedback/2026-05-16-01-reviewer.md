# Review: Task 41 SPIRE diagnostic relation guards (922)

## Summary

First post-strategy migration packet. Migrates ~14 SPIRE diagnostic
SQL functions to `open_valid_ec_spire_index_guard` using the
**explicit `drop(index_relation)`** idiom rather than the `{}`
capture form used in earlier packets. Baseline 4,579 → 4,539 (−40),
the largest single-packet delta of the run.

I'm satisfied with the slice. The cross-cutting summary at the end
covers the whole 915-922 sequence.

## Confirmations

- **Pattern adopted: explicit `drop`.** Every migration follows the
  shape:
  ```
  let index_relation = open_valid_ec_spire_index_guard(oid, name);
  let result = unsafe { am::call(index_relation.as_ptr(), ...) };
  drop(index_relation);
  // ... build owned TableIterator
  ```
  This is the form I suggested in my 914 feedback. It makes the
  guard's lifetime end visible at the callsite (no nested block
  needed) and is more debuggable in stack traces than implicit
  end-of-scope drop.
- **No relation lifetime extension.** Spot-checked all hunks (lines
  9197, 9271, 9347, 9403, 9471, 9544, 9626): every AM call returns
  owned data into a local, the `drop` runs immediately after, then
  the `TableIterator` builds from the owned `Vec`. No `as_ptr()`
  captured into a closure, no SPI between open and drop.
- **Baseline arithmetic.** −40 across ~14 migrations averages
  ~2.8 SAFETY-annotated entries per prior function (consistent with
  the `index_open` + `index_close` + intermediate raw deref pattern
  the previous code carried).
- **Strategy compliance.** This is the kind of slice the 921
  strategy targets first: read-only AM diagnostics with no
  validation-only or paired-resource complexity. The remaining
  raw `open_valid_ec_spire_index` callsites (still many in 9700+,
  10000+, 13000+) are queued for the next slices.

## Findings

No blockers.

### Cross-cutting confirmations

- `AccessShareIndexRelation` from packet 911 has scaled cleanly
  through 22 migrated functions. No new `unsafe` impl debt.
- Three callsite idioms emerged:
  1. `let guard = ...; AM call; drop(guard);` (preferred — used in
     922 throughout, in 920 local branch).
  2. `let result = { let guard = ...; AM call };` (used in 915-919
     and 914 catalog summary). Equivalent semantics.
  3. `{ let _index_relation = ...; }` validation-only (used in 912
     and 914 for catalog readers and the descriptor registration
     validator). 921 strategy item (4) flags converting these to a
     named `validate_ec_spire_index` helper.
- Drop-on-`pgrx::error!`-unwind is the load-bearing invariant
  across all 22 migrations. It is correct under pgrx 0.17 but
  implicit. Task 41 closeout should either:
  - add a focused regression test (open a guard with a deliberately
    wrong `relam`, assert relation refcount returns to pre-call
    value after the elog-caught ERROR), or
  - an ADR noting the pgrx version contract so a future pgrx bump
    audits this.
