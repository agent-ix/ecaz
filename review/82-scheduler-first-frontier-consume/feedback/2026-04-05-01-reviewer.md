# Feedback: Scheduler-First Frontier Consume

Request:
- `review/82-scheduler-first-frontier-consume.md`

**Reviewer:** Claude (Opus)
**Date:** 2026-04-05

## Answers to Review Questions

### Is scheduler-first consume with cached-index fallback the right intermediate contract?

**Yes.** Verified in `consume_candidate_frontier_head` (scan.rs:649-659):

```rust
let head = bootstrap_expansion_mut(opaque)
    .peek_best()
    .and_then(|candidate| find_candidate_frontier_index(opaque, candidate.node))
    .or(opaque.candidate_frontier_head)?;

let consumed = candidate_frontier_mut(opaque).remove(head);
bootstrap_expansion_mut(opaque).forget_queued(consumed.element_tid);
recompute_candidate_frontier_head(opaque);
```

The flow is:
1. Ask beam scheduler for its best queued node
2. Map that node to a frontier Vec index via `find_candidate_frontier_index`
3. If either step fails, fall back to cached `candidate_frontier_head`
4. Remove from Vec, forget from beam scheduler, recompute head

This makes the beam scheduler authoritative for *which* node to consume while using the Vec for the actual `ScanCandidate` data. The `.or(opaque.candidate_frontier_head)` fallback handles the case where the beam scheduler and Vec are temporarily out of sync (should be rare but safe).

The `forget_queued` call at scan.rs:656 keeps the beam scheduler aligned after Vec removal. This is the right synchronization point â€” the node is removed from both containers in the same operation.

### Does the debug helper expose correct "actually consumed" candidate semantics?

The debug helpers in `scan_debug.rs` call `consume_and_refill_bootstrap_frontier` (scan.rs:708-717) which calls `consume_candidate_frontier_head`. Since consumption now goes through the scheduler-first path, the debug helpers report the actually consumed candidate as determined by the beam scheduler, not the cached Vec index. This matches runtime behavior.

### Should remaining vector-index assumptions move into the shared search seam?

**This is the key remaining transition.** The current architecture has two parallel containers:
- Frontier Vec (`candidate_frontier: *mut Vec<ScanCandidate>`) â€” holds full candidate data, indexed by position
- Beam scheduler (`bootstrap_expansion: *mut BeamSearch<ItemPointer>`) â€” holds `(node, score, source)`, ordered by BinaryHeap

They're kept in sync through:
- `seed_discovered_candidates` â€” adds to both
- `consume_candidate_frontier_head` â€” removes from both
- `find_candidate_frontier_index` â€” maps beam nodes to Vec indices

The next natural step is to eliminate the Vec by enriching `BeamCandidate` or `BeamSearch` to carry the materialization-relevant fields (`source_tid`, `score_valid`). Then:
- `peek_best` returns the full candidate data
- `pop_best` removes and returns it
- `forget_queued` operates on the single container
- `candidate_frontier_head` becomes unnecessary
- `find_candidate_frontier_index` becomes unnecessary

This would simplify the scan state machine significantly and remove the O(n) `forget_queued` drain-and-rebuild issue.

## Additional Findings

**Potential edge case:** If `peek_best()` returns a node that `find_candidate_frontier_index` can't find in the Vec (e.g., the beam scheduler has a stale entry), the code falls through to `opaque.candidate_frontier_head`. If that's also `None`, the entire `consume` returns `None` â€” correct behavior, the frontier is effectively empty. But the stale beam entry stays in the scheduler. Subsequent calls to `recompute_candidate_frontier_head` would hit the same stale entry again, causing repeated fruitless lookups until something else clears it.

In practice this shouldn't happen because `seed_discovered_candidates` and `forget_queued` keep the structures synchronized. But adding a `forget_queued` call for the stale node when `find_candidate_frontier_index` fails would be a defensive improvement:

```rust
let head = bootstrap_expansion_mut(opaque)
    .peek_best()
    .and_then(|candidate| {
        find_candidate_frontier_index(opaque, candidate.node)
            // If beam has node but Vec doesn't, clean up beam
            .or_else(|| { bootstrap_expansion_mut(opaque).forget_queued(candidate.node); None })
    })
    .or(opaque.candidate_frontier_head)?;
```

This is not blocking â€” just a defensive note for when the dual-structure is still in use.
