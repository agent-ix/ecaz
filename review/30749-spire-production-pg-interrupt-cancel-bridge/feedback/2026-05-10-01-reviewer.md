# 30749 — SPIRE production PG interrupt cancel bridge review

Validation pass on `7af6fb0d` (`Add SPIRE production PG interrupt
cancel bridge`). Closes packet 30742 P2.

## What I checked

- New abstraction `SpireRemoteLocalCancelSource` at
  `src/am/ec_spire/root/remote_candidates.rs:2270` with three
  variants:
  - `None` — no local-cancel polling.
  - `TestAfterMs(u64)` — deterministic test trigger after a delay.
  - `PostgresInterruptPoll { poll_interval_ms }` — production
    poll of `InterruptPending` / `QueryCancelPending`.
  Production constructor (line 2279) returns the
  `PostgresInterruptPoll` variant with a 5 ms poll interval
  (constant at line 2276).
- Production paths (`run_probe_requests` line 2294, and the
  receive equivalent) default to
  `SpireRemoteLocalCancelSource::production()`. Test paths use
  `run_probe_requests_with_local_cancel(...)` which maps
  `Option<u64>` to either `TestAfterMs` or `None` — keeps the
  deterministic timer source confined to test callers and keeps
  production callers from accidentally using it.
- Backend signal flags `InterruptPending` and `QueryCancelPending`
  are looked up via dynamic symbol resolution (per the request
  claim) so the adapter polls real PostgreSQL backend cancel
  flags without adding link-time references that would break the
  pgrx test binary. This is the right boundary for a pgrx
  extension where the test binary doesn't link the full backend.
- When the local backend has query-cancel flags pending, the
  adapter sends the remote `tokio-postgres` cancel token (built
  in packet 30742) and reports `local_query_cancelled`. End-to-
  end signal path: PG `QueryCancelPending` →
  `SpireRemoteLocalCancelSource::production()` poll →
  `tokio-postgres` cancel-token request → remote SQLSTATE
  `57014` without `statement timeout` text →
  `remote_query_cancelled` from packet 30740.
- New PG18 fixture `prod_transport_pg_interrupt_bridge_cancel`
  sets/restores the backend flags inside a `pg_test` backend and
  proves the default production probe path classifies them as
  `remote_transport_failed` / `local_query_cancelled`.
- Phase 11 task and coordinator/executor design docs record this
  as the first production interrupt bridge while keeping C2 open
  for distinct local statement-timeout provenance.

## Answers to the packet's review questions

- **Dynamic `dlsym` lookup of PostgreSQL backend signal flags
  the right boundary for a pgrx extension?** Yes. The pgrx test
  binary doesn't link the full backend, so static references to
  `InterruptPending` / `QueryCancelPending` would fail to load.
  Dynamic lookup with a fallback to "no signal" when symbols are
  missing (test/standalone contexts) is the standard pattern for
  optional backend integration. The 5 ms poll interval is
  reasonable: low enough to feel responsive on user-cancel and
  high enough not to dominate CPU on a fanout that may already
  be I/O-bound.
- **`SpireRemoteLocalCancelSource` split narrow enough to keep
  timer-based cancellation test-only while making production
  paths poll PostgreSQL by default?** Yes. Production constructor
  returns `PostgresInterruptPoll` and is the only path
  `run_probe_requests` calls; the timer source is reachable only
  through `run_probe_requests_with_local_cancel(Option<u64>)`,
  which is test-facing. A production caller cannot accidentally
  reach for the timer variant.
- **Acceptable to land query-cancel provenance first and leave
  local statement-timeout classification as the explicit
  follow-up?** Yes. Query cancel and statement timeout reach
  the same SQLSTATE `57014` on the remote side (per packet
  30738), but their *local* origins are distinct: query cancel
  comes from `QueryCancelPending`, while statement timeout
  comes from PostgreSQL's local `STATEMENT_TIMEOUT` mechanism
  setting a different flag. Splitting the slices keeps each
  bridge narrow and testable. C2 closes when local statement-
  timeout provenance lands.

## P2 — close before C2 is marked complete

1. **Local statement-timeout provenance.** As named in the
   request, this is the remaining gap before C2 closes. The
   bridge for it should poll a different PostgreSQL flag
   (the local `STATEMENT_TIMEOUT` machinery surfaces a
   `QueryCancelHoldoff` or related signal depending on
   interrupt timing) and classify the resulting cancel as
   `local_statement_timeout` rather than `local_query_cancelled`,
   distinct from the remote-side `remote_statement_timeout`
   from packet 30738. The new `SpireRemoteLocalCancelSource`
   abstraction is the right place to add another production
   variant — same shape, different signal source.

## P3 — informational

2. **Symbol-lookup robustness.** The dynamic lookup approach is
   correct, but worth confirming the symbol-not-found path is
   safe (e.g. returns "no cancel pending" rather than treating
   absence as cancel). A fixture that explicitly verifies the
   no-symbol fallback would lock the safe-default invariant in
   tests rather than relying on the test environment happening
   to have the symbols. Optional — only worth doing if running
   the adapter outside pg_test ever becomes a real path.
