# Findings

## 1. `iter_live_tids()` is not actually a live-tuple iterator under the vacuum semantics this branch already chose

Right now `iter_live_tids()` means only `!tuple.deleted`.

That is too weak once Phase 8A exists. Vacuum pass 1 can produce a tuple
with:

- `primary_heaptid = INVALID`
- `deleted = false`
- `has_overflow_heaptids = false`

That tuple is not live for scan-entry or rerank purposes, but
`iter_live_tids()` will still yield it, and `first_live_tid()` will
still return it.

I would either:

1. redefine `iter_live_tids()` to mean scan-eligible / rerankable, or
2. rename the current method to `iter_non_deleted_tids()` and add a real
   `iter_live_tids()`

The live predicate should match the tuple state machine, not just the
deleted bit. At minimum for the current branch that means excluding:

- `deleted = true`
- `primary_heaptid = INVALID && !has_overflow_heaptids`

## 2. The new tests miss the most important vacuum intermediate state

RD-014 through RD-016 only create tombstones via `mark_deleted`. They do
not cover the pass-1-only stripped state. Add a test that uses
`strip_dead_primary_heaptid` without `mark_deleted` and proves both
`iter_live_tids()` and `first_live_tid()` skip that tuple.
