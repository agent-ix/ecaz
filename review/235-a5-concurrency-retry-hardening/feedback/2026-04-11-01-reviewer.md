# Feedback: A5 Concurrency Retry Hardening

**Reviewer:** Claude (Opus 4.6)
**Date:** 2026-04-11
**Commit reviewed:** `064829a`

## Scope match

Matches the packet. `BacklinkMutation` gains `target_element_tid`,
`apply_backlink_mutation` returns `BacklinkMutationOutcome::{NoChange,
Changed, RetryReplan}` instead of `bool`, the write-path split becomes
`plan_backlink_mutations → apply_backlink_mutations` looped
`MAX_BACKLINK_REPLAN_PASSES = 3` times, and replanning happens strictly
outside any data-page `EXCLUSIVE` lock. Two deterministic unit tests
cover the stale-drift → retry → replan arc against the inner mutation
helper. ADR-026 grows Decision #7 (the retry rule) and renumbers
metadata-last to #8. FR-016, the task tracker, and `plan/status.md` mark
A5 closed at 100%.

## Findings

### 1. Retry loop topology respects ADR-026

Tracing the new control flow:

```
add_backlinks_to_forward_neighbors
  for _ in 0..MAX_BACKLINK_REPLAN_PASSES:
    mutations = plan_backlink_mutations(index_relation, planner, pending)   // read-only, no page EXCL
    if mutations.is_empty(): break
    pending = apply_backlink_mutations(index_relation, &mutations, ...)     // per-page EXCL + WAL
    // pending now contains only RetryReplan targets
```

The crucial invariant is that `plan_backlink_mutations` runs entirely
outside the data-page exclusive lock scope. Inside
`apply_backlink_mutations → add_backlinks_on_page`, the page is held
`EXCLUSIVE` and the `RetryReplan` detection happens under the lock (it
has to — comparing `layer_slice` against `expected_slice` requires a
lock-protected read), but the *consequences* of that detection are just
pushing the target into a local `retries: Vec`, which is returned *after*
`UnlockReleaseBuffer`. Replanning then picks the vec up in the outer
loop, no lock held. That is exactly what ADR-026 Decision #7 requires
and it keeps the deadlock surface unchanged.

One subtle correctness detail worth naming: `retries.extend(...)` and
`sort_and_dedup_forward_selections(&mut retries)` happen in
`apply_backlink_mutations` *after* all per-page write loops finish, not
interleaved. So the retry set is always drained into a clean
post-ordered form before the next plan pass. Correct.

### 2. `target_element_tid` addition is the right data-flow fix

Before this slice, `BacklinkMutation` only carried `neighbor_tid` (the
on-disk tuple location). Re-planning needs to reload three things:
target element, its neighbor tuple, its layer slice. Carrying
`target_element_tid` directly on the mutation avoids a second lookup
through a reverse index from `neighbor_tid → element_tid` (which didn't
exist anyway). The retry path just rebuilds a `LayerForwardSelection {
layer, element_tid: mutation.target_element_tid }` and hands it back to
`plan_backlink_mutations`, which then does the full `load_graph_element
→ load_graph_neighbors → plan_backlink_mutation` dance on the fresh
state. Clean symmetry.

The extended sort comparator now has three keys: `(neighbor_tid, layer,
target_element_tid)`. The third key is defensive — in a healthy index,
`neighbor_tid → target_element_tid` is 1:1 (each element owns its
neighbor tuple), so the tertiary comparator is a no-op in practice. But
if a future refactor ever made them many-to-one (e.g., shared neighbor
pages), the sort would still be deterministic. Low-cost insurance.

### 3. Bounded 3-pass retry — correct limit but silently drops on exhaustion

`MAX_BACKLINK_REPLAN_PASSES = 3` is a magic constant with no comment
explaining the choice. The rationale is presumably "under normal
concurrent contention, 1 retry is enough; 2 handles adversarial races; 3
is insurance" — but that's a guess from the reviewer side. Add a
one-line comment: `// Three passes: one initial plan + up to two retries
against concurrently-drifted targets. Further drift falls out of the
retry loop and is silently dropped under the same contract as
InsertIfFree on a full slice.`

**Real concern:** if all three passes produce `RetryReplan` for the same
target (e.g., a hot neighbor under constant churn), the fourth iteration
would require replanning, but the loop bound stops it. The target's
backlink is then silently dropped — same failure mode as a full slice
that never gets pruned. This matches the packet's stated semantics
("retry is bounded"), but there is *no observability* for exhausted
retries. A pathological workload can lose backlinks systematically
without any signal visible in the admin snapshot or logs.

**Recommended follow-up (not a blocker):** add a counter or a `pgrx::warning!`
emission when the loop exits with `pending.is_empty() == false` after
three passes, and surface that count through
`tqhnsw_index_admin_snapshot` alongside `inserted_since_rebuild`. This
is the kind of metric that only matters under concurrent workloads, but
when it matters, it matters a lot — A6 vacuum repair would otherwise be
the only way to discover that a live insert session was silently
dropping edges. One line of logging + one admin-snapshot field; file on
an A6 adjacent slice.

### 4. Test coverage exercises the inner contract but not the outer loop

Both new tests are unit tests on `apply_backlink_mutation` directly:

- `rewrite_full_slice_requests_retry_when_snapshot_drifted`: builds a
  mutation whose `expected_slice = [(1,1), (1,2), (1,3), (1,4)]`, mutates
  the layer to `[(5,5), (1,2), (1,3), (1,4)]` (simulating a concurrent
  drift), and asserts the outcome is `RetryReplan` *and* the layer is
  left untouched. The "left untouched" assertion is the load-bearing one
  — it proves that the stale-detection step happens *before*
  `copy_from_slice` runs, not after. Without it, a regression that
  reorders the drift check could silently corrupt the layer. Strong.

- `rewrite_full_slice_applies_after_replanning_against_current_slice`:
  builds a mutation whose `expected_slice` matches the drifted layer
  (i.e., the replanner already re-read the layer), and asserts the
  mutation now returns `Changed` and the layer is rewritten. This is
  the positive case for a *replanned* mutation after the retry.

Together they cover:
- stale plan → detected → retries + layer preserved ✓
- fresh plan on drifted layer → applies correctly ✓

**What they don't cover:** the full outer loop in
`add_backlinks_to_forward_neighbors`. A test that constructs a real
index fixture, runs two concurrent insert sessions, and asserts the
*second* session's insert observes drift and retries is not here. The
packet explicitly says "without introducing extra cross-session harness
machinery into the checkpoint," and I think that's the right call for
this slice — cross-session pgrx testing has its own cost and the unit
tests prove the mutation-level semantics. But file on the A6 kickoff: once
vacuum repair adds an independent concurrency story, the
cross-session harness becomes useful for both lanes simultaneously,
and this slice's retry-loop correctness should be retrospectively
validated against that fixture.

### 5. The concurrency contention concern from 234 is NOT resolved here

234's always-take-metadata-EXCL change made every successful live
insert serialize on the metadata page. I flagged that as a named
concurrency concern for 235 to own. 235 does not address it — the
metadata lock is still unconditionally taken in the closure at
`src/am/insert.rs:163`, and no insert-latency benchmarks under
concurrent load are included.

That's not a defect of *this* slice's stated scope (the packet describes
retry hardening for stale full-slice plans, not metadata-page contention
reduction), but it is a discrepancy with the slice's *name*. "A5
Concurrency Retry Hardening" reads as "this slice is the one that makes
concurrent insert work end-to-end," but what it actually delivers is
"stale full-slice plans retry correctly under *this process*'s
optimistic assumption of a concurrent writer." No other session has to
exist for `RetryReplan` to be triggered — it only needs to believe one
did.

**Required for closing A5 for real:** either

a. Run a multi-session insert benchmark (2+ concurrent `INSERT`s, same
   index, ≥100 rows each) with both contention-sensitive latency
   collection and deadlock timeout assertion, and attach the numbers to
   the A5 closeout ADR. The point is not to set a NFR-001 gate; it's to
   prove empirically that the metadata-always-EXCL change doesn't
   reintroduce a serialization cliff at small session counts. Or,

b. Document explicitly in FR-016-AC-3 or the A5 closeout that the
   concurrency story in this slice is bounded to "single writer, stale
   plan retry," and that multi-writer contention/deadlock validation is
   deferred to an explicit future slice.

The FR-016 spec update here takes option (b) implicitly — the deferred
concern just isn't named anywhere. I'd be more comfortable if it were
named. One paragraph in FR-016-AC-3 stating "multi-writer
deadlock-focused validation is out of scope for A5 and scheduled as a
post-A6 runtime slice once vacuum concurrency is also in play" would
close the gap.

### 6. ADR-026 Decision #7 is the right durable rule

The new Decision #7:

> If a full-slice rewrite sees a stale live layer under the page lock,
> do not replan while that page is locked. Record the target for retry,
> finish the current ordered page pass, then re-enter read-only planning
> for those targets before another ordered write pass begins.

This is exactly the rule that keeps the retry loop compatible with the
rest of the lock ordering. I'd sharpen one thing: add a sentence saying
*what data* the write-path is allowed to capture for retry without
extending the lock scope. The current implementation captures exactly
`(target_element_tid, layer)` into a local `Vec`, which is a 48-bit TID
+ a u8 — trivially copyable and read-only from the caller's perspective.
If a future author tries to capture a decoded neighbor tuple or a scored
candidate list "for efficiency," that would effectively hold
page-derived data past `UnlockReleaseBuffer`, and the retry plan would
then be running against stale bytes from the prior write pass. One
sentence: "The retry payload may capture target TID and layer only; no
page-derived or lock-protected data may escape the write phase into the
replan phase."

## Verdict

**Approve.** The retry loop is correctly structured around ADR-026, the
inner-mutation semantics are proven by the two new unit tests, and the
deferred-work documentation is now honest about what this slice
delivered versus what's still open. The load-bearing architectural win
is that `plan_backlink_mutations` runs entirely outside any data-page
`EXCLUSIVE` lock scope, which lets the write path detect drift without
being able to *resolve* it in-place — and that separation is what makes
bounded retry compatible with the lock ordering.

Follow-ups, none blocking A5 closure:

1. **Item 3 (retry exhaustion observability):** add a counter +
   `pgrx::warning!` when the retry loop exits with unresolved pending
   targets, and expose the count through `tqhnsw_index_admin_snapshot`.
   File on the A6 kickoff or a A5-adjacent cleanup slice.
2. **Item 3 (constant comment):** one-line comment on
   `MAX_BACKLINK_REPLAN_PASSES = 3` explaining the choice.
3. **Item 4 (cross-session harness):** cross-session retry-loop test
   should happen once A6 concurrency harness lands, not now.
4. **Item 5 (contention from 234):** either run a small concurrent-insert
   latency benchmark and attach to A5 closeout, or add a one-paragraph
   deferred-scope note in FR-016-AC-3 saying multi-writer deadlock
   validation is explicitly out of scope for A5.
5. **Item 6 (ADR-026 retry payload rule):** one sentence on Decision #7
   restricting the retry payload to `(target_element_tid, layer)` only,
   no page-derived data.

Regarding item 5 in particular: the concern is real but the packet is
a defensible close of A5 as written. Just don't let "A5 closed" become
"concurrent multi-writer insert is validated" in anyone's memory — they
are different assertions and A5 only makes the first one.
