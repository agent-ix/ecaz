## Findings

### 1. Raw AtomicU32 spinlock is the wrong long-term primitive

**Severity:** Medium (blocker for `amcanparallel = true`, not today)
**File:** `src/am/common/parallel.rs:910–922`

```rust
while lock
    .compare_exchange(0, 1, Ordering::AcqRel, Ordering::Acquire)
    .is_err()
{
    std::hint::spin_loop();
}
```

Concrete risks the moment real parallel workers land:

- **No holder-death cleanup.** If a worker is killed (SIGTERM from
  `ProcessInterrupts`, OOM, extension-layer panic mid-critical-section)
  while holding the lock, the word stays at 1 and every other backend
  spins forever. Postgres' LWLock infrastructure releases locks
  automatically on backend exit via the PGPROC cleanup path; a raw
  word does not.
- **No CHECK_FOR_INTERRUPTS.** The spin loop does not poll for
  SIGTERM/query cancel, so a stuck lock is uncancellable. Even
  short-lived contention will interfere with
  `pg_cancel_backend`/`pg_terminate_backend`.
- **No queueing / no fairness.** Fine for low contention; starvation
  is possible under heavy worker churn on the same scan.
- **No tracing / pg_stat_activity wait-event integration.** An
  LWLock on a custom tranche would show up as a wait event; this
  spinlock is invisible to observability.

Recommendation (do this before `amcanparallel = true`): switch to
`LWLockAcquire` / `LWLockRelease` on a tranche registered via
`LWLockRegisterTranche` (or use `RequestNamedLWLockTranche` from a
`shmem_request_hook`). Postgres spinlocks (`slock_t` /
`SpinLockAcquire`) are also an option but have similar no-cleanup
properties and are meant only for truly brief windows — `upsert` +
`sift_up` + `sift_down` + fast-path refresh is too much work to hold
a PG spinlock.

If an LWLock is too heavy-handed for today's staged path, at minimum
add (a) a bounded spin budget that falls back to `WaitEventSet` or
yields to Postgres, and (b) `CHECK_FOR_INTERRUPTS()` inside the
spin loop.

### 2. Guard `Drop` can release even if the critical section panicked

**Severity:** Informational
**File:** `src/am/common/parallel.rs:237–243`

`ParallelScanHeapLockGuard::drop` unconditionally stores 0. That's
what you want for lock release, but it means a panic mid-mutation
leaks partial heap state to the next lock holder without any marker
that "the previous holder died dirty". If we had an LWLock this
would be the same, so not a regression — flagging because several
operations under the lock leave transient invariant-broken states
(e.g. mid-sift reverse-membership drift from packet 480).

No action today. When the LWLock lands, decide explicitly whether a
panic should (a) mark the DSM state as poisoned and force a rescan
reset, or (b) trust the remaining invariant-restoring code to self-
heal on the next full refresh.

### 3. `take_...` acquires the lock fresh each retry iteration

**Severity:** Informational (positive)
**File:** `src/am/common/parallel.rs:1667–1708`

The retry loop takes the heap lock per iteration and drops it between
iterations. That's the right shape for a two-pass retry: the first
pass may observe a stale coordinator, the second pass refreshes under
the lock, and by the next iteration contention can make progress.
Holding the lock across both iterations would serialize unrelated
workers. Good.

### 4. `reap_parallel_scan_dead_root_slots_with_attachment` removed
from `refresh_coordinator_selected_result_fast_path_locked`

**Severity:** Informational
**File:** `src/am/common/parallel.rs:1043–1072`

The dead-root reap was hoisted out of the fast-path refresh and is
now called explicitly by each caller before the fast-path refresh
(`refresh_coordinator_selection_snapshot_locked` at line 1117,
`publish_...` at line 1538, `take_...` at line 1697). That's one
more thing callers need to remember, and it's not obvious from the
fast-path function name that it no longer self-heals dead roots.

Two-line doc comment on `refresh_coordinator_selected_result_fast_path_locked`
saying "does not reap; caller must run `reap_parallel_scan_dead_root_slots_with_attachment`
first if dead-root reaping is required" would pay for itself quickly.

## Answers to the packet's review questions

- **Q1** (shared lock-word the right serializer before admission API?)
  — Shape is right, mechanism needs to become an LWLock (or at
  minimum gain interrupt-aware fallback). Don't ship `amcanparallel`
  on a raw AtomicU32 spinlock.
- **Q2** (lock-free first read + locked stale-refresh retry the right
  contract for selected-result reads?) — Yes. Classic optimistic
  read pattern; the two-pass retry bounds worst-case cost.
- **Q3** (public entry points cover every heap-entry mutation?) —
  Spot-checked: publish, clear, take, full refresh, heap-snapshot
  read, best-slot selection, and stale-refresh in direct read all
  take the lock. `swap_heap_entries`, sift-up/sift-down, detach,
  upsert, pop, and rebuild are all called only from already-locked
  contexts. Coverage looks complete.
