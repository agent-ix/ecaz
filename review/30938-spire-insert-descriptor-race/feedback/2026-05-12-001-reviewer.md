## Summary

Closes 30836 P2 / 30896 H1 with the actual parallel-INSERT
race fixture I asked for. The 30927 packet pinned SQLSTATE
`40001` for the descriptor-refresh race; this packet swaps the
old `Err(...)` return for a proper `pgrx::ereport!(ERROR,
ERRCODE_T_R_SERIALIZATION_FAILURE, ...)` raise with the

The fixture (`test_ec_spire_insert_descriptor_race_sql`):

## Findings

### P1 — `pgrx::ereport!(ERROR, ...)` instead of `Err(_)` return

The `if !descriptor_refreshed` path now raises directly with
SQLSTATE `40001` instead of returning `Err`. That's the right
shape — `ereport!(ERROR, ...)` triggers PG's normal error
flow, the calling transaction aborts, the registered Abort
xact callback fires `ROLLBACK PREPARED` on the remote, and
the application-side error has the correct SQLSTATE for
auto-retry tooling.

A subtle correctness benefit: returning `Err(_)` from
`Spi::connect_mut` would have unwound through Rust error
handling that may not consistently trigger PG's transaction
abort cleanup. Direct `ereport!(ERROR, ...)` is the
PG-native pattern.

### P2 — Helper `spire_insert_prepared_count` searches by GID prefix

Uses `gid LIKE 'ec_spire_insert_<oid>_<node>_%'` to count
SPIRE prepared xacts for a specific descriptor. Robust against
the GID format change (30914 dropped the pid suffix); the
remaining suffix `<served_epoch>_<top_xid>` doesn't collide
across descriptors because the prefix already pins
`(index_oid, node_id)`.

Worth a one-line comment naming why the LIKE pattern is safe
(both segments after the underscore-prefix are coordinator-
local identifiers, not user-supplied).

### P2 — Race fixture as the H1 closeout proof

Combined with 30927 (SQLSTATE pin), this fixture is the
load-bearing evidence H1 has actually shipped. The rest of
the feedback chain (30836 → 30927 → 30938) finally has all
three pieces:
- Stable SQLSTATE callers can catch on (30927).
- Direct `ereport!` raise that triggers proper xact abort
  (30938).
- Real fixture proving the loser rolls back its remote
  prepared xact under contention (30938).

### P3 — Test setup uses 30930's env-var helper

`tests.ec_spire_test_set_env_var(SECRET_KEY, conninfo)`
propagates the conninfo secret to external backends. Pattern
already established in 30928; same constraints apply
(test-only, gated behind `pg_test`).

### P3 — Cleanup hook in test setup

The fixture's setup `DO $$ BEGIN ... DELETE FROM
ec_spire_placement WHERE index_oid = idx; DELETE FROM
ec_spire_remote_node_descriptor WHERE coordinator_index_oid =
idx; END $$;` ensures clean state for re-runs. Important for
race fixtures that may leave partial state across runs.
