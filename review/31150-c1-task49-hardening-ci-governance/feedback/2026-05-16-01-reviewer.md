# Review: Task 49 hardening CI governance (910)

## Summary

Retired synthetic harnesses (rudra/flux/loom/shuttle) are removed from
git, `hardening-nightly-local` no longer aggregates them, `make test`
is back to plain `cargo test`, and `test-hardening-local` is preserved
behind `make test-local` for the macOS pgrx loader workaround.
`scripts/hardening_validate.sh` enforces three rules: retired lane
dirs have no `Cargo.toml`/`src/lib.rs`, retained hardening crates
import real `src/` via `#[path = "../../../src/"]`, and `make test`
is wired to `cargo test`. Retained crates (`hardening/careful`,
`hardening/kani`) both still import real `src/storage/page.rs` and
their tests pass (artifacts confirm 8 + 331 unit tests green in
`test-local.log`). Governance doc and tiers report match the live
Makefile inventory.

I am satisfied with the direction; the findings below are tightenings
on the guard surface rather than blockers.

## Findings

### 1. `hardening_validate.sh` doesn't gate the `hardening-local` aggregate

`scripts/hardening_validate.sh:30` only matches
`^hardening-nightly-local:.*\b(loom|shuttle|flux|rudra)\b`. The
`hardening-local` aggregate (`Makefile:262`) is the equally exposed
target for someone reintroducing a retired lane ŌĆö the script will not
catch `hardening-local: ... loom ...`. Suggest extending the regex
(or factoring the retired-lane list and looping over both aggregates)
so both aggregate-level reintroductions trip the gate.

### 2. `make test` recipe match is brittle

`scripts/hardening_validate.sh:35-43` requires the literal indented
line `cargo test$`. Any reasonable evolution ŌĆö `cargo test
--workspace`, `RUST_LOG=info cargo test`, splitting into two `cargo
test` lines for `-p`-targeted runs ŌĆö fails the gate even though the
intent is preserved. Consider matching `cargo test\b` (allow trailing
args) and a `! ...test-hardening-local` negative check, so the
"silently weakened to local subset" regression is what fails rather
than any expansion of the recipe.

### 3. Retired lane directories still exist on disk

`hardening/rudra`, `hardening/flux`, `hardening/loom`, and
`hardening/shuttle` still exist as filesystem directories ŌĆö empty
`src/` for rudra, and `Cargo.lock` + `target/` leftovers for the
other three (all untracked, so absent from a fresh clone). Two small
options:

- `git rm -r` the empty dirs in this same packet (and ensure
  `hardening/<retired>/` is in `.gitignore` so locally-cached
  `Cargo.lock` doesn't reappear), or
- broaden the validate check to also fail if any file under
  `hardening/{rudra,flux,loom,shuttle}/` is git-tracked (`git
  ls-files`), not just `Cargo.toml`/`src/lib.rs`.

Either is fine; leaving the working-tree leftovers as-is is also
acceptable since `git ls-files hardening/` confirms only `careful`
and `kani` remain tracked.

### 4. Minor: PR-tier budget claim vs. `test` lane budget

`docs/hardening-governance.md` says PR-tier lanes must be `<5 min`
wall clock. `make hardening-tiers-report` lists `test` with budget
`variable`. The intent is presumably "PR lanes are <5 min except
`cargo test`, which can grow as the suite grows" ŌĆö but a future
reader sees a contradiction. Either tighten `test`'s budget claim or
add a one-line carveout in `hardening-governance.md` ("the `test`
lane is the single exception, budgeted by CI wall clock rather than
the 5-minute PR rule").
