# Feedback: Consolidated PG18 Diagnostics Snapshot

**Reviewer:** Claude Opus 4.6
**Date:** 2026-04-07

## Findings

### 1. Consolidation is the right instinct but applied too late

This snapshot consolidates EXPLAIN and stats scaffolding into one PG18 diagnostics view. That's the right instinct — but it means the separate EXPLAIN snapshot and stats snapshot were premature. The consolidation should have been the first move, not a layer on top.

### 2. Layer cake of snapshot functions is becoming unwieldy

The planner branch now has: admin snapshot, explain snapshot, cost snapshot, stats snapshot, PG18 upgrade snapshot, PG18 diagnostics snapshot, and planner integration blockers snapshot. That's **seven** SQL functions, most of which return static `false` flags. This is surface-area debt that will need cleanup.

**Recommendation:** For D1, you need at most three SQL functions:
1. `tqhnsw_index_admin_snapshot(regclass)` — operational: ef_search, tuning source, node count, block count
2. `tqhnsw_index_cost_snapshot(regclass)` — cost model: modeled vs gated costs, inputs
3. `tqhnsw_planner_readiness()` — consolidated: all readiness flags, blocker strings, PG18 status

Everything else is premature surface area.

## Net

The consolidation idea is right. Apply it more aggressively — collapse the seven functions into three.
