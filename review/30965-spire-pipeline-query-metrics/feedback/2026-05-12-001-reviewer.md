# Review: SPIRE Pipeline Query Metrics (30965)

## Summary

Phase 12.9 bench harness extension. Adds to
`ecaz bench spire-pipeline`:

- `--include-query-metrics` — runs coordinator KNN SQL per
  sampled query, reports latency min/p50/p95/p99/max per
  `nprobe` sweep value;
- `--include-recall` — loads exact local truth, reports
  recall@k in the query-metrics section;
- `--query-metric-k` — independent of remote diagnostic
  `--top-k`.

CLI/parser coverage; no live PostgreSQL run. Tracker update
remains a sub-bullet under the still-open Phase 12.9
benchmark row.

## Findings

### P2 — One CLI entry point for readiness capture

Keeping query metrics + recall + routing/local/remote
counters + `--remote-tuple-transport` (30964) all in
`ecaz bench spire-pipeline` is the right primitive. Future
packet-local readiness runs collect everything from one
command and one artifact directory. Splitting across
multiple scripts would force operators to coordinate
timing/parameters by hand, which is the failure mode this
phase is supposed to prevent.

### P2 — `--query-metric-k` independent from `--top-k`

Independent k controls is the right call. The query metric
k (what the SQL `LIMIT` is) and the remote diagnostic top-k
(what the remote pipeline reports) measure different things:

- `--query-metric-k`: latency/recall at the SQL boundary
  (what end users see).
- `--top-k`: candidate count from remote pipeline (what the
  fanout produces).

Conflating them would force operators to either over-fetch
on the SQL side or under-instrument on the diagnostic side.
Independent is honest.

### P2 — Distinct report sections

Routing/local/remote counters and query metrics in
**distinct sections** is the right rendering choice. They
answer different questions:

- counters: "what did the pipeline do?"
- query metrics: "what did the user experience?"

Operators triaging a regression need to see both without
having to mentally separate them from one column-mix. Good.

### P2 — Recall@k via local truth

Loading exact local truth for the sampled queries and
reporting recall@k is the right correctness anchor.
Without it, latency improvements could mask correctness
regressions (a faster path that returns wrong answers
"improves" latency by skipping work). Recall@k catches
that.

Worth pinning in the doc that **recall@k is local-truth-
only** — it does not validate remote-origin correctness
beyond what the SQL returns. For multi-instance correctness
testing, the Stage E lifecycle matrix from 30961 is the
authoritative oracle.

### P3 — Tracker scope as sub-bullet

Correctly remains a sub-bullet under the still-open Phase
12.9 benchmark row. Does NOT close the parent row. The
parent closes when a packet runs the harness and captures
packet-local readiness evidence. Honest layering.

### P3 — Latency percentile selection

min/p50/p95/p99/max is the right default percentile set.
Worth one sentence noting the absence of p99.9 — for the
local production-readiness smoke target profile (30952's
1-concurrent-session limit), p99 is sufficient sample
density. AWS scale (Phase 13) may want to add p99.9, but
that's a Phase 13 follow-up.

### P3 — Sampled-query reproducibility

If the `--include-query-metrics` path samples queries from
a corpus, the seed/sample-size knobs need to be recorded
in the artifact header for reproducibility. From the
request the existing harness handles corpus selection;
worth confirming the existing seed plumbing carries
through to the new query metrics output.

## Reviewer recommendation

Accept. Right harness extension: one entry point,
distinct sections, independent k controls, recall anchor.
Sub-bullet tracker placement is honest about what's
infrastructure vs what's evidence.
