# Review: Packet 268 — C1 Cached-Plan Query Timing

Reviewer: agent-ix (via code-review)
Date: 2026-04-12

## Assessment: Clean negative result — closes the last launcher-side hypothesis

### What this packet did right

1. **Systematically eliminated the planning overhead hypothesis.** After
   packet 267 ruled out EXPLAIN instrumentation, the natural follow-up is
   "does fresh statement parsing/planning per query add measurable cost?"
   The answer is no (cached-plan: `11.041ms` vs plain-server: `11.020ms`).
   This is a ~0.02ms difference — deep in the noise.

2. **The per-cell restriction is correct.** Cached-plan mode only makes sense
   with session reuse, since the cached plan is a session-local prepared
   statement. Requiring `--session-mode per-cell` prevents a nonsensical
   per-query cached-plan configuration that would prepare + discard a plan
   each time.

3. **The plpgsql helper is a good approach.** Using `pg_temp.tqv_latency_cached_plan`
   to wrap the ordered query in a function that reuses the planned statement
   across the cell is the idiomatic PostgreSQL way to test plan reuse. The
   `clock_timestamp()` timing inside the function body measures only execution,
   not the one-time plan cost.

### The three-way comparison is now the definitive warm-surface characterization

```
explain:      mean=11.111ms
plain-server: mean=11.020ms
cached-plan:  mean=11.041ms
```

All three modes agree within ~0.1ms. This means:

- **EXPLAIN overhead: ~0.09ms** (negligible)
- **Per-statement planning overhead: ~0.02ms** (negligible)
- **The entire 11ms is AM execution time.** Parse, plan, and executor
  instrumentation together account for less than 1% of the warm surface.

This is the strongest possible evidence that the remaining C1 gap is inside
the ordered scan code path itself. Every future optimization must target
`amrescan` / graph loading / scoring / beam bookkeeping.

### Diminishing returns on launcher-side investigation

Packets 261 (warm seam), 267 (plain timing), and 268 (cached plan) together
represent three consecutive launcher-focused slices. Each produced useful
infrastructure but no latency improvement. The packet correctly concludes
that the next work should return to executor / AM hot-path changes.

## Suggestions

1. **Merge the timing modes.** All three modes are useful as diagnostic tools,
   even though cached-plan didn't move the needle. Having them available in
   the harness costs nothing and aids future investigation.

2. **Close the launcher investigation thread.** This packet, combined with
   267, definitively answers "is the measurement harness part of the
   bottleneck?" with "no." Future packets should not revisit this question
   unless the harness or PG version changes.

3. **Next slice should target allocation overhead.** With the scorer near-
   optimal (packet 266) and the harness eliminated (packets 267-268), the
   remaining 11ms surface is dominated by per-element allocation/decode
   overhead during graph loading. Packet 264's ideas #4 (inline heaptids)
   and #5 (scratch buffers) are the right next targets — they address the
   ~4% `Vec::extend_from_slice` cost and the ~120+ per-query allocations
   that accumulate across beam search expansions.
