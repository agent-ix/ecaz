## TL;DR for the agent

The eight in-scope files read cleanly against the native builder: FR-008
and FR-021 are now accurate, the parallel-build text correctly says
"not yet" without overcommitting, and the architecture snapshot matches
the current module layout. But the packet stopped short of the ADR
record itself — ADR-042 is still `PROPOSED` despite being shipped, and
ADR-002 / ADR-003 (which specifically codify the retired `hnsw_rs`
build path) are still marked `DECIDED`. A reader reaching for the ADR
index comes away with a different answer than the spec text gives them.

## Findings

### 1. ADR status record lags the shipped code

**Severity:** Medium
**Files:**
- `spec/adr/ADR-042-native-hnsw-build-path.md` — still `status: PROPOSED`
- `spec/adr/ADR-002-hnsw-rs-no-delete.md` — still `status: DECIDED`
- `spec/adr/ADR-003-hnsw-rs-serialization.md` — still `status: DECIDED`

ADR-042 lists its own impact as "Affects FR-008 (HNSW build), FR-016
(HNSW insert), FR-021 (parallel build), ADR-002, ADR-003, ADR-030."
The packet updated FR-008 and FR-021 to reflect ADR-042's outcome — so
the decision is visibly accepted in the downstream spec — but the ADR
itself still says PROPOSED. ADR-002 / ADR-003 have not been marked
SUPERSEDED BY ADR-042. This is exactly the state this packet's
problem statement describes ("docs lagging the actual code and task
state"), just at the ADR layer instead of the FR layer.

Fix: flip ADR-042 to `DECIDED` (or whatever the repo's accepted terminal
state is — one sweep of `spec/adr/*.md` for the convention will answer),
and add `Superseded by: ADR-042` headers to ADR-002 and ADR-003 with a
one-sentence note. No content edits beyond the status block.

### 2. FR-009 / FR-016 reference `hnsw_rs` only to negate it

**Severity:** Low
**Files:**
- `spec/functional/FR-009-hnsw-scan.md:18` — "… All scan operations work
  directly on Postgres buffer pages — no `hnsw_rs` involvement."
- `spec/functional/FR-016-hnsw-insert.md:22` — "… This does NOT use
  `hnsw_rs` — it operates directly on Postgres buffer pages…"

These aren't in the packet's scope, but the packet's own review focus
asks: "Whether any still-live doc/spec surface is still implying
hnsw_rs remains part of the production build path." These two lines do
the opposite — they *negate* hnsw_rs involvement — which is correct, but
they still anchor the reader to a name that's no longer part of the
system. Once ADR-002/003 are marked superseded (finding #1), drop the
"no hnsw_rs" clauses entirely; they become a reference to a named thing
the new reader has no context for.

Not a merge blocker for this packet; optional cleanup in the same
follow-up that addresses #1.

### 3. FR-021 parallel-build framing is correctly scoped

**Severity:** Informational (positive)
**File:** `spec/functional/FR-021-parallel-build.md`

Worth calling out: the rewrite says parallel build is *off today* and
frames future work as "parallelize Ecaz's own builder" rather than
"replace hnsw_rs". That's the right shape — avoids the temptation to
treat parallelization as a crate swap. No changes requested.
