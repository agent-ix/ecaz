# Review: Task 41 HNSW scan_debug index guards (31181)

## Summary

First migration into the largest unsafe-heavy file in the codebase
(`src/am/ec_hnsw/scan_debug.rs`, 478 entries pre-packet). Covers
`debug_begin_end_scan` through `debug_gettuple_after_rescan_result`
ÔÇö ~10 helpers that open an index, run scan lifecycle calls
(`ec_hnsw_ambeginscan` Ôćĺ `ec_hnsw_amrescan` Ôćĺ `ec_hnsw_amendscan` Ôćĺ
`pg_sys::IndexScanEnd`), and previously closed the index manually
at the end. Uses the consolidated `IndexRelationGuard::access_share`
from 31180 throughout. Baseline 4,301 Ôćĺ 4,284 (Ôłĺ17); `scan_debug.rs`
478 Ôćĺ 461.

This is also the first **post-consolidation** migration that
exercises the new primitive in a real packet. The shape works:
import the primitive, call `access_share(oid, "caller_name")`, use
`as_ptr()` at the AM-call boundary, let Drop close at function
return. No new module-local guard struct was introduced.

## Confirmations

- **Guard lives across scan lifecycle.** Each helper opens the
  guard once at the top, then `ec_hnsw_ambeginscan(guard.as_ptr(),
  ...)` returns a scan descriptor that internally references the
  relation. The guard stays alive through `amrescan`, the
  test-specific reads (query state, opaque pointer checks, etc.),
  `amendscan`, `IndexScanEnd`, and only drops at function return
  ÔÇö same effective lifetime as the prior explicit `index_close`.
  Ôťö
- **Error-path helpers correctly rely on Drop.**
  `debug_rescan_null_query` calls `ec_hnsw_amrescan` with a
  `SK_ISNULL` orderby, which the AM rejects via `pgrx::error!`. No
  explicit close was present before (the function intentionally
  panics); now the guard's `Drop` closes the relation as the panic
  unwinds. Same behavior, but now we no longer rely on
  "this function panics so the close doesn't matter" ÔÇö Drop runs
  regardless. Strictly safer. Ôťö
  
  Same for `debug_rescan_with_index_qual` and the other
  intentionally-panicking rescan helpers.
- **Caller-name argument used correctly.** Each call passes a
  unique `caller` string (`"debug_begin_end_scan"`,
  `"debug_rescan_query_dimensions"`, etc.) ÔÇö if a null open ever
  panicked, the message would name the test that triggered it.
  Good practice.
- **Scope is intentionally narrow.** Packet covers ~10 helpers;
  the request explicitly lists "heap-backed scan, active snapshot,
  tuple slot, and later scan-debug helper lifetimes" as follow-up
  slices. `scan_debug.rs` is 478 entries; this slice retires 17.
  More slices are coming.
- **Baseline arithmetic.** Ôłĺ17 across 10 helpers Ôëł ~1.7 entries
  retired per helper (open + close + occasional intermediate
  SAFETY comments). Consistent.
