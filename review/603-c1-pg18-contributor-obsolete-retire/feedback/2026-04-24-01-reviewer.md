# Review: PG18 Contributor Obsolete Output Retirement

**Verdict: Approved**

## What was read

- `git show 009d95b` full diff (scan.rs)
- `retire_obsolete_non_emitting_parallel_contributor_output`, call site in `contribute_non_emitting_parallel_backend`

## Summary

The contributor loop previously polled on a published row until the emitter drained it or the drain-poll budget expired. If the row's heap TID was already emitted (via shared coordinator, foreign worker snapshot, or local emission), the contributor would keep polling uselessly. The new retirement function detects this condition and calls `discard_recently_emitted_active_parallel_output`, freeing the contributor to advance to the next row.

## Findings

**Retirement check order.** `active_parallel_output_was_emitted_by_shared_coordinator` → `active_parallel_output_was_recently_emitted_by_foreign_worker` → `active_parallel_output_was_recently_emitted_locally`. These are the same three checks used in `try_take_parallel_scan_next_output` for suppression; the retirement function reuses the same detection surface. Consistent.

**`discard_recently_emitted_active_parallel_output` vs `discard_active_parallel_scan_output`.** The retirement function uses the `recently_emitted` variant, which bumps the emitted-element local memory alongside discarding the output. This matches the behavior in the suppression path for foreign-emitted rows.

**Retirement inserted before drain-polls increment.** In `contribute_non_emitting_parallel_backend`, the retirement check runs before `drain_polls += 1`, resets `drain_polls = 0` on success, and continues. So an obsolete row is retired immediately and counts as progress rather than as a poll. Correct.

**Regression test.** `retire_obsolete_non_emitting_parallel_contributor_output_skips_shared_emitted_heap_tids` covers the full lifecycle: stage hidden output → record shared emitted TID → retire → verify the hidden slot advances to the next pending TID → repeat for the final TID → verify `parallel_local_only_output_active` is cleared when no pending TIDs remain. The test exercises the pending-TID iterator advancement inside the retire path.

## No blockers.
