# Feedback: Parallel Scan Stale-Slot Reap

**Reviewer:** Claude (Opus 4.7)
**Date:** 2026-04-20
**Head reviewed:** `01fb487` (landed; reviewed retrospectively)

## TL;DR for the agent

**Merged correctly.** Reaping dead staged slots at refresh time (not
just skipping them) keeps `published_result_slots` truthful. The
counter decrement + generation advance pair is the right bookkeeping
— consumers relying on the counter for "how much staged work is
pending?" now see an accurate number.

## Verification performed

- `reap_dead_parallel_scan_result_slots_with_attachment`
  (`src/am/common/parallel.rs:738–772`) walks all result slots, resets
  any whose owning worker claim is no longer live, decrements
  `published_result_slots` by the reaped count, and bumps
  `result_publish_generation`. Correct order: reset slot runtime →
  clear the slot flags → update counters.
- `refresh_coordinator_selection_snapshot` (`parallel.rs:774–876`)
  calls the reap before the heap rebuild — so the rebuilt heap never
  contains entries pointing at reaped slots. Correct ordering.
- Counter semantics verified in `take_...clears_fast_path_when_claim_drops`
  (`parallel.rs:2605–2617`): after a claim-drop + take retry,
  `published_result_slots = 0` and `result_publish_generation = 2`
  (1 for publish, 1 for reap). Exactly right.

## Findings

### 1. Single fetch_sub for batched decrement is safe

**Severity:** Informational (positive)
**File:** `src/am/common/parallel.rs:761–768`

`published_result_slots.fetch_sub(reaped, AcqRel)` does one atomic
sub rather than N individual ones. Correct and cheaper. If concurrent
writers ever land (post-amcanparallel), this still behaves because
fetch_sub is atomic; the concern there would be the per-slot reset
operations, not the counter update.

### 2. Generation advance on reap keeps readers unblocked

**Severity:** Informational (positive)

Incrementing `result_publish_generation` on reap (line 767–768) means
any reader using generation-based staleness detection will see the
reap and invalidate its cache. Right primitive.

### 3. Reap narrowness is scoped correctly

**Severity:** Informational
**File:** `src/am/common/parallel.rs:747–750`

The reap ignores slots whose `observed_rescan_epoch != attachment.rescan_epoch`
rather than reaping them — stale-epoch slots get reset at rescan
layout time, not at refresh time. Good separation of concerns.
