### 1. `src/tests/` per-file cap policy

`src/tests/remote_search.rs` (2,634 lines) and
`src/tests/dml_frontdoor.rs` (2,562 lines) already pierce the
2,500-line cap, and more concern fixtures are queued for both. The
exit-criteria cap in the task is phrased for `src/am/ec_spire/`; it is
ambiguous whether the same cap applies under `src/tests/`.

**Decision needed before more 12b.2 packets land:** state explicitly
whether the 2,500-line cap applies to `src/tests/`. If yes, schedule
sub-splits (`remote_search/contracts.rs`,
`remote_search/tuple_payload.rs`, etc.) into 12b.2. If no, document the
carve-out so a future reader does not flag it.

### 2. `src/lib.rs` "<2,000 lines" target

The task's exit criterion (`task30-phase12b-spire-cleanup.md:209`)
says `src/lib.rs` should land "under 2,000 lines." Current residual is
17,812: `bench_api` re-exports, `ecvector`/`tqvector` pg_extern
wrappers, `unit_tests`, and the `pg_test` scaffold. These belong in
`lib.rs` by `pgrx` convention; the target as stated is unreachable
without further sub-module moves.

**Decision needed:** either revise the target to something achievable
(e.g. "fixture bodies absent; `lib.rs` may retain pgrx registration,
`bench_api` re-exports, and `pg_test` scaffold"), or open a sub-row in
12b.2 to move `bench_api` and the pg_extern scoring wrappers into
their own modules so the original target is reached. Whichever — the
tracker should not silently miss its own number.

### 3. `tuple_transport_status: "ready"` is a static literal

`src/am/ec_spire/custom_scan/explain.rs:33`. Functionally correct for
12b.3 per the request packet (the request explicitly states this is
a minimal stable shape, not a live state probe). Flagging only so a
Phase 13 reader does not mistake the EXPLAIN field for a live runtime
diagnostic. Optional: a one-line code comment naming the constraint
would future-proof the misread.
