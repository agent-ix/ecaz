# Review: SPIRE DML Frontdoor Classify Diagnostic (30854)

## Summary

`ec_spire_dml_frontdoor_classify_sql(sql text)` — operator-facing
SQL diagnostic that:
- Parses + analyzes one SQL statement (rejects multi-statement,
  empty parse, multiple analyzed queries).
- Pulls the target heap relation OID via 30851's helper.
- Loads the SPI-backed relation context.
- Builds a `SpireDmlFrontdoorQueryContext` and invokes
  `classify_dml_frontdoor_query`.
- Returns `(target_relation_oid, relation_status, supported,
  operation, kind, status, error, hint, next_step)`.

Two graceful-degradation paths:
- No target relation: returns `unsupported_target_relation` with a
  clear "rewrite as single-table" message.
- Classifier returns `None` (non-DML command): returns
  `unsupported_operation`.

The packet correctly keeps the production planner hook
pass-through — the SPI-backed relation-context lookup is
re-entrant unsafe inside `planner_hook`, so the diagnostic is the
right vehicle for end-to-end shape probing until 30855's catalog-
backed path is wired into the hook.

The three review-focus questions:
- Diagnostic bridge useful and appropriately scoped before hook-
  side replacement: yes — operators get a "what would the hook do"
  surface that's safe to call from any session.
- Single-statement parser/analyzer rejection is the right safety
  default: yes.
- Acceptable that relation context still uses SPI lookup here
  while the planner-hook stays pass-through: yes — that's exactly
  what 30855's catalog path is built to fix.

## Findings

### P2 — `pg_analyze_and_rewrite_fixedparams` runs full analysis

The diagnostic runs the full PG analyzer/rewriter pipeline. That's
correct for a representative classification, but means the
diagnostic can fail with PG's normal analyzer errors (table
doesn't exist, column doesn't exist, type mismatch, etc.) before
any front-door logic runs. Worth documenting in the ADR/operator
docs that the diagnostic surfaces standard PG analysis errors
upstream of the front-door classifier, so an operator running it
against a typo'd query sees the same error PG would surface for
the actual query.

### P2 — `Spi::connect` inside the diagnostic

The function calls `am::spire_dml_frontdoor_relation_context_row`
which uses the SPI-backed loader. Fine for a diagnostic, but worth
noting that calling this from the planner_hook would deadlock
(planner inside SPI inside planner). The pass-through hook
correctly avoids this, and 30855's catalog path is the right fix.

### P3 — `unsupported_target_relation` message text

`"ec_spire_distributed: DML front door requires one target heap
relation"` — clear. Worth pinning a stable error registry so
future changes don't break monitors that grep on this text.

### P3 — `unsupported_operation` returns relation OID + status

The shape includes the resolved relation OID + status even when
the operation is rejected. Useful — operators can see "yes I
recognized your table, but this command shape isn't supported."

### P3 — `next_step` text strings

`"wire planner hook to CustomScan executor replacement"` for
supported shapes vs `"rewrite query to fit the ADR-069 v1 DML
front-door shape"` for unsupported. Both reasonable.

## Reviewer recommendation

Accept. This is the right operator-facing surface for the gap
between "primitives exist" and "transparent hook is wired." Once
the planner hook starts replacing plans, this diagnostic remains
useful as the "what would happen" probe.
