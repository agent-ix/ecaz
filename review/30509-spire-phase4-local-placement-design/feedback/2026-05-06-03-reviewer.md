## Code-level observations from this round

### 1. `prefetch_store_object_read_group` (singular) is now dead production code

scan/snapshot.rs:317. Only called from
`scan/tests/candidates.rs:638`. Production calls
`prefetch_store_object_read_groups` (plural) at scan/snapshot.rs:228
which already handles the singular case as `n=1`.

Recommendation: delete the singular helper; have the test call the
plural form with a one-element slice. Trivial cleanup, but
duplicate-helper smell is exactly the pattern earlier feedback
flagged — easier to delete now than after another caller appears.

### 2. Recall gate fixture choice — note

Packet 30530 uses `local_store_count = 1`. That is the right baseline
for the recall claim (multi-store should not change recall, and 30533
confirms it doesn't). It does mean the closure of the Phase 1 gate is
specifically against the single-store path. If the SPIRE storage
format changes for multi-store indexes (it does not currently — same
codecs across stores), the gate would need to be rerun.

Worth a one-line note in the task tracker: "Phase 1 gate measured on
single-store; multi-store path covered by 30533 with recall parity."
The current tracker text is close to this but does not explicitly
call out the parity claim.

### 3. Local placement benchmark — what 10k hides

The same-device and extra-drive lanes show ~140 ms p50 at nprobe=24,
~62 ms at nprobe=8, indistinguishable from each other. At
10k × 1536-dim TurboQuant, the working set is small enough to be
OS-page-cache resident after warmup. So this benchmark proves "no
regression and recall parity" but cannot distinguish disk parallelism
from CPU-bound query path. The packet correctly disclaims production
multi-NVMe claims; the framing is honest.

For the eventual production benchmark on cloud hardware, suggest:

- corpus large enough that the index does not fit in page cache
  (probably 1M+ × 1536-dim);
- cold-buffer measurement lane (drop_caches between iterations);
- the comparison that actually matters: same-corpus, different
  store_count under the same `nprobe` budget.

This is forward-looking, not blocking on this packet.

### 4. Storage-debt aggregation — relation-close ordering

`root/snapshots.rs:456–464` calls `relation_close` unconditionally in
the `if opened` branch *before* propagating `scan_result?`. That
pattern is correct: even if scan_object_tuples returns Err, the
relation closes deterministically. Good.

One residual concern: a panic inside the closure (e.g., from
unanticipated arithmetic) would still leak the lock. The closure
currently uses `checked_add(...).ok_or_else(...)?` and
`u64::try_from(...).map_err(...)?` for all fallible ops, so no
panic path is reachable from current code. Still, wrapping the
relation-open/close pair in a Drop-guard or `defer!`-style helper
would make this robust to future closure additions. Not blocking.

Worth surfacing: packet 30529 (large routing object chain) was a hard
build blocker uncovered by actually running the recall gate. 1536-dim
× 32 nlists routing objects exceed the single-tuple page limit; the
fix chains them into segment tuples behind the existing V1 logical
codec. This is exactly the value of running the gate that was
deferred for several rounds — there was a real correctness issue
hiding behind the unmeasured baseline. Worth keeping in mind for
future "should we measure now or later" judgment calls.
