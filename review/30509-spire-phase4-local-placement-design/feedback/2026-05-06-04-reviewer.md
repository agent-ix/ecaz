### Structural debt â€” `include!` module split

The directory layout under `src/am/ec_spire/` looks like real Rust
submodules (`storage/`, `update/`, `scan/`, `meta/`, `build/`,
`root/`), but the implementation uses `include!` macros to splice
files together at compile time:

- `src/am/ec_spire/mod.rs:59â€“67`: `include!("root/lifecycle.rs")` Ă—9
- `src/am/ec_spire/storage.rs:22â€“34`: `include!("storage/...")` Ă—13
- Same pattern in `update.rs`, `scan.rs`, `meta.rs`, `build.rs`

Implications:

- **`pub(super)` semantics are misleading.** A `pub(super)` item in
  `storage/leaf_v2.rs` is *actually* `pub(super)` from `storage.rs`
  (resolves to `ec_spire`), not `pub(super)` from
  `storage::leaf_v2` (which would resolve to `storage`).
  The directory structure suggests a privacy boundary that does
  not exist.
- **Circular references between files are silently allowed** because
  they are the same module.
- **Dependency graph is invisible** to tooling and to humans
  scanning `mod` declarations â€” a maintainer cannot see "what does
  scan depend on" without reading every file.
- **Compile-time encapsulation is zero.** `update/scheduler.rs` can
  reference any `update/...` private item without an explicit `use`.

Why this happened: the round-2 file split was the cheapest
mechanical translation from the original 11.6k-line `update.rs` to
multiple files. `include!` preserves all references unchanged, no
`pub(crate)` adjustments needed.

Recommendation: **track as immediate post-landing cleanup.** Convert
each `include!` to a real `mod ...;` declaration. The migration
will surface real privacy issues that need explicit `use` lines but
should be mechanical. The longer this is deferred the more code
piles on top of the synthetic structure. Not blocking on Phase 4
landing â€” the code compiles correctly and tests pass â€” but flagging
firmly because this is the kind of debt that quietly compounds.

## Code quality â€” assessment

### Good

- **No `unwrap()` in production paths sampled.** Errors propagate
  via `Result<_, String>` consistently. `unwrap()` appears only in
  tests.
- **Real implementations everywhere; no mocks.** Tests use
  `SpireLocalObjectStore` (in-memory backend) for unit work and
  `SpireRelationObjectStore` for PG18 integration, with the trait
  boundary being `SpireObjectReader`. This is the right test
  architecture.
- **Naming consistency.** `Spire` prefix throughout; `local_store_id`,
  `store_relid`, `pid`, `vec_id`, `epoch`, `object_version` are
  uniform across all 107 files.
- **Newtype safety wrappers** like `SpirePidAllocator`,
  `SpireLocalVecIdAllocator` enforce monotonicity invariants in the
  type system.
- **Error messages carry identifiers** (PIDs, relids, store ids) â€”
  reviewable from logs without re-running.
- **Module sizes stayed tight after split.** Largest non-test file
  is `update/tests/merge_execution.rs` at 961 lines (test); largest
  production file is `update/routing.rs` at 959.

### Gaps

- **Zero `// SAFETY:` comments on 97 unsafe blocks in `page.rs`,
  85 in `relation_store.rs`, 52 in `vacuum.rs`.** Postgres FFI is
  unsafe by necessity, but Rust convention is one safety comment
  per `unsafe` block stating why the inputs satisfy the invariants
  the called function expects. Without these, future maintainers
  must walk back through caller chains to verify safety. Adding
  these is mechanical but takes time. **Track as post-landing
  cleanup, not a landing blocker.**
- **Effectively no rustdoc.** `grep -c "^/// "` returns 0â€“3 across
  the major files. For a 37k-line subsystem implementing a
  published design (ADR-049), the absence of type-level and
  module-level docstrings is a real maintainability gap.
  Recommendation: at minimum, write a 5-line docstring on each
  major struct (`SpirePublishCoordinatorInput`,
  `SpireRelationObjectStoreSet`, `SpireValidatedEpochSnapshot`,
  `SpireLocalStoreConfig`, `SpirePlacementDirectory`) and a
  paragraph at each `mod` head explaining what it owns.
  Post-landing.
- **`Result<_, String>` everywhere instead of a typed error
  enum.** Callers cannot pattern-match on error categories. For a
  subsystem this size, a `SpireError` enum (with `thiserror` or
  manual `Display` impl) would help diagnostics, log filtering,
  and future telemetry. Post-landing, low priority.

### Constructor leak â€” pre-landing fix

`SpireRelationObjectStoreSet::for_index_relation_and_config`
(line 1098) and `for_index_relation_and_placements` (line 1145)
build `opened_relations: Vec<(Relation, LOCKMODE)>` locally before
moving into `Self`. If the loop opens k relations successfully then
errors on k+1, the `Vec` is dropped *without* calling
`relation_close` â€” neither the tuple type nor `Vec` carries a
destructor for this case.

This is filed in detail in
`review/30537-spire-aux-store-autovacuum-guard/feedback/`. The
fix is a small RAII wrapper. Pre-landing if practical, otherwise
fast follow.
