## 1. Helper proliferation has crossed into a design smell

Phase 2's task list and the Phase 4 packets share a recognisable shape:
near-identical entries differing only along a small number of axes
(relation/local × split/merge/replacement × execution-input / draft /
snapshot / publish-plan / publish-lock helper). Each cell of that matrix
became its own pure helper, its own commit, and its own review packet.
The result:

- `src/am/ec_spire/update.rs` — 11586 lines
- `src/am/ec_spire/scan.rs` — 5328 lines
- `src/am/ec_spire/storage.rs` — 4624 lines
- `src/am/ec_spire/build.rs` — 4052 lines
- `src/am/ec_spire/mod.rs` — 3885 lines
- `src/am/ec_spire/meta.rs` — 2260 lines

The combinatorial naming is the giveaway that a missing trait or enum
axis is being expressed via duplication instead. Concrete evidence in
Phase 4 alone: `SpireStoreLeafRouteGroup` and `SpireStoreObjectReadGroup`
carry the same payload and one is a wrapper around the other (see 30519
feedback).

Recommendation, in order:

1. Pure file split first (no logic changes) along the axes the helper
   names already imply: `update/{split,merge,replacement,scheduler,
   maintenance}.rs`, `scan/{routing,grouping,quantized,reranked}.rs`.
   Mechanical move plus `pub use`s. Reviewable in one packet.
2. Once the split exposes the duplication visually, decide if a trait
   collapse is worth doing before more Phase 4 helpers. Optional — do
   not let it block Phase 4 if the right boundary is not obvious.

The split is cheap *now*. It gets more expensive every day Phase 4 work
lands inside the existing files.

## 2. Land file split before more Phase 4 code

Independent of the trait question: any Phase 4 diff currently lands inside
files that already exceed reviewer working memory. After a pure split, the
hash-routed write diff lives in `update/replacement.rs` or
`storage/placement.rs` and is legible on its own. Doing Phase 4 first
means every Phase 4 line you write gets touched again by the eventual
split and the diff is harder to verify as behavior-preserving.

## 3. Review-packet cadence is too fine-grained

Phase 4 has produced ~10 review packets so far for what is structurally
one feature slice (local multi-store placement). Several packets are
~150-line diffs containing a single planner/helper. Each packet has its
own `Plan X` + `Add X review packet` commit pair, inflating history and
reviewer load without obviously improving safety. The pre-persistence
architecture gate was the right call; the per-helper gate is overkill.

Recommendation: bundle related helpers into one packet at the
feature-slice grain — for example, all "store-relation planning" pieces
(tablespace plan, relation-name plan, descriptor publish plan) belong in
one packet, not three. Reviewer attention scales with packet count, not
LOC.

## 4. Easy structural work is landing ahead of the validation that
   proves the design

Pattern across phases: scaffolding ships fast, then the items that
actually prove the choice — measured recall/latency, physical
reclamation, parallel fetch, local placement benchmark, post-Phase-1
grouped-PQ scorer — get deferred. Phase 1's "measured recall/latency
evidence remains open" is the most concerning instance: Phase 2/3/4 code
is being built on a foundation whose recall has not been measured against
a real corpus. The single-level foundation is the load-bearing claim of
Task 30 and is unverified.

Stop and run the recall/latency gate on the Phase 1 surface before more
Phase 4 plumbing lands. If recall or latency is materially off, every
hour of Phase 4 helper work is being spent on a foundation that needs a
correction first.

## 5. Phase 4 is half-decorated

Done so far: option surfaces, hash *planning*, descriptor *planning*,
relation-name *planning*, scan grouping *primitive*, object-store
local-store-id *threading*. Open: actual hash-routed writes (verify
`store_for_pid` is on the write hot path, not just present in the codebase
— see meta.rs:374), actual store relations, actual parallel fetch, actual
placement diagnostics with per-store-relation byte/object counts, actual
multi-store benchmark.

The plan walked the boundary in pure-function order without ever crossing
it. Recommendation: pick one PID, write a real object to a *second* store
relation, fetch it on the scan path, end-to-end. Get the through-path lit,
then come back to fill in the planning helpers. This single end-to-end
test will surface ten times more design issues than another ten planning
helpers will.

## 6. `spire_pid_hash` is an undocumented load-bearing on-disk format

`meta.rs:441` is splitmix64 (`0xbf58_476d_1ce4_e5b9` / `0x94d0_49bb_1331_11eb`
constants). The cross-platform stability test asserts golden values, which
is good. But:

- The function carries no comment naming the algorithm. Future-you will
  not know whether it is safe to swap.
- `pid_hash % store_count` is fine for fixed `store_count`, but if
  `store_count` ever changes (user adds an NVMe and reindexes, or a
  future online rebalance lands), every PID's home store moves. The
  Phase 4 design note should explicitly state: "store_count is fixed at
  build; changing it requires REINDEX." If that constraint is not
  documented next to `store_for_pid`, someone will assume otherwise.

Both are one-line fixes. They are load-bearing because the hash output is
implicitly persisted via every placement decision.

## 7. Strengths — keep doing

- Strict-mode visibility, retired-manifest copies, and epoch-bundle
