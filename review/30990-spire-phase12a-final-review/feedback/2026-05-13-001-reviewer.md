## What's solid

### 12a.1 prepared-xact reaper (`30988`, `35240a79`) — the load-bearing item

The 2PC in-doubt window is properly closed:

- `ec_spire_remote_prepared_xact_intent` is a **regular durable
  relation** (`sql/bootstrap.sql:156`), not unlogged/temp, with
  `PRIMARY KEY (gid)` and state CHECKs.
- Intent row is written with state `prepare_requested` **before**
  the libpq `PREPARE TRANSACTION` dispatch in
  `coordinator_insert_prepare_remote_sql_batch`. After ack, state
  advances to `prepare_acked`. On `PreCommit` xact callback the row
  flips to `commit_local` **before** the Commit callback fires the
  remote resolve. This is the correct ordering — an ack lost between
  remote WAL-flush and coordinator visibility leaves the intent in
  `prepare_requested` with no live coordinator xid, which is the
  reaper's rollback target.
- `ec_spire_reap_orphaned_remote_prepared_xacts(node_id)` and the
  all-remotes sweep `ec_spire_reap_all_orphaned_remote_prepared_xacts()`
  both ship. Decision logic uses `pg_stat_activity.backend_xid` /
  `backend_xmin` for liveness and preserves rows with intent
  `commit_local` (operator escalation, not silent rollback).
- Fixture `test_ec_spire_reaper_resolves_lost_prepare_ack_fixture`
  simulates the lost-ack window directly.
- ADR-069 records the operator-driven (not background-worker)
  cadence decision.

## Caveats (none block closure)

### P3 — 12a.5 retired-transport-status e2e fixture gap

`src/am/ec_spire/root/remote_candidates.rs:8791`
(`remote_tuple_transport_tests`). The new
`SPIRE_REMOTE_STATUS_TUPLE_TRANSPORT_RETIRED` category is exercised
only by struct-level unit tests against synthetic `endpoint_identity`
values. The audit-flagged closure item — a pg_test fixture stubbing a
real json-only remote — did not land. Low risk because the production
path is a single `Err(...)` branch trivially proven correct by
inspection, and the hint plumbing is independently tested through
`degraded_skip_report_hints_retired_tuple_transport_upgrade`. Worth
adding when convenient, but not closure-blocking.

### P3 — 12a.4 CI path-filter pattern

`.github/workflows/ci.yml:74`. The path filter for the Stage E subset
runs **after** matrix expansion via an in-job `git diff` script with
`if: steps.changes.outputs.run == 'true'`. On non-SPIRE PRs, four jobs
allocate runners and immediately return. Functionally correct;
wastes CI minutes. The idiomatic pattern is a job-level `if:` keyed on
a `paths-filter` action, or a workflow-level `paths:` filter. Cosmetic
hygiene only.

### P3 — 12a.2 pre-cap row-vector allocation

`try_get::<_, Vec<String>>("payload_values_hex")` at
`remote_candidates.rs:9690` materializes the full hex `Vec<String>`
before the byte-cap check. The cap then bounds the bulk `hex::decode`
expansion, but the initial Vec is bounded by libpq protocol limits,
not the GUC. Acceptable for P2 in practice (the underlying libpq driver
applies its own limits and the post-decode expansion is the
dominant memory pressure), but not literally "before allocation."
Document as "before bulk decode" or tighten with a streaming receive
if Phase 13 measurement shows pressure.

### P3 — 12a.6 per-dispatch echo cost

The chosen design queries remote schema on every dispatch. Decision is
ADR-069-recorded. At Phase 13 cross-AZ scale this adds one round-trip
of latency per coordinator-routed mutation. Carry as a Phase 13
perf-measurement watch-item; do not change the design without scale
evidence.
