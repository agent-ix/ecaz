## Additional Reference: prism-mcp TurboQuant Implementation

A third independent TurboQuant implementation exists in `~/dev_bak/prism-mcp/`. It is a pure
TypeScript port used for Prism MCP's "Quantized Agentic Memory" feature (v5.0). Comparing it
alongside TurboQuantDB strengthens several conclusions from review 204 and surfaces one new
finding.

### prism-mcp Architecture Summary

Source: `~/dev_bak/prism-mcp/src/utils/turboquant.ts` (~840 lines)

- **Rotation**: QR decomposition (Householder) of a random Gaussian matrix — NOT FWHT/SRHT
- **Dimension**: 768 (Gemini text-embedding-004), no power-of-2 constraint
- **Codebook**: Gaussian N(0, 1/d) approximation of Beta, parameterized by actual dimension d
- **Encoding**: all d=768 dimensions, no padding, no truncation
- **QJL**: full d×d Gaussian projection matrix with explicit residual reconstruction
- **Centroid lookup**: binary search via boundaries
- **Numeric precision**: Float64Array throughout

### Three-Implementation Comparison

| Property | tqvector (current) | TurboQuantDB | prism-mcp |
|---|---|---|---|
| Rotation method | SRHT (tiled 3×512) | Full SRHT (padded 2048) | QR decomposition |
| Rotation cost per vec | O(d log d) | O(d log d) | O(d²) |
| Rotation memory | ~6 KB (signs) | ~8 KB (signs) | ~4.7 MB (d×d matrix) |
| Power-of-2 required | Yes | Yes | No |
| Dimension | 1536 | 1536→2048 | 768 |
| Codebook PDF | Beta(d/2) | Beta(n/2) | Gaussian N(0, 1/d) |
| Codebook dim param | `dim`=1536 | `n`=2048 | `d`=768 |
| Codebook matches rotation? | **No** (Beta(768) vs actual ~Beta(256)) | **Yes** | **Yes** |
| Encoded dims | 1536 | 2048 | 768 |
| Dims discarded | 0 (tiled) | 0 (all-n stored) | 0 |
| QJL implementation | 14 lines, thin | 412 lines, stateful | Full d×d projection |
| Centroid lookup | linear scan | binary search | binary search |

### What prism Validates

**1. The algorithm works when rotation and codebook are correctly matched.**

prism at d=768 achieves >85% Pearson correlation between true and estimated cosine similarity,
>95% top-5 retrieval accuracy, and <0.05 mean estimator bias — all verified in
`tests/turboquant.test.ts`. These results are achieved with a straightforward implementation
that has no padding, no tiling, and no dimension mismatch.

This confirms that tqvector's quality gap is in the FWHT implementation choices (tiling scope
and codebook mismatch), not in the TurboQuant algorithm itself.

**2. Codebook parameterization must match the actual post-rotation distribution.**

prism uses `getCodebook(config.d, this.mseBits)` where `config.d` = 768, the actual dimension.
The QR rotation produces coordinates distributed as N(0, 1/768) — and the codebook is built
for exactly that distribution.

TurboQuantDB uses `lloyd_max(bits-1, n)` where `n` = 2048, the padded dimension. The full SRHT
produces coordinates distributed as Beta(1024, 1024) ≈ N(0, 1/2048) — codebook matches.

tqvector uses `lloyd_max(bits-1, dim)` where `dim` = 1536. But the tiled FWHT (3×512 blocks)
produces per-coordinate marginals closer to Beta(256, 256) ≈ N(0, 1/512). **The codebook
is built for a distribution ~3× narrower than reality.** This is the same finding as review
204's recommendation #1, now independently corroborated by a working reference implementation.

**3. Gaussian approximation is equivalent to exact Beta for d≥64.**

prism uses `gaussianPdf(x, d) = N(0, 1/d)` instead of the exact Beta((d-1)/2, (d-1)/2) PDF.
tqvector and TurboQuantDB use the exact Beta. At d≥64, the KL divergence between the two is
negligible (~1e-6). This means the codebook distribution choice (Gaussian vs Beta) is not a
quality-relevant divergence. The dimension parameter is what matters.

### What prism Does Differently (Not Directly Portable)

**QR rotation is too expensive for tqvector's use case.**

prism's QR approach eliminates all FWHT dimension constraints, but at substantial cost:

| Cost | SRHT (tqvector) | QR (prism) |
|---|---|---|
| One-time matrix generation | ~0 (ChaCha8 PRNG) | O(d³) ≈ 50ms at d=768 |
| Per-vector rotation | O(d log d) ≈ ~15K ops | O(d²) ≈ ~590K ops |
| Memory per index | ~6 KB | ~18 MB at d=1536, ~32 MB at d=2048 |

For prism (1 vector per save call, server singleton), O(d²) is fine. For tqvector (bulk HNSW
build encoding thousands of vectors, per-query preparation in PostgreSQL), the per-vector cost
is prohibitive at d=1536+. The memory footprint is also unsuitable for PostgreSQL backend
processes.

SRHT remains the correct rotation choice for tqvector. The lesson from prism is not "use QR"
but "match your codebook to your actual rotation dimension."

**Float64 precision is a non-factor.**

prism uses Float64Array throughout. tqvector uses f32. For quantization to 3-4 bit indices,
the extra precision is irrelevant — the quantization noise dominates by orders of magnitude.

### Updated Assessment of Review 204 Recommendations

The prism comparison reinforces all four review 204 recommendations:

1. **Fix codebook parameterization (zero-cost)** — now supported by evidence from two
   independent implementations (TurboQuantDB and prism) that both parameterize the codebook
   by the actual post-rotation dimension. tqvector is the only implementation that gets this
   wrong.

2. **Measure codebook fix before committing to full+all-n** — still the right sequencing.
   prism shows good quality at d=768 with correct codebook alone (no full FWHT advantage at
   play since QR gives full decorrelation at any dimension). If the codebook fix at d=1536
   tiled recovers most of the gap, full+all-n may be unnecessary.

3. **Adopt full+all-n only if decorrelation gap >2pp** — unchanged. prism's QR gives
   theoretically perfect decorrelation, which SRHT approximates. Tiled FWHT gives weaker
   decorrelation. The empirical measurement will show whether this matters.

4. **Keep tiled FWHT as supported mode** — unchanged. For 1536 specifically, tiling is the
   right tradeoff vs the 33% storage cost of full+all-n.

### New Finding: QJL Implementation Gap

prism's QJL is substantially more complete than tqvector's:

- **prism**: full d×d Gaussian projection matrix S, explicit `residual = vec - mseReconstruction`,
  `projected = S × residual`, sign bits packed, correction term =
  `residualNorm × sqrt(π/2) / m × Σ(S@q)_i × sign(S@r)_i`
- **TurboQuantDB**: 412-line stateful QjlQuantizer with n=2048 projection
- **tqvector**: 14 lines in `qjl.rs`, `qjl_project` and `decode_mse_only`

Review 201 showed QJL contributes only ~1% recall. But that measurement was taken with the
codebook mismatch in place — the MSE reconstruction was poor, leaving a large residual that
1-bit QJL correction could not meaningfully recover. **After the codebook fix improves MSE
reconstruction, QJL's marginal contribution may increase.** This should be re-measured after
the codebook fix lands.

This is low priority compared to the codebook fix but worth noting as a potential follow-up
quality lever.

## Paper Reference: Rotation Is QR, Not FWHT

Source: `~/dev_bak/turboquant-2504.19874.pdf` (arXiv:2504.19874, ICLR 2026)

The TurboQuant paper specifies QR decomposition, not FWHT:

> "We start by randomizing this vector by multiplying it with a random rotation matrix
> **Π ∈ ℝ^{d×d}**. We can generate Π by applying QR decomposition on a random matrix with
> i.i.d Normal entries."

The paper says nothing about FWHT, SRHT, tiled transforms, non-power-of-2 dimensions, padding,
or truncation. The post-rotation distribution is explicitly Beta:

> "each coordinate of Π·x follows a Beta distribution" with PDF
> `f(x) = Γ(d/2) / (√π · Γ((d-1)/2)) · (1 - x²)^((d-3)/2)`

The codebook is solved for this distribution in dimension d. Since QR produces a d×d orthogonal
matrix, rotation dimension = codebook dimension = d. No mismatch is possible.

### FWHT Is an Implementation Optimization, Not Paper-Specified

SRHT/FWHT was introduced as a practical substitute by TurboQuantDB (and adopted by tqvector).
The substitution is mathematically valid — SRHT approximates a random rotation — but it
introduces constraints the paper never had to address:

1. Power-of-2 input length requirement
2. Padding or tiling for non-power-of-2 dimensions
3. The question of how many rotated dimensions to store
4. Codebook-rotation dimension mismatch when padding or tiling changes the effective dimension

### Why tqvector Cannot Use the Paper's QR Approach

The paper's QR rotation requires storing and applying a d×d dense orthogonal matrix:

| Dimension | Matrix size | Per-vector multiply |
|---|---|---|
| 768 (prism) | ~4.7 MB | O(d²) ≈ 590K ops |
| 1536 (tqvector) | ~18 MB | O(d²) ≈ 2.4M ops |
| 2048 | ~32 MB | O(d²) ≈ 4.2M ops |

For prism (1 vector per save call, server singleton, d=768), this is acceptable — 4.7 MB
resident memory, ~50ms one-time setup.

For tqvector (PostgreSQL backend, bulk HNSW build encoding thousands of vectors, per-query
preparation, d=1536+), this is prohibitive:

- **18 MB per index** in backend process memory for the rotation matrix alone
- **O(d²) per encode** during bulk build — at 10k vectors, that's 24 billion FP ops just
  for rotation, vs ~165 million with SRHT
- **O(d²) per query preparation** — adds ~2.4M ops to every search query
- Multiple concurrent backends each holding their own copy

SRHT at O(d log d) with O(d) memory is the correct choice for tqvector. The tradeoff is
accepting the power-of-2 constraint and the associated tiling/padding decisions.

### Implication for the Codebook Fix

The paper's implicit assumption — rotation dimension equals codebook dimension — holds
trivially with QR (both are d) and with TurboQuantDB's full SRHT (both are n=2048). It breaks
in tqvector because tiled FWHT creates an effective rotation dimension (512) that differs from
both the original dimension (1536) and the codebook dimension (1536).

The codebook fix (review 204 recommendation #1) restores this invariant by parameterizing the
codebook with the tile dimension. This is the cheapest path to matching the paper's implicit
contract within tqvector's SRHT-based architecture.

|---|---|---|---|---|---|

### Relevance to tqvector

tqvector's current best exact-only recall at 1536-dim 4-bit is ~81.5% on clustered 1k
(review 203). TurboQuantDB hits 89.15% at 4-bit on 50k real embeddings with HNSW. These
numbers are not directly comparable (different corpora, different scales, HNSW vs exact-only),
but the direction is consistent: **a correctly-matched codebook + full SRHT + all-n-dims
encoding yields materially higher recall.**

The ~6pp gap between TurboQuantDB's "Balanced" (89.15%) and "Fast Build" (83.35%) is entirely
from QJL — fast_mode sets gamma=0 and skips the QJL residual. This is a larger QJL
contribution than tqvector's ~1% (review 201), likely because TurboQuantDB's MSE reconstruction
is already good enough that the residual is small and the 1-bit correction is effective.

## TurboQuantDB Deep-Dive: Code-Level Architecture

Full source read of `~/dev_bak/TurboQuantDB/src/quantizer/` and `src/linalg/hadamard.rs`.

### README Describes QR, Code Uses SRHT

The README says "QR rotation" in the write path description:
> **Write path:** insert() → quantize (QR rotation → MSE → Gaussian QJL) → WAL

But the actual code uses SRHT throughout. `MseQuantizer` (mse.rs:13) documents:
> "SRHT rotation: apply H * D * (x, 0) where H is the Walsh-Hadamard matrix and D is a
> seeded random diagonal ±1 matrix."

The QjlQuantizer also uses SRHT (qjl.rs:10: `use crate::linalg::hadamard::srht`). There is
no QR decomposition anywhere in the codebase. The README's "QR rotation" label is inaccurate
shorthand for the SRHT.

### MseQuantizer (mse.rs)

```rust
pub fn new(d: usize, b: usize, seed: u64) -> Self {
    let n = d.next_power_of_two();           // 1536 → 2048
    // rotation_signs: n=2048 random ±1 values
    let centroids = lloyd_max(b, n, 20_000); // codebook uses n=2048, NOT d=1536
    ...
}
```

Key design choices:
- `n = d.next_power_of_two()` — always pads to power of 2
- `lloyd_max(b, n, ...)` — codebook parameterized by **padded dimension n**, not original d
- `rotation_signs` length = n (2048 for d=1536)
- `quantize()` returns **n indices** (all 2048 dimensions encoded)
- `dequantize()` applies inverse_srht to recover only the first d=1536 dimensions
- RNG: `StdRng::seed_from_u64(seed)` (not ChaCha8Rng like tqvector)

### QjlQuantizer (qjl.rs)

```rust
pub fn new(d: usize, seed: u64) -> Self {
    let n = d.next_power_of_two();           // 1536 → 2048
    // projection_signs: n=2048 random ±1 values (independent seed from MSE)
}
```

- Uses a **second SRHT** with independent signs (not a dense Gaussian matrix like prism)
- Encodes the residual in n=2048 space → `ceil(n/8)` = 256 bytes of QJL signs
- Dequantize uses `sqrt(π / (2*n))` scaling factor

### ProdQuantizer (prod.rs)

```rust
pub fn new(d: usize, b: usize, seed: u64, fast_mode: bool) -> Self {
    let mse_quantizer = MseQuantizer::new(d, b, seed);
    let qjl_quantizer = QjlQuantizer::new(d, seed ^ 0xdeadbeef);
    let n = d.next_power_of_two();
    ...
}
```

- `prepare_ip_query` builds LUT of size `n × 2^b` (2048 × 16 at b=4)
- `prepare_ip_query_lite` rotates query into n-dim SRHT space for code-to-code scoring
- `score_ip_encoded_lite` iterates over `self.n` = 2048 dimensions
- AVX2+FMA SIMD path for both MSE centroid gather and QJL sign expansion
- `fast_mode`: skips QJL entirely, stores gamma=0 — saves ~50% scoring work per README


- Uses exact Beta PDF: `f(x) = Γ(d/2) / (√π · Γ((d-1)/2)) · (1-x²)^((d-3)/2)`
- Grid-based Lloyd-Max (20,000 points) instead of Simpson's-rule integration (prism)
- Initial centroid spacing bounded by `3/sqrt(d)` — adapts to dimension automatically
- Returns `Vec<f64>`, cast to `Vec<f32>` by MseQuantizer

### Hadamard (linalg/hadamard.rs)

```rust
pub fn srht(x: &[f32], signs: &[f32], out: &mut [f32]) {
    let d = x.len();
    let n = d.next_power_of_two();
    // out[0..d] = x[i] * signs[i]
    // out[d..n] = 0.0           ← zero padding
    fwht(out);                    ← full FWHT on n dimensions
    // normalize by 1/sqrt(n)
}
```

- Full FWHT with AVX2 SIMD path (8-wide butterfly operations)
- `inverse_srht` uses thread-local temp buffer to avoid allocation per dequantize call
- No tiling support — always operates on the full padded dimension

### tqvector vs TurboQuantDB: Line-by-Line Divergences

| Code point | tqvector | TurboQuantDB | Impact |
|---|---|---|---|
| `prod.rs` new() codebook | `lloyd_max(bits-1, dim)` | `lloyd_max(b, n)` | **Critical**: tqvector uses wrong dim |
| MSE bits | `bits - 1` (1 bit reserved for QJL) | `b` (full bits for MSE) | TurboQuantDB gives all bits to MSE; QJL has its own separate encoding |
| Encoded dims | 1536 | 2048 | 33% more signal in TurboQuantDB |
| FWHT | tiled 3×512 | full on 2048 + AVX2 | Full decorrelation in TurboQuantDB |
| QJL projection | SRHT-based (14 lines) | SRHT-based (412 lines, batch, SIMD) | Same algorithm, much more mature impl |
| QJL seed | `seed` (same as MSE) | `seed ^ 0xdeadbeef` | TurboQuantDB ensures independence |
| Centroid lookup | linear scan O(k) | `partition_point` O(log k) | Minor at k=8-16 |
| MSE RNG | ChaCha8Rng | StdRng | Different sign vectors for same seed |
| Scoring SIMD | none (scalar only) | AVX2+FMA for MSE gather + QJL sign expand | Performance only |
| Residual computation | thin | full: `dequantize()` → subtract → QJL encode | TurboQuantDB computes exact residual |
| fast_mode | N/A | gamma=0, skip QJL (~30% faster ingest) | Nice optimization |

### Critical Finding: MSE Bit Allocation

Review 204's request listed `bits-1` for MSE as a shared property, but a closer reading shows
the implementations differ in what "bits" means:

- **TurboQuantDB**: `MseQuantizer::new(d, b, seed)` receives the full MSE bit budget directly.
  `ProdQuantizer::new(d, b, seed)` passes `b` straight through to MseQuantizer. At b=4, MSE
  gets **4 full bits (16 centroids)**. QJL is a separate 1-bit-per-dim encoding on top.
- **tqvector**: `lloyd_max((bits - 1) as usize, dim, 20_000)` — at bits=4, MSE gets
  **3 bits (8 centroids)**. The 1-bit "reservation" for QJL is baked into the MSE budget.

This means **TurboQuantDB at b=4 has 2× the MSE centroids** (16 vs 8) compared to tqvector at
bits=4. The total storage per dimension is 4 MSE bits + 1 QJL bit = 5 bits in TurboQuantDB
vs 3 MSE bits + 1 QJL bit = 4 bits in tqvector. TurboQuantDB's "4-bit" is really 5 bits per
coordinate.

This reframes the benchmark comparison: TurboQuantDB's 89.15% recall at "b=4" uses 5 bits/dim
total, while tqvector's 81.5% at "bits=4" uses 4 bits/dim total. They are not at the same
compression ratio. TurboQuantDB's 70 MB disk for 50k×1536 at b=4 includes the extra QJL byte
overhead.

**This is not a bug in tqvector** — ADR-007 explicitly chose the `bits-1` MSE + 1-bit QJL
split. But it means direct recall comparisons at the same "bits" label are misleading. A fair
comparison would be tqvector bits=5 (4 MSE + 1 QJL) vs TurboQuantDB b=4 (4 MSE + 1 QJL).

## Disposition

Review 204's analysis and recommendations hold. The deep-dive code comparison, prism-mcp
reference, paper review, and published benchmarks provide converging evidence:

1. **The TurboQuant algorithm achieves 89-97% recall on real embeddings** when rotation and
   codebook are correctly matched (TurboQuantDB benchmarks on DBpedia OpenAI 50k×1536)
2. **FWHT/SRHT is an implementation optimization** not specified by the paper, which uses QR.
   TurboQuantDB substitutes SRHT. Both prism (QR) and TurboQuantDB (SRHT) get the codebook
   right; tqvector does not.
3. **QR is not viable for tqvector** due to O(d²) cost and O(d²) memory at d=1536+
4. **tqvector has two codebook-related defects:**
   - Codebook dimension uses 1536 instead of tile_dim=512 (Beta(768) vs Beta(256))
   - MSE gets `bits-1` centroids where TurboQuantDB gives `bits` full centroids — a design
     choice from ADR-007, not a bug, but it means "4-bit" in tqvector is lower fidelity than
     "b=4" in TurboQuantDB
5. **The codebook dimension fix is the immediate next step** — zero storage cost, restores
   the invariant that the paper assumes and both reference implementations preserve
6. **Re-evaluating the bits-1 MSE budget** is a secondary question worth measuring after the
   codebook fix: what does tqvector recall look like at bits=5 (to match TurboQuantDB's
   effective rate)?

Coder-1's ongoing tests against the review 204 request will determine the empirical magnitude
of these findings.
