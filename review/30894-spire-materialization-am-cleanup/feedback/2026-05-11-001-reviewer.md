# Review: SPIRE Materialization AM Cleanup (30894)

## Summary

Removes the remaining superseded row-materialization AM cleanup
paths after 30892 dropped the catalog. Big delete: ~352 lines
out of `remote_candidates.rs`, ~282 lines out of `tests.rs`.

Key behavioral changes:
- Removed row-materialization and mirror-sync SQL contract
  functions and their operator-entrypoint contract rows.
- Removed the catalog-backed AM materialization provider and
  its tests.
- Removed `SPIRE_REMOTE_MATERIALIZED_HEAP_RESOLUTION` from the
  AM tuple delivery branch — remote-origin rows now route only
  to CustomScan tuple payload delivery.
- Status string rename: `requires_remote_row_materialization` →
  `requires_custom_scan_tuple_delivery`. The AM cursor still
  classifies remote-origin candidates as "not deliverable
  through this path" but now points operators at CustomScan
  instead of the dropped materialization mechanism.

The three review-focus questions:
- No runtime / SQL surface still exposes the superseded
  contract: confirmed by the diff (all
  `remote_row_materialization` symbols deleted; only the
  zero-shim columns in the orphan summary remain, by design).
- Remote-origin rows directed to CustomScan tuple delivery
  rather than AM cursor / materialization: yes — the
  `custom_scan_access` branch dispatches remote-origin to
  `custom_scan_store_remote_tuple_payload`; AM cursor's
  remote-origin branch errors with the new "requires CustomScan
  tuple delivery" diagnostic.
- `row_materialization_*` columns in catalog cleanup
  diagnostics as zero-returning compat shims acceptable: yes
  (already discussed in 30892).

## Findings

### P2 — Status rename touches the operator-facing diagnostic
contract

`requires_remote_row_materialization` is replaced with
`requires_custom_scan_tuple_delivery` in user-visible diagnostic
SQL. Any monitoring / alerting that grep'd for the old string
will silently stop matching. Worth:
- A one-line note in the 0.1.1→0.1.2 migration / changelog that
  the diagnostic status string changed, with a
  before/after mapping (operators can update grep regex in one
  pass).
- Confirming the new string is documented in the operator
  runbook.

The rename direction is correct (the new string accurately
names what the AM cursor would need); the concern is silent
operator-monitoring breakage.

### P3 — `~352 + ~282 lines deleted`

Healthy cleanup. The remote_candidates.rs file shrinks by
materialization-mapping validation that was load-bearing for
ADR-064/065/066 but never reached production. Worth confirming
the remaining `remote_candidates.rs` still passes its own focused
tests after the delete.

### P3 — `MATERIALIZED_HEAP_RESOLUTION` constant

Removed from the access-method dispatch matcher. Worth
confirming the constant itself is still referenced anywhere
(some test fixtures may still reference it for assertions about
"this used to be a materialized path"). If not, the constant
itself can be deleted.

### P3 — Mirror-sync contract entry-point row

The `ec_spire_remote_row_materialization_mirror_sync_contract`
operator-entrypoint row is dropped. Operators querying
`ec_spire_remote_node_contract_strings()` (or whatever the
contract registry is named) will see one fewer row. Acceptable
removal; same compat-window concern as the status rename.

## Reviewer recommendation

Accept. The cleanup is thorough — removes both the catalog
surface (30892) and the AM-side consumers (30894), retiring the
ADR-064/065/066 direction completely. The P2 about the
diagnostic-string rename is an operator-facing concern worth
folding into the next docs pass.
