### What went well in the C1 arc

1. **Measurement-first discipline.** Packets 251–253 profiled before
   optimizing. Every subsequent slice had measured evidence justifying the
   target. This is exactly the right way to work.

2. **Narrow, verifiable slices.** Each packet changed one thing, validated it,
   and measured the result. No "optimistic refactors." The score cache (254)
   and greedy upper-layer (256) slices are textbook examples.

3. **Benchmark integrity work was real.** Packets 244–250 are not glamorous,
   but discovering that the benchmark was timing Seq Scan, fixing per-cell
   planner verification, and correcting the ef_search=200 crossover prevented
   weeks of wasted optimization against the wrong surface.

4. **Honest reads.** Every packet that didn't move the needle said so (255
   fast-hash: "essentially flat within noise at ef_search=40"). That honesty
   is more valuable than optimistic framing.

### What needs attention

1. **The remaining 10x gap is structural, not incremental.** The C1 arc has
   been excellent at wringing out per-query overhead, but each slice gets
   diminishing returns on the same fundamental path: score each candidate once,
   cache it, reduce bookkeeping cost. The next 10x will not come from another
   cache layer or hash swap.

2. **Cold-cache vs warm-cache divergence is a red flag.** Packet 251 showed
   warm-cache SQL at ef_search=40 is already ~6.8ms. The cold-cache verified
   surface is ~50ms. That ~8x gap means the dominant remaining cost is
   **page-level I/O latency** (or at least buffer-miss penalty), not CPU-side
   scoring or traversal. This needs to be measured and stated explicitly before
   the next optimization slice.

3. **scan_debug.rs is growing large (2548 lines).** The debug/profile
   infrastructure was necessary for measurement, but it now represents
   significant code mass behind `#[cfg(test)]` gates. Worth tracking as
   technical debt — it should not creep into the production binary.

## Packet 257 Detail: QJL-Disabled 4-bit Score Fast Path

### Code review

The implementation is correct and well-scoped:

- `score_ip_from_split_parts` now detects `!qjl_enabled && bits == 4` and
  dispatches to `score_ip_from_split_parts_no_qjl_4bit`
- The fast path scores directly from packed nibbles against `prepared.rotated`
  and the 16-entry codebook, avoiding the generic `mse_index_at` + LUT
  indirection
- The old scalar path remains as the reference fallback
- No other code paths were disturbed

The 79% microbench improvement (68μs → 14μs per score at d1536_b4) is
well-evidenced and consistent with the macro improvement.

### Concern: the fast path is still scalar

`score_ip_from_split_parts_no_qjl_4bit` iterates byte-by-byte with scalar
nibble extraction and scalar `codebook[nibble] * prepared.rotated[dim]`
accumulation. At 1536 dims, that's 768 loop iterations with two dependent FP
multiplies and adds each.

The existing AVX2 path for QJL-enabled scoring (line 576+) already
demonstrates the SIMD pattern this project uses: `_mm256_permutevar8x32_ps`
for codebook gather, `_mm256_fmadd_ps` for accumulation, 4-way unrolled
accumulators. An equivalent 4-bit no-QJL AVX2 path could process 32 dimensions
per iteration (one 16-byte load → 32 nibbles → 4×8 lane gathers + FMAs).

This would likely cut the per-score cost by another 4–8x, and since candidate
scoring is still the second-largest hot-path bucket (3.2ms at ef_search=40),
that could translate into real scan improvement.

### Validation

Green on all required gates. No concerns.

### Results quality

Excellent. Microbench, hot-path probe, representative SQL, and full verified
surface are all present and consistent.

## Optimization Suggestions for Subsequent C1 Work

The remaining gap is ~10x. Here's what the evidence suggests is worth pursuing,
in order of expected impact:

### 1. Measure and close the cold/warm gap (highest priority)

Packet 251 showed warm-cache ~6.8ms at ef_search=40. The verified surface
shows cold-cache ~50ms. Before any more CPU optimization, coder-1 should run
the hot-path profiler in both warm and cold states and report exactly where
the difference is. If it's buffer-miss latency:

- **PG-level tuning** (shared_buffers sized to hold the index, effective_io_concurrency)
  may be worth more than any code change
- **Read-ahead / prefetching** during graph traversal could hide I/O latency
  on cold paths. The `GraphPrefetchState` and `LinearPrefetchState` are already
  present in TqScanOpaque — are they actually issuing prefetch advice to PG?
- The NFR spec says "Report warm-cache and cold-cache results separately when
  feasible." The C1 surface should already be doing this.

If the warm-cache latency is 6.8ms and the target is 5ms, the warm-path gap
is only ~1.4x, which is much more tractable than 10x.

### 2. SIMD 4-bit no-QJL scoring

As noted above, the new fast path is scalar. An AVX2 version using the same
patterns as the existing 3-bit QJL-enabled path (permute-based codebook
gather + FMA) would process 32 dims per iteration instead of 2. At 1536 dims
that's ~48 iterations vs ~768. The codebook is 16 entries, which fits in two
`__m256` registers — the 4-bit nibble extraction is simpler than the 3-bit
bit-packing.

This is a natural follow-on from packet 257 and should be straightforward to
implement and bench.

### 3. Reduce graph traversal I/O via page-locality-aware beam ordering

The hot-path data shows layer-0 seed search is now the largest single bucket
(5.1ms at ef_search=40 after 257). Buffer hits are 361, but the access pattern
during beam search is random across graph pages. If candidates in the beam are
sorted or prioritized by page proximity (block number), buffer locality
improves and the OS/PG buffer manager has better hit rates.

This is speculative — measure first.

### 4. Consider whether NFR-001 targets should be re-evaluated against corpus size

NFR-001 specifies p50<5ms on a **50K-vector** table. The current C1 surface is
measured on a **10K-vector** table. The 10K corpus is a valid measurement lane,
but the final NFR target table will be 5x larger with a taller graph, more
pages, and more candidates per query. If the warm-cache gap on the 10K table
is already close (6.8ms vs 5ms), the 50K table will likely be harder. This
should be acknowledged in the C1 read.

### 5. LUT-based scoring instead of codebook*rotated

The current 4-bit fast path computes `codebook[nibble] * prepared.rotated[dim]`
per dimension. The `PreparedQuery` already builds a LUT
(`prepared.lut[dim * 16 + centroid_index]`) during `prepare_ip_query`. That
LUT precomputes all 16 `codebook[c] * rotated[d]` products for each dimension.
The non-QJL 4-bit path could use a single LUT lookup per dimension instead of
a multiply, which would be a pure table-lookup accumulation loop — potentially
even faster under SIMD where you'd use `vpshufb` / `vpermb` for 4-bit LUT
lookups across 32 lanes in one instruction.

This is worth checking — the LUT is already allocated, but the fast path
bypasses it.

## Summary

C1 work quality is high. The measurement-first, narrow-slice discipline is
exactly right. The remaining gap is structural (cold-cache I/O dominance) and
the next slice should characterize that gap before committing to more CPU-path
work. If warm-cache latency is already near target, the optimization strategy
shifts from "make scoring faster" to "make the index warm faster / keep it
warm."
