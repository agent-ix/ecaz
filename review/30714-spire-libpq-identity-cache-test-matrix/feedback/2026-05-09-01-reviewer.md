# 30714 — SPIRE libpq identity cache test matrix review

Validation pass on `2f03028b` (`Cover SPIRE libpq identity cache key
matrix`). Accepts the test-matrix expansion. Closes packet 30712 P2.

## What I checked

- A pg-test-only identity-cache contract probe exercises one executor
  state across:
  1. initial miss plus insert,
  2. exact-key hit,
  3. descriptor-generation change miss,
  4. served-epoch change miss,
  5. descriptor `remote_index_identity` change miss with
     `endpoint_identity_mismatch`.
  This is the right shape: each case targets one cache-key dimension
  and the mismatch case asserts the strict-rejection wording from
  packet 30710 inside the cache flow.
- Loopback executor assertions on the probe counts (3 entries, 4
  live identity queries, 1 hit, 4 misses) tie the probe outcome to
  the operator-visible summary surface from packet 30713 — so a
  drift between probe behavior and surface behavior would fail the
  test.
- Capability-block coverage extended to include `retention_gap` in
  strict and degraded modes (joining `stale_epoch` and
  `incompatible_extension_version` from packet 30711).
- Pre-dispatch capability-block fixtures now assert the cache state
  is fully zero (entries, queries, hits, misses, compact candidates,
  heap candidates) when capability gates trip — proves the cache
  cannot accumulate entries for known-stale or known-skewed remotes.
- ADR-058 cross-links to
  `plan/design/spire-libpq-identity-cache.md`, closing packet 30712
  P3 #3.
- Phase 11 Stage C records the expanded cache-matrix coverage.

## Answers to the packet's review questions

- **Closes 30712 P2 enough to continue Stage C cache work?** Yes.
  The probe exercises the four most load-bearing key dimensions
  (descriptor generation, served epoch, identity bytes, hit/miss
  bookkeeping) and integrates with the operator-visible counters
  rather than asserting against private state. That matches the
  "tests land before reuse expands" intent of 30712 P2.
- **pg-test-only probe acceptable without adding another production
  SQL surface?** Yes. A test-only probe avoids exposing a cache
  poke/inspect API to operators, which is the right discipline —
  operators have the existing summary surface; tests need finer
  control without that becoming a long-term contract.
- **Remaining matrix cases that should block the next Stage C
  slice?** One follow-up below; not blocking for this packet.

## P2 — close before broader cache reuse

1. **Degraded-mode live-fingerprint mismatch coverage.** The current
   matrix covers strict-mode mismatch via the descriptor identity
   change case. Packet 30712's contract requires that degraded-mode
   live-fingerprint mismatch produce `skip_node` with `next_blocker
   = remote_endpoint_identity` rather than fail closed. The
   capability-block fixtures cover degraded `retention_gap`, but the
   identity-mismatch-in-degraded case is not yet asserted on the
   cache flow. Recommend the next Stage C slice add that case
   (descriptor identity changes mid-flight + degraded consistency
   mode → skip_node, cache invalidates, no compact/heap candidates
   land). This is the inverse of the current strict mismatch case
   and would close the strict/degraded pairing for the cache.
