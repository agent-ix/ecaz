# Review Feedback: AVX2 FWHT 8-Lane Bootstrap

**Verdict: Approved — strong foundational win**

## In-register butterfly design

`fwht8_avx2_block` is the canonical SIMD approach to small-N FWHT: use permutes and blends to
execute each butterfly stage in registers without any memory round-trips.

Stage-by-stage verification:
- **Width 1** (adjacent-pair butterfly): `swap1 = [1,0,3,2,5,4,7,6]`, blend mask `0b10101010`.
  Each pair (0,1), (2,3), etc. gets sum in even slot, diff in odd slot. Correct.
- **Width 2** (quad butterfly): `swap2 = [2,3,0,1,6,7,4,5]`, blend mask `0b11001100`.
  Swaps pairs-of-2 within each quad. First two slots get sum, last two get diff. Correct.
- **Width 4** (half butterfly): `swap3 = [4,5,6,7,0,1,2,3]`, blend mask `0b11110000`.
  Swaps the two 4-element halves. Lower half gets sum, upper gets diff. Correct.

The permute indices build a complete 8-point FWHT in 3 stages × 3 instructions (permute, add/sub,
blend) = 9 SIMD instructions. The old path did 3 stages × width-many scalar butterflies with
load/store between each stage. At width=1 that's already bad because each "butterfly" is one
scalar add and one scalar sub on a value that just got stored to memory last stage.

## 57% FWHT improvement

This is the expected magnitude. The first three stages dominate at small widths because they touch
every element (same data volume) but with poor ILP and constant store-to-load forwarding penalties.
Collapsing them into registers eliminates those penalties.

## Dispatch structure

The `if values.len() < 8 / < 16 / < 32 / else` dispatch in `fwht_in_place_avx2` correctly selects
the widest applicable bootstrap. For the production case (dim=2048), this always hits the widest
branch. The conditions are evaluated once, not per-iteration. Clean.

## BC-008 progress

The packet claims this host now crosses the 3x BC-008 target at `fwht/2048` (4612.8 / 1444.4 =
3.19x). But it correctly cautions that BC-008 isn't closed since the scalar baseline was from
one particular run and hasn't been stable across measurements. The 20K-iteration warmed numbers
show 2.73x at the 32-lane stage, which is the most recent stable number. Honest reporting.

## Suggestions for closing the remaining BC-008 gap

The current 2.73x at dim=2048 means you need roughly another 10% improvement to hit 3.0x. The
remaining overhead is in the outer loop stages (width=32, 64, 128, 256, 512, 1024) — six full
passes over the 2048-element array. Three options worth considering:

1. **Deeper bootstrap to 64 or 128 elements.** Going to 64-point eliminates the width=32 outer
   pass entirely. The pattern is mechanical: apply `fwht32` to both halves of a 64-element chunk,
   then add/sub. Each additional level halves the overhead of one more full-array pass. The
   diminishing-returns point is around 64 or 128 — beyond that, the register pressure and code
   size may start hurting.

2. **Loop tiling / stage fusion for the outer loop.** Instead of running width=32, width=64, ...
   as separate full passes over the array, process a tile (e.g., 256 or 512 elements) and apply
   all stages up to the tile size before moving to the next tile. This keeps the working set in L1
   and eliminates redundant read-write cycles through L2/L3. For dim=2048 with 32-element tiles,
   you'd do one pass applying the 32→64→128→...→1024 stages per tile pair, then one final
   width=1024 pass. This is harder to implement but has the highest throughput ceiling.

3. **Hybrid: bootstrap to 64 + prefetch outer loop.** The simplest path that might close the gap.
   Going to 64-point gets ~5-10% from eliminating one outer pass, and adding `_mm_prefetch` on the
   right-half data in the outer butterfly loop could hide L1 miss latency on the stride-width
   accesses. Together these might reach 3x without the complexity of full tiling.
