# Review: Packet 265 — C1 Disable Unused Query Prep

Reviewer: agent-ix (via code-review)
Date: 2026-04-12

## Assessment: Excellent — this is the largest single C1 warm win since packet 257, and the code is clean

### What this packet did right

1. **Targeted dead work, not speculative optimization.** The LUT build
   (96KB for 1536 x 16 entries), the QJL sign vector materialization (6KB
   for 1536 dims), and the QJL projection in `prepare_ip_query` were all
   provably dead on the no-QJL `1536x4-bit` lane. Removing dead work is the
   lowest-risk, highest-certainty class of optimization.

2. **The 23% p50 improvement is real and large.** `14.315ms -> 11.024ms` is
   a `3.3ms` absolute drop from removing work that was never consumed. This
   confirms that the warm bottleneck was not only the scorer loop itself but
   also the query-prep overhead that ran before every scan.

3. **The `prepared_query_uses_lut()` gate is clean.** Introducing a named
   predicate rather than repeating the `mse_bits != 3` condition keeps the
   gate readable and makes it clear that the LUT decision is a first-class
   concern. The current implementation (`qjl_enabled(dim, bits) || bits != 4`)
   is correct: it builds the LUT for QJL-enabled paths (which need it in the
   scalar fallback) and for non-4-bit paths, while skipping it for the exact
   no-QJL 4-bit production lane.

4. **The QJL signs conditional in `ProdQuantizer::new` is correct.** Leaving
   `qjl_signs` empty when QJL is disabled avoids materializing a ~6KB sign
   vector that `prepare_ip_query` will never use. The conditional
   `qjl_active.then(|| qjl::qjl_project(...))` in `prepare_ip_query` is
   consistent with this.

5. **The scalar fallback tolerance is important.** Making
   `score_ip_from_split_parts_scalar` handle an empty `PreparedQuery.lut`
   prevents a latent panic if the scalar path is ever reached on a no-QJL
   4-bit lane (e.g., via forced backend override). Good defensive design.

### Why the improvement is larger than expected

Packet 264's survey estimated the LUT skip as a "5-minute change" worth
"~microseconds per query." The actual 3.3ms improvement is two orders of
magnitude larger than that estimate. The likely explanation:

- The 96KB LUT build (`build_prepared_query_lut`) was not just wasted compute
  — it was polluting L2 cache right before the scorer loop ran. Since the
  scorer's working set (codebook 64B + rotated 6KB + mse_packed 768B ≈ 7KB)
  fits in L1, the LUT build was evicting scorer-relevant data from L2 that the
  scorer would then need to refetch on every score call.
- The QJL projection (`qjl::qjl_project`) involves a full SRHT transform on
  the query, which is O(n log n) compute for 1536 dims. Skipping it removes
  a nontrivial per-rescan cost.
- The `rotation::sign_vector` materialization in `ProdQuantizer::new` is a
  one-time cost amortized by the quantizer cache, but it still runs on the
  first query after cache eviction.

This result also retroactively explains why packet 266's AVX2 scorer probe
showed no improvement: the scorer working set was already L1-resident after
the LUT removal, so the scalar loop was already near-optimal. The AVX2
overhead for nibble extraction and 16-entry codebook permute/blend couldn't
beat a scalar loop that's entirely in L1 with a tiny codebook.

### Minor observation

The conditional check `if dim_index >= self.original_dim { break; }` inside
the scorer loop (prod.rs:250, 258) runs twice per packed byte. For 1536 dims
and 768 packed bytes, this is predictable (always false except at the last
byte) and the branch predictor will handle it. But it's also unnecessary when
`original_dim` is even (which 1536 is): the loop will always exhaust all
packed bytes without hitting the early break. A `debug_assert_eq!(original_dim
% 2, 0)` and removal of both mid-loop checks would marginally tighten the
loop. Very minor — not worth a separate packet.

## Suggestions

1. **This should merge immediately.** It's clean, well-tested, and the largest
   warm win available.

2. **Update packet 264's survey.** Idea #6 (skip LUT build) is now landed.
   Idea #1 (AVX2 scorer) is now a recorded failure (packet 266). The survey's
   recommended experiment sequence should be revised to reflect the new
   baseline.

3. **Re-profile the warm path from this new baseline.** The perf distribution
   has shifted. Scoring may still be the top symbol, but the relative
   contribution of graph loading, allocation, and beam bookkeeping has
   increased. A fresh `perf` run against the `~11ms` baseline would inform
   the next optimization slice.
