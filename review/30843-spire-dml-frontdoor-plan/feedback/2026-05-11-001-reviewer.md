## Hook boundary direction (response to coder Q)

The three coder questions — answered:

### Q1: UPDATE/DELETE via planner/ModifyTable hook recognizing narrow bigint-PK equality?

**Yes.** A planner-level hook is the right v1 boundary, not row
triggers. Reasoning:

- Row triggers (`BEFORE UPDATE`, `BEFORE DELETE`) only fire when
  PG's executor finds a matching row in the coordinator heap.
  Remote-owned rows aren't there, so the trigger never fires —
  the DML silently reports `0 rows affected`. That is the
  wrong-results trap you correctly flagged.
- Statement-level triggers see the qual via transition tables but
  not the WHERE clause itself, so they can't extract the PK to
  route.
- A `planner_hook` (or `set_join_pathlist_hook` for a nested
  case) can inspect the `Query` tree, detect the narrow
  `UPDATE / DELETE FROM tbl WHERE pk_col = const` shape, and
  rewrite the plan to a `CustomScan` (or a custom `ModifyTable`-
  shaped node) that calls the existing primitive instead of the
  base heap path.

Concretely: install a `planner_hook` that runs after
`standard_planner` (or wraps it). For ec_spire-marked tables,
walk the `Query`:
- `UPDATE`: confirm exactly one rangetable entry, single-table
  qual `pk_col = const`, no subquery / join / RETURNING, all
  target-list entries are non-embedding columns. If yes, replace
  the plan with a no-op base scan + a `CustomScan` that calls
  `ec_spire_forward_coordinator_update_tuple_payload(...)` on
  Exec. If no, **error** — don't fall through to the base plan,
  because the base plan would silently miss remote-owned rows.
- `DELETE`: same shape detection; replace with CustomScan calling
  `ec_spire_prepare_coordinator_delete_tuple_payload(...)`.

The "fail closed if the shape is unsupported" rule is the
load-bearing safety property. Without it, an operator running an
unsupported UPDATE shape gets wrong results, not a clear error.

### Q2: PK-keyed SELECT via planner hook, OR view?

**Planner hook with CustomScan, not view.** Reasoning:

- A view (even an updatable proxy view) creates a recursive
  visibility surface and complicates `EXPLAIN` / cursor /
  prepared-statement semantics.
- The CustomScan path you already built for vector ORDER BY is
  the obvious template. For PK-equality SELECT against a marked
  table, the planner hook generates a CustomScan that calls
  `ec_spire_forward_coordinator_select_tuple_payload(...)` on
  Exec, skipping the base heap scan entirely. The CustomScan
  populates the scan slot from the JSON payload via the typed
  conversion path (or direct typed call once that lands).
- This unifies vector-ORDER-BY and PK-equality through one
  CustomScan-provider shape, which simplifies the operator
  mental model: any time the planner picks
  `Custom Scan (EcSpireDistributedScan)`, it's the SPIRE path.

### Q3: First implementation supports only the v1 narrow shape, fail closed otherwise?

**Yes.** That's the safety property that makes this approach
acceptable. Concretely:

- v1 supported: `WHERE pk_col = const_bigint`, single-table, no
  joins, no subqueries, no RETURNING, and for UPDATE only
  non-embedding columns in the target list (the embedding-UPDATE
  rejection is already on the primitive).
- Anything outside that shape: error with `ec_spire_distributed:
  <description> not yet supported in v1` plus a HINT pointing at
  the relevant ADR-069 section. **Do not fall through to the base
  plan** — that is where wrong results would slip in.

## Suggested next-packet shape

Recommend a narrow scaffold packet with these slices:

1. Add a `set_query_pathlist_hook`-equivalent (the right hook name
   for v1 PG18) that detects ec_spire-marked tables in
   UPDATE/DELETE/SELECT queries.
2. Add a "shape classifier" returning
   `(supported, kind, pk_value, projection_columns, error)`. Land
   this *first* with comprehensive tests — the classifier is the
   piece that's hard to get right.
3. Once the classifier is reviewed, add the executor wiring per
   operation in separate slices.

The diagnostic surface from this packet (`coordinator_dml_frontdoor_plan`)
should advance each row from `frontdoor_pending` to `ready` as the
hooks land, so operators can see incremental progress.
