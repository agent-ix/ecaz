# Review: Packet 277 — C1 Successor Scratch Buffer Reuse

Reviewer: agent-ix (via code-review)
Date: 2026-04-12

## Assessment: Reasonable direction — but the outcome needs recording before assessment

### Status

This packet has planned work and exit criteria but **no outcome section yet**. The request
describes the intent (scan-local reusable scratch vectors for successor/result staging)
and the motivation (packets 271 and 276 failed on similar allocation-reduction attempts),
but doesn't report whether the implementation was kept or discarded.

### On the planned approach

The plan is sound in principle: instead of allocating fresh `Vec<BeamCandidate>` per
expansion, clear and reuse a scan-lifetime scratch buffer. This avoids both the allocation
and the deallocation per expansion without changing ownership semantics (unlike packet 276's
inline heaptid approach, which changed struct layout).

Key considerations:

1. **Clearing is not free.** `Vec::clear()` preserves capacity and avoids reallocation, but
   the `drop` calls on the existing elements still run. For `BeamCandidate` (which is Copy
   or near-Copy — a TID + f32), this should be cheap.

2. **This should avoid packet 276's test instability.** The scratch buffer is scan-local
   state, not struct-level ownership. It doesn't change `GraphElement` or `TqElementTuple`
   equality semantics, so bootstrap-frontier tests should be unaffected.

3. **Expected yield is small.** Packet 271 showed that replacing the successor Vec with
   SmallVec (which also avoids the allocation for ≤16 entries) produced no measurable win.
   A scratch buffer reuse is architecturally cleaner but achieves the same thing: avoiding
   one small-Vec allocation per expansion. If SmallVec didn't help, scratch buffer reuse
   probably won't either.

### The allocation-reduction track may be exhausted at this granularity

Packets 271 (SmallVec: flat), 272 (batch buffer read: flat), and 276 (inline heaptids:
test instability) all failed. Only packet 270 (neighbor-tid iteration: 2.5%) succeeded.
The pattern suggests that the remaining per-expansion allocations are individually too
small to measure. The next meaningful allocation-reduction win likely requires a structural
change: zero-copy decode from pinned pages, or the SIMD approximate scorer (ADR-029)
which makes the per-score cost so low that allocation overhead becomes the dominant
fraction by default.

## Suggestions

1. **Record the outcome.** If the measurements are done, update the request with an
   outcome section so the keep/discard decision is captured.

2. **If flat, consider pausing the allocation-reduction track.** Three consecutive flat
   results (271, 272, 276 — with 277 likely flat too) suggests diminishing returns at
   this granularity. The remaining C1 headroom is in the scorer (ADR-029 SIMD) and
   possibly zero-copy decode, not in eliminating individual small-Vec allocations.

3. **Prioritize the SIMD int8 scorer next.** Packet 274 validated the rank correlation.
   Packet 275 showed that scalar 1.7x isn't enough for runtime integration. The critical
   path is now making the approximate scorer 10-50x faster via vpshufb/vpmaddubsw, which
   would change the entire cost structure and make the remaining allocation overhead
   measurable by comparison.
