This is **scaffolding, a seam, a catalog table, and tests**. It

the AM cursor blocking with `requires_remote_row_materialization`,

**zero operator-observable behavior change** at the SQL surface
unless something populates the catalog.

The packets are good. They are not enough.

## What "Stage D done" actually requires

The Stage D goal from the task file: *"make the coordinator-
visible result stream production-correct"* and end-to-end SQL
queries against `ec_spire` indexes with remote placements
return remote rows under MVCC.

For that to be true, three things still need to land:

### 1. End-to-end SQL fixture (likely in flight as packet 30799)

A PG18 fixture that:

- creates a coordinator `ec_spire` index with at least one
  remote placement on a real second cluster,
- inserts the base-table row that proxies the remote candidate,
- registers a mapping via
  `ec_spire_register_remote_row_materialization(...)`,
- issues a real
  `SELECT id FROM tbl ORDER BY embedding <-> $1 LIMIT k;`
  through PostgreSQL's executor,
- asserts the result set includes the remote-origin row with
  the `coordinator_materialized_heap` owner attribution.

Until this fixture exists and passes, **the end-to-end claim
"distributed SPIRE queries return rows" has no evidence**. The
existing 22 Stage E fixtures all exercise the executor state
machine or diagnostic SQL surfaces — none of them prove a
PostgreSQL `SELECT` returns remote rows.

Coder's WIP 30799 directory (`spire-am-materialized-remote-row`
with `test_ec_spire_prod_scan_am_delivers_materialized_remote_row`
in the artifact log) appears to be exactly this. Land it.

### 2. Operator-owned mirror sync mechanism (the gating slice)

This is the actual production-readiness gap. ADR-064 specifies
that materialized rows must be created by *epoch maintenance /
operator mirror lifecycle outside `amrescan` / `amgettuple`*.
ADR-065 specifies the catalog shape. Neither names the
mechanism.

Until something runs that mirror sync — replicating
remote-origin rows into the coordinator's base table and
registering the corresponding mappings — the catalog stays
empty in any real deployment. The register function exists,
but nothing calls it.

Concretely, this needs:

- **A design ADR** (call it ADR-066) selecting one of the
  candidate mechanisms:
  - background worker triggered on epoch publish,
  - explicit `ecaz` operator command driven by a runbook,
  - per-query lazy materialization (probably not — defeats
    the "no scan-time writes" rule, but might be permissible
    if it runs outside the scan cursor in a separate
    transaction),
  - logical replication subscription configured per remote
    placement,
  - or some other mechanism the coder chooses with reasoning.
- **The implementation** that runs the chosen mechanism.
- **A PG18 fixture** that proves end-to-end: epoch advance →
  mirror sync runs → catalog populated → `SELECT` returns
  remote rows without explicit `register_remote_row_materialization`
  calls in the test.

This is the slice that turns "remote rows can be returned if
something registers them" into "remote rows are returned in
production."

### 3. Coverage for the catalog under realistic conditions

The current tests for 30798 cover the register function and
batch-load happy path. Stage D needs:

- a stale-mapping case (catalog entry exists but coordinator
  heap row is gone — vacuumed, DELETEd, never inserted),
- a concurrent register + scan case (mapping registered
  mid-query — what does the in-flight scan see?),
- epoch retirement cleanup (mapping for retired epoch is
  removed when epoch retires, not kept indefinitely),
- catalog-size bounds (registered mappings don't grow
  unbounded; old epochs get cleaned),
- the operator-rotation case (mirror sync re-creates a
  mapping after the proxy row is vacuumed).

Some of these are covered by existing Stage E fixtures
indirectly (the `remote_heap_resolution_failed` path
exercises stale data). But Stage D's catalog-lifecycle
correctness needs its own fixtures.

## Priority order

1. **Land 30799** (end-to-end SQL fixture with explicit
   `register_remote_row_materialization` call). This proves
   the *mechanism* works.
2. **Land the mirror sync ADR + implementation + fixture**.

3. **Land the catalog-lifecycle coverage** (stale, concurrent,
   epoch retirement, bounds, rotation).
4. **Only then** open Stage F (multi-NVMe hardening) or
   Stage G (harness + runbook + AWS gate).

Stage F and Stage G are explicitly **scoped after Stage D
exit** in the task file's production landing sequence. Do not
start them until Stage D is feature-complete by the definition
above.
