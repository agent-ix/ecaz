# 03 — score_query_inner_product still allocates temporary payload

**Severity:** Medium  
**File:** `src/lib.rs:310–327`

## Finding

`score_query_inner_product` reconstructs a `(gamma || code_bytes)` payload `Vec` on every call and passes it to `score_ip_encoded`:

```rust
let mut payload = Vec::with_capacity(4 + codes.len());
payload.extend_from_slice(&gamma.to_le_bytes());
payload.extend_from_slice(codes);
Ok(quantizer.score_ip_encoded(&prepared, &payload))
```

The `ProdQuantizer` now exposes `score_ip_from_parts(prepared, gamma, code_bytes)` (added to address pass-1 item #16 for the scan hot path), which avoids this allocation entirely. The SQL-facing `tqvector_query_inner_product` still uses the old allocating path.

## Impact

For a SQL `ORDER BY` query over N rows, this allocates and drops one `Vec` per candidate. The scan path already avoids this allocation via `score_ip_from_parts`. The SQL scorer should use the same zero-allocation path.

## Suggested fix

```rust
fn score_query_inner_product(candidate: &[u8], query: &[f32]) -> Result<f32, String> {
    let (dim, bits, seed, gamma, codes) = unpack(candidate)...;
    ...
    let quantizer = ProdQuantizer::cached(dim as usize, bits, seed);
    let prepared = quantizer.prepare_ip_query(query);
    Ok(quantizer.score_ip_from_parts(&prepared, gamma, codes))
}
```
