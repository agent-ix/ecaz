# Review: SPIRE Typed Tuple Endpoint Scalar (30915)

## Summary

Slice 1 of the 30913 typed tuple transport design. Adds
`ec_spire_remote_search_tuple_payload_typed(...)` beside the
existing JSON endpoint. Implementation matches the design:

- Per-attribute binary I/O via dynamically-resolved `typsend`
  function from `pg_attribute` + `pg_type` + `pg_proc` /
  `pg_namespace` (`send_function_sql` is schema-qualified and
  identifier-quoted).
- Returns the design's typed projection arrays:
  `payload_attnums`, `payload_names`, `payload_type_oids`,
  `payload_typmods`, `payload_collations`, `payload_nulls`,
  `payload_values`, `payload_formats`.
- Emits `tuple_transport = 'pg_binary_attr_v1'` and
  `tuple_transport_status = 'ready'` for supported scalar rows.
- Reuses the same local heap candidate path as the JSON
  endpoint, so origin-node visibility behavior is identical.
- Out-of-band NULLs via `payload_nulls`; null payload values get
  empty bytea placeholder (length-checked against the column
  count).
- Per-row fail-closed length validation:
  `payload_nulls.len() != columns.len() || payload_values.len()
  != columns.len()` errors loudly.
- `tuple_payload_missing` flag preserved for visibility-skew
  cases.

PG18 fixture: scalar parity for `bigint` and `text`; typed bytes
compared to `int8send(...)` and `textsend(...)` — direct
byte-for-byte oracle.

JSON endpoint and the existing CustomScan receive path
unchanged. JSON regression test in the artifacts.

The three review-focus questions:
- Endpoint scaffold matches `pg_binary_attr_v1` shape from
  30913: yes — every named field present, in correct types.
- Dynamic SQL around per-column `typsend` is acceptable for
  first endpoint: yes (see findings below).
- Scalar fixture proves JSON parity without overclaiming
  array/composite/NULL/domain coverage: yes — 30916 follows up
  with NULL + array.

## Findings

### P2 — Dynamic SQL builds CASE expressions per column

`payload_value_exprs` builds a `CASE WHEN heap.X IS NULL THEN
''::bytea ELSE typsend.func(heap.X) END` per requested column
and joins them into one `ARRAY[...]::bytea[]`. For N requested
columns the SQL string size scales O(N * column_name_length).
For typical N≤30 this is fine; for very wide projections worth
profiling.

The alternative (one SPI call per column) would be N round-trips
instead. The current shape is the right tradeoff.

### P2 — `pgrx::spi::quote_qualified_identifier` for the send function

The send function name is built via
`quote_qualified_identifier(send_schema, send_name)`. Good —
this handles user-defined `typsend` functions in non-`pg_catalog`
schemas (e.g., custom range types, extension-provided types)
without identifier injection risk.

### P3 — `unsupported_type_binary_io` for missing typsend

Types without a `typsend` function (`InvalidOid`) fail closed
with `unsupported_type_binary_io for column "<name>"`. Matches
the 30913 design's named status label. Worth confirming the
error message includes both the column name AND the type OID
or name for operator diagnostics.

### P3 — Empty projection edge case

If `requested_columns` is empty, `projected_columns = ""`,
`found_projection = "true AS __ec_spire_found"`, and
`payload_null_array` / `payload_value_array` use the empty-
collection literals (`ARRAY[]::boolean[]` /
`ARRAY[]::bytea[]`). The fixture probably doesn't exercise
this; worth confirming the empty-projection case returns
metadata-only rows without erroring.

### P3 — `iter_deny_null` on the bytea[] column

`pgrx::datum::Array<&[u8]>` with `.iter_deny_null()` —
appropriately strict. A NULL element in `payload_values` would
indicate a SPIRE bug (empty bytea is the placeholder for SQL
NULL columns, not array NULL). Failing closed is correct.

### P3 — `tuple_transport_status` only `'ready'` in this slice

Slice 1 sets `tuple_transport_status = 'ready'` for all rows.
Future slices (30916+ for unsupported_type_binary_io paths,
later for schema_drift) will populate the other named statuses
from 30913. The status field is correctly future-extensible.

## Reviewer recommendation

Accept. Slice 1 lands cleanly: the typed endpoint is
production-ready for scalar columns; metadata is correct;
fail-closed paths are named per the design; JSON regression
unaffected.

Slice 2 (30916, NULL + array) follows on directly.
