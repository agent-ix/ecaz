# Feedback: Visible-Seed Top-Up Search Seam

**Reviewer:** Claude (Opus 4.6)
**Date:** 2026-04-13
**Commit reviewed:** `c196f2a`

## Verdict: Approve — correct seam extraction

This is a clean refactor that moves the visible-seed top-up path from scan
into a search-owned helper. The scope is narrow and the non-goals are
correct (graph-owned expansion, scan-owned visited/expanded sets, result
adjudication, and linear fallback all unchanged).

## Findings

### 1. Seam placement is correct

`search::VisibleFrontier::top_up_from_visible_seeds(...)` is the right
owner for this logic. The visible-seed top-up is a frontier-management
concern: it filters unexpanded seeds, checks remaining capacity, marks
expanded sources, and registers discovered candidates. All of those
operations are frontier-internal bookkeeping.

The scan side should be calling into the frontier with "here are visible
seeds, top up if you have capacity" — which is what this does.

### 2. Test coverage is appropriate

The pure search test covering unexpanded-seed selection, capacity
propagation, expanded-source marking, and successor seeding gives the
seam its own correctness surface independent of the full scan integration.
This is the right testing strategy for a seam extraction.

### 3. Scan-side semantic preservation

The request states that the scan-side rewiring preserves current runtime
semantics. Since this is a refactor (not a behavior change), the existing
benchmark and validation surfaces remain authoritative. No new measurement
is needed.

### 4. This is the right smallest slice before bootstrap fill-policy work

Extracting the top-up seam first means that any future fill-policy changes
(e.g., adaptive top-up budgets, priority-based seed selection) can be
implemented inside `VisibleFrontier` without touching scan.rs. That's a
clean dependency ordering.

## One minor note

The request doesn't mention whether `top_up_from_visible_seeds` handles
the case where there are zero unexpanded seeds (no-op path). This is
presumably covered by the test, but it's worth confirming that the helper
returns cleanly rather than doing unnecessary work when all seeds are
already expanded.

## Suggestion

No changes needed. This is a well-scoped seam extraction that sets up the
next layer of frontier-policy work cleanly.
