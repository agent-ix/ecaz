# Review: SPIRE Insert Descriptor Refresh (30836)

## Summary

Closes the manual descriptor re-registration caveat from 30834.
The remote-prepare primitive now probes the remote index identity
and epoch *inside* the prepared remote transaction (after the
INSERT, before `PREPARE TRANSACTION`), and the public helper
advances `ec_spire_remote_node_descriptor` in the coordinator
transaction before staging placement.

The refresh is gated on monotonic generation
(`WHERE descriptor_generation < $3::bigint`) so concurrent inserts
that race won't downgrade. If no row updates,
`descriptor_refresh_did_not_advance_descriptor_generation` errors and
the local txn aborts — the registered Abort callback rolls back the
remote prepared transaction. Multicluster fixture no longer calls
`ec_spire_register_remote_node_descriptor` after insert.

The three review-focus questions:
- Refresh in the public helper rather than the remote endpoint:
  yes — the remote endpoint stays a stable typed-INSERT surface;
  the descriptor refresh is a coordinator-side bookkeeping concern.
- Insert metadata probe weaker than the read-path validator: yes —
  decodes endpoint fingerprint and extension version without
  requiring rabitq read-readiness, so non-rabitq write targets
  aren't blocked.
- Local descriptor update ordered before placement staging and
  fails closed: yes; SPI returns false → helper errors → local txn
  aborts → Abort callback fires.

## Findings

### P2 — Generation-gated UPDATE handles concurrency but loses out-of-order writes

The `descriptor_generation < $3` predicate ensures only newer
generations win. If two concurrent INSERTs land at generations 5
and 6, the order matters: gen-6 commits first, gen-5 then sees
"did not advance" and errors. That's correct *for the descriptor*
(gen 6 is newer), but the gen-5 INSERT's local transaction aborts
even though its remote INSERT prepare succeeded. The Abort callback
then ROLLBACK PREPAREDs gen-5's remote transaction.

Net effect: under concurrent INSERT bursts, some INSERTs will
silently roll back due to descriptor-generation races. Worth one
of:
- Re-load the live descriptor row, compare generations explicitly,
  and if the live one is newer or equal *but compatible*, accept
  without UPDATE (the descriptor doesn't need to advance for this
  INSERT to succeed).
- Document this as expected v1 behavior and pin "concurrent INSERTs
  may roll back" in ADR-069 with the recommendation to retry.

The cheap fix is the first; the strict fix is to serialize INSERTs
per (index, node) in the helper. For v1 with low INSERT
concurrency the current behavior is acceptable, but the failure
mode is non-obvious and worth documenting.

### P2 — Descriptor probe runs inside the remote prepared transaction

The probe SELECTs from `ec_spire_remote_search_endpoint_identity`
within the same remote transaction as the INSERT. That's the right
ordering (probe sees the post-INSERT state). Worth confirming the
probe doesn't acquire locks that would interfere with concurrent
INSERTs from the same coordinator session — `endpoint_identity`
should be a read-only catalog lookup, but worth a one-line
confirmation.

### P3 — `last_seen_at = clock_timestamp()` and not txn time

`clock_timestamp()` returns the actual wall time, not the txn
start time. Since the local txn may not commit immediately, the
"last seen" timestamp is when the probe ran, not when the row was
made durable. That's the right choice for diagnostic purposes;
worth a one-line comment confirming this is intentional vs
`now()` / `transaction_timestamp()`.

### P3 — `last_error = 'none'` resets prior error state

Successful refresh clears `last_error`. Good — operators looking at
the descriptor row see clean state after a successful INSERT
through the helper. Worth confirming the field semantics: is it
"last error seen by any path" or "last error since last successful
INSERT"? Different operational meanings.

### P3 — Test fixture asserts descriptor_generation advanced and identity nonempty

`assert_eq!(descriptor_row, "17|true|true|true")` — clean shape
proof. The `17` is the new generation, the three booleans confirm
last_served_epoch / min_retained_epoch / nonempty identity.
Effective regression guard.

## Reviewer recommendation

Accept the descriptor refresh. The P2 concurrent-generation race is
the most operationally surprising behavior — fold a docs note into
ADR-069 in the next docs pass, and consider the live-row check
optimization when concurrent-INSERT throughput becomes a Stage F
benchmark target.
