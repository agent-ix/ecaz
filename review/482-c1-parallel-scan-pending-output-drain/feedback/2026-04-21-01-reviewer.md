## TL;DR for the agent

**Merged correctly.** Staging the full inline pending-output heap-TID
buffer in the shared worker-result slot and draining one heap TID at
a time (advancing in place when more pending remain, clearing only

per-slot advance preserves heap ordering without touching the heap
entries themselves. Version bump 6→7 is correctly paired with the

## Findings

### 1. In-place advance mutates slot runtime state under the "heap" lock

**Severity:** Informational
**File:** `src/am/common/parallel.rs:1800–1815`

The advance path stores `pending_index` and
`heap_block_number`/`heap_offset_number` while holding the heap lock.
That's correct for writer-writer exclusion, but readers calling
`load_coordinator_result_slot_snapshot` take no lock and can observe
interleaved state: `pending_index` advanced but `heap_tid` not yet
updated, or vice versa. `store_parallel_item_pointer` is two separate
atomic stores, so even the heap TID itself is non-atomic.

Today's readers (the coordinator fast-path in 483) validate the
snapshot against the cached pending output and refresh on mismatch,
so this is self-healing. Worth flagging for the amcanparallel=true
slice: either (a) reader-side validation must stay in place, or (b)
the advance must be atomic against readers via generation-based
retry (already present: `result_publish_generation.fetch_add`).

### 2. Lock name is becoming a misnomer

**Severity:** Low
**File:** `src/am/common/parallel.rs:910`

`acquire_parallel_scan_heap_lock` now covers not just heap-entry
mutation but also slot-runtime mutation (pending advance, heap_tid
store). The lock is effectively a "coordinator mutation" lock. When
the LWLock replacement lands (see 481 feedback), name it
accordingly — `acquire_parallel_scan_coordinator_lock` or similar —
so future readers don't assume it's scoped to the heap array.

### 3. Scan-side publish stages pending[0] as the live heap_tid

**Severity:** Informational (positive)
**File:** `src/am/ec_hnsw/scan.rs:1165–1171`

The publish path now picks
`pending_heap_tids[pending_index]` (falling back to
`current_result.heap_tid()` if invalid) as the staged `heap_tid`
rather than always using `current_result.heap_tid()`. This keeps the
slot's `heap_tid` fast-path field in sync with the current
pending-output slot across republish. Correct.

### 4. Pending-output take emits the current TID then advances

**Severity:** Informational (positive)
**File:** `src/am/common/parallel.rs:1772–1815`

The take snapshots `pending_output` first (from `pending_index`),
then advances to `pending_index + 1` (or clears if exhausted). That's
the right ordering — callers receive TID N, and the next take returns
TID N+1. The snapshot returned is the *pre-advance* state; that
matches the "consume then move cursor" convention used elsewhere.

### 5. Regression coverage is proportionate

**Severity:** Informational (positive)

Three focused tests: (a) drain within a slot without clearing, (b)
clear only after last pending TID, (c) scan-side staging of pending
buffer. All three lock down the right invariants. Nothing missing
for this slice.
