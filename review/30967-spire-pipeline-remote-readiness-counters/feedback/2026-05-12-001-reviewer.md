# Review: SPIRE Pipeline Remote Readiness Counters (30967)

## Summary

Phase 12.9 harness extension for remote readiness capture.
`ecaz bench spire-pipeline` now:

- reads `ec_spire_remote_search_endpoint_identity(...)`
  once per run and renders tuple transport
  capability/default/status + `pg_binary_attr_v1_ready`
  summary;
- when remote diagnostics enabled, calls
  `ec_spire_remote_search_degraded_skip_report(...)` per
  sampled query, aggregates by `(nprobe, node_id)`;
- reports degraded skip query count, requested epoch
  stability, skipped PID sum, first skip category, status.

CLI/report wiring; no live fixture. Tracker update is the
artifact-capture row sub-bullet.

## Findings

### P2 — Endpoint identity once per run

Right cadence — endpoint identity is per-remote and
relatively stable. Calling it once per run captures the
capability set at run-start; if it changed mid-run, the
degraded-skip output would catch the resulting failures.
No need for per-query polling.

The `pg_binary_attr_v1_ready` summary is the load-bearing
flag for the 30963 typed-receive cutover. An operator
running a benchmark and seeing `pg_binary_attr_v1_ready=false`
knows to investigate why the remote isn't advertising
typed before reading throughput numbers.

### P2 — Degraded skip aggregation by `(nprobe, node_id)`

Right granularity. From 30954 the degraded skip report is
per-skipped-dispatch (one row per `(node_id,
skipped_pid_count, first_skip_category, status)`). Adding
`nprobe` as outer key gives "at this nprobe, did this
remote degrade-skip more often?" — useful for spotting
nprobe values that exceed remote capacity.

`(nprobe, node_id)` is right granularity; not aggregating
by `first_skip_category` is also right because category is
**evidence about why**, not a key. Operators want to see
"node 2 skipped at nprobe=64, category=stale_epoch" —
flattening category into the key would multiply rows
without value.

### P2 — Reported counters

`degraded_skip_query_count`, `requested_epoch_stability`,
`skipped_pid_sum`, `first_skip_category`, `status` — covers
the operator-actionable axes:

- query count → frequency
- epoch stability → did the epoch move during the run?
- PID sum → cost (work that didn't go out)
- category + status → what kind and current state

Worth confirming `requested_epoch_stability` reports
*whether the epoch was stable across the run* (boolean or
"first vs last epoch" pair). If epoch advanced mid-run,
operators need to know — degraded skips after an epoch
advance are categorically different from steady-state
skips.

### P2 — Non-live wiring acceptable

Correctly scoped. CLI/report wiring without live fixture
artifact is the right shape for this slice — proves the
surface compiles and renders without committing to a live
artifact. The artifact-capture row only closes when a
packet runs the bench and produces evidence (the same
pattern as 30965 and 30966).

### P3 — Capability summary as bundle entry

`pg_binary_attr_v1_ready` is the single most useful
header field for the future production-readiness bundle —
an auditor looking at a bundle artifact wants the
capability surface visible at the top. Pin a tracker note
that the final readiness bundle (still-open) should
include the endpoint summary in its header section.

### P3 — Strict-mode behavior

The 30954 degraded skip report is degraded-mode only.
Worth confirming the bench doesn't accidentally call it in
strict mode and report empty results as "no skips" when
the actual mode wouldn't have skipped (it would have
failed). If the harness records the mode in the header,
this is self-resolving — operators know to interpret an
empty skip section by mode.

## Reviewer recommendation

Accept. Right cadence (endpoint once, skip per query),
right grouping (`(nprobe, node_id)`), right counter
selection, right header surface for future bundle
inclusion. Closes the remote-readiness portion of the
harness.
