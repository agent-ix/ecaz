# Review: Task 42 tuple fixtures (9046)

## Summary

Extends `fixtures/on-disk/` from metadata to HNSW and DiskANN
tuple/codebook payloads:
- `hnsw_element_tuple_v3.hex`
- `hnsw_neighbor_tuple_v3.hex`
- `hnsw_grouped_codebook_tuple_v3.hex`
- `diskann_vamana_node_tuple_v3.hex`
- `diskann_vamana_codebook_tuple_v3.hex`

Integration test decodes each, plus byte-swaps the DiskANN
Vamana node `neighbor_count` field and asserts the decoder
rejects the mutated fixture because the decoded count would
exceed `graph_degree_r`.

## Confirmations

- **Tuple coverage matches the layout-contract scope.** Every
  tuple kind that has a 9042/9043 size_of assertion now has a
  golden fixture too — same coverage at both gates.
- **`neighbor_count` mutation is the right targeted swap.**
  Validates that the decoder's count-vs-degree-bound check
  actually triggers on garbage input. Catches a real
  corruption class (truncated/expanded tuple).
- **Hand-pinned little-endian bytes.** Same convention as
  9045.

## Findings

No blockers.

### Pattern note

The `neighbor_count > graph_degree_r` rejection check is a
stronger end-to-end test than the metadata version-swap from
9045 — it exercises a semantic invariant of the codec, not
just a format-tag check. Worth adding similar semantic-invariant
mutations to subsequent fixtures (e.g., centroid tuples with
mutated dimensions, IVF posting tuples with mutated counts).
