# Review: SPIRE Tuple Transport Bench Knob (30964)

## Summary

Phase 12.2 throughput-measurement groundwork. Adds session-
level transport selector
`ec_spire.remote_tuple_transport`:

- `auto` (current production behavior — typed when remote
  advertises ready);
- `json_tuple_payload_v1` (forces JSON compat path);
- `pg_binary_attr_v1` (forces typed; still requires remote
  capability advertisement).

Plus `ecaz bench spire-pipeline --remote-tuple-transport`
flag and selected mode in benchmark header.

Does not claim throughput results or close the parent row.

## Findings

### P2 — Capability gate preservation under override

Critical detail: the typed override **still requires the
remote endpoint to advertise ready `pg_binary_attr_v1`**.
This means the `pg_binary_attr_v1` selector is not a
"force-typed regardless" knob — it's a "prefer typed and
fail if not available" knob.

That's the right design for a measurement selector: it
prevents an operator from accidentally setting the GUC and
breaking a production read against a remote that doesn't
support typed. If they want measurement without capability
gate, they need a remote that actually supports typed.

The JSON override does not have this gate (compat path is
always available by design), which is also correct.

### P2 — `auto` preserves production default

The `auto` mode preserves current behavior — typed when
remote advertises ready, JSON otherwise. Setting the GUC to
`auto` should be a no-op vs not setting it. Worth confirming
the test suite asserts this — i.e., that the default
behavior path through the negotiation is identical with
`auto` set vs unset.

### P2 — CLI flag and benchmark header

Printing the selected transport mode in the benchmark
header is the right primitive for packet-local reproducibility.
Future tuple-heavy measurement packets can have a
`json_tuple_payload_v1` run and a `pg_binary_attr_v1` run
with identical other parameters, and the artifact header
makes the comparison auditable.

If the bench output also writes the transport mode into
the per-row metric output (e.g., a `transport_mode` column
in the latency CSV), an analyst can mix runs and still
attribute correctly. Optional but useful.

### P3 — Tracker scope as groundwork

Correctly does NOT claim a throughput result. The tracker
update reflects measurement *infrastructure* in place, not
measurement performed. Honest. The actual throughput row
stays open until a packet runs comparable JSON vs typed
sweeps and reports the delta.

### P3 — Force-JSON regression hook

The `json_tuple_payload_v1` override is a useful
regression hook even after typed becomes default —
operators triaging "did this regress because of the typed
path?" can flip the GUC and re-test. Worth noting in docs
that the JSON override remains available throughout the
compat window AND should be retired together with the JSON
endpoint (otherwise the GUC outlives the path it
references).

### P3 — Future compatibility-window deadline

When the JSON endpoint retires (post-Phase 12.2 close),
both `json_tuple_payload_v1` and `auto`'s JSON-fallback
behavior need to be removed together. Worth pinning a
follow-up that the GUC value list and the negotiation
logic share a removal point. Otherwise an orphaned
`json_tuple_payload_v1` value will return a confusing
"capability not available" error rather than "this
transport mode no longer exists."

## Reviewer recommendation

Accept. Right groundwork primitive: capability gate
preserved on typed, no gate on JSON, `auto` preserves
default, CLI selector with header reporting. Sets up the
30965+ measurement packets to do honest before/after work.
