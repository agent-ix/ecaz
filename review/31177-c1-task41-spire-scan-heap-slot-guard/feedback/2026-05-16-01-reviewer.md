# Review: Task 41 SPIRE scan heap slot guard (31177)

## Summary

Wraps the heap tuple slot used by SPIRE heap rerank scan
candidate preparation in `src/am/ec_spire/scan/relation.rs`.
Introduces module-local `HeapTupleSlot` — owns a
`*mut pg_sys::TupleTableSlot` from `MakeSingleTupleTableSlot`,
drops via `ExecDropSingleTupleTableSlot`. Removes the manual
`ExecDropSingleTupleTableSlot` call after candidate prep.
Baseline 4,315 → 4,313 (−2); `scan/relation.rs` 36 → 34.

## Confirmations

- **Drop happens after last closure use.** The closure that
  iterates rows calls `slot.as_ptr()` per row to load source
  vectors; Drop runs at the slot's end of scope (after closure
  returns and outer function continues). ✔
- **No returned value borrows from the slot.** Candidate
  preparation returns owned `Vec`s and primitives; the slot is
  scratch space only. ✔
- **Per-row `ExecClearTuple` semantics preserved.**
  `load_indexed_source_vector_from_heap_row` still clears the
  slot between rows; the new code didn't change row-internal
  state management.
- **Behavioral safety improvement.** As the request notes,
  errors during candidate preparation now drop the slot through
  RAII rather than leaking it. Strictly safer than the prior
  shape (which had no error path that could exit before the
  manual `ExecDropSingleTupleTableSlot`, but the manual call
  pattern was a regression hazard).
- **Baseline arithmetic.** −2 (one open + one close site
  collapsed).

## Findings

### 1. (cross-cutting, see 31175) Second tuple-slot wrapper in
the codebase

31175 introduced `TupleTableSlotGuard` in
`src/am/ec_spire/custom_scan/planner.rs` six minutes before this
packet introduced `HeapTupleSlot`. Both wrap the same
`*mut pg_sys::TupleTableSlot` resource, both drop via
`ExecDropSingleTupleTableSlot`, both have the same SAFETY
contract. The naming differs (`TupleTableSlotGuard` vs
`HeapTupleSlot`) but the structure is identical.

This is the relation-guard proliferation pattern repeating, one
level out, for a new resource family. The strategic note in my
31175 feedback applies: snapshot, scan, and slot wrappers
should be promoted to `src/storage/` (or sibling files) before
they multiply further.

Suggested unified shape (matches the
`IndexRelationGuard`/`HeapRelationGuard` pattern):

```rust
// src/storage/slot_guard.rs (or extend relation_guard.rs)
pub(crate) struct OwnedTupleTableSlot {
    slot: *mut pg_sys::TupleTableSlot,
}
impl OwnedTupleTableSlot {
    pub(crate) fn make_single(
        tupdesc: pg_sys::TupleDesc,
        ops: *const pg_sys::TupleTableSlotOps,
    ) -> Self { ... }
    pub(crate) fn as_ptr(&self) -> *mut pg_sys::TupleTableSlot { self.slot }
}
impl Drop for OwnedTupleTableSlot { ... }
```

Both 31175 and 31177 collapse into one type. No functional
change.
