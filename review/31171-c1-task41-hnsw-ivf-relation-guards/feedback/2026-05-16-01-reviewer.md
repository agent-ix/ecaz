# Review: Task 41 HNSW/IVF relation guards (31171)

## Summary

Closeout slice for the HNSW/IVF compatibility helpers. Migrates the
last four production wrappers (`ec_ivf_index_page_ownership`,
`ec_ivf_index_cost_snapshot`, `ec_hnsw_index_cost_snapshot`,
`ec_hnsw_planner_integration_snapshot`) plus several HNSW test
callers in `src/tests/ec_hnsw_build.rs` and
`src/tests/ec_hnsw_recall_debug_exports.rs`, then **deletes**:

- `unsafe fn open_valid_ec_hnsw_index`
- `unsafe fn open_valid_ec_ivf_index`
- `AccessShareIndexRelation::into_raw` — the escape hatch from 31151.

Baseline 4,390 → 4,351 (−39); `src/lib.rs` baseline 185 → 177.

Removing `into_raw` is the consequential change: there is no longer
a supported way to get a raw `pg_sys::Relation` out of the guard
without going through `as_ptr()` (which borrows, never transfers
ownership). The "raw pointer takes over close obligation" path is
**gone**.

## Confirmations

- **Zero remaining raw callers.** `rg "open_valid_ec_(hnsw|ivf)_index\(|into_raw\("
  src/lib.rs src/tests` returns nothing at this HEAD. Deletion is
  safe.
- **HNSW tests hold guard across multiple raw-pointer graph reads.**
  In `test_pq_fastscan_graph_reads`, the guard opens once, then
  `am::graph::with_graph_storage_tuple(index_relation.as_ptr(), ...)`,
  `am::graph::load_grouped_graph_adjacency(...)`, and
  `am::graph::load_grouped_graph_element(...)` all run while the
  guard is alive. `drop(index_relation)` runs at the test's last
  statement. Same effective lifetime as the prior single
  `index_close`. ✔
  
  This is the reviewer-focus item that mattered most — graph
  loaders walk the index via raw pointer reads inside their
  unsafe blocks; closing the relation early would have produced
  dangling reads. The guard correctly stays in scope.
- **Cold-rerank payload test.** Same shape — guard opens once,
  `load_grouped_graph_element` and `load_grouped_rerank_payload`
  both run with guard alive, drop at the end.
- **Production wrappers.** All four use the standard explicit-`drop`
  idiom; AM call returns owned data, drop, then `TableIterator::*`.
- **`into_raw` deletion is safe.** No `into_raw(` callsites remain
  anywhere. The compatibility surface is gone.
- **Baseline arithmetic.** −39 across the migration + helper
  deletion is the expected mix of per-callsite retirements plus
  the helper-body retirement.

## Findings

No blockers.

### 1. DiskANN compatibility helper status

This packet closes HNSW and IVF; it does NOT touch
`open_valid_ec_diskann_index`. I called this out in the 31170
closeout note — `open_valid_ec_diskann_index` is still defined in
`src/lib.rs:~320` (or thereabouts) per the original 31151
introduction. Either it has no remaining callers and can be
deleted in a one-line follow-up, or there's a residual caller
worth migrating. The next slice should resolve.

### 2. `into_raw` removal — explicit milestone worth noting

This is the first time since 31151 that the relation guard cannot
escape its owning scope at all. The "raw pointer with no
lifetime" failure mode I keep flagging in reviews is now
**impossible to introduce via the guard's own API** — callers can
only use `as_ptr()` (borrowing). The remaining hole is callers
that themselves call `pg_sys::index_open` directly bypassing the
guard; the broader strategy needs to deny that pattern at lint
level eventually. The 921/31161 strategy doc should record this
as a structural improvement.

### 3. HNSW test guard lifetime is the strongest case for the
"long guard is acceptable" exception

`test_pq_fastscan_graph_reads` keeps the guard alive across 3
graph-loader calls plus assertions. This is exactly the case where
the long lifetime is necessary — the loaders return references
that point into the relation's pages, not owned copies. Worth a
one-line note in the 31161 strategy about "graph-loader callers
must hold guard across all loads; explicit `drop` only after the
last load completes."
