  `src/am/ec_spire/scan/callbacks.rs:68` now calls
  `production_scan_result_stream_am_outputs(&stream)` to

  `src/am/ec_spire/scan/types.rs`:

  - Owned by the runtime state struct via the `cursor`

  pass standalone.

## Answers to the packet's review questions

- **Is `SpireScanOutputCursor` the right scan-opaque contract
  for final `amgettuple` delivery?** Yes. Owning the cursor
  in the runtime-state struct gives `amgettuple` a stateful
  iterator across calls without re-running the production
  fanout, and putting it next to existing scan opaque state
  keeps lifetime tied to the scan descriptor. The `Default`
  impl at line 597 makes empty-result and not-yet-rescanned
  cases distinguishable from positioned cursors.
- **Does `amrescan` now depend on the production stream
  without accidentally allowing remote-origin heap
  coordinates into `xs_heaptid`?** Yes. The fail-before-
  `xs_heaptid` rule on any blocker, combined with the
  classifier-driven filter to local-only outputs, means
  remote-origin coordinates can't slip through even if the
  production stream returned them. Belt-and-suspenders with
  packets 30757/30758/30761.
- **Is the AM top-k default through the production stream's
  scan-plan candidate limit appropriate?** Yes for v1.
  Reusing the existing scan-plan candidate limit avoids
  introducing a separate AM-side knob that could drift from
  the executor-side budget. If a future packet needs an AM-
  specific bound (e.g. a planner-supplied LIMIT pushdown),
  it can layer on top of this default.
- **Are remote materialization blockers surfaced early
  enough for the remaining implementation slice?** Yes.
  Failing at `amrescan` rather than `amgettuple` means the
  blocker reaches PostgreSQL's executor at scan-start, not
  after partial tuple delivery — operator sees the error
  before any row is returned, which is the expected
  semantics for a hard contract violation.

## P3 — informational

1. **`SpireScanOutputCursor` advance-and-clear discipline.**
   `amgettuple` will need to advance the cursor and
   ultimately exhaust it. Worth confirming with a future
   test that the cursor's `Default` state, an exhausted
   cursor, and a re-`amrescan`'d cursor are all
   distinguishable in the runtime state — the current
   `Default` impl exists but I haven't seen the
   advance/exhaust paths yet. Will land naturally with the
   `amgettuple` slice.

2. **Stream-rebuild-on-rescan semantics.** PostgreSQL may
   call `amrescan` multiple times on the same scan
   descriptor (e.g. for parameter-driven re-runs). Worth
   confirming the production stream is rebuilt on each
   rescan rather than reused from a prior run; otherwise a
   second rescan with different scan keys could return the
   first run's stream. Optional check before parameter-
   driven scans land.
