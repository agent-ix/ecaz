## Note: Code Has Evolved Further

The intermediate wrapper types (`VisibleCandidateFrontier`, `VisibleCandidateFrontierRef`) described in this review have been absorbed into `VisibleCandidateFrontierState` by review 95. The methods (`clear`, `push`, `extend`, `remove_node`, `len`, etc.) are now directly on the state type. `visible_frontier_mut` returns `&mut VisibleCandidateFrontierState` rather than a wrapper.

## Response to Review Focus

### Is this container seam the right narrow boundary before deeper ownership transfer?

**Yes.** The container seam was the correct structural step. Whether it passed through intermediate wrapper types or landed directly on the state type, the key achievement is: no scan path directly mutates `Vec<ScanCandidate>` except through named methods on a container type. This means the Vec can be replaced (with a different backing store, or moved behind `search.rs`) without changing the mutation surface.

The current method set on `VisibleCandidateFrontierState` (scan.rs:327-389):
- `clear`, `push`, `extend` — write operations
- `remove_node` — TID-driven removal (encapsulates `position()` + `Vec::remove()`)
- `len`, `is_empty`, `iter`, `slot` — read operations
- `contains_node`, `best_tid_by_score` — query operations

This is the right minimum surface. Every operation scan needs is here; nothing extra.

### Are the chosen methods the right minimum surface?

**Yes.** Each method maps to a clear runtime need:
- `clear` — rescan reset
- `push` / `extend` — seeding via `seed_discovered_candidates`
- `remove_node` — consume via `take_candidate_frontier_node`
- `len` — capacity check in `top_up_bootstrap_frontier`
- `contains_node` — beam-node visibility check in `scheduler_best_frontier_node`
- `best_tid_by_score` — Vec fallback head in `current_candidate_frontier_head_tid`
- `iter` / `slot` — debug snapshot and positional access

No operation is missing, and none is dead weight.

### Do any remaining paths manipulate the frontier too directly?

**No.** All runtime mutation goes through `visible_frontier_mut(opaque)` (scan.rs:407-413) which returns the state type. All reads go through `visible_frontier_ref(opaque)` (scan.rs:399-405). The only remaining direct `.candidates` access is in `candidate_frontier_ref` (scan.rs:395-397) which returns a raw `&[ScanCandidate]` slice — used by some unit tests and internally for the old `VisibleCandidateFrontierRef` bridge. This is a minor residual.

## Additional Findings

No issues found. The container seam is clean and correctly staged.
