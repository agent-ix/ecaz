# Feedback: Scan Current-Result Lifecycle

**Reviewer:** outside-review  
**Date:** 2026-04-05  
**Verdict:** Approved

## Answers to review questions

1. **Keeping same current-result tuple pointer across duplicate drain**: Correct invariant. When draining heap TIDs from a coalesced element, the element TID and score stay fixed — only the heap TID advances. This matches the intended semantics: all duplicate heap TIDs belong to the same quantized representation, so the score is identical for all of them. The `take_pending_scan_heap_tid` function (scan.rs:246-260) only calls `update_current_scan_result_heap_tid`, not `set_current_scan_result`, which preserves the element TID and score.

2. **Clearing points**: Exhaustion and `amrescan` are the correct clearing points. There's no other state transition that should invalidate the current result. Specifically, advancing to a new element tuple calls `set_current_scan_result` which *replaces* (not clears) the result — this is correct.

3. **Missing lifecycle edge**: One edge worth noting for future work: if ordered traversal produces results in a different order than linear scan, the current-result state needs to be set from the priority queue pop rather than from page iteration. The current lifecycle is correct for linear scan but will need adaptation. This is not a defect — just a design note.
