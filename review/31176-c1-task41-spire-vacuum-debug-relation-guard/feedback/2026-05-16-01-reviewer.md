# Review: Task 41 SPIRE vacuum debug relation guard (31176)

## Summary

Wraps pg_test-only SPIRE vacuum debug relation opens in
`src/am/ec_spire/vacuum/mod.rs`. Introduces
`ShareUpdateExclusiveIndexRelation` — module-local guard, fixed
to `ShareUpdateExclusiveLock`. Keeps the index relation live
across `ec_spire_ambulkdelete` and (for the remove helper)
`ec_spire_amvacuumcleanup`. Baseline 4,319 → 4,315 (−4);
`vacuum/mod.rs` 39 → 35.

## Confirmations

- **Lockmode preserved.** The previous code used
  `ShareUpdateExclusiveLock`; the new guard hardcodes the same
  lock. ✔
- **`info.index` borrow lifetime correct.** The vacuum
  callbacks take `info.index` (a `pg_sys::Relation` pointer);
  the guard owns the index for the whole helper, so the borrow
  is valid across both calls. Drop at function return closes
  the relation. ✔
- **Null-open behavior is appropriate for debug helpers.**
  `pgrx::error!` on null is reasonable for pg_test-only code
  that expects a real index oid passed in by the test driver.
- **Baseline arithmetic.** −4 across two helpers ≈ 2 each
  (open + close). Consistent.

## Findings

### 1. (cross-cutting, see 31180 feedback) Now superseded by primitive

`ShareUpdateExclusiveIndexRelation` is **mechanically
replaceable** by:

```rust
use crate::storage::relation_guard::IndexRelationGuard;

let index = IndexRelationGuard::open(
    oid,
    pg_sys::ShareUpdateExclusiveLock as pg_sys::LOCKMODE,
    "debug_helper_name",
);
```

This packet predates 31180 by ~20 minutes, so the local struct
was the right call at write time — but post-31180 it should be
collapsed into the primitive. Tracked in the
"Migrate remaining module-local relation guards" follow-up.

No functional concern. The migration is correct; this is purely
a consolidation note.
