## Note: Line Numbers Updated

## Answers to Review Questions

### Is the removal of representative-heap gamma reads complete?

**Yes — fully complete.** Confirmed that `heap_tqvector_gamma` no longer exists anywhere in `src/`. The two hot paths are now:

1. **Scan scoring** (scan.rs:954): reads `element.gamma` from the decoded `TqElementTuple`, then passes it to `score_scan_element_result`. The gamma comes directly from the index page, not the heap.

2. **Duplicate detection** (insert.rs:319): compares `element.gamma.to_bits() == gamma.to_bits()` using the decoded element tuple. No heap fetch involved.

Both the graph-traversal candidate scoring path (`refill_candidate_frontier_from_source`, scan.rs:670) and the linear scan path (scan.rs:933) use persisted gamma via `GraphElement.gamma` and `TqElementTuple.gamma` respectively. No heap-derived gamma path remains.

### Is `score_ip_from_parts` the right long-term scorer API?

**Yes.** `ProdQuantizer::score_ip_from_parts(prepared, gamma, code_bytes)` (prod.rs:164-169) reads gamma and code bytes in-place without building an intermediate `[gamma][code_bytes]` buffer. This eliminates the per-candidate Vec allocation that was identified in earlier reviews. The scan hot path (scan.rs:965) calls it directly with a negation for the `<#>` convention.

The API is clean: `prepared` carries the query-side state, `gamma` and `code_bytes` are the element-side inputs. This is exactly the interface that ordered traversal needs — score a candidate from its index-stored parts without any intermediate allocation.

The quantizer-level tests (prod.rs:489-507) verify that `score_ip_from_parts` matches the payload-based scorer and that gamma actually influences the score. Good.

### Remaining correctness or lifecycle edges?

No remaining edges found. The gamma value is immutable once persisted (set at encode time, never updated except via element-tuple replacement during duplicate coalescing). The `to_bits()` comparison for duplicate detection handles +0.0/-0.0 and NaN correctly. The scoring path consumes gamma as a plain f32, which is the right type for the quantizer's affine correction term.

## Additional Findings

No issues found. This is a significant hot-path improvement — eliminating heap I/O per candidate is the critical enabler for efficient ordered traversal.
