# Review: SPIRE Local DELETE Placement Handling (30842)

## Summary

Closes the coordinator-local half of the DELETE primitive. After
30824 allowed `node_id = 0` in the placement catalog and 30839/30838
gave UPDATE / SELECT local-node branches, DELETE is now consistent.
A new shared heap-delete helper (factored out of the remote DELETE
endpoint) is reused by both the remote endpoint and the
coordinator-local DELETE branch.

For `node_id = 0` placements, `ec_spire_prepare_coordinator_delete_tuple_payload`:
- Deletes the row directly from the coordinator heap.
- Removes the placement-directory row in the same local
  transaction.
- Skips remote dispatch and `PREPARE TRANSACTION`.
- Returns `local_delete_applied` / `done`.

Remote-owned DELETE behavior unchanged.

The three review-focus questions:
- Local DELETE path should not use a remote prepared transaction:
  yes — there's no remote to coordinate with, and using 2PC for a
  pure-local operation would be both incorrect (no remote `PREPARE
  TRANSACTION` to issue) and overkill.
- Local heap delete + placement-directory delete in same local
  transaction is sufficient: yes — both writes are
  same-coordinator-database, so PG's normal atomicity covers them.
- Remote DELETE behavior preserved: yes; the shared helper
  refactor doesn't change the remote prepared path.

## Findings

### P3 — Local DELETE inherits the same `count == 1` strictness

The shared heap-delete helper presumably enforces the same
`deleted_count == 1` guard for the local path. If so, concurrent
DELETE races on local-node rows will fail loudly the same way the
remote path does. Same operational concern as the 30839 P2 (worth
ADR-069 pinning the policy: error vs idempotent-clean-up).

### P3 — Three local-vs-remote branches now (UPDATE/DELETE/SELECT)

Each primitive (UPDATE/SELECT/DELETE) has its own `node_id == 0`
branch with its own helper. Worth pinning a tracker note that the
front-door hook should consume these primitives via a single
local/remote dispatch shape so the three branches don't diverge.
The helpers themselves look uniform; the concern is more that
future maintainers might fix one branch and miss the others.

### P3 — Local placement DELETE removes the directory row

`DELETE FROM ec_spire_placement WHERE index_oid = $1 AND pk_value
= $2`. Standard. The unique `(index_oid, pk_value)` PK guarantees
one or zero matches.

## Reviewer recommendation

Accept. UPDATE/SELECT/DELETE local-node parity is now consistent;
DELETE is the last of the three primitives to gain it.
