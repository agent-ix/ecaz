## Answers to the packet's review questions

- **Is comparing descriptor `remote_index_identity` to live endpoint
  `profile_fingerprint` the right first production binding for
  Stage B?** Yes. The descriptor is the coordinator's stable view of
  what the remote should be serving; the live endpoint fingerprint is
  what the remote claims to be serving right now. Comparing the two
  catches the most common production failure modes — descriptor
  staleness after remote rebuild, remote epoch advance without
  descriptor refresh, and accidental cross-shard misregistration —
  without needing additional protocol fields.
- **Acceptable that SQL-visible connection-plan diagnostics expose
  only byte count while internal dispatch carries bytes?** Yes. The
  bytes are an exact-equality token between coordinator and remote;
  exposing them via SQL would invite operators to compare them by
  hand, and the libpq executor already does that comparison
  authoritatively. The byte count keeps the operator-facing surface
  honest about whether a binding exists at all.
- **Does the strict mismatch fixture cover the right boundary?** Yes.
  It runs through the full preflight on a ready RaBitQ remote with
  the wrong descriptor identity, so this is exclusively the
  binding-side correctness signal.

## P2 — close before Stage C cache lands

1. **Identity-byte refresh policy.** Once Stage C caches validated
   remote index identity, the descriptor↔endpoint binding becomes
   "the cache holds a known-good fingerprint snapshot for a known
   epoch." Recommend documenting in Stage C's cache contract:
   - cache key includes `(node_id, remote_index_identity,
     served_epoch)`;
   - any dispatch that observes a different live fingerprint
     invalidates the entry and forces a reseat through the descriptor
     refresh path;
   - registering or updating a remote-node descriptor invalidates the
     cache entry for that node.
   Without this, Stage C can ship a cache that masks an
   `endpoint_identity_mismatch` and no test will catch it.
