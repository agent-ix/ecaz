## TL;DR for the agent

**Merged correctly.** Caching the selected pending output directly in
the coordinator state (heap TID + score + optional approx/comparison/
rank) is the right next seam — readers now get the global next
pending output from one coordinator load instead of reconstructing it
from the selected slot. The within-slot advance → fast-path refresh
fix is a real bug fix, not just an optimization. One non-blocking
naming hazard: the coordinator flag word now overlaps with the
result-slot flag namespace.

## Findings

### 1. Coordinator flag word now overlaps with the result-slot flag namespace

**Severity:** Low (maintenance hazard)
**File:** `src/am/common/parallel.rs:759–797, 1203–1215`

`load_coordinator_snapshot` reads bits out of `coordinator.flags` and
interprets them using result-slot flag constants:
`EC_PARALLEL_RESULT_SLOT_APPROX_SCORE_VALID` (1 << 2),
`EC_PARALLEL_RESULT_SLOT_COMPARISON_SCORE_VALID`,
`EC_PARALLEL_RESULT_SLOT_APPROX_RANK_VALID`. The writer
(`refresh_coordinator_selected_fast_paths_locked`) stores those same
bits into `coordinator.flags`. Bit 0 (`COORDINATOR_SELECTED_RESULT_VALID`)
and bit 1 (`COORDINATOR_SELECTED_PENDING_OUTPUT_VALID`) coexist with
bits 2/3/4 which are defined for the result-slot flag word.

Functionally correct — both writer and reader use the same bit
positions — but this couples two flag namespaces that were previously
separate. Bits 0 and 1 of the coordinator flag word now also have
collision potential with `EC_PARALLEL_RESULT_SLOT_PUBLISHED` (1 << 0)
/ `EC_PARALLEL_RESULT_SLOT_SCORE_VALID` (1 << 1) if anyone ever
reuses the constants cross-word.

Recommendation: introduce dedicated coordinator-flag constants for
the pending-output validity bits
(`EC_PARALLEL_COORDINATOR_PENDING_APPROX_SCORE_VALID`,
`..._COMPARISON_SCORE_VALID`, `..._APPROX_RANK_VALID`) rather than
reusing slot-scope constants in coordinator-scope reads/writes. Same
bit positions are fine; the name is the hazard.

### 2. `coordinator.flags.store(...)` overwrites the whole word

**Severity:** Low
**File:** `src/am/common/parallel.rs:1216, 1252`

Both the success and None branches now use `.store(...)` on
`coordinator.flags` rather than the earlier `fetch_or` / `fetch_and`
combination. This is fine today because the coordinator flag word
only carries the selection and pending-output validity bits. But
any future flag added to `coordinator.flags` that isn't scoped to
"selection state" will be clobbered on every fast-path refresh.

Add a short comment at the store site ("coordinator.flags currently
only carries selection validity; add new flags here when that
invariant changes") or switch to explicit set/clear of only the
selection-related bits.

### 3. Within-slot advance fast-path fix is a real bug fix

**Severity:** Informational (positive)
**File:** `src/am/common/parallel.rs` take pending-output path

The packet claims the prior slice could leave the coordinator cached
pending output on the *just-emitted* TID after a within-slot advance.
Confirmed: in 482 the take path advanced `pending_index` and
`heap_tid` on the slot but did not refresh the coordinator fast path,
so a subsequent `read_parallel_scan_selected_pending_output_snapshot`
— added in this slice — would have seen the stale cached TID. The
fix correctly refreshes `refresh_coordinator_selected_fast_paths_locked`
after the in-place advance. Good.

### 4. Rank round-trip via `to_ne_bytes` / `from_ne_bytes`

**Severity:** Informational
**File:** `src/am/common/parallel.rs:792–797, 1191–1196`

`i32` ↔ `u32` via `to_ne_bytes`/`from_ne_bytes` is correct and
portable but verbose. `i32 as u32` / `u32 as i32` or
`i32::cast_unsigned` is equivalent and a lot more readable. Pure
style; not blocking.

### 5. Score comes from `selected_result_score_bits`, not a dedicated pending score

**Severity:** Informational
**File:** `src/am/common/parallel.rs:769–774`

`EcParallelPendingOutputSnapshot.score` is loaded from
`selected_result_score_bits` (the selected result slot's score),
not a dedicated `selected_pending_score_bits`. That's the right
reuse — within a slot the score is constant across pending TIDs;
only approx/rank/comparison vary. No bug; flagging because an
unfamiliar reader might wonder why pending-output doesn't have its
own score field.

## Answers to the packet's review questions

- **Q1** (caching selected pending output in coordinator state the
  right seam?) — Yes. One lock-free coordinator load now yields
  both "which slot" and "which TID" — the natural read surface for
  the eventual shared top-K admission path.
- **Q2** (fast-path refresh owns the within-slot-advance contract?)
  — Yes. The new regression pins it. Without this fix, the cached
  fast path would lag one TID behind.
- **Q3** (direct read helper narrow enough?)
  `read_parallel_scan_selected_pending_output_snapshot` is the
  minimum surface: reads the cached output, validates against the
  slot, refreshes on mismatch. Does not take. Won't have to change
  signature when admission lands.
