# Pre-Checkpoint Review: SPIRE DML PK Extraction Centralization (30874)

## Note

Code committed in `bcb8de5f Centralize SPIRE DML PK CustomScan
extraction` but no `request.md` exists yet. Pre-checkpoint
feedback per `review/REVIEWER.md` so the coder sees feedback
before the packet is finalized.

## Summary

Closes the 30873 P1 #2: the baserestrictinfo-aware PK predicate
walker that previously lived in `custom_scan.rs` is moved into
the existing `dml_frontdoor.rs` scaffold as
`dml_frontdoor_pk_select_primitive_plan_expr_from_baserel(root,
rel)`. `custom_scan.rs` shrinks by ~190 lines and now consumes
this helper instead of re-implementing predicate extraction.

The new helper:
- Takes `PlannerInfo*` + `RelOptInfo*` (the planner-side inputs
  available from `set_rel_pathlist_hook`).
- Confirms the query's `commandType` is `CMD_SELECT`.
- Resolves the target relation OID from the rangetable via the
  rel's `relid` (not via `resultRelation`, since SELECT doesn't
  set that).
- Loads the catalog-backed relation context (no SPI).
- Walks `baserestrictinfo` first (planner-normalized
  RestrictInfo list — what the hook sees), falling back to
  `jointree.quals` (what the existing scaffold's analyzed-Query
  walker sees).
- Returns `Option<Result<SpireDmlFrontdoorPrimitivePlanExpr,
  String>>` so callers can distinguish "not a candidate" from
  "candidate but malformed."
- Composes through the existing scaffold:
  `dml_frontdoor_replacement_decision_from_shape` →
  `dml_frontdoor_primitive_plan_from_replacement_decision` →
  `SpireDmlFrontdoorPrimitivePlanExpr`.

The two PK-extraction paths now share the same predicate walker;
divergence risk is closed.

## Findings

### P2 — Two `dml_frontdoor_pk_predicate_*` helpers

The diff shows `dml_frontdoor_pk_predicate_from_baserestrictinfo`
(new) plus the existing `dml_frontdoor_pk_predicate` (which
walks `jointree.quals`). The new
`dml_frontdoor_pk_select_query_detail_from_baserel` tries the
baserestrictinfo helper first, falling back to the jointree
helper:

```rust
let predicate = dml_frontdoor_pk_predicate_from_baserestrictinfo(...)
    .unwrap_or_else(|| dml_frontdoor_pk_predicate(query_ref, ...));
```

That's the right precedence: planner-normalized RestrictInfo
takes priority when the hook is invoked from
`set_rel_pathlist_hook`. The classify-from-Query path (used by
the diagnostic `_sql` surface and the hook's observation pass)
still uses `dml_frontdoor_pk_predicate` for parity with the
analyzed-Query expectations.

Worth a one-line comment naming why the two walkers exist:
"baserestrictinfo for hook-time planner inputs, jointree.quals
for the SQL diagnostic that operates on a fresh analyzed Query."

### P2 — `target_rtindex = i32::try_from(rel.relid).ok()?`

`rel.relid` is `pg_sys::Index` (u32). `i32::try_from` will fail
for relids > i32::MAX (~2 billion), which is implausible but the
function silently returns `None` on overflow. Worth either a
panic with context or documenting that overflow → not a
candidate.

### P2 — Catalog-backed `relation_context_catalog_row` could fail-closed

If the catalog loader returns `Err`, the helper returns `Some(Err(...))`
with a generic message. The caller (`custom_scan.rs`) can then
fail closed. Worth confirming the planner hook surfaces this as
an ADR-069 planner error (matching the existing fail-closed
guard from 30861) rather than silently passing through.

### P3 — `dml_frontdoor_query_has_join_shape(query_ref, PkSelect)`

The shape classifier now calls a per-operation join-shape
helper. Worth confirming the implementation matches the existing
`single_table` detection (which checks `jointree.fromlist`
length and types). If they diverge, the centralization fix
introduces a different drift.

### P3 — `custom_scan.rs` shrinks by ~190 lines

That's exactly the right outcome — the duplicated
predicate-walk logic moves into the scaffold module and the
hook code becomes a thin caller. Healthy.

### P3 — `Option<Result<...>>` return type

Three states:
- `None` — not a DML PK SELECT candidate (cheapest reject).
- `Some(Err(_))` — looks like a candidate but failed to
  extract.
- `Some(Ok(plan_expr))` — ready for the CustomPath builder.

Clear and matches PG idioms. Worth pinning that the hook should
treat `Some(Err(_))` as a planner error (not a silent skip) so
malformed candidates surface loudly.
