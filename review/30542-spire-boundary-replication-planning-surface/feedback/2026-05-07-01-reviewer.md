Reloption parsing, diagnostic exposure, and the route-map helper land
cleanly. No blocking findings.

## Confirmations

1. Driving `scan_dedupe_mode` from `boundary_replica_count > 0` at plan
   resolution time keeps the default primary-only scan path free of
   `HashMap` allocations — verified against
   `scan/candidates.rs::rank_routed_leaf_rows_by_ip` (line ~695:
   `NoReplicaDedupeDisabled => None`).
2. `SpireSingleLevelRouteMap::route_boundary_assignment_for_vector`
   (`build/routing_plan.rs:120-166`) is the right place to centralize
   primary + bounded secondary selection. The PID-already-seen guard
   (line 145) protects against degenerate routing tables that surface the
   same PID twice — necessary because the build callers later trip
   `replica pids must not include primary pid` if they did not.
3. SQL options snapshot now exposes the three columns operators need to
   see whether an index is replica-capable.

## Findings

- **Minor — `validate_route_vector` zero-norm threshold.**
  `build/routing_plan.rs:275` rejects when `norm_sq <= f64::EPSILON`
  (~2.2e-16). For f32 inputs that is effectively "exact zero" rejection.
  Fine for now, but if normalized embeddings ever underflow during
  preprocessing, the boundary path will produce an error that the primary
  path does not (because the primary path no longer flows through this
  helper for `boundary_replica_count = 0` if the design decision were
  ever revisited). Consider raising the threshold to something tied to
  `f32` precision (`f64::from(f32::EPSILON).powi(2)` or similar) to
  match upstream HNSW tolerance.

- **Cosmetic — repeated `usize::try_from(boundary_replica_count)`** in
  `route_boundary_assignment_for_vector`. The same conversion appears
  three times (`saturating_add(1).min(...)`, the `selected_pids.len() ==`
  loop bound, etc.). Hoist once for readability.
