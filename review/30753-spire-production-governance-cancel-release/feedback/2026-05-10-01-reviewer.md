## Answers to the packet's review questions

- **Acquiring governance permits inside the production async
  adapter, before conninfo parsing and socket open, the right
  Stage C ownership boundary?** Yes. The "fail before doing
  expensive work" pattern is correct here — conninfo
  resolution and socket open both have their own latency and
  failure modes; gating on governance first means saturated
  fanout doesn't pollute conninfo or connect diagnostics with
  unrelated failures, and doesn't burn coordinator I/O on
  remotes that won't run anyway. Mirrors the same precedence
  rule packet 30715 established for budget gates.
- **`remote_executor_overload` strict/degraded actions correct
  for C5 to consume from the production fault matrix?** Yes.
  Strict fail-closed is correct because the request specified
  more concurrency than the coordinator allows; silently
  proceeding would violate the cap. Degraded skip is correct
  because degraded mode opted into "best-effort with whatever
  ready remotes responded" — a saturated node is just another
  reason a node didn't respond, identical from the AM scan's
  perspective to a stale-epoch skip. Both behaviors slot
  cleanly into the C5 matrix consumer.
- **Local-cancel tests prove RAII release for global and
  per-node permits before C5 routes the AM scan path?** Yes.
  The two `_local_cancel_remote_cancel` tests exercise the
  fault path that's hardest to get right: cancel during a
  held permit. If RAII release weren't correct, those tests
  would either deadlock subsequent test attempts (permit
  leak) or fail subsequent fixtures with phantom overload
  (permit not released). Both tests passing through the SQL
  wrapper proves the release path holds across local-cancel
  early returns.
