# Review: Task 41 SPIRE page read buffer guard (31230)

## Summary

**Major milestone:** introduces `LockedBufferGuard` in
`src/storage/buffer_guard.rs` — the **first buffer/page/WAL
primitive in the shared module**. Uses it for three read-only
SPIRE object page helpers (`read_root_control_page`,
`read_partition_object_page`, `read_placement_directory_page`).
This is the start of Task 41 invariant #3's buffer/page/WAL
subtrack — the next major track the 31161 strategy named after
relation-resource closeout.

## Confirmations

- **Primitive design is clean.**
  `LockedBufferGuard::read_main(rel, block, mode, lockmode) ->
  Option<Self>` combines `ReadBufferExtended` +
  `BufferIsValid` check + `LockBuffer` in one call. Drop runs
  `UnlockReleaseBuffer` (which both unlocks and releases the
  pin in one PG API call). ✔
- **API surface.** Three accessors:
  - `page() -> pg_sys::Page` — for typical page header /
    tuple reads
  - `buffer() -> pg_sys::Buffer` — for callers that need the
    Buffer handle directly
  - `page_size() -> usize` — convenience for sizing checks
  
  All three are safe-fn dereferences of `unsafe` PG calls,
  with SAFETY comments anchoring the "guard owns valid locked
  buffer" invariant.
- **`unsafe fn` constructor.** `read_main` is `unsafe fn`
  because the caller supplies the relation pointer — the
  primitive can't verify relation liveness statically. Same
  pattern as the relation guards' `try_open`. ✔ Correct: this
  is a "caller knows the relation is live" assertion.
- **Three migrations are mechanical.** Each
  `read_*_page` helper opens the guard, reads through
  `guard.page()`, drops at function return. Same shape as the
  relation guard migrations.
- **`UnlockReleaseBuffer` chosen correctly.** Combines
  `LockBuffer(BUFFER_LOCK_UNLOCK)` and `ReleaseBuffer` — the
  right Postgres API for a guard that owns both pin and lock.

## Findings

### 1. (significant milestone) Buffer/page/WAL subtrack started

This is the **seventh shared primitive** in `src/storage/` and
the first one in the buffer family. Task 41 invariant #3
(`plan/tasks/41-ffi-safety-boundary.md:91-98`) names four RAII
wrapper targets: `BufferPin`, `LwLockGuard`, `Snapshot`,
`SpiTuptable`. The snapshot family already has two guards
(`ActiveSnapshotGuard`, `RegisteredSnapshotGuard`). This packet
starts the buffer family.

`LockedBufferGuard` is more specific than the task doc's
`BufferPin` — it combines pin + lock, while `BufferPin` would
be pin-only. That's a deliberate choice: every current SPIRE
read path acquires the lock immediately after the pin, so a
combined guard is cleaner than two separate guards. If a
pin-only access pattern emerges, a sibling `BufferPinGuard`
can be added without disturbing this.

### 2. (recommendation) Consider sibling primitives

Three follow-ups likely needed to cover the rest of the buffer
family:

- **`PinnedBufferGuard`** — `ReadBufferExtended` + `ReleaseBuffer`
  only, no lock. For paths that pin a buffer to walk it without
  needing exclusive access.
- **`WriteBufferGuard`** or extended `LockedBufferGuard`
  variants — for pages opened with `BUFFER_LOCK_EXCLUSIVE`.
  Either a separate type or constructor variants on
  `LockedBufferGuard::read_main` that take the lockmode.
  Actually re-reading the API: `read_main` already takes
  `lockmode: i32`, so it can construct an exclusive-locked
  guard. `read_main` is the constructor name though — a
  `write_main` alias / constructor for exclusive locks would
  read more idiomatically.
- **`GenericXLogState`** wrapper — for the WAL-write path
  (`GenericXLogStart` / `GenericXLogFinish` / `GenericXLogAbort`).
  This is the WAL part of "buffer/page/WAL" — bigger refactor
  but the same shape.

None of those block 31230; they're queued territory.

### 3. Production read paths

The three migrated helpers (`read_root_control_page`,
`read_partition_object_page`, `read_placement_directory_page`)
are SPIRE production read paths — every SPIRE scan reads the
root control page, partition object pages for child fan-out,
and the placement directory. So the primitive is exercised by
real workloads on day one.
