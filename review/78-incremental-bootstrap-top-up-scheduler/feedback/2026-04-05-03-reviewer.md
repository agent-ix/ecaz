## Answers to Review Questions

### Does the incremental scheduler preserve the current bootstrap expansion contract?

**Yes.** Verified: `top_up_bootstrap_frontier` (scan.rs:592-616) uses `bootstrap_expansion_mut(opaque).expand_one(|_| empty())` to select the next source to expand. This replaces the previous pattern of calling `next_bootstrap_expand_index` on each iteration. The expansion contract is preserved:
1. Pop the best unexpanded candidate from the beam scheduler
2. Check `expanded_contains_source` — skip if already expanded (scan.rs:610-611)
3. Mark expanded and refill from that source's adjacency (scan.rs:613-614)
4. `seed_discovered_candidates` (called inside `refill`) adds new candidates to both the frontier Vec and the beam scheduler

The loop terminates when frontier capacity is reached or the beam scheduler is empty. This is semantically identical to the previous behavior but routes source selection through the beam scheduler's `BinaryHeap` instead of linear scan.

### Is the interaction between `expanded_source_tids` and the beam scheduler sound?

**Yes.** The `expanded_source_tids` set and the beam scheduler track different things:
- `expanded_source_tids`: which element TIDs have been used as expansion sources (to avoid redundant adjacency loads)
- Beam scheduler `visited`: which element TIDs have been scored and added to the frontier (to avoid duplicate candidates)

The check at scan.rs:610 (`expanded_contains_source`) guards against re-expanding a source that `expand_one` popped from the beam scheduler. This can happen when the beam scheduler holds a node that was already expanded during a prior refill cycle. The `continue` skips the redundant expansion correctly.

One subtlety: `expand_one` permanently removes the node from the beam scheduler (it calls `pop_best` internally). So a skipped source is gone from the scheduler — it won't be popped again. This is correct behavior: the node was already expanded, and the scheduler shouldn't waste time on it again.

### Is this the right step before moving more frontier ownership out of `scan.rs`?

**Yes.** This slice establishes the beam scheduler as the source-selection authority while keeping the frontier Vec as the storage and materialization authority. The next natural step is to make the beam scheduler also authoritative for head selection and consumption (reviews 81-82), then eventually replace the frontier Vec entirely.

No correctness issues found. The incremental scheduler is a clean integration step.
