## One concern that survived this packet — constructor leak

Reviewing `SpireRelationObjectStoreSet::for_index_relation_and_config`
(`storage/relation_store.rs:1098–1143`) and the parallel
`for_index_relation_and_placements` (line 1145–1196):

```rust
let mut opened_relations = Vec::new();
for descriptor in &config.stores {
    ...
    let relation = unsafe { pg_sys::relation_open(relid, lockmode) };
    if relation.is_null() {
        return Err(format!(...));     // <-- partial Vec dropped here
    }
    opened_relations.push((relation, lockmode));
    ...
}

Ok(Self { ..., opened_relations })
```

`Vec<(pg_sys::Relation, LOCKMODE)>` has no custom Drop — the inner
tuple type is `(*mut RelationData, i32)`, neither of which carries a
destructor. So if the loop succeeds for stores 0..k and then errors
on store k+1, the `Vec` is dropped without calling `relation_close`
on the k relations already opened. Those relations stay open with
their locks held until the backend exits or the transaction
commits/rollbacks (which closes them via the resource manager —
but that's *implicit* cleanup the code is silently relying on).

The Drop impl on the constructed `Self` is correct and was the right
addition. The bug is that the constructor's local `Vec` is not the
same thing as `Self.opened_relations` — it has no destructor until
it gets moved into Self.

Fix: wrap `opened_relations` in a small guard struct with its own
Drop that closes any not-yet-moved relations, then `mem::take` (or
similar) into Self on the success path. Or, more simply, restructure
so `Self` is constructed incrementally and the Drop on the partial
`Self` handles cleanup. Trivial change, prevents a real-but-rare
leak.

In practice this only triggers when a relation OID becomes invalid
between the placement-directory read and `relation_open` (concurrent
DROP, catalog corruption). Backend transaction abort *will*
eventually clean up via the resource owner. So this is a
defense-in-depth fix, not a critical-path bug — but it is the kind
of thing that bites in tests under contention, and the cleanest
landing shape is to fix it before it becomes habitual code style
elsewhere.

Not blocking on this packet. Worth a small standalone packet next.

## REINDEX tracker — confirmed

The task tracker entry at line 1346:

> **Explicit multi-store REINDEX lifecycle.** Multi-store REINDEX
> must create and publish a fresh auxiliary-store relation set, then
> retire stale auxiliary store relations. Internal dependencies are
> not enough to make existing auxiliary stores participate correctly
> in PostgreSQL REINDEX.

This is the right framing. One additional ask before landing Phase 4:
**reject REINDEX explicitly with a clear ereport** until the lifecycle
is implemented. Currently a user running `REINDEX INDEX
my_spire_idx` against a multi-store SPIRE index would get a
quietly broken result — the root index gets rebuilt but the aux
relations remain at the previous build's state. A one-line
ereport in the appropriate ambuild/amrebuild path is cheaper than
a support ticket. Track as a small pre-landing follow-up.
