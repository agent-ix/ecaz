### Handoff blockers (must close before coder-1 starts performance work)

1. **Unit-norm assumption is unconditional in the distance wrapper but
   is not validated at build time.** `ambuild.rs:319` and
   `insert.rs:1691` both compute `d = max(0, C − ip)` with
   `C = ECDISKANN_UNIT_NORM_DISTANCE_BIAS = 1.0`. The design doc
   (`plan/design/diskann-build-algorithm.md:176`) is explicit that the
   `C = 1` branch holds "when vectors are unit-normalized" — the code
   applies it unconditionally with no guard. On non-unit corpora this
   reproduces the exact class of bug `11078` just fixed: the
   `.max(0.0)` floor silently collapses every `ip > 1` pair into the
   same zero-distance bucket, and the α-dominance inequality
   `α · d(p*, v) > d(p, v)` no longer holds the Euclidean-like
   properties `robust_prune` was written against.

   This is the first thing a coder-1 tuning pass will fall over on —
   the moment they try a non-unit fixture to characterize recall vs.
   corpus shape, the graph will silently go bad and the
   measurement will look like a tuning problem rather than a
   precondition violation.

   Recommended minimum before handoff:
   - Sample a bounded prefix of build-time source vectors, compute
     `‖v‖`, and `pgrx::error!` (or `warning!` if you want to keep it
     permissive for v0) if the mean norm is not within `[1 − ε,
     1 + ε]` for a small ε.
   - Name `ECDISKANN_UNIT_NORM_DISTANCE_BIAS` in one place (`mod.rs:53`)
     with a short comment pointing at the design-doc precondition and
     at the rename options when we lift it (per-corpus `C`, cosine
     normalization, etc.).
   - Either close `#ifs` around this as a v0 precondition in the task
     doc "Out of scope" section, or file a follow-up packet explicitly
     tracking the non-unit corpus lift. As written today it's neither
     validated nor called out.

2. **Task plan DoD line "Insert + vacuum round-trip survives the
   concurrency scratch script" has no artifact.** The phase 7 + 8
   plan rows name `scripts/diskann_insert_concurrency_scratch.sh`
   and `scripts/diskann_vacuum_concurrency_scratch.sh`
   (`plan/tasks/17-diskann-access-method.md:488–493, 536–538`).
   Neither file exists in `scripts/`. The task file says "Final
   performance benches on the faster machine" is the remaining
   signoff work, but the concurrency scratch DoD is separate from
   recall/latency and it has no packet, no artifact, and no pg_test
   obviously standing in for it.

   Either:
   - Land the two 60-second scratch scripts and capture one run
     each in a new packet (handoff slice for coder-2), or
   - Explicitly downgrade the DoD requirement in the task file and
     point at which existing `#[pg_test]` in `routine.rs` covers
     the concurrent-INSERT-+-VACUUM+scan case today.

   Handing off to coder-1 with this DoD row still unchecked and no
   script on disk makes it coder-1's hidden cleanup — they'll discover
   it the first time they try to reproduce a reviewer concern about
   concurrent-insert safety and find nothing to run.

3. **Baseline recall / latency numbers are in packet prose only.**
   `11078` captured `list_size ∈ {64, 128, 200, 400, 800}` →
   `Recall@10 ∈ {0.928, 0.931, 0.9315, 0.9315, 0.9315}` at
   `graph_degree=32, build_list_size=100, alpha=1.2` on the real
   10k fixture. `11083` captured the post-vacuum repeat at
   `list_size=128`. Those are the numbers coder-1 will measure
   against on the faster box. Right now they live exclusively in
   `review/11078/artifacts/*.log` and `review/11083/artifacts/*.log`
   — not in any durable `plan/measurements/*` or `plan/tasks/17-…`
   baseline table.

   Recommend lifting the `11078` post-distance-sweep table and the
   `11083` pre/post-vacuum pair into a short baseline block on the
   task file's "Current status" section (or a new
   `plan/measurements/task17-diskann-baseline-2026-04-21.md`).
   Without that, the first faster-machine run has no canonical
   number to compare to, and the "~0.95 preferred" target becomes
   impossible to evaluate because the slower-box `0.9310` isn't
   itself on record as a best-effort reference.
