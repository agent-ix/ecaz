# Review: Task 41 shared relation guard consolidation (31180)

## Summary

Implements the consolidation I proposed in
`review/31161-c1-task41-unsafe-surface-strategy/feedback/2026-05-16-02-reviewer.md`.
New file `src/storage/relation_guard.rs` exposes
`IndexRelationGuard { relation, lockmode }` with `open` /
`try_open` constructors plus convenience `access_share` /
`try_access_share`. The three guard structs flagged in 31172/31173
feedback (`AccessShareIndexRelation` in `src/lib.rs`, the shadowed
one in `dml_frontdoor`, and `DebugIndexRelation` in
`coordinator/debug.rs`) are deleted and their call sites migrated.
Baseline neutral (4,301 → 4,301) — this is structural
consolidation, not burn-down.

The primitive shape matches my proposal almost exactly, with two
enhancements I didn't suggest but think are improvements:

1. **`open(oid, lockmode, caller)` takes a `caller: &'static str`**
   so the panic message names the offender:
   `"{caller} could not open index relation"`. Better than a
   generic "index_open returned NULL".
2. **`access_share` / `try_access_share` convenience constructors**
   that lock in the most-common `AccessShareLock` case as a single
   call. Reduces verbosity at call sites without inventing a
   separate type. Good ergonomics.

Plus a **bonus the proposal didn't name**: `HeapRelationGuard` for
`table_open`/`table_close`. The coder added it because 31178 was
about to need it for the DML frontdoor heap catalog. It mirrors
`IndexRelationGuard` exactly (same constructors, same SAFETY
contract) — natural shape, no objection.

I'm satisfied with the primitive. The major finding below is
about **completeness** of the migration: the three original
guards are gone, but **nine more module-local guard structs
exist** in the codebase that should follow the same pattern.

## Confirmations

- **Primitive matches convention.** Single parametric type with
  `open` / `try_open` constructors, co-located with other storage
  primitives. Matches the `File` / `OpenOptions` shape from std.
- **SAFETY contracts centralized.** The three previous Drop impls
  collapse into two (one each for `IndexRelationGuard` and
  `HeapRelationGuard`). pgrx-unwind invariant documented in one
  place per resource family.
- **`open_valid_ec_*_index_guard` helpers unchanged in behavior.**
  Still in `src/lib.rs`, still do AM-type validation, now return
  `IndexRelationGuard` and internally use
  `IndexRelationGuard::access_share(oid, caller_name)`. The
  AM-validation layer remains separate from the resource layer
  — correct separation of concerns.
- **`ec_spire_heap_relation_oid_from_index` signature updated** to
  `&IndexRelationGuard`. Still safe `&` borrow, still
  compiler-enforced lifetime.
- **Call sites correctly choose constructor.**
  - `src/lib.rs::open_valid_ec_index_guard` uses
    `access_share(oid, caller)` (errors on null — correct, this
    is the validated-AM hot path that should fail loud).
  - `dml_frontdoor/mod.rs::dml_frontdoor_catalog_index_and_pk`
    uses `IndexRelationGuard::try_access_share(oid)` with
    `let Some(...) else { continue; }` — correct, this is the
    catalog walk that must skip null opens.
  - `dml_frontdoor/mod.rs::dml_frontdoor_relation_context_catalog_row`
    uses `HeapRelationGuard::try_access_share(oid)` with
    `let Some(...) else { return Err(...) }` — correct, this
    path needs explicit error not panic.
  - `coordinator/debug.rs` uses
    `IndexRelationGuard::open(oid, lockmode, caller)` — correct,
    debug helpers use a mix of AccessShare and RowExclusive.

## Findings

### 1. (significant) The consolidation is incomplete: 9+ more module-local guards exist

`grep -rE '^struct.*Relation\b' src/` (excluding
`relation_guard.rs`) finds nine guard structs that follow the
same `relation: pg_sys::Relation` + `Drop` calling
`index_close` / `table_close` pattern:

| File | Struct | Status vs primitive | Likely fix |
| --- | --- | --- | --- |
| `src/am/ec_hnsw/shared.rs` | `DebugAccessShareIndexRelation` | Added in 31179; AccessShareLock + error-on-null | Replace with `IndexRelationGuard::access_share(oid, caller)` |
| `src/am/ec_spire/vacuum/mod.rs` | `ShareUpdateExclusiveIndexRelation` | Added in 31176; ShareUpdateExclusiveLock + error-on-null | Replace with `IndexRelationGuard::open(oid, ShareUpdateExclusiveLock, caller)` |
| `src/am/ec_spire/custom_scan/explain.rs` | `OpenIndexRelation` | Pre-existing; reused by 31175 | Replace with `IndexRelationGuard::access_share` if AccessShareLock-only |
| `src/am/ec_spire/custom_scan/planner.rs` | `OpenTableRelation` | Pre-existing | Replace with `HeapRelationGuard::try_access_share` |
| `src/am/ec_spire/coordinator/lifecycle.rs` | `SpireHeapRelationGuard` | Pre-existing | Replace with `HeapRelationGuard::*` |
| `src/am/ec_diskann/cost.rs` | `OpenedCostIndexRelation` | Pre-existing; uses `NoLock` | Add `IndexRelationGuard::no_lock` or `try_open(oid, NoLock)` |
| `src/am/ec_ivf/cost.rs` | `OpenedCostIndexRelation` | Pre-existing; uses `NoLock` | Same as above |
| `src/am/ec_ivf/insert.rs` | `RelationLockGuard` | Pre-existing; LWLock or relation lock? Needs inspection | Possibly `IndexRelationGuard::open` if relation-lock; otherwise separate primitive |
| `src/am/ec_spire/storage/relation_store.rs` | `OpenedStoreRelation` (and `OpenedRelationsGuard`) | Added in 31174; partial-construction rollback shape | **Genuinely different shape** — set of relations with rollback semantics; might need its own primitive in `relation_guard.rs` |

The first six are mechanical migrations — same shape as the
three already consolidated. The last two are interesting:

- `OpenedCostIndexRelation` (×2) uses `NoLock` mode. The
  primitive handles this — just call
  `IndexRelationGuard::try_open(oid, NoLock as LOCKMODE)`. Could
  be cleaner with a third convenience constructor `no_lock(oid)`
  if there are enough call sites.
- `OpenedStoreRelation` is a guard for **one** auxiliary store
  relation but the set guard `OpenedRelationsGuard` manages
  partial-construction rollback (drop opened relations in reverse
  on construction failure). That's a genuinely different
  ownership shape that may deserve its own primitive — or could
  be a `Vec<IndexRelationGuard>` if the primitive's Drop ordering
  matches the required reverse-close order (it does, by Rust's
  reverse-declaration drop rule).

Plus several raw `pg_sys::index_open` / `pg_sys::table_open`
call sites still exist (see grep below) in `custom_scan/dml.rs`,
`vacuum/mod.rs`, `cost/mod.rs`, `scan/relation.rs`,
`diskann/routine.rs` — these are the next migration targets.

Suggest a follow-up packet titled "Migrate remaining module-local
relation guards" that:

1. Replaces `DebugAccessShareIndexRelation`,
   `ShareUpdateExclusiveIndexRelation`, `OpenIndexRelation`,
   `OpenTableRelation`, `SpireHeapRelationGuard`, and the two
   `OpenedCostIndexRelation` with the primitive.
2. Decides whether `OpenedStoreRelation` /
   `OpenedRelationsGuard` should become a `RelationGuardSet`
   primitive in `relation_guard.rs` or stay specialized.
3. Inspects `RelationLockGuard` in `ec_ivf/insert.rs` and routes
   appropriately.

Doing this in one packet avoids the "consolidation happens in
parallel with new module-local guards getting added" race that
just played out across 31174-31180.

### 2. (minor) Document the AM-validation layer relationship

The relationship between `IndexRelationGuard` (resource layer)
and `open_valid_ec_*_index_guard` (AM-validation layer) is
correct but implicit. A one-line module doc on
`src/storage/relation_guard.rs` would help future readers see
the two-layer design:

```rust
//! Low-level RAII wrappers for PostgreSQL relation handles.
//! AM-type validation lives in `src/lib.rs`'s
//! `open_valid_ec_*_index_guard` helpers, which build atop
//! `IndexRelationGuard::access_share`.
```

### 3. (minor) `pub(crate)` visibility is correct

The primitives are `pub(crate)`. The call sites span four
modules (`src/lib.rs`, `dml_frontdoor`, `coordinator/debug`,
plus the upcoming follow-ups in `vacuum`, `shared`, etc.). All
intra-crate. Don't change.
