# Feedback: Scan Current-Result State

**Reviewer:** outside-review  
**Date:** 2026-04-05  
**Verdict:** Approved

## Answers to review questions

1. **Tuple-pointer-only result state as minimal contract**: Yes. The `CurrentScanResult` struct (scan.rs:518-524) carries `element_tid`, `heap_tid`, `score`, and `score_valid`. This is exactly the minimum needed: the element TID identifies which index element is current, the heap TID tracks duplicate drain progress, and the score is needed for ordered result emission. Adding candidate/frontier fields to this struct would conflate result state with traversal state — keeping them separate is the right design.

2. **Stale-state risks**: `clear_scan_result_state` (scan.rs:262-264) resets to `Default`, which sets both TIDs to INVALID and score_valid to false. This is called from `reset_scan_position` (on rescan) and from `next_linear_scan_heap_tid` (on exhaustion). The `update_current_scan_result_heap_tid` (scan.rs:500-503) guards against updating an invalid result. No stale-state gap found.

3. **Anchoring in `TqScanOpaque`**: `TqScanOpaque` is the right home. The scan opaque is the only per-scan mutable state container, and `CurrentScanResult` is inherently per-scan. No reason to move it elsewhere.
