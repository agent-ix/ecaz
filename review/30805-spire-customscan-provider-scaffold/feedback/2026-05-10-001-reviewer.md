# Review: SPIRE CustomScan Provider Scaffold (30805)

## Summary

Accept. The scaffold is the right shape: provider registration in
`_PG_init`, a chained `set_rel_pathlist_hook` that defers entirely to
the previous hook, fail-closed Begin/Exec callbacks that error rather
than silently produce zero rows, and a `ec_spire_custom_scan_status()`
diagnostic that reports both registration state and the explicit
`path_generation_enabled = false` / `exec_wiring_enabled = false`
markers so reviewers can see partial state at a glance.

The three reviewer-focus questions answer yes:
- `_PG_init` registration is acceptable for the first slice.
- Fail-closed executor callbacks (`pgrx::error!` in Begin and Exec) are
  the right temporary safety boundary.
- Status surface makes partial state visible.

## Findings

### P3 — `CUSTOM_PATH_METHODS` is dead code in this slice

`CUSTOM_PATH_METHODS` (and the `ec_spire_plan_custom_path` callback it
points at) is declared but never reached from any code path in this
slice — nothing creates a CustomPath, so the planner never consults the
path methods. It compiles because it's a `static mut`, but it's load-
bearing only for the next packet (planner path generation). Two
acceptable options:

- Leave as-is. The next packet will start exercising it; the diagnostic
  surface already reports `path_generation_enabled = false` so reviewers
  know the static is currently inert.
- Move `CUSTOM_PATH_METHODS` and the `ec_spire_plan_custom_path` stub
  out of this slice and into the planner-path packet, so the scaffold
  ships only registered surfaces.

Mild preference for option 2 (smaller scaffold, no inert statics), but
not a blocker.

### P3 — `set_rel_pathlist_hook` installed before any path is generated

The hook is installed in `_PG_init` and runs for every relation in
every query, but currently does nothing except chain the previous hook.
That's correct fail-closed behavior — there's no risk of a stray
CustomScan plan reaching execution because no path is added. The cost
is one indirect function call per relation per query for every backend
the extension is loaded into, even on databases that don't have any
`ec_spire` index.

Acceptable for the scaffold and arguably necessary to prove the chain
is correct before the path-generation packet adds the actual work.
Worth a brief comment on the function noting that the empty body is
deliberate (no path generation in this slice) so a future reader does
not delete the hook as no-op.

### P3 — `static mut` ergonomics

The four `static mut` items (`PREVIOUS_SET_REL_PATHLIST_HOOK`,
`CUSTOM_SCAN_REGISTERED`, `REL_PATHLIST_HOOK_INSTALLED`, plus the three
methods structs) match the existing pattern in this codebase for
`_PG_init` registration, so this is consistent. Worth flagging that
Rust 2024 edition tightens `&raw const` on `static mut` — when the
crate eventually moves edition, these will need a wrap in `OnceLock`
or `addr_of`. Not for this packet; a known migration item.

### P3 — Status row test is unit-only

The Rust unit test only verifies the static row shape; the SQL `pg_test`
asserts the live registration result. That split is fine. Consider
naming the unit test
`custom_scan_status_static_row_shape_is_pre_registration` so a future
reader does not mistake it for a live-registration assertion (the unit
test runs without `_PG_init` and so always reports `registered = false`,
hook_installed = false`, which is *not* the production reality).

## Reviewer recommendation

Accept and proceed to the next slice. Either fold `CUSTOM_PATH_METHODS`
into the planner-path packet or land it as-is — both are fine, but if
left here please add a one-line comment indicating it's intentionally
inert until path generation lands.
