# Review: Packet 261 — C1 Warm-Cache Verified Surface

Reviewer: agent-ix (via code-review)
Date: 2026-04-12

## Assessment: Strong infrastructure packet — the honest warm seam is now the authoritative surface

### What this packet did right

1. **Identified the session-reuse confound.** The old harness opened a fresh
   `psql` per query, so backend-local state (plan cache, buffer pins, hash
   tables, quantizer cache) was rebuilt on every timed query. That conflated
   cold-backend churn with steady-state scan latency. Making this explicit and
   fixing it with `--session-mode per-cell` is exactly right.

2. **The warmup-pass control is clean.** `--warmup-passes 3` gives the PG
   shared-buffer pool and the backend itself a fair chance to reach steady
   state before timing begins. This matches how production ORDER BY queries
   behave after the first few calls.

3. **Honest framing of results.** The packet doesn't overstate the win.
   `p50=14.315ms` is still ~3x the NFR target, and the packet says so plainly.
   The temptation to cherry-pick the single-query `p50=12.488ms` spot check
   was correctly resisted.

4. **The per-query vs per-cell gap is useful data.** `p50=15.883ms` (per-cell,
   1 warmup) vs `p50=50.246ms` (per-query, 1 warmup) quantifies exactly how
   much of the old cold surface was session-setup cost. This frames every
   subsequent C1 packet.

### Concern: the warm surface should be the default going forward

The packet correctly adds warm measurement as an option, but the C1 reporting
still centers the cold surface in its exit criteria. Since NFR-001 specifies
warm steady-state targets, every future C1 packet should benchmark against the
warm `per-cell` surface as its primary gate and report cold only as secondary
context.

### Concern: the query-limit=1 spot check is a risky reference point

The single-query warm spot check (`p50=12.488ms`) is mentioned as a
"representative" data point. Be careful with this — one query can be an outlier
(short neighbor chain, entry point near the target, lucky page co-location).
The full query-set p50 is the honest number to track.

## Suggestions

1. **Pin the warm seam as the default for future packets.** Standardize on
   `--session-mode per-cell --warmup-passes 3` as the canonical warm
   measurement. Document this in the review so later packets don't drift back
   to per-query mode.

2. **Record the shared-buffer hit rate alongside latency.** The warm surface
   assumes all pages are in shared buffers. If a future corpus size exceeds
   `shared_buffers`, the "warm" surface degrades silently. Adding `BUFFERS`
   output (already available via `EXPLAIN ANALYZE BUFFERS`) to the warm summary
   would catch this.
