## Findings

### 1. Data directory is preserved across runs

**Severity:** Low
**File:** `scripts/run_pg18_preload_pgstat_test.sh:83-85`

```bash
if [[ ! -f "${data_dir}/PG_VERSION" ]]; then
    "${initdb_bin}" -D "${data_dir}" -A trust -U postgres >/dev/null
fi
```

initdb is skipped on subsequent runs. That's a fine perf optimization,
but it means an earlier run's binaries baked into the catalog (via
`CREATE EXTENSION ecaz`) survive — and if the ecaz binary on disk has
changed meaningfully (new control-file version, new SQL surface), the
next run starts in a state where the catalog and the `.so` disagree.
The `DROP EXTENSION IF EXISTS ecaz CASCADE; CREATE EXTENSION ecaz;`
pair at line 170-171 covers most cases, but not a control-file version
bump or a dropped SQL function the previous catalog still thinks it
owns.

Two reasonable fixes:

- Add an opt-in `--fresh` flag that deletes `${data_dir}` before
  running. Default stays cached for dev iteration speed.
- Or detect an out-of-date catalog by comparing the installed
  `ecaz.control` version against `extversion` from `pg_extension`,
  and force an `ALTER EXTENSION ecaz UPDATE` / drop-recreate when
  they differ.

Option 1 is simpler and matches how CI would want to run this.

### 2. Baseline is non-zero across reruns — assertion still correct

**Severity:** Informational (not a bug)
**File:** `scripts/run_pg18_preload_pgstat_test.sh:190-205`

The script reads baseline → runs scan → reads shared, and asserts
`shared > baseline`. That's robust against pre-existing shared counter
state, which is the right design choice given #1. Worth a one-line
comment in the script so the next reader sees the intent.

### 3. `next_pg18_blocker` literal-string assertion is brittle

**Severity:** Low
**File:** `scripts/run_pg18_preload_pgstat_test.sh:188`

```bash
assert_eq "${next_pg18_blocker}" "no merged PG18 blocker remains on main" \
    "planner snapshot should clear the preload-only blocker under preload"
```

The assertion compares against the exact prose from
`src/am/ec_hnsw/shared.rs:797`. That means any future wording
adjustment to the user-facing snapshot message silently breaks this
test. Two cheap options:

- Assert on `pg18_diagnostics_surface_ready = 't'` only (already doing
  this at line 187 — arguably sufficient by itself), and relax the
  literal check to "does not contain `custom pgstat kind registration
  requires`" (the actual blocker string).
- Or move the "cleared" sentinel to a constant in
  `src/am/ec_hnsw/shared.rs` and reference it from test inventory so
  the script + source share one truth.

Not a correctness issue — the current string is what the code
produces — but it's the kind of coupling that rots silently.

### 4. Three-row fixture may not exercise the graph path

**Severity:** Low
**File:** `scripts/run_pg18_preload_pgstat_test.sh:172-178`

With three rows, the `ec_hnsw` index's graph layer is effectively
empty and the scan falls through to the linear-bootstrap path. That's
fine for proving `record_distance_calc` fires (it still runs in
scoring regardless of which path), but it means this script doesn't
actually exercise `record_graph_hop` or the graph-prefetch ReadStream.
If the packet's intent is "prove preload pgstat works across all
hot-path counters," bump the fixture to ~1000 rows and add assertions
on `total_graph_hops` and `total_linear_pages`. If the intent is
narrowly "prove cross-backend shared counter visibility at all," the
current three-row fixture is fine.

The `request.md` review focus says "proves the shared pgstat path
rather than merely rechecking the backend-local fallback," which the
current script *does* do — but a richer fixture would future-proof
against regressions that only affect a subset of counters.

### 5. Port retry loop succeeds on any start failure other than
"Address already in use"

**Severity:** Medium
**File:** `scripts/run_pg18_preload_pgstat_test.sh:92-106`

```bash
if "${pg_ctl_bin}" ... start >/dev/null 2>/dev/null; then
    selected_port="${candidate_port}"
    break
fi

if ! grep -q "Address already in use" "${log_file}"; then
    cat "${log_file}" >&2
    exit 1
fi
```

The `>/dev/null 2>/dev/null` on `pg_ctl start` swallows stderr, which
means the only signal you have for why it failed is the log file.
That's the right choice. But: `pg_ctl` exits non-zero also when
`shared_preload_libraries = 'ecaz'` itself fails to load (e.g., the
`.so` is a PG17 binary under a PG18 install). The script does dump
the log to stderr in that case (line 103), which is good, but the
exit code (1) is the same as the "no free port" case (line 110). Make
the failure modes distinguishable — either different exit codes or a
clear prefixed error before `cat "${log_file}"`.

### 6. Trap ordering: data survives failed startup

**Severity:** Low
**File:** `scripts/run_pg18_preload_pgstat_test.sh:75-85`

The `trap stop_cluster EXIT` is set before the port-search loop, and
`stop_cluster` only runs `pg_ctl stop` when `pg_ctl status` reports
the cluster is up. If the cluster failed to start (e.g., a stale
postmaster pid from an earlier crash), the trap does nothing and the
stale state is left behind. Consider adding a `pg_ctl stop -m
immediate` or `rm -f ${data_dir}/postmaster.pid` recovery step at the
top of the script. Low-severity because the next run usually succeeds
once the postmaster process has actually died.

### 7. Script has no teardown of the test fixture

**Severity:** Informational
**File:** `scripts/run_pg18_preload_pgstat_test.sh:170-178`

The `DROP TABLE ... CASCADE; DROP EXTENSION ... CASCADE; CREATE ...`
pattern re-creates the fixture each run. Because the cluster is
cached (#1), the data directory accumulates WAL and other test state
over many runs. Consider an optional `vacuumdb` or periodic `--fresh`
run. Not a correctness issue.
