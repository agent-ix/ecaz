# Review: hnsw_rs Wildcard Version Dependency

**File:** `Cargo.toml:24`
**Severity:** Medium (build reproducibility)
**Category:** Dependency management

## Finding

```toml
hnsw_rs = "*"
```

The `hnsw_rs` dependency uses a wildcard version, meaning any version of the crate will satisfy the dependency. This creates several risks:

1. **Build non-reproducibility:** Different builds at different times may pull different versions with different behavior or bugs. While `Cargo.lock` pins the resolved version, anyone doing a fresh `cargo update` or building without the lockfile gets an unpredictable version.

2. **Breaking changes:** `hnsw_rs` has no stability guarantees at `*`. A semver-breaking release could silently change graph construction behavior, affecting index quality.

3. **Supply chain risk:** A compromised or yanked version could be pulled in.

4. **`deny.toml` present but version unconstrained:** There's a `deny.toml` for license checks, but the wildcard version undermines reproducibility goals.

## Recommendation

Pin to a specific version or at minimum a semver range:

```toml
hnsw_rs = "0.3"  # or whatever version is currently in Cargo.lock
```

Check `Cargo.lock` for the currently resolved version and pin to that major.minor.

## Action Required

Replace `"*"` with the actual version currently locked in `Cargo.lock`.
