# Review: aminsert Metadata Update and Data Write Not Atomic

**File:** `src/am/mod.rs:188-250`
**Severity:** High (crash safety)
**Category:** WAL safety / correctness

## Finding

In `tqhnsw_aminsert`, the metadata page and data pages are modified in separate WAL transactions:

1. `with_locked_metadata_page` acquires an exclusive lock on the metadata page, runs the closure, writes the metadata page via GenericXLog, and releases the lock.
2. Inside the closure, `append_heap_tuple` is called (line 242), which writes the data page(s) in a **separate** GenericXLog transaction.
3. The metadata page write (entry_point update, shape initialization) happens in the `with_locked_metadata_page` WAL transaction.

If the backend crashes between:
- Writing the data page WAL record (the new element tuple is durable) and
- Writing the metadata page WAL record (entry_point update)

...then after recovery, the data page contains an element tuple that the metadata doesn't know about. The entry_point might still be INVALID (for the first insert into an empty index).

Conversely, if metadata is written first but data page write fails, the entry_point could reference a non-existent tuple.

**Specific crash scenario:**
1. Empty index, first insert
2. `with_locked_metadata_page` is entered, metadata shows `dimensions=0, bits=0`
3. Closure sets `metadata.dimensions`, `metadata.bits`, `metadata.seed`
4. `append_heap_tuple` writes element+neighbor to data page, WAL committed
5. **CRASH before `with_locked_metadata_page` writes metadata WAL**
6. After recovery: data page has tuples, metadata still shows `dimensions=0`

On next insert, the `dimensions == 0 && bits == 0` check (line 213) will trigger again and overwrite with the new tuple's shape, which is correct. But the existing data tuples are orphaned -- they'll be scanned by linear scan but the entry_point is INVALID.

Wait -- actually looking more carefully: `append_heap_tuple` is called inside the `with_locked_metadata_page` closure at line 242, but it opens its own buffer and creates its own WAL transaction. This means we have **nested WAL transactions** -- the data page WAL record is separate from the metadata page WAL record.

GenericXLog only supports one transaction at a time per backend, but since `append_heap_tuple` creates its own `GenericXLogTxn` and finishes it before returning, this should work sequentially. The issue is that they're not atomic.

## Recommendation

This is a known limitation of GenericXLog (which doesn't support multi-page atomic writes without custom WAL records). The current approach is acceptable for v0.1 with the understanding that:

1. The index can tolerate orphaned tuples (linear scan still finds them)
2. The worst case is a stale entry_point, not data loss
3. Proper crash recovery should be added before production use

Document this limitation in an ADR.

## Action Required

Document the non-atomicity in an ADR. Consider adding a consistency check in `ambeginscan` or `amrescan` that validates the metadata page against the actual data pages.
