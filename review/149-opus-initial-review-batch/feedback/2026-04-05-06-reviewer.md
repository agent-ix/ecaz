# Review: relation_options Uses SPI in Insert Path

**File:** `src/am/mod.rs:664-696`
**Severity:** Medium (performance)
**Category:** Optimization

## Finding

`relation_options` executes an SPI query to read reloptions:

```rust
let reloptions = pgrx::Spi::get_one_with_args::<Vec<String>>(
    "SELECT reloptions FROM pg_class WHERE oid = $1",
    &[unsafe { pgrx::datum::DatumWithOid::new(relid, pg_sys::OIDOID) }],
)
```

This function is called on every `aminsert` (line 202). SPI is a relatively expensive interface: it involves query parsing, catalog lookups, snapshot management, and executor overhead. For a high-throughput insert workload, this adds significant overhead per tuple.

The reloptions for an index relation are available directly from the relation descriptor's `rd_options` field, which is already cached in shared memory by PostgreSQL's relcache. The `tqhnsw_amoptions` callback is specifically designed to parse these into the `TqHnswReloptions` struct.

## Recommendation

Replace the SPI-based option reading with direct access to `(*index_relation).rd_options`:

```rust
unsafe fn relation_options(index_relation: pg_sys::Relation) -> TqHnswOptions {
    let rd_options = unsafe { (*index_relation).rd_options };
    if rd_options.is_null() {
        return TqHnswOptions::DEFAULT;
    }
    let relopts = unsafe { &*(rd_options as *const TqHnswReloptions) };
    // parse m, ef_construction, build_source_column from relopts
}
```

This eliminates the SPI overhead entirely and is the standard pattern used by pgvector and other index extensions.

## Action Required

Replace `relation_options` with direct `rd_options` access. This is a significant performance fix for the insert path.
