# Feedback: ef_search Control Surface And Planner Gate Scaffolding

Request:
- `review/126-ef-search-control-surface-and-planner-gate-scaffolding.md`

**Reviewer:** Codex
**Date:** 2026-04-06

## Findings

### 1. Rebase onto current `main` before merge

This branch was cut before the newer PG18 optimization spec/docs landed on `main`, so relative to
current `main` it appears to delete a large shared spec surface. That looks like stale-base drift,
not an intentional product decision, but it still cannot merge safely as-is.

Action:
- rebase or merge current `main` into `planner-integration-lane`
- preserve the broader PG18 optimization docs/FRs/use cases/ADRs from `main`
- keep this branch's narrower planner/admin additions layered on top

### 2. Keep ADR-011 as a hard gate

The code direction in this slice is good as long as the planner gate remains real. The runtime lane
still does not have credible planner-visible ordered execution semantics; it is improving, but it
is still staged bootstrap-first plus linear fallback.

Action:
- keep `TQHNSW_PLANNER_SCAN_ENABLED = false` or equivalent hard-disabled planner seam
- do not let the new `ef_search` control surface imply planner readiness
- treat this slice as scaffolding for costing/explain/statistics, not as planner enablement

### 3. The GUC/reloption precedence direction looks right

Using the session GUC only as an override over the existing per-index reloption is the right
current shape. It gives the planner/productization lane a real tuning surface without forcing
runtime ownership changes into `scan.rs`.

What to preserve:
- explicit precedence rules
- observable effective `ef_search` in admin/debug surfaces
- no hidden planner side effects from changing the GUC

## Net

The slice is conceptually on the right track. The main steering correction is integration safety:
rebase first, preserve the newer PG18 spec set from `main`, and keep the planner gate firmly on.
