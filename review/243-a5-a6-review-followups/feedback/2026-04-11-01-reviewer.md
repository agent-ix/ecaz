### 1. Post-quiesce harness assertion (finding 3, critical)

Landed and strengthened. `scripts/vacuum_concurrency_scratch.sh:236`
now runs `VACUUM (ANALYZE)` after workers exit, and then lines 283–298
enforce **two** assertions that together form a real row-conservation
proof:

- `final_live_elements == reference_live_elements` (strict equality,
  line 283). Both values come from
  `tqhnsw_index_admin_snapshot(...).total_live_nodes` — one on the
  mutated index, one on a freshly `CREATE INDEX`-ed reference on the
  same post-quiesce table data. Because both indexes are built against
  the same underlying rows, admin-snapshot live-node counts **must**
  agree. If pass 2 ever silently drops an element, this is a hard
  failure at check time. This is the strict equality I wanted, on the
  right metric.

- `final_reachable_live_elements * 100 < reference_reachable_live_elements * 90`
  (line 295). This is the 90%-of-reference reachability floor on the
  new `tests.tqhnsw_debug_reachable_live_element_count(...)` helper.

The review asked for `final_live_rows == final_scan_result_count`. The
packet correctly explains why that exact equality is wrong: the scan
helper is bounded by `ef_search` and returns 40 rows on a 2000-row
fixture regardless of index health. Using it as an equality gate would
have been a bogus proof. Replacing it with a reference-build comparison
on a metric that actually corresponds to graph connectivity is a
strictly better response than the literal ask. Thank you for pushing
back on that.

One methodology note worth recording for future readers — not a
blocker:

`debug_layer0_reachable_live_element_tids` (`src/am/scan_debug.rs:772`)
does a **layer-0-only BFS from `metadata.entry_point`**. That entry
point is a top-layer node, but the BFS only follows
`load_layer0_neighbor_tids(...)`. So the metric counts "nodes reachable
via layer-0 edges from the entry-point's own layer-0 slot set," not
"nodes reachable via a real scan" (which descends through upper layers
first). That's why the observed ratios show ~55–67% of total live
elements are reachable — it's a property of the metric, not a bug in
repair.

For the harness purpose that's fine: both the live and reference index
get measured the same way, and a ratio-based threshold is stable under
that asymmetry. But `final_reachable_live_elements` should not be read
as "rows a real query can find" outside the harness context. A
one-line comment on the helper explaining the measurement shape would
help whoever next touches this — not required.

## Answers to packet review questions

### Is the proof correction sound?

**Yes, and it's stronger than the literal ask.** The review asked for
equality between "live rows in the table" and "scan result count",
which would have been a bogus check because the scan helper is
ef-search-bounded. Coder-1 investigated, discovered the asymmetry,
and replaced it with:

1. A strict `final_live_elements == reference_live_elements` equality
   on admin-snapshot `total_live_nodes`. This IS the row-conservation
   proof — both indexes are built against the same final table data,
   so their live-node counts must agree or the live index lost rows.
2. A 90%-of-reference floor on layer-0 reachability, which catches
   graph-connectivity regressions.
3. Keeping the live scan helper in the worker mix (not the assertion),
   so the concurrent mix still exercises the real
   `ambeginscan / amrescan / amgettuple` path.

This is a better design than what I proposed. Good outcome.

### Is 90% the right threshold for TC-215?

**Acceptable as a regression gate, with one observation worth recording.**

The observed numbers from the 5s and 60s runs are illuminating:

| Duration | live_reachable | reference_reachable | ratio |
|---------:|---------------:|--------------------:|------:|
| 5s       |           1284 |                1262 |  101% |
| 60s      |           2207 |                1819 |  121% |

In both runs, the live mutated index is *more* reachable than the
fresh reference build. A plausible explanation: vacuum repair is
**fill-only and additive** — it only adds to free layer-0 slots, never
evicts. So after hundreds of DELETE+VACUUM cycles, the live layer-0
graph accumulates edges from (insert-time) + (repair-time), while a
fresh build only has (insert-time) edges. That's consistent with the
A6 design choice to keep repair fill-only.

The 90% floor catches under-reachability regressions (the
correctness-critical direction), which is the primary purpose.
Consider as a future refinement: a symmetric band like `80% <= ratio
<= 150%` that would also catch the inverse case — "repair is adding so
many edges that layer-0 is bloating." A bloated layer-0 isn't a
correctness regression, but it would affect scan latency and storage.
Not required for A6 closure; worth noting in the A6 follow-ups list.
