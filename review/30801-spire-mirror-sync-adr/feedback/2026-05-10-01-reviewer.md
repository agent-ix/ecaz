## Answers to the packet's review questions

- **Is explicit operator-owned refresh the right v1
  mechanism before implementation starts?** Yes. Three
  reasons:
  1. **Smallest surface area**: a SQL primitive +
     `ecaz` wrapper is the most contained mechanism — no
     new daemon, no replication subscription setup, no
     per-query state. Easy to reason about, easy to
     test, easy to roll back.
  2. **Operator-visible control**: when something goes
     wrong with materialization (stale data, missing
     mappings, catalog drift), an operator has an
     explicit command to invoke and inspect. Background
     workers and replication subscriptions are harder to
     debug under partial failure.
  3. **Right ordering for Phase 11**: a background
     worker or replication mechanism is a Phase 12+
     productionization choice that benefits from real
     operational experience with v1. Locking the simpler
     mechanism first creates the test surface and
     diagnostic vocabulary that more complex mechanisms
     can build against.
- **ADR-066's rejected alternatives accurately scoped?**
  Yes. Background workers add lifecycle complexity that
  v1 doesn't need; lazy per-query materialization
  violates ADR-064's no-scan-time-writes rule; logical
  replication adds substantial PostgreSQL infrastructure
  config; user-managed register calls leak the catalog
  implementation into application code. All four are
  the right things to reject for v1.
- **Implementation slice order matches the Stage D
  finish direction from 30800?** Yes:
  1. **Profile/dry-run diagnostics** first — operators
     can see what *would* be materialized without
     actually writing. Matches "build the inspection
     surface before the mutation surface" discipline
     used throughout Stage C / D.
  2. **Refresh SQL primitive** — the actual mechanism.
  3. **`ecaz` wrapper** — operator UX.
  4. **No-explicit-register PG18 fixture** — proves
     end-to-end production behavior (catalog populated
     by `ecaz` invocation → SQL `SELECT` returns
     remote rows without test-only register calls).
     This is the **acceptance signal for Stage D
     feature-complete**.
  5. **Catalog lifecycle coverage** — completes the
     30800 action items.

## P2 — close before refresh primitive lands

1. **Concurrency contract between refresh and live
   scans.** ADR-066 establishes that refresh runs
   outside AM scans. Worth pinning before the primitive
   lands:
   - Can refresh run while a scan against the same
     `ec_spire` index is in flight? (Probably yes — the
     scan reads the catalog at scan-start; refresh
     inserts/updates rows; live scans see their
     snapshot's view.)
   - What lock-mode does the refresh need on the
     coordinator base table and on the
     `ec_spire_remote_row_materialization` catalog?
     (Probably row-level for catalog updates, no lock on
     the base table since the proxy rows are written by
     normal INSERT/UPDATE.)
   - What happens if refresh is interrupted mid-run?
     (Should be a transaction — partial mappings either
     all commit or all roll back; can't have
     half-populated catalog entries.)
   These are answerable in the ADR or the primitive's
   implementation packet, but should land before
   downstream packets depend on them.

## P3 — informational

2. **Profile/dry-run output format.** When the dry-run
   diagnostic lands, the output should be machine-
   readable (rows: pid / origin_node / vec_id /
   row_locator / mapping_exists / staleness_reason) so
   the `ecaz` wrapper and Stage G runbook can consume it
   directly. Worth pinning the schema in the diagnostic
   packet.

3. **Refresh idempotency.** Running refresh twice in
   succession should be a no-op the second time. Worth
   asserting in the no-explicit-register fixture and
   noting in the ADR that idempotency is part of the
   contract — operators will want to re-run refresh
   freely without worrying about duplicate-mapping
   conflicts.
