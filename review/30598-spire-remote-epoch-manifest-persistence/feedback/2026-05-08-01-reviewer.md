# Remote epoch manifest persistence review

Two new extension-owned tables (`ec_spire_remote_epoch_manifest`,
`ec_spire_remote_epoch_manifest_entry`) plus
`ec_spire_persist_remote_epoch_manifest()` as the writer and two
catalog readback functions. Real persistence boundary, not
diagnostic. No blocking findings.

## Confirmations

1. **Fail-closed before persistence.**
   `ec_spire_persist_remote_epoch_manifest`
   (`src/lib.rs:1919-1932`) refuses to write unless the manifest
   summary reports `emit_distributed_epoch_manifest`. Plus an
   explicit guard against zero included remote entries
   (line 1937-1941). A blocked or local-only manifest cannot be
   accidentally persisted.
2. **DELETE + INSERT entry replacement is atomic** within the
   `Spi::connect_mut` block (lines 1986-2046). Concurrent
   readers see the prior entry set until commit, then the new
   set — no half-state visibility.
3. **Foreign key on the entry table** with `ON DELETE CASCADE`
   keeps header and entries consistent. PK shape
   `(coordinator_index_oid, active_epoch[, node_id])` is right
   for the access patterns.
4. **CHECK constraints** on both tables are tight:
   `active_epoch > 0`, nonempty status/scope/decision,
   non-negative counts, `manifest_action = 'include_remote_node'`
   (the only persisted action — blocked decisions never reach
   the writer).
5. PG18 coverage exercises both ready persistence + readback and
   the blocked fail-closed path.

## Findings

- **Active-epoch advance race window.**
  `persist_remote_epoch_manifest` reads
  `spire_remote_epoch_manifest_summary` (line 1922) and
  `spire_remote_epoch_manifest_plan` (line 1923) before opening
  the SPI write block. If a concurrent build/insert advances the
  active epoch between the summary read and the
  `INSERT ... ON CONFLICT` write, this call would persist a
  manifest tagged with the *old* active epoch.

  In practice this is mostly self-healing: the next persist call
  observes the new active epoch and writes a fresh row. But a
  caller treating "persist returned true" as a guarantee that
  the *current* active epoch has a manifest is wrong by a small
  window.

  Suggested mitigation: re-read root control inside the SPI
  block and assert the read epoch matches `summary.active_epoch`
  before INSERT, or document the race in the function's
  contract. Not blocking — the writer is idempotent and the
  readback summary will correctly detect stale entries via
  30600.

- **`persisted_at_micros` precision.** Computed via
  `(extract(epoch from clock_timestamp()) * 1000000)::bigint`.
  Microsecond-level. CHECK requires `> 0`. Float-to-bigint cast
  truncates rather than rounds, which is fine for monotonicity
  but means two persists within the same microsecond would tie.
  Not a real concern given normal call cadence.
