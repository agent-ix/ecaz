# Review: Packet 260 — C1 AM Startup Boundary Reconciliation

Reviewer: peter (via code-review)
Date: 2026-04-12

## Assessment: This is the pivotal C1 finding. The cold/warm gap IS the remaining bottleneck.

### The key result

The earlier packet-259 reading of ~46ms startup is **not reproducible** on the
current build with a warm cache. The actual warm-cache numbers are:

- AM `amrescan` total: **3.289ms** at ef_search=40
- SQL `Index Scan` startup: **4.1ms** at ef_search=40
- Gap between AM and SQL: **~0.8ms** (executor overhead, not tqhnsw)
- Repeated plain scans: **~1.1ms/query** over 5,000 iterations

**This means tqhnsw warm-cache query latency is already below the NFR-001
p50<5ms target on the 10K corpus.** The entire remaining C1 gap in the
cold-cache verified surface (~50ms) is buffer-miss / page-fault latency, not
algorithmic or CPU overhead.

### What this changes about C1 strategy

This is a fundamental finding that should redirect subsequent C1 work:

1. **The CPU optimization arc (packets 252–257) has succeeded.** Warm-cache
   latency is at or below target. Further scoring/traversal optimization has
   near-zero marginal value for the cold-cache surface.

2. **The cold-cache surface is dominated by I/O.** The ~50ms cold-cache vs
   ~4ms warm-cache gap means ~46ms is spent on buffer misses during graph page
   reads. This is not something that faster scoring or better hash maps can
   fix.

3. **The next C1 work should be one of:**
   - **Declare warm-cache NFR-001 success** and report cold/warm separately as
     the spec already requires. If the warm surface is below 5ms and the cold
     surface is reported honestly, C1 may be closeable now.
   - **Reduce the cold-cache penalty** via prefetch/readahead during graph
     traversal (the `GraphPrefetchState` in TqScanOpaque suggests this was
     already planned), or by PG-level tuning (shared_buffers sizing).
   - **Move to the 50K corpus** to see whether the warm-cache target still
     holds on the normative NFR-001 table size.

### `perf` data is valuable

The backend `perf` capture showing 31.78% in `score_ip_from_split_parts` on
repeated warm queries is useful context. It confirms that under warm cache,
scoring is still the single largest CPU consumer — which means the SIMD
fast-path suggestion from the 257 review would directly improve the warm
surface further. But the warm surface is already at target, so this is
a nice-to-have, not a blocker.

### Code review

No concerns. The changes are purely instrumentation (timing probes in
`amrescan`) and a new debug SQL surface (`tqhnsw_debug_scan_heap_tids`). Both
are clean and narrow.

### Critical feedback on packet presentation

This packet's "In-Progress Findings" section buries the most important C1
result to date inside an update block. The fact that **warm-cache query
latency is at NFR-001 target** should be the headline, not a sub-bullet. The
packet should have an explicit "Revised C1 Read" section (like packet 247
does for its reruns) that states:

- Warm-cache p50 at m=8, ef_search=40 is ~4ms, below NFR-001's 5ms target
- The verified cold-cache surface (~50ms) is dominated by I/O, not CPU
- The ~46ms reading from packet 259 was a cold-cache artifact, now explained
- The C1 CPU optimization arc is functionally complete for the warm surface

### Suggestions for next steps

1. **Run the verified surface in warm-cache mode** (or at least report a
   warm-cache p50 alongside the cold-cache surface). NFR-001 says "Report
   warm-cache and cold-cache results separately when feasible." The C1 arc has
   only reported cold-cache numbers so far.

2. **Consider whether the cold-cache verified surface is the right primary
   metric.** A real production workload will have the working set warm after
   the first few queries. The cold-cache surface is a worst-case bound, not a
   steady-state measurement.

3. **If C1 continues, target I/O not CPU.** `posix_fadvise` / PG ReadStream
   prefetch during graph traversal, or `shared_buffers` tuning to keep the
   10K index resident, are the leverage points now.

4. **Move to the 50K corpus.** NFR-001 specifies 50K vectors. The 10K lane
   has been an excellent development surface, but the target measurement needs
   to happen on the normative size.
