# Phase 7 remote candidate merge review (also Phase 5/6/7 closeout summary)

Per AGENTS.md, the cross-cutting summary for this review batch lands on
the most-recent packet (this one). Localized findings are in the other
nine feedback files.

## 30554 — local findings

- **Score-ascending ranking is correct.** Internal scan candidate
  scoring stores `score = -ip` (`scan/candidates.rs:308, 542, 599,
  673, 717`), so lower-score-first in
  `remote_search_candidate_cmp` (`root/remote_candidates.rs:9-22`)
  selects higher-IP candidates first. Verified against the candidate
  emission path in `remote_search_candidates`
  (`hierarchy_snapshots.rs:159-170`), which forwards the same `score`
  field unchanged.

- **`assignment_flags` role rank uses only the boundary bit.**
  `remote_candidate_assignment_role_rank` returns 1 if the
  `BOUNDARY_REPLICA` bit is set, 0 otherwise — even if `PRIMARY` is
  also set on the same row. With current writers that never set both
  primary and boundary bits, this is correct. If the producer-side
  validation ever changes (e.g. allowing both bits to coexist), the
  rank computation should be revisited.

- **Tie-break by `vec_id` is dead code.** Earlier tie-breakers cover
  every distinguishing field for a duplicate group; by the time we
  fall through to `left.vec_id.cmp(&right.vec_id)`, both candidates
  share the same vec_id (otherwise dedupe would have separated them
  into different `HashMap` entries). The line is harmless but
  unreachable except for byte-identical rows, where it is also a
  no-op. Drop it or comment why it is retained.

- **`HashMap<Vec<u8>, ...>` clones each `vec_id`** at
  `remote_candidates.rs:53` — `candidate.vec_id.clone()` for every
  input row. With small per-node candidate fanout this is fine, but
  for large coordinator merges consider deferring the clone until
  insertion (e.g. `entry().or_insert_with(...)` pattern), or hashing
  on `&[u8]` via a wrapper. Non-blocking.

- **Validation surface is right-sized for v1.** Finite score and
  nonempty `vec_id` are the two contract bits the helper actually
  needs to function. Adding stricter checks (positive `served_epoch`,
  nonzero `pid`, nonzero `node_id`, etc.) belongs at the libpq
  receive boundary once that lands, not here. The request flagged
  this question — current scope is correct.

- **`input_count` and `duplicate_vec_id_count` are useful first
  diagnostics.** Both should flow into a future EXPLAIN/diagnostic
  surface so operators can detect coordinator fanout that is
  over-replicating or that has stale-node skew.

## Cross-cutting summary — Phase 5, 6, and start of Phase 7

### What's solid

- **Phase 5 boundary replication** is correctly default-off, gated by
  one bounded reloption, and uses a single shared `vec_id` for the
  primary plus replicas across all writer surfaces (build, recursive
  build, post-build insert, split replacement). Scan dedupe is
  derived from the active epoch's replica-capable metadata, not from
  user state. The 30548 recall study honestly reports that on real-10k
  the replication is not compelling for recall today, and the
  implementation correctly preserves it as an availability primitive
  for Phase 7.
- **Phase 6 top-graph** is opt-in (`top_graph_enabled = 0` default),
  fail-closed on missing/duplicate visible graph objects, and
  validated thoroughly at both the build-input and codec layers. The
  three-way binding (build, scan, diagnostic) is consistent and the
  default scan path is unaffected.
- **Phase 7 design + storage-node endpoint + merge helper** are
  correctly disjoint. The design states the model, the SQL endpoint
  proves a single-node leaf-scan slice works against the existing
  manifest/object-store surface, and the merge helper is a pure
  testable function with a clean validation surface.

### Cross-cutting findings the coder should track

1. **Inner-product helper duplication.** `build/top_graph.rs:237-239`
   redefines `top_graph_inner_product` after the 30551 followup
   consolidated `rank_centroid_routes_by_ip` to share IP scoring
   across build/scan/split paths. Re-use the routing-plan helper or
   hoist a single primitive — see 30550 and 30551 feedback.

2. **`vec_id` global-uniqueness precondition is now load-bearing.**
   The merge helper dedupes by raw `vec_id` bytes; remote-search
   produces vec_ids from each node's local allocator. Until SPIRE
   lands global vec_id (called out as deferred in 30552), the merge
   helper is only safe when the coordinator targets a single node.
   Recommend adding an assertion or doc-comment to
   `merge_remote_search_candidates` that the input set must come
   from a single coordinator-scoped namespace, so a future caller
   does not invoke it across nodes prematurely. See 30552 and 30554
   feedback.

3. **Insert-path boundary fanout dedupes implicitly via tree
   hierarchy.** The build path explicitly dedupes routed PIDs
   (`route_boundary_assignment_for_vector` in
   `build/routing_plan.rs:120-166`), but the post-build insert path
   relies on the implicit "each leaf has one parent" hierarchy
   invariant to keep replica PIDs disjoint from the primary. Future
   hierarchy work (e.g. shared-children, top-graph-aware insert)
   could break this. Either dedupe defensively in `insert.rs` or
   document the invariant at the call site. See 30544 feedback.

4. **Distance-offset construction at top-graph build is O(n²) over
   centroids.** Acceptable now (root-only nodes), but should be
   replaced with a `max ‖c‖²` pass before the design's larger
   "graph-searchable top routing child set" lands. See 30550
   feedback.

5. **Top-graph not consulted by aminsert.** Inserts route through the
   recursive (non-graph) router even when the published epoch
   contains a graph object. That is acceptable for v1 but means
   "graph-searchable scan" and "linear-routed insert" can disagree on
   the favored leaf for a given query. Worth a code comment so the
   asymmetry is intentional. See 30544 feedback.

### Validation gaps

- 30553 (`ec_spire_remote_search`) currently has only one PG18 SQL
  test. Coverage for strict-mode unavailable placements and for
  consistency-mode mismatch would harden the fail-closed contract
  before coordinator fanout integrates.
- The merge helper has unit coverage (3 tests) but no end-to-end
  test that walks `ec_spire_remote_search → merge_remote_search_candidates`
  yet. That is appropriate for this slice — but worth flagging as
  the gate for the next coordinator slice.

No blockers for the current slices to remain landed. The notes above
are forward-looking — they should be addressed before the next
significant Phase 7 step (libpq fanout) lands.
