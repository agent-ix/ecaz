# Review: Task 41 coordinator relation guards (919)

## Summary

Six SPIRE coordinator/routing SQL functions migrated. Includes the
non-trivial `ec_spire_prepare_coordinator_insert_tuple_payload`
which chains `spire_classify_centroid` followed by
`spire_coordinator_insert_prepare_remote_tuple_payload` and then
enters `Spi::connect_mut`. Baseline 4602 → 4590 (−12).

## Confirmations

- **Guard correctly dropped before SPI.** In
  `ec_spire_prepare_coordinator_insert_tuple_payload` the `{}`
  block scopes the two AM calls and returns
  `((node_id, centroid_id, served_epoch), prepare_row)`; the guard
  drops at block exit, then `Spi::connect_mut(|client| {...})`
  proceeds. Matches the prior explicit-close point exactly. ✔
- **Inner-block error paths unwind through Drop.** The two
  `.unwrap_or_else(|e| pgrx::error!("{e}"))` calls inside the `{}`
  block panic during unwind; the guard's `Drop` runs as the panic
  propagates through the `{}` boundary on its way to pg_guard.
  Same close behavior as the old `index_close` immediately before
  the `pgrx::error!` (which was followed by an explicit close on
  the previous error-path in some callers). ✔
- **Read-only diagnostics.** The other five migrations
  (`root_routing_snapshot`, `routing_centroid_snapshot`,
  `classify_centroid`, `plan_coordinator_insert`,
  `plan_coordinator_insert_dispatch`) follow the same single-AM-call
  `{}` capture pattern as 915-918. ✔

## Findings

No blockers.

- A comment at the `{}` block in
  `ec_spire_prepare_coordinator_insert_tuple_payload` would help a
  future reader spot the intentional pre-SPI close — e.g. "guard
  dropped before SPI; see Task 41 strategy matrix". Stylistic.
- The variable rebinding `((node_id, centroid_id, served_epoch),
  prepare_row) = { ... };` is slightly opaque vs. the older two-let
  shape. Considered acceptable since the early-drop intent it
  encodes is the load-bearing change.
