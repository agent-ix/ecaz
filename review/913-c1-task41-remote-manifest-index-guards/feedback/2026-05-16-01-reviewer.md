# Review: Task 41 remote manifest index guards (913)

## Summary

Migrates nine SPIRE remote-node/epoch diagnostics plus
`ec_spire_persist_remote_epoch_manifest` (the only function in this
packet that holds the guard across a `Spi::connect_mut` block).
Validation-only catalog reader
`ec_spire_remote_epoch_manifest_catalog` switches to the
`{}`-scoped validator pattern. Baseline 4684 → 4660 (−24).

## Confirmations

- **Persist path: relation lifetime correct across SPI write.**
  `ec_spire_persist_remote_epoch_manifest` opens the guard once
  (line 2611-2612), reads `spire_remote_epoch_manifest_summary` and
  `spire_remote_epoch_manifest_plan` before entering SPI, then calls
  `am::spire_active_epoch(index_relation.as_ptr())` **inside** the
  `Spi::connect_mut(|client| {...})` closure for the active-epoch
  recheck (line 2643). This requires the guard to remain live through
  the SPI block — and it does, because `index_relation` is captured
  by the closure via `as_ptr()` (the guard itself is owned by the
  outer scope and dropped at function return). AccessShareLock is
  legal to hold across SPI in the same transaction. ✔
- **Error branches now rely on Drop.** The previous code closed
  before `pgrx::error!` for the `manifest_decision !=
  "emit_distributed_epoch_manifest"` and empty-included-rows cases
  (deleted at lines 2618 and 2630 of the old code). Both branches
  now unwind through the function-scoped guard's `Drop`, which is
  correct under pgrx 0.17 panic-to-elog semantics. ✔
- **Catalog reader.** `ec_spire_remote_epoch_manifest_catalog`
  validates with a `{}`-scoped `_index_relation` then uses
  SPI-only catalog SQL. Matches the old immediate-close-after-open
  pattern. ✔
- **Eight read-only diagnostics.** The remaining migrations
  (`ec_spire_remote_node_descriptor_readiness`, its summary,
  `ec_spire_remote_node_capability_plan`, capability summary,
  `ec_spire_remote_epoch_publish_plan`, publish readiness,
  publish gate summary, manifest plan, manifest summary) are
  mechanical: one AM read, then `TableIterator::*`. Guard drops at
  function return, same as the prior explicit close. ✔
- **Baseline arithmetic.** −24 entries roughly matches nine
  migrated functions plus the persistence function's pair of
  pre-error closes.

## Findings

No blockers.

- The persistence function's hold-across-SPI semantics are now
  implicit (guard lifetime = function lifetime via `as_ptr()`
  capture). A one-line comment at the `Spi::connect_mut` site
  noting "guard must live until the closure returns" would protect
  this against a future refactor that pre-extracts `as_ptr()` into a
  local and accidentally drops the guard early. Not required.
- The two error-path `index_close` removals depend on the
  Drop-on-`pgrx::error!`-unwind invariant (carried over from 911).
  Worth flagging in any Task 41 closeout that this is now the load-
  bearing contract across the migrated surface.
