# Feedback: Parallel Scan Local-Only Retry Publication

**Reviewer:** Claude (Opus 4.7)
**Date:** 2026-04-21
**Head reviewed:** `5f5211a` (landed), plus follow-up `b7d8751`

## TL;DR

**Merged correctly.** `parallel_local_only_output_active`
(scan.rs:5706) gates whether the current row is published into the
shared coordinator result slot. When set, the worker still
publishes runtime state but hides the row from shared visibility
while it sits in the foreign-owner local-only fallback. Follow-up
commit `b7d8751` correctly keeps the retained blocker visible in
the shared worker snapshot even while the row itself is hidden.

## Verification performed

- Gate at scan.rs:1179: `current_result.has_element()
  && !opaque.parallel_local_only_output_active` controls whether
  pending heap tids and current result are published. The flag
  cleanly separates "this worker is producing output" from "this
  output is shared-visible."
- Flag clear points: shared retry entry (3670), successful consume
  (3721), emit_materialized entry (3739), discard (3760), parallel
  bind/clear (1285, 1307). Symmetric with the 507 retained-blocker
  lifecycle.
- Flag set points: graph local-direct emit (scan.rs:5011), linear
  local-direct emit (5084). Only on the `KeepLocalEmit` fallback,
  never on success paths.
- Follow-up commit `b7d8751` at scan.rs:1126–1132 publishes the
  retained blocker when `parallel_local_only_output_active` is set
  — so the shared snapshot still says "blocked on X" even though
  the row itself is not visible. Important: without this the
  retained blocker would vanish from shared view between retries
  and multi-worker coordinators would see "nothing blocking" when
  something definitely is.

## Findings

### 1. Split between runtime state and row publication is clean

Worker runtime state (frontier sizes, phase, counters) keeps
publishing. Only the *row* is hidden. That matches the intent —
"local-only between retries" should be invisible to the shared
merge but still observable as a runtime signal.

### 2. Follow-up commit is load-bearing, not cosmetic

`b7d8751` fixes a real signal-loss hazard introduced by the 508
baseline: stale shared blocker visibility. Reviewers on the next
slice should treat the snapshot publication at 1126–1132 as the
canonical pattern.

### 3. Test coverage pins both halves

Tests assert (a) local-only fallback keeps worker snapshot active
and (b) shared coordinator slot is cleared while local-only
fallback is active. Both are necessary invariants.
