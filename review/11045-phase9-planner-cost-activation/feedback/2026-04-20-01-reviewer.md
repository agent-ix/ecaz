## Feedback: Planner Cost Activation (Phase 9) — task-17 closeout pass

Read `src/am/ec_diskann/cost.rs`, `src/am/ec_diskann/mod.rs`, the `amcostestimate`
wire-up in `src/am/ec_diskann/routine.rs`, and cross-checked against
`src/am/common/cost.rs` and `src/am/ec_hnsw/cost.rs`. Also swept the module as
a whole since this packet is framed as the task-17 AM-surface closeout.

### What's right

- `compute_amcostestimate` delegates to `am::common::cost::estimate_planner_cost`
  with the same shape as `ec_hnsw_amcostestimate`. `tree_height = 1.0` is the
  correct single-layer Vamana value — not `0.0` (which would trigger the
  seqscan-branch) and not a synthetic multi-layer fallback.
- Empty-index gate uses `block_count <= FIRST_DATA_BLOCK_NUMBER`, matching the
  `ec_hnsw` convention. The `test_ec_diskann_empty_index_remains_planner_gated`
  pg test pins that boundary.
- The large-table test does not flip `enable_seqscan` off — natural planner
  selection, which is the right bar for "gate is lifted."
- `ECDISKANN_PLANNER_SCAN_ENABLED = true` is cosmetic rather than load-bearing,
  but it matches reality now and is the right thing to flip.

### Concerns (task-17 closeout blockers, not this-packet blockers)

1. **`src/am/ec_diskann/mod.rs:1–7` module header is stale.** It still reads
   "Phase 1A landing: skeleton + handler registration only. All scan, insert,
   build, and vacuum callbacks intentionally error at runtime with `not yet
   implemented` until subsequent phases of task 17 fill them in." Every
   callback in this module is now live (including this packet's
   `amcostestimate`). A reader landing on the module today would be actively
   misled. Suggest rewriting the header to describe the module's current
   responsibility (Vamana AM: build, scan, live insert, vacuum, planner cost)
   and dropping the Phase 1A language. Scope is one file, trivially in-scope
   for this closeout slice.

2. **Recall@10 DoD measurement is not captured anywhere I could find.** Task
   plan lines 586–594 name "Recall@10 ≥ 0.90 at default tuning on the real
   10k fixture (baseline); ~0.95 preferred" as a DoD gate, and Phase 6's
   plan text (line 435) calls the same number out as a "phase-6 blocker."
   Packets 11031/11032 land the scan callbacks and 11045 flips the planner
   gate, but I don't see a recorded measurement artifact in any packet (no
   appendix, no `plan/measurements/` entry, no cited number). Activating
   the planner without the recall number on file means the ADR-011-style
   gate has been lifted with no independent recall signoff — which is the
   specific concern the plan calls out. Before ADR-034 flips to ACCEPTED
   this should be measured and filed (either as an addendum packet or as
   a note appended to this one).

3. **Strict-clippy baseline deferral is accumulating.** Every packet from
   11041 onward carries the same "known baseline" list in its Verification
   section: `unnecessary_sort_by` in `reader.rs` / `scan.rs` / `vamana.rs`,
   test-only `unnecessary_cast` / `needless_borrows_for_generic_args` in
   `scan.rs`, `needless_range_loop` in `vacuum.rs`. That's ~6 strict-clippy
   violations inside `src/am/ec_diskann/` that fail
   `cargo clippy --all-targets --no-default-features --features pg17 --
   -D warnings`. Task-17 closeout should land a dedicated hygiene packet
   so the module passes strict clippy cleanly. The individual per-slice
   deferrals were reasonable, but the aggregate is now a task-17 debt that
   should not cross the DoD line.

4. **`cost.rs:53–54` uses `pgrx::error!` on metadata-read failure during
   planning.** If the metadata page is corrupt or the persisted chain has
   drifted, the planner callback will now abort the query with a `PgError`
   rather than falling back to the gated `f64::MAX` estimate. Today that is
   fine because `read_metadata_page` errors only when the index itself is
   structurally broken — at which point failing loud is arguably correct —
   but the control-flow choice is different from the `ec_hnsw` sibling
   (which ultimately resolves through `shared::read_metadata_page` and lets
   pgrx unwrap guarantees handle it). Worth a comment in `cost.rs` naming
   why the planner is allowed to `error!` here, so a future reader doesn't
   "soften" it by accident and mask a real corruption during `EXPLAIN`.

### Non-blocking nits

- The packet's "Boundary after this packet" bullet "planner cost activation"
  is accurate. Worth adding one line under "Not doing in this packet" naming
  that strict-clippy baseline cleanup and the recall measurement are
  explicitly deferred to a separate closeout packet, so a future reviewer
  doesn't read this slice as the final task-17 packet.
- `DISKANN_SINGLE_LAYER_TREE_HEIGHT = 1.0` is a fine constant to name, but
  consider dropping it behind a short comment (`// Single-layer Vamana;
  see task plan phase 6.`). Future maintainers looking for why HNSW uses
  `metadata_fallback_tree_height(max_level)` while DiskANN hardcodes will
  appreciate the one-line pointer.

### Not blocking signoff on this packet

The planner activation itself is correct and well-tested. The four concerns
above are task-17 closeout items that belong on the path to ADR-034 flipping
to ACCEPTED, not on merging this specific slice. Concern 1 (mod.rs header)
and concern 4 (`cost.rs` error! comment) are narrow enough to fold into this
packet; concerns 2 (recall measurement) and 3 (strict-clippy baseline) want
their own packets.
