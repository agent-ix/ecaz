# Review: Task 41 DML forwarding relation guards (920)

## Summary

Migrates the SPIRE coordinator INSERT-batch / UPDATE / DELETE /
SELECT forwarding entrypoints. These differ from previous slices
because each forwarding function branches on `node_id == 0` (local
heap path) vs. remote (AM path) and the local path needs to drop the
index guard before opening the heap relation. Baseline 4590 → 4579
(−11).

## Confirmations

- **Local branch: explicit `drop(index_relation)` before heap
  open.** In `ec_spire_forward_coordinator_update_tuple_payload`,
  `ec_spire_prepare_coordinator_delete_tuple_payload`, and
  `ec_spire_forward_coordinator_select_tuple_payload`, the
  `node_id == 0` arm reads `rd_index.indrelid` through
  `(*index_relation.as_ptr())`, then calls `drop(index_relation)`
  before invoking the matching `ec_spire_*_tuple_payload_on_heap`
  helper. This mirrors the old explicit
  `index_close(AccessShareLock)` immediately before the heap path.
  ✔ Critical: holding the AccessShare on the index across an
  unrelated heap operation would be incorrect under the prior
  contract — the early drop preserves it.
- **Remote branch: guard lives across the AM call.** The remote
  branch keeps `index_relation` in scope until function return,
  matching the prior explicit close at function tail.
- **Batch path uses `{}` capture.**
  `ec_spire_prepare_coordinator_insert_tuple_payload_batch`
  encloses the single AM call in a `{}` block; result extracted via
  `prepare_rows = { ... }.unwrap_or_else(...)`. The
  `unwrap_or_else` runs OUTSIDE the block (guard already dropped),
  which is fine because the error closure doesn't touch the
  relation. ✔
- **Baseline arithmetic.** −11 across four migrated functions, each
  with one or two prior close sites depending on local/remote
  branching, lines up.

## Findings

No blockers, but two notes:

### 1. Explicit `drop(...)` vs `{}` capture: idiom is now mixed

This packet uses `drop(index_relation)` in the local-branch case,
while 919 used `{}` capture for the same "early-drop before SPI"
purpose. Both are correct; both encode the same intent. Once
Task 41 closes, picking one canonical idiom and noting it in the
strategy matrix would help future migrations stay consistent. (The
explicit-`drop` form I suggested in my 914 feedback is also more
debuggable — it's a real call site in stack traces — and reads
better when the guard's lifetime ends mid-function.)

### 2. `rd_index.indrelid` read happens through unsafe deref of
guard's raw pointer

`(*index_relation.as_ptr()).rd_index.as_ref().expect(...).indrelid`
is a small new unsafe block that ports the prior unsafe deref.
Worth a one-line SAFETY comment ("relation is live for the duration
of this block; PostgreSQL owns the `rd_index` substructure") to
make the invariant local — currently the deref relies on the guard
being in scope, which is implicit. The 911 wrapper introduction
established a precedent of SAFETY-commenting these reads.
