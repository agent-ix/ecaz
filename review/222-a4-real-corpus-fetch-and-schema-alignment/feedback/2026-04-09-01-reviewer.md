## Overall Assessment

Very solid progress. The three packets collectively close the structural
gap that made review 218 a blocker: the dataset is staged, the converter
handles the real schema, the loader verifies a real manifest, and the
metric contract inside the probe now matches NFR-003 semantics. Finding
the iter_batches memory path on the real 1M corpus was the right fix —
`pyarrow.dataset(...).to_table()` at 1536 dims × 1M rows is genuinely not
viable at ~5 GB RSS, and pinning that as "landed path" in the packet is
useful institutional memory.

What is genuinely good:

- **`ExternalRecallContext` refactor** (`src/lib.rs:6306`) is exactly the
  shape item 4 asked for. Corpus + queries + fp32 ground truth + ctid map
  + precomputed exact-quantized top-10 row indices all live in one struct
  that gets built once inside a single `Spi::connect` and passed to four
  gate config probes. The cost of running 4 configs is now ~1.02× the
  cost of 1 config instead of 4×. That shaves the gate from "tens of
  minutes" back toward "minutes" on 50k×1k at 1536 dims.
- **Item-wise MAE via operator-facing `<#>` scores** at `src/lib.rs:6561`.
  Using `debug_gettuple_scan_heap_tids_with_scores` and negating the
  ascending operator score back to inner-product space is the right call:
  it compares the quantizer's *estimated* score against the fp32 truth
  *for the same items*, which is definition (a) from review 218 item 2.
  The column name `mean_abs_score_error` now matches the computation.
- **Canonical selection rule** pinned in `docs/RECALL_REAL_CORPUS.md:47-53`
  with explicit rows and explicit lexicographic ordering, and implemented
  deterministically by `scripts/qdrant_dbpedia_to_tsv.py` via the
  `_ReverseSourceId` / `heapq` max-heap pattern at
  `qdrant_dbpedia_to_tsv.py:81-224`. This avoids materializing the full
  1M id column to RAM just to pick the smallest 51k. Nice.
- **Schema auto-detect with the real column names** in
  `ID_COLUMN_CANDIDATES` and `VECTOR_COLUMN_CANDIDATES`
  (`qdrant_dbpedia_to_tsv.py:34-42`). Including
  `text-embedding-3-large-1536-embedding` explicitly means the one-shot
  helper actually works zero-flag against the Hugging Face release, which
  is the "single command" promise review 221 made and review 222
  delivered.
- **Manifest verification** (`scripts/load_real_corpus.py:239`) enforces
  `manifest_version`, `prefix`, `dimension`, basename, row count,
  SHA-256, and first/last id. The bad-manifest rejection trace in review
  221 evidence is exactly the guardrail this lane needed. The
  `_derive_manifest_path` auto-discovery for canonical
  `<basename>_{corpus,queries}.tsv` naming is a nice touch — it makes the
  happy path zero-flag too.
- **Unit norm verification** (`VectorNormStats`, online mean) closes
  review 219's "new finding A" — the loader now logs
  `count=50000 mean=1.000000 min=1.000000 max=1.000000` and would warn if
  the real corpus diverged from unit norm. That directly explains why the
  smoke-test MAE~8 observed in review 219 was a non-unit-input symptom
  rather than a metric bug.
- **Structural memory fix**. Moving from `pyarrow.dataset(...).to_table()`
  to `ParquetFile.iter_batches(columns=..., batch_size=...)` is the right
  long-term pattern for this dataset. Reading only `_id` and
  `text-embedding-3-large-1536-embedding` (and skipping `title` / `text`)
  is why the converter now actually completes on the 1M corpus.
- **Recording the failed paths** in review 222 itself (numeric id
  assumption, generic schema inference, dataset scan memory blowup) is
  exactly the right institutional-memory framing. Future reviewers can
  see the trap without having to rediscover it.

## What is still open on this lane

### Open 1 — NDCG convention still hybrid (review 218 item 5)

`src/lib.rs:6409-6439` still has:

```rust
let relevance: HashMap<usize, f32> =
    true_top_k.iter().take(k).map(|(i, s)| (*i, *s)).collect();
// gain = fp32_score if item ∈ truth top-k, else 0
```

That is still "score-weighted hit rate at rank", not standard graded NDCG
and not standard binary NDCG. This is the last metric-contract drift on
the lane. For cross-library comparability against ann-benchmarks /
big-ann-benchmarks numbers it should be either:

- **Binary NDCG:** `relevance = 1.0` for items in the true top-k, else 0.
  Directly comparable to ann-benchmarks. Cleanest.
- **Standard graded NDCG:** build `relevance` from a wider band (e.g.
  true top-100), and for a predicted item look up its true fp32 inner
  product regardless of whether it is in the top-10 gate set.

Pick one. Binary is safer if we want to quote "recall-like" numbers next
to NDCG; graded is only worth the complexity if NFR-003 explicitly
requires it.

### Open 2 — ann-benchmarks one-time reference anchor (review 218 item 7)

Still worth opening a `#[ignore]`d probe that points at a published

`build_external_recall_context` or the converter has a subtle bug, the

## New observations on landed code

### N1 — Manifest verification re-reads the full TSV before load

`scripts/load_real_corpus.py:208` streams the whole corpus + queries
files end-to-end once through `_inspect_vector_file` to hash them, then
the `COPY` path in `_load_corpus` / `_load_queries` reads them again for
the actual load. On a 50k × 1536 TSV that is roughly 150 MB read twice
per load. It is a one-time pre-load pass so in absolute terms the cost is
fine and the manifest integrity check is the whole point — but on the
full 1M × 1536 TSV (~3 GB) the double-read starts to cost real seconds.

**Not blocking.** Possible tightening later: inline the hash update into
the `COPY`-streaming loop inside `_load_corpus` / `_load_queries` and
defer the `sha256()` equality check until after the load completes,
rolling back the table if the hash differs. That keeps the guardrail and
halves the disk traffic. Only worth doing if the full-1M lane becomes the
default; for 10k / 50k the current shape is fine.

### N2 — Review-cycle hygiene: converter in 221 was untested against real parquet

The converter landed in review 221 without being exercised against the
actual Hugging Face parquet. Review 222 then had to rediscover both the
`_id` string-id failure and the `text-embedding-3-large-1536-embedding`
schema mismatch. That is a cheap-to-notice gap: we had the canonical
dataset URL in `docs/RECALL_REAL_CORPUS.md:19` and the Hugging Face
metadata is visible without pulling all 27 GB, so the schema could have
been sanity-checked with a single `pq.read_schema()` call against one
shard before the 221 review packet went out. Mentioning it here only so
that future "I added a converter" reviews include a schema-sanity
subsection before being declared "landed".

### N3 — `_load_selected_rows` makes two full scans of the parquet

`qdrant_dbpedia_to_tsv.py:202` scans once for `_load_sorted_ids` (id
column only) and then `qdrant_dbpedia_to_tsv.py:227` scans a second time
for `_load_selected_rows` (id + vector columns). On the 1M corpus that is
two full iter_batches passes over 1M rows of id plus one full pass over
1M rows of 1536-dim vectors. It is correct and bounded-RAM, but on 1M the
second scan dominates wall clock.

**Not blocking.** Current 10k and 50k subsets complete fast enough to
ignore this. If the full 1M lane ever becomes a supported profile, the
selected-id set is known up front, so the selection pass and the
materialization pass could be fused into one scan by maintaining both
the smallest-ids heap and a `dict[str, list[float]]` of already-seen
selected rows, evicting vectors that fall out of the heap. Not worth
doing until 1M is a real target.

### N4 — `source_parquet` manifest path is ephemeral

`qdrant_dbpedia_to_tsv.py:410` records the absolute input path via
`os.path.abspath(args.parquet)`. For the real fetch this is the local
per-developer `/home/peter/dev/datasets/...` path. The manifest is
expected to be committed alongside the first official DBpedia run, which
means a reviewer on a different machine will see a path that does not
exist on their filesystem. Consider either stripping this to the
dataset name / shard list, or keeping it but adding a
`source_parquet_basename` field that is portable. Cosmetic, low
priority.

### P2 — ann-benchmarks reference anchor probe (review 218 item 7)

Stand up a new `#[ignore]`d probe, e.g.
`tqhnsw_graph_scan_recall_ann_benchmarks_reference`, that:

- documents in a new `docs/RECALL_ANN_BENCHMARKS_ANCHOR.md` which

`scripts/load_real_corpus.py:_load_corpus` / `_load_queries`. Expected

Scope is small (one test function, maybe a shared helper), isolated from

### P5 — Recall investigation harness for the "below 0.89" scenario

Pre-stage the tools we will want if the first real 50k DBpedia number
lands below the 0.89 gate. This is speculative scaffolding, but the whole
point of this lane is that we do not know the real number yet, and if
the number is bad we want the investigation tools ready before we start
investigating. Concretely:

- a per-query recall histogram surface that reports "how many queries
  landed at recall 10/10, 9/10, ..., 0/10" (already almost free from
  `ExternalRecallContext` but not exposed)
- an `ef_search` sweep helper that runs a single context against a
  parameter grid `[40, 64, 100, 128, 160, 200, 300, 500]` and emits one
  row per configuration
- an "exact vs approximate" diff query: "for queries where graph recall
  < K, what was exact-quantized recall?"

These are all pure additions on top of the already-landed
`ExternalRecallContext` and do not touch the hot path or the loader.
They turn the first DBpedia run from a yes/no result into a diagnosable
result.

### P6 — Portable manifest fields (observation N4)

Add a `source_parquet_basename` field to the manifest and stop pinning
an absolute per-developer path as the only source reference. Isolated
two-line change in `_write_manifest`, plus a compatibility note in the
loader's `_verify_manifest`. Makes the committed first-run manifest
portable across reviewers' machines. Cosmetic but coder-2-sized and
isolated.
