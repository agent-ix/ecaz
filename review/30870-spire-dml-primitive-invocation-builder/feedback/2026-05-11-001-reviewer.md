# Review: SPIRE DML Primitive Invocation Builder (30870)

## Summary

`SpireDmlFrontdoorPrimitiveInvocation` — the executor-ready
argument bundle. Combines:
- typed primitive plan (30866) — index OID, mode, primitive name,
  PK argument, column payloads
- runtime PK byte resolution (30868) — bytea for both const and
  param paths

`dml_frontdoor_primitive_invocation_from_plan(plan, params)`
builds the bundle from a typed plan and `ParamListInfo`. Reuses
the existing const/param byte conversion so both paths share one
bytea boundary. Re-exported through SPIRE AM module surfaces.

PG18 coverage: const PK SELECT invocation; bound-parameter PK
invocation.

The three review-focus questions:
- Right handoff for upcoming DML CustomScan executor: yes — one
  typed input bundle, all bytea/string fields resolved.
- Const/parameter PK byte evaluation at the right executor-
  adjacent boundary: yes.
- Builder/coverage slice without enabling rewrite or dispatch:
  confirmed.

## Findings

### P3 — `pk_value` becomes `Vec<u8>` here

The invocation owns `pk_value: Vec<u8>`. Cheap (8 bytes for
bigint). Same allocation pattern as 30864's encoder. Worth
confirming this is the only allocation path so a future change
to fixed-size arrays could land in one place.

### P3 — Coverage covers happy paths only

The two PG18 fixtures are happy-path const + happy-path param.
Worth focused negative coverage:
- `ParamBigint` with NULL parameter (already covered indirectly
  via 30868).
- Plan with `index_oid == InvalidOid`.

The next executor packet (30873 in flight) presumably exercises
these via the actual primitive dispatch.

## Reviewer recommendation

Accept.
