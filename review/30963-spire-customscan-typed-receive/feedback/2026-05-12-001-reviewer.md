# Review: SPIRE CustomScan Typed Receive (30963)

## Summary

**Major Phase 12.2 milestone.** Two rows close together:

- endpoint negotiation prefers typed `pg_binary_attr_v1`
  payload only when remote advertises ready;
- `EcSpireDistributedScan` slot materialization now uses
  per-attribute binary receive for typed payloads instead
  of JSON/text coercion.

JSON path retained as **migration fallback** only. Fixture
extends remote row projection to include `text[]`, a
domain, and a named composite — all round-trip through
typed transport with `typed_payload_probe=ready,pg_binary_attr_v1,t,t`.

This is the production cutover for typed transport receive.
Throughput measurement and JSON endpoint retirement remain
open (correctly).

## Findings

### P1 — Capability gate is the load-bearing primitive

The negotiation rule "prefer typed only when remote
advertises ready" is the right safety primitive for a
mixed-version cluster. A coordinator that defaulted to
typed unconditionally would break against any remote that
hadn't yet shipped the typed endpoint. The advertise-then-
prefer rule means the cutover is forward-compatible with
older remotes during the migration window.

Confirm one detail: the capability check should treat
"endpoint identity unreachable" as **not ready** (i.e., fall
back to JSON), not as "assume ready." If the endpoint
identity call fails or returns absent, defaulting to typed
would force a remote to a code path it may not support.
From the request the negotiation reads endpoint identity
explicitly, which suggests the right behavior — worth
verifying via the test suite that the absent/error case
falls back rather than fails.

### P1 — Fixture column coverage spans the type-system axes

The fixture covering `text[]` (array), domain, and named
composite in one round-trip is the right coverage shape.
These three plus the existing scalar coverage from 30915
hit:

- scalar (bigint, text);
- array (`text[]`);
- domain (over base type);
- composite (named record);
- NULL (covered separately);
- empty projection (covered separately).

That's the full v1 type-class matrix. If remote-origin
typed transport works for all six classes through CustomScan
slot materialization, the production cutover is type-
complete.

### P2 — Failed-attempt artifact discipline

The note that earlier `customscan-read-typed*` artifact
directories document failed debug attempts and are
**intentionally not cited as success evidence** is exactly
the right disclosure shape. Same pattern as 30961's setup-
retry note. Preserves debug provenance (intermediate remote
heap resolution + typed collation decode failures) while
keeping success evidence unambiguous.

This is a good template for future cutover packets where
the path from "intermediate failure" to "final fix" is
non-obvious — the failed attempts become the trail an
auditor can follow to understand what the fix actually did.

### P2 — Tracker scope discipline

Closes only the two rows it actually proves:

- endpoint negotiation: yes (capability gate live).
- CustomScan typed receive: yes (slot materialization
  proven via fixture).

Does NOT claim:

- tuple-heavy throughput measurement;
- production JSON endpoint retirement.

Both correctly remain open. Honest scoping.

### P3 — JSON fallback exit path

The note that the JSON path is "retained as migration
fallback" should pin **when it can be removed**. From 30896
P1 (typed transport design) the exit criterion was:

> JSON fallback may remain production-reachable for one
> minor-version compatibility window after the typed
> endpoint release; removal requires all scalar, array,
> composite, NULL, and domain fixture classes to pass
> through typed transport and a reviewer-accepted packet
> that names any unsupported type gaps.

With this packet, the fixture-class condition is met
(scalar 30915 + NULL/array 30916 + domain/composite 30917 +
empty 30918 + production CustomScan 30963). The remaining
gate is the minor-version compatibility window. Worth a
tracker note that the JSON retirement row is now
**unblocked except for the compat window** so the next
JSON-retirement packet can move directly to retirement
once the window closes.

### P3 — Per-attribute binary receive performance assumption

The cutover assumes per-attribute `typsend`/`typreceive` is
faster than JSON for tuple-heavy workloads. That's the
hypothesis behind the cutover. The 30964 transport-selector
knob makes the JSON-vs-typed comparison measurable, which
is the right setup for proving the assumption with 30965+
bench infrastructure.

If 30964/30965 measurement shows typed is *not* faster for
some subset of workloads, the negotiation logic is the
right place to special-case (e.g., per-payload-shape
preference). For now, prefer-typed-when-available is the
right default.
