# Review: Task 41 SPIRE custom scan planner guards (31175)

## Summary

Wraps PostgreSQL resources in
`src/am/ec_spire/custom_scan/planner.rs`. Reuses the pre-existing
`OpenIndexRelation` from `custom_scan/explain.rs` for planner
candidate index eligibility. Introduces **four new module-local
RAII guard structs** for the planner's
`custom_scan_index_has_sql_placement` helper:

1. `OpenTableRelation` — `table_open`/`table_close` pair
2. `ActiveSnapshotGuard` — `GetLatestSnapshot` +
   `RegisterSnapshot` + `PushActiveSnapshot` /
   `PopActiveSnapshot` + `UnregisterSnapshot`
3. `IndexScanGuard` — `index_beginscan` / `index_endscan`
4. `TupleTableSlotGuard` — `MakeSingleTupleTableSlot` /
   `ExecDropSingleTupleTableSlot`

Baseline 4,321 → 4,319 (−2); `planner.rs` 39 → 37.

This is **the first multi-resource-family packet** — it
introduces three brand-new RAII wrapper types for three resource
families (snapshot, index scan, tuple slot) that Task 41
invariant #3 explicitly names as targets. The migration itself is
correct, but it locks in a per-module pattern for resource
families that are about to repeat across the codebase. See
finding #1.

## Confirmations

- **Drop order is correct.** In
  `custom_scan_index_has_sql_placement`, the guards are
  declared in order:
  1. `placement_table` (OpenTableRelation)
  2. `placement_index` (OpenIndexRelation)
  3. `snapshot` (ActiveSnapshotGuard)
  4. `index_scan` (IndexScanGuard)
  5. `slot` (TupleTableSlotGuard)
  
  Rust drops in reverse declaration order: slot, scan, snapshot,
  index, table. That matches the prior explicit cleanup order
  (drop slot, end scan, pop+unregister snapshot, close index,
  close table). ✔
- **`OpenIndexRelation` reuse from explain.rs is fine for this
  packet** (request flagged this) — it's pre-existing, has the
  same shape as 31180's `IndexRelationGuard::access_share`
  conceptually. Worth migrating in the follow-up.
- **Early-return cleanup is now drop-owned.** Previously the
  function had manual cleanup at each early return; the new
  shape relies on guards going out of scope, which handles all
  paths including `?` propagation and `pgrx::error!` unwind.
  Strictly safer. ✔

## Findings

### 1. (significant) Three new resource families landed as module-local guards

This packet is the prototype for the buffer/page/WAL track and
the wider invariant #3 work. Task 41's task file
(`plan/tasks/41-ffi-safety-boundary.md:91-98`) explicitly names
`BufferPin`, `LwLockGuard`, `Snapshot`, and `SpiTuptable` as
RAII wrapper targets. Three of those (snapshot, slot — which is
adjacent to SpiTuptable — and a new one, IndexScan) just shipped
as module-local types.

Each of these resource families will appear in many places:

| Family | Likely callers |
| --- | --- |
| Snapshot (`Register`/`Push`/`Pop`/`Unregister`) | every place that runs index scans against a non-current snapshot — planner, vacuum, custom scans, debug helpers |
| IndexScan (`beginscan`/`endscan`) | similar set |
| TupleTableSlot (`MakeSingleTupleTableSlot`/`ExecDrop...`) | scan execution, custom scans, fetch helpers, test fixtures |
| (incoming) BufferPin | `src/am/ec_*/page.rs`, scan readers, build/insert/vacuum paths |
| (incoming) LwLockGuard | shared/parallel state |

If each one gets a module-local copy in every module that needs
it, the codebase ends up with 20-30 RAII guard structs. The same
proliferation problem the relation guards just demonstrated.

31177 already added a **second** tuple-slot wrapper (`HeapTupleSlot`
in `src/am/ec_spire/scan/relation.rs`) that duplicates this
packet's `TupleTableSlotGuard`. That confirms the trajectory.

**Recommendation:** alongside the
"Migrate remaining module-local relation guards" follow-up I
proposed in 31180 feedback, the next strategy iteration should
add a section to `src/storage/relation_guard.rs` (or split into
sibling files: `relation_guard.rs`, `snapshot_guard.rs`,
`scan_guard.rs`, `slot_guard.rs`) for these resource families.
Each follows the same pattern as `IndexRelationGuard`:

```rust
pub(crate) struct ActiveSnapshotGuard { snapshot: pg_sys::Snapshot }
impl ActiveSnapshotGuard { pub(crate) fn register_latest() -> Option<Self> { ... } }
impl Drop for ActiveSnapshotGuard {
    fn drop(&mut self) { unsafe { pg_sys::PopActiveSnapshot(); pg_sys::UnregisterSnapshot(self.snapshot); } }
}

pub(crate) struct IndexScanGuard { scan: pg_sys::IndexScanDesc }
impl IndexScanGuard {
    pub(crate) fn begin(table: &HeapRelationGuard, index: &IndexRelationGuard, snapshot: &ActiveSnapshotGuard, nkeys: i32, norderbys: i32) -> Option<Self> { ... }
}
impl Drop for IndexScanGuard { ... }

pub(crate) struct OwnedTupleTableSlot { slot: *mut pg_sys::TupleTableSlot }
impl OwnedTupleTableSlot {
    pub(crate) fn make_single(tupdesc: pg_sys::TupleDesc, ops: *const pg_sys::TupleTableSlotOps) -> Self { ... }
}
impl Drop for OwnedTupleTableSlot { ... }
```

Establishing these now in the shared module — before the
buffer/WAL track lands its own buffer-pin and LWLock guards in
parallel — keeps the convention from drifting.

### 2. (minor) Drop-order documentation

The function relies on Rust's reverse-declaration drop order to
match the prior explicit cleanup ordering. This is correct and
stable, but since the order is semantically meaningful (slot
must drop before scan, scan before snapshot, etc.), a one-line
comment near the guard declarations would protect against a
future refactor reordering them:

```rust
// Drop order is reverse declaration order: slot, scan,
// snapshot, index, table. Matches the prior cleanup sequence.
let placement_table = ...;
let placement_index = ...;
let snapshot = ...;
let index_scan = ...;
let slot = ...;
```
