# Review: SPIRE Descriptor Refresh Retry SQLSTATE (30927)

## Summary

Closes 30896 H1. The descriptor-refresh race that I flagged from
30836 now surfaces a **stable SQLSTATE callers can catch on**:

- `ec_spire_register_remote_node_descriptor(...)` raises SQLSTATE
  `40001` (`serialization_failure`) when the descriptor-
  generation monotonic guard rejects a stale generation.
- Error message stable; error detail tells callers to retry the
  whole coordinator write after the winning descriptor refresh
  commits.
- pg_test now catches the specific SQLSTATE and asserts the
  retry detail (not just panic text).
- ADR-069 + `docs/SPIRE_DIAGNOSTICS.md` document:
  - retry contract;
  - failed transaction has not published placement state;
  - failed transaction's remote prepared transaction rolls back.

The three review-focus questions:
- SQLSTATE `40001` is the right stable retry signal for the v1
  descriptor-refresh race: **yes** — `40001` is
  `serialization_failure`, which PG documents as "the
  transaction must be retried." Standard DBI/JDBC drivers
  already auto-retry on `40001`, so applications that rely on
  generic-retry middleware get free coverage.
- Retry contract scoped to the whole coordinator write, not
  just descriptor registration: yes — descriptor registration
  is part of an atomic coordinator transaction; retrying just
  the registration would leave the rest of the coordinator
  transaction state inconsistent.
- Tracker row complete with this focused behavior + docs: yes.

## Findings

### P3 — `40001` matches PG's normal "deadlock-style retry" semantics

A descriptor-refresh race is conceptually the same shape as a
PG serialization conflict — two transactions independently
think they have the latest state, one wins, the other must
retry. `40001` is exactly what PG raises in those cases, so
SPIRE inheriting it is the right symmetry. Application retry
loops that handle PG's regular `40001` cases now handle this
one too.

### P3 — Retry contract matters for connection pools

Most connection-pool clients have built-in `40001` retry. Worth
pinning in the runbook that this is automatic for those tools.
Apps using raw libpq need to opt in.

### P3 — Failed transaction publishes nothing

The doc explicitly says "the failed transaction has not
published placement state." That's the load-bearing safety
property — without it, retry could double-write the placement
row. Confirmed by the existing 2PC abort callback which rolls
back the prepared remote.

## Reviewer recommendation

Accept. Closes H1 cleanly with a SQLSTATE that integrates with
existing PG retry tooling.
