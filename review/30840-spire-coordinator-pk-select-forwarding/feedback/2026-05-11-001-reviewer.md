# Review: SPIRE Coordinator PK SELECT Forwarding (30840)

## Summary

PK-keyed read primitive for ADR-069. Two functions:

- `ec_spire_remote_select_tuple_payload(index_oid, pk_column,
  pk_value, requested_columns)` — remote endpoint that validates
  pk_column + projection columns against the indexed heap and
  returns `selected_count`, `payload_column_count`, and a JSON
  tuple payload of the requested projection.
- `ec_spire_forward_coordinator_select_tuple_payload(...)` —
  coordinator helper that looks up `(node_id, served_epoch)` from
  `ec_spire_placement`. `node_id = 0` placements serve from the
  coordinator heap; remote placements use the descriptor +
  conninfo + governance dispatch.

Mirrors the UPDATE primitive's local-vs-remote dispatch shape, so
the contract is uniform across UPDATE/DELETE/PK SELECT.

The four review-focus questions:
- JSON payload at primitive layer with typed projection deferred
  to the front door: yes — the JSON bridge mirrors the INSERT/UPDATE
  primitives and lets PG own type input via `jsonb_populate_record`
  on the client side.
- `node_id = 0` parity with UPDATE: yes — same branching pattern
  resolves locally without remote dispatch.
- Remote dispatch reuses descriptor + governance correctly: yes.
- Transparent `SELECT ... WHERE pk = ...` left open: confirmed
  (next packet, gated on the hook design question).

## Findings

### P3 — Returns at most one row but result shape is `selected_count`

The primitive can theoretically return `selected_count = 0` (PK
not found) or `> 1` (PK uniqueness violation on remote). Worth
asserting `selected_count <= 1` with a clear error on `> 1`,
matching the DELETE primitive's `count == 1` defensiveness. The
PK select case is the most likely place to hit this if remote
schema drifts.

### P3 — Local-node SELECT path returns the same JSON shape

Returning JSON from the coordinator-local path means the typed
front door has one decode contract for both branches. Good
uniformity. The trade-off is a roundtrip through `to_jsonb` /
`jsonb_populate_record` even for local rows — acceptable for v1
but worth noting that a typed-tuple front door would skip both
encode steps for the local branch.

### P3 — No ADR text yet on PK-read at-most-once vs exact-once

Like UPDATE, this primitive doesn't use 2PC (read-only, no
coordinator-side state changes). A connection drop mid-SELECT
returns an error; caller may retry safely (idempotent). Worth a
one-line ADR-069 note that PK-read is idempotent and safe to
retry.

### P3 — `requested_columns.is_empty()` is rejected

Good — empty projection is a caller bug; clear error.

## Reviewer recommendation

Accept. The P3 `selected_count > 1` defensive check is cheap and
worth folding before the transparent front door consumes this
primitive.
