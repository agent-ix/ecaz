# Review: Packet 258 — C1 Layer-0 Search Bookkeeping Breakdown

Reviewer: peter (via code-review)
Date: 2026-04-12

## Assessment: Good profiling slice, but it opened a bigger question than it answered

### What this packet did right

1. **Identified the accounting gap.** The tracked buckets (upper-layer seed
   1.2ms + layer-0 seed 5.1ms + graph loads 1.8ms + scoring 3.2ms = ~11.3ms)
   don't explain the SQL startup of ~46ms at ef_search=40. Making that gap
   explicit is the right first move.

2. **Finer-grained startup breakdown.** The new timing probes for
   `prepare_query`, `reset_state`, `initialize_entry`, `stage_ordered_results`,
   `initial_prefetch`, `frontier_consume`, and `graph_result_materialize` give
   a more complete picture of where `amrescan` time goes.

3. **Correctly stopped before optimizing.** The data showed the gap wasn't
   where expected (layer-0 bookkeeping), so the packet didn't speculatively
   rewrite the beam search. That discipline continues to be correct.

### Concern: instrumentation weight

The scan.rs diff for this batch adds another ~130 lines of `#[cfg(test)]` /
`#[cfg(not(test))]` timing pairs. Each new bucket is a pair of functions
(test + no-op) plus `Instant::now()` + elapsed conversion in the hot path.

This is getting heavy:

- `ScanDebugProfile` now has **16 fields**
- The `amrescan` function body is interleaved with timing probes to the point
  where the actual scan logic is harder to follow
- The `prefetch_next_graph_result_from_frontier` refactor (while → loop +
  explicit break) was driven by instrumentation, not by correctness

I'm not saying remove it — the data it produces is clearly valuable. But this
is technical debt that should not grow further without consolidation. Consider
a macro or wrapper that compiles to `Instant::now()` under test and vanishes
in release, rather than continuing to add function pairs.

### The 46ms reading

The packet notes that the SQL startup is ~46ms but the AM-internal counters
only explain ~11ms. That's a 4x gap. This is the right question to chase, but
the packet doesn't yet propose where the remaining time might be:

- PostgreSQL executor overhead between `amrescan` and the first `amgettuple`?
- Buffer miss latency during graph page reads (cold cache vs warm cache)?
- The `EXPLAIN (ANALYZE)` overhead itself?
- Subquery evaluation (the `(SELECT source FROM ...)` subplan)?

Packet 259 correctly picks up this thread.

## Suggestions

1. **Consider a timing-probe macro.** Something like:
   ```rust
   timed!(opaque, prepare_query_elapsed_us, {
       store_scan_prepared_query(opaque, &query, &metadata);
   });
   ```
   that compiles to a no-op in release. This would cut the per-bucket
   boilerplate from 8 lines to 1.

2. **Include warm-cache numbers alongside cold.** Every hot-path probe should
   report whether the reading is warm or cold. The 46ms vs 5.6ms gap is almost
   certainly a warm/cold divergence, and mixing the two makes the accounting
   look worse than it is.
