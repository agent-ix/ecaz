# Review: Beam-Driven Bootstrap Top-Up From Visible Frontier (Review 165)

**Reviewer:** Claude Opus 4.6
**Date:** 2026-04-08
**Scope:** Review 165 — replace production top-up loop with graph-owned beam expansion
**Commit:** `e0e1620`

## Summary

Replaces the remaining production bootstrap top-up loop with a graph-owned layer-0 beam expansion seeded from the still-visible frontier. After consumed-source refill, graph-owned beam expansion discovers additional candidates from remaining visible seeds. The old generic top-up helper is retained for tests only.

**Verdict: Approved with one observation.**

## Assessment

This completes the graph-owned traversal handoff for the production bootstrap path. The three integration points are now:
1. Entry seeding → `run_layer0_beam_search` (review 161-164)
2. Post-consume refill → `load_layer0_refill_successors` (review 162)
3. Visible-seed top-up → `expand_layer0_visible_seeds` (this review)

The expansion budget of `bootstrap_limit - visible_len` (passed as `max_successor_candidates`) is the right contract — it fills the frontier up to the limit without exceeding it.

The `Layer0VisibleSeedExpansion` struct (graph.rs:142-146) cleanly separates the two pieces of information scan needs from the expansion: which sources were expanded (for the expanded-source tracking set) and which new candidates were discovered (for the frontier). This avoids scan having to parse a raw `BeamTrace`.

### Observation: test-only `fill_bootstrap_frontier` and `top_up_bootstrap_frontier`

These functions are now gated behind `#[cfg(any(test, feature = "pg_test"))]` (lines 660-672, 674-699). This is acceptable as a temporary split — the test helpers exercise the old loop-based expansion contract, which is useful for regression coverage of the frontier invariants independent of the graph I/O layer.

However, as the graph-owned traversal stabilizes, the test helpers should be migrated to use the same `*_with_successors` internal helpers that production uses (like `refill_successors_with_successors` and `expand_visible_seeds_with_successors`). This would let the tests exercise the actual production code path with mock successors, rather than maintaining a parallel expansion loop.

Not blocking — this is cleanup work for a future review.

## Arc Summary (Reviews 158–165)

This 8-commit arc successfully extracts graph traversal ownership from scan into graph:

| Layer | Before | After |
|---|---|---|
| Neighbor loading | Inline in scan | `graph::load_layer0_neighbor_tids` |
| Successor discovery | Inline in scan | `graph::load_layer0_successor_candidates` |

| Post-consume refill | Manual successor loading | Single-step beam trace |
| Visible-seed top-up | Manual scheduler loop | Beam expansion from visible seeds |

The scan module is now a consumer of graph traversal results rather than an implementor of traversal logic. This is the right architecture for adding multi-layer traversal, prefetch, and read-stream integration later.
