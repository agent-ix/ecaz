# Review: SPIRE Local Store Read-Overlap Harness (30948)

## Summary

Phase 12.8 read-overlap harness row closure. Adds
`ec_spire_index_scan_local_store_read_overlap_harness` as a
narrow companion that exposes per-`(node_id, local_store_id)`:

- `prefetched_object_bytes`
- `read_batch_count`
- `delta_decode_count`

PG18 SQL fixture asserts a two-store post-build delta scenario
returns two touched groups, one read batch per group, positive
bytes, one selected delta decode.

## Findings

### P3 — Narrow surface vs widening placement snapshot

Correct call to keep `ec_spire_index_scan_placement_snapshot`
unchanged. Same reasoning as 30946 — the placement snapshot is
already near pgrx row-shape limits, and read-overlap counters
are a different conceptual axis (I/O accounting vs placement
identity). Three narrow snapshots correlated by
`(index_oid, query)` is more sustainable than one wide one.

### P3 — `read_batch_count` semantics

Defining `read_batch_count` as **one current backend
prefetch/read group per touched `(node_id, local_store_id)`**
is the right framing — it's an I/O-batching counter, NOT a
parallelism indicator. Combined with the
`local_store_execution_mode = 'sequential_backend'` label
from 30946, an operator looking at the harness output can
correctly read it as: "I/O is grouped, CPU is not." Worth a
single sentence in the doc that explicitly contrasts
`read_batch_count` from any future
`concurrent_store_executor_count` so the future label is
unambiguous.

### P3 — `delta_decode_count` reuse contract

The harness `delta_decode_count` should equal the number of
**selected** delta object reads, matching the 30947 reuse
contract (each selected delta object loaded once per route
set). The fixture's "one selected delta decode" assertion is
the load-bearing proof of that invariant under harness
observation. If harness instrumentation accidentally double-
counts on a future `load_delta_rows_for_routes` refactor, this
fixture catches it. Keep.

### P3 — Multi-store fixture coverage

The two-store post-build delta scenario covers the joint
behavior of placement (two stores) + delta path (one delta
object) in one shot. Consider whether a future fixture should
exercise:

- Two stores, both with delta objects (current covers one
  delta object across the route set).
- Two stores where one has zero touched PIDs (does the
  harness emit a row with `read_batch_count = 0` or skip the
  group?).

The skip-vs-zero-row distinction is operator-visible behavior
that's worth a doc line. Optional follow-up if not already
covered.

## Reviewer recommendation

Accept. Closes the Phase 12.8 read-overlap harness row with
honest semantics (I/O batching, not parallelism) and the
right reuse-contract assertion.
