# Review: SPIRE Degraded Skip Report (30954)

## Summary

Phase 12.7 dry diagnostic adding
`ec_spire_remote_search_degraded_skip_report(...)`. Reuses
production libpq dispatch *planning* (no conninfo secret
resolution, no socket open). One row per pre-dispatch skipped
remote in degraded mode:

- `node_id`
- `skipped_pid_count`
- `first_skip_category`
- `status`

## Findings

### P2 — "Dry" property is the load-bearing primitive

The fact that the report does NOT resolve secrets or open
sockets is what makes it safe to call from operator
diagnostics on a coordinator that's degraded for connectivity
reasons. If it tried to open sockets, it would amplify the
degraded condition during diagnosis. Honest design.

Worth a one-line guarantee in the docs: "this function will
not attempt remote connections; safe to call when nodes are
unreachable." Operators reading the diagnostic surface
should know it's a planning-only oracle.

### P2 — Report shape satisfies the row

The four fields cover the Phase 12.7 ask:

- `node_id` → identity
- `skipped_pid_count` → cost (how much work didn't go out)
- `first_skip_category` → reason (operator can categorize)
- `status` → state machine surface

`first_skip_category` is the right granularity — listing all
categories per row would multiply rows for the same node
without operator value. First-category gives the primary
trigger, and operators can drill in via the executor state
surfaces if they need more.

### P2 — Pre-dispatch scope is correct

Scoping to **pre-dispatch** degraded skips (stale epoch,
incompatible version, etc.) is right — these are the
categories the planner can determine without opening
sockets. Live transport/candidate/heap skips happen *after*
dispatch and have different observability needs (they're
already covered by the existing executor state surfaces).

The split is clean: dry pre-dispatch report here, live
post-dispatch via executor state. An operator triaging
"why didn't we dispatch?" goes here; "why did we dispatch but
get nothing back?" goes to the executor state surface.

### P3 — Should this remain degraded-skip-only?

Yes, keep it scoped. Returning all remote dispatch rows
(including ready-and-dispatched) would conflate this with
placement snapshot territory. The diagnostic value is "what
DIDN'T go out and why" — which is a different question than
placement diagnostics answer.

If a future operator workflow needs "all remotes considered,
with dispatch outcome," that's a *different* function (e.g.,
`ec_spire_remote_search_dispatch_decision_report`). Keep this
one narrow.

### P3 — Strict mode behavior

The function is documented for degraded mode. Worth confirming
strict-mode behavior is either:

- empty result (because strict mode fails closed instead of
  skipping); or
- error matching the strict-mode failure SQLSTATE.

Either is reasonable, but the doc should pick one and pin it
so operators don't get a silent empty-result in strict mode
and conclude "no problems" when the actual behavior was a
strict-mode failure they should be looking elsewhere for.

## Reviewer recommendation

Accept. Clean dry diagnostic with the right scope (pre-
dispatch, degraded only). The "won't open sockets" property
is the safety primitive that makes this deployable to
operator diagnostics under exactly the conditions you'd want
to call it.
