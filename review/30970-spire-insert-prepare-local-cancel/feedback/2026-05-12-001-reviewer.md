# Review: SPIRE INSERT Prepare Local Cancellation (30970)

## Summary

**Closes Phase 12.4 INSERT 2PC dispatch cancellation row** —
brings INSERT writes to parity with Stage C read
cancellation. Cancellation watcher around
`coordinator_insert_prepare_remote_sql`:

- polls local `InterruptPending` /
  `QueryCancelPending`;
- sends sync libpq cancel token to remote;
- classifies result as local cancellation;
- on cancel, rolls back open remote transactions;
- if cancel observed immediately after `PREPARE
  TRANSACTION`, explicitly rolls back the prepared txn
  before registering local commit/abort callbacks.

PG18 loopback fixture asserts `local_query_cancelled`, no
matching remote `pg_prepared_xacts`, no visible remote row.

## Findings

### P1 — No-orphan contract is the load-bearing primitive

The whole point of this work is "if the local backend
cancels, no remote prepared transaction is left behind."
The fixture's three assertions are the right oracles:

- **`local_query_cancelled`**: the cancellation classification
  worked (we know we cancelled, not failed).
- **no matching remote `pg_prepared_xacts`**: the cleanup
  ran (no orphan).
- **no visible remote row**: the rollback was effective
  (no half-applied write).

Three independent assertions catch three different bug
classes:

- assertion 1 catches a misclassified failure (cancel
  reported as crash);
- assertion 2 catches a missed rollback (orphan);
- assertion 3 catches a successful prepare that wasn't
  rolled back (would commit on remote restart).

Right oracle set.

### P1 — Cancellation window coverage

The four windows the request names — remote SQL failure,
descriptor metadata failure, `PREPARE TRANSACTION`
failure, and **cancellation observed immediately after
prepare success** — cover the full prepare path. The last
one is the trickiest:

- Local backend issued `PREPARE TRANSACTION`.
- Remote prepared successfully.
- Cancel signal arrived in the narrow window between
  prepare-acknowledged and local commit/abort callback
  registration.
- Without explicit rollback in this window, the prepared
  transaction would survive the local cancel as an
  orphan.

The fix (explicit rollback before registering
commit/abort callbacks) closes that window. This is the
right place to handle it — registering callbacks first
would let the local backend clean up via abort callback,
but the abort callback wouldn't fire if the cancel
short-circuits the path before callback registration.
Explicit rollback in the cancel handler is the correct
fix.

### P2 — Sync libpq watcher is right parity mechanism

Stage C read cancellation uses an async libpq path; INSERT
prepare uses sync libpq. The watcher polling
`InterruptPending` / `QueryCancelPending` and sending the
libpq cancel token is the sync-path equivalent of what
Stage C does asynchronously. Different mechanism, same
contract.

This is the right shape for a sync path — async would
require restructuring the prepare code. The watcher
approach preserves the existing prepare flow and adds
cancellation as an overlay rather than a rewrite.

### P2 — Loopback fixture without shell scripts

Doing this as a `cargo pgrx test pg18` fixture instead of
a shell-driven multicluster setup is right. The cancel
behavior is testable against a single PostgreSQL process
acting as both coordinator and remote (loopback), which is
the simplest reproduction. Shell scripts would add
infrastructure variability without proving anything more
about the cancel path.

The fixture's "remote SQL cancels coordinator while remote
prepare is sleeping" pattern is the right reproduction —
deterministic, fast, no external timing dependency.

### P3 — Coverage of all four cancellation windows

The request names four windows. The fixture exercises one
(cancel during sleeping prepare). Worth confirming the
unit/integration tests cover the other three:

- remote SQL failure with cancel pending;
- descriptor metadata failure with cancel pending;
- `PREPARE TRANSACTION` failure with cancel pending;
- cancel immediately after prepare success (this one is
  the fixture).

If only the fourth has a fixture, the other three are
implicit-by-design. Worth one tracker note that the
fixture covers the trickiest window (post-prepare); the
other three rely on the cleanup logic being shared by
construction.

### P3 — Phase 12.4 lane closure

With this packet AND 30969:

- 13/14 rows closed.
- Only async INSERT dispatch (P9) remains open.

P9 (the async/pipelined INSERT dispatch) is the parallel
to making remote prepares non-serializing. That's a
performance concern, not a correctness gap. The
correctness work in 12.4 is **complete** with this packet.

## Reviewer recommendation

Accept. **Closes the INSERT 2PC dispatch cancellation
row.** Right oracle set, right window coverage (especially
the post-prepare narrow window), right sync-path parity
mechanism. Combined with 30969, Phase 12.4 correctness
work is complete; only P9 async dispatch remains as a
performance follow-up.
