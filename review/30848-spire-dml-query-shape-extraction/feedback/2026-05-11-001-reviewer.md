# Review: SPIRE DML Query Shape Extraction (30848)

## Summary

Adds the bridge from a PG `Query` tree to the 30847 shape
classifier. `classify_dml_frontdoor_query(query, context)`:
- Maps `commandType` to `Update`/`Delete`/`PkSelect`.
- Validates single range-table shape (one entry, `T_RangeTblRef`,
  no joins).
- Detects subquery shape via `hasSubLinks`, `hasModifyingCTE`,
  `hasRecursive`, non-null `cteList`, non-null `setOperations`.
- Detects RETURNING via `returningList`.
- Walks `jointree.quals` looking for an `OpExpr` whose `opno`
  resolves via `pg_sys::get_opcode(...)` to `F_INT8EQ`.
- Extracts the constant or param value kind for the predicate's
  RHS.
- Pulls UPDATE target columns / SELECT projected columns from
  `targetList` non-junk `TargetEntry`s.
- Handles the `target_rtindex` distinction (UPDATE/DELETE use
  `query.resultRelation`; SELECT uses the single range-table
  index).

3 new unit tests cover the query-layer helpers. Coordination with
the 30847 classifier is clean — extraction → input struct →
classification.

The three review-focus questions:
- `pg_sys::Query` extraction boundary is right next layer: yes.
- Strict single-table + bigint-PK equality matches v1 hook shape:
  yes.
- Target-column extraction by non-junk `TargetEntry.resno`: yes
  for v1; the relation-context layer (30850) supplies the
  attnum→name mapping.

## Findings

### P2 — Subquery detection rules out CTEs entirely

`!query.cteList.is_null()` rejects any query with a CTE, including
read-only WITHs that don't modify data. v1 acceptable (the
classifier is conservative by design), but worth pinning that even
`WITH cte AS (SELECT 1) SELECT * FROM tbl WHERE pk = $1` will be
rejected. ADR-069 should either:
- Document that CTE-prefixed DML is not supported in v1; or
- Note that read-only CTEs may be supported in a future packet
  once the classifier learns to walk them.

The `hasModifyingCTE` flag is the more important reject; the
plain `cteList` non-null reject is broader than necessary. Worth
narrowing later.

### P2 — `pg_sys::get_opcode((*op_expr).opno) == pg_sys::Oid::from(pg_sys::F_INT8EQ)`

`F_INT8EQ` is the function OID of the `int8eq` operator's
implementation function. Comparing `get_opcode(opno) == F_INT8EQ`
correctly identifies the equality operator for `bigint = bigint`
regardless of which schema the operator is in. Good — that's the
robust check.

But: this rejects equality between `int4` and `int8` if the
predicate is `pk = 5` where PG implicitly casts the int4 literal
to int8. That should still resolve to `int8eq` after the cast, so
this is fine. Worth a focused test asserting `WHERE pk = 5`
(int4 literal) is recognized as `ConstBigint`.

### P3 — `query_layer_recognizes_bigint_const_and_param_values`

Test name confirms both `Const` and `Param` paths are exercised.
Important — the hook needs to handle prepared statements with
parameter placeholders too.

### P3 — Returning Query tree fact accessor pattern

The `dml_frontdoor_*` helpers each take a `&pg_sys::Query` slice
and return a single fact. That makes adding new safety facts
incremental. Worth a one-line module comment naming the pattern
("each helper extracts one fact, returns Option to signal
unsupported, classifier composes them") for future maintainers.

### P3 — `range_table_ref` returns `Option<i32>`

Good API: `None` means "not a single-table query," which the
classifier translates to `has_join = true`. Cleaner than a
bool/index pair.

### P3 — `target_rtindex` for SELECT defaults to `range_table_ref.unwrap_or_default()`

If `range_table_ref` is `None`, `target_rtindex = 0`. Then PK
predicate extraction uses `target_rtindex = 0` for a non-existent
relation, which won't match any column. The classifier ultimately
rejects via `predicate_column != Some(pk_column)`. Works but is
indirect — worth either short-circuiting earlier on
`range_table_ref.is_none()` for SELECT, or commenting why the
default-zero is OK.

## Reviewer recommendation

Accept. The CTE-rejection breadth (P2) is the only operational
nuance worth pinning in ADR-069 before any application uses
WITH-clauses against an ec_spire-marked table.
