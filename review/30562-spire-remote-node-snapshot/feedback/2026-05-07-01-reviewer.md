# Remote node snapshot review

`ec_spire_remote_node_snapshot` projects per-node placement counts
and a placeholder descriptor state. Correctly diagnostic-only. No
blocking findings.

## Confirmations

1. Local node 0 reports `node_kind = 'local'`, `descriptor_state =
   'active'`, `status = 'ready'` without implying that a remote
   descriptor exists.
2. Nonzero node IDs are visible as `descriptor_state = 'missing'`,
   `status = 'requires_remote_node_descriptor'`, with placement counts
   carried forward from the active epoch placement directory.
3. The coordinator fanout manifest loader is the right boundary —
   `load_relation_epoch_manifests_for_coordinator_fanout` does not
   reject remote placements the way the regular local-store loader
   does, so remote-node placements stay visible to diagnostics
   instead of erroring through local-store validation.
4. No conninfo strings or libpq behavior surfaced.

## Findings

- **`extension_version` source.** `remote_node_capability_plan_row`
  (`root/snapshots.rs:860, 879`) and the empty-row helper at
  `snapshots.rs:908` use `env!("CARGO_PKG_VERSION")` for the local
  side, and the literal `"unknown"` for the remote side until a
  descriptor lands. That is reasonable for now, but should be
  documented (in a comment) that the local value is build-time
  constant from Cargo metadata so a deployment that mixes coordinator
  and storage-node binaries from different builds will see the
  coordinator's view as authoritative until descriptor metadata
  exists.

- **Empty-row helper.** `remote_node_snapshot_empty_row`
  (`snapshots.rs:891-935`) is good defensive scaffolding. Worth
  noting that local node 0 with zero placements still reports
  `status = 'ready'`, which is correct (an empty index is not
  "missing"), but the future libpq executor must not interpret
  `ready` as "has placements" — it should also gate on
  `placement_count > 0` if it intends to dispatch.
