packet describes this as `ForeignAdmittedHead` handoff only, with

intentionally deferred. The **code handles both**

confirms this is intentional.

## Findings

### 1. Packet vs. code mismatch on scope — flag for the implementer

The packet says: "When the blocker kind is `ForeignAdmittedHead`,
that helper now drains the next globally admitted row..." and
lists foreign selected-pending handoff as deferred. The code and
tests both cover `ForeignSelectedPending`. This isn't a correctness
bug, but it misdescribes what landed. Future reviewers reading the
packet alone would underestimate the scope. Worth correcting the
packet (or the code, if the broader coverage was unintended).

### 2. Cross-slot consistency: foreign worker must republish

Taking via the non-owner-aware path for `ForeignSelectedPending`
drains the foreign worker's pending from **coordinator state**,
but the foreign worker's local `TqScanOpaque.result_state` is not
reached. The foreign worker will still see its pending row locally
until its next `publish_parallel_scan_worker_slot_snapshot` call,
at which point it republishes a now-stale pending into the slot.

In practice the coordinator will then see the same pending
re-published and can re-admit / re-take it — meaning the drained
row can briefly reappear as selected-pending from the foreign
worker's slot after handoff. The `PUBLISHED` flag and generation
counters should prevent duplicate emission (the local worker's
emitted-set filters by element_tid), but this is a subtle
eventual-consistency seam. Worth an explicit test that asserts
"after handoff-drain of foreign selected-pending, foreign worker
republishes next row without re-emitting the drained one."

### 3. Handoff-first ordering is right

At scan.rs:3722 the `Blocked` arm tries handoff before recording
explain counters / publishing blocker state. That way, if handoff
succeeds, the Blocked path is skipped entirely and the worker
treats the result as an emit. If handoff returns `None` (e.g.,
concurrent drain emptied the queue), the worker falls back to the
blocker recording path. Correct ordering.

### 4. Non-blocker cursor preservation verified

Both tests assert `local.result_state.current().element_tid()` and
`pending_index()` unchanged after handoff drain. That's the key
invariant for handoff — the local worker still owns its own row,
it just got a different row from the shared queue first.
