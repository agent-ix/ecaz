# Review: Task 41 SPIRE tuple-payload relation guard (31166)

## Summary

Introduces `ec_spire_index_heap_relation_oid(oid, caller)` — a
small validation-only helper that opens a SPIRE index guard,
copies `rd_index.indrelid`, and returns the OID. Drops the guard
at function return. Four tuple-payload entrypoints
(`ec_spire_remote_{insert,update,delete,select}_tuple_payload`)
now call this helper instead of repeating the open/`rd_index`
deref / `index_close` triplet inline. Baseline 4,479 → 4,467
(−12).

I'm satisfied with this slice. It directly addresses the finding I
left on 31160 (DML forwarding) about the duplicated `rd_index`
deref pattern needing a SAFETY comment / consolidation.

## Confirmations

- **OID-only escape.** The helper returns `pg_sys::Oid` (a `Copy`
  value type). No relation pointer leaves the function. Anything
  the SPI work downstream does with the OID is decoupled from the
  guard's lifetime. ✔
- **Guard scope is correct.** The unsafe `(*as_ptr()).rd_index`
  deref happens inside the helper while the guard owns the
  relation. Guard drops at the helper's `}` before returning the
  OID — same effective lifetime as the prior explicit
  `index_close` immediately after the `indrelid` read. ✔
- **SAFETY comment is local and accurate.** The new SAFETY note
  on line ~14671 ("`index_relation` is a live opened index
  relation for the duration of this copy, and `rd_index` is
  PostgreSQL's index metadata for it") is the only unsafe
  justification needed for this pattern, in exactly one place
  rather than four. ✔
- **Validation-only intent preserved.** The four callers all only
  needed the heap OID; they didn't need the relation across SPI.
  The helper enforces that intent by returning before any SPI work
  is possible.
- **Baseline arithmetic.** −12 across four migrations ≈ 3 entries
  retired each (open + indrelid deref + close) consolidated into
  the helper's single unsafe block.

## Findings

No blockers.

- This is the first instance of the "named validation-only helper"
  pattern I suggested in my 31161 strategy feedback (finding #4).
  The companion suggestion was a generic
  `validate_ec_spire_index(oid, caller)` for callsites that need
  only the relkind/AM check with no metadata read. That second
  helper could land in a follow-up packet to convert the remaining
  `{ let _index_relation = ...; }` callsites (31152 / 31154 in the
  manifest catalog functions).
- Naming nit: `ec_spire_index_heap_relation_oid` follows the
  ECAZ-prefix convention but reads as "the heap relation OID *of*
  the index" rather than "*lookup of* the heap relation OID *by*
  index". `lookup_spire_index_heap_oid(index_oid, caller)` or
  `spire_heap_oid_for_index(...)` would be a touch more obvious.
  Not worth changing if 31167 already extends the same name.
