## F2 — `connection_check` performs live libpq probes

At `src/lib.rs:9536–9551` the function reads each row's `provider_lookup_key`
from `std::env::var(...)` and, if a non-empty value is present, calls
`postgres::Client::connect(&conninfo, postgres::NoTls)` and immediately drops
the client. Failures are bucketed into `libpq_connection_failed` /
`requires_conninfo_secret_resolution` / `requires_libpq_executor`.

Consequence: `SELECT * FROM ec_spire_remote_pipeline_steps(...)` opens (and
discards) a real libpq connection to *every* configured remote node on each
call. Operators reaching for the consolidation as a cheap "where am I stuck"
overview will be paying for active remote probes. On a busy cluster, repeated
diagnostic calls add real load to remotes.

**Why this is the right time to flag.** P2's stated goal was a single
consolidated row-per-step surface so operators don't have to chain narrow
queries. F2 means the consolidation is *more* expensive than the narrow
surfaces (each narrow surface probes once; the consolidation probes once
*plus* fans out to executor candidate + heap candidate calls — see F4). The
consolidation should be the *cheapest* place to land first.

**Proposed mitigation (one of):**
- Add an optional `probe_connections boolean default false` parameter that
  defaults to *not* opening real libpq sessions; operators who want live
  probe coverage opt in. The unconnected path still reports
  `requires_libpq_executor` / `requires_conninfo_secret_resolution` based on
  env-var presence, which covers most "where am I stuck" cases.
- Or split into two surfaces: `ec_spire_remote_pipeline_steps_dry(...)`
  (cheap, no socket I/O) and the current
  `ec_spire_remote_pipeline_steps(...)` retained as live-probe.

Non-blocking but worth resolving before this becomes the de-facto operator
runbook entrypoint.

## F3 — count semantics for produced rows

For `candidates`/`heap_candidates`:

```rust
ready_count: candidate_count,
blocked_count: if candidate_count == 0 && recommendation != "none" { 1 } else { 0 },
```

This makes `ready_count` the count of rows produced, not the count of "ready
items in this step". For comparable steps like `dispatch_plan`
`ready_count = pipeline_dispatch_count` (count of dispatch *plans* that are
ready) and `blocked_count = missing_descriptor_dispatch_count` (count of
dispatch *plans* that are blocked). The unit on `ready_count` is therefore
inconsistent across steps:

| step             | `ready_count` unit                         |
|------------------|--------------------------------------------|
| `dispatch_plan`  | dispatch plans                             |
| `connection_check` | connection rows                          |
| `candidates`     | candidate rows produced                    |
| `heap_candidates`| heap candidate rows produced               |
| `manifest_apply` | manifest receive plans ready               |
| `coordinator_result` | coordinator result row count           |

Operators who `SUM(ready_count)` across rows will get a meaningless number.
Recommend either (a) documenting that these counts are step-local and not
comparable, or (b) adding a separate `produced_count` column for steps that
actually return data.

## F4 — separate libpq round-trips for candidates and heap_candidates

The function calls `spire_remote_search_libpq_executor_candidate_rows`
*and then* `spire_remote_search_libpq_executor_heap_candidate_rows`. Both
issue independent libpq queries to the remote node. The consolidation does
not consolidate that network cost — and `heap_candidates` re-runs the
candidate selection internally (per the heap candidate executor's contract).
For diagnostic use this is wasteful but acceptable; for steady-state operator
runbooks it adds amortized load.

Consider in a future slice: a single libpq round-trip that returns both
candidate-only and heap-resolved rows tagged by `result_kind`, and have the
narrow surfaces project from that. Out of scope for 30655.
