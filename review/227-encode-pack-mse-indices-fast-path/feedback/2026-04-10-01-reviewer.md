# Feedback: Encode `pack_mse_indices` Bytewise Fast Paths

**Reviewer:** Claude (Opus 4.6)
**Date:** 2026-04-10
**Commit reviewed:** `ba9cc2f`

## Scope match

Dispatch structure matches the task prompt exactly: a `match bits_per_index`
with closed-form fast paths for {2, 3, 4, 5} and a renamed
`pack_mse_indices_generic` fallback for every other width. Function
signature unchanged. `unpack_mse_indices` and `mse_index_at` untouched
per task out-of-scope.

## Findings

### 1. Bit-order correctness — hand-verified across all four fast paths

I traced each fast path against `write_bits_le`'s little-endian layout:

- **2-bit:** `(d3<<6)|(d2<<4)|(d1<<2)|d0` — d0 at bits 0-1, d1 at 2-3,
  d2 at 4-5, d3 at 6-7. Matches. ✓
- **3-bit:** `u32` accumulator shifted `value << (offset*3)`, then the
  low `ceil(chunk*3/8)` bytes copied via `to_le_bytes`. Partial chunks
  are handled by recomputing `total_bytes` from the chunk length. Traced
  len ∈ {1, 2, 3, 9} — all cases write exactly the bytes the generic
  loop would leave. ✓
- **4-bit:** `(hi<<4)|lo` where lo is `chunk[0]` low nibble, hi is
  `chunk[1]` low nibble (default 0 on odd-length tail). ✓
- **5-bit:** `u64` accumulator shifted `value << (offset*5)`, low 5
  bytes via `to_le_bytes` for full chunks, `total_bytes` recomputed for
  partial tails. ✓

`out = &mut out[total_bytes..]` correctly walks the destination slice
across chunks. Output buffer is zero-initialized before filling, matching
the generic path's zero-init + OR-in semantics.

### 2. Regression test coverage is rigorous

`pack_mse_indices_fast_paths_match_generic`:

- `bits ∈ 2..=7` (note: 6 and 7 trivially match because they both hit
  the generic path — see item 4 below)
- `len ∈ {1, 2, 3, 7, 8, 9, 16, 17, 257, 1536}`
- Every (bits, len) pair: asserts dispatched path produces byte-equal
  output to `pack_mse_indices_generic`, AND round-trips through
  `unpack_mse_indices` to the original indices.
- Deterministic seed (`ChaCha8Rng::seed_from_u64(0xC0FFEE)`).

Covers production size (1536), boundary cases (1, 7, 8, 9), and
non-aligned tails (17, 257). This is stronger than the existing
`mse_pack_unpack_roundtrip_all_widths` test and is the right shape for
proving byte-exactness. ✓

### 3. **Microbenchmark skipped — contrary to task Step 5**

Task L117-124 explicitly asks:
> Add an `#[ignore]`-gated microbenchmark in `src/quant/prod.rs#tests`
> that calls `pack_mse_indices` with 1536 4-bit indices in a 100_000-
> iteration `Instant` loop on both paths and prints the wall clock for
> each. Target: at least **5× speedup** for the 4-bit path. If the
> speedup is below 3×, the bit loop was not as bad as profiling
> suggested — record the actual number and decide whether the
> complexity is worth it.

The packet declines this in §"Microbenchmark — not run on this branch"
with the argument that "the mechanical claim — replace 6144 bit-shift
iterations with 768 byte-stores — is self-evident from the code." That
argument proves the fast path does *less work*, not that the runtime
actually dropped. On modern out-of-order x86 the branching bit-shift
loop could already retire close to 1 bit/cycle; the fast path could be
memory-bound on the 768-byte output. Without measurement we do not know
which.

**This is not a landing blocker** (the change is bit-exact, worst case
is "we added four helper functions that are the same speed"), but it is
a scope deviation from the task spec, and the same deviation appears in
task 10062. The pattern should not be accepted silently across both.

**Request:** land an `#[ignore]`-gated microbench that measures
`pack_mse_indices_generic` vs. the dispatched path on the 1536×4-bit
shape, records the wall-clock numbers in-tree (as a `// measured on …`
comment next to the `#[ignore]`), and notes the ratio. If the ratio is
below 3×, the packet's justification needs a second look — at that point
the complexity of four extra helpers and a dispatch match may not be
worth the gain. Run this as a follow-up in the same branch as the
10062 microbench to keep both optimizations' measurements together.

### 4. Test trivially-passes for widths 6 and 7

The test iterates `bits ∈ 2..=7`, but widths 6 and 7 fall through to
`pack_mse_indices_generic` in both the dispatched path and the reference
call, so the comparison is `generic == generic` at those widths — always
true. The round-trip check is still meaningful for 6/7 (it proves the
generic path itself is correct), but the dispatched-vs-generic cross-check
does no useful work at those widths.

Not a bug. Minor nit: the test docstring could note that 2..=5 is where
the cross-check is load-bearing and 6..=7 is round-trip only. Low
priority.

### 5. `#[allow(dead_code)]` on `unpack_mse_indices`

Unchanged by this branch, but noting: `unpack_mse_indices` carries a
`#[allow(dead_code)]` even though the new test calls it. Worth checking
whether the allow is still needed after this branch lands.

## Verdict

**Approve with follow-up required.** Item 3 is the only item that needs
action: land an `#[ignore]`-gated microbench (paired with the
corresponding 10062 microbench in a single follow-up) and record the
measured speedup in-tree. Items 4 and 5 are polish.

The correctness work on this branch is strong — bit-order verification,
exhaustive regression, partial-chunk math all check out. The only gap is
that we have not actually proven the "win" part of the optimization,
which is the part the task explicitly asked for.
