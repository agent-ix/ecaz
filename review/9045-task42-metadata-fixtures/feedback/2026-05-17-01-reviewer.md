# Review: Task 42 metadata fixtures (9045)

## Summary

Introduces the **golden-fixture lane** for Task 42. Adds
`fixtures/on-disk/hnsw_metadata_v3.hex` and
`fixtures/on-disk/diskann_vamana_metadata_v3.hex` plus the
`tests/on_disk_fixtures.rs` integration suite and a new
`make on-disk-fixtures` target.

Each fixture test does two things:
1. Decodes the hand-pinned hex bytes into expected in-memory
   metadata values.
2. Byte-swaps the fixture's `format_version` field and asserts
   the decoder **rejects** the mutated bytes.

This is the second half of Task 42's stability story — the
static-layout assertions from 9042-9044 lock the byte
**positions**, and the golden fixtures lock the byte
**contents**. Together they catch both layout drift and silent
encoder behavior changes.

I'm satisfied. The "swap-version-and-reject" pattern is the
right shape for endian / version-tag enforcement.

## Confirmations

- **Hex fixtures are hand-pinned, not regenerated.** The test
  reads the `.hex` files at runtime; they live in the repo as
  committed bytes. A future code change that breaks the
  encoder would fail the decode-and-compare test, not silently
  regenerate the fixture.
- **Swap-version test exercises endian failure mode.** Byte-
  swapping the `format_version` field is exactly the
  "what happens if a big-endian host writes these bytes" case
  Task 42 calls out. The decoder rejecting the mutated bytes
  means it's actually checking the version tag, not just
  reading any 16-bit value as success.
- **Test surface.** `tests/on_disk_fixtures.rs` is 844 lines —
  reasonable for the breadth of types covered.
- **Make target.** `make on-disk-fixtures` is a clean CI-able
  entry point. Pairs with `make layout-check` for the two
  halves of Task 42's gate.

## Findings

### 1. Pattern is solid; extend it methodically

The shape — `(decode → assert values) + (mutate format_version
→ assert rejection)` — works well for metadata. The follow-up
packets (9046-9049) extend this same shape to tuple payloads
and SPIRE-specific types, which is the right way to build the
coverage matrix.

For tuple payloads, the analogous "mutate field, assert
rejection" check is more interesting because there are more
fields to mutate. 9046's `neighbor_count` mutation is a good
example — see review on that packet.

### 2. Future addition: cross-version compatibility tests

When ECAZ adds format v4 in some future release, the v3
fixtures should continue to decode (forward compatibility),
and v4 fixtures should be added without invalidating v3. The
current single-version fixtures lock down v3 — that's correct
for the current state — but the multi-version matrix is the
next Task 42 deliverable (`(format_version, AM)` upgrade
matrix in the task doc).

Not actionable in 9045; flagging for sequencing.
