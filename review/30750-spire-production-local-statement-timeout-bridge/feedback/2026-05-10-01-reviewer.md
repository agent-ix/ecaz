- `postgres_local_cancel_failure_category()` at line 2735 is the
  source-of-truth dispatcher: when a query-cancel is pending, it
  checks `get_timeout_indicator(STATEMENT_TIMEOUT_ID, false)`
  (line 2732, with `STATEMENT_TIMEOUT_ID = 3` at line 2721) and
  returns `local_statement_timeout` if the indicator is pending,
  else `local_query_cancelled`. Right boundary ‚Äî
  `STATEMENT_TIMEOUT` raises both flags (cancel + timeout
  indicator), so checking the timeout indicator after observing
  the cancel disambiguates the two cases.
- Critically: `get_timeout_indicator(STATEMENT_TIMEOUT_ID,
  false)` passes `reset = false`. Resetting would consume
  PostgreSQL's timeout indicator and leave subsequent code paths
  (including PostgreSQL itself) thinking the timeout never
  fired. Non-reset is the right choice for a *classifier* that
  doesn't own the indicator's lifecycle.
- Symbol lookup (line 2727) uses `dlsym` for

  pattern from packet 30749 for `InterruptPending` /
  `QueryCancelPending`. Symbol-not-found returns `false` (line

  asserting timeout ‚Äî the safe default.
- New PG18 fixture dynamically schedules a local

  `remote_transport_failed` / `local_statement_timeout`.
- Existing timer-triggered cancellation tests for both

- Phase 11 task and design docs mark the backend interrupt

## Answers to the packet's review questions

- **Non-resetting `get_timeout_indicator(STATEMENT_TIMEOUT,
  false)` the right PostgreSQL boundary?** Yes. The classifier
  does not own the timeout indicator's lifecycle; PostgreSQL
  itself or other consumers may still need to read it.
  `reset = true` would silently break those readers. The non-
  reset call is exactly what a passive classifier should use.
- **Acceptable to classify statement timeout inside the
  existing production `PostgresInterruptPoll` source rather
  than adding a separate variant?** Yes ‚Äî better than a
  separate variant. PostgreSQL's `STATEMENT_TIMEOUT` machinery
  raises the same query-cancel flags (`InterruptPending` /
  `QueryCancelPending`); a separate poll source would race the
  query-cancel poll. Classifying inside the existing source
  preserves a single production interrupt poll while still
  giving operator-distinct categories.
- **Should the parent Phase 11.4 cancellation task now be
  marked complete, or stay open until C5 AM bridge consumes
  this adapter behavior?** I'd keep 11.4 task-level open until
  C5 wires the AM scan path, but mark the **adapter-layer**
  bridge complete (which the docs already do). The reason: a
  reviewer skimming "11.4 complete" might assume cancellation
  works under real query workloads, but until C5 runs the
  executor under a real AM scan, the bridge is only proven
  through `pg_test` fixtures. Per-stage completion with C5 as
  the integration gate keeps the readiness story honest.
