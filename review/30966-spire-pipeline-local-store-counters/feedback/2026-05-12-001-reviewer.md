# Review: SPIRE Pipeline Local Store Counters (30966)

## Summary

Phase 12.9 harness extension. Adds
`--include-local-store-overlap` to `ecaz bench spire-pipeline`,
aggregating
`ec_spire_index_scan_local_store_read_overlap_harness(...)`
per sampled query and `nprobe` sweep:

- route, leaf-route, delta-route sums;
- candidate row sums;
- prefetched object-byte sums;
- read-batch sums;
- delta-decode sums.

Distinct `Local store overlap counters` report section. CLI
parsing/wiring/render coverage; no live PG run. Tracker
update is CLI capture capability, not benchmark evidence.

## Findings

### P2 — Distinct section from local pipeline counters

Right rendering choice. The local store overlap counters
answer "I/O behavior per `(node_id, local_store_id)` group"
which is a different question than the existing local
pipeline step counters answer ("what did each stage of the
pipeline produce?"). Blurring them into one section would
force operators to mentally separate I/O behavior from
stage behavior. Distinct sections preserve the operator
mental model.

### P2 — Aggregation by `(nprobe, node_id, local_store_id)`

Right grouping. The harness from 30948 emits one row per
touched `(node_id, local_store_id)`. Aggregating by adding
`nprobe` as the outer key gives operators "for this nprobe
setting, how did each store group behave?" That's the
question the bench output should answer.

Worth confirming the report renders the three-level grouping
in a stable order (e.g., nprobe ascending, then node_id,
then local_store_id) so diff-of-two-runs is meaningful.
Random row order would defeat artifact comparison. From
the CLI render coverage it likely does.

### P2 — Object-byte and read-batch sums

These two counters are the load-bearing I/O metrics for
the throughput-measurement work that 30964/30965 are
setting up:

- **object-byte sums**: total bytes read per group →
  bandwidth ceiling proxy.
- **read-batch sums**: count of distinct read groups → I/O
  call overhead proxy.

Together they answer "is throughput limited by bandwidth
or by I/O syscall count?" That's the right diagnostic for
sizing future async/parallel store-group execution work
(the future primitive named in 30946).

### P2 — Delta-decode sums

Per the 30947 reuse contract, delta-decode count should
equal selected-delta-object reads (each loaded once per
route set). Aggregating across sampled queries gives
operators visibility into how often delta paths are
exercised vs base-routed work — useful for capacity
planning and for spotting scenarios where delta accumulation
dominates I/O.

### P3 — Tracker scope as CLI capability

Correctly scoped to "CLI capture capability," NOT actual
packet-local benchmark evidence. The harness command
exists; the artifact-capture row only closes when a
packet runs the harness and reports findings. Honest
sub-bullet placement under the still-open Phase 12.9
artifact row.

### P3 — Future per-store-class breakdown

Once async/parallel store-group execution lands, it might
be useful to add a column distinguishing "store groups
that ran sequentially in this backend" vs "store groups
that ran concurrently" — to validate the parallelism
diagnostic. Optional future extension; not needed now.

## Reviewer recommendation

Accept. Right rendering separation, right grouping key,
right counter selection for the throughput measurement
that 30964/30965 are setting up. Honest sub-bullet
placement.
