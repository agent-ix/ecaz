# Feedback: Build Detoast Copy Detection

**Reviewer:** outside-review  
**Date:** 2026-04-05  
**Verdict:** Approved

## Answers to review questions

1. **Remaining detoast ownership edge cases**: Pointer comparison (`!std::ptr::eq(varlena, original)`) is the correct ownership test. PostgreSQL's `pg_detoast_datum_packed` returns the original pointer if no detoast was needed, or a freshly palloc'd copy if it was. There's no third case where the return value aliases the original but still needs freeing. The same pattern is used in `heap_tqvector_gamma` (mod.rs:595), which is consistent. No remaining edge case identified.

2. **Factor into shared helper**: At two call sites (build_heap_tuple and build_heap_tuple_with_source), the inline pattern is clear enough. The third site in `heap_tqvector_gamma` also uses the same pattern. A shared helper would be justified now — something like `with_detoasted_tqvector<T>(datum, f: impl FnOnce(&[u8]) -> T) -> T` that handles detoast + pfree + return — but this is an optional cleanup, not a correctness issue.

3. **Missing toasted-datum regression test**: This is genuinely hard to test in pgrx integration tests because PostgreSQL only toasts values that exceed the TOAST threshold (~2KB). With 1536-dim 4-bit tqvectors at 772 bytes, they fit inline. You would need a wider vector or lower quantization bits to trigger TOAST. Not blocking for this stage.
