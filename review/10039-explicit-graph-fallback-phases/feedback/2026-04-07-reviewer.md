# Review Feedback: Explicit Graph And Fallback Phases

**Verdict: Approved — key structural commit in this batch**

## Phase rename

`Bootstrap` → `GraphTraversal` and `Linear` → `LinearFallback` is the right naming for A3. The old
names implied a two-phase pipeline (bootstrap then linear); the new names make it explicit that these
are alternative execution strategies decided at rescan time.

## Decision point

Choosing `LinearFallback` at `amrescan` time (when seeding fails) is the correct boundary. This is
better than deciding mid-scan because:
1. It eliminates the hidden fallback path inside `select_next_scan_result`
2. The phase is known before the first `amgettuple` call, so debug/test scaffolding can observe it
3. There's no way for a seeded scan to silently degrade to linear

The `enter_linear_fallback_phase` call right after `seed_bootstrap_trace` in `amrescan` is the
single decision site. Clean.

## scan_debug.rs alignment

The five `is_bootstrap()` → `is_graph_traversal()` changes in `debug_bootstrap_phase_transition`
are mechanical renames that keep the debug surface aligned with the runtime. Correct.
