# Review: SPIRE CustomScan Pivot ADRs (30803)

## Summary

Accept the pivot. ADR-067/068/069 are the right shape: CustomScan is the
correct integration point for tuple-direct return without `xs_heaptid`
indirection, and the storage scale-out / coordinator write-pressure
arguments against the AM+mirror path are convincing. The bulk of Stage A–E
work cleanly maps onto the new shape (state machine, transport, fault
matrix, lifecycle matrix all reusable).

The four reviewer-focus questions all answer yes:
- CustomScan vs continuing AM+mirror: correct pivot.
- Coordinator-as-router + remote-shard: matches operator intent.
- v1 write scope (coord-routed CRUD + 2PC, embedding-UPDATE rejected, bulk
  separate): right shape.
- Deferred items list in ADR-069: covers the right Phase 12+ surface.

The findings below are clarifications to fold into the implementation
packets, not blockers on accepting the ADRs.

## Findings

### P2 — ADR-068: tuple_payload missing-row semantics undefined

ADR-068 declares the tuple_payload as a side-channel keyed by
`(node_id, vec_id)` returned alongside the existing 18-column envelope,
but does not state what the coordinator does when a candidate is present
in the envelope but its tuple_payload entry is missing (e.g., the row
disappeared between origin-side scoring and origin-side projection, or
the remote materialized only a subset because the heap row vanished
under MVCC).

Implementation packet should pin one of:
- treat missing tuple_payload as `remote_heap_resolution_failed` (strict
  fails closed; degraded skips), reusing the existing fault category; or
- introduce a new fault category `remote_tuple_payload_missing` with
  strict/degraded behavior identical to heap-resolution failure.

Without this, the CustomScan path has a silent gap that the AM cursor
never had (the AM blocked on `requires_remote_row_materialization`).

### P2 — ADR-069: PREPARE TRANSACTION operational prerequisites

ADR-069 mandates two-phase commit on coordinator-routed INSERT and DELETE
but does not call out that PostgreSQL `PREPARE TRANSACTION` requires
`max_prepared_transactions > 0` on every remote (default is 0, which
errors out at PREPARE time). The ADR-068 operator setup sketch should
include this GUC on each remote, and the v1 write packet should add a
preflight that validates the remote has `max_prepared_transactions` set
high enough for expected coordinator concurrency. Otherwise first-INSERT
on a fresh remote fails with a confusing `max_prepared_transactions = 0`
error rather than a clear ADR-069-shaped diagnostic.

### P3 — ADR-068 setup sketch uses `real[]` instead of `ecvector`

The illustrative SQL on lines 187 and 210 of ADR-068 uses `embedding
real[]`, but this codebase's canonical row type is `ecvector(dim)`
(ADR-043). Cosmetic, but operators copy-pasting from the sketch will
hit a column-type mismatch with the coordinator's `ec_spire` index. Fix
the sketch to `embedding ecvector(N)` or add a note that the example is
schematic.

### P3 — ADR-069: placement directory growth and contention

The placement directory is a coordinator-local table with one row per
distributed row, hot-written on every INSERT/DELETE. ADR-069 doesn't
mention the long-tail consequences:
- table size scales linearly with total distributed dataset size, on the
  coordinator;
- under sustained insert concurrency, the placement insert and the 2PC
  commit are serialized through the coordinator's WAL.

Probably acceptable for v1 (coordinator routing is by definition the
bottleneck), but the ADR should note the expected ceiling and what
operators should monitor (placement-directory size, WAL pressure,
prepared-transaction backlog). A future ADR may want placement-directory
partitioning.

### P3 — ADR-067 path keys / cost are open questions

ADR-067 acknowledges path-key declaration and cost estimation as open
questions. Worth flagging that ordered output from a vector-distance
operator does not have a standard pathkey representation; the
implementation packet will need to either declare the distance ordering
as a pathkey on the vector column (so downstream `LIMIT k` consumes
it without re-sort) or accept an explicit Sort node above the
CustomScan. The latter is cheaper to implement but loses one of the
benefits of CustomScan over the AM cursor. Worth deciding before the
planner-path packet.

### P3 — Bulk COPY semantics not addressed

ADR-069 covers INSERT, UPDATE, DELETE, and bulk-load via the
classification helper, but `COPY documents FROM STDIN` against the
coordinator is the most common Postgres bulk-ingest pattern. ADR-069
should at minimum state whether COPY routes through the same
coordinator-routed INSERT path (slow but correct) or rejects with a
hint pointing at the bulk-load tooling. Operators will hit this on
day one of any non-trivial migration.

## Reviewer recommendation

Move ADR-067/068/069 from PROPOSED to ACCEPTED, with the P2 findings
folded into the corresponding implementation packets (tuple_payload
missing-row contract into the Stage B endpoint extension; 2PC
prerequisites into the v1 INSERT packet). The P3 items are docs-only
follow-ups.
