# DML Front-Door Integration Question

The CustomScan read path and PK SELECT path are now wired far enough that the
remaining ADR-069 gap is transparent non-embedding UPDATE and DELETE. Before
enabling any UPDATE/DELETE CustomPath, I need one architectural point pinned
down.

ADR-069 says transparent `UPDATE ... WHERE pk = ...` and
`DELETE ... WHERE pk = ...` need a "ModifyTable/view-hook integration" because
remote-owned rows are not present in the coordinator heap and row-level table
triggers cannot capture them.

The current CustomScan baserel path can build an executor-ready DML primitive
invocation, but placing that scan under PostgreSQL `ModifyTable` appears unsafe
as the final integration point:

- If the CustomScan performs the remote UPDATE/DELETE and returns no tuple,
  PostgreSQL likely reports `UPDATE 0` / `DELETE 0`.
- If it returns a virtual remote tuple, `ModifyTable` still expects row identity
  for the coordinator heap path (`ctid`/tableoid style semantics) and may try to
  update/delete a coordinator row that intentionally does not exist.
- Row-level triggers remain insufficient for remote-owned rows because the base
  heap scan finds no coordinator row to trigger on.

Which integration should be treated as the ADR-069 v1 target for transparent
UPDATE/DELETE?

1. A view front door with INSTEAD OF trigger/rule semantics for UPDATE/DELETE
   and a separate table underneath.
2. A planner/executor hook that replaces the whole DML statement with a
   provider-owned plan node and preserves command tags/rowcount.
3. A coordinator stub-row strategy so PostgreSQL `ModifyTable` has local row
   identity while the SPIRE trigger/CustomScan path forwards side effects.
4. Another PostgreSQL hook surface intended by "ModifyTable/view-hook" that I
   should implement instead.

Until this is clarified, I am keeping UPDATE/DELETE CustomScan path generation
disabled. The current code only prepares the shared CustomScan DML metadata and
primitive-invocation boundaries used by the already-live PK SELECT path.
