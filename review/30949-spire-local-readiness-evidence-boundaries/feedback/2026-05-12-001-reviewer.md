# Review: SPIRE Local Readiness Evidence Boundaries (30949)

## Summary

Phase 12.9 docs slice that publishes the three evidence
labels:

- **local functionality evidence** (correctness fixtures);
- **local production-readiness smoke evidence** (operator
  workflow + capacity smoke at local scale);
- **AWS/RDS product-scale evidence** (only valid claim
  surface for "ready for product workloads").

Pins Phase 13 entry boundary: AWS verification cannot
backfill missing Phase 12 hardening; accepted Phase 12
deferrals must repeat in AWS reports.

## Findings

### P2 — Strict label separation is the right primitive

The three-label structure prevents the most common project
failure mode in this lane: **a local fixture passing and
someone reading it as "ready for AWS."** By naming local
production-readiness smoke as a distinct middle tier, the doc
gives Phase 12.9 a clear "we did the operator workflow at
small scale" claim without leaking into product-scale claims.

The "may not support" lines are the load-bearing part. Ensure
each label has at least one explicit "may not claim" item
(e.g., local smoke MAY NOT claim concurrent multi-store
throughput parity; AWS MAY NOT skip a Phase 12 deferral
without re-listing it).

### P2 — Phase 13 entry boundary

"AWS/RDS verification should not start by implementing
missing Phase 12 hardening" is the right gate. This protects
against the failure mode where AWS tests find a gap and the
gap gets fixed in the AWS branch instead of being kicked
back to a Phase 12 packet with proper local coverage.

The "accepted Phase 12 deferrals must be repeated in AWS
reports" rule is what makes the deferred items (e.g.,
async/parallel store executor, prepared-xact recovery
helper, credential rotation) visible in AWS evidence rather
than silently dropped. Good.

### P3 — Local smoke checklist scoping

Worth confirming the local production-readiness smoke
checklist in the doc names exactly the still-open Phase 12.9
rows (libpq runbook ✓ from 30951, capacity targets ✓ from
30952, plus the still-open benchmark/contention/async-write
items) without claiming the open ones are done. From the
request the wording sounds correct — "matches the remaining
Phase 12.9 bundle and counter rows without pretending those
rows are already complete."

### P3 — Cross-link from packet templates

If the project has packet/request templates, a single line
"cite which evidence label this packet contributes to" would
make label drift catch itself at packet creation time.
Optional process improvement.

## Reviewer recommendation

Accept. The three-tier label separation is the right
primitive for keeping local evidence from accidentally
becoming product-scale claims, and the Phase 13 entry
boundary rule prevents Phase 12 gaps from hiding inside AWS
work.
