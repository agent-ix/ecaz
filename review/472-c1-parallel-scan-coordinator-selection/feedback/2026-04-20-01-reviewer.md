## Findings

### 1. Heap rebuild is not concurrent-safe

**Severity:** Medium (blocker for `amcanparallel = true`; no-op today)
**File:** `src/am/common/parallel.rs:797–814, 816–839`

`sift_up_heap_entry` uses `std::ptr::swap` on raw `*mut u32` entries
without atomics, and the rebuild loop clears the heap entries and
repopulates them in sequence. Two callers of
`refresh_coordinator_selection_snapshot` racing would corrupt the heap
arbitrarily. Callers today are inside `publish_...result_slot_runtime`
and `clear_parallel_scan_result_slot_with_attachment`, which are
themselves only called from a single worker per scan, so no race is
currently possible.

Two options for the `amcanparallel = true` slice:

- **Option A (simplest):** add a lightweight LWLock owned by the
  coordinator, taken around every refresh. Reads
  (`read_parallel_scan_selected_result_slot_snapshot`) can use the
  cached coordinator selection without taking the lock; the two-pass
  retry at `parallel.rs:1361–1387` already covers the stale-selection
  case.
- **Option B:** restructure the heap operations around CAS on each
  entry + a `heap_generation` that readers can use to detect an
  in-progress rebuild and retry. More code, avoids the lock hop.

Option A is probably the right default given the rest of this module
already assumes coordinator writes happen under some logical serializer.

### 2. Packet text understates the implementation

**Severity:** Informational
**File:** `review/472-c1-parallel-scan-coordinator-selection/request.md:25–33`

The request describes selection as "scan the staged coordinator result
slots / ignore invalid / choose lowest score / tie-break by slot
index". The code actually implements:

- A live-slot min-heap rebuild (`heap_entries` array backing).
- Dead-slot reaping (`reap_dead_parallel_scan_result_slots_...`).
- `heap_generation` and `published_result_slots` counter maintenance.
- A cached `selected_result_slot_index` plus two-pass retry on read.

This is strictly more than the request implies. Not a bug — the selection
behavior matches the spec — but if someone searches for "select
coordinator result" expecting a linear scan, they'll be confused. Worth
a line in Task 18's notes or an `///` doc comment on
`refresh_coordinator_selection_snapshot` summarising the heap-based
implementation.

### 3. `total_cmp` tie-break is deterministic — good

**Severity:** Informational (positive)

Using `f32::total_cmp` with slot-index tie-break means two workers
publishing the same score to different slots will always resolve
identically across runs. That matters for test reproducibility.

### 4. NaN handling

**Severity:** Informational
**File:** `src/am/common/parallel.rs:777–786`

`total_cmp` orders NaNs consistently but they will still rank as the
largest (negative-NaN) or nearly-largest (positive-NaN) score. If a
worker ever publishes a NaN score (unexpected but possible from a
buggy scorer), the slot won't be selected — which is the safe failure
mode. No action needed; documenting for future readers.
