## Answers to the packet's review questions

- **Right boundary for C5 while Stage D remote heap resolution
  is absent?** Yes. Exposing this as a `_summary(...)` SQL
  surface (rather than wiring directly into `amrescan`) lets
  reviewers and operators inspect the production scan path's
  state end to end without making it depend on Stage D rows.
  When Stage D lands, `amrescan`/`amgettuple` can call into the
  same executor state machine without needing this surface to
  change shape.
- **Routing-only leaf-count helper preserves coordinator/remote
  ownership boundary cleanly enough for deeper multi-layer
  SPIRE routing?** Yes. `count_scan_plan_routable_leaf_pids`
  walks the plan without reading leaf payload objects, which
  is the boundary packet 30736's
  `collect_scan_plan_selected_leaf_pids` established. Deeper
  routing slices (recursive + boundary replicas) will reuse
  the same boundary because the count helper depends on the
  same snapshot/object-store contract that already filters
  non-local placements.
- **Production executor handoff status model clear enough for
  the future move into `amrescan`/`amgettuple`?** Yes. See P3
  below for one ergonomic suggestion before Stage D lands.

## P3 — informational

1. **Internal state object for the AM scan path.** The current
   summary surface returns a wide row (effective_nprobe,
   selected_pid_count, local/remote split, dispatch/receive
   counts, merged candidate count, final status, blocker).
   When `amrescan`/`amgettuple` consume the same flow, they
   probably want a narrower internal struct rather than going
   back through SQL. Worth scoping a small `SpireScanHandoff`
   (or similar) Rust-side type that the SQL summary serializes
   *from* — same data, different consumer ergonomics.
   Optional, but the Stage D AM integration packet is the
   natural place to land it.

2. **Two design fixes ride along.** The
   `load_relation_epoch_manifests_for_coordinator_fanout(...)`
   refactor and the `count_scan_plan_routable_leaf_pids(...)`
   helper are both used at 5+ existing call sites, so this
   packet is doing more refactoring work than the title
   suggests. Worth confirming that the old call sites
   continued to behave correctly after the swap — the
   handoff fixture exercises one path, but the other four
   call sites in coordinator fanout could quietly regress.
   The pgrx-test sandbox issue from packet 30753 prevented a
   broad `cargo pgrx test` run; recommend the next review
   cycle includes a broader pgrx-test pass once the sandbox
   loader issue is resolved.
