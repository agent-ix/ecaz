# Review: Task 41 SPIRE pipeline relation guards (31165)

## Summary

Migrates `spire_remote_pipeline_step_rows` (the live-pipeline
helper) plus five SPIRE remote-search receive/merge/heap-resolution
diagnostics. The live-pipeline helper is the most interesting case
in the Task 41 chain so far: it previously held **one long guard**
across two unconditional AM calls, env-variable lookup, libpq
connection probing, and two conditional AM calls. The new code
shortens this into **up to three short, narrowly-scoped guards**.

## Findings

### 1. Multiple-open pattern: acceptable here, worth a strategy note

Opening the same index 1ÔÇô3 times per pipeline invocation is more
overhead than a single open. For a diagnostic helper this is fine
(AccessShareLock acquisition is cheap, this is not a hot path, and
the previous behavior had its own concrete safety issue ÔÇö holding
the lock across libpq). But the strategy matrix in 31161 does not
discuss when re-opening is preferable to keeping a long guard. A
one-line addition would help future migrations decide: "Prefer
re-opening short guards over a long guard when the long guard would
span env-var lookups, FFI calls into non-Postgres code, or large
control-flow regions."

### 2. Function-call repetition cost considered

Each `open_valid_ec_spire_index_guard` call re-validates the AM
type, builds an owned `String` on error, etc. For the live-pipeline
helper this happens up to three times per invocation. Could be
factored into one open + cached `relam` check, but doing so would
require either re-introducing the long guard or building a shared
`SpireValidatedIndexHandle` that re-opens without re-validating.
Not worth doing for diagnostic code ÔÇö flagging only so it's not a
surprise if this pattern propagates into a hot path later.

### 3. Failure ordering across guards

If the AM type changed between guard opens (impossible in practice
for a fixed `index_oid` inside one transaction with AccessShareLock
held repeatedly), the second and third guards would re-detect the
mismatch and error there rather than at function start. This is
harmless under the current invariants but worth noting if anyone
extends the helper to accept multiple index OIDs.
