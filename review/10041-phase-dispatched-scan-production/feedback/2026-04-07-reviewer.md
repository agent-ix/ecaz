# Review Feedback: Phase-Dispatched Scan Production

**Verdict: Approved**

## Dispatch structure

The `match opaque.execution_phase` in `produce_next_scan_heap_tid` is the right shape. Each phase
owns its full production path:
- `GraphTraversal` → `produce_next_graph_traversal_heap_tid` → `materialize_next_bootstrap_frontier_result`
- `LinearFallback` → `produce_next_linear_fallback_heap_tid` → `select_next_linear_scan_result`
- `Exhausted` → false

No shared selector, no late fallback decision. This is easier to reason about than the previous
shared "select then materialize" shell.

## Production boundary

`materialize_next_bootstrap_frontier_result` was previously `#[cfg(any(test, feature = "pg_test"))]`.
Promoting it to production code and having graph-phase production call it directly is the right move.
The `mark_scan_exhausted` call when the frontier has nothing left is the correct termination
semantics — graph traversal owns the scan to exhaustion, no fallback.

## Linear fallback path

The linear path still does `select_next_linear_scan_result` → `materialize_selected_scan_result` →
`emit_pending_scan_heap_tid`. Duplicate drain and current-result semantics are preserved because
this path didn't change structurally, it just got its own dedicated call site.

## No blocking issues
