# Review: Task 41 shared snapshot and slot guards (31195)

## Summary

Promotes `ActiveSnapshotGuard` and `TupleTableSlotGuard` into
`src/storage/snapshot_guard.rs` and `src/storage/slot_guard.rs`.
Migrates the SPIRE custom scan planner helper from its local
copies (introduced in 31175) to the shared types. Baseline
neutral (4,239 → 4,239) — pure structural consolidation, exactly
as expected.

This implements the consolidation I've been pushing for since
31175 / 31177 / 31182. Three resource families now have shared
primitives: relations (3 variants in `relation_guard.rs`),
snapshot (here), and tuple slot (here). The convention from my
31161 strategy feedback now applies to two more resource
families.

I'm satisfied with the structure. One follow-up note and one
small finding.

## Confirmations

- **`ActiveSnapshotGuard::latest` preserves register/push and
  pop/unregister ordering.** Constructor:
  `RegisterSnapshot(GetLatestSnapshot())` then `PushActiveSnapshot`.
  Drop: `PopActiveSnapshot()` then `UnregisterSnapshot(snapshot)`.
  Same as the prior 31175 inline version, plus the pgrx-unwind
  SAFETY note now lives at the shared site rather than module-local.
  ✔
- **`TupleTableSlotGuard` has both constructors.** Two constructor
  variants in one type:
  - `create(relation)` → `table_slot_create(relation, null)` —
    matches the planner.rs original.
  - `single_for_heap(relation)` →
    `MakeSingleTupleTableSlot((*relation).rd_att,
    table_slot_callbacks(relation))` — matches `HeapTupleSlot`
    (scan/relation.rs, 31177) and `DebugTupleSlotGuard`
    (scan_debug.rs, 31182).
  
  Both drop via the same `ExecDropSingleTupleTableSlot`, so one
  Drop impl covers both constructors. ✔
- **Planner migration is mechanical.** `custom_scan/planner.rs`
  drops 60+ lines of local guard impls; imports
  `crate::storage::{snapshot_guard::ActiveSnapshotGuard, slot_guard::TupleTableSlotGuard}`.
  Lifetime semantics preserved.
- **Module surface preserved.** `src/storage/mod.rs` re-exports
  the two new modules; the `relation_guard.rs` family + 2 new
  files form a coherent storage-resource namespace.
- **Baseline arithmetic.** Neutral, as expected for a structural
  consolidation that moves rather than retires unsafe sites.

## Findings

### 1. `IndexScanGuard` not yet promoted

The planner.rs file still has a module-local `IndexScanGuard`
(lines ~115-150 of the new file). This is the third resource
family from 31175 that hasn't been promoted yet. Same shape
applies: it's just `pg_sys::IndexScanDesc` with
`index_beginscan` / `index_endscan` paired ownership, no
caller-specific state.

Suggest a one-line follow-up packet
"Promote index scan guard to storage" that:

1. Creates `src/storage/scan_guard.rs` with `IndexScanGuard`
   (the constructor takes `&HeapRelationGuard`,
   `&IndexRelationGuard`, `&ActiveSnapshotGuard` — note the
   compiler-enforced lifetime borrows now possible thanks to
   the typed shared guards).
2. Migrates the planner.rs callsite.
3. Re-exports from `storage/mod.rs`.

That completes the snapshot/scan/slot family consolidation. The
scan-state bundle mentioned as deferred in 31194 then has all
three typed components to build on.

### 2. Pending: 4 module-local slot/snapshot variants to migrate

The new `TupleTableSlotGuard::single_for_heap` constructor
matches both `HeapTupleSlot` (scan/relation.rs, 31177) and
`DebugTupleSlotGuard` (scan_debug.rs, 31182). The
`DebugActiveSnapshotGuard` (scan_debug.rs, 31182) is structurally
identical to the new shared `ActiveSnapshotGuard::latest` modulo
the `CommandCounterIncrement` preamble — could be a second
constructor `latest_with_cci(label)` or callers can do the CCI
explicitly before calling `latest()`.

These are all addressable in the same "migrate remaining
module-local snapshot/slot guards" follow-up. Tracked.

### 3. Convention is now established for ~5 wrapper types

This is the third consolidated resource family
(`IndexRelationGuard`/`HeapRelationGuard`/`RelationGuard`,
`ActiveSnapshotGuard`, `TupleTableSlotGuard`). When the
buffer/page/WAL track lands its own `BufferPinGuard` and
`LwLockGuard`, the precedent is now solid: shared module in
`src/storage/`, single type per resource family with multiple
constructors where needed, pgrx-unwind SAFETY note at the
Drop. The 31161 strategy convention is no longer hypothetical —
it's the working pattern.
