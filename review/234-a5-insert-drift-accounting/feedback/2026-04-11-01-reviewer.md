# Feedback: A5 Insert Drift Accounting

**Reviewer:** Claude (Opus 4.6)
**Date:** 2026-04-11
**Commit reviewed:** `e651617`

## Scope match

Matches the packet. `MetadataPage` gains an `inserted_since_rebuild: u64`
field (`METADATA_BYTES` recomputed, encode/decode/roundtrip tests updated),
`BuildState::new` and `flush_build_state` reset it to `0` on every
`CREATE INDEX`, `tqhnsw_aminsert` advances it with `saturating_add(1)` on
every successful append, `IndexAdminSnapshot` exposes it alongside a derived
`insert_drift_fraction`, and `tqhnsw_index_admin_snapshot(regclass)` is
wired as a SQL admin surface. FR-016-AC-4 observability is now wired; the
concurrency half of FR-016-AC-3 is explicitly deferred to slice 235.

## Findings

### 1. Metadata format growth is clean and roundtrip-proof

`MetadataPage` adds one `u64` field, `METADATA_BYTES` recomputes to
`2 + 2 + ITEM_POINTER_BYTES + 2 + 1 + 1 + 8 + 8`, and the on-disk
encode/decode pair is updated at both call sites symmetrically. The
property roundtrip at `tests/proptest_page.rs` gains `inserted_since_rebuild:
seed` in its strategy and asserts it in `prop_assert_eq!`, so a future
change that forgets to encode/decode the new field will fail
deterministically on `cargo test -p tqvector proptest_page`.

The `src/am/scan.rs` test fixture update (`inserted_since_rebuild: 0`) is
the one other place `MetadataPage` is literal-constructed, and the compiler
catches any other forgotten site via non-exhaustive struct literal errors.
Good hygiene.

### 2. Non-empty insert now always takes the metadata EXCLUSIVE lock — call this out

This is the largest behavior change in the commit and it's buried in an
otherwise innocuous "add a counter" slice. Before 234:

```rust
// only reacquire metadata EXCL if we actually need to repair
if metadata_snapshot.entry_point == INVALID || insert_level > snapshot.max_level {
    with_locked_metadata_page(...)
}
```

After 234:

```rust
// unconditional metadata EXCL on every successful non-empty insert
with_locked_metadata_page(index_relation, |metadata| {
    metadata.inserted_since_rebuild = ... saturating_add(1);
    if metadata.entry_point == INVALID || insert_level > metadata.max_level {
        metadata.entry_point = element_tid;
        metadata.max_level = insert_level;
    }
});
```

Every successful non-empty insert now serializes on the metadata page for
a final write phase. On a workload with N concurrent insert sessions, the
metadata page becomes a contention bottleneck proportional to insert rate.
The *same* lock that was previously taken 1-in-many times (only on
promotion events) is now taken 1-in-1.

I don't think this is the wrong call — counting drift requires a durable
increment, and splitting the counter onto its own page would be a much
bigger slice. But:

a. **Concurrency slice 235 needs to own this as a named concern.** The
   retry-hardening packet should measure insert throughput under ≥4
   concurrent sessions and include a latency-lane before/after on the
   non-empty path. If metadata-page contention dominates, 235 may need to
   introduce `LWLockAcquire(MetadataLock, LW_SHARED)` + counter bump with
   an atomic, or move the counter off the metadata page entirely. Flag it
   explicitly in 235's prompt.

b. **The comment on the `with_locked_metadata_page` block should state
   the contention tradeoff**, not just "piggybacks on that same lock
   scope." A future reader looking at a contention flame graph should find
   the rationale here. Three-line comment: "every successful insert now
   takes this lock; prior versions only took it on promotion events; the
   cost is accepted in exchange for a durable drift counter; see FR-016-AC-4
   / slice 235 for the concurrency story."

### 3. Duplicate coalescing correctly bypasses the counter

Traced: duplicate path early-returns at `src/am/insert.rs:130` before the
`add_backlinks_to_forward_neighbors` + metadata-write block is reached.
The test `test_tqhnsw_index_admin_snapshot_tracks_insert_drift` asserts
explicitly that a duplicate-TID coalesce leaves `inserted_since_rebuild`
at `1` (not `2`) and `total_live_nodes` at `4` (not `5`). That is the
right semantic choice — the counter is "net new element tuples since
rebuild," not "aminsert calls since rebuild" — and the test makes the
semantic machine-checkable. Strong.

### 4. `insert_drift_fraction` can exceed 1.0 under delete+vacuum — document the
   invariant, not the fix

The fraction is `inserted_since_rebuild / total_live_nodes` where
`total_live_nodes` is computed by `count_element_tuples(index_relation)`,
which walks the heap and counts live tuples. The counter is monotonic and
only reset by a full `CREATE INDEX`; the denominator can *shrink* after
vacuum repair (A6) removes dead element tuples.

Concrete scenario once A6 lands: build 100 rows, live-insert 50 more,
delete 80 rows, vacuum. State: `inserted_since_rebuild = 50`,
`total_live_nodes = 70`. Reported drift fraction = `0.714`, which makes
sense as "50 of your 70 live nodes arrived after build." Now delete
another 60, vacuum. State: `inserted_since_rebuild = 50`,
`total_live_nodes = 10`. Reported drift fraction = `5.0`.

That's not *wrong* — it accurately reports "5× more live inserts than
the post-vacuum node set" — but it *is* surprising for a field named
`drift_fraction`, and downstream consumers (alerting, planner cost) that
treat it as a [0, 1] probability will break.

**Required before A6 lands:** either document explicitly in the
`insert_drift_fraction` docstring / admin snapshot column comment that
the fraction is unbounded and semantically "fresh inserts per live node,"
or clamp it at `1.0` in the helper. I'd pick document-and-don't-clamp
because the unclamped value is actually more useful for "do I need to
rebuild" analysis.

Today (before A6, no vacuum repair), this is latent — the test
`test_tqhnsw_index_admin_snapshot_tracks_insert_drift` passes because
the denominator only grows. File the doc note on the 235 slice or the A6
kickoff, not here.

### 5. `IndexAdminSnapshot` lost its `Eq` derive — intentional and acceptable

Adding `insert_drift_fraction: f64` forced a derive change from
`PartialEq, Eq` to `PartialEq` only. No current caller compares
`IndexAdminSnapshot` values by `Eq`, so nothing broke. But if someone in
the future tries to put `IndexAdminSnapshot` in a `HashSet` or uses it as
a `HashMap` key, the derive gap will surface as a compile error rather
than a silent semantic gotcha. Acceptable.

Small improvement worth a follow-up: the f64 field doesn't need to be a
field at all since it's *derived* from the other two. A method
(`snapshot.insert_drift_fraction() -> f64`) would remove the Eq ambiguity
entirely and force callers to compute on demand. Low priority.

### 6. `usize::try_from` on the counter is defensive, not lossy

`usize::try_from(metadata.inserted_since_rebuild)` converts `u64 → usize`,
which is identity on 64-bit targets and potentially lossy on 32-bit.
The `unwrap_or_else` calls `pgrx::error!` on truncation, so a 32-bit
build that manages to insert >4B rows would loudly fail rather than
silently truncate. Unreachable in practice, but the error path is
correctly hooked rather than `unwrap()`-ing. Good defensiveness.

The mirror direction (usize → i64 in `lib.rs`) uses `i64::try_from` with
`.expect("inserted-since-rebuild should fit in i64")`. Also unreachable,
also correctly wired.

### 7. `saturating_add(1)` is the right overflow discipline

`u64::MAX` inserts to a single index is unreachable (at 1M inserts/sec it
takes ~580,000 years), so `saturating_add` is effectively equivalent to
`+1`. The saturating form has the property that if some pathological
future state *does* hit it, the counter sticks at `u64::MAX` rather than
wrapping to zero and pretending the index just got rebuilt. Correct
choice for a drift counter specifically — the failure mode of "counter
stopped moving" is much louder on monitoring dashboards than "counter
wrapped to zero." Keep.

### 8. Empty-index path also increments, and the test covers it

`test_tqhnsw_index_admin_snapshot_counts_empty_first_insert` is the test
that specifically exercises the shape-init code path (first insert into
an empty index where `metadata.dimensions == 0 && metadata.bits == 0`).
The shape-init path is a distinct branch from the non-empty path and has
its *own* `saturating_add(1)` call at `src/am/insert.rs:89-90`. The test
asserts that after the first insert, `inserted_since_rebuild = 1` and
`total_live_nodes = 1`, so `drift_fraction = 1.0`.

Notably, this test exercises the case "empty CREATE INDEX, then live
insert the first row" rather than "CREATE INDEX on populated table." The
distinction matters because the populated-CREATE-INDEX path goes through
`flush_build_state` which initializes the counter to 0; the empty
`CREATE INDEX` path initializes no metadata at all and instead sets it
up lazily on first insert. Both paths must produce `inserted_since_rebuild = 1`
after one live insert, and both tests together cover that.

### 9. Rejection test is shallow but appropriate

`test_tqhnsw_index_admin_snapshot_rejects_non_tqhnsw_index` uses
`#[should_panic(expected = "tqhnsw_index_admin_snapshot requires a tqhnsw index")]`.
That's a substring match on the error message — if the error text ever
changes, the test fails silently (well, loudly, but not for the reason
the test was written). The standard workaround is a more specific match;
but the error message is owned by `open_valid_tqhnsw_index` and changes
there would be caught by existing tests elsewhere. Acceptable as-is.

- **FR-016 spec:** promoted from "may expose … with explicit NULL" to

  with concurrency hardening for FR-016-AC-3 as staged follow-on." Crisp.

- **`review/README.md`:** `aminsert` description now lists graph-aware

No spec drift, no status inflation.

## Verdict

**Approve.** The drift counter lands in a place that's forward-compatible
with both concurrency hardening (235) and vacuum repair (A6), the
duplicate-coalesce semantic is explicit in the test, and the metadata
format migration is roundtrip-proof. The one thing this slice *does*
that wasn't highlighted in the packet is that it turns the metadata
exclusive lock into an always-path on non-empty insert — that needs to
be acknowledged in 235's prompt as a named concern.

Recommended follow-ups (none blocking):

1. **Item 2 (concurrency contention):** 235's prompt should name the
   metadata-lock-always-taken change as a contention risk and include a
   concurrent-insert latency lane; either keep and measure, or move the
   counter off the metadata page.
2. **Item 2 (comment):** add a 2-3 line comment on the `with_locked_metadata_page`
   block in `src/am/insert.rs` documenting the contention tradeoff.
3. **Item 4 (drift fraction semantics):** document in the admin snapshot
   column that `insert_drift_fraction` is unbounded post-vacuum, before
   A6 lands. One paragraph on the `insert_drift_fraction` helper or the
   SQL function docstring.
4. **Item 5 (derive cleanup):** consider making `insert_drift_fraction`
   a method rather than a field so `IndexAdminSnapshot` can re-derive
   `Eq`. Low priority.

The load-bearing observability win is that FR-016-AC-4 is now
machine-checkable from SQL: a monitoring job can poll
`tqhnsw_index_admin_snapshot(idx)` and drive rebuild scheduling off
`inserted_since_rebuild`. That closes the observability half of
FR-016 even while the concurrency half is still in flight.
