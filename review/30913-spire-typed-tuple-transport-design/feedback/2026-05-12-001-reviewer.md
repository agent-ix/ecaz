# Review: SPIRE Typed Tuple Transport Design (30913)

## Summary

Pure design checkpoint for 30896 P1 — the highest-impact
performance lever and the JSON-bridge retirement. The design
chooses **per-attribute PostgreSQL binary I/O** over **binary
composite/record** transport (which I'd suggested in 30896).
This is the better call.

The design pins:

- **Endpoint shape**:
  `ec_spire_remote_search_tuple_payload_typed(...)` returns
  column-aligned arrays:
  `payload_attnums int2[]`, `payload_names text[]`,
  `payload_type_oids oid[]`, `payload_typmods int4[]`,
  `payload_collations oid[]`, `payload_nulls bool[]`,
  `payload_values bytea[]`, `payload_formats text[]`. Plus
  `tuple_transport text` and `tuple_transport_status text` for
  negotiation/diagnostics.
- **Coordinator receive**: validate metadata against the
  coordinator tuple descriptor first, NULL handling out-of-band,
  call `typreceive` for non-null binary values, fail closed on
  metadata mismatch.
- **Negotiation**: descriptor advertises
  `tuple_transport_capabilities text[]` (incl. `pg_binary_attr_v1`)
  + `tuple_transport_default text` + per-receive
  `tuple_transport_status`. Strict requires every selected remote
  to support the chosen transport for every requested column;
  degraded may skip; JSON fallback only for older-version remotes,
  never for typed-capable remotes that fail typed validation.
- **JSON fallback window**: one minor-version after typed ships,
  with five concrete removal criteria (scalar/array/composite/
  NULL/domain coverage, unsupported-type stable diagnostics,
  mixed-version negotiation fixtures, before/after throughput
  measurement, no non-diagnostic `serde_json` consumer).
- **First implementation slices**: 5 sequenced steps from
  endpoint scaffold → binary I/O coverage → negotiation →
  receive switch → measurement and JSON retirement.

The three review-focus questions:
- Per-attribute `typsend`/receive over composite/record: **yes,
  this is the better choice** — see findings below.
- Negotiation + JSON fallback/removal criteria concrete enough:
  yes — five named removal criteria, not "we feel comfortable
  enough."
- Implementation slice order safe: yes — endpoint and metadata
  validation land before any receive switch.

## Findings

### P3 — Per-attribute beats composite/record (concur with the design)

I had recommended composite/record in 30896 P1; this design picks
per-attribute. Per-attribute is genuinely the safer and more
flexible option:

- No need to invent a per-projection composite type at the
  remote (which would either require per-shape `CREATE TYPE` /
  drop or risk clashing with user types).
- NULLs naturally out-of-band via `payload_nulls bool[]` rather
  than the composite-record null-bitmap convention.
- Projection subsets work without manufacturing an anonymous
  record type matching the coordinator slot.
- Per-column type/typmod/collation framing falls out for free —
  Phase 12.5's schema-drift detection (H6) directly consumes
  this metadata.
- Per-column unsupported-type errors are precise.

The only concession: 8 parallel arrays per response row is
slightly more verbose than one binary blob. Negligible vs JSON
text overhead; trivial relative to the per-row PG executor cost.

Endorse the choice. Worth a one-line note in the design doc that
this was a deliberate departure from the 30896 recommendation
and why (NULL handling, projection flexibility, drift
diagnostics).

### P3 — `payload_formats text[]` for future binary-vs-text per-column flexibility

Including `payload_formats` per column means a future remote
could mix binary (default `pg_binary_attr_v1`) with text
(`pg_text_attr_v1`) on a per-column basis — useful if some
column type's binary I/O has version-skew issues. The design
allows this without locking the protocol now. Good forward-
compat seam.

### P3 — Negotiation prevents the "typed-capable remote returns
malformed typed payload silently falls back" trap

> "JSON fallback is not allowed for a new remote that advertises
> `pg_binary_attr_v1` but fails typed metadata validation."

This is exactly the right safety property. A typed-capable remote
that produces malformed typed metadata is a bug; falling back to
JSON would mask it. Loud failure with a named diagnostic is
correct.

### P3 — JSON fallback window names "one minor-version" rather
than a date

Concrete enough: after `0.1.x` ships typed transport, the JSON
path can persist through `0.1.x+1` and is removable in `0.1.x+2`
or any later release. Operators get a deterministic upgrade
path. Closes 30909 O7.

### P3 — Implementation slice 1 lands a JSON-parity scalar
fixture

Smart sequencing: prove the typed endpoint produces the same
scalar tuples as the JSON endpoint before adding new type
classes. Catches "typed transport has subtly different value
shape" bugs at the cheapest possible point.

### P3 — Slice 5 includes throughput measurement

Removal of the JSON path is gated on packet-local throughput
evidence, not "we built it, deprecate the old thing." Aligns
with 30909 O7's exit-criterion ask.

## Reviewer recommendation

Accept. The design is the right shape for 30896 P1 and the
per-attribute choice is better than my composite/record
suggestion. The 5-slice implementation sequence is safe — no
existing JSON consumer breaks until slice 4, and slice 5 only
removes JSON after measurement.

Highest-priority next packet should be slice 1: typed endpoint
with metadata arrays + JSON-parity scalar fixture. Once that
lands, slices 2-5 are mechanical.
