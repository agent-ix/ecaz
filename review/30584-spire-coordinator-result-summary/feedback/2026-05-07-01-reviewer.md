# Coordinator result summary review (closeout for 30569–30584)

Final composing summary for the Phase 7 coordinator stack. This is
the most-recent packet in the stream, so the cross-cutting summary
lands here. Localized findings are in 30569–30583.

## 30584 — local findings

`remote_search_coordinator_result_summary_row`
(`hierarchy_snapshots.rs:525-582`) composes the gate summary with
the local heap candidate summary into a single row that an
operator can use as the integration check before dispatch.

- **Pipeline redundancy is now structural.**
  `coordinator_result_summary_row` calls both
  `remote_search_coordinator_gate_summary_row` and
  `remote_search_local_heap_candidate_summary_row`, with full
  `query.clone()` and `selected_pids.clone()` for each. Each of
  those summaries internally composes execution + merge +
  finalization summaries (gate side), or heap resolution +
  candidate rows (local heap side). Each of those, in turn, calls
  `remote_search_coordinator_local_candidates_result` either
  directly or through a sub-summary.

  Conservatively, a single `coordinator_result_summary` SQL call
  re-runs the fanout planner ≥4 times, opens a new
  `AccessShareLock` object store ≥2 times, and re-executes leaf
  scoring + merge ≥2 times. For diagnostic queries this is
  measurable but not a correctness bug — every redundant call
  validates `requested_epoch == active_epoch` before doing real
  work, so an epoch advance between sub-calls produces an error
  rather than inconsistent rows.

  The right fix is structural: factor out a single
  `SpireCoordinatorPipeline::execute_once(...)` that returns a
  bundle (plan, gate state, merge state, heap-resolution rows)
  and have every summary surface project from that bundle. That
  is a larger refactor than this packet; the recommendation is
  to land it before adding any further composing summaries.

- **`result_source` taxonomy.** The literal strings
  `"local_heap_candidates"`, `"blocked"`, and the central
  `SPIRE_REMOTE_NONE` are not yet in the central registry from
  30570. Move them in for consistency.

## Cross-cutting summary — 30569–30584

### What's solid

- **Every cross-cutting finding from the 30556–30568 closeout was
  addressed in 30569–30573.** Empty-input snapshot validation,
  hardcoded `"available"` rendering, derived result-column count,
  tie-breaker label, rustdoc relocation, stale-placement
  coverage, and combined-fixture readiness test all landed
  cleanly. 30570 centralizes ~24 status/transport/format/policy
  strings into a constants block. 30572 introduces
  `SpireRemoteCountRollup` and routes four summaries through it.
  30573 adds the tie-breaker invariant test.
- **The descriptor-contract scaffolding (30574/30575)** lays out
  a 12-field schema with explicit required/optional flags and
  named validators, plus a 9-step registration contract — both
  enforce the design's "no raw conninfo storage" invariant via
  the `conninfo_secret_name` indirection.
- **Executable local-finalization path lands (30579/30582).**
  Decoded heap block/offset work items now flow alongside ranked
  candidates, with the remote-target gate kept fail-closed.
- **Real bug fix (30583).** Degraded-skipped placement plus
  local executable work was previously collapsed to `ready`,
  losing the operator-facing degradation signal. Now preserved
  through every summary layer.

### Cross-cutting findings (forward-looking, non-blocking)

1. **Pipeline redundancy is now the dominant scaling concern.**
   See 30584 local finding. Every summary that composes prior
   summaries re-runs the coordinator-local pipeline. With
   16+ summary surfaces, calling the topmost gate
   (`coordinator_result_summary`) likely re-executes the leaf
   scoring 4–6 times. Recommend a one-shot
   `SpireCoordinatorPipeline::execute_once(...)` bundle before
   the next composing summary lands. Also see 30577, 30579,
   30580, 30582 feedback.

2. **Status precedence logic is open-coded across summaries.**
   30583 patched the missing `degraded_ready` arm in two
   summaries. Other summaries derive their status with similar
   if/else chains. The 30572 rollup helper handles counts;
   recommend extending it (or adding a sibling
   `derive_summary_status(rollup, top_k_zero, blocked_first)`)
   so future precedence changes touch one place. See 30583
   feedback.

3. **Descriptor-state literals (`"missing"`, `"active"`) are
   not yet in the central registry.** 30570/30571 covered the
   search-readiness taxonomy; descriptor lifecycle strings
   remain literal. Track for the next round when libpq
   registration starts producing additional states. See 30571,
   30575 feedback.

4. **Recommendation strings are duplicated across packets.**
   "register remote node descriptor before libpq fanout
   execution" appears in multiple files. Consider a
   `recommendation_for(blocker_status)` helper. See 30575
   feedback.

5. **Degradation policy contract (30576) is a static list with
   no invariant test.** The actual decision logic lives in
   `fanout_should_skip_placement` and peers. Without an
   invariant assertion (similar to the 30573 tie-breaker test),
   the contract claim could drift silently from the
   implementation. See 30576 feedback.

6. **Locator decode error context.** `ItemPointer::decode` errors
   in `remote_search_local_heap_resolution_plan_rows` /
   `_candidate_rows` bubble out as raw decode strings without
   pid/vec_id context. Forward-looking; only matters if a
   corrupt locator ever reaches this path. See 30579 feedback.

### Test coverage

55 PG18 tests passing under `cargo test --lib remote --features
pg18`. Coverage matrix is broad: ready, degraded skipped,
descriptor missing, libpq blocked, mode mismatch, strict
unavailable, stale, mixed local/remote precedence, mixed
local/degraded-skipped status preservation. Solid.

### Verdict

No blockers for the 30569–30584 stream to remain landed. Findings
are forward-looking. Two should be addressed before adding more
composing summaries: pipeline redundancy (1) and status-precedence
centralization (2). The other findings can ride along with the
next descriptor or libpq slice.

The Phase 7 SQL diagnostic surface is now extensive (≥30
SQL-visible diagnostic functions across `remote_candidates.rs` and
`snapshots.rs`). The contract clarity is high, but the layering of
summaries-of-summaries-of-summaries has reached the point where a
small refactor pays off both for performance and for future
contract maintenance.
