## Answers to the packet's review questions

- **Boundary between diagnostic libpq SQL surfaces and the
  production AM executor?** Right shape. The diagnostic surfaces
  remain operator-facing read paths that compose existing dispatch
  planning, identity probing, and budget gating into rows; the
  production executor owns its own state machine and the AM scan
  path. They share the receive-attempt vocabulary but not the
  control flow.
- **C0–C6 granular enough for review while broad enough to avoid
  premature implementation detail?** Yes. C0 is a state contract
  (no I/O), C1 introduces the transport adapter as a single seam,
  C2/C3/C4 are independent capabilities, C5 is the scan-path cut,
  C6 is harness/runbook. Each can be reviewed as one packet.
- **Cancellation, timeout, strict/degraded, counter requirements
  strong enough before async/pipeline starts?** Yes for C0/C1 and
  mostly for C2/C4. See P2 below for one tightening item before
  async lands.

## P2 — close before C1 async/pipeline implementation lands

1. **Connection-lifetime contract.** The plan acknowledges
   per-query connect/close as acceptable for C1 (line 225: "per-query
   connect/close is acceptable for..."). For C1 to ship safely,
   the design doc should pin one of these explicitly:
   - per-query connect / per-dispatch close (simplest; matches
     current diagnostic shape);
   - executor-state-scoped pool with connect/disconnect at executor
     start/end (better tail latency, more state);
   - longer-lived pool with explicit invalidation triggers (highest
     reuse, hardest invalidation).
   Without this pinned, C1 could ship a connection model that C2's
   cancellation contract has to retrofit. Recommend stating "C1
   uses per-query connect/close; pool design is C1.5 or later"
   so cancellation has a known target shape.

## P3 — informational

2. **C5 AM scan integration ordering.** C5 is currently last after
   C4 strict/degraded; consider whether a smaller "C5a: AM scan
   reads dry production-state summary" slice should land before
   C5 full integration, so the AM path can observe production
   state for diagnostic purposes before it depends on production
   transport. Optional; lands naturally if the dry-state work in
   30723 grows that direction.

3. **C6 runbook content cross-reference.** The C6 runbook will need
   to consume the budget summary, identity-cache summary,
   production-state summary, and the receive-attempt diagnostics.
   Worth adding a "Surfaces this runbook reads" subsection now so
   any new diagnostic added between now and C6 lands with a
   matching runbook entry.
