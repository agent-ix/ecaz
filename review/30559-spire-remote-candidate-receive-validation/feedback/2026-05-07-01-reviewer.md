# Remote candidate receive validation review

`validate_remote_search_candidate_batch` and
`merge_validated_remote_search_candidate_batches`
(`root/remote_candidates.rs:1328-1476`) bracket the libpq receive
boundary correctly. One real finding below.

## Findings

- **Misplaced rustdoc — global vec_id warning is now attached to the
  validator, not the merge helper.** The doc-comment at
  `root/remote_candidates.rs:1322-1327`
  (`/// Merges candidates that share one coordinator-scoped vec_id
   namespace ...`) was originally added on `merge_remote_search_candidates`
  in commit `510749b6`. When `validate_remote_search_candidate_batch`
  was inserted between the doc and the merge function, the doc became
  rustdoc on the validator instead. The validator does not "merge
  candidates," so the doc text now misleads any reader of
  `cargo doc`.

  Fix: move the existing doc-comment back onto
  `merge_remote_search_candidates` (line 1403) and add a separate
  appropriate doc-comment to `validate_remote_search_candidate_batch`
  describing the validator's contract (per-batch
  epoch/node/PID/visibility/vec_id/row_locator/score checks). The
  existing doc-comment on
  `merge_validated_remote_search_candidate_batches` (line 1454-1457)
  already cites the same global-vec_id precondition, so the
  cross-batch case stays documented.

## Confirmations

1. The validator covers the receive-boundary surface:
   - `requested_epoch != 0` and matches every candidate's `served_epoch`
   - `expected_node_id` matches every candidate's `node_id`
   - selected PIDs nonzero and unique
   - candidate `pid` belongs to the selected set
   - `object_version != 0`
   - `is_visible_scored_assignment_flags(assignment_flags)` — exactly
     one of `PRIMARY` or `BOUNDARY_REPLICA` set, no delete flags
   - `vec_id` and `row_locator` nonempty
   - `score.is_finite()`

   That is the right minimum for a receive boundary against an
   unauthenticated remote producer.
2. `merge_validated_remote_search_candidate_batches` validates every
   batch before any flattening. A failed batch does not produce
   partial merge state.
3. The multi-node global vec_id precondition is explicitly preserved
   in the new `merge_validated_remote_search_candidate_batches`
   doc-comment.
4. Validator is target-scoped: each batch is checked against its
   declared `node_id` and the fanout-selected PID set for that node,
   so cross-target leakage (a node returning candidates for another
   node's PIDs) is rejected.
