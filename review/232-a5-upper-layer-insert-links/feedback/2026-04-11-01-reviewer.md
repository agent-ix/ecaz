# Feedback: A5 Upper-Layer Insert Links

**Reviewer:** Claude (Opus 4.6)
**Date:** 2026-04-11
**Commit reviewed:** `34ab846`

## Scope match

Matches the packet. `discover_insert_forward_neighbor_slots` now returns
`(slots, selections)`; upper-layer search runs top-down from
`metadata.max_level` via `graph::search_layer_result_candidates`, and the
backlink write path carries `(neighbor_tid, layer)` through grouping, dedup,
and the on-page rewrite. `insert_backlink_for_layer` replaces
`insert_layer0_backlink` with a layer-indexed variant that reuses the same
free-slot semantics.

The slice still does **not** prune — full upper-layer slices silently drop the
backlink, matching the 231 contract.

## Findings

### 1. Slot layout bookkeeping is consistent across write/read/test

Three helpers all compute the same layer-to-slot mapping:

- `backlink_slot_bounds(m, total, layer)` — used by `insert_backlink_for_layer`
  (write side)
- `selected_forward_slot_bounds(m, total, layer)` — used by
  `write_layer_forward_candidates` (the layer-0 case narrows to `[0, m)` to
  preserve the 230 contract of "only first M layer-0 slots populated on a new
  node"; upper-layer cases use the full `[start, start+m)` range)
- `layer_neighbor_slice(neighbor_tids, m, layer)` — test-side accessor, not
  production code, but implements the exact same math

The three definitions agreeing is load-bearing. **Required follow-up (not
blocker):** collapse to one source. The cleanest form is to move
`backlink_slot_bounds` into `src/am/page.rs` alongside `neighbor_slots`, next
to the layout it encodes, and have all three callers use it. That also makes
the test helper stop silently re-implementing a piece of the on-disk format.
File it as a 10-minute cleanup on whichever slice next touches slot layout.

### 2. Upper-layer descent duplicates work with `greedy_descend_from_entry`

`discover_insert_forward_neighbor_slots` runs in this order:

1. `populate_upper_layer_forward_slots` — walks layers `[max_level..=1]` via
   `search_layer_result_candidates` with `ef_construction` width, keeping
   `seeds` updated between layers
2. `greedy_descend_from_entry` — walks *the same* entry candidate down all
   layers via internal greedy descent (loads entry element again), producing
   a single `descended_seed`
3. `search_layer0_result_candidates` — layer-0 beam search seeded from
   `descended_seed`

Step 2 re-traverses the upper layers that Step 1 just covered, and discards
Step 1's `seeds` when Step 3 runs (Step 3 uses `[descended_seed]`, not
`seeds`). This is the classic HNSW insert flow *minus* the optimization where
the upper-layer search's final result feeds the next layer's seeds.

**Correctness impact:** Step 2's greedy descent produces one candidate that
may be worse than Step 1's last `seeds` list. The layer-0 search then starts
from that one (possibly worse) seed. Standard HNSW would feed the last
upper-layer beam state into layer 0.

**Performance impact:** upper layers are walked twice — once beam-width, once
greedy-width. For small M and typical trees `~3–4` levels deep, it's maybe a
dozen extra page reads per insert. Not catastrophic, but not free.

**Recommended fix (next slice, not this one):** after
`populate_upper_layer_forward_slots`, if `seeds.is_empty()` is false, hand
`seeds` directly to `search_layer0_result_candidates` and skip
`greedy_descend_from_entry` entirely. If `seeds` is empty (e.g.,
`insert_level == 0` or `max_level == 0`), fall through to the current greedy
path. One branch, deletes duplicate work, gives layer 0 better seeds.

I would put this on the overflow pruning slice (233) since that's the next
touch on the insert search path and the search's "quality of candidates"
story starts mattering once pruning exists.

### 3. Upper-layer search loop descends correctly but throws away intermediate quality

The loop in `populate_upper_layer_forward_slots`:

```rust
for current_layer in (1..=metadata.max_level).rev() {
    seeds = search_layer_result_candidates(..., current_layer, ..., seeds, ...);
    if current_layer <= insert_level {
        write_layer_forward_candidates(slots, selections, current_layer, m, seeds.clone());
    }
    if seeds.is_empty() { break; }
}
```

The `if current_layer <= insert_level` guard is correct: we walk the full
upper-layer tree regardless of `insert_level` (to give the layer-0 starting
point the best candidates), but we only *persist* forward slots for layers
the new node actually occupies. This is exactly the HNSW insert contract.

Two small notes:

- **`seeds.clone()` before `write_layer_forward_candidates`** is only needed
  because `seeds` is still being reassigned the next loop iteration, and the
  write function takes an owned `IntoIterator`. The clone cost is `O(m)` and
  happens ≤ `max_level` times per insert — tolerable, but a small
  optimization would be to structure the write function to take `&[...]` and
  clone inside the helper. Not worth churn in this slice.
- **`seeds.is_empty()` break** is the right early-exit. If a layer produces
  no scored candidates, lower layers can't improve from nothing. Good.

### 4. `BacklinkMutation` grouping and dedup are correct

`add_backlinks_to_forward_neighbors` builds a flat `Vec<BacklinkMutation>`
across *all* layers, sorts by
`(neighbor_tid.block, neighbor_tid.offset, layer)`, dedups, groups by block
for the on-page rewrite, and within a page groups by `neighbor_tid` so one
tuple is decoded/re-encoded only once even when multiple layer updates target
it. This is the right shape for the workload — if the new node sits on layer
2 and selects an existing element that also participates in layer 2, that
element's neighbor tuple gets one decode, multiple `insert_backlink_for_layer`
calls against different sub-ranges of `neighbor.tids`, and one re-encode.

The inner loop correctness matters because `tuple_changed` is OR-ed across
the per-layer mutations, and the tuple is only rewritten if *any* layer took
effect. Crucially, if layer 2 had free capacity and layer 1 was full, the
layer 2 update still lands. Traced the logic — correct.

### 5. Selection-vs-slot coupling needs a rename or a note

`LayerForwardSelection { layer, element_tid }` is emitted both for layer-0
slots (`[0..m)` only — the 230 narrow contract) and upper-layer slots (the
full `[start, start+m)` range). Two distinct "these are what got written"
concepts share one struct, and the only way to tell them apart is
`selection.layer == 0` vs `> 0`.

This is fine today because every consumer (just
`add_backlinks_to_forward_neighbors`) treats all selections uniformly: each
selection → one backlink attempt on the corresponding layer. But the
asymmetry is a trap for the next author who adds a new consumer and assumes
"selections represents what exists in slots" — for layer 0, slots `[m..2m)`
can be empty even when the layer-0 selection list has `m` entries, because
the selection covers only the narrow range.

**Recommended follow-up:** rename `LayerForwardSelection` to
`LayerForwardBacklink` (or similar) so it's explicit that the struct names
the backlink mutation side, not the slot-population side. Alternatively, add
a `// NOTE: selections represent the backlink targets, not the slot set.` on
the struct. Either is enough to stop the next slice from conflating them.

### 6. Test `test_tqhnsw_insert_populates_upper_layer_links_when_available`
   is the right shape

Walks live inserts 1..=192 with a deterministic level formula until it finds
a state where:

1. the index *already* has `max_level > 0` (graph has an upper layer to
   search against)
2. the *new* insert also draws `level > 0` (so upper-layer forward links are
   actually populated)

Then for the inserted element:

- asserts the level matches the pre-computed expected level (guards against
  insert-level / storage-level drift)
- walks `layer_neighbor_slice(..., layer=1)` and finds the populated tids
- for each layer-1 forward target, loads its element, confirms
  `element.level >= 1` (guards against "we linked to a level-0 node in a
  layer-1 slot"), then walks *that* element's layer-1 slice and counts
  matching backlinks
- asserts at least one reciprocal backlink landed
- asserts the `[m..2m)` half of the new element's layer-0 slice is still
  `INVALID` (preserves the 230 narrow-forward contract)

The reciprocity assertion ("at least one forward target receives the
matching backlink") is the specific thing that could go wrong if the write
path misplaced layer identity. It catches exactly the class of bug the new
`(neighbor_tid, layer)` grouping is supposed to prevent. Strong.

**Gap worth closing in 233:** the assertion is `layer1_backlink_targets > 0`,
not `== forward_tids.len()`. A target that was full (no INVALID slot in its
layer-1 sub-range) would count as "silently dropped," which matches this
slice's contract but is not asserted directly. Once 233 lands, convert the
test to assert either "every forward target received a backlink" or "every
forward target that wasn't already full received a backlink", whichever
matches the final contract.

### 7. Regression of existing layer-0 tests — tightened, not weakened

Both prior tests (`test_tqhnsw_insert_populates_forward_links_from_live_entry_seed`
and `test_tqhnsw_insert_populates_forward_links_against_built_graph`) now
use `layer_neighbor_slice(..., 0)` instead of `row.tids.iter().take(...)`,
and the assertion "remaining slots invalid" now narrows specifically to
"second half of layer-0 slice invalid" rather than "everything after the
first M invalid." That's correct — because upper-layer slots *may* now be
populated on nodes with `level > 0`, the old assertion "everything after M
invalid" would have been wrong for any node that happened to draw a higher
level. The rewritten test is more specific and still fails under the bugs it
was meant to catch.

## Verdict

**Approve.** The slice is internally consistent: upper-layer search, selection
carrying `layer`, and write path all agree on the slot layout, and the new
test directly exercises the reciprocal backlink contract. Item 2 (duplicate
descent) and Item 1 (three definitions of slot bounds) are the two things
worth cleaning up in the next A5 touch but neither is a blocker on this
slice.

Recommended follow-ups, all on 233 or later:

1. **Item 1 (slot bounds dedupe):** move `backlink_slot_bounds` into
   `page.rs` as the one canonical helper; update both call sites and the test
   helper to use it.
2. **Item 2 (duplicate descent):** use the upper-layer search's final `seeds`
   as the layer-0 search seed instead of re-running
   `greedy_descend_from_entry`. Delete the entry-element second load in the
   process.
3. **Item 5 (selection rename):** rename `LayerForwardSelection` to
   `LayerForwardBacklink` or add a comment pinning the semantics.
4. **Item 6 (test sharpening):** tighten the upper-layer test's reciprocal
   assertion from `> 0` to "all non-full targets received the backlink" once
   233 lands.

The load-bearing correctness win here is that the write path now carries
layer identity end-to-end, and one neighbor tuple can be updated in multiple
logical layers under a single buffer lock + WAL txn. That's the
infrastructure 233 needs before it can start reasoning about "this target is
full, prune it."
