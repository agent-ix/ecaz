# Review: SPIRE CustomScan Planner Path (30809)

## Summary

The path-generation slice is the right shape: extends the rel-pathlist
hook to discover ec_spire indexes via `rel.indexlist`, gates on
`sort_pathkeys` + `limit_tuples` + the eligibility-surface signal,
adds a `CustomPath`, and implements `PlanCustomPath` enough to build a
`CustomScan` plan node so EXPLAIN works. `BeginCustomScan` correctly
becomes a no-op so EXPLAIN-without-ANALYZE doesn't trip the
fail-closed Exec error.

The fail-closed boundary is preserved at the `ExecCustomScan` layer;
the EXPLAIN test passes by relying on `enable_seqscan = off` /
`enable_indexscan = off` to force the planner toward the new path.

The four review-focus questions:
- Planner gate appropriate? Mostly yes for the slice, with one
  important shape gap (P1 below) that is closed by the immediately
  following packet 30810. Calling that out so it isn't buried.
- CustomPath cost / pathkey handling: provisional and biased; flag
  for the cost packet (P2 below).
- BeginCustomScan no-op for EXPLAIN: correct.
- Tuple delivery still open: acknowledged.

## Findings

### P1 â€” Planner gate does not verify ORDER BY shape

`custom_scan_candidate_index_oid` checks
`!sort_pathkeys.is_null()` and `limit_tuples >= 0.0`, then calls
`custom_scan_index_eligibility_result` to confirm an ec_spire index
with a remote-available placement exists. It does **not** verify that
the sort pathkey is the vector-distance ordering against the embedding
column of the ec_spire index. This means in this slice in isolation,
any `ORDER BY id LIMIT k` (or any other ordered-output query) against
a table that happens to have an ec_spire index with remote placements
gets a CustomScan path generated.

The next packet (30810) closes this gap by adding
`custom_scan_orderby_query_expr` and calling it from
`custom_scan_candidate_index_oid`, which validates the sort target is a
single relation-var `<op>` constant-`real[]` expression. So at HEAD this
is fixed. But at this commit alone, a `BeginCustomScan` no-op for
EXPLAIN is the only thing preventing wrong-results behavior â€” any
query reaching `ExecCustomScan` would have errored, but only because
*all* exec paths were errored, not because the path was correctly
gated.

If 30809 lands separately (e.g., bisected), this slice is shape-unsafe.
For history, this would be cleaner if the eligibility check and the
ORDER-BY-shape check landed together. Not asking for a re-spin; flagging
so the bisect-failure mode is recorded.

### P2 â€” CustomPath cost is artificially low

`add_custom_scan_path` sets `total_cost = rows.max(1.0)` (where rows
is `limit_tuples` or `rel.rows`). With LIMIT 1 the cost is 1.0, which
beats almost any other path. The EXPLAIN test masks this by disabling
seqscan and indexscan, but in production the planner will pick
CustomScan over a perfectly fine local index path simply because the
declared cost is fictional.

This is acknowledged scaffolding (request says "first path slice"),
but pin in the cost packet that the cost model needs to reflect:
- routing-traversal cost on the coordinator;
- per-remote dispatch latency Ă— fanout count;
- bounded heap-rerank cost.

Otherwise an `ec_spire`-indexed table with all-local placements but
some remote placements still hits CustomScan when a local index scan
would be faster.

### P2 â€” Pathkey declaration trusts the executor's output ordering

`custom_path.path.pathkeys = root_ref.sort_pathkeys` declares the
CustomScan output is ordered by the planner's requested sort key. The
production executor stream does return rows in score order with the
deterministic tie-breaks, so for the vector-distance pathkey on the
correct column this is true. But the same ORDER-BY-shape gap from P1
applies: if the planner pathkey is *not* the vector-distance ordering
this declaration is a lie. With the 30810 shape check this becomes
true-by-construction; without it, the declared pathkeys can mismatch
the actual output ordering, producing wrong LIMIT-k results without
an explicit Sort to catch it.

### P3 â€” `custom_scan_index_eligibility_result` runs per-relation per-query

The hook calls into eligibility on every relation it sees, which
means every query plan against any base relation pays for at least
one `index_open` + placement-directory page read per ec_spire-indexed
table the relation lists. Probably fine for v1 (placement directories
are small), but for catalogs / tables with many indexes this adds up.
A planner-time cache (per-snapshot or per-backend) keyed on
`(index_oid, active_epoch)` would amortize cleanly. Not blocking.

### P3 â€” `CustomScan` node copies clauses to both `qual` and `custom_exprs`

`custom_scan.scan.plan.qual = clauses` and
`custom_scan.custom_exprs = clauses` carry the same list pointer. This
is the conventional pattern for a CustomScan that wants normal qual
filtering and also needs the executor to fix expression varnos in
`custom_exprs`, but it does mean the same node list is referenced
twice. Postgres list nodes are reference-semantic and copyObjectImpl
will deep-copy if needed downstream; just confirming this is
intentional rather than a missed copy.

## Reviewer recommendation

Accept, noting that the P1 ORDER-BY-shape gap is closed by 30810.
Pin the cost-model and pathkey-validity work for the next costing
packet so production deployments don't get surprised by an artificially
cheap CustomScan path.
