## Answers to the packet's review questions

- **Closed in-flight async query connections should map to
  `remote_backend_terminated`?** Yes. Backend termination
  (`pg_terminate_backend`, OOM kill, crash, etc.) presents to the
  client as a closed connection rather than a SQLSTATE; without
  this category, the failure would have folded into
  `connect_failed` (wrong — connect succeeded) or
  `remote_query_failed` (wrong — no SQL error was returned).
  Having a dedicated category lets operators distinguish "remote
  was unhealthy mid-query" from "remote was unreachable" or
  "remote returned an error", which have different remediation.
- **Transport-path coverage enough for this narrow slice?** Yes.
  The compact-receive path uses the same async adapter, so the
  classification is shared by construction; a separate fixture
  for the receive path is optional. The broader fault matrix in
  Stage E will cover both paths under the production gate.
