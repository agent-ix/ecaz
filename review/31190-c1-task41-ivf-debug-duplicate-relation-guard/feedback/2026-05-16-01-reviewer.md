# Review: Task 41 IVF duplicate debug relation guard (31190)

## Summary

Migrates `debug_ec_ivf_validate_no_duplicate_heap_tid` in
`src/am/ec_ivf/insert.rs` from manual `index_open` /
`index_close` to `IndexRelationGuard::access_share`. Baseline
4,254 → 4,252 (−2).

## Confirmations

- **`access_share` chosen correctly.** The prior code opened with
  `AccessShareLock`; the new helper goes through the convenience
  constructor with the same effective lockmode and an error-on-null
  failure (matches the prior explicit null check that called
  `pgrx::error!`).
- **Guard lifetime covers metadata read + duplicate check.** The
  helper reads IVF metadata then walks heap TIDs to assert no
  duplicate is present; both happen with the guard alive, drop at
  function return.
- **Fully-qualified import.** The cfg-gated helper uses
  `crate::storage::relation_guard::IndexRelationGuard` qualified
  rather than a `use` at the top, to avoid an unused-import
  warning in non-test builds. Pragmatic; matches the standard
  cfg-gating pattern. ✔
- **Baseline arithmetic.** −2 (open + close).

## Findings

No blockers.

### Note on remaining `RelationLockGuard` in this file

`src/am/ec_ivf/insert.rs` still defines a module-local
`RelationLockGuard` struct (one of the items I flagged in my
31180 finding #1). This packet migrated only the debug helper,
not the production insert path. The production
`RelationLockGuard` may have different lock semantics or
lifetime than `IndexRelationGuard` — worth a follow-up packet
that either:

1. Inspects `RelationLockGuard` and routes its uses through
   `IndexRelationGuard::open(... lockmode ...)` if the shape
   matches, or
2. Documents why `RelationLockGuard` needs to stay specialized
   (e.g., if it wraps a different resource like `LockRelationOid`
   rather than a relation handle).

Not blocking. Tracked.
