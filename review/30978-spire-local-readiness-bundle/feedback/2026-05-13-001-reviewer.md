# Review: SPIRE Local Readiness Bundle Attempt (30978)

## Summary

**Bundle attempt, not closeout.** Captures fresh packet-
local evidence through `target/debug/ecaz` and identifies
three blockers that prevent honestly closing Phase 12.9.

**Passing**: CustomScan distributed read with
`typed_payload_probe=ready,pg_binary_attr_v1,t,t`;
helper-mode INSERT + read; two-remote transport overlap
with `fast_completed_before_slow=true`; remote statement
timeout (strict + degraded); local cancel (strict +
degraded); endpoint metrics with p50/p95/p99 latency,
payload bytes, route/candidate/heap counters.

**Blockers**:

1. Trigger-mode `insert-read-after-customscan-pg18` exits
   nonzero (`coordinator_row_count=1`); focused PG18
   trigger test passes — needs investigation at fixture/
   transaction boundary.
2. `ecaz bench spire-pipeline` can't automate measurements:
   - tokio-postgres query-metrics path hits v1 DML
     frontdoor;
   - fresh local readiness DB hits same guard fetching
     exact truth for recall;
   - older `tqvector_bench` corpus lacks
     `ec_spire_remote_search_endpoint_identity(oid)`.
3. SQL recall sanity reports `recall_at_10 = 0.0000` even
   on one-list/full-rerank — not acceptable as readiness
   pass.

## Findings

### P1 — Right discipline: don't claim closure under blockers

This is exactly the right way to handle a Phase 12.9
attempt that hits real obstacles. Three different
discipline failures avoided:

- Did NOT close the rows on partial evidence.
- Did NOT massage the failing artifacts into looking
  passing.
- Did NOT defer-with-vagueness ("we'll come back to
  this").

Instead: captured what passed, named what blocked,
attributed each blocker to specific causes, and **kept
the rows open**. That's the audit-friendly shape — a
future reader can see exactly what's done and what isn't.

This packet is more valuable as a **blocker manifest** than
it would have been as a forced closure.

### P1 — Blocker 1: trigger-mode fixture (resolved by 30979)

The `coordinator_row_count=1` mismatch is exactly the
class of bug that the BEFORE ROW + AFTER STATEMENT trigger
pair from 30973 should have prevented (suppressed local
heap row → coordinator row count = 0). The PG18 unit
fixture passes, so the suppression logic works in the test
harness. The live fixture failing means either:

- the live fixture's assertion is wrong (it's reading
  through CustomScan instead of heap directly); or
- transaction visibility differs between the live setup
  and the unit fixture (e.g., autocommit boundaries).

30979 (next packet) addresses this — the live fixture's
heap-count query was `WHERE id = 303`, which is a
distributed PK-select shape that reads through CustomScan,
not the local heap. So the assertion was observing the
remote row via the distributed path, not checking heap
suppression. **Fixture bug, not product bug.**

The right disclosure: 30978 honestly named "needs
investigation"; 30979 did the investigation and confirmed
fixture-side. That's the right escalation pattern — name
the blocker, then the next packet either fixes or
escalates further.

### P1 — Blocker 2: bench CLI v1 DML frontdoor

Three sub-blockers all rooted in the same product-path
gap: the v1 DML frontdoor fail-closed guard rejects shapes
the bench client wants to issue via tokio-postgres.

This is the **same issue 30975 disclosed** as a workaround
note. The measurement work has been routing around the
bench client by using `ecaz dev sql` + psql instead. That
worked for one-shot measurement packets, but for *automated
recurring readiness*, the bench client needs to either:

- have its query shapes widened to fit the v1 DML
  frontdoor; or
- use a documented bypass (e.g., a "diagnostic mode" that
  skips frontdoor enforcement for known-safe SELECTs).

The third sub-blocker (older `tqvector_bench` corpus) is a
separate fixture-version skew issue — pin a follow-up to
either upgrade the corpus surface or document which corpora
support which endpoints.

This is a **real product-path gap** (not just fixture).
Worth pinning a Phase 12.9 follow-up tracker note: the
bench-client KNN shape needs widening before the final
readiness bundle can be automated. Manual artifact capture
via `ecaz dev sql` is acceptable for now but doesn't scale.

### P1 — Blocker 3: recall_at_10 = 0.0000

This is the most concerning blocker. On a one-list /
full-rerank fixture, recall@10 should be near-perfect (the
one-list-full-rerank case is the trivial-correctness
oracle — every query exact-matches because everything
gets reranked). Recall=0 means either:

- the **truth set** is wrong (different corpus, different
  PK mapping, different query embeddings);
- the **lookup** is wrong (joining on the wrong key);
- the **storage_format/index params** don't match between
  fixture build and recall lookup;
- something in the local pipeline is genuinely returning
  no overlap with truth.

Right call to NOT mark the row complete. Recall=0 is not
"low recall" — it's "we measured the wrong thing." A
final readiness bundle reporting recall=0 would be a
correctness alarm, not a readiness checkmark.

Worth a focused investigation packet on this specifically:
build a minimal one-list/full-rerank corpus, manually
compute exact truth, run the recall path against it, and
diff the truth vs the recall results to find where they
diverge. The recall path is load-bearing for the bundle —
without it, the bundle can't claim correctness.

### P2 — Passing evidence is substantively complete

The passing evidence covers most of the readiness surface:

- distributed read end-to-end (with typed transport
  verified via probe);
- helper INSERT + read end-to-end;
- two-remote transport overlap (fast/slow ordering);
- two fault categories in both strict + degraded modes;
- live endpoint identity + per-query metrics + pipeline
  counters.

What's NOT covered (because of the blockers):

- trigger-mode INSERT (resolved by 30979);
- automated recall (blocker 3);
- automated repeated bench (blocker 2).

If the recall path is fixed and the bench client is
widened, the existing passing evidence + automated
recall + automated bench = final bundle. The bundle is
about 70% there in evidence terms, gated on the three
blockers.

### P2 — Tracker discipline

Correctly leaves both Phase 12.9 rows open:

- live packet-local artifact capture: blockers 2 and 3
  prevent automation.
- final production-readiness bundle: blocker 3 prevents
  honest claim.

Honest open status. The packet is artifact, not closure.

### P3 — Where to go from here

Suggested sequencing for closing the remaining Phase 12.9
rows:

1. **Blocker 3 first**: investigate recall=0. Without
   correct recall measurement, the bundle can't claim
   correctness, so this is the gating concern.
2. **Blocker 2 second**: widen bench client KNN shape (or
   document diagnostic-mode bypass) so the bundle can be
   automated. This unblocks the artifact capture row.
3. **Blocker 1 already resolved by 30979** (verify in
   that packet's review).
4. **Final bundle**: with 1-3 done, re-run the
   manifest-of-evidence pattern from this packet, this
   time as a closure rather than an attempt.

This is roughly 1-2 weeks of work depending on what the
recall investigation finds.

## Reviewer recommendation

Accept as **blocker manifest, not closure**. Right
discipline (no false closure, name blockers specifically,
attribute causes, keep rows open). The packet's value is
the honest open-row inventory — closing on partial
evidence would have been worse than this honest defer.
