# Review: SPIRE Coordinator Insert Planning Primitive (30828)

## Summary

First narrow ADR-069 INSERT slice. The new
`ec_spire_plan_coordinator_insert(index_oid, pk_value, embedding,
source_identity)` SQL function:

- Validates `pk_value` is non-empty.
- Validates `source_identity` is exactly 16 bytes (per ADR-063).
- Calls the same `am::spire_classify_centroid(...)` the
  `ec_spire_classify_centroid()` SQL helper uses, sharing the
  centroid-routing path so the planner and the eventual write path
  cannot disagree.
- Returns `(index_oid, pk_value, node_id, centroid_id, served_epoch,
  source_identity)` matching the placement-directory tuple shape
  exactly.

Side-effect-free: no remote dispatch, no placement-directory
mutation, no PREPARE TRANSACTION. Tests cover happy path
(round-tripping the planned tuple), empty-PK rejection, and bad
source-identity rejection. ADR-069 gains a description of the
primitive and the Phase 11 tracker marks the planning sub-slice
complete.

The three review-focus questions:
- Side-effect-free primitive as the right first boundary: yes —
  separates "what would I do" from "do it" cleanly. The mutating
  INSERT path can wrap this primitive plus a remote-2PC dispatcher,
  and tests can exercise the planning logic without a remote
  cluster.
- Returned fields match `ec_spire_placement` tuple contract: yes,
  field order and types are identical, including the `index_oid` and
  `pk_value` echo.
- Validation limited to PK bytes and identity length: agreed for
  this primitive; duplicate-PK detection rightly belongs to the
  catalog `INSERT`, and remote dispatch failures belong to the
  later mutating path.

## Findings

### P2 — Function declared `STRICT` but errors on empty Vec instead

`#[pg_extern(stable, strict)]` makes a NULL argument short-circuit
to a NULL row. So `pk_value = NULL` returns NULL silently instead
of erroring. But `pk_value = bytea ''` (empty bytea, non-NULL)
errors with `pk_value must not be empty`. The asymmetry is a bit
surprising for a planning primitive: NULL → silent NULL row;
empty → loud error. Two acceptable shapes:

- Drop `strict`, surface NULL `pk_value` with the same error message
  ("pk_value must not be empty/NULL"), so the failure mode is
  consistent across "the operator handed me garbage" cases.
- Keep `strict` and document it explicitly: NULL-elision is for
  composition (e.g., the future INSERT-as-SQL-function path will
  treat NULL inputs as a no-op).

Mild preference for the first; the future mutating INSERT will need
explicit NULL handling at its own boundary either way.

### P2 — `source_identity` length check, not byte-pattern check

Length is enforced at exactly 16 bytes, which matches ADR-063, but
the all-zeros source identity (`\x00\x00...\x00`) is currently
accepted. ADR-063 doesn't reserve a sentinel value, so this is
fine in v1, but if any future writer convention uses zero as
"unset", the planning primitive should reject it before it lands in
the placement directory. Worth pinning either way: either
"all-zero source_identity is a valid 16-byte identity" or "reject
all-zero". The placement catalog has no constraint on this, so the
contract has to live in the planning helper.

### P2 — Planning primitive duplicates the open `index_relation`

The body opens the index, calls the classifier, then closes. The
mutating INSERT path will also need to open the index (to forward
the row to the remote and to write the placement directory). Worth
factoring out a helper that takes `Relation` + `embedding` and
returns the planned tuple, so the mutating path doesn't pay for
two opens. Cosmetic refactor; not blocking the v1 INSERT slice.

### P3 — `index_oid` echoed in the return row

The function returns `(index_oid, pk_value, ...)` with the same
`index_oid` it received. Convenient for INSERT-into-placement
SELECT-from-plan composition (e.g., `INSERT INTO ec_spire_placement
SELECT * FROM ec_spire_plan_coordinator_insert(...)`). Confirmed
via the test's `format!("{}:010203:7:{expected_centroid_id}:...",
u32::from(index_oid))` assertion.

### P3 — No coverage for "no active epoch"

The classifier already errors with "ec_spire_classify_centroid
requires an active ec_spire routing epoch" when called against a
freshly-built index that hasn't published an epoch. The planning
primitive inherits that behavior. Worth a brief test that asserts
the propagated error wording so a future classifier-error rewrite
doesn't silently change the planning primitive's surface.

### P3 — `should_panic(expected = "...")` matches substring

Both error tests use substring matching. Good practice for catching
the contract without overconstraining the message text.

## Reviewer recommendation

Accept. The primitive is the right first boundary for the v1
INSERT path — it's testable in isolation, side-effect-free, and
shape-compatible with the placement catalog. The P2 items
(strict-vs-empty asymmetry, source_identity sentinel question) are
cleanups before the mutating INSERT slice consumes this primitive
in anger.
