# Review: Task 41 SPIRE search tuple-payload guards (31167)

## Summary

Splits the 31166 helper into two layers:
`ec_spire_heap_relation_oid_from_index(&AccessShareIndexRelation)
-> Oid` (operates on an already-open guard) and the original
`ec_spire_index_heap_relation_oid(oid, caller) -> Oid` (now thin —
opens a guard and forwards). Migrates
`ec_spire_remote_search_tuple_payload` and
`ec_spire_remote_search_tuple_payload_typed`, which need **both**
the heap OID **and** an AM read against the same guard. Baseline
4,467 → 4,461 (−6).

## Confirmations

- **Helper split is the right factoring.** Functions that need
  only heap OID call `ec_spire_index_heap_relation_oid`. Functions
  that need heap OID *and* an AM call get to open one guard, copy
  the OID, do the AM call, drop. No redundant index_open.
- **`&AccessShareIndexRelation` borrow is a real safety upgrade.**
  The new helper takes `&AccessShareIndexRelation` (Rust borrow
  with a real lifetime) rather than `pg_sys::Relation` (raw
  pointer, no lifetime). The caller can no longer pass a dangling
  reference — the compiler enforces the guard outlives the call.
  This is a small but meaningful instance of moving from
  convention-enforced to type-enforced lifetime.
- **Per-function guard scope is correct.** Both migrated functions
  open one guard, copy the heap OID, run the AM call with
  `as_ptr()`, then `drop(index_relation)` **before** the
  `ec_spire_validate_tuple_payload_columns` / SPI fetch work. The
  validation reading is OID-only at that point. ✔
- **Single-guard-for-both-uses is justified.** The reviewer focus
  asks whether one guard for both heap-metadata and AM is correct
  vs. two guards; the answer is yes — both reads need the same
  validated index, no FFI/env work in between, and avoiding a
  second `index_open` is a free win.

## Findings

No blockers.

- The pattern of `&AccessShareIndexRelation` accepting borrows is
  worth lifting into the 31161 strategy matrix as the preferred
  shape for any future helper that does "operate on an already-open
  index" (vs. helpers that open the guard themselves). It's the
  cleanest safety contract this subtrack has produced so far.
- Same naming nit as 31166. `ec_spire_heap_relation_oid_from_index`
  is descriptive but verbose; `spire_heap_oid(&guard)` or similar
  would read more like a property accessor.
