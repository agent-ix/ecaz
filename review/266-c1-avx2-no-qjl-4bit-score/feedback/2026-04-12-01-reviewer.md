# Review: Packet 266 — C1 AVX2 No-QJL 4-bit Score

Reviewer: agent-ix (via code-review)
Date: 2026-04-12

## Assessment: Valuable negative result — correctly discarded, and the data changes the optimization strategy

### What this packet did right

1. **Ran a clean experiment.** Implemented the AVX2 scorer, verified
   correctness (scalar-vs-dispatched agreement after the nibble lane ordering
   fix), measured on both the isolated microbench and the full warm SQL
   surface, found no improvement, discarded.

2. **Recorded the result instead of discarding silently.** This packet is
   exactly the kind of failed experiment that should stay in the review chain.
   It prevents the same idea from being re-attempted, and it provides evidence
   for why the remaining optimization strategy needs to shift.

3. **Correctly identified the nibble lane ordering bug.** The
   `low0, high0, low1, high1, ...` interleave is a common trap in packed
   nibble extraction. Catching it in the test harness before benchmarking is
   good practice.

### Why AVX2 didn't help — and why the survey's prediction was wrong

Packet 264's survey predicted a 4-8x speedup from AVX2 on the scorer. The
actual result was scalar=14082ns vs AVX2=14301ns (slightly slower). This is
worth understanding:

1. **The L1-resident working set neutralizes SIMD's main advantage.** After
   packet 265 removed the LUT build, the scorer operates on codebook (64B) +
   rotated (6KB) + mse_packed (768B) ≈ 7KB, all permanently in L1. The
   scalar loop's bottleneck is throughput, not memory latency. Modern x86
   out-of-order cores can sustain ~2 loads + 1 FMA per cycle on L1 data,
   which is close to what the scalar nibble loop achieves.

2. **The 16-entry codebook lookup is expensive in AVX2.** With 16 centroids,
   a single `_mm256_permutevar8x32_ps` covers only 8 entries. The
   two-permutevar + cmpgt + blendv approach for 16 entries costs ~6-8 cycles
   per 8 dimensions. The scalar codebook[nibble] lookup costs ~1-2 cycles per
   dimension (L1 hit, no branch). So the AVX2 lookup overhead per dimension
   is comparable to the scalar cost, eliminating the expected throughput gain.

3. **Nibble extraction is not free.** The broadcast + variable shift + mask
   sequence to extract 8 nibbles from 4 packed bytes adds ~3-4 instructions
   per group of 8. The scalar code pays ~2 instructions per nibble (mask +
   shift). The per-dimension instruction count is similar.

4. **The packet 264 estimate was wrong because it assumed SIMD would reduce
   instructions-per-dimension by 4-8x.** In reality, the AVX2 path trades
   simple scalar instructions for more complex vector instructions with
   similar latency characteristics when the data is already in L1. The 8-wide
   execution is real, but the per-lane overhead for codebook lookup + nibble
   extraction eats the width gain.

### What this means for the remaining C1 strategy

The scorer is now near-optimal for the current data layout. Further scoring
improvements would require a different data layout:

- **Pre-expanded codebook values.** Instead of storing 4-bit indices and
  looking up the codebook at score time, store pre-multiplied values during
  encoding. This trades storage for compute but changes the on-disk format.
- **LUT-based scoring with a cache-friendly LUT layout.** The 96KB LUT was
  removed because it polluted L2, but a smaller LUT (e.g., 16 f32 values per
  cache line, tiled to match the access pattern) might work if carefully
  designed. This is speculative.
- **Reduced dimensionality.** Fewer dimensions means fewer score iterations.
  This is an application-level decision, not a runtime optimization.

Given that the scorer is now near-optimal, the remaining `~11ms` gap to the
5ms target must come from non-scoring costs: graph loading, allocation
overhead, beam bookkeeping, and PG buffer manager overhead.

## Suggestions

1. **Update the packet 264 survey to mark idea #1 as a recorded failure.**
   Note the specific reasons (L1-resident working set, 16-entry permute
   overhead) so future SIMD attempts on different data layouts can learn from
   this.

2. **Re-profile from the 11ms baseline.** With scoring optimized as far as
   the current layout allows, the perf distribution should look very different
   from the earlier 31.78%-in-scoring profile. A fresh `perf` run will show
   where the remaining time goes.

3. **Consider whether the scorer can be called fewer times rather than made
   faster.** Score caching (already present), early termination in beam
   search, or a coarser pre-filter could reduce the number of score calls.
   This is an algorithmic rather than microarchitectural approach.
