## Bug fix uncovered by the fixture

The fixture exposed a production AM blocker that existed
**before** scan execution: PostgreSQL planning calls SPIRE
cost / diagnostic paths before `amrescan`, and those paths
still assumed remote-owned placements were unreadable
locally. The fix changes planner/diagnostic metadata to:

- **Load the coordinator fanout manifest** (per the helper
  introduced in packet 30754) for remote-owned placements
  during planning/diagnostic reads.
- **Read the coordinator's published metadata copy** so
  the planner has cost data without requiring local
  payload reads.
- **Preserve remote placement ownership for execution** —
  the change is read-only at planning time; nothing about
  scan/receive/heap-resolution flow changed.

The scoping is right. This is the kind of bug fixture work
should uncover: a path that nobody hit until the AM cursor
delivered a real tuple to PostgreSQL's executor.
