# Review: SPIRE DML Catalog Relation Context (30855)

## Summary

Catalog/relcache-backed sibling to the SPI-backed relation-context
loader. `dml_frontdoor_relation_context_catalog_row(heap_oid)`:
- Opens heap with `table_open(AccessShareLock)`.
- Walks `RelationGetIndexList(heap)` to find ec_spire indexes via
  `get_index_am_oid(c"ec_spire", true)`.
- Errors with the same v1 multi-index message as the SPI loader
  if more than one ec_spire index exists.
- Pulls primary-key column from `pg_index.indisprimary && indnkeyatts == 1`
  on each index, returning the first single-column bigint PK.
- Pulls ordinary heap columns from the open relation's tuple
  descriptor.
- Pulls embedding columns from `indkey[0..indnkeyatts]` of the
  selected ec_spire index, preserving INCLUDE-column exclusion.

PG18 fixture compares the catalog context against the SPI-backed
context for status, PK column, and embedding columns — locking
parity between the two paths.

The reason this matters: the planner hook can now consume catalog
metadata without recursive SPI inside `planner_hook`. That unblocks
the next slice (the actual plan rewrite) cleanly.

The three review-focus questions:
- Relcache/index metadata path safe as planner-hook metadata
  source: yes — uses standard PG catalog primitives that are
  re-entrant safe under the planner.
- v1 index selection semantics match SPI-backed loader: yes (same
  multi-index error text + same single-bigint-PK gate).
- `indnkeyatts` boundary preserves ordinary-column / embedding-
  column parity: yes — the catalog walker calls
  `*index_form.indkey.values.as_ptr()` only for slot 0 since
  `indnkeyatts == 1`.

## Findings

### P2 — Two relation-context loaders now exist

The SPI-backed loader (`dml_frontdoor_relation_context_row`) and
the catalog-backed loader
(`dml_frontdoor_relation_context_catalog_row`) implement the same
contract via different mechanisms. The PG18 parity fixture
compares them, but as the implementations evolve, drift is
possible.

Two acceptable shapes long-term:
- Keep both, with the parity fixture as a regression guard. The
  SPI loader stays for operator diagnostic; the catalog loader is
  the production hook path. Document the split in a module
  comment.
- Migrate the SPI loader to delegate to the catalog loader once
  the SPI re-entrance concern is resolved (it doesn't apply
  outside the planner hook), so there's a single implementation.

For v1 the first is fine and matches the existing pattern.

### P2 — `RelationGetIndexList` lock ordering

The walker opens each index via `index_open(AccessShareLock)` and
closes it. That's the standard PG pattern. Worth a one-line
comment confirming that `RelationGetIndexList` returns a private
copy of the index list (it does) so the loop's index-open / close
pattern is safe even if the relcache evicts during the walk.

### P3 — `indnkeyatts != 1` rejection of multi-column PKs

If a heap has a multi-column PK (`PRIMARY KEY (tenant_id, id)`),
the catalog walker correctly returns `None` for primary_key. The
status then becomes `unsupported_pk_shape`. Same behavior as the
SPI loader; consistent.

### P3 — `index_oid.to_u32() < ec_spire_index_oid.to_u32()`

Selects the lowest-OID ec_spire index when multiple exist (which
is then immediately rejected by the multi-index check). The
"select lowest OID" tie-break is dead code given the immediate
rejection, but harmless — kept for parity with the SPI loader.
Worth removing once both loaders enforce single-index strictly.

### P3 — `pg_sys::INT8OID` direct comparison

Standard. Multi-tenant deployments with custom int8 domain types
won't match — that's the v1 contract: bare `bigint` only, not
domains over `bigint`. Worth pinning in ADR-069 if not already.

### P3 — Catalog loader returns `Result<...>`, SPI loader too

Same error type (`String`). Caller pattern is identical between
the two loaders, which is exactly right for the parity fixture.

## Reviewer recommendation

Accept. With this in place, the planner hook now has everything it
needs to wire actual plan replacement: target relation extraction
(30851), catalog-safe relation context (30855), query shape
extraction (30848), and pure shape classification (30847). The
next packet should land the plan-rewrite logic itself, replacing
the eligible UPDATE/DELETE/PK SELECT plans with a CustomScan that
calls the existing primitives.
