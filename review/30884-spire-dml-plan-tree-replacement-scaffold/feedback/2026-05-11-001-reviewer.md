# Pre-Checkpoint Review: SPIRE DML Plan-Tree Replacement Scaffold (30884)

## Note

Code committed in `0e63d2c3 Scaffold SPIRE DML CustomScan plan
replacement` but no `request.md` exists yet. Pre-checkpoint
feedback per `review/REVIEWER.md`.

## Summary

**This is the slice I recommended in 30803 seq 004.** The
planner_hook now:

1. Runs `dml_frontdoor_observe_planner_query(parse)` before
   delegation (existing behavior).
2. If the decision says "supported UPDATE/DELETE plan-tree
   replacement," extracts the primitive-plan expression via
   `dml_frontdoor_primitive_plan_expr_catalog_row(query)`.
3. Calls the chained previous_hook / `standard_planner` to get
   the standard `PlannedStmt`.
4. **Replaces `planTree`** via
   `custom_scan_dml_replacement_plan(plan_expr, fallback_plan)`
   — the new helper builds a CustomScan node sized like the
   fallback plan, no ModifyTable above it.
5. Records `last_hook_action = "plan_tree_replaced_customscan"`
   in the diagnostic surface.

`plan_rewrite_enabled = true` flips to `true` in the hook status.

**Execution still fails closed:** UPDATE/DELETE executor
branches haven't been wired yet. The PG18 fixture confirms
`EXPLAIN UPDATE/DELETE ... WHERE id = 5` reports the
plan-tree-replaced action, but actually running the UPDATE
errors with the existing executor guard.

## Findings

### P1 — `custom_scan_dml_replacement_plan` cost copying

The replacement CustomScan inherits `disabled_nodes`,
`startup_cost`, and `total_cost` from the fallback plan
(`(*fallback_plan).total_cost` etc.). That's reasonable for
preventing PG from disabling the plan, but the *fallback* plan
is the original ModifyTable, which has a cost reflecting heap
tuple processing — usually higher than what the CustomScan
will actually cost. PG won't re-pick another plan because the
CustomScan replaces planTree directly (not as a competing path),
so cost only matters for `EXPLAIN` output and PG's
decision-making in nested contexts.

Acceptable for v1. Worth a comment that the cost copy is
"inherit so EXPLAIN looks reasonable; planner doesn't actually
choose between plans here."

### P1 — Replaced plan's `targetlist = NULL` and `qual = NULL`

The replacement CustomScan sets:
- `scan.plan.targetlist = NULL`
- `scan.plan.qual = NULL`
- `scan.scanrelid = 0`

For UPDATE/DELETE that's fine because the user query doesn't
project anything (UPDATE has no top-level targetlist; DELETE has
none either). But: PG's executor walks `planTree.targetlist` to
build the `EState.es_processed` machinery for command-tag
purposes, and a NULL targetlist may cause unexpected behavior in
some PG paths.

Worth confirming that executing this plan with `commandType =
CMD_UPDATE / CMD_DELETE` and a NULL plan targetlist doesn't
trigger PG executor assertions. If it does, populate a minimal
targetlist (a single junk entry) so the executor has something
to walk. Specific test: run actual `UPDATE`/`DELETE` against the
fixture once the executor branch is wired in 30884's follow-up
and confirm `UPDATE 1`/`DELETE 1` reports correctly.

### P1 — `plan_rewrite_enabled = true` is now a load-bearing
diagnostic

The hook status flips to `true`. Operators reading the status
surface will now see "rewrite enabled" — but actual rewrites
don't happen until the executor wiring lands. Worth either:
- Renaming the field to `plan_tree_replacement_scaffold_enabled`
  to make the partial state explicit; or
- Keep the name but pin in the diagnostic comment that
  `plan_rewrite_enabled = true` means "supported shapes plan as
  CustomScan; execution is gated on per-mode wiring."

### P2 — Fallback plan disposal

The `custom_scan_dml_replacement_plan(plan_expr, fallback_plan)`
helper takes the fallback plan but only reads its cost fields.
The original plan tree is never freed — PG's planner allocates
in a memory context that gets reset, so this isn't a real leak,
but worth confirming the dropped plan tree doesn't carry
references to relations / locks that need explicit cleanup. PG
generally handles this via memory-context lifetime, so probably
fine.

### P2 — Test fixture proves plan replaces; doesn't yet prove
behavior

The fixture asserts `last_hook_action == "plan_tree_replaced_customscan"`
for both UPDATE and DELETE EXPLAIN, then runs an actual UPDATE
and catches the executor guard error. Good — this proves the
plan replacement works without claiming behavior. The next
packet's executor-wiring test should run the actual UPDATE
through to completion and assert:
- `UPDATE 1` rowcount.
- The remote heap reflects the change.
- The placement-directory row is unchanged (non-embedding UPDATE
  doesn't change placement).

### P3 — `dml_frontdoor_uses_plan_tree_replacement(decision)`

Predicate is correctly narrow:
`decision.supported && custom_scan_mode in ("update_payload",
"delete_payload")`. PK SELECT is excluded — it has its own
existing CustomPath path at the rel-pathlist level. Clear
operation/path-shape split.

### P3 — `LAST_HOOK_ACTION = Some("plan_tree_replaced_customscan")`

New action string. The `test_ec_spire_dml_frontdoor_hook_status_installed_pass_through`
test was updated to accept this in its allowed-statuses list.
Good.

## Reviewer recommendation

**Accept the milestone scaffold.** This delivers the plan-replace
shape recommended in 30803 seq 004 cleanly. The two P1 items
(cost-copy comment, targetlist=NULL behavior) are correctness
concerns to verify when the executor-wiring slice lands and runs
actual UPDATE/DELETE statements.

The path forward is now mechanical:
1. Wire UPDATE executor branch in `ec_spire_custom_scan_access`
   (or earlier in `BeginCustomScan`): construct the
   `SpireDmlFrontdoorPrimitiveInvocation`, call
   `ec_spire_forward_coordinator_update_tuple_payload(...)`,
   increment `estate->es_processed`, return NULL.
2. Same shape for DELETE with the matching primitive.
3. Multicluster fixtures.
4. ADR-069 docs slice for v1 trigger / RETURNING / transition-
   table limitations.

This is the load-bearing structural change for transparent DML
v1. Once the executor branches land, applications can run
`UPDATE distributed_tbl SET ... WHERE id = $1` and have it Just
Work — no view, no stub rows, no application-layer routing.
