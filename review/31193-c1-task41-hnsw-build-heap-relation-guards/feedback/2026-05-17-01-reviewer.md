# Review: Task 41 HNSW build heap relation guards (31193)

## Summary

Migrates two HNSW build heap relation opens in
`src/am/ec_hnsw/build.rs` from manual `table_open` /
`table_close` pairs to `HeapRelationGuard::try_access_share`.
Both sites occur during build-time setup
(`BuildState::new`'s indexed-vector-kind probe and
`validate_grouped_rerank_source_column_for_empty_build`).
Baseline 4,245 → 4,241 (−4).

## Confirmations

- **`try_access_share` chosen correctly.** Both prior sites used
  `AccessShareLock`; both new call sites use the same effective
  lock through the convenience constructor. ✔
- **Null-open semantics preserved by explicit
  `pgrx::error!`.** The previous code didn't have a null check —
  `table_open` either returns or raises ERROR inside Postgres.
  The new code adds a defensive `let Some(...) else { pgrx::error!(...) }`
  guard pattern, which is consistent with the rest of the Task 41
  surface. Acceptable defensive code.
- **Guard lifetime covers the live-relation reads.** In
  `BuildState::new`, the guard is alive across
  `source::resolve_indexed_vector_attribute(heap_relation.as_ptr(), ...)`;
  the returned `IndexedVectorKind` is owned (enum value), so
  guard can drop at the `else` block boundary. ✔
- **Validation site mirrors.** `validate_grouped_rerank_source_column`
  takes a raw relation pointer; the guard is alive across the
  call, drops at function return. ✔
- **Baseline arithmetic.** −4 across two sites ≈ 2 each
  (open + close).

## Findings

No blockers.

- This is the first migration that touches the HNSW build path
  (`src/am/ec_hnsw/build.rs`). Build code is unsafe-heavy
  generally — the file has more raw operations beyond these
  relation opens — but only the relation lifetimes are in scope
  for the current relation-resource subtrack. Other unsafe sites
  in build.rs (page writes, parallel build coordination, etc.)
  remain for later subtracks.
