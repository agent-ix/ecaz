# Review: SPIRE Placement Directory Catalog (30817)

## Summary

Catalog surface lands cleanly: the `ec_spire_placement` table with
the ADR-069 column shape, the `(index_oid, pk_value)` primary key,
the `(index_oid, source_identity)` lookup index, and per-column
constraints (non-empty `pk_value`, positive `node_id`, non-negative
`centroid_id`, positive `served_epoch`, 16-byte `source_identity`).
Both `sql/bootstrap.sql` and the `0.1.0--0.1.1` upgrade get the same
DDL. Cleanup diagnostics (`ec_spire_remote_catalog_orphan_summary`,
`ec_spire_remote_catalog_orphan_cleanup`, drop-index event trigger)
correctly include the new table without changing existing
read/executor behavior.

The three review-focus questions:
- Table shape and identity index match ADR-069: yes — column types,
  PK, and identity index match the ADR's example DDL exactly.
- Cleanup diagnostics cover placement rows without changing existing
  behavior: yes; the orphan summary now reports five counts instead
  of four and the cleanup function deletes orphaned placement rows
  alongside the existing categories.
- Task tracker scope is accurate (catalog only, not write
  forwarding): yes — explicitly notes classifier and coordinator-routed
  writes remain open.

## Findings

### P2 — `node_id > 0` excludes the local node by ADR-068 convention

The constraint `CHECK (node_id > 0)` rejects `node_id = 0`, but
`SPIRE_LOCAL_NODE_ID = 0` is the existing convention for a row that
lives on the coordinator's own shard (router-plus-shard
deployments). ADR-069's example DDL doesn't constrain `node_id` at
all, and the placement directory ought to contain the local-shard's
own rows so PK-keyed reads and DELETEs can find them on the
coordinator without a routing detour.

If the intent is "v1 ADR-069 only routes to *remote* nodes and a
coordinator-local placement is implicit," document that and rename
the column / status to make the implicit-local-shard contract
explicit. Otherwise relax to `CHECK (node_id >= 0)`. As-is, a future
PK-keyed SELECT for a coordinator-local row has nowhere to look.

### P2 — Catalog placed in bootstrap of every install, including local-only

`sql/bootstrap.sql` adds the table unconditionally, so a local-only
`ec_spire` install (no distributed deployment) gets an empty
`ec_spire_placement` table forever. That's harmless — but combined
with the 16-byte source_identity constraint and the orphan-cleanup
event-trigger registration, the local-only install pays a small
cost (catalog row, event trigger fire on every DROP INDEX) for a
feature it doesn't use.

Acceptable for v1; flag for the eventual "distributed mode" reloption
work — at that point, the placement table can become
distributed-mode-only.

### P3 — Constraints fire at INSERT time, not at registration

The `CHECK` constraints are good defense for the bulk-load path
(packet 30819) but only after the row reaches the table. If
`ec_spire_register_placement_batch` (next packet) does the validation
client-side too, error messages can be clearer than the catalog
constraint's default text. Worth a follow-up that wraps the catalog
errors with caller-context (which entry index in the batch, which
column).

### P3 — `pk_value bytea` opaqueness

ADR-069 stores PKs as `bytea` for type-agnosticism. The 30817 SQL
test inserts hex-encoded PKs which round-trip cleanly. Worth pinning
in a docs slice that the canonical PK encoding is the type's binary
output (`COPY ... BINARY` form, or `typsend`/`typreceive`) rather
than ad-hoc hex / text — otherwise different INSERT paths could
encode the same logical PK differently and PK lookups would miss.

### P3 — Identity index is plain B-tree

`(index_oid, source_identity)` as a plain B-tree means
`source_identity` lookups within an index do equality scans. That's
correct for the ADR-063 16-byte identity payload. No follow-up.

## Reviewer recommendation

Accept the catalog shape, with the P2 `node_id > 0` constraint as
the one thing to revisit before any code path tries to register a
coordinator-local-shard row. Either the constraint relaxes or the
design explicitly excludes local-shard placements from this
directory; pick one and document it.
