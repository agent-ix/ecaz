# Reviewer Note: Merge Gating Changed — A1–A10 Required Before Merge

This note supersedes the merge dispositions in the prior reviewer
feedback files for this branch. Prior files said the branch was
approved for merge with optimization work tracked as post-merge
follow-up. **That guidance is rescinded.** The user has clarified that
the optimization and quantizer work is **part of this task**, not a
follow-up, and **must land before this branch merges to `main`**.

## What changed

Prior feedback files described the optimization arc (cost-model
audit, vacuum scaling, quantizer dispatch validation, posting-scan
early-stop, head-to-head quantizer comparison, scale measurement) as
post-merge follow-up work. The user has now stated:

> "these are not follow-ups[—]this is work that is considered part of
> the task and will be done now before landing it on main."

The user has also added a quantizer-quality requirement that was not
in any prior reviewer note:

> "I need you to TEST all of the quants. RaBitQ, TurboQuant,
> PQFastScan… We want an honest assessment of the quality of all of
> them. TurboQuant is not even likely to be the best option based on
> its size and speed often being worse than the other two options."

## Where the work is tracked now

`plan/tasks/28-ivf-competitive-substrate.md` has been updated to make
the gating explicit. The tracked items are A1 through A10 in that
file. Summary:

- **A1** — audit `cost.rs` constants against measured runtime cost.
- **A2** — replace full-list-into-memory vacuum with streaming
  bulkdelete.
- **A3** — add physical-compaction support to vacuum.
- **A4** — replace `exact_score_mode_name()` String comparison with
  typed enum match.
- **A5** — audit `ProdQuantizer::cached` cache key correctness across
  scans.
- **A6** — revisit planner cost behavior at higher `nlists` with a
  cross-test matrix.
- **A7** — add posting-scan early-stop with score-bound pruning.
- **A8** — wire **PQ-FastScan and RaBitQ** into `ec_ivf` alongside
  TurboQuant as additional `IvfQuantizerProfile` variants. **Purely
  additive.** TurboQuant is not removed, deprecated, or replaced.
  PQ-FastScan and RaBitQ implementations in `crate::quant` are not
  changed. The work is to make all three *selectable* on an `ec_ivf`
  index via the `quantizer` reloption.
- **A10** — honest head-to-head quantizer assessment across all three
  variants on identical fixtures, with the recommendation on which
  variant should back `quantizer = 'auto'` grounded in the measured
  numbers, not in which variant is currently the default.
- **A9** — re-measure `ec_ivf` at 100k+ scale.

Sequencing is recorded in the same file.

## Disposition

**Merge of `task28-ivf` to `main` is blocked on A1 through A10.** Each
item lands as its own commit (or commit set) with a packet recording
implementation, validation, and measurement where applicable.

When all 10 items are recorded as complete in
`plan/tasks/28-ivf-competitive-substrate.md` with corresponding
packets, this disposition will be revisited. Prior "approve for merge"
language in `2026-04-27-01-reviewer.md`,
`2026-04-27-02-reviewer.md`, and
`review/30047-task28-ivf-review-followups/feedback/2026-04-27-02-reviewer.md`
no longer applies.

## Language note for future feedback

Earlier feedback used the phrase "drop in" to mean "add" or "wire up"
(specifically: "Drop PQ-FastScan as the second quantizer profile,
exercising the dispatch seam"). That phrasing is dangerous in a
software context where "drop" reads as remove/deprecate (DROP TABLE,
drop a feature). Going forward in this packet trail and elsewhere,
"drop" is reserved for its literal meaning. When the intent is
additive, use "add," "wire up," "land," "integrate," or "register."
