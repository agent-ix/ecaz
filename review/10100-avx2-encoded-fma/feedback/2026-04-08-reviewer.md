# Review Feedback: AVX2 Encoded Scorer FMA

**Verdict: Approved**

## FMA application

Replacing `mul_ps + add_ps` with `fmadd_ps` in the 3-bit encoded scorer is the textbook
improvement: one fewer micro-op per iteration, one fewer register consumed for the intermediate
product, and slightly better precision from the fused rounding. The 8% improvement is in the
expected range for a latency-bound loop where one instruction is on the critical path.

The `target_feature(enable = "avx2,fma")` is already on the function — FMA is guaranteed available
in this codepath.

## Narrowing to encoded-only

Correct decision to leave lite scoring alone. The lite scorer does `codebook_permute(a) *
codebook_permute(b)` — both operands come from permutes, and the multiply feeds directly into the
accumulator add. The FMA fusion there is `permute * permute + acc`, which has the same dependency
chain as `mul(permute, permute); add(result, acc)` because the two permutes are independent.
On this microarchitecture the FMA regression on lite scoring suggests the scheduler was already
overlapping the separate mul+add effectively, and the fused version may have introduced a false
dependency. Either way, not touching it is right.

## Noisy first run

The first 20K-iteration pass at 919 ns versus the immediate rerun at 790 ns is a ~16% cold-start
delta — exactly what the warmup pass in 10101 is meant to fix. The review packet documents both
data points honestly. Good.

## Suggestion: non-3-bit FMA

The generic-width AVX2 path (the `else` branch) still uses `add_ps(acc, loadu_ps(...))`. This
doesn't have a multiply to fuse, but the QJL accumulation in that branch does:
`mul_ps(sq, sign_lanes)` then `add_ps(qjl_acc, ...)`. That could become an FMA. Low priority
since the generic path is rarely used at 4-bit, but if you're looking for micro-wins it's free.
