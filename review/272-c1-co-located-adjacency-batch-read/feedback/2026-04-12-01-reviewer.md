# Review: Packet 272 — C1 Co-Located Adjacency Batch Read

Reviewer: agent-ix (via code-review)
Date: 2026-04-12

## Assessment: Correct discard — the hypothesis was well-motivated but the warm buffer path is too cheap to optimize

### What this packet did right

1. **Targeted a real structural inefficiency.** Two separate `ReadBufferExtended` +
   `LockBuffer` + `UnlockReleaseBuffer` cycles for co-located tuples is genuinely
   redundant. The test `pg_test_build_keeps_element_neighbor_local` confirms that the
   co-located case is common, so the optimization had a valid premise.

2. **Clean discard.** Two valid runs effectively flat — first trims the tail slightly,
   second gives it back. No signal.

### Why it didn't help

On a warm shared buffer pool at 10K vectors, `ReadBufferExtended` for an already-pinned
page is essentially a hash table lookup + pin increment. `LockBuffer(BUFFER_LOCK_SHARE)`
on an uncontended shared lock is ~50-100ns. The combined cost of the second buffer read
for a co-located neighbor tuple is ~200-400ns.

At ~200-400 expansions per query, saving ~200ns per expansion saves ~40-80μs total. That's
**0.04-0.08ms** — well within measurement noise on an 11ms surface.

This is the same lesson as pgvector's approach: pgvector also does separate
`ReadBuffer` calls for element and neighbor tuples (`HnswLoadElementImpl` +
`HnswLoadNeighborTids`), and it's not a bottleneck for them either. The PG buffer manager
is fast for warm shared-buffer reads.

### What this rules out

Buffer manager call reduction is not a productive C1 seam at 10K warm. The per-call overhead
is too small. The remaining graph-loading cost is dominated by what happens *between* the
`LockBuffer` and `UnlockReleaseBuffer`: the `.to_vec()` copy, the decode allocations, and
the `Arc::new()` wrapper. Those are the bytes-and-allocations costs, not the lock costs.

## Suggestions

1. **Record and move on.** The discard is correct.

2. **Do not revisit buffer manager call reduction for warm C1.** This result, combined with
   packets 267-268 (executor overhead is negligible), closes the "PG infrastructure overhead"
   hypothesis. The remaining cost is in the Rust decode/allocation path, not in PG syscalls.
