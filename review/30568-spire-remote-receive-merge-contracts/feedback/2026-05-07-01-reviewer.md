# Remote receive/merge contracts review (closeout for 30556–30568 stack)

This is the most-recent packet in the Phase 7 coordinator-transport
groundwork stream. Per AGENTS.md, the cross-cutting summary lands
here. Localized findings are in the per-packet feedback files
30556–30567.

## 30568 — local findings

- **Tie-breaker label drift in `merge_input_summary`.**
  `remote_search_merge_input_summary_row` reports
  `tie_breaker: "score_then_primary_assignment_then_input_order"`
  (`root/remote_candidates.rs` near line 1235). The actual
  `remote_search_candidate_cmp` (line 5-22) tie-breaks on:

      score, then assignment role rank, then served_epoch DESC,
      then node_id ASC, then pid ASC, then object_version DESC,
      then row_index ASC, then row_locator ASC

  There is no "input_order" semantic in the comparator — the merge
  uses a `HashMap<vec_id, ...>` so input order is explicitly *not*
  preserved across same-vec_id duplicates. The label should describe
  the full ordering or be cut to the meaningful prefix
  (`"score_then_primary_then_epoch_desc_then_node_pid"` or similar).
  This is a contract-string / implementation drift, not a
  correctness bug — but the diagnostic stack is precisely where
  these claims should be accurate.

- **Result contract column count is correctly 9 and matches the
  storage-node endpoint** (`remote_search_libpq_result_contract_rows`
  at line 1076-1152). Each row's `validator` string names the
  in-process check from the 30559 receive validator. Good.

- **Receive plan validator linkage.** `remote_search_receive_plan_rows`
  (line 1154) sets `validator_function:
  SPIRE_REMOTE_SEARCH_RECEIVE_VALIDATOR` — that constant names the
  Rust function the libpq executor must call after row decode.
  Centralizing this name in one constant is the right pattern, and
  exactly the pattern I recommend extending to status strings.

- **Row locator policy.** `row_locator_policy =
  "opaque_origin_node_bytes"` is consistent with the 30553/30555
  documentation. Good.

## Cross-cutting summary — 30556–30568

### What's solid

- **Phase 7 receive boundary is fully validated before any candidate
  enters merge.** `validate_remote_search_candidate_batch` enforces
  epoch / node-id / selected-PID / object-version / visibility /
  vec_id / row_locator / score on every received row, and
  `merge_validated_remote_search_candidate_batches` validates every
  batch atomically before flattening. A buggy or hostile remote node
  cannot smuggle non-selected PIDs, mismatched epochs, or invisible
  rows past the boundary.
- **The fanout planner is pure Rust over a validated snapshot, with
  deterministic node-id and selected-PID ordering.** Same code path
  used by every diagnostic surface and by the executable
  coordinator-local endpoint, so drift between diagnostics and
  execution is structurally limited.
- **Coordinator-local execution path is real and fail-closed.**
  `ec_spire_remote_search_coordinator_local` and its summary
  exercise the same fanout planner, receive validator, and merge
  helper that the libpq path will use, and refuse to run when any
  remote target survives planning. That is the right "real
  end-to-end test of the v1 contract minus transport" shape.
- **Test coverage for fail-closed surfaces is now strong.** Strict
  unavailable, degraded skipped, mode mismatch, descriptor missing,
  and nonlocal-target rejection all have PG18 tests under
  `cargo test --lib remote_search`/`remote_node`, totaling well over
  20 tests in this stream.

### Cross-cutting findings (forward-looking, non-blocking)

1. **Status string proliferation — central registry needed.**
   Across 30558–30568 the SQL diagnostic surfaces use literal
   status strings (`ready`, `requires_libpq_transport`,
   `degraded_ready`, `degraded_skipped`, `empty_top_k`,
   `requires_remote_node_descriptor`, `local_ready`,
   `no_candidate_batches`, `missing_descriptor`,
   `local_direct`, `libpq_pipeline`, `opaque_origin_node_bytes`,
   `ec_spire_remote_search_v1`, etc.). The constants
   `SPIRE_REMOTE_SEARCH_RECEIVE_VALIDATOR` and
   `SPIRE_REMOTE_SEARCH_MERGE_FUNCTION` already exist — extend the
   pattern: lift every status/transport/format/policy literal into
   a single `pub(crate) const` block in `remote_candidates.rs` (or
   a new `remote_diagnostics_constants.rs`). Once the libpq
   executor lands, it will need to read and emit some of these
   strings, and a single source of truth prevents drift between
   diagnostic claims and runtime behavior. See per-packet
   findings on 30561 and 30566.

2. **`remote_candidates.rs` is now 1476 lines, ~75% diagnostic SQL
   row builders.** The production code (planner, validator, merge,
   merge-with-validation) is ~150 lines; the rest is
   `remote_*_rows` and `remote_*_summary_row` projections. Consider
   splitting the diagnostic SQL surface into
   `root/remote_diagnostics.rs` so the production module stays
   focused. The split is mechanical given the existing function
   boundaries.

3. **Misplaced rustdoc — global vec_id warning is now attached to
   `validate_remote_search_candidate_batch` instead of
   `merge_remote_search_candidates`.**
   See 30559 feedback for details. Cosmetic but produces wrong
   `cargo doc` output.

4. **Result-column-count and SQL-template drift risk.** The libpq
   request envelope (30567) hardcodes `9` for the result column
   count and `"SELECT served_epoch, node_id, ..."` for the SQL
   template, while 30568's `remote_search_libpq_result_contract_rows`
   maintains the actual column list. Derive the count from the
   contract rows so a future v2 schema bump cannot diverge.

5. **Diagnostic-summary count rollup duplication.** Every
   readiness/execution/merge summary recomputes ready/blocked/
   local/remote/skipped counts inline. A
   `SpireRemoteCountAggregator` helper would consolidate the
   rollup logic. See 30565 feedback.

6. **Diagnostic claim "tie_breaker" string disagrees with the
   comparator.** See 30568 local finding above. Fix the label or
   the comparator, but they should match exactly.

### Test coverage suggestions

- One combined-fixture test that drives a single plan with a local
  target, a remote-target needing descriptor, and a degraded-skipped
  placement at once, asserting the per-row status precedence and the
  summary count math (see 30563 feedback).
- A `tie_breaker` invariant test that asserts the diagnostic string
  matches a serialized form of the comparator field order — would
  catch finding (6) automatically.

### Verdict

No blockers for this stream to remain landed. The findings are
forward-looking — most should be addressed before the libpq pipeline
executor lands and starts emitting these same strings/contracts at
runtime, because that is the point at which any drift becomes a
real-system observability problem rather than a code-review concern.
