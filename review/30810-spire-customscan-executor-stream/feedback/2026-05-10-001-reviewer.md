# Review: SPIRE CustomScan Executor Stream (30810)

## Summary

Substantial slice. Closes the 30809 ORDER-BY-shape gap by validating
the sort target is `<rel-var> <op> <constant real[]>` before declaring
a candidate, plumbs the index OID + LIMIT into `custom_private` and
the copied query expression into `custom_exprs`, and routes
`ExecCustomScan` through PostgreSQL's `ExecScan` so qual / projection
machinery stays standard. The wiring genuinely reaches
`SpireRemoteFanoutExecutor` — the `#[should_panic]` test proves the
call lands at the production blocker rather than the scaffold "not
wired" error.

The four review-focus questions:
- State allocation/drop pattern: structurally correct but has a
  cleanup-on-error gap (P1 below).
- planner/executor contract: works; minor shape nits (P3).
- `ExecScan` integration + guarded callbacks: correct; one rescan/recheck
  semantics question (P2).
- Parameterized queries / remote tuple-payload deferral: explicit,
  documented in `next_step` and the executor error.

## Findings

### P1 — `EndCustomScan` cleanup may not run on executor error

`SpireCustomScanExecState` contains owned `Vec<f32>` and
`Vec<SpireRemoteProductionScanOutputRow>` fields. The state is allocated
via `palloc0` + `ptr::write` (correct for Rust-typed members), and
freed via `ptr::drop_in_place` + `pfree` in `ec_spire_end_custom_scan`.

PostgreSQL does not guarantee `EndCustomScan` is called when the query
errors out (e.g., the `pgrx::error!` in `custom_scan_ensure_outputs`
for a non-`SPIRE_REMOTE_NONE` blocker, or a remote transport failure
mid-scan, or backend cancellation). On error, the surrounding
ExecutorState memory context is reset, which `pfree`s the state's
palloc'd region — but it does **not** call `drop_in_place` on the Rust
fields, so the heap-allocated `Vec` buffers leak through Rust's
allocator.

For a single error this leak is small. For a backend that runs many
CustomScan queries that fail (e.g., during Stage E fault-matrix
fixtures), the leak accumulates per-query.

Pattern fix: register a `MemoryContextCallback` against the executor
state's per-tuple or per-query memory context that runs
`drop_in_place(state)` on context reset. This is the standard pgrx
idiom for "Rust-owned Vec inside palloc'd PG state." See e.g.
`pgrx::memcxt::PgMemoryContexts::register_reset_callback` patterns.

### P1 — Snapshot visibility check on `table_tuple_fetch_row_version` may under-deliver

`ec_spire_custom_scan_access` resolves each output's heap TID against
`(*estate).es_snapshot` via `table_tuple_fetch_row_version` and
silently skips invisible rows in the loop. The production executor
stream is supposed to have already resolved heap visibility on the
origin node. For local outputs (`SPIRE_REMOTE_LOCAL_HEAP_RESOLUTION`)
the coordinator's snapshot is the authority, so re-checking is
correct. For materialized-remote outputs
(`SPIRE_REMOTE_MATERIALIZED_HEAP_RESOLUTION`), the coordinator's
snapshot may have moved relative to when the materialized row landed,
and silently dropping these means the user gets fewer than `top_k`
rows with no diagnostic.

Two concerns:
1. No counter / log line indicates rows were dropped at this
   visibility check. Operators see "fewer rows than expected" with no
   trail.
2. Once we drop below `top_k` we do not request more from the producer
   stream — `outputs` is loaded once at first call and never refilled.
   If half the outputs are silently invisible, `LIMIT 10` returns 5.

For v1 with materialized-only delivery this is probably fine because
the materialization mirror is short-lived. But once remote tuple-payload
slots land and the visibility surface widens, this loop needs to either
loop back into the producer or surface the drop count.

### P2 — Recheck callback always returns `true`

`ec_spire_custom_scan_recheck` unconditionally returns `true`. For
v1 with no quals beyond the LIMIT this is fine, but in EvalPlanQual
re-evaluation under SERIALIZABLE / UPDATE-on-vector-table scenarios,
this means any recomputed tuple is accepted regardless of the original
ORDER BY filter. Since CustomScan output has no Postgres-level qual
to re-check, returning `true` is the conventional default — but it's
worth a one-line comment noting that the recheck is a no-op because
the CustomScan output is already filtered/ordered by
SpireRemoteFanoutExecutor, not by Postgres expressions.

### P2 — `ec_spire_custom_scan_rescan` clears outputs but Begin is not re-called

Rescan correctly clears `outputs`, `next_output`, and
`loaded_outputs`. But `index_oid`, `top_k`, and `query` are not reset
to zero / empty between rescans — they remain from the original
Begin. That is the intended behavior (rescan reruns the same query
with the same parameters), and matches how index AMs handle
`amrescan` of a parameterized scan. But because parameterized
queries are explicitly rejected by the planner shape check, this
slice's rescan path is only exercised when the same backend runs the
same plan multiple times (e.g., inside a join's inner loop), not when
parameters change. Worth a comment that re-execution under bound
parameters will need a Begin-level invalidation when that lands.

### P2 — Producer stream error surfaces as `pgrx::error!` mid-`ExecScan`

When `stream.summary.next_blocker != SPIRE_REMOTE_NONE`,
`custom_scan_ensure_outputs` calls `pgrx::error!`. This longjmps out
of `ExecScan`, which then unwinds to the surrounding executor. The
error message names `status`, `next_blocker`, and `recommendation` —
useful diagnostics. The concern is the cleanup-on-error question
above (P1): the `ExecState` palloc'd region is freed by the executor
context reset, but the Rust heap allocations leak.

Combined with P1 fix, this becomes safe.

### P3 — `top_k` round-trip through `Oid` is misleading

`top_k` is encoded into `custom_private` as `Oid::from(u32)` and
read back via `pg_sys::list_nth_oid(...).to_u32()`. This works
because `Oid` is structurally a `u32`, but it conflates the OID type
with an unrelated counter. PostgreSQL has `lappend_int` /
`list_nth_int` (or `lappend` with `makeInteger`) for exactly this
case. Cosmetic but reduces "wait, is that an OID?" head-scratching
for the next reader.

### P3 — Planner-time cache for eligibility now matters

The planner-path slice (30809) called eligibility on every base
relation. With the executor wired, that gate stays load-bearing —
worth pinning the eligibility-cache work suggested in the 30809 review
before this becomes hot.

### P3 — `varno` comparison breadth

`u32::try_from(var.varno).ok() == Some(relid)` correctly excludes
negative special varnos. Note that `INNER_VAR`, `OUTER_VAR`, and
`INDEX_VAR` are large positive constants (65000+) that fit in u32 but
do not match any real `relid`; the equality check filters them out
naturally. Comment-worthy if not already noted somewhere.

## Reviewer recommendation

Accept the wiring shape — this is a clear step forward and the
fail-closed boundaries are honest. The P1 items are required before
this becomes a long-running production path: (1) memory-context
callback for Vec drop, and (2) decide what to do when post-resolution
visibility check drops rows below the requested top-k. Both can land
in the same follow-up packet ahead of the tuple-payload slot work.
