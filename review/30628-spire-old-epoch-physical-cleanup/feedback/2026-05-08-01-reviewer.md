# Old-epoch physical cleanup review

`ec_spire_index_epoch_cleanup_run(index_oid)` adds the first
real destructive operation in SPIRE: physical reclamation of
old-epoch object tuples. This is the most safety-critical
slice in the Phase 8 work. **No blocking findings, but several
correctness checks worth recording.**

## Correctness review

I walked through the cleanup pipeline carefully because this
is destructive. Here is what I checked and what holds.

### Lock discipline

- Takes `lock_publish_relation` (the SPIRE publish lock,
  `snapshots.rs:713`). Same lock that publish/persist use, so
  cleanup serializes against publish operations.
- Per-storage-relation `RowExclusiveLock` for non-index
  storage relations (`snapshots.rs:756`). Correct level for
  DELETE-style operations.
- Matched `relation_close` with the same lock mode (line 772).

### WAL and crash safety

- `delete_object_tuples_no_compact` (`page.rs:316-411`) uses
  `wal::GenericXLogTxn` with `GENERIC_XLOG_FULL_IMAGE`. Crash
  safe — replay reproduces the page state.
- Critically, `PageIndexTupleDeleteNoCompact` operates on the
  WAL-registered scratch buffer. Drop without `finish()` (i.e.
  on error) discards changes — verified the error paths
  (`snapshots.rs:362-369, 372-376, 384-389`) all `drop(wal_txn)`
  before `UnlockReleaseBuffer`, so partial in-memory mutations
  cannot leak to other backends.
- Buffer lock is exclusive while modifying. Other backends
  cannot read the in-progress page.

### TID stability

- Uses `PageIndexTupleDeleteNoCompact`, not the compacting
  variant. **This is the critical choice for SPIRE**: TIDs of
  remaining tuples (active and retained-epoch object tuples)
  must remain stable, otherwise concurrent scans following
  manifest pointers would dereference moved tuples.
  Verified.
- Free space recorded via `RecordPageWithFreeSpace` so the FSM
  picks up the freed bytes for future inserts. Reuse without
  TID drift.

### Validation guards

- Line 322: refuses to delete from blocks below
  `FIRST_DATA_BLOCK_NUMBER` — protects metadata blocks
  (root_control, manifests).
- Line 362: bounds check against `max_offset`.
- Line 371: null item-id check.
- Line 384: tuple bounds check against page size.
- Line 379: skips already-dead line pointers (`lp_flags == 0`).

### Cleanup eligibility

- `collect_physical_cleanup_candidates` filters by retention
  window AND active-query state (the same eligibility helper
  used by the existing summary surface). So a query holding a
  long-lived snapshot on epoch N prevents N from being a
  cleanup candidate.
- The `protected` set explicitly excludes any tuple chain
  reachable from active or retained epoch manifests — verified
  via the call site, this includes objects, placement entries,
  and manifest tuples at all retained epochs.

### Status taxonomy

- `not_required` (active_epoch == 0)
- `blocked_by_retention` (eligibility filter found nothing)
- `reclaimed` (removed_count > 0)
- `no_candidates` (eligible epochs but no actual tuples to
  remove)

The four states cover the realistic combinations.

## Findings

### Non-blocking

- **`physical_cleanup_supported` hardcoded to `true`** in
  `lib.rs:9229` rather than reading from the result. Today
  cleanup IS supported, so the constant is correct. But
  `SpireIndexEpochCleanupRunResult` doesn't carry a
  `physical_cleanup_supported` field — it's implied by the
  function existing. Future-proofing: if a build configuration
  ever disables cleanup, the SQL surface should report
  `false`. Cosmetic.

- **Cleanup runs hold the publish lock for the full duration.**
  For an index with many cleanup-eligible epochs and many
  storage relations, this could create a measurable pause
  window for concurrent inserts. Acceptable for v1 (operator-
  driven cron, presumably during low-traffic windows). Worth
  documenting in the docs slice (30627) that cleanup is a
  publish-lock-holding operation.

- **Test coverage is thin for failure injection.** PG18 tests
  cover the happy path (cleanup removes tuples + post-cleanup
  scan still works) and uses `debug_age_retired_manifests` to
  artificially make epochs eligible. Missing: a test that
  injects an active-query simulation and asserts cleanup
  refuses to remove that epoch's tuples. The eligibility
  helper is exercised elsewhere, but a focused cleanup-with-
  active-query test would harden the contract.

- **`removed_tuple_bytes` measures lp_len bytes, not full
  on-disk impact.** After deletion the line pointer remains
  (just in LP_DEAD-style state). The measurement is "freed
  payload bytes," not "freed total bytes." Probably the right
  thing for capacity planning, but the name could mislead an
  operator into thinking they're getting line-pointer bytes
  back too. Cosmetic.

### Suggestions for follow-up

- **Cleanup throughput / observability.** A
  `cleanup_duration_micros` field would help operators decide
  cron cadence based on actual cleanup cost.
- **Per-epoch result rows.** Currently the run returns a
  single summary row. For an operator triaging a cleanup that
  freed less than expected, per-epoch breakdowns would help.
  Could be a follow-up `ec_spire_index_epoch_cleanup_per_epoch`
  surface.

## Confirmations

- This is the right design: explicit operator-driven cleanup
  rather than implicit on-publish reclamation. Operators get
  a controlled pause rather than surprise long publishes.
- Symmetric with the maintenance scheduler — both run via the
  `*_run` verb, both honor the publish lock, both report
  detailed result rows.
- Phase 8 old-epoch reclamation item correctly marked
  complete.
