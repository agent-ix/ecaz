## Response to Review Focus

### Is the ordered-result regression test the right durable A4 coverage?

**Yes.** `test_tqhnsw_graph_first_scan_emits_distance_sorted_scores` checks monotonic score ordering, no duplicate heap TIDs, and that every emitted TID maps back to a table row. It's cheap (8 rows, 4-dim), deterministic, and independent of the recall gate threshold. This is the correct persistent test to keep on `main` — it guards ordering invariants without coupling to the expensive 10K corpus.

One limitation: the tiny corpus won't catch traversal bugs that only manifest at scale. That's acceptable — the recall gate test covers scale, and the regression test covers invariants.

### Do the report/probe surfaces cleanly demonstrate the failure is in graph traversal?

**Yes.** The probe function (`build_graph_scan_recall_probe`) returns three orderings side-by-side — fp32 brute-force truth, graph-first predicted top-10, and exact quantized `<#>` top-10 — which directly isolates where divergence occurs. The request notes that the exact quantized scorer still overlaps meaningfully with truth while graph-first diverges sharply from both, which pins the failure to traversal/runtime rather than scorer accuracy.

The report function (`run_graph_scan_recall_gate`) cleanly sweeps across four `(m, ef_search)` configs with one gated threshold. The env-var gating (`TQVECTOR_RUN_RECALL_GATE`) for the expensive test is appropriate.

### Do the plan/status updates accurately describe the gate as failing?

**Yes.** Plan, status, and task file all consistently mark A4 as "in progress — failing" with the specific 1.7% number, and downstream work (A5, A6, SIMD merge, D2) stays blocked. The investigation notes in `05-graph-scan.md` correctly record the UNLOGGED table correction and the conclusion that failure is in traversal, not the scorer.

1. **Minor duplication in scan_debug.rs.** `debug_gettuple_scan_heap_tids_with_scores` (lines 477-503) is near-identical to `debug_gettuple_scan_heap_tids` (lines 449-473) with one added score field. A shared inner function could reduce this. Low priority since it's test-only code.
