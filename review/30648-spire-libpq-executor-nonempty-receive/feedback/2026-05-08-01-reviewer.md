## Finding F2 (non-blocking) — `raw_conninfo_exposed` column is structurally dead in 38fae3e9

`ec_spire_remote_epoch_manifest_libpq_executor_results` returns
a `raw_conninfo_exposed: bool` column. In all six early-return
tuples and the success-path return, this position is the
literal `false`. There is no code path that can set it to
`true`. The matching loopback assertion
(`assert!(!raw_conninfo_exposed)`) is therefore always-true.

**Why:** The contract assertion is real (the executor never
serializes the resolved conninfo back to the caller), but
encoding a *runtime invariant* as a *runtime column whose
value is constant* gives operators a false signal — a `false`
return is indistinguishable from "the field was never wired
up." Two acceptable resolutions:

- Drop the column. The invariant is enforceable by code review
  and a static-string lint, not by runtime read-back.
- Or keep the column but make it carry a different signal —
  e.g. `conninfo_lookup_kind` ∈ `{secret_provider, env_inline,
  not_attempted}` — so it actually distinguishes states.

Same observation applies to the search-side
`ec_spire_remote_search_libpq_executor_connection_check`
return (already shipped in 30646), so a fix should consider
both surfaces together.

## Finding F3 (non-blocking) — Inline 10-tuple early-returns in 38fae3e9 will not scale

The manifest executor has six inline-literal early returns
shaped `(active_epoch, node_id, false,
"blocked_before_connection", 0_i64, …, "manifest_dispatch",
"blocked", "resolve manifest publication blockers …")`. With
ten fields per tuple and increasing branch coverage as more
failure modes are added, this is the kind of code that
silently loses a column-position invariant during a refactor.

**How to apply:** Introduce a small `ManifestExecutorResultRow`
struct (or a builder with named setters) inside the function
and have each branch construct one explicitly. The TableIterator
then converts struct → tuple at the bottom in a single place.
This is the same shape the rest of the codebase uses for
multi-column SQL returns and removes the column-position risk.
