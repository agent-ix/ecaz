# Review: SPIRE Remote PK Read Isolation (30937)

## Summary

Closes 30896 H5 — the EvalPlanQual / isolation decision I
flagged. The packet:

- Adds `test_ec_spire_remote_pk_select_isolation_contract_sql`,
  which actually drives the question: a coordinator transaction
  reads a remote-owned row, a concurrent backend updates it,
  the same coordinator transaction reads again, and asserts
  the second read **observes the new value** even at
  `REPEATABLE READ` and `SERIALIZABLE`.
- Fixture confirms the plan is `Custom Scan
  (EcSpireDistributedScan)` (not a local stand-in).
- **Decision pinned**: `ec_spire_custom_scan_recheck` stays
  unconditionally `true` for v1, with a code comment tying
  that choice to the virtual-tuple-without-coordinator-heap-
  identity reality.
- ADR-068 + ADR-069 document the v1 isolation contract:
  remote CustomScan payloads do not carry coordinator heap row
  identity, remote statements do not inherit the coordinator
  transaction snapshot, and EvalPlanQual cannot rerun against
  the remote origin row.

The three review-focus questions:
- Fixture exercises the real remote PK-read CustomScan path:
  yes — loopback coordinator + remote shard table, descriptor
  registered, EXPLAIN-asserted CustomScan plan.
- Expected v1 behavior is acceptable (`REPEATABLE READ` /
  `SERIALIZABLE` can observe newer remote rows): yes — see
  below for the operator-facing call.
- ADR-068/069 state the limitation clearly without overstating
  broader distributed-query guarantees: yes.

## Findings

### P2 — Honest limitation, but operationally significant

The decision is correct given the architecture (remote
statements run in their own remote PG snapshots, distinct from
the coordinator's), but the operator-visible effect is real:

- An application that runs `BEGIN ISOLATION LEVEL
  SERIALIZABLE; SELECT WHERE id=$1; ... SELECT WHERE id=$1;
  COMMIT;` against a distributed table can see two different
  values for the same row in one transaction.
- This violates PG's normal SERIALIZABLE / REPEATABLE READ
  guarantees for *local* tables. Operators and application
  developers used to PG's stricter guarantees may be
  surprised.

The ADR docs name this clearly. Worth pinning in the operator
runbook (`docs/SPIRE_DIAGNOSTICS.md`) that:
- Distributed table reads provide **read-committed semantics
  only** regardless of the surrounding transaction's isolation
  level.
- Applications that need PG's normal SERIALIZABLE guarantees
  for distributed tables must implement application-level
  locking or accept the v1 limitation.

If this isn't already in the diagnostics doc, worth a
follow-up packet.

### P2 — `recheck` callback unconditional `true`

For v1 with no qual-recheck semantics beyond the PK
predicate, this is correct. EvalPlanQual rerun against a
remote-origin tuple would require either:
- Re-running the primitive against the coordinator's snapshot
  (impossible — remote PG owns its own snapshot).
- Inheriting the coordinator's snapshot to the remote (ADR
  out-of-scope work).

Pinning the choice + adding the code comment + documenting in
the ADRs is the right v1 disposition.

### P3 — Future ADR hook

If SERIALIZABLE distributed reads ever matter (cross-shard
financial transactions, etc.), the future ADR would need to
either:
- Push the coordinator's snapshot to the remote via
  `BEGIN ISOLATION LEVEL ... USING <snapshot>` (PG supports
  exporting/importing snapshots, but only between local
  backends).
- Re-execute the read against a coordinator-side cache that
  participates in the coordinator's MVCC.

Both are substantial scope. v1 deferral is honest.

### P3 — Fixture also covers `READ COMMITTED`

The fixture exercises all three isolation levels in one PG18
test. Good coverage; the `READ COMMITTED` behavior matches
PG's normal contract (each statement sees latest committed),
so the contract is consistent across local and distributed
tables for that level.

## Reviewer recommendation

Accept. **Closes H5 honestly** — the v1 limitation is named,
the recheck callback's reasoning is documented in code, and
the fixture proves the actual behavior. The follow-up runbook
entry on read-committed-only distributed reads is worth a
small docs slice before AWS.
