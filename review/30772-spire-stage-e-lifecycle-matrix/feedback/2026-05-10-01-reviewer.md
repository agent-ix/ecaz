## Answers to the packet's review questions

- **DROP / REINDEX CONCURRENTLY / CREATE INDEX
  CONCURRENTLY split at the right timing boundaries?** Yes.
  The before/in-flight split for both DROP and REINDEX is
  the correct cut — different in-flight remote operations
  trigger different detection paths (the executor either
  sees a missing remote index, or a fingerprint mismatch on
  return). Splitting them lets fixtures exercise each path
  separately rather than racing.
- **Strict/degraded actions and statuses correct,
  especially deferring new CIC descriptors for already-
  planned fanout?** Yes. Deferring new descriptors is the
  right behavior — including a freshly-registered
  descriptor in an already-planned fanout would change the
  query's PID set mid-flight, which violates the snapshot
  semantics the coordinator promised the caller. Strict and
  degraded behaviors otherwise match the corresponding
  fault-matrix rows (DROP-after-fanout → fail-closed or
  skip-node based on degraded selection; REINDEX-in-flight
  → endpoint_identity_mismatch on return).
- **`remote_index_unavailable`, `endpoint_identity_mismatch`,
  and `requires_remote_node_descriptor` the right required
  detections?** Yes for all three:
  - `remote_index_unavailable` from packet 30732 for
    missing remote regclass (DROP cases);
  - `endpoint_identity_mismatch` from packet 30710 for
    fingerprint divergence (REINDEX-in-flight case);
  - `requires_remote_node_descriptor` from the existing
    capability gate (missing-descriptor case).
  Each detection has an existing production code path; the
  matrix uses them rather than inventing new categories.
- **Phase 11 task update scoped honestly?** Yes —
  "definition" vs "fixture logs still open" is the right
  language for this slice.
