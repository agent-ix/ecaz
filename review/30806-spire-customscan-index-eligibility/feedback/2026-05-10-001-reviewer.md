# Review: SPIRE CustomScan Index Eligibility Surface (30806)

## Summary

Accept. The eligibility surface uses the right source of truth
(`load_relation_epoch_manifests_for_coordinator_fanout` + the placement
directory), correctly bypasses
`ec_spire_index_placement_snapshot(...)` which is still gated on the
superseded AM-local materialization path, and the four-status taxonomy
(`no_active_epoch`, `local_only`, `no_available_remote_placements`,
`customscan_candidate`) cleanly reflects the planner-gate cases.

The three reviewer-focus questions answer yes:
- Right source of truth.
- Bypassing the materialization-gated snapshot is appropriate under
  ADR-067.
- `customscan_candidate` is correctly limited to indexes with an active
  epoch and at least one available remote placement.

## Findings

### P2 â€” Loads epoch + object manifests just to read placement directory

`custom_scan_index_eligibility_row` calls
`load_relation_epoch_manifests_for_coordinator_fanout(...)` and then
discards `_epoch_manifest` and `_object_manifest`, using only
`placement_directory`. This is a diagnostic surface that operators
will run interactively and that the planner-path packet will likely
call on every plan. Loading both manifests just to discard them
imposes I/O for no benefit.

Suggestion: extract a narrower helper that loads only the placement
directory for the active epoch, and have the existing fanout loader
call into that helper. Same source of truth, lighter cost path.

Not a correctness blocker â€” just don't want this becoming a hot-path
overhead once the planner integration lands.

### P2 â€” Eligibility threshold may be too permissive for the planner gate

`eligible_for_custom_scan = remote_available_placement_count > 0`
returns true when even one remote placement is `Available`, regardless
of how many are `Stale`, `Unavailable`, or `Skipped`. The diagnostic
status `no_available_remote_placements` only fires when *all* remote
placements are non-`Available`.

For a diagnostic this is fine. For the planner gate that the next
packet will build on, this is probably too loose: a CustomScan plan
that fans out to a centroid space where most leaves are unreachable
will silently return degraded results (in degraded mode) or fail
strict mode at execution time. Consider exposing a finer signal:
- `eligible_for_custom_scan` stays as "at least one available" (covers
  the basic local-vs-distributed dispatch);
- add a separate `all_remote_placements_available: bool` or a
  `remote_unavailable_placement_count` counter so the planner packet
  can decide policy without re-deriving it.

### P3 â€” `remote_node_count` semantics

`remote_node_count` counts the unique `node_id`s across all remote
placements regardless of state. That matches "remotes that have ever
held a placement on this index" but not "remotes the planner can fan
out to right now." Consider renaming to `remote_node_seen_count` or
adding `remote_available_node_count` for clarity. Cosmetic.

### P3 â€” SQL test only covers two of the four status branches

The `pg_test` covers `local_only` â†’ `customscan_candidate` after
rewriting one PID. Missing branches:
- `no_active_epoch` (an index immediately after CREATE before first
  build / publish);
- `no_available_remote_placements` (an index where all remote
  placements are `Stale` or `Unavailable`).

The third branch is hard to construct in a focused fixture, but
`no_active_epoch` should be a one-line addition: assert eligibility
status against an `ec_spire` index before any data is built into it.
Worth adding before the planner-path packet starts depending on these
status strings.

### P3 â€” Comment on bypass rationale

The body of `custom_scan_index_eligibility_row` does not document why
it deliberately avoids `ec_spire_index_placement_snapshot(...)`. That
context lives in the request packet but the next contributor reading
the code will not have it. Consider a short comment above the
`load_relation_epoch_manifests_for_coordinator_fanout` call pointing at
ADR-067 and noting that the snapshot helper is intentionally bypassed
because it errors when remote placements are present (Shape-A gating).

## Reviewer recommendation

Accept and proceed. The P2 items (lighter loader, finer eligibility
signal) should be folded into the planner-path packet rather than
re-spinning this one. The P3 items are quick polish for the next code
touch on the file.
