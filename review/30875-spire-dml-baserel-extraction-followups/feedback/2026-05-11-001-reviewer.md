# Pre-Checkpoint Review: SPIRE DML Baserel Extraction Hardening (30875)

## Note

Code committed in `6efe8f84 Harden SPIRE DML baserel extraction
errors` but no `request.md` exists yet. Pre-checkpoint feedback
per `review/REVIEWER.md`.

## Summary

Closes three of the four 30874 P2/P3 items in one slice:

- **`i32::try_from(rel.relid)` overflow** now returns
  `Some(Err(...))` with a typed message naming the failure
  ("relid exceeds planner rtindex range") instead of silently
  returning `None`.
- **`Some(Err(_))` no longer silently skips** in the candidate
  extractor — `custom_scan.rs` now propagates the error via
  `pgrx::error!("{err}")`, so malformed candidates surface
  loudly at planner time instead of getting dropped to a
  pass-through plan.
- **Two-walker comment** added: explains why
  `dml_frontdoor_pk_predicate_from_baserestrictinfo` (hook view)
  and `dml_frontdoor_pk_predicate` (analyzed-Query view)
  coexist — `set_rel_pathlist_hook` sees planner-normalized
  RestrictInfo while diagnostic SQL helpers operate on a
  freshly analyzed Query.

Bonus structural cleanup:
- Mode/supported short-circuit moved before `pk_value_expr`
  unwrapping, so unsupported shapes return early without
  triggering a "missing PK expression" error for queries that
  legitimately don't have a PK predicate.
- `has_join` computed once and reused in both `single_table` and
  `has_join` shape inputs.

## Findings

### P3 — `pgrx::error!` propagation in the planner hook

`custom_scan.rs` now calls `pgrx::error!("{err}")` from inside
`set_rel_pathlist_hook` when the extraction returns an error.
That's the right shape — matches the 30861 fail-closed guard
pattern. Worth a one-line comment in `dml_pk_select_candidate_index_oid`
naming "errors here become planner errors via pgrx::error!" so
a future reader doesn't try to convert these back to
`Option::None` for "graceful degradation."

### P3 — Mode short-circuit before pk_value_expr

The reorder is correct: a query that doesn't classify as
`coordinator_pk_select_tuple_payload` shouldn't error on missing
pk_value_expr; it should just return None. The new shape:

```rust
let decision = ...;
if !decision.supported || decision.custom_scan_mode != "coordinator_pk_select_tuple_payload" {
    return None;
}
let pk_value_expr = pk_value_expr.ok_or_else(...)?;
```

Reads cleanly. Mode check is the cheap first filter; expression
extraction only runs when we know it's needed.

### P3 — `i32::try_from` error message

The error names "planner rtindex range" which is informative
for a PG developer but might not parse well for an end-user
operator. Worth confirming this error is intentionally
internal-only (i32 overflow on `rel.relid` essentially means
"the planner gave us a >2-billion-relid which shouldn't happen
in practice"). If so, document as an internal invariant
violation; if it can surface to operators, soften the wording.

## Reviewer recommendation

Accept. Tight follow-up packet that closes the 30874 P2/P3 items
cleanly. The packet should land its request.md so the small
structural shifts (mode short-circuit, has_join refactor) get
their own validation receipt.
