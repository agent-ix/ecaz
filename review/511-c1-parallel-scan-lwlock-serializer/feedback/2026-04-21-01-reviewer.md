## Findings

### 1. Release guard: gated on `InterruptHoldoffCount > 0` — subtle

`release_parallel_scan_lwlock` at parallel.rs:1464–1467:

```rust
#[cfg(not(test))]
if unsafe { pg_sys::InterruptHoldoffCount } > 0 {
    unsafe { pg_sys::LWLockRelease(lock) };
}
```

This is defensible as a guard against double-release after ereport
unwind (PG's `LWLockReleaseAll` during abort resets the count to
zero, so a subsequent Drop would skip release). But it is **not**
the standard idiom — most extension code releases unconditionally
because `LWLockAcquire` bumps the count and normal Drop-order
release decrements it.

Concrete failure mode: if something bumps/decrements holdoff count
independently between acquire and drop (unlikely but possible via
PG_TRY/PG_CATCH or future nested LWLock work), this guard could
**silently skip a legitimate release**, leaking the lock. A leak
here is far worse than a double-release.

Recommendation: either (a) add a comment documenting this as an
ereport-unwind defense and anchor it to `LWLockReleaseAll` behavior,
or (b) drop the guard and rely on `LWLockReleaseAll` to have
already cleared the lock from the held list (the idiomatic
approach). Non-blocking, but worth a follow-up before
`amcanparallel = true`.

### 2. Test shim does not prove runtime lock semantics

The `#[cfg(test)]` path uses a spinloop CAS on the embedded
`state.value` and skips `LWLockAcquire`/`LWLockRelease` entirely.
That's the right choice for unit tests (avoids pgrx's cross-thread
FFI guard), but it means unit tests do not exercise:
- wait-queue semantics
- `CHECK_FOR_INTERRUPTS` inside acquire
- `LWLockReleaseAll` interaction on ereport

Those are the exact behaviors that matter for `amcanparallel =
true`. Worth an integration-level test at PG level before the
enablement flip.

### 3. Tranche re-registration is idempotent

`register_parallel_scan_lwlock_tranche` (parallel.rs:1413–1428)
early-returns if the stored id already matches. Attach-time
re-register is safe to call repeatedly from multiple backends.
Good.
