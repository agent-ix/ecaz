# Review Feedback: AVX2 FWHT 32-Lane Bootstrap

**Verdict: Approved with suggestions**

## Compositional structure

Same pattern as 16-lane: `fwht32_avx2_block` calls `fwht16_avx2_block` on both halves, then
butterflies between them. Four __m256 registers in, four out. The recursive decomposition is
correct.

## 10.9% improvement over 16-lane

Stronger than the 16→16 step (5.2%) because this eliminates two outer-loop passes (width=8 and
width=16 are now both inside the bootstrap). At dim=2048, this is meaningful.

## Current standings vs targets

Cumulative SIMD batch position:

| Metric | Baseline (pre-SIMD) | Current (this checkpoint) | Target |
|---|---|---|---|
| `score_ip_encoded/d1536_b4` | ~17,000 ns | ~790 ns | BC-010: >200K/s (~5000 ns) |
| `score_ip_codes_lite/d1536_b4` | ~12,800 ns | ~1,380 ns | — |
| `fwht/2048` | ~4,600 ns | ~1,237 ns | BC-008: ≥3x (~1,537 ns) |
| `fwht/4096` | ~9,600 ns | ~2,786 ns | — |

BC-010 is passing comfortably (1.26M scores/sec vs 200K target). BC-008 is at 2.73x vs 3.0x
target — 10% short.

## Register pressure and complexity ceiling

At 32 lanes, the bootstrap block takes 4 × __m256 inputs and produces 4 × __m256 outputs. Inside
the call tree, the deepest nesting (fwht8_avx2_block) uses 3 permute-index registers plus the
working value — roughly 6-7 __m256 registers per level. With 16 YMM registers available on x86_64,
the 32-lane block is approaching the limit where the compiler can keep everything in registers
without spilling. Going to 64-lane would double the input/output count to 8 × __m256, which is
exactly half the register file — feasible but tight.

## Suggestions for the next FWHT push

Given BC-008 is 10% short:

**Option A — 64-lane bootstrap (simplest, moderate win).** Mechanical extension of the same
pattern. Eliminates the width=32 outer pass. Expected to get another ~5-8% given the pattern from
16→32. This alone might close BC-008 if the scalar baseline cooperates, but it's tight. The risk
is register spilling — profile the generated assembly to check.

**Option B — Streaming store for outer stages.** The outer loop (width ≥ 32) reads and writes the
full array per stage. At dim=2048 (8 KiB), the array fits in L1 (typically 32-48 KiB), so cache
misses aren't the bottleneck — store-to-load forwarding latency is. Using `_mm256_stream_ps`
for the writes (bypassing cache, writing directly to memory) would be *worse* here because L1
is warm. But if you ever bench at larger dims (4096+ → 16+ KiB), this becomes relevant.
**Probably not useful at dim=2048.**

**Option C — Dual-pass tiling (highest ceiling, most complexity).** Instead of N outer passes
over the full array, process pairs of 32-element blocks and apply all remaining stages that
fit within a tile before moving on. For dim=2048 with 32-element blocks:
- First pass: apply bootstrap-32 to each block (already done)
- Second pass: for each adjacent pair of 32-element blocks, apply width=32 butterfly, then
  continue up to width=64 within the same 64-element working set. Then for adjacent pairs of
  64-element blocks, apply width=64 butterfly and continue to width=128. Etc.

This is essentially a cache-oblivious FWHT. It reduces the number of full-array passes from 6
(width=32 through 1024) to ~2-3 tiled passes. This would close BC-008 with margin but is
significantly more complex to implement correctly.

**My recommendation:** Try Option A first (64-lane bootstrap). If that gets to ~2.9x, the
combination of that plus pinned CPU frequency on better hardware will likely close BC-008. If it's
still short, skip to Option C for a definitive fix.

## One structural observation

The `fwht_in_place_avx2` dispatch now has a 3-way `if/else if/else` for input size, plus a
`if/else if/else` for starting width. These duplicate the same thresholds. Consider collapsing
to a single match on a bootstrap level:

```rust
let bootstrap_width = if values.len() < 8 { 0 } else if values.len() < 16 { 8 } else if values.len() < 32 { 16 } else { 32 };
```

Then dispatch bootstrap by `bootstrap_width`, and start the outer loop at `width = bootstrap_width`.
Not blocking, but it would make adding 64-lane cleaner.
