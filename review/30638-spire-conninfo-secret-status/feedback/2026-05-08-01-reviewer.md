# Conninfo secret status surface review

**First executable secret-provider lookup.**
`ec_spire_remote_conninfo_secret_resolution_status(conninfo_secret_name)`
maps a secret reference to env var
`EC_SPIRE_REMOTE_CONNINFO_<UPPERCASED+SANITIZED>`. v1 provider
is environment variables. The raw conninfo is read internally
to determine length but never returned through SQL. No blocking
findings on the security model.

## Security review

I walked the function carefully because this is the first SPIRE
surface that touches a real secret.

### What's safe

1. **No raw conninfo in the SQL return type.** The row exposes
   `provider_lookup_key`, `resolved_conninfo_bytes` (length),
   `raw_conninfo_exposed: false`, status, and recommendation.
   The actual conninfo never crosses the SQL boundary.
2. **Env var read at call time.** No caching. If the operator
   rotates the secret (`pg_reload_conf` style), the next call
   sees the new value.
3. **Env var key derivation is deterministic and visible.**
   `EC_SPIRE_REMOTE_CONNINFO_` prefix + uppercased reference
   with non-alphanumeric → `_`. Operators can verify their env
   config matches the lookup key.
4. **Tristate status:** `resolved`, `requires_secret`
   (configured but empty), `requires_secret` (env not set).
   Clear failure signal.
5. **PG test asserts no-raw-conninfo invariant** — closes the
   contract claim.

### What to watch

- **Env-var-derived secret in process memory.** Once
  `std::env::var(&key)` succeeds, the conninfo lives in the PG
  backend's heap during the function call. A core dump would
  expose it. Same risk class as any secret in PG memory; no
  worse than `postgres_fdw` user-mapping passwords. Worth an
  operator doc note.
- **Lookup-key collisions.** A `conninfo_secret_name` of
  `node-1` produces lookup key
  `EC_SPIRE_REMOTE_CONNINFO_NODE_1`. A name of `node_1`
  produces the same key. The byte-by-byte
  `is_ascii_alphanumeric` filter treats `-` and `_` identically.
  Two descriptors with names that collide on the lookup key
  would resolve to the same conninfo. Recommend either:
  - validating `conninfo_secret_name` at registration time to
    reject names whose mangled form collides with an existing
    descriptor, or
  - documenting the mangling rule in the descriptor contract
    so operators avoid colliding names.
  Not blocking; worth a follow-up.
- **Multi-byte UTF-8 chars get per-byte underscore replacement.**
  `é` → `__`. Cosmetic; ASCII is the realistic input.

## Confirmations

- `provider_policy = "external_executor_secret_provider"` matches
  the 30630/30633 contract.
- The function name and behavior are consistent with the
  contract claim "executor-owned secret resolution."
- PG18 coverage exercises both missing and resolved env entries.
