## Answers to Review Questions

### Is the gamma field position reasonable?

**Yes.** Gamma is placed at `page.rs:150` after `heaptids` and before `neighbortid` in the `TqElementTuple` struct. The field is encoded as 4 bytes (f32 LE) in the tuple payload. Given the pre-v1 format status, exact byte position is not a long-term concern, but placing gamma adjacent to the code bytes is sensible for scoring cache locality — both are read together during candidate evaluation.

The `encoded_len` computation correctly accounts for the 4-byte gamma field, and decode/encode roundtrip tests (page.rs:630-655, 707-755) exercise distinct gamma values. The layout is coherent.

### Are there remaining element tuple construction or decode sites that should have moved?

**No.** All element writers now include gamma:
- Build-time staging: `build.rs:499-503` — sets `gamma: tuple.gamma`
- Live insert (new element): `insert.rs:159-163` and `insert.rs:231-235` — both set `gamma: tuple.gamma`
- Graph read surface: `graph.rs:38` — propagates `element.gamma` into `GraphElement`

All decode sites (`TqElementTuple::decode`, `DataPage::read_element`, `DataPageStore::read_element`) produce the gamma field. No remaining construction or decode site was missed.

### Is the follow-on boundary clear?

**Yes.** This slice persists gamma in the index without changing any hot-path consumer. The next slice (review 52) switches scan scoring and duplicate detection to use persisted gamma instead of heap fetches. The boundary is clean: structural change first, then hot-path cutover.
