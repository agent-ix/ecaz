# Feedback: A5 Layer-0 Backlinks and Lock Ordering

**Reviewer:** Claude (Opus 4.6)
**Date:** 2026-04-11
**Commit reviewed:** `51d40c0`

## Scope match

Matches the packet. `tqhnsw_aminsert` now:

1. computes forward links (prior slice)
2. appends the new element and releases its page lock
3. calls `add_layer0_backlinks_to_forward_neighbors` — loads up to `M` forward
   target elements, extracts their neighbor tuple TIDs, sorts ascending by
   `(block, offset)`, dedups, groups by block, and rewrites each group under
   one buffer `EXCLUSIVE` lock + one `GenericXLog` txn
4. `insert_layer0_backlink` writes into the first `INVALID` slot in the `2M`
   layer-0 window, no-ops if the new node is already present or the slice is
   full
5. then and only then re-acquires metadata exclusive to promote entry point /
   max level

`ADR-026` lands alongside with the full durable rule (traversal → append →
release → sort → per-page write in ascending physical order → metadata last).

## Findings

### 1. Lock protocol implementation is faithful to the ADR

Walking the insert path against ADR-026 Decision #1-#8:

- **#1 traversal read-only first:** ✓ (prior slice; `discover_insert_forward_neighbor_slots` runs before any write)
- **#2 append under one data-page EXCL:** ✓ (`append_heap_tuple` holds exactly the target page buffer lock)
- **#3 release append lock before backlinks:** ✓ (the buffer is `UnlockReleaseBuffer`-ed before `add_layer0_backlinks_to_forward_neighbors` runs)
- **#4 materialize + sort backlink targets:** ✓ (`sort_unstable_by(compare_item_pointers)` then `dedup()`)
- **#5 rewrite one page at a time, ascending block order:** ✓ (grouping loop in `add_layer0_backlinks_to_forward_neighbors`; one `GenericXLog` txn per group)
- **#6 within a page, ascending offset order:** ✓ (slice is sorted by `(block, offset)` before grouping, iteration preserves order)
- **#7 stale full-slice replan rule:** N/A this slice (no pruning yet)
- **#8 metadata lock last:** ✓ (the `if snapshot invalid || level > snapshot.max_level` block comes after backlink writes)

There is no path in the current code that takes a data-page EXCL and a
metadata-page EXCL simultaneously. The ADR's load-bearing deadlock-avoidance
invariant holds.

### 2. Per-page tuple rewrite is sound

`add_layer0_backlinks_on_page` is the subtle bit and it's correct:

- Opens buffer with `RBM_NORMAL`, asserts `BufferIsValid`, takes
  `BUFFER_LOCK_EXCLUSIVE`, starts one `GenericXLog` txn registering the page
  with `GENERIC_XLOG_FULL_IMAGE`.
- For every tuple in the group: decodes via `TqNeighborTuple::decode`, calls
  `insert_layer0_backlink`, re-encodes, copies back into the same
  `(tuple_offset, tuple_len)` slot.
- Critically: `if encoded.len() != tuple_len { pgrx::error!(...) }` — this
  catches any future change that might resize the neighbor tuple (e.g., adding
  a count field) and ensures we cannot corrupt the line pointer length by an
  in-place overwrite. Because `TqNeighborTuple` serialization is
  `tag + count + slot_count * 6 B` and `insert_layer0_backlink` only mutates
  tids in already-allocated slots, the length is in fact invariant. Defensive
  check is the right hedge.
- Only commits (`wal_txn.finish()`) if `changed == true`; otherwise drops the
  WAL txn with no write. Clean.

One thing worth hardening: `shared::page_item_id` does no bounds check against
`pd_lower` / line pointer count. If a corrupt neighbortid ever pointed past the
last item, the read of `item_id.lp_flags()` would be a wild read, and the
`lp_flags() == 0` guard would silently accept garbage. This is not reachable
from an honest `load_graph_element` because the element we loaded held that
neighbortid, but it's the kind of check that pays off the first time someone
wires `add_layer0_backlinks_on_page` from a different caller. **Recommended
follow-up:** have `add_layer0_backlinks_on_page` call
`page_line_pointer_count(page_ptr)` once and assert each neighbor's
`offset_number <= lp_count` before the dereference. Three lines, no hot path
cost (runs once per backlink target), catches exactly the "someone passed me a
bad TID" class of bugs.

### 3. `.take(m)` is correct, but mismatch with the forward slice is subtle

`add_layer0_backlinks_to_forward_neighbors` uses `.take(usize::from(m))` — i.e.
only the first `M` forward slots receive backlinks, not all `2M`. That matches
the prior slice, which only fills slots `[0..M)`. So the `.take(m)` is
equivalent to "take all populated forward slots" *today* but drifts the moment
the next slice widens the forward fill to `2M`. When the later slice lands,
this `.take(m)` must become `.take(2 * usize::from(m))` (or
`.take(layer0_width)`) to match, otherwise new inserts will only backlink half
their neighbors.

**Recommended follow-up:** replace `.take(usize::from(m))` with a named constant
or helper (`forward_layer0_width(m)`) that is the *one* definition of "how many
layer-0 forward slots are populated on a new node," so when that slot count
changes in the prune slice it only changes in one place. Call this out in the
A5 roadmap as a coupling point before 233 lands.

### 4. `.dedup()` after sort is load-bearing and correct

`backlink_neighbor_tids` is sorted, then deduped. Because two distinct
elements always have distinct neighbor TIDs (each element owns its neighbor
tuple), the dedup is in practice "defensive against
`forward_neighbor_slots[i] == forward_neighbor_slots[j]`." That duplicate is
*itself* unreachable today (the forward-slot search returns distinct
candidates), but the cost is one `O(n)` scan on a list ≤ M long. Keep it.

### 5. Capacity check — free slots only — is the right interim boundary

`insert_layer0_backlink` reads the `2M` layer-0 slice, checks for the new TID
(idempotency), then finds the first `INVALID` slot and fills it. If the slice
is full, the function returns `false` and no mutation happens on that target.

That is exactly the constraint the slice's title ("Layer-0 Backlinks and Lock
Ordering") implies: we're installing backlinks where there is room, without
making a pruning decision. The tradeoff is visible: a node whose `2M` layer-0
slice is already full will silently reject this new backlink, so the new node
becomes forward-reachable-to-it but not back-reachable-from-it. Call this out
explicitly in the 233 prompt as the failure mode that overflow pruning exists
to fix. Not a defect here.

One implicit assumption is worth documenting in a comment: `insert_layer0_backlink`
only touches `neighbor_tids[..layer0_width]`, so *upper-layer slots are
invisible to this helper*. If someone later tries to use the same helper for
upper-layer backlinks (slice 232) by passing a different slice of the same
vec, they'll need a different entry point that addresses a specific layer's
sub-range. Worth a one-line doc comment saying "layer-0 only; upper-layer
backlinks need their own helper."

### 6. Graph reachability test is the strongest assertion of the packet

`test_tqhnsw_live_insert_is_graph_reachable_via_backlinks` does what the other
two can't:

1. Builds a 4-row graph via `CREATE INDEX` (build path, not insert)
2. Walks live inserts 5..=32 with deterministic level-0 draws until it finds
   one where `debug_insert_level_for_heap_tid` returns 0 (so the insert is
   *not* promoted to entry point — reachability must come from backlinks
   alone, not from being the entry)
3. Asserts `metadata.entry_point != inserted_element_tid` — this is the part
   that makes the test real
4. Drives the scan-side `debug_rescan_candidate_frontier` with the inserted
   row's raw vector as the query
5. Asserts the inserted element appears in the graph frontier — meaning the
   runtime's graph-seeded traversal actually reached it

This is the test the packet needed. It is not a "we wrote the bytes" test; it
is a "the scan path reaches the row" test. It specifically excludes the
entry-point promotion shortcut. Strong.

One follow-up improvement for the next A5 slice, not this one: drive the same
assertion through a real `SELECT ... ORDER BY embedding <#> $1 LIMIT 1` SQL
query rather than `debug_rescan_candidate_frontier`, so that the assertion
rides the full amgettuple path. The debug entry point is fine for isolating
the traversal step, but an end-to-end SELECT is the thing a user will actually
run.

### 7. ADR-026 scope vs implementation scope

The ADR describes the *full* A5 insert protocol (traversal → append →
backlink → overflow pruning → stale retry → metadata), which is wider than
what this slice implements (traversal → append → backlink → metadata, no
pruning, no retry). That's intentional: ADR-026 is the durable rule that later
slices have to respect, not just a checkpoint note. Reviewing it as written:

- The `impact: Affects FR-016, ADR-019` line should probably be extended to
  name FR-020 (planner cost gate) and Task 06 directly, since the ADR's
  decisions bind behavior that both of those observe. Minor.
- Decision #7 ("stale full-slice plan") is not yet exercised by code. Mark
  with a `<!-- TODO(A5 slice 234): implement retry path for full-slice
  overflow -->` comment inside the ADR so a later reader knows which slice is
  expected to close this gap.
- "First-insert initialization remains on the existing metadata-lock path
  because it does not yet touch multiple data pages" in Consequences > Neutral
  is accurate and important. Keep.

### 8. Tests are good, minor gap

- `test_tqhnsw_insert_populates_forward_links_from_live_entry_seed` is updated
  to also check that row1's neighbor tuple gained `row2`'s TID in its layer-0
  slice — this is the round-trip check on the small case.
- `test_tqhnsw_live_insert_is_graph_reachable_via_backlinks` is the
  large-fixture behavioral check.

**Gap not worth blocking on:** no test exercises the "layer-0 slice is full,
backlink is silently dropped" path. Today the test fixtures don't build a
layer-0 slice to capacity. Once 233 (overflow pruning) lands, that test
becomes mandatory — until then, document in the 233 prompt that a fixture
that saturates `2M` on a pre-existing node is the regression the pruning code
needs to own.

## Verdict

**Approve.** The lock protocol implementation is faithful to ADR-026, the
per-page GenericXLog rewrite is sound, and the graph reachability test
directly exercises the runtime frontier rather than just checking persisted
bytes. ADR-026 is the right durable artifact to land with this slice and its
scope ahead of implementation is the correct trade (later slices can't
retroactively invent the rule).

Follow-ups, all for the next A5 slice rather than blockers on this packet:

1. **Item 2 (`page_item_id` bounds):** add a `page_line_pointer_count` check
   before the first `item_id` read in `add_layer0_backlinks_on_page`.
2. **Item 3 (`.take(m)` coupling):** extract "how many forward layer-0 slots
   are populated" into one named helper before 233 touches the forward fill.
3. **Item 5 (helper layer scope):** one-line doc comment on
   `insert_layer0_backlink` stating it's layer-0 only.
4. **Item 6 (SQL-level reachability):** add an end-to-end
   `SELECT ... ORDER BY ... <#> $1` test in the closeout packet, not a
   debug-frontier test, so the amgettuple path gets its own assertion.
5. **Item 7 (ADR cross-refs):** extend `impact:` to name FR-020 / Task 06;
   mark Decision #7 with a TODO until the retry path lands.
6. **Item 8 (capacity path):** the "layer-0 full → silently dropped backlink"
   path becomes a test in 233, not a gap in 231.

The load-bearing win here is that `tqhnsw_live_insert_is_graph_reachable_via_backlinks`
is actually testing what its name says, using a non-promoted inserted element.
That is the hardest assertion to write correctly in this family of tests and
the one most likely to get confused with "the entry point got promoted, so of
course we find it."
