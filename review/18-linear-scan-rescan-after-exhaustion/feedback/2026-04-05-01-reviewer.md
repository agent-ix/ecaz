# Feedback: Linear Scan Rescan After Exhaustion

**Reviewer:** Claude (Opus)

## Response to Review Notes

### Rescan after exhaustion with duplicate-coalesced final element

**Agree.** The reviewer correctly identified that `reset_scan_position` zeroes `pending_heaptid_count` and `pending_heaptid_index`, so the mid-drain rescan edge case is handled. A targeted regression test for "exhaust while mid-drain, then rescan" is a worthwhile addition — it documents a contract that's currently only implicitly covered.

### `scan_block_count` staleness on rescan

**Agree — this should be documented.** The cached `scan_block_count` is not refreshed on rescan, so concurrent inserts adding pages after the original `amrescan` would be invisible to the rescanned pass. This is acceptable for the bootstrap linear scan and consistent with MVCC snapshot semantics (a scan should see a consistent point-in-time view). Add a one-line note to ADR-008 or ADR-009 as suggested.

### ADR documentation

**Agree.** A one-liner in ADR-008 about the cached block count behavior during rescan would be the right home.

## Additional Findings

No correctness issues. The `scan_block_count` staleness is a known limitation, not a bug — but documenting it prevents future confusion when ordered scan work begins.
