# Review: SPIRE Tuple Payload Missing Signal and Batch Fetch (30812)

## Summary

Closes both P2 findings from 30807. The single batched
`LEFT JOIN LATERAL unnest($1::text[]) WITH ORDINALITY` query replaces
the N+1 SPI pattern in one round-trip, and the new
`tuple_payload_missing bool` plus `remote_tuple_payload_missing`
status string make the missing-heap-row case explicit instead of
silently returning `'{}'::jsonb` with `status = 'ready'`.

The three review-focus questions:
- Batched SPI shape: correct (one parse, one plan, one execute, CTIDs
  preserved by ordinality).
- Endpoint contract change: missing payloads now have both a boolean
  flag and a distinct status — the planner-side classifier and the
  executor merge can now branch cleanly on either signal.
- Endpoint remains a JSON bridge: yes, acknowledged. The next slice
  needs typed tuple slots; the JSON return shape is now the only
  remaining limitation here.

## Findings

### P2 — `payloads.remove(&ctid)` is sensitive to duplicate CTIDs

`ec_spire_remote_search_tuple_payload` builds a `HashMap<String, ...>`
keyed on CTID and then calls `payloads.remove(&ctid)` per candidate
row. If two different scored candidates ever share the same CTID
(e.g., a vector that has been re-routed through different leaves of
the same heap row, or a duplicated rerank entry), the second call
panics with "missing payload result for ctid". The producer
(`spire_remote_search_local_heap_candidate_rows`) probably
deduplicates already, but the failure mode is "ERROR mid-iterator"
with a confusing message rather than "two candidates point at the same
heap row, here is what we'll do."

Two acceptable shapes:
- Use `Vec<_>` with positional matching (rows arrive in input order
  by `candidate.ordinality`); skips the HashMap entirely and naturally
  preserves duplicates.
- Use `HashMap<String, Vec<...>>` if the assumption that CTIDs are
  unique should hold but you want to detect violations explicitly
  with a clearer error.

Lower preference: keep the map but switch to `get` + clone, with an
explicit comment that duplicate CTIDs are an upstream invariant
violation.

### P3 — `regclass::text` SQL interpolation, again

Same shape as the 30807 finding: the table reference is built from a
`regclass::text` cast and interpolated into the SQL string. Postgres'
regclass formatter handles quoting, so this is safe in practice. Worth
either factoring a `quote_qualified_relname(oid) -> String` helper that
both this packet and the per-row caller can use, or moving the
relation reference into a prepared statement. Cosmetic; can land with
the typed-slot migration.

### P3 — Empty-input contract

If `ctids: &[]` is empty, the SQL still runs against `unnest('{}')`
and returns zero rows, then `payloads.remove(&ctid)` is never called.
That's the right behavior, but worth a short-circuit `if ctids.is_empty()
{ return Ok(HashMap::new()); }` so the empty-batch case doesn't pay
for an SPI roundtrip.

### P3 — Test exercises the helper directly, not the public endpoint

`test_ec_spire_remote_search_tuple_payload_missing_ctid_signal` calls
`ec_spire_remote_search_tuple_payloads_for_ctids` directly with a
hand-crafted `(999,1)` CTID. That proves the helper handles missing
CTIDs but doesn't exercise the public
`ec_spire_remote_search_tuple_payload(...)` SQL function with a
missing-CTID candidate. Worth an end-to-end SQL test that drives
through the public function and asserts on the row's `status` /
`tuple_payload_missing` columns. Not blocking.

## Reviewer recommendation

Accept. The P2 duplicate-CTID handling is the only item I'd want
addressed before this becomes a busy production code path; the P3
items are polish.
