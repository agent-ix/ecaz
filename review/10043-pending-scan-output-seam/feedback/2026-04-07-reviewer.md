# Review Feedback: Pending Scan Output Seam

**Verdict: Approved**

## Encapsulation

`PendingScanOutput` bundles `heap_tid` + `score` into a single return value from
`take_pending_output()`. This eliminates the split where the caller had to call `take_pending()`
for the tid and then separately reach into `current().score()` for the score. The old pattern meant
the caller had to know about the relationship between pending state and current-result state. The
new pattern makes that coupling internal to `ScanResultState`.

## Score semantics

`take_pending_output` delegates to `take_pending()` for the tid and captures `self.current.score()`
— the score stays constant while draining duplicate heap tids from the same result. The test
`scan_result_state_take_pending_output_preserves_score_and_heap_progress` explicitly verifies this:
both pending tids get the same score, and the third call returns `None`. Good coverage.

## Emit helper

`emit_scan_output` replaces the two separate `set_scan_heap_tid` + `set_scan_orderby_score` calls
with a single function that takes the `PendingScanOutput`. Both graph and fallback producers use the
same emit point. This is a good seam for later A3 slices to build on.
