# Feedback: ADR-020 Dimension Operating Points

Request:
- `review/10106-adr020-dimension-operating-points/request.md`

**Reviewer:** Claude (Opus)
**Date:** 2026-04-08

## Response to Review Focus

### Does ADR-020 reflect the intended 1024 / 1536 / 2048 design exploration?

**Yes.** The ADR is well-grounded in repo facts rather than competitive speculation. The three operating points are clearly positioned: 1024 as speed candidate, 1536 as current baseline, 2048 as quality candidate. The decision to leave the default open until measurement exists is the right call.

### Is the query-prep vs per-candidate cost distinction clear?

**Yes.** The ADR makes this unambiguous in multiple places:
- Section "Context" point 2: "FWHT is not a per-candidate search cost"
- Section "What Not To Assume" point 1: "Do not model tqvector search as 'an FWHT per distance calculation'"
- The table separates `transform_dim` (FWHT work), payload bytes (storage), and code bytes (scoring) into distinct columns

The key insight — that 1536 and 2048 share the same FWHT transform length but differ by 33% in per-candidate scoring cost — is clearly stated.

### Does the ADR balance repo facts vs unmeasured hypotheses?

**Yes.** The "Explicit Working Hypotheses" section clearly labels what is known vs what requires measurement. The "Required Validation" section lists five concrete measurement categories that must be completed before the operating point decision can be revisited. No overclaiming.

## Additional Findings

1. **This ADR should land on `main` promptly.** It affects how both coder-1 (runtime lane) and coder-2 (SIMD lane) frame their benchmark targets. Keeping it branch-only risks the two lanes diverging on dimension assumptions.

2. **The table's "AVX2 3-bit hot query bytes" column header says 3-bit but the surrounding text says `bits = 4`.** This appears to refer to the AVX2 prepared-query state layout which uses a different encoding than the storage bit-width. If so, a brief clarifying note in the table header or a footnote would prevent confusion. Minor.

## Verdict

No blocking issues. Well-written ADR that correctly recenters the dimension discussion on implementation-grounded costs. Recommend landing on `main`.
