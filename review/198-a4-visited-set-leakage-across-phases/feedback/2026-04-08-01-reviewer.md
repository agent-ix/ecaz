# Feedback: A4 Visited Set Leakage Across Search Phases

Request:
- `review/198-a4-visited-set-leakage-across-phases/request.md`

**Reviewer:** Claude (Opus)
**Date:** 2026-04-08

## Verdict: No Leakage Bug Found

### Traced the visited set lifecycle

1. `reset_scan_position` → `reset_scan_visited_state` ��� **clears** the visited set
2. `greedy_descend_from_entry` (`graph.rs:133-155`) — **does not touch** the scan-level visited set.
   The descent uses its own local loop state; it doesn't call `mark_visited_element`.
3. `search_layer0_result_candidates` (`graph.rs:332-401`) — uses an **internal local** `HashSet`,
   completely independent of the scan-level visited set. The scan-level filter
   `|neighbor_tid| !visited_contains_element(opaque, neighbor_tid)` is passed but the scan visited set
   was just cleared, so nothing is filtered.
4. `stage_ordered_graph_results` → `seed_discovered_candidates` — marks discovered candidates as
   visited in the scan-level set. This happens **after** the search completes.

There is no cross-phase leakage. The upper-layer descent and layer-0 beam search operate with
independent visited tracking.

### The refill concern

The review correctly notes that bootstrap trace seeding marks all discovered candidates as visited,
which prevents rediscovery during later refill. However:

- The production scan path (`prefetch_next_graph_result_from_frontier`) does **not** use refill. It
  only drains the initial results from `search_layer0_result_candidates`.
- The refill functions (`consume_and_refill_bootstrap_frontier`,
  `refill_candidate_frontier_from_source_into`) are gated behind `#[cfg(any(test, feature = "pg_test"))]`.
- So visited-set overpopulation from bootstrap cannot affect production recall.

### Reference implementation comparison

The review's comparison with hnswlib-rs and instant-distance is useful context but the concern doesn't
apply here: tqvector's layer-0 search creates a fresh local visited set, matching the reference
pattern.

## Disposition

**Closed — no bug.** The visited set lifecycle is correct. The scan-level visited set is only
populated after the layer-0 search completes, and the production path never refills from the graph
after initial staging.
