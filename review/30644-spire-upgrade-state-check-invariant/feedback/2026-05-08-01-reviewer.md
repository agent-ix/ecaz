# Upgrade state check invariant review

Closes my 30637 finding about the descriptor-state CHECK
constraint being duplicated between `sql/bootstrap.sql` and
`ecaz--0.1.0--0.1.1.sql` without a drift guard. Test parses
both SQL files via `include_str!` and compares to the catalog-
state contract. No blocking findings.

## Confirmations

1. **Three-way invariant.** Bootstrap SQL, upgrade SQL, and
   the Rust state-source contract are all asserted equal in
   one PG18 test (`lib.rs:20338-20385`). A future change to
   any of the three that doesn't update the others fails at
   test time.
2. **`include_str!`** keeps the test self-contained — no
   filesystem reads at test runtime. Matches the existing
   30603 invariant pattern.
3. The parser uses a sentinel marker
   (`descriptor_state text NOT NULL CHECK (\n        descriptor_state IN (`)
   plus comma-split + quote-strip. Brittle to whitespace
   changes in the SQL but correct for the current shape.

## Findings

- **Parser brittleness.** If a future SQL formatter run
  reformats the CHECK clause (e.g. inline vs. multi-line),
  the marker string match would fail. Acceptable trade-off —
  the test would fail loudly rather than silently miss drift.
  Worth a comment in the parser noting the sentinel form is
  deliberate. The packet's review focus question asks
  whether parsing is sufficient or whether a generated
  state-list source would be better; for v1 this works, but
  a future migration generator that emits the state list
  from the Rust contract would eliminate the parsing
  altogether.
