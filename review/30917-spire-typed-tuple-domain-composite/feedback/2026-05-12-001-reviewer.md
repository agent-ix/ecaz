# Review: SPIRE Typed Tuple Domain/Composite (30917)

## Summary

Closes the 30916 P3 — composite + domain coverage that I flagged
should land before negotiation work assumes type-class
completeness.

The fixture creates a text domain (`ec_spire_typed_label_domain`),
a named composite (`ec_spire_typed_pair`), and a table mixing
domain + composite + bigint + ecvector columns. Then asserts
`ec_spire_remote_search_tuple_payload_typed(...)` returns:

- correct `payload_type_oids` for the **domain** (preserves the
  domain OID, not the base type OID — important for receive-side
  validation)
- correct `payload_type_oids` for the named composite
- aligned `payload_attnums`, `payload_names`, `payload_nulls`,
  `payload_formats`
- domain value bytes match the **base** type's binary send
  (right call — domains use the base type's I/O functions; the
  domain identity stays in metadata)
- named composite bytes match `record_send(ROW(...)::named)`
  byte-for-byte

The three review-focus questions:
- Domain bytes via base-type binary send while preserving
  domain OID in metadata: yes — this is the canonical PG
  convention. `typsend` for a domain delegates to the base
  type's typsend; receive-side `typreceive` does the same. The
  domain check (RLS-style domain CHECK constraints) runs on
  receive, not at the wire format.
- `record_send(ROW(...)::named_composite)` as oracle: yes —
  this is exactly what PG would do internally for a composite
  column.
- No overclaimed coverage: confirmed — negotiation, CustomScan
  typed receive, throughput measurement, and JSON retirement
  remain open.

## Findings

### P3 — All four type-class gaps from 30816 now closed

The original 30816 scalar gate rejected: arrays (closed by
30916), composites (closed by 30917), domains (closed by
30917), and… arguably nothing else. Scalars work via 30915. So
the typed endpoint covers every type class the JSON bridge
couldn't handle.

Worth pinning in the tracker: "All v1 distributed table column
types now have typed-transport coverage. Negotiation can
proceed without per-type capability gaps."

### P3 — `record_send` for anonymous composites?

The fixture uses a **named** composite. A future user could write
`SELECT (id, title) FROM tbl` against a coordinator-routed
distributed table — the row would carry an anonymous composite.
`record_send` works for both, but typed-receive on the
coordinator needs to either (a) reconstruct the field types from
the composite's oid lookup, or (b) reject anonymous composites
in v1. Worth pinning in the design doc which v1 chooses.

### P3 — Domain CHECK constraint enforcement on receive

The packet doesn't claim the receive side enforces the domain's
CHECK constraint. It shouldn't have to — `typreceive` for a
domain is the base type's `typreceive`, and PG runs the domain
CHECK separately when storing into a domain column. A future
fixture could exercise "remote sends a base-type value that
violates the coordinator's domain CHECK" and confirm the right
SQLSTATE fires; not load-bearing for v1.

## Reviewer recommendation

Accept. Closes the type-class coverage gap. Negotiation work
(30919 first slice) and CustomScan typed receive can now
proceed without worrying that some type class will surprise
them.
