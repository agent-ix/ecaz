# Feedback: Gamma-Aware Duplicate Coalescing

**Reviewer:** outside-review  
**Date:** 2026-04-05  
**Verdict:** Approved with findings

## Answers to review questions

1. **Using first stored heap TID as representative gamma source**: This is an acceptable interim contract. The first heap TID is always present (heaptids is non-empty for non-deleted elements), and all coalesced heap rows share the same `(gamma, code_bytes)` by definition. However, if the representative heap row is later deleted by vacuum or HOT pruning while other heap TIDs in the element survive, the gamma fetch will fail. This is a known limitation documented in ADR-010 — future vacuum work needs to handle representative row migration.

2. **`SnapshotAny` for the heap fetch**: `SnapshotAny` (mod.rs:564) means the fetch sees all tuple versions including dead ones. For duplicate *detection* during insert, this is conservative — it avoids falsely concluding no duplicate exists because the representative row is not yet visible to the inserter's snapshot. However, it also means a deleted-but-not-yet-vacuumed representative row will still match. This is acceptable: the element tuple is still live, so coalescing into it is correct. The risk would be fetching a gamma from a truly gone (vacuumed) tuple, but `heap_fetch` would return `found=false` in that case and the code correctly errors out.

3. **Persisting gamma in element tuples**: **This is the most important follow-up.** The current design requires a heap fetch to recover gamma for every duplicate candidate check during `aminsert` and for every element scored during scan. For `find_duplicate_element_tid`, this is O(elements) heap fetches — expensive. For ordered traversal, scoring every candidate requires a heap fetch for gamma. Persisting gamma as a 4-byte field in `TqElementTuple` would:
   - Eliminate heap fetches during duplicate detection (compare element.gamma directly)
   - Eliminate heap fetches during scan scoring (read gamma from the index tuple)
   - Remove the representative-row dependency for gamma recovery
   - Cost only 4 bytes per element tuple (trivial compared to code_len)
   
   **Recommend prioritizing this before ordered traversal lands.** It is a page-layout change, so it's better to do it now while the on-disk format is still unstable.

## Additional notes

- The `gamma.to_bits()` comparison (mod.rs:483) is correct for exact bitwise match, including distinguishing +0.0 from -0.0 and handling NaN. This is the right choice for duplicate semantics.
- The O(n) linear scan in `find_duplicate_element_tid` is a known scalability limitation. It works for small indexes but will need a hash or sorted lookup for larger ones. Not blocking for v0.1.
