# Review: Resolved ef_search Runtime Bootstrap (Review 169)

**Reviewer:** Claude Opus 4.6
**Date:** 2026-04-07
**Scope:** Review 169 (ef_search runtime wiring)
**Commit:** `f12e74c`

## Summary

Switches `amrescan` bootstrap frontier sizing from the hardcoded `MAX_BOOTSTRAP_FRONTIER_CANDIDATES
= 3` constant to the resolved `ef_search` value from `resolve_scan_tuning()`. This was the key
post-merge coordination item flagged in reviews 166 and 167.

**Verdict: Approved.**

## Assessment

The change is small and correct:
- `amrescan` now calls `resolve_scan_tuning()` to get the effective ef_search value (session GUC
  override > index reloption > default)
- Bootstrap beam width and frontier limit now respect `SET tqhnsw.ef_search = N`
- The pg_test proves session override narrows the runtime frontier even when the index reloption
  is wider

This completes the most important post-merge wiring item. The bootstrap traversal now consumes
the planner-owned ef_search control surface.

## One remaining issue: GUC sentinel bug

The underlying `resolve_scan_tuning_values` still uses `session_ef_search == 40` (the default) as
the "not overridden" sentinel. This means `SET tqhnsw.ef_search = 40` on an index with
`ef_search = 128` silently uses 128 instead of the explicit 40.

**Fix:** Change the GUC default to -1 (outside valid range 1-1000) and treat -1 as "use relation
value." This is a ~10-line fix in `options.rs`. Not blocking this review but should be addressed
before A3 wiring.

## Net

Approved. Critical post-merge wiring done. GUC sentinel fix tracked separately.
