## Findings

### 1. Two-refresh-variant split is worth naming explicitly

**Severity:** Low
**File:** `src/am/common/parallel.rs:1019, 774`

The module now has two refresh paths:

- `refresh_coordinator_selection_snapshot` — full heap rebuild + reap +
  fast-path update.
- `refresh_coordinator_selected_result_fast_path` — root-only reap +
  fast-path update.

Both exist for good reason (take doesn't need a full rebuild), but the
naming only hints at the distinction. A two-line doc comment on each
explaining "use full refresh when publishing/clearing an arbitrary
slot; use fast-path refresh when the root is the only thing that
changed" would save the next reader ten minutes of tracing.

### 2. `refresh_selection_snapshot: bool` argument is a subtle handshake

**Severity:** Low
**File:** `src/am/common/parallel.rs:1309–1313`

`clear_parallel_scan_result_slot_with_attachment` grows a bool 4th
parameter controlling whether it fires the full refresh. This
handshake works but is easy to get wrong — the caller must remember
to pair `false` with an explicit `refresh_coordinator_selected_result_fast_path`
call. Consider two named entry points instead:
`clear_and_refresh_full` / `clear_without_refresh` — the boolean
becomes the function name. Not a correctness issue; readability.

### 3. Test assertions include `result_publish_generation`

**Severity:** Informational (positive)
**File:** `src/am/common/parallel.rs:2612, 2715-ish`

Tests explicitly assert the `result_publish_generation` counter
increments as expected across publish/clear/take sequences. That's the
right invariant to lock down — generation-based invalidation depends
on it being monotonic.
