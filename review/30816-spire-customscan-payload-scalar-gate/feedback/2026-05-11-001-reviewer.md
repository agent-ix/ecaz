# Review: SPIRE CustomScan Payload Scalar Gate (30816)

## Summary

Closes the P1 array-column input bug from the 30814 review and the P2
per-row `fmgr_info` waste. `custom_scan_validate_tuple_payload_attr`
fails closed at `BeginCustomScan` for any column whose type is array
or composite/row, with a column-named error message. The
`SpireCustomScanPayloadAttrInput` cache populated at Begin lifts
`getTypeInputInfo` and `fmgr_info` out of the per-row loop. The
JSON-value branch for `Array | Object` now errors instead of producing
a nonsense PG array literal.

The four review-focus questions:
- Fail-closed type gate as interim contract: yes — the documented
  ADR-068 direction is typed-tuple transport, and erroring early at
  Begin is much better than mid-scan parse failures.
- `SpireCustomScanPayloadAttrInput` lifetime: lives on the exec state
  which is per-scan, freed in `EndCustomScan`. The `FmgrInfo`
  internals (e.g., `fn_mcxt`) are populated against the current
  per-query memory context, which outlives the scan. Correct.
- Validation only rejects requested payload columns: yes — array
  columns that aren't projected are not visited by the validator, so
  `SELECT id FROM tbl_with_array ORDER BY embedding ...` still works.
- Open items (degraded-mode skip accounting, remote projection
  pushdown, JSON-bridge removal): noted; deferring is appropriate.

## Findings

### P2 — `ecvector` columns will hit the array-type gate

`get_element_type((*attr).atttypid) != InvalidOid` is the standard
"type X is an array of element type Y" check. If `ecvector` is
defined as a `CREATE TYPE ... AS ARRAY` over `real4` (or anything
that exposes a non-zero `typelem`), it will be rejected by the gate
even though it's a domain-specific scalar. Worth checking
`pg_type.typelem` for `ecvector` and either:
- Allow-list `ecvector` (and other extension scalar types that look
  like arrays under PG metadata) in the validator, or
- Document that `ecvector` columns cannot be projected through the
  CustomScan tuple-payload path in v1, with the typed-transport
  follow-up explicitly listing it.

If `ecvector` is internally a base type (single I/O pair, no
`typelem`), the check passes and this is moot. The fixtures don't
project `embedding` so the issue is masked either way.

### P2 — `type_is_rowtype` blocks composite columns; that includes JSONB?

`type_is_rowtype((*attr).atttypid)` is the right check for relational
composite types (e.g., user-defined `CREATE TYPE foo AS (a int, b
text)`). It does *not* block JSON / JSONB / range / enum / domain —
those go through `InputFunctionCall` cleanly. Worth a one-line
comment in the validator listing what the gate covers and what it
intentionally lets through, so a future reviewer doesn't reach for
"why is JSONB allowed but composite isn't?"

### P3 — `tuple_payload_inputs` lifetime tied to exec state

`tuple_payload_inputs: Vec<Option<SpireCustomScanPayloadAttrInput>>`
is owned by the exec state, freed via `drop_in_place` in
`EndCustomScan`. Same memory-context-callback concern from the 30810
review applies: if `EndCustomScan` is skipped on error, the `Vec`
backing buffer leaks through Rust's allocator. The `FmgrInfo` itself
sits in the palloc'd struct but its `fn_mcxt`-resident fields are
already managed by PG's memcxt, so this leak is bounded to the
`Vec`'s allocation rather than every flinfo's internals.

This continues to argue for the memory-context-callback wrapping
suggested previously. Not a regression here, but the cache makes the
state struct heavier so the leak grows.

### P3 — Test name reflects the gate

`SELECT tags text[] ... ORDER BY embedding ... LIMIT` proves the gate
fires at Begin, not silently at first row. Good evidence. If
`ecvector` projections need to be validated explicitly, add a similar
`SELECT embedding FROM tbl ORDER BY embedding ... LIMIT` test and
record whether the gate fires (it should fail-closed in v1; if it
silently succeeds, that's a wrong-results path).

### P3 — Status next-step is now the multi-instance fixture / write path

`next_step = "add multi-instance CustomScan read fixture and ADR-069
write path"` aligns with the 30815 + 30817/30818/30819 packets in
flight. Good narrative thread; the diagnostic now points at the next
real work.

## Reviewer recommendation

Accept. The P2 ecvector-vs-array-type-gate question is the only thing
that could surprise an operator: confirm `ecvector` doesn't trip
`get_element_type` and either allow-list it or document the
limitation before the typed-transport packet lands.
