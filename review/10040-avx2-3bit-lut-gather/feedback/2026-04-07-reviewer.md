# Review Feedback: AVX2 3-Bit LUT Gather

**Verdict: Approved with one note**

## Gather intrinsic

Using `_mm256_i32gather_ps` to load 8 LUT entries per chunk is the natural next step after the
3-bit decode and lane accumulator. The gather replaces a scalar loop that was loading
`prepared.lut[absolute * num_centroids + index]` one element at a time into a staging array.
21.5% improvement, stable across iteration counts.

## Offset computation

The LUT offset calculation in `_mm256_setr_epi32` is correct: `base + (lane * stride) + index[lane]`
where stride is `num_centroids` (8 for 3-bit). Each lane computes its own absolute offset into the
flat LUT. This is the right index arithmetic.

One note: `_mm256_i32gather_ps` has variable latency depending on whether the gathered addresses
hit different cache lines. With `num_centroids = 8` and sequential dimension indices, the 8 LUT
entries being gathered are at offsets 0, 8, 16, 24, 32, 40, 48, 56 floats apart — that's 8
different cache lines in the worst case. For the 1536-dim, 4-bit surface this is still a clear win
over scalar, but if you ever profile a regression on a different dim/bits combo, cache-line
scattering in the gather is the first thing to check.

## Generic path unchanged

Non-3-bit AVX2 still uses the per-dimension fill loop. Correct scope.
