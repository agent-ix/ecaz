# Coordinator local remote search review

First executable coordinator-side path
(`root/hierarchy_snapshots.rs:241-356`,
`ec_spire_remote_search_coordinator_local`,
`ec_spire_remote_search_coordinator_local_summary`). Cleanly threads
through fanout planner → leaf scoring → receive validator → merge
helper, and fails closed if a remote target ever sneaks into the
plan. No blocking findings.

## Confirmations

1. Local-only failure is explicit: `plan.remote_targets.is_empty()`
   guard at `hierarchy_snapshots.rs:304-309` errors with a clear
   "requires libpq transport for N remote target(s)" message before
   any object store opens.
2. Coordinator-local execution uses the same fanout, receive
   validation, and merge boundaries the future libpq path will use —
   `merge_validated_remote_search_candidate_batches` is invoked even
   for a single local batch, so the receive-validation contract is
   exercised on the local path too.
3. The result-helper refactor (sharing
   `remote_search_candidates_result` between storage-node and
   coordinator entries) is a clean extraction. The storage-node
   endpoint keeps the same caller-visible behavior.
4. `degraded_ready` is the right status name for plans that complete
   without transport because a degraded skip removed remote targets;
   it is distinct from `requires_libpq_transport` which fires when a
   nonzero remote target survives planning.
5. Test-only placement-node rewrite (`debug_spire_rewrite_placement_node`)
   is `#[cfg(any(test, feature = "pg_test"))]` and only flips the
   `node_id` field; it does not weaken local-store validation in
   production paths.

## Findings

- **Redundant top-k truncation.**
  `collect_quantized_selected_leaf_candidates` is called with
  `Some(top_k)` at `hierarchy_snapshots.rs:330`, producing at most
  `top_k` rows. Those rows are then handed to
  `merge_validated_remote_search_candidate_batches` with
  `Some(top_k)` again at line 352, which truncates to top_k a second
  time. With only one batch the second truncate is a no-op except
  when the leaf collector returned fewer than top_k rows (in which
  case the second truncate is also a no-op).

  This is correct but redundant. Once libpq lands and multiple
  batches arrive, the leaf collector will run per-target and the
  top-k cap on the merge step becomes load-bearing — at that point
  consider whether per-batch top-k pre-truncation should be a fixed
  multiple of the merge-side cap (to allow score competition across
  nodes). Worth a comment now describing the intended division of
  labor.

- **Summary helper does full planning even for `top_k = 0`.**
  `remote_search_coordinator_local_summary_result:387` does not
  short-circuit on `top_k == 0` the way
  `remote_search_coordinator_local_candidates_result:275` does.
  This is intentional (the summary should report counts even for
  empty-top-k probes), but the asymmetry is worth a comment, since a
  reader could otherwise assume the candidate path's short-circuit
  is a contract.
