# Review: Packet 281 — C1 ADR-031 Cached Binary Prefilter Runtime

Reviewer: agent-ix (via code-review)
Date: 2026-04-12

## Assessment: Major breakthrough — but the dominant win is not what the packet title suggests

### What landed

Integrated the ADR-031 cached binary prefilter into ordered-scan runtime:

1. Scan-local prepared binary query (`BinarySignNoQjl4BitQuery`)
2. Cached binary words on `CachedGraphElement`
3. Lazy exact scoring — newly loaded elements defer exact scoring until after
   the binary prefilter
4. Source-local binary survivor gate in `cached_scan_successor_candidates_for_layer`

Result: p50=2.87ms, mean=2.83ms at 10K — a 3.7x improvement over the 10.51ms
baseline.

### The most important observation

The `drop=0/2/4` budget sweep is effectively flat:

- `drop=0`: mean=2.828ms
- `drop=2`: mean=2.826ms
- `drop=4`: mean=2.824ms

This means **the binary rejection step contributes almost nothing to the
latency win**. The dominant improvement comes from making exact scoring lazy —
not computing the 14,000ns exact score eagerly on every graph cache miss.

The packet notes this ("the dominant win comes from making exact scoring lazy
for newly loaded graph elements"), which is good. But the architectural
implication deserves emphasis: the ADR-031 binary prefilter is primarily
valuable as a **justification for lazy scoring**, not as a meaningful filter in
its own right on the warm path.

### Why this matters for next steps

Since the binary filter itself isn't the main contributor:

1. **Investing in better binary code quality** (full RaBitQ, larger binary
   dimensions) is unlikely to improve warm latency further. The filter already
   passes almost everything through.
2. **The cold path is where binary filtering should matter more**, because cold
   misses pay the full buffer-read + decode cost. Persisted sidecars (packet
   285) matter more for cold than warm.
3. **The remaining ~2.8ms at 10K** is dominated by buffer I/O, decode overhead,
   and per-element allocation — not by scoring cost. Further gains require
   structural changes to how elements are loaded and cached.

### On the implementation shape

The `LoadedElementState` enum (`None | ExactScore(f32) | ExactPayload(...)`) is
a clean way to defer the scoring decision. The `BinaryPrefilterCandidate`
staging with sort-and-truncate is straightforward.

One allocation to note: `LoadedElementScoreInput` still copies 768 bytes of
code via `element.code.to_vec()` (`scan.rs:760`) for every cache miss on the
binary path. This copy exists because the PG buffer is released before scoring
happens. Eliminating this copy (by scoring while the buffer is still pinned)
would remove the largest remaining per-element allocation on the hot path.

### Verdict

This is the most important packet in the C1 arc. The 3.7x improvement puts the
10K seam well below NFR-001 and proves the lazy-scoring architecture works. The
binary prefilter provides the structural framework for further optimization even
though the filter itself isn't the main latency contributor.
