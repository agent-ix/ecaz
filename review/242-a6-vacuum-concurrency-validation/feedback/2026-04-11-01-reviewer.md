## Findings

### 1. The `pg_test`-only SQL seam is the right narrow opening

`tqhnsw_debug_scan_result_count` is a thin `#[pg_extern]` that returns
`i32`. It is gated behind `#[cfg(any(test, feature = "pg_test"))]`
(inherited by being inside the `tests` module), so it will not appear
in a release build. That is exactly the right boundary — the harness
needs a cheap scalar that forces a real tqhnsw scan from SQL, and this
is the smallest possible extension of the existing debug helper.

One small inefficiency: the wrapper calls `open_valid_tqhnsw_index(...)`
to validate the relation is a tqhnsw index, then *immediately* closes
it with `AccessShareLock`, and `debug_gettuple_scan_heap_tids` then
opens the same relation *again* with `AccessShareLock` from scratch
(`src/lib.rs:10557–10564` vs `src/am/scan_debug.rs:461`). So every
harness probe does two `index_open` / `index_close` pairs instead of
one. Harmless for harness use, but if the validation helper returned
the validated `Relation` and the caller threaded it into the debug
helper, this would collapse to one open/close. Not a blocker for A6.

The paired regression
(`test_tqhnsw_debug_scan_result_count_matches_scan_helper`) verifies
the SQL wrapper and the Rust helper return the same count on the same
fixture. Good — this prevents the SQL seam from silently drifting from
the underlying helper. It intentionally uses a 3-row fixture so equality
is unambiguous.

### 2. Harness concurrency topology is structurally correct

The concurrent mix (1 INSERT worker, 1 VACUUM-after-DELETE worker, 2
different scan workers) against the same table exercises the A6
invariants that matter:

- INSERT takes `RowExclusiveLock`, VACUUM takes
  `ShareUpdateExclusiveLock`, scans take `AccessShareLock`. All three
  are compatible with each other at the table level, so they actually
  run concurrently — this is the real MVCC story, not a simulated one.
- `DELETE + VACUUM` followed by live-row and scan-row counting exercises
  pass 1 (heap-TID strip), pass 2 (dead-edge unlink + layer-aware
  replacement fill), pass 3 (finalize dead elements), and then a
  subsequent scan that must traverse the repaired graph.
- Repeated INSERT during VACUUM exercises the pass-2 planner's
  read-only snapshot vs a moving write front — if the planner or
  unlinker had a lock-ordering or aliasing bug, the `ShareUpdateExclusive`
  overlap window is exactly where it would surface.
- Two distinct fixed query vectors run in parallel, so the scan
  workers have different traversal paths into the graph each iteration.

That said, there are quality gaps in the harness that I want flagged
before this becomes the `TC-215` primary proof — see findings 3–5.

### 3. CRITICAL: the harness never asserts
       `final_live_rows == final_scan_count`

The end of
`scripts/vacuum_concurrency_scratch.sh:216–224` reads:

```bash
final_live_rows="$(run_sql ... "SELECT count(*) FROM ${table_name}")"
final_scan_count="$(
    run_sql ... "SELECT tests.tqhnsw_debug_scan_result_count(...)"
)"

echo "vacuum concurrency harness passed"
echo "duration_seconds=${duration}"
echo "final_live_rows=${final_live_rows}"
echo "final_scan_count=${final_scan_count}"
```

It prints both values and exits 0. If graph repair silently loses a
row — the exact bug class A6 is supposed to catch — the harness would
print `final_live_rows=10844, final_scan_count=10843` and still report
`vacuum concurrency harness passed`. The match the packet quotes
(10844 = 10844) is *shown in the text of the packet*, but the harness
itself does not enforce equality.

This is the single biggest gap in the final A6 slice. Please add:

```bash
if (( final_live_rows != final_scan_count )); then
    echo "final_live_rows (${final_live_rows}) != final_scan_count (${final_scan_count})" >&2
    exit 1
fi
```

Stronger still: after quiescing writes (kill insert_worker and
vacuum_worker), issue `VACUUM (ANALYZE) ${table_name}` once more, then
assert the equality on that post-quiesce snapshot. The current midair
snapshot is subject to MVCC skew between the two `count(*)` calls — a
tiny difference might be legitimately explained by an in-flight
insert, and that is exactly the kind of ambiguity that would tempt
someone to weaken the assertion later. A post-quiesce equality check
is unambiguous.

Secondary: the scan workers assert only `result_count > 0`
(`scripts/vacuum_concurrency_scratch.sh:154–157`). During a vacuum that
correctly unlinks dead nodes, that does technically catch "graph went
empty" but not "graph lost half its rows but a few rows remain
reachable". A per-iteration comparison against the concurrent live
count would be racy, but the post-quiesce final assertion would solve
this cleanly.

### 4. Harness throughput is dominated by psql spawn cost, not by the
       index operations under test

Every `run_sql` call (`scripts/vacuum_concurrency_scratch.sh:56–60`)
execs `scripts/pg17_scratch_psql.sh` which in turn `exec`s a fresh
`psql` process (`scripts/pg17_scratch_psql.sh:80–82`). So each worker
iteration is:

- insert_worker: 1 psql spawn per iteration
- vacuum_worker: 3 psql spawns per iteration (delete, vacuum, count)
- scan_worker: 1 psql spawn per iteration

On my back-of-the-envelope model of pgrx scratch cluster psql startup
(~30–70ms cold per call), a 60-second run probably only gets a few
hundred iterations per worker, not thousands. The packet's
"60 seconds of concurrent load" is narrower than it sounds —
effectively, the harness proves that the concurrent mix *does not
corrupt state over ~hundreds of cycles*, which is meaningful but not a
stress test.

Two concrete improvements, neither blocking:

- Replace per-iteration `run_sql` with a single long-lived psql per
  worker using `psql --single-transaction=off -f <(fifo)` or just a
  Python/`psycopg` worker — throughput probably jumps 10–50x and the
  same 60-second budget becomes a real soak.
- Report iterations-per-worker at the end (the worker functions already
  count them in `iterations=...`, but the harness discards that echo in
  favor of just exit status). Printing the counts would expose this
  regression class in future runs.

As a `TC-215` proof, what the harness does today is "no obvious
breakage under concurrent load" rather than "resilient under pressure".
The packet's framing ("concurrency validation") is honest about that,
but I'd flag this for the A6 follow-up list so nobody later assumes
this harness is a soak test.

### 5. Harness delete pattern and query set have low diversity

- `vacuum_worker` always deletes `ORDER BY id LIMIT 2`
  (`scripts/vacuum_concurrency_scratch.sh:167–173`). This means pass-2
  unlink / replacement fill only ever removes the lowest-id tail of the
  table. INSERT-side allocation always appends new ids, so the delete
  window and insert window are far apart in id-space. Whether they are
  far apart in *vector space* (i.e., actually neighbors in the graph)
  is a function of the random insert worker vectors, not anything the
  harness controls.

  The practical consequence: the harness exercises "delete something,
  repair, vacuum, repeat" as a pipeline but does not reliably create
  the harder scenario "delete a row whose persisted neighbor tuple is
  *currently being inserted into* by the concurrent insert_worker",
  which is the race slice-237's duplicate-reanimation guard was meant
  to protect against.

  A better delete pattern for stress is `ORDER BY random() LIMIT k`, or
  occasional deletes targeting rows with high connectivity. Neither is
  needed for A6 to close, but I'd note it in the A6 followups list.

- The two scan workers use fixed query vectors
  (`scripts/vacuum_concurrency_scratch.sh:200–201`). That's fine for
  correctness — the scan path is query-agnostic — and does increase
  cache residency, which is arguably a harder case for graph mutation
  races (the scan revisits the same entry point repeatedly). Leave as
  is.

### 6. Edge-case walkthrough across the three-pass stack

I re-walked pass 1 / pass 2 / pass 3 looking for anything the harness
would miss.

**Pass 1 (mark / strip heap TIDs).** Walks every data page under
`ShareUpdateExclusiveLock` + per-page `EXCLUSIVE`, rewrites element
tuples with filtered `heaptids`, writes WAL via `GenericXLogTxn`. The
only concurrent writer that could observe a half-stripped page is
another vacuum, which the `ShareUpdateExclusiveLock` on the table
already excludes. `aminsert` holds the target page `EXCLUSIVE` for its
own write window, so a pass-1 page rewrite is serialized against any
concurrent append on the same page — good.

Pass-1 also correctly handles the "element becomes fully dead as of
pass-1" case: `heaptids.is_empty()` elements are *not* immediately
deleted in pass 1; they are left for pass 3. That is intentional
because pass 2 still needs to read the persisted neighbor edges in
order to plan unlinks, and toggling `deleted = true` mid-pass-1 would
confuse pass 2's scan.

**Pass 2 (unlink + replacement fill).** Planned outside any write lock,
applied under per-page `EXCLUSIVE`, in ascending physical order
(ADR-027 rules 1–4). The scan that populates `LayerRepairRequest`s
only runs on live elements and only records neighbor TIDs that pass-1
already marked dead (`heaptids.is_empty()`). So:

- New inserts during pass 2 cannot produce *new* dead-neighbor refs,
  because `aminsert` does not insert with empty heaptids. A newly
  inserted element's backlink cannot be pointing at a dead element
  because the dead element's tid set was fixed by pass 1.
- Deletes observed by a later vacuum — but not *this* vacuum — cannot
  affect this pass at all, because `heaptids` only becomes empty via
  pass 1.
- The repair planner reads neighbor tuples without holding an
  EXCLUSIVE lock (ADR-027 rule 1). That is safe because the apply
  phase re-validates state under the lock: `apply_repair_plans_on_page`
  clears any deleted refs still present and only writes into `INVALID`
  slots. If a concurrent insert filled one of the slots we planned to
  write, the fill-only check rejects our write for that slot — we lose
  the planned replacement, but correctness is preserved (the slot just
  holds the insert-time neighbor instead).

  Worth confirming once with a targeted unit test: "planner picks slot
  k on page P; insert wins slot k first; apply phase skips slot k and
  moves on" — I did not find this exact test in the existing coverage.
  The existing fill-only regression exercises the *rule* but not the
  *race*. For A6 closure this is acceptable; file it under followups.

- Pass-2 never writes the metadata page. That is ADR-027 rule 7, and
  is crucial because the metadata-last rule from ADR-026 makes the
  metadata page the ordering anchor for inserts — vacuum must not
  contend on it.

**Pass 3 (finalize).** Only flips elements with `heaptids.is_empty()`
and no live layer-0 inbound edges to `deleted = true`. The
duplicate-reanimation guard (slice 237) means that finalize will *not*
flip an element whose heaptid set is empty but whose code still
appears in a live backlinked neighbor — which is the case a concurrent
insert could create if it coalesced into a stale element. I re-read
the guard and it's still correct: the check walks all live neighbors
at layer 0 and fails finalization if any of them still reference the
candidate. There is one subtle question about whether the guard is
sufficient for multi-layer inbound edges, but since pass 2 already
cleared upper-layer references in the unlink phase, by the time we get
to finalize only layer-0 references could still be pointing at us, and
those are what the guard checks. So the layer-0-only check is
sufficient *given pass 2 has run*. That's a load-bearing invariant —
worth a comment on the finalize guard noting "relies on pass 2 clearing
upper-layer refs first".

**WAL safety.** Every page rewrite lives inside a `GenericXLogTxn`
whose Drop impl aborts unfinished transactions (`src/am/wal.rs`).
Panics inside an apply phase will therefore not leak a half-written
WAL record. Confirmed by inspection.

### 7. Performance concerns across the full A6 stack

I don't think any of these are blocking, but they should be on the
record before A6 closes:

- **`collect_repair_requests` is O(live_elements × avg_page_line_pointers)
  per vacuum.** For every live element on every data page, we decode
  the persisted neighbor tuple and scan its layer slices. On a 1M-row
  index with ~500 elements per page (~2000 pages) this is a full index
  read per vacuum. There's no smarter way to do this without indexing
  the dead-ref set into a side structure, and the wall cost is
  dominated by the subsequent page rewrite anyway — but if vacuum
  latency ever becomes a problem on large indexes, this is the first
  place to look.

- **`plan_repair_replacements` runs a beam search per affected live
  node.** On a big vacuum (say, 10% of rows deleted on a 500k index)
  that's ~50k beam searches at `ef_construction`-ish candidate widths.
  Order-of-magnitude that's comparable to a small index build. Again,
  not wrong, just expensive. Worth noting in FR-022 as a known
  scale-out concern so it isn't discovered in production.

- **Linear top-up fallback** for upper layers is `filter(element.level
  >= planner.layer)` over the full index. At layer k, the set of
  candidates shrinks roughly geometrically (HNSW level distribution),
  so the top-up cost drops sharply with `k`. That's good, and the
  current implementation correctly benefits from it. No concern.

- **Harness throughput** is psql-spawn-bound as discussed in finding 4.
  The harness `duration_seconds` value is not a useful proxy for the
  number of concurrent operations executed.

## Answers to the packet's review questions

> Is the dedicated scratch-database harness the right durability
> boundary for TC-215, or should this proof move elsewhere later?

As a *checkpoint* proof, yes — it's honest about being a multi-session
scratch harness rather than an in-process `#[pg_test]`, and the
tradeoff is defensible: a real MVCC concurrency story needs multiple
backends, which the `#[pg_test]` runtime does not support cleanly. The
scratch-database isolation (as opposed to using `postgres`) is
correct and the packet's stated reason — avoiding stale-extension-state
false failures across reinstall cycles — is exactly right.

As a *long-term* TC-215 home, I would push harder. Specifically:

- The harness should be runnable from CI, not only from a manually
  started scratch cluster. A short-lived `initdb` + `pg_ctl start` +
  harness + `pg_ctl stop` wrapper would make this a real gate.
- The missing `final_live_rows == final_scan_count` assertion
  (finding 3) must land before this is treated as the primary
  regression for `TC-215`, or the harness is false coverage.
- A stress-mode variant (psql persistent connection, random deletes,
  higher row counts, iteration-count reporting) should be considered
  before promising anyone that TC-215 actually soaks the concurrency
  story.

For now the scratch harness closes A6, but `TC-215` should be recorded
as "covered by a smoke harness, not a soak", and promoting it to a soak
should be on the B-lane backlog.

> Is the `pg_test`-only SQL count wrapper an acceptable narrow seam for
> driving real tqhnsw scans from the harness?

Yes. The wrapper is minimal, gated on `pg_test`, schema-scoped under
`tests.`, and exercises the real `ambeginscan / amrescan / amgettuple`
path via an existing debug helper. The one regression asserts the SQL
wrapper matches the Rust helper, so future drift is caught. I'd only
ask for the small open/close micro-cleanup from finding 1, and only
when someone is already touching this code.

> Does any A6 concurrency risk remain materially unexercised by the
> current INSERT + scan + VACUUM worker mix?

Three are visible but not materially unexercised:

1. **Planner-vs-insert race on a single slot.** The pass-2 planner
   reads without a lock, then applies under lock with fill-only
   semantics. A concurrent insert grabbing the same slot first is a
   legitimate race. The fill-only contract makes it safe, but the
   harness does not deliberately force this race and we have no unit
   test that drives it directly. File under "desirable followup".

2. **Delete-then-insert-same-vector race.** The duplicate-reanimation
   guard from slice 237 is specifically designed for this case. The
   harness does not deliberately trigger it (inserts use random
   vectors; deletes target `ORDER BY id LIMIT 2`). The guard's unit
   test covers correctness, but the concurrent story is not directly
   exercised. Low probability of regression, but this is the exact
   case the guard exists to protect.

3. **Multi-layer write batching under single EXCLUSIVE.**
   `apply_repair_plans_on_page` now writes multiple layer slices of the
   same neighbor tuple in one EXCLUSIVE-lock window. ADR-027 still
   describes this correctly (the rule is "one data page EXCLUSIVE at a
   time", not "one layer slice"), but the ADR doesn't explicitly
   document that multi-layer batching is legal. A sentence in ADR-027
   noting "multiple logical-layer slices of the same persisted
   neighbor tuple may be rewritten in a single EXCLUSIVE window" would
   close the doc gap.

None of these are concurrency *risks* in the code — the invariants
hold. They are unexercised *in the harness*, which is a coverage gap
rather than a correctness gap. A6 can close on the current mix; the
unexercised cases go on the followups list.

**Close A6.** The concurrency slice lands the required seam, the
harness structurally exercises the right mix, and the invariants hold
when I trace the three-pass path under concurrent INSERT + scan +
VACUUM. The SQL wrapper and its regression are well-scoped.
