## Answers to Review Questions

### Is the scheduler-first / vector-fallback policy the right intermediate contract?

**Yes.** Verified in `recompute_candidate_frontier_head` (scan.rs:618-647):

```
1. Try: beam scheduler peek_best() → find_candidate_frontier_index() → set head
2. Fallback: linear scan of frontier Vec → set head
```

The beam scheduler's `peek_best()` (O(1) on `BinaryHeap`) returns the best queued node. `find_candidate_frontier_index` (scan.rs:350-360) maps the beam node back to a Vec index. If the mapping fails (beam and Vec are out of sync), the fallback linear scan picks the best valid candidate from the Vec directly.

This dual-path design is correct for the transition period: the beam scheduler becomes more authoritative with each slice while the Vec scan provides a safety net. The fallback will eventually be dead code once the dual-structure is unified.

### Does the unit test capture the ownership shift?

Verified: the unit test `consume_candidate_frontier_head_reselects_then_clears` (scan.rs:1095+) initializes both the frontier Vec and the beam scheduler (`reset_bootstrap_expansion_state`), seeds candidates, and verifies consumption behavior. The test exercises the scheduler-first path implicitly because `recompute_candidate_frontier_head` is called after each consume and the beam scheduler is populated.

A more targeted test that explicitly verifies "scheduler disagrees with Vec order → scheduler wins" would strengthen the ownership-shift contract, but is not blocking.

### Should remaining frontier-head logic move fully behind the search seam?

**Not yet.** The `candidate_frontier_head: Option<usize>` is still needed because the frontier Vec is the materialization source (the Vec holds the `ScanCandidate` structs with `source_tid` and other scan-specific fields). The beam scheduler holds `BeamCandidate<ItemPointer>` which only has `node`, `score`, `source`. Until the beam scheduler carries all the fields needed for materialization, the Vec index is necessary.

The path forward: either enrich `BeamCandidate` to carry scan-specific fields, or make `consume_candidate_frontier_head` operate entirely through `pop_best` + `find_candidate_frontier_index` (which is already the primary path at scan.rs:650-653).

## Additional Findings

No correctness issues found. The scheduler-first head selection is a clean ownership shift.
