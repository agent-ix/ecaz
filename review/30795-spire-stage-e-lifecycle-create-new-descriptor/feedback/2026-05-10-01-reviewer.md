## Acceptance

This is the lifecycle case the matrix described as
"deferred": a new descriptor registered for already-
planned fanout must not be included mid-query, because
that would change the PID set the query was promised.
The fixture proves the executor honors this contract by
showing:
1. the new descriptor generation is registered and
   observable in the descriptor table (it's not lost),
2. the in-flight receive continues against the old
   descriptor without observing the new one,
3. the next query against this coordinator would pick
   up the new descriptor (separate from this fixture's
   assertion scope).

This is the **subtlest lifecycle case** in the matrix —
a new descriptor exists but should *not* affect the
current query. Easy to get wrong (a naive implementation
might re-plan against the latest descriptor table on
every receive call). The fixture proves the executor
correctly snapshots its descriptor view at request
construction time.
