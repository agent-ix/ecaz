# 30734 — SPIRE production receive request state review

Validation pass on `cd7b1919` (`Add SPIRE production receive
request state`). Accepts the receive-state composition slice.

## What I checked

- `SpireRemoteProductionDispatch` retains selected PIDs,
  `conninfo_secret_name`, and `remote_index_regclass` copied from
  the dispatch plan row. This is the right ownership boundary —
  the executor reuses the dispatch plan's already-validated values
  rather than re-reading diagnostic rows.
- `SpireRemoteFanoutExecutor::compact_candidate_receive_requests(...)`
  (line 2839) builds internal async receive-adapter requests from
  `TransportReady` dispatches. C5 AM scan can call this method
  rather than reconstruct the request shape.
- Raw conninfo isolation: dispatch state stores only the
  `conninfo_secret_name`; raw conninfo is resolved only when
  building internal receive-adapter requests, never persisted on
  the dispatch state. This preserves ADR-058's no-raw-conninfo
  diagnostic boundary into production state.
- Missing secret resolution becomes a per-dispatch
  `CandidateReceiveFailed` transition with failure category
  `requires_conninfo_secret_resolution` (line 31 constant; test
  assertion at line 7135). Per-dispatch isolation matches the
  packet 30729 invariant.
- `consistency_mode` changed from `&'static str` to owned
  `String`. Right call — AM scan callers in C5 will likely have a
  dynamic consistency mode (parsed from query options), and
  forcing static literals would push the conversion responsibility
  outward.
- Phase 11 Stage C checklist item for dispatch-owned receive
  request state flips.
- Test coverage: `compact_candidate_receive_requests` exercised at
  lines 7081 and 7122.

## Answers to the packet's review questions

- **Should the executor own selected PIDs, secret reference, and
  remote regclass as dispatch state?** Yes. These are derived from
  the dispatch plan row at admission time and don't change
  thereafter. Owning them on the dispatch removes the temptation
  for C5 to maintain parallel `Vec<plan_row>` state outside the
  executor.
- **Resolving raw conninfo only during internal receive-request
  build preserves the no-raw-conninfo diagnostic boundary?** Yes.
  ADR-058 forbids raw conninfo through SQL surfaces; storing only
  `conninfo_secret_name` on the dispatch and resolving in the
  receive-request build path keeps that contract uniform across
  diagnostic and production paths.
- **Missing secret isolation should become `CandidateReceiveFailed`
  rather than aborting the whole receive request build?** Yes.
  Aborting the whole build would break the per-node isolation
  invariant from packet 30729 — one missing secret on one node
  would suppress receive on every other ready node. Per-dispatch
  failure is the right shape.

## Status

Closed. Production receive request state is composable; C5 AM
scan can call the executor methods rather than duplicating
bookkeeping.
