# Feedback: Direct Discovered-Candidate Beam Seeding

Request:
- `review/83-direct-discovered-candidate-beam-seeding.md`

**Reviewer:** Claude (Opus)
**Date:** 2026-04-05

## Response to Review Focus

### Is direct beam seeding at discovery time the right next ownership shift?

**Yes.** `seed_discovered_candidates` (scan.rs:556-577) adds new candidates to both the frontier Vec and the beam scheduler atomically. The function:
1. Filters for `score_valid` candidates
2. Extends the frontier Vec (scan.rs:568)
3. Marks each candidate visited (scan.rs:569-571)
4. Seeds the beam via `seed_many` with the same candidates mapped through `scan_candidate_to_beam_candidate` (scan.rs:572-576)

This eliminates the previous "discover in Vec, then scrape into beam" pattern. The beam now has immediate knowledge of all frontier candidates at the moment they're discovered, which means `peek_best()` and `expand_one()` always reflect the complete frontier state. No stale-beam window.

The helper is called from two sites:
- `refill_candidate_frontier_from_source` (scan.rs:712) — for adjacency-discovered successors during refill
- `initialize_scan_entry_candidate` (scan.rs:494) — for the initial entry candidate

Both are the correct discovery points.

### Should remaining vector-tail reseeding logic be removed?

**Already done.** `top_up_bootstrap_frontier` (scan.rs:603-631) no longer has any Vec-tail diffing or reseeding. The beam scheduler is consulted directly via `expand_one()` (scan.rs:617) for ordering, and `seed_discovered_candidates` handles beam population at discovery time. The old "before/after Vec len, reseed appended range" pattern is gone.

The only remaining vector-to-beam sync points are:
- `seed_existing_frontier_into_expansion` (scan.rs:579-587) — for helpers that start from a prebuilt frontier (addressed in review 84)
- `forget_queued` calls in `consume_candidate_frontier_head` (scan.rs:665) — to remove consumed candidates from the beam

Both are structurally necessary and well-placed.

### Should `seed_discovered_candidates` move behind a search-facing seam?

**Not yet.** The function needs to touch both the frontier Vec and the beam scheduler, which are both scan-owned state in `TqScanOpaque`. Moving it behind a search seam would require either:
1. The search module managing the Vec (premature — the Vec is still the data-carrying container)
2. A callback interface where the search module notifies scan of new candidates (overengineered for the current architecture)

The right time to move this is when the beam scheduler becomes the primary container and the Vec is eliminated. That's a later transition step.

## Additional Findings

No issues found. Clean consolidation of the dual-container seeding path.
