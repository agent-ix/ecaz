# Review: SPIRE DELETE Not-Found Idempotence (30920)

## Summary

Closes 30896 H2. ADR-069 picks the right policy: **DELETE-not-found
returns zero affected rows as idempotent success**, matching PG's
default DELETE semantics. `ec_spire_prepare_coordinator_delete_tuple_payload(...)`:

- Returns `delete_not_found_noop` when the placement row is
  already absent (no remote dispatch, zero rows affected).
- If placement exists but the owning remote heap row is gone:
  removes the placement row and returns zero rows.
- Accepts zero local or remote deleted rows as success.
- Preserves the hard error if `count > 1` (uniqueness violation,
  schema drift — these should still fail loudly).
- Reports local vs remote not-found statuses distinctly so
  diagnostics aren't flattened.

Placement-row deletion is now idempotent (`placement_deleted =
false` instead of raise).

PG18 fixture covers both missing placement and stale local
placement cleanup paths.

The three review-focus questions:
- Idempotent not-found behavior matches ADR-069 v1 policy: yes.
- No-placement sentinel row (`node_id = -1`, `served_epoch = 0`,
  `prepared_gid = 'none'`) acceptable for the helper surface:
  see findings below — minor.
- Allowing zero remote deletes as prepared success is the right
  v1 behavior: yes — coordinator placement cleanup happens
  regardless, so the placement directory stays consistent with
  reality even when the remote already lost the row.

## Findings

### P3 — Sentinel row uses `node_id = -1`

The catalog's `CHECK (node_id >= 0)` constraint (per 30824) is on
the `ec_spire_placement` table, not on the helper's return shape.
The helper returning `node_id = -1` in its result row is fine
because the result is a SQL function row, not a placement-table
INSERT. Worth a one-line comment in the helper or its docs that
this sentinel is shape-compatible with the table's columns but
deliberately sentinel (negative) so callers can detect "no
routing happened" without a separate boolean.

### P3 — `count > 1` still hard-errors

Right call. PK uniqueness is a structural invariant of the
distributed table; `count > 1` means schema drift or PK
constraint violation on the remote, both of which should
surface loudly rather than silently delete extra rows.

### P3 — Remote not-found vs placement not-found distinct

Operator monitoring can now distinguish "the row was already
gone on the remote" from "we never had a placement entry for
this PK" — useful because they signal different upstream
failure modes (concurrent remote delete vs application bug).

## Reviewer recommendation

Accept. Closes H2 cleanly with PG-aligned semantics.
