# Review: SPIRE JSON Production Path Retirement (30977)

## Summary

**Closes Phase 12.2 JSON endpoint production-path retirement
row.** Production tuple-payload dispatch can no longer
select the JSON SQL template:

- legacy SQL endpoint
  `ec_spire_remote_search_tuple_payload(...)` retained for
  compatibility/measurement;
- `ec_spire.remote_tuple_transport = json_tuple_payload_v1`
  GUC value retained for measurement;
- production dispatch reaches typed only — non-typed-ready
  endpoints fail closed with the existing endpoint-identity
  failure category;
- new `remote_tuple_payload_production_sql(...)` enforces
  the typed-only production branch;
- unit test asserts typed-ready selects typed; JSON-default
  or missing-capability fail closed.

`serde_json` retention noted (DML CustomScan JSON update
payload + JSONB manifest surfaces still need it).

## Findings

### P1 — Right shape for retirement

The retirement is **production-path-only**, not endpoint
removal. Three layers preserved:

1. **SQL endpoint** stays callable for measurement
   reproducibility (30975 used it for the JSON arm of the
   typed-vs-JSON comparison; future re-measurements can
   too).
2. **GUC value** stays selectable so operators can force
   the legacy path for diagnostic comparison.
3. **Production dispatch** can no longer reach JSON.

This is exactly the right scope for retirement under the
30975 finding. The original 30896 P1 retirement argument
was performance-based; 30975 measured no speedup. Retiring
the **production path** still makes sense on:

- code-surface reduction (one path through
  `remote_tuple_payload_production_sql`, not two);
- type-system correctness (typed preserves binary fidelity
  for arrays/composites better than JSON text);
- cutover-completion (the typed cutover from 30963 has
  type-class coverage; production should commit to it).

But removing the endpoint entirely would defeat the 30975
measurement reproducibility, so keeping the SQL endpoint
+ GUC value is the right balance. **Honest retirement
shape.**

### P1 — Fail-closed for non-typed-ready endpoints

Critical detail: when production tuple-payload reaches an
endpoint that doesn't prefer ready
`pg_binary_attr_v1`, the coordinator now **fails closed**
with the existing endpoint-identity failure category.

This is the right failure mode. Three reasons:

- **No silent fallback**: if production silently fell back
  to JSON, the retirement would be cosmetic — the JSON
  path would still be reachable in production, just
  hidden.
- **Operator visibility**: the existing endpoint-identity
  failure category is what operators already grep for. No
  new SQLSTATE to learn.
- **Migration window contract**: per ADR-069, the
  typed cutover requires remote endpoints to advertise
  `pg_binary_attr_v1` ready. A coordinator hitting a
  remote that doesn't advertise this is in an
  unsupported deployment configuration, and failing
  closed surfaces that misconfiguration immediately.

The existing 30951 libpq runbook + 30954 degraded skip
report should already document the operator response to
this failure category. Worth a tracker note that the
retirement may produce *new occurrences* of this existing
category in deployments where some remotes haven't been
upgraded — operators should expect and triage that
specifically.

### P2 — `serde_json` retention note

The note that `serde_json` remains required for:

- DML CustomScan JSON update payload handling;
- JSONB manifest surfaces;

is the right honest disclosure. Phase 12.2's "drop
`serde_json` if no other runtime path needs it" exit
clause was conditional, and other paths still need it.
Retiring `serde_json` would be a separate, larger refactor
across DML and manifest code paths.

The conditional clause is satisfied: production tuple-
payload dispatch no longer needs JSON, but the dependency
removal gate isn't met because of the other consumers.
Tracker reflects this correctly.

### P2 — Test coverage shape

Three unit tests cover the load-bearing paths:

- `remote_tuple_transport_auto_uses_endpoint_default`:
  default behavior preserved.
- `remote_tuple_transport_session_override_keeps_capability_gate`:
  the GUC override still respects capability gating (so
  forcing typed against a non-typed remote still fails
  closed, not bypasses).
- `remote_tuple_payload_production_sql_requires_typed_transport`:
  the production branch can't select JSON.

These three close the regression surface for the three most
likely future bugs:

- "we accidentally added a JSON code path back";
- "we made the GUC bypass the capability gate";
- "we made the default behavior diverge from
  capability-driven negotiation."

Right test coverage shape.

### P3 — Compatibility window vs immediate retirement

Per the ADR-069 exit criterion noted in 30896 P1, JSON
fallback "may remain production-reachable for one
minor-version compatibility window." This packet retires
production immediately, ahead of the compatibility window.

That's the right call IF the operator deployment
expectation is that all remotes upgrade together. If
there's a possibility of mixed-version deployments where
some remotes haven't shipped typed yet, those will now
fail closed instead of falling back.

Worth confirming the operator runbook (30951) names this
expectation: "all remotes must advertise
`pg_binary_attr_v1` ready before upgrading the
coordinator past version X." If it doesn't, mixed-version
deployments will produce surprise failures.

If the deployment expectation is "upgrade everything
together," fine. If it's "rolling upgrade with mixed
versions allowed," the retirement should add a
compatibility-window flag (e.g., `ec_spire.allow_legacy_json_dispatch`)
that defaults off but lets operators flip it for the
duration of a rolling upgrade.

This is the one detail worth confirming before the packet
lands as the final word on retirement.

## Reviewer recommendation

Accept. **Closes Phase 12.2 fully.** Right retirement
shape (production path only, endpoint+GUC kept for
measurement/compat), right failure mode (fail-closed via
existing category), right test coverage. The one detail
worth confirming is whether the deployment expectation is
"upgrade everything together" — if mixed-version rolling
upgrades are supported, a temporary GUC bypass for the
window may be needed.
