## Answers to the packet's review questions

- **Cache key sufficient for Stage C without being overbroad?** Yes.
  The five-field key plus descriptor-generation binding is exactly
  what the receive-side validator already cross-checks; nothing in
  the validator is left out, so a hit on the cache is provably
  equivalent to a fresh preflight at the moment the entry was
  inserted. The other identity fields are bound (must match) rather
  than keyed, which is the right shape â€” they are derivable from
  `(node_id, remote_index_identity)` for a given descriptor
  generation.
- **Invalidation trigger set complete?** Yes. The 11 triggers cover
  every input to either the descriptor or the endpoint identity that
  the validator reads. The local-extension-upgrade trigger is
  particularly worth calling out because it would otherwise be a
  silent-correctness hole on rolling extension upgrades.
- **Backend-local bounded cache as first implementation?** Yes.
  PostgreSQL backend-local memory has well-defined lifetime
  (statement / transaction / backend) and avoids cross-backend
  invalidation infrastructure. A shared cache should only land if a
  measured benefit appears in Stage G harness data; otherwise the
  invalidation surface is more risk than benefit.
