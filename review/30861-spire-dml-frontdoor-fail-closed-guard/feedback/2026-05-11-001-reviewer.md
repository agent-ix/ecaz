# Review: SPIRE DML Frontdoor Fail-Closed Guard (30861)

## Summary

Critical safety milestone. The hook now **errors** at planner time
when an unsupported DML shape targets a SPIRE front-door candidate,
instead of falling through to the coordinator heap path (which
would silently miss remote-owned rows — the wrong-results trap I
flagged in 30843).

The guard:
- Computes the replacement decision via the catalog-backed loader.
- Classifies the action via `dml_frontdoor_hook_action(decision)`:
  - `pass_through_until_rewrite` if `decision.supported`;
  - `planner_error_fail_closed` if the relation is a SPIRE
    front-door candidate (`index_oid != InvalidOid` OR
    `kind == "relation_context_error"`);
  - `pass_through_not_spire_frontdoor` otherwise.
- Raises `ERRCODE_FEATURE_NOT_SUPPORTED` with the decision's error
  + hint for the fail-closed case.

Hook status surface gains `unsupported_shape_fail_closed_enabled =
true` and `last_hook_action`, so operators see what happened on
the last classification.

PG18 fixture proves an embedding-column UPDATE on an indexed
distributed table fails closed with the exact ADR-069 error/hint;
the same UPDATE shape on a plain (non-`ec_spire`) table still
passes through.

The three review-focus questions:
- Guard only errors for SPIRE front-door candidates, not ordinary
  heap tables: yes — the
  `dml_frontdoor_decision_is_spire_frontdoor_candidate` predicate
  gates strictly on `index_oid != InvalidOid` (or context-load
  error).
- Supported ADR-069 shapes still pass through while plan rewriting
  is disabled: yes — `pass_through_until_rewrite` is the action
  for `decision.supported`.
- Hook-status fields give visibility into the interim guard:
  yes.

## Findings

### P2 — `relation_context_error` triggers planner-error fail-closed

When the catalog loader fails (corrupt catalog, missing index
metadata, etc.), the predicate
`decision.kind == "relation_context_error"` fires the planner
error. That's defensible — failing to read context is suspicious
enough to fail loudly — but in some operational scenarios (e.g.,
relation just dropped between hook entry and lock acquisition) the
fail-closed could surface as a confusing error during otherwise-
normal CASCADE drops or schema migrations.

Worth a focused fixture that exercises the
`relation_context_error` → planner-error path and confirms the
error message mentions context-load failure (so operators
distinguish "I don't support this DML shape" from "I couldn't
read your catalog"). The current decision row uses the message
`"ec_spire_distributed: relation context could not be loaded"`
which is accurate but worth confirming the hook surfaces this
verbatim in the planner error.

### P2 — Guard runs *before* the previous planner_hook delegation

The hook's flow is: observe → fail-closed-or-pass → delegate to
previous_hook / standard_planner. So if a previous hook installed
by another extension would normally rewrite the same query, the
SPIRE guard preempts it for unsupported shapes targeting SPIRE
candidates. That's the right ordering for safety (SPIRE has the
strongest opinion about these queries) but worth pinning in the
ADR or hook docs that the planner-error fires before any other
extension's rewrite chance.

### P3 — `last_hook_action` static is per-backend

Same backend-local visibility as the other hook statics. Useful
for "what did my session see" diagnostics; doesn't aggregate
across backends. Consistent with the existing pattern.

### P3 — `unsupported_shape_fail_closed_enabled` always returns true

The field is a static `true` because the guard is
unconditionally enabled when the hook is installed. Worth keeping
the field anyway (consistent diagnostic shape for future
configurability), but the value is informational.

### P3 — `dml_frontdoor_raise_planner_error` builds `ErrorReport`

Mirrors the embedding-update rejection in
`ec_spire_forward_coordinator_update_tuple_payload`. Same SQLSTATE
(`ERRCODE_FEATURE_NOT_SUPPORTED` / `0A000`). Operators see
consistent error shape regardless of whether the rejection
happens at the planner or at the primitive layer.

## Reviewer recommendation

Accept. **This is the safety milestone.** Wrong-results trap is
now closed: an operator running an unsupported DML shape against
a SPIRE-marked table gets a loud ADR-069 error at plan time
instead of silently missing remote-owned rows. The plan-rewrite
slice that follows now lands on a safe foundation — even if it's
buggy or incomplete, the worst case is "supported shapes don't
get rewritten yet" not "unsupported shapes return wrong results."

The fail-closed guard plus the typed decision row plus the
catalog-safe relation context plus the shape classifier — all the
pieces are now in place for the actual plan-rewrite slice to land
cleanly.
