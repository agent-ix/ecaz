# Review: SPIRE Boundary Replica Manifest Freshness (30958)

## Summary

Closes Phase 12.7 boundary-replica freshness fixture row.
`test_ec_spire_boundary_replica_manifest_freshness_sql`
exercises `ec_spire_remote_epoch_manifest_freshness()`
through three states:

- **missing**: pre-persistence →
  `requires_remote_epoch_manifest_persistence`;
- **ready**: post-persistence;
- **stale**: drift the persisted entry →
  `stale_remote_epoch_manifest` + next action
  `refresh_remote_epoch_manifest`.

Pairs the freshness surface with the boundary identity
snapshot from 30957 to prove boundary-replica metadata is
actually being exercised.

## Findings

### P2 — Three-state coverage is the right matrix

Missing → ready → stale is the load-bearing freshness state
machine for operator workflow:

- **Missing**: operator hasn't run persistence yet (fresh
  install or post-rebuild). Diagnostic must surface a clear
  next-action.
- **Ready**: steady state. Diagnostic must NOT false-alarm
  here.
- **Stale**: operator did persistence but coordinator/remote
  drifted (DDL, manual catalog edit, version bump).
  Diagnostic must surface the corrective next-action.

These three states cover the operator state machine. A
future fourth state worth considering is **partial** — some
remotes persisted, some not — but that's a multi-remote
fixture and likely Phase 12.7 multi-instance work, not this
single-fixture row.

### P2 — Pairing with identity snapshot

Calling `ec_spire_index_boundary_replica_identity_snapshot`
mid-fixture to confirm boundary-replica metadata is present
is the right cross-check. Without it, a freshness fixture
could pass against a non-boundary-replica index by accident
(freshness API doesn't itself require boundary replicas).
The pairing pins the fixture to the actual feature under
test. Good.

### P2 — Drift mechanism

The fixture drifts the persisted catalog entry to trigger
`stale_remote_epoch_manifest`. Worth confirming the drift
mechanism mirrors the realistic drift causes (epoch advance
on remote, manual catalog UPDATE for testing) rather than
deleting/corrupting in a way the production stale path
won't see. From the request it sounds like a controlled
catalog edit, which is the right primitive — production
stale comes from epoch advance, and a controlled edit
simulates that without needing live epoch progression.

### P3 — Tracker row closure scope

Correctly does NOT claim:

- live remote object reads;
- degraded replica-placement diagnostics (30959 covers).

Closure is freshness-fixture only. Honest.

### P3 — `refresh_remote_epoch_manifest` next-action

The stale state surfaces `refresh_remote_epoch_manifest` as
the next action. Worth one sentence in the operator doc
naming the actual command/function the operator runs to
perform that refresh — "next action: X" is more useful when
operators know what X invokes.

## Reviewer recommendation

Accept. Three-state coverage with the right cross-check
pairing. Closes the Phase 12.7 freshness fixture row
honestly.
