# Feedback: A4 Real-Corpus Recall Lane

Request:
- `review/218-a4-real-corpus-recall-lane/request.md`

**Reviewer:** Claude (Opus)
**Date:** 2026-04-09

## Overall Assessment

The lane is well-architected and broadly correct. It cleanly resolves
review 217's contradiction, reuses the existing recall primitives
(`recall_fixture_ident`, `ctid_id_map`, `RECALL_GATE_CONFIGS`,
`debug_gettuple_scan_heap_tids`) consistently with the synthetic gate, and
the idempotency / byte-identical-rerun contract is real. The packet's
"open question for the reviewer" framing on the gate target is exactly the
right time to flag it.

The findings below are ordered by severity. Items 1 and 2 should land
**before** the first real DBpedia run gets recorded as a review-218 result,
because both can change the headline number. Items 3–6 can land in a
single follow-up commit on the same branch.

## Critical

### 1. The default working subset has no canonical selection rule

`docs/RECALL_REAL_CORPUS.md:30-42` says the default is `tqhnsw_real_50k`
(50k corpus + 1k queries), but **does not say which 50k**. The whole
"credibility surface" argument from review 217 depends on the subset being
canonical: two reviewers staging "the same" fixture from the Qdrant 1M
parquet will pick different subsets and get different recall numbers, and
the gate becomes unreproducible across humans. The contract is currently
dataset-agnostic at the file level, which is good for the loader, but the
*gate fixture* needs to be pinned at the dataset level.

This is the single biggest gap in the packet. Concretely, the doc should:

- Specify selection method: e.g. "first 50k rows of the parquet ordered by
  Qdrant `id` ascending; queries are rows 50_000..51_000".
- Pin a manifest: corpus row count + query row count + a SHA-256 of the
  concatenated `id\t<json>` lines (or of the `source` column bytes after
  f64→f32 cast). Commit the manifest to the repo even though the corpus
  binaries stay out.
- Promise that future loader runs that produce a different manifest hash
  will refuse to rerun the gate without an explicit override.

Without this, the lane is *technically* dataset-agnostic but the A4 gate
is not actually reproducible — which is exactly the failure mode review
217 was about.

The companion missing piece is a tiny **converter script** —
`scripts/qdrant_dbpedia_to_tsv.py` or similar — that takes the Qdrant
parquet and emits the canonical `tqhnsw_real_50k_corpus.tsv` /
`tqhnsw_real_50k_queries.tsv` against a documented selection rule.
Otherwise every user reinvents their own parquet→TSV bridge and the
manifest goes stale immediately. The repo doesn't ship the corpus
binaries; it should still ship the deterministic recipe that produces
them.

### 2. `mean_abs_score_error` semantics may not match NFR-003

In `probe_graph_scan_recall_external_summary_for_relation`, the MAE block:

```rust
let predicted_top_10_scores: Vec<f32> = predicted_top_10_ids
    .iter()
    .map(|idx| dot_product(query, &corpus[*idx as usize]))
    .collect();
let truth_top_10_scores: Vec<f32> =
    truth.iter().take(RECALL_K).map(|(_, s)| *s).collect();
// ... position-wise abs diff
```

This computes "how much worse is the score at each *rank position*", i.e.
the L1 distance between two sorted score profiles. That is a meaningful
quality metric, but it is **not** the conventional "MAE between predicted
and true scores of the same items".

The conventional NFR-style `mean_abs_score_error` for an ANN system is one
of:

- **(a) Per-item quantizer fidelity:** for each predicted item,
  `|quantized_score(item) − fp32_score(item)|`. Tells you how much the
  quantizer distorts scores, item-by-item.
- **(b) Item-identity diff:** for the union of predicted and true top-k,
  `|fp32_score − fp32_score|`. Trivially zero, not useful.
- **(c) Score-curve L1 distance at matching ranks:** what is currently
  implemented. Tells you how close the predicted score profile is to the
  optimal score profile.

Each is defensible, but (a) and (c) tell you very different things, and
the `NFR-003` spec text presumably picks one. **Action:** open
`spec/non-functional/NFR-003-recall-quality.md`, find the MAE definition,
and either rename this column to match (e.g. `score_curve_l1_distance`)
or rewrite the computation to match (a). As it stands the column name and
the computation may diverge silently.

## High

### 3. `id_to_row_index_lookup` is rebuilt per query

In the per-query loop inside
`probe_graph_scan_recall_external_summary_for_relation`:

```rust
for (query, truth) in queries.iter().zip(ground_truth_top_k.iter()) {
    // ... graph scan ...
    // ... exact-quantized SPI fetch ...
    let id_to_row_index_lookup: HashMap<i64, i64> = row_index_to_id  // <-- per query
        .iter()
        .enumerate()
        .map(|(idx, id)| (*id, idx as i64))
        .collect();
    // ... use it for one query, drop it ...
}
```

This is `O(N·Q)` where it should be `O(N + Q)`. For 50k corpus × 1k
queries that is 50M HashMap inserts of redundant work. Hoist the map out
of the loop into the function prologue (right after `corpus_ids` is
built), use the same map type as `id_to_row_index` constructed for
`ctid_to_row_index`, or just merge the two: `corpus_ids` already has
everything needed.

### 4. Gate report reloads the corpus N times

`run_graph_scan_recall_gate_from_external` calls
`probe_graph_scan_recall_external_summary_for_relation` once per
`RECALL_GATE_CONFIGS` row. The probe **reloads the entire corpus and
queries from Postgres each time**, then **recomputes the brute-force fp32
truth from scratch each time**. For 50k×1k×1536 that is ~77 GFLOPs of dot
products per call repeated 4 times, plus ~600 MB of redundant SPI
transfers per gate run.

Recommended refactor: factor the probe into

```rust
struct ExternalRecallContext {
    corpus_ids: Vec<i64>,
    corpus: Vec<Vec<f32>>,
    queries: Vec<Vec<f32>>,
    ground_truth_top_k: Vec<Vec<(usize, f32)>>,
    ctid_to_row_index: HashMap<(u32, u16), usize>,
    id_to_row_index: HashMap<i64, usize>,
}

fn build_external_recall_context(
    corpus_table: &str,
    query_table: &str,
) -> ExternalRecallContext { ... }

fn probe_graph_scan_recall_external_summary_for_context(
    ctx: &ExternalRecallContext,
    corpus_table: &str,
    index_name: &str,
    m: i32,
    ef_search: i32,
) -> GraphScanRecallExternalSummaryRow { ... }
```

Then `run_graph_scan_recall_gate_from_external` builds the context once
and passes it to four probe calls. The single-config pg_extern wrapper
(`tqhnsw_graph_scan_recall_external_summary`) builds a one-shot context
and discards it — same shape as today, no API change. For the 50k×1k gate
this is the difference between minutes and tens of minutes.

This refactor also makes the smoke test's "byte-identical rerun"
assertion stronger, because both runs would share the same context
computation surface.

## Medium

### 5. NDCG uses non-standard graded relevance with truth-set gating

`ndcg_at_k_external` defines gain as "the truth score of the predicted
item, *if* the predicted item is in the true top-k, else 0":

```rust
let relevance: HashMap<usize, f32> =
    true_top_k.iter().take(k).map(|(i, s)| (*i, *s)).collect();

let dcg: f32 = pred_ids
    .iter()
    .take(k)
    .enumerate()
    .map(|(rank, idx)| {
        let rel = *idx as usize;
        let score = relevance.get(&rel).copied().unwrap_or(0.0).max(0.0);
        score / ((rank as f32 + 2.0).ln() / 2.0_f32.ln())
    })
    .sum();
```

This conflates two NDCG conventions: graded relevance (gain = true score,
even for items outside top-k) and binary relevance with truth-set gating
(gain = 1 if in truth set, else 0). The mix gives "score-weighted hit
rate at rank", which is not wrong but is also not what most ANN benchmark
papers report.

Pick one:

- **Standard graded NDCG:** `relevance[item]` = fp32 inner product score
  regardless of whether the item is in the true top-k. Build `relevance`
  from a wider band (e.g. top-100). Gain reflects "this item is *somewhat*
  relevant even if not in top-10".
- **Standard binary NDCG:** gain = 1 if item ∈ truth, else 0. Cleaner,
  comparable to ann-benchmarks numbers.

The current hybrid will be hard to compare to any external NDCG number on
the same dataset, which defeats half the point of moving to a real
corpus. Recommend binary NDCG for cross-library comparability and keep
graded as a secondary metric only if NFR-003 explicitly requires it.

### 6. The pg_extern surfaces are gated behind `pg_test`

The new `tqhnsw_graph_scan_recall_external_summary` and
`tqhnsw_graph_scan_recall_external_gate_report` functions live inside
`mod tests` (`#[cfg(any(test, feature = "pg_test"))]`). This is consistent
with the existing fixture-backed gate functions, so it is the intended
pattern — but `RECALL_REAL_CORPUS.md:124-161` shows them being called
from `psql` as if they were regular extension functions.

The user therefore needs to install the **pg_test build** of tqvector
against `tqvector_bench`, not a normal `cargo pgrx install`. This is
fine, but `RECALL_REAL_CORPUS.md` should add a one-line note: "Requires
the pg_test feature build of the extension; install with
`cargo pgrx install --features pg_test`" (or whatever the actual install
incantation is — verify against an existing benchmark doc).

Without this, a fresh user follows the doc, runs
`SELECT * FROM tqhnsw_graph_scan_recall_external_gate_report(...)`, and
gets `function does not exist` — a frustrating first impression for a
lane whose value depends on being easy to run.

### 7. The ann-benchmarks one-time reference probe is missing

Earlier conversation recommended a split: Qdrant HF for the gate fixture,
**plus** a one-shot ann-benchmarks reference probe (`#[ignore]`d, full
untouched 1M split) to anchor "our reference impl matches a published
number". This branch only lands the Qdrant lane, not the ann-benchmarks
anchor. That is a defensible scope cut — the gate is the urgent thing —
but it leaves a gap: if the brute-force ground truth or subset loader has
a subtle bug, every gate run will be silently wrong with no external
sanity check to catch it.

Suggest opening a follow-up review (`219-a4-ann-benchmarks-reference-anchor`
or similar) that lands the ignored probe and records the published-vs-
measured number once. Does not have to block this PR, but it should not
be forgotten.

## Low

### 8. Dead alias `row_index_to_id`

```rust
let row_index_to_id: Vec<i64> = corpus_ids.clone();
```

Just use `corpus_ids` directly downstream. The clone is a 50k × 8B = 400

### 9. Smoke test runtime (163 s for 500 rows)

The smoke test inserts 500 corpus rows + 25 query rows via per-row
`Spi::run("INSERT ... VALUES ...")`. This dominates the 163 s runtime.
Since the smoke test is `#[ignore]`d and runs on demand this is
acceptable, but if it ever needs to land in CI it should batch via `COPY`
or multi-row `VALUES` the same way the real loader does. Not blocking.

## What is right and should not be touched in a refactor pass

For balance:

- **Truth computed from actually loaded vectors, not regenerated from a
  seed.** The packet calls this out explicitly and it is the right call —
  it removes a whole category of "ground truth drift" bug that the
  synthetic fixture probe would have hit if extended naively.
- **`build_source_column = 'source'` discipline** for the indexes —
  separates "is the graph good" from "is the quantizer good", consistent
  with review 215.
- **Identifier validation on the loader side** (`SQL_IDENT_RE`) before
  any SQL composition — small but correct, and keeps the SQL layer trust
  surface narrow.
- **Idempotency contract is real:** loader skips reload when row count
  matches, skips index rebuild when reloptions match, drops and reloads
  empty leftover tables. Smoke test asserts byte-identical reruns. These
  two together genuinely deliver the "one-time-load / repeated-rerun"
  promise the doc claims.
- **The "open question for the reviewer"** at the bottom of the request
  packet — acknowledging that the 0.89 gate target was tuned against
  synthetic and may need re-tuning against the first real DBpedia run.
  That is exactly the right framing and the right time to flag it.
- **Schema is minimal and correct:** `(id bigint pk, source real[],
  embedding tqvector)`. No surprises, no over-engineered columns.
