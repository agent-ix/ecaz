# Feedback: Scan Prepared-Query Cache

**Reviewer:** outside-review  
**Date:** 2026-04-05  
**Verdict:** Approved with note

## Answers to review questions

1. **Caching `PreparedQuery` at `amrescan` time**: Correct. `PreparedQuery` contains the pre-rotated lookup table and QJL projection. These are derived purely from the query vector and the quantizer parameters — both of which are known at rescan time. Caching this avoids re-preparing on every `amgettuple` call, which would be O(dim * num_centroids) wasted work per tuple.

2. **Stale-state or lifetime risk on repeated rescans**: `store_scan_prepared_query` calls `free_scan_prepared_query` first (scan.rs:202), which drops the previous `Box<PreparedQuery>` if present. The new prepared query is then `Box::into_raw`'d into the opaque pointer. `amendscan` also frees it. The lifecycle is correct: free-before-replace on rescan, free on endscan. No leak or double-free.

3. **PostgreSQL-managed vs. boxed Rust object**: Storing as `Box::into_raw` is fine for this stage. The `PreparedQuery` contains `Vec<f32>` fields (lut, sq) which are Rust-heap-allocated. Moving these into `palloc`'d memory would add complexity without benefit — the scan descriptor's lifetime is well-defined and Rust's `Box::from_raw` + `drop` handles cleanup correctly. PostgreSQL memory contexts would only matter if the scan needed to survive transaction boundaries, which it doesn't.

## Additional notes

- The `prepared_query` pointer in `TqScanOpaque` is `*mut PreparedQuery`. For future traversal, the entire prepared query could be reused across multiple candidate scorings within one `amgettuple` call, which is the intended design.
