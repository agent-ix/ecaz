# Pre-Checkpoint Review: SPIRE DML PK SELECT CustomScan (30873)

## Note

Code committed in `ca4fa198 Wire DML PK select through SPIRE
CustomScan` but no `request.md` exists yet — leaving this
pre-checkpoint feedback per `review/REVIEWER.md` so the coder
sees feedback before the packet is finalized.

## Summary

**Major milestone — first DML front-door plan replacement
through CustomScan.** The packet:

- Adds `SpireCustomScanPlanMode::{VectorOrderLimit,
  DmlPkSelectTuplePayload}` so one CustomScan provider handles
  both vector-ORDER-BY and DML PK SELECT shapes via a discriminator
  in `custom_private[0]`.
- Extends `set_rel_pathlist_hook` to also call
  `dml_pk_select_candidate_index_oid` and add
  `add_dml_pk_select_custom_scan_path(...)` for eligible base
  relations.
- New `plan_dml_pk_select_custom_path(...)` builds a CustomScan
  plan node carrying the PK value expression in `custom_exprs[0]`
  via `copyObjectImpl`.
- New `custom_scan_dml_pk_select_access(state, scan_state)`
  invokes `ec_spire_forward_coordinator_select_tuple_payload(...)`
  via SPI and stores one row through the existing virtual-tuple
  payload writer.
- Mode dispatch in `Begin` / `Exec` / `Rescan` correctly branches
  between vector and DML paths; the `DmlPkSelectTuplePayload`
  path resolves PK column from the catalog relation context (no
  custom_private storage needed) and PK value via
  `ExecInitExpr` on the `custom_exprs[0]` expression at Begin
  time.
- New helpers
  `custom_scan_dml_pk_qual_value_expr` and family detect the
  narrow `WHERE pk_col = const_or_param_bigint` shape, including
  via `baserestrictinfo` walk first then `jointree.quals`
  fallback.

Eligibility gate now also considers placement-directory
existence (`custom_scan_index_has_sql_placement`) so the
CustomScan only intercepts SELECTs against tables that have at
least one placement row.

The fact that this lands without breaking the existing vector-
ORDER-BY path (the existing 30815 multicluster fixture should
still pass) is the load-bearing structural change here.

## Findings

### P1 — Eligibility-gate uses SPI from inside `set_rel_pathlist_hook`

`custom_scan_index_has_sql_placement(index_oid)` calls
`Spi::connect(...).select(...)`. This runs from
`ec_spire_set_rel_pathlist_hook` which is invoked during
planning. SPI inside `planner_hook` has historically been the
re-entrance trap that 30855 was specifically built to avoid —
the catalog-backed relation-context loader exists to keep SPI
out of the hook.

Two concerns:
- **Re-entrance**: SPI inside the rel-pathlist hook can cause
  recursive planning if the SPI query is itself rewritten by the
  hook. `SELECT EXISTS(... FROM ec_spire_placement WHERE ...)`
  doesn't target `ec_spire`-marked tables so the recursion case
  is theoretical here, but the pattern is fragile to add. A
  future CustomScan against the placement table itself would
  recurse.
- **Cost**: Per-query SPI roundtrip on every relation that has
  an ec_spire index. Adds planner overhead.

Suggested shape: replace the SPI probe with a direct catalog
lookup (`pg_class` + `pg_index` → does any row in
`ec_spire_placement` have this `index_oid`?) using PG's relcache
or a direct `index_open` over the placement table's
`(index_oid, pk_value)` PK. Or, weaker: cache the result per
backend in a small hashmap keyed on `(index_oid,
relcache_invalidation_xid)`.

For v1 ship the SPI shape if testing shows no recursion, but
flag this as the highest-priority follow-up before the packet
sees production traffic.

### P1 — Two parallel PK-extraction paths now exist

`custom_scan_dml_pk_qual_value_expr` (this file) and the
`dml_frontdoor.rs` query-extraction layer both walk PG
queries to find the bigint-PK equality predicate. They have
slightly different shapes:
- The custom_scan version walks `baserestrictinfo` then
  `jointree.quals`, returns the matched expression.
- The dml_frontdoor version walks `jointree.quals` only via
  `dml_frontdoor_pk_predicate(...)`.

The existing dml_frontdoor extraction layer was the whole point
of the 30847→30866 scaffold. The custom_scan code should
delegate to it via `dml_frontdoor_primitive_plan_expr_catalog_row(...)`
(which is exactly the helper 30872 just added) and avoid
re-implementing the predicate walk.

Recommend refactoring `dml_pk_select_candidate_index_oid` and
`plan_dml_pk_select_custom_path` to consume
`dml_frontdoor_primitive_plan_expr_catalog_row(query)` →
`SpireDmlFrontdoorPrimitivePlanExpr` → use the typed primitive
plan to produce the CustomPath. Otherwise the two extractors
will drift over time and their accept/reject sets will diverge.

### P2 — Eligibility check fires on every base relation

`dml_pk_select_candidate_index_oid` runs:
1. Walk indexlist to find an `ec_spire` index.
2. SPI probe for placement existence.
3. Walk `baserestrictinfo` / `quals` to find the PK predicate.

For every base relation in every query. The vector-ORDER-BY
candidate check already runs first and short-circuits when
matched. Worth confirming the order is "cheap relation-class
check → ec_spire AM existence (relcache hit) → placement existence
(can become catalog-cached) → predicate walk."

### P2 — Mode discriminator stored as `Oid`

`custom_private[0] = Oid::from(CUSTOM_SCAN_PLAN_MODE_VECTOR_ORDER_LIMIT |
DML_PK_SELECT)`. Same hack as the 30810 P3 about `top_k` being
stored as Oid. The List of Oids is convenient, but a List of
Integer nodes (`makeInteger(1)`/`makeInteger(2)`) would more
honestly express "this is an enum value, not an OID." Lower
priority but worth pinning before more modes get added.

### P2 — `pk_value_expr` evaluation runs at Begin time, not Exec

`custom_scan_dml_pk_value_from_plan` calls `ExecInitExpr` and
evaluates the param at Begin time. For prepared statements
re-executed with new parameter values that's stale — the value
doesn't refresh on the next execute. The `Rescan` callback
clears `dml_payload_loaded` / `dml_payload_emitted` but does NOT
re-evaluate the PK value, since `dml_pk_value` was set in
`Begin`.

Two acceptable shapes:
- Move PK value resolution into `Exec` so each rescan re-fetches
  the parameter value.
- Keep value at Begin but document that v1 doesn't support
  re-execute with different params (which would also need a
  parameterized fixture asserting the limitation).

The first is the production-correct shape; the second is a
documented v1 limitation.

### P2 — Const-only PK value resolution in this slice?

`custom_scan_bigint_expr_value` accepts both `T_Const` and
`T_Param` and handles both. But the runtime parameter path
inside the executor uses `(*node).ss.ps.ps_ExprContext` for
evaluation. Worth confirming this expression context is the
right one for executor PK parameters (it should be — that's
the per-scan ExprContext that PG initializes).

### P3 — `selected_count != 1` returns empty payload silently

```rust
Ok::<Option<String>, String>((selected_count == 1).then_some(payload).flatten())
```

For `selected_count = 0` (PK not found): returns `None`, the
ExecCustomScan returns no rows. Correct shape — matches PG's
PK-not-found semantics where SELECT returns zero rows.

For `selected_count > 1`: also returns `None`. But the
30846/30840 primitives error on `selected_count > 1`, so this
case shouldn't actually trigger. Worth confirming the SPI call
surfaces the primitive's error rather than silently mapping it.

### P3 — `dml_pk_value` length zero check at Exec time

`if state_ref.dml_pk_value.is_empty() { error }` — defensive,
but the value was just assigned in Begin from a
`dml_frontdoor_bigint_pk_value_bytes(...)` call which always
returns 8 bytes. The empty check is unreachable in practice. Fine
as defense in depth.

### P3 — `extract_actual_clauses` change

The vector-ORDER-BY plan also flips from `qual = clauses` to
`qual = extract_actual_clauses(clauses, false)`. That's the
right pattern for clauses coming from the planner's restrictinfo
list (need to extract the OpExpr from RestrictInfo). Worth a
focused EXPLAIN regression to confirm the existing vector-ORDER-BY
test still picks the same plan.

## Reviewer recommendation

**Accept the milestone direction with two P1s before merge to the
production-fixture lane:**

1. **Replace the SPI probe in the planner hook** with a relcache-
   safe placement existence check. SPI inside the hook is the
   re-entrance trap the entire 30855 catalog scaffold was built
   to avoid; landing it here re-opens that risk.
2. **Delegate PK extraction to the existing dml_frontdoor
   scaffold** (use `dml_frontdoor_primitive_plan_expr_catalog_row(...)`
   from 30872) instead of duplicating the extraction logic. The
   30847→30872 packet chain exists precisely so the rewrite
   slice doesn't have to re-derive these facts.

The structural shape of the slice is right — discriminated
CustomScan provider, single Exec dispatch, virtual-tuple writer
reuse, mode-aware Begin/Rescan. P2 items (PK value evaluation
at Exec vs Begin, mode discriminator type) are the next-priority
follow-ups.

This is the load-bearing executor work the entire scaffold has
been building toward — please open the request packet so the
multicluster fixture coverage can land alongside.
