## Direction Assessment

**Coder-1's direction is correct.** The graph-first prefetched cursor is the right A3 architecture:

1. **FR-009 alignment**: "Load results in rescan, drain in amgettuple" — this is exactly what the arc achieves. Graph results are materialized during `amrescan` and post-emit advancement; `amgettuple` only drains.

2. **Read-stream readiness**: The prefetched cursor pattern is what read-stream integration needs. When `ReadStream` delivers pages asynchronously, the graph cursor can prefill from the stream during rescan/advancement, and `amgettuple` remains a synchronous drain.

3. **Clean phase boundaries**: Graph and fallback now have fully independent result-state lifecycles. This means the graph phase can be replaced with a search-owned ordered cursor without touching fallback code.

4. **Reduced scan-owned state**: Removing `graph_traversal_seeded` and deriving control from live graph/search state (reviews 10050-10051) is the right direction — scan should be a thin shell over graph-owned traversal, not a parallel state machine.

**Next logical steps** for coder-1 would be:
- Move graph result-state ownership into a dedicated cursor struct (separate from `TqScanOpaque`)
- Wire read-stream prefetch into the graph cursor refresh path
- Eventually collapse the remaining scan shell into "dispatch to cursor, emit from cursor"
