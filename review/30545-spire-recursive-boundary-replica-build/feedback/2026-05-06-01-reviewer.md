## Inconsistent route metric between recursive default and replica paths

`build_recursive_leaf_rows_by_pid` (build/recursive.rs:226) splits on
`boundary_replica_count`:

- `== 0`: rows are grouped by `assignments_by_centroid`, which comes from
  `common_training::assign_vector_to_centroid` (Euclidean k-means
  assignment).
- `> 0`: every source vector is re-routed through
  `route_boundary_assignment_for_vector`, which uses **inner product**.

So enabling boundary replication on a recursive index can shift the
*primary* leaf placement of rows whose Euclidean-nearest centroid and
IP-nearest centroid differ — not only add secondary placements. That is
not wrong against the design (which says primary = top-IP leaf), but it
means the knob does more than fan out, and `boundary_replica_count=0`
recursive builds use a different metric than `boundary_replica_count>0`
recursive builds and than the populated single-level path (which always
uses the route map).

Two reasonable resolutions:

1. Route the recursive default through `route_boundary_assignment_for_vector`
   with `boundary_replica_count=0` so the primary placement metric is
   consistent across replica counts and across single-level vs recursive
   builds.
2. Document the divergence explicitly in
   `plan/design/spire-boundary-replication.md` — operators turning the
   knob on should know primaries can move.

Option 1 is the smaller change and removes a silent behavior shift on
existing indexes if they ever rebuild with `boundary_replica_count > 0`.
