# Review: SPIRE Boundary Storage Accounting (30547)

Reviewed at code commit `c3497c76` on branch `task-30-spire`.

The new diagnostic columns (`base_primary_assignment_count`,
`base_boundary_replica_assignment_count`,
`delta_boundary_replica_insert_assignment_count`,
`effective_boundary_replica_assignment_count`) cleanly separate physical
overhead from logical row count. The recall-study artifacts confirm the
counters land where expected: 10000 / 10000 / 0 for boundary off and
10000 / 10000 / 10000 for replica=1.

Delta accounting correctly excludes delete deltas (the
`SPIRE_ASSIGNMENT_FLAG_DELTA_INSERT` gate at `root/snapshots.rs:675`
ensures only insert deltas are counted, and the
`SPIRE_ASSIGNMENT_FLAG_BOUNDARY_REPLICA` check is nested inside it).
`effective_boundary_replica = base + delta_insert` matches the design
while delete-delta replica rows remain invalid.

## Header vs. row-counted assignment totals are not cross-checked

`apply_leaf_snapshot_base_row` (root/diagnostics.rs:420) sets
`base_assignment_count = u64::from(header.assignment_count)` from the
object header, while `base_primary_assignment_count` and
`base_boundary_replica_assignment_count` come from re-iterating the
assignment rows in `count_leaf_snapshot_base_assignment_roles`. If a
future codepath ever publishes a leaf where header count and row-iterated
count disagree, the diagnostic will silently report inconsistent numbers
and downstream operators have no signal that something drifted.

Either return `Err` from the snapshot when
`base_primary + base_boundary_replica != base_assignment_count`, or — if
that's too strict because of fields like `TOMBSTONE` — at minimum
debug-assert the relationship that *is* invariant for current writers
(every base assignment row has exactly one of `PRIMARY` or
`BOUNDARY_REPLICA`). The recall-study artifacts show today they match,
so the check would not be lossy on landed data.

## v1/v2 fallback order is asymmetric with the scan path

`count_leaf_snapshot_base_assignment_roles` (root/snapshots.rs:742) tries
`read_leaf_object` (v1) first and falls back to `read_leaf_object_v2`.
The scan path (`scan/candidates.rs:328` `append_quantized_leaf_candidates_for_pid`)
does the opposite — v2 first, fall back to v1. Both succeed because every
leaf is one or the other, but the asymmetry is a footgun: a future change
that only updates one site won't show up until a v1↔v2 mixed-version
index hits production.

Pick one canonical order (v2-first matches where the codebase is moving)
and either factor a shared `read_leaf_object_versioned` helper or at
least add a comment at both sites pointing at each other.

## SQL surface placement

The new columns are appended to `ec_spire_index_leaf_snapshot` in a
sensible order — base counts grouped together, delta counts grouped
together, effective counts at the end. Named callers will pick them up
without breakage; positional callers see them after the existing columns.
Looks fine.
