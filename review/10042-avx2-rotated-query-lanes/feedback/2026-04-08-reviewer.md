# Review Feedback: AVX2 Rotated Query Lanes

**Verdict: Approved — architecturally important change**

## Mathematical equivalence

The old path computed `lut[dim * num_centroids + index]` where `lut` was precomputed as
`codebook[centroid] * rotated[dim]` for all (dim, centroid) pairs. For the 3-bit case
(num_centroids=8), this was a 1536×8 = 12,288-entry table, and the AVX2 path gathered from it
with `_mm256_i32gather_ps` hitting up to 8 different cache lines per iteration.

The new path caches `rotated[dim]` directly on `PreparedQuery` and does
`codebook_permute(indices) * rotated_load(dim)` — a single-cycle permute from a register-resident
8-entry codebook plus a contiguous 8-lane load. This is fundamentally better memory behavior:
the codebook never leaves registers, and the rotated query is a sequential linear scan.

26% improvement is the right magnitude for replacing scattered gathers with contiguous loads.

## PreparedQuery expansion

Adding `rotated: Vec<f32>` (1536 extra f32s = 6 KiB) is acceptable for B1. The LUT is still
needed for the scalar path and generic-width AVX2, so this is additive rather than replacing.
If you want to shrink `PreparedQuery` later, the LUT becomes redundant for 3-bit-only workloads
since `lut[dim * 8 + idx] == codebook[idx] * rotated[dim]`. But that's a future cleanup.

## Discarded experiments

16-dimension unroll and sign-mask XOR replacement both regressing on this host is useful to
document. The unroll regression is slightly surprising — it could be register-pressure related.
Worth retrying if the loop body shrinks further (e.g., after FMA).
