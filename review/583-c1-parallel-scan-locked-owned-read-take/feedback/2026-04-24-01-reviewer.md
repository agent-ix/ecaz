# Review: Parallel Scan Locked Owned Read-Take

## Summary

The refactor correctly closes the TOCTOU window. The old call site in `try_take_parallel_scan_next_output` separated "read readiness" (one coordinator-lock section) from "take output" (a second lock section), leaving a window where a better hidden foreign row could interpose and trigger the fatal `ready state produced no owned output to take` path.

The new `take_parallel_scan_owned_ready_output_snapshot` acquires the lock once per outer iteration and calls both `read_parallel_scan_owned_output_state_locked` and `take_parallel_scan_owned_next_output_snapshot_locked` under the same lock hold. The Empty/Blocked returns are terminal; only the Ready branch needs the take, so the combined function correctly drops the lock on each non-terminal case.

## Findings

**Lock hold across inner retry loop.** `take_parallel_scan_owned_next_output_snapshot_locked` (the extracted inner helper) loops up to 4 times calling `admit_pending_output_locked` and `take_admitted_result_locked` under a single held coordinator lock — whereas the original `take_parallel_scan_owned_next_output_snapshot` re-acquired the lock per iteration. This is a stricter serialization and is the correct behavior for the combined operation, but it extends the worst-case lock hold to cover `4 * (admit + take)`. At current coordinator state complexity this is bounded, but worth noting as the coordinator grows.

**Public wrapper preservation.** `read_parallel_scan_owned_output_state` and `take_parallel_scan_owned_next_output_snapshot` are preserved as public wrappers that each acquire-then-delegate. The internal `_locked` variants are `unsafe fn` (not `pub`) so the lock discipline is enforced by convention at the call sites. This is clear and consistent with the rest of the module.

**Regression test.** `take_parallel_scan_owned_ready_output_snapshot_preserves_hidden_foreign_blocker` covers the specific failure mode: a hidden foreign row that would block the owner's output is visible when readiness is read, but the combined function returns `Blocked` instead of proceeding to drain the owner row. Tight coverage of the stale-ready path.

## No blockers.
