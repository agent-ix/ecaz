# 30765 — SPIRE row materialization mapping contract review

Validation pass on `e415b20e` (`Expose SPIRE row materialization
mapping contract`). Accepts the mapping contract surface.

## What I checked

- New SQL surface
  `ec_spire_remote_search_row_materialization_mapping_contract()`
  at `src/lib.rs:11672`, registered in the operator
  entrypoint contract at
  `src/am/ec_spire/root/remote_candidates.rs:386`.
- Contract pins five invariants before AM may return a
  remote-origin row through `xs_heaptid`:
  1. **Mapping identity**: requested epoch + served epoch +
     origin node + global vec-id + opaque row locator must
     match the heap-resolved remote output (full quintuple,
     not just locator).
  2. **Same-relation TID rule**: the materialized TID must
     belong to the heap relation being scanned by the index
     scan — matches the v1 AM-compatible rule from ADR-064
     and packet 30761.
  3. **Scan snapshot visibility**: stale or vacuumed
     materialized rows remain blocked.
  4. **No scan-time heap writes**: `amrescan` /
     `amgettuple` cannot create the materialized row —
     matches ADR-064's no-scan-time-writes invariant.
  5. **Strict/degraded behavior**: missing or stale mapping
     fails closed in strict, skips the dispatch in degraded.
- This is a provider boundary, not actual remote-origin AM
  delivery. Scan path continues to block remote-origin rows
  with `remote_row_materialization` until a real
  epoch-scoped materialized-row provider exists. Honest
  scoping.

## Answers to the packet's review questions

- **Mapping identity complete enough to reject stale or
  mismatched materialized rows?** Yes. The five-element
  identity (requested epoch + served epoch + origin node +
  global vec-id + opaque row locator) covers every dimension
  along which a stale mapping could pass single-element
  comparison:
  - epoch divergence (rebuild on origin),
  - node-id divergence (replica drift),
  - vec-id divergence (cross-vector confusion),
  - row-locator divergence (origin-side heap rewrite).
  Requiring exact match on all five rejects every realistic
  staleness mode.
- **Same-relation TID rule explicit enough for PostgreSQL
  index AM correctness?** Yes. PostgreSQL's index AM
  contract requires `xs_heaptid` to identify a row in the
  scanned relation; the contract names this directly rather
  than relying on the implementation to remember the
  invariant. Matches the layered enforcement at packets
  30757/30758/30761/30762.
- **Distinction preserved between provider boundary and
  actual remote-origin AM delivery?** Yes. The contract is
  the rule a provider must satisfy; the delivery still
  fails closed with `remote_row_materialization` until a
  provider lands. No premature claim of working delivery.

   The strict-fail-closed and degraded-skip behaviors for

   node, etc. — to lock the contract against single-field-
   only comparison drift. Naturally lands with the provider.
