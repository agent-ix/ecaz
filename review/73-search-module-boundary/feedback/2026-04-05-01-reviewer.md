# Feedback: Search Module Boundary

## Answers to Review Questions

### Is the `scan` vs `search` boundary the right long-horizon seam?

**Yes.** Verified: `search.rs` (610 lines) has zero pg imports — only `std::cmp`, `std::collections`, `std::hash`. The `BeamSearch<NodeId>` struct is generic over `NodeId: Copy + Eq + Hash`, which means it can be tested with plain `u64` nodes (as the unit tests do) without any pg infrastructure. Meanwhile `scan.rs` (1618 lines) handles all pg-specific concerns: scan descriptors, buffer management, WAL, palloc'd memory.

The integration point is clean: `scan.rs` imports `use super::search` and maps between `ScanCandidate` (pg-aware, with `page::ItemPointer`) and `search::BeamCandidate<page::ItemPointer>` via the `scan_candidate_to_beam_candidate` converter (scan.rs:533-544). This adapter pattern is the right boundary — the search module doesn't know about pages, buffers, or scan descriptors.

### Does `BeamSearch` capture the right minimal traversal responsibilities?

**Yes.** The struct owns:
- `frontier: BinaryHeap<Reverse<QueuedCandidate<NodeId>>>` — best-first priority queue
- `visited: HashSet<NodeId>` — deduplication
- `discovery_order: Vec<BeamCandidate<NodeId>>` — insertion-order trace for debugging
- `sequence: u64` — stable tie-breaking within the heap

The API surface is minimal:
- `seed` / `seed_many` — add initial candidates
- `peek_best` / `pop_best` — read/consume the best queued candidate
- `expand_one` — pop best + add its neighbors (the core greedy-descent step)
- `run` — repeat `expand_one` up to `ef_search` times
- `forget_queued` — remove a node from the frontier (for scan-side consumption sync)
- `frontier_snapshot` / `frontier_len` / `visited_count` — observability

No pg concerns leak in. The `source: Option<NodeId>` on `BeamCandidate` provides provenance without knowing what nodes represent.

### Are the tests sufficient to lock in best-first, dedup, and bounded-expansion behavior?

**Yes.** Verified 7 unit tests in `search.rs:226-609`:
- `beam_search_expands_best_first_from_seeds` — validates expansion order on a 7-node graph
- `beam_search_deduplicates_self_loops_and_parallel_edges` — self-loops and duplicate edges don't create duplicate discoveries
- `beam_search_respects_expansion_budget` — `ef_search=2` caps at 2 expansions
- `beam_search_seed_many_accepts_unique_candidates_only` — duplicate seeds rejected
- `beam_search_expand_one_matches_incremental_best_first_progress` — incremental stepping matches batch behavior
- `beam_search_run_matches_repeated_expand_one` — `run()` === repeated `expand_one()`
- `beam_search_forget_queued_removes_frontier_node_and_allows_reseed` + `ignores_expanded_or_unknown_nodes` — forget semantics

Coverage is comprehensive for the current API surface.

## Additional Findings

**One design note on `forget_queued`**: The implementation (search.rs:98-122) drains the entire `BinaryHeap`, filters out the target, and rebuilds. This is O(n) per call. For the current bootstrap frontier sizes this is fine, but when `ef_search` grows to 40-1000 candidates, repeated `forget_queued` calls during consume-refill cycles could become expensive. Consider whether `forget_queued` is a temporary integration bridge or a long-term API — if the latter, a lazy-deletion approach (mark-then-skip on pop) might be more efficient.
