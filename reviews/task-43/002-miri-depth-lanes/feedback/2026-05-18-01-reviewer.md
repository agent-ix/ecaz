# Review: Task 43 Miri Depth Lanes (packet 002)

## Summary

Adds `miri-tree`, `miri-many-seeds`, and `miri-full` Make targets;
points `hardening-nightly-local` at `miri-full`; switches
`scripts/hardening.sh` to PATH-based `cargo`/`rustup` discovery with
the Homebrew paths kept as fallback. Docs and governance updated.
No new test code — pure infrastructure checkpoint.

## Confirmations

- **Make targets exist and shell-validate.** `make -n miri-tree
  miri-many-seeds miri-full hardening-nightly-local` succeeds.
- **`hardening-nightly-local` correctly depends on `miri-full`.** The
  dry-run output shows `bash scripts/hardening.sh miri-full`, which
  fans out to default Miri + Tree Borrows + many-seeds.
- **PATH discovery is the right unblocker for Linux hosts.** The
  Homebrew fallback preserves macOS behaviour, so this is a strictly
  additive change.
- **`MIRIFLAGS` composition is correct.** The `miri-tree` lane sets
  `-Zmiri-tree-borrows` without dropping caller-provided MIRIFLAGS;
  the `miri-many-seeds` lane composes similarly.

## Findings

- **Many-seeds lane is unverified in this packet.** The packet does
  not run `miri-many-seeds` end-to-end — it only validates the make
  dry-run. That's fine for an infrastructure-only slice, but it means
  the actual seed-range syntax bug (a real `=0..128` issue caught in
  packet 005) was not exposed here. A faster smoke run (even one seed)
  would have caught it sooner.
- **No documented test of the disagreement-triage path.** The Make
  target is wired and docs explain it, but the packet does not
  demonstrate that the lane *would* report a disagreement clearly if
  one occurred. Acceptable for now since SB and TB end up agreeing
  across all 35 tests, but worth a one-shot test of the triage
  failure mode in a follow-up.

## Verdict

Approve. Infrastructure scaffolding is correct, dry-run validation is
adequate for this slice, and the docs cover the new lanes. The
follow-on packet 005 correctly catches and fixes the
`-Zmiri-many-seeds=0..128` syntax that this packet's dry-run-only
validation did not exercise.
