# Review: Task 43 Remote Typed Payload Parser Extraction (packet 009)

## Summary

Closes G4. Extracts SPIRE remote typed-payload validation out of the
`postgres::Row` decoder into a Row-independent field-level decoder, so
the validation logic that previously could only run with a live libpq
row is now Miri-able. Production libpq path delegates to the new pure
decoder. Adds Miri tests for valid decoding (with and without
collation), odd-length and invalid hex rejection, decoded-byte caps,
width mismatches across every typed payload vector, invalid attnums,
invalid type OID and collation text, unsupported tuple transport,
non-ready transport status, and unsupported per-column payload format.

## Confirmations

- **Pure helper extraction was actually done.** This was the explicit
  strategy-packet recommendation that earlier closeouts had skipped.
  Production path now uses the pure decoder; row-shaped wrapper just
  marshals fields into the decoder's input.
- **Adversarial coverage is broad.** Hex parsing, attnum bounds, OID
  parsing, collation parsing, transport status, format constraints —
  each gets a rejection test, not just a happy-path test.
- **Miri logs ground all claims.**
  `miri-remote-typed-payload-fields.log` 2/0, `miri-remote-payload-caps.log` 1/0.
- **Mutation probe lands in packet 013.** Tracker correctly marks
  cargo-careful mirror as blocked on pgrx/Oid dependency rather than
  declaring G6 done here.

## Findings

- **The pgrx-blocker for cargo-careful mirroring is now well-scoped.**
  Tracker entry will be readable enough to plan a future SPIRE careful
  micro-harness. Good. Not a packet-009 concern.
- **No backward-compat shim.** The libpq path is rewritten in place to
  delegate. That's the right call — no two parsers to keep in sync —
  but reviewers of any future libpq-row regression should know the
  pure decoder is the authority.

## Verdict

Approve. Cleanest extraction of pgrx-bound code into pure-Rust I've
seen in the task-43 series. Closes G4.
