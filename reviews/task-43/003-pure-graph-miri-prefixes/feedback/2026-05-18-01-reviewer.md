# Review: Task 43 Pure Graph Miri Prefixes (packet 003)

## Summary

Promotes four existing bounded unit tests into the `miri_` prefix
covering DiskANN Vamana (`robust_prune` alpha dominance,
`greedy_search` convergence) and HNSW (`BeamSearch` dedupe,
`VisibleFrontier` live-node preference). No algorithm change; rename
only.

## Confirmations

- **All four tests run cleanly under Miri.** Validation logs are
  packet-local under `artifacts/`.
- **Tests are genuinely bounded.** Each uses a small fixture graph
  (≤ a dozen nodes), which is the right size for Miri's per-instruction
  cost.
- **Promotion-over-creation is the right choice for these surfaces.**
  HNSW search and DiskANN Vamana are pure Rust today; existing tests
  already cover the algorithmic core. Writing parallel Miri tests
  would just duplicate.

## Findings

- **Coverage is narrow within each surface.** The task spec listed
  HNSW pure candidates `BeamSearch`, `VisibleFrontier`,
  `select_next_with_refill`, deterministic frontier ordering, and
  stale-candidate removal. This packet covers BeamSearch (dedup
  scenario) and VisibleFrontier (live-node preference) only.
  `select_next_with_refill`, frontier-ordering invariants, and
  stale-candidate removal are not in the Miri lane. Similarly for
  DiskANN: `build_vamana_graph_with_stats` and
  `build_vamana_graph_with_pass1_extra_candidates` from the strategy
  packet did not get Miri prefixes.
- **Limited bug-injection signal.** Miri's value on pure search code
  is correctness under Stacked / Tree Borrows aliasing rules. The
  promoted tests don't include adversarial inputs that would force
  unusual aliasing patterns — e.g., graphs with cycles, parallel
  edges across many levels, or empty layers — so a regression that
  preserves the algorithmic happy path but introduces UB through e.g.
  a borrow split is not certain to be caught.
- **No mutation test or sanity check.** The task spec's validation
  bullet says each `miri_` test should "fail predictably if an obvious
  bug is injected." This packet does not demonstrate that.

## Verdict

Approve as a representative first slice. The promoted tests are
genuinely pure and pass under Miri. Reservations are about *breadth*
within each surface, not about correctness of what landed. Recommend
follow-up work to extend HNSW coverage to `select_next_with_refill`
and frontier-ordering invariants, and DiskANN coverage to the
build-graph helpers, when there is appetite for a third pass.
