# Standards Compliance Review

## Findings

### P1 - AC-level traceability is still not standards-complete

`spec/spec.md` now requires AC-level traceability, allowing grouped ranges only
when the same section also records individual FR or AC evidence rows or an
explicit gap ([spec.md lines 322-324](../../../spec/spec.md#L322-L324)). The
matrix still uses grouped rows as the only evidence for many completed claims:
`FR-001..FR-006`, `FR-007..FR-018`, and `FR-028..FR-038` are summarized by
range ([tests.md lines 92-113](../../../spec/tests.md#L92-L113)), and the SPIRE
rows map whole requirements to broad test groups such as `TC-020 SPIRE` and
`TC-024` ([tests.md lines 114-126](../../../spec/tests.md#L114-L126)).

I ran a local inventory of requirement headings and found 192 acceptance
criteria under StR/US/FR/NFR files, with 161 not mentioned literally in
`spec/tests.md`. Examples include `FR-048-AC-2..8`,
`FR-055-AC-1..3`, `FR-056-AC-1..3`, `FR-057-AC-1..3`,
`FR-058-AC-1..3`, and `FR-059-AC-2..9`. This leaves the ISO/IEC/IEEE 29148 and
local `/spec-matrix` coverage claim incomplete even where rows say `Complete`
or `Implemented`.

Action: either split the matrix into per-AC rows for every completed/implemented
claim or downgrade those rows to partial with explicit `GAP-*` entries until
individual AC evidence exists.

### P1 - TC-034 and NFR-004 overstate hardening evidence provenance

`NFR-004-AC-4` requires hardening-lane change packets to store raw tool logs and
record which lanes passed, skipped, or remain manually gated
([NFR-004 lines 120-123](../../../spec/non-functional/NFR-004-safety-and-stability.md#L120-L123)).
The Task 34 request claims passed results for `make test`, `make
hardening-local`, `make hardening-nightly-local`, `make cargo-vet`, `make
cargo-geiger`, `make afl-decoders`, sanitizer lanes, Miri, cargo-careful, Kani,
Flux, Rudra, MIRAI, Loom, and Shuttle
([request.md lines 54-79](../../30034-task34-comprehensive-hardening/request.md#L54-L79)).
The packet artifact manifest, however, records durable logs only for installer
attempts plus MIRAI, Flux, and Rudra-family lanes
([manifest.md lines 10-176](../../30034-task34-comprehensive-hardening/artifacts/manifest.md#L10-L176)).

That is not enough to support `TC-034` as `Implemented for local-first baseline`
([tests.md line 190](../../../spec/tests.md#L190)) or the coverage-baseline claim
that Task 34 landed all named hardening lanes
([tests.md lines 29-30](../../../spec/tests.md#L29-L30)). It also weakens the
ISO/IEC/IEEE 15939 measurement/provenance target for hardening claims
([spec.md line 284](../../../spec/spec.md#L284)).

Action: packet the missing raw logs or change `TC-034`/`NFR-004` coverage status
to reflect that only a subset of lanes currently has packet-local evidence.

### P2 - IEEE 828 lifecycle control is incomplete for deleted requirement IDs

The master spec says identifier gaps that reached review or task planning
require a retained artifact or explicit tombstone, including whether the old
requirement was deprecated, superseded, or folded into another artifact
([spec.md lines 338-340](../../../spec/spec.md#L338-L340)). The current
functional requirement inventory has `FR-001..FR-038`, then `FR-044..FR-060`;
`FR-039..FR-043` have no retained requirement files or tombstone entries. This
is visible in the matrix ordering as well, where `FR-044..FR-047` appear after
`FR-060` ([tests.md lines 127-130](../../../spec/tests.md#L127-L130)).

The ADR index also preserves historical duplicate numeric IDs, for example
`ADR-016` and `ADR-048` each identify two separate decisions
([adr/index.md lines 8-17](../../../spec/adr/index.md#L8-L17),
[adr/index.md lines 24-26](../../../spec/adr/index.md#L24-L26)). The index
documents a forward-looking no-reuse policy
([adr/index.md line 77](../../../spec/adr/index.md#L77)), but the duplicate
legacy numbers still mean numeric-only references are ambiguous.

Action: add lightweight `FR-039..FR-043` tombstones or a tombstone table in the
master spec, and require ADR references to include filename/topic wherever a
historical duplicate number exists.

### P2 - NFR frontmatter and scope do not meet the declared NFR discipline

The declared 25010 target says NFRs are scoped to a primary quality attribute
([spec.md line 283](../../../spec/spec.md#L283)). The local NFR skill also
requires impacted FRs in `relationships:`. `NFR-004` combines crash behavior,
memory safety, WAL correctness, licensing, supply chain, fuzzing, formal/model
checking, SQLsmith, and analyzer promotion policy in one artifact
([NFR-004 lines 16-100](../../../spec/non-functional/NFR-004-safety-and-stability.md#L16-L100)),
but its frontmatter constrains only `StR-002`
([NFR-004 lines 7-10](../../../spec/non-functional/NFR-004-safety-and-stability.md#L7-L10)).

`NFR-015` has the same relationship-shape problem: it constrains `StR-006` and
extends `NFR-007`, but the targets still use the old `tqvector` namespace rather
than the current `ecaz` namespace
([NFR-015 lines 7-13](../../../spec/non-functional/NFR-015-benchmark-reporting-standard.md#L7-L13)).
Those relationship edges are not reliable enough for a standards trace graph.

Action: split or explicitly classify the multi-quality parts of `NFR-004`, then
add direct `constrains` edges to affected FRs such as `FR-011` and
`FR-049..FR-059`. Normalize `tqvector` relationship targets that are part of the
current Ecaz spec.

### P3 - Several US artifacts still fail the current user-story artifact shape

The current test matrix says every AC should trace to a TC or documented gap
([tests.md lines 10-17](../../../spec/tests.md#L10-L17)), but legacy user-story
files still use numbered criteria rather than `US-XXX-AC-N` headings and legacy
`traces:` frontmatter. For example, `US-001` has no `artifact_type`, uses
`traces:`, and lists unnumbered acceptance criteria as ordinary numbered items
([US-001 lines 1-19](../../../spec/usecase/US-001-store-compressed-embedding.md#L1-L19)).

`US-021` is a current cloud story but has `type: use-case`, no `artifact_type:
US`, no As/I-want/So-that story text, and no `US-021-AC-*` acceptance criteria
([US-021 lines 1-76](../../../spec/usecase/US-021-cloud-benchmark-cycle.md#L1-L76)).
The matrix maps it by prose flow instead of AC IDs
([tests.md line 86](../../../spec/tests.md#L86)).

Action: migrate legacy stories incrementally, but do not mark their traceability
as standards-complete until frontmatter and AC IDs are normalized. Prioritize
`US-021` because it is still planned/current work rather than purely historical
legacy context.

## Residual Risks

- SPIRE storage/wire/process specs are much more precise after this checkpoint,
  but coverage remains matrix-level rather than AC-level for faithful
  reimplementation evidence.
- Product-hardening confidence should not use `TC-034` as a broad proof until
  the missing hardening logs are packeted or the status language is narrowed.
- Stale `ix://agent-ix/tqvector/...` references may continue to hide broken
  dependency edges as the Ecaz spec grows.

## Open Questions

- Should legacy `traces:` artifacts be migrated before any future standards
  checkpoint, or should the master spec explicitly mark the legacy section as
  outside the standards-complete claim?
- Should Task 34 be reopened for an evidence-only packet that captures the
  missing hardening logs, or should `TC-034` be split into separate TC rows by
  lane maturity?
