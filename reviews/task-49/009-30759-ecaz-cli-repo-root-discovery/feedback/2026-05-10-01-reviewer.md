# 30759 — ecaz CLI repo root discovery review

Validation pass on `c923e3bb` (`Fix ecaz CLI repo root
discovery`). Closes packet 30756 P2 — operator install path is
unblocked for Stage G.

## What I checked

- New `repo_root()` function at
  `crates/ecaz-cli/src/commands/dev/support.rs:19` walks
  upward from three anchors:
  1. current working directory,
  2. compile-time `CARGO_MANIFEST_DIR`,
  3. current executable path (`std::env::current_exe()`).
- `find_repo_root_from(...)` (line 36) accepts only
  directories that contain *both* the workspace `Cargo.toml`
  and `crates/ecaz-cli/Cargo.toml` — two markers, so a bare
  crate directory or unrelated workspace cannot be mistaken
  for the repo root.
- Two unit tests (lines 244, 258) cover the positive
  (nested-checkout-path) and negative (crate-directory-only)
  cases. Both run as plain `cargo test -p ecaz-cli` — no pgrx
  involvement, so the recurring `SPI_finish` loader issue
  doesn't gate verification of this fix.
- After-fix proof reports successful install with
  `repo=/home/peter/dev/ecaz` and a recorded backend SHA256
  matching the assertion. Same operator command that failed
  in packet 30756 now runs cleanly.

## Answers to the packet's review questions

- **Marker-based repo-root discovery the right operator
  behavior for an installed binary run from a checkout?** Yes.
  The three-anchor walk handles every realistic operator
  invocation:
  - `cd $REPO && ecaz dev install ...` (CWD anchor),
  - `cd /tmp && /path/to/ecaz dev install ...` (executable
    anchor),
  - in-tree `cargo run -p ecaz-cli` (manifest anchor).
  Anchoring on `current_exe()` covers the `target/release`
  binary case which was the original failure.
- **Repo markers specific enough to avoid false positives?**
  Yes. Requiring **both** workspace `Cargo.toml` and
  `crates/ecaz-cli/Cargo.toml` rejects:
  - any unrelated workspace that happens to be in the
    walk-up path (single-marker false positive),
  - the `crates/ecaz-cli/` crate directory itself
    (single-marker false positive — covered by the negative
    test).
  Could be tightened further by requiring a third marker
  (e.g. `plan/tasks/30-spire-ivf-foundation.md`), but two
  markers is enough discrimination for the realistic
  ambiguity surface.
- **Sufficient to close 30756 P2 before Stage G harness
  depends on `ecaz dev install ecaz-pg-test`?** Yes. Stage G
  runbook depends on operators being able to run
  `ecaz dev install` without knowing about
  `CARGO_MANIFEST_DIR` quirks; that's now resolved. No
  outstanding installer-side blockers for Stage G.
