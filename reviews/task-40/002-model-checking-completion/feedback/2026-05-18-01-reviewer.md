## Non-blocking follow-ups

1. **Raise Shuttle iteration budget.** `hardening/shuttle/src/lib.rs:70`
   and `:122` use `shuttle::check_random(..., 128)`. 128 is the low end
   for Shuttle; the state spaces here are small enough that
   `shuttle::check_dfs` would explore exhaustively, or a budget of
   1000+ random schedules would be a reasonable raise.
2. **Add a seed sweep to sim-spire.** Task 40 spec asked for "1000
   randomized network schedules per scenario"; the current packet runs
   each Turmoil scenario once with the default deterministic seed.
   Parameterize the seed and add a `--seeds N` loop (or wrap each `#[test]`
   in a seed-iteration helper) so the lane really exercises the schedule
   space, not a single point in it.
3. **Strengthen the Shuttle epoch-publish concurrency model.** Writer and
   scanner currently share a single `Mutex<SpireEpochPublishModel>`, so
   Shuttle effectively explores schedule ordering of a serialized model
   — the "scanner never observes Begun/Partial" property reduces to a
   single-thread API check exercised across thread orderings. A version
   using `Arc<RwLock<...>>` or separate atomic fields would exercise
   true concurrent observation.
4. **Document injected-bug counterexamples for shuttle-real and
   sim-spire-remote.** Loom's discovered race is recorded in 001; for
   completeness, the closeout should record at least one
   counterexample input per lane (e.g. a deliberately broken
   dedupe-winner choice that Shuttle catches, or a deliberately reordered
   transport flag set that sim-spire catches). Useful for future
   regression watch and confirms the lane actually has teeth.
5. **Orthogonal note:** `src/am/ec_hnsw/build_parallel.rs:62`
   `PgLockedDsmInsertStateCell::compare_exchange_acqrel_acquire` is
   implemented as load-then-store, not a real CAS. The Loom-modeled
   algorithm assumes atomic CAS, so the model is honest about its
   target semantics. The `ConcurrentDsm` enum carrying this impl is
   currently `#[allow(dead_code)]`, so this isn't yet live. When the
   path is activated, the PG-locked variant either needs to actually
   perform an atomic CAS (under-the-lock) or the algorithm needs to be
   re-verified to be correct against a "load-then-store under the PG
   spin/LWLock" model. Worth tracking in the Task 41 / build_parallel
   follow-up surface, not in Task 40.
