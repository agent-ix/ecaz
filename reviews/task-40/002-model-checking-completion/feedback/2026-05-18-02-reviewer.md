### #5 — Decide the fate of `PgLockedDsmInsertStateCell` / `ConcurrentDsm`

The user notes they expected a parallel build was already shipped. To
clarify the current state:

- `src/am/ec_hnsw/build_parallel.rs` already implements a
  **leader-aggregate parallel build**: multiple worker processes
  ingest tuples through `shm_mq` queues, the leader assembles the
  serial graph from the aggregated tuples. That is the `SerialLeader`
  variant of `EcHnswBuildGraphAssembly` (`build_parallel.rs:106`), and
  it is the live mode in production.
- `EcHnswBuildGraphAssembly::ConcurrentDsm` and the surrounding
  `EcHnswConcurrentDsmGraphHeader`/`Node`/`GraphParts`/`Attachment`/
  `NodeLayout` are scaffolding for an additional mode where workers
  insert into a DSM-resident graph concurrently (no leader-assemble
  step). It is not wired into the build executor today; the variant is
  `#[allow(dead_code)]`.
- `PgLockedDsmInsertStateCell::compare_exchange_acqrel_acquire`
  (`build_parallel.rs:62`) is the PG-side trait impl that would bridge
  the Loom-modeled state machine to a real `pg_atomic_uint32`. As
  shipped it is load-then-store, **not** an atomic CAS. The Loom model
  assumes a real CAS, so this impl is unsound for its model the moment
  the variant goes live.

Pick one of the two cleanup paths and execute it now:

(a) **Fix in place.** Replace the load-then-store with PostgreSQL's
    native `pg_atomic_compare_exchange_u32` from `pg_sys::`. Run
    `make loom-real` against the corrected impl. Add a brief comment
    citing the lifted `concurrent_dsm_state.rs` algorithm and naming
    `pg_atomic_compare_exchange_u32` as the contract. Best option if
    concurrent DSM build is on the near-term roadmap.

(b) **Delete the scaffolding.** Remove the `ConcurrentDsm` variant
    from `EcHnswBuildGraphAssembly`, remove
    `PgLockedDsmInsertStateCell`, and the seven `#[allow(dead_code)]`
    DSM layout / attachment types in `build_parallel.rs:202–264`. Keep
    `src/am/ec_hnsw/concurrent_dsm_state.rs` (the lifted module) and
    the `hardening/loom` coverage — they document the intended protocol
    and the algorithm has been model-checked, so when someone later
    revives concurrent DSM they restart from a model-checked baseline,
    not a half-finished scaffold with a known broken impl.

The user has signalled openness to a future parallel-build optimization
slice, so (a) is reasonable if there's appetite for it in the next
slot; otherwise (b) eliminates the soundness bomb cleanly. **Do not
leave the broken impl behind `#[allow(dead_code)]` after this packet.**
