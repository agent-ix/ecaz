## Findings

### 1. (verification) Floor values appropriately conservative
for burn-in

Task 47's task doc names "documented recall floors per AM"
as the deliverable. The `docs/recall-floors.md` table sets
initial floors at conservative values for burn-in. **Report-
first** framing is critical here — IVF/DiskANN's first runs
will produce some natural variance and shouldn't trigger
red CI immediately. The "PR candidate, report-first" cadence
captures this correctly: the gate runs, output is captured,
but no PR is blocked until the variance band is understood.

### 2. (verification) Cost-gate positivity check is the right
opening move

The cost gate's first wiring is a **positivity check** — it
fails if any AM produces a non-positive modeled total cost.
That's correct because:
- Negative or zero costs would indicate a cost-model bug
  (the planner can't compare them sensibly)
- It's binary — no threshold tuning needed
- It catches a real class of bugs without false positives

The packet acknowledges that **baseline JSON / per-node cost
drift detection** is deferred to a follow-up — that's the
right sequencing. Drift detection needs:
1. A baseline (which doesn't exist yet)
2. A tolerance band (which depends on cross-platform variance
   data not yet collected)

Don't ship drift detection before the baseline is committed.

### 3. (suggestion) Cross-AM gate Jaccard / Kendall-tau details

The task doc mentions "cross-AM consistency (Jaccard /
Kendall-tau)" as part of the cross-AM gate's scope. The
current `cross-am-gate-small.json` and recall-floors.md don't
explicitly mention the consistency metric used. Worth one
line in `docs/recall-floors.md` clarifying:
- Whether the cross-AM gate currently uses Jaccard, Kendall-tau,
  or just pairwise recall comparison
- If it's deferred to a follow-up, name it explicitly

### 4. (suggestion) Fixture preload mechanism

The fixture contract section names the expected SQL prefixes
(`task47_gate_hnsw`, `task47_gate_ivf`, `task47_gate_diskann`).
But the loading mechanism isn't named in this packet. A
reviewer running the gate needs to know:
- Which `ecaz corpus` command preloads these fixtures
- Whether the gate fails loudly when a fixture is missing
  (vs. silently succeeding with no rows)

If the fixture-loading is in a separate packet (e.g., the
upcoming live-small-gate packet 002), worth a forward
reference. If it's already in `ecaz bench suite run`, worth
noting the failure-mode behavior in `docs/recall-floors.md`.

### 5. (sequencing note) Task 36 SIMD differential overlap

Task 36 (SIMD scalar differential) is another correctness
gate landing in the same time window. The two tasks have
non-overlapping scope:
- Task 36: scalar-vs-SIMD per-AM differential (catches SIMD
  encoding bugs)
- Task 47: exact-KNN-vs-AM recall floor (catches recall
  drift / ANN correctness)

But both eventually write to `target/gates/` or similar. Worth
ensuring the output paths don't collide. Not actionable here,
just a heads-up for the burn-in window when both lanes start
producing data.
