## Findings

### 1. Cost-gate now meaningfully discriminates per-AM

With three AMs baselined, the cost-gate can catch a regression
that's per-AM-specific (e.g., a DiskANN cost-model tweak that
flips the plan choice for IVF-shaped queries). Before this
packet, DiskANN drift was invisible. Now it's gated.

### 2. (carries from prior feedback) Per-node cost decomposition
still pending

This packet adds an aggregate row for DiskANN. The per-node
cost decomposition (item 12 in my 003 plan) — checking
`EXPLAIN (FORMAT JSON)` node-by-node rather than aggregate
totals — remains open. With three AMs now baselined at
aggregate level, per-node would be the next logical
refinement.

### 3. (cross-reference) Real-corpus rows still pending

Item 11 in my 003 plan calls for real-corpus cost rows
(against the `m5_diskann_real10k` fixtures or DBpedia
OpenAI corpus). All current rows are 512-row synthetic.
Real-corpus cost numbers would expose cost-model behavior
that synthetic fixtures can't trigger (e.g., reltuples in
the millions, cost models that depend on selectivity
estimates from real distributions).

Tracked as separate work; not blocking for this packet.

### 4. Extension version bump for symbol availability

The packet notes that the new SQL function isn't picked up
by an already-installed extension version 0.1.1 — the dev
extension version didn't bump. This is a minor operational
detail but worth flagging:

- For production: when this lands in main, the next ECAZ
  release will need either a version bump (forcing reinstall)
  or an upgrade SQL script that registers the new function
- For dev: the `register-diskann-cost-snapshot.sql` workaround
  is fine for local; CI should ideally bump version when new
  SQL surface is added

If there's a Task 49 governance rule about SQL-surface
additions requiring version bumps, this packet should
reference it. If not, worth a one-line note in
`docs/contributing.md` (separate scope).
