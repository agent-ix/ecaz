## Confirmations

- **Wilson 95% CI is the right interval.** Better than
  normal-approximation for small samples — recall is bounded
  in [0, 1] and the normal approximation can give nonsensical
  bounds (negative or >1.0) at small n or extreme p. Wilson
  doesn't. Right call.
- **Trial count = `queries × k`.** Each (query, rank-position)
  pair is a Bernoulli trial. CI over `queries × k` trials is
  a tighter bound than over `queries` alone, and it's
  defensible because each rank position is a separate
  observed hit/miss event.
- **Missing predictions = zero-recall queries.** Critical
  detail. If 64 queries are expected but the runner only
  emitted 60 predictions, the gate shouldn't ignore the 4
  missing ones — they're worst-case failures. Counting them
  as zero-recall preserves worst-case visibility. ✓
- **Legacy field preserved.** `recall@k` still in the table.
  Existing gate configs that threshold on `recall@k` still
  work. Backward compatibility maintained.
- **`recall_ci95_low` threshold is deferred.** The packet
  explicitly does NOT change gate behavior — adds data only.
  This is exactly the right discipline for the burn-in phase:
  collect the data, see how it correlates with `recall@k`
  across multiple runs, then decide threshold semantics.
  Avoids the "we added a new gate metric and it failed CI
  immediately because we didn't know the variance band yet"
  failure mode.

## Findings

### 1. (substantive verification) Trial-count choice

Two reasonable interpretations of "trial count" for Wilson CI:
- **Per-query trial** (n=64): one observation per query,
  observed recall@10 for that query
- **Per-rank-position trial** (n=64×10=640): one observation
  per (query, top-k slot)

This packet uses the second (per-rank). That gives tighter
CI bounds. Reasonable tradeoffs:
- Per-rank assumes rank-position outcomes are independent
  Bernoulli trials. In practice they're correlated (a query
  with 0 recall has 0 hits at all 10 positions), so the
  true variance is higher than i.i.d. Wilson assumes.
- Per-query is more honest about independence but gives
  wider bounds.

**Both are defensible.** Per-rank is more standard in IR
recall literature. Worth a one-line note in
`docs/recall-floors.md` (or the implementation comment)
naming the choice — future readers might wonder.

### 2. (cross-reference) Per-query percentiles

Item 20 in my Task 47 003 plan called for `p10`, `p50`, `p90`,
`min`, `max`, `mean` per the standard recall-table shape. This
packet lands the percentiles (p10, p50, p90) but doesn't
mention `min` / `max` / `mean`. They might be implicit:
- `mean` = `recall@k` (already there)
- `min` and `max` could be derived from `p0` / `p100` —
  acceptable to defer

Suggest adding `recall_min` / `recall_max` to the table in a
follow-up if the per-query distribution shape becomes
debugging-relevant. Not blocking.

### 3. (verification) Calibration deferral is explicit

`docs/recall-floors.md` updated to document the new fields
and explicitly states "hard-thresholding on `recall_ci95_low`
is deferred until calibration, because current floors came
from single observed runs."

This is the right framing. My 004 feedback flagged the
"single-measurement floor at tight margin" concern; this
packet adds the CI data so future calibration packets have
the raw signal needed to set defensible thresholds.

The calibration packet itself (item 9 in 003 plan) is the
next logical step.
