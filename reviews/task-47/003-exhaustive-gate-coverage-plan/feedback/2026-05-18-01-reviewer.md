### Items 1-2: exact-KNN baseline cache

**Spec from task doc:**

> `(query, k)` pair and caches the result under
> `fixtures/exact-knn/`. Cache keyed on `(corpus_hash,

**Implementation sketch:**
```

  <corpus_hash>/
    <query_hash>-<k>-<metric>.bin    # serialized top-k tuples
    manifest.json                     # human-readable index
```

**Acceptance:**

- Hash key includes all four fields (corpus, query, k, metric)
- Cache hit short-circuits computation; cache miss recomputes
  and writes
- Gate runs reference cached truth; CI doesn't recompute

### Items 5-7: cross-AM metrics

**Jaccard** = |A ∩ B| / |A ∪ B| on top-k membership.
Measures "do they agree on which results to return" without
caring about order.

**Kendall-tau** measures rank-order correlation. Penalizes
"AM A puts result X first, AM B puts it tenth" even if both
include X.

**Why both:**
- Jaccard catches membership drift (one AM finds a result the
  other missed)
- Kendall catches ordering drift (both find the same set but
  rank them differently)

A planner change that swaps tie-breaking is Kendall-visible
but Jaccard-invisible. A scoring bug that drops one result
from the set is Jaccard-visible but may also drop Kendall.

**Threshold policy:**
- Jaccard ≥ 0.7 per pair at k=10 → no log
- 0.5 ≤ Jaccard < 0.7 → log to packet, no fail
- Jaccard < 0.5 → gate fails (real regression suspected)
- Kendall similar bands

These are starting values; calibrate per item #9.

### Item 8: confidence intervals

Single-measurement floor at 0.85 doesn't account for
variance. With 64 queries, the standard error on a 0.85
recall estimate is ~√(0.85·0.15/64) ≈ 0.045. So a single
0.85 observation is consistent with true recall anywhere
in [0.76, 0.94] at 95% CI.

**Implementation:**
- Wilson score interval (better than normal-approx for small
  samples)
- Report `recall = X (95% CI: [low, high], n=Q)` in gate
  output
- Floor comparison: observed.low ≥ floor (conservative) OR
  observed ≥ floor (current behavior; relaxed)

**Decision required:** which comparison mode is the gate?
Recommend conservative (observed.low ≥ floor) to avoid
false-pass on small samples.

### Item 9: floor recalibration

**Current floors are from single measurement on synthetic
CI fixture.** That's:
- Conservative for the CI fixture (single run could overshoot)
- Aggressive for any other fixture (synthetic CI is uniform —
  easier than real data)
- Uncalibrated for variance

**Calibration approach:**
- Each gate × each fixture × 3 random seeds
- Measure recall@k per run
- Floor = `observed_mean - 2σ` OR empirical 5th percentile,
  whichever is more conservative

**Output:** `docs/recall-floors.md` raw table committed in the
calibration packet, plus updated floor values.

### Item 14: adversarial corpus

This is the hard one. Approach:

1. Generate cluster centers at random in d=32 space
2. Place query points midway between adjacent clusters (so
   nearest neighbor is in either cluster with ~50% probability)
3. Add high-dim isotropic noise to vectors so distance
   ordering is brittle near the query
4. Verify exact-KNN baseline still resolvable (queries should
   have unambiguous true top-k even if heuristics struggle)

**Acceptance:** floors on the adversarial corpus are
documented separately and may be much lower (e.g., HNSW 0.65,
IVF 0.60) — the point is to lock the floor, not match the
uniform-corpus floor.

### Item 20: per-query + quartile reporting

Aggregate recall hides worst-case behavior. A gate that
reports `recall = 0.85` could be:
- Every query at 0.85 (uniformly mediocre — usually safe)
- 90% at 1.00 and 10% at 0.0 (catastrophic on a subset)

**Extended report shape:**
```
HNSW ef_search=128 k=10 (n=64):
  mean:    0.8500 (95% CI [0.76, 0.91])
  p10:     0.7000
  p50:     0.9000
  p90:     1.0000
  min:     0.5000  <-- worst query
  max:     1.0000
```

Floor can apply to mean OR p10 OR min. **Decision required**:
which gate triggers? Recommend p10 ≥ floor for a tighter
guarantee against worst-case regressions.
