# Review: Packet 285 — C1 ADR-031 Persisted Binary Sidecar Feasibility

Reviewer: agent-ix (via code-review)
Date: 2026-04-12

## Assessment: Solid feasibility study and first implementation — two key gaps to close

### What landed

1. **Feasibility analysis**: Confirmed that a 192-byte binary sidecar fits as
   an optional trailing payload on element tuples without requiring an index-v2
   format change. Element grows from 842B to 1034B, still fits on 8KB page with
   m=8 neighbors.

2. **First implementation slice**:
   - `TqElementTuple` / `TqElementTupleRef` support optional trailing binary
     words
   - Bulk build writes persisted binary words on the no-QJL 4-bit lane
   - Scan prefers persisted sidecar when present, derives as fallback
   - Insert path writes old shape (`binary_words = []`) for now

3. **All gates green** (cargo test, pgrx test pg17, clippy)

### On the backwards compatibility story

The approach is sound: decode distinguishes old vs new tuples from tuple length,
`update_raw_tuple` enforces same-length rewrites so old tuples stay old, new
inserts into old indexes can write new-format tuples without breaking old-format
reads. This is the right level of compatibility for a contained extension.

### Gap 1: Missing cold-path measurement

The packet explicitly flags this: "no measurement yet for whether persisted
sidecars materially improve the cold real-corpus seam." This is the most
important open question. The warm path already works well with derive-on-miss
(packet 281 showed derive adds negligible cost when elements are cached). The
persisted sidecar is primarily justified by cold-path performance.

**Before investing more packets in sidecar polish, the cold-path measurement
must happen.** If persisted sidecars don't measurably help cold startup, the
192B per element is wasted storage overhead (~23% larger element tuples).

### Gap 2: Insert path not wired

Incrementally inserted elements write `binary_words = []`, so any index with
ongoing inserts will have a mix of sidecar-present (bulk-built) and
sidecar-absent (inserted) elements. Scan handles this correctly via the
derivation fallback, but it means:

- Inserted elements always pay the derivation cost on first cache miss
- The index is in a permanently mixed state until rebuilt

This is acceptable for validation but should be closed before the sidecar is
considered production-ready.

### On the page-fit analysis

The numbers check out:

- 842B (current) + 192B (sidecar) = 1034B element
- With m=8 neighbor payload (99B), element+neighbor pair = 1133B
- 8KB page with header/alignment overhead has ~8000B usable
- Comfortably fits, with room for the line pointers

The higher-level insert cap would shrink slightly, but the packet correctly
notes this is not catastrophic. At m=8 with max_level typically 3-4, the
impact is minimal.

### Structural recommendation

The persisted sidecar is the right long-term shape for binary codes — it
eliminates derivation cost on both cold and warm paths and enables future
optimizations like direct-from-page binary scoring without copying. But the
sequencing should be:

1. Measure cold-path impact first (next step flagged by the packet)
2. Wire insert-path sidecar writes
3. Only then consider the sidecar production-ready

### Verdict

Good feasibility study and clean first implementation. The two gaps (cold-path
measurement, insert-path writes) are correctly identified by the packet itself.
The backwards-compatible trailing-payload approach is the right design choice.
