# Feedback: Zero-Allocation Code Scoring

**Reviewer:** outside-review  
**Date:** 2026-04-05  
**Verdict:** Approved

## Answers to review questions

1. **Long-term boundary for `score_ip_codes_lite`**: Yes, this is the right boundary. Build-time code-to-code scoring only needs the MSE component (symmetric, no query-side rotation). The QJL component is inherently asymmetric (query-to-candidate), so omitting it in code-to-code scoring is correct per ADR-007. The name `codes_lite` communicates that this is the MSE-only path.

2. **Remaining callers constructing fake payloads**: `score_scan_element_result` in scan.rs:471-474 still constructs a `Vec<u8>` payload with `[gamma][code_bytes]` for the prepared-query scorer. This is a different path (asymmetric query-to-candidate), so the allocation is justified — the prepared query scorer needs the full payload including gamma and QJL. However, when scan scoring becomes a hot path in ordered traversal, this per-element Vec allocation will need attention. A `score_ip_from_parts(prepared, gamma, code_bytes)` method that avoids the intermediate allocation would be the right next step.

3. **Assert-based validation**: `assert!`-based invariants for raw code length are appropriate for this internal path. The caller (`score_code_inner_product`) already validates inputs at the SQL boundary. Internal assert is the right level.

## Additional notes

- Clean performance fix. The MSE-only code-to-code path is well-justified by the quantizer design.
