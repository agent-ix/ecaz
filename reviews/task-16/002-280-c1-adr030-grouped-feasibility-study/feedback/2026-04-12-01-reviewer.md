# Review: Packet 280 — C1 ADR-030 Grouped Scoring Feasibility Study

Reviewer: agent-ix (via code-review)
Date: 2026-04-12

## Assessment: Valuable negative result — correctly defers ADR-030

### What landed

Extended `approx_score_study` with grouped-f32 and grouped-u8 modes at group
sizes 8, 16, and 32 on the real 10K corpus. Compared directly against the exact
scorer on the same surface as the ADR-031 study in packet 279.

### On the results

The critical finding is that the rank correlation degrades rapidly with group
size:

- `group_size=8`: rho=0.798
- `group_size=16`: rho=0.702
- `group_size=32`: rho=0.625

All substantially worse than ADR-031's rho=0.932. The speedups only become
meaningful at larger group sizes where quality is already unacceptable.

### The right diagnosis

The packet correctly identifies that the problem is **not** LUT quantization
(grouped-u8 tracks grouped-f32 almost exactly at group_size=16). The problem is
the grouped reinterpretation itself on tqvector's per-dimension scalar-coded
format. This is the right conclusion — it means ADR-030 cannot be a
"just reinterpret the existing codes" change. It needs a different encoding.

### What this means for sequencing

ADR-030 is now correctly positioned as a future effort requiring:

1. New grouped encoding (closer to true grouped PQ or FAISS-style IVF-PQ4)
2. Likely a new build-time encoding pass
3. Possibly a page layout change for the grouped code format

That makes it a much larger project than ADR-031, which works on the existing
encoding. The deferral is the right call.

### One forward-looking note

If ADR-030 is ever revisited, the grouped encoding would need to coexist with
the current scalar encoding (for backwards compatibility with existing indexes)
or require an explicit index rebuild. The metadata page currently has no format
version field (noted in packet 285's storage readout). Adding one before ADR-030
would simplify the migration story.

### Verdict

Clean negative result. Correctly rejects grouped scoring on the current format
and correctly identifies what ADR-030 would actually require. Good use of the
study harness for cheap feasibility validation before wasting runtime
integration effort.
