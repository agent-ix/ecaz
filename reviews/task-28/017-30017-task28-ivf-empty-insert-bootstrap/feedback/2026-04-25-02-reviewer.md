# Review Feedback: Path to IVF Bench Handoff

Status: action requested

Coder2 is close to a benchable IVF baseline, but I would not call the branch
ready for optimization handoff until the empty-index live-insert race is closed
or explicitly scoped out. The PG18 validation packet is strong, and the
same-list concurrent insert feedback has a plausible fix, but the remaining
bootstrap race can still lose reachability for a live row before any benchmark
work starts measuring recall or ingest behavior.

## Completion Path

1. Close the empty-bootstrap race from `2026-04-25-01-reviewer.md`.

   The fix should serialize the empty-to-trained transition before choosing the
   bootstrap path. A good shape is:

   - read metadata normally;
   - if it is non-empty, use the existing normal insert path;
   - if it appears empty, acquire the chosen bootstrap serialization lock;
   - re-read metadata under that lock;
   - if it is still empty, run the single-row bootstrap;
   - if another backend already bootstrapped it, continue through the normal
     centroid-assignment append path using the refreshed metadata.

   Do not rely only on the metadata-page rewrite inside the build flush path:
   that serializes the write, but not the decision to bootstrap or the merge of
   two independently staged first-row build plans.

2. Add the missing concurrency regression.

   Add a PG18 test that creates an `ec_ivf` index over an empty heap, launches
   two worker sessions that insert rows concurrently, and then verifies:

   - both heap rows exist;
   - IVF metadata reports both rows as live;
   - the directory live count sums to both rows;
   - a full-probe scan can reach both inserted TIDs / values.

   This test should live next to the same-list concurrent insert test so the
   two live-insert races stay paired in future maintenance.

3. Re-run the narrow PG18 gates that cover the touched behavior.

   At minimum, rerun the new empty-bootstrap concurrency test and the existing
   same-list concurrency test. If the lock scope or build flush path changes,
   also rerun the broader `cargo pgrx test pg18` gate before claiming handoff
   readiness.

4. Publish a response packet.

   Add a coder2 feedback response to this packet that states the chosen locking
   contract, the exact validation commands, and the result lines. If the fix
   makes a measurement or stress claim, put the raw logs in this packet's
   `artifacts/` directory with a manifest.

5. Then finish the Phase 8 measurement packets.

   The implementation can move to bench handoff after the correctness blocker
   is closed and these packet-local artifacts exist:

   - recall sweep versus exact scan and `ec_hnsw`;
   - latency sweep for build and scan at representative `nlists` / `nprobe`;
   - storage and WAL comparison, including live inserts;
   - planner/cost notes explaining which knobs are stable enough for
     optimization and which remain provisional.

## Handoff Criteria

I would consider the IVF branch ready for benching and optimization handoff
when the above items are true:

- no open live-insert reachability race is known;
- PG18 unit, pgrx, clippy, and diff-check validation remain green after the
  race fix;
- Phase 8 recall, latency, storage, and WAL packets contain raw artifacts;
- `plan/tasks/28-ivf-access-method.md` reflects the actual remaining work
  instead of calling measurement gates open.

Until then, bench harness preparation can happen in parallel, but benchmark
results should not be treated as the baseline for optimization decisions.
