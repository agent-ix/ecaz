# Review Feedback: Concurrent Empty-Index Bootstrap Race

Status: action requested

## Finding

- `src/am/ec_ivf/insert.rs:18` reads metadata before any bootstrap
  serialization, and `src/am/ec_ivf/insert.rs:38` sends every inserter that
  sees `metadata.dimensions == 0` through `bootstrap_empty_index`. That path
  stages a single-row build and calls `build::flush_build_plan`
  (`src/am/ec_ivf/insert.rs:137`), which appends data pages and then rewrites
  the metadata page (`src/am/ec_ivf/build.rs:380`).

  Two concurrent inserts into an index built empty can both observe the empty
  metadata page, append independent one-row data page chains, and then race the
  final metadata rewrite. The last metadata writer wins, so the losing
  inserter's posting pages remain on disk but are not reachable from
  `centroid_head` / `directory_head`; scans, vacuum, and admin snapshots only
  see the winner's first row. This is the empty-index version of the same-list
  directory race: the later metadata write is serialized at the page level, but
  the decision to bootstrap and the build-plan merge are not serialized.

## Requested Change

Serialize the transition from empty metadata to trained metadata before
declaring live insert complete:

- take a relation-level or metadata-page-level bootstrap lock before deciding
  that `metadata.dimensions == 0` should stage a single-row build; then re-read
  metadata under that lock;
- if the index is still empty, perform the single-row bootstrap;
- if another session already bootstrapped it, route the insert through the
  normal non-empty centroid-assignment append path against the refreshed
  metadata.

Please add PG18 coverage with two worker sessions inserting into an IVF index
created over an empty heap. The test should assert heap count, metadata live
count, directory live count, and full-probe scan reachability for both inserted
rows.
