## Code Read

I read the code at the three commits cited by the packet, focusing on the
rerank surface.

### `src/am/ec_ivf/scan.rs` `rerank_probe_candidates` (lines 691–710)

```rust
super::options::RerankMode::HeapF32 => {
    let rerank_len = resolve_rerank_len(index_options.rerank_width, candidates.len());
    unsafe { rerank_probe_candidates_heap_f32(scan, opaque, &mut candidates[..rerank_len]) };
    candidates[..rerank_len].sort_by(candidate_cmp);
}
```

**Sharp edge — heterogeneous score domains in the returned vector.**
`candidates` is pre-sorted by approximate (probe) score before this call.
After this branch runs, the first `rerank_len` entries hold exact f32 IP
scores and are sorted; the tail (indices `rerank_len..candidates.len()`)
keeps the approximate quantized scores from the probe pass and is *not*
re-sorted against the exact-scored head.

For the lane being measured (`LIMIT 10`, `rerank_width = 25`) this is
correct: the executor only ever reads from the exact-scored head. But the
shape leaks: a caller using `LIMIT 50` with `rerank_width = 25` would
receive a result whose first 25 are correctly ranked by exact score and
whose next 25 are approximate-scored and not order-comparable to the head.
The two score domains are not strictly comparable (negative IP with
quantization noise vs. negative IP exact), so an approximate score in the
tail can be smaller than the worst exact score in the head and not
actually correspond to a better neighbor.

**Recommendation (non-blocking for this packet, blocking before broader
recall claims):** when `rerank = 'heap_f32'` and `rerank_width > 0`,
truncate `candidates` to `rerank_len` instead of keeping the approximate
tail. Either:

```rust
candidates.truncate(rerank_len);
candidates.sort_by(candidate_cmp);
```

or document `rerank_width` as the effective scan-output cap (and reject
`LIMIT > rerank_width` plans, or warn). Truncation is the simpler and
more conservative fix; the tuning summary's frontier already implies
rerank_width is the effective top-k.

### `src/am/ec_ivf/scan.rs` `resolve_scan_snapshot` (lines 795–810)

```rust
unsafe fn resolve_scan_snapshot(scan: pg_sys::IndexScanDesc) -> (pg_sys::Snapshot, bool) {
    if !unsafe { (*scan).xs_snapshot }.is_null() {
        return (unsafe { (*scan).xs_snapshot }, false);
    }
    let active_snapshot = unsafe { pg_sys::GetActiveSnapshot() };
    if !active_snapshot.is_null() {
        return (active_snapshot, false);
    }
    let registered_snapshot = unsafe { pg_sys::RegisterSnapshot(pg_sys::GetLatestSnapshot()) };
    ...
}
```

**Concern — falling back to `GetLatestSnapshot()` for heap rerank fetches.**
`xs_snapshot` is set by the executor before `amgettuple`; the
`GetLatestSnapshot()` fallback would only fire under a code path that has
no executor-supplied snapshot, in which case fetching the heap row at the
"latest" snapshot may return a row version newer than what the caller's
MVCC view should see. For an IVF scan that's about to score-and-return a
heap TID, that drift can change the score and (worse) score against a
heap row that is logically invisible to the user query.

**Recommendation:** keep the `xs_snapshot` and `GetActiveSnapshot()` legs;
remove the `GetLatestSnapshot()` leg in favor of `pgrx::error!`. Under
normal Postgres index-scan execution, one of the first two is always
populated. If a deferred path needs a different policy (e.g. tooling
calling the AM directly), prefer making the snapshot explicit at that
call site than papering it over here.

### `resolve_rerank_len` and `rerank_probe_candidates_heap_f32`

Both look correct:

- `resolve_rerank_len` clamps to `candidate_len` and treats `<= 0` as
  "rerank everything" — matches the `rerank_width = 0` documentation.
- The fetch loop's resource discipline is clean: `drop(source_vector)`
  before `ExecClearTuple(slot)` (the slice borrows into the slot datum),
  `ExecDropSingleTupleTableSlot` after the loop, owned-snapshot and
  owned-relation flags release in the right order.
- `negative_inner_product` matches the lower-is-better candidate-heap
  convention used elsewhere in the AM.

Minor nit: the loop calls `source::resolve_indexed_ecvector_attribute`
once outside the loop (good) and pays one heap fetch per candidate. At
`rerank_width = 25`, that's up to 25 random heap reads per query; the
measured p50 (100–435 ms across the 10k/25k slices) is dominated by this
and by the unaccelerated f32 IP that the SQL-facing rerank path
intentionally preserves. No action needed; just naming the latency
source.

via the cited commit and packet IDs. Numbers are internally consistent:
recall@10 = 1.0000 at `nlists=32, nprobe=32, width=25` on the 10k slice
(p50 ≈ 178 ms) and the same configuration holds recall on the 25k slice

45.0 s). `nlists=64`/`128` did not buy anything on this fixture, which is
the expected shape for IVF on a uniformly-clustered embedding when the
list count exceeds the natural cluster count of the corpus.

The summary correctly flags that this is a *local* tuning lane on a
DBPedia-derived fixture and not a substitute for the ann-benchmarks /
Graviton product surface.
