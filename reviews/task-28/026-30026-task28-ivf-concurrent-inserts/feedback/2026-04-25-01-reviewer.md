# Review Feedback: Same-List Live Insert Serialization

Status: action requested

## Finding

- `src/am/ec_ivf/insert.rs:50` loads the assigned list directory before the
  append, then `src/am/ec_ivf/insert.rs:70` applies insert stats to that stale
  in-memory copy and `src/am/ec_ivf/insert.rs:72` rewrites the whole directory
  tuple. The new metadata helper fixes the global counter race by re-reading
  metadata under the metadata-page exclusive lock, but there is no equivalent
  read-modify-write path for the list directory tuple.

  Two concurrent inserts assigned to the same IVF list can both read the same
  `live_count`, `inserted_since_build`, and `tail_block`, append postings under
  the page lock, and then race their directory rewrites. If both postings fit on
  the existing tail page, the list counters can lose one increment. If one
  insert allocates a new posting block and the older directory rewrite wins
  last, the directory can move `tail_block` back to the old block, making the
  newly appended posting block unreachable by scans and vacuum. This is more
  than an admin-stat drift issue because scan materialization bounds its result
  heap from directory `live_count` and reads posting blocks only through the
  persisted head/tail range.

## Requested Change

Add same-list serialization before treating concurrent live insert as complete:

- either update the directory tuple through an exclusive read-modify-write
  helper that re-reads the latest directory state under the directory-page lock
  after the append, then merges the new posting block/count into that latest
  state;
- or take a per-list relation/advisory lock around the directory load, append,
  and directory rewrite path.

Please add PG18 coverage with two concurrent sessions inserting vectors that
route to the same list. The test should verify metadata live count, that the
assigned directory live/inserted counters reach both increments, and that a
full-probe scan can reach both inserted heap TIDs after the concurrent inserts.

## Validation

- `cargo check --no-default-features --features pg18 --tests`
