# Feedback: A5 Insert Level Allocation and Metadata Promotion

**Reviewer:** Claude (Opus 4.6)
**Date:** 2026-04-11
**Commit reviewed:** `45bdb21`

## Scope match

Matches the packet: `tqhnsw_aminsert` now derives a real HNSW level from
`splitmix64(seed ^ encode_heap_tid(heap_tid))`, `append_heap_tuple` pre-sizes
neighbor tuples to `neighbor_slots(level, m)` with `INVALID` fills, and the
post-append metadata repair promotes both `entry_point` and `max_level` when
either is invalid or the new node outranks it. No forward/back edges, no
neighbor mutation. The check stays within the declared slice.

## Findings

### 1. Level formula is the standard HNSW draw — correct

`level_from_random_bits` computes `floor(-ln(u) / ln(m))` where
`u = (bits + 1) / (u64::MAX + 1)`. That is the `floor(-ln(u) * mL)` form with
`mL = 1/ln(M)` — the HNSW paper's level sampler. The `+1` numerator nudges
`bits = u64::MAX` to `u = 1.0` so `-ln(1) = 0` and the level is `0`, avoiding a
`-0.0` edge; `bits = 0` gives `u ≈ 2^-64`, which produces a large level that
then clamps to `max_level`. Both ends are safe.

Minor note worth a comment: `(u64::MAX as f64) + 1.0 == u64::MAX as f64` because
`f64` rounds `2^64 - 1` up to `2^64`, so the denominator is exactly `2^64`. The
arithmetic still works, but a one-liner in the helper explaining the `+1` on the
numerator (protecting against `ln(0)`) would keep the next reader from
"simplifying" it.

### 2. Determinism across heap TID reuse is acceptable for this slice — flag it

`choose_insert_level` mixes only `seed` and `heap_tid`, so if a heap TID is ever
recycled (VACUUM FULL, cluster rewrite, HOT update landing on the same
`(block, offset)`), the new row gets the *same* level as the old one. For this
slice that's fine — the packet is pre-delete-support — but it will become a
correctness concern once delete/rebuild lands, because level assignment will no
longer be independent of prior fixture state. Worth recording in
`plan/tasks/06-graph-insert.md` as a A5 follow-up so the next slice that touches
insert doesn't accidentally ship a stable level per TID.

### 3. Page-fit cap is correct but much tighter than it looks at realistic `m`

`max_insert_level_that_fits` walks down from `max_level_that_fits(m, page_size)`
until `neighbor_tuple + element_tuple` for that level fits on one fresh page. At
`m=2, dim=4, 4-bit code` (the test fixture), levels up to several fit. At
production `m=16, dim=1536, 4-bit` the neighbor tuple alone at level `L` is
`2*16 + 16*L = 32 + 16L` tids × 6 bytes + header, plus an element tuple of
`~800 B`. On an 8 KiB page the pair stops fitting around `level ≈ 7`, and in
practice `-ln(u)/ln(16)` rarely draws above ~2, so this cap is mostly a
correctness backstop rather than a live limiter. Not a defect; just worth
confirming with a one-line assert-or-log in the next slice so the cap's silent
truncation can't mask a regression where the whole draw collapses to level 0.

### 4. Metadata repair/promotion preserves the invariant — but the re-check is
   subtly racy on `max_level` promotion only

The fast path takes the snapshot outside the metadata lock and only reacquires
the exclusive metadata lock when *either* `entry_point` is invalid *or*
`insert_level > snapshot.max_level`. Inside the lock it re-checks the same
predicate against the live metadata. That's correct.

However, the new element tuple has already been appended (and its TID is
`element_tid`) before the metadata is repaired. If two concurrent inserts both
draw `level = 5` against a current `max_level = 3`, both will take the slow
path, both will re-check under the lock, the second one will see
`metadata.max_level = 5` after the first commits, and the predicate
`insert_level > metadata.max_level` will be *false* — so the second insert will
*not* promote. Both nodes exist at level 5, but only the first is advertised as
the entry point. That is the intended outcome (entry_point is a single TID), and
it preserves the `metadata.max_level == element_at(entry_point).level` invariant
as long as the first insert's element tuple is still alive. No bug; calling out
that the invariant hinges on "entry_point is still valid when promotion happens"
is worth a `// SAFETY:` comment on the re-check block so the next slice
(delete/vacuum) doesn't break it by allowing the entry element to be GC'd before
max_level is updated.

### 5. Pre-sized neighbor tuples now carry `INVALID` fills — good, but confirm
   scan-layer tolerance is actually tested elsewhere

The packet's claim is that the scan/runtime layer-slicing helpers already
"expect stable `2M` / `M` slot boundaries." Before this change, live-insert
neighbor tuples had `count = 0` and empty `tids`, so there was never an
`INVALID` tid in a live-insert path to hit. Now there are. The packet's own
tests assert *shape* (slot count matches `neighbor_slots(level, m)`), not *scan
behavior* over the new shape.

**Required check before this pattern becomes load-bearing in 230+:** point at
the specific scan/iteration site that skips `INVALID` neighbor tids and confirm
it's covered by an existing test. From the 10050-range cleanup I'd expect
`skip_invalid_successor_neighbor_refs` (review packet 50) to handle this, but
the request should name the guarantee explicitly rather than leave it inferred.
If it is already tested, one sentence in the packet pointing at the test is
enough; if not, add a single assertion in
`test_tqhnsw_insert_neighbor_tuple_sizing_matches_levels` that runs an actual
SELECT via the index against the fixture and confirms results come back (proves
the scan survives the INVALID fills, not just that the tuples were written).

### 6. Test coverage is appropriate for the slice

- `test_tqhnsw_insert_neighbor_tuple_sizing_matches_levels` — 128 inserts, m=2,
  walks until an upper-layer node appears, then verifies *every* element's
  neighbor tuple matches `neighbor_slots(level, m)`. Good shape test.
- `test_tqhnsw_insert_promotes_entry_point_on_level_up` — walks the deterministic
  sequence, reads the pre-insert `max_level`, and after every insert checks that
  if the expected level exceeded the previous max, metadata got repointed. This
  is the right test for the promotion rule.
- `test_tqhnsw_insert_repairs_invalid_entry_point_after_shape_init` — updated to
  decode elements + neighbors and cross-check that the promoted entry TID
  actually identifies a live element at `metadata.max_level`. Strong.
- `test_tqhnsw_insert_reuses_new_tail_page_after_rollover` — rewritten to use
  `debug_insert_level_for_heap_tid` for the page-fit search so it picks a `dim`
  that still fits the new level-aware tuple size. Necessary; previously the
  search assumed `level = 0` and would have silently picked a `dim` that
  couldn't hold a real level-5 draw.

None of these tests mock the insert path; they all run real SPI inserts and
decode the on-disk pages. That's the standard this packet holds itself to, and
it holds.

### 7. `debug_insert_level_for_heap_tid` gating is right

The debug helper is `#[cfg(any(test, feature = "pg_test"))]` and re-exported
through `am::mod.rs` under the same cfg. It doesn't leak into release builds.

## Verdict

**Approve.** The slice lands exactly what the packet describes, the level
formula is the canonical HNSW draw, and the promotion rule preserves the
metadata invariant. Three items to record in `plan/tasks/06-graph-insert.md` as
follow-ups rather than blockers on this packet:

1. (Item 2) heap-TID-stable level assignment becomes a concern once delete
   lands — note it in the A5 delete slice prompt so the next author doesn't
   accidentally ship stable-per-TID levels across recycles.
2. (Item 4) add a `// SAFETY:` comment on the re-check block explaining the
   "entry_point element must still be alive when promotion commits" invariant
   before vacuum/delete touches insert.
3. (Item 5) in the 230 packet (forward links on new node), cite the test that
   proves the scan layer tolerates `INVALID` neighbor fills at `2M`/`M` stride
   — the guarantee is now load-bearing for every subsequent insert.

Item 1 and 3 (the comment + formula clarification) are both one-line edits that
could land on this branch or the next; they don't need to block 229.
