# Feedback: 11039-phase7g-insert-metadata-bookkeeping

Reviewed against head `5e7bd18` (impl commit `ee98778`).

## Verdict

Approved. Minimal, correct, and exactly the seam the 11002 reviewer
guidance asked for.

## Review focus answers

1. **Is the bookkeeping seam scoped right?**

   Yes. The helper touches exactly one counter
   (`inserted_since_rebuild`), reuses `with_locked_metadata_page(...)`
   for locking + WAL registration, uses `checked_add` so an overflow
   errors instead of wrapping silently, and deliberately does not
   touch `needs_medoid_refresh`. That matches the ADR-046 feedback on
   11002 verbatim — live insert owns the drift counter, maintenance
   owns the refresh flag.

2. **Are the pg assertions both-sided?**

   Yes. Bootstrap sees `== 1`, the second distinct insert sees `== 2`,
   and the duplicate-after-append case stays at `2`. All three also
   assert `needs_medoid_refresh == false`, which pins down the
   "insert does not write this flag" half of the contract.
   Together those three assertions make it impossible to accidentally
   collapse the seam later without a failing test.

## Notes

- Call-site ordering in `ec_diskann_aminsert` is right: the counter
  increments only after append and free-capacity backlink steps both
  succeed. If either errored, `pgrx::error!` would have unwound
  already, so the counter cannot advance for a failed unique insert.
- Reading metadata through `scan_state::materialize_chain_from_index`
  in the pg tests is a reasonable choice given the off-limits
  admin-snapshot surface; the helper is private to the test module
  and doesn't expand any public API.
- Duplicate-path untouched: the duplicate probe still errors before
  any state mutation, so "duplicate attempts don't advance drift" is
  a direct structural consequence, not just something the test
  happens to observe.
