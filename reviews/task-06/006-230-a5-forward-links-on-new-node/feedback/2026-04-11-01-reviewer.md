# Feedback: A5 Insert Forward Links on the New Node

**Reviewer:** Claude (Opus 4.6)
**Date:** 2026-04-11
**Commit reviewed:** `241a87d`

## Scope match

Matches the packet. `tqhnsw_aminsert` now calls
`discover_insert_forward_neighbor_slots`, which loads the metadata entry
element, scores it, descends via `graph::greedy_descend_from_entry`, then runs
`graph::search_layer0_result_candidates` with `ef_construction.max(1)` as the
window, and fills only the first `M` slots of the new node's layer-0 slice.
Remaining layer-0 slots and all upper-layer slots stay `INVALID`. No backlinks,
no shrinking, no upper-layer insert search. The empty-index path is unchanged.

## Findings

### 1. Traversal seam reuse is the right call

`greedy_descend_from_entry` + `search_layer0_result_candidates` are the same
helpers the scan path uses, so there is now exactly one layer-0 expansion
implementation in the crate. That is the seam this packet should be pinning. No
duplicate search loop, no insert-only variant. Good discipline.

The minor redundancy worth noting: `load_insert_entry_candidate` calls
`load_graph_element` on the entry TID, and `greedy_descend_from_entry` also
calls `load_graph_element` on the same TID to read `entry_element.level` before
it begins walking. That's one duplicated page read per insert against the
metadata entry, not a correctness issue, but it's a visible cost on the
hot-path. A follow-up slice could let the helper accept a pre-loaded
`GraphElement` (or return one from a combined
`load_entry_and_descend(...)` helper) so insert and scan share the same single
load.

### 2. Scoring function matches build, not scan — intentional and consistent

`score_insert_graph_element` calls `crate::score_code_inner_product`, which
dispatches to `ProdQuantizer::score_ip_codes_lite`. That scorer computes the IP
from the MSE code bytes only, ignoring the QJL tail and the per-element gamma.
Scan-side, in contrast, uses `score_ip_from_parts(prepared_query, gamma,
code_bytes)` which *does* factor in gamma.

I'd normally flag an insert/scan metric mismatch as a concern, but
`BuildCodeDistance::eval` at `src/am/build.rs:515` already uses the same
`score_code_inner_product` path during offline build, so the graph now has one
consistent build-time metric between `CREATE INDEX` and live insert. Both
diverge from scan-side scoring the same way, which is the convention this
crate's build path has always held.

**Required in the A5 closeout (not this packet):** promote this decision from
"implicit convention" to something ADR-shaped. Review packet
`195-a4-score-function-divergence` already raised "what metric does search
actually use" as a question; the insert path now concretely anchors the build
side to the answer. The ADR should state, in one paragraph: "graph construction
(both build and insert) scores by MSE-only IP via
`score_code_inner_product`; scan scores by the full IP with gamma via
`score_ip_from_parts`; the divergence is deliberate because gamma is a
per-query rotation that doesn't apply during construction." That gives future
maintainers a single place to re-derive the rule instead of rediscovering it in
two different modules.

### 3. Sign convention is correct but fragile — one-line comment would fix it

The scorer returns `Some(-crate::score_code_inner_product(...))`. Scan-side at
`scan.rs:1259` also returns `-quantizer.score_ip_from_parts(...)`. Both make the
"lower is better" direction match `BeamCandidate`'s ordering. Because the sign
flip lives three function calls deep in an `Option`-returning helper, a future
refactor that drops the leading `-` would silently invert every ordering in the
insert path without breaking any test that doesn't compare against a known
*ordered* list. The two existing tests only check "the link points at an
existing element" and "the second insert linked to the first" — both survive an
inverted score.

**Recommendation:** drop a single-line comment on `score_insert_graph_element`
("negate so lower-is-better matches BeamCandidate ordering"), or — stronger —
add a third test that inserts three distinct rows into a pre-built graph where
one candidate is *clearly* closer by IP than the others, and asserts the closer
candidate occupies slot 0. That gives the ordering a direct assertion.

### 4. `ef_construction.max(1)` is a defensive floor — verify it can't hide a
   config bug

The call passes `usize::from(metadata.ef_construction).max(1)` as the beam
width. If `ef_construction` is stored as zero (bad metadata, or a
pre-initialization read), the traversal silently degrades to a single-candidate
beam instead of failing loudly. Given that metadata is written during build and
validated elsewhere, this is probably dead defensiveness. But if
`ef_construction = 0` is actually unreachable, the `.max(1)` should be a
`debug_assert!` instead — a hidden minimum is a quiet way to mask a metadata
corruption bug. If it *is* reachable (e.g., via old indexes predating a schema
bump), then there's a silent behavior change for those indexes that should get
its own regression test.

Ask: which of the two is it? One line in the insert helper or the packet would
resolve it.

### 5. First-`M` truncation is the right interim contract

Filling only slots `[0..M)` matches the slice's own invariant: "we know the
top-M layer-0 candidates; we do not yet know the bottom-M shrink set." Leaving
the remaining slots `INVALID` preserves the "scan must skip INVALID neighbor
tids" contract raised in 229, and it makes the next slice (backlinks) trivially
identifiable — any slot that's still `INVALID` on a mature node is either "not
yet backlinked" or "backlink overflow pruned" depending on which slice lands
first.

Worth stating in the A5 roadmap comment thread so the next author doesn't
accidentally fill the second half of the layer-0 slice from the same candidate
list under the assumption that "we had ef_construction candidates anyway."

### 6. Tests are load-bearing and correct

- `test_tqhnsw_insert_populates_forward_links_from_live_entry_seed`: builds an
  empty index, inserts one row, inserts a second, then asserts the second
  row's first-M slots are exactly `[row1_element_tid]` and the rest are
  `INVALID`. This directly exercises the metadata-entry → single-element
  neighbor-fill path.
- `test_tqhnsw_insert_populates_forward_links_against_built_graph`: builds a
  4-row index via `CREATE INDEX` (so the graph is populated by the build path,
  not insert), inserts a fifth row live, then asserts `row5`'s first-M slots
  are all pre-existing element TIDs, none self-reference `row5`, and every
  slot beyond `M` is `INVALID`. This is the real test — it proves live insert
  finds neighbors in a graph it didn't build.

Both tests decode pages directly and compare TIDs rather than asserting on
scan output, so they cannot be masked by a downstream scan-path regression.

One gap (item 3 above): neither test asserts the *order* of the filled slots,
so an inverted score convention would still pass. Adding a third ordering-aware
test is the cheapest closing of that gap.

### 7. The no-self-reference assertion is doing real work

`populated_layer0_slots.iter().all(|tid| *tid != row5_element_tid)` catches
exactly the class of bugs where the layer-0 search accidentally returns the
just-appended element as its own neighbor (e.g., because the candidate filter
didn't exclude it). Since the new element was appended *before*
`discover_insert_forward_neighbor_slots` was called, this is unreachable on the
current code path — `search_layer0_result_candidates` is walking the graph
*from an existing entry*, and `row5`'s tuple wasn't yet linked into the graph
— but the assertion guards against a future refactor that swaps the order of
append-vs-discover. Keep it.

## Verdict

**Approve.** The packet lands exactly the slice it describes using the right
seams, and both tests actually execute the insert-side search path.

Recommended follow-ups (none block this packet, all should be tracked on A5):

1. **Item 2 (ADR):** fold the build-time/insert-time vs scan-time scoring
   divergence into the A5 closeout as an ADR or an addendum to the existing
   195 review. The insert path now ties this decision directly to user-visible
   runtime behavior — it deserves one paragraph of written rationale.
2. **Item 3 (ordering test):** add a three-element-at-different-distances test
   to the next A5 slice so the sign convention has a direct assertion; cheap
   insurance against a silent inversion later.
3. **Item 4 (`.max(1)`):** decide whether `ef_construction = 0` is reachable,
   and either promote to `debug_assert_ne!` or add a regression test.
4. **Item 1 (duplicate entry load):** fold into whatever slice touches the
   upper-layer insert search, since that will need to reuse the entry element
   anyway — the refactor pays for itself when two more helpers start calling
   `load_graph_element(metadata.entry_point, ...)`.

The build-time/insert-time metric alignment is the load-bearing win here: the
graph now has *one* construction metric, not two. That's a structural
improvement hidden inside a 3-slot forward-link fill.
