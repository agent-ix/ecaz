# Feedback: Insert Metadata Lock Scope

**Reviewer:** Codex (GPT-5)
**Date:** 2026-04-10

## Update

The lock-scope fix is now implemented in `src/am/insert.rs` and validated on current `main`.

### 1. The hot insert path no longer holds the metadata exclusive lock across the duplicate scan

`tqhnsw_aminsert` now:

- snapshots metadata with `read_metadata_page(...)`
- keeps the old exclusive path only for first insert / shape initialization
- validates shape against the snapshot once the index is initialized
- runs `find_duplicate_element_tid(...)` without the metadata exclusive lock held
- reacquires the metadata exclusive lock only to repair `entry_point` when the snapshot still showed
  `INVALID`

The accepted race remains the same one the request allowed: concurrent same-code inserts can
double-insert instead of coalescing if they overlap between scan and append. That is acceptable for
this packet and does not break query correctness.

### 2. The new regression closes the main test gap

Added `test_tqhnsw_insert_repairs_invalid_entry_point_after_shape_init` in `src/lib.rs`.

This forces an already-shaped index into `entry_point = INVALID`, then verifies that a subsequent
live insert repairs metadata back to a valid element tuple. That is the strongest practical
behavioral regression for the new fast-path recheck without adding invasive lock-test hooks.

### 3. The benchmark moved in the expected direction

Local concurrent insert workload against the scratch pg17 cluster:

- workload: 4 concurrent backends inserting 1,000 total rows into the same `tqhnsw` index
- before: `301 ms`
- after: `180 ms`

That is about a `40%` wall-clock reduction on this machine for the contention-sensitive path this
packet was targeting.
