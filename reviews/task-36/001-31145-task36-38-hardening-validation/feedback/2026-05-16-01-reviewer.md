# Review Feedback: Task 36 SIMD Differential + Task 38 Fault Smoke

Reviewer: Claude (loop cycle 1, 2026-05-16)
Head SHA reviewed: `a919bb21` (impl) + `6ea5efd7` (packet)
Reviewer focus from `request.md`:
1. SIMD tolerance choices and scalar reference isolation
2. Whether the Task 38 provider boundary is explicit enough for follow-up
3. Whether application-name based postcondition checks are the right minimum
   live leak checks for the built-in fault lanes

## Verdict

**Task 36**: directionally correct foundation, but scope claim in the task
header overstates coverage. Several enumerated SIMD surfaces have no
differential test yet, and the design's "force every backend via test-only
entry points" requirement is not met. **Status should read "partial" rather
than "implemented locally"** until those gaps are closed or explicitly
descoped.

**Task 38**: the operator surface, provider boundary, and crate layout are
good. However, the live smoke lanes that *do* run mostly exercise PostgreSQL
infrastructure rather than ECAZ AM code paths — they will not catch the bug
classes the task was designed to surface. Provider boundary documentation is
fine; cancellation/timeout/resource probes need workload changes before they
mean what the request claims.

## Task 36 — SIMD/scalar differential

### Strengths

- `score_ip_from_parts_scalar_reference`, `score_ip_codes_lite_scalar_reference`,
  and `fwht_in_place_scalar_reference` are correctly gated behind
  `cfg(any(test, feature = "bench"))` and route to the existing private scalar
  paths instead of duplicating logic. Reference isolation is clean.
- `tests/simd_diff.rs` is `#![cfg(feature = "bench")]`, so production dispatch
  is unaffected and the harness cannot run in a release build by accident.
- Tolerances (`1e-5` absolute/relative) are documented in `docs/hardening.md`
  with a change-requires-justification policy — matches the task's exit
  criterion.
- `assert_close` uses `max(1.0)` to avoid pathological-zero false positives
  while remaining tight for unit-bounded vectors. Good choice.

### Gaps vs. task scope

1. **Per-backend pinning is missing (task §Approach step 2).** The task
   explicitly calls for `score_ip_from_split_parts_avx2_fma_for_test`,
   `_avx512_for_test`, `_neon_for_test` entry points so the host CPU does not
   silently skip a backend. The current harness only invokes the dispatched
   entry, which on this Linux/x86 host means **AVX-512 and NEON paths are
   never exercised**. On a NEON host AVX paths are skipped instead. The
   "production_1536_4bit_score_path_matches_scalar_reference" test only
   compares whichever backend the dispatcher selects.

   Either expose per-backend test entry points (preferred — that is what the
   task asks for) or document in `docs/hardening.md` and the packet that the
   current lane covers only the host-reachable backend, and gate the gap on a
   follow-up task with an issue link.

2. **Enumerated SIMD surfaces with zero differential coverage:**
   - `unpack_mse_indices` SIMD paths — only a single 3-bit bit-exact fixture
     (`three_bit_unpack_fixture_stays_bit_exact`). No SIMD-vs-scalar diff
     across 2..=8 bits, no proptest sweep.
   - `encode` and vectorized inner loops in the quantizer hot path — no
     differential test.
   - `src/quant/rotation.rs` — no differential test (task allows "if it grows
     arch-specific variants" — note explicitly whether that is the case
     today; if not, drop it from the task body so the scope reads honest).
   - `src/am/ec_*/scan.rs` SIMD distance accumulators (DiskANN/IVF/HNSW/
     SPIRE) — no differential test.

   Task 36 lists these by name in "Coverage targets". Either land them or
   move them into a "Phase 2" subsection of the task with the rationale.

3. **`mse_bits ∈ {3, 4}` not pinned.** The proptest range is `2u8..=8`, so
   the two production-critical widths are exercised stochastically across 24
   cases rather than always-on. Add explicit `prop_oneof![Just(3), Just(4),
   any_in_range]` or split into separate test fns; recall regressions for
   prod live in those two widths.

4. **Miri reference-path check (task §Approach step 3) not visible.** The
   task wants the *scalar* reference itself run under Miri to prove it is
   UB-free. There is no `simd_diff` Miri lane in the packet, and the
   `cfg!(miri)` fallback in `prod.rs` covers production dispatch, not the
   bench-feature reference functions. Add a small pure-Rust harness (or use
   `miri_` prefixed tests in `src/quant/prod.rs`) that calls
   `score_ip_from_split_parts_scalar` directly.

5. **Negative-control demonstration missing (task §Validation bullet 3).**
   "A deliberately mutated SIMD path (flip a sign in one branch) is caught by
   `simd-diff` and reported as a non-trivial diff in the packet." Nothing in
   `artifacts/` shows this. A one-off mutation diff captured to
   `simd-diff-mutation-control.log` would close the loop on "this lane
   actually catches the class of bug it's designed for".

### Nits

- `tests/simd_diff.rs` imports `EncodedTq` via `bench_api` but the type is
  re-exported anyway — fine, but a doc comment naming which scalar entry
  points are bench-only (so future maintainers don't accidentally make them
  `pub` in production) would help.
- The hardcoded production test uses `seed = 0x1536` / `0x4B17` for query/
  candidate. Pin those in a comment as "magic numbers tied to documented
  baseline, do not change without re-baselining tolerance".

## Task 38 — PG fault injection

### Strengths

- Clean crate layout: `ecaz-fault-injection` is pgrx-free and only owns
  fault model, workload SQL, leak probes, and provider env construction.
- Provider as a separate C source built by `build.rs` is the right call —
  avoids polluting the workspace with a Linux-only Rust dep.
- `assert_provider_marker` enforcement on slow-disk is exactly the discipline
  the request advertised: the lane *cannot* pass against a stock postmaster.
- The CLI refuses non-dry-run for `Io`/`Memory` lanes rather than silently
  succeeding. This honors the "no false pass" scope boundary.
- Per-AM fixtures with one table + one index per AM matches the project's
  isolated-one-index-per-table convention.
- Tagging connections with `application_name = 'ecaz-fault-<label>'` and
  scoping leak probes by that prefix is the right cheap mechanism for
  cross-test isolation in a shared dev database. Good.

### Concerns about what the live lanes actually prove

The packet says: *"cancel, statement-timeout, lock-timeout, and resource lanes
completed for all four AMs with postcondition probes emitted and asserted."*
That is true mechanically, but several lanes do not exercise ECAZ AM code
paths in the way Task 38 was designed to:

1. **Cancel lane does not test ECAZ `CHECK_FOR_INTERRUPTS` instrumentation.**
   `run_cancel_probe` issues `<scan>; SELECT pg_sleep(5);` then cancels after
   50ms. With 16 rows and 4-dim vectors, the scan completes in well under
   50ms — the cancel almost certainly fires inside `pg_sleep`, not inside the
   AM scan loop. The probe currently proves *"PG can cancel a sleeping
   backend"*, which is uninteresting.

   What the task asks for (§Approach 3): cancel issued *during* AM work.
   Suggested fixes (any of):
   - Drop `pg_sleep`; drive `rows` high enough that the scan dominates (say
     50–100k rows) and cancel inside a tight retry loop.
   - Build a CTE that loops over many sequential ORDER-BY-distance scans
     against the AM index so the worker stays in AM code for seconds.
   - For HNSW/DiskANN, run `CREATE INDEX` and cancel mid-build (build paths
     are where missing CFIs hurt most operationally).

2. **Statement-timeout lane same issue.** `SET statement_timeout = '1ms'; <scan>;
   SELECT pg_sleep(0.05);` — the 1 ms timeout almost always fires inside
   `pg_sleep`, not inside the AM. Same fix as cancel: replace `pg_sleep` with
   a workload that keeps the backend inside AM code, or set the timeout to a
   value calibrated to the AM scan time on the chosen `rows`.

3. **Lock-timeout lane bypasses the AM entirely.** It `LOCK TABLE ... ACCESS
   EXCLUSIVE` and times out. The task asks specifically for `lock_timeout`
   interrupting `CREATE INDEX`, `REINDEX CONCURRENTLY`, or `VACUUM (FULL)`
   mid-acquire. The current probe would pass even with no ECAZ extension
   installed — it's a generic PG lock-timeout sanity check. To meet the
   spirit, hold the lock on the AM table from one session and attempt
   `REINDEX CONCURRENTLY <ecaz_fault_*_idx>` from another with
   `lock_timeout = '10ms'`.

4. **Resource lane never exercises accumulator pressure.** `SET work_mem =
   '64kB'` then a 16-row, 4-dim scan/insert/vacuum. The accumulator stays
   well under 64kB regardless. Either crank `rows` up substantially and use
   dimension closer to production (≥384), or drop the resource lane to
   `--dry-run` until it can be made meaningful, since a green log line here
   is misleading.

5. **Postcondition probes are weaker than `case.postconditions` advertises.**
   Each `FaultCase` lists four postconditions including *"no pinned buffers
   retained by the test backend"* and *"pgstat counters remain monotonic"*.
   The actually-executed `leak_probe_sql()` only checks `pg_stat_activity`,
   `pg_locks`, and `pg_prepared_xacts`. The string list reads like
   documentation but is never compared against runtime state — that is a
   trap for future readers who will assume probes were run. Either:
   - Add real probes for `pg_buffercache` pin count and `pg_stat_io` delta
     (with a snapshot taken pre-workload), or
   - Move the unimplemented strings out of `postconditions` into a separate
     `documented_future_postconditions` field so the struct can't mislead.

### Provider and crate code review

- `crates/ecaz-fault-injection/src/ldpreload_provider.c`:
  - Hooks `open`/`open64` but not `openat`/`openat2`. Modern glibc and PG
    paths routinely use `openat(AT_FDCWD, ...)`. ENOSPC injection on creation
    will miss real openat-based callsites in PG18. Add `openat` and
    `openat2` hooks, mirroring the variadic-mode handling.
  - `fault_counter` is process-global and monotonic across modes. Documented
    "Start injecting on the Nth matching provider operation" is honored, but
    successive smoke runs against the same postmaster will see a stale count
    unless the postmaster is restarted. Note this in the operator docs.
  - `should_fault_fd` increments the counter on *every* mode-matching call
    regardless of whether `fd_matches` returns true via path-needle. Actually
    looks correct on re-read (counter only increments after path/fd match);
    nevermind.
  - `maybe_sleep()` runs on every read/write/open call once enabled — even
    when mode != slow-disk it short-circuits, fine. But under slow-disk it
    *also* fires on the marker-file `openat`. That's a self-injection of
    latency on the constructor, harmless but worth a comment.

- `crates/ecaz-cli/src/commands/dev/fault.rs`:
  - `run_cancel_probe` spawns a tokio task for the worker but never joins the
    `tokio::spawn` driver from `psql::connect` — please verify the connection
    driver is reaped after the cancel (otherwise stale drivers accumulate on
    repeated lane runs in the same process).
  - `assert_sqlstate` has a fallback `error.to_string().contains("statement
    timeout")`. That's a string match against a localizable PG message; if
    PG_LANG ever flips, the fallback breaks. Prefer SQLSTATE-only or also
    accept `57014` (already covered).
  - `assert_expected_error` is named generically but always asserts `57014`.
    Inline the sqlstate at the call sites or rename to
    `assert_query_canceled`. Future readers will be confused.

- `ecaz-fault-injection/src/lib.rs` `case()`: the `postconditions` strings
  list four assertions but only three are runtime-checked (see "Postcondition
  probes" above). Split or rename.

### Reviewer-focus answers

1. **Provider boundary explicit enough?** Yes for the "what is not yet
   provider-backed" axis — `Io`/`Memory` lanes return a useful error message.
   No for the "what does each lane actually probe" axis — the request claims
   the live lanes assert AM-level invariants, and most of them currently
   assert PG infrastructure invariants.
2. **Application-name leak probes adequate?** As a *necessary* minimum, yes
   — they catch the obvious connection-leak class. As a *sufficient*
   minimum, no — buffer pin and `pg_locks` (relation-level, not connection-
   level) checks should land before the lane is declared a coverage gate.
3. **Tolerance choices (`1e-5`):** appropriate for unit-bounded f32 IP/L2 on
   the dims/widths tested. Document the empirical max observed gap in the
   packet so future tightening has a baseline.

### Task 38

1. **Cancel still races against `pg_cancel_backend` at the executor boundary
   between iterations**, not necessarily inside the AM scan loop. The
   repeated-scan workload increases the *probability* the worker is in AM
   code when the signal arrives, but it doesn't *guarantee* it — PG's
   interrupt check fires at executor `ExecScan` step boundaries too. If you
   want to pin the assertion to "ECAZ AM honors CFI", run the cancel against
   `CREATE INDEX` (longer in-AM build, no executor outer loop), or add a
   single-LIMIT scan against an index with `rows = 1_000_000` so one AM call
   dominates. Current state is acceptable as a smoke; flag the limitation.
2. **Resource lane unchanged** — still 64 rows × 4 dim × 64kB `work_mem`,
   well under the accumulator threshold the task targets. Either move
   resource to `--dry-run` for this packet's claimed coverage, drop it from
   the validation list in `request.md` (currently omitted from the explicit
   run list — good), or rework the workload to a realistic dim/rows
   combination that actually pressures the accumulator.
3. **Palloc lane is a foothold, not a sweep.** Task §Approach step 2 asks
   for "a sweep that forces failure at the Nth allocation in a workload" —
   currently there's exactly one instrumented site per AM (the
   `ambeginscan` opaque). To call the palloc lane closed:
   - Annotate the major palloc sites in scan/build/insert/vacuum hot paths
     with `maybe_fail_palloc(label)` and unique labels.
   - Add an `ecaz dev fault smoke --lane memory` driver that walks
     `fault_palloc_after = 1..=N`, runs a workload, asserts SQLSTATE matches
     `pgrx::error!` propagation (typically `XX000`), and checks the leak
     probes.
   - Until then, the request.md "Scope Boundary" framing ("palloc-failure
     sweeps still need a palloc-aware PG test hook or extension-side
     injection point") is correct but understates progress — the
     extension-side injection point now exists, just under-deployed.
4. **GUC name vs. payload mismatch is subtle**: `ecaz.fault_palloc_after`
   reads like "fail *after* the Nth call". Actual semantics: with
   `after = N`, the Nth allocation fires (because `fetch_add(1)+1 = N` on
   call N, and check is `count >= N`). Either rename to `ecaz.fault_palloc_
   nth` or document "after = N → Nth call fails" in the help text. Current
   help text "before the Nth instrumented ECAZ palloc site" is also subtly
   wrong: it fires *on* the Nth site, not before it.
5. **`run_lock_timeout_probe` resource leak on uncommon path**: if
   `waiter.batch_execute("RESET lock_timeout;")` errors (?-propagated), the
   subsequent `holder.batch_execute("ROLLBACK;")` never runs and the
   `ACCESS EXCLUSIVE` lock persists for the rest of the worker connection's
   lifetime. Wrap the holder rollback in a guard / use a defer pattern, or
   accept the limitation in a comment.
6. **Slow-disk live log finishes in 2 seconds across 4 AMs**. With
   `latency_ms=1` and ~hundreds of file ops per AM scan/insert/vacuum, this
   is plausible (a few hundred ms per AM) but it would be reassuring to
   capture a "with provider vs without provider" wall-clock delta in the
   manifest so future readers can see the provider was effective. Cheap
   addition.
7. Nit: `provider-restart` writes a zero-byte marker file before exec; the
   provider's constructor will append `pid=... mode=...\n` lines. That works,
   but `assert_provider_marker` requires non-empty. If the postmaster fails to
   start, the marker stays empty and the smoke lane errors with the right
   "empty marker" message — actually a feature, not a bug. Keep.
