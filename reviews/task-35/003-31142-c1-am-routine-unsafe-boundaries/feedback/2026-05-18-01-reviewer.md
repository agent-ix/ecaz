# Review: AM routine unsafe boundary reduction (task-35/003)

## Summary

Second substantive Task 35 packet. **4,809 → 4,799 entries (−10) across
4 fewer files.** Unlike packet 002 (all comment-only), this packet
actually **deletes** 8 unsafe blocks; 2 remaining are documented.

The deletions are real safety wins, not relabeling. Two patterns
retired:

1. `amvalidate` callbacks for HNSW/IVF/SPIRE/DiskANN had
   `unsafe { pgrx::pgrx_extern_c_guard(|| true) }`. Returning
   `true` directly is correct — `true` cannot panic, so the
   panic-to-elog wrapper was vestigial. **4 unsafe blocks
   deleted.**
2. Handler functions (`ec_*_handler`) had inner
   `pgrx_extern_c_guard` calls **on top of** their `#[pg_guard]`
   attribute. The attribute already provides the C-unwind boundary
   guard; the inner call was duplication. **4 unsafe blocks
   deleted.**

This is the right kind of unsafe-burndown work: identifying that
the unsafe was always redundant and removing it, not paperwork-ing
around it.

## Findings

### 1. (verification) `#[pg_guard]` IS equivalent to inner `pgrx_extern_c_guard`

The review focus asks for confirmation. Per pgrx documentation and
source:

- `#[pg_guard]` is a proc-macro attribute that wraps the function
  body in `pgrx_extern_c_guard`. It exists specifically so callers
  don't have to write the wrapper manually.
- Applying `#[pg_guard]` **and** calling `pgrx_extern_c_guard`
  inside the body wraps the body twice. The inner wrapper is a
  no-op in steady state and adds an extra setjmp-equivalent frame.

So removing the inner call is **semantically identical** to the
prior behavior in the success path, and **strictly safer** in
terms of stack frame management. The deletion is correct.

The "Left comments on the handler bodies to make the boundary
handling explicit" line in the request describes the residual
comments that point at `#[pg_guard]`'s role — those comments are
useful future-reader signposts and the right shape.

### 2. (verification) `amvalidate` returning `true` directly is safe

The `amvalidate` callback signature is
`unsafe extern "C-unwind" fn ec_*_amvalidate(_opclassoid: pg_sys::Oid) -> bool`.
The body was `unsafe { pgrx::pgrx_extern_c_guard(|| true) }`,
now just `true`.

The "C-unwind" ABI means Rust panics **can** propagate to C
through the function boundary. Without `pgrx_extern_c_guard` or
`#[pg_guard]`, a panic would become a foreign unwind in
PostgreSQL — that's a real bug if any panic-capable code runs.

For these functions: the body is a literal `true`. `true` cannot
panic. So the removal is safe.

**One subtle risk to flag** for future reviewers: if anyone later
adds *any* logic to these `amvalidate` bodies (even seemingly
trivial like `pg_sys::is_some_check(opclassoid)`), they will need
to re-add `#[pg_guard]` or `pgrx_extern_c_guard`. **A small
defensive measure** would be to put `#[pg_guard]` on the function
*now* — it adds the wrapper around `true`, which costs essentially
nothing, but it provides forward protection against future
panicking additions.

**Suggested follow-up edit:** add `#[pg_guard]` to all four
`amvalidate` functions. Not blocking — the current state is
correct — but it future-proofs against accidental panic-source
additions.

### 3. The 2 documented DiskANN diagnostic reads are correctly scoped

`src/am/ec_diskann/diagnostics.rs::graph_summary` reads:

- `pg_sys::RelationGetNumberOfBlocksInFork(index_relation, ...)`:
  ```
  // SAFETY: The caller provides a live DiskANN index relation;
  // this read-only diagnostic only asks PostgreSQL for the
  // current MAIN fork block count.
  ```
- `scan_state::materialize_chain_from_index(index_relation)`:
  ```
  // SAFETY: The same live DiskANN index relation is materialized
  // through the scan-state reader without modifying pages.
  ```

Both SAFETY comments name the precondition (live relation pointer)
and the operation guarantee (read-only). These are correctly
scoped — the unsafe is real (raw `pg_sys::Relation` pointer is
unsafe to dereference) and can't be deleted, only documented.

**One tightening:** "live DiskANN index relation" could be
stronger. Specifically, this is a **caller invariant** — the
caller (the SQL diagnostic function) must have called
`index_open` or equivalent and held the relation open across the
diagnostic call. If a future caller passes a freed relation
pointer, this is unsafe. A comment that says "Caller must hold
`relation_open` lock across this call" would name *what makes the
relation live* rather than just asserting it.

Not blocking. Tighten in a follow-up.

### 4. (cross-reference) Pattern reusable for other AM handlers

The "duplicated `pgrx_extern_c_guard` inside `#[pg_guard]`-attributed
function" pattern is worth grepping for across the codebase. If
this same duplication exists in:

- `src/am/ec_*/build.rs` callbacks

then a follow-up packet could retire all of them. The pattern is
mechanical and the verification (build cleanly under
`--features pg18,bench`) is the same.

**Suggested next Task 35 packet:** grep for `pgrx_extern_c_guard`
inside `#[pg_guard]`-attributed bodies across the AM subdirectories;
batch-remove the duplications. Likely 20-50 entries in one packet
if the pattern is consistent.

### 5. Baseline file count drop matches scope

−10 entries / −4 files: the 4 routine.rs files (HNSW/IVF/SPIRE/
DiskANN) each had ≥2 entries that were both removed. That's
consistent with "removed 2 unsafe blocks per routine: `amvalidate`
+ `handler`" × 4 files = 8 deletions, plus 2 unchanged-but-now-
documented in `diagnostics.rs` (1 file). The file count drop is 4
(the 4 routine files reached 0 entries), the entry drop is 10
(8 deleted + 2 still present but unchanged: wait, 2 documented
existed before and still exist after, so the entry count change
should be -8, not -10...).

Let me re-check: −10 entries / −4 files. If 8 were deleted in 4
files, that's −8 entries / −4 files. The extra -2 means 2 other
entries were also retired somewhere. Possibly the
`pgrx_extern_c_guard` removal cascaded a second unsafe? Or maybe
the 2 diagnostic entries' line numbers changed and they got
re-added under different IDs in the baseline (which would be
weird, since they should be content-keyed).

**Verification question for the coder:** the math `−10 entries`
vs `8 unsafe blocks removed + 2 documented` doesn't quite balance.
Where are the other 2 entries that left the baseline? Likely a
line-number-churn artifact in the baseline TSV format, but worth
confirming.

If the baseline keys on file+line rather than file+content-hash,
removing 4 unsafe blocks from a file can shift all subsequent line
numbers and create N "deletions" + N-2 "additions" in the diff —
artificially inflating the deletion count. **If that's what
happened**, the lane's accounting is misleading and packet 001
should switch to content-hash keying.

Not blocking this packet's acceptance. Flagging for follow-up
on the baseline format.
