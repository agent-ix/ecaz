## Series-level notes

### Strategy assessment: micro-slicing is acceptable for this file

**Pros confirmed:**

- Each slice is independently reviewable in ~5 minutes.
- Layer boundaries (read / read-stream / visitor / append / mutate /
  rewrite / WAL) became visible in the packet topics — they're not
  arbitrary cuts, they trace real architectural seams in the file.
- Smaller blast radius per commit, easier rollback granularity.
- Allowed the coder to interleave with other work without holding a
  big slice in flight.

**Cons observed:**

- 10× the per-packet overhead (request.md, manifest, artifacts,
  validation logs) for one file. The bookkeeping cost is non-trivial.
- Risk of losing cross-section invariants: e.g., does packet 029's
  "exclusive buffer lock required before mutation" contract align
  with packet 028's claim that the append-range logic selected
  `block_number`? I spot-checked this and yes — but no single
  packet's SAFETY comments enumerate the *full* lock acquisition
  graph for the file. Acceptable, but a closing summary packet would
  help.

### Recommendation for packet 035 (closing the file)

When packet 035 lands the final 20 entries on `page.rs`:

1. **Honest accounting for absorbed line drift.** Packet 022
   absorbed +1 entry on `page.rs` (133 → 134) from upstream line
   drift. The series so far hasn't subtracted that absorption — net
   real reduction from 133 → 20 is −113, not −114. Small, but worth
   stating cleanly in packet 035's summary.
2. **Series-closing summary.** Packet 035's `request.md` should
   include a short "series closeout" paragraph: total entries
   processed (113), packets in series (11 if 035 lands the last 20),
   architectural seams covered (read / read-stream / visitor /
   append / mutate / WAL / rewrite / debug / metadata helpers), and
   a single end-state per-file count = 0.
3. **Cross-section lock graph.** Brief note inside packet 035 or in
   a follow-on doc summarizing which functions in this file require
   which lock modes (share / exclusive) and which functions establish
   the WAL transaction boundary. This is the "architectural fact"
   that the series' per-slice SAFETY contracts collectively assume
   but nowhere state in one place. Real reviewer protection comes
   from being able to read this once and audit the rest.

### Structural opportunities (Task 50, deferred)

Three patterns repeat across the series that are candidates for
structural reduction once page.rs hits baseline zero:

1. **`read_page_tuple` + decode pair.** Every entrypoint reader
   (centroid, list-directory, PQ-codebook, posting) wraps
   `read_page_tuple` with a tag-specific decode. A typed
   `read_page_tuple_as::<IvfCentroidTuple>(...)` helper could absorb
   the unsafe at the call site by encoding the tag at the type
   level.
2. **Exclusive buffer guard + WAL transaction pair.** The
   `exclusive_buffer_lock` + `GenericXLogTxn::start` +
   `register_buffer` sequence in append/mutate/rewrite paths
   appears 5+ times. A `with_wal_full_page!(index_relation,
   block_number, |page| { ... })` closure-style helper would absorb
   the unsafe and tie the lock and WAL lifetimes together
   structurally.
3. **`RecordPageWithFreeSpace` call sites** appear after every
   posting-page mutation. A small `record_free_space_after(page,
   block)` wrapper would document the post-mutation freespace
   invariant once.

These belong in Task 50 (post-burndown), not in any remaining
Task 35 page.rs work.
