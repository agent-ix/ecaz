## Decision

Approve packet 022 at commit `ce898c89`. The SAFETY contracts added to
`src/quant/hadamard.rs` meet the Task 35 standard of *specific*

restatement of the unsafe operation.

## What I verified

**Comment quality (sampled across the diff):**

- **Dispatch layer (lines ~17, 19):** cites `backend()`'s contract that
  `Avx2Fma` / `Neon` are only returned after process-level runtime
  feature detection. Correct chain of trust.
- **Test-only entrypoints (lines 51, 60):** cites the immediately-prior
  `is_x86_feature_detected!` / `is_aarch64_feature_detected!` guard as
  the AVX2/FMA / NEON precondition.
- **Two-level dispatch (1024/2048/4096):** cites the fixed widths
  dividing the exact power-of-two input length. The 1024-element case
  declares `(512, 256)` divides 1024 and bootstraps complete tiles —
  the comment captures the divisibility fact, which is the real
  invariant.
- **Tile loops (lines ~125, 152):** cite `tile_width >= 128` /
  `inner_chunk >= 64`, so each chunk is large enough for the bootstrap
  to return a complete 64-lane stage width. This is the right level —
  it explains why width 64 is a safe second argument.
- **Unaligned load/store blocks (lines ~187-211, 194-198, etc.):**
  cite chunk lane count (8 / 16 / 32 / 64 f32 values), offset
  arithmetic within the chunk (0, +8, +16, +24, ...), and the
  `_mm256_loadu/storeu_ps` unaligned permission. Each store is paired
  with the corresponding load's region. This is exactly what a
  reviewer needs to confirm pointer-arithmetic safety without
  re-deriving the geometry.
- **Recursive stage helpers (`fwht_in_place_avx2_stages`,
  `fwht_in_place_avx2_stage_width`):** cite the stage-doubling
  invariant (`width *= 2` and `width % 8 == 0`) and the
  `offset + step <= len` chunking invariant. Real preconditions,
  not paraphrases.
