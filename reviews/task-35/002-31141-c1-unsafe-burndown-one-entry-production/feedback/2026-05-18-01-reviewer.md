## Findings

### 1. SAFETY comment quality is **mostly good**, one needs tightening

The review focus asks whether each comment "states a real
caller/callee invariant rather than restating the unsafe
operation." Spot-checking:

**Good (states invariant):**
- `src/standalone_pg_backend_stubs.rs` `ecaz_test_pg_backend_panic`:
  > `// SAFETY: The C test stub passes a null-terminated error
  > message pointer or null; the null case is handled above before
  > constructing CStr.`

  Names the precondition (null-terminated or null) and points at
  the null-guard. This is the right shape.

**Needs tightening:** `src/am/ec_ivf/quantizer.rs`
`load_pq_fastscan_model`:
```
// The caller provides a live IVF index relation.
// SAFETY: `next_tid` comes from the metadata/codebook chain and is
// checked against `ItemPointer::INVALID` before each read.
```
Two issues:

1. The "The caller provides a live IVF index relation" line is
   **outside** the `SAFETY:` comment block. It should be inside —
   per the codebase convention and Rust style, the entire
   invariant statement goes after the `SAFETY:` marker so the
   `check_unsafe_comments.sh` lint and any future tooling can
   parse it as a single attached comment.
2. "checked against `ItemPointer::INVALID` before each read" —
   *where* is that check? Naming the check function/line would
   make this auditable. Right now I have to grep to verify.

**Suggested rewrite:**
```rust
// SAFETY: The caller provides a live IVF index relation. `next_tid`
// comes from the metadata/codebook chain; the surrounding
// `while !next_tid.is_invalid()` loop guarantees we never read
// past chain end.
```

Not blocking — the invariant is real and the comment is on the
right track. Tighten in a follow-up commit on the same packet.

### 2. The 7 entries are all "comment-only" — what about deletions?

This packet's pattern is `SAFETY:` annotation. Packet 003 (sibling
in this batch) actually *deletes* 8 unsafe blocks. Both are valid
burndown moves, but the lane will move faster if subsequent
packets prefer the deletion variant where possible.

For the 7 entries in 002 specifically:
- `ec_ivf/quantizer.rs::load_pq_fastscan_model`: invariant depends
  on caller; can't be deleted, comment is correct.
- `ec_spire/coordinator/.../resolve.rs`,
  `result_contracts.rs`, `scan/types.rs`,
  `update/materialization.rs`, `update/routing.rs`: would need
  to inspect each to know if a deletion variant exists. The fact
  that all 5 SPIRE sites got comment-only treatment suggests
  there's a shared invariant pattern here that **might** be
  abstractable into a Task 41-style RAII wrapper.

**Suggested follow-up:** before the next SPIRE-focused Task 35
packet, audit the 5 SPIRE coordinator/scan/update sites for a
shared invariant pattern. If they all share the "live SPIRE
relation pointer through coordinator boundary" precondition, that's
an abstraction opportunity (a `SpireCoordinatorRelation` guard,
or similar). One abstraction can retire many unsafe sites.

### 3. (cross-reference) Net file count drop matches entry drop

−7 entries / −7 files means each removed entry was the only entry
in its file. That's a sign these were the lowest-hanging fruit —
files with isolated single unsafe blocks. The harder packets will
have N-of-many removals where the file stays in the baseline.
Expected; not a finding.

### 4. (cross-reference) `cargo check` warnings note

The request notes "existing warnings from PostgreSQL headers and
currently unused SPIRE re-exports in `src/am/mod.rs`." Confirmed
not introduced by this packet. Worth a separate cleanup at some

packet, since it has nothing to do with unsafe burndown. Track as a
separate hygiene item.
