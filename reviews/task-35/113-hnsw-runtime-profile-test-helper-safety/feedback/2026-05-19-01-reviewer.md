## Quality observations

### Macro-consolidation pattern extended (113–118)

The same per-file `unsafe`-wrapping macro pattern from 108–112
landed in 6 more HNSW test files. Each request.md explicitly
states the structural choice:

- 113: "by routing repeated HNSW runtime/profile debug helper
  calls through one documented local macro"
- 114: "by routing repeated HNSW recall/debug helper calls
  through one documented local macro"
- 115: "by routing HNSW metadata, page, vacuum, frontier, and
  scan debug helper calls through one documented local macro"
- 116: "by routing HNSW index page debug helpers and relation-
  backed graph/codebook reads through one documented local
  macro"
- 117: "by routing SQL-visible HNSW test/debug export calls
  through one documented local macro after the exported helpers
  validate their HNSW index OID"
- 118: "by routing runtime comparison debug calls through one
  documented local macro and documenting the binary receive
  rejection fixtures in `hnsw_misc.rs`"

This is the same structural reduction surfaced in 108–112 (and
called out as Task 50 prototype in my prior wave feedback). **Six
more HNSW test files now have single-macro audited safety
contracts** covering tens of callsites each. The audit-surface
collapse continues.

### 119–120 are honest hold-outs from the macro pattern

**Packet 119** (`remote_search/*`) cannot easily use a single
macro because the surface spans multiple test files. A shared
macro would require restructuring the test module layout. The
coder opted for repeated SAFETY comments instead — same prose
text appears ~5 times. The repetition is honest given the
constraint, but it is technically a comment-only slice.

**Non-blocking observation:** A shared `remote_search_debug!`
macro hoisted to a small `src/tests/remote_search/helpers.rs`
module would have absorbed the repetition. Modest Task 50
candidate — not worth a code change now, but worth noting that
the repetition is the cost of not promoting the helper module.

**Packet 120** (general residuals) covers genuinely scattered
sites across `src/tests/build.rs`, `custom_scan.rs`,
`custom_scan_concurrency.rs`, etc. — different surfaces
(SPIRE debug, tuple-slot reads, interrupt guards, vacuum
diagnostics). No single macro could cover them; per-site SAFETY
comments are the right call.
