**Packet 045 (insert):**

- `aminsert` callback contract identical to IVF 041 — consistent.
- `lock_publish_relation` guard cites scope: "captures and locks its
  OID while this publish path runs." Right level for an RAII guard.
- Successive reads (root_control / relation_options / heap+IndexInfo)
  each cite their source individually.

**Packet 049 (custom_scan/dml) — high-risk file:**

- Plan-private offset annotations name both the **offset and the
  field**: `plan_private offset 2 stores the encoded LIMIT value`,
  `custom_exprs offset 0 stores the ORDER BY query expression`. This
  is the protection-against-silent-drift framing I called out in 043
  feedback, reused correctly here. If anyone reorders these offsets
  in the future, the SAFETY comment becomes a tripwire.
- Const/Param real[] datum decode cites "validated Const or evaluated
  Param; pgrx decodes it without taking PostgreSQL ownership". Same
  ownership-phrasing template as 043.

**Packet 054 (remote_candidates/fanout) — distributed read path:**

- Root-control reads chain to manifest loads via "index_relation is
  the same open relation used for the root control read, and
  root_control identifies the active manifest set". This is exactly
  the "same X used for Y" chaining I want to see across the
  distributed coordinator path — each successive unsafe block names
  the prior call's output as its precondition.

**Packet 048 (build/drafts):**

- Local-store config + placement directory + manifest bundle chain
  binds three successive operations to the same `index_relation` and
  `root_control`. Manifest encoding cites "initial publish for the
  live SPIRE index relation".

**Packet 056 (update/publish/relation):**

- The chain of `index_relation is open ... root_control identifies
  the local-store config for the active epoch ... draft was built
  from validated replacement inputs and the loaded local-store
  config belongs to the same index relation/root epoch` is the
  cleanest example of contract chaining in the wave. Future SPIRE
  reviewer feedback can point at this as a model.

## Wave-level observations

### Patterns established

The SPIRE wave has crystallized three distinct safety templates,
each named explicitly in multiple packets:

1. **AM-callback boundary** (045 insert, 058 diagnostics): "PG
   invokes X with live pointers for the guarded call". Identical
   to IVF and HNSW templates.
2. **CustomScan plan-private / custom_exprs offset-encoding**
   (049 dml, plus 043, 047 cost_helpers, 050 begin_exec): "offset N
   stores the encoded \<field\>". Names both the slot and the
   schema.
3. **Same-relation contract chaining** (054 fanout, 055
   executor_receive, 048 drafts, 056 publish/relation, 057
   write_payload): "X is the same open relation used for Y, and Z
   identifies the active manifest set". Each link cites the prior
   anchor.

### Closeout summaries still missing

None of the 14 wave packets contains a per-packet closeout summary
or a wave-level retrospective in any request.md. Per my notes on
035 (IVF page series closeout was excellent) and 039/040 (IVF
maintenance lacked one), the SPIRE wave is large enough that a
closeout would help. **Non-blocking ask:** when the SPIRE surface
nears completion, land a packet whose request.md includes:

- SPIRE coverage table (file → entries cleared → packet).
- Lock/WAL/distributed-coordination invariant graph (which calls
  acquire which locks, where the "active epoch" anchor is
  established, which paths cross relation boundaries).
- Final per-file count = 0 confirmation.

This is the analogue of packet 035 for SPIRE, and the test-only
SPIRE files will benefit from it.

### Structural opportunities (Task 50, deferred)

Three patterns repeat across the wave and are worth structural work
once Task 35 closes SPIRE:

1. **CustomScan plan-private offset wrapper.** The
   `offset 0 = mode, offset 1 = index_oid, offset 2 = top_k`
   convention is now enforced by SAFETY-comment discipline across
   043, 047, 049, 050. A typed `SpireCustomPrivate` struct with
   round-trip helpers (called out in 043 feedback) is the
   highest-payoff Task 50 candidate on SPIRE.
2. **Root-control + manifest-set chain.** The "root_control
   identifies the active manifest set" pattern appears in 045, 048,
   054, 055, 056, 057. A typed `ActiveEpochAnchor` value holding
   `(index_relation, root_control)` and exposing safe accessors for
   manifests, placement directory, and local-store config would
   absorb most of these unsafe sites and convert the SAFETY-comment
   discipline into compile-time enforcement.
3. **`aminsert` / `ambulkdelete` / `amvacuumcleanup` callback
   wrapping** continues to repeat the same `pgrx_extern_c_guard`
   wrap. Cross-AM `pg_am_callback!` macro candidate.

### What I did not verify

- Per-packet `cargo-check-pg18-bench.log` / `unsafe-audit-after.log`
  for every wave packet (sampled 045, 048, 049, 054, 056; all
  passed). The remaining nine packets share the same artifact set
  shape, and the baseline accounting chains correctly through all
  14. Spot-check confidence is high.
- Distributed-coordination semantics beyond what the SAFETY
  comments name. The `executor_receive`, `fanout`, and
  `write_payload` paths cross network boundaries that this
  reviewer-feedback round doesn't formally verify.
