## Notes (non-blocking)

1. **Admin diagnostic / page substrate boundary is correct.** The
   request asks whether admin diagnostic reads of metadata and
   directory pages are documented at the right layer "given the
   larger IVF page substrate remains scheduled separately." Answer:
   yes. The admin SAFETY comments stop at the page reader call
   boundary ("page readers validate the metadata/directory layout")
   rather than re-stating the page reader's internal invariants. That
   keeps the admin and page slices in different scopes. When the
   page substrate slices (025+) land, they own the page-reader-side
   invariants and admin's pointer at them stays accurate.
2. **Structural opportunity (Task 50, deferred).** The four AM
   callbacks in `cost.rs` all wrap the same
   `unsafe { pgrx::pgrx_extern_c_guard(|| { ... }) }` pattern. A
   single safe macro `pg_am_callback! { ... }` could eliminate the
   per-call `unsafe` block. Same pattern likely appears across HNSW
   and (incoming) SPIRE AM callback files. Worth a future
   cross-cutting Task 50 packet.
3. **`relation_options` is `unsafe fn` itself** — the callers carry
   the contract that `index_relation` is live. The SAFETY comment
   inside cites callers; that's correct for the current shape, but a
   future structural pass could replace `pg_sys::Relation` with a
   safe `&PgIndexRelation` wrapper across IVF (mirroring the existing
   `IndexRelationGuard` / `AccessShareIndexRelation` work on the
   HNSW side). Out of scope here.
