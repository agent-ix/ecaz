## Process flag: packet 077 was a single packet for 159 entries

My feedback on the 059–071 wave explicitly recommended:

> "`src/am/ec_spire/dml_frontdoor/mod.rs` (159) — the single
>  largest residual file in the project. **Slice it like the IVF
>  page series:** layer-by-layer with an explicit 035-style closeout
>  packet on the final slice. Suggested seams: write path, PK
>  resolution, replacement decision, primitive plan, query context."

Packet 077 instead landed all 159 entries in a single commit with
152 SAFETY comments. **No architectural-seam slicing, no closeout
summary.**

### Why I'm still approving

I sampled the 152 comments at the start, middle, and tail of the
diff. Quality is consistently high throughout:

- **Planner hook installation:** "planner_hook is PostgreSQL
  backend-local process state; install only once during extension
  initialization and preserve the previous hook" — names backend-
  local nature, install-once invariant, and previous-hook capture.
- **Relcache callback:** "callback registry is backend-local; this
  flag keeps registration idempotent for the current backend."
- **HeapRelationGuard scope chains:** "heap_relation is held open
  by HeapRelationGuard in the caller; rd_id and descriptor fields
  are copied while that guard is live" — names the *guard's owner*,
  not just "live relation".
- **Index relation handling:** "IndexRelationGuard holds the index
  open while rd_index and indkey values are inspected; heap_relation
  is open for attribute-name lookup" — two simultaneous guards
  explicitly tied to their respective reads.
- **Tuple descriptor bounds:** "attr_index is in 0..natts for this
  tuple descriptor" — names the iteration bound at every access
  site.
- **Recursive expression decoding:** "recursive decoding is bounded
  by `DML_FRONTDOOR_MAX_COERCION_WRAPPER_DEPTH`" — names the bound
  *constant*. That's the right framing — if the constant ever
  becomes ineffective, the comment becomes a tripwire.
- **NodeTag dispatch:** identical template to 043
  (plan_private) — NodeTag confirmation before cast, type-check
  before downcast.
- **C-string copy semantics:** "resname was checked non-null and
  points at a PostgreSQL C string for the target entry; it is
  copied immediately."

The work is high-quality. The slicing strategy is what I flagged.

### Why the process flag still matters

Future reviewers (human or agent) reading this packet cold are
asked to audit 152 SAFETY contracts in a single diff with no
seam-level reading order. If quality had degraded in the middle of
the file (a real risk when bulk-annotating), the bisect-and-revert
granularity for fixing the regression would be the entire file.

Concrete asks for next time:

1. When a file's residual count is ≥ 50 entries and the file has
   identifiable architectural layers, slice by layer. Use the IVF
   page series (025–035) as the template.
2. The 035-style closeout summary (drift accounting, lock/WAL
   graph, per-layer enumeration) is especially valuable when the
   surface is large.
3. If the coder genuinely believes a large file is *not*
   layer-sliceable (i.e., the unsafe sites are interleaved enough
   that any slice would touch every layer), the request.md should
   state that explicitly and justify the single-packet choice. The
   077 request doesn't do this — it just lists the topics covered.

The packet stays approved because the work is correct and the
priority surface advances significantly. The flag is to prevent
this becoming the default approach for future large files.

### Closeout summary still missing across SPIRE

This wave essentially closes the major SPIRE files. **The SPIRE
surface is now substantially documented**, with only smaller
residuals + test-only files left. This is precisely the moment my
prior reviews said a closeout packet would pay off: a series-end
retrospective covering the full SPIRE coverage table, the
distributed-coordination invariant graph (active-epoch anchor +
manifest set + placement directory + heap-rerank state), and the
lock/WAL summary for SPIRE's pages and aux stores.

**Ask:** before declaring SPIRE done, land a `0XX-spire-closeout`
packet matching the 035 template. Include:

- Full SPIRE coverage table (~30 files × entries cleared × packet).
- The active-epoch anchor + manifest set invariant graph.
- Lock/WAL summary for SPIRE pages, aux store metadata, and
  publish lock.
- RAII guards inventory (`HeapRelationGuard`, `IndexRelationGuard`,
  publish lock OID-copy guard, etc.).

### Structural opportunities (Task 50, deferred)

Wave 072–081 surfaces three additional Task 50 candidates beyond
those already noted in prior reviews:

1. **`HeapRelationGuard` + `IndexRelationGuard` composition.** The
   077 packet shows these used in pairs (heap open + index open) in
   multiple PK-resolution sites. A composite
   `RelationPairGuard { heap, index }` returning safe accessors
   would eliminate the per-call unsafe at every dual-guarded site.
2. **Active-epoch anchor lifetime.** The chain
   `index_relation` → `root_control` → `active manifest set` →
   `placement directory` → `local store config` is *the* SPIRE
   distributed-read invariant. Captured by SAFETY-comment discipline
   across 074, 075, 078, 079 (and earlier 045, 048, 054–057). A
   typed `ActiveEpochAnchor` struct (also noted in prior reviews) is
   now overwhelmingly justified — it appears in nearly every SPIRE
   file.
3. **NodeTag-dispatched expression decoder** in 077 has the same
   shape as 043 plan_private. A trait or macro
   `PgNodeDispatch::decode_const_or_param(...)` could absorb the
   unsafe at the dispatcher level. Cross-file consolidation.
