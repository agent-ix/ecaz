## Findings

### 1. `docs/hardening.md` addition is **legitimate policy** — accepted

This packet adds a new "Unsafe Quality Burndown" subsection to
`docs/hardening.md` at lines 78-87 of the new revision. **This is
the right kind of content for that doc** — it establishes the
contract for a new hardening lane (what the lane does, what
packets must cite, what counts as a blocker).

To be clear about the rule (re-stated for Task 35's audience since
my version of the rule landed only on 2026-05-18):

- **Allowed in `docs/hardening.md`:** policy/contract for a lane —
  what tools the lane uses, what artifacts review packets must
  produce, what counts as a blocker, what the lane gates.
- **Not allowed in `docs/hardening.md`:** per-packet ratchet-source
  bullets, per-packet decision text, hand-edited live snapshots of
  the baseline counts (those mirror `scripts/unsafe_comment_baseline.txt`
  the same way the coverage table mirrors the coverage TSV).

The 001 addition is the policy variant. Accepted.

### 2. (suggestion) Baseline count should not be embedded in docs

The 001 docs/hardening.md text correctly does *not* say "current
baseline is 4,816 entries" inline. It says "Use `make
unsafe-baseline-report` before and after each packet" — i.e., the
*command*, not a frozen number. That's the right framing.

If a future packet is tempted to amend the doc with "current
baseline is N" or "as of packet 0XX, the baseline is M": refuse
that edit. The number lives in `scripts/unsafe_comment_baseline.txt`;
the doc references how to *measure* it.

### 3. Packet sequencing question — "right packet sequence and acceptance criteria"

The review focus asks whether Task 35 has the right sequence. The
sequence is now de facto:

1. 001 — scaffold (this packet)
2. 002 — small proof slice (7 entries, sibling packet in this batch)
3. 003 — larger subsystem batch (10 entries, AM routine boundaries)

**Pattern is sound.** Three observations:

- **Don't standardize on "10 per packet."** Packet 002's 7 entries
  was a sanity-check size; 003's 10 was a coherent AM routine
  cleanup. Future packets should be sized by *coherent scope*
  (one subsystem / one invariant / one pattern) not by
  entry-count budget.
- **Define a "merge floor" for reviewer time.** Some packets will
  be ~3-5 entries (tight invariant cleanups); others may be 100+
  entries (mass-delete of mechanical wrappers). Both are valid.
  Reviewers should accept variation rather than push back on
  "small" packets.
- **Reserve the highest-density buckets for late.** `src/lib.rs`
  at 512 entries in a single file is a sign that the file has
  generated or auto-wrapped unsafe (likely macro expansion). That
  pattern won't yield to one-entry-at-a-time review and needs a
  different packet shape (likely "audit and bulk-remove the
  pattern that generates these entries"). Don't run that packet
  until the easy wins are gone.

### 4. (suggestion) Acceptance criteria for the lane

Task 35 plan should state explicit acceptance for each packet:

- Net count reduction is **negative** (after - before < 0)
- All removed entries either:
  (a) have their unsafe block actually deleted (preferred),
  (b) have a `SAFETY:` comment naming the caller-side invariant the
      unsafe relies on (with the invariant in observable terms, not
      restatements of the operation), or
  (c) are wrapped in a safer abstraction (e.g., one of the storage
      RAII guards from Task 41).
- `make fmt-check`, `cargo check --features pg18`, and
  `bash scripts/check_unsafe_comments.sh` all pass.

The packets in this batch (002, 003) already follow this de facto;
worth codifying in `plan/tasks/35-unsafe-quality-burndown.md` if
not already.

### 5. (cross-reference) Lane intersects Task 41

The storage RAII guards from Task 41 (`src/storage/*_guard.rs`)
are exactly the abstraction Task 35 packets should reach for when
unsafe code crosses a PG resource boundary. As callers migrate to
the guards (which encapsulate the unsafe), Task 35's baseline
shrinks naturally. **Suggested coordination:** Task 35 packets that
touch storage-relation/snapshot/buffer/lock should prefer migrating
to a Task 41 guard rather than commenting the raw unsafe.

This is not blocking — just the right direction for the lane.
