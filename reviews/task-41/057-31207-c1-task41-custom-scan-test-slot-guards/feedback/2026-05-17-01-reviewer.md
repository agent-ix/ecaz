# Review: Task 41 custom scan test slot guards (31207)

## Summary

Migrates the custom-scan payload-slot pg_test in
`src/tests/custom_scan.rs` to the shared `HeapRelationGuard` and
`TupleTableSlotGuard::single_for_heap`. **Also narrows the
remaining `unsafe` blocks** to the specific operations that need
them (payload store access, slot attribute reads, datum decodes)
rather than the previous broad test-level `unsafe { ... }` wrap.

This is the first packet that **reduces the unsafe-block
granularity** beyond what RAII migration alone requires. Notable
because narrower `unsafe` blocks are a longer-term Task 35
direction (residual SAFETY-comment quality) — but here it's a
free side-effect of the guard migration, since the slot/relation
ownership no longer needs to live inside `unsafe`.

## Confirmations

- **Shared guards applied correctly.**
  `HeapRelationGuard::try_access_share` replaces the manual
  `table_open` / `table_close` pair.
  `TupleTableSlotGuard::single_for_heap` replaces the manual
  `MakeSingleTupleTableSlot` / `ExecDropSingleTupleTableSlot`. ✔
- **Narrower unsafe blocks.** The previous test wrapped a
  large body in `unsafe { ... }`; the new code restricts
  `unsafe` to specific calls that genuinely need it (raw FFI
  into test-only payload store, slot attribute reads, datum
  decodes), each with its own SAFETY comment.
- **Baseline impact.** The combined "fewer manual close
  sites" + "narrower unsafe blocks" gives a larger reduction
  than a pure guard migration would.

## Findings

### 1. Pattern worth replicating

The "broad `unsafe { ... }` block around a function body"
pattern shows up in several other tests and debug helpers. A
follow-up sweep could narrow those the same way 31207 did —
opportunistically, as later packets touch each file. Worth
noting in the 31161 strategy as a secondary deliverable of the
relation-resource subtrack.
