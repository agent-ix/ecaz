# Review: Task 41 HNSW graph prefetch buffer guard (116)

## Summary

Adds **`PinnedBufferLockGuard<'a>`** to `src/storage/buffer_guard.rs`
— a new buffer-family primitive that owns a lock on a buffer
**borrowed from a `PinnedBufferGuard`**. The lifetime parameter
`<'a>` with `PhantomData<&'a PinnedBufferGuard>` compiler-enforces
that the lock guard cannot outlive the pin guard. Uses it for
the PG18 HNSW graph-prefetch path in `scan.rs` and `graph.rs`.

This is **the first compiler-enforced lifetime relationship
between two members of the buffer family** — the lock-on-pin
ownership is now type-system-checked, not convention-checked.
Significant architectural step.

## Confirmations

- **Lifetime parameter prevents pin-outliving-lock.** The
  `PhantomData<&'a PinnedBufferGuard>` ties the lock guard's
  lifetime to the pin guard's borrow. If a caller tries to keep
  the lock guard past the pin guard, the borrow checker rejects
  it. Same shape as 31198's `IndexScanGuard::begin` taking
  typed `&Guard` references.
- **Read-stream + lock pattern.** PG18 prefetched buffers come
  back pinned but unlocked; the caller needs to lock them to
  read pages. Two-step ownership: pin (read-stream) + lock
  (caller-acquired). The new guard correctly models this as
  "lock that borrows a pin".
- **Drop releases the lock only.** The pin is owned by the
  outer `PinnedBufferGuard`; this guard drops via `LockBuffer(buf, BUFFER_LOCK_UNLOCK)`,
  leaving the pin intact. ✔

## Findings

### 1. Convention milestone

`PinnedBufferLockGuard<'a>` is the **second** typed-borrow
constructor in the codebase (first was 31198 `IndexScanGuard::begin`
taking `&HeapRelationGuard, &IndexRelationGuard, &ActiveSnapshotGuard`).
Worth promoting the convention to the strategy doc: **"resource
guards that depend on other resources should take typed
references / carry lifetime parameters."**

The buffer family alone now has 3 primitives:
- `LockedBufferGuard` (independent — pin+lock together)
- `PinnedBufferGuard` (independent — pin only)
- `PinnedBufferLockGuard<'a>` (depends on `&'a PinnedBufferGuard` — lock layered on pin)

This handles the three orthogonal cases (pin-only, pin+lock-together, lock-on-pinned). The API surface is symmetric and complete for current production patterns.
