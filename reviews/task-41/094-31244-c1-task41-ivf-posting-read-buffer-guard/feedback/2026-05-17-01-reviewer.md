# Review: Task 41 IVF posting read buffer guard (31244)

## Summary

**Adds `PinnedBufferGuard::read_main`** for non-PG18 builds (the
PG18 read-stream path already uses `from_pinned` from 31235),
and migrates `visit_ivf_postings_for_list_block` in
`src/am/ec_ivf/page.rs`.

## Confirmations

- **`read_main` constructor on `PinnedBufferGuard`.** Mirrors
  `LockedBufferGuard::read_main` — same constructor shape, but
  pin-only ownership (no lock acquired). Right addition: the
  non-PG18 fallback path needs to pin without locking, and a
  dedicated constructor is cleaner than calling
  `ReadBufferExtended` raw and then `from_pinned`. ✔
- **PG18 vs non-PG18 parity.** Both code paths now go through
  `PinnedBufferGuard` — the PG18 path via `from_pinned` on
  read-stream-yielded buffers, the non-PG18 path via the new
  `read_main`. Same Drop semantics either way.
- **Non-trivial coverage.** This packet was the missing
  non-PG18 fallback case for the buffer family.

## Findings

No blockers.

### Constructor pair note

`PinnedBufferGuard` now has two constructors:
- `read_main(rel, block, mode)` — caller-initiated pin
- `from_pinned(buffer)` — adopt PG-pinned buffer

Same shape as `LockedBufferGuard`'s three. The buffer family
API surface is consistent and complete for the current
production patterns.
