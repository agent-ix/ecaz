# Review: Task 41 HNSW debug heap scan resource guards (31182)

## Summary

Wraps the heap-backed scan helpers in
`src/am/ec_hnsw/scan_debug.rs`. `DebugHeapBackedScan` was a
five-field struct of raw pointers plus a `pushed_registered_snapshot`
bool used as a manual cleanup flag; it now stores four typed
guards. Manual error-cleanup blocks deleted. Uses the shared
`IndexRelationGuard` and `HeapRelationGuard` from
`src/storage/relation_guard.rs` for the two relation fields. Adds
three **new module-local guard structs** for the
remaining resource families:

- `DebugActiveSnapshotGuard` — `RegisterSnapshot` +
  `PushActiveSnapshot` / `Pop` + `UnregisterSnapshot`
- `DebugIndexScanGuard` — `index_beginscan` / `index_endscan`
- `DebugTupleSlotGuard` — `MakeSingleTupleTableSlot` /
  `ExecDropSingleTupleTableSlot`

I'm satisfied with the migration — the struct-of-guards
refactor is a meaningful safety win (manual flag-based cleanup
gone). The substantive finding is about where those three new
guard types should live.

## Confirmations

- **Drop order correct.** `DebugHeapBackedScan` field order is
  `scan`, `_snapshot`, `_index_relation`, `_heap_relation`. Rust
  drops in reverse declaration order: heap, index, snapshot,
  scan. That matches the prior manual cleanup order
  (`ExecDropSingleTupleTableSlot` → `IndexScanEnd` →
  `PopActiveSnapshot` + `UnregisterSnapshot` → `index_close` →
  `table_close`). ✔
- **`pushed_registered_snapshot` bool is gone.** Previously the
  helper tracked whether the snapshot was pushed via a flag so
  cleanup could conditionally pop. Now `DebugActiveSnapshotGuard::Drop`
  unconditionally pops+unregisters because the guard is only
  constructed via `latest()` which performs both. Strictly
  simpler — no more flag-state divergence possible. ✔
- **Shared relation guards used.** The struct stores
  `IndexRelationGuard` and `HeapRelationGuard` from
  `src/storage/relation_guard.rs`, not module-local copies.
  Validates the 31180 primitive in the heap-scan path.
- **Error unwind through Drop.** Where the prior code had manual
  cleanup before each `pgrx::error!`, the new code lets the
  guards' Drop run during unwind. Same pgrx-unwind invariant as
  the rest of Task 41.

## Findings

### 1. (significant) Three new module-local guards repeat the
relation-guard proliferation, one resource-family layer up

`DebugActiveSnapshotGuard`, `DebugIndexScanGuard`, and
`DebugTupleSlotGuard` are **structurally identical** to the
guards 31175 introduced in
`src/am/ec_spire/custom_scan/planner.rs`
(`ActiveSnapshotGuard`, `IndexScanGuard`, `TupleTableSlotGuard`)
— and `DebugTupleSlotGuard` is **also** structurally identical
to `HeapTupleSlot` from 31177 in
`src/am/ec_spire/scan/relation.rs`. The only differences:

| Resource | Variants in tree | Difference |
| --- | --- | --- |
| Active snapshot | `ActiveSnapshotGuard` (planner.rs), `DebugActiveSnapshotGuard` (scan_debug.rs) | `Debug*` adds `CommandCounterIncrement` before register |
| Index scan | `IndexScanGuard` (planner.rs), `DebugIndexScanGuard` (scan_debug.rs) | identical shape; different `begin()` arg signatures |
| Tuple slot | `TupleTableSlotGuard` (planner.rs), `HeapTupleSlot` (scan/relation.rs), `DebugTupleSlotGuard` (scan_debug.rs) | trivially different constructor surfaces |

This is the same proliferation pattern the relation guards just
demonstrated. The `Debug*` prefix isn't a meaningful
distinction — these guards do exactly what the production
versions do, they're just `#[cfg(any(test, feature = "pg_test"))]`.

If the buffer/page/WAL track also rolls its own
`DebugBufferPinGuard` / `BufferPinGuard` / etc. per module, the
final count is in the dozens.

**Recommendation:** the next strategy iteration should add
section 2 (the relation-guard consolidation worked, here's the
shape for the next four resource families) to
`src/storage/relation_guard.rs` (or sibling files:
`snapshot_guard.rs`, `scan_guard.rs`, `slot_guard.rs`). The
production guards land first, the debug variants either use them
directly or compose them with the `CommandCounterIncrement`
preamble where needed.

Sketch I proposed in 31175 / 31177 / 31180 feedback:

```rust
// src/storage/snapshot_guard.rs
pub(crate) struct ActiveSnapshotGuard { snapshot: pg_sys::Snapshot }
impl ActiveSnapshotGuard {
    pub(crate) fn register_latest() -> Option<Self> { ... }
    pub(crate) fn register_latest_with_cci(label: &'static str) -> Self {
        unsafe { pg_sys::CommandCounterIncrement() };
        Self::register_latest().unwrap_or_else(|| pgrx::error!("{label}"))
    }
}

// src/storage/scan_guard.rs
pub(crate) struct IndexScanGuard { scan: pg_sys::IndexScanDesc }
impl IndexScanGuard {
    pub(crate) fn begin(
        heap: &HeapRelationGuard,
        index: &IndexRelationGuard,
        snapshot: &ActiveSnapshotGuard,
        nkeys: i32,
        norderbys: i32,
    ) -> Option<Self> { ... }
}

// src/storage/slot_guard.rs
pub(crate) struct OwnedTupleTableSlot { slot: *mut pg_sys::TupleTableSlot }
impl OwnedTupleTableSlot {
    pub(crate) fn make_single(
        tupdesc: pg_sys::TupleDesc,
        ops: *const pg_sys::TupleTableSlotOps,
    ) -> Self { ... }
    pub(crate) fn single_for_heap(heap: &HeapRelationGuard, label: &'static str) -> Self { ... }
}
```

Each absorbs 2-3 current module-local variants.

### 2. (minor) `_field` naming for "just-keep-alive" guards

`DebugHeapBackedScan` uses the `_snapshot`, `_index_relation`,
`_heap_relation` naming pattern for fields that aren't read
directly — they exist purely for their `Drop` side effects. This
is the standard Rust idiom (silences "unused field" lints
without requiring `#[allow]`). Just noting for the record; it's
correct.

### 3. (carries from 31180) Validates the consolidation
infrastructure works

The fact that the relation fields use the shared primitive
without ceremony confirms that putting the resource wrappers in
`src/storage/` is the right home. Extending the same module
(or sibling files) for snapshot/scan/slot will follow the
established pattern naturally.
