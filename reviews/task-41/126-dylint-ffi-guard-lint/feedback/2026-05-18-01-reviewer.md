> 2. **dylint enforcement.** Author a `dylint` lint:
>    `ecaz_panic_across_ffi` — denies any `extern "C" fn` body
>    that can reach a panic without a guard frame.

## Confirmations

- **Syntactic check is the right scope.** The lint denies
  `extern "C"` / `extern "C-unwind"` Rust function bodies that
  lack `#[pg_guard]`, `pgrx::pgrx_extern_c_guard`, or
  `std::panic::catch_unwind`. Syntactic enforcement catches the
  vast majority of regressions (someone adds an `extern "C" fn`
  and forgets the guard); a semantic reachability analysis would
  catch a wider class but at much higher implementation cost.
  Syntactic is the right pragmatic choice.
- **Negative fixture proves the lint fires.** Per 125's
  fixtures plus 126's own negative case — a deliberately
  unguarded callback is rejected by the lint. Same teeth-sharpness
  test as the audit script's self-test.
- **Workspace exclusion is correct.** Dylint requires nightly
  Rust and rustc-private bindings. Including it in the main
  workspace would break stable-Rust builds. Routing through
  `scripts/run_dylint.sh` and excluding from the normal workspace
  keeps the build graph clean. ✔
- **`pg_finfo_*` exclusion.** Generated metadata functions
  return a static struct and can't reach a panic. Lint correctly
  ignores them — matches the same exclusion list the audit
  script uses (121).
- **All gates green.** Per the request:
  - `ffi audit passed: 101 direct C ABI functions, 288 pgrx-managed SQL entrypoints`
  - `ffi audit self-test passed`
  - `ffi lint self-test passed`
  - `ffi lint passed: raw PostgreSQL resource APIs are confined to guard modules`
  - `dylint self-test passed`

### 3. Dylint follow-up suggestion

The current lint enforces one rule (`ecaz_panic_across_ffi`).
The `crates/ecaz-lints` crate is now a place to add more lints
without re-spinning infrastructure. Two natural additions
already implied by the campaign:

- `ecaz_raw_resource_open` — denies raw `pg_sys::index_open` /
  `pg_sys::table_open` / `pg_sys::ReadBufferExtended` /
  `pg_sys::LWLockAcquire` outside `src/storage/`. (Today the
  Python `ffi-lint` enforces this via regex; a dylint version
  would catch it earlier with better diagnostics.)
- `ecaz_palloc_borrow_escape` — denies returning a borrowed
  slice from a function that constructs it from palloc. (Would
  enforce Agent IX's closure-CPS invariant #2 pattern at the
  lint level.)

Both are follow-ups, not blockers.
