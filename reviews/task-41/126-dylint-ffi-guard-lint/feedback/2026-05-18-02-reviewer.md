## Findings

### 1. (gap, low-severity) "Verified weekly" cadence is unenforced

Exit criterion #3 in the task doc says:

> `docs/ffi-inventory.md` is authoritative and **verified weekly**.

There's no cadence enforcement. The audit script auto-regenerates
the inventory when run with `--write`, but **nothing fails if
the committed inventory drifts from the live source tree**. A
new `extern "C" fn` added to `src/` is caught at PR time by
`make ffi-audit --check`, but the inventory artifact's
*content* (function names, line numbers, exception list) only
matches reality if someone runs `--write` and commits.

Suggested fix (carries from my 121 finding #3): make
`ffi-audit --check` fail if the on-disk
`docs/ffi-inventory.md` doesn't match `--write` output. That
turns the inventory into a self-enforcing artifact.

### 2. (gap, low-severity) Inventory exception list isn't auto-detected

The 6 documented exceptions are hard-coded by symbol name. When
a new `pg_finfo_*` metadata symbol lands (e.g., a new
`pg_finfo_ecvector_*` for an additional SQL type), the audit
will fail until a human adds the exception. That's the right
default behavior (fail-closed), but it means new metadata
symbols require a code change to `ffi_audit.py`'s exception
list, not just a script regeneration.

Suggested fix: auto-recognize `pg_finfo_*` as a known-safe
prefix in the script, rather than enumerating each symbol.
The test-panic-stub stays as an explicit exception.

### 3. (scope shift, blocking-on-Task-38) `make ffi-leak-smoke`
is scaffolding only

The Validation section of the task doc says:

> A deliberately leaked `Buffer` pin under forced ERROR is
> caught by the leak smoke lane.

`make ffi-leak-smoke` exists (124) but has **no test
content** — it's an aggregate target with nothing aggregated
yet. The task doc explicitly notes this is "paired with Task 38
(fault injection)" and depends on its plumbing. So this isn't
**undone** in invariant #1's scope; it's **deferred** to
Task 38. But the validation criterion as written reads like an
invariant-#1 deliverable.

Recommend updating the invariant #1 closeout language to say
"`make ffi-leak-smoke` scaffolding ready; content lands with
Task 38" rather than treating leak-smoke as a fully-shipped
invariant #1 item. Otherwise a future reader scanning Task 41
status might think the validation criterion is fully satisfied.

### 4. (scope shift, deliberate) Dylint scope narrower than spec

The task doc's Approach #2 says:

> Author a `dylint` lint: `ecaz_panic_across_ffi` — denies any
> `extern "C" fn` body that **can reach a panic** without a
> guard frame.

The shipped dylint is **syntactic** (per 126's request): it
denies `extern "C"` bodies that don't *literally name* one of
the guard mechanisms. It doesn't do reachability analysis to
ask "can this guarded body still reach an unguarded panic?"

This is a deliberate scope narrowing — semantic reachability
analysis would have been substantially harder and the audit
script + lint together still catch the common regression. I'm
**not flagging this as a bug**; just noting that the dylint is
weaker than the spec's literal wording. If the campaign ever
wants strict reachability checking, that's a Task 41 followup,
not Task 41 closeout work.
