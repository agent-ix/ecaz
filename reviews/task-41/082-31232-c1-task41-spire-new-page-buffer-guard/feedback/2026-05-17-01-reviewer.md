# Review: Task 41 SPIRE new page buffer guard (31232)

## Summary

Adds `LockedBufferGuard::read_main_locked` constructor for read
modes that return an already-locked buffer (e.g.,
`RBM_ZERO_AND_LOCK`). Migrates the remaining new-page paths in
`src/am/ec_spire/page.rs` to use it.

## Confirmations

- **`read_main_locked` constructor matches PG semantics.**
  `RBM_ZERO_AND_LOCK` returns a buffer with the exclusive lock
  already held — calling `LockBuffer` on top would deadlock or
  fail. The new constructor skips the explicit `LockBuffer`
  call and trusts PG's already-locked invariant. ✔
- **Drop behavior identical.** Same `UnlockReleaseBuffer` on
  Drop — the unlock is symmetric regardless of which
  constructor locked the buffer.
- **First "PG returns it already locked" case covered.** This
  unlocks all subsequent new-page allocation migrations.

## Findings

### 1. Constructor naming

`read_main` (caller-locks) vs `read_main_locked` (PG-locks) —
the names invert the action ("we lock" vs "it comes locked")
in a subtle way. The current names are correct but a reader
might initially expect `read_main_locked` to mean "we lock it,
then read." A one-line doc on each constructor distinguishing
the lock-acquisition responsibility would clarify:

```rust
/// Read mode where the caller passes a lockmode and this guard
/// acquires the lock after the read.
pub(crate) unsafe fn read_main(...) { ... }

/// Read mode where PostgreSQL returns the buffer already locked
/// (e.g. RBM_ZERO_AND_LOCK). The guard adopts the existing lock.
pub(crate) unsafe fn read_main_locked(...) { ... }
```

Not actionable, just helps future readers.
