## Findings

### 1. (small lint gap) Missing rule for `GenericXLog(Start|Finish|Abort)`

`src/storage/wal.rs` already provides an RAII wrapper for the
WAL generic-xlog API trio. But `ffi_lint.py` doesn't have a
`RawApiRule` confining `pg_sys::GenericXLogStart`,
`pg_sys::GenericXLogFinish`, or `pg_sys::GenericXLogAbort` to
`src/storage/wal.rs`. Currently 0 raw callsites outside that
module — but **no forward-protection** against future code
bypassing the wrapper.

This is the same coverage-gap pattern that finding #3 fixed
for snapshot/scan/slot APIs. ~6-line addition:

```python
RawApiRule(
    "WAL generic xlog APIs",
    re.compile(r"pg_sys::GenericXLog(?:Start|Finish|Abort)\b"),
    frozenset({"src/storage/wal.rs"}),
),
```

Plus a negative fixture (any other module calling
`GenericXLogStart` directly fails) and an allowed fixture
(`wal.rs` itself).

This is the **last remaining coverage gap** for the wrapped-
resource families. After this, every resource family that has
an RAII wrapper in `src/storage/` has both:
1. A migrated production surface (0 raw callsites outside the
   wrapper), and
2. Lint enforcement that prevents future bypass.

### 2. (verification) Closeout claim about live smoke is
appropriately scoped

127's request says: "The live smoke here is PG18 HNSW evidence,
not exhaustive all-AM runtime coverage. ... Exhaustive source
coverage is provided by the static gates."

That's accurate framing. The static gates (`ffi-audit`,
`ffi-lint`, dylint) are the exhaustive coverage; the live
smoke is targeted evidence. The split is correct because:
- Static gates run in `<1s` and catch regressions on every PR
- Live smoke needs a PG18 cluster and is the appropriate place
  for "did this actually work at runtime" evidence

No issue here.

### 3. (verification) SPI tuptable claim

127's request also says: "no direct `pg_sys::SPI_*` acquisition
sites outside `src/storage/spi_guard.rs`."

Verified by grep. ECAZ uses `pgrx::Spi::connect` /
`pgrx::Spi::get_one` exclusively. `SpiTupleTableGuard`'s
`#[allow(dead_code)]` on `from_owned`/`as_ptr` is appropriate
forward-protection — same status as ResourceOwner.
