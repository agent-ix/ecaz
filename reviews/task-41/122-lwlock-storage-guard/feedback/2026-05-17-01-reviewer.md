# Review: Task 41 LWLock storage guard (122)

## Summary

Directly executes my 120 finding #1: promotes the module-local
`EcHnswConcurrentDsmLockGuard` into a shared
`src/storage/lock_guard.rs::LwLockGuard`. Migrates the one
existing caller. **First LWLock primitive in the shared
storage family.** Baseline 3,690 → 3,687 (−3).

This brings the shared storage family to **eight primitives**
across **five resource families** (relations, snapshots, slots,
scans, buffers, plus now locks).

## Confirmations

- **Constructor surface.** Four constructors covering the
  expected call shapes:
  - `acquire_shared(lock)` — `LWLockAcquire(LW_SHARED)`
  - `acquire_exclusive(lock)` — `LWLockAcquire(LW_EXCLUSIVE)`
  - `from_acquired(lock)` — adopts a pre-acquired lock with
    the standard `LWLockRelease`
  - `from_acquired_with_release(lock, release)` — escape hatch
    for the test-lock-injection case in `build_parallel.rs`
- **Design choice on release dispatch.** My 120 finding #2
  suggested storing `lockmode` and dispatching internally to a
  single `LWLockRelease`. The coder kept the function-pointer-
  release pattern but **buried it behind cleaner default
  constructors**. Result: normal callers never see the function
  pointer; only `from_acquired_with_release` exposes it for the
  test-injection use case. That's the better design — preserves
  the test-injection capability without forcing every caller to
  reason about release dispatch.
- **Test injection preserved.** HNSW concurrent DSM still
  supports injecting no-op test locks (via the
  `_with_release` constructor) while production uses the
  shared default-release path. Both go through the same Drop.
  ✔
- **Raw `LWLockAcquire`/`LWLockRelease` consolidated.** Verified
  by inspection — only `src/storage/lock_guard.rs` calls these
  PG APIs directly now.
