# Deep Audit: Task 41 invariant #3 (resource release on early return)

Re-review of invariant #3 closeout state. Goal is to catch
**undone, underscoped, or scope-shifted** work vs the Task 41
task doc spec (`plan/tasks/41-ffi-safety-boundary.md`).

## Verdict

Invariant #3 is **~90% complete by spec coverage** with **two
concrete issues**:
1. A **real leak** — one raw `MakeSingleTupleTableSlot`
   callsite outside `src/storage/`, missed because the
   ffi-lint doesn't have a rule for slot APIs.
2. `SpiTupleTableGuard` exists but has **no callers** — the
   wrapper landed but the migration sweep didn't happen.

Plus a handful of less-severe items detailed below.

## What I verified

| Spec resource | Wrapper | Existing callers | Raw outside `src/storage/` |
|---|---|---|---|
| BufferPin | `PinnedBufferGuard`, `LockedBufferGuard`, `PinnedBufferLockGuard<'a>` | many | 0 (lint enforces) |
| LwLockGuard | `LwLockGuard` | 1 (HNSW concurrent DSM) | 0 (lint enforces) |
| Snapshot | `ActiveSnapshotGuard`, `RegisteredSnapshotGuard` | many | 0 (lint enforces `RegisterSnapshot`/`UnregisterSnapshot`) |
| SpiTuptable | `SpiTupleTableGuard` | **0** | 0 (lint enforces `SPI_freetuptable`) |
| ResourceOwner handles | none | n/a | n/a — ECAZ doesn't use ResourceOwner |

## Findings

### 1. (REAL LEAK — medium severity) Raw `MakeSingleTupleTableSlot`
outside the wrapper

`grep -rE 'pg_sys::MakeSingleTupleTableSlot\(' src/` reveals
**one callsite that bypasses `TupleTableSlotGuard::single_for_heap`:**

```
src/am/ec_hnsw/source.rs:444
```

This is a regression from invariant #3's resource-release
guarantee. The function allocates a slot via raw FFI and
presumably has a corresponding manual `ExecDropSingleTupleTableSlot`
somewhere downstream — but on any `pgrx::error!` path between
the two, the slot leaks.

Two fixes needed:

1. **Migrate the callsite** to
   `TupleTableSlotGuard::single_for_heap` so Drop handles
   release.
2. **Extend `scripts/ffi_lint.py`** with a `RawApiRule` for
   slot APIs:
   ```python
   RawApiRule(
       "tuple slot allocation APIs",
       re.compile(r"pg_sys::(?:MakeSingleTupleTableSlot|table_slot_create|ExecDropSingleTupleTableSlot)\b"),
       frozenset({"src/storage/slot_guard.rs"}),
   ),
   ```

Without rule 2, future regressions slip past CI just like this
one did.

### 2. (UNDERSCOPED — high severity) `SpiTupleTableGuard` has
zero callers

`src/storage/spi_guard.rs` defines `SpiTupleTableGuard` with
`#[allow(dead_code)]` on both `from_owned` and `as_ptr` —
**no caller anywhere in `src/`** uses it. The wrapper exists
but the migration sweep that adopts it never happened.

This means:
- The lint rule `pg_sys::SPI_freetuptable` confines that API
  to the wrapper's Drop. ✓ But there are presumably callers
  that **don't free the tuptable at all** today — they rely
  on pgrx's `Spi::connect` context-cleanup, which works for
  the happy path but not necessarily for every error path.
- Audit needed: every place that calls `SPI_execute*`,
  `SPI_cursor_*`, or `SPI_palloc` and gets a tuptable back
  should either route through `Spi::connect` (pgrx-managed) or
  through `SpiTupleTableGuard` (our managed). Mixed
  responsibility is the easy regression.

Suggested follow-up packet: "SPI tuptable migration sweep" —
inventory all SPI tuptable acquisition sites, classify each as
pgrx-managed-OK or needs-guard, migrate the latter.

### 3. (LINT COVERAGE GAP — low severity) Several wrapped APIs
have no lint rule

The lint enforces 5 rules. The wrapper namespace has more
resource types than the lint rules cover. APIs not yet
guarded by lint (currently 0 raw callsites for each — purely
forward-protection):

| Raw API | Should be confined to | Lint rule? |
|---|---|---|
| `table_open` / `table_close` | `relation_guard.rs` (HeapRelationGuard) | **No** |
| `relation_open` / `relation_close` | `relation_guard.rs` (RelationGuard) | **No** |
| `PushActiveSnapshot` / `PopActiveSnapshot` | `snapshot_guard.rs` (ActiveSnapshotGuard) | **No** |
| `index_beginscan` / `index_endscan` | `scan_guard.rs` (IndexScanGuard) | **No** |
| `MakeSingleTupleTableSlot` / `table_slot_create` / `ExecDropSingleTupleTableSlot` | `slot_guard.rs` | **No** (caught the leak in Finding 1) |
| `GetLatestSnapshot` (could be misused) | — | (low priority) |

None of these have current leakage callsites (other than the
slot one above), but the lint doesn't prevent future
regressions. Suggested follow-up: extend `RAW_API_RULES` to
cover all wrapped APIs. ~30-line script change.

### 4. (SCOPE SHIFT — informational) ResourceOwner not addressed

The task doc spec calls out:

> 3. **Resource release on early return.** Buffer pins,
>    LWLocks, **ResourceOwner handles**, snapshots, and SPI
>    tuptables must be released on every exit path...

But:
- No `ResourceOwnerGuard` exists
- `grep -rE 'ResourceOwner' src/` returns 0 — ECAZ doesn't
  acquire ResourceOwner handles directly

So this is a **scope-shift correct by virtue of non-applicability**.
ECAZ relies on pgrx-managed and PG-default ResourceOwner
handling. Nothing actionable, but worth noting that the spec
mentioned this and ECAZ deliberately didn't implement it
because the surface is empty. A one-line comment in the
ffi-inventory or a note in the task doc closeout would prevent
a future reader from wondering "did we forget ResourceOwner?"

### 5. (SCOPE SHIFT — informational) Drop-order audit
content depends on Task 38

The task doc's Approach #5 calls for:

> - force ERROR mid-scan and assert no leaked buffer pin
> - force ERROR mid-build and assert no leaked maintenance
>   memory context

These tests don't exist yet — they're scheduled to land via
Task 38's fault-injection infrastructure under
`make ffi-leak-smoke`. So this is **deferred not undone** per
the task doc's dependency note, but worth tracking as the
remaining invariant #3 validation criterion that isn't
satisfied yet. Same point as my invariant #1 audit Finding 3.

### 6. (NEW FAMILY NOT IN SPEC) WAL state wrappers

ECAZ uses `GenericXLogStart` / `GenericXLogFinish` /
`GenericXLogAbort` for WAL replay safety. The task doc's
Approach #4 didn't enumerate this, but it's the same RAII
shape (paired start/finish, must be balanced on every exit
path). `src/storage/wal.rs` exists (95 lines) but doesn't
have a generic `GenericXLogStateGuard` wrapping the API in
RAII shape.

This is outside the strict spec (the four named resources
are buffer/LW/snapshot/SPI) but is in the spirit of
invariant #3 ("Resource release on early return"). Worth a
follow-up Task 41 packet — or accept that WAL state wrapping
falls under Task 41's continuation rather than its closeout.
