# Review: Task 41 SPIRE coordinator debug index guards (31173)

## Summary

Wraps `index_open`/`index_close` pairs in
`src/am/ec_spire/coordinator/debug.rs` (test-only debug helpers).
Introduces `DebugIndexRelation` — a third guard struct in the
codebase, differentiated from `AccessShareIndexRelation` by
**storing the lockmode dynamically** rather than fixing it to
`AccessShareLock`. The debug helpers use a mix of
`RowExclusiveLock` (for mutation-roundtrip helpers) and
`AccessShareLock` (for read-only helpers like
`debug_spire_root_control`), so a fixed-lockmode guard would not
fit. Baseline 4,347 → 4,325 (−22); module-local
`debug.rs` baseline 60 → 38.

## Confirmations

- **RowExclusive mutation helpers keep guard alive across all
  writes.** Spot-checked `debug_spire_relation_object_tuple_roundtrip`,
  `debug_spire_relation_leaf_v2_roundtrip`,
  `debug_spire_empty_manifest_publish_roundtrip`,
  `debug_spire_rewrite_placement_state`: each opens
  `DebugIndexRelation` once with `RowExclusiveLock`, runs the
  full read/write/read sequence inside the closure that produces
  the result tuple, then `unwrap_or_else(...)` consumes the
  result after the closure returns. Drop runs at function return
  — after all writes and reads complete. ✔
- **Dual-guard test correctness.**
  `debug_spire_relation_two_store_scan_roundtrip` opens
  `root_relation` then `aux_relation`. Drop order is **reverse
  declaration order in Rust** — so `aux_relation` drops first,
  then `root_relation`. This **matches the prior explicit close
  order** in the old code (`index_close(aux_relation)` then
  `index_close(root_relation)`). ✔ Both guards stay alive across
  the entire closure body, which uses both relations
  (`SpireRelationObjectStore` over each, plus
  `SpireRelationObjectStoreSet` over `root_relation` with the
  placement directory).
- **AccessShare read-only helpers preserve lockmode.**
  `debug_spire_root_control` uses `AccessShareLock`;
  `debug_spire_rewrite_placement_state` uses `RowExclusiveLock`.
  The new `DebugIndexRelation::open(oid, lockmode)` takes the
  lockmode explicitly, so each caller's choice is preserved
  verbatim. Drop uses `self.lockmode` to match. ✔
- **Error-path Drop unwind.** Every callsite has an inner
  closure that returns `Result<_, String>` followed by
  `.unwrap_or_else(|e| pgrx::error!("{e}"))`. The `pgrx::error!`
  panics with the guard still in scope; Drop runs as the panic
  propagates. Same Drop-on-unwind invariant as the rest of the
  Task 41 surface. ✔
- **Baseline arithmetic.** −22 in `debug.rs` (60 → 38) across
  ~11 helpers ≈ 2 entries per helper, matching the prior
  `index_open` + `index_close` pattern.

## Findings

### 1. (carries from 31172) `DebugIndexRelation` is now the third guard struct

This packet's `DebugIndexRelation` plus 31172's module-local
`AccessShareIndexRelation` and the original `src/lib.rs` version
makes three guard structs total. See my 31172 finding #1 for the
full proposal — consolidating into a single
`IndexRelationGuard { relation, lockmode }` with `open` (errors
on null) and `try_open` (returns Option) constructors would
absorb all three current variants:

| Existing variant | Replaced by |
| --- | --- |
| `src/lib.rs::AccessShareIndexRelation` | `IndexRelationGuard::open(oid, AccessShareLock)` |
| `dml_frontdoor::AccessShareIndexRelation` | `IndexRelationGuard::try_open(oid, AccessShareLock)` |
| `debug::DebugIndexRelation` | `IndexRelationGuard::open(oid, lockmode)` |

The `open_valid_ec_*_index_guard` helpers in `src/lib.rs` (which
add AM-type validation) stay as thin wrappers atop the primitive.

`DebugIndexRelation`'s `#[cfg(any(test, feature = "pg_test"))]`
gate means it doesn't bloat release builds — but if the
consolidation happens, the primitive doesn't need a cfg gate
either (it's a small RAII type with no test-only state).

### 2. Drop-order documentation

The two-store test relies on Rust's reverse-declaration drop
order. This is correct and stable across Rust versions, but
since the close order **is** semantically meaningful here (the
aux store is closed before the root store), a one-line
comment near the `DebugIndexRelation::open` calls in
`debug_spire_relation_two_store_scan_roundtrip` would protect
against a future refactor that reorders the declarations. E.g.:

```rust
// Drop order is reverse declaration order: aux drops first,
// then root. Preserves the prior explicit close order.
let root_relation = DebugIndexRelation::open(root_index_oid, lockmode);
let aux_relation = DebugIndexRelation::open(aux_store_oid, lockmode);
```

Not required; the current code is correct. The comment is
insurance against silent regression.

### 3. Test-only `#[cfg]` gate on the guard

`DebugIndexRelation` is gated on `cfg(any(test, feature = "pg_test"))`.
This is fine for its current location, but if finding #1
consolidation happens, the primitive should be unconditionally
compiled (test code can use it without bringing the helper
behind a cfg). Not actionable in this packet.
