## Confirmations

- **Audit script catches the right surface.** The regex matches
  `extern "C"` and `extern "C-unwind"` declarations, plus pgrx
  attribute macros. The "guarded" check looks for `#[pg_guard]`,
  `pgrx::pgrx_extern_c_guard`, or `std::panic::catch_unwind`.
  That's the canonical invariant #1 enforcement set named in
  the Task 41 task doc. ✔
- **Six documented exceptions are correctly limited.** Five
  `pg_finfo_*` symbols (these return a static `Pg_finfo_record`
  — pure metadata, no executable Rust code that could panic) +
  one test-only panic stub (`ecaz_test_pg_backend_panic`,
  intentionally panicking in unit tests). All correctly excluded.
- **Fixed callbacks preserve semantics.** Spot-checked:
  - `ec_amestimateparallelscan` (returns `Size`): wrapped with
    `pgrx_extern_c_guard` which preserves the return type. ✔
  - Relcache + vacuum dead-TID callbacks: void-returning C
    callbacks; the guard wrapping doesn't change return semantics.
    ✔
- **Audit fails on regression.** `make ffi-audit` runs
  `python3 scripts/ffi_audit.py --check` which exits non-zero
  if any unguarded fn appears. A new `extern "C" fn` without a
  guard will fail CI on the next run. ✔
- **`docs/ffi-inventory.md` is the artifact the task doc names.**
  Generated by `--write` mode of the audit script. Single source
  of truth for the FFI surface.

### 2. (suggestion) Wire `make ffi-audit` into `ci-quick`

Per Task 49 governance, PR-blocking lanes live under `ci-quick`.
If `make ffi-audit` isn't already there, it should be — the
audit takes <1s (just a Python regex scan), well within the
PR-tier budget. A one-line follow-up packet.

### 3. (suggestion) Inventory regeneration discipline

The inventory's accuracy depends on `--write` being run when
new FFI entry points are added. Two options to prevent drift:

1. Bake the regen into `make ffi-audit --check`: if the on-disk
   inventory differs from what the script would generate now,
   fail. Forces a regen before merge.
2. Add a pre-commit hook or CI step that runs `--write` and
   diffs the result.

Option 1 is cleaner. Worth a quick follow-up.

### 4. Documented-exception expansion path

The six exception entries are currently hard-coded by symbol
name in the script (likely). When a new `pg_finfo_*` lands, the
script needs to be updated. That's fine for the metadata case
(rare additions), but worth documenting that the exception list
isn't auto-detected — it's a deliberate allow-list.
