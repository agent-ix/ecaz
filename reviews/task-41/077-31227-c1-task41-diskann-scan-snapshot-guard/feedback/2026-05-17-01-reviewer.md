# Review: Task 41 DiskANN scan snapshot guard (31227)

## Summary

**Adds a new primitive** `RegisteredSnapshotGuard` to
`src/storage/snapshot_guard.rs` and uses it for
`ResolvedScanSnapshot` when a DiskANN scan must register a
latest snapshot itself. Borrowed snapshots remain raw pointers.

## Confirmations

- **Sixth primitive in the shared module.**
  `RegisteredSnapshotGuard` joins `IndexRelationGuard`,
  `HeapRelationGuard`, `RelationGuard`, `ActiveSnapshotGuard`,
  `TupleTableSlotGuard`, `IndexScanGuard`. Convention from
  31161 continues to scale.
- **Difference from `ActiveSnapshotGuard`.**
  `RegisteredSnapshotGuard::latest` calls only
  `RegisterSnapshot(GetLatestSnapshot())` — no
  `PushActiveSnapshot` / `PopActiveSnapshot`. Drop is just
  `UnregisterSnapshot`. This is the right separation: scan paths
  that need a registered snapshot for executor visibility don't
  always need to push/pop the active-snapshot stack.
- **`ResolvedScanSnapshot` shape.** Uses
  `Option<RegisteredSnapshotGuard>` for owned, raw pointer for
  borrowed — same `Option<Guard>` pattern as
  `ResolvedScanHeapRelation` (31209) and
  `ResolvedVacuumHeapRelation` (31216).

## Findings

### 1. Naming distinction is the right one

`ActiveSnapshotGuard` (push to active stack) vs
`RegisteredSnapshotGuard` (just registered) is a meaningful
semantic distinction in PostgreSQL's snapshot lifecycle. Worth
a one-line doc comment at the top of `snapshot_guard.rs`
distinguishing them:

```rust
//! Two snapshot families:
//!   - `RegisteredSnapshotGuard`: Register + Unregister only.
//!     For scans that need a snapshot for executor visibility
//!     but don't push it to the active stack.
//!   - `ActiveSnapshotGuard`: Register + Push, then Pop +
//!     Unregister on drop. For SPI / call sites that need the
//!     snapshot as backend-active.
```

Not required.

### 2. Three packets reuse this primitive (31228, 31229)

The companion IVF and HNSW migrations land next, validating
cross-AM applicability immediately.
