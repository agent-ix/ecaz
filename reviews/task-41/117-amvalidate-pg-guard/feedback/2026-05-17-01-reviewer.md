## Confirmations

- **Four AM `amvalidate` callbacks now have `pg_guard`.** This
  is the standard pgrx unwinding boundary ÔÇö without it, a panic
  in `amvalidate` would unwind across the C FFI and corrupt PG's
  state. With it, the panic is caught and translated to a PG

  no body changes. Safe by construction.
- **Coverage scope is narrow.** Only `amvalidate` callbacks ÔÇö not

## Findings

### 1. Convention milestone but scope question

This is a great starting move but the Task 41 task doc names a
broader inventory:

> Inventory.** Generate the complete list of FFI entry points:
> - `rg -n 'extern "C" fn' src/` + every `#[pg_extern]`,
>   `#[pg_aggregate]`, `#[pg_operator]`, `IndexAmRoutine` field,
>   `CustomScanState` callback, `RegisterXactCallback`,
>   `RegisterSubXactCallback`, etc.
> - Cross-reference against `#[pg_guard]` (or pgrx-managed
>   equivalents).
> - The diff is the panic-unsafe surface; first packet eliminates it.

This packet covers ~4 callbacks. The full inventory is much
larger. Worth a follow-up packet (or a strategy doc update)
that:

1. Generates the inventory at `docs/ffi-inventory.md` ÔÇö the
   task doc explicitly names this artifact.
2. Cross-references which entry points are already guarded vs.
   unguarded.
3. Establishes the migration tracking surface so subsequent
   packets can be slice-able by file or by callback family.

The dylint (`ecaz_panic_across_ffi`) and `make ffi-audit`
target remain unstarted but should follow the inventory.

### 2. Reviewer can't verify completeness without inventory

Without the inventory artifact, this packet's scope ÔÇö
"amvalidate callbacks" ÔÇö is identified by the request but
can't be cross-referenced against "amvalidate callbacks across
the codebase." Maybe there are amvalidate-equivalent functions
in test fixtures or debug helpers that should also be guarded.
A short grep-based check in the artifacts (e.g., "grep
'amvalidate'" output showing all four covered) would make the
"all amvalidate callbacks covered" claim verifiable.

### 3. Pattern note

`#[pg_guard]` is the **lightest-weight** invariant #1 fix ÔÇö
single-attribute add per function. The inventory will likely
turn up:
- Functions that just need `#[pg_guard]` added (lightest)
- Functions that need `catch_unwind` retrofitted (medium)
- Functions whose panics shouldn't be caught and need restructuring (heaviest)

This packet handles the first class. Sequencing follow-up
packets to take similar broad sweeps (e.g., "all
`CustomScanState` callbacks", "all `RegisterXactCallback`s") is
the right approach.
