# Review: Task 41 pinned read-stream buffer guard (31235)

## Summary

**Adds a sibling primitive** `PinnedBufferGuard` to
`src/storage/buffer_guard.rs` — pin-only ownership, no lock.
Uses it for PG18 read-stream loops in
`src/am/ec_spire/scan/relation.rs` and `src/am/ec_ivf/page.rs`.

## Confirmations

- **Pin-only vs pin-plus-lock distinction is the right shape.**
  `PinnedBufferGuard` owns just the pin (releases via
  `ReleaseBuffer`); `LockedBufferGuard` owns pin + lock
  (releases via `UnlockReleaseBuffer`). Two PG APIs, two
  guards — no premature unification.
- **`from_pinned(buffer)` adapter.** Adopts a buffer that PG
  already pinned (e.g., `read_stream_next_buffer` results).
  Right shape — the read-stream API hands buffers to the caller
  pre-pinned, and the guard takes ownership of releasing them.
  Drop iteration on the stream is safe by construction.
- **Eighth shared primitive in `src/storage/`.** The pattern
  continues to scale: each PG resource family gets its own
  guard, all in one module.

## Findings

### 1. Sibling-primitive shape vindicated

This is the second guard in the buffer family. The naming
distinction `PinnedBufferGuard` vs `LockedBufferGuard` is
explicit about ownership — much like the `ActiveSnapshotGuard`
(register + push) vs `RegisteredSnapshotGuard` (register only)
split. Following the convention.

When a third buffer family appears (e.g., for `BAS_BULKREAD`
or `RBM_NORMAL_NO_LOG` semantics), the same shape applies.

### 2. Cross-AM read-stream coverage

Two AMs use this primitive at landing time — confirms the
read-stream pattern is shared infrastructure, not an
AM-specific shape. Worth a one-line doc comment near
`from_pinned` naming the typical PG callers that produce
pre-pinned buffers (read_stream_next_buffer is the main one).
