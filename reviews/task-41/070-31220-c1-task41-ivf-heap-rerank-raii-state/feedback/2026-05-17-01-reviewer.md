# Review: Task 41 IVF heap rerank RAII state (31220)

## Summary

Replaces the raw heap-rerank fields on `EcIvfScanOpaque`
(`Relation`, owned booleans, `Snapshot`, raw slot pointer,
attnum) with **one boxed `IvfHeapRerankState`** owning three
typed guards: `TupleTableSlotGuard`, `ResolvedIvfScanSnapshot`,
`ResolvedIvfScanHeapRelation`. Scan opaque keeps a nullable raw
pointer to the boxed state. Baseline 4,105 → 4,102 (−3).

**This is the first "composite RAII state" packet** — instead of
six raw fields plus boolean ownership flags managed by manual
cleanup helpers, the entire heap-rerank lifecycle is encoded as
one Rust-owned struct of guards. The C-side opaque just holds a
nullable `*mut IvfHeapRerankState` and drops via
`free_heap_rerank_state`.

## Confirmations

- **`PgBox::alloc0` safety preserved.** The scan opaque is
  zero-initializable because it stores only a raw pointer to
  the boxed state, not the guards inline. ✔ Same safety
  property the prior raw-field layout had, but without the
  cleanup booleans.
- **State field drop order preserves slot → snapshot → relation.**
  Field declaration order in `IvfHeapRerankState`: slot first,
  snapshot, relation last. Rust drops in reverse declaration
  order — relation first, snapshot, slot last. Wait — this is
  the opposite of what the request describes. Let me re-read.
  
  Request says "state field drop order preserves slot cleanup
  before snapshot and relation cleanup." For slot to drop
  FIRST, slot must be declared LAST. **Verify the field order
  in the struct matches this intent** — if declaration is
  `slot, snapshot, relation`, drop order is
  `relation, snapshot, slot`, which would drop snapshot **before**
  slot. Worth a closer look (Finding #1 below).
- **Cleanup paths.** `free_heap_rerank_state` drops the boxed
  state on rescan and endscan paths. Guards inside the state
  handle their own Drop.
- **Borrowed cases.** `ResolvedIvfScanHeapRelation` and
  `ResolvedIvfScanSnapshot` use the `Option<Guard>` pattern
  from 31209/31216/31217 — borrowed cases store `None`, owned
  cases store the guard.

## Findings

### 1. Verify field declaration order matches drop-order intent

The request states "slot cleanup before snapshot and relation
cleanup" should be preserved. Rust drops fields in **reverse
declaration order**. So to get `drop(slot) → drop(snapshot) →
drop(relation)`, the struct must declare fields in order
`relation, snapshot, slot`. If the declaration order is the
opposite — `slot, snapshot, relation` — the drop order flips.

A one-line comment above the struct fields anchoring this
intent would prevent a future refactor from silently reordering
the cleanup sequence:

```rust
// Field order is intentional: relation/snapshot drop AFTER slot,
// because Rust drops fields in reverse declaration order.
struct IvfHeapRerankState {
    _relation: ResolvedIvfScanHeapRelation,   // drops 3rd (last)
    _snapshot: ResolvedIvfScanSnapshot,       // drops 2nd
    slot: TupleTableSlotGuard,                // drops 1st
}
```

Worth verifying the on-disk field order matches this intent
before declaring the migration complete. (Same comment applies
to the HNSW analog in 31221.)

### 2. Composite RAII state is the right shape

This pattern — boxed Rust-owned state struct + nullable raw
pointer in the C-side opaque — is the natural evolution of the
RAII work. It scales to any number of resource families
co-managed by a single scan lifecycle. The DiskANN scan-state
work in 31217 also moves in this direction. Worth a one-line
note in the 31161 strategy doc as the "compose multiple guards
into a state struct" idiom for any future scan / vacuum / build
lifecycle that owns multiple PG resources.
