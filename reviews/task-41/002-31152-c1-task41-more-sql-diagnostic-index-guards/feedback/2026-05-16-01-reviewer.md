# Review: Task 41 more SQL diagnostic index guards (912)

## Summary

Mechanical migration of seven SPIRE/IVF SQL diagnostics
(`ec_spire_custom_scan_index_eligibility`,
`ec_ivf_index_drift_snapshot`, `ec_ivf_index_admin_snapshot`,
`ec_spire_index_active_snapshot_diagnostics`,
`ec_spire_index_allocator_snapshot`,
`ec_spire_index_placement_snapshot`,
`ec_spire_remote_node_snapshot`) from
`open_valid_ec_*_index` + manual `index_close` to
`open_valid_ec_*_index_guard`. Each function is read-only AM access
followed by `TableIterator::*`, so dropping the guard at function
return is correct. Baseline drops 4700 → 4684 (−16).

## Confirmations

- **No extended lifetime.** Each migrated function holds the guard
  only across one AM call and ends with either `TableIterator::once`
  or `TableIterator::new` over an owned `Vec`. No SPI, no raw-pointer
  storage, no escape through closures. The relation is closed by
  `Drop` at function return — same effective lifetime as the old
  explicit `index_close` at the same point. ✔
- **Validator-only case.** `ec_spire_register_remote_node_descriptor`
  (line 2005-2010 of the new code) now opens the guard in a `{}`
  scope and binds it to `_index_relation` so it drops immediately
  after validation, matching the old open/close-only pattern. The
  subsequent `Spi::connect_mut` work then proceeds without an open
  AccessShare lock on the index. ✔
- **Remaining raw callers unchanged.** `grep` finds ~80 callsites
  still using the raw `open_valid_ec_*_index` helpers (line 9188
  onward); they retain manual `index_close` responsibility as
  intended. Migration scope-limited as stated.
- **Baseline arithmetic.** −16 entries lines up with seven migrated
  functions × the SAFETY-annotated `index_close` (and a few
  `index_open` notes) each previously carried.

## Findings

No blockers. Two minor notes carried forward from 911:

- The Drop-on-`pgrx::error!`-unwind invariant continues to be the
  load-bearing piece. Within the migrations in this packet there are
  no error paths between guard open and the AM read (the helper
  itself raises on validation failure), so this is moot here — but
  for future migrations of fallible functions, audit any branch
  between guard open and end-of-scope to ensure Drop is the only
  needed close.
- Task 41 plan should track removal of the residual raw helpers
  (`open_valid_ec_hnsw_index`, `open_valid_ec_ivf_index`,
  `open_valid_ec_spire_index`, `open_valid_ec_diskann_index`) as an
  explicit closeout step once their last callers are migrated.
