# Strategy: consolidate the three index-relation guards into one parametric primitive

Follow-up to my first 31161 feedback. Now that packets 31172 and
31173 have landed two additional guard structs alongside the
original `AccessShareIndexRelation`, the proliferation pattern is
concrete enough to act on. The recommendation below is grounded
in the std/parking_lot/Rust API Guidelines convention rather than
local preference.

## Decision: unify

Adopt a single `IndexRelationGuard { relation, lockmode }`
primitive with two constructors (`open` errors on null,
`try_open` returns `Option`). Replace the three current
variants. Keep the `open_valid_ec_*_index_guard` AM-validation
helpers in `src/lib.rs` as a layer above the primitive.

## Current state (post-31173)

| File | Type | Lockmode | Null behavior | Lines |
| --- | --- | --- | --- | --- |
| `src/lib.rs:222` | `AccessShareIndexRelation` | Fixed AccessShareLock | `pgrx::error!` | ~30 |
| `src/am/ec_spire/dml_frontdoor/mod.rs:197` | `AccessShareIndexRelation` (shadowed name) | Fixed AccessShareLock | Returns `None` | ~30 |
| `src/am/ec_spire/coordinator/debug.rs:5` | `DebugIndexRelation` | Variable | `pgrx::error!` | ~30 |

All three:

- wrap `pg_sys::Relation`,
- close via `pg_sys::index_close`,
- expose `as_ptr() -> pg_sys::Relation`,
- depend on the same pgrx-unwind SAFETY contract.

The only differences are a `LOCKMODE` value and whether `open`
returns `Self` or `Option<Self>`.

## Why unify rather than keep separate (Rust convention)

The std/parking_lot guard pattern uses **separate types when
variants differ in capability**:

- `MutexGuard` vs `RwLockReadGuard` vs `RwLockWriteGuard` are
  distinct types because a write guard can mutate (a read guard
  can't), and `RwLock::downgrade()` converts a write guard to a
  read guard â€” needing distinct signatures.
- parking_lot is internally **parametric** over `RawMutex` and
  exposes the per-mode names as type aliases; the open RFC
  [rust-lang/rfcs#3377] proposes pushing genericity even further.

Convention uses **one parametric type with constructor variants
when variants differ only in configuration**:

- `std::fs::File` is one type; read-only vs read-write is
  configured by `OpenOptions`, not a separate `ReadOnlyFile`
  type.
- `BufReader<R>` is one type parameterized on the underlying
  reader, not `BufReaderForFile` / `BufReaderForTcp`.

The three index-relation guards here expose the **same
operations** (`as_ptr` is the only method, `Drop` is the only
behavior). Lockmode is a value, null behavior is a constructor
choice. This is the `File`/`OpenOptions` shape, not the
`Mutex`/`RwLock` shape.

The Rust API Guidelines also push against the alternative of a
`guards::*` module ("Names of items within a module should not
be prefixed with that module's name to avoid stuttering").
Co-locate the primitive with the resource it wraps â€” i.e., next
to other relation/storage primitives â€” rather than in a
dedicated guards namespace.

## Proposed shape

```rust
// src/storage/relation_guard.rs (or extend existing src/storage/relation.rs)

pub(crate) struct IndexRelationGuard {
    relation: pg_sys::Relation,
    lockmode: pg_sys::LOCKMODE,
}

impl IndexRelationGuard {
    /// Open an index relation. Errors via `pgrx::error!` on null.
    /// Use for callers where a null open is a hard failure.
    pub(crate) fn open(
        index_oid: pg_sys::Oid,
        lockmode: pg_sys::LOCKMODE,
    ) -> Self {
        // SAFETY: PostgreSQL owns the relation cache entry; this
        // guard owns the matching close for `lockmode`.
        let relation = unsafe { pg_sys::index_open(index_oid, lockmode) };
        if relation.is_null() {
            pgrx::error!("index_open returned NULL");
        }
        Self { relation, lockmode }
    }

    /// Open an index relation. Returns `None` on null.
    /// Use for callers that need to skip dropped/invalid indexes
    /// without aborting (e.g., catalog walks with `continue`).
    pub(crate) fn try_open(
        index_oid: pg_sys::Oid,
        lockmode: pg_sys::LOCKMODE,
    ) -> Option<Self> {
        // SAFETY: same as `open`; the null branch returns `None`
        // and no close is owed.
        let relation = unsafe { pg_sys::index_open(index_oid, lockmode) };
        if relation.is_null() {
            return None;
        }
        Some(Self { relation, lockmode })
    }

    pub(crate) fn as_ptr(&self) -> pg_sys::Relation {
        self.relation
    }
}

impl Drop for IndexRelationGuard {
    fn drop(&mut self) {
        // SAFETY: `relation` was returned by `index_open` in
        // `IndexRelationGuard::{open, try_open}`; this guard
        // owns the matching close. Relies on pgrx ERROR paths
        // unwinding Rust frames so `Drop` runs; re-audit on
        // pgrx bumps or `pg_guard` behavior changes.
        unsafe { pg_sys::index_close(self.relation, self.lockmode) };
    }
}
```

Call-site migration mapping:

| Old | New |
| --- | --- |
| `AccessShareIndexRelation::open(oid)` (`src/lib.rs`) | `IndexRelationGuard::open(oid, AccessShareLock)` |
| `AccessShareIndexRelation::open(oid)` (`dml_frontdoor`) | `IndexRelationGuard::try_open(oid, AccessShareLock)` |
| `DebugIndexRelation::open(oid, lockmode)` (`debug.rs`) | `IndexRelationGuard::open(oid, lockmode)` |

The `open_valid_ec_*_index_guard` AM-validation helpers in
`src/lib.rs` keep their bodies, just construct
`IndexRelationGuard::open(oid, AccessShareLock)` internally
instead of inlining the struct. The
`ec_spire_index_heap_relation_oid` /
`ec_spire_heap_relation_oid_from_index` helpers from 31166/31167
take `&IndexRelationGuard` instead of
`&AccessShareIndexRelation` â€” minor signature update, same
borrow semantics.

## Impact

| Metric | Now | After |
| --- | --- | --- |
| Guard impl LoC | ~90 (3 Ă— ~30) | ~50 |
| Distinct guard types | 3 (two share a name) | 1 |
| SAFETY contracts to audit on pgrx bump | 3 | 1 |
| Type-confusion risk in stack traces / IDE | High (shadowed name) | Zero |
| New buffer/WAL track wrappers can follow the same shape | No precedent | Yes (`BufferPinGuard`, `LwLockGuard` follow the parametric pattern) |

Migration is mechanical: each existing variant becomes one of
the two constructors. No behavior change. No public-API churn
(these are all `pub(crate)` or private).

## Why now, before the buffer/WAL track

Task 41 invariant #3 explicitly names four more RAII wrapper
families to land
(`plan/tasks/41-ffi-safety-boundary.md:91-98`): `BufferPin`,
`LwLockGuard`, `Snapshot`, `SpiTuptable`. If the
"module-local copy" precedent from 31172/31173 carries forward,
each family will get its own per-module variants â€” potentially
6â€“10 RAII structs in the codebase, each carrying its own
SAFETY contract and per-module Drop impl. Establishing the
"shared primitive, parametric over the resource state" shape
**now** sets the convention for the next four wrapper families
to follow. Doing it after they land is a wider refactor across
more modules with higher drift risk.

## Suggested sequencing

1. One packet adds `src/storage/relation_guard.rs` (or
   equivalent) with the primitive, no migrations yet.
2. One packet migrates `src/lib.rs::AccessShareIndexRelation`
   call sites and deletes the original struct.
3. One packet migrates `dml_frontdoor` and `debug.rs` to the
   primitive and deletes both module-local structs.
4. Document the pattern in the 31161 strategy matrix as the
   shape for the next wrapper families.

Each packet is small, baseline-neutral or slightly negative
(fewer SAFETY lines), and reversible. Total surface: ~5 files
touched.
