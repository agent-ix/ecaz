# Review: Task 41 unsafe surface strategy (921)

## Summary

Pivots Task 41 from "open the next wrapper slice" to a strategy
matrix that groups the 4,579-entry residual baseline by safety
mechanism: relation resources, buffer/page/WAL resources, scan
descriptor / tuple slot / detoast lifetimes, sync primitives,
SIMD/quantization, and test-only. Defers sync to Task 40, defers
pointer-heavy residuals behind Task 43 proof coverage, and keeps
test-only counts out of production sequencing. Strategy directly
addresses the "raw helpers pending removal" and "what's next" notes
I left on 911/912.

I'm satisfied with the strategy direction. The findings below are
sharpenings, not redirections.

## Confirmations

- **Structural deletion ahead of comment backfill.** The matrix
  pushes the remaining `open_valid_ec_*_index` callers to the front,
  with raw-helper deletion as the closeout. That's the right
  ordering — Task 35 (residual SAFETY comments) is correctly
  blocked behind shrinking the wrappable surface first.
- **Task 35 deferral is explicit.** Section "Synchronization and
  Parallel State" and the SIMD section both name Task 40 / Task 43
  as the gating dependencies and forbid piecemeal annotation. Good
  — this prevents racing comment-writers from claiming proof of
  invariants that the lifting tasks will restate.
- **Next-slice selection rule.** Item (1) says "migrate remaining
  callers, split validation-only from AM callers, delete raw
  helpers." Combined with the explicit-`drop` idiom landing in 920
  / 922, the next-slice rule is concrete enough for another agent
  to pick up.

## Findings

### 1. No projected entry deltas per cluster

The strategy lists clusters but not the entry-count budget each is
expected to retire. Today the baseline is 4,579; after a full
relation-resource sweep, what should it look like? After the buffer/
WAL wrapper track? Without numbers, "we're making progress" can hide
unequal yields.

Suggest adding to `strategy-matrix.md` a "projected baseline after
this track" estimate per cluster. The baseline report already has
per-file counts (`scan_debug.rs` 478, `lib.rs` 366, `ec_ivf/page.rs`
224, etc.), so the projection is straightforward:

| Track | Projected baseline after |
| --- | --- |
| Relation resources (finish) | ~4,300 (lib.rs raw helper deletions) |
| Buffer/page/WAL wrappers | ~3,700 |
| Scan descriptor / tuple slot | ~3,400 |
| Task 40 sync lift (external) | ~3,200 |
| SIMD / Task 43 proof | ~3,100 |
| Test-only sweep | ~2,500 |

(Numbers illustrative; the coder should pick them based on the
top-files report.)

### 2. Buffer/page/WAL track lacks an entry point

Section says "Start with page-codec and WAL-heavy files such as
`src/am/ec_ivf/page.rs`, `src/am/ec_spire/page.rs`, and
`src/storage/wal.rs`. Target paired resources first: buffer
pin/release, lock/unlock, generic WAL start/finish/abort." Good
ordering, but no concrete first slice. Recommend an explicit "next
packet" identification: e.g. "first slice: introduce
`BufferPinGuard` for `src/storage/page.rs` and migrate the N
callers in `ec_ivf/page.rs`."

### 3. Strategy does not name a stop condition

When is Task 41 done? "Baseline = 0" is unrealistic given the
intrinsic-heavy SIMD and SAFETY-required cases. Suggest defining
closeout: "Task 41 closes when all paired-resource surfaces have
RAII wrappers, raw helpers are deleted, and the residual is
restricted to (a) cases gated on Task 40, (b) cases gated on Task
43, and (c) SAFETY-commented intrinsic blocks." This converts the
burndown into a satisfiable condition rather than an open-ended
chase.

### 4. Validation-only helper not yet specified

Section (1) says "add a validation-only helper for callers that only
need AM/relkind checks." Three callsites already use the
`{ let _index_relation = ...; }` workaround (914 entry_catalog,
freshness; 912 register_remote_node_descriptor). A small named
helper would make their intent explicit and let the wrapper carry a
SAFETY comment in exactly one place. Suggest:

```rust
fn validate_ec_spire_index(index_oid: pg_sys::Oid, caller: &'static str) {
    let _ = open_valid_ec_spire_index_guard(index_oid, caller);
}
```

Promotes the pattern to first-class and signals "validation only,
no AM read intended" at the callsite.

## Cross-cutting note (carries to 922)

The Drop-on-`pgrx::error!`-unwind invariant introduced in 911 is now
load-bearing across the entire migrated surface (910 through 922).
Task 41 closeout should include either a regression test that
opens-and-errors a guard (asserting the relation reference count
returns to baseline) or an ADR documenting the pgrx version
contract. Currently the invariant is implicit; on a pgrx bump it
would silently break.
