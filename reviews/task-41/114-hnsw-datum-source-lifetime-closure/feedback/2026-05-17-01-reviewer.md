# Review: Task 41 HNSW datum source lifetime closure (114)

## Summary

**Major milestone:** the first packet to land Task 41 invariant
#2 (memory-context lifetime safety). Adds explicit lifetimes to
`FlatFloat4ArrayRef`, `FlatFloat4VarlenaRef`, and
`FlatFloat4SourceRef` in `src/am/ec_hnsw/source.rs`. Makes the
raw `from_datum` constructors private to that module. Replaces
return-a-wrapper APIs with **higher-ranked closure helpers**:
- `with_flat_float4_source_from_datum`
- `with_source_from_heap_row`
- `with_indexed_ecvector_from_slot`

Migrates HNSW build / insert / vacuum / scan and IVF heap-rerank
callers so Datum-backed slices are consumed (copied or scored)
inside the borrow closure rather than escaping it.

This is the design pattern I described at the strategic level in
the Task 41 progress conversations — instead of a `PallocCtx<'cx>`
newtype, the codebase is going with the **CPS / closure-with-borrow**
shape (the `with_foo(|borrow| ...)` idiom). Both encode "the
borrow cannot escape" — closure form is lighter-weight and
doesn't require threading a context type through every signature.

I'm satisfied with this slice as the invariant #2 opening move.
Substantive findings below.

## Confirmations

- **Higher-ranked closure correctly prevents escape.** Each
  `with_*` helper takes a `for<'a> FnOnce(&'a Wrapper<'a>) -> R`
  shape. The `R` cannot capture `'a`, so the wrapper cannot
  leave the closure. The compiler enforces this at the type
  level — much stronger than a doc comment or convention. ✔
- **Raw `from_datum` constructors are now `pub(super)` /
  private to `source.rs`.** External callers can't bypass the
  closure helpers. Type-system enforcement, not convention. ✔
- **Callers migrated correctly.** Spot-checked:
  - `build.rs` source-scan path: closure does the copy-to-Vec
    before slot clear ✓
  - `insert.rs` heap-source scorer: scores inside the closure
    then drops the wrapper ✓
  - `scan.rs` heap-rerank: same pattern, score-and-drop ✓
  - `vacuum.rs` source scorer: same shape as insert ✓
  - `ec_ivf/scan.rs` heap-rerank: cross-AM consumer, same
    pattern ✓
- **Build / test passes.** `cargo check --features pg18`
  recorded in `artifacts/`.

## Findings

### 1. Pattern note: this is the closure-CPS form of invariant #2

The Task 41 task doc describes invariant #2 as "Newtype palloc'd
lifetimes: introduce `PallocCtx<'cx>` and require any `&'cx [u8]`
/ `&'cx str` to carry it." This packet takes a different but
equivalent approach: instead of an explicit context type
threaded through signatures, use **higher-ranked closures** that
own the borrow's lifetime locally.

Both encode the same invariant. The closure form is the more
pragmatic choice for a Rust-on-PG codebase because:

- No need to propagate `'cx` through every function that touches
  source data
- The closure body is the natural place to do the copy-to-owned
  or score-immediately work that's the *point* of the invariant
- It composes cleanly with the existing pgrx FFI surface

The downside is that the closure form only catches **direct
borrow escape**. A caller that copies the raw pointer out of the
wrapper inside the closure (via `unsafe`) and stashes it in an
outer-scope variable would still escape. The `PallocCtx<'cx>`
type-propagation form would catch that too. For now the closure
form is sufficient because every migrated caller copies to
owned `Vec` immediately — but worth noting in case a future
caller wants to be clever.

### 2. Suggested follow-up: extend the pattern to other source
types

This slice covers HNSW's FlatFloat4 family. The Task 41
invariant #2 surface is broader:
- DiskANN binary sidecar + payload encoding
- SPIRE leaf assignment row decoding
- Generic `Datum`-wrapping that returns wrappers from `from_datum`

Each is a candidate for the same `with_*` closure treatment.
Worth a follow-up packet that catalogs the remaining Datum-
borrow sites (analogous to the `grep`-based inventory I
suggested for invariant #1's FFI surface).

### 3. The "scored immediately" invariant is implicit per
callsite

For each migrated caller, the proof that "scoring happens before
slot clear" is per-callsite — there's no central enforcement.
That's fine for an opening packet, but a future Task 35 SAFETY
comment pass could codify it: each `with_*` callsite gets a
one-line comment saying "score-and-drop inside the closure;
slot will be cleared on next loop iteration."

### 4. Cross-cutting: invariant #2 progress signal

This is **the first invariant #2 packet**. Until now, Task 41
was entirely invariant #3 (resource release on early return).
With this packet, invariant #2 (memory-context lifetimes) goes
from 0% to ~10% — HNSW's hottest source-data path is now type-
enforced. Worth recording as a Task 41 milestone in any
status-tracking artifact.
