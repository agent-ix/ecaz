### Deferred work: grouped heap-rerank scan-state relation bundle

The request notes that "the longer-lived grouped heap-rerank
scan-state relation remains for a separate resource-state slice
because it stores relation, snapshot, and slot state across scan
operations." This is the right boundary ÔÇö the scan-state bundle
combines all three resource families I've been pushing to
consolidate into `src/storage/`:

- relation (handled by `HeapRelationGuard`/`IndexRelationGuard`
  ÔÇö primitives exist)
- snapshot (still module-local across multiple files ÔÇö see my
  31175 / 31182 findings)
- tuple slot (still module-local in 31175 / 31177 / 31182)

When the scan-state slice lands, it would benefit from having
the snapshot and slot wrappers already promoted to
`src/storage/`. Otherwise the scan-state struct will end up
storing module-local guard types from three different files,
which makes the lifetime story harder to audit.

**Suggested sequencing for the next non-mechanical packet:**

1. Promote `ActiveSnapshotGuard` and `OwnedTupleTableSlot` (or
   names of choice) into `src/storage/snapshot_guard.rs` and
   `src/storage/slot_guard.rs` (sketches in my 31182 feedback).
2. Migrate the 2-3 existing module-local copies of each to the
   shared types.
3. Then build the scan-state struct on top of the consolidated
   set.

Doing this in the order above means the scan-state struct
already has typed, compiler-verified `&Guard` borrow APIs to
build on, rather than re-inventing them locally.
