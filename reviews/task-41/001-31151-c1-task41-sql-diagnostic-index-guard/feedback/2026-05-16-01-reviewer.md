# Review: Task 41 SQL diagnostic index guard (911)

## Summary

Introduces `AccessShareIndexRelation` (`src/lib.rs:222-253`) as an
RAII wrapper around `index_open(... AccessShareLock)` / `index_close`
and centralizes AM validation through `open_valid_ec_index_guard`.
The four per-AM helpers (`hnsw`/`ivf`/`spire`/`diskann`) collapse to
one shared body. Existing unsafe-pointer callers continue to compile
through `into_raw()`. Two diagnostics (`ec_hnsw_index_admin_snapshot`,
`ec_diskann_index_graph_summary`) now hold the guard directly and the
manual error-path `index_close` calls are gone. Baseline drops 4725
→ 4700 (−25), matching `unsafe-baseline-{before,after}.log`.

I'm satisfied with the slice. The findings below are notes for the
remainder of the migration, not blockers.

## Confirmations

- **One close per open.** `AccessShareIndexRelation::open` calls
  `index_open` once; `Drop` closes once; `into_raw` consumes via
  `std::mem::forget` so no double-close on the raw-pointer path.
  Search confirms no `index_close(... AccessShareLock)` callsites
  outside the guard's `Drop` (`src/lib.rs:252` is the only one).
- **Relation alive across `rd_rel`/`relname` reads.** In
  `open_valid_ec_index_guard`, `index_relation` is a stack-owned
  guard. `rd_rel` is read through `as_ptr()` and the relname `CStr`
  is converted to an owned `String` before `pgrx::error!` is invoked,
  so the format argument outlives the relation. ✔
- **`into_raw` preserves the old close contract.** Raw helpers
  (`open_valid_ec_hnsw_index`, etc.) still return
  `pg_sys::Relation`; callers retain responsibility for
  `index_close`. No churn for the not-yet-migrated SQL diagnostics.
- **Baseline arithmetic.** 25 grandfathered entries removed lines up
  with the four collapsed validation bodies (each previously held
  several SAFETY-annotated `unsafe` blocks for inline error-path
  closes and `rd_rel` reads).

## Findings

### 1. Drop-on-unwind depends on `pgrx::error!` panic semantics

The new design deletes the inline `index_close` calls on the error
paths and relies on `Drop` running during the `pgrx::error!` unwind.
In pgrx 0.17 (locked in `Cargo.lock`), `pgrx::error!` is a Rust
panic caught at the `pg_guard` FFI boundary, so destructors do run.
This is correct today, but it's also the precise contract that would
silently break if a future pgrx bump (or a custom panic hook) reverted
to a direct `ereport(ERROR)` longjmp.

Two cheap insurance options:

- Add a SAFETY block comment on
  `impl Drop for AccessShareIndexRelation` (or above
  `open_valid_ec_index_guard`) documenting "Drop must run during
  `pgrx::error!` unwind in pgrx ≥ 0.13; revisit on every pgrx bump."
- Or a focused regression test that opens with a deliberately wrong
  `relam` and asserts the relation reference count returns to its
  pre-call value after the elog-caught ERROR.

The point isn't that the code is currently wrong — it isn't — it's
that the invariant we now depend on is implicit rather than enforced.

### 2. `if relation.is_null()` is unreachable today

`index_open` raises `ERROR` (longjmp) on bad oids/missing relations
before returning, so the null check at `src/lib.rs:230` cannot fire
on a real PG runtime. Fine as defensive code, but worth a comment if
it's intentional (otherwise a future cleanup will delete it as dead).

### 3. Two of the four raw `open_valid_ec_*_index` helpers are now
single-call thin wrappers

`open_valid_ec_hnsw_index` (line 294) and `open_valid_ec_ivf_index`
(line 308) only exist to satisfy not-yet-migrated callers (planner
cost callbacks, etc.). That's the stated intent. Recommend tracking
their removal as an explicit closeout step in the Task 41 plan so
this scaffolding doesn't fossilize.

### 4. Caller name string formatted in panicking branch only

`relation_name` is built (`to_string_lossy().into_owned()`) only on
the `relam` mismatch branch; the `relkind` mismatch path doesn't
include the offending relname. Minor, but the diagnostic value for a
user staring at a "requires an index relation" elog without knowing
which relation they pointed at could be improved by including the
relname there too (same CStr read).
