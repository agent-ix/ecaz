# Review: Task 41 generic relation guard (31191)

## Summary

Adds `RelationGuard` to `src/storage/relation_guard.rs` as the
third primitive in the shared module — alongside
`IndexRelationGuard` and `HeapRelationGuard` — covering the
generic `pg_sys::relation_open` / `pg_sys::relation_close` pair
that aux-store callers need. Minimal initial API (`try_open`,
`as_ptr`); panic-on-null `open` deferred until a production
caller wants it. Migrates first one-shot aux-store users in
`src/am/ec_spire/storage/relation_plan.rs` and
`src/tests/build.rs`. Baseline 4,252 → 4,251 (−1).

## Confirmations

- **Same shape as existing primitives.** `RelationGuard` mirrors
  `IndexRelationGuard` and `HeapRelationGuard`: `{ relation,
  lockmode }` fields, `try_open(oid, lockmode) -> Option<Self>`,
  `as_ptr(&self) -> pg_sys::Relation`, identical SAFETY contract
  on Drop (including the pgrx-unwind re-audit note). Consistent
  with the convention I sourced in my 31161 strategy feedback. ✔
- **Minimal API surface.** Only `try_open` and `as_ptr` exposed.
  The request explicitly notes that an erroring `open` can be
  added later "when production callers want that behavior" —
  this is the right discipline (don't add API without a caller).
- **Aux-store + test migrations are mechanical.**
  `relation_plan.rs` drops 6 lines of manual open/close handling;
  `tests/build.rs` does the same. Both still hold the guard
  across the read that needs the live relation.

## Findings

No blockers.

### 1. Naming: `RelationGuard` is the generic-name slot already used

This is a minor signal-vs-noise observation, not a request to
rename. The three primitives now in the module are:

| Type | Backed by | Lockmode |
| --- | --- | --- |
| `IndexRelationGuard` | `index_open` / `index_close` | configurable |
| `HeapRelationGuard` | `table_open` / `table_close` | configurable |
| `RelationGuard` (new) | `relation_open` / `relation_close` | configurable |

`RelationGuard` is the most generic name and goes with the
lowest-level Postgres open function. That's fine, but a future
reader scanning the module might assume `RelationGuard` is the
"main" type and the other two are specializations — when
actually all three are siblings backed by different PG entry
points (each entry point applies different relcache + lock
semantics, even with the same lockmode). A one-line module-level
note distinguishing the three by intended caller would help:

```rust
//! Three sibling primitives:
//!   - `IndexRelationGuard`: index-only opens via `index_open`.
//!   - `HeapRelationGuard`: heap/table opens via `table_open`.
//!   - `RelationGuard`: any relkind via `relation_open` — pick
//!     when the relkind isn't known statically (aux-stores,
//!     planner reflection).
```

Not required.

### 2. Defer of erroring `open`

The packet correctly defers the erroring `open` constructor.
Watch for an inconsistency once a production caller lands: in
`IndexRelationGuard` the convention is `open(...)` panics and
`try_open(...)` returns `Option`. `HeapRelationGuard` currently
only has `try_open` / `try_access_share`. `RelationGuard`
matches `HeapRelationGuard`. When the first erroring caller
arrives, add `RelationGuard::open(oid, lockmode, caller)`
mirroring `IndexRelationGuard::open` so the naming stays
predictable.

### 3. The `coordinator/snapshots.rs` helper in 31192 uses this primitive

This packet's primitive is exercised by 31192's
`open_storage_relation_or_index` — the production aux-store
walker. That validates the design: same module of three sibling
primitives covers the three needed call shapes.
