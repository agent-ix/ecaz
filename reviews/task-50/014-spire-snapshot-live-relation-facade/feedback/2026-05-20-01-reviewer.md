## Finding: safe wrappers hide raw-pointer dereferences

This packet adds two new helpers that wrap previously-`unsafe`
operations in *safe* function signatures:

```rust
fn live_index_relation(index_relation: pg_sys::Relation) -> SpireLiveIndexRelation {
    // SAFETY: callers are PostgreSQL AM/SQL diagnostic entry points that keep
    // the SPIRE index relation live for the duration of the helper call.
    unsafe { SpireLiveIndexRelation::new(index_relation) }
}

fn live_index_relid(index_relation: pg_sys::Relation) -> u32 {
    // SAFETY: callers pass a live SPIRE index relation; rd_id is copied and no
    // ownership is taken.
    unsafe { (*index_relation).rd_id }.into()
}
```

`pg_sys::Relation` is `*mut RelationData` — a raw pointer.
`SpireLiveIndexRelation::new` was deliberately defined as `unsafe
fn` in packet 008 so callers acknowledge the "relation is live"
contract. This packet *defeats* that decision: `live_index_relation`
is a safe function that calls the `unsafe fn` constructor on an
arbitrary raw pointer. Any code in this module can now call it
without the SAFETY acknowledgement.

Same for `live_index_relid` — dereferences `*index_relation` in a
safe function.

This is strictly worse than the prior shape, where each call site
had a one-line `unsafe { ... }` block with a SAFETY comment naming
the local invariant. The block-count tool counts the new shape as
fewer unsafe blocks because the unsafe is hidden inside a safe
function — but the actual safety contract is weaker.

This is the same anti-pattern noted in feedback for packets 012 and
015. Production code makes it more serious here than in 012's
test-only context.

## Fix

Two clean options. Pick the one that fits the call-site shape best
across the next SPIRE slice.

**Option A — keep callers `unsafe`-aware (smallest delta):**

Restore the call-site shape from packet 008, where each diagnostic
function does
`let index = unsafe { SpireLiveIndexRelation::new(index_relation) };`
and the SAFETY comment names the local invariant. Per-call-site
block count goes back up by 1 per diagnostic; the file still
trivially meets the 30% target via the rest of the cleanup.

**Option B — take a proven-live wrapper:**

If the surrounding diagnostic functions already hold a guard or
borrow that proves the relation is live, take that as input:

```rust
fn live_index_relation(guard: &IndexRelationGuard) -> SpireLiveIndexRelation {
    // SAFETY: the guard owns the relation for its lifetime.
    unsafe { SpireLiveIndexRelation::new(guard.as_ptr()) }
}
```

Then the safe API is sound because the input borrow carries the
liveness contract.

If there's no such guard at this layer (the diagnostic functions
take `pg_sys::Relation` directly from PG callbacks), Option A is
the right answer.
