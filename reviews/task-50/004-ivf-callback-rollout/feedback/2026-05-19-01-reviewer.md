## Structural question: `am_callback` vs `pg_am_callback!`

Packet 004 introduced a *second* helper alongside the Slice 1a
function:

```rust
// src/am/common/callback.rs
#[inline]
pub(crate) fn am_callback<R>(callback: impl FnOnce() -> R) -> R { ... }

macro_rules! pg_am_callback {
    ($body:block) => {{ unsafe { pgrx::pgrx_extern_c_guard(|| $body) } }};
}
```

Both wrap `pgrx_extern_c_guard`; the macro expands to the same
closure shape the function takes. Callers now mix the two:

```text
src/am/ec_ivf/cost.rs:123  am_callback(ivf_tree_height_callback_value)
src/am/ec_ivf/cost.rs:131  am_callback(|| { ... })
src/am/ec_ivf/cost.rs:71   pg_am_callback!({ ... })
src/am/ec_ivf/scan.rs:317  pg_am_callback!({ ... })
src/am/ec_spire/cost/mod.rs:523  am_callback( ... )
src/am/ec_spire/cost/mod.rs:543  am_callback(|| { ... })
```

I cannot see a structural difference between the two paths.
`return`, `?`, and never-type behavior are identical because the
macro also wraps `$body` in a closure (`|| $body`). The only call
site advantage of the macro is dropping `||` from the syntax.

**Request:** pick one and roll the other out. My recommendation is
the **function** `am_callback`, because:

1. It is a real Rust item — `cargo doc`, IDE jump-to-def, refactor
   tooling all work; macros lose all three.
2. `unsafe` is contained in one inline body, not duplicated in every
   macro expansion (the block-count tool already conflates the
   macro definition's `unsafe { ... }` with one fixed cost; the
   function form has the same single-cost shape).
3. Reviewers and grep-based audit tools find `am_callback(` more
   easily than `pg_am_callback!(`.

If there is a load-bearing reason the macro form must exist (for
example, a borrow checker case I missed where a `FnOnce` capture
fails but a block does not), please cite the call site in the
follow-up packet and document why both helpers must coexist.

This is **non-blocking** — packet 004 is approved as-is. The
consolidation can land as part of Slice 1c (packet 005 is also
already on the branch) or as a separate cleanup packet.
