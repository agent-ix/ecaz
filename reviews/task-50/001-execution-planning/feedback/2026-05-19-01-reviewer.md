# Approve with revisions: Task 50 execution-planning bridge

## Decision

Approve the planning packet **conditionally** — the four documents
(`consolidated-candidates.md`, `slice-sequence.md`,
`bench-baseline-plan.md`, `block-count-tooling.md`) cover the right
ground and are a strong first draft. Eight points below need
revision before a coder agent can land Slice 1 cleanly. Most are
doc-only edits to the existing files; no new artifacts required.

## What's strong (preserve these)

- All four documents present, well-structured, source-evidence-
  cited per candidate (closeout references, packet numbers, file
  counts).
- Ranking formula `(blocks absorbed × callsites × cross-AM
  applicability) / (impl risk + perf risk)` applied honestly, with
  the **product-priority modifier** (RaBitQ/SPIRE/IVF outrank
  cleanup-only HNSW/DiskANN) as a sensible tie-breaker addition.
- Bench tolerances anchored to existing M5 tasks (Task 31/32/33,
  Task 30 phase 13d) rather than invented.
- Block-count tooling spec includes both Make target and optional
  script form, with a top-15 snapshot at HEAD baked in.
- Slice 5 deliberately deferred-decision (reloptions vs. vector
  datum, depending on Task 39/47 outcomes) — honest about
  dependencies.

## Eight requested revisions

### 1. Slice 1 is too broad — split into 1a / 1b / 1c

`slice-sequence.md` rolls Slice 1 into "IVF
cost/build/scan/vacuum/insert + SPIRE scan/cost/insert/vacuum +
common." That's three packets' worth of work bundled. Bug-bisect
across 4 AMs is hard; a subtle inlining regression in the helper
would mark every downstream packet bad.

**Revision:** split Slice 1:

- **Slice 1a:** introduce the callback helper in `src/am/common`
  with the smallest possible first user (pick one IVF callback).
- **Slice 1b:** IVF rollout to remaining callbacks.
- **Slice 1c:** SPIRE rollout to remaining callbacks.

Each independently reviewable. Pattern matches HNSW 094–097's
build_parallel layered slicing.

### 2. Reconcile top-15 vs. priority sequence

The Task 50 task doc's exit criterion is "top-15 densest modules
each see ≥30% block-count drop." The planning packet's priority
sequence is RaBitQ/SPIRE/IVF first; HNSW/DiskANN consume shared
helpers later. **Six of the top 10 densest files are HNSW**
(`scan_debug.rs` 356, `scan.rs` 226, `build_parallel.rs` 203,
`insert.rs` 133, `vacuum.rs` 99, `source.rs` 78). Under the
priority sequence, those reduce only via shared-helper rollout —
not via direct touch.

**Revision:** add a `top-15-coverage-map.md` (or extend
`block-count-tooling.md`) mapping each top-15 file to which
slice(s) will reduce its count, with projected delta per file.
Without this, "did we hit the exit criterion" is guesswork.

### 3. Add a risk register

Each candidate carries different perf and correctness risk. The
planning packet's "Expected payoff" sections are useful but don't
say what could go wrong.

**Revision:** short table — candidate × failure mode × mitigation
× verification. Examples:

- **AM callback macro** → closure inlining inhibited → `#[inline]`
  + disassembly check on hot path → bench delta verification.
- **Heap source scorer** → accidental per-candidate allocation →
  borrow not own where possible → alloc count via `dhat` or
  instrumented build.
- **ActiveEpochAnchor** → hidden lock acquisition costs moved into
  the type → don't move lock decisions into the wrapper → read-
  efficiency bench.
- **SIMD newtypes** → AVX2 vs NEON inlining divergence → keep
  `#[target_feature]` on the same function, not on the newtype
  method → cross-arch bench gate (see point 5).

### 4. Subdivide Slice 3 (ActiveEpochAnchor)

SPIRE 083's active-epoch chain runs through dozens of files
(coordinator/snapshots, hierarchy_snapshots, scan/relation,
fanout, write_payload, publish/relation, etc.). Designing a single
type that absorbs that chain in one slice is a major architectural
change.

**Revision:** subdivide:

- **Slice 3a:** type definition + smallest single user (e.g., one
  scan callback site).
- **Slice 3b:** coordinator/snapshots + hierarchy_snapshots
  rollout.
- **Slice 3c:** read-efficiency lane rollout (production read
  path).

Avoid a single packet that touches every SPIRE coordinator file.

claimed the bench-baseline packet didn't exist. **It does — the
captures Task 50 needs are mostly already in the repo.**

Existing AWS baselines (Graviton 4 / m8g.large / PG18.3 / gp3 EBS,
captured 2026-05-16):

- `benchmarks/cloud-10k-graviton-preopt-baselines/` — DBpedia
  10k+50k, ec_ivf TurboQuant + RaBitQ **kernel attribution**.
- `benchmarks/cloud-10k-real-baselines/` — DBpedia 10k+50k on
  m8g.large real data.
- `benchmarks/cloud-scaling-multi-am/` — **the canonical
  comparator**: scaling curve at 10k/50k/100k/1M across
  `ec_hnsw + ec_ivf (TurboQuant/RaBitQ/PqFastScan) + ec_diskann`.
  Explicitly captioned as "the full pre-optimization scaling
  curve."
- `benchmarks/comparators-50k-100k-1m/` — pgvector / pgvectorscale
  / vchord competitor latencies.

The planning packet's `bench-baseline-plan.md` proposes a fresh

| Slice | Existing comparator | New capture needed? |

| 4 heap source scorer (SPIRE side) | none | yes — folds into the Slice 3 SPIRE capture |

**Revision:** rewrite `bench-baseline-plan.md` to:

   comparators for slices 1, 2, 4-IVF, 5.
2. **Identify the SPIRE baseline gap explicitly** and propose
   either reusing Task 30 phase 13d evidence (if durable per
   NFR-007) or capturing a new SPIRE-specific cloud baseline

3. Confirm Graviton 4 / m8g.large remains the canonical AWS bench

### 6. Local vs. cloud bench policy

The planning packet doesn't say whether local benches are
required, optional, or required-with-asterisk. They should be one
of the three. Recommended policy:

- **Default (most slices):** cloud comparison against the
  existing baselines is the regression gate. Local benches are
  not required and are not cited in request.md.
- **Smoke gate:** local pre-commit bench is recommended for fast
  iteration during development — catch obvious 2× regressions
  before AWS round-trip. Not load-bearing evidence.
- **SIMD slices (Slice 8 / 9 candidates):** **both** local x86_64
  and cloud Graviton aarch64 measurement required. The cloud
  baseline exercises only the NEON path; AVX2/FMA codepaths can
  only be measured on x86. Reuse the existing `_for_test` entry
  points in `src/quant/hadamard.rs` and `src/quant/prod.rs`.

**Revision:** add a short "local smoke gate" subsection to
`bench-baseline-plan.md` capturing this policy.

Task 50's task doc explicitly excludes Task 41 (PG resource RAII),
Task 40 (concurrency lifts), Task 43 (proof coverage). The
planning packet doesn't say which existing wrappers each slice
*consumes* or what each slice must NOT step on.

**Revision:** add a one-line "Coordination" note to each slice:

- **Slice 1 (callback wrapper):** consumes `pgrx_extern_c_guard`
  directly; does not duplicate Task 41's relation guards.
- **Slice 2 (IVF page visitor):** consumes Task 41's
  `AccessShareIndexRelation` where applicable; does not duplicate.
- **Slice 3 (ActiveEpochAnchor):** must not step on Task 30 phase
  13d ongoing edits to `coordinator/snapshots.rs` —
  cross-check active branches before starting.
- **Slice 4 (scorer):** consumes existing
  `HeapRelationGuard` / `IndexRelationGuard`; does not duplicate.

### 8. `rg` install-check + Slice 1 callsite metric clarification

Two smaller polish items:

**8a.** `block-count-tooling.md` uses `rg --count-matches`. `rg` is
not installed by default on this dev host (I had to
`cargo install ripgrep` mid-Task-35). Add an install check
prerequisite to the Make target or a `grep -c` fallback. Don't
trust ripgrep's presence.

**8b.** `slice-sequence.md`'s Slice 1 "Expected first target
count" says "IVF callback guard sites: 20 current matches" — this
is file-counted, not callsite-counted. The 30% reduction target
needs to clarify whether it's at the **block-count granularity
the tool measures** or at the **callsite granularity the helper
absorbs**. The former is the canonical metric (per the task doc);
the latter is the human-meaningful metric. Recommend both: the
gate is block-count, the narrative cites callsites.
