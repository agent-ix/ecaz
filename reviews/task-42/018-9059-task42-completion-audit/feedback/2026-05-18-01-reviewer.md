# Review: Task 42 partial closeout audit (018)

## Summary

Records the Task 42 partial closeout audit and downgrades the
framing from "complete" to "partial smoke checkpoint" after
reviewer feedback. Maps every original Task 42 requirement to
concrete evidence (fixture file lists, test counts, CI run IDs).
Links the version-matrix framing to NFR-016-EV-3 and the WAL
framing to ADR-070 / Task 37.

The "partial smoke checkpoint" framing is **correct** and an
improvement over the original "complete" framing. The original
claim was overreach; the revised claim is honest.

## Confirmations

- **Status downgrade is the right call.** Original commit said
  "Mark Task 42 complete." The audit now says "partial smoke
  checkpoint." This matters because: pg_upgrade has only a
  4-row HNSW corpus with top-2 ID equality, qemu lane has not
  burned in over time, and Task 37 hasn't shipped custom WAL
  records yet. None of those are *failures* — they're explicitly
  conditional future work. Calling that "complete" would have
  obscured what's actually shipped vs. scaffolded.
- **Every prompt requirement has concrete evidence.** The
  Prompt-to-Artifact Checklist in
  `artifacts/completion-audit.md` is unusually well-structured:
  each row names a specific file or CI run ID, not a vague
  "covered." That's the audit format other task closeouts should
  copy.
- **The "Conditional Future Work" framing is the right structural
  choice.** Rather than carrying "remaining gaps" forward (which
  implies the task isn't done), the audit reclassifies them as
  *conditional* (activate when X happens). That cleanly separates
  "Task 42 didn't ship this" from "Task 42 correctly deferred this
  until a trigger event."
- **CI run IDs are cited.** `26003647665` / `76431237004` for
  the qemu lane. That's durable evidence — anyone can re-fetch
  the CI logs years from now and reproduce the green state.
- **NFR-016-EV-3 + ADR-070 cross-references.** The audit links the
  version-matrix work to a normative spec (NFR-016-EV-3) and the
  WAL framing to an ADR (ADR-070). That's the right level — Task
  42's deliverables are now tied to specs that outlive any
  individual task, not floating evidence.

## Findings

### 1. (verification) The reviewer focus question

> Does the audit correctly distinguish the current Task 42 smoke
> checkpoint from future conditional work that only activates when
> CI is steadier, new durable byte contracts ship, or new
> incompatible writable versions ship?

**Yes.** Each conditional row in the checklist names its trigger:

- pg_upgrade richer corpus: "satisfied only trivially until a
  richer corpus lands" → trigger is corpus availability.
- Version matrix: "Live upgrade rehearsal activates when a second
  writable version ships per NFR-016-EV-3" → trigger is a
  second version landing.
- WAL: "Per-record contracts activate when Task 37 lands custom
  WAL records" → trigger is Task 37 landing.
- CI burn-in: "extensive CI burn-in deferred until CI is steadier"
  → trigger is CI stabilization.

Each conditional has a named trigger event. That's exactly how
deferred work should be framed.

### 2. (verification) Is any current on-disk surface missing?

The review focus asks: "Is there any current on-disk surface
missing from the fixture/static/matrix evidence chain?"

Cross-referencing the audit's checklist against the Task 42 scope
in `plan/tasks/42-on-disk-format-invariants.md`:

| Scope item from plan | Audit coverage |
|---|---|
| `src/storage/page.rs` (generic page header / tuple layout) | ✓ (fixtures, layout assertions) |
| `src/am/ec_hnsw/page.rs` (HNSW page format) | ✓ (010 hot fixtures, 001 layout) |
| `src/am/ec_diskann/page.rs` (DiskANN metadata, V1-V3 versioning) | ✓ (002 layout, 011 overflow) |
| `src/am/ec_spire/storage/**` (SPIRE V2 / object / placement / epoch) | ✓ (003 layout, 007/008/009 fixtures) |
| **Codebook persistence (Lloyd-Max, OPQ rotation matrices)** | **NOT explicitly covered in audit** |
| Future WAL record encoding | ✓ (017 policy + Task 37 handoff) |

**Codebook persistence appears to be a gap.** The plan lists
"Codebook persistence (Lloyd-Max, OPQ rotation matrices)" as
scope, but the audit checklist doesn't have a row for it.

Possible explanations:
- Codebook persistence is covered under IVF (006) or SPIRE
  (007-009) fixtures and just not called out by that name. If
  so, the audit should rename one of those rows to make it
  explicit.
- Codebook persistence is genuinely deferred and should be added
  to the "Conditional Future Work" section with a trigger.
- Codebook persistence is not yet implemented — in which case it
  shouldn't be in the plan's scope.

**Suggested clarification:** add a row to the audit checklist for
codebook persistence with the actual coverage status (covered /
deferred / not implemented). Right now it's ambiguous.

### 3. The `pg_upgrade` corpus is meaningfully thin

> Narrow PG18 same-binary smoke: HNSW-only, four-row corpus, top-2
> ID equality. The original recall-floor criterion is satisfied
> only trivially until a richer corpus lands.

This is the most honest line in the audit. A 4-row corpus with
top-2 ID equality is a smoke test, not a recall validation. The
audit calls this out explicitly — good.

For future planning: "richer corpus" needs a specific target. Is
it `m5_diskann_real10k` (10k rows, available locally per
`fixtures/`)? Is it DBpedia OpenAI 100k? Without a target, "richer
corpus" can drift indefinitely.

**Suggested addition to "Conditional Future Work":** name the
target corpus and the recall floor it would establish (e.g.,
"pg_upgrade smoke on `m5_diskann_real10k` at recall@10 ≥ 0.95
post-upgrade").

### 4. (cross-reference) Audit format is reusable

The structure of this audit (Prompt-to-Artifact Checklist +
Conditional Future Work + cross-references to specs) is the
right template for other task closeouts. Task 41 closeout, Task
35 (when it nears completion), Task 39 (when its 003 plan
items are fully closed), and Task 47 (similar gate-coverage plan)
would all benefit from copying this format.

**Suggested project-level move:** add a `reviews/CLOSEOUT-AUDIT-TEMPLATE.md`
that captures this structure for reuse. Not blocking — but the
pattern is too good to leave as a one-off.
