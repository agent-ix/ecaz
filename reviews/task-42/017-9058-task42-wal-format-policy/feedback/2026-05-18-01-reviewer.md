# Review: Task 42 WAL format policy (017)

## Summary

Closes the Task 42 WAL version-tag item **without** inventing
custom WAL records that ECAZ does not currently emit. Documents
that current writes use `GenericXLog` page images, defines the
byte-0 version-tag convention for any future custom WAL records,
and ships a validator (`validate_custom_wal_record_format_version`)
that Task 37 will wire into custom redo/replay paths when those
land.

**Right call.** Don't pre-specify a wire format for records that
don't exist; do pre-specify the *header byte* convention so the
first custom record automatically has version-tag rejection
behavior. This is the inverse of the trap that bit packet 013
(pgrx coverage feasibility): rather than defer the work, do the
shippable subset and define the seam for the deferred part.

## Confirmations

- **`ECAZ_CUSTOM_WAL_RECORDS_ENABLED = false`** is honest. Today
  ECAZ emits only `GenericXLog` records (verified by reading
  `src/storage/wal.rs` — the `GenericXLogTxn` RAII wrapper from
  Task 41 emits `GenericXLogStart`/`GenericXLogFinish` pairs, no
  custom record types). Setting the constant to `false` until that
  changes is the right state.
- **Byte-0 version-tag convention is the right seam.** Every
  on-disk format invariant Task 42 covers uses a byte-0 (or
  near-zero) version tag — `index_format_version`, payload-flag
  bits, etc. WAL records following the same convention means Task
  37 inherits the same auditing/rejection pattern.
- **Validator function shipped.** `validate_custom_wal_record_format_version`
  is the entry point Task 37 needs to call when consuming a custom
  WAL payload. Task 37 doesn't have to design the rejection logic
  — it's already here, tested by `tests/wal_policy.rs`.
- **`tests/wal_policy.rs`: 2 tests pass.** Tests the policy state
  (`false` today) and the reject path (validator rejects missing/
  unknown version). Right minimal test surface.
- **`docs/on-disk-format.md` records the policy** (15-line
  addition; not a per-packet log — this is the inventory's WAL
  section, which belongs there).

## Findings

### 1. (cross-reference) The `LockBuffer` symbol issue is the same class as packet 013's `_BufferBlocks`

The request notes:
> `cargo test --lib storage::wal` was not used as final validation
> because the lib-test binary loads pgrx callback symbols and fails
> at process startup with the existing unresolved `LockBuffer`
> symbol.

This is the **same loader-level macOS issue** that packet
`reviews/task-39/013-pgrx-coverage-feasibility/` documented:
pgrx lib-test binaries can't resolve Postgres backend symbols when
loaded standalone. The 013 packet's "not feasible" framing was
reversed in my revised feedback — setup issues are not deferral
grounds.

Same rule applies here. **However:** for this specific packet
(017), the integration-test workaround (`cargo test --features
bench --test wal_policy`) is a legitimate fix, not a deferral.
The test runs through the integration-test harness which bypasses
the lib-test loader issue. So the policy *is* tested, just not
through the lib-test path.

**No reversal needed on this packet.** But worth flagging that the
same loader issue blocks two different packets across two
different tasks — this is a project-wide setup gap that someone
should own. Most natural owner: Task 39 if it's about coverage
instrumentation, or a new "developer environment hardening"
packet if it's about the dev cluster setup more broadly.

Suggested follow-up: a packet that picks one of the dyld
workarounds (link with `-undefined dynamic_lookup`, or run pgrx
tests via `--runas postgres`, or `DYLD_FORCE_FLAT_NAMESPACE=0`)
and ratifies it across both lib-tests and instrumented coverage.
Not blocking 017.

### 2. (verification) WAL policy aligns with current production

Reading `src/storage/wal.rs` against the policy:

- Current production code uses `GenericXLogTxn` (a Task 41 RAII
  wrapper) which encapsulates `GenericXLogStart` → `GenericXLogRegisterBuffer`
  → `GenericXLogFinish`. No custom record types are emitted.
- The validator function is gated to apply only when
  `ECAZ_CUSTOM_WAL_RECORDS_ENABLED == true`. Today it's `false`,
  so the validator is a no-op in production.

Policy and reality match. Right state.

### 3. (verification) ADR-070 alignment

The closeout audit (packet 018) references ADR-070 and Task 37
for the WAL framing. Reading the packet text against that
framing:

- ADR-070 (per the task-49/013 packet title) is about "On-Disk
  Format Evolution Discipline."
- This packet's WAL policy is consistent with that: define the
  contract today, defer the implementation to Task 37 when
  custom records ship.

The cross-reference is correct. If/when ADR-070 specifies WAL
header conventions explicitly, this packet's byte-0 convention
should match — worth verifying when both are landed.

### 4. (suggestion) Future-proof the validator

The validator currently takes a version-byte and returns
`Result<(), String>`. For future Task 37 callers, consider:
- A typed error enum (`enum WalFormatError { Missing, Unknown(u8), Unsupported(u8) }`)
  rather than `String`, since callers may want to discriminate
  these cases for telemetry.
- A length-aware variant that takes the full record bytes and
  validates header structure end-to-end, not just the version
  byte. The byte-0 convention is right; a `validate_custom_wal_record_header`
  that *parses* the header (version + length + checksum if any)
  would be more useful to Task 37.

Not blocking — Task 37 can extend the API when it consumes it.
Worth a note in `docs/on-disk-format.md` so Task 37's author sees
the design space.

### 5. (cross-reference) WAL is the last on-disk surface

After this packet, Task 42 has covered:
- Page format (storage/page.rs) — 010 fixtures
- AM metadata (HNSW/DiskANN/SPIRE) — 001, 002, 003 layout contracts +
  004-009 fixtures
- IVF (006), SPIRE V2 chain (009), HNSW hot tuples (010), DiskANN
  overflow (011)
- Upgrade matrix (012)
- CI lanes (013) + qemu (014-015)
- PG upgrade (016)
- **WAL (this packet) — last surface**

The closeout audit (018) correctly identifies WAL as a
"scaffolded" surface rather than fully complete — and that's the
right framing because no custom records exist to test yet.
