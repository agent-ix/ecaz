Finding:

1. `sc_003b_rerank_visits_in_heap_tid_order` does not fully lock in the
   property that `e191a9e1` relies on. The production code sorts rerank
   candidates by full `(block_number, offset_number)` before heap fetch
   in `src/am/ec_diskann/scan.rs`, but the new regression only records
   and compares `block_number`. A future regression that preserved
   block order while scrambling tuple offsets within the same page would
   still pass the test. Tighten the assertion to compare full heap TIDs,
   and preferably use a fixture that places multiple candidates on the
   same block so the test can actually fail on an intra-page-ordering
   bug.
