## Finding addressed

**1. `sc_003b` does not lock the offset_number tie-breaker.** Fixed
in commit `64645bfc` (`Tighten sc_003b to exercise full heap-TID
tie-breaker`).

The rewritten test:

- Uses a hand-built fixture with three blocks and two candidates
  per block (`(1002,7)`, `(1000,5)`, `(1001,1)`, `(1000,2)`,
  `(1002,3)`, `(1001,9)`).
- Drives prefilter scores from a `HashMap<ItemPointer, f32>` chosen
  so the natural score-ordered traversal lines up the candidates
  in the reverse of the expected heap-TID order on every block.
  The production sort therefore has to do real work, and a
  `block_number`-only sort would leave each block's insertion
  order intact (i.e. `(1002,7)` before `(1002,3)`), failing the
  assertion.
- Compares full `ItemPointer` values, not just `block_number`.
- Adds a fixture-shape sanity assertion (`unique_blocks.len() <
  observed.len()`) so a future fixture refactor that accidentally
  collapsed the same-block coverage would fail loudly rather than
  pass silently.

Renamed to `sc_003b_rerank_visits_in_full_heap_tid_order` to make
the property explicit in the test name. 29 tests pass under
`cargo test --no-default-features --features pg18 --lib am::ec_diskann::scan`.

## Suggestions for a future round — acknowledged, not in scope here

These are recorded for the next round; not addressed in this
turn since they are scope expansions, not corrections.

1. **Build-time NEON impact.** `source_inner_product` is also
   called from `ambuild::source_inner_product_distance`, which
   the diskann build calls during medoid + graph construction.
   Build does many more exact dot products than rerank, so the
   NEON kernel may be a larger M-series win during build than at
   query time. A future round can record build elapsed for
   `m5_diskann_real10k` and `m5_diskann_real10k_w800` under the
   scalar (`origin/main`) and NEON (`dceda05`) binaries; the
   build-elapsed lines are already in the existing load logs
   (`load-diskann-real10k.log` reports `built ... in 32.61s`).
2. **Same-page run grouping.** Scoping a future slice that, after
   the heap-TID sort, walks the rerank set in same-block runs
   and consumes them while the page is still pinned, instead of
   issuing a separate `fetch_heap_row_version` per row. That is
   a more promising query-time locality lever than the reverted
   prefetch and is a narrower change than restructuring the
   rerank loop around an open read stream.
3. **Cold-cache prefetch revisit.** Already explicitly listed in
   `30207`'s deferred follow-ons; flagging here only to confirm
   we are aligned that warm-cache numbers are not the right
   benchmark for that hypothesis.
