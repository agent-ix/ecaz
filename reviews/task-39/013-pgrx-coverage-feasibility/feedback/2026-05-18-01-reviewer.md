## Findings

### 1. The `_BufferBlocks` symptom is a Postgres-extension loader issue

`_BufferBlocks` is a Postgres backend symbol (the shared buffer
pool entry point). The pgrx test harness builds the extension as a
dylib that expects backend symbols to be resolved at runtime by the
postmaster. The lib test binary tries to load the extension
standalone, which is why the loader can't find `_BufferBlocks`.

This is a known class of issue for `cargo pgrx test` on macOS with
`flat_namespace` linking. Workarounds documented in the wider
ecosystem:

- Build with `-undefined dynamic_lookup` (defer symbol resolution
  to runtime).
- Use `cargo pgrx test --runas <user>` inside a real postmaster
  instead of the lib-test loader.
- Skip lib tests and run instrumented `pgrx-pg-sys` against a real
  PG instance via cargo-test with explicit `LD_PRELOAD`/`DYLD_*`.

**None of these need to be in this packet** — it's a feasibility
*decision*, not a fix. But if/when a future packet tackles this,
those are the leads.
