# Review: Task 39 pgrx coverage feasibility (013) — REVISED

**Supersedes `2026-05-18-01-reviewer.md` on the deferral question.**

## Reversed decision

The previous feedback accepted "not feasible on the current macOS PG18
setup" as a valid Task 39 outcome and treated items 14-16 as
formally deferred. **That is incorrect.** Setup issues are not grounds
for deferring coverage work — they are work items themselves.

The two blockers documented in this packet are both **fixable**:

1. **`dyld` cannot resolve `_BufferBlocks`** when the pgrx lib-test
   binary tries to load standalone. This is a macOS flat-namespace
   linker issue, not a fundamental coverage-instrumentation problem.
   Standard fixes:
   - Link with `-undefined dynamic_lookup` so backend symbols
     resolve at runtime (the postmaster provides them)
   - Run instrumented tests through `cargo pgrx test --runas
     postgres` against a real backend rather than the standalone
     lib-test loader
   - Use `DYLD_INSERT_LIBRARIES` / `DYLD_FORCE_FLAT_NAMESPACE=0` to
     bring backend symbols into the loader's search path
2. **`LLVM_PROFILE_FILE` must be absolute.** Already known — just
   set it absolute in the lane wrapper. One-line fix.

Neither blocker is a "this can't work" — both are routine extension
setup for instrumented pgrx tests on macOS.

Create a follow-up packet (next ordinal after 014) that actually

## Why this matters

Allowing "setup issues = deferral" sets the wrong precedent for the
whole hardening campaign. If the bar for shipping a coverage row is
"works on the dev's current laptop," gaps compound silently. The
Task 39 plan called for live PG18 callback coverage as a concrete
deliverable — that deliverable stays open until callback files have
real numbers.

The previous "honest gap" framing was wrong in this case: gaps are
honest only when there is no path to close them. Here there is a
path; it just hasn't been taken yet.
