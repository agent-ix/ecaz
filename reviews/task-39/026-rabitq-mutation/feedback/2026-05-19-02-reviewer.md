## Findings

### 1. Checkpoint vs closeout — the 0-missed discipline

Prior mutation packets (014, 015, 016, 017, 018) all closed at
**0 missed**. This packet files at 9 missed + 2 timeouts. The
work is substantial — accepting the checkpoint framing makes
sense given the file's size and surface — but **the 0-missed
discipline shouldn't drift into "checkpoint is good enough."**

**Required:** a follow-up packet that closes the 9 + 2 residuals
before RaBitQ mutation is declared done. The triage already
specifies what each test needs:

- `encode_code:409` / `encode_code:429`: zero-vector code-tail
  assertions (zero-norm + zero `o_dot` branch boundaries)
- `encode_code_centered:585`: asymmetric-center accumulator
  assertion
- `encode_code_centered:588`: zero-residual centered-code tail
- `CenteredScorer::score_at:728`: non-unit query vector
  assertion
- `CenteredScorer::score_at:750`: `o_dot` floor equality boundary
- `CenteredScorer::score_at:752` (×2 mutants): query-residual
  floor with below/equal/above coverage
- `quantize_level:993`: scale-term non-roundtrip expected-value
  assertion
- `estimate_ip_impl:1127` (×2 timeouts): extract smaller helper
  for the candidate `o_dot` floor check, OR bump cargo-mutants
  timeout for these mutants specifically

That's 9 + 2 targeted tests, each with a named acceptance
criterion. Should be one tight follow-up packet.

### 2. The 2 timeouts are the same pattern as packet 017

Packet 017 (HNSW mutation) hit timeout-class mutants on a test
helper that searched for a max-fitting boundary in a loop; the
fix was rewriting the test to assert against a fixed expected
value (no loop to mutate, no timeout).

`estimate_ip_impl:1127`'s `<` operator likely sits inside a
similar loop or iterates over a range. Extracting the boundary
check into a small helper that asserts directly against expected
edges (below / equal / above) should eliminate the timeout class
without bumping the global mutation timeout.

This is the same lesson, second application. Worth documenting
in the follow-up packet's triage that this pattern recurs.

### 3. `--in-place` workaround is fine but worth a note

> `--in-place` was used because this checkout has large unrelated
> untracked fixture directories; scratch-copy mutation would copy
> those into each mutant workspace.

Accepted as a workaround. **Cleanup recommendation, not
blocking:** the untracked fixture directories
(`fixtures/m5_diskann_real10k/`, `m5_diskann_real100k/`,
`m5_diskann_synth10k/`, `artifacts/`) are large and may be
breaking other workflows. A `.gitignore` rule + a `make
mutants-clean` that excludes them would let scratch-copy mutation
work again. Not in scope for this packet.
