## Findings

### 1. `quant/simd.rs` gap (48% vs target 70%)

The packet acknowledges this. Worth understanding why and
whether it's expected:

**Likely cause:** `simd.rs` contains runtime-dispatched
intrinsics (AVX2, AVX-512, NEON). Even on a fast host, only
the dispatch path matching the running CPU's features
executes. So:
- On x86_64 with AVX2 but not AVX-512: NEON paths + AVX-512
  paths uncovered
- On ARM with NEON: AVX2 + AVX-512 paths uncovered
- No single host can cover all paths

**Options to address:**
1. Use `target_feature` test attributes to force-enable each
   path under cross-compilation
2. Accept the per-arch ceiling and document expected coverage
   per CPU class (e.g., "60% on x86_64-avx2, 70% on x86_64-avx512,
   55% on aarch64-neon")
3. Defer `simd.rs` to Task 36 (SIMD scalar differential) which
   has full per-path coverage as its primary goal

Recommend option 3 — Task 36 IS the per-SIMD-path coverage
mechanism. Task 39 coverage tooling can't easily exercise
inactive code paths; Task 36's differential harness can. The
TSV entry for simd.rs should annotate this with "Task 36
provides per-path coverage; Task 39 measures the union from
the host CPU."

Tracked as a refinement to item 5 in the 003 plan + adds
cross-coordination note to Task 36.

### 2. `quant/mod.rs` at 0.00%

Module file (`mod.rs`) typically only contains `pub use`
re-exports and module declarations — there's no executable
code to cover. Coverage reports `0.00%` because there are no
"reachable" lines. This isn't a bug, just a measurement
artifact.

**Recommend documenting:** add a note in the TSV row or
`docs/hardening.md` that `mod.rs` files are expected to
remain at 0% (no-op coverage). Otherwise a future reader sees
"quant/mod.rs 0.00%" alongside "quant/prod.rs 93.02%" and
wonders why one module is uncovered.

### 3. `storage/page.rs` at 76.57% (close to 80% target)

3.43pp under the 80% target. Looking at the value, it's
probably the AM-specific tuple-kind paths (each AM has tuple
kinds that page.rs dispatches on) that aren't all exercised
by the careful harness. Item 11 in my 003 plan covers SPIRE
storage helpers; an extended careful harness with all AM
tuple kinds should push this to 80%+ naturally.

Not a blocker; mark as a soft target in the next packet.

### 4. (verification) Merged coverage total of 4.51% looks
correct

The merged `summary.txt` reports 4.51% total line coverage.
That's the union of careful (~60-90% on critical files) +
root harness (~0% on AM/SPIRE/coordinator) weighted by file
size.

The bulk of `src/` line count is in `src/am/*` (3000+ entries
per the unsafe baseline survey), of which careful covers
very little. As pgrx coverage lands (item 13 in 003 plan),
this total will jump substantially. Don't read 4.51% as a
final state — it's the current pure-Rust ceiling.

### 5. (suggestion) Ratchet mode CI gate

`--ratchet` mode exists, but the packet doesn't say whether
it's wired into the `test-quality-coverage` GHA job. Per the
003 plan item 17:

> CI runs `--ratchet` only on commits to `main`, not PRs

If the GHA job currently runs delta-check on every PR (no
ratchet) but doesn't have a separate main-only ratchet step,
the baseline won't auto-raise even when coverage improves.

Worth verifying:
- PR job: `check_coverage_delta.sh` without `--ratchet` (fails
  if drop > 2pp)
- Main-push job: `check_coverage_delta.sh --ratchet` (rewrites
  TSV when actual > baseline + 2pp)

If only the PR job is wired, the next packet should add the
main-push ratchet step. Otherwise baseline stays at this
packet's values forever even as future work raises actual
coverage.
