## Findings

### 1. The 600-line `page.rs` diff is mostly test code

`src/am/ec_hnsw/page.rs` grew by ~580 lines net. Skimming the
file structure that emerges from the triage:

- Test additions for metadata page/content minimum length and

  boundary tests

  tuple wrappers
- Exact-minimum current metadata page decode assertion (kills
  the last survivor from rerun-2)

That's the right place for these tests — `#[cfg(test)] mod tests {
... }` inline keeps them with the codec they're testing, and the
careful harness already imports the file. **Not in scope for this
review**, but the file is now approaching 2,640 lines (was 2,040
when packet 012 landed) and may eventually be worth splitting:
`page.rs` for the codec, `page_tests.rs` (or a `#[path = ...]`
external module) for the tests. Not urgent.

### 2. Timeout-class fix in rerun-1 is well-handled

> The timeout mutants were caused by a test helper that searched
> for the maximum fitting element tuple length with a loop. The
> test was rewritten to assert the fixed default-page boundary
> directly.

This is the right call. A test helper that loops to find a
boundary value is mutation-vulnerable on *the loop* — the
mutation harness times out the loop because the mutated arithmetic
keeps it from terminating. Rewriting the test to assert against a
fixed, computed expected value:
- Eliminates the timeout class (no loop to mutate)
- Makes the test's intent explicit (the expected boundary is
  named, not searched for)
- Tightens the contract — if the boundary changes, the test
  fails loudly with a concrete expected vs. actual

Worth flagging as a **mutation-testing best practice**: test
helpers should compute expected values from invariants, not
search for them. The 6 timeouts in rerun-1 disappeared because
the loop did. Pattern is worth noting for ec_ivf and ec_spire
mutation packets when they land.

### 3. Literal-constant rewrites: same structural-equivalence trick as 016

Packet 016 rewrote `1 << 0` → `0b0000_0001` to retire a
shift-by-zero equivalent mutant. This packet generalizes the
pattern:

- "Derived HNSW metadata byte constants" → literals (the triage
  doesn't name them individually, but the pattern is consistent
  with 016)
- Binary-sidecar flag expression → literal

The trade-off, just like 016: literal constants are less
*derivable* (a future reader can't see "this is `sizeof(field_a)
+ sizeof(field_b)`") but are *mutation-stable* and have to be
updated explicitly when the format changes. For on-disk wire
constants, the mutation-stable + explicit-update properties win
— the on-disk format is exactly the kind of thing that should
not silently follow type-size changes.

**One follow-up to consider:** for each rewritten constant,
adding a sibling `const_assert!` (or `debug_assert!` in a test)
that the literal matches the previously-derived expression would
preserve the "this is what it represents" documentation while
keeping the literal mutation-stable. Not blocking. Just a thought
if the literal becomes hard to audit later.

### 4. (verification) Review focus answered

The review notes ask two things:

> Whether the literal byte-size constants are preferable to the
> previous derived expressions for mutation gating

**Yes**, for the reason above: derived expressions in on-disk
constants invite silent format drift when a contributing type
size changes. Literals require explicit format updates, which is
what we want for a wire contract.

> Whether the added layout/boundary assertions pin the intended
> HNSW wire layout without overfitting implementation details

Mostly yes — the new assertions land on:
- Metadata payload sizes (wire-visible)
- Tuple field offsets (wire-visible)
- Payload flag bit positions (wire-visible)
- Min/max boundary cases for tuple kinds (decoder behavior, not
  representation)

The boundary tests are about decoder *behavior at edges*, not
internal data structure layout, so they're not overfitting. The
risk would be tests that assert on internal `Vec` capacity or
similar implementation details — none of those appear in the
triage classes.
