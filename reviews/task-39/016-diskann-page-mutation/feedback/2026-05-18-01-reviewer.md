## Findings

### 1. The `1 << 0` → `0b0000_0001` rewrite is the right call

The triage correctly identifies `1 << 0` as an **equivalent
mutant** target: `1 << 0` and `1 >> 0` both evaluate to `1`, so no
test can distinguish them at runtime. The two options were:

- (a) Mark it `equivalent` in the triage and move on.
- (b) Rewrite the source to a literal that has no `<<` operator,
  so the mutation operator has no expression to attack.

The packet picks (b), which is the better call **for a bit-flag
constant** because:
- Literal `0b0000_0001` is more readable than `1 << 0` (the shift
  was always vestigial here)
- The mutation operator can't generate a `>>` mutant against a
  bit-literal — the equivalence is structurally impossible, not
  just behaviorally identical
- Future mutation runs won't need to re-classify the equivalence

For non-trivial shifts (e.g., `1 << 3`), the calculus is different
— there the shift expresses bit position and shouldn't be flattened
to a literal. So this is the right call specifically because the
shift was by zero.

### 2. The other bit-flag (cold_rerank at bit 2) got an assertion

`PAYLOAD_FLAG_COLD_RERANK_PAYLOAD == 0b0000_0100` is the killing
assertion in `la_005b_empty_clears_cold_rerank_flag`. That test
also serves as the wire-bit contract for the cold-rerank flag.
**Worth noting:** this assertion ties the constant value to the
on-disk format. Any future code change that "improves" the
constant (say, switching to a different bit position) will trip
this test and surface the on-disk-format break before it ships.
That's the right behavior — wire bits in metadata are part of the
format contract.
