# Review: Task 39 leaf-V2 meta validate() coverage (029)

## Summary

Adds direct coverage for 9 `SpireLeafPartitionObjectV2Meta::validate()`
error branches in `src/am/ec_spire/storage/leaf_v2_parts.rs`:
zero published-epoch-backref, zero object-bytes-total, LocalU64
wrong stride, GlobalBytes stride below min / above max, non-empty
meta with zero segment count / INVALID locator / NONE payload
format / zero payload stride.

Each sub-assert twists exactly one field on a known-good baseline
(with a sanity `is_ok()` check that pins the baseline itself).

## Confirmations

- **Each error path now has direct coverage**, not just indirect
  via the success path.
- **Isolated-field-twist pattern is good**: ensures each
  assertion targets one specific guard, not aggregate behavior.
- **No `docs/hardening.md` edits.**
- Coverage delta tracked in 032's baseline ratchet.

## Findings

### The "one-twist-on-known-good" pattern is reusable

Most validate-style guards in this codebase have similar shape:
many independent field checks against a single struct. The
pattern this packet uses — define a `valid_meta()` baseline,
sanity-check it, then assert error on each twisted copy — is
the right template for any future packet covering similar
validate functions.

Worth flagging for reuse in any other AM that has a
multi-field validate function (DiskANN metadata, SPIRE meta
states, etc.). Same shape, different fields.

## Disposition

Accepted. Tight, clean, real correctness coverage. After 032's
ratchet, `leaf_v2_parts.rs` crosses the 80% floor (77.52 →
81.03).
