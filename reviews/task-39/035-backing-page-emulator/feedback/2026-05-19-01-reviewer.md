## Findings

### 1. The emulator is the right Phase-1 shape

Reviewer direction asks for confirmation. **Yes** ÔÇö exactly the
Phase-1 shape from 033's review:

- Per-`Buffer` `Box<[u8; 8192]>` raw bytes Ôťô
- `Vec<Box<ItemIdData>>` for stable line-pointer heap addresses Ôťô
- Real `PageInit` / `PageAddItemExtended` / `PageGetItem` /
  `PageGetItemId` / `PageGetMaxOffsetNumber` / `PageGetFreeSpace`
  / `PageGetSpecialPointer` / `PageGetSpecialSize` /
  `PageIndexTupleDeleteNoCompact` Ôťô
- `ReadBufferExtended` with `P_NEW + RBM_ZERO_AND_LOCK` allocation
  path Ôťô
- Registry indexed by `(rd_id, block_number)` with reverse `*mut
  u8 Ôćĺ *mut BackingPage` lookup Ôťô
- `RelationGetNumberOfBlocksInFork` sourced from registry Ôťô
- `reset_buffer_registry` wired into `reset_counters` Ôťô

The stable heap-address requirement (`Box<ItemIdData>`, not
`Vec<ItemIdData>`) is the load-bearing detail ÔÇö `PageGetItemId`
returns a `*mut ItemIdData` that the production code dereferences
later, so the address must survive vector resizes. Right call.

### 4. Test serialization note

The emulator uses thread-local registries plus a global
`TEST_LOCK`. Each test calls `reset_counters` (which now also
resets the buffer registry). That's the right serialization
approach ÔÇö parallel tests across threads on a global registry
would race; one global serialization point per test is the
simplest correct contract.
