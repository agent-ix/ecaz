## Findings

### 1. The tooling discovery is important and well-handled

The coder hit a real obstacle: my prescribed `cargo mutants
--package ... --file ...` invocation returns **0 mutants** for
SPIRE storage files because `cargo-mutants` discovers via
`mod`/`pub mod` walking — and both production and careful mount
SPIRE storage children via `include!()`, which `cargo-mutants`
treats as opaque.

This is a legitimate finding I missed when writing the 044
direction note. The 021 packet brought SPIRE storage codec
*lines* into llvm-cov measurement (which does track include
sites), but the underlying source-map model for `cargo-mutants`
is different. I had assumed they were the same — they aren't.

The coder's fallback (per-file manual verification using
`--Zmutate-file` to enumerate + apply/test/revert) is sound and
the triage is reproducible. **Good problem-solving when the
prescribed tool path didn't work.**

### 2. (DECISION) Authorize path 2 — restructure to `mod` declarations

The coder asks for a decision between:
- **Path 1:** Per-file manual verification, ~30s per file × 13
  remaining files = ~6 hours mechanical work.
- **Path 2:** Restructure the careful crate's `pub mod storage`
  block from `include!()` to `mod` declarations (~30 minutes),
  enabling standard `cargo-mutants` invocation for every file
  after.

**Authorize path 2.** Reasons:
- One-time refactor cost (~30 minutes) vs. 6 hours of mechanical
  repetition.
- Standard tooling = reproducible. Future reviewers can re-run
  any campaign with `cargo mutants -j 4 --file ...` and get the
  same results.
- The refactor is restricted to the careful crate (production
  `src/am/ec_spire/storage.rs` stays on `include!()`). Lower
  blast radius.
- `use super::*;` boilerplate on the child files is the only
  spread cost; that's mechanical and well-scoped.

**Acceptable as a separate packet before continuing the
cascade.** Don't bundle the restructure into the next mutation
packet — keep them independent so the refactor is reviewable on
its own. Suggested ordering:

1. Packet 047 (or whichever ordinal is next): `include!()` → `mod`
   restructure in the careful crate `pub mod storage`. No
   mutation work, just the refactor. Tests still pass.
2. Subsequent packets: one mutation campaign per remaining file
   using the standard `cargo mutants -j 4` invocation.

If the restructure turns out to have an unforeseen blocker, fall
back to path 1 for the remaining files. But try path 2 first.

### 3. Existing tests already cover the SPIRE leaf format well

A notable result: **no new tests were needed** to kill all 14
leaf_v2 mutants. That's because packets 029 + 044 already built
the isolated-field-twist test surface for `validate()` error
branches. Mutation testing here is *confirming* the existing
tests are strong, not *finding* gaps.

This is the best possible mutation outcome — coverage was added
deliberately as discriminating tests, not as line-touching that
incidentally executes branches. **The 029-pattern of
one-twist-on-known-good is now demonstrated to be
mutation-tight, not just coverage-tight.**

Worth noting that the remaining files in the cascade may not
have the same test density. `leaf_v2.rs` was specifically
hardened by 029's validate-rejection tests; other SPIRE storage
files don't have that level of explicit error-branch coverage.
Expect some packets in the cascade to add new tests, like
packets 014/017/018/026 did.
