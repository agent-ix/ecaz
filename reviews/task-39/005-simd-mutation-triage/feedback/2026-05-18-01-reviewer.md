## Findings

### 1. Boolean-gate extraction is the right ARM-coverage
pattern — worth documenting

The `e1b8a77a` / `f82589c1` pattern (extract per-arch
boolean gates into named, testable functions) is the general
technique for getting non-zero ARM coverage on x86-only code
paths. It composes well:

- x86 path: gate function returns `true`, body executes (real
  coverage)
- ARM path: gate function returns `false`, body skipped, but
  the gate function itself was called and tested (gate
  coverage)
- Mutation on gate: catchable on ANY host

Worth a note in `docs/hardening.md` or `docs/test-quality-strategy`
(once that lands as canonical doc) — pattern called something
like "per-arch decision points extracted for testability."
Future SIMD additions can follow the same shape from day one
rather than retrofitting.

### 2. Mutation triage table is the precedent

`triage.md` contains 9 mutant rows with verdicts (per
katexec/triaged-with-followups status). Right shape for
discipline item #19 from my 003 plan. The pattern:
- Mutant identifier (file + line + transformation)
- Outcome (caught / unviable / missed)
- Killer test or rationale for unviable

When future mutation packets land (item 19 cont.), they should
follow this packet's triage table structure.

### 3. `simd.rs` baseline raise to high value

Per `fixtures/quality/coverage-baseline.tsv`, `simd.rs` is now
at its 95.18% measurement. With 2pp tolerance, the floor is
~93%. That's defensible:
- A real regression dropping coverage below 93% requires
  packet justification (good — forces deliberate code
  changes through review)
- Normal SIMD additions that extend the file shouldn't drop
  coverage below 93% unless they add intrinsic bodies that
  can't run on the test host (in which case, per Finding 1,
  extract the boolean gate)

### 5. Mutation-run cadence question

`9 mutants in 19s` for `simd.rs` is fast — well within an
on-PR budget. But the Task 39 task doc and CI wiring put
mutation testing on a **weekly** cadence. With this packet
proving sub-minute mutation runs for small modules, worth
considering whether **per-module mutation could be a PR
gate** for the most-leveraged modules (prod.rs, page.rs)
rather than only weekly full sweeps.

Tradeoff: full prod.rs mutation might be much slower
(it has many more lines than simd.rs); a budget probe would
tell. Not actionable here; flagging for future sequencing.
