## Findings

### 1. Equivalent-mutant verdict on `|→^` at line 418:60 is correct

The mutated expression is `DELTA_INSERT ^ DELTA_DELETE` vs the
original `DELTA_INSERT | DELTA_DELETE`. The two flags are
**disjoint single-bit constants** (`0x0008` and `0x0010` per the
packet). For any pair of disjoint single-bit values:

```
a | b = a ^ b   (because a & b == 0)
```

So `0x0008 | 0x0010 == 0x0008 ^ 0x0010 == 0x0018`. No input can
distinguish the two expressions. Equivalent verdict accepted.

If the codebase wanted to retire this mutant *structurally* (the
same play as packets 016/017/027 used to retire equivalent
arithmetic mutants), the move would be `DELTA_FLAGS_MASK: u16 =
0x0018` and use the literal. But that loses the readability of
the named flags — for bit-flag masks, keeping the `|` form is
the right call. **Accept as equivalent, no source change.**

### 2. (decision closure) Continuing with manual verification + script is fine

In packet 046's review I authorized the `include!` → `mod`
restructure (path 2). The coder chose path 1 + automation —
generalized `run-leaf-v2-parts-mutations.py` from the 046 script,
~7 min wall time per packet. **That's a valid trade.**

The script-driven automation closes the speed gap that motivated
path 2. Standard `cargo mutants -j 4` would have been mildly
nicer for reproducibility (anyone reruns with one command), but
the script + enumeration file + verification log makes the
campaign equally reproducible.

**Decision: continue with the per-file manual + helper-script
methodology. The `include!` → `mod` restructure is no longer
required.** The generic `run-spire-mutations.py` from packet 048
is the canonical helper for the remaining cascade.

### 3. Existing test surface continues to discriminate well

After packets 029 + 044 hardened the SPIRE storage codec test
surface, mutation campaigns are catching most or all mutants
without new tests:

- Packet 046 (leaf_v2): 14/14 with no new tests.

That's a strong signal that the one-twist-on-known-good pattern
