## Findings

### 1. (verification) Coverage scope description accurately
limited

The Task 39 task doc and the packet both correctly frame this
initial coverage scope as **local pure-Rust coverage**, not
pgrx callback coverage. `cargo-llvm-cov` instrumentation
doesn't reach across the pgrx FFI boundary in any meaningful
way for `cargo pgrx test pg18`-style runs without further
plumbing. The packet acknowledges this explicitly in
`docs/hardening.md` (per packet 002's preview); the first lane
is the `ecaz-cli` + `hardening/careful` subset which is
genuinely pure-Rust runnable.

That avoids the failure mode where someone reads "coverage
86%" and assumes it covers AM callback execution. Good
discipline.

### 2. (verification) Mutation target list is appropriately
focused

The Task 39 task doc names "critical correctness modules" as
the scope: `src/quant/prod.rs`, `src/am/`, `src/storage/`.
First mutation target list per the packet is `src/quant/prod.rs`
(via `MUTANTS_MODULE` env var). That's the highest-leverage
starting point because:
- It's the lowest-level encoding/decoding code that everything
  else builds on
- Bugs there silently corrupt every AM that uses ProdQuantizer
- It's pure Rust (no pgrx fixture needed) so mutation runs
  quickly

The `MUTANTS_MODULE` parameterization is the right abstraction
— it lets follow-up slices add modules without rewriting the
target.

### 3. (suggestion) Make `mutants` cadence explicit in the doc

The `docs/hardening.md` section names coverage but the mutation
cadence (nightly? on-demand? PR?) isn't explicit. Mutation
testing is expensive (per-mutant test-suite reruns) and
shouldn't be a PR gate yet. Worth a one-line note pinning
"mutation: nightly / on-demand, not PR" so a future reader
doesn't wire it into `ci-quick`.

### 4. (suggestion) Coverage delta gate placement

Task 49 governance puts PR gates under `ci-quick` with <5min
budget. `make coverage` itself is probably too slow for that
(full coverage runs of `ecaz-cli` + `hardening/careful` take
minutes), but the **delta gate** mentioned in packet 002's
preview (`scripts/check_coverage_delta.sh` per the stashed
work I saw on main) is a different shape: it just diffs
against a baseline. The delta gate could fit `ci-quick` once
the baseline is committed. Worth sequencing the burn-in
explicitly:
- 39/001: lanes shipped (this packet) — not in `ci-quick`
- 39/002: baseline + delta gate — eligible for `ci-quick` once
  baseline is committed
- 39/003+: cadence promotion after burn-in window

### 5. (out-of-scope but relevant) Coverage of newly RAII'd code

Task 41 just landed ~12 RAII guard primitives in `src/storage/*_guard.rs`.
Worth flagging in the Task 39 follow-up that those modules
are the highest-priority new coverage target — they're small,
critical (drop semantics are load-bearing for the whole Task 41
correctness story), and the kind of code where mutation testing
would actually catch interesting bugs (e.g., flipping the
`UnlockReleaseBuffer` to `ReleaseBuffer` and verifying a test
notices the lock leak).

This isn't actionable in 001 — it's a sequencing note for
when mutation target list expands.
