## Findings

### 1. (substantive) Baseline is 0.00% on every critical module

The TSV pins **every critical module at 0.00% line coverage**:
- All `quant/*` (codebook, grouped_pq, hadamard, mse, prod,
  qjl, rabitq, rotation, simd)
- `storage/page.rs`
- All `am/*/page.rs` (diskann, hnsw, ivf, spire)
- All SPIRE storage helpers

The packet's key results section confirms this: total line
coverage 0.09%, critical modules 0.00%. **This is honest
reporting, not a measurement bug** — the pure-Rust lane
(`ecaz-cli` + `hardening/careful`) genuinely doesn't exercise
these modules at all. The 0.09% total comes from the small
slice of `ecaz-cli` Rust code that does run.

But it means the **delta gate is currently a no-op** for
critical modules. A baseline of 0.00 with a 2pp tolerance
allows actual values in `[-2.00, 0.00]` which clamps to 0.00
since coverage can't go negative. So:

- The gate can't catch regressions (there's nothing to
  regress)
- The gate can't ratchet coverage upward (no mechanism to
  raise baselines as coverage lands)
- The gate currently enforces "stay at 0% or above" which
  is trivially satisfied

This is **fine as a baseline slice** — it captures the
honest current state. But the gate isn't enforcing anything
useful yet. Worth being explicit about that in the packet
status so a reader doesn't think "delta gate active" means
"coverage now ratcheted."

### 2. (sequencing) The next Task 39 packet's substance is
"how do these modules get covered"

A 0.00% baseline implies the actual Task 39 work — making
critical modules *covered* — hasn't started. The lanes
exist, the gate exists, but coverage doesn't. The natural
next slice is:

1. **Identify which critical modules can be exercised in
   pure-Rust without pgrx.** Some `quant/*` modules
   (`hadamard`, `codebook`, `prod`'s codec paths) are
   computationally independent of PG and can be unit-tested
   directly. Adding such tests would move those baselines off
   0.00 immediately.
2. **Decide what's irreducibly pgrx-dependent.** `am/*/page.rs`
   uses pg_sys types and only meaningfully exercises through
   `cargo pgrx test pg18` runs. Pure-Rust coverage can't
   touch these without significant fixturing.
3. **Add pgrx-instrumented coverage as a separate lane
   later.** The task doc names this; it's the heavier lift.

Suggest a follow-up packet "Task 39 quant pure-Rust coverage"
that picks the highest-leverage `quant/*` modules and lands
unit tests sufficient to move baselines to non-zero. Then the
delta gate has something to defend.

### 3. (suggestion) Mechanism to raise baselines when coverage
lands

Once non-zero baselines exist, there needs to be a clear
mechanism for **raising the baseline** when coverage improves.
Two patterns work:

- **Manual baseline-update commits.** Coverage jumps to 35%
  after a test PR? Commit a baseline-update changing the TSV
  from 0.00 → 35.00 with a note like "raised by 002-...".
  Pro: explicit, reviewable. Con: requires discipline.
- **Auto-ratchet on PR.** The delta gate could optionally
  rewrite the baseline TSV when actual > baseline + 2pp,
  guarded by an explicit `RATCHET=1` env. Pro: less friction.
  Con: easier to accidentally relax (someone runs with
  `RATCHET=1` on a partial test run).

Worth one-line in `docs/hardening.md` or `check_coverage_delta.sh`
naming the chosen pattern. Currently both `docs/hardening.md`
and the script silently assume manual updates.

### 4. (verification) GitHub Actions wiring

Reviewed `.github/workflows/ci.yml` (or equivalent). Three
jobs added:
- `test-quality-coverage` — per-PR
- `test-quality-mutants` — workflow_dispatch + `cron: 37 10 * * 1`
  (Mondays 10:37 UTC)
- `test-quality-flake-hunt` — workflow_dispatch + `cron: 37 9 * * *`
  (nightly 9:37 UTC)

Cron times use minute 37 to avoid the :00 / :30 herd (consistent
with the ScheduleWakeup off-minute guidance — same hygiene
applies here for "every CI runner in the world hits the queue
at :00"). ✓

Timeouts:
- mutants: 90 min — reasonable for `cargo-mutants --all-features
  --in-place` on critical modules
- flake-hunt: 90 min — generous; `FLAKE_HUNT_SEEDS=8` × 8 fuzz
  targets × 10s = ~16 min compute, rest is build overhead

Reasonable defaults.

### 5. (low-priority) Coverage of Task 41 RAII guards

`src/storage/*_guard.rs` modules aren't in the baseline TSV.
Those are the RAII primitives Task 41 just shipped (~600 lines
across 9 files). They're pure-Rust except for the
`unsafe { pg_sys::... }` calls in constructors/Drops — the
test-only logic (e.g., the lint self-test fixtures) doesn't
exercise the Drop semantics at runtime.

Two options:
- **Add to baseline at 0.00%.** Captures the honest state
  and creates a future ratcheting target.
- **Document why excluded.** If the team's view is "these are
  always going to be 0% in pure-Rust because they're FFI
  wrappers," that decision should be explicit.

Currently silently absent. Recommend adding at 0.00% with the
same baseline note, since they're as critical as the page
codecs (Task 41 correctness rides on their Drop semantics).
