# Review: Task 39 coverage doc refresh (023)

## Summary

Refreshes the per-row coverage snapshot table in
`docs/hardening.md` after packets 019-022. Updates `quant/mod.rs`
to 100%, splits the SPIRE storage row into covered codecs +
recorded `relation_store.rs` gap, splits DiskANN `build`/`scan`
(covered) from `routine.rs` (recorded gap).

**Review note from the packet:** "Please focus on whether the
doc now states the remaining Task 39 gaps precisely without
treating the coverage baseline section as a packet-by-packet
changelog."

That review-note framing is **right in spirit but wrong in
execution.** The fix is not to maintain the snapshot table; the
fix is to remove it.

## Findings

### 1. (BLOCKING) Wrong direction — maintenance instead of deletion

Per the standing rule established 2026-05-18:
`docs/hardening.md` is **policy/contract for hardening lanes**.
It is not a log, ratchet-source changelog, or live baseline
snapshot. Two sections in that doc currently violate this:

- **"Baseline sources" per-packet bullet list** (lines ~256-280):
  one bullet per ratchet-source packet. This packet did **not**
  touch it. Packets 010, 011, 012, 015, 018 have been
  accumulating entries here. Every entry duplicates the
  annotation column of `fixtures/quality/coverage-baseline.tsv`.
- **"Critical area" per-row snapshot table** (lines ~283-296):
  this packet's edits. Mirrors the baseline TSV. Every coverage
  ratchet requires a hand-edit here to keep it from drifting.

**023's mistake:** instead of removing these sections, it
updates the snapshot table. That perpetuates the rule
violation rather than fixing it.

### 2. Required change

The right doc-refresh is **deletion**:

1. **Delete the "Baseline sources" list entirely** (the
   `Baseline sources:` heading + all bullet entries through the
   next blank line).
2. **Delete the "Critical area" table entirely.**
3. The remaining `docs/hardening.md` content is policy — what
   the coverage lane does, where the TSV lives, how the delta
   gate works, what the ratchet sequence is. **That stays.**
4. If a future reader wants to know "what's the current coverage
   on X?" — they read the TSV. If they want to know "which
   packet ratcheted X?" — they read the TSV's annotation
   column. Both already work without these sections.

### 3. Don't pretty-print the gaps either

This packet's diff adds the new bullet "live relation-store I/O"
to the existing "remain recorded gaps" sentence. That sentence
is policy (lane scope) — fine to keep. But **don't enumerate
which files are gaps inline** in the sentence; the TSV does
that with 0.00% rows. The sentence should describe the *kind*
of gap (PG callback paths, live relation I/O), not the specific
files.

### 4. (cross-reference) The cleanup is broader than 023

Packet 018 added a "Baseline sources" bullet for the IVF page
ratchet. Packet 015 added one for cost.rs. Packet 010 added one
for storage/page. All of those edits should come out as part of
this cleanup. Don't piecemeal it — one cleanup packet, full
delete, done.

## Disposition

**Reject the framing.** The "doc refresh" is the wrong move; the
right move is to **delete** the per-packet log + snapshot table
from `docs/hardening.md` entirely.

Required follow-up packet:
- Delete "Baseline sources" list (lines ~256-280).
- Delete "Critical area" snapshot table (lines ~283-296).
- Keep the policy/contract text describing the coverage lane.
- That packet supersedes 023; no further "refresh" edits.

Until that cleanup ships, every new coverage packet (018 was
the most recent example) will keep extending these sections,
and reviewer feedback will keep flagging it. The cleanup ends
the cycle.
