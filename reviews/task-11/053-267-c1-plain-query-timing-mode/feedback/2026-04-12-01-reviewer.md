# Review: Packet 267 — C1 Plain Query Timing Mode

Reviewer: agent-ix (via code-review)
Date: 2026-04-12

## Assessment: Good infrastructure — the negative result is valuable and the seam is worth keeping

### What this packet did right

1. **Tested the right hypothesis.** The question "is EXPLAIN instrumentation
   overhead part of the 11ms warm surface?" is the obvious next thing to rule
   out after the scorer-level work plateaued. The answer is no (explain:
   `11.111ms` vs plain-server: `11.020ms`), which is important because it
   redirects the next slice back to AM hot-path work.

2. **The planner verification is preserved.** The `plain-server` mode still
   runs the plan check before timing. This prevents the failure mode where a
   new timing mode silently measures a seqscan instead of the tqhnsw index.

3. **The MATERIALIZED subquery approach is clean.** Using
   `clock_timestamp()` around a `MATERIALIZED` subquery ensures the server
   measures only the ordered-scan execution time without client round-trip
   latency. This is the correct PostgreSQL pattern for server-side timing
   without EXPLAIN.

4. **Documentation was updated.** Both the recall docs and NFR-001 spec now
   acknowledge the new timing mode. Keeping the spec in sync with the harness
   prevents future confusion about what "latency" means.

### The result confirms: the warm bottleneck is in the AM, not the executor

The `~0.09ms` difference between explain and plain-server modes means EXPLAIN
instrumentation adds less than 1% overhead to the warm scan. This eliminates
the executor instrumentation layer as a target and firmly places the remaining
cost inside the AM rescan path.

Combined with packets 265 (dead query-prep removal) and 266 (scorer SIMD
failure), the picture is now clear: the `~11ms` warm surface is dominated by:

- Score evaluations: ~300-500 calls x ~14us each ≈ 4-7ms
- Graph element loads (cache misses): ~200-400 calls x ~10-13us each ≈ 2-5ms
- Graph neighbor loads: ~100-200 calls x ~8us each ≈ 0.8-1.6ms
- Beam/frontier bookkeeping: ~0.5-1ms

These estimates align with the packet 253 hot-path profile data, adjusted for
the packet 265 query-prep savings.

## Suggestions

1. **Keep the seam, but default new benchmark runs to `plain-server` for warm
   measurements.** Since EXPLAIN overhead is negligible, either mode works for
   warm. But `plain-server` is conceptually cleaner for steady-state reporting
   because it doesn't include explain output generation.

2. **Consider adding a `--timing-mode` default in the verified launcher.**
   Right now the default is `explain`. Since the warm surface is the primary
   C1 gate, and EXPLAIN overhead is confirmed negligible, this is a
   documentation concern rather than a correctness one.
