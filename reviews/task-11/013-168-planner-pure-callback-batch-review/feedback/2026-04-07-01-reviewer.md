# Batch Review: Planner Pure Callback Scaffolding (Reviews 10015-10025)

**Reviewer:** Claude Opus 4.6
**Date:** 2026-04-07
**Scope:** Reviews 10015-10025 (pure AM callback and struct scaffolding)
**Commit range:** `0202984` through `761bbb9`

## Summary

This batch addresses the review 10012 feedback directive to shift from snapshot SQL functions toward real D1 deliverables: pure AM callback helpers, reusable counter/stats structs, and EXPLAIN emission contracts. The direction is correct — this is the right work to be doing. Every slice is a pure Rust function with unit tests and no runtime wiring, which is exactly the planner lane's contract.

## Assessment by group

### Reviews 10015, 10021: ReadStream callback scaffolding (stream.rs)

Adds `GraphPrefetchState`, `LinearPrefetchState`, `ReadStreamCallbackSignature`, `ReadStreamCallbackResult`, plus pure callback functions `graph_prefetch_callback` and `linear_prefetch_callback`.

**Verdict: Approved.** These are the right abstractions for PG18 ReadStream integration. The graph prefetch state holds a `Vec<u32>` of block numbers and yields them in order; the linear state advances sequentially. Both return `EndOfStream` when exhausted, matching the PG18 `InvalidBlockNumber` sentinel contract.

**One note:** `GraphPrefetchState` stores blocks in a `Vec<u32>` and advances with an index. This is fine for scaffolding but the real PG18 integration will need to handle the case where new blocks are discovered during graph traversal (beam search discovers new neighbors while prefetching). The current design assumes blocks are known upfront. Flag this for the PG18 lane — it may need a channel or callback-driven block source.

### Review 10016: TqExplainCounters struct (explain.rs)

Adds `TqExplainCounters` with 7 fields, `record_*` methods, `reset()`, and `explain_properties()` that produces an array of `ExplainProperty` values.

**Verdict: Approved.** This is the concrete struct that will live in `TqScanOpaque`. The field names match the counter definitions from review 10014. The `explain_properties()` method produces the exact output shape for the EXPLAIN hook. Clean, minimal, ready for runtime wiring.

### Review 10017: Tree-height callback helper (cost.rs)

Adds `amgettreeheight_callback_value(max_level: u8) -> i32`. Trivial wrapper converting `u8` to `i32`.

**Verdict: Approved.** The function name matches the PG18 callback convention. Test confirms identity conversion and boundary (u8::MAX).

### Review 10018: Expanded compare-type contract (cost.rs)

Expands `PlannerCompareType` from `{Invalid, Lt}` to the full PG18 set: `{Invalid, Lt, Le, Eq, Ge, Gt, Ne, Overlap, ContainedBy}`. Updates `amtranslatecmptype_callback` to return 0 for all non-Lt types.

**Verdict: Approved.** Matching the full `CompareType` enum from PG18 is correct. Only `Lt` maps to strategy 1 (our ORDER BY `<->` operator); all others correctly return 0 (unsupported). Tests cover every variant exhaustively.

### Review 10019: Explain property emission skeleton (explain.rs)

Adds `ExplainPropertyValue`, `ExplainProperty`, and `TqExplainCounters::explain_properties()` method.

**Verdict: Approved.** The property names ("Bootstrap Expansions", "Bootstrap Pages Read", etc.) are user-facing EXPLAIN output. They're readable and match the counter semantics.

### Review 10020: Cumulative stats struct (stats.rs)

Adds `TqStatsCounters` (7 `u64` fields) with `record_*` methods and `reset()`.

**Verdict: Approved.** Parallel structure to `TqExplainCounters` but with `u64` for cumulative session-level stats (vs. `u32` for per-scan EXPLAIN counters). The field overlap (both have `quantizer_cache_*`) is intentional — EXPLAIN tracks per-scan, stats tracks cumulative.

### Review 10022: PG18 callback naming alignment (cost.rs)

Renames `strategy_to_compare_type` → `amtranslatestrategy_callback` and `compare_type_to_strategy` → `amtranslatecmptype_callback`.

**Verdict: Approved.** Naming now matches the PG18 AM callback names, which is the right convention for functions that will eventually become actual callback implementations.

### Review 10023: Explain output group contract (explain.rs)

**Verdict: Approved.** The `opened_with`/`closed_with` fields document the PG18 API calls that will bracket the EXPLAIN output. Minor: consider whether "TQVector Stats" or "tqhnsw Stats" is the right user-facing label — the access method name is "tqhnsw" but the extension is "tqvector". Not blocking.

### Review 10024: Stats summary derived rates (stats.rs)

Adds `TqStatsSummary` with derived `bootstrap_hit_rate` and `quantizer_cache_rate`. Implements `TqStatsCounters::summary()`.

**Verdict: Approved.** Division-by-zero handling is correct (returns 0.0 when denominators are zero). The derived rates are useful diagnostic output.

## Cross-cutting findings

### 1. Good course correction from review 10012 feedback

The prior review cycle flagged snapshot SQL function proliferation and recommended shifting to pure AM callback scaffolding. This batch does exactly that — 11 slices of pure Rust structs, helpers, and tests with zero new SQL functions. The planner agent followed the feedback.

### 2. Counter field naming: `stats_` prefix inconsistency

`TqExplainCounters` prefixes all fields with `stats_` (e.g., `stats_bootstrap_expansions`). `TqStatsCounters` does not prefix fields (e.g., `total_distance_calcs`). The `stats_` prefix on EXPLAIN counters is slightly misleading — they're per-scan counters, not cumulative stats. Consider dropping the `stats_` prefix from `TqExplainCounters` fields to avoid confusion when both structs appear in `TqScanOpaque`. Not blocking.

### 3. Slice granularity is still very fine

11 review requests for what is ~500 lines of pure Rust structs and methods. The slices are individually correct but could have been 3-4 batches (counters/struct group, callback helpers group, naming alignment). Same observation as review 156 on main — the per-commit granularity is good for bisect but the review overhead is high.

### 4. Snapshot consolidation still pending

The earlier feedback to consolidate from 7 SQL functions to 3 has not been addressed in this batch. These new slices correctly avoid adding more SQL functions, but the existing ones remain. Track this as a post-merge cleanup item.

### 5. ef_search GUC sentinel bug still open

The `resolve_scan_tuning_values` default-as-sentinel issue flagged in review 167 has not been addressed. Still needs fixing before the GUC is used in production paths.
