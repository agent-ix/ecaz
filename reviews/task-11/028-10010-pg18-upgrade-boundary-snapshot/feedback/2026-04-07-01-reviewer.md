# Feedback: PG18 Upgrade Boundary Snapshot

**Reviewer:** Claude Opus 4.6
**Date:** 2026-04-07

## Findings

### 1. Scope creep: FR-026 and FR-027 are new spec artifacts

The planner branch is creating new functional requirements (FR-026 PG18 module identity, FR-027 pgrx PG18 upgrade) and a new ADR (ADR-017). These weren't in the original plan. Creating new FRs should go through a spec review, not emerge from a scaffolding branch.

**Recommendation:** The PG18 upgrade boundary is real and needs tracking, but it belongs in a spec discussion or an ADR, not as SQL-surface scaffolding. Defer the SQL function, keep the ADR/FR creation as a separate discussion with the project owner.

### 2. The SQL function reports only static data

`tqhnsw_pg18_upgrade_snapshot()` returns the extension name, cargo version, default feature, and three `false` flags. None of this data is dynamic or requires SQL to observe — it's all derivable from `Cargo.toml` and the feature flags.

### 3. No boundary violation

The function doesn't touch graph search territory. Clean.

## Net

This slice goes beyond D1 scope by creating new FRs and ADRs for PG18 module identity. The functional work is correct but the scope expansion should be flagged to the project owner. The SQL function itself has minimal value over documentation.
