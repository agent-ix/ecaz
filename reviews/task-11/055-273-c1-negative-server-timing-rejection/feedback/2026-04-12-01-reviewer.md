# Review: Packet 273 — C1 Negative Server Timing Rejection

Reviewer: agent-ix (via code-review)
Date: 2026-04-12

## Assessment: Necessary infrastructure fix — prevents a class of silent measurement corruption

### What this packet did right

1. **Responded to a real incident.** Packet 271's first run produced `mean=3.636ms` with
   `min=-799.355ms` — a plausible-looking summary that would have been accepted if only
   p50/p95/p99 were checked. This is exactly the kind of silent corruption that undermines
   confidence in all prior and future measurements.

2. **The fix is in the right place.** Rejecting negative samples in the summary path (not
   the collection path) means the raw data is preserved for debugging while the reported
   statistics are protected. A non-zero exit makes the failure visible to the operator.

3. **EXPLAIN mode is correctly excluded.** EXPLAIN timing comes from PG's own instrumentation,
   not from `clock_timestamp()` differences, so it's not subject to wall-clock steps.

4. **The regression test is useful.** Testing the inline summary Python with synthetic
   negative samples catches future regressions without requiring a WSL2 environment.

### WSL2 timing reliability

The `clock_timestamp()` backward step on WSL2 is a known issue with the Hyper-V clock
source. This fix prevents the symptom from corrupting results, but it doesn't fix the
underlying clock instability. On WSL2, any `clock_timestamp()` delta smaller than ~1ms
should be treated with suspicion.

This is worth keeping in mind when evaluating small C1 deltas (like packet 271's flat
result or packet 269's 0.05ms improvement). The measurement floor on this host is probably
~0.2-0.5ms for trustworthy deltas.

## Suggestions

1. **Merge.** Essential infrastructure hardening.

2. **Consider logging the rejected sample count rather than failing immediately.** A single
   negative sample in a 100-cell run might be acceptable if the percentile statistics are
   computed from the remaining valid samples. Failing the entire run for one bad sample is
   conservative — correct for now, but worth revisiting if WSL2 clock steps are frequent
   enough to make valid runs hard to complete.
