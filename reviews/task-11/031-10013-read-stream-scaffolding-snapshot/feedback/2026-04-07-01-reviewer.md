# Feedback: ReadStream Scaffolding Snapshot

**Reviewer:** Claude Opus 4.6
**Date:** 2026-04-07

## Findings

### 1. Fold into planner-integration snapshot, do not keep as standalone

A dedicated `tqhnsw_read_stream_snapshot()` SQL function is not the right D1 seam. ReadStream is PG18-only and entirely future work. The three readiness bits (`callback_surface`, `scan_wiring`, `vacuum_wiring`) are already surfaced through `pg18_read_stream_surface_ready` in the planner integration snapshot. That is sufficient. Adding a seventh standalone snapshot function deepens the proliferation problem flagged in review 10012 feedback.

**Action:** Remove `tqhnsw_read_stream_snapshot()` from `lib.rs`. Keep `stream.rs` as the internal module — it's a clean seam for the future PG18 lane. Expose it only through the consolidated planner integration snapshot.

### 2. Mode names and access patterns are premature to expose via SQL

The strings `"random_access"`, `"sequential_access"`, `"block_range"`, `"single_block"` are reasonable internal labels but should not be in a SQL-visible surface until ReadStream binding exists. Keeping them internal to `stream.rs` is fine.

### 3. The three readiness bits are the right long-lived boundary

`pg18_callback_surface_ready`, `pg18_scan_wiring_ready`, `pg18_vacuum_wiring_ready` are the correct decomposition. These map cleanly to three distinct PG18 integration work items. Keep these.

**Note:** The subsequent reviews 10015 and 10021 add pure callback helpers and prefetch state structs to `stream.rs`, which is exactly the right follow-up — keep the internal module, remove the SQL surface.

## Verdict

The `stream.rs` module is approved. The standalone SQL function should be removed as part of snapshot consolidation.
