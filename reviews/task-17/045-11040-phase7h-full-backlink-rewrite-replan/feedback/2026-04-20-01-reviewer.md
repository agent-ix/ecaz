# Feedback: 11040-phase7h-full-backlink-rewrite-replan

Reviewed against head `ec1f334` (impl commit `112cb03`).

## Verdict

Approved. The plan/apply split lands the ADR-046 rewrite contract
cleanly and the write window stays page-local. One sizing concern
flagged below but it is not a merge blocker.

## Review focus answers

1. **Is the write window actually narrow?**

   Yes. Heap fetches, source-vector materialization, and the exact
   prune (`select_insert_forward_neighbors`) all run during
   `plan_backlink_mutations(...)` with no page lock held. The
   lock-holding path — `apply_backlink_mutations(...)` — only reopens
   the target tuple, runs the staleness check against the snapshotted
   `(expected_neighbors, expected_neighbor_count)`, and either writes
   the replacement bytes or returns `RetryReplan`. That is the right
   shape for an ADR-046 rewrite.

2. **Does the stale-detect / replan seam actually work?**

   Yes, and the three in-module tests pin the three states that
   matter: kept-after-prune rewrite (`in_013`), drift-detected retry
   (`in_014`), and successful rewrite after replan against fresh
   state (`in_015`). The retry-pass cap of `3` is enforced by the
   outer `for _ in 0..MAX_BACKLINK_REPLAN_PASSES` loop and errors
   loudly when exhausted — that matches the 11002 feedback directly.

3. **Does `apply_backlink_mutation` handle the "slot reopened" case
   for a full-slice plan?**

   Yes — it retries the free-slot path before deciding the plan is
   stale. That is the right cheap fallback for the window where
   another transaction freed a slot between plan and apply, and it
   avoids an unnecessary full rewrite.

4. **Is the pg test fixture sized right?**

   Yes. The 5-D shape is load-bearing because insert-side distance is
   `(-ip).max(0.0)`, so a sloppier fixture would produce zero-clamp
   ties and silently stop exercising the rewrite. Calling that out in
   the request was the right move — a future reader tempted to
   "simplify" this test to a 2-D or 3-D shape will now know why not
   to.

## Notes

- The "full slice may shrink after rewrite" behavior is intentional
  and documented; the pg test correctly does not assert that the
  post-rewrite slice stays at full capacity. Good catch during
  development — the earlier assumption would have made this test
  spuriously tight against the exact prune contract.
- Retries carry only the pending target tids forward and replan from
  a freshly materialized chain each pass. That keeps retry state
  target-local and avoids rerunning already-applied mutations.
- Stale targets (INVALID primary heaptid, dead tuples, unreadable
  neighbors) are silently skipped during planning. That is correct
  for this seam — a dead target should not receive a backlink — but
  it means a concurrent delete racing insert can quietly drop a
  planned backlink. Worth a follow-up note when entry-point /
  overflow maintenance lands, but not an issue for this slice.

## Sizing concern (non-blocking)

- `plan_backlink_mutations(...)` fetches heap source vectors for
  every live neighbor of every full target. At default
  `graph_degree_r = 32` with all forward targets full, that is
  O(R²) ≈ 1024 heap probes per insert before any actual rewrite.
  Each probe hits `SnapshotSelfData` via
  `fetch_heap_source_vector`, which is cheap individually but not
  free in aggregate. If a later real-corpus insert-throughput
  measurement flags this as hot, the fix is to cache planning-pass
  source vectors by `(target, neighbor_tid)` across the <=3 retry
  passes. Not required for merge — but worth writing down before the
  loop grows.

## Suggestions (non-blocking)

- `sort_and_dedup_backlink_targets` (in `insert.rs`) and
  `sort_and_dedup_item_pointers` (in `routine.rs`) are the same
  comparator on `ItemPointer` over different containers. Consider
  collapsing to a single `fn cmp_item_pointer_physical(a, b)` shared
  helper. Cosmetic only.
- `BacklinkMutationKind::RewriteFullSlice` stores the full
  `expected_neighbors: Vec<ItemPointer>` plus
  `replacement_neighbors: Vec<ItemPointer>` per target. For
  `graph_degree_r = 32` that is two 32-wide vecs per mutation held
  across the pass. Fine at current degree bounds; note it if the
  degree ceiling is raised.
