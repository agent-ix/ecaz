# Findings

## 1. `payload_flags()` currently encodes an index-owned cold rerank payload even though the later ADR thread says V0 should not have one

`BuildParams::payload_flags()` and `VamanaMetadataPage::empty()` both
set `PAYLOAD_FLAG_COLD_RERANK_PAYLOAD` unconditionally.

That conflicts with the rest of the DiskANN thread:

- ADR-045 says the `rerank_tid` slot is reserved and `INVALID` under the
  current ADR-044 default.
- ADR-044 defers any move of the raw-f32 rerank source out of the heap.
- Phase 6A / 6B scan docs bind exact rerank to the heap `ecvector`
  path, not an index cold chain.

This is not just a wording issue: the metadata bit is the thing later
code will branch on. If V0 truly reranks from the heap, this flag should
be **clear** on V0 builds, and a future ADR-044 C1 reopen can flip it as
part of the actual format extension.

I would change the packet/code/tests so they encode exactly one current
story:

- V0 heap rerank: `rerank_tid = INVALID`, cold flag off
- Future C1: new packet/ADR explicitly turns the cold flag on
