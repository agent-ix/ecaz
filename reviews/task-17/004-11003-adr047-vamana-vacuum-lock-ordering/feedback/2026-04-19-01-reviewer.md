# ADR-047 review answers for implementation

I reviewed the ADR-047 draft plus the later gap list on the
`adr034-diskann-access-method` branch. The most important correction is
that **pass 3 should not depend on an inbound-edge orphan scan**. The
later Phase 8A vacuum primitives and the existing HNSW A6 closeout both
point to a simpler and safer rule: finalize when the tuple is fully
dead, not when the graph proves it has zero inbound references.

## Findings

1. **Step 9 is currently too strong and likely wrong for the intended
   implementation.**
   Requiring "no live inbound neighbor references" forces either an
   O(N*R) accounting pass or a pass-2 side data structure. The branch's
   Phase 8A primitives already define the durable finalization seam as:
   `primary_heaptid == INVALID && !has_overflow_heaptids`.
   That should be the ADR rule.
2. **Cold rerank payload handling has the same contradiction as ADR-046.**
   ADR-045 reserves `rerank_tid` for a future ADR-044 C1 reopen.
   ADR-047 currently reads as if vacuum already owns a hot+cold payload
   chain. For Phase 8B, I recommend the same answer as ADR-046:
   **V0 has no index-side cold rerank chain.**

## Answers

### G1 — pass 1 discovery vs write split

Pass 1 should be defined as **one ordered block scan with an interleaved
page-local strip write**, not as two global passes.

Concretely:

1. open page under `SHARE`
2. discover dead primary heap TIDs on that page
3. if any need stripping, release `SHARE`, reopen that same page alone
   under `EXCLUSIVE`, strip them, WAL-log once, release
4. continue to the next page in block order

That matches the Phase 8A primitive boundary and keeps the pass count
honest: pass 1, pass 2, pass 3.

### G2 — replacement planning when the medoid is dead

Use a live fallback entry point.

If the metadata entry-point tuple is in the delete-set or already
tombstoned, read-only planning should fall back to the **lowest-block,
lowest-offset live tuple** in the chain. The later scan work on the same
branch already introduced exactly this `resolve_entry_point ->
first_live_tid` rule. Vacuum should use the same rule rather than
planning from a dead medoid.

### G3 — pass 3 orphan detection algorithm

Do **not** compute orphanhood via inbound-edge accounting.

Recommendation:

- pass 2 remains responsible for unlinking delete-set neighbors and
  best-effort fill
- pass 3 finalizes any tuple that is **fully dead**
  (`primary_heaptid == INVALID && !has_overflow_heaptids`)
- tombstoned tuples may still retain neighbor payload and may still be
  traversed until later reaping; that is acceptable and already matches
  the Phase 8A `mark_deleted` contract

This mirrors the HNSW A6 rule and avoids an unnecessary O(N*R)
bookkeeping surface.

### G4 — cold rerank payload lifecycle at pass 3

For Phase 8B, no action: **there is no cold rerank chain in V0**.

Pass 3 flips the hot tuple's `deleted` bit only. If ADR-044 C1 is
adopted later, then pass 3 can grow an optional hot-then-cold cleanup
rule, but that should be added together with the actual C1 wire-format
decision, not assumed early here.

### G5 — retry cap on replan

Name the cap and reuse the insert-side precedent:

- **`MAX_REPAIR_REPLAN_PASSES = 3` total ordered repair passes per
  vacuum run**

When the cap is hit, vacuum should log loudly and leave the remaining
dead/`INVALID` slots for a later vacuum rather than spinning.

### G6 — vacuum/insert concurrency invariant

State it explicitly:

> `ecdiskann` vacuum and live insert may run concurrently. Both use the
> same ordered-page-pass discipline and resolve write/write races through
> bounded read-only replanning rather than global serialization.

### G7 — `needs_medoid_refresh` ownership

Freeze ownership now:

- vacuum sets `needs_medoid_refresh = true` when the current entry-point
  tuple becomes fully dead / tombstoned
- insert does **not** set this flag
- rebuild (or a future explicit medoid-refresh maintenance path) is the
  only operation that clears it

That gives the flag monotonic semantics and prevents ADR-046/047 from
fighting over it.

### G8 — RobustPrune in planning vs fill-only in writes

Keep `RobustPrune` in planning. Do **not** fall back to nearest-first.

The right composition is:

1. read-only planning forms the candidate pool and runs `RobustPrune`
   against the current live neighbors plus proposed candidates
2. write phase consumes only the accepted candidates needed to fill the
   slots that are dead / `INVALID`
3. live neighbors are never evicted under the write lock
4. if the accepted set is smaller than the number of free slots, the
   remainder stays `INVALID`

This means vacuum repair is **best-effort connectivity repair**, not a
full restoration of the global Vamana alpha invariant. That is the
correct conservative contract for a concurrent vacuum path.

## Additional implementation answers

- **Generation counters are not required.** Insert can detect a stale
  vacuum interaction by reopening the tuple under `EXCLUSIVE` and
  comparing the live neighbor array against the read-only plan. The
  tuple contents already carry enough information.
- **Pass 3 resurrection concern:** coder-2 should make Phase 7 duplicate
  lookup ignore tuples that are tombstoned or have
  `primary_heaptid == INVALID`. Once pass 1 strips the heap TID, the
  node is no longer eligible for duplicate binding; pass 3 then becomes
  monotonic.

## Recommendation

I would not flip ADR-047 to `ACCEPTED` until the ADR text incorporates:

1. pass 1 as an interleaved share-scan / page-local strip write
2. medoid fallback to first live tuple
3. pass-3 criterion = fully dead, not inbound-edge orphanhood
4. explicit statement that V0 vacuum has **no** cold rerank chain work
5. named retry cap of `3`
6. explicit insert/vacuum concurrency sentence
7. explicit ownership and monotonic semantics for `needs_medoid_refresh`
