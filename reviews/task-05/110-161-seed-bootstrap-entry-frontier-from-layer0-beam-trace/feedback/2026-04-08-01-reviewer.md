# Review: Seed Bootstrap Entry Frontier From Layer-0 Beam Trace (Review 161)

**Reviewer:** Claude Opus 4.6
**Date:** 2026-04-08
**Scope:** Review 161 — scan consumes graph beam trace for entry seeding
**Commit:** `2643ade`

## Summary

Makes `scan.rs` consume the graph-owned layer-0 beam runner during bootstrap entry seeding when `ef_search > 1`. Seeds visible frontier, visited state, expanded-source state, and scheduler state from the resulting beam trace. Keeps the old single-entry path for `ef_search == 1`.

**Verdict: Approved.**

## Assessment

Entry seeding is the right first runtime seam. It's the least disruptive place to introduce beam trace consumption because it runs exactly once per scan (in `initialize_scan_entry_candidate`), before any result production or refill logic triggers. If it breaks, the scan returns no results — easy to detect, no partial corruption.

The `seed_bootstrap_trace` helper (line 608) correctly seeds all four state machines from the trace: visible frontier, BeamSearch scheduler, visited set, and expanded-source set. This is the full handshake needed for subsequent refill-on-consume to work.

Keeping `ef_search == 1` on the old path is a pragmatic compatibility boundary. The single-entry case doesn't benefit from beam search (only one candidate to expand), and keeping it separate avoids regressions in the trivial scan configuration.

No issues found. Good first runtime integration point.
