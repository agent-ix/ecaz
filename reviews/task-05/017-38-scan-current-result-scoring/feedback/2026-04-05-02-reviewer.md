# Feedback: Scan Current-Result Scoring

**Reviewer:** outside-review  
**Date:** 2026-04-05  
**Verdict:** Approved with findings

## Answers to review questions

1. **`<#>` value vs. support-function inner-product**: Storing the negated inner product (the `<#>` operator value) is correct. PostgreSQL's ordered index scan expects `xs_orderbyvals` to contain operator-class-compatible values. Since the planner will eventually emit `ORDER BY embedding <#> query`, the scan must produce the same value. The negation in `score_scan_element_result` (scan.rs:474, `-quantizer.score_ip_encoded(...)`) produces the `<#>` convention.

2. **Representative-heap-row gamma recovery as temporary bridge**: This is the most expensive part of the current scan path. `heap_tqvector_gamma` (mod.rs:514-609) does a full `heap_fetch` + detoast + unpack for every element scored. For the bootstrap linear scan this is acceptable (results are unordered anyway), but for ordered traversal with `ef_search=40` exploring hundreds of candidates, this becomes the dominant cost. **Strongly recommend persisting gamma in `TqElementTuple` before ordered traversal lands** (same recommendation as review 29).

3. **Heap-relation fallback correctness**: The fallback path (mod.rs:520-531) resolves the heap OID via `IndexGetRelation`, opens with `AccessShareLock`, fetches the tuple, then closes. This is correct for manual scan descriptors in tests where `heapRelation` is NULL. In production scans, PostgreSQL populates `heapRelation` on the scan descriptor, so the fallback path is not exercised. However, there is a subtle resource concern: `table_open`/`table_close` acquires and releases a lock on every scoring call when the fallback is used. For tests this is fine, but ensure this path is never hit in production scan execution.

## Finding: Per-element Vec allocation in scoring hot path

**Severity:** Medium (performance, not correctness)  
`score_scan_element_result` (scan.rs:471-474) allocates a `Vec<u8>` for every element scored:
```rust
let mut payload = Vec::with_capacity(4 + code_bytes.len());
payload.extend_from_slice(&gamma.to_le_bytes());
payload.extend_from_slice(code_bytes);
```
For the linear bootstrap scan, this is O(elements) allocations per full scan. For ordered traversal, this will be O(candidates_explored) allocations per `amgettuple` call. Consider adding `ProdQuantizer::score_ip_from_parts(prepared, gamma, code_bytes)` that reads gamma and code bytes in place without concatenation. This is a quantizer-level change that avoids the allocation entirely.

## Additional notes

- The `scan_seed` caching alongside prepared query is good — it avoids re-reading metadata for quantizer construction.
- Score lifecycle (valid on production, cleared on exhaustion/rescan) is correct.
