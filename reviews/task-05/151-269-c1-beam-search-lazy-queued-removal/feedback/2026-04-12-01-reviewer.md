# Review: Packet 269 — C1 Beam Search Lazy Queued Removal

Reviewer: agent-ix (via code-review)
Date: 2026-04-12

## Assessment: Clean structural improvement — the code is better even though the latency win is small

### What this packet did right

1. **Replaced the O(n) heap drain with O(1) lazy removal.** The old
   `forget_queued` drained the entire `BinaryHeap`, filtered, collected to a
   `Vec`, and rebuilt. The new implementation removes from a `HashSet<NodeId>`
   in O(1) and lets stale entries stay in the heap until they're naturally
   popped and skipped in `peek_best` / `pop_best`. This is textbook lazy
   deletion and it's correct.

2. **The `queued` set is well-integrated.** Looking at the current code:
   - `is_empty()` and `frontier_len()` use `self.queued` (not heap len)
   - `peek_best()` loops and pops stale entries
   - `pop_best()` loops and checks `self.queued.remove()`
   - `push_candidate()` inserts into `queued`
   - `forget_queued()` removes from `queued` in O(1), then does a linear
     scan + retain on `discovery_order`

   This is all consistent. The heap can contain stale entries, but they're
   never visible through the public API because every read path checks
   `queued`.

3. **The measured result is honest.** The `~0.05ms` mean improvement is small
   but directionally positive across p50/mean/p95. The packet doesn't
   overclaim. At ef_search=40, the old heap had at most ~40 entries, so even
   the O(n) drain was fast in absolute terms.

### Remaining concern: `discovery_order` retain is still O(n)

The new `forget_queued` still does:

```rust
self.discovery_order.retain(|discovered| discovered.node != node);
```

This is O(n) in the discovery order length, which grows with every
`push_candidate` call. At ef_search=40 this is at most ~40 entries, but at
ef_search=200 it could be ~200+. The retain also shifts elements in the Vec,
which is a memcpy.

For the steady-state frontier consumption pattern, `forget_queued` is called
from `peek_best_matching` and `take_best_matching` when the scheduler head
doesn't match the visible frontier. Each call removes one entry from
`discovery_order`. If many stale entries accumulate at the top of the heap,
this could be called repeatedly.

This is not a problem at the current scale, but it's worth noting that
`discovery_order` is the remaining linear-time bottleneck in the scheduler.
If it's only used for `snapshot_frontier` (diagnostic purposes), it could be
lazily cleaned or deferred to snapshot time.

### The 11ms baseline is now well-characterized

After packets 265-269, the warm `10K` `m=8` `ef_search=40` surface sits at
approximately `p50=10.9ms`, `mean=11.0ms`. The breakdown of what's been ruled
out:

- EXPLAIN instrumentation: ~0.09ms (packet 267)
- Statement planning: ~0.02ms (packet 268)
- Beam heap drain: ~0.05ms (packet 269)
- Dead query-prep work: ~3.3ms (packet 265, already removed)
- AVX2 scorer: no improvement (packet 266)

The remaining ~11ms is genuine AM execution time — scoring, graph loading,
allocation, and beam traversal.

## Suggestions

1. **Merge.** The code is structurally better and doesn't regress. The lazy
   deletion pattern is standard and well-tested.

2. **Consider whether `discovery_order` can be decoupled from `forget_queued`.**
   If `discovery_order` is only consumed by `snapshot_frontier` and diagnostic
   queries, the retain inside `forget_queued` could be deferred. This would
   make `forget_queued` truly O(1) instead of O(n). Low priority at current
   scale.

3. **The next C1 slice should not target more scheduler micro-optimizations.**
   The scheduler is no longer a meaningful contributor to the warm surface.
   The remaining headroom is in graph loading overhead (per-element allocation,
   PG buffer manager calls) and the aggregate cost of ~300-500 score
   evaluations that can't be made individually faster.
