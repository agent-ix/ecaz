# Feedback: Entry Candidate Lifecycle

**Reviewer:** outside-review  
**Date:** 2026-04-05  
**Verdict:** Approved

## Answers to review questions

1. **Entry candidate stable during partial scan progress**: Correct for the current design. The entry candidate is search initialization state, not scan cursor state. The bootstrap linear scan doesn't consume or mutate it — it walks pages independently. For future ordered traversal, the entry candidate will be consumed by the graph search initialization, but that's a different state machine. Keeping it stable during the bootstrap scan is the right groundwork.

2. **Clearing on exhaustion**: See finding in review 43 — this is acceptable now but should be reconsidered when ordered traversal replaces the linear scan.

3. **Missing lifecycle edge around rescan-after-partial-progress**: The existing `debug_entry_candidate_lifecycle` helper verifies stability through partial progress and clearing on exhaustion. A rescan-after-partial-progress test would confirm that `amrescan` re-seeds the entry candidate even when the previous scan didn't fully exhaust. This is implicitly covered by the existing rescan tests (which always re-seed via `initialize_scan_entry_candidate`), but an explicit test linking partial-progress + rescan + verify-re-seeded would be a clean addition.
