# Findings

## 1. `resolve_entry_point()` inherits the too-weak live predicate from `first_live_tid()`

The implementation currently accepts the preferred TID when
`!tuple.deleted`, and otherwise falls back to `reader.first_live_tid()`.

That is not enough once vacuum pass 1 strips the heap TID before pass 3
tombstones the tuple. In that state the tuple is still `!deleted`, but
it is not a valid scan entry point and does not carry a rerankable heap
row.

So both branches of the resolver are wrong in the same way:

- preferred TID may be accepted too early
- fallback TID may also be selected too early

The resolver needs the same stronger predicate as the reader's
"live-tuple" iterator.

## 2. The packet needs a strip-without-tombstone regression test

SC-012..SC-014 only exercise the `mark_deleted` case. Add the vacuum
intermediate-state regression explicitly:

1. strip the preferred tuple's `primary_heaptid`
2. leave `deleted = false`
3. assert the resolver falls back instead of returning that stripped
   tuple

Without that test, the current bug will slip through the exact scenario
ADR-047 says the scan must tolerate.
