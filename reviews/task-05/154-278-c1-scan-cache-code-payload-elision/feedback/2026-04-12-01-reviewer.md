# Review: Packet 278 — C1 Scan Cache Code Payload Elision

Reviewer: agent-ix (via code-review)
Date: 2026-04-12

## Assessment: Clean keep — correct final zero-copy seam

### What landed

Introduced `TqElementTupleRef` (borrowed decode view) in `page.rs` and
`with_graph_element_tuple` in `graph.rs`, then reworked the scan-local graph
element cache to score from the borrowed page bytes and cache only metadata —
not the 768-byte code payload.

### On the result

The ~2% improvement (10.72ms -> 10.51ms) is modest, and the packet correctly
calls this the last clearly-defensible copy/payload reduction on the current
scan cache shape. That self-awareness is good — it prevents further diminishing-
returns churn on the same seam.

### Structural value beyond the latency number

The `TqElementTupleRef` / `with_graph_element_tuple` API introduced here turns
out to be the critical enabler for everything that follows:

- Packet 281 uses it to cache binary words from borrowed tuple data
- Packet 285 uses it to read persisted binary sidecars from the trailing tuple
  payload

The borrowed-decode pattern is the foundation layer for the ADR-031 runtime
path. The packet doesn't claim this — it just ships the seam — but in hindsight
this is the most important packet in the 278-285 arc for structural reasons.

### One concern for the future

The scan cache still copies `heaptids: Vec<ItemPointer>` per element
(`scan.rs:163`). With `HEAPTID_INLINE_CAPACITY=10` and 6 bytes per TID, this is
a 60-byte heap allocation per cache miss. Not a priority now, but when pursuing
further allocation reduction, inlining heaptids as `[ItemPointer; 10]` +
`count: u8` on `CachedGraphElement` is the obvious next step.

### Verdict

Clean keep. Correctly exhausts the zero-copy line before pivoting to ADR-031.
