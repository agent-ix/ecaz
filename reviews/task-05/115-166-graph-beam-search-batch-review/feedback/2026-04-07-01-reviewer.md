## Assessment by group

### Reviews 158-160: Graph primitives (graph.rs)

Three clean primitives added:
- `load_layer0_neighbor_tids`: extracts valid neighbor TIDs, filters INVALID
- `load_layer0_successor_candidates`: loads neighbors, filters deleted/empty, scores via callback
- `run_layer0_beam_search`: runs `BeamSearch::run()` wired through the above

**Verdict: Approved.** File boundary is correct — `graph.rs` owns graph topology access, `scan.rs` owns query/scoring/visited state. The generic `KeepFn`/`ScoreFn` callbacks give scan full control without graph needing scan-internal types.

Three unit tests cover the internal helpers. The `run_layer0_beam_search` function itself is untested in isolation (it needs `pg_sys` page access), which is acceptable — it's covered via the scan-level `pg_test` tests.

**One minor note:** `load_layer0_neighbor_tids` loads the full graph element (element + neighbor tuple) but only uses the neighbor TIDs. The element's `code`, `gamma`, `heaptids` are decoded and discarded. This is functionally correct and only affects initialization cost, not hot-path. Not worth optimizing now, but worth noting if profiling shows element decode cost.

### Reviews 161-163: Beam trace seeding (scan.rs)

Replaces the old manual bootstrap expansion loop with beam-search-driven initialization:
- `seed_bootstrap_trace`: consumes a `BeamTrace` to initialize the visible frontier
- `refill_candidate_frontier_from_source`: runs beam search (ef_search=1) to refill after consume
- `initialize_scan_entry_candidate`: now runs a full beam search from the entry point

**Verdict: Approved with two findings.**

**Finding 1 (minor): `refill_candidate_frontier_from_source` seeds with score 0.0 (scan.rs:739)**

```rust
[search::BeamCandidate::new(source_tid, 0.0)],
```

The source node is seeded with a hardcoded `0.0` rather than its actual score. With `ef_search=1`, the source is immediately expanded (its score only matters relative to its own neighbors), so this is harmless today. But if `ef_search` is raised here later, the zero score would misrepresent the source's priority relative to discovered neighbors. Consider passing the consumed candidate's actual score, or adding a comment explaining why 0.0 is safe.

**Finding 2 (correctness note): `seed_bootstrap_trace` only marks the first candidate as expanded (scan.rs:628-630)**

The beam trace's `expanded` vec contains all nodes expanded during the search, but `seed_bootstrap_trace` only marks the entry candidate (first discovered) as expanded. This means later `refill_bootstrap_frontier_after_success` may re-expand nodes that the initialization beam search already visited. The test at line 2158 confirms this is intentional. The rationale should be documented: the refill operates with different visited state, so re-expansion can discover new candidates that the initialization search couldn't see.

### Reviews 164-165: Unified entry seeding and visible-seed top-up (scan.rs)

- `initialize_scan_entry_candidate` unified to always go through `run_layer0_beam_search` → `seed_bootstrap_trace`
- `top_up_bootstrap_frontier_from_visible_seeds`: multi-seed beam expansion from unconsumed visible frontier candidates
- `refill_bootstrap_frontier_after_success`: orchestrates consumed-source refill + visible-seed top-up

**Verdict: Approved.**

The `top_up_bootstrap_frontier_from_visible_seeds` function builds a `Vec` of seed candidates, then a `HashSet` of their nodes for post-search filtering. The double-collection is mildly wasteful but clear. No action needed.
