# Feedback: Two-Slot Candidate Frontier

**Reviewer:** outside-review  
**Date:** 2026-04-05  
**Verdict:** Approved with design notes

## Answers to review questions

1. **Two-slot frontier as staging structure**: Yes, this is the right intermediate step. A `BinaryHeap` or sorted `Vec` would be premature — the frontier semantics (insert, pop-best, check-visited) aren't implemented yet. The fixed two-slot container is honest about the current capability while providing the structural home that a future heap will replace.

2. **Lifecycle edges with entry and successor in same container**: `clear_scan_candidate_state` (scan.rs:266-268) zeroes the entire `CandidateFrontier` and sets `candidate_frontier_head` to `u8::MAX`. `recompute_candidate_frontier_head` (scan.rs:323-340) correctly handles all four combinations of (slot0 valid, slot1 valid). No lifecycle gap found.

3. **Communicating that traversal ordering is not implemented**: The `candidate_frontier_head` mechanic (pick the lower-scoring slot) technically *is* a trivial ordering over two candidates. The comment and review request text make clear this is structural scaffolding. One suggestion: when the transition to a real frontier happens, the `CandidateFrontier` struct and `candidate_frontier_head` field should be completely replaced rather than extended. Don't try to grow a 2-slot array into a heap — replace the container wholesale.

## Design notes for the next traversal slice

The current frontier structure reveals the shape of the upcoming graph traversal state machine. The next several slices will likely need:

1. **Visited set**: A `HashSet<ItemPointer>` or bitmap tracking which element TIDs have been scored. Without this, traversal will re-score previously visited neighbors. This should live in scan-owned state alongside the frontier.

2. **Candidate heap**: Replace `CandidateFrontier` with a `BinaryHeap<ScanCandidate>` ordered by score. The `candidate_frontier_head` field becomes unnecessary — the heap's `peek()` is the best candidate.

3. **Result buffer**: Separate from the candidate heap. Candidates that have been fully scored and are in the top-k become results. The current `CurrentScanResult` is a single-slot version of this — it will need to become a bounded priority queue of size `ef_search`.

4. **Gamma in element tuples**: Repeated across reviews 29 and 38 — the heap fetch per candidate will be the bottleneck. Persisting gamma in the element tuple is the critical enabler for efficient traversal.

5. **Layer-aware neighbor access**: The current flat neighbor list needs a layer interpretation for greedy descent through upper layers. This requires knowing the element's level and the index M parameter, then slicing the neighbor TID array accordingly.

The current groundwork (entry seed, successor seed, scoring, frontier container) provides the right scaffolding for these additions.

## Additional notes

- `recompute_candidate_frontier_head` uses `<=` comparison (scan.rs:333), preferring slot 0 on tie. This is arbitrary but consistent. For a real heap, ties should be broken by insertion order or element TID to ensure deterministic behavior.
