# Feedback: Scan Page-Level Neighbor Access

**Reviewer:** outside-review  
**Date:** 2026-04-05  
**Verdict:** Approved with finding

## Answers to review questions

1. **Element and neighbor tuple decode checks**: The `read_page_tuple_bytes` helper in graph.rs validates: offset in range (line 100), slot not unused (line 110-113), tuple bounds within page (line 117). This is sufficient defensive validation for a shared graph-read primitive. The BUFFER_LOCK_SHARE is correctly acquired and released on all paths including error paths.

2. **Keeping API flat-neighbor-only**: Correct for the current layout. The `TqNeighborTuple` stores a flat array of neighbor TIDs. The layer-segmented interpretation (layer 0 gets 2M slots, higher layers get M each) is a *logical* overlay that depends on knowing the element's level and the index's M parameter. Until the graph read API needs to distinguish layers (for greedy descent through upper layers), exposing only the flat list is the right boundary.

3. **Missing edge cases**: 

**Finding: No validation that neighbor TIDs point to element tuples**

**Severity:** Low  
`load_graph_neighbors` returns TIDs from the neighbor tuple's `tids` array, but does not validate that each TID actually points to a live element tuple. The caller (`initialize_scan_entry_candidate` in scan.rs:300-317) loads each neighbor via `load_graph_element`, which would error if the TID pointed to a non-element tuple or an invalid slot. This is acceptable defensive behavior for now — the error message would identify the corrupt graph link. However, for production traversal, consider adding a `TQ_ELEMENT_TAG` check in the graph read layer rather than relying on the decode error.

Also: neighbor TIDs that are `INVALID` (e.g., from unfilled higher-layer slots) should be filtered out before the caller tries to load them. The current caller in `initialize_scan_entry_candidate` iterates `neighbors.tids` which is already sliced to `count`, so unfilled slots beyond `count` are excluded. But INVALID TIDs *within* the active count (from deleted neighbors not yet pruned) would cause `read_page_tuple_bytes` to attempt reading block `u32::MAX`. This would likely error from PostgreSQL's buffer manager, but adding an explicit INVALID check in the traversal loop would be cleaner.

## Additional notes

- The flat neighbor access is the right stopping point before actual traversal.
