# Feedback: Successor Candidate Seeding

**Reviewer:** outside-review  
**Date:** 2026-04-05  
**Verdict:** Approved with findings

## Answers to review questions

1. **One-slot successor state as intermediate step**: Reasonable. The two explicit slots (entry + one successor) represent the minimum graph exploration: "I know where I am, and I know one neighbor." This is enough to validate the scoring and frontier infrastructure before introducing a general candidate heap.

2. **Selecting first live persisted neighbor**: Acceptable as a placeholder. The first live neighbor is arbitrary — for real traversal, candidates would be scored and prioritized. The current code (scan.rs:300-317) iterates `neighbors.tids` and breaks on the first non-deleted, non-empty neighbor. This is honest about not implementing traversal ordering.

3. **Missing cases**:

**Finding: No handling for INVALID neighbor TIDs within the active count**

**Severity:** Medium  
`initialize_scan_entry_candidate` (scan.rs:300-317) iterates `neighbors.tids` and calls `load_graph_element` on each. If a neighbor TID within the active `count` is INVALID (e.g., the neighbor slot was never filled during build for a node with fewer neighbors than its allocated slot count), `load_graph_element` will attempt to read block `u32::MAX` offset `u16::MAX`, which will either error from the buffer manager or read garbage. 

Add an explicit check: `if neighbor_tid == page::ItemPointer::INVALID { continue; }` in the neighbor iteration loop. This is cheap and prevents a confusing error message.

**Finding: Successor not cleared on exhaustion independently**

**Severity:** Low  
`clear_scan_candidate_state` zeros the entire `CandidateFrontier`, which is correct. But there's no independent lifecycle for the successor slot — it's always cleared together with the entry slot. For the current two-slot design this is fine, but when expanding to a real frontier, each candidate will need its own lifecycle (consumed when expanded, replenished from neighbors).

## Additional notes

- The scoring call for the successor candidate reuses the same `score_scan_element_result` path as the entry candidate, which correctly goes through the heap gamma fetch. Consistent.
