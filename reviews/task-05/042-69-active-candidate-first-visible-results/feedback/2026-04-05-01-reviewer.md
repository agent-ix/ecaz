# Feedback: 69 — Active Candidate First Visible Results

**Reviewer:** Claude Opus  
**Date:** 2026-04-05

## Note: Code Has Evolved

The visible amgettuple path no longer uses the active_candidate staging. `materialize_next_bootstrap_frontier_result` (scan.rs:784) directly consumes and materializes frontier candidates. The active_candidate path (`maybe_consume` + `materialize_active_candidate_result`) remains for debug/helper flows only. Line numbers have shifted significantly.

## Overall assessment

This is the most consequential scan slice so far — it makes frontier candidates produce visible tuple output and introduces the emitted-element dedup set as the correctness seam between candidate-driven and linear-scan paths. The design is correct and well-tested.

## Findings

### 1. Candidate-first materialization ordering is correct

The `amgettuple` flow (lines 148–167):
```
maybe_consume_bootstrap_frontier_candidate  → sets active_candidate
materialize_active_candidate_result         → loads heaptids, clears active, marks emitted
take_pending_scan_heap_tid                  → returns first heap tid
... (falls through to linear scan if no candidate or materialization fails)
```

Each step has clear preconditions:
- `materialize_active_candidate_result` gates on `active_candidate.score_valid` and `pending_heaptid_count == 0`
- On success, it clears the active slot before loading heaptids, so re-entry drains pending before consuming the next candidate
- The `maybe_consume` guard (line 641–647) correctly blocks consumption when pending drain is in progress, when the scan is exhausted, or when an active candidate already exists

No interleaving hazard — all steps execute synchronously within one `amgettuple` call.

### 2. Emitted-element dedup is the right boundary

`mark_emitted_element` is called from `set_current_scan_result` (line 840), which is the single point where result state transitions to active. This means both the candidate materialization path and the linear scan path mark elements as emitted through the same seam. The linear scan's `emitted_contains_element` check at line 783 then skips those elements.

One correctness property worth noting: the emitted check fires AFTER the cursor is advanced (lines 773–778 set `next_block_number` and `next_offset_number` before line 783). So skipping an emitted element correctly advances the linear cursor past it without stalling.

### 3. "Every heap tid exactly once" invariant is well-covered

The `test_tqhnsw_gettuple_returns_all_duplicate_heap_tids` and `test_tqhnsw_gettuple_duplicate_scan_spans_multiple_pages` tests both sort observed vs expected tids and assert set equality. This correctly verifies completeness (no missing tids) and uniqueness (no duplicates) regardless of output order.

### 4. Dead code in linear scan (carried from review 68)

As noted in the review-68 feedback, the active-candidate match branch at lines 786–796 in `next_linear_scan_heap_tid` is now unreachable. It should be cleaned up in a follow-on slice. No correctness impact.

### 5. Edge case: element deleted between frontier seeding and materialization

If an element is seeded into the frontier during `amrescan` but deleted (or its heaptids emptied) before `materialize_active_candidate_result` runs, the materialization correctly detects this at line 666–668 and returns false. The active candidate is cleared at line 665 regardless, and the linear scan falls through normally. The emitted set is NOT populated for this element (since `set_current_scan_result` is never called), so the linear scan will re-encounter and correctly skip the deleted element via `element.deleted` at line 769.

This is correct behavior — no stale emitted entry, no phantom result.

### 6. Minor: emitted set grows monotonically

The emitted set is allocated and reset during `reset_scan_position` (called by `amrescan`) and freed during `amendscan`. Within a single scan pass, it grows monotonically. For the current bootstrap frontier (3 candidates), the overhead is negligible. When ordered traversal returns many candidates before the linear fallback, the set will hold every returned element. This is bounded by the number of distinct elements in the index, which is fine.

## Verdict

Correct and well-staged. The emitted-element dedup is a clean correctness seam that will scale naturally to larger frontier sizes. The only cleanup is the dead bridge code from review 68.
