# Findings

## 1. The current "no tombstone filtering" stance is too weak for the API the packet exposes

`vamana_scan` / `greedy_descent` push `ScanCandidate`s that carry only:

- `tid`
- `primary_heaptid`
- `score`

The later rerank stage receives only `primary_heaptid`.

That means the packet's current design gives the rerank closure no way
to reject a candidate that is:

- `deleted = true`, or
- pass-1 stripped (`primary_heaptid = INVALID`) but not yet tombstoned

The packet currently says tombstone filtering belongs to the pgrx
boundary, but the rerank closure does not receive enough information to
do that filtering correctly. In a concurrent vacuum window, the current
shape can surface an invalid heap TID into rerank.

Coder-2 should resolve this before Phase 6B:

- either make the scan shell filter non-rerankable tuples itself, or
- carry tuple liveness / tombstone state forward in `ScanCandidate`
  so rerank can skip them safely

As written, Phase 6B cannot wire this cleanly.

## 2. The test surface is missing the strip-only vacuum state

The resolver tests cover tombstoned tuples via `mark_deleted`, but the
dangerous concurrent state is earlier:

- `primary_heaptid = INVALID`
- `deleted = false`

That is exactly what pass 1 produces before pass 3 finalizes. Add a test
that constructs this state and prove the scan does not emit it as a
rerank candidate or final result.
