## Post-v0.1 Follow-up Notes for A3

These items are not blockers for v0.1 but are worth tracking for future cycles:

### 1. Raw pointer aliasing in `GraphTraversalPrefetchContext::run`
The `self as *mut Self` pattern (line 452) is sound but fragile — it exists because `select_next_with_refill` takes multiple closures that need mutable access to the same context. In a future cycle, consider whether the frontier selection API can accept a single trait object or visitor instead of a closure bundle, which would eliminate the self-aliasing. Not urgent — the aliasing is contained within one method and the SAFETY comment is adequate.

### 2. Graph exhaustion → fallback transition policy
Currently, graph exhaustion always marks `Exhausted` (via `mark_scan_exhausted` in `ensure_prefetched_output`, line 345). There is no graceful fallback to linear for partial-graph indexes (e.g., indexes with sparse connectivity or very few layers). For v0.1 this is fine — partial-graph behavior is an edge case that should surface through A4 recall testing first. In a future cycle, consider whether some graph-exhaustion conditions should transition to `LinearFallback` instead of `Exhausted`, especially after graph-aware insert (A5) introduces indexes that may have connectivity gaps during live writes.

### 3. `with_visible_frontier_mut_and_bootstrap_expansion` borrow splitting
The `with_visible_frontier_mut_and_bootstrap_expansion` helper (used in `GraphTraversalPrefetchContext::prefetch_next`, line 441) exists to work around Rust's borrow checker for disjoint field access through `TqScanOpaque`. In a future cycle, if `TqScanOpaque` is refactored to separate graph-phase state into its own struct (which the cursor extraction arc has laid groundwork for), this helper can be replaced with direct field borrows. Low priority — the current approach is correct and well-documented.

### 4. `GraphTraversalCursor` lifetime scope
The cursor is currently reconstructed on each call via `graph_traversal_cursor(opaque)` (line 373), which borrows `opaque.result_state`. This means cursor construction happens multiple times per `amgettuple` call (has_prefetched_output, emit, needs_refresh, ensure_prefetched). In a future optimization cycle, consider whether the cursor can be held across the full `produce_next_graph_traversal_heap_tid` body to avoid repeated borrow re-entry. Unlikely to matter for perf — the compiler almost certainly inlines these — but it would make the code read more naturally.

### 5. Integration-level ordered-result test coverage
The cursor mechanics have strong unit coverage, but there is no integration test that confirms the graph-first path produces heap TIDs in distance-sorted order across a non-trivial built index. This should come naturally during A4 recall measurement, but a persistent regression test at that level would be valuable after A4 passes.
