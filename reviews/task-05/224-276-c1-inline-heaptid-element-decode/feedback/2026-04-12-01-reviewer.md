# Review: Packet 276 — C1 Inline Heaptid Element Decode

Reviewer: agent-ix (via code-review)
Date: 2026-04-12

## Assessment: Correct discard — the idea was right but the ownership change had unintended side effects

### What this packet did right

1. **Targeted the right field.** `heaptids: Vec<ItemPointer>` is allocated on every element
   decode for a fixed-capacity array of at most 10 entries (60 bytes). Replacing it with
   `[ItemPointer; HEAPTID_INLINE_CAPACITY]` + count is the textbook fix.

2. **The local codec tests passed.** `element_tuple_roundtrip` and `inline_heaptids`
   confirmed that the encoding/decoding is correct in isolation.

3. **Discarded when the full test suite showed instability.** The bootstrap-frontier failures
   (`pg_test_tqhnsw_frontier_head_refills_from_consumed_neighbors` and
   `pg_test_tqhnsw_rescan_builds_bootstrap_candidate_frontier`) under full-suite ordering
   — but not in isolation — indicate a subtle state interaction, not a codec bug. Walking
   away from this rather than debugging an order-dependent failure for a small allocation
   win is the right priority call.

### What likely went wrong

The bootstrap-frontier tests depend on graph element equality or identity checks somewhere
in the frontier seeding path. Changing `Vec<ItemPointer>` to `[ItemPointer; 10]` + count
changes:

- `PartialEq` / `Eq` semantics: a Vec compares only its live elements; an inline array
  with a count field compares differently if unused slots contain stale data
- `Clone` behavior: a Vec clone is a fresh allocation; an inline array clone is a memcpy
  including unused slots
- `Debug` output: affects test failure messages and any debug-path comparisons

If `GraphElement` or `TqElementTuple` is compared anywhere in the frontier path (even
transitively via `PartialEq` derives), the unused array slots could cause false inequality.
The fix would be to implement `PartialEq` manually to compare only the first `count`
entries, but that's more complexity for a small win.

### The broader inline-decode direction remains valid

This specific attempt failed on test stability, not on performance or correctness. The
heaptid Vec allocation is real overhead (~60 bytes + allocator metadata per element load).
A future attempt could:

- Use a newtype wrapper around `[ItemPointer; 10]` + `u8` count with correct `PartialEq`
- Or avoid the ownership change entirely by decoding heaptids in-place from the pinned
  page buffer (zero-copy), which skips both the Vec and the inline array

## Suggestions

1. **Record and move on.** Do not re-attempt inline heaptids until the zero-copy decode
   path is designed. The zero-copy approach (holding the buffer pin through decode) would
   make both the Vec and the inline array unnecessary.

2. **If re-attempted, implement `PartialEq` manually.** The derive-based equality on an
   inline array with unused slots is the most likely cause of the order-dependent test
   failures.
