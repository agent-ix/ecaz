# Batch Review: Search Seam Extraction (Reviews 10026-10032)

**Reviewer:** Claude Opus 4.6
**Date:** 2026-04-07
**Scope:** Reviews 10026-10032 (post-merge search seam extraction)
**Commit range:** `92e6814` through `c196f2a`

## Summary

Seven commits extracting scan-internal bootstrap traversal logic into search-owned and graph-owned
APIs. Each commit moves a narrow slice of state management from `scan.rs` into `search.rs` or
`graph.rs`. The direction is correct — this work is making `scan.rs` thin enough that A3 (wiring
graph search as the primary scan path) will be a clean, small change rather than major surgery.

**Overall verdict: Approved with directive to cap the extraction and shift to A3.**

## Assessment by group

### Review 10026: Visible Frontier Search Seam (search.rs)

Introduces `VisibleFrontier<NodeId>` as the search-owned container for scan-visible candidates.
Moves `best_candidate` and `consume_best` behind search-owned APIs. Scan delegates frontier reads
and consumption.

**Verdict: Approved.** The `VisibleFrontier` abstraction is the right boundary — it owns the Vec
of candidates and coordinates with `BeamSearch` for scheduler-aware selection. The stale-scheduler
pruning path (fall back to Vec when scheduler is drained) is preserved through the
`best_candidate_by_score` fallback.

### Review 10027: Visible Frontier Progression (search.rs)

Adds `advance_after_consume` on `VisibleFrontier`. Moves post-consume refill/top-up orchestration
behind the search seam. Scan-side refill helpers refactored to operate on already-borrowed frontier
and scheduler.

**Verdict: Approved.** The callback-heavy signature on `advance_after_consume` (4 closures) is
complex but justified — it lets search own the progression protocol while scan retains control of
graph I/O, visited marking, and expanded tracking. The alternative (search knowing about scan
internals) would be worse.

### Review 10028: Layer-0 Top-Up Graph Seam (graph.rs)

Adds `load_layer0_refill_successors` and `expand_layer0_visible_seeds` to graph.rs. These
consolidate the scan-owned direct `run_layer0_beam_search` calls into graph-owned helpers.

**Verdict: Approved.** Good separation. Graph owns topology traversal, search owns candidate
management, scan owns scoring and visited state. The `Layer0VisibleSeedExpansion` return type
cleanly separates expanded sources from discovered candidates.

Unit tests cover best-first refill ordering and visible-seed expansion filtering. The
`refill_successors_with_successors` internal helper reuses beam search with ef=1, which is a
clever way to get single-hop expansion without a separate code path.

### Review 10029: Bootstrap Selection Search Seam (search.rs)

Adds `select_next_with_refill` on `VisibleFrontier`. Owns the consume-until-selected loop that
was previously in scan.rs.

**Verdict: Approved.** This is the most significant seam in the batch — it moves the core
bootstrap result-selection loop into search. The callback signatures (`RefillFn`, `TopUpFn`,
`SelectFn`) correctly preserve scan's authority over graph I/O and result adjudication while
search owns the iteration protocol.

### Review 10030: Discovered Candidate Seeding (search.rs)

Adds `seed_discovered` on `VisibleFrontier`. Unifies the pattern of extending visible frontier +
marking visited + seeding scheduler.

**Verdict: Approved.** Three call sites in scan.rs (bootstrap seed, single-source refill,
visible-seed top-up) now go through one search-owned path. The visited-marking callback correctly
lets scan maintain its own HashSet.

### Review 10031: Bootstrap Trace Seeding (search.rs)

Adds `seed_bootstrap_trace` on `VisibleFrontier`. Owns conversion from a `BeamTrace` into
initial visible-frontier and scheduler state.

**Verdict: Approved.** Correctly truncates trace to `max_candidates`, seeds via `seed_discovered`,
and marks the entry candidate as expanded. Test covers truncation, visited marking, and
entry-expansion semantics.

### Review 10032: Visible-Seed Top-Up (search.rs)

Adds `top_up_from_visible_seeds` on `VisibleFrontier`. Owns the visible-seed filtering,
capacity math, expanded-source marking, and discovered-candidate registration for the post-success
top-up path.

**Verdict: Approved.** This completes the search seam extraction — all bootstrap traversal
protocol is now behind search-owned APIs. Scan's remaining role is: (1) provide graph I/O
callbacks, (2) maintain visited/expanded/emitted sets, (3) adjudicate results, (4) drive the
linear fallback.

## Cross-cutting findings

### 1. Seam extraction is complete — shift to A3

The `VisibleFrontier` type now owns:
- Candidate storage and iteration
- Best-candidate selection (scheduler-aware + Vec fallback)
- Consume and advance protocol
- Bootstrap trace seeding
- Discovered candidate registration
- Visible-seed top-up orchestration
- Select-next-with-refill iteration

This is the full bootstrap traversal protocol. There is no remaining scan-internal traversal logic
worth extracting before A3. **Directive: stop extracting seams and start A3 (wire graph beam
search as primary scan execution path).** The seam work has achieved its goal — scan.rs is now thin
enough that replacing the linear fallback with graph-first execution is a contained change.

### 2. Dead code from review 166 still not gated

Review 166 flagged 5 dead production functions in scan.rs that should be `#[cfg(any(test,
feature = "pg_test"))]` gated:
- `fill_bootstrap_frontier`
- `top_up_bootstrap_frontier`
- `refill_bootstrap_frontier_after_consume`
- `seed_existing_frontier_into_expansion`
- `BootstrapExpandPolicy` enum

These were not addressed in this batch. They should be gated before A3 to keep the production
binary clean. This is a 5-minute task.

### 3. Slice granularity observation (repeated)

7 review requests for ~400 lines of structural refactoring. Each is individually correct but
carries more review overhead than the code changes justify. For future refactoring batches of this
nature, consider batching 2-3 related seam extractions per review request. The per-commit
granularity in git is fine; the per-commit review request granularity is excessive.

### 4. Test coverage is good

Each seam extraction includes pure search-level unit tests that cover the delegated behavior
without needing pg_sys page access. The test strategy of "pure search tests for protocol, pg_test
for integration" is correct and should continue into A3.
