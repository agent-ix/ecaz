# Review: SPIRE Embedding UPDATE Rejection (30841)

## Summary

Wires the ADR-069 embedding-UPDATE rejection into the shared
coordinator UPDATE primitive. New `ec_spire_index_key_column_names`
helper looks up the `ec_spire` index's key columns (i.e., the
columns that the index is built on, including the embedding
column). If any `updated_columns` overlap with the key columns,
the primitive raises the exact ADR-069 error message via pgrx's
`ErrorReport` with `ERRCODE_FEATURE_NOT_SUPPORTED`:

> `ec_spire_distributed: UPDATE of indexed embedding column is not
> supported on a distributed ec_spire table. Use DELETE + INSERT.`
> `HINT: Cross-shard atomic moves will be available in a future release.`

PG18 fixture catches the error and asserts both message and hint
text via the pgrx `ErrorReport` interface.

The three review-focus questions:
- Rejection at the shared UPDATE primitive boundary is right: yes
  — the front-door hook will call this primitive, so guarding here
  catches all callers (trigger, future ModifyTable hook, direct
  SQL invocation).
- Using the `ec_spire` index key columns as the embedding-column
  detector matches v1 table shape: yes — the indexed column *is*
  the embedding by construction. Generalizes cleanly if INCLUDE
  columns or expression indexes get added later.
- Error message and hint match ADR-069 exactly: yes.

## Findings

### P3 — `index_key_column_names` returns ALL key columns, not just the embedding

If a future packet adds a multi-column ec_spire index (e.g.,
`USING ec_spire (embedding, category)`), this rejection would also
fire on UPDATEs to `category`, which arguably should be allowed
(non-vector key column). For v1 the ec_spire AM only supports a
single key column, so this is moot. Worth a one-line comment in
`ec_spire_index_key_column_names` that "all key columns" is
conservative and that future multi-column ec_spire indexes will
need to distinguish vector-distance keys from non-vector keys.

### P3 — `INCLUDE` columns not affected

`indkey` excludes INCLUDE columns (they're in `indkey` past
`indnkeyatts`, but the SQL filter `key_column.attnum > 0` plus
the `unnest(idx.indkey)` shape covers all `indkey` slots). Worth
confirming via an explicit `LIMIT idx.indnkeyatts`-style guard
in the SQL so a future PG version that changes the array layout
doesn't accidentally start treating INCLUDE columns as key columns.
Minor.

### P3 — `pgrx::PgSqlErrorCode::ERRCODE_FEATURE_NOT_SUPPORTED`

SQLSTATE `0A000` matches the meaning. Operators / drivers can
catch on this SQLSTATE explicitly. Good.

### P3 — Test fixture pipes error and hint through `ErrorReport`

The test asserts both error message and hint via a pipe-separator
("error|hint"). Catches both halves of the contract. Effective
regression guard.

## Reviewer recommendation

Accept.
