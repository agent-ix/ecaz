# Remote node descriptor contract review

`ec_spire_remote_node_descriptor_contract()` lays out the 12-field
schema for the future descriptor catalog. Contract-only; no
persistence yet. No blocking findings.

## Confirmations

1. `conninfo_secret_name` is `text` with semantic role
   `indirect_connection_secret` and validator
   `must_be_nonempty_secret_reference` — confirming the design's
   "no raw conninfo storage" invariant.
2. `state` validator (`must_be_active_or_draining_for_reads`)
   matches the 30552 design's eligibility model.
3. `last_seen_at` and `last_error` are correctly `required:false`
   and `validator:diagnostic_only` — health timing and error text
   should not gate libpq dispatch.
4. `min_retained_epoch` validator
   (`must_not_exceed_requested_epoch`) and `last_served_epoch`
   validator (`must_cover_requested_epoch`) bracket the served
   epoch window correctly.
5. `coordinator_index_oid` validator
   (`must_equal_local_index_oid`) prevents a descriptor from one
   index from being applied to another — important when multiple
   SPIRE indexes share the same coordinator session.

## Findings

- **`remote_index_regclass` validator
  `must_resolve_on_remote_node`.** The validator name is correct in
  intent — the regclass is a remote-side OID that must resolve in
  the storage node's catalog. But the diagnostic surface might
  give an operator the impression that local catalog resolution is
  the check. A short doc-comment in the row builder noting "this
  validator runs at descriptor-registration time against the
  remote node, not the coordinator catalog" would prevent
  misinterpretation. Forward-looking, not blocking.
