# Phase 6 top-graph build draft review

Large slice covering codec, build draft, scan-side route helper,
object-store integration, recursive build binding, scan candidate
binding, options, and the SQL diagnostic. The defaults remain
inert and the gate (`top_graph_enabled = 1` + `recursive_fanout >= 2`)
is properly fail-closed. No blocking correctness findings, but
several non-blocking observations below.

## Findings

- **`max_centroid_inner_product` is O(n²) and recomputed at every
  build.** `build/top_graph.rs:227-235` runs an `n²` IP across every
  pair of node centroids to compute the distance offset. With root-only
  node counts this is fine; once the design pulls in a wider top
  routing-child set (per 30549's deferred scope), this becomes
  expensive. A cheaper upper bound is
  `max_norm_sq = max ‖c_i‖²`, and the IP between any two unit-bounded
  centroids is bounded by `max_norm_sq`. Replacing the double-loop with
  one `max ‖c‖²` pass gives the same nonnegative-distance guarantee
  with linear cost. Not blocking — but worth landing before scaling
  the node-set.

- **Distance offset and Vamana metric assumptions.** Vamana's
  `RobustPrune` uses `alpha`-pruning that assumes a metric distance
  (or at least a quasi-metric). `(distance_offset - ip).max(0.0)` is
  *order-preserving with respect to IP*, but it is not a metric: there
  is no triangle inequality, and the diagonal entries are
  `max_ip - ‖c_i‖²`, which need not equal zero. For "closeness routing"
  this is empirically fine and the design accepts it, but the design
  note (30549) should state explicitly that alpha-pruning is being run
  against a non-metric pseudo-distance derived from IP and that the
  recall study on routing centroids is what justifies this.

- **`top_graph_inner_product` duplicates `inner_product` in
  `build/routing_plan.rs:281`.** The followup packet (30551) just
  consolidated centroid IP ranking into one helper across build/scan/
  split paths. The top-graph's standalone copy reintroduces the same
  primitive. Reuse the routing-plan helper (or hoist an
  `am::ec_spire::math::inner_product`) to keep the followup's
  consolidation intact. Cosmetic but easy.

- **Codec — `reserved` field rejection.**
  `storage/top_graph.rs:99-104` reads a `u16` reserved word and
  rejects nonzero values. Good — this preserves a forward-compatible
  byte slot and refuses to accept indeterminate bytes. The encode side
  unconditionally writes `0_u16` (`storage/top_graph.rs:71`), which
  matches.

- **Scan binding fail-closed.** The scan-side path
  (`scan/candidates.rs::collect_top_graph_scan_plan_reranked_candidates`)
  calls `collect_snapshot_top_graph_routed_probe_leaf_rows`, which
  bails out with an explicit error
  ("ec_spire scan snapshot has no available top graph object") if the
  graph object is missing. The behavior matches the design's
  fail-closed contract.

- **Diagnostic SQL surface.**
  `ec_spire_index_top_graph_snapshot` reports both
  `configured_search_list_size` and `effective_search_list_size`,
  which is necessary because `top_graph_search_list_size = 0` means
  "derive later from `nprobe`" — operators will rely on this
  distinction during tuning. Good shape.

## Confirmations

- Validation surfaces (root PID, dims, degree, build-list size, alpha,
  child PID and centroid ordinal uniqueness, finite centroids, neighbor
  bounds, no self-neighbors, neighbor-count ≤ degree) all line up
  between `SpireTopGraphBuildInput::validate` and
  `SpireTopGraphPartitionObject::validate`. The codec rejects ragged
  bodies via the `cursor != tail.len()` check at the end of decode.
- Sort order for graph-routed children — graph distance ascending,
  centroid ordinal, child PID, then graph node ordinal — gives a
  deterministic route ordering that matches the rest of SPIRE's
  tie-break conventions.
- Object PID is allocated **after** leaf placements in the explicit
  recursive build variant
  (`build/recursive.rs` graph-publish), so the graph object PID does
  not collide with leaf-allocator state.
- Reloption defaults: `top_graph_enabled = 0`, search-list `0` deriving
  from `nprobe`, are correctly inert until opted in. `top_graph_enabled
  = 1` without `recursive_fanout >= 2` fails build, matching the
  design.
