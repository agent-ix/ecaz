# Review: SPIRE Boundary Replica Remote Identity (30957)

## Summary

Closes Phase 12.7 boundary-replica multi-instance proof
row. Extends `ec_spire_index_boundary_replica_identity_snapshot`
to read coordinator metadata copies for synthetic remote
placements while still reporting the original placement
node/local-store span.

Fixture builds a `source_identity = 'include'`,
`boundary_replica_count = 1`, two-local-store index;
rewrites one leaf to remote node/store `2`; asserts:

- four ready global identity rows;
- 1 primary + 1 boundary-replica per source identity;
- at least one ready global identity spans node IDs `0..2`.

## Findings

### P2 — Metadata-read for synthetic remote placements

This is the right boundary for an *identity propagation*
proof. Identity propagation is a coordinator-side metadata
property (does the global vec_id appear in primary +
boundary placements across nodes?) — it does NOT require
live remote object reads to verify. Reading the coordinator
metadata copy is sufficient and avoids the live-read
prerequisite that would block this proof.

The cleanly named separation matters: identity propagation
ships now, live remote read verification ships when remote
objects exist. Two different rows, two different proofs.

### P2 — Original node/store reporting

The diagnostic continues to report the *original* remote
`node_id` and `local_store_id` rather than the local
metadata-read override. This is critical — if it reported
the override, an operator inspecting the diagnostic would
incorrectly think the boundary replica is local-only.
Keeping the original identity preserves the meaning of the
diagnostic: "where does this boundary replica think it
lives?" not "where did we read the metadata from?"

Worth one doc sentence explicitly calling out that the
node_id/store_id columns are placement identity, not
read-source identity, so a future maintainer doesn't
"helpfully" make them match the read source.

### P3 — Tracker row scoping

Closure is correctly scoped to **identity/metadata proof
only**. Open items NOT claimed:

- live remote object reads;
- degraded replica freshness;
- boundary-replica placement diagnostics (that's 30959).

The Phase 12.7 lane is being closed item-by-item with
honest scope per row. Good.

### P3 — Four-ready-row assertion specificity

Asserting "four ready global identity rows" pins the
expected count exactly — meaning the fixture catches both
under-counting (lost identity) and over-counting (duplicate
identity) regressions. Cleaner than ">= 4" or ">= 1". Keep.

### P3 — Boundary replica count = 1

The fixture uses `boundary_replica_count = 1`, which proves
single-replica identity span. Worth pinning a follow-up to
verify identity span with `boundary_replica_count > 1` if
the production target supports >1 — multi-replica identity
is a *different* invariant (each source identity maps to N
replica placements, all sharing global vec_id) and a
1-replica fixture doesn't prove it.

If `boundary_replica_count = 1` is the v1 maximum, ignore.

## Reviewer recommendation

Accept. Right boundary (metadata, not live read), right
identity reporting (original placement, not override), right
fixture shape (exact-count assertions). Closes the row
honestly.
