## Answers to the packet's review questions

- **Transport fixture models remote statement timeout at
  the right layer?** Yes. The failure originates inside
  the remote PostgreSQL backend (its own
  `statement_timeout` machinery cancels the query mid-
  execution) and surfaces at the *transport* layer
  through SQLSTATE `57014`, before any candidate row is
  returned. Exercising it at the transport-probe layer
  matches the production code path from packet 30738.
- **Strict fail-closed and degraded skip aligned with
  the Stage E matrix?** Yes. Strict surfaces
  `remote_transport_failed` with category
  `remote_statement_timeout` (transport-stage status,
  packet 30738 category); degraded surfaces a
  `DegradedSkipped` dispatch with category preserved.
  Same shape as packet 30778's
  `simulated_network_partition` row — transport-stage
  failures consistently flow through the same state
  transformation.
- **Separate transport-fault fixture family the right
  shape for remaining transport rows?** Yes. The
  transport-fault family will absorb
  `connect_reset_mid_batch`,
  `remote_backend_termination`, and `remote_oom` as
  additional `--case` values — all four are transport-
  time failures that surface as
  `remote_transport_failed` with different categories.
  Keeping them in one family script means the
  setup/teardown logic for the slow/timeout production
  transport probe stays in one place.
