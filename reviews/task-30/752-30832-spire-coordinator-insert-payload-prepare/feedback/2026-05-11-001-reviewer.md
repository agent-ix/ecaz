# Review: SPIRE Coordinator Insert Payload Prepare (30832)

## Summary

Composes 30830 (remote prepare primitive) + 30831 (remote insert
endpoint). New `coordinator_insert_prepare_remote_tuple_payload`:

- Resolves the descriptor via the existing dispatch readiness gate.
- Builds remote SQL `SELECT * FROM ec_spire_remote_insert_tuple_payload(
  '<remote_index_regclass>'::regclass, '<json>'::jsonb,
  ARRAY['col1','col2']::text[])` via `coordinator_insert_remote_tuple_payload_sql`.
- SQL-literal-quotes the regclass, JSON payload, and column names
  using `quote_sql_literal` (single-quote doubling).
- Hands the constructed SQL to `coordinator_insert_prepare_remote_sql`
  from 30830, which runs it inside a `BEGIN; ...; PREPARE TRANSACTION`
  on the remote and registers the xact callbacks.

This replaces table-specific raw INSERT SQL with a stable
endpoint call before the transparent DML hook lands.

The four review-focus questions:
- Coordinator boundary correct (descriptor → remote SPIRE index;
  endpoint derives heap): yes.
- Remote SQL quoting (descriptor regclass, JSON payload, column
  list): identifier→ literal correctly; JSON-string single quotes
  are escaped; the endpoint then re-quotes column names as
  identifiers. Two-stage quoting is right.
- 2PC ordering preserved: yes — endpoint call → remote prepare →
  local placement staging.
- Test scope honest: confirms prepared endpoint composition + local
  placement staging, doesn't claim DML hook or read-after-insert.

## Findings

### P2 — JSON payload escaping on the wire

`row_payload_json: &str` is escaped via `quote_sql_literal`
(doubles `'`). PG's JSON literal `'<json>'::jsonb` parses single
quotes inside double-quoted JSON strings as literal `'`. The
combination works for ASCII payloads with only `'` quoting needs.
Concern: PG's libpq `batch_execute` sends the raw query string;
non-ASCII control characters in the JSON payload will pass
through. PG's JSON parser handles them but may reject `\x00`
NULs in JSON string values silently. Worth a one-line note that
the helper assumes the JSON payload was produced by a JSON
serializer that escapes control chars (which serde_json does).

Lower-risk alternative: use a parameterized query
(`client.execute_with_args`) instead of building the SQL string
inline. PG libpq parameter binding handles escaping at the protocol
level, no quoting needed. Worth migrating in the next slice.

### P2 — Empty column name rejected; what about empty payload?

`coordinator_insert_remote_tuple_payload_sql` rejects empty
`requested_columns` and empty individual column names. Doesn't
reject empty `row_payload_json` (caller could pass `"{}"`). The
endpoint then runs `jsonb_populate_record(NULL::heap, '{}'::jsonb)`
which returns a record with all NULL fields, attempts INSERT of
all NULLs, and either fails on NOT NULL constraints or inserts a
row of NULLs. Probably acceptable since the helper is internal,
but worth a comment that payload validation is the caller's
responsibility.

### P3 — Reuses the dispatch readiness check; descriptor generation drift

The function calls `coordinator_insert_dispatch_plan_row` then
immediately calls `coordinator_insert_prepare_remote_sql` which
re-invokes the same dispatch check internally. Two readiness
checks, slightly redundant. Either:
- Pass the dispatch row to `coordinator_insert_prepare_remote_sql`
  to skip the second lookup; or
- Document the redundancy as defensive (descriptor could change
  between calls within the same coordinator txn — though within
  one Rust function call that's effectively impossible).

Cosmetic.

### P3 — Constructed SQL is large and per-row

For 5 columns and a 2KB JSON payload, the constructed SQL string
runs ~3KB per INSERT. Each invocation builds it from scratch. For
v1 single-row INSERTs that's fine. For bulk-load this would benefit
from a prepared statement on the remote side or the parameterized
protocol shape mentioned above.

### P3 — ADR-069 docs note the composition

The new ADR text describes the layered shape (descriptor →
endpoint call → remote prepare). Good narrative thread for an
operator reading the ADR end-to-end.

## Reviewer recommendation

Accept. The parameterized-query migration (P2) is the single most
useful next-step optimization; it eliminates a class of escaping
correctness concerns entirely and shrinks the wire payload.
