## Answers to the packet's review questions

- **Runtime creation as the only batch-level failure the right
  adapter contract?** Yes. Runtime creation failure is a
  coordinator-side resource problem (out of file descriptors, etc.)
  that affects the entire batch, so a batch-level error is the
  right shape. Per-node failures must be rows so a single failed
  remote cannot suppress a ready remote — which is the C1
  isolation invariant this slice locks in.
- **Initial failure categories specific enough for the upcoming
  strict/degraded state machine?** Yes. The four categories map
  cleanly to C4 strict/degraded behavior:
  - `conninfo_parse_failed` → coordinator config bug; both modes
    must error (this is not a remote-side fault).
  - `connect_failed` → strict: fail closed; degraded: skip_node.
  - `statement_timeout_setup_failed` → strict: fail closed;
    degraded: skip_node (connection succeeded but is unusable).
  - `remote_query_failed` → strict: fail closed; degraded:
    skip_node, possibly with secondary classification by error code
    later.
- **Missing-socket + loopback fixture adequate coverage?** Yes for
  the isolation invariant. The fixture is the minimum-viable proof
  that one failed remote cannot suppress a ready remote at the
  transport layer. Stage E will need broader fault-matrix
  coverage, but this is the right C1 unit.

## P3 — informational

1. **`conninfo_parse_failed` classification.** This category covers
   coordinator-side parse failures of operator-provided conninfo.
   When C4 strict/degraded lands, consider whether to keep this
   as a strict-only error in *both* modes (since degraded skipping
   would silently drop a misconfigured remote without operator
   notice) or surface it alongside `skip_node` with a stronger
   `recommendation` field. Worth a one-line note in the C4 packet.

2. **Failure-reason payload size.** `failure_reason` will carry
   raw error text from `tokio-postgres` and `tokio::io`. Worth a
   tests/diagnostics convention on truncation or sanitization
   before C5 lands the AM scan integration, since AM scan results
   may end up in `EXPLAIN` or `pg_stat_*` paths where unbounded
   error text is unwelcome. Not blocking — natural to land in C4
   alongside the other strict/degraded contract work.
