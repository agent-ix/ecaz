# Review: SPIRE Local Capacity Targets (30952)

## Summary

Phase 12.9 docs slice publishing the local production-
readiness smoke target profile:

- 8 ready remotes per coordinator query;
- 256 remote leaf PIDs per query;
- 64 selected PIDs per remote;
- 1 concurrent distributed-read coordinator session;
- 8 concurrent remote-search dispatches across coordinator
  backends;
- 1 concurrent remote-search dispatch per remote;
- 1 concurrent coordinator-routed writer workload;
- 1 unit of work per remote (one read OR one prepared write
  branch).

Pinned as the active local capacity profile; raising targets
requires packet-local benchmark or contention logs.

## Findings

### P2 — "Local-only" claim discipline

The profile correctly *only* claims local production-
readiness smoke targets, not AWS or product-scale capacity.
This sits cleanly under the 30949 evidence label tier. The
"raise targets only with packet-local benchmark or
contention logs" rule is the load-bearing primitive that
prevents target inflation by reading.

### P2 — Coherence of 8 global / 1 per-node

The 8 dispatches global / 1 per-node is coherent for one
fully fanned-out 8-remote local query. Specifically:

- One coordinator session × 8 ready remotes × 1 per-remote
  dispatch ≤ 8 global dispatches. ✓
- A second concurrent session would need to either (a) wait
  for the first session's per-remote slots to free, or (b)
  hit the 8-global cap if disjoint remotes — but the 1
  concurrent distributed-read session line already prevents
  case (b).

So the per-node 1 IS the binding constraint, and the global
8 is a sanity ceiling. That's correct for v1.

### P2 — One-at-a-time read/write workload limits

Acceptable while:

- benchmark row open;
- write cancellation row open;
- placement contention row open;
- async write-dispatch row open (P9 in 30896).

The 1-writer / 1-reader-session rule is honest about what's
been verified rather than aspirational. When the four open
rows close, the targets can be raised with proper evidence.

The doc should make the dependency explicit: "These targets
will be revisited when {benchmark, write cancellation,
placement contention, async write dispatch} rows close." If
the dependency is implicit only, operators won't know the
ceiling is provisional.

### P3 — Required nonzero remote-search budget GUCs

Naming the required nonzero GUCs in the doc is the right
operator-facing primitive — local smoke fails closed without
them, which the 30954 degraded skip report can surface.
Worth confirming the GUC names listed match the actual GUC
names in the source (typo here = silent skip in production).

### P3 — Strict vs degraded overload behavior

Worth one sentence on what an operator should *see* when
hitting these caps:

- Strict mode: probably a SQLSTATE failure naming the cap.
- Degraded mode: probably a degraded-skip report row + a
  partial result.

If the doc names those expected behaviors, an operator can
test their own ceiling-hit assertions. If not, this would be
a useful one-liner.

## Reviewer recommendation

Accept. Target profile is honest about local scope, the
1-per-node binding constraint is coherent with the 8-global
ceiling, and the "raise targets only with evidence" rule is
the right gate. The P2/P3 items are wording sharpenings.
