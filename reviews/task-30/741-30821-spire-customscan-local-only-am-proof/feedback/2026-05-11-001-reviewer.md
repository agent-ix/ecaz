# Review: SPIRE CustomScan Local-Only AM Proof (30821)

## Summary

Tight focused proof: with seqscan disabled, indexscan enabled, and
no remote placements registered, an `ORDER BY embedding <#> $ LIMIT
1` against an `ec_spire`-indexed table plans as `Index Scan`, not
`Custom Scan (EcSpireDistributedScan)`. Closes the tracker bullet
that ADR-067's "local-only AM unchanged" promise was actually
preserved by the planner gate. The assertion checks both negative
(no CustomScan) and positive (Index Scan present) cases.

The three review-focus questions:
- Plan-shape assertion strong enough: yes for "AM path preserved"
  in the user-visible sense; the planner-internal hooks are
  exercised indirectly because the eligibility gate runs in the
  hook regardless of outcome.
- `enable_seqscan = off` with indexscan enabled is the right
  pressure: yes — without disabling seqscan the planner might
  legitimately pick seqscan over Index Scan on a tiny test table,
  which would mask whether the AM path is even available.
- Tracker update doesn't overstate remaining work: confirmed; ADR-069
  writes, production cost model, Stage E migration, and cleanup all
  remain explicitly open.

## Findings

### P3 — `Index Scan` substring match is broad

`plan.contains("Index Scan")` matches `Index Only Scan` and
`Bitmap Index Scan` too. Probably fine for this fixture (the only
ec_spire index path is the AM cursor scan), but a stricter assert
like `Index Scan using ec_spire_customscan_local_only_idx` would
nail down "the planner picked the ec_spire AM specifically, not
some other index path." Cosmetic; the negative assertion (no
CustomScan) is the load-bearing one and that one is exact.

### P3 — Two-row fixture is the smallest possible

The fixture inserts two rows. With `nlists = 2` and an index
build on two rows, the routing tree is one-level-deep with
trivial centroids. That is sufficient for the planner-gate
assertion (eligibility surface returns `local_only` regardless of
row count when no placements are remote), but doesn't exercise the
case where some non-leaf routing objects exist locally. Acceptable
since the gate doesn't depend on tree depth, just on placement
state. Worth a one-line note in the test confirming that the gate
is placement-state-driven, not topology-driven.

### P3 — No coverage of "local-only with `nprobe > 1`"

The fixture uses defaults; doesn't set `nprobe`. The CustomScan
path generation gate is the same regardless, but a follow-up
fixture that exercises a richer query shape (`SELECT
non_indexed_column FROM tbl WHERE ... ORDER BY embedding ... LIMIT
k`) would lock the contract that quals on non-indexed columns
don't accidentally wake up the CustomPath. Optional.

### P3 — Tracker bullet wording

The phase-11 tracker line for this fixture should explicitly call
out it is the *negative* proof (CustomScan does not capture
local-only scans), distinct from the *positive* proof in 30820
(CustomScan captures remote-placement scans). The fixture pair
forms the planner gate's full evidence; cross-link them in the
tracker so a future reader sees the proof on both sides at once.

## Reviewer recommendation

Accept. P3 items are polish for the next docs touch on the test or
tracker.
