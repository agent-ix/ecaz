# Review: SPIRE DML Planner Hook Scaffold (30849)

## Summary

Installs the planner hook chain in `_PG_init()`. The hook itself
is currently a pass-through: chains any prior `planner_hook` and
delegates to `standard_planner` when no prior hook exists. No plan
rewriting yet. Status surface
(`ec_spire_dml_frontdoor_hook_status()`) reports
`planner_hook_installed = true`, `query_shape_classifier_enabled =
true`, `plan_rewrite_enabled = false` with status
`pass_through_query_classifier_ready`.

The three review-focus questions:
- Pass-through chain/delegate behavior acceptable before rewrite
  slice: yes — installing the hook now lets reviewers see that
  the registration, chaining, and status surface all work before
  any plan is touched.
- Status surface advertises hook-installed-but-rewrite-disabled:
  yes — three explicit booleans and a status string distinguish
  the partial state.
- Next slice should bind relation metadata and replace eligible
  UPDATE/DELETE/PK SELECT plans with CustomScan: yes (closed by
  30850 for the metadata; CustomScan executor replacement is the
  next packet's work).

## Findings

### P3 — `static mut PLANNER_HOOK_INSTALLED: bool` ergonomics

Same pattern as the CustomScan registration. `_PG_init` is called
once per backend so the static-mut is safe in practice; Rust 2024
edition migration remains the long-term concern.

### P3 — Hook doesn't currently classify queries

The pass-through hook delegates without invoking
`classify_dml_frontdoor_query`. That's the right slice scope (only
install the chain), but worth a comment in
`ec_spire_dml_frontdoor_planner_hook` saying "classification +
rewrite lives in the next packet" so a future reader doesn't
think the hook is supposed to do more.

### P3 — `query_shape_classifier_enabled = true` even though the hook doesn't call it

The flag means "the classifier exists and could be called," not
"the hook calls it now." Worth either adding a finer boolean
(`query_shape_classifier_invoked_by_hook`) or documenting in the
status doc string that the flag tracks code presence, not runtime
invocation.

### P3 — `next_step` text repeated across status / frontdoor_plan rows

`"wire relation metadata and CustomScan executor replacement"` now
appears in both `ec_spire_dml_frontdoor_hook_status` and the per-
operation rows of `coordinator_dml_frontdoor_plan`. Fine — but
the next slice should advance these in lockstep so operators see
consistent status across both diagnostics.

## Reviewer recommendation

Accept. The hook chain is the right shape; rewrite logic should
land in a separate slice with its own evidence, per the
recommended scaffold direction.
