# Review: SPIRE DML CustomScan UPDATE Executor (30886)

## Summary

**MILESTONE — first transparent ADR-069 UPDATE executes through
the planner-hook plan-tree replacement.** `UPDATE
distributed_tbl SET col = ... WHERE id = $1` now actually
performs the remote update via the existing primitive and
reports the right rowcount.

Concrete shape:
- UPDATE target value expressions copied into `custom_exprs`
  alongside the PK expression
  (`custom_scan_dml_custom_exprs_from_plan_expr` builds the
  list).
- PK column metadata stored in `custom_private[4]` so
  UPDATE/DELETE replacement plans don't need to re-derive it
  from the scan relation context (which would require a heap
  open at exec time).
- UPDATE executor branch:
  - Builds `SpireDmlFrontdoorPrimitiveInvocation` from runtime
    state.
  - Converts const + parameter SET values to JSONB row payload.
  - Calls `ec_spire_forward_coordinator_update_tuple_payload(...)`
    via SPI exactly once.
  - Increments `estate->es_processed` from
    `remote_updated_count`.
  - Returns NULL — no tuple, just the rowcount side effect.
- Row-dependent SET expressions (e.g., `SET counter = counter +
  1` requiring the OLD row) are explicitly fail-closed for v1
  with a named error.
- PK SELECT candidate probe softens UPDATE/DELETE mode
  mismatches from `pgrx::error!` to `return None` so the
  rel-pathlist hook doesn't fight the planner-hook plan-tree
  replacement.

The four review-focus questions:
- UPDATE execution maps CustomScan state to coordinator update
  primitive without reintroducing a coordinator heap path: yes.
- `estate->es_processed` increment in the right place: yes — in
  the UPDATE access branch, after the primitive returns,
  before returning the (cleared) tuple slot.
- Const/parameter-only SET-value boundary as v1 fail-closed
  scope: yes — row-dependent expressions need the PG executor's
  expression context with NEW/OLD bindings, which would require
  a much richer SET-expression evaluator. Documented limitation.
- PK SELECT candidate probe silently declines UPDATE/DELETE
  mode mismatch: correct ownership split — DML planner-hook
  owns UPDATE/DELETE plan generation; PK SELECT rel-pathlist
  hook stays focused on SELECT shapes.

## Findings

### P2 — `dml_update_value_exprs: Vec<*mut pg_sys::Expr>` lifetime

The exec state holds raw PG `Expr*` pointers from the plan tree.
Those expressions live in PG's per-query memory context, which
outlives the scan, so the borrowing is safe. Worth a `# Safety`
comment on the field naming the lifetime guarantee, since a
future refactor that copies Vec to a different memory context
would be a use-after-free trap.

### P2 — Row-dependent expression rejection scope

The error message says "row-dependent SET expressions are not
supported in v1." Worth confirming the detection is robust —
specifically, that `SET col = $1::text || ' suffix'` (a
const/param expression with a string operator) classifies as
const/parameter, not row-dependent. The classifier should walk
the expression tree and reject only on `Var` references to the
target relation.

### P2 — UPDATE's `es_processed` increment via
`remote_updated_count`

`remote_updated_count` is `i64` from the primitive's return.
The increment is `es_processed.saturating_add(deleted_count)`
(in the DELETE path; assume same for UPDATE). For the v1 PK
equality shape this is always 0 or 1, so no overflow concern.
Worth pinning in the executor comment that v1 expects 0/1
rowcount and any other value would surface a primitive-side bug
loudly.

### P3 — PK column in custom_private[4]

The metadata layout grows from 4 to 5 string entries. Worth
updating the module comment from 30885 to enumerate the slots
explicitly: `[0] mode, [1] index_oid, [2] updated_columns_json,
[3] projected_columns_json, [4] pk_column`.

### P3 — `dml_update_value_exprs` empty for non-UPDATE modes

PK SELECT and DELETE construction skips populating
`dml_update_value_exprs`. Good — invocation builder branches on
mode, so empty is correct for non-UPDATE.

## Reviewer recommendation

**Accept the milestone.** Transparent UPDATE works. The P2
items (lifetime comment on `Expr*` Vec, row-dependent
classifier robustness) are worth folding into the row-
dependent-expression follow-up packet whenever that lands.
Until then the v1 const/param-only contract is documented and
fails closed.
