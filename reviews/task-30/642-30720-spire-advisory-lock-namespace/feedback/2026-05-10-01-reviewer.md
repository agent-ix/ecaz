# 30720 — SPIRE advisory lock namespace review

Validation pass on `c9afb529` (`Document SPIRE governance lock
namespace`). Accepts the namespace contract. Closes packet 30717
P2 and packet 30717 P3 (operator inspection hint).

## What I checked

- `plan/design/spire-libpq-executor-budget.md` line 55 ff. adds an
  `Advisory Lock Namespace` section recording:
  - **Global dispatch slot range:**
    `class_id ∈ 730000000..=730004095`, `object_id = 0`. Slot `n`
    maps to `(730000000 + n, 0)`.
  - **Per-node dispatch slot range:**
    `class_id ∈ 731000000..=731004095`,
    `object_id = bit_preserving_i32(node_id)`. Slot `n` for
    `node_id` maps to
    `(731000000 + n, bit_preserving_i32(node_id))`.
  - 4096 slots per axis is well above any plausible Stage C cap.
- The `pg_locks` inspection hint requested as 30717 P3 #2 is
  present at line 74:
  `pg_locks WHERE locktype = 'advisory' AND classid::bigint
  BETWEEN 730000000 AND 731004095`. This is exactly the operator
  query that should land in the Stage G runbook.
- ADR-058 has the same reserved-range contract at lines 55–56:
  `730000000..=730004095` (global) and `731000000..=731004095`
  (per-node). ADR is the durable home; the design doc carries the
  operator-facing detail.
- Phase 11 task file marks the namespace documentation closed
  without flipping any other Stage C box.
- Notes that "other extension features, operator scripts, and
  runbooks must not use these class ranges" — this is the
  cross-extension contract I was concerned about.

## Answers to the packet's review questions

- **Class ranges and object-id mapping precise enough for
  operator/runbook use?** Yes. `bit_preserving_i32(node_id)` is the
  right encoding because `pg_advisory_lock(int4, int4)` and
  `pg_try_advisory_lock(int4, int4)` take signed `int4` for the
  object id, so a `node_id: u32` must be reinterpreted bitwise to
  fit. Naming the function in the doc keeps that explicit instead
  of leaving operators to guess endianness/sign. Future review of
  the implementation should confirm the bit-preserving cast does
  not depend on platform-specific behavior — `i32::from_ne_bytes(
  node_id.to_ne_bytes())` is the portable form.
- **ADR-058 the right durable home, with the detailed range table
  in the budget design doc?** Yes. ADR-058 owns the boundary
  contract (no raw conninfo through SQL, advisory-lock namespace
  reserved); the budget design doc owns the operator-facing
  detail. Splitting them keeps the ADR readable while letting the
  operator-detail doc evolve as new slots get assigned.

## P3 — informational
