# Review: SPIRE CustomScan Cost Calibration (30976)

## Summary

Closes Phase 12.3 cost calibration + benchmark logs rows.
Calibrated against the 30975 loopback fixture. Constants in
`src/am/ec_spire/custom_scan.rs`:

- remote dispatch CPU units: `32.0` → `1024.0`
- merge CPU units: `4.0` → `0.5`
- projected tuple-width cost: `0.001` CPU units per
  projected byte (new term)

Goal: **conservative planner costing**, not a product
benchmark claim. Fixed remote dispatch dominates one-remote
loopback reads; monotonic cost growth preserved across
fanout, placements, output rows, projected width.

Measured (p50_ms): id_only k=10 31.674, k=100 42.521;
title_body k=10 31.775, k=100 42.454.
Modeled total cost: id_only k=10 2.838, k=100 4.977;
title_body k=10 2.842, k=100 5.019.

## Findings

### P1 — Conservative-costing framing is the right discipline

The framing matters more than the numbers. The packet
explicitly does NOT claim:

- the cost model predicts execution time;
- the constants are "correct";
- the calibration generalizes beyond the loopback fixture.

It claims only that the new constants:

- keep fixed remote dispatch as the dominant term for the
  loopback (which the measurement supports — fixed
  dispatch >> projected width contribution);
- preserve monotonic growth across the four cost axes
  (fanout, placements, rows, width);
- are conservative — the planner will overestimate
  CustomScan cost rather than under, biasing PostgreSQL
  toward not over-using SPIRE-fronted reads in plans where
  alternatives exist.

This is exactly what a planner cost should do under
uncertainty: **fail safe by overestimating**. A planner that
underestimated would push SPIRE-fronted reads into
contexts where their actual cost dominates and the wrong
path wins selection. Overestimating means SPIRE gets
chosen only when there's no obvious alternative, which is
the safe default.

### P1 — Constant magnitudes vs measurement

Cross-checking the constants against the measurement:

- **Remote dispatch 1024.0**: dominates the modeled total
  (k=10 model is ~2.84, of which 2.6 is startup — the
  remaining ~0.24 is per-row work). For the measured 31ms
  p50, even at PostgreSQL's nominal 0.01 ms/cpu-unit
  conversion, 1024 cpu-units corresponds to ~10ms — under
  the measured wall time, which is right for a cost that
  PostgreSQL planner uses comparatively (not as ms). The
  ratio matters, not the absolute units.
- **Merge 0.5**: small, since merge is local CPU work
  amortized over k. Matches the observation that k going
  from 10→100 only added ~10ms (mostly per-row work, not
  merge overhead).
- **Projected width 0.001/byte**: tiny per-byte cost. For
  a 31KB payload that's ~31 cpu-units total — a small
  contribution to the ~2.84 total. This matches the
  measurement showing **id_only and title_body have nearly
  identical p50** (31.7 vs 31.8 at k=10), meaning width
  contributes very little to actual time on this fixture.

The constants are calibrated to be **proportional to the
measured behavior**, not predictive. That's the right
shape.

### P1 — Projected width term is appropriately small

Per the measurement, id_only and title_body p50 differ by
~0.1ms at k=10 (31.674 vs 31.775) and 0.07ms at k=100
(42.521 vs 42.454). Width matters very little to actual
time on this fixture. A `0.001/byte` term over 31KB =
~31 cpu-units is small relative to fixed dispatch (1024)
and the per-row work. **Right magnitude.**

A larger width term would over-penalize wide projections,
biasing the planner against payload-heavy reads in cases
where (per the measurement) width genuinely doesn't matter
much. Keep the term small until evidence shows wider
payloads change the picture.

### P2 — Tracker closure scope

Closes the calibration row + benchmark logs row. Does NOT
claim:

- the constants are AWS-validated;
- this is a final cost model (could re-calibrate as more
  fixtures land);
- the cost model is good for production query selection
  beyond avoiding pathological cases.

That's the right scope. Phase 13 AWS work may need re-
calibration for product-scale fanout, network-bound
dispatch, etc. Pinning these as **conservative local
constants** keeps the door open without forcing a Phase 13
gate.

### P2 — Co-evolution with 30975 measurement

This calibration is grounded in 30975's measurement
fixture. That coupling is good for now (both packets cite
the same fixture, so the calibration is reproducible from
the same evidence) but creates a future maintenance
dependency: if 30975's fixture changes shape, this
calibration's basis shifts.

Worth a comment in `custom_scan.rs` near the constants
naming the **calibration evidence packet** (30975 +
30976) so a future re-calibration packet can find the
prior basis. Without this, a year from now someone will
look at `1024.0` and have no provenance for it.

### P2 — Test coverage

`cargo test custom_scan_cost` passes. Worth confirming the
test exercises the four cost axes (fanout, placements,
rows, width) — the monotonic-growth claim is load-bearing
and a test that asserts cost grows in each axis would be
a good regression. From the test name alone I can't tell
the coverage shape; if it's only checking constants, a
follow-up to add growth assertions would protect against
a future "simplification" breaking the monotonic property.

### P3 — PostgreSQL conventional cost units

Worth a comment in the constants noting whether `1024.0`
is in PostgreSQL's `cpu_tuple_cost`-style units (typically
0.01 default) or some SPIRE-native unit. If they're meant
to be comparable to PostgreSQL's other costs (so the
planner ranks SPIRE plans against alternatives correctly),
the units must match. If the comparison is meant to be
purely SPIRE-internal, that's a different framing.

From the conservative-overestimate framing it sounds like
SPIRE costs are meant to **rank above** alternative plan
shapes for SPIRE-eligible queries (so SPIRE wins selection
when applicable, but not preferred over heap-only plans
when a heap query exists). That's the right ranking;
worth confirming the units agree with PG's defaults so the
ranking actually works.
