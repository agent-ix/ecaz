# Planner cost model review

Replaces the gated `cost::gated_planner_cost_estimate(block_count)`
with a SPIRE-specific estimator that factors in `nlists`,
effective `nprobe`, `local_store_count`, storage format, rerank
width, hierarchy depth, routing child distribution, leaf
assignment count, and routing/leaf object bytes from the active
snapshot. PG18 `amgettreeheight` now reports the actual SPIRE
hierarchy depth instead of zero. Real planner change. No blocking
findings.

## Confirmations

1. Cost shape is sound: routing-side work as startup cost,
   selected-leaf candidate work as run cost. Matches IVF/HNSW
   convention.
2. `compute_amcostestimate` opens with `NoLock` (line 65) — the
   correct discipline for cost estimation called inside
   `set_baserel_pathlist_hook`. Closes with `NoLock`. ✓
3. Snapshot reads (`active_snapshot_diagnostics`,
   `index_hierarchy_snapshot`) are pure reads against the
   already-opened relation. No additional lock escalation.
4. Pure Rust estimator tests (probe count, recursive depth,
   local-store fanout) cover the parameter sensitivity.
5. `amgettreeheight` is feature-gated behind `pg18` and falls
   back to `hierarchy_snapshot` for older PG versions.

## Findings (non-blocking)

- **`active_snapshot_diagnostics` is called on every planner
  cost estimate.** That snapshot read traverses the manifest
  + placement directory pages. For a hot OLTP workload that
  re-plans frequently (prepared-statement invalidation, etc.),
  this could become noticeable. Worth profiling once the
  benchmark harness from 30622-30624 is hooked up; not a
  blocker.

- **Cost scale constants** (`SPIRE_ROUTING_SCORING_DIMENSION_SCALE`
  etc., lines 11-14) are magic numbers. They will need
  empirical calibration against the scale packet (30629)
  results. Today they are placeholders; once measurement
  evidence exists, these should ideally be either documented
  with the calibration source or pulled into reloptions for
  operator tuning. Forward-looking.
