## 1. Architecture review

### What's been built

Phase 7 has assembled a layered control plane on top of the
landed Phase 1–6 search engine. The layering is clean:

```
┌────────────────────────────────────────────────────────────┐
│ Operator surface (SQL-callable verbs)                      │
│   ec_spire_register_remote_node_descriptor                 │
│   ec_spire_persist_remote_epoch_manifest                   │
│   ec_spire_apply_remote_epoch_manifest    (validator only) │
└────────────────────────────────────────────────────────────┘
                            │
┌────────────────────────────────────────────────────────────┐
│ Diagnostic surface (~80 ec_spire_remote_* SQL functions)   │
│   per-step plans  → readiness → contracts → summaries      │
│   end-to-end gate: descriptor → secret → connection →      │
│   pipeline → send → receive → merge → heap → result        │
└────────────────────────────────────────────────────────────┘
                            │
┌────────────────────────────────────────────────────────────┐
│ Persistence (extension-owned tables)                       │
│   ec_spire_remote_node_descriptor                          │
│   ec_spire_remote_epoch_manifest{,_entry}                  │
└────────────────────────────────────────────────────────────┘
                            │
┌────────────────────────────────────────────────────────────┐
│ Production helpers (Rust, pure)                            │
│   plan_remote_search_fanout                                │
│   validate_remote_search_candidate_batch                   │
│   merge_remote_search_candidates                           │
│   merge_validated_remote_search_candidate_batches          │
│   SpireRemoteCountRollup, descriptor state registry        │
└────────────────────────────────────────────────────────────┘
                            │
┌────────────────────────────────────────────────────────────┐
│ Storage-node API                                           │
│   ec_spire_remote_search   (real, executable)              │
│   ec_spire_apply_remote_epoch_manifest (payload validator) │
└────────────────────────────────────────────────────────────┘
```

### Architecture strengths

- **Contract-first design.** Every executor action — descriptor
  resolution, secret resolution, connection open, pipeline
  enter, request send, receive validation, merge handoff —
  has a SQL-visible contract row before the executor exists.
  That gives operators an inspection surface and the executor
  itself can be a thin orchestrator over already-decided
  specs.
- **Drift-guard discipline is now uniform.** Tie-breaker (30573),
  degradation policy (30588), descriptor state CHECK (30603),
  search executor steps (30603), manifest executor steps
  (30603). Every contract claim is paired with an invariant
  test that fails if the implementation drifts. This should
  carry forward to the executor itself.
- **Conninfo opacity preserved.** The descriptor table stores
  only `conninfo_secret_name`. No raw connection string flows
  through SQL surfaces, bind plans, or contracts. This is a
  load-bearing design property — once the executor lands, it
  must look up the secret indirectly (vault / pg role / etc.)
  rather than receive a connection string from the SQL layer.
- **Status taxonomy is centralized.** Constants block in
  `remote_candidates.rs:23-65` plus the descriptor-state
  registry. Status precedence is derived through one helper
  (`SpireRemoteCountRollup::summary_status`).
- **Symmetry between search and manifest paths.** Both sides
  expose: request plan, request summary, dispatch plan,
  dispatch summary, bind plan, bind summary, executor work
  plan, executor work summary, executor readiness, receive
  plan, receive summary, gate summary, result summary. The
  parallelism makes the executor's job simpler — one
  orchestration pattern handles both.

### Architecture concerns

1. **Diagnostic surface size.** ~80 `ec_spire_remote_*` SQL
   functions. The pre-execution contract is over-specified
   relative to the executor that will consume it. This is
   defensible — it ensures the executor has zero design
   decisions left to make — but operationally noisy.
   Recommend documenting the "operator entry-point" subset
   (probably `coordinator_gate_summary`,
   `coordinator_result_summary`, descriptor catalog reads, the
   register/persist verbs) so users know which 6–8 functions
   to hit, not all 80.

2. **Pipeline redundancy compounds.** The composing summaries
   each transitively re-run the underlying coordinator-local
   pipeline. With the gate now reporting receive_count
   (30612) and the result summary inheriting receive
   (30613), the highest-level call chain runs the fanout
   planner ≥6 times, opens the AccessShareLock object store
   ≥3 times, runs leaf scoring ≥2 times. Self-healing on
   active-epoch changes (each call re-validates), but
   measurably wasteful. The
   `SpireCoordinatorPipeline::execute_once(...)` bundle
   suggested in earlier reviews is now overdue. Recommend
   landing it before the libpq executor — the executor itself
   will want a single bundle to orchestrate, not a tree of
   re-runs.

3. **Active-epoch advance race in persistence.** Both
   `ec_spire_persist_remote_epoch_manifest` and
   `ec_spire_register_remote_node_descriptor` read state
   outside their SPI write blocks and assume the coordinator
   epoch is stable. Self-healing on retry; documented but not
   guarded.

4. **No connection lifecycle / pooling design yet.** The libpq
   contracts specify a pipeline-mode dispatch but say nothing
   about: connection-per-query vs. pool, idle timeout,
   reconnect on transient failure, or PostgreSQL backend
   resource limits on the storage node side. These are
   executor-implementation decisions, but the contract surface
   should at least name the policy (e.g.
   `connection_lifecycle_policy: per_query | per_session |
   pooled` in a future contract row).

5. **`ec_spire_apply_remote_epoch_manifest` is a payload
   validator but named `apply_*`.** Calling it twice is
   idempotent only because nothing is written. When real
   apply lands, the function's behavior changes silently.
   Either rename to `ec_spire_validate_remote_epoch_manifest_payload`
   or add a remote-side applied-manifest catalog now. (Earlier
   feedback already raised this; tracking.)

6. **Authentication / secret lookup mechanism unspecified.** The
   descriptor's `conninfo_secret_name` is a name, not a
   resolution. Where does the secret come from? Postgres role
   password lookup? `postgres_fdw` user-mapping? Vault?
   This is the single biggest open architectural decision
   between the current state and a working executor. It
   doesn't need to be answered now, but it should be a
   labeled blocker on the Phase 7 task list.

## 2. Edge case review

Things I checked or want to flag, organized by surface:

### Empty / boundary indexes

- **Empty index (`active_epoch == 0`).**
  - `ec_spire_persist_remote_epoch_manifest` errors via the
    `manifest_decision != "emit_distributed_epoch_manifest"`
    guard (lib.rs:1926). ✓
  - `ec_spire_register_remote_node_descriptor` does not check
    that the index has any active epoch — it accepts
    descriptors against an empty index. Probably OK
    operationally (registering descriptors before the first
    publish makes sense), but worth a comment.
- **Single-node graph (top_graph).** Verified earlier; entry
  node ordinal validation handles `node_count == 1`.
- **boundary_replica_count = 0** — explicit `NoReplicaDedupeDisabled`
  scan plan; primary-only path bypasses HashMap allocation.

### Concurrency / TOCTOU

- **Active-epoch advance during persist.** Documented; not
  guarded. Self-healing.
- **Two concurrent `ec_spire_register_remote_node_descriptor`
  calls** for same `(coordinator_index_oid, node_id)` with
  same generation: both could see no row, both INSERT, one
  PK conflict and rollback. With *advancing* generation: the
  second loses on the WHERE clause, errors out cleanly. ✓
- **Descriptor delete during query.** The descriptor table
  has no DELETE protection / soft-delete. If an operator
  deletes a descriptor while a coordinator is mid-query (with
  the descriptor already loaded into the connection plan),
  the future libpq executor would attempt to dispatch
  against a removed descriptor. Worth either a foreign-key
  protection or an explicit "soft-delete" via state
  transition (set to `disabled`/`failed` rather than DROP).
- **Manifest entry FK + DELETE CASCADE** is correct: deleting
  a manifest header automatically removes its entries. But:
  manifest-only DELETE without index DROP leaves orphan
  catalog rows (no FK from `coordinator_index_oid` to
  `pg_class`). Operationally a `DROP INDEX` on the SPIRE
  index will leave behind catalog rows for the now-missing
  OID. **Real bug class** — see Plan to Complete §4.

### Failure modes

- **`disabled` / `failed` descriptor states** — fail-closed in
  the libpq connection loader since 30595/30596 hardening. ✓
- **`stale` placement state in degraded mode** — errors per
  `fanout_should_skip_placement` (degradation invariant test
  pins this). ✓
- **Mode mismatch (strict vs degraded request).** Storage-node
  endpoint errors before doing any work. ✓
- **Selected PID 0 / duplicate selected PIDs** — fanout
  planner rejects. ✓
- **Empty payload to apply function.** Returns
  `empty_manifest_entries` — soft error, not a panic. ✓
- **Malformed JSONB to apply.** Falls through to
  `invalid_manifest_payload` / `invalid_manifest_payload_format`
  / `missing_manifest_epoch` etc. ✓

### Cross-cluster / lifecycle

- **`pg_dump` + restore.** OIDs change. The
  `coordinator_index_oid` column in
  `ec_spire_remote_node_descriptor` and
  `ec_spire_remote_epoch_manifest` is an OID, so after
  restore those columns reference dead OIDs. Operators must
  re-register descriptors. Worth a doc note or a
  pg_extension cleanup hook.
- **Extension upgrade.** No upgrade path script exists for
  the new tables yet. When a future ecaz `0.1.x` -> `0.1.y`
  upgrade lands, `ec_spire_remote_node_descriptor` and
  `_manifest{,_entry}` will need an upgrade migration.
- **`DROP INDEX` on SPIRE index.** No cleanup of catalog
  rows — see Concurrency note above.
- **Backup of catalog tables.** They are regular Postgres
  tables, so they ride along with `pg_basebackup`. WAL
  replay should reconstruct them correctly. Worth verifying
  on the next big test pass.

### Numeric overflow

- All count-rollup math uses `checked_add` (verified earlier).
- `descriptor_generation` is `bigint` — max u64 / 2 worth of
  generation bumps. No realistic concern.
- `active_epoch` is `bigint`. Same.

### Vec-id semantics

- **Global vec_id is still a production blocker for cross-node
  merge.** Documented in `merge_remote_search_candidates`
  rustdoc. Until the global vec_id format lands, the merge
  helper is only safe when used against a single node's
  output (which is exactly what `coordinator_local` does
  today).

## 4. Plan to complete Phase 7

In rough dependency order:

### A. Pre-executor refactor (1 slice)

Land the `SpireCoordinatorPipeline::execute_once(...)` bundle
the prior reviews have been recommending. Composing summaries
should consume the bundle rather than re-run their inputs.
This is the right time — before the libpq executor itself
starts composing on top.

### B. Authentication / secret resolution decision (1 design slice)

Pick one of:
1. **Postgres user mapping**: reuse `postgres_fdw`-style user
   mapping. Lowest friction; users already know it.
2. **Vault / external secret store**: `conninfo_secret_name`
   resolves through an external lookup hook.
3. **In-extension secret table**: a separate
   `ec_spire_remote_node_conninfo` table with
   superuser-only access.

This decision blocks the executor. Recommend a design
checkpoint packet with the trade-offs.

### C. Lifecycle hooks (1 slice)

- Add a `DROP INDEX` event trigger or extension cleanup hook
  that removes orphan rows from
  `ec_spire_remote_node_descriptor` and
  `ec_spire_remote_epoch_manifest{,_entry}` keyed on the
  dropped OID. Without this, catalogs grow unbounded.
- Add a `pg_dump` extension hook (or a doc that operators
  must re-register after restore).
- Add an extension upgrade migration script for the new
  tables (currently they exist only on fresh
  `CREATE EXTENSION`).

### D. Libpq remote search executor (2–3 slices)

The actual implementation. Reads from existing surfaces:

- Inputs: `ec_spire_remote_search_libpq_executor_work_plan`
  (per-node action), the descriptor catalog (conninfo
  secret reference), the bind plan (per-bind values).
- Steps (already named in the executor step contract):
  1. resolve_remote_node_descriptor (already done by the
     loader)
  2. resolve_conninfo_secret_reference (depends on §B)
  3. open_libpq_connection
  4. enter_libpq_pipeline_mode (PQenterPipelineMode)
  5. send_remote_search_request (PQsendQueryParams per node)
  6. receive_remote_search_candidates (PQgetResult loop)
  7. validate via `validate_remote_search_candidate_batch`
  8. handoff to `merge_remote_search_candidates`
- Then a remote-row resolution path: each merged remote
  candidate's row_locator must be resolved by an additional
  per-origin-node lookup. This is its own slice.

### E. Libpq manifest publication executor (1–2 slices)

Same shape as §D but simpler:
- Send `ec_spire_apply_remote_epoch_manifest($1, $2, $3)` per
  node with the materialized JSONB payload.
- Receive `applied_entry_count` + `status`.
- Persist results into a new
  `ec_spire_remote_epoch_manifest_apply_result` table (also
  closes the apply-function-naming concern from 30602).

### F. Remote heap row resolution (1 slice)

When a coordinator query selects a remote candidate, a
follow-up query to the origin node is needed to fetch the
heap row. Today the row_locator is opaque bytes only the
origin node can interpret. Plan: a SPI-callable
`ec_spire_remote_resolve_heap_rows(...)` that takes batches
of opaque locators and returns heap rows.

### G. Cross-node integration tests (ongoing)

PG18 unit tests have taken the contract layer as far as it
can go without real sockets. The next gating test should be
a multi-PG-cluster fixture (one coordinator, two storage
nodes) that:

1. Builds a SPIRE index with placements on both remote
   nodes (using fixture-only descriptor catalog rewrites
   like the existing `debug_spire_rewrite_placement_node`).
2. Executes a real `SELECT` through the coordinator gate
   and asserts results match a reference single-machine
   build.
3. Kills one storage node mid-query, asserts strict mode
   fails closed and degraded mode skips with explicit
   diagnostics.

This is the test that actually proves Phase 7. Likely needs
infrastructure work (multi-cluster pgrx fixture, libpq mock
or lightweight in-process stand-in) — could justify its own
infra packet before §D.

### Estimated remaining slice count: ~8–12

- 1 pre-executor refactor (A)
- 1 secret-resolution design + 1 implementation (B)
- 1 lifecycle (C)
- 2–3 search executor (D)
- 1–2 manifest executor (E)
- 1 remote heap fetch (F)
- 1–2 multi-cluster integration test infra + fixture (G)

These can pipeline: the design checkpoint (B) and the
lifecycle (C) can land in parallel with §A; §G's
infrastructure can land alongside §D.

### Verdict

**Phase 7 has reached the point where every architectural
decision has been made, every contract has been written, and
every input the future executor needs has been specified.
What remains is the executor itself plus a small set of
lifecycle/operational gaps that I'd argue are first-class
Phase 7 work, not Phase 8.**

The contract-first approach has paid off: the executor's
implementation will be mechanical translation of existing
specs, and almost every failure mode the executor will
encounter has a pre-existing diagnostic row that already
distinguishes it. The remaining open question — secret
resolution mechanism — is genuinely an architectural
decision, not an oversight.

No blockers for the 30603–30617 stream to remain landed.
