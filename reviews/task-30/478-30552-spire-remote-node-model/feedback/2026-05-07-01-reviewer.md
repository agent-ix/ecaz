# Phase 7 remote node model design review

Design checkpoint is the right shape and stays correctly disjoint from
the libpq transport, remote DDL, and global vec_id rewrite work. No
blocking findings.

## Confirmations

1. `node_id = 0` reserved for the local coordinator, with nonzero IDs
   coordinator-scoped and detached from DSNs / OIDs / hostnames /
   pod identities, is the right separation. It keeps node identity
   stable across deployment churn — the alternative (binding identity
   to PG OID or DSN) would force an identity rewrite on every
   reconnect or instance recreation.
2. The four eligibility states (`active`, `draining`, `disabled`,
   `failed`) cover the operational lifecycle. Importantly, `draining`
   serving retained-active placements while rejecting new ones is the
   correct shape — without it a graceful node retire forces query
   errors during the drain window.
3. Keeping remote placements in the existing
   `pid -> node_id -> local_store_id -> object` map (vs. a separate
   remote-placement table) preserves one source of truth and reuses
   existing manifest/visibility surfaces. The coordinator's view of
   `local_store_id` and `store_relid` as diagnostic-only metadata is a
   sensible boundary.
4. Strict fail-closed + degraded `Skipped`/`Unavailable` diagnostic
   modes match FR-041/FR-042 semantics and the existing local
   degraded-skip behavior, so the remote case does not invent a new
   visibility model.
5. Deferred list (replicated partition objects, automatic discovery,
   remote DDL, rebalancing, row fetch, libpq retry policy, global
   vec-id rewrite) is correctly out-of-scope for this checkpoint.

## Findings

- **Global `vec_id` precondition is the load-bearing constraint.**
  The design references global `vec_id` as a future requirement for
  durable cross-node merge claims. The current merge helper (30554)
  dedupes by raw `vec_id` bytes — which is correct *only* when the
  bytes are globally unique. Today every node uses local-sequence
  vec_ids (`SpireVecId::local`), so two different rows on two
  different nodes can share the same `vec_id` bytes. The design
  document should call this out explicitly as a *production* blocker
  for multi-node merge, not just a future generalization, so a future
  reader does not assume the merge helper is safe to invoke across
  nodes today.

- **Stale-node conditions vs. retained-epoch window.** The design lists
  served-epoch + retained-epoch + extension version + node generation
  as inputs to staleness, but keeps retained-epoch serving deferred.
  Recommend the design state that until retained-epoch serving lands,
  the only valid `requested_epoch` for cross-node fanout is the
  coordinator's published active epoch — which matches what 30553
  enforces.
