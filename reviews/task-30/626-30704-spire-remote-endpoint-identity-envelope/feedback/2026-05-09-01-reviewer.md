## Answers to the packet's review questions

- **Is the 18-column endpoint envelope the right v1 shape before
  production coordinator enforcement?** Yes. The envelope is exactly
  what packet 30702's contract gate named as the missing wire-shape
  binding (fingerprint + opclass + scoring profile + status), and
  enforcement is a separate slice (30705).
- **Is it correct to validate protocol/version/nonempty identity
  fields in libpq decode while leaving `endpoint_status` enforcement
  for the next coordinator gate?** Yes. Splitting "decode validity"
  from "merge readiness" lets diagnostic SQL surface non-ready rows
  for operators while production merge fails closed in 30705.
- **Does the remote-node-model fingerprint documentation satisfy
  packet 30703 P2?** Yes — both P2 #1 (inputs + byte order) and
  P2 #2 (training-stat extension path = protocol bump) are now in
  the design doc, before the fingerprint becomes the stable wire
  contract.

## P3 — informational

1. **FNV-1a digest collision posture.** A 16-hex (64-bit) FNV-1a
   digest is fine for v1 because all inputs are protocol-controlled
   strings, not user input — collisions cannot be adversarially
   triggered. Worth a one-line note in the design doc that the
   digest is *not* a cryptographic identity and should not be relied
   on as one if a future input is operator-supplied bytea.

2. **Numeric input encoding edge cases.** Decimal ASCII is the right
   choice; the doc should say whether negative numbers are allowed
   (none of the listed inputs go negative, but stating "all numeric
   inputs are non-negative integers" closes the question). Same for
   leading-zero canonicalization on `pq_group_size = 0`.
