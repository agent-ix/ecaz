# Review: SPIRE Boundary Replica Placement Diagnostics (30959)

## Summary

Closes Phase 12.7 boundary-replica operator-diagnostic row.
`ec_spire_index_boundary_replica_placement_diagnostics(index_oid)`
groups placement health by global `vec_id` and reports:

- primary + boundary-replica assignment coverage;
- stale/unavailable/skipped replica counts;
- node span;
- status, degraded-mode action, recommendation.

PG18 fixture covers four states: missing, unavailable
(`skip_and_report`), skipped (`skip_and_report`), stale
(`fail_closed`).

## Findings

### P2 — Degraded-mode action policy

The split is conservative and correct:

- **Stale**: `fail_closed` — stale data is wrong data; in
  degraded mode, returning known-wrong replica answers
  silently is the worst failure mode. Failing closed forces
  operator response.
- **Missing**: `fail_closed` (implied for boundary-replica
  index without coverage) — same reasoning. Missing replica
  in a configured-replica index means a setup or rebuild
  failure that needs operator attention.
- **Unavailable**: `skip_and_report` — placement exists,
  remote is down. This is exactly the case degraded mode is
  designed for: continue with reduced fanout, surface the
  skip count.
- **Skipped**: `skip_and_report` — already-pre-dispatch-
  blocked. Same shape as 30954 degraded skip report.

The policy matrix is what it should be: don't degrade-skip
correctness failures, do degrade-skip availability failures.

### P2 — Inspection without weakening strict path

The diagnostic must read placement metadata permissively
enough to surface stale/unavailable/skipped states (which
the strict published-snapshot check would reject), without
weakening the strict check itself. The request says the
test-only placement rewrite helper now decodes manifests
permissively for fixture state simulation — that helper's
permissiveness must NOT leak into the production strict
check.

Worth confirming the permissive decode lives in
`debug.rs`/test-only paths and is not reachable from
production. From the file list it does (`src/am/ec_spire/root/debug.rs`).
Good — the asymmetry is "production = strict, diagnostic =
permissive read" which is the right separation.

### P2 — Grouping by global `vec_id`

Right primitive. Operators triaging boundary-replica health
think in terms of "is this vector well-replicated?" not "is
this placement row OK?" — grouping by `vec_id` means each
row answers the operator-level question. The per-vec_id
counts (stale/unavailable/skipped) preserve the underlying
placement detail without forcing operators to manually
group.

### P3 — Status + recommendation columns

Pairing `status` with a `recommendation` column is what
makes this diagnostic actionable rather than just
informational. An operator reading "stale, recommendation:
refresh manifest" knows the next step without consulting
the runbook. Worth confirming the recommendation strings
are the same vocabulary used in
`docs/SPIRE_DIAGNOSTICS.md` so grep-driven operator
workflow stays consistent.

### P3 — Test-only helper permissiveness scope

The "decode permissively to rewrite an already-invalid
state into the next invalid state" change to the rewrite
helper is necessary for the fixture matrix but worth a
comment in `debug.rs` calling out that:

- this is for fixture state staging, not production;
- production strict decode must remain unchanged;
- if production decode ever loosens, this helper should be
  re-evaluated (because they may now overlap and hide
  real bugs).

### P3 — Phase 12.7 lane closure status

With this packet, the Phase 12.7 boundary-replica trio
(identity 30957 + freshness 30958 + diagnostics 30959) is
closed. Combined with placement metadata (30956), strict
no-mix epoch (30955), and degraded skip report (30954),
the Phase 12.7 lane is now down to:

- ecaz multi-instance setup/teardown command;
- Stage E rerun against current CustomScan path.

Two open items, both ergonomics not feature gaps.

## Reviewer recommendation

Accept. **Closes the Phase 12.7 boundary-replica operator-
diagnostic row.** Conservative degraded-mode action policy,
correct strict/permissive separation, operator-friendly
vec_id grouping. With this and the prior trio, boundary-
replica health is end-to-end observable.
