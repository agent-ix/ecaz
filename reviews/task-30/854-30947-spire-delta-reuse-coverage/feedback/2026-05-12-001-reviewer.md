# Review: SPIRE Delta Reuse Coverage (30947)

## Summary

Phase 12.8 delta-decode reuse row closure. Documents and
fixture-asserts that selected-leaf delta decoding is shared
between:

- delete suppression (existing path);
- delta-insert candidate scoring (the path under coverage);
- remote local-heap and tuple-payload candidate resolution
  before origin-node payload lookup.

Plus existing unit coverage that
`load_delta_rows_for_routes(...)` reads each delta object
once.

## Findings

### P3 — Fixture scope is correct

`test_ec_spire_remote_search_local_heap_resolution_plan`
extension exercises:

- post-build delta row insert;
- active-epoch + selected-leaves recompute;
- `ec_spire_remote_search_local_heap_candidates(...)`
  returns three rows with one delta object and valid local
  locators.

That's the targeted assertion for "delta-decode reuse
extends to remote candidate resolution," and it does NOT
require tuple-payload compatibility (which is correctly
deferred to Phase 12.2). Scoping is right.

### P3 — "Read each delta object once" invariant

The unit coverage that `load_delta_rows_for_routes(...)`
reads each selected delta object once is the load-bearing
performance contract for this reuse boundary — without it,
"reuse" is just "called twice with the same key." Keep it.

If the implementation later changes to a per-call cache vs
single-pass collector, that test should fail loud rather
than silently degrade to N reads.

### P3 — Boundary description in design note

The design note phrasing — "selected delta rows are loaded
once per selected delta route set, then reused for delete
suppression and candidate scoring" — is the right level of
abstraction. Names *what's reused* (selected delta rows),
*the cache key* (selected delta route set), and *the two
consumers* (delete suppression, candidate scoring). A
future maintainer adding a third consumer has a clean spot
to extend the contract.

### P3 — Cross-packet wiring

Worth a one-line cross-reference in the design note to the
30945 multi-store diagnostic key so future readers tracking
"why does this collector group by route set?" can find the
placement-grouping rationale. Optional.

## Reviewer recommendation

Accept. Closes the Phase 12.8 delta-decode reuse row
without crossing into 12.2 tuple-payload territory. Fixture
+ unit coverage together prove both correctness (three rows,
one delta object) and the per-object single-read invariant.
