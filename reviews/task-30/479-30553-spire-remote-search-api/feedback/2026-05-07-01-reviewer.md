# Phase 7 remote search API review

`ec_spire_remote_search` provides the first executable storage-node
endpoint. Validation, fail-closed surfaces, and degraded skipping all
look correct. Notes below are non-blocking but worth addressing before
coordinator integration.

## Confirmations

1. SQL contract — `selected_pids`, `requested_epoch`, `top_k`,
   `consistency_mode` — matches the coordinator-fanout shape from the
   30552 design.
2. Fail-closed: wrong epoch, duplicate selected PIDs (`scan/candidates.rs:97-101`),
   PID 0, non-leaf selected objects, and consistency mode mismatch
   all error before any scoring runs. Good.
3. Degraded skip: in degraded mode `should_skip_placement` returns
   `Ok(true)` for unreadable placements and the loop drops them
   silently
   (`scan/candidates.rs::selected_leaf_routes_from_snapshot:104-109`),
   matching the design contract that degraded skips produce no
   candidates rather than reading stale objects.
4. Candidate identity fields (`served_epoch`, `node_id`, `pid`,
   `object_version`, `row_index`, `assignment_flags`, `vec_id`,
   `row_locator`, `score`) match the merge helper's input contract in
   30554.
5. Active-epoch-only serving is the right initial shape; the SQL
   function errors loudly if the requested epoch differs.

## Findings

- **Heap TID encoded as `row_locator` is opaque to the coordinator.**
  `root/hierarchy_snapshots.rs:90-94` (`remote_search_row_locator`)
  encodes the local heap TID directly. This is fine for v1 because the
  contract treats the locator as opaque bytes, but it means the
  coordinator cannot perform exact rerank fetch without round-tripping
  back to the storage node (which is the planned path). The design
  doc and the request both say row fetch is deferred — recommend
  recording in code (a doc-comment on `remote_search_row_locator`)
  that **the locator is opaque to coordinators** and must only be
  resolved by the originating node, so a future caller doesn't try
  to decode it locally.

- **`assignment_flags` returned as `i32`, internally `u16`.**
  `lib.rs:1593`: `i32::from(row.assignment_flags)`. The widening is
  fine, but this means the coordinator reads a 32-bit value and must
  remember the high 16 bits are always zero. Consider switching to
  `i16`/`smallint` so the SQL surface reflects the underlying type.

- **`top_k = 0` returns empty rows silently.**
  `root/hierarchy_snapshots.rs:108-110` short-circuits `top_k == 0`
  to `Ok(Vec::new())` while the SQL prelude only rejects negative
  values. That is consistent with the merge helper accepting an
  empty input, but the request says "validates nonnegative top-k" —
  it might be worth documenting (in `request.md` or a trace log)
  whether `top_k = 0` is a legitimate "ping the endpoint" pattern or
  should error like `requested_epoch = 0` does.

- **`AccessShareLock` symmetry.** `lib.rs:1584` closes with
  `AccessShareLock`, and the relation object store opens at the same
  level (`hierarchy_snapshots.rs:139`). Symmetric and correct. Worth
  noting if a future variant ever takes a stronger lock to wait for an
  in-flight publish, the close-mode would need to follow.

- **Top-graph routing is bypassed for remote search.**
  `collect_quantized_selected_leaf_candidates` uses the
  `selected_leaf_routes_from_snapshot` path, not the recursive or
  top-graph routers. That is correct — selection is the coordinator's
  job — but worth a comment that this storage-node endpoint never
  participates in graph-driven leaf selection. The graph route lives
  in the coordinator-side fanout planner that has not landed yet.
