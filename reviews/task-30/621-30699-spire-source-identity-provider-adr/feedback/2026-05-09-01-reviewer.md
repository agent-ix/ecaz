## P2 — close before ADR-063 transitions to ACCEPTED

1. **Add an Alternatives section.** ADR-063 has Decision, Required
   Invariants, Rationale, and Consequences but no Alternatives.
   Reviewers and future readers benefit from seeing the rejected
   shapes:
   - System-level identity (xmin/oid/system column) — rejected because
     not stable under logical replication or restore.
   - DDL identity via `WITH (source_identity_column = '<col>')` rather
     than INCLUDE — rejected because (rationale).
   - Generated columns vs included physical column — captured in passing
     under "Expression-derived identities" but worth promoting.
   - Hash-based canonicalization of text/composite keys — deferred,
     already explained in Rationale; restate in Alternatives so the
     defer is visible.

2. **Logical replication / REPLICA IDENTITY note.** "Stable across
   logical replication" is listed as an invariant, but stability under
   logical replication actually depends on the operator's `REPLICA
   IDENTITY` choice on the source table. Recommend an operator-facing
   note: when the table participates in logical replication and the
   source-identity column is not part of `REPLICA IDENTITY`, the
   operator is responsible for ensuring updates that preserve logical
   row identity also preserve the source-identity column value. This
   is not blocking, but should be in the ADR before it transitions
   from PROPOSED to ACCEPTED.

## P3 — informational

3. **Reloption change semantics.** ADR-063 covers DDL accept/reject
   for new indexes. It does not state behavior when an existing
   `ec_spire` index is REINDEXed after the reloption changes, or when
   a CREATE INDEX CONCURRENTLY uses the reloption while base writes
   are concurrent. Stage E covers fanout-side lifecycle but the DDL
   contract itself should say: changing `source_identity` requires a
   full rebuild because in-place transition of vec ID namespace is
   not supported.

4. **`amcaninclude = true` cost.** Enabling `amcaninclude` opens a
   small surface (planner may try to use INCLUDE in unrelated ways;
   pg_dump representation). The Stage A DDL validation should explicitly
   reject `amcaninclude`-enabled indexes that do not match the v1 shape
   (one vector key column, zero or one identity included column) so
   future user mistakes do not depend on opclass behavior.

5. **Three-class diagnostics naming.** Stage A names "local-only,
   global-capable but not yet remote-published, and global-writer
   active". Worth pinning the exact diagnostic strings now (e.g.
   contract-row values) so packets that follow do not invent
   competing names.
