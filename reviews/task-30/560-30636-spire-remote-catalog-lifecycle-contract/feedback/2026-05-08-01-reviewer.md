# Remote catalog lifecycle contract review

`ec_spire_remote_catalog_lifecycle_contract()` documents the
four lifecycle events: `pg_dump_restore`, `drop_index`,
`basebackup_wal_replay`, `extension_upgrade_0_1_0_to_0_1_1`.
Each row spells out OID stability, catalog risk, operator
action, cleanup surface, migration surface, status, and
recommendation. Closes my prior findings about pg_dump,
DROP INDEX, and extension upgrade lifecycle gaps. No blocking
findings.

## Confirmations

1. **`pg_dump_restore`** is correctly classified as
   `oids_reassigned` → `requires_operator_reregistration`.
   Cleanup surface points at the orphan helpers from 30634.
2. **`drop_index`** points at both the exact-OID cleanup
   (30639) and the orphan sweep, with `migration_surface =
   future_drop_index_hook` flagging the deferred event-trigger
   work.
3. **`basebackup_wal_replay`** is `supported` — physical
   backups preserve OIDs, so catalog rows replay correctly.
4. **`extension_upgrade`** is `supported_after_upgrade_script`
   thanks to 30637's migration entry.
5. Invariant test pins all four rows + key statuses.

## Findings (forward-looking)

- **No row for `ALTER INDEX ... RENAME` / `REINDEX`.** Neither
  changes the OID, so they should be implicitly safe. Worth a
  quick test that confirms remote catalog rows survive a
  REINDEX; not blocking.

- **No row for `DROP EXTENSION`.** Dropping the extension
  removes the catalog tables (and their rows). The lifecycle
  is implicit. Worth documenting that DROP EXTENSION is the
  "nuclear" cleanup that doesn't need orphan surfaces.
