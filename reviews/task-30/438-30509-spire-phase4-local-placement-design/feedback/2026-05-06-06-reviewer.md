### #1 REINDEX explicit rejection — packet 30538

`storage/relation_plan.rs:181–186` rejects multi-store REINDEX when
the auxiliary store relation already exists in `pg_class`. Error
message names REINDEX explicitly and points at the workaround.

Tests:

- `test_ec_spire_multistore_reindex_rejected` (`src/lib.rs:6756`)
  uses `#[should_panic(expected = "ec_spire multi-store REINDEX is
  not supported yet: auxiliary local store relation")]` against the
  exact rejection string.
- `test_ec_spire_singlestore_reindex_succeeds` (`src/lib.rs:6718`)
  proves single-store REINDEX still works end-to-end (REINDEX +
  ordered scan + result check).

Correctness check: the rejection condition is "auxiliary store
relation already exists in this namespace." That fires on REINDEX
because the multi-store build path tries to recreate the same
`ec_spire_store_<index_oid>_<store_id>` relation name and the
`<index_oid>` (rd_id) is stable across REINDEX. Single-store builds
never enter this code path because `relation_plan.len() == 1` is
short-circuited at line 148.

### #2 Constructor leak RAII fix — packet 30538

`OpenedRelationsGuard` (`storage/relation_store.rs:1097–1123`):

- `push` adds opened relations as construction proceeds
- `into_inner()` uses `mem::take` to swap in an empty Vec, leaving
  the guard's Drop a no-op on the success path
- Drop closes any remaining relations in reverse order

Both constructors (`for_index_relation_and_config` line 1126,
`for_index_relation_and_placements` line 1187) use the guard.

Correctness check on the success path: `opened_relations.into_inner()`
moves the Vec into `Self`. The guard's Drop runs after move on the
emptied guard (no-op). `Self`'s Drop continues to handle normal
scope-exit cleanup. No double-close, no leak.

Correctness check on the partial-failure path: any early `Err`
return drops the guard, which closes opened relations in reverse.
Same lock ordering as the success-path Drop on `Self`.

### #9 Autovacuum boundary check — packet 30540

`test_ec_spire_aux_store_relcache_disables_autovacuum`
(`src/lib.rs:6548`) — better than the timing-dependent autovacuum
launcher test I had originally suggested.

Instead of trying to trigger autovacuum and observe behavior (flaky
in tests), it asserts at the parsed-options boundary that
PostgreSQL's autovacuum launcher actually consults:

```rust
let enabled = (*rd_options.cast::<pg_sys::StdRdOptions>())
    .autovacuum
    .enabled;
assert!(!autovacuum_enabled);
```

That proves `autovacuum_enabled=false` is visible at the relcache
options layer, not just stored as raw text in `pg_class.reloptions`.
PostgreSQL's autovacuum launcher reads the parsed `StdRdOptions`,
so this assertion covers the actual decision boundary
deterministically.

This is a better test than I asked for. Right call.
