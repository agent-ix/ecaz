## #1 — REINDEX explicit rejection

**Problem.** A user running `REINDEX INDEX my_spire_idx` against a
multi-store SPIRE index today rebuilds the root index but leaves the
auxiliary store relations at the previous build's state. Internal
catalog dependencies do not participate in REINDEX. The result is a
silently broken multi-store index. The Phase 4 task tracker
acknowledges this as a follow-up but the live code path does not
reject it.

**Fix.** Detect the REINDEX path in the appropriate ambuild entry
(or whatever currently dispatches into the build coordinator with a
multi-store config) and `ereport(ERROR, ...)` with a clear message.
Single-store SPIRE indexes (`local_store_count = 1`) should still
allow REINDEX since they have no auxiliary relations to manage.

Detection: check whether the index already exists with auxiliary
store relations via `pg_class` lookup, or via the build-time flag
PostgreSQL passes through `IndexInfo->ii_Concurrent` /
ambuild-concurrent path. The standard pattern in other extensions is
to inspect `IsBootstrapProcessingMode()` and the relation's existing
`relfilenode` change.

**Verification.**

- New `#[pg_test]` `test_pg18_ec_spire_multistore_reindex_rejected`
  builds a 2-store index, runs `REINDEX INDEX <name>`, asserts the
  ereport message.
- `test_pg18_ec_spire_singlestore_reindex_succeeds` confirms the
  single-store path is unaffected.

**Effort.** ~30 minutes including tests.

## #2 — Constructor leak RAII fix

**Problem.** `SpireRelationObjectStoreSet::for_index_relation_and_config`
(`src/am/ec_spire/storage/relation_store.rs:1098–1143`) and
`for_index_relation_and_placements` (line 1145–1196) both build:

```rust
let mut opened_relations = Vec::<(pg_sys::Relation, LOCKMODE)>::new();
for descriptor in &config.stores {
    let relation = unsafe { pg_sys::relation_open(relid, lockmode) };
    if relation.is_null() {
        return Err(...);     // <-- partial Vec dropped, leaks k opens
    }
    opened_relations.push((relation, lockmode));
}
Ok(Self { ..., opened_relations })
```

`Vec<(*mut RelationData, i32)>` has no destructor. If the loop
opens k relations successfully then errors on k+1, the local Vec is
dropped without `relation_close` calls. Backend transaction abort
will eventually release locks via the resource owner, but the code
should not silently rely on that.

**Fix.** Wrap `opened_relations` in a guard struct that closes on
Drop, then `mem::take` (or a dedicated `into_inner`) to move the
inner Vec into Self on the success path, suppressing the guard's
Drop:

```rust
struct OpenedRelationsGuard {
    relations: Vec<(pg_sys::Relation, pg_sys::LOCKMODE)>,
}

impl OpenedRelationsGuard {
    fn push(&mut self, r: pg_sys::Relation, m: pg_sys::LOCKMODE) {
        self.relations.push((r, m));
    }
    fn into_inner(mut self) -> Vec<(pg_sys::Relation, pg_sys::LOCKMODE)> {
        std::mem::take(&mut self.relations)
    }
}

impl Drop for OpenedRelationsGuard {
    fn drop(&mut self) {
        for (r, m) in self.relations.drain(..).rev() {
            unsafe { pg_sys::relation_close(r, m) };
        }
    }
}
```

Both constructors switch to the guard, push as relations open, and
call `into_inner()` only on the Ok-construct path.

**Verification.**

- New unit test that exercises the partial-failure path. Hardest
  part is forcing `relation_open` to return null mid-loop. Easiest
  approach: use a placement directory containing one valid relid
  and one synthetic invalid one, run the constructor, assert Err
  return *and* that no spare locks are held (check via
  `pg_locks`).
- Alternatively, a lower-fidelity unit test that exercises the
  guard's Drop with mock relations is acceptable as a defensive
  smoke test.

**Effort.** ~1–2 hours.

## #3 — `include!` → `mod` conversion

**Problem.** Every directory under `src/am/ec_spire/` looks like a
real Rust submodule but is implemented via `include!` at:

- `src/am/ec_spire/mod.rs:59–67` (root/...)
- `src/am/ec_spire/storage.rs:22–34` (storage/...)
- `src/am/ec_spire/scan.rs:29–36` (scan/...)
- `src/am/ec_spire/update.rs:38–46` (update/...)
- `src/am/ec_spire/meta.rs:30–38` (meta/...)
- `src/am/ec_spire/build.rs:36–44` (build/...)

`pub(super)` resolves through the include site, not the file path,
so privacy boundaries the directory layout suggests do not exist.
Dependency graph is invisible. Cross-file references work without
explicit `use` because the files are the same module.

**Fix.** Convert mechanically:

1. For each parent file (`storage.rs`, `scan.rs`, `update.rs`,
   `meta.rs`, `build.rs`):
   - Replace each `include!("foo/bar.rs")` with `mod bar;` (and
     for items that need to be re-exported, add `pub(super) use
     bar::*;` or named re-exports).
   - Move `storage/bar.rs` etc. unchanged otherwise.
2. For `mod.rs`'s `include!("root/...")`: this is the trickier
   one because `root/` items are currently in the *ec_spire*
   namespace, not a `root` submodule. Two options:
   - **a)** Create a real `root` submodule, move items there, add
     `pub(super) use root::*;` at `mod.rs` to preserve external
     callers.
   - **b)** Keep the items at `ec_spire` level by inlining them
     into `mod.rs` directly (smallest semantic change).
   Recommend (a) — it matches the directory layout and matches
   the other directories' conversion.
3. After conversion, run `cargo check` and fix every privacy and
   import error that surfaces. Each one is a place where a file
   was reaching across what *should* have been a module boundary.
   Some will need `pub(super)` widened to `pub(crate)`. Some will
   need explicit `use`. Some will reveal genuine
   layering inversions worth flagging — note these but do not try
   to fix them in this packet.
4. Run the full test suite. The migration must be behavior-
   preserving.

**Anticipated issues.**

- Tests in `*/tests.rs` and `*/tests/*.rs` reach freely into their
  parent's private items today. After conversion, the test files
  need to be either `#[cfg(test)] mod tests { ... }` siblings of
  the parent module, or accept widened visibility for testable
  items.
- The `SpireObjectReader` trait in `storage/routing_delta.rs`
  (oddly named, see #5) is implemented across multiple files.
  Trait method visibility may need adjustment.
- Items that today are referenced cross-directory (e.g.,
  `scan/snapshot.rs` referencing `storage/local_store_set.rs`
  internals) will need explicit `super::storage::...` paths.

**Verification.**

- `cargo build` clean.
- `cargo clippy --all-targets --no-default-features --features pg18 -- -D warnings` clean.
- `cargo pgrx test pg18` full suite passes.
- `cargo test --lib` full suite passes.
- A grep for `include!` in `src/am/ec_spire/` returns nothing.

**Effort.** 1–2 days. The mechanical edit is fast; the surfacing
of privacy issues is what takes time. Land as one packet.

## #4 — `// SAFETY:` comments on unsafe blocks

**Problem.** Files with high unsafe density carry zero safety
documentation:

- `src/am/ec_spire/page.rs` — 97 unsafe blocks
- `src/am/ec_spire/storage/relation_store.rs` — 85 unsafe blocks
- `src/am/ec_spire/vacuum.rs` — 52 unsafe blocks
- `src/am/ec_spire/scan/relation.rs` — 52 unsafe blocks
- `src/am/ec_spire/root/snapshots.rs` — 45 unsafe blocks
- (other files with smaller counts — see
  `find ... | xargs grep -c "unsafe " | sort`)

Postgres FFI is unsafe by necessity, but Rust convention is one
`// SAFETY:` comment per unsafe block stating why the inputs satisfy
the called function's invariants.

**Fix.** Walk each unsafe block in the listed files and add a
single-line (or short multi-line) `// SAFETY:` comment immediately
above. Common patterns:

- `relation_open` — `// SAFETY: relid was validated against pg_class via lookup;
  caller holds the appropriate lockmode.`
- `pg_sys::PageGetItemId` — `// SAFETY: page is a buffer-pinned PostgreSQL
  page; offset is in [FirstOffsetNumber, PageGetMaxOffsetNumber].`
- `pg_sys::heap_create_with_catalog` — `// SAFETY: namespace_oid, owner_oid,
  tablespace_oid all read from the index relation's pg_class row;
  tuple_desc constructed by CreateTupleDescCopy and freed before
  return.`

Group adjacent unsafe blocks under one SAFETY block where they share
the same justification.

**Verification.**

- `grep -c "// SAFETY:"` on each touched file should approach the
  unsafe block count for that file.
- Add `#![warn(clippy::undocumented_unsafe_blocks)]` to the crate
  root or to the `ec_spire` module so future unsafe blocks are
  flagged. (Then fix the small handful that won't be in the
  initial pass.)

**Effort.** ~1 day. Mechanical-additive, no behavior change.

## #6 — `SpireError` typed enum

**Problem.** Every error path in `src/am/ec_spire/` returns
`Result<_, String>`. Callers cannot pattern-match on error
categories; logging cannot filter by class; future telemetry
cannot bucket failure modes.

**Fix.** Introduce a `SpireError` enum (suggest `thiserror` for
brevity, or hand-rolled `Display + Error` if you want to avoid the
dep). Categorize existing error sites into variants such as:

- `ConfigInvalid(String)` — placement directory inconsistencies,
  store config validation
- `EncodeFailure { context: &'static str, reason: String }` —
  partition-object encode/decode failures
- `RelationOpenFailed { relid: u32, reason: String }`
- `PlacementMissing { pid: u64, context: &'static str }`
- `IntegerOverflow(&'static str)` — checked_add / try_from cases
- `StateInvariantViolated(String)` — should-be-impossible states
- `Other(String)` — catch-all for the long tail; aim to whittle
  this down post-conversion

Convert error sites in tranches: do `meta/`, `storage/`, `scan/`,
`update/`, `build/` separately. After each, fix all call-site
match arms. Keep `String` at AM-callback boundaries that
ultimately go to `pgrx::error!` — convert at the boundary.

**Verification.**

- `grep -rn "Result<.*, String>" src/am/ec_spire/` should drop to
  near-zero outside the AM-callback boundary layer.
- Existing tests still pass (test assertions on error strings
  should be updated to match the variant's `Display` impl, or
  the test should pattern-match on the variant directly).
- `cargo doc` shows the enum and its variants.

**Effort.** 2–3 days. The cascading match arms are the time sink.
Land as one packet or as a series of submodule-scoped packets.

## #8 — Property/fuzz tests on partition-object encoders

**Problem.** V1 row-format leaf, V2 column-major leaf, V2 segment
chain, routing object, delta object, manifest bundles — these are
durable on-disk formats. Unit tests cover hand-crafted cases but no
property/fuzz coverage. A subtle decode bug at an edge case
(unaligned field, max-value field, empty assignment list) would
ship.

**Fix.** Two layers:

1. **Property-style round-trip tests** using `proptest` (already
   in the `dev-dependencies` for other crates in the repo, verify):

   ```rust
   proptest! {
       #[test]
       fn roundtrip_leaf_v2(rows in vec(arb_assignment_row(), 0..256)) {
           let original = SpireLeafPartitionObjectV2::from_rows(...rows.clone(), ...);
           let encoded = original.encode().unwrap();
           let decoded = SpireLeafPartitionObjectV2::decode(&encoded).unwrap();
           prop_assert_eq!(decoded.assignment_rows().unwrap(), rows);
       }
   }
   ```

   Add one for each codec: V1 leaf, V2 leaf, V2 segment chain,
   routing object, routing chain, delta object, epoch manifest,
   object manifest, placement directory.

2. **Fuzz target** under `fuzz/` (already exists in the repo) for
   each decoder taking arbitrary bytes and asserting "either
   returns Ok with a struct that re-encodes to the input, or
   returns Err — never panics, never UB":

   ```rust
   fuzz_target!(|data: &[u8]| {
       if let Ok(decoded) = SpireLeafPartitionObjectV2::decode(data) {
           let _ = decoded.encode();  // re-encode shouldn't panic
       }
   });
   ```

   Run for at least 5 minutes per target locally; integrate into CI
   if the fuzz lane is wired up.

**Verification.** New tests pass. No fuzz crashes after a 5-minute
local run per target.

**Effort.** 1–2 days for the property tests; fuzz targets are an
afternoon each.

## #9 — Autovacuum-enabled regression test

**Problem.** The autovacuum guard fix (#30537) is verified by
asserting `pg_class.reloptions @> ARRAY['autovacuum_enabled=false']`
in `test_ec_spire_populated_build_hash_routes_logical_store_set`.
That asserts the *reloption is present*, not the *behavior under
autovacuum*. PostgreSQL's autovacuum launcher honors the reloption,
so the catalog assertion is sufficient defensively, but a
behavioral test would be a stronger guarantee.

**Fix.** A `#[pg_test]` that:

1. Builds a multi-store SPIRE index.
2. Sets `autovacuum_naptime = 1`, `autovacuum_vacuum_threshold = 0`,
   `autovacuum_analyze_threshold = 0` for the test database.
3. Triggers conditions that would normally invite autovacuum
   (heavy update churn, low thresholds).
4. Waits a few seconds for the autovacuum launcher to wake.
5. Asserts: index is still scannable, no panic in logs, and
   `pg_stat_user_tables` shows the aux relations have
   `last_autovacuum IS NULL` (proving they were skipped).

This is moderately tricky because controlling autovacuum timing in
a test is nondeterministic. A robust shape: poll
`pg_stat_user_tables.last_autovacuum` for the *root index relation*
on a regular heap table to confirm autovacuum did fire somewhere,
then assert the aux relations stayed `NULL`.

**Verification.** Test passes deterministically across at least 10
local runs.

**Effort.** ~half day, plus iteration to make the timing reliable.
