# Review: SPIRE UPDATE Primitive Edge Coverage (30845)

## Summary

Tightens fixture coverage for
`ec_spire_forward_coordinator_update_tuple_payload(...)` before the
transparent UPDATE front door consumes it. Three covered cases:

- Multi-column local `node_id = 0` UPDATE: proves the local-heap
  branch updates two non-embedding columns in one helper call.
- Missing-placement fail-closed: proves a `pk_value` not present in
  the placement directory surfaces the explicit error
  `ec_spire coordinator update placement row is missing` instead
  of silently no-oping.
- Existing remote-owned UPDATE forwarding fixture preserved.

All three live under one `cargo test forward_coordinator_update`
filter, so the regression suite for the UPDATE primitive runs
together.

## Findings

### P3 — Missing-placement error wording

`ec_spire coordinator update placement row is missing` is fine for
diagnostics but surfaces as a free-text error string. Worth pinning
either the SQLSTATE the helper raises (so callers can catch on it)
or, if it's just `XX000`, a stable error-code constant in the Rust
side that the planner-hook front door can re-raise consistently.

### P3 — Multi-column UPDATE asserts both columns updated

The fixture should assert both column values after the helper call,
which it does. Worth confirming that the test also re-reads from
the placement directory to confirm placement metadata wasn't
mutated (UPDATE non-embedding shouldn't touch the placement row).

## Reviewer recommendation

Accept.
