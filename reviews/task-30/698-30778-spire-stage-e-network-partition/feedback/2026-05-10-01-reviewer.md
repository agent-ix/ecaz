## Answers to the packet's review questions

- **`fault-pg18 --case simulated_network_partition` the
  right CLI shape for extending the rest of packet 30770's
  fault matrix?** Yes — definitively. Matches my 30776
  P3 #1 recommendation (one subcommand per matrix family,
  one `--case` flag for the matrix-row key). Future
  packets can add `--case epoch_mismatch`,
  `--case backend_terminated`, etc. without changing the
  subcommand surface. Consistent with the per-row evidence-
  file convention from packet 30773.
- **Transport-summary fixture draws the right boundary for
  this first row, or should the next row move closer to
  full operator/heap materialization?** Transport-summary
  is the right boundary for this row. `connect_failed`
  fires before any candidate is received, so heap
  materialization is irrelevant to the failure mode being
  tested. A later row whose failure happens *after* receive
  (e.g. fingerprint mismatch on returned rows) is the
  right place to extend the fixture into receive +
  heap-resolution stages.
- **Strict/degraded counters strong enough to prevent
  fail-closed vs skip-node policy regressions?** Yes. The
  observed-summary strings contain seven distinct counter
  fields (dispatches / ready / completed / failed /
  failure_category / degraded_skipped /
  first_degraded_skip_category / next_step / status), so
  any regression that conflates strict and degraded — or
  changes the strict failure category, or breaks the
  degraded skip counter — will fail the assertion. The
  pass marker plus the recorded counter strings give the
  fixture a strong negative-detection surface.
