# Review: SPIRE DML Replacement Decision (30859)

## Summary

Adds the typed decision row that the executor-replacement slice
will consume. `SpireDmlFrontdoorReplacementDecisionRow` carries
`(target_relation_oid, index_oid, supported, operation, kind,
status, custom_scan_mode, primitive, error, hint, next_step)`,
which composes:
- relation context (catalog-backed) from 30855;
- query shape classification from 30847/30848;
- operation→primitive mapping for the three supported v1 shapes;
- explicit `custom_scan_mode = "none"` + ADR-069 planner-error
  next_step for unsupported shapes.

Mappings:
- `update_non_embedding` → `coordinator_update_tuple_payload` →
  `ec_spire_forward_coordinator_update_tuple_payload`
- `delete` → `coordinator_delete_tuple_payload` →
  `ec_spire_prepare_coordinator_delete_tuple_payload`
- `pk_select` → `coordinator_pk_select_tuple_payload` →
  `ec_spire_forward_coordinator_select_tuple_payload`

Bonus fix: actual `UPDATE ... FROM` queries are now recognized as
joined and rejected via the `resultRelation` extraction, with a PG18
fixture proving rejection.

`ec_spire_dml_frontdoor_replacement_sql(sql text)` is the operator-
facing diagnostic so the decision row is inspectable from SQL.

The three review-focus questions:
- Each supported v1 shape maps to the correct CustomScan mode and
  primitive: yes — names align with the existing primitive
  function signatures.
- Unsupported shapes preserve the fail-closed ADR-069 planner-error
  path: yes — `custom_scan_mode = "none"` plus `next_step` text
  pointing at "raise ADR-069 planner error" guides the executor
  slice.
- UPDATE/DELETE `resultRelation` extraction matches PG analyzed
  trees and rejects joined DML: yes — fixture covers
  `UPDATE ... FROM` rejection.

## Findings

### P3 — `relation_context_error` returns a synthetic decision row

When the catalog loader returns Err, the helper returns a row with
`kind = "relation_context_error"`, `index_oid = InvalidOid`, and
the ADR-069 hint. The fail-closed guard (30861) then needs to
distinguish "this isn't a SPIRE table" (pass through) from "this
is a SPIRE table but I couldn't read its context" (planner-error).
Worth confirming the guard's `decision.index_oid != InvalidOid ||
decision.kind == "relation_context_error"` predicate handles the
context-error case correctly even with `index_oid = InvalidOid`.

Looking at 30861: yes, the predicate explicitly OR's on
`decision.kind == "relation_context_error"`. Correct shape.

### P3 — `custom_scan_mode = "none"` for unsupported

The mode string acts as a discriminator for the executor slice.
Worth pinning in ADR-069 the full enumeration of `custom_scan_mode`
values (`coordinator_update_tuple_payload`,
`coordinator_delete_tuple_payload`,
`coordinator_pk_select_tuple_payload`, `none`) so a future packet
that adds a new mode knows the contract.

### P3 — `dml_frontdoor_classify_query_with_relation` shared helper

Refactors classification into a shared helper used by both the
replacement decision and the existing classify path. Avoids drift
between the two callers — exactly the right pattern.

## Reviewer recommendation

Accept. The decision shape gives the executor slice a single typed
input, which is the cleanest possible boundary for the rewrite
work.
