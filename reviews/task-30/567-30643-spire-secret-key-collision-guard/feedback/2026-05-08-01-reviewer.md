# Secret key collision guard review

Closes my 30638/30642 finding about secret-name collisions
(e.g. `node-1` and `node_1` mangling to the same env-var key).
Implementation is clean and concurrency-safe. No blocking
findings.

## Confirmations

1. **Shared lookup-key derivation.** The mangling logic moved
   into one helper
   (`remote_conninfo_secret_provider_lookup_key`) used by
   both registration validation and the status surface.
   Single source of truth — drift between the two paths is
   structurally impossible.
2. **`LOCK TABLE ec_spire_remote_node_descriptor IN SHARE ROW
   EXCLUSIVE MODE`** (`lib.rs:1443`) is the right level. SHARE
   ROW EXCLUSIVE conflicts with itself but allows concurrent
   SELECTs. Two concurrent registrations cannot both pass the
   collision check before either inserts.
3. **Self-exclusion in the collision scan.**
   `node_id <> $2::integer` (line 1453) excludes the row being
   re-registered, so updating an existing descriptor doesn't
   false-positive against itself.
4. **Same secret name across nodes is allowed** (line 1473-1475
   continues past matching names). Only different names that
   alias to the same lookup key are rejected. Matches the
   review-focus question in the request.
5. **Contract validators updated** from
   `must_be_nonempty_secret_reference` to
   `must_be_nonempty_noncolliding_secret_reference`. The
   contract claim now matches the runtime check.
6. PG18 test
   `test_ec_spire_remote_secret_key_collision_rejected`
   exercises `node-1` then `node_1` and asserts the second
   call panics with the shared lookup key in the error
   message.

## Findings

- **None.** Tight slice; closes my finding cleanly.
