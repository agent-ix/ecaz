# Target/request/readiness summary review

Joins target/request rows from 30561 to the per-node descriptor view
from 30562. The descriptor-aware status precedence is correct. No
blocking findings.

## Confirmations

1. Status precedence is right: degraded skipped placements stay
   `degraded_skipped` regardless of node descriptor state, executable
   targets inherit node-readiness (`requires_remote_node_descriptor`)
   before transport readiness, and local targets remain `ready`.
   That ordering ensures a node-descriptor failure does not silently
   reclassify a degraded placement, and a degraded placement does not
   mask a missing remote descriptor.
2. Target-readiness rows preserve the 30561 selected-PID arrays per
   target; the join is on `node_id` only, so PID order within a
   target is stable.
3. The readiness summary aggregates local/remote/skipped/missing
   descriptor/blocked counts that line up with the per-row contract.
4. No conninfo or libpq execution surfaced.

## Findings

- **Status precedence semantics — verify with a test.** The status
  precedence "degraded_skipped > requires_remote_node_descriptor >
  ready" is asserted via individual PG tests (mixed local/remote,
  degraded-skipped, blocked) but not against a single fixture that
  combines all three at once. A future change that flips the
  precedence (e.g. by checking node readiness before placement
  state) would still pass the existing tests. Suggest one targeted
  test that drives a fixture with one local available leaf, one
  remote-node leaf needing descriptor, and one degraded-skipped leaf
  in the same plan, asserting the three status strings appear in
  the expected per-row positions.
