# 30764 — SPIRE standalone pgrx loader stubs review

Validation pass on `afd97885` (`Fix standalone PG18 test loader
stubs`). Closes the recurring `SPI_finish` loader issue from
packets 30753 / 30756 / 30757 / 30758 / 30761 / 30762.

## What I checked

- `csrc/standalone_pg_backend_stubs.c` exports the
  additional pgrx/PostgreSQL symbols needed for direct
  `cargo test --features pg18` to load.
- Symbol classification:
  - **Inert stubs** for safe helper symbols: memory-context
    globals, minimal memory-context allocation, transaction
    id helpers, binary coercion, type formatting, SPI
    globals. These return reasonable defaults so pure-Rust
    tests can compile and link.
  - **Panicking stubs** for SPI execution/accessor
    functions: an explicit "unavailable outside a
    PostgreSQL backend" panic so direct tests cannot
    silently fake database execution and produce false
    positives.
- The split is the right discipline — anything that touches
  real backend state panics; only inert helpers (counters,
  memory contexts, type info) return defaults.
- Validation chain in the request:
  - `cargo test production_scan_result_stream_am_outputs
    --no-default-features --features pg18` runs cleanly
    after stubs (closing the recurring loader issue);
  - `cargo test row_materialization_contract --no-default-features
    --features pg18` similarly clean;
  - `nm -u target/debug/deps/ecaz-...` returns no unresolved
    symbols matching the previously missing set after the
    focused build.
- `cargo check` with both `pg18` and `pg18 pg_test` features
  pass.
- `pg_test` lane and `cargo pgrx test` continue to be the
  authoritative path for backend-facing tests; this fix
  restores the *direct* pure-Rust path for non-backend
  tests, not the backend lane.

## Answers to the packet's review questions

- **Newly exported symbols narrow enough for the standalone
  pure-Rust test lane?** Yes. The inert stubs cover only the
  symbols that pure-Rust tests' transitive dependencies
  reach for during initialization (memory contexts, type
  metadata, etc.) — not the wider PostgreSQL backend
  surface. A test that needs anything more than this
  surface is, by construction, a backend test that belongs
  in `cargo pgrx test`.
- **Correct for SPI execution/accessor stubs to panic
  instead of returning fake data?** Yes — definitively. A
  stub that returned fake SPI results would let a pure-Rust
  test exercise database paths that the test environment
  cannot actually validate, producing silent false
  positives. A panic with explicit wording forces the
  developer to either:
  1. recognize the test belongs in `cargo pgrx test`, or
  2. refactor the code-under-test to not require SPI in
     the pure-Rust scope.
  Both are correct outcomes.
- **Does this leave the separation clear between direct
  `cargo test` for pure-Rust behavior and `cargo pgrx test`
  for backend behavior?** Yes. The discipline is now: if
  your test panics with "unavailable outside a PostgreSQL
  backend", move it to the pgrx lane; if it doesn't, it
  works in both lanes. That's the right invariant — pure-
  Rust tests are now actually pure-Rust.
