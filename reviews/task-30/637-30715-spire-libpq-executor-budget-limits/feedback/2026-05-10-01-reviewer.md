## Answers to the packet's review questions

- **Row-granular budget blocking is the right Stage C behavior
  instead of partial PID truncation?** Yes. Truncation would silently
  serve a smaller answer than requested, which is a correctness
  hazard — the coordinator would merge an under-fan-out result and
  the operator would have no visible signal of degraded recall.
  Marking the row `remote_executor_overload` /
  `blocked_before_dispatch` keeps strict mode fail-closed and lets
  the operator either lift the budget or reduce the request; both
  are safer than silent truncation.
- **`remote_executor_overload` precedence through dispatch, secret,
  readiness?** Right. The block is pre-secret (so a secret-not-found
  for an over-budget remote does not pollute diagnostics) and
  pre-dispatch (so we never open a libpq connection we are about to
  reject). Same shape as the 30711 capability gate.
- **Timeout settings scoped correctly as first connection-helper
  enforcement, with global concurrency/cancellation still open?**
  Yes. Connect/statement timeouts are a per-call discipline that
  belongs at the connection helper; global concurrency caps and
  cancellation propagation need executor-state plumbing that is
  Stage C ongoing work. Splitting them is correct.

## P3 — informational

1. **Default values for the three caps.** If the GUC defaults are
   `0` (unbounded), the budget surface is opt-in for operators.
   Worth a one-line note in `spire-libpq-executor-budget.md` (and
   eventually the Stage G runbook) stating that the default is
   unbounded and naming a recommended starting value (e.g.
   `max_nodes = 8`, `max_pids = 256`, `max_pids_per_node = 64`)
   tied to the capacity-target work mentioned in packet 30694 P3 #9.

2. **Interaction with capability gate.** When a request is
   capability-blocked on one remote and budget-blocked on another,
   the `next_blocker` precedence in the receive-attempt row should
   be deterministic. The current implementation gates capability
   first, then budget, then identity preflight; that order is sane
   because capability failures are cheaper to evaluate. Worth
   pinning the precedence in the design doc so future surfaces
   (Stage E fault matrix, Stage G harness) can rely on the
   ordering.
