# Review: SPIRE CustomScan Cost Model (30827)

## Summary

Replaces the placeholder `startup=0, total=LIMIT` cost from packet
30809 with a calibrated planner model that accounts for the work
the CustomScan path actually performs:

- **routing traversal** — `min(remote_placements, 64) *
  cpu_operator_cost`;
- **per-remote dispatch startup** — `fanout * 32 * cpu_operator_cost`;
- **bounded heap rerank** — `min(LIMIT * fanout, rel_rows) *
  (cpu_tuple_cost + cpu_operator_cost)`;
- **merge CPU** — `LIMIT * log2(fanout) * 4 * cpu_operator_cost`;
- **tuple delivery** — `LIMIT * cpu_tuple_cost`.

`startup_cost = routing + dispatch`, `total_cost = startup + heap_rerank
+ merge + tuple_delivery`. The remote-fanout count and remote
available placement count flow in via the eligibility row that the
hook already reads, so no extra catalog work. The unit tests cover
two structural invariants: cost increases with remote fanout, and
the eligibility-driven inputs are wired through. The PG18 EXPLAIN
test still selects `Custom Scan (EcSpireDistributedScan)` for an
eligible query, so the model isn't over-pricing the path.

The three review-focus questions:
- Cost terms cover the intended categories without over-penalizing
  small-LIMIT queries: see findings; mostly yes, with one concern
  about the dispatch-startup term scaling against any eligible
  query regardless of LIMIT.
- Fanout/output-row monotonicity tests are right unit coverage:
  yes — they lock the structural shape without claiming exact
  numbers.
- `next_step = "add ADR-069 write path"`: accurate.

## Findings

### P2 — Per-remote dispatch startup penalizes small-LIMIT queries hardest

`remote_dispatch_cost = fanout * 32 * cpu_operator_cost` lands
entirely in `startup_cost`. For LIMIT 1 against a 4-remote fanout,
that's `4 * 32 * 0.0025 = 0.32` units of pure startup. Compared to
a competing index scan whose startup is typically `< 0.1`, this
makes CustomScan look like a worse choice for small LIMITs even
when remote dispatch latency is the dominant real cost. The
relative-pricing direction is correct (remote fanout *should* cost
more than a local index lookup at startup), but `32 *
cpu_operator_cost` is small enough that the model probably won't
shift planner choices much. Worth either:
- Calibrating the constant against measured dispatch latency in the
  Stage F harness (which is the planner-cost packet's natural
  follow-up); or
- Documenting that `CUSTOM_SCAN_REMOTE_DISPATCH_CPU_UNITS = 32` is
  a structural placeholder, not a benchmark-derived value, so the
  next packet that touches it knows the dial is symbolic.

### P2 — Routing traversal is bounded at 64 placements

`CUSTOM_SCAN_ROUTING_SCORE_BOUND = 64.0` caps the routing-traversal
cost regardless of how many remote placements exist. For an index
with thousands of remote placements that does coarse pruning before
fanout, 64 is fine as a coarse upper bound. For a degenerate case
where the planner sees an index with very few remote placements
(say 2), the cost still computes correctly (`min(2, 64) = 2`).
Worth a one-line comment explaining the 64 bound — without context,
a reader might think it's the routing tree fanout cap rather than a
cost-estimation bound.

### P2 — `merge CPU` uses `log2(fanout).max(1.0)`

The `.max(1.0)` floor means the merge term is always at least
`output_rows * 4 * cpu_operator_cost`, even with fanout = 1 (where
log2(1) = 0). That's defensible: even single-remote merge has some
CPU cost. But "fanout = 1" should mean *no merge* (just stream the
single remote's output through). Either drop the floor and let the
zero-merge case zero out, or document that the floor models "stream
+ tie-break overhead" that exists even at fanout 1.

### P3 — Test suite uses `eligible_cost_row()` helper

Not shown in the diff slice I read, but referenced by the unit
tests. Assumed to be a builder that fills sensible defaults. Worth
confirming it sets `eligible_for_custom_scan = true` and a non-zero
`remote_available_placement_count` so the tests aren't accidentally
exercising the no-fanout branch.

### P3 — `current_planner_cost_constants()` extraction

The cost function pulls `cpu_operator_cost` etc. from the
common-cost helper and keeps `cpu_tuple_cost` separate. Consistent
with how other AM cost models in this codebase handle the same
constants. Good factoring.

### P3 — Cost model is calibrated, not benchmark-derived

The packet says so explicitly. Worth pinning a Stage F packet that
re-tunes these constants once the production benchmark harness
exists, so the symbolic-vs-measured distinction doesn't get lost.

## Reviewer recommendation

Accept. The model is the right *structural* shape; the constants
are placeholders that need a follow-up calibration pass against the
Stage F harness. P2 items (dispatch-startup constant, routing
bound) are the levers most likely to mis-price in the meantime —
worth a one-line comment naming each as symbolic so the next reader
doesn't treat them as load-bearing values.
