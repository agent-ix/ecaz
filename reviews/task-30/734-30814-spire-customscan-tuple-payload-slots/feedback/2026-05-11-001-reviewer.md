# Review: SPIRE CustomScan Tuple Payload Slots (30814)

## Summary

Right shape and a real step forward. Tuple payloads now flow:
remote endpoint → libpq receive
(`SPIRE_REMOTE_SEARCH_LIBPQ_TUPLE_PAYLOAD_SQL_TEMPLATE`) →
`SpireRemoteSearchLocalHeapCandidateRow.tuple_payload_json` →
`SpireRemoteProductionScanOutputRow` → `custom_scan_store_remote_tuple_payload`
→ JSON-decoded `tts_values` / `tts_isnull` → `ExecStoreVirtualTuple`.
Local / materialized outputs still go through
`table_tuple_fetch_row_version`, dispatched on `heap_lookup_owner`.
The libpq dispatch correctly forwards the requested column list as the
seventh parameter only when the request is a CustomScan-style request,
keeping the existing Stage B compact-candidate path intact.

The three review-focus questions:
- AM vs CustomScan endpoint dispatch: correct boundary; AM callers
  remain on the heap-candidate endpoint and CustomScan callers carry
  `Some(tuple_payload_columns)` through to the libpq SQL template.
- Slot materialization path: structurally right but has a P1 that
  breaks for any array column (see below) and over-projects.
- Multi-instance fixture deferred: noted; that is the correct next
  packet.

## Findings

### P1 — JSON → array column input is broken

`custom_scan_json_value_to_datum` stringifies the JSON `Value` and
feeds the text into `InputFunctionCall`:

- `serde_json::Value::Array(_) | Object(_) => value.to_string()`

For an array-typed column (for example `text[]`, `int[]`, or any other
array column), `value.to_string()` produces JSON-style
`[1, 2, 3]`, but PostgreSQL's array input function expects
PostgreSQL's array literal `{1,2,3}`. This will fail at
`InputFunctionCall` with a parse error. Same hazard applies to
`ecvector` — its JSON serialization (whatever it is) is unlikely to
match its `typinput` text format.

Two acceptable shapes:
- Switch to `jsonb_populate_record` server-side (the remote node
  populates a typed record from the JSON and ships it via libpq's
  composite type). This keeps decode in PostgreSQL where the type
  conversion rules are correct.
- Build per-column conversion that maps `serde_json::Value` to the
  attribute type using `typsend`/`typreceive` paths or attribute-aware
  formatters (e.g., translate JSON arrays to `{...}` form for array
  types).

Lower preference: explicitly reject array / non-scalar columns at
`custom_scan_tuple_payload_columns` time and document the limitation.
That at least fails closed instead of producing a confusing
input-function error mid-scan.

The `test_ec_spire_customscan_tuple_payload_stores_virtual_slot` test
masks this because the test relation (`id bigint, title text,
embedding ecvector`) only stores `id` (number) and `title` (string)
in the JSON payload, with `embedding` deliberately absent.

### P1 — `custom_scan_tuple_payload_columns` over-projects

`custom_scan_tuple_payload_columns(node)` walks the relation's full
tuple descriptor (`(*relation).rd_att`) and requests *every*
non-dropped column. So `SELECT id FROM tbl ORDER BY embedding <#> $1
LIMIT 10` requests `(id, title, embedding, ...)` from the remote.
Both wire bytes and remote-side `to_jsonb(heap_row)` cost scale with
the full row width, not the projection.

Should derive from the projection / scan target list instead:
- `(*node).ss.ps.plan.targetlist` (filtered to relation `Var`s); or
- `(*custom_scan).custom_scan_tlist` if you decide to declare a
  CustomScan target list separately.

Without this, the storage / wire savings the CustomScan pivot was
meant to deliver are partially given back at scan time.

### P2 — `InputFunctionCall` machinery rebuilt per row per column

`custom_scan_json_value_to_datum` calls `getTypeInputInfo` and
`fmgr_info` for every (row × column). Both are deterministic per
attribute and should be cached at `Begin` time into a per-attribute
`(FmgrInfo, typioparam)` array stored on `SpireCustomScanExecState`.
For a 10-column projection over a 1000-row scan, this is currently
~10000 `fmgr_info` lookups; cached, it's 10. Cheap fix, real cost
delta once the scan width grows.

### P2 — Missing-payload always errors regardless of consistency mode

`custom_scan_store_remote_tuple_payload` calls `pgrx::error!` when
`output.tuple_payload_missing` is true. That's the correct strict-mode
behavior, but degraded mode is supposed to skip degraded outputs
rather than fail the whole scan. The producer's `consistency_mode` is
already threaded through the executor, but the per-row dispatch in the
CustomScan access callback doesn't consult it.

Suggested shape: in `ec_spire_custom_scan_access`, on
`tuple_payload_missing`, branch on the session consistency mode
(`current_session_remote_search_consistency_mode_name()`):
strict → error; degraded → continue the loop, skipping this output and
incrementing a per-scan counter that the diagnostic surface can
expose.

### P2 — `to_jsonb(heap_row)` materializes the full row before projection

The remote-side endpoint runs `to_jsonb(heap_row)` over the full
heap row and the outer query then `->>`-projects the requested
columns out. That keeps the wire payload small (the SELECT projects
the JSON object down to requested keys) but the remote side still
serializes the whole row to JSON. For wide tables with large
non-projected columns (e.g., a `body text` column when the projection
only wants `id, title`), this is wasted work on the remote.

A future optimization would push the projection into the remote
SELECT (e.g., `SELECT row_to_json((SELECT r FROM (SELECT id, title)
r))` or build the JSON object via `jsonb_build_object(VARIADIC ...)`).
Not for this slice; pin for the typed-tuple slot follow-up.

### P3 — `serde_json` is now a top-level dependency

Cargo.toml adds `serde_json = "1"`. Used only by `custom_scan.rs` for
parsing the JSON payload. That is an acceptable runtime dep, but
worth noting that the eventual typed-tuple-slot migration (where the
remote returns a composite tuple instead of JSON) should also remove
this dependency. Otherwise we keep the JSON encoder around purely
because the decoder once needed it.

### P3 — `tts_values` / `tts_isnull` writes after `ExecClearTuple`

`ExecClearTuple` resets the slot, then the loop writes
`tts_isnull[i]` and `tts_values[i]` directly. After clearing, the
slot's underlying arrays remain allocated (slot lifetime owns them),
so this is the correct virtual-tuple pattern. Worth a one-line
comment that the writes-after-clear pattern is intentional, since a
casual reader might worry the cleared slot has reset the arrays.

### P3 — Unused `tuple_payload_columns: Vec<String>` lifetime

`SpireCustomScanExecState.tuple_payload_columns` is populated in
`Begin` but only used in `custom_scan_ensure_outputs`. Once the
result stream is loaded the field is effectively dead for the
remainder of the scan. Acceptable; a future packet that adds rescan
re-projection can reuse it.

## Reviewer recommendation

Accept the wiring shape. The P1 items (array-column input bug,
over-projection) are blockers before the multi-instance fixture
exercises a row with any array-typed column or wide projection. P2
items (per-row fmgr setup, degraded-mode dispatch, remote
projection-pushdown) should fold into the same follow-up packet so
the typed-tuple-slot direction lands cleanly without leaving the
JSON path as a permanent fallback.
