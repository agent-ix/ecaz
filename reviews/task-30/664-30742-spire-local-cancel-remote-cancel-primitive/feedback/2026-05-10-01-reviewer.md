# 30742 — SPIRE local-cancel remote-cancel primitive review

Validation pass on `be5be41b` (`Add SPIRE local cancel remote cancel
primitive`). Accepts the first C2 propagation slice.

## What I checked

- `Self::run_query_with_optional_local_cancel<T, F>(...)` at
  `src/am/ec_spire/root/remote_candidates.rs:2601` is the shared
  `tokio-postgres` cancel-token wrapper used by both transport
  probe (line 2363) and compact-candidate receive (line 2485)
  query futures. Same primitive on both paths so cancellation
  semantics cannot drift between them.
- Test-only local-cancel triggers exist for PG18 fixtures
  (`local_cancel` test pattern matches `pg_test test_ec_spire_*_local_cancel*`
  per the request).
- A `local_query_cancelled` result from transport or candidate
  receive transitions **all** dispatches to
  `remote_executor_cancelled` rather than recording a per-node
  failure. This is the right shape: a local cancel request is a
  whole-query event, not a per-remote one.
- The Phase 11 task/design docs record this as the first C2
  primitive while explicitly leaving the real PostgreSQL
  interrupt bridge open. No over-claim.

## Answers to the packet's review questions

- **Adapter-side cancel-token primitive appropriately scoped for
  the first C2 slice with the real PostgreSQL interrupt bridge
  still open?** Yes. Wrapping the query future with a
  `tokio-postgres` cancel-token is the narrowest production-side
  primitive that gives us deterministic cancellation behavior
  inside the executor. The actual PostgreSQL interrupt bridge
  (e.g. backend `CHECK_FOR_INTERRUPTS()` → executor cancel)
  needs to be handled in the AM scan integration packet (C5)
  where the executor runs under a real backend, not in this
  adapter primitive.
- **Should `local_query_cancelled` continue to cancel all
  dispatches globally from both transport and compact-candidate
  receive outcomes?** Yes. A local cancel is a whole-query event:
  if any dispatch observes it, every other in-flight dispatch
  should also stop. Per-dispatch cancellation would only be
  meaningful if cancels could target a single remote, which is
  not how PostgreSQL query cancellation works.
- **For per-query connection lifecycle, is requesting remote
  cancel and then closing/aborting the connection task
  acceptable rather than draining a cancelled query response?**
  Yes. Per packet 30724's connection lifetime contract
  (per-query connect / per-dispatch close), the connection has
  no future use after cancel, and draining a cancelled query
  response only consumes time. The remote receives the cancel
  request and the local side drops the connection task — clean
  termination.

## P2 — close before C5 AM scan integration

1. **PostgreSQL backend interrupt bridge.** The current primitive
   relies on test-only triggers to call cancel on the token. In
   production under AM scan, the cancel signal must arrive from
   PostgreSQL's `CHECK_FOR_INTERRUPTS()` / `QueryCancelPending`
   path. C5 should pin how that bridge connects: either
   - the AM scan poll loop checks `QueryCancelPending` and calls
     the cancel-token; or
   - a tokio task watches a signal that the AM scan side raises
     in its interrupt callback.
   Without this bridge, local cancel won't propagate into the
   adapter under real workloads, only under test triggers. Worth
   pinning the bridge shape in
   `plan/design/spire-production-coordinator-executor.md` before
   C5 starts.

## P3 — informational

2. **Cancel-token reuse across retries.** If C2 future work adds
   retry-on-transient-error, ensure the retry creates a fresh
   cancel-token per attempt; a token re-used across retries
   would let an already-cancelled token affect a retry attempt
   that the operator may not have intended to cancel. Optional;
   only relevant if retries land.
