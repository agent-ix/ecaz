## Answers to the packet's review questions

- **Should candidate receive always preflight endpoint
  identity before remote candidate SQL, matching heap
  receive?** Yes — definitively. The symmetry argument is
  the strongest: heap receive's preflight from packet
  30709 was added precisely so a rebuilt remote couldn't
  bypass identity validation; candidate receive needed
  the same gate for the same reason. Before this fix,
  the REINDEX-in-flight case would have masked as a
  generic `remote_query_failed` — exactly the silent-
  identity-drift failure mode the four-layer enforcement
  is meant to prevent. With this fix, candidate receive
  and heap receive now share the same preflight
  discipline.
- **Does the in-flight REINDEX fixture prove the
  request-build versus receive timing boundary?** Yes.
  The fixture captures the pre-REINDEX fingerprint into
  the receive request, then runs REINDEX before the
  receive executes, then asserts the receive observes
  the new fingerprint and fails. This is the
  mid-lifecycle window between request construction and
  remote execution — packet 30772's matrix correctly
  identifies it as a distinct timing case from
  REINDEX-before-fanout (packet 30791), where the
  receive request itself is built against the new
  fingerprint.
- **Do strict/degraded summaries correctly classify the
  rebuilt endpoint as `endpoint_identity_mismatch`?**
  Yes. Same category and status shape as packet 30791,
  proving the timing boundary doesn't change the
  observed failure classification.
