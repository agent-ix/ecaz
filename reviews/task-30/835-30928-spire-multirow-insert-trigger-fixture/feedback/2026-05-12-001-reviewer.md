# Review: SPIRE Multi-Row INSERT Trigger Fixture (30928)

## Summary

Closes 30896 H11. PG18 fixture
`test_ec_spire_trigger_multirow_commits_prepares_sql` covers
multi-row coordinator INSERT through the trigger front door:

- Two-leaf coordinator index, leaves rewritten to **different
  remote node IDs**, so the v1 prepared GID scope
  `(index_oid, node_id, served_epoch, top_xid)` stays unique
  per row even when both rows share the same `top_xid`.
- Two-row INSERT in one explicit coordinator transaction.
- Asserts after commit:
  - 2 placement rows staged with expected PK bytes, node IDs,
    centroid IDs, served epoch, source identities.
  - Duplicate-PK probes on the remote table affect zero rows
    (proves both remote rows committed).
  - `pg_prepared_xacts` has zero remaining SPIRE prepared
    transactions (proves both prepared remotes resolved).
- Adds test-only `tests.ec_spire_test_set_env_var(...)` helper
  so the loopback backend executing the trigger can resolve the
  conninfo secret in its own process environment.

The three review-focus questions:
- Fixture covers per-row trigger dispatch + commit-time
  prepared-transaction resolution for current v1 GID scope:
  yes.
- Two-node setup is right scope for this tracker row, leaving
  same-node multi-row batching to wide-fanout/async dispatch
  work: yes — H11 was specifically about per-row trigger
  dispatch, not about same-node prepare-batching (that's P9
  async dispatch territory).
- Test-only env helper acceptable for external loopback
  backend secret resolution: see findings.

## Findings

### P2 — Test-only `tests.ec_spire_test_set_env_var(...)` helper

The helper exists because the trigger runs in an external libpq
backend (loopback to the same cluster) which has its own process
environment, so the test-runner's env vars aren't visible. Test-
only helpers in a `tests.` schema are the right pattern (already
established for `tests.ec_spire_test_rewrite_placement_node` in
30820), but `set_env_var` is a sharper edge — it mutates process
state.

Worth confirming:
- The helper is gated behind `#[cfg(any(test, feature = "pg_test"))]`
  so it never lands in production builds.
- The `tests.` schema isn't created by production bootstrap.
- The helper is idempotent / safe to call multiple times in
  parallel test runs.

If all three hold, this is the right shape. If any are unclear,
worth a one-line comment on the helper documenting the
constraints.

### P2 — Same-`top_xid` multi-node setup forces unique GIDs

Clever construction: by routing the two rows to different node
IDs, the GID's `(index_oid, node_id, served_epoch, top_xid)`
tuple stays unique even though `top_xid` is shared. This locks
down a real concern about the GID scope — if H11 had used a
single-node setup with two rows, GIDs would have collided and
the second `PREPARE TRANSACTION` would error.

Worth pinning in the GID design (ADR-069) that v1's GID
**relies on per-row routing decisions producing different
node_ids**. Same-node multi-row INSERTs in one transaction
would currently fail at PREPARE. That's another argument for
P9 async batching (one prepare per node per transaction, not
one per row).

### P3 — `pg_prepared_xacts` empty after commit

Strong assertion — proves both prepared transactions resolved
via the `COMMIT PREPARED` xact callback. The classic 2PC
correctness guarantee.

### P3 — Duplicate-PK probe trick

`INSERT ... ON CONFLICT DO NOTHING` style probe on the remote
table to check the row exists without coupling the assertion
to the specific row content. Reasonable shape; subtle.

## Reviewer recommendation

Accept. Closes H11 cleanly. The same-node multi-row INSERT
limitation surfaced by the GID construction is a real concern
worth folding into ADR-069 and the P9 async-dispatch packet —
suggest one ADR-069 line that says "v1 multi-row INSERT in a
single coordinator transaction requires per-row routing
decisions to produce different node_ids; same-node multi-row
INSERTs are unsupported until P9 async batching consolidates
multiple rows into one PREPARE per node."
