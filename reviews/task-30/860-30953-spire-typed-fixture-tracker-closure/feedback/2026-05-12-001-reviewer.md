# Review: SPIRE Typed Fixture Tracker Closure (30953)

## Summary

Tracker-only closure of Phase 12.2 typed-transport fixture
*parent* row, citing the four already-landed fixture packets:

- 30915 — scalar JSON-parity (`bigint`, `text`).
- 30916 — NULL + `text[]` via `payload_nulls` + `array_send`.
- 30917 — domain + named composite via `record_send`.
- 30918 — empty requested projection (typed metadata/value
  arrays aligned without JSON fallback).

Explicitly leaves open: endpoint negotiation, CustomScan
typed binary receive, tuple-heavy throughput measurement,
production JSON retirement.

## Findings

### P3 — Citation completeness

The four cited packets together cover the fixture matrix:

- scalar happy path (30915);
- NULL + array (30916);
- domain + composite (30917);
- empty projection edge (30918).

That spans the type-system axes (scalar, null, array,
domain, composite, empty) at the fixture level. Closing the
*fixture parent* row is justified.

### P3 — Tracker wording — what's NOT closed

The four still-open Phase 12.2 rows are correctly named:

1. **Endpoint negotiation** — coordinator/remote handshake to
   pick typed transport. Without this, typed transport is
   reachable only via fixture, not in production.
2. **CustomScan typed binary receive** — the consumer side
   on the coordinator. Fixture currently asserts the *send*
   side via SQL functions, not the CustomScan code path.
3. **Tuple-heavy throughput measurement** — the perf claim
   for typed-vs-JSON. Without measurement, "typed is faster"
   is unverified.
4. **Production JSON retirement** — the actual flip that
   makes typed the default and removes the JSON fallback.

Closing the parent row without those four open rows is a
real risk only if a downstream packet treats "fixture parent
closed" as "typed transport done." Worth a one-line note in
the tracker that the parent row is **fixture coverage only**,
not transport readiness.

### P3 — JSON retirement is the load-bearing gate

Of the four open rows, **production JSON retirement** is the
one that determines whether typed transport is real or
just a parallel code path. Until that flips, both paths
exist and divergence bugs are possible. Pin a follow-up that
the JSON path should be exercised by at least one fixture
until retirement, so divergence is caught.

## Reviewer recommendation

Accept. Tracker closure is honest — the four cited packets
cover the fixture matrix, and the four still-open rows are
correctly named as separate work. Just make sure downstream
packets read "fixture parent" as fixture-only, not transport
readiness.
