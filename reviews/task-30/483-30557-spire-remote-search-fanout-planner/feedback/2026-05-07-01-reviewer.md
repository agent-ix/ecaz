# Remote search fanout planner review

`plan_remote_search_fanout` (`root/remote_candidates.rs:58-124`) is the
right shape: pure Rust over the validated published-epoch snapshot,
splits selected PIDs into local/remote groups by `placement.node_id`,
and records degraded skips with `(node_id, pid, state)` diagnostics.
No blocking findings.

## Confirmations

1. Local/remote split keys on `meta::SPIRE_LOCAL_NODE_ID` (line 100),
   matching the design's `node_id = 0 == coordinator-local` rule.
2. Strict-mode unavailable/skipped placements return `Err` from
   `fanout_should_skip_placement`, while degraded silently skips. Stale
   in degraded mode also errors — that matches the storage-node
   endpoint's behavior in 30553 and is the right "fail closed when
   the placement state is ambiguous" choice.
3. `BTreeMap<u32, Vec<u64>>` for `remote_by_node` (line 74) gives
   deterministic node-id ordering for the libpq executor while
   preserving selected-PID order within each node.
4. Duplicate selected PIDs and `pid == 0` are rejected before any
   placement lookup happens, which keeps error messages informative.
5. Planner-only scope: no public coordinator API is exposed.

## Findings

- **Empty-input early return skips snapshot validation.**
  `plan_remote_search_fanout:62-69` returns Ok with empty groups
  before calling `from_snapshot`. The caller paths (in 30558 / 30560 /
  30561) re-validate consistency mode and active epoch upstream, so
  this is harmless today. But the helper now has two execution paths
  with different invariants — empty-input does not validate the
  snapshot, populated-input does. If a future caller invokes the
  helper on a freshly loaded but unvalidated snapshot with empty
  selected_pids, validation drift will not be caught.

  Suggest either (a) calling `SpireValidatedEpochSnapshot::from_snapshot`
  before the empty-input branch, or (b) adding a doc-comment that the
  caller must validate the snapshot first.
