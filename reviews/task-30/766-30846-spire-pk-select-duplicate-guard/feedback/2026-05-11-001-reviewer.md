# Review: SPIRE PK SELECT Duplicate Guard (30846)

## Summary

Closes the 30840 P3. `selected_count > 1` guards added to the
coordinator-local branch, the coordinator-remote branch, and the
direct remote endpoint. PG18 fixture builds a heap with a duplicate
PK (intentionally bypassing PK constraint, presumably) and proves
the helper errors instead of returning an ambiguous tuple. Updated
ADR-069 with PK-read idempotent retry behavior + duplicate-match
rejection.

The three review-focus questions:
- Coordinator and remote `selected_count > 1` guards as fail-closed:
  yes — surfaces schema drift loudly.
- `selected_count = 0` remains a valid "not found" result: yes
  (assumed; fail-closed is for ambiguous duplicates only).
- ADR-069 PK-read retry/idempotency note sufficient: yes.

## Findings

### P3 — Same guard could be lifted to a shared helper

UPDATE/SELECT/DELETE all benefit from this `count == 1 vs > 1`
shape. Worth folding into a common `ec_spire_assert_unique_match`
helper so the three primitives share the rejection text and any
future SQLSTATE choice.

### P3 — Duplicate-match test setup

The test must construct a state where a `bigint pk` column has
duplicate values, which usually requires bypassing the PK
constraint. Worth confirming the test setup is unmistakably
artificial (e.g., creates a table without a PK, or uses a UNIQUE
INDEX bypass) so a future reader doesn't think duplicate-PK is a
supported state.

## Reviewer recommendation

Accept.
