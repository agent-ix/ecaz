# Review: SPIRE Classifier Leaf-Pid Contract (30826)

## Summary

Closes the 30818 P2 cleanly. ADR-069 now states explicitly that
`centroid_id` is "the active-epoch routing leaf pid selected by
`ec_spire_classify_centroid`," scoped to `(index_oid, served_epoch)`,
and warns that future epoch retraining must rewrite or reclassify
affected placement rows before using the value for pruning. The
classifier gains a traversal-depth bound derived from the root
routing object's `level + 1`, and a recursive PG18 fixture with
`nlists = 4` and `recursive_fanout = 2` exercises the multi-level
walk by independently scoring root children → leaf children through
`ec_spire_index_routing_centroid_snapshot(...)` and asserting the
classifier picks the same `(node_id, leaf_pid, epoch)` after rewriting
the placement.

The three review-focus questions:
- Documenting `centroid_id` in ADR rather than renaming the column:
  acceptable for v1 — renaming the column would churn the placement
  catalog before any production caller depends on it. The ADR text
  is now the contract.
- Recursive fixture covers multi-level traversal and the
  `parent.header.level == 1` leaf-parent gate: yes — at `nlists = 4`
  with `recursive_fanout = 2` the routing tree has at least one
  internal level above leaves.
- Root-level traversal bound (`max_routing_depth = root.level + 1`)
  is the right defensive limit: yes — combined with the existing
  visited-pid set, this catches both pid-cycles and depth blow-ups.

## Findings

### P3 — Depth bound is `root.level + 1`

`usize::from(parent.header.level).saturating_add(1)` derives the
bound from the root object's level. If `level` is 1-indexed
(level 1 = direct parent of leaves), then a tree with N internal
levels above leaves has root level = N, and the bound = N+1, which
allows root → ... → leaf-parent → leaf walk. That matches the
loop's terminating `parent.header.level == 1` gate. If `level`ing
ever changes to 0-indexed elsewhere (e.g., a future flat-routing
shape), the bound silently shifts. Worth a one-line comment
recording that the bound depends on `level` being 1-indexed at the
leaf parent so a future schema change knows what to update.

### P3 — Fixture asserts equivalence to SQL routing snapshot

The recursive test re-derives the expected leaf pid via
`ec_spire_index_routing_centroid_snapshot(...)` joined against an
`unnest(...)`-based SQL inner-product computation, then rewrites
that placement to `node_id = 9` and asserts the classifier returns
`(9, expected_leaf_pid, active_epoch)`. Strong test pattern — any
classifier vs snapshot drift is caught. Same shape as 30818's
single-level fixture; consistent.

### P3 — Visited-set check still in place

The depth bound is additive to the visited-pid cycle check from
30818. Belt-and-suspenders against malformed routing objects.
Worth a sentence in the function doc that the two guards cover
different failure modes (cycle through repeated pids vs unbounded
chain through fresh pids).

### P3 — ADR note about future retraining

The new ADR text says retraining "must either rewrite affected
placement rows or reclassify them." That's the right
forward-compat hook; it implicitly means the rebalance work has to
include a placement-directory migration step. Worth surfacing this
as a tracker item in the deferred ADR list (rebalance-on-retrain)
so it isn't lost.

## Reviewer recommendation

Accept.
