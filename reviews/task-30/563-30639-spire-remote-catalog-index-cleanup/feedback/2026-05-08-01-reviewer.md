# Remote catalog index cleanup review

`ec_spire_remote_catalog_index_cleanup(index_oid)` is the exact-
OID variant of 30634's orphan sweep — the narrow target a
future `DROP INDEX` event trigger can call without sweeping
unrelated rows. No blocking findings.

## Confirmations

1. **Exact-OID semantics.** The WHERE clauses filter by
   `coordinator_index_oid = $1::oid`, so the function only
   touches rows for the specified index. A coordinator with
   many SPIRE indexes can drop one without affecting the
   others.
2. **Same DELETE pattern as 30634** (manifest header → FK
   cascade for entries → descriptors). Consistent with the
   orphan helper.
3. **Counts measured before deletion** in the same SPI
   transaction; consistency preserved.
4. **Lifecycle contract row for `drop_index` now names both**
   surfaces (exact + orphan), giving operators the right
   choice for the situation.

## Findings (forward-looking)

- **The actual DROP INDEX event trigger has not landed yet.**
  The lifecycle contract row says
  `migration_surface = future_drop_index_hook`. Until that
  hook lands, operators must call this function manually
  after each DROP INDEX. Worth tracking as a follow-up slice.

- **Same authorization concern as 30634** — no role check.
  The function deletes catalog rows with the calling role's
  privileges. Worth either a superuser check or doc note.
