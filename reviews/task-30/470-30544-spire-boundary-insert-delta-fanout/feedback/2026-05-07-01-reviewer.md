# Boundary insert-delta fanout review

aminsert now routes with `nprobe = boundary_replica_count + 1`,
allocates a single `vec_id` for the inserted row, and writes one delta
object per selected leaf. Slice is well-scoped, but flagging two
findings the coder should look at.

## Findings

- **Correctness/defensiveness — primary/replica PID collision risk
  in the recursive routing path.**
  `insert.rs:101-115` calls
  `scan::collect_snapshot_routed_probe_leaf_rows`, then takes the first
  routed leaf as the primary and the rest as replicas. The build path
  goes through `route_boundary_assignment_for_vector`, which dedupes
  PIDs explicitly (`build/routing_plan.rs:145`). The insert path does
  not.

  Today this is safe in single-level mode because
  `route_root_object_to_leaf_pids` (`scan/routing.rs:368-377`) dedupes
  within one root, and in recursive mode each leaf has a single parent
  so unique routes produce unique leaves. But the insert path then calls
  `build_boundary_insert_delta_assignment_placements`, which validates
  `replica_pids.contains(&primary_pid) → error`
  (`assign.rs:222-228`). If a future routing change (e.g. shared
  children across boundary parents in a richer hierarchy, or a top-graph
  insert path) ever produces a duplicate leaf in the routed list, every
  affected `INSERT` errors with
  `ec_spire boundary assignment replica pids must not include primary pid`.

  Recommend either:
  1. Have `insert.rs` dedupe the routed leaf PID list before splitting
     into primary + replicas, mirroring the build-path helper, or
  2. Document the "tree hierarchy guarantees unique leaves per query"
     invariant in a comment on the insert call site so future
     hierarchy changes don't silently break aminsert.

- **Top-graph + insert interaction.**
  This insert path always uses the recursive (non-top-graph) router
  even when `top_graph_enabled = 1`. That is the right v1 choice, but
  because the published epoch can include a `TopGraph` object,
  scans-after-insert use the graph router while inserts route through
  the linear recursive path. The two routers can pick different
  primaries for the same query, so an inserted row may be routed to a
  leaf that is not the top-graph-favored leaf for nearby reads. Worth a
  one-line note in the design or in the insert path that this is
  intentional and acceptable until top-graph is wired into aminsert.

## Confirmations

1. `nprobe = boundary_replica_count + 1` is the right route width.
2. One delta object per target leaf, parented to the corresponding base
   leaf placement, is consistent with the existing placement-directory
   semantics (`update::publish` routes each delta to the same store as
   its base leaf via `insert_delta_object_for_base_placement`).
3. Delta-flag validation (primary OR boundary, exactly one scored role)
   is neither too permissive nor too narrow. Delete deltas correctly
   continue to reject `BOUNDARY_REPLICA`.
4. Regression coverage demonstrates both physical fanout (snapshot
   counts) and logical scan dedupe (deduped heap rows).
