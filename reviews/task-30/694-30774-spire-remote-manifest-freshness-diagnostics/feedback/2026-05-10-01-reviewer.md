## Answers to the packet's review questions

- **Reasonable composition over existing plan/catalog/
  publication surfaces, or should it live lower in the
  Rust snapshot layer?** SQL-surface composition is the
  right choice here. The underlying four surfaces are
  already snapshot-stable Rust readers; rolling them up at
  the SQL layer makes the rollup operator-inspectable
  (essential for Stage E fixture assertions) without
  changing any of the source surfaces' shapes. A
  Rust-side rollup would force every consumer to read four
  surfaces individually — that's more boilerplate for the
  same answer. SQL composition is correct.
- **`freshness_status` and `next_action` categories
  sufficient for Stage E fixture assertions?** Yes for the
  current case set (ready / missing-persistence / stale).
  When per-boundary-replica fixture work lands, additional
  categories may be needed (e.g. `replica_lag`,
  `replica_divergence`); easy to add to the rollup without
  breaking existing assertions.
- **Task/docs wording avoid overclaiming boundary-replica
  fixture evidence?** Yes. The task file flags this as
  node-level coverage with per-boundary-replica still
  open. Honest scoping — node-level freshness is a
  necessary but not sufficient signal for replica
  correctness.
