# Review: SPIRE DML Feedback Hardening (30857)

## Summary

Closes the 30853 P2 (depth-1 coercion walk) and the 30854 P2 (ADR
docs about analyzer-error surface).

Coercion walking is now bounded-recursive: nested
`FuncExpr` / `RelabelType` / `CoerceViaIO` wrappers around an int8
result get unwrapped until a `Const` / `Param` is reached. Outer
non-`int8` wrappers still fail closed as `Other` so the broader
predicate still rejects.

Unit regression confirms `int4 → numeric → int8` (or similar
multi-step coercion) classifies as `ParamBigint` when the outer
result is int8.

ADR-069 now documents that `ec_spire_dml_frontdoor_classify_sql`
runs PG parse / analysis / rewrite first, so standard analyzer
errors surface before any SPIRE-specific diagnostic row.

The three review-focus questions:
- Bounded recursive walk handles intended coercion stack without
  widening accepted predicate values beyond int2/int4/int8: yes.
- ADR wording accurately describes the diagnostic without implying
  planner-hook SPI use: yes.
- Regression coverage sufficient for 30853 P2: yes.

## Findings

### P3 — Recursion bound

Worth confirming the walk has an explicit depth cap (e.g., 8 or
16) so a pathological deeply-stacked coercion chain doesn't recurse
unboundedly. PG's planner doesn't normally produce such trees, but
defense in depth is cheap.

### P3 — `Other` classifier outcome

Predicates the wrapper walk doesn't recognize fall through to
`SpireDmlFrontdoorValueKind::Other`, which the upstream classifier
maps to `unsupported_pk_predicate`. Operator sees a clear error.
Consistent with prior shape.

## Reviewer recommendation

Accept.
