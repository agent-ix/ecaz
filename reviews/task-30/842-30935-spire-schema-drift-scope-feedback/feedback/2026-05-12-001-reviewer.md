# Review: SPIRE Schema Drift Scope Feedback (30935)

## Summary

Closes both 30933 P2s:

- **Migration backfill semantics**: ADR-069 now says existing
  descriptors are backfilled when their coordinator index still
  exists; descriptors left with `unset` fingerprint **fail closed
  for coordinator-routed INSERT until re-registered**. That's
  the safer of the two options I flagged in 30933 P2 — no silent
  guard-skip on existing descriptors.
- **UPDATE/DELETE coverage**: Phase 12.5 tracks extending the
  descriptor-bound schema-drift coverage to UPDATE/DELETE as a
  follow-up (closed by 30936 immediately after, see that
  review).
- **Fingerprint canonicalization**: ADR-069 names the canonical
  fingerprint inputs and clarifies the digest is a deterministic
  drift token, not a security boundary. Closes 30933 P3.
- ADR-069 + `docs/SPIRE_DIAGNOSTICS.md` explicitly state the
  landed guard (at this commit) is INSERT-scoped.

The three review-focus questions:
- Migration/backfill semantics explicit enough for operators:
  yes — `unset` → fail-closed-INSERT-until-re-registered is the
  unambiguous default; no silent skip.
- Tracking UPDATE/DELETE schema-drift as follow-up: yes —
  scope honest, immediately closed by 30936.
- Canonical fingerprint wording matches implementation: yes
  (per 30933's column inputs).

## Findings

None blocking.

## Reviewer recommendation

Accept. The migration choice (fail-closed-on-unset) is the
right default — operators get a loud signal to re-register
existing descriptors after upgrade, vs silently running with
the guard disabled.
