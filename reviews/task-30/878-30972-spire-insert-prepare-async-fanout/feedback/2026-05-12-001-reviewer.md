# Review: SPIRE INSERT Prepare Async Fanout (30972)

## Summary

First Phase 12.4 P9 slice. Moves
`coordinator_insert_prepare_remote_sql(...)` onto a reusable
Tokio fanout adapter:

- builds remote INSERT prepare requests from dispatch rows;
- opens remote libpq/tokio connections concurrently;
- runs `BEGIN`, INSERT SQL, descriptor refresh, `PREPARE
  TRANSACTION`;
- bridges local cancel/timeout into remote cancels;
- rolls back open remotes on pre-prepare failure;
- rolls back any prepared remotes if another batch dispatch
  fails before local callback registration;
- registers local commit/abort callbacks **only after every
  remote prepare in the batch succeeds**.

Single-row helper now routes through the batch adapter.
P9 tracker row stays open pending trigger-level batching.

## Findings

### P1 — Callback-after-all-prepares is the right boundary

This is the load-bearing primitive. Three reasons:

- **Atomicity**: callbacks fire on local commit/abort. If
  callbacks register before all prepares succeed, a partial-
  prepare failure couldn't roll back the already-prepared
  remotes through the callback path — they'd be left as
  orphans waiting for a callback that never fires
  correctly.
- **Cancel-window symmetry**: 30970 fixed the post-
  prepare narrow window for the **single-remote** case. The
  batch case widens that window to "prepared remote N
  while remote N+1 is still preparing." The same pattern
  applies — explicit rollback in the failure handler before
  the callback registration window opens.
- **No-orphan contract**: combined with the explicit
  rollback of any prepared-but-pre-callback remotes on
  partial failure, no `pg_prepared_xacts` row outlives a
  failed batch. Matches the contract the Phase 12.4 row
  requires.

This is the right boundary.

### P1 — Cancellation parity across concurrent prepares

Bridging local cancel/timeout into **remote cancels** for
each in-flight prepare is the multi-remote analog of
30970. Worth confirming the cancel signal fans out to
**all** open remote backends, not just the one whose
response the coordinator is currently awaiting. If only the
"current" remote is cancelled, the others continue
prepare, succeed, and become orphans.

From the request's "bridges local query cancel / statement
timeout into remote cancel requests" wording (plural
"requests"), this looks correct. Worth one fixture in a
follow-up that exercises cancel mid-fanout with at least
two concurrent remotes.

### P2 — Single-row helper routing through batch adapter

Routing the single-row helper through the batch adapter is
the right code-path consolidation. Two paths (single +
batch) would diverge over time and the cancel/rollback
logic would have to be maintained twice. Single
implementation, two call shapes.

The single-row case is just a batch of size 1; the adapter
handles both shapes uniformly. Correct refactor.

### P2 — P9 tracker row staying open

Right call. The async fanout adapter exists, but the
trigger still calls it once per row. Until the trigger
queues rows and flushes at statement-end, M-row INSERT
still produces M serial dispatch calls (each internally
async, but consecutive). The fanout speedup only manifests
when M dispatches share a fanout call.

Keeping P9 open until trigger-level batching lands (30973
addresses this) is the honest scope. Closing P9 here would
overclaim — the adapter's existence isn't the same as
"M remote prepares no longer serialize."

### P2 — Reviewer-feedback processing acknowledgment

The packet notes processing P3 notes from 30969 and 30970
in the same commit (placement contention same-txn shape
comment, INSERT cancel post-prepare cleanup tracker
notes). This is the right way to absorb small follow-ups
without spawning new packets. Honest disclosure that the
slice contains both the new feature and the prior
follow-ups.

### P3 — Adapter reuse for read path?

The "reusable Tokio fanout adapter" framing suggests the
read path's existing concurrent libpq pattern could be
unified with this adapter eventually. Not in scope for
this packet, but worth a tracker note that the read and
write paths now have parallel concurrent-libpq machinery
that may converge.

### P3 — Failed-prepare diagnostic surface

If a batch fails because remote N rejected prepare while
remotes 1..N-1 succeeded, the operator needs to know
which remote failed and why. The error path should surface
the originating remote's `node_id` and the SQLSTATE.
Without that, batch failures look like unattributed batch-
level errors. From the existing remote prepare error
handling (which already names node_id) this likely carries
through, but worth confirming the fanout adapter doesn't
swallow per-remote error context.

## Reviewer recommendation

Accept. Right boundary (callback-after-all), right
cancellation primitive (fan out cancel signal), right
scope (P9 stays open pending trigger batching). Sets up
30973 to actually close P9.
