## What I checked

- New focused test
  `scan_opaque_rescan_replaces_exhausted_output_cursor` at
  `src/am/ec_spire/scan/tests/runtime_state.rs:356`. The
  test exercises three transitions:
  1. **Default state**: a fresh `SpireScanOpaque` has no
     output rows and has not been rescanned.
  2. **Reset cursor advance/exhaust**: after
     `reset_for_outputs(...)` (line 361), the cursor
     advances row-by-row, reports remaining rows, and stays
     exhausted once drained.
  3. **Rescan replacement**: a later `reset_for_outputs(...)`
     call (line 397) replaces the exhausted cursor and
     query state with the new rescan stream — proving the
     stream-rebuild-on-rescan invariant from my 30762 P3 #2.
- The test runs in the standalone lane (`cargo test
  --features pg18`), now possible after packet 30764 fixed
  the `SPI_finish` loader issue. Validates that infrastructure
  win immediately.
- Phase 11 task file note correctly scopes this as
  **coverage**, not implementation — no claim that the AM
  cursor is feature-complete, only that the existing
  cursor-state transitions are tested.

## Answers to the packet's review questions

- **Covers the important AM cursor state transitions before
  the real materialized-row provider lands?** Yes. The
  three-transition matrix (default → reset → exhaust →
  rescan-replace) is the minimum viable coverage for
  `amrescan`/`amgettuple` cursor state. Any future
  materialized-row provider that re-feeds the cursor will
  go through `reset_for_outputs`, so the rescan-replacement
  invariant is the right one to lock now.
- **`reset_for_outputs` clearly replaces prior stream
  state on every rescan?** Yes — the test asserts
  exhausted-cursor state is *not* preserved across
  rescan. A second `reset_for_outputs` with new outputs
  yields a fresh cursor positioned at the new rows, not a
  continuation of the exhausted state.
- **Phase 11 note scoped correctly as coverage rather than
  implementation?** Yes. The task file correctly
  distinguishes "AM cursor coverage" from the still-open
  "remote row materialization implementation." No
  over-claiming.
