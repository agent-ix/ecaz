# 30798 — SPIRE row materialization catalog provider review

Validation pass on `c2dea420` + `5a2da23c`. **Accepts the
catalog-backed Stage D remote row materialization
provider.** This is the slice that lets remote-origin rows
actually reach `xs_heaptid`. Closes packet 30797 P2.

## What I checked

- **ADR-065** at
  `spec/adr/ADR-065-spire-remote-row-materialization-catalog.md`
  records the catalog storage decision, registered in
  `spec/adr/index.md:54` as PROPOSED. Closes 30797 P2.
- **Catalog table** `ec_spire_remote_row_materialization`
  defined in both `sql/bootstrap.sql:221` and the upgrade
  migration `ecaz--0.1.0--0.1.1.sql:181` (so existing
  installs upgrade correctly).
- **Register function**
  `ec_spire_register_remote_row_materialization(...)` at
  `src/lib.rs:6221` with strict input validation (lines
  6234-6260):
  - `requested_epoch` and `served_epoch` must both be > 0,
  - `origin_node_id` must be > 0,
  - `vec_id` and `row_locator` must be nonempty,
  - `materialized_heap_block` and `_offset` must fit
    PostgreSQL's TID ranges,
  - `scan_heap_relation_oid` must match the indexed heap
    relation (line 6271 — this is the AM-correctness
    invariant from ADR-064 enforced at register time so
    a malformed mapping cannot be inserted in the first
    place).
- **Catalog read function**
  `ec_spire_remote_row_materialization_catalog(index_oid)`
  at `src/lib.rs:6314` exposes sanitized catalog state for
  operator/diagnostic use (read-only).
- **AM result stream batch-load**: per the packet text,
  the AM result stream batch-loads candidate mappings
  from the catalog **for the current scan** and validates
  the materialized coordinator heap TID under the
  executor scan snapshot before setting `xs_heaptid`.
  Right discipline:
  1. one batch load per scan (not per row → no per-row
     SPI),
  2. validates 7-element identity from the provider seam
     (packet 30797),
  3. validates `xs_heaptid` visibility under the actual
     executor scan snapshot before delivery (the most
     subtle AM correctness invariant — a mapping row can
     be ready in the catalog but the coordinator heap row
     may be invisible/dead under the active snapshot).
- **Cleanup**: orphan / index drop / remote-catalog
  cleanup all remove materialization rows alongside the
  rest of the remote catalog state. Right lifecycle —
  matches the "cleanup outside scan cursor" rule from
  ADR-064.
- **ADR-064 still intact**: AM path performs no
  scan-time writes. Provider is a *catalog lookup* +
  *visibility check*. Rows without a ready validated
  mapping → block with `remote_row_materialization`.
  Rows with stale or invisible coordinator heap mappings
  → fail closed before delivery.

## Answers to the packet's review questions

- **ADR-065 enough durable rationale for the catalog-
  backed mapping design before broader operator-owned
  mirror lifecycle work depends on it?** Yes. It pins
  the catalog storage shape, the column set
  (index/epoch×2/origin node/vec_id/locator/heap
  relation), and the relationship to ADR-064's lifecycle
  rules. Future packets that build the operator-owned
  mirror sync only need to write into this catalog —
  they don't have to re-litigate the schema.
- **AM path remains read-only with respect to
  materialization state and does not introduce
  scan-time writes?** Yes. The AM cursor reads from the
  catalog and reads the heap row for visibility check —
  both are reads. The `register` function is the *write*
  path, and it's invoked outside the AM cursor (by the
  future mirror sync). ADR-064 invariant preserved.
- **Catalog lookup sufficiently batched for the current
  result stream and does not perform per-row SPI?** Yes
  per the packet description — one batch load per scan.
  Important for performance: a per-row SPI would
  dominate scan latency. The batching also enables
  result-stream merge to see all valid mappings up
  front, which keeps the strict-merge-rejection rule
  from packet 30731 working correctly when some remote
  rows have mappings and others don't.
- **Cleanup semantics cover orphan, index, and drop
  lifecycle for the new table?** Yes per the packet
  description — the three cleanup paths (orphan
  detection, index drop, remote-catalog drop) all
  cascade to materialization rows. This is essential for
  Stage F multi-store hardening, which will exercise
  cleanup paths under load.
- **Fail-closed behavior correct when a mapping row
  exists but the coordinator heap row is not visible
  under the scan snapshot?** Yes — the visibility check
  is the AM-correctness invariant from ADR-064 packet
  30761. A snapshot-invisible TID cannot be returned via
  `xs_heaptid` because downstream operators expect
  visible rows under the same snapshot the index AM
  used. Failing closed in this case is the only safe
  behavior.

## What this slice unlocks

**A real PostgreSQL `SELECT` query against an `ec_spire`
index with remote placements can now return remote rows**
— provided:
1. A coordinator heap row exists in the indexed relation
   that proxies the remote candidate, AND
2. `ec_spire_register_remote_row_materialization` has
   been called to register the mapping, AND
3. The mapping is current with respect to the requested
   epoch + served epoch + origin node + vec_id + locator,
   AND
4. The proxy heap row is visible under the executor's
   scan snapshot.

The future operator-owned mirror lifecycle work
(Stage F or a separate maintenance slice) will populate
the catalog by replicating remote-origin rows into the
coordinator heap and registering the mappings. Until
then, this catalog is in place but empty — same
fail-closed behavior as before the catalog landed,
because no mappings exist to validate.

## P2 — close before claiming end-to-end SQL remote rows

1. **Operator-owned mirror sync mechanism.** ADR-065
   establishes the catalog; ADR-064 establishes the
   lifecycle ownership ("epoch maintenance / operator
   mirror lifecycle outside `amrescan` and
   `amgettuple`"). The remaining design question:
   *which subsystem* runs the mirror sync (background
   worker / explicit operator command via `ecaz` /
   triggers on epoch publish events). Recommend an ADR
   or design doc selecting this before the sync
   implementation lands. Until the sync runs, the
   catalog stays empty in practice.

## P3 — informational

2. **End-to-end SQL fixture.** Once any mirror-sync

   - creates a local `ec_spire` index with a remote
     placement,

   executor. That's the Stage D feature-complete signal.

## Status

Closed. **Catalog-backed materialization provider is
acceptance-quality.** The seam from packet 30797 now has
a real provider; default behavior (no mappings registered)
