## Findings

| ID | Severity | Notes |
|----|----------|-------|
| F1 | non-blocking, medium | The diagnostic surface (`recursive_routing_diagnostics_levels` and friends in `scan/routing.rs:626+`) is a *parallel* routing path that mirrors the production routing logic but accumulates counters. The request states the collector is "threaded through the same … semantics" — in practice it is a separate function with the same shape. If the production routing logic changes (e.g., dedup before vs after expansion cap, score-sorting tie-breaks), the diagnostic path can drift silently. **Mitigation:** either (a) extract the per-level traversal into one helper that takes a `Collector` trait so production and diagnostic paths share the same loop, or (b) add a property test asserting that the diagnostic `selected_child_count` and `deduped_route_count` match the production routing's actual leaf/child set on a fixture. |
| F2 | non-blocking, small | `ec_spire_index_scan_routing_snapshot` is a separate SQL surface from `ec_spire_index_scan_placement_snapshot`. The request asks whether it should be joined later. For Phase 9 they are correctly separate (placement is per-leaf, routing is per-level). For Phase 10 perf work, a unified `ec_spire_index_scan_pipeline_snapshot` joining placement + routing + candidates rows by step ordinal would mirror the `pipeline_steps` shape on the remote side. Defer to Phase 10. |
