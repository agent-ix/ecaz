- Profile fingerprint is a deterministic digest over: protocol,
  extension, opclass, storage format, assignment format, quantizer
  profile, scoring profile, key build options, and active epoch.

- Phase 11 task file records conservative Stage B progress.

## Answers to the packet's review questions

- **Is the identity surface the right contract object before
  expanding remote candidate rows?** Yes. Promoting fields into the
  candidate row before the served identity is testable in SQL would
  ship an under-specified wire shape. Pinning the identity in SQL
  first is the correct sequencing.
- **Is `requires_rabitq_storage_format` the right fail-closed
  behavior for default/non-RaBitQ SPIRE indexes?** Yes. Phase 11 only
  supports RaBitQ for remote serving; defaulting to a `not_ready`
  status with an explicit reason matches the rest of the contract
  posture.
- **Is the current profile fingerprint scope acceptable?** Yes for
  the first served identity. Training-stat fingerprinting can
  replace or extend the digest later without changing the surface
  shape, since the digest is opaque from the operator's side.

## P2 — close before Stage B candidate-row promotion

1. **Document the fingerprint inputs and stability rules.** The
   fingerprint digest currently mixes nine inputs (protocol,
   extension, opclass, two formats, two profiles, key build options,
   active epoch). Recommend adding one short doc paragraph (in the
   identity surface help or a small ADR addendum) listing exactly
   which fields contribute, in what byte order, and how the active
   epoch is incorporated. Without it, future cross-version mismatch
   debugging will require reading the digest implementation. This
   should land before the fingerprint is promoted into the candidate
   row, because once it's on the wire, changing the input set is a
   breaking change.

2. **Training-stat extension path.** The packet correctly notes that
   training-stat fingerprinting may replace or extend the digest
   later. Recommend recording, in the identity surface or an ADR
   note, whether such a future extension would bump the protocol
   version (clean break) or extend the digest input set (silent
   broadening) — operators need to know which behavior to expect.

## P3 — informational

3. **Recommendation row consumers.** The identity row exposes a
   recommendation field. Worth confirming with a future packet
   whether any operator runbook or Stage G harness consumes that
   field as machine-readable, or whether it is intended for human
   reading only. Affects whether values must be stable strings
   (machine) or can drift in wording (human).

## Status

Closed for the Stage B identity surface. P2 #1 (fingerprint inputs
documented) is the gating action before the fingerprint moves into
the candidate row wire shape. RaBitQ-only fail-closed posture is
correct for Phase 11.
