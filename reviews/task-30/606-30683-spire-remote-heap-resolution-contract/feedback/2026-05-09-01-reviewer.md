# 30683 — Remote heap resolution contract review

Validation pass on `4db3f35b` + ADR-059. Phase 10.6 ADR pinning
remote heap resolution at the origin node and tying production
cross-node dedupe to the writer-side global vec ID slot from
ADR-055.

## Verified

- ADR-059 status `ACCEPTED`, dated 2026-05-09. Two sharp
  decisions:
  1. Coordinator treats remote `row_locator` bytes as **opaque**
     and must not decode them directly. Remote heap resolution is
     origin-node-owned.
  2. Production cross-node boundary-replica dedupe requires
     writer-side global `0x02 || stable_global_payload_bytes` vec
     IDs per ADR-055. The local `0x01` vec ID stays valid for
     diagnostics and single-node compatibility but is not
     sufficient for a production cross-node claim.
- Until a production origin-node heap resolver lands, remote heap
  rows remain explicitly blocked/deferred via
  `requires_remote_heap_resolution`. No silent partial-row
  delivery.
- Production heap resolver requirements are listed: only
  heap-visible rows from the origin node under the requested
  epoch and consistency contract; non-visible rows omitted or
  reported as degraded/blocked, never as partial.
- Phase 10 task file points at the contract.

This **closes my prior 30667 F2** (writer-side global ID
allocation tracked as production blocker). The forward gap is now
named in an accepted ADR, not just inline in a task file.

No findings.
