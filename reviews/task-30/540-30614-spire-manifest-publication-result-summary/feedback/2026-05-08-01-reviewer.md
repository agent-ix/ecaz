# Manifest publication result summary review

`ec_spire_remote_epoch_manifest_publication_result_summary`
composes the publication gate into a result_source taxonomy:
`not_required`, `pending_libpq_executor`,
`remote_manifest_validation_result`, or `blocked`. No blocking
findings.

## Confirmations

- Result-source taxonomy is symmetric with the search-side
  `coordinator_result_summary` (`local_heap_candidates`,
  `blocked`, `none`).
- `pending_libpq_executor` is the right marker for "everything
  is ready except the executor itself" — correctly distinct
  from `blocked` (something else is wrong) and
  `not_required` (local-only).
- `remote_manifest_validation_result` is forward-leaning
  scaffolding for when the apply call has actually run on
  remote nodes; today no caller produces this status, but the
  taxonomy is in place.

## Findings

- **`remote_manifest_validation_result` is unreachable today.**
  The status is in the taxonomy and the contract (30616) lists
  it, but no code path emits it because the apply step does not
  execute remotely yet. Acceptable as forward scaffolding;
  worth a comment in the contract row's recommendation column
  noting "v1: synthesized once remote apply executor lands."
