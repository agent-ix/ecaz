# Review: SPIRE Typed Tuple NULL And Array (30916)

## Summary

Slice 2 of the 30913 typed transport design. Extends the typed
endpoint fixture coverage from 30915:

- New PG18 fixture for out-of-band SQL NULL handling — proves
  `payload_nulls = true` for NULL attributes and that the
  payload byte placeholder is correct (empty bytea, ignored on
  receive).
- New `text[]` array fixture — verifies typed bytes match
  PostgreSQL's own `array_send(...)` output. This is the key
  proof that arrays now work through the typed transport
  (previously rejected by the 30816 scalar gate on the JSON
  bridge).
- Mixed projection metadata fixture: `id bigint`, `title text`,
  `tags text[]` — verifies per-column metadata (attnum, name,
  type_oid, typmod, collation) is correctly returned for a
  realistic shape.

The three review-focus questions:
- NULL representation matches the accepted design
  (out-of-band via `payload_nulls`, payload bytes ignored for
  NULLs): yes.
- `array_send(...)` is suitable as the byte-level oracle for
  the first `text[]` fixture: yes — `array_send` is PG's
  canonical binary serialization for arrays; matching it
  byte-for-byte proves the typed endpoint produces the value
  PG's own `array_recv` will deserialize.
- No overclaimed coverage: confirmed — composite, domain,
  negotiation, CustomScan receive, and JSON retirement all
  remain explicitly open.

## Findings

### P3 — Empty bytea placeholder for NULL columns

The `CASE WHEN heap.X IS NULL THEN ''::bytea ELSE typsend(heap.X)
END` from 30915 stores a 0-length bytea when a column is NULL.
The receive side ignores `payload_values[i]` when
`payload_nulls[i] = true`, so the placeholder is never
consumed. Worth confirming the design doc names this as the
NULL placeholder convention so a future receive implementation
doesn't try to call `typreceive` on the empty bytea.

### P3 — `text[]` is the "easy" array case

PG's `array_send` for `text[]` produces a stable binary format
that round-trips cleanly. Worth confirming that `int4[]`,
`int8[]`, and `numeric[]` work too — these have different
element-size conventions and would catch bugs in the typed
endpoint's array handling. Consider folding into the next
slice's coverage.

### P3 — Composite type coverage still open

Composite types and domains are explicitly noted as remaining
open. Composites in particular have non-trivial binary
serialization (record_send walks the tuple descriptor at the
remote, which the coordinator must reconstruct in
record_recv). Worth a focused composite-type fixture in the
next slice — composites are where binary I/O semantics get
subtle, and a fixture catches it before negotiation work
depends on the assumption that composites round-trip.

### P3 — Metadata fixture exercises real `pg_attribute` shape

The mixed `id bigint, title text, tags text[]` projection runs
through the same `ec_spire_tuple_payload_column_metadata` SPI
walker as production. Good — proves the metadata pipeline
handles array types correctly without special-casing.

## Reviewer recommendation

Accept. Slice 2 covers NULL and array — two of the four
remaining type-class gaps from 30816 (the others being
composite and domain). The next slice should land composite +
domain coverage before negotiation work begins.

Slice 3 (descriptor negotiation + strict/degraded fallback)
can land in parallel since it's mostly orthogonal — but the
typed endpoint should cover the major type classes before any
remote starts advertising `pg_binary_attr_v1` capability.
