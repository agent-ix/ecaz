# 30751 — SPIRE production strict/degraded fault matrix review

Validation pass on `0cbf841c` (`Add SPIRE production strict
degraded fault matrix`). Closes packet 30744 P3 #2 (matrix table
in design doc) and packet 30748 P3 #1 (consistency_mode_mismatch

## What I checked

- New SQL surface
  `ec_spire_remote_search_production_fault_matrix()` at
  `src/lib.rs:9862` returns a dry policy table — no SPI, no
  remote calls.
- Matrix coverage spans every category landed across packets
  30724–30750 plus reserved Stage D categories:
  - Connect / auth / cert-style transport failures (30725).
  - Conninfo secret failures (30734).
  - `remote_statement_timeout` / `local_statement_timeout`
    (30738, 30750).
  - `remote_backend_terminated` (30739).
  - `remote_query_cancelled` / `local_query_cancelled` (30740,
    30742).
  - Candidate `decode` / `validation` failures (30727, 30728).
  - `endpoint_identity_mismatch` (30710, 30743).
  - Protocol / extension version skew (30711).
  - Stale / served epoch mismatch (30745).
  - `requested_epoch_mismatch` (line 88 constant, claimed by
    the matrix).
  - `consistency_mode_mismatch` (30748).
  - `remote_index_unavailable` (30732).
  - Reserved Stage D remote heap-resolution categories (so the
    AM boundary has stable names before Stage D emits rows).
- Row shape per the request: category + scope + executor step
  + strict action/status + degraded action/status +
  recommendation. That's the right column set for C5 to consume
  the policy as data — every column maps to a decision the AM
  bridge needs to make per failure.
- Local cancellation is **query-wide** in both strict and
  degraded modes. This is correct: `local_query_cancelled` /
  `local_statement_timeout` are coordinator-side events, not
  per-remote events. A degraded mode that "skipped" local
  cancellation would be incoherent.
- Most remote-node failures fail closed in strict mode and skip
  the affected node in degraded mode. The exceptions
  (`consistency_mode_mismatch` and `requested_epoch_mismatch`)
  fail closed in *both* modes — see review-question answer
  below.
- `production_fault_matrix_covers_required_categories` (line
  7666) and PG18 `production_fault_matrix_contract` test both
  pass. The Rust-side contract test is a useful guardrail
  against a future packet adding a category to the executor
  without adding a row to this matrix.

## Answers to the packet's review questions

- **Row shape sufficient for C5?** Yes. C5 will consume this
  table either as a runtime lookup (read once at scan start,
  branch per failure during merge) or as a generation source
  for compile-time match arms; either way, the columns provided
  cover every decision. If C5 needs a tie-break ordering across
  multiple failures, that's a separate concern that doesn't
  affect this row shape.
- **Should reserved Stage D heap categories stay in the C4
  matrix now, or wait until the heap executor emits them?**
  Stay now. Reserving the names in the matrix locks the
  AM-boundary contract before Stage D code lands, so Stage D
  has a target to emit against rather than inventing categories
  and back-fitting them into the matrix. This is the same
  discipline that worked for packets 30738/30739/30740 reserving
  C2 categories before claiming emission paths.
- **`consistency_mode_mismatch` and `requested_epoch_mismatch`
  correctly fail-closed in degraded mode?** Yes —
  unambiguously. Both are *publisher-declared* invariants:
  - `consistency_mode_mismatch` means the publisher said the
    epoch is strict-only; a degraded session cannot override
    that without violating the publisher's contract (locked in
    packet 30746).
  - `requested_epoch_mismatch` means the coordinator asked for
    an epoch the remote cannot serve at all; degraded mode
    cannot "skip" this because the request itself is wrong, not
    the remote.
  Treating these as degraded skips would let a query silently
  succeed against the wrong assumption. Fail-closed in both
  modes is the only safe shape.

## P3 — informational

1. **Document the matrix as the C5 source of truth.** Worth
   one sentence in the C5 design contract noting that AM scan
   integration must consume this matrix (via SQL surface or
   Rust-side equivalent) rather than re-encoding strict/
   degraded behavior in the AM scan code path. Otherwise a
   future packet might add new behavior in two places that
   could drift. Optional — natural to land in the C5 packet.

2. **Reserved Stage D row visibility.** Operators reading the
   matrix today will see Stage D rows with statuses like
   `requires_remote_heap_resolution_implementation` or similar.
   Worth confirming the matrix labels those rows as `reserved`
   rather than `ready` so an operator doesn't think Stage D is
   already implemented. The wording matters for runbook
   accuracy.
