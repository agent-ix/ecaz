# Remote summary rollups review

Addresses cross-cutting finding #5 from 30568. `SpireRemoteCountRollup`
(`root/remote_candidates.rs:49-156`) is the right shape: typed counts,
checked overflow on every increment, and explicit error returns for
unknown target/status strings. No blocking findings.

## Confirmations

1. Four summaries (request, readiness, execution, libpq request)
   now route through `record_target` and `record_status`. Each
   summary's status precedence is preserved exactly — the rollup
   collects counts but does not derive the final status string.
2. `record_target` rejects unknown `target_kind` strings; combined
   with the central registry from 30570, this is a structural
   guard against stringly-typed drift between rollup callsites and
   the public diagnostic surface.
3. `executable_pid_count` correctly errors on overflow rather than
   wrapping.

## Findings (non-blocking)

- **`record_status` rejects unknown statuses but `record_target` is
  exhaustive.** `record_status` matches on
  `READY | DEGRADED_SKIPPED | REQUIRES_DESCRIPTOR | REQUIRES_LIBPQ`
  and falls through to "unknown status" for anything else. That's
  correct today, but `degraded_ready` (used by other summaries) is
  not in the match arms. If a rollup ever consumes a row with
  `status = "degraded_ready"`, it will error. Currently no rollup
  caller passes that string, so this is forward-looking — but worth
  either extending the arm or documenting that "degraded_ready" is
  a *summary-output* status, not a *target-input* status.
