# Review: SPIRE Coordinator Insert SQL Helper (30833)

## Summary

End-to-end SQL composition of the v1 INSERT path before the
transparent DML hook lands. `ec_spire_prepare_coordinator_insert_tuple_payload`:

1. Validates `pk_value` (non-empty) and `source_identity` (16
   bytes).
2. Opens the SPIRE index and runs `spire_classify_centroid` to pick
   `(node_id, centroid_id, served_epoch)`.
3. Calls `spire_coordinator_insert_prepare_remote_tuple_payload`
   (30832), which goes through dispatch readiness → libpq → BEGIN /
   remote-endpoint / PREPARE TRANSACTION → registers Commit/Abort
   xact callbacks.
4. Closes the index relation.
5. INSERTs into `ec_spire_placement` via SPI.
6. Returns the placement tuple plus prepared GID, remote_insert_sent,
   remote_prepared, placement_staged, status, next_step.

Real loopback PG18 fixture proves the prepared composition + local
placement staging. ADR-069 gets the helper documented.

The four review-focus questions:
- Useful boundary for the eventual transparent DML hook: yes — the
  hook will need to construct PK bytes / source identity / JSON
  payload / column list from the executor tuple, then call this
  helper.
- Remote prepare before local placement staging: yes, in that
  order in the function body.
- Local placement staging failure aborts the txn → registered
  Abort callback rolls back the prepared remote: yes, by
  construction (any `pgrx::error!` in step 5 unwinds the local
  txn, which fires the abort callback registered in step 3).
- SQL surface honest about scope: yes — caller still provides PK
  bytes, source identity, row JSON, and column list.

## Findings

### P2 — Status string `remote_insert_prepared` is ambiguous about commit state

The function returns `status = "remote_insert_prepared"` (from
30830) once the remote PREPARE TRANSACTION succeeds and the local
placement row is staged. But the **coordinator transaction has not
yet committed** — the remote prepared xact is only resolved when
the calling transaction commits or aborts. A caller reading the
returned status as "the row is now durable" is wrong; the row is
durable only after the calling transaction commits.

Suggestion: rename to `remote_insert_prepared_pending_local_commit`
or add an explicit `pending_commit: true` flag in the row so the
caller cannot mistake "prepared" for "durable." Especially load-
bearing for the eventual SQL surface (operators will run this in
psql sessions and see the status before they `COMMIT`).

### P2 — Cumulative xact callbacks per transaction

Each call to this helper inside one coordinator transaction
registers a fresh pair of `Commit` / `Abort` callbacks (one per
prepared remote xact). For a transaction that calls the helper N
times — e.g., a multi-row INSERT loop — there are N callback pairs
firing at commit/abort. Each one opens a fresh libpq connection to
its remote and runs `COMMIT/ROLLBACK PREPARED`. That's:
- N libpq connections open/close at commit time (latency adds up).
- Failure of one callback's COMMIT PREPARED is silently swallowed
  (per 30830 design); subsequent callbacks still run.

For v1 single-row INSERT this is fine. For the mass-INSERT path or
a coordinator hook that batches rows per remote, worth grouping
prepared xacts by `(node_id, conninfo)` and resolving them in one
connection per node at commit time.

### P2 — `Spi::connect_mut` placement INSERT inside the function body

The placement-directory INSERT happens *after* the remote prepare
returned. If the local INSERT triggers a duplicate-PK error against
`ec_spire_placement_pkey`, the function `pgrx::error!`s and the
local txn aborts → Abort callback rolls back the prepared remote.
That's correct.

But: the placement-directory INSERT *can* race with another
coordinator session inserting the same `(index_oid, pk_value)`
between two calls. Both sessions would each have prepared their
remote xacts. The one that loses the placement-directory PK race
rolls back its remote prepared correctly. The winning session
commits both. End state: one row remote, one row local. OK — but
worth a one-line note that PK conflicts on placement directory are
the resolution boundary (vs trying to detect the race earlier on
the remote).

### P3 — Returned row echoes `index_oid`, `pk_value`, `source_identity`

Convenient for callers building a follow-up `RETURNING`-style
result; field order matches the placement catalog.

### P3 — Validation matches 30828 planning primitive exactly

`pk_value.is_empty()` → error and `source_identity.len() != 16` →
error. Same checks. Worth factoring into a shared helper to avoid
drift between the planning-only and the prepare paths.

### P3 — Test fixture uses ScopedEnvVar pattern

Matches the 30815 multicluster fixture pattern. Consistent and
clean.

### P3 — `next_step` after this helper

The status comes back from 30830 as
`local_placement_directory_write`. After this helper that step is
complete; should the next_step advance to `commit_coordinator_transaction`
or `await_local_commit`? Currently the status string still says
"do the placement-directory write" even though it just happened.
Worth advancing to a post-staging next_step so the caller knows the
helper's work is done.

## Reviewer recommendation

Accept. The composition is the right shape, but the pre-commit /
post-commit naming is the most operator-facing concern: the status
string and the next_step both need to clearly distinguish "prepared
+ staged, awaiting local commit" from "committed and durable."
Otherwise an operator running this in a session and seeing
`remote_insert_prepared` will reasonably believe the row is live.
