# Review: SPIRE Centroid Classification Helper (30818)

## Summary

`ec_spire_classify_centroid(embedding, index_oid)` returns
`(node_id, centroid_id, epoch)` by walking the active-epoch routing
hierarchy from the local root down through internal routing objects
to a leaf, scoring children by max inner product with deterministic
tie-breaks `(centroid_index, child_pid)`. Validates active epoch,
decodes epoch / object / placement manifests, validates the snapshot
through the existing `SpireValidatedEpochSnapshot::new`, and walks
exclusively through local routing objects until it hits the leaf
(remote leaves are returned; remote non-leaf routing objects fail
closed).

The three review-focus questions:
- `centroid_id = selected leaf pid` is the right durable
  interpretation as long as no rebalance / centroid retraining
  changes leaf pids — and ADR-069 explicitly defers retraining
  rebalance. Acceptable for v1.
- Helper bypasses only the superseded local-heap materialization
  gate and still validates active routing metadata: yes — uses the
  full `SpireValidatedEpochSnapshot` path, not the lighter
  custom-scan eligibility loader.
- Recursive routing behavior: local routing objects are traversed,
  remote leaves return, remote non-leaf routing objects fail closed
  with a named error. Correct.

## Findings

### P2 — `centroid_id == leaf_pid` couples to physical routing layout

ADR-069's column comment is "centroid_id bigint NOT NULL CHECK
(centroid_id >= 0)". The classifier returns the *leaf pid* under
that name. As long as routing objects are stable across epoch
publishes, `placement.centroid_id` is meaningful and lookups work.
But two failure modes:

- **Epoch rebuild / re-train**: ADR-069 defers this, but when it
  lands, the placement directory's `centroid_id` becomes a stale
  reference to a leaf pid that may no longer exist. Updaters must
  either rewrite the column or re-classify on read.
- **Coordinator-routed UPDATE/DELETE flows** (packet 30819 +
  successors) will read `centroid_id` purely as an opaque routing
  hint or also use it to prune to a single leaf during scan? The
  current packet doesn't expose the read-side semantics of
  `centroid_id` — pin the contract before the read paths consume it.

Suggestion: rename the field `routing_leaf_pid` in the placement
directory and the helper output, or document explicitly that
`centroid_id` semantically is "active-epoch leaf pid the row routes
to" and that retraining requires migration. Either is fine; the
risk is silent post-retrain drift later.

### P2 — `embedding.iter().any(|c| !c.is_finite())` rejects NaN/inf at entry

Good for the routing-side: an embedding with a non-finite component
would corrupt all inner-product comparisons. But applications often
pass `NaN` or `inf` accidentally (model produced bad output, etc.).
Currently the classifier ERRORs the whole INSERT. Coordinator-routed
INSERT (next packet) will need to surface this error message in a
way the application can attribute to the specific row, not the
whole transaction. Worth a follow-up note in the write-path packet.

### P2 — Cycle detection via visited set; no max-depth bound

`visited.insert(parent.header.pid)` returns false on revisit, which
yields a clear "routing cycle" error. Good defensive coding. Worth
also adding a max-depth bound (e.g., the snapshot's reported tree
height + small margin) so a pathological routing object that points
back to root via *different* pids gets bounded — visited-set alone
doesn't catch a cycle through pid renaming.

### P2 — `parent.header.level == 1` is the leaf-parent gate

The terminating condition is `level == 1`, after which any selected
child is a leaf. Worth confirming that `level` on internal/root
routing objects is consistently 1-indexed over depth (level 1 =
parent of leaves, level 2 = parent of internals at level 1, etc.).
If `level` is 0-indexed elsewhere in the codebase, this
classification fails to recurse correctly. The test exercises a
`nlists = 2` (probably one-level) index, which doesn't traverse
multiple levels — a multi-level fixture would be the safer
regression.

### P2 — Walks all manifest entries to find the root

`for manifest_entry in &snapshot.object_manifest().entries { ... if
header.kind != Root { continue; } ... }` iterates the entire object
manifest looking for the unique local root, with a duplicate-root
check. For large indexes this is O(N) on the manifest at every
classification. The root pid is presumably reachable through a
narrower path (root_control or epoch manifest). Worth either:
- caching the root lookup per-snapshot;
- or making the root identifier explicit on the snapshot to avoid
  the linear scan.

Coordinator-routed INSERTs will call this on every row; the linear
scan dominates classification cost on wider hierarchies.

### P3 — `score.total_cmp` is the right tie-breaker for f32

Using `total_cmp` rather than `partial_cmp` means NaN sorts
deterministically rather than panicking; combined with the
finite-check at entry this is belt-and-suspenders. Good.

### P3 — Test asserts via classifier vs SQL routing snapshot agreement

The test computes the expected `centroid_id` by reading the
`ec_spire_index_routing_centroid_snapshot` and joining against an
`unnest(...)`-based scoring SQL — a parallel implementation in SQL.
Then asserts the classifier returns that same pid after rewriting
its placement to `node_id = 7`. Strong test: any classifier vs
snapshot drift is caught.

## Reviewer recommendation

Accept the helper shape. The P2 `centroid_id` semantics question is
the most load-bearing for downstream packets — pin the meaning
before INSERT / UPDATE / PK-read paths consume it, otherwise the
classifier's leaf-pid output and the placement directory's
`centroid_id` semantics could drift apart. The root-lookup linear
scan and the multi-level fixture are scaling/regression hardening
for a follow-up.
