# Pre-checkpoint review: SPIRE Top Graph Codec

This feedback was written before the coder created `request.md` for this
topic. Reviewing the in-progress working tree on `task-30-spire` after
commit `d200e97d` (phase 5 review feedback). The slice on disk:

- new `src/am/ec_spire/storage/top_graph.rs` (305 lines):
  `SpireTopGraphPartitionObject` + `SpireTopGraphNodeRecord` with
  encode/decode/validate;
- new `src/am/ec_spire/storage/tests/top_graph.rs` (86 lines): 3 unit
  tests;
- new variant `SpirePartitionObjectKind::TopGraph = 5` threaded through
  every match-arm consumer (12 files) as no-ops or carry-forward.

This is a codec-only slice. No writer, no reader on the object store,
no manifest emission, no scan binding yet. That's a sensible starter
shape for Phase 6.

## What's solid

- Header invariants enforced cleanly: `kind == TopGraph`,
  `parent_pid == root_pid`, `level > 0`, `assignment_count == 0`,
  `flags == 0`, `child_count == nodes.len()`.
- Vamana parameter constraints: `graph_degree > 0`,
  `build_list_size > 0`, `alpha` finite and `>= 1.0`,
  `dimensions > 0`. Matches `plan/design/spire-top-level-graph.md`.
- Per-node validation: pid uniqueness, centroid-ordinal uniqueness,
  neighbor count ≤ `graph_degree`, no self-loops, no duplicate
  neighbors, no out-of-range neighbors. Directed edges (no symmetry
  requirement) is the right Vamana shape.
- Body layout: `8+2+2+4+4+4+4 = 28` bytes prefix matches
  `TOP_GRAPH_OBJECT_BODY_PREFIX_BYTES`. The `reserved: u16` at offset
  10..12 gives a forward-compat slot, validated as zero on decode.
  Trailing-bytes check rejects garbage suffixes.
- Match-arm threading is exhaustive — every consumer site picked up
  the new variant. No silent fall-through.

## Findings

### 1. `centroid_ordinal` cross-reference is unbounded at codec layer

`validate()` enforces uniqueness across nodes but not that
`centroid_ordinal < N` for some N tied to the root routing object's
child count. A corrupted top-graph object could carry ordinals that
point past the routing object's child list, and decode would accept
it.

The check belongs at bind time (when the top graph is paired with the
root routing object), but add a comment in `validate()` saying so
explicitly so a future reader doesn't assume the codec self-validates
that link.

### 2. Carry-forward through replacement and compaction stages a stale-graph footgun

`update/leaf_rows.rs::plan_replacement_epoch_placement_directory` and
`vacuum.rs::publish_compacted_delta_epoch_if_needed` both add
`TopGraph` to the carry-forward group alongside Root/Internal/Leaf.
That's correct for current scope — split/merge changes leaves, not
top-level centroids. But:

- A recursive-routing rewrite (which *does* change top-level
  centroids) would invalidate the graph's `centroid_ordinal`
  references. There is no guard yet that fails or rebuilds the top
  graph in that case.
- Even pure leaf split/merge can change the centroid count if
  Phase 6's eventual "rebuild graph" trigger isn't wired before that
  happens.

Not a defect of this slice (no writer exists), but worth a
`// TODO(phase6): invalidate on routing rewrite` at both carry-forward
sites so the assumption is auditable.

### 3. Two adjacent empty arms in `vacuum.rs:237`

```rust
SpirePartitionObjectKind::Root | SpirePartitionObjectKind::Internal => {}
SpirePartitionObjectKind::TopGraph => {}
```

Two identical empty arms; could be combined. `root/snapshots.rs:693`
*did* fold `TopGraph` into the existing `Root | Internal` group —
pick one style and apply it consistently. Cosmetic.

### 4. Test coverage is thin for a new on-disk codec

Three tests cover round-trip, neighbor-out-of-range, and duplicate
centroid ordinal. The validator has roughly 16 distinct error paths.
Add at minimum one test per Err arm:

- `entry_node >= node_count`
- self-loop rejection
- duplicate neighbor rejection
- `alpha < 1.0` and `alpha = NaN`
- `graph_degree = 0` and neighbor count exceeding `graph_degree`
- `dimensions = 0`
- `level = 0`
- non-zero `assignment_count` / `flags`
- non-zero `reserved` bytes on decode
- trailing bytes on decode
- empty `nodes`
- duplicate `child_pid`
- truncated body (short header, short prefix, short neighbor block)

Codec validators are durable on-disk contracts; one `#[test]` per Err
arm is the cheapest insurance against silent format drift.

### 5. `decode()` uses `.expect(...)` after length checks (very minor)

`try_into().expect("…")` calls are unreachable given the explicit
length checks above them. Idiomatic alternatives: `.unwrap()` inside
already-bounded slices, or `tail.split_first_chunk::<8>()` etc. Style
only; current code is correct.

## Suggested checkpoint scope before landing

If you want this codec slice to land independently before writers and
readers, the bar is:

1. Address the test coverage gap (finding 4) — most of the cost.
2. Add the `centroid_ordinal` cross-reference comment (finding 1).
3. Drop the carry-forward TODO comments (finding 2).

Findings 3 and 5 are stylistic; don't block on them.
