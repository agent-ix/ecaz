# Review: SPIRE Placement Local Node Zero (30824)

## Summary

Closes the 30817 P2: `ec_spire_placement.node_id` constraint relaxes
from `> 0` to `>= 0` in both `sql/bootstrap.sql` and the
`0.1.0--0.1.1` upgrade SQL. Tests now insert and round-trip both a
remote `node_id = 2` row and a local `node_id = 0` row through both
the direct catalog INSERT and the
`ec_spire_register_placement_batch(...)` path. The catalog test
also asserts the constraint text via `pg_constraint`, locking the
contract against silent regression.

The three review-focus questions:
- `node_id >= 0` is the right invariant: yes — matches
  `meta::SPIRE_LOCAL_NODE_ID = 0` and aligns the placement
  directory's identity space with the rest of the codebase.
- Tests cover both direct writes and the batch helper: yes.
- No other constraints need adjustment: agreed for v1 — `pk_value`
  non-empty, `centroid_id >= 0`, `served_epoch > 0`, and 16-byte
  `source_identity` all hold for local placements identically to
  remote ones.

## Findings

### P3 — `pg_constraint` LIKE-match locks against the literal text

The new assertion uses `pg_get_constraintdef(oid) LIKE '%node_id >=
0%'`. That depends on the constraint definition emitting that exact
spacing. Postgres normalizes constraint definitions consistently, so
this is robust — but worth a comment that the test asserts the
*relaxed* form, distinguishing it from the previous `> 0` form so a
future "cleanup" doesn't simplify the LIKE to just `>%` and lose the
regression guard.

### P3 — Test data uses bytea-encoded PKs `01` and `02`

Consistent with 30817's tests. Worth pinning in the docs slice that
single-byte `pk_value`s are illustrative; real PK encodings will be
the type's binary form. Already noted for follow-up; not regressed
here.

## Reviewer recommendation

Accept.
