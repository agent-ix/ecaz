# 30731 — SPIRE production compact merge handoff review

Validation pass on `1c079675` (`Add SPIRE production compact merge
handoff`). Accepts the C5 precursor.

## What I checked

- `SpireRemoteProductionDispatch` now retains the ready
  `SpireRemoteSearchCandidateBatch` on `CandidateReceiveReady`
  transitions and clears it on `CandidateReceiveFailed` — the
  executor owns the batch so C5 doesn't need parallel state.
- `SpireRemoteFanoutExecutor::merge_ready_candidate_batches(...)`
  exists at `src/am/ec_spire/root/remote_candidates.rs:2830` and
  is reachable via `executor.merge_ready_candidate_batches(limit)`
  at line 3196. The merge reuses the existing remote candidate
  merge contract (the same path the diagnostic libpq merge uses),
  so production and diagnostic merge cannot drift.
- Strict-merge precondition: rejected if any dispatch is still
  `Planned`, `TransportReady`, `TransportFailed`,
  `CandidateReceiveFailed`, or `BlockedBeforeDispatch`. Only
  `CandidateReceiveReady` is accepted. This locks the
  receive-to-merge handoff invariant from packet 30730 in code.
- The hard-coded `remote_heap_resolution` string is now a named
  executor-step constant — small but legitimate cleanup that
  lets C2/C5 reference the step name without duplicating string
  literals.
- AM scan callback wiring deliberately not yet performed; merge
  helper is callable and tested but not yet on the AM scan path.
- `cargo test production_executor_` reports 11 passed (up from 9
  in packet 30728), so the new merge helper has its own focused
  unit tests.

## Answers to the packet's review questions

- **Storing `SpireRemoteSearchCandidateBatch` on the ready dispatch
  the right owner boundary for AM scan integration?** Yes. The
  alternative — letting C5 maintain parallel candidate-batch state
  — would force two-place bookkeeping that the executor's state
  transitions cannot reason about. With the batch on the dispatch,
  the state machine has a single source of truth for the
  candidate-batch lifecycle.
- **Strict merge reject all non-ready dispatch states, or tolerate
  any pre-C4 degraded shape?** Strict reject is correct. C4 owns
  degraded skip semantics; bundling skip-tolerance into the merge
  helper would couple two contracts that should land separately.
  When C4 lands, it should add a *named* method (e.g.
  `merge_ready_or_degraded_skipped_dispatches`) rather than
  loosening this strict helper, so callers can tell which
  contract they're invoking.
- **Compact merge helper narrow enough, or should the next slice
  expose a SQL-visible summary?** Narrow is correct for now. C5
  AM scan wiring is the natural place for a SQL-visible merge
  summary — exposing one before AM scan integration would create
  a third surface (planner, executor, AM) that all need to stay
  in sync. Wait for C5.

## P2 — close before C5 AM scan integration

1. **Cancellation interaction with retained batches.** When C2
   adds cancellation as a transition, a `CandidateReceiveReady`
   dispatch carrying a candidate batch may be cancelled before
   merge. The C2 packet should specify whether cancellation
   clears the retained batch (matching `CandidateReceiveFailed`
   behavior) or freezes it as-is. Recommend clearing on cancel,
   so a cancelled dispatch cannot accidentally contribute to a
   later merge. Not a 30731 issue per se, but the contract
   choice should be made before C2 lands so this slice's batch
   ownership doesn't accidentally outlast cancellation.

## P3 — informational

2. **Test coverage for non-ready merge rejection.** The 11 unit
   tests cover the happy-path merge plus state transitions.
   Worth confirming there's a test for each rejected state
   (Planned, TransportReady, TransportFailed,
   CandidateReceiveFailed, BlockedBeforeDispatch) — those are
   the contract's negative cases and they should be locked in
   tests rather than implementation. Optional if already covered
   by the 11 tests; check the test names.
