# Review: SPIRE Read Pass-Through and Local Recall Fix (30980)

## Summary

**Resolves both remaining 30978 blockers in one packet.** Two
distinct bugs, one commit:

1. **Bench client false rejection**: ordinary SPIRE
   `SELECT` reads were being failed by the DML frontdoor
   guard. Now non-PK SELECTs pass through as
   `non_pk_select_pass_through`; PK-select frontdoor edge
   shapes (e.g., `id IN (...)`, `id = numeric`, `OR`
   equality) still fail closed.
2. **Recall=0 root cause**: production heap-resolution
   exact rerank was returning **raw positive inner
   product**, opposite of SQL `<#>` operator semantics. Now
   returns **negative inner product**, matching `<#>`
   ordering. Recall jumps from 0.0 to 1.0.

`ecaz bench spire-pipeline` completes end-to-end with
`recall@k = 1.0000` and full bench artifact (endpoint
tuple transport readiness, p50/p95/p99, route/candidate/
heap counters, local-store I/O counters, fanout status).

## Findings

### P1 — Bug 2 (sign error) is the most important finding

The recall=0 result from 30978 was a real correctness
alarm, not just a measurement glitch. Production heap
rerank returning **the wrong sign** of inner product means:

- **Worst rows ranked first**: highest positive inner
  product (most similar) was treated as "lowest score"
  under the `<#>`-style ordering convention (where lower
  is better).
- **All recall comparisons inverted**: truth set ranked by
  `<#>` (lower = better match), production-path output
  ranked by raw inner product (higher = better match).
  Comparing them gave zero overlap.
- **Local-only manifestation**: this would NOT show up in
  remote candidate paths (which use a separate
  `remote/local heap candidate merging` code path that
  apparently already had the right sign). It only bit
  the local AM output and the local-rerank join with
  remote candidates.

The fix — returning negative inner product to match SQL
`<#>` ordering and the remote candidate score semantics
— makes the local and remote paths agree on what "lower
is better" means. This is the right shared-convention
primitive.

**Critical question for the reviewer-focus:** does this
bug affect any *prior* Stage E or fixture results?

- Unit fixtures using exact-rank assertions on small
  toy data may have been false-positives (the wrong sign
  could still pass a fixture that asserted the wrong
  ranking).
- Stage E lifecycle/fault matrices are about presence/
  absence of rows, not ranking, so they wouldn't catch
  this.
- The 30975 measurement was throughput, not recall —
  unaffected.
- The 30961 Stage E rerun didn't measure recall — also
  unaffected.

So no prior packet is invalidated. But worth explicitly
checking: **were there any fixtures asserting top-k
ordering that "passed" because they were wrong about the
expected order?** If yes, those fixtures need re-validation
under the new (correct) sign. If the test suite passes
unchanged after this fix, then no — fixtures were testing
presence/count, not order.

This is the kind of bug that AWS verification would have
caught (recall would have been 0 there too), so finding it
in Phase 12 instead of Phase 13 is actually a discipline
win.

### P1 — Bug 1 (frontdoor over-rejection) is the right scope split

The original DML frontdoor was over-broad: it was
treating *any* `SELECT` against a SPIRE-indexed table as
needing PK-select frontdoor classification, when really
only PK-shaped predicates need it.

The new boundary:

- **Non-PK SELECT** (e.g., `ORDER BY embedding <#>
  $query LIMIT k`) → `non_pk_select_pass_through` (let
  CustomScan or normal scan handle it).
- **PK-select frontdoor edge shapes** (`id IN (...)`,
  `id = numeric`, `OR` equality) → fail closed (existing
  Phase 12.6 negative-classifier behavior preserved).
- **PK-select valid shape** (`WHERE id = $1::int8`) →
  PK-select frontdoor handling (existing).

This is the right three-way split. The original code
collapsed (1) and (2) into one rejection. The 30978
blocker found that collapse, this packet fixes it.

**Confirm the regression risk**: the Phase 12.6 negative
classifier fixtures (30929 — `IN`, NULL, OR, numeric
coercion, etc.) should still pass. The frontdoor bypass
for non-PK SELECTs must NOT also bypass these PK-edge
rejections. From the request's "PK-select predicate
detection still catches malformed PK-frontdoor shapes,"
this is preserved. Worth confirming the test suite runs
the 30929 fixtures unchanged with green results.

### P1 — Both fixes ship together

Bundling these into one commit is the right call:

- The non-PK pass-through alone wouldn't surface the recall
  bug (without the bench client running, the recall path
  isn't exercised).
- The sign fix alone wouldn't be measurable (without bench
  client running, can't measure recall).
- Together: bench client runs, recall is measurable, sign
  fix takes effect, recall@k = 1.0.

The fact that **both bugs were latent and surfaced by the
same readiness-bundle attempt** is the expected outcome of
the readiness work — it's specifically supposed to find
this kind of integration-level bug. 30978's discipline
(name blockers, don't force closure) was what enabled
this packet to address the right things.

### P2 — Recall=1.0000 on one-list/full-rerank

Recall=1.0 on a one-list/full-rerank fixture is the
expected result for that configuration — every query
should exact-match because the entire corpus is reranked
exactly. Anything less than 1.0 would indicate either
truth-set divergence or product-path bugs.

That this now reports 1.0 is the load-bearing proof that
the sign fix is correct. The same fixture reporting 0.0
before and 1.0 after, with no other change to the
truth-set or pipeline, attributes the change to the sign
fix unambiguously.

### P2 — Bundle row consideration

The request asks: "Confirm packets 30978, 30979, and 30980
together satisfy the Phase 12.9 local readiness artifact
rows without making AWS/product-scale claims."

My read:

- **Live packet-local artifact capture row**: yes, this
  packet provides the evidence. Closure justified.
- **Final production-readiness bundle row**: borderline.
  The three packets together do contain bundle-shaped
  evidence, but a "final bundle" packet typically means a
  single curated artifact set with a manifest. The
  evidence is split across three packets; an operator
  looking for "the bundle" would need to know to read
  all three.

Recommendation: a small follow-up packet that **assembles
a manifest pointing at the three** as the bundle, with
explicit "local production-readiness smoke" labels per
30949's evidence-tier rules, would close the bundle row
honestly. Alternatively, if the tracker reads "bundle =
the union of these three packets," and the README/runbook
says where to find them, that's also acceptable.

Either way, scope-discipline-wise the closure must avoid
claiming AWS/product-scale evidence — these are local
loopback packets per 30949's middle tier ("local
production-readiness smoke"), not Phase 13 AWS evidence.
