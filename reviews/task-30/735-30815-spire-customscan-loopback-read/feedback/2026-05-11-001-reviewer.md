# Review: SPIRE CustomScan Loopback Read Fixture (30815)

## Summary

Major milestone slice. The first end-to-end PG18 fixture proves
`EcSpireDistributedScan` returns a remote-origin row through the
ADR-068 tuple-payload path with no row-materialization catalog or
register call on the read path. The fixture sets up two real heap
relations (coordinator + loopback remote), rewrites coordinator
placements to `node_id = 2`, registers the remote descriptor, asserts
`Custom Scan (EcSpireDistributedScan)` in EXPLAIN, and verifies the
returned `(id, title)` tuple is the **remote** alpha row, not the
coordinator row.

Three secondary improvements ride along:
- Adds `heap_block` / `heap_offset` to
  `ec_spire_remote_search_tuple_payload(...)` for executor handoff.
- Replaces the duplicate-CTID-fragile HashMap from 30812 with a Vec +
  zip pattern preserving positional order, plus a row-count guard. P2
  from the 30812 review is closed.
- Narrows `custom_scan_tuple_payload_columns` to the projected target
  list when simple `Var`s are extractable, falling back to the full
  relation descriptor otherwise. P1 from the 30814 review is partially
  closed — see findings.

The four review-focus questions:
- `heap_block` / `heap_offset` on the endpoint: right shape; the
  production executor needs these for the deduplication / tie-break
  invariants downstream.
- Positional resolution: correct. The new "rows returned ≠ CTIDs sent"
  assert is good defense against unexpected SQL behavior.
- Projected-column selection: works for the common case; doesn't yet
  cover INCLUDE-only queries or expressions over `Var`s. Acceptable
  fallback.
- Loopback vs separate-instance lane: noted as still open.

## Findings

### P2 — Projection narrowing falls back to full relation on any expression

`custom_scan_tuple_payload_columns` collects `attr_numbers` from
target-list entries that are non-junk simple `Var`s (`varattno > 0`).
If *any* target entry is an expression (e.g., `SELECT id+1, title
FROM ...`), the simple-Var pass yields a partial set, but only adds
those it found — actually re-reading: `if !attr_numbers.is_empty() &&
!attr_numbers.contains(&attnum)` — so if at least one Var is found,
columns are filtered to *just* those Vars and the expression's input
columns are dropped. That breaks any expression that depends on a
column not also referenced as a bare `Var` in the target list.

Concrete failure case:
```sql
SELECT title || ' (boosted)' FROM tbl ORDER BY embedding <#> $1 LIMIT 1
```
- `target_list` has one entry, the `OpExpr` for `||`.
- The simple-Var collector finds none.
- `attr_numbers` is empty, so the code falls back to all relation
  columns. OK, this case is safe.

But:
```sql
SELECT id, title || ' (boosted)' FROM tbl ORDER BY embedding <#> $1 LIMIT 1
```
- `target_list`: `id` (Var) and `OpExpr(||, title, ...)`.
- `attr_numbers = {id_attnum}`.
- `title` attribute is dropped from the requested column list because
  the OpExpr is not a bare `Var`.
- Remote payload missing `title`; expression evaluates against NULL.

Suggested fix: walk expressions for `Var` references too (use
`pull_varattnos` or `expression_tree_walker`) so the requested column
set is the union of all `Var`s in the targetlist regardless of
position. Alternatively, fall back to "all columns" when *any* target
entry is non-`Var`, which is safer but less efficient.

Without this, the fixture happens to work because the SELECT projects
only bare `Var`s; the bug is masked.

### P3 — Fixture relies on a session-shared loopback conninfo env var

The test sets `EC_SPIRE_REMOTE_CONNINFO_SPIRE_REMOTE_CUSTOMSCAN_PAYLOAD`
under `_env_lock`. That works under the existing serialized env-var
test pattern, but the conninfo string gets baked into the registered
remote descriptor for the lifetime of the test database. If a parallel
or follow-up test reuses the same descriptor name, results may bleed
across test cases. Consider naming the descriptor with a per-test
unique suffix derived from the test name.

### P3 — `_conninfo_secret` is held for the full test scope but not consumed

`let _conninfo_secret = ScopedEnvVar::set(...)` parks the guard for
RAII unset-on-drop. That's the right pattern for env-scoped tests; a
short comment noting that the underscore-bind is intentional (not a
"discarded" value) helps a future reader.

### P3 — Loopback fixture asserts only the first ranked row

The assertion `assert_eq!(row, (10, "remote alpha".to_owned()))` proves
the ranked-1 row is the remote shard's alpha. Worth a follow-up that
asserts top-2 for both `(10, "remote alpha")` and `(20, "remote beta")`
to prove the remote path returns multiple rows in correct order, not
just the single top result. Not blocking — the single-row proof is the
load-bearing claim — but the multi-row case is the next fixture.

### P3 — Empty CTID short-circuit added; helpful

`if ctids.is_empty() { return Ok(Vec::new()); }` closes the P3 from
30812. Good follow-through.

## Reviewer recommendation

Accept. The P2 projection-narrowing gap should land before the
multi-instance fixture exercises any non-trivial projection, otherwise
expression-based SELECTs will silently return NULL for inputs.
