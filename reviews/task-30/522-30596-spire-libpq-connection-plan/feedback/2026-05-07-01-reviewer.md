# Libpq connection + dispatch plan review

`ec_spire_remote_search_libpq_connection_plan` and `_dispatch_plan`
plus their summaries. The dispatch plan threads pre-I/O actions
into the coordinator gate, which is the right shape for the
future executor. **Inherits the descriptor_state filter bug from
30595.**

## Findings

### 1. Inherits the 30595 descriptor_state filter gap

The connection plan's readiness check is `descriptor.is_some()`,
with no descriptor_state gate. A `disabled` or `failed`
descriptor row will surface as
`conninfo_resolution = ready`, `pipeline_mode = libpq_pipeline`.
See 30595 finding #2 for full context — the fix lands in
`load_remote_libpq_connection_descriptors`, which is consumed
here. Once 30595's loader is fixed, this surface inherits the
fix.

### 2. Dispatch plan reports `open_pipeline_and_send_remote_search`
   for descriptors that exist but may not be in a readable state.
   Same root cause as #1 — the dispatch action should be gated on
   `descriptor_state ∈ {active, draining}`, not just descriptor
   presence.

## Confirmations

1. `ec_spire_remote_search_libpq_connection_summary` and
   `_dispatch_summary` correctly aggregate
   descriptor-resolved / missing-descriptor / pipeline /
   remote-PID / blocked-PID counts through the existing rollup
   helper.
2. The dispatch plan exposes the receive validator function
   name (`validate_remote_search_candidate_batch`) — same
   pattern as 30568's libpq result contract, eliminates drift
   between the executor and the diagnostic claim.
3. Coordinator gate now reports libpq dispatch count/status, so
   one top-level query can answer "is the next blocker
   descriptor, transport, or already-ready?".
4. SQL prelude correctly fail-closes for missing descriptors;
   the bug is specifically that `disabled`/`failed` descriptors
   are not "missing" but should still fail closed.
