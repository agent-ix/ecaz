# Target/request/summary plan SQL review

Three layered SQL projections over the 30557 fanout planner:
target-grouped, request-shape (binds query/top-k/endpoint metadata),
and request summary. No blocking findings.

## Confirmations

1. Target rows preserve fanout planner's `BTreeMap`-derived node-id
   ordering and selected-PID order within a node, so the libpq
   executor will get a stable input contract.
2. Skipped degraded placements are grouped by `(node_id,
   placement_state)` and surfaced as distinct rows from executable
   local/remote targets, matching 30559's receive-validator stance
   that skipped placements never enter the merge path.
3. Request-plan rows correctly bind targets to
   `ec_spire_remote_search` shape (query dimension, top-k budget,
   consistency mode, endpoint function) without copying the query
   vector into the row — `endpoint_function = "none"` for
   degraded-skipped is the right marker.
4. Summary statuses (`ready`, `requires_libpq_transport`,
   `degraded_ready`, `empty_top_k`) match the rest of the Phase 7
   diagnostic stack.

## Findings (non-blocking)

- **Status string proliferation.** Across this packet and 30558, 30562,
  30563, 30565–30568 there is a growing vocabulary of literal status
  strings (`ready`, `requires_libpq_transport`, `degraded_ready`,
  `degraded_skipped`, `empty_top_k`, `requires_remote_node_descriptor`,
  `local_ready`, `no_candidate_batches`, etc.). Once libpq lands and
  these strings start being parsed by the executor or by external
  operators, drift between packets will be hard to detect.

  Recommend pulling these into a single
  `pub(crate) const SPIRE_REMOTE_STATUS_*: &str` block (analogous to
  the existing `SPIRE_REMOTE_SEARCH_RECEIVE_VALIDATOR` constant), so
  there is one source of truth and the constants can be re-exported
  to documentation. Not blocking, but the longer this proliferates
  the more painful the consolidation gets. Cross-cutting summary on
  30568.
