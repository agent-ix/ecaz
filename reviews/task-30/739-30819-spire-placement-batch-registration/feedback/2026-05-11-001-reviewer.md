# Review: SPIRE Placement Batch Registration (30819)

## Summary

Adds the ADR-069 bulk-load post-write registration primitive: the
`ec_spire_placement_entry` composite type and the
`ec_spire_register_placement_batch(index_oid oid, entries
ec_spire_placement_entry[]) RETURNS bigint` SQL function. The
function `unnest`s the entries array and INSERTs into the catalog
from packet 30817, returning the inserted row count. Both bootstrap
and the `0.1.0--0.1.1` upgrade get the type and function. Strict
INSERT semantics rely on the catalog's primary key for duplicate
detection.

The three review-focus questions:
- Composite shape matches ADR-069: yes — five fields in the
  ADR-specified order and types.
- Strict insert acceptable for v1: yes for the bulk-load primitive
  (bulk callers should be inserting fresh rows). Pin the semantics
  if any caller needs upsert.
- Belongs in both bootstrap and upgrade SQL: yes — same install
  surface, same migration path.

## Findings

### P2 — Upgrade SQL should be defensive against re-runs

`CREATE TYPE ec_spire_placement_entry AS (...)` and
`CREATE FUNCTION ec_spire_register_placement_batch(...)` in
`ecaz--0.1.0--0.1.1.sql` will fail if a partial upgrade or a manual
DDL touched these objects already. The other definitions in the
same upgrade file (e.g., the row-materialization table earlier)
appear to be similarly unguarded, so this is consistent with the
file's style — but worth pinning whether `CREATE OR REPLACE
FUNCTION` and a guarded `DO $$ BEGIN CREATE TYPE ... EXCEPTION
WHEN duplicate_object THEN NULL; END $$;` are the project's
preferred upgrade pattern. If extension upgrade scripts only run
once per version this is moot; if they can be re-applied, the lack
of guards bites.

### P2 — `STRICT` semantics interaction with NULL elements in the array

The function is declared `STRICT`, so a NULL `entries` argument
short-circuits to NULL. Good. But individual array elements that
are NULL (`ARRAY[NULL::ec_spire_placement_entry, valid_row]`) pass
the strictness check (the array itself is not NULL) and `unnest()`
yields a NULL composite. The downstream INSERT then sees NULL
values for every column, which fail one-by-one against the
`NOT NULL` constraints from the catalog. Operators get a column-level
NOT NULL error rather than "entries[N] is NULL".

Suggested fix: add a `WHERE entry IS NOT NULL` clause (or, better,
explicitly assert no NULL elements with a CASE-RAISE) so the error
message points at the array index that is malformed. Optional —
the existing failure is loud, just not informative.

### P2 — No transaction-boundary guarantees stated

`ec_spire_register_placement_batch` is a SQL function (not
procedure), so it runs in the caller's transaction; the entire
batch INSERT either commits or rolls back atomically. That's
correct for the bulk-load semantics ADR-069 implies. Worth pinning
in a docs slice that the function is transactional and that callers
should not call it in autocommit mode if they expect partial-batch
recovery — otherwise a single PK conflict near the end aborts the
whole batch.

### P3 — `RETURNING 1` vs `RETURNING *`

`RETURNING 1` is fine and minimizes work. Slight readability win
to write `RETURNING index_oid` (or any single small column) so a
future reader sees what's being returned without having to guess
the wire-format. Cosmetic.

### P3 — Composite type ordering pins the wire format

The `ec_spire_placement_entry` field order is `(pk_value, node_id,
centroid_id, served_epoch, source_identity)`. Once any caller
constructs `ROW(...)::ec_spire_placement_entry` literals, the order
is locked — any future field addition has to be at the end.
`ALTER TYPE ... ADD ATTRIBUTE` doesn't exist in PostgreSQL for
composite types; the only path forward is `DROP TYPE` then
`CREATE TYPE`, which breaks dependent functions and any application
that uses the literal form. Pin "the entry shape is frozen for v1;
extensions go through a new typename if needed" so a future packet
doesn't try to add fields in place.

### P3 — Test exercises only the happy path

`test_ec_spire_register_placement_batch_sql` proves two valid rows
land. Worth adding regression tests for:
- Empty array input (`ARRAY[]::ec_spire_placement_entry[]`):
  expected return 0.
- Duplicate `pk_value` within the same batch: expected PK-conflict
  error message.
- Invalid entry (e.g., `node_id = 0` or `source_identity` of wrong
  length): expected catalog `CHECK` constraint violation.

These are quick adds that lock the contract and would catch a
relaxation of the catalog constraints downstream.

## Reviewer recommendation

Accept. The P2 NULL-element handling and the upgrade-script
defensiveness are worth folding into the next write-path packet
that calls this primitive in anger; the P3 items are docs / test
polish.
