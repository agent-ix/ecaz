# Feedback: SPIRE Auxiliary Local Store Relations (30527)

Code review of `4be77293` at HEAD `f8159e49`. Files inspected:
`src/am/ec_spire/storage/relation_plan.rs`, `src/am/ec_spire/page.rs`,
`src/lib.rs`.

The structural pieces — single-store compatibility, internal dependency
direction, `CommandCounterIncrement` placement, and same-tablespace
support for the same-device baseline — all look correct. Catalog
sequence is `heap_create_with_catalog` → `recordDependencyOn(INTERNAL)`
→ `CommandCounterIncrement` → `relation_open(AccessExclusiveLock)` →
`initialize_root_control_page` → close. That ordering is fine.

Two concerns, one serious.

## 1. Autovacuum interaction with a `RELKIND_RELATION` heap that is
   not actually a heap — **needs action before any real workload**

The aux stores are created with:

- `relkind = RELKIND_RELATION`
- `accessMethod = HEAP_TABLE_AM_OID`
- standard `PageInit` block layout (page.rs:59, 350)
- SPIRE object bytes placed in line-pointer slots via
  `PageAddItemExtended` (page.rs:295, 361)

PostgreSQL will treat these relations as ordinary heap tables for
all catalog-driven operations — including autovacuum. autovacuum
walks `pg_class`, finds these as RELKIND_RELATION, and at the
configured threshold will dispatch the heap_am vacuum callbacks
against them. heap vacuum will:

- read each block via the buffer manager (fine — pages have valid
  PageHeaderData);
- iterate line pointers via `PageGetItemId` and decode the item bytes
  as `HeapTupleHeaderData`;
- evaluate `HeapTupleSatisfiesVacuum` against the synthesized
  `(xmin, xmax, t_ctid, t_infomask, t_hoff, t_bits)` it reads from
  whatever your SPIRE object bytes happen to contain at those offsets.

The first ~24 bytes of every SPIRE object tuple are about to be
reinterpreted as a heap tuple header by an unrelated subsystem. The
outcomes range from "autovacuum silently does nothing useful" to
"autovacuum decides every tuple is dead and tries to compact pages"
to "ERROR: invalid memory alloc request size" or worse. No path
through that decode is correct, because the bytes are not a heap
tuple.

This will not show up in the current test matrix because tests don't
trigger autovacuum. It will show up the first time a real workload
sits on a multi-store SPIRE index for long enough to cross
`autovacuum_vacuum_threshold`.

The packet description says PostgreSQL rejected `RELKIND_INDEX`
sidecars because there is no matching `pg_index` row. That is the
right diagnosis but the wrong landing point. Options, in rough order
of preference:

1. **Disable autovacuum on the aux relations.** Set the
   `autovacuum_enabled = false` reloption at creation. The setting
   is per-relation and applies to both autovacuum and autoanalyze.
   Cheapest fix, contains the blast radius. Does not stop a manual
   `VACUUM <auxrel>` from breaking things — that needs (3) too.
2. **Add a real `pg_index` row for each aux relation** so they can
   carry `RELKIND_INDEX`, with a synthetic indrelid pointing at the
   index's heap. This is closer to what aux indexes actually are
   (extension-internal sidecars of an index). It makes them invisible
   to autovacuum and gives the right semantic for `pg_dump`,
   `pg_class.relkind` filters, and CLUSTER. More invasive.
3. **Reject `VACUUM <auxrel>` in a hook.** Not sufficient on its own
   — autovacuum is dispatched by the launcher, not by a user-level
   command — but useful as a defense-in-depth wrapper around (1).

Whichever path you take, please add a test that toggles
`autovacuum_naptime` low and proves that running PG with autovacuum
enabled against a populated multi-store SPIRE index does not corrupt
or panic. Without that, this is a latent failure mode that the
existing Phase 4 SQL VACUUM coverage (30535) does not catch — that
test exercises the SPIRE *index* vacuum callback through
`amvacuumcleanup`, not the heap-am autovacuum dispatched against the
*aux relation*.

This is the only blocking-class issue I see in this review.

## 2. `initialize_root_control_page` on aux stores — naming smell

Each aux relation gets `initialize_root_control_page(aux_relation,
SpireRootControlState::empty())` immediately after creation
(relation_plan.rs:236). Functionally this reserves block 0 with the
shape the existing object-tuple append/read helpers expect. Logically
the aux store does not own a "root control" — only the index relation
holds the active epoch manifest.

So this is using a function named after a single role to do double
duty as "reserve block 0 with the SPIRE metadata header." That works,
but the name is now misleading: someone reading
`initialize_root_control_page` on an aux store will assume the aux
store has an independent root control. It does not — that empty state
on the aux relation is dead state by design.

Suggested cleanup (not blocking): rename to
`initialize_spire_metadata_block_zero` or split into two helpers —
`initialize_root_control_page_for_index_relation` and
`initialize_metadata_page_for_aux_store`, where the aux variant is
explicit about leaving the root control empty. The alternative —
naming the role something neutral like
`reserve_spire_metadata_block_zero` — is probably the right move
since the helper is now used for two semantically different things.

## 3. Minor — REINDEX semantics

`recordDependencyOn(store, index, INTERNAL)` ties drop lifecycle
correctly. Worth confirming explicitly that REINDEX on the root index
is not a code path SPIRE supports today — INTERNAL deps do not
participate in reindex, so a REINDEX would rebuild the index relation
in place and leave the aux relations as an orphaned, stale snapshot
of the previous build. If REINDEX is intended to be unsupported for
multi-store SPIRE indexes, reject it explicitly in the appropriate
ambuild/amrebuild path with a clear error rather than letting the
user produce a quietly broken multi-store index. If supporting it is
on the roadmap, that is a future-task tracker entry.

This is forward-looking, not a blocker on this packet.

## Validation gaps to consider

- No test runs autovacuum against a populated multi-store index
  (concern 1).
- No test exercises a manual `VACUUM <auxrel>` (would also be
  blocked by concern 1, but should be tested as an explicit
  rejection).
- No test exercises `REINDEX` against a multi-store SPIRE index
  (concern 3).
- The two-store fixture is excellent for the build/scan path — these
  three gaps are catalog-lifecycle paths that the fixture does not
  cover.

## Net

Build/scan/diagnostics shape is right. The autovacuum concern is real
and needs to land before any production-shaped workload sits on a
multi-store SPIRE index. The naming and REINDEX items are cleanup, not
blockers.
