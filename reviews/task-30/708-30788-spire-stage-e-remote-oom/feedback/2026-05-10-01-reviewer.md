## Answers to the packet's review questions

- **SQLSTATE 53200 acceptable local fixture for the remote-
  OOM row without forcing an actual memory exhaustion
  event?** Yes. The only thing the fixture needs to prove
  is that an OOM-class remote SQL error flows through the
  transport classifier correctly and surfaces a sanitized
  category. Actually triggering an OOM would require
  remote-side memory pressure injection (e.g. allocating
  GBs in the remote backend), which adds infrastructure
  cost with zero behavioral payoff — the classifier
  doesn't care *why* the SQLSTATE was raised, only that
  it was.
- **Does the evidence prove sanitized coordinator
  categorization well enough?** Yes. The coordinator-
  visible transport row shows only `remote_query_failed`
  — the raw SQLSTATE 53200 text from the remote backend
  does not leak through the transport row. That's the
  contract from packet 30725's failure-category mapping
  (categories are normalized strings, not raw error
  text).
- **Strict/degraded state handling aligned?** Yes — same
  shape as packets 30783 / 30784: transport-stage
  failures convert to `DegradedSkipped` with category
  preserved when degraded mode is active.

   in a future taxonomy refinement.** Right now SQLSTATE
   `53200` and other resource-exhaustion conditions fold
   into the generic `remote_query_failed`. Operators

   existing taxonomy if real operator feedback shows
   confusion. Optional — not blocking; the current
