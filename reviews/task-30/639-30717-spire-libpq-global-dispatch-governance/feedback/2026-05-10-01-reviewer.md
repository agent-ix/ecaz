## What I checked

  - `ec_spire.remote_search_max_concurrent_dispatches`
  - `ec_spire.remote_search_max_concurrent_dispatches_per_node`
  Each has a `current_session_remote_search_max_concurrent_*`
  accessor (lines 796–800), parallel to the per-query budget
  GUCs from packet 30715.
- Saturated governance produces `remote_executor_overload` status
  with `next_blocker = remote_executor_governance` — the runtime
  blocker is distinct from the pre-dispatch
  `remote_executor_budget` blocker, so an operator can tell

  irrelevant secret/connection failures.
- The mechanism is `pg_try_advisory_lock`-based and nonblocking, so
  a saturated slot trips immediately with the overload status
  rather than queueing inside the executor and consuming a backend
  worker slot.
- `ec_spire_remote_search_libpq_executor_budget_summary(...)` now
  reports the active concurrency caps alongside per-query fanout
  and timeout caps; one operator query covers both governance and
  per-query admission.

  `am::remote_search_libpq_global_governance_advisory_key_for_test`

- `plan/design/spire-libpq-executor-budget.md` and Phase 11 Stage C

## Answers to the packet's review questions

- **Advisory-lock slot admission acceptable as first local
  cross-backend governance primitive before async/pipeline lands?**
  Yes. PostgreSQL advisory locks are session-scoped, lightweight,
  and visible in `pg_locks`, which gives operators a free
  diagnostic surface (number of held vs requested) without us
  building one. Nonblocking `pg_try_advisory_lock` is the right
  variant — a blocking variant would queue inside a backend
  worker and complicate cancellation. A future async/pipeline
  executor can replace this with an in-process semaphore once the
  executor owns its own queueing discipline; until then,
  cross-backend coordination is what we actually need.
- **Saturated governance shares `remote_executor_overload` status
  but uses `remote_executor_governance` as the runtime blocker,
  distinct from pre-dispatch `remote_executor_budget`?** Yes.
  Sharing the status keeps the strict-mode fail-closed posture
  uniform; splitting the blocker keeps the two failure causes
  diagnosable. Operators should be able to query "all overload
  events" by status and "concurrency-driven" specifically by
  blocker.
- **Remaining Stage C split right?** Yes. Cancellation
  propagation and true overlapped libpq execution are the next
  pieces; both need executor-state plumbing that builds on this
  governance primitive.

## P2 — close before async/pipeline executor lands

1. **Lock-class registration and conflict policy.** PostgreSQL
   advisory locks share a 64-bit namespace per database. The
   coordinator should publish in
   `plan/design/spire-libpq-executor-budget.md` (or a small ADR)
   the exact `(class_id, object_id)` ranges this governance uses
   so other extensions or operator scripts cannot accidentally
   collide. The test fixture already references
   `remote_search_libpq_global_governance_advisory_key_for_test`,
   which is fine for tests; production needs a documented range.

## P3 — informational

2. **Operator diagnostic note in the runbook.** Stage G runbook
   should mention that `pg_locks WHERE locktype = 'advisory'` is
   the way to inspect the SPIRE coordinator's current concurrency
   utilization. Free observability we should not hide.

3. **Symmetric per-node accounting.** The per-node concurrency cap
   bounds dispatches by `node_id`. Worth confirming with a future
   fixture that two distinct `node_id`s with the same per-node cap
   admit independently — i.e., one node saturated does not block a
   second node's dispatch — to lock the per-node-vs-global
   isolation invariant. Optional; the fixture as is verifies the
   global cap.
