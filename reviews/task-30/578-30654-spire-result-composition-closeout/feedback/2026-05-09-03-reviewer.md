## `3cb45efc` regression fix audit (5 issues found, all real)

The broader pg18 pass exposed five genuine pre-existing bugs that the narrow
2026-05-09 closeout test did not exercise. All five fixes are correct.

### 1. Assignment input validation gap — `src/am/ec_spire/assign.rs`

**Pre-fix:** `validate_assignment_input` logic lived inline in
`build_assignment_row` but was *not* called by
`build_boundary_assignment_placements` or `build_allocated_assignment_rows`.

**Risk:** boundary-replica and pre-allocated assignment paths could accept
invalid heap_tid (`ItemPointer::INVALID`), non-finite gamma, or
encoded-payload lengths exceeding `u32`. These would burn vec-ids from the
local allocator before the eventual error surfaced downstream — i.e. silent
allocator drift on bad input.

**Fix:** extract `validate_assignment_input` as a free function, call it from
all three construction sites. Behavior preserved for the existing primary
path; new validation closes the gap on the other two. **Sound.**

### 2. Recursive primary placement diverged from centroid plan — `src/am/ec_spire/build/recursive.rs`

**Pre-fix:** `build_recursive_leaf_rows_by_pid` computed primary PIDs from
`route_map.route_boundary_assignment_for_vector(..., boundary_replica_count)`
on each assignment's source vector, ignoring `centroid_plan.assignment_indexes`
(which records the centroid each assignment was deterministically routed to
during clustering). Boundary plan also returned exactly
`boundary_replica_count` replicas — but those replicas were derived from the
same vector-distance ranking, so the primary slot could disagree with the
centroid plan when re-routing produced a different best leaf.

**Risk:** in recursive (multi-level) builds, primary leaf placement could
silently disagree with the centroid plan that drove the route map's
construction. This would break the invariant that *each centroid's
assignments live at that centroid's leaf*. Visible at scan time as recall
loss when retrieved leaves miss the rows the cost model expects.

**Fix:** look up `primary_pid` directly from
`route_map.get(assignment_index).pid` (deterministic, matches the clustering),
then call boundary routing with `boundary_replica_count.saturating_add(1)` and
filter out the primary from the returned set, taking up to
`boundary_replica_count` distinct replicas. The `+1 / filter / take` shape
keeps replica count semantics intact even when the boundary plan happens to
include `primary_pid` among its top picks.

**Sound,** and the explicit `assignment_indexes.len() != assignments.len()`
guard added at the entry is good defensive shape.

### 3 & 4. Two unit tests reading pgrx GUCs from parallel threads — `src/am/ec_spire/scan/tests/candidates.rs`

**Pre-fix:** Tests used `prepare_single_level_snapshot_scan_candidates`, which
internally resolves session GUCs (`ec_spire.nprobe` etc.). Pure-Rust unit
tests run multi-threaded; pgrx GUC accessors require a SPI context.

**Fix:** decompose at the test level into:
- `count_snapshot_single_level_leaf_pids(...)` for leaf count;
- `resolve_single_level_scan_plan_values(leaf_count, options, -1, -1)`
  (newly made `pub(super)`) for plan resolution with explicit `-1` GUC
  sentinels — bypassing the GUC reader entirely;
- `collect_single_level_scan_plan_reranked_candidates` /
  `collect_top_graph_scan_plan_reranked_candidates` (the latter newly made
  `pub(super)`) for the actual candidate collection.

**Sound.** The visibility relaxations (`pub(super)`) are minimal — both
helpers stay crate-private. The `-1, -1` sentinel call site mirrors the
existing GUC-unset path. Test contract assertions are preserved (leaf_count,
nprobe, candidate ordering, scores). **No production behavior change.**
