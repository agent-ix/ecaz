# Review: SPIRE DML Baserel Mode Wrappers (30879)

## Summary

Three operation-specific wrappers around the generic baserel
primitive-plan helper from 30876:
- `dml_frontdoor_pk_select_primitive_plan_expr_from_baserel`
  (existing, refactored to use the shared guard).
- `dml_frontdoor_update_primitive_plan_expr_from_baserel` (new).
- `dml_frontdoor_delete_primitive_plan_expr_from_baserel` (new).

Shared mode guard returns operation-specific fail-closed errors
when the extracted plan mode doesn't match the requested
wrapper. Each wrapper preserves the same `None` (not a
candidate) vs `Some(Err)` (candidate but malformed) semantics.

The three review-focus questions:
- UPDATE/DELETE wrappers preserve None vs Some(Err) semantics:
  yes.
- Shared mode guard keeps the existing PK SELECT fail-closed
  boundary intact: yes.
- Wrapper-only slice does not enable any new planner path:
  confirmed (UPDATE/DELETE generators not yet wired).

## Findings

None blocking. The three wrappers form the right callable
surface for the upcoming UPDATE/DELETE plan-replace slice (see
the 30803 seq 004 reviewer feedback for the recommended
plan-replace shape).

## Reviewer recommendation

Accept. Wrapper trio + shared guard is the right shape; the
plan-replace slice that follows can call the matching wrapper
per command type without re-deriving operation semantics.
