## Findings (carry-over + new)

### P2 — Placement-existence direct scan correctness

`custom_scan_index_has_sql_placement` opens
`ec_spire_placement`, builds one `ScanKeyData` keyed on
`index_oid` via `F_OIDEQ`, runs `table_beginscan_catalog`, fetches
one row. Several quick correctness notes:

- `table_beginscan_catalog` is the read-only catalog-scan
  variant. Worth confirming `ec_spire_placement` is a regular
  user table (not in the system catalog) — `table_beginscan_catalog`
  may set `rs_flags = SO_TYPE_CATALOG` which influences snapshot
  selection. If the placement table needs MVCC visibility from
  the planner's snapshot (it should), ensure the catalog scan
  uses the right snapshot. PG's `table_beginscan_catalog` uses
  `SnapshotNonVacuumable`-equivalent; works for "is there at
  least one row" but might count uncommitted INSERTs from other
  backends. For the eligibility gate that's actually fine — even
  uncommitted rows mean SPIRE is being used — but worth a one-
  line comment that the gate is intentionally permissive.
- `EC_SPIRE_PLACEMENT_INDEX_OID_ATTNO = 1` hardcodes the column
  position. If the placement table layout ever changes, this
  silently scans the wrong attribute. Worth a `static_assertions`-
  style debug check (or a comment that PK column ordering is
  load-bearing) so a future ALTER doesn't silently break the
  eligibility gate.

### P2 — Sequential scan over `ec_spire_placement`

`table_beginscan_catalog` is a heap (sequential) scan, not an
index scan. For a placement table with millions of rows,
checking "does any row exist for this index_oid" via a seqscan
is O(N) per planner call. The placement directory has the
`PRIMARY KEY (index_oid, pk_value)` btree which would make this
O(log N).

For v1 with small-to-medium placement directories this works.
For ADR-069's v1 production target (which scales with row
count), this becomes a planner-time hot spot. Worth pinning a
follow-up that uses `index_beginscan` against the placement PK
or the `(index_oid, source_identity)` secondary index, or
caches the existence-by-index_oid result per planner invocation.

### P3 — `EC_SPIRE_PLACEMENT_INDEX_OID_ATTNO = 1`

See P2 above. Could be derived from the table's tuple descriptor
at `_PG_init` time and cached in a static OnceLock. Cosmetic
but lock-step with the placement table layout invariant.

### P3 — Acknowledged P1 #2 deferral to 30874

The request.md is honest about the scope split. The 30874
packet (review forthcoming) does the actual centralization.
Worth noting that until 30874 lands, the duplicated extractor
in `custom_scan.rs` is a live drift risk if anyone changes the
predicate-walk rules in `dml_frontdoor.rs` without updating
both. Pin in the tracker that 30874 must land before any
UPDATE/DELETE rewrite slice.

## Reviewer recommendation

**Accept the milestone.** P1 #1 closed cleanly via the direct
catalog scan. P1 #2 cleanly deferred to 30874 with a clear
follow-up note. The two new P2s (catalog-scan snapshot
semantics, seqscan scaling) are worth folding into 30874 or a
sibling packet before the placement table grows large in a
production deployment.

This is the load-bearing executor work. With PK SELECT live
through CustomScan, UPDATE and DELETE rewrite slices have a
clear template to follow.
