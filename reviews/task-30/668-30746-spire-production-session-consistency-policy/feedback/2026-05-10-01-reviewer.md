## Answers to the packet's review questions

- **Is a Userset enum GUC the right first AM-boundary source for
  strict/degraded mode, or should C5 insist on a query option
  instead?** GUC is the right *first* source. Reasoning:
  - Userset GUCs honor `SET LOCAL`, so per-query scoping works
    without inventing a new syntax surface.
  - Operators get a familiar override mechanism that
    `pg_settings` exposes for runbooks.
  - Future query-level options (e.g. a `WITH (consistency =
    'degraded')` table option, or a planner hint) can land
    on top of this GUC without breaking the executor's single
    source.
  C5 should explicitly state in the design doc that the GUC is
  the contract and any future query-level option would *override*
  the GUC for a single statement, not replace it. Worth pinning
  before query-level options become a compatibility question.
- **Correct that session degraded mode must still match the
  active epoch's published consistency mode before dispatch
  planning succeeds?** Yes. The published consistency mode is
  the strongest constraint a remote can observe; if the epoch
  was published as strict-only, allowing a session to override
  that to degraded would let an opt-in fall back to partial
  results that the publisher explicitly forbade. Rejecting at
  dispatch-planning time is the safe default; if a future
  workflow needs override, it should be an explicit
  publisher-side capability.
- **Compact session summary enough proof for this slice, or
  should the full production state summary grow a
  `consistency_mode_source` column too?** Compact summary is
  sufficient *for this slice*. See P3 below for the natural
  follow-up.

## P3 — informational

1. **Promote `consistency_mode_source` to the full production
   state summary.** The compact session summary surfaces the
   GUC source, but
   `ec_spire_remote_search_production_executor_state_summary` is
   the surface a C5 AM scan caller will read for full
   per-dispatch state. Adding `consistency_mode_source` and
   `consistency_mode` columns to the full summary lets one
   query answer both "what mode is the executor running" and
   "what's the per-dispatch state". Worth landing in the next
   C5 slice rather than now.

2. **Active-epoch policy mismatch category.** When a session
   declares degraded mode but the active epoch is strict-only,
   the dispatch-planning rejection should map to a named
   `failure_category` consistent with the C2/C3 vocabulary
   (e.g. `consistency_mode_mismatch` rather than a generic
   error). If it already does, this is a no-op note; if not,
   worth aligning before C5 reads the surface.
