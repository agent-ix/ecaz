# Phase 5 design checkpoint review

Design note `plan/design/spire-boundary-replication.md` is consistent with
the implementation that lands in 30542–30551 and with the recall study in
30548. No blocking findings.

## Confirmations

1. `boundary_replica_count` reloption (no GUC) is the right control surface.
   It lives on the index, so build/insert fanout cannot drift out of sync
   with scan dedupe across sessions.
2. Top-N nearby leaves is acceptable as the first predicate. The recall
   study (30548) shows negligible recall lift at 2× storage cost on real-10k
   at the current operating point, which validates that a margin-based
   predicate is not worth introducing yet.
3. Deriving scan dedupe from `boundary_replica_count > 0` is correct.
   The default primary-only path keeps `NoReplicaDedupeDisabled`, avoiding
   the HashMap allocation hot path.
4. PID-based local placement (no replica-specific store planner) is correct
   for v1: replicas already live at distinct leaf PIDs, so the existing
   hash-by-PID store router gives natural NVMe spread.
5. Diagnostics surface (storage accounting in 30547) is sufficient for the
   first recall/storage packet — base/boundary/delta primary and replica
   counts are visible separately.

## Suggestions (non-blocking)

- The design document should call out explicitly that the recall study
  rationale (replication is a distributional/availability hedge, not a
  recall multiplier on real-10k) is what keeps the default off — that
  framing is currently in 30551 but worth pulling forward into the design
  note so future readers do not re-litigate it.
