# Review: SPIRE CustomScan Eligibility Planner Readiness (30808)

## Summary

Closes both P2 findings from the prior 30806 review:
- The narrow `load_custom_scan_placement_directory` helper reads only
  the placement directory tuple, dropping the wasted epoch-manifest /
  object-manifest decode the eligibility surface used to do.
- The added planner-facing signals (`remote_available_node_count`,
  `remote_unavailable_placement_count`, `all_remote_placements_available`)
  let a future cost model gate a partial-availability index without
  re-deriving the same counts.

Also addresses the P3 missing-coverage finding by adding the
`no_active_epoch` and `no_available_remote_placements` SQL test
branches, and adds the requested comments on intentionally-inert
`CUSTOM_PATH_METHODS` and the empty hook body. Good follow-up.

The three review-focus questions:
- Narrow placement-directory read is the right helper — yes, the
  planner gate doesn't need epoch/object-manifest validation. The
  active-epoch consistency check on the placement-directory tuple is
  enough for the gate.
- `eligible_for_custom_scan` semantics: reasonable. "At least one
  available remote placement" keeps the dispatch simple while
  `all_remote_placements_available` exposes the partial state. Cost
  model can decide policy.
- Planner-facing signals are the right shape before CustomPath
  generation lands.

## Findings

### P3 — Defensive `active_epoch == 0` check in the new helper is unreachable

`load_custom_scan_placement_directory` returns
`Err("ec_spire cannot load placement directory for empty active epoch")`
when `root_control.active_epoch == 0`, but the only caller
(`custom_scan_index_eligibility_row`) already checked the same
condition and short-circuited on the no-active-epoch branch above.
Either drop the helper's check or, if the helper is meant to grow
other callers, document that it is the canonical gate and remove the
redundant check from the caller. Minor cleanliness; not blocking.

### P3 — Helper bypasses the existing fanout loader's manifest validation

The new helper deliberately skips the epoch-manifest and
object-manifest decode that the fanout loader does. That is the right
call for the planner gate (cheap and the manifests aren't needed
here), but the placement-directory contents themselves can become
inconsistent with the active epoch in ways that the fanout loader
would catch. The new helper validates only `placement_directory.epoch
== root_control.active_epoch`, which is the most important invariant.
Worth adding a one-line comment that explicitly cross-references
where the deeper validation is performed (executor path), so a future
reader does not reintroduce the heavy load thinking the lighter path
is unsafe.

### P3 — `remote_node_count` vs `remote_available_node_count` semantics

The new pair is good. Worth a one-line note in the SQL function
docstring (or in the planner-path packet that consumes this) clarifying
that `remote_node_count` is the *historical/total* set of remotes that
have ever owned a placement at this active epoch, while
`remote_available_node_count` is the planner-relevant subset. Otherwise
operators reading the diagnostic may use the wrong column.

## Reviewer recommendation

Accept. P3 items are doc/comment polish for the next code touch on
this file.
