# Review: SPIRE Placement Write Contention (30969)

## Summary

Closes the Phase 12.4 H12 placement-table write-contention
row. `test_pg18_ec_spire_placement_write_contention_distinct_pk_dml`:

- 8 psql workers behind advisory barrier;
- each worker: distinct-PK INSERT + DELETE in one txn,
  plus matching `ec_spire_placement` INSERT/DELETE;
- asserts: worker success, PK lookups visible, final
  placement count, no ungranted `ec_spire_placement`
  locks, unchanged `pg_stat_database.deadlocks`, p99
  completion < 20s.

Also pins the partition-decision: keep shared
`ec_spire_placement` table until packet-local evidence
crosses the threshold.

## Findings

### P2 — 8-writer / 20s threshold

Reasonable for Phase 12.4 local contention gate, given
30952's local capacity profile (1 concurrent writer
workload as the smoke target). 8 writers is an
**8x overload** of the documented capacity target — this
is the right pressure level for a contention oracle:
catches degradation before scaling the production target,
without requiring AWS-class scale.

20s p99 is generous (8 short transactions should complete
in single-digit seconds on local hardware) which avoids
flake risk while still catching pathological contention.
Worth confirming the threshold was chosen *before* running
(per the tracker exit criterion in 12.4 — "choose the
fixture's writer count and p99 latency threshold before
running it"). From the request wording ("predeclared 20s
threshold") this is the case. Good — the threshold isn't
back-fit to observed values.

### P2 — Coverage of both contention surfaces

The fixture exercises:

- **app-table distinct-PK DML** (the user-visible write
  path); and
- **direct `ec_spire_placement` INSERT/DELETE in same txn**
  (the placement-row hot page).

Together these reach the H12 question "does shared
placement table cause hot-page contention under
distinct-PK writes?" If both pass without deadlocks or
ungranted locks under 8x overload, the shared-table
hypothesis holds. Honest coverage.

### P2 — Partition deferral is justified

Deferring `index_oid` partitioning until packet-local
evidence shows lock waits/deadlocks is the right call
under the data this fixture produces:

- no ungranted `ec_spire_placement` locks → no contention
  visible at the lock manager;
- unchanged deadlock counter → no deadlock cycle;
- p99 < 20s under 8x overload → bounded latency growth.

Three independent oracles, all passing. Partitioning ahead
of evidence would add operational complexity (per-index
shard, vacuum scheduling, statistics) for no measured
benefit. Wait for evidence.

The exit criterion (cross threshold OR show lock
waits/deadlocks) is correctly preserved in the tracker, so
a future evidence-driven decision can still trigger
partitioning.

### P3 — Same-transaction app + placement writes

Combining app-table and placement-table writes in one
transaction is the production write shape (BEFORE INSERT
trigger writes both). The fixture mirrors that, which is
correct. If a future refactor splits the writes across
transactions for any reason, this fixture would no longer
exercise the joint contention shape — pin a comment in
the fixture about the production-shape parity so future
maintainers know not to split them for "cleanup."

Accept. Right pressure level (8x overload), right
threshold (predeclared, generous), right coverage (joint
app+placement), and partition-deferral is justified by
the three independent oracle outcomes.
