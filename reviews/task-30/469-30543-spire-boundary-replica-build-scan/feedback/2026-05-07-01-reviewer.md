# Boundary build + scan review

Build-side fanout writes one shared `vec_id` for the primary plus
bounded `BOUNDARY_REPLICA` rows; the scored-visible predicate keeps
update/vacuum on primary-only semantics. No blocking findings.

## Confirmations

1. `build_boundary_leaf_assignment_placements` (`assign.rs:189-211`)
   allocates exactly one `vec_id` per `SpireBoundaryLeafAssignmentInput`
   and threads it across the primary placement and each replica
   placement — verified at `assign.rs:239-249`.
2. Scored-visible vs. primary-only predicate split is the right
   boundary: identity-ownership operations (delete, vacuum, vec_id
   uniqueness) keep using the primary predicate, while scan candidate
   collection sees both primary and replica rows.
3. Populated single-level relation-backed scope is acceptable as the
   first runtime fanout slice.
4. Task tracker wording is clear about the remaining insert / recursive /
   replacement gaps; subsequent packets close them.

## Findings

- **Performance — payload cloning per replica placement.**
  `assign.rs:213-252` (`build_boundary_assignment_placements`) clones
  `input.assignment` (which carries `encoded_payload: Vec<u8>`) once per
  placement: once for the primary and once per replica. With
  `boundary_replica_count` capped at 8 and current encoded payload sizes
  this is fine, but with V2 leaves and larger scannable payloads in
  flight it is worth noting. Consider an `Rc<SpireLeafAssignmentInput>`
  or consuming `assignment` for the primary then cloning only for
  replicas; not blocking.

- **Validation — replica PID ordering.**
  The function rejects `replica_pids.contains(&primary_pid)` and
  duplicate replica PIDs but does not require any specific ordering of
  replica PIDs. That is consistent with the route map producing
  ordered-by-IP results, but if replicas ever come from a different
  caller, encode order will affect the distribution across stores. Worth
  documenting the implicit "caller passes route order" contract.
