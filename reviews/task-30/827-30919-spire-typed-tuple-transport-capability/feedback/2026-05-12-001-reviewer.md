# Review: Typed Tuple Transport Capability Advertisement (30919)

## Summary

First negotiation slice for Phase 12.2. Adds typed transport
capability advertisement on the live endpoint identity surface
without changing CustomScan receive behavior. The receive path
still consumes JSON; this packet only makes typed-capable remotes
visible to the coordinator.

What changed:

- `ec_spire_remote_search_endpoint_identity(...)` now returns:
  - `tuple_transport_capabilities = ARRAY['pg_binary_attr_v1']`
  - `tuple_transport_default = 'pg_binary_attr_v1'`
  - `tuple_transport_status = 'ready'`
- `ec_spire_remote_search_endpoint_contract()` documents the
  three new tuple transport contract rows.
- Endpoint identity PG18 fixture asserts the new fields.
- Receive-contract PG18 fixture asserts the new contract rows
  and the updated endpoint contract row count.

The three review-focus questions:
- Endpoint identity is the right first advertisement surface:
  yes — that's where descriptor-side identity validation already
  consumes remote capability fields. Coordinator-side selection
  rules (strict requires capability match, etc., per the 30913
  design) can plug into the existing identity-cache path.
- Profile fingerprint should remain unchanged by tuple
  transport capability metadata: yes — the profile fingerprint
  hashes the immutable scoring/storage shape; transport
  capability is a runtime negotiation surface, not part of the
  semantic identity that gates merge eligibility.
- Tracker wording doesn't overclaim full negotiation or
  CustomScan typed receive: confirmed — the parent negotiation
  item stays open until receive switches.

## Findings

### P3 — Capability advertised even when remote can't actually deliver

Every remote running this build now advertises
`pg_binary_attr_v1` regardless of whether the remote heap
relations actually have columns with `typsend` functions defined.
Per-column unsupported-type-binary-io errors fire at request
time (30915), so the advertisement isn't actively wrong — it
means "I support the protocol; specific column-type queries may
still fail." Worth a one-line note in the design that
capability is protocol-level, not column-shape-level.

### P3 — `tuple_transport_default` per-descriptor

If a future operator wants to force JSON transport for a
specific remote (debugging, version-skew workaround), the
default is currently hardcoded `pg_binary_attr_v1`. Worth
pinning whether `tuple_transport_default` is settable per
descriptor (would require a writer surface) or is always the
build's preferred default. Cosmetic for v1; operationally
useful eventually.

### P3 — Profile fingerprint stability

Confirmed: `tuple_transport_*` fields are NOT in the
fingerprint hash, so descriptor-identity validation continues
to behave the same after this packet. A 0.1.x deployment that
upgrades through 0.1.2→0.1.3 sees its descriptor fingerprint
unchanged. Good migration shape.

### P3 — Receive-contract row count assertion

The fixture asserts the updated endpoint contract row count
(presumably +3 for the new transport rows). Worth pinning the
contract count as a stable monitor — operators tracking
contract drift catch the change cleanly, but the assertion will
need updating each time contract rows are added/removed.

## Reviewer recommendation

Accept. Capability advertisement is the right first negotiation
slice — receive switching depends on the coordinator knowing
which remotes advertise typed support. The next slice should
land the strict/degraded selection rules in the executor, then
the receive switch itself.

After that, the JSON retirement criteria from 30913 (five
named gates) become measurable.
