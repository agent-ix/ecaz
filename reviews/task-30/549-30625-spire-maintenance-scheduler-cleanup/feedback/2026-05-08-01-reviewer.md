# Maintenance scheduler + cleanup summary review

`ec_spire_index_maintenance_scheduler_plan/run` plus
`ec_spire_index_epoch_cleanup_summary` close the Phase 8
background-scheduler item. Operator-controlled periodic-job
policy (one of the three options the original spec allowed).
No blocking findings.

## Confirmations

1. **Lock discipline preserved.**
   `ec_spire_index_maintenance_scheduler_run` delegates to the
   existing `ec_spire_index_maintenance_run`, which already
   honors the lock-time reload/recheck contract from Phase 2.
   No new lock paths introduced.
2. **Scheduler policy is honest.** `scheduler_policy =
   'operator_periodic_job'` advertises that this is not a
   background worker — operators must drive the cron. The
   recommendation string explicitly says
   "call ec_spire_index_maintenance_scheduler_run from an
   operator-controlled periodic job."
3. **`scheduler_status` derived from planner_status.**
   `'due'` only when the planner selects split/merge work;
   `'idle'` otherwise. No false-positive cron triggers.
4. **`blocked_not_implemented` for old-epoch cleanup** is the
   right v1 status. Matches reality at the time of this slice
   (later closed by 30628).

## Findings (non-blocking)

- **The "operator periodic job" policy is the simplest of the
  three options the spec allowed.** Considered: background
  worker (more PG infrastructure, restart semantics), VACUUM
  hook (couples maintenance to autovacuum cadence),
  operator-controlled periodic job (cron / pg_cron / external
  scheduler). The chosen option keeps SPIRE deployment-
  agnostic — reasonable v1 trade-off, but worth noting that
  operators on cloud environments without external cron
  (e.g. RDS without pg_cron preinstalled) will need separate
  setup. Track for the docs in 30627.
