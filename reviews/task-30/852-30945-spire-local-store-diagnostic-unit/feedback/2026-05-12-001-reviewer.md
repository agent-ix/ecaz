# Review: SPIRE Local Store Diagnostic Unit (30945)

## Summary

Phase 12.8 multi-store diagnostic unit row closure. The
`(node_id, local_store_id)` and
`(node_id, local_store_id, store_relid)` keys are now
asserted distinct in the two-store SQL VACUUM fixture, plus
scan placement diagnostics show only local-node groups for
query-selected stores.

## Findings

### P3 — Diagnostic-only post-VACUUM verification

Validating post-VACUUM state through SPIRE diagnostics
rather than heap SELECTs is **correct given the fail-closed
DML frontdoor policy**. A heap SELECT against an
ec_spire-indexed coordinator table now goes through the
planner-hook fail-closed for unsupported predicates, which
would obscure VACUUM correctness behind frontdoor errors.
Diagnostic surface is the right oracle.

Suggest one sentence in the fixture comment explaining *why*
heap SELECT is avoided, so a future maintainer doesn't
"helpfully" add a `SELECT * FROM coord_table` and trip the
frontdoor.

### P3 — Phase 12.8 checklist scoping

The wording is correctly scoped to:

- `(node_id, local_store_id)` scheduling/diagnostic unit;
- `(node_id, local_store_id, store_relid)` lookup map;
- bounded local store lookup rows.

It explicitly does NOT claim:

- read-overlap harness coverage;
- multi-NVMe performance evidence;
- async/parallel store-group execution.

Good — those remain open and are correctly tracked
separately (see 30946 for the sequential-execution
diagnostic row). No overclaim.

### P3 — Two-store VACUUM coverage edges

The fixture asserts no active delta object or delta
assignment debt post-insert AND post-delete. Worth
confirming the post-delete branch also passes through a
case where the delete actually retires a delta-routed row
(not just a base-routed row), since that's the path that
produces the most placement-table churn.

If the existing fixture already covers that, ignore. If it's
base-only deletes, pin a follow-up to add a delta-routed
delete pass.

## Reviewer recommendation

Accept. The diagnostic unit closure is honest about scope —
distinct keys + bounded lookups + scan group placement —
without overclaiming the still-open Phase 12.8 lanes.
