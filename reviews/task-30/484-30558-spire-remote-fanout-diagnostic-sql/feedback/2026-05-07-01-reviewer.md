# Remote fanout diagnostic SQL review

`ec_spire_remote_search_fanout_plan` is a thin SQL projection over the
30557 planner. No blocking findings.

## Confirmations

1. SQL prelude (`remote_search_fanout_plan_rows:159-181`) mirrors the
   storage-node endpoint's fail-closed checks for `requested_epoch` and
   active consistency mode before invoking the planner.
2. Local-only fanout is visible as `target_kind = 'local'` rather than
   suppressed, which matches the request focus (#3) — operators tuning
   selection should see the whole plan, not only remote rows.
3. Row count is preallocated correctly:
   `local + sum(remote_per_target) + skipped`
   (`remote_search_fanout_plan_rows:188-196`).
4. The diagnostic does not imply remote transport has landed; status
   strings are descriptive (`requires_libpq_transport` arrives in
   later packets, not this one).

## Findings

- **Status string `"available"` is hardcoded for local and remote
  rows** (`remote_search_fanout_plan_rows:203, 213`). Today the
  planner only routes selected PIDs through these branches when their
  placement is in fact `Available` (degraded skipped placements take
  the third branch). That is correct. But because the rendering
  hardcodes the literal rather than reading
  `lookup.placement.state`, a future change in
  `fanout_should_skip_placement` that lets a non-Available state into
  the local/remote branches would silently emit a wrong
  `placement_state` string. Worth using
  `fanout_placement_state_name(lookup.placement.state)` in those
  branches too, even though the value is currently invariant.
