# 30675 — SPIRE eager bounded scan contract review

Validation pass on `47559184` + ADR-056. Phase 10.2 records that
`ec_spire` keeps the current eager-`amrescan` / drain-`amgettuple`
shape, with the eager work now genuinely bounded by the Phase 10.1
route + candidate budgets. ADR-only; no behavior change.

## Verified

- ADR-056 status `ACCEPTED`, dated 2026-05-09, scoped to Task 30 Phase
  10 execution architecture. Affects ADR-049/050 lineage (not
  conflicting).
- Decision is stated narrowly: `amrescan` owns
  snapshot/object-store/routing/candidate/heap-rerank work;
  `amgettuple` drains a pre-ranked `(heap_tid, orderby_score)`
  cursor.
- Memory ceiling now bounded: route frontier (capped by
  `beam_width / max_routing_expansions / max_leaf_routes`) plus
  candidate cursor (capped by `candidate_limit = min(rerank_width,
  max_candidate_rows)`). The pre-Phase-10 unbounded shape is
  explicitly out of scope.
- Streaming `amgettuple` is deferred and tied to a future ADR
  covering snapshot/object-store ownership across executor calls.
  Right disposition — the ownership model is the actual hard part.
- Phase 10.2 checklist marked `[x]` against this ADR.
