# Review: SPIRE DML Target Relation Extraction (30851)

## Summary

Adds the bridge from a `pg_sys::Query` to its target heap relation
OID. `dml_frontdoor_target_relation_oid(query)`:
- For `UPDATE` / `DELETE`: walks the rangetable to the entry at
  `query.resultRelation` and returns `RangeTblEntry.relid`.
- For single-table `SELECT`: walks the single `RangeTblRef` from
  the FROM list and returns the corresponding `RangeTblEntry.relid`.
- For joined SELECT (multiple FROM entries): returns `None` so
  the planner hook can fail closed before any catalog lookup.

PG18 fixture analyzes real SQL for UPDATE / DELETE / SELECT and a
joined SELECT rejection.

The three review-focus questions:
- Target relation extraction semantics for UPDATE/DELETE vs
  single-table SELECT: yes — uses `resultRelation` for DML and the
  single `RangeTblRef` for SELECT, which is the correct PG idiom.
- Reject joined SELECT before metadata lookup: yes — short-circuits
  cleanly without paying for catalog SPI.
- Next slice should combine target relation + relation context to
  classify real planner-hook queries: confirmed — and now
  delivered by 30854 (classify diagnostic).

## Findings

### P3 — Function name vs return semantics

`dml_frontdoor_target_relation_oid(query) -> Option<Oid>` returns
`None` for any rejection path (joined, non-DML, missing
resultRelation). Worth a one-line comment that `None` is the
"this query cannot be the front door" signal — the planner hook
should pass through to standard_planner for `None` and only attempt
classification when a non-`None` OID is returned.

### P3 — Joined UPDATE / DELETE not yet exercised in the test

The test covers joined SELECT rejection but not joined UPDATE /
DELETE (`UPDATE a SET ... FROM b WHERE ...` shapes). PG handles
UPDATE-with-FROM via additional rangetable entries and the
classifier should reject those via the existing `single_table` /
`has_join` checks downstream. Worth a focused fixture for the
UPDATE-from-join shape so the rejection is regression-locked.

### P3 — `RangeTblEntry.relid` is the heap OID

Standard. Returns `InvalidOid (0)` for non-relation rangetable
entries (e.g., subqueries, function calls). The downstream
classifier rejects those via `ec_spire_distributed_table = false`.
Worth confirming the helper returns `None` (not `Some(InvalidOid)`)
for those cases so the contract is "either a real heap OID or
None."

## Reviewer recommendation

Accept.
