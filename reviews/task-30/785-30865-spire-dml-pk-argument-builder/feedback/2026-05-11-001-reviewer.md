# Review: SPIRE DML PK Argument Builder (30865)

## Summary

Adds the typed PK argument shape: `SpireDmlFrontdoorPkArgument {
pk_column, value }` with `value:
SpireDmlFrontdoorPkValuePlan::{ConstBigint(i64), ParamBigint(i32)}`.
`dml_frontdoor_pk_argument_from_replacement_decision(decision)`:
- Rejects unsupported decisions.
- Requires non-empty `pk_column`.
- Branches on `pk_value_kind` (`const_bigint` /
  `param_bigint`); rejects other kinds.
- For `param_bigint`, validates `param_id > 0`.

PG18 coverage: supported PK SELECT decision builds the typed
argument; unsupported embedding UPDATE decision fails closed
before any executor construction.

The three review-focus questions:
- Typed argument shape sufficient for upcoming executor: yes —
  the executor branches on `value` to decide const-vs-param
  byte resolution.
- Const/parameter validation matches the replacement-decision
  contract: yes — same kind strings and same int2/int4/int8 →
  bigint coercion rules.
- Unsupported decisions fail closed before executor construction:
  yes.

## Findings

### P3 — `pk_column` is owned (`String`)

The argument struct owns `pk_column`. Cheap clone per query but
not per row, since the argument is built at plan time. Fine.

### P3 — `param_id <= 0` rejection

PostgreSQL parameter ids are 1-based; `<= 0` is a malformed
decision row. Defensive shape; harmless.

## Reviewer recommendation

Accept.
