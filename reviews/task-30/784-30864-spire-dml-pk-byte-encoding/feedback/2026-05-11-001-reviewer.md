# Review: SPIRE DML PK Byte Encoding (30864)

## Summary

Pins the bigint PK byte encoder for the future DML CustomScan
executor: `dml_frontdoor_bigint_pk_value_bytes(i64) -> Vec<u8>`
returns `value.to_be_bytes().to_vec()`. PG18 fixture compares the
helper output to PostgreSQL `int8send(...)::bytea` for `0`,
positive, negative, `i64::MAX`, `i64::MIN` — proves byte-for-byte
match across the value range.

The three review-focus questions:
- Byte encoding matches placement-directory and primitive
  contract based on `int8send(...)::bytea`: yes — `int8send`
  returns big-endian 8 bytes; `to_be_bytes()` is the same shape.
- Appropriate shared boundary for the upcoming DML CustomScan
  executor: yes — every consumer (placement insert, primitive
  call, DELETE forwarding) takes bytea PK from the same encoder.
- No planner/executor behavior changes: confirmed.

## Findings

### P3 — Helper allocates per call

`to_be_bytes().to_vec()` allocates a fresh 8-byte `Vec<u8>` per
call. For the high-volume executor path that's 8 bytes plus the
Vec metadata per row. Acceptable for v1; worth pinning a future
optimization that takes a `&mut [u8; 8]` writer if profiling shows
allocation pressure here.

### P3 — i128/u64/u128 explicitly out of scope

The encoder is `i64`-only, matching the v1 bigint PK contract.
Any future PK type (UUID, etc.) needs its own encoder. Worth a
one-line comment naming the v1 scope.

## Reviewer recommendation

Accept.
