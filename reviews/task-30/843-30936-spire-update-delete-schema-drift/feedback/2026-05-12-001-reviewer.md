# Review: SPIRE UPDATE/DELETE Schema Drift (30936)

## Summary

Generalizes the descriptor-bound schema-drift guard from
INSERT-only to all coordinator write paths:

- Same pre-dispatch fingerprint comparison runs for remote
  UPDATE and DELETE payload paths.
- Guard placed before remote SQL construction, conninfo
  resolution, libpq transaction setup, and **DELETE placement-
  directory removal** — so a drift-rejected DELETE doesn't
  half-cleanup the placement directory.
- Fixture proves:
  - Remote UPDATE fails with schema drift; remote row
    unchanged.
  - Remote DELETE fails with schema drift; remote row present.
  - No SPIRE prepared transaction left behind.
  - DELETE does not remove the placement row before the guard
    fires.
- ADR-069 + diagnostics + Phase 12.5 tracker now describe the
  guard as covering INSERT, UPDATE, **and** DELETE remote
  write paths.

Closes 30933 P2 about UPDATE/DELETE coverage in one slice.

The three review-focus questions:
- UPDATE and DELETE guard placement early enough: yes — fires
  before any remote work or local placement-directory mutation.
  The DELETE-doesn't-remove-placement-before-guard assertion is
  load-bearing.
- Reusing INSERT shape fingerprint for UPDATE/DELETE: yes —
  the column shape that matters for `to_jsonb(NEW)` /
  `jsonb_populate_record` round-trip is the same one that
  matters for UPDATE SET column lists and DELETE row identity.
  One fingerprint covers all three.
- Tracker no longer overstates open UPDATE/DELETE schema-drift
  gap: yes.

## Findings

### P3 — DELETE-before-guard-fires assertion

The fixture explicitly checks "DELETE does not remove the
placement row before the guard fires." That's the right
defensive shape — without it, a drift-rejected DELETE could
leave the placement directory inconsistent with the actual
remote heap (placement gone, remote row still present).

### P3 — Single guard covers all three operations

The descriptor-bound `coordinator_insert_shape_fingerprint`
field name now applies to UPDATE/DELETE too. The "insert" in
the field name is now slightly misleading. Worth pinning in
ADR-069 that the field is named for historical reasons but
applies to all coordinator write paths. A future rename to
`coordinator_write_shape_fingerprint` or
`coordinator_payload_shape_fingerprint` could land in 0.2.x.

## Reviewer recommendation

Accept. **30896 H6 is now fully closed across all coordinator
write paths.** Combined with H1 (descriptor refresh SQLSTATE),
H7 (trigger payload type round-trip), H10 (DDL ordering), and
H11 (multi-row INSERT), schema-drift handling is now
production-ready.
