# Review: SPIRE Selected PID Placement Map (30956)

## Summary

Closes Phase 12.7 placement-metadata row. Adds
`ec_spire_index_selected_pid_placement_snapshot(index_oid,
selected_pids)` returning per-requested-PID placement
metadata: epoch, ordinal, pid, node_id, local_store_id,
store_relid, placement_state, object_version, object_bytes.

Uses the coordinator-fanout manifest loader so it can
inspect remote placements pre-fanout without tripping
`custom_scan_tuple_delivery`.

## Findings

### P3 — Manifest-loader boundary is correct

Using the coordinator-fanout manifest loader rather than
the local-heap delivery path is the right choice for a
diagnostic that must work for remote placements. The
local-heap guard exists to enforce CustomScan tuple
delivery at execution time; a *diagnostic* function asking
"where is this PID?" should not need execution-time
guarantees. Honest separation.

Worth a one-line doc note that this diagnostic answers
"where would this PID be served from?" not "is the data
currently readable end-to-end?" — those are different
questions and conflating them is the most likely operator
misuse.

### P3 — Object-version agreement validation

The pre-return validation of manifest/directory pair plus
per-PID object-version agreement is the load-bearing
correctness primitive. Without it, a stale manifest could
report a placement that disagreed with the directory and
operators would chase ghosts. Keep.

If a future operator reports a PID returns no row from this
diagnostic but appears in scan results, the most likely
cause is the validation rejecting it — worth pinning that
in a troubleshooting note ("if this returns fewer rows
than requested, check object-version mismatch").

### P3 — Scope vs broader dispatch decision report

Correctly scoped as **selected-PID placement** rather than a
general dispatch decision report. Dispatch decisions involve
fanout state, degraded skips, executor budgets — orthogonal
axes that should not be wedged into a placement function.
The 30954 degraded skip report is the right surface for the
dispatch-decision question; this one answers placement only.

### P3 — Tracker row closure justification

Fixture rewrites one selected PID to remote node/local
store `2` and verifies the returned map includes both local
and remote placements. That's the minimal proof of "selected
PID maps span remotes," which is exactly what the Phase 12.7
row asks for. Closure justified.

### P3 — `selection_ordinal` semantics

Worth confirming the doc names what `selection_ordinal`
means relative to the `selected_pids` input — typically
"position in the input array, 1-indexed" or "position in the
canonical ranking." Operators using this to correlate with
ranking order need to know which.

## Reviewer recommendation

Accept. Narrow placement diagnostic with the right loader
boundary and the right validation. Closes the Phase 12.7
placement-metadata row honestly.
