# Review: SPIRE Split Replacement Boundary Fanout (30546)

Reviewed at code commit `61b4816e` on branch `task-30-spire`.

The split materialization slice does what the packet claims. Replacement
leaf input validation correctly accepts primary + boundary rows for the
same `vec_id` while still rejecting duplicate rows in a single leaf and
multiple primaries (`update/materialization.rs:382` and downstream
`validate_replacement_leaf_object_inputs`). `split_replacement_assignment_with_flags`
clones the original assignment so `vec_id`, `heap_tid`, `payload_format`,
`gamma`, and `encoded_payload` are shared across replica rows. Merge
remains primary-only as documented.

## `route_split_replacement_boundary_indexes` skips source-vector validation

`update/materialization.rs:422` computes inner product over the raw source
vector and the trained centroids without rejecting non-finite or zero-norm
inputs. Compare with `routing_plan.rs:218` `validate_route_vector`, which
rejects both, and `build/recursive.rs` paths that go through the same
helper.

Today this is unreachable through the normal path because k-means would
have already rejected pathological inputs upstream, but the asymmetry is
gratuitous and silently cooperates with NaN ordering via `total_cmp`. A
small `validate_route_vector(source_vector, dimensions)?` (or the same
inline checks) at the top of `route_split_replacement_boundary_indexes`
keeps the contract uniform with the build/insert paths and makes future
refactors safe.
