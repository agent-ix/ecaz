# Remote search execution plan review

`ec_spire_remote_search_execution_plan` is the final pre-libpq
contract row: target kind, transport, endpoint, remote index source,
conninfo source, candidate format, status. No blocking findings.

## Confirmations

1. Local targets correctly report `transport = 'local_direct'`,
   `endpoint = 'ec_spire_remote_search'`, candidate format from the
   v1 string.
2. Remote targets report `transport = 'libpq_pipeline'`, with
   `remote_index_source` and `conninfo_source` both
   `remote_node_descriptor`, status carried through as
   `requires_remote_node_descriptor` from 30564.
3. Skipped degraded targets report `transport = 'none'`, `endpoint =
   'none'`, `candidate_format = 'none'`, summarizing as
   `degraded_ready`.
4. The execution summary correctly aggregates the per-row contract.

## Findings

- **`libpq_pipeline` transport string is asserted before any libpq
  code exists.** The diagnostic claims pipeline mode will be the
  transport, matching the 30552 design intent. That is fine for now,
  but the actual libpq executor (in a future packet) must use this
  same string when reporting itself, or the diagnostic will diverge
  from the implementation. Tracking item: when libpq lands, ensure
  the executor reads the transport string from the same source as
  this row builder rather than re-stringifying. Cross-cutting summary
  on 30568.
