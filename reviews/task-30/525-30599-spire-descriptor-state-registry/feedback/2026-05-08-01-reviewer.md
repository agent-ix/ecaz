# Descriptor state registry review

`ec_spire_remote_node_descriptor_state_contract()` exposes all
five descriptor states (`active`, `draining`, `disabled`,
`failed`, `missing`) with their source (`catalog` vs.
`synthetic`), read eligibility, snapshot status, and
recommendation. Closes the descriptor lifecycle string-literal
finding from 30586/30595/30596. No blocking findings.

## Confirmations

1. The contract is now the single source of truth for descriptor
   state semantics. `remote_node_descriptor_catalog_state_is_supported`
   (`snapshots.rs:979-983`) routes registration validation
   through the contract instead of a literal `matches!` arm.
2. `state_source = 'catalog'` vs. `'synthetic'` correctly
   separates persistable states from the `missing` placeholder
   that only exists when no row is in the catalog.
3. `read_eligible` is `true` only for `active` and `draining`,
   matching the 30574 contract validator
   `must_be_active_or_draining_for_reads` and the descriptor_state
   filter fix from commit 9866d033.

## Findings

- **`sql/bootstrap.sql:144` CHECK constraint still has the
  literal state list.** The packet's notes correctly observe
  this — PostgreSQL DDL cannot consume the Rust registry. The
  contract row is now the comparison surface for reviewers and
  operators verifying alignment, but a future test that asserts
  the SQL CHECK constraint and the Rust contract agree on the
  catalog state set would close the drift class. Same pattern
  as the 30573 tie-breaker invariant and the 30588 degradation
  policy invariant.

- **Vec allocation per registration.**
  `remote_node_descriptor_catalog_state_is_supported` allocates
  the contract `Vec` on every call. For one-shot registration
  this is fine; for any future batch path that registers many
  descriptors in one transaction, the contract should be cached
  in a `static` or computed once. Forward-looking only.
