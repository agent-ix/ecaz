# Review: SPIRE DML Primitive Plan Diagnostic (30869)

## Summary

`ec_spire_dml_frontdoor_primitive_plan_sql(sql text)` —
operator-facing surface that exposes the typed primitive plan
shape:
- For supported const-PK shapes, returns CustomScan mode,
  coordinator primitive name, PK argument metadata, operation-
  specific column payloads, and `pk_value_bytes` (computed via
  the const helper).
- For unsupported shapes, returns `primitive_plan_not_ready`
  with the typed builder error instead of constructing a
  half-formed handoff object.

PG18 coverage: ready PK SELECT primitive plan; unsupported
embedding UPDATE diagnostic path.

The three review-focus questions:
- Diagnostic exposes the right executor handoff fields without
  implying planner replacement is enabled: yes — the `_sql`
  surface naming and the `not_ready` branch make it clear this
  is a probe, not behavior.
- Constant `pk_value_bytes` gives a useful review signal: yes —
  operator can see the exact bytea the primitive will receive.
- Unsupported shapes fail through the typed primitive-plan
  builder rather than bypassing ADR-069 validation: yes — the
  diagnostic invokes the same builder as the future executor,
  so any new validation rule lands in both at once.

## Findings

### P3 — Param-PK plans show what?

For a `WHERE id = $1` shape, the const helper returns Err
("requires runtime parameter evaluation"). The diagnostic
presumably surfaces this as the typed builder error. Worth a
focused fixture asserting that a param-PK SQL classifies as
ready-but-no-bytes (the plan is ready, the bytes resolve at
runtime). Operators running the diagnostic against a prepared
statement need to see "this is a supported shape, params will
resolve at execute time."

### P3 — Diagnostic surface is now a thick chain

`ec_spire_dml_frontdoor_classify_sql` (30854) and
`ec_spire_dml_frontdoor_replacement_sql` (30859) and now
`ec_spire_dml_frontdoor_primitive_plan_sql` form three diagnostic
SQL surfaces, each consuming progressively more typed output of
the front-door pipeline. Worth a one-line cross-reference in each
function's docs naming the others, so an operator picking the
right surface knows the levels.

### P3 — Status string `primitive_plan_not_ready`

Distinct from `unsupported_shape` and `not_distributed_table`
(used by `classify_sql`). Worth ensuring the documentation on
`coordinator_dml_frontdoor_plan` enumerates all the diagnostic
status strings in one place so operators have a complete index.

## Reviewer recommendation

Accept. The diagnostic chain is now complete: classify (shape) →
replacement decision → primitive plan → typed bytea PK. The next
packet should land the actual planner-hook plan rewrite that
calls these helpers from the hook and replaces the eligible base
plans with a CustomScan.
