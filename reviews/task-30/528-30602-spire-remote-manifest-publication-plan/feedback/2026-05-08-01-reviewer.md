## Prior-finding verification (30595/30596 hardening, commit 9866d033)

**Both blocking-like findings from the prior review round are
fixed.** Verified directly in the diff.

1. **Generation monotonicity** — `ec_spire_register_remote_node_descriptor`
   now uses `INSERT ... ON CONFLICT DO UPDATE WHERE
   EXCLUDED.descriptor_generation > ec_spire_remote_node_descriptor.descriptor_generation
   RETURNING true AS registered`. The Rust caller checks the
   RETURNING result and errors with
   `"descriptor_generation must advance existing descriptor_generation"`
   when the WHERE blocks the update. PG18 test
   `test_ec_spire_remote_node_descriptor_stale_generation_rejected`
   pins the contract.

2. **descriptor_state filter** —
   `load_remote_libpq_connection_descriptors`
   (`remote_candidates.rs:1107-1124`) now filters
   `descriptor_state IN ('active', 'draining')`. Plus
   `remote_search_target_readiness_rows` adds an explicit
   guard that maps `disabled`/`failed` to
   `requires_remote_node_descriptor`. The bug class is closed
   on both the loader and the readiness projection.

Bonus: the same hardening commit also distinguishes
`remote_epoch_window` from `requires_remote_node_descriptor` in
`remote_epoch_publish_readiness` based on `missing_descriptor_node_count`,
which makes the publish-side blocker taxonomy precise.

## 30602 — local findings on the publication plan + apply surface

`ec_spire_remote_epoch_manifest_publication_plan/_summary` plus
the libpq request/result/parameter/dispatch contracts and
`ec_spire_apply_remote_epoch_manifest`. 13 new SQL functions in
one slice. No blocking findings; one note on the apply surface
worth tracking.

- **`ec_spire_apply_remote_epoch_manifest` is payload validator
  only.** The function (`src/lib.rs:3359-3444`) validates
  the JSONB envelope (format string, epoch match, entries
  array, per-entry node_id/manifest_action/status), counts
  validated entries, and returns `"ready"`. It does NOT
  persist any remote apply state.

  Today this is intentional and the request notes call it out
  ("does not persist remote apply state"). But the function is
  named `ec_spire_apply_*` — a future caller could
  reasonably expect that calling apply, then querying the
  remote-side state, would show evidence of application. As
  it stands, calling apply twice with the same payload returns
  the same `"ready"` count each time.

  Suggestions for the next slice: either rename to
  `ec_spire_validate_remote_epoch_manifest_payload` while it is
  pure validation, or add a remote-side "applied manifests"
  table (mirroring the coordinator-side
  `ec_spire_remote_epoch_manifest`) that the apply function
  writes to. Either makes the contract honest with the
  function name.

- **Apply does no capability cross-check.** A payload
  declaring "node 7 should serve epoch 5" passes apply
  validation even on a remote node where the local active
  epoch is 3 or where this node identity does not match
  node_id 7. That is correct for a payload-shape validator,
  but a future "real apply" must add the cross-check. Worth
  documenting in the apply function's doc-comment.

  remote-node descriptor catalog (30595, hardened by 9866d033)
  and the remote-epoch-manifest catalog + entry tables (30598)
  give Phase 7 real durable state. Both fail closed before

  catalog states + the synthetic `missing` state have a

  the contract instead of a literal arm.
- **Manifest catalog stale detection (30600)** covers content
  drift, not just row-count drift, via the per-entry mismatch
  scan.
- **Libpq executor readiness (30601)** decomposes the broad

  The coordinator gate now answers "what is the next executor

- **Manifest publication plan (30602)** layers the publication

  materialization, dispatch action, and pre-I/O apply
  validator into one chain — the future libpq publisher has a
  fully specified pre-execution contract.

### Forward-looking findings

1. **Active-epoch advance race in persistence functions.**
   `ec_spire_persist_remote_epoch_manifest` reads the manifest
   summary then writes to the catalog with the read epoch. A
   concurrent epoch advance between read and write would tag
   the manifest with the old epoch. Self-healing on next
   call; worth documenting or guarding with an in-SPI
   epoch re-read assert. See 30598 finding.

2. **Apply-function naming vs. behavior.**
   `ec_spire_apply_remote_epoch_manifest` validates payload
   shape but does not persist applied state. Either rename or
   add a remote-side applied catalog. See 30602 finding.

3. **SQL CHECK constraint vs. Rust state registry drift.**
   `bootstrap.sql:144` and the Rust contract are independent
   sources for the catalog state taxonomy. An invariant test
   that asserts both agree would close the drift class. See
   30599 finding.

4. **Six new executor action strings outside the central
   registry (30601).** `"resolve_conninfo_secret_reference"`,
   `"open_libpq_connection"`, etc. are literals next to the
   `SPIRE_REMOTE_EXECUTOR_STEP_*` constants. Cosmetic; track
   with the ongoing string-registry work.

5. **No invariant test pinning the executor step contract
   against the readiness output.** Same pattern as the 30573
   tie-breaker and 30588 degradation invariants. See 30601
   finding.
