# Remote node capability plan review

`ec_spire_remote_node_capability_plan` is the per-node capability
projection: required epoch window, candidate format, extension
version, conninfo source, and per-check statuses. No blocking
findings.

## Confirmations

1. Local node 0 reports `conninfo_source = 'local'`,
   `required_candidate_format = 'local'`, all per-check statuses set
   to either `ready` or `not_required`, and overall status `ready`.
2. Nonzero node IDs report `conninfo_source =
   'remote_node_descriptor'`, every per-check status as
   `missing_descriptor`, and overall status
   `requires_remote_node_descriptor` with a clear recommendation
   string.
3. `required_last_served_epoch == required_min_retained_epoch ==
   active_epoch` is correct for v1: until retained-epoch serving
   lands, the only valid epoch for cross-node fanout is the
   coordinator's published active epoch (from 30552 design).
4. No raw conninfo or libpq execution exposed.

## Findings

- **Candidate format string `"ec_spire_remote_search_v1"` is hardcoded
  in `remote_node_capability_plan_row` (`snapshots.rs:878`).** When a
  v2 candidate format ever lands, this constant will need to track
  the actual emitted format from `ec_spire_remote_search`. Today the
  storage-node endpoint emits the 9-column shape from
  `remote_search_libpq_result_contract_rows` (30568) — those are
  effectively a single "v1" surface. Consider lifting both the v1
  candidate-format string and the contract-row schema next to each
  other so a future format bump touches both.
