# Phase 6 top-level graph design review

Design checkpoint is consistent with the implementation in 30550 and
with the Phase 0 partition-object boundary. No blocking findings.

## Confirmations

1. Single-layer Vamana/DiskANN over routing centroids is the right
   first choice. Reusing `ec_diskann`'s pure graph core
   (`build_vamana_graph_with_stats`, `approximate_medoid`) instead of
   nesting an `ec_diskann` access method keeps the storage and
   visibility lifecycle inside SPIRE's epoch/object model — which is
   load-bearing for fail-closed manifest handling later.
2. Deferring HNSW + build-time graph algorithm selection until a second
   real graph implementation exists is correct. Premature selectability
   would force a generic abstraction over one implementation.
3. Treating the graph as a SPIRE epoch object (not a nested PG child
   index) is correct: it inherits the same publish/retain/cleanup
   semantics as routing/leaf objects, avoiding a second visibility
   model.
4. Scan fallback rules — fail-closed on missing/duplicate visible graph
   objects, default-disabled, gated on `recursive_fanout >= 2` — give
   operators an explicit revert path and prevent silent degradation.

## Suggestions

- The design discusses graphs only over the **root** routing children.
  When SPIRE later adds a wider top routing-child set (the
  "graph-searchable top routing child set" callout in the request),
  the distance-offset trick used in
  `build/top_graph.rs::max_centroid_inner_product` will become an
  O(n²) precondition. Worth recording, in this design note, that the
  offset construction is an artifact of using IP through a Vamana core
  designed for distance — and that growing the node-set will require a
  cheaper offset (e.g. derive from `max ‖c‖²`) before that scale
  matters.

- The design note should call out explicitly that boundary replication
  (Phase 5) and top-graph (Phase 6) are independent: the top-graph
  input is the routing centroid set, not the assignment set, so
  `boundary_replica_count` does not affect graph node count or graph
  build cost. Useful for future readers reasoning about combined
  feature interactions.
