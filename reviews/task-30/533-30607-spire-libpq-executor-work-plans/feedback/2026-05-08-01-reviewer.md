# Libpq executor work plans review

Per-node work-plan rows that compose dispatch + bind readiness +
executor readiness into a single row that names the next action
(work_action) and effective status. No blocking findings.

## Confirmations

- Symmetric search and manifest work-plan shape: same
  selected_pids / bind count / dispatch action / next executor
  step / executor status / work action / effective status fields.
- The work_action is the eventual executor's "do this next"
  pointer; together with `next_executor_step` it gives both the
  fine-grained step name (executor contract vocabulary) and a
  call-to-action string.

## Findings (non-blocking)

- **Action verbs (`open_pipeline_and_send_remote_search`,
  `open_pipeline_and_send_remote_epoch_manifest`,
  `await_remote_descriptor`, etc.) live next to the executor
  step constants but as inline literals.** Same pattern as the
  30601 finding; consider hoisting alongside the
  `SPIRE_REMOTE_EXECUTOR_STEP_*` constants when libpq lands and
  the executor needs to read them.
