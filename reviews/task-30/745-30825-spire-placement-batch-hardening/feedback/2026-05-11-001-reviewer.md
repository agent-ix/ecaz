# Review: SPIRE Placement Batch Hardening (30825)

## Summary

Closes the 30819 P2 and several P3 items. The function migrates from
`LANGUAGE sql` to `LANGUAGE plpgsql`, adds an explicit pre-INSERT
scan via `generate_subscripts(input_entries, 1)` plus
`input_entries[i]::text IS NULL` to detect NULL composite elements,
and raises `RAISE EXCEPTION 'ec_spire_register_placement_batch
entries[%] is NULL'` with `ERRCODE = '22004'` (null_value_not_allowed)
and a HINT pointing at the fix. Returns row count via
`GET DIAGNOSTICS inserted_count = ROW_COUNT` instead of a CTE
`SELECT count(*)`.

Test coverage now includes empty batch (returns 0), happy path
(returns 2), NULL element rejection at index 1, duplicate PK
violation surfacing as the `ec_spire_placement_pkey` constraint
name, and invalid `source_identity` length surfacing as
`ec_spire_placement_source_identity_check`.

ADR-069 gains durable notes pinning:
- caller-transaction semantics (whole batch aborts on any error);
- frozen v1 `ec_spire_placement_entry` field order;
- forward-compat path requires a new type or append-only contract.

The three review-focus questions:
- Explicit NULL-entry error: correct contract; the error message
  names the array index, the SQLSTATE is the right standard code,
  and the HINT is actionable.
- Preserving PK / catalog constraint violations rather than
  per-field wrapping: yes — the constraint name in the error message
  is enough for callers to attribute the failure without the
  function reimplementing every check. Catalog is the source of
  truth.
- ADR notes pin transaction semantics and field-order freeze: yes,
  in the right place.

## Findings

### P3 — `input_entries[i]::text IS NULL` detects "row of all NULLs" too

`composite::text` for a composite that is structurally non-NULL but
all of whose attributes are NULL formats as `(,,,,)` not NULL. So
the IS NULL check correctly identifies *the array element itself
being NULL*, not "an element with all-null fields." Good — the
boundary is exactly what the reviewer asked for, and ROW(NULL,
NULL, ...) values still fall through to the per-column NOT NULL
constraints with the catalog's existing error messages.

Worth a one-line comment naming why `::text IS NULL` is the right
check (composite rows are NULL only when the whole element is NULL,
which is what `IS NULL` on the composite literally tests already —
you could even drop the `::text` cast). Cosmetic.

### P3 — `ALIAS FOR $1` / `$2` is older PL/pgSQL style

Modern PL/pgSQL prefers named parameters in the `CREATE FUNCTION`
signature (the function already names `index_oid` and `entries`).
The `DECLARE input_index_oid ALIAS FOR $1` pattern works but is
slightly anachronistic. Not blocking; matches existing style in
other PL/pgSQL functions if any.

### P3 — `STRICT` plus PL/pgSQL inert when array is non-NULL but contains NULL

`STRICT` is the function attribute "if any *argument* is NULL,
return NULL without invoking the body." The body is reached when
both arguments are non-NULL arrays — even if those arrays contain
NULL elements. The new NULL-element check fills the gap precisely.
Worth a comment that `STRICT` covers "entries IS NULL", and the
explicit scan covers "entries[i] IS NULL".

### P3 — Test uses `#[should_panic(expected = "ec_spire_placement_pkey")]`

Constraint names are stable (the catalog assigns `tablename_pkey`
deterministically) but rely on PostgreSQL not changing that
convention. Acceptable; if the expected text ever drifts the test
fails loudly.

### P3 — Empty batch returns 0; nice symmetric assertion

`empty_registered_count == 0` proves the empty-array path doesn't
spuriously RAISE. Good defensive coverage.

## Reviewer recommendation

Accept. The hardening is exactly the shape the next coordinator-
routed-INSERT packet needs to lean on.
