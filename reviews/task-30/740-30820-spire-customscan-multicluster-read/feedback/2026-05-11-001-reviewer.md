# Review: SPIRE CustomScan Multicluster Read Fixture (30820)

## Summary

Closes the final read-path evidence gap. The script
`run_spire_multicluster_customscan_read_pg18.sh` provisions two
genuine PG18 clusters on separate ports and Unix sockets, installs
the test extension into both, builds matching coordinator and remote
heap/index pairs, rewrites the coordinator's leaf placements to
`node_id = 2`, registers the remote descriptor with the **real**
endpoint identity fingerprint pulled from
`ec_spire_remote_search_endpoint_identity(...)` (not a placeholder),
asserts `Custom Scan (EcSpireDistributedScan)` in EXPLAIN, asserts
the returned row is `10,remote alpha` (the remote shard's row), and
independently probes the tuple-payload endpoint on the remote to
verify the side-channel returns `id` and `title` fields. Plus
sanity-checks that the remote and coordinator agree on
`active_epoch` and `leaf_pids` before registering the descriptor.

Status `next_step` advances cleanly to "ADR-069 write path and
production CustomScan cost model" — the handoff is honest.

The three review-focus questions:
- Multicluster setup/teardown vs Stage E smoke scripts: consistent
  pattern (PG_CTL initdb + start, trap cleanup, ARTIFACT_DIR
  conventions, env-var conninfo).
- Real fingerprint: yes, pulled live from the remote endpoint.
- Read-path-only claim, no ADR-069 overstatement: confirmed; tracker
  bullets explicitly leave write/cost/Stage E open.

## Findings

### P2 — Default ports collide on concurrent / repeated runs

`REMOTE_PORT=39228` and `COORD_PORT=39229` are static defaults. Two
parallel runs (e.g., a background loop plus an interactive run, or
two CI shards) clash on the listening port and the env-var
conninfo. CLI flags `--remote-port` / `--coord-port` allow override,
but the defaults bite first. Two acceptable shapes:
- Derive default ports from `RUN_ID` (e.g., hash modulo high port
  range), so runs with different `RUN_ID`s never clash; or
- Pick free ports at runtime via `ss -ltn` / `python -c
  "socket; ..."` and reflect them in the conninfo env var.

The Unix socket dir (`SOCKET_DIR`) per-run is already correct since
PG_CTL `-k` makes the listening socket file per-RUN_DIR; the listen
port is the only shared resource.

### P2 — `tests.ec_spire_test_rewrite_placement_node` namespace

The script calls
`tests.ec_spire_test_rewrite_placement_node(...)`. The schema
prefix `tests.` suggests this is a test-only helper in a `tests`
schema. Confirm:
- This function is only created when the extension is built with
  `--features pg_test` (the script does pass `--features "pg18
  pg_test"` to `cargo pgrx install`), so the function exists in this
  multicluster proof binary; and
- A future production install (without `pg_test`) does not surface
  the `tests.` schema or this function, otherwise operators have an
  unsupported placement-rewrite primitive in their database.

If both are true, this is a known-test-binary fixture and fine.
Worth a one-line comment in the script noting the dependency on
`pg_test`-built extension.

### P2 — Cleanup leaves `RUN_DIR` data behind

`trap cleanup EXIT` stops both PG_CTL clusters but does not remove
`RUN_DIR`. That is intentional for evidence packets (data dirs are
useful for post-mortem) but a developer running the script
repeatedly without `--run-dir` overrides accumulates one full
PG18 cluster directory per `RUN_ID`. Worth either:
- Documenting in `--help` that `RUN_DIR` persists by design, or
- Adding a `--cleanup-data` flag that wipes `RUN_DIR` on success.

Not blocking; the `RUN_DIR already exists` early-exit prevents
silent re-use, which is the more important guard.

### P3 — `cargo pgrx install` default is heavy

The script defaults to running `cargo pgrx install` unless
`ECAZ_SKIP_INSTALL=1` or `--skip-install` is set. That's a
multi-minute build for a smoke test that often re-runs against an
already-installed extension. Consider flipping the default: skip
install unless `--reinstall` is passed. Existing artifact log shows
this isn't the bottleneck for the proof itself, but for iteration
ergonomics it matters.

### P3 — Hard-coded `PGBIN` path in `--help`

`Default: $PGBIN.` in the usage prints the literal `$PGBIN`
variable rather than resolving it. Cosmetic; either expand the
default in the heredoc with `cat <<USAGE` (no quotes around USAGE)
or compute and substitute the resolved value at print time.

### P3 — Payload probe assertion via `==` glob

`[[ "$payload_probe" == ready,2,*'"id": 10'* ]]` and the
`'"title"'` glob are brittle to JSON-key spacing variations. PG's
`jsonb::text` output is canonical (`"id": 10`), so the glob holds
today. If a future PG version compacts JSONB output (e.g., removes
the space after `:`), the assertion silently breaks. Worth a
`jsonb_object_field_text` or `tuple_payload->>'id' = '10'` form on
the SQL side that produces a stable scalar, then asserts on that.

## Reviewer recommendation

Accept. The P2 default-port collision is the only thing likely to
bite in real usage (CI parallelism, a developer running this
alongside another smoke); fold a port-selection improvement into
the next multicluster fixture or a small follow-up. The
`tests.`-schema gating is a quick verification rather than a code
change.
