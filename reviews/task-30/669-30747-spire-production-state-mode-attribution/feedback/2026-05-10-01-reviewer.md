## Answers to the packet's review questions

- **Rust-side mode attribution in
  `SpireRemoteProductionExecutorStateSummaryRow` enough for C5,
  with SQL attribution kept on the compact session summary?**
  Yes. C5 will primarily consume the executor state through Rust
  callers (the AM scan integration is in-process Rust), not
  through SQL. Operators get attribution through the session
  summary, which is the right place for "what mode is this
  session running" as a top-level question. Avoiding a tuple
  width fight with pgrx is the right call when there's a clean
  alternative.
- **Active-epoch policy mismatch as a separate row-returning
  preflight rather than changing existing planner functions?**
  Yes. Changing dispatch-planning fail-fast to row-returning
  would break the existing diagnostic SQL contract and force
  every caller to handle a new failure shape. A dedicated
  preflight surface (which packet 30748 lands) is the right
  shape: callers that want operator-visible policy state call
  the preflight; callers that just want a planning result keep
  the fail-fast semantics they have today.

## Status

Closed. Mode attribution is on the Rust executor state where C5
needs it; SQL attribution remains on the compact session summary
where operators read it. The active-epoch preflight is in packet
30748.
