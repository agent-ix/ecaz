## Findings

### P2 â€” `paramFetch` fills caller-provided workspace, not allocated buffer

The code does:
```rust
let mut workspace = pg_sys::ParamExternData::default();
unsafe { fetch(params, param_id, false, &mut workspace) }
```
The returned `param` pointer points to the local `workspace`
which lives in the function's stack frame. Then the code reads
through that pointer and immediately consumes the value via
`DatumGetIntN`. That's safe because the read happens before the
function returns. Worth a one-line comment noting that
`workspace` lifetime is intentionally function-scoped â€” a future
refactor that returns the param pointer to a caller would
introduce a use-after-free.

### P2 â€” Out-of-range int4/int2 â†’ i64 coercion

`i64::from(int16/int32)` is lossless. But: if a `bigint` PK is
queried with a `WHERE id = $1::int8 = 9223372036854775807`, the
runtime returns the full i64 value through `DatumGetInt64`. Worth
a focused fixture asserting `i64::MAX` and `i64::MIN` round-trip
through both the param path AND the const path so the boundary
is regression-locked at both endpoints.

### P3 â€” `ptype` mismatch error names the offending OID

The error message includes the OID number (`other.to_u32()`),
which is not human-readable. Worth either:
- Calling `format_type_be(typoid)` for a readable name in the
  error.
- Documenting that operators should look up the OID via
  `SELECT typname FROM pg_type WHERE oid = $1`.

Cosmetic.

### P3 â€” `paramFetch` callback NULL check

The check pattern is `if let Some(fetch) = params_ref.paramFetch
{ ... }`. Standard. Worth confirming pgrx's binding for
`paramFetch` is `Option<unsafe extern "C" fn(...)>`; if it's a
nullable function pointer the check is correct.

## Reviewer recommendation

Accept. The runtime parameter resolution is the last piece the
DML CustomScan executor needs before it can call the existing
primitives with bytes. Fold the i64-boundary fixture (P2) into
the executor packet's coverage.
