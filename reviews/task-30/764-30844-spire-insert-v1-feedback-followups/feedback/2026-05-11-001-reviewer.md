# Review: SPIRE INSERT V1 Feedback Followups (30844)

## Summary

Closes the two P2 docs/test items from the 30835 + 30836 reviews:

1. **bytea source-identity round-trip test**: New
   `test_ec_spire_insert_trigger_source_identity_json_roundtrip_sql`
   proves the trigger's `to_jsonb(NEW)` payload preserves an exact
   16-byte `bytea` source-identity through `jsonb_populate_record`
   on the remote side. Closes the 30835 P2 silent-corruption
   concern.
2. **ADR-069 v1 type-scope note**: Documents that the v1 transparent
   INSERT trigger pins the front door to bigint PK + ecvector
   embedding + exact-16-byte bytea source identity, with UUID PK
   support deferred. Operators reading the ADR end-to-end now know
   what works in v1 and what's planned.
3. **ADR-069 descriptor-generation race note**: Documents that
   concurrent INSERTs racing for the same descriptor can fail
   closed if a newer generation wins first; v1 callers should
   retry, Stage F can switch to live-row equivalence or per-node
   serialization.

The three review-focus questions:
- JSON roundtrip test covers the bytea source-identity concern:
  yes — directly tests "encode bytea → JSON → decode bytea via
  jsonb_populate_record returns the same bytes." Closes the 30835
  P2.
- ADR-069 v1 type-scope note precise enough for operators: yes —
  explicit about what's supported and what's deferred.
- Descriptor-generation race note acceptable as v1 docs while
  higher-concurrency INSERT is a Stage F concern: yes — names the
  failure mode, the correct caller behavior (retry), and the
  Stage F upgrade path.

## Findings

### P3 — Round-trip test as regression guard

The test asserts the byte-for-byte equality after the JSON
roundtrip. Worth pinning that this test must not be relaxed
without a follow-up ADR — the source-identity round-trip is a
correctness invariant for ADR-063 dedupe, not just a feature.

### P3 — UUID deferral note location

The v1 type-scope note lives in ADR-069's transparent-INSERT
section. Worth a cross-reference from ADR-063 (source identity)
that says "v1 INSERT trigger only supports bytea source identity;
uuid INCLUDE columns are still allowed for the read path." That
way an operator reading ADR-063 to understand identity options
isn't surprised by the trigger's bytea-only constraint.

### P3 — Descriptor race note suggests "retry"

ADR-069 should also pin what failure surface a caller sees when
the race fires (the exact error message), so applications can
catch on a stable error pattern rather than parsing free-text.
Currently the helper errors with `ec_spire coordinator insert
descriptor refresh did not advance descriptor_generation`. Worth
either documenting that string as a stable contract or moving it
to a SQLSTATE the caller can catch on.

## Reviewer recommendation

Accept. The roundtrip test and ADR docs close the open 30835/30836
P2 items cleanly.
