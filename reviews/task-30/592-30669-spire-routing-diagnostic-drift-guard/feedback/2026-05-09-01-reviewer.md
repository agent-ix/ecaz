# 30669 — Routing diagnostic drift guard review

Validation pass on `261cf408`. Test-only. Closes my prior 30664 F1
(parallel diagnostic path drift risk) by adding a fixture-level
property test rather than refactoring to a shared helper. Acceptable
shape.

## Verified

- The fixture now invokes both paths on the same input:
  `collect_scan_routing_diagnostics` (the diagnostic path) and
  `route_recursive_routing_objects_to_leaf_routes_with_budget` (the
  production path). Both run with the same `recursive_nprobe_policy`
  and `recursive_route_budget` from the resolved scan plan, so the
  comparison is genuinely apples-to-apples.
- Three drift-detecting assertions:
  1. **Root-level dedupe count** must equal the count of distinct
     parent PIDs the production path actually visited.
  2. **Root-level selected child count** must equal the count of
     children that the production path's child routing produces from
     the same root.
  3. **Leaf-level selected child count** must equal the production
     candidate count summed across actually-visited parents.
  4. **Leaf-level dedupe count** must equal the production leaf-route
     count exactly.
- Production leaf PID order is also pinned (`vec![FIRST_PID+11,
  FIRST_PID+21]`) so a routing-comparator change that reorders ties
  fails this test loudly.

## Findings

| ID | Severity | Notes |
|----|----------|-------|
| F1 | non-blocking, small | The drift guard catches divergence in *counts* at root and leaf levels, but not at intermediate levels for hierarchies of depth > 2. The current fixture is two levels (root → internal → leaf), so the leaf-level checks cover the only intermediate level. When recursion goes 4–6 levels deep (Phase 9 architecture target), the test should be parametrized over depth or a deeper fixture should be added. Track in Phase 10 alongside the existing shared-helper refactor item. |
| F2 | informational | The shared-traversal-helper refactor remains tracked in `task30-phase10-spire-execution-performance.md` per the request. With this property test in place, the refactor is now optional/cleanup rather than a correctness gate — that's the right disposition. |

## Review focus answers

> Confirm the fixture meaningfully catches production-vs-diagnostic
> recursive routing drift for selected and deduped route counts.

Yes. Both `selected_child_count` and `deduped_route_count` are
compared to production-path values at root and leaf levels. A
diagnostic that miscounted would fail the assertion before reaching
the production-path comparison. F1 above is the one shape gap.

> Confirm deferring the shared traversal/helper refactor remains
> acceptable for Phase 10 while the property guard is in place.

Yes. With the test guarding the contract, the refactor becomes a
maintenance simplification rather than a correctness fix. Phase 10 is
the right home.
