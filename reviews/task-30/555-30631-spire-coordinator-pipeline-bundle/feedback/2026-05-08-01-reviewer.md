# Coordinator pipeline bundle review

`SpireCoordinatorPipeline::execute_once(...)` closes the pipeline-
redundancy concern from the 30584 closeout and the prior
comprehensive review's Plan §A. The coordinator gate now runs
the readiness chain once and derives execution + dispatch +
receive + merge + finalization + executor readiness from one
shared bundle. No blocking findings.

## Confirmations

1. **Single readiness pass.** `execute_once` calls
   `remote_search_request_readiness_rows` once
   (line 2600-2609); every downstream summary is derived from
   the result via the new `_from_*` projection helpers. No
   re-running of the leaf scoring or fanout planning.
2. **Field passthrough preserves SQL contract.** The bundle
   exposes the same five summary types the public functions
   were already producing (execution, dispatch, receive,
   finalization, executor readiness). Public SQL surfaces are
   unchanged.
3. **First-attempt bug captured.** The packet notes the empty-
   dispatch summary fallback originally used a synthetic zero
   query vector for query metadata; the fix carries the
   original query. Good test discipline — caught before merge.

## Findings

- **Manifest publication path is not yet bundled.** The bundle
  is search-only. The manifest path's gate summary
  (30610) and result summary (30614) still re-run their
  inputs. Reasonable v1 scope; track for a follow-up bundle.

- **Receive rows are stored as `Vec<...>` in the bundle**, not
  a summary struct. Callers that need the summary still
  compute it locally. Slight asymmetry with the other four
  fields but probably correct — receive rows are the
  per-batch contract, not just an aggregate.
