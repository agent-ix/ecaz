# Remote catalog orphan cleanup review

`ec_spire_remote_catalog_orphan_summary()` and
`_cleanup()` close my "DROP INDEX leaves orphan catalog rows"
finding from the prior comprehensive architecture review (Plan §C).
No blocking findings.

## Confirmations

1. **Live-index detection is correct.** The `live_spire_index`
   CTE joins `pg_class` to `pg_am` on `amname = 'ec_spire'`
   and `relkind = 'i'`. A descriptor whose
   `coordinator_index_oid` is no longer a live SPIRE index
   is correctly classified as orphan.
2. **Snapshot consistency.** Count and DELETE both run inside
   one `Spi::connect_mut` transaction, so they observe the
   same snapshot — the count cannot diverge from what gets
   deleted.
3. **FK cascade handles entries.** Manifest-entry deletion
   piggybacks on the `ON DELETE CASCADE` from
   `ec_spire_remote_epoch_manifest`. No need for a third
   DELETE. Verified via the entry count being computed
   independently — matches the cascaded count.
4. **Symmetric `_index_cleanup(index_oid)`** (30639) is the
   exact-OID variant for a future `DROP INDEX` event trigger
   to call without sweeping unrelated orphans.
5. **PG18 coverage** uses a synthetic dead OID and asserts
   summary counts, cleanup counts, and post-cleanup ready
   status. The packet also captures a test-shape bug (cleanup
   was being called once per returned column) and the fix
   (single SQL call returning all four columns).

## Findings (cosmetic)

- **Functions are not marked `volatile`** — `pg_extern` default
  is volatile, so this is correct. But the orphan summary
  reads pg_class, which is system-stable within a transaction;
  technically `stable` would be allowed for the summary. Not
  worth changing.

- **No `superuser_only` or role check.** The cleanup function
  runs as the calling role's privileges. A non-superuser with
  USAGE on the extension can call it and delete catalog rows.
  Worth either a role check or doc note that catalog cleanup
  is an operator function. Forward-looking.
