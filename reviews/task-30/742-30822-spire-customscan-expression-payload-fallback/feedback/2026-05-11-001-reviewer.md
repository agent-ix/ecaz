# Review: SPIRE CustomScan Expression Payload Fallback (30822)

## Summary

Closes the P2 from the 30815 review. The narrowing logic now treats
projection narrowing as valid only when *every* non-junk target entry
is a simple user-column `Var` (`varattno > 0`). Any expression, junk
entry already excluded, or system-column Var (`varattno <= 0`)
flips `can_narrow_projection = false` and the function falls back to
requesting all relation columns. The new loopback-fixture extension
proves `SELECT id, title || ' (boosted)' FROM tbl ORDER BY embedding
... LIMIT 1` returns `(10, "remote alpha (boosted)")`, demonstrating
the previously-dropped `title` column is now actually fetched.

The three review-focus questions:
- Mixed Var-plus-expression target lists fall back to all columns:
  yes — first non-Var entry breaks the loop and clears `attr_numbers`.
- Loopback assertion covers the wrong-results shape: yes — the
  expression result depends on the remote payload containing
  `title`, which the previous narrowing stripped.
- Doesn't claim final pushdown: confirmed; this is a correctness
  fallback, not an optimization.

## Findings

### P3 — Subtle: `can_narrow_projection` only matters when at least one Var was collected

The flag is set to `true` at the start of the targetlist iteration
and only flipped to `false` on a non-Var or system-column Var.
After the loop:

```rust
if !can_narrow_projection {
    attr_numbers.clear();
}
```

If the targetlist is empty or contains only junk entries (continue
branch), `can_narrow_projection` stays `true` and `attr_numbers`
stays empty. The natural fallback below
(`if !attr_numbers.is_empty() && !attr_numbers.contains(...)`)
already handles empty `attr_numbers` as "include everything". So
the `clear()` here is correct but redundant in the
empty-target-list edge case. Not a bug — just a minor logic
duplication that could be simplified to "any non-Var → fallback;
empty Var set → fallback" without the boolean.

Cosmetic.

### P3 — System-column Var fallback path is untested

The fix correctly flips the fallback flag on `var.varattno <= 0`
(system-column Vars like `ctid`, `xmin`, `tableoid`), but there is
no test covering that branch:

```sql
SELECT ctid, id FROM tbl ORDER BY embedding ... LIMIT 1
```

Worth a one-line addition either to the loopback fixture or as a
sibling test. Not blocking — the failure mode if the branch
regresses is the same wrong-results bug just for a different
target-entry shape.

### P3 — Comment could explain why the fallback exists

`custom_scan_tuple_payload_columns` now has subtly different
semantics from before: a partial Var-only narrowing is preferred
when possible, otherwise full-relation fallback. Without context, a
future reader might "simplify" the fallback flag back into the
narrow-only path and reintroduce the bug. A two-line comment
referencing the 30815 P2 finding (or "fallback exists because
narrowing must be a *superset* of every Var consumed by any
expression in the target list") would be load-bearing for
maintenance.

### P3 — Acknowledges deferred remote-projection pushdown

The packet correctly notes this is not the final shape — eventual
typed-tuple transport or remote projection pushdown will let the
narrow-projection optimization survive expression target lists by
walking expression-tree Var references. That follow-up should also
remove the conservative fallback once the walker is in place.

## Reviewer recommendation

Accept. The P3 items are documentation / coverage polish; the
correctness gap from 30815 is closed.
