# 30732 — SPIRE production remote index resolution review

Validation pass on `b2c901d9` (`Resolve SPIRE production remote
index in receive adapter`). Accepts the receive-side regclass fix.

## What I checked

- `SpireRemoteProductionCandidateReceiveRequest` carries
  `remote_index_regclass` (a stable descriptor reference) instead
  of a coordinator-local `remote_index_oid`. The receive adapter
  resolves `to_regclass($1)::oid` on the **remote connection**
  immediately before invoking `ec_spire_remote_search(...)`.
- New failure category constant
  `SPIRE_REMOTE_PRODUCTION_REMOTE_INDEX_UNAVAILABLE =
  "remote_index_unavailable"` at line 78. The category is reported
  by the adapter (line 5118) when remote regclass resolution fails,
  separate from `remote_query_failed`.
- Receive-isolation fixture extended to interleave malformed input,
  missing conninfo, connect failure, missing remote index, remote
  query failure, decode failure, and batch-validation failure with
  one ready remote that still returns candidates — the per-node
  isolation invariant from packet 30729 holds for the new failure
  category.
- Phase 11 Stage C checklist item for remote-side regclass
  resolution flips.

## Answers to the packet's review questions

- **Resolving the remote index OID on the remote connection the
  right production boundary?** Yes — definitively. A remote
  PostgreSQL node owns its own index OID namespace, so any
  coordinator-local OID is meaningless on the remote side. The
  loopback fixture happens to share an OID space with the
  coordinator, which is exactly the kind of accidental coupling
  this fix removes before AM scan integration locks it in.
- **`remote_index_unavailable` the right isolated failure category
  for missing remote regclass?** Yes. Operator remediation differs
  from `remote_query_failed` (the index actually doesn't exist on
  the remote vs. it exists but a query against it failed). C4
  strict/degraded mapping will eventually treat both as fail-closed
  / skip-node, but the diagnostic distinction matters now.
- **Fixture exercises the important distinction between missing
  remote index, remote query failure, candidate decode failure,
  and candidate validation failure without broadening the
  diagnostic executor?** Yes. The fixture stays inside the
  test-only adapter probe surface; no production SQL surface is
  widened.

## Status

Closed. The receive adapter is now correctly remote-side OID
resolved; loopback-only OID assumptions are out of the production
path before C5 AM scan integration.
