## What's solid

  the sign convention for `<#>` (negative-inner-product) is
  fixed and documented at the point of use; pin test
  `remote_heap_exact_score_uses_orderby_negative_inner_product`
  at `root/tests.rs:114-118` will catch regression. No
  other distance operators exist in
  `sql/bootstrap.sql:704-741`, so the single-opfamily
  scope is honest.
- **DML frontdoor pass-through** (`dml_frontdoor.rs:1685-1755`,
  `classify_dml_frontdoor_shape`): the gate widened only
  for non-PK SELECT shapes; INSERT/UPDATE/DELETE/embedding-
  UPDATE rejection paths are untouched. The classifier
  cannot overshoot into unsafe DML.
- **Trigger fixture invariant** (`sql/bootstrap.sql:319-481`,
  `ec_spire_enable_coordinator_insert`): BEFORE ROW returns
  NULL, suppressing the coordinator heap row;
  `coordinator_row_count = 0` is the correct invariant.
  Per-statement queue is a temp table with
  `ON COMMIT DELETE ROWS`, AFTER STATEMENT flush goes
  through the async fanout adapter. Clean.
- **Async 2PC fanout cancellation** (`remote_candidates.rs:3437-3477`,
  `run_insert_prepare_requests_with_local_cancel_source`):
  any single-remote prepare failure triggers a
  `ROLLBACK PREPARED` sweep over already-prepared peers
  before the local commit callback is registered.
- **Schema-drift fingerprint** (`sql/bootstrap.sql:247-264`,
  enforcement at `remote_candidates.rs:2647/2794/2853`):
  attnum-ordered name/typid/typmod/collation/notnull tuple,
  applied uniformly to INSERT, UPDATE, and DELETE.
- **Hygiene**: no `todo!()` / `unimplemented!()` / `FIXME`
  in `src/am/ec_spire/`. No `.unwrap()`/`.expect()` on
  remote-supplied data in hot paths. The `#[ignore]`
  markers in the test suite are HNSW recall fixtures, not
  SPIRE.

## Real risks (none block Phase 12 acceptance; flagged for
Phase 13)

### P2 — In-doubt prepared txn on PREPARE-network-timeout

`run_one_insert_prepare_request`
(`src/am/ec_spire/root/remote_candidates.rs:4154-4198`).
If `PREPARE TRANSACTION` ack is lost AFTER the remote
WAL-flushed the prepare-state but BEFORE the client sees
the ack, the Err arm fires; the row is **not** in
`prepared_rows`, so the outer rollback sweep
(`remote_candidates.rs:3463-3472`) does not visit it.
`coordinator_insert_resolve_remote_prepared`
(`remote_candidates.rs:2230`) silently swallows connection
failures. No `pg_prepared_xacts` reaper exists for
`ec_spire_insert_*` GIDs. Classic 2PC in-doubt window;
will be visible at AWS scale under flapping links.

**Phase 13 ask:** background reaper that scans
`pg_prepared_xacts` on each remote and rolls back GIDs
whose top-transaction OID is no longer live on the
coordinator.

### P2 — Unbounded coordinator memory on adversarial payload

`decode_remote_search_typed_tuple_payload_pg_row`
(`remote_candidates.rs:9468-9581`) validates cross-array
width but enforces no per-row or per-batch byte cap on
`payload_values_hex` or `selected_pids`. A misbehaving or
hostile remote can return arbitrarily large payloads and
OOM the coordinator.

**Phase 13 ask:** `spire.max_remote_payload_bytes` (per-row)
and `spire.max_remote_payload_rows_per_batch` GUCs,
enforced before allocation.

### P2 — Cost constants hardcoded, not GUC

`src/am/ec_spire/cost.rs:11-14`
(`SPIRE_ROUTING_SCORING_DIMENSION_SCALE`,
`SPIRE_LEAF_SCORING_DIMENSION_SCALE`,
`SPIRE_INDEX_PAGE_COST_SCALE`,
`SPIRE_LOCAL_STORE_PAGE_FANOUT_SCALE`) and the scoring
multipliers at `cost.rs:377/385` are `const`. The 30981
reviewer correctly noted that AWS recalibration is needed
under cross-AZ fanout RTT; iterating that knob is
currently a code-change-and-release cycle. Local readiness
does not need this; Phase 13 will, fast.

**Phase 13 ask:** `spire.cost_*` GUCs with the current
values as defaults.

### P3 — Stage E fault matrix bitrot risk

The Phase 12 entry criterion ("Stage E fault matrix
(11 cases) and lifecycle matrix (6 cases) pass against the
CustomScan build in packet `30895`") is true at packet
`30895`'s SHA. The Phase 12.9 closeout bundle (`30978`)
re-ran only 2 of the 11 fault cases live
(`remote_statement_timeout`, `local_cancel`). The
remaining 9 are pinned by archived logs against a different
SHA; nothing in CI re-runs them. The
`scripts/run_spire_multicluster_stage_e_*.sh` scripts have
no GHA hook.

A reader who interprets the tracker as "matrix is gated
in CI" would be wrong. Suggest tracker line 26 add a
parenthetical: "(matrix is archived in `30895`; live
re-run cadence is reviewer-requested, not CI-gated)."

**Phase 13 ask:** wire at least the lightweight half of
Stage E into CI (the pre-dispatch fault script and the
transport fault script are bash-driven against two PG18
clusters; feasible in GHA `services:` with the existing
`ecaz dev spire-multicluster` wrappers).

### P3 — JSON fail-closed error category is generic

`remote_tuple_payload_production_sql`
(`remote_candidates.rs:8637-8645`) returns
`SPIRE_REMOTE_STATUS_ENDPOINT_IDENTITY_MISMATCH` when the
remote does not advertise `pg_binary_attr_v1`. The
operator gets an "endpoint identity mismatch" status,
which does not immediately suggest "upgrade the remote to
a version that speaks `pg_binary_attr_v1`."

**Suggested:** add `SPIRE_REMOTE_STATUS_TUPLE_TRANSPORT_RETIRED`
with a hint string naming the required capability. Small,
cheap, improves runbook-driven recovery.

### P3 — Schema-drift fingerprint is coordinator-only

`coordinator_write_current_shape_fingerprint`
(`remote_candidates.rs:2579`) reads from the coordinator;
the descriptor's cached fingerprint was also captured from
the coordinator at register/refresh time. Silent **remote**
drift (someone runs `ALTER TYPE` on a remote without
re-registering) is caught only at SQL-execution time on
the remote, not pre-dispatch. Acceptable v1 boundary if
the DDL ordering runbook is followed strictly; tighten for
Phase 13 by adding a `remote_shape_fingerprint` echoed on
descriptor refresh.
