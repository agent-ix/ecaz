## Answers to the packet's review questions

- **pg_test-only namespace GUC the right way to isolate
  advisory-lock tests without changing production lock
  behavior?** Yes. Two design choices make this clean:
  1. **Compile-time gating** of the read path on test-only
     features — production builds literally cannot read the
     GUC, so production behavior cannot be affected by any
     misconfiguration of the GUC value.
  2. **`SET LOCAL` scope** in fixtures — namespace doesn't
     leak across tests within the same pg_test run.
  The alternative (running tests serially) would mask future
  parallelism regressions and slow down CI.
- **Namespace offset preserves the production governance
  path being tested?** Yes. Advisory-lock semantics are
  identical regardless of which `class_id` range is used —
  the offset just moves the keys into a non-colliding band.
  The production code path through `pg_try_advisory_lock`,
  RAII release, and overflow handling is exercised
  unchanged.
- **Phase 11 validation claim scoped correctly to the
  production pgrx filtered set, not the full remote-search
  suite?** Yes. The task-file note names
  `test_ec_spire_prod_` as the passing filtered surface
  rather than claiming the entire remote-search suite has
  cleared. Honest scoping.
