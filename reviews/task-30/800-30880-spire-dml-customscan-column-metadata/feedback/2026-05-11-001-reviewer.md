# Review: SPIRE DML CustomScan Column Metadata (30880)

## Summary

Serializes operation-specific column metadata (UPDATE
`updated_columns`, PK SELECT `projected_columns`) into
`CustomScan.custom_private` after the existing mode and index
OID entries. Stores column names as JSON string nodes (not
comma-delimited) so column names with embedded commas / special
characters round-trip cleanly. Decoded during `BeginCustomScan`
for all DML plan modes.

The three review-focus questions:
- JSON string nodes in `custom_private` as serialization
  boundary: yes — JSON is overkill for column-name lists but
  it's lossless and matches the tuple-payload JSON pattern used
  elsewhere.
- Additional plan-private entries preserve mode/index OID
  layout consumed by vector and PK SELECT paths: yes — entries
  are appended after the existing positional layout, no
  reordering.
- Metadata-only slice doesn't enable UPDATE/DELETE execution:
  confirmed.

## Findings

### P3 — JSON for a column-name list is heavy

A list of column names doesn't have any structural complexity
that requires JSON. PG's existing `lappend(list, makeString("name"))`
pattern would store strings as `T_String` nodes natively.
JSON-encoded strings work but require parse/serialize per call.

Acceptable for v1; worth pinning a future cleanup that uses
native PG string list nodes (or an Integer count + N
string nodes positionally) once the metadata layout settles.

### P3 — Empty-column fail-closed

The unit test confirms empty column lists fail closed at the
metadata boundary. Good defense.

## Reviewer recommendation

Accept.
