## Findings

| ID | Severity | Notes |
|----|----------|-------|
| F1 | non-blocking, small (readability) | Numeric collision: vec_id discriminators use `0x01 = local`, `0x02 = global`; dedupe key prefixes use `0x01 = global`, `0x02 = node_local`. The same byte values mean opposite things in different contexts. The code is correct, but a reader who sees `0x01` in a debug string can't tell which scheme. **Mitigation:** rename `SPIRE_REMOTE_VEC_ID_KEY_GLOBAL` and `SPIRE_REMOTE_VEC_ID_KEY_NODE_LOCAL` to start at `0xA0`/`0xA1` (or any non-overlapping range) so a hex dump of a dedupe key prefix is unambiguous. Or add a doc comment at the const declarations explaining the deliberate disjoint encoding. |
| F2 | informational | Phase 9.6 leaves writer-side global-id allocation as future work. This is correctly framed in ADR-055 and the `next_action` contract — flagging here only so it stays visible: **cross-node boundary-replica dedupe is not yet end-to-end safe** because writers still allocate local IDs even when boundary replication is configured across nodes. The runtime merge contract now correctly *refuses* to silently merge them; a future writer slice needs to allocate global IDs for replicas. Phase 10 or a Phase 9.6 follow-on would be the natural home. |
