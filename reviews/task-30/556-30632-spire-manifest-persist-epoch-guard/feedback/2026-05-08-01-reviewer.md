# Manifest persist epoch guard review

Closes my "active-epoch advance race" finding from 30598. The
persist function now keeps the index relation open through the
SPI write block and rechecks the active epoch immediately
before catalog writes via a new root-control-only helper. Fails
closed with a retryable error if the epoch advanced. No blocking
findings.

## Confirmations

1. The recheck happens **inside** `Spi::connect_mut`
   (`lib.rs:1953-1959`), not before. So the window between
   re-read and INSERT is bounded by the SPI transaction
   snapshot.
2. The narrow root-control-only helper avoids re-running the
   full diagnostic stack — the original first attempt used
   `active_snapshot_diagnostics` which validates remote
   placement state and was over-validating for this guard's
   needs. The packet captures that learning.
3. Error message names the old and new epoch and tells the
   caller to retry — operationally clear.
4. Index relation is now closed only after the SPI block
   finishes (or in the early-error branches at 1926/1938).

## Findings (cosmetic)

- The early-error branches at lines 1926 and 1938 close the
  index relation explicitly before `pgrx::error!`. The success
  path closes after the SPI block. Consistent, but a `defer`-
  style closer would prevent a future early-return from
  forgetting the close. Not blocking.
