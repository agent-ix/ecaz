# Degraded heap status preservation review

Real bug fix: `merge_input_summary_row` and
`finalization_summary_row` previously emitted `ready` even when a
plan had degraded-skipped placements alongside local executable
work, losing the `degraded_ready` signal. Two-line fix in
`remote_candidates.rs` plus a focused PG18 test. No blocking
findings.

## Confirmations

1. `merge_input_summary_row` now checks
   `execution_summary.degraded_skipped_plan_count > 0` before
   falling through to the `ready` branch — degraded skips with
   local work now surface as `degraded_ready`.
2. `finalization_summary_row` carries the merge summary's status
   forward instead of overwriting it with `ready`.
3. PG18 coverage drives a fixture with one local available leaf
   plus one degraded-skipped leaf and asserts the per-summary
   `degraded_ready` propagation through merge → finalization →
   heap resolution → local heap candidate summary.
4. Final local heap fetch status remains `local_ready` because
   the executable local work is still locally finalizable — the
   degraded signal is observability, not a blocker.

## Findings

- **None on this slice.** This is exactly the kind of
  status-preservation drift my 30568 closeout warned about. The
  fact that this bug existed undiscovered until this packet
  motivates centralizing the status-precedence logic into the
  count rollup helper or a dedicated status-derivation function,
  rather than open-coding the precedence in every summary. The
  rollup helper from 30572 already covers count math; consider
  extending it to status derivation in a future slice.
