# Review: SPIRE DML Primitive Plan Builder (30866)

## Summary

Composes the typed handoff object the future DML CustomScan
executor will consume. `SpireDmlFrontdoorPrimitivePlan`:
- `index_oid: pg_sys::Oid`
- `mode: SpireDmlFrontdoorCustomScanMode` (typed enum:
  `CoordinatorUpdateTuplePayload` /
  `CoordinatorDeleteTuplePayload` /
  `CoordinatorPkSelectTuplePayload`)
- `primitive: &'static str` (must equal mode-derived expected
  primitive name)
- `pk_argument: SpireDmlFrontdoorPkArgument`
- `updated_columns: Vec<String>`
- `projected_columns: Vec<String>`

`dml_frontdoor_primitive_plan_from_replacement_decision(decision)`:
- Rejects `index_oid == InvalidOid`.
- Builds the typed PK argument via 30865.
- Maps `decision.custom_scan_mode` string to the typed enum.
- Asserts `decision.primitive == expected_primitive` (cross-check
  between the decision row's free-text mode/primitive pair and
  the typed enum's contract).
- Operation-specific column validation:
  - UPDATE requires non-empty `updated_columns` and empty
    `projected_columns`.
  - DELETE requires both empty.
  - PK SELECT requires non-empty `projected_columns` and empty
    `updated_columns`.

PG18 coverage: UPDATE / DELETE / PK SELECT happy paths,
mode/primitive mismatch rejection, unsupported embedding UPDATE
rejection.

The three review-focus questions:
- Primitive plan shape sufficient as executor handoff object:
  yes — the executor consumes one typed input.
- Mode/primitive validation matches ADR-069 replacement-decision
  contract: yes — the cross-check guards against decision-row
  drift.
- Operation-specific column validation prevents wrong-payload
  consumption: yes.

## Findings

### P3 — Mode/primitive cross-check guards against decision drift

The validator asserts `decision.primitive == expected_primitive`
where `expected_primitive` is derived from the typed mode enum.
That catches any future drift between
`dml_frontdoor_replacement_decision_from_shape` and the typed
mapping. Belt-and-suspenders pattern; good.

### P3 — Empty `updated_columns` allowed only for DELETE

The shape catches the common error "build a UPDATE plan with no
target columns" or "build a SELECT plan with no projection"
before the executor hits a confused primitive call. Worth a
one-line comment naming the operation-specific invariants so
future maintainers see the contract.

### P3 — `Vec<String>` clones for column payloads

`updated_columns.clone()` and `projected_columns.clone()` per
query. Acceptable at plan time. For very wide projections
(50+ columns) consider `Arc<[String]>`; cosmetic for v1.

## Reviewer recommendation

Accept.
