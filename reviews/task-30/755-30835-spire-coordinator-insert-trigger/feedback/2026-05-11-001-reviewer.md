# Review: SPIRE Coordinator Insert Trigger (30835)

## Summary

First transparent `INSERT INTO coordinator_table ...` front door,
implemented via the trigger option ADR-069 explicitly allows.
`ec_spire_enable_coordinator_insert(table_oid, index_oid, pk_column,
embedding_column, source_identity_column = 'source_identity')`:
- Validates that the supplied index is `ec_spire` and references
  the supplied table.
- Pins v1 column types: `bigint` PK, `ecvector` embedding, `bytea`
  source_identity. Rejects with `42804` for any drift.
- Installs `BEFORE INSERT FOR EACH ROW` trigger calling
  `ec_spire_coordinator_insert_forward_trigger`.

The trigger function:
- Encodes the bigint PK via `int8send` for canonical bytea.
- Casts `ecvector` embedding to `real[]` for classifier input.
- Builds tuple JSON with `to_jsonb(NEW)` and the requested column
  list from live `pg_attribute` rows.
- Calls `ec_spire_prepare_coordinator_insert_tuple_payload(...)`,
  reusing 30832's prepare + 2PC path.
- Returns `NULL` so the coordinator heap is not mirrored.

The four review-focus questions:
- Trigger-based front door acceptable before a ModifyTable hook:
  yes — ADR-069 names triggers as the v1 option and the lower-level
  hook is explicitly deferred.
- v1 type constraints explicit: yes (see findings below for
  scope).
- `RETURN NULL` preserves "no coordinator-side mirror" invariant:
  yes — that's the documented PG semantics for BEFORE INSERT row
  triggers.
- Tracker leaves descriptor refresh + multicluster `INSERT INTO`
  open: confirmed (closed by 30836 / 30837).

## Findings

### P2 — V1 type constraints exclude UUID PKs and other shapes

ADR-068's example DDL uses `id bigint PRIMARY KEY`, but ADR-069 also
talks about `uuid` PKs as a possibility. The trigger rejects
anything but `bigint` for PK, `ecvector` for embedding, and `bytea`
for source_identity. That's a fine v1 minimum but worth pinning in
the ADR what shapes are *not yet* supported and why. Otherwise
operators creating a `id uuid PRIMARY KEY` table will see `42804
v1 requires a bigint pk column` and will reasonably ask "when?"

Suggested follow-up shape:
- ADR-069 lists supported v1 PK / embedding / identity types.
- Future packet adds a `uuid` PK encoder (pg `uuid_send` returns 16
  bytes natively, simpler than `int8send`).

### P2 — `to_jsonb(NEW)` includes the source_identity bytea

`to_jsonb(NEW)` serializes the entire row including the
`source_identity bytea`. PG serializes bytea as a `\x...` hex string
inside JSON. The remote endpoint runs
`jsonb_populate_record(NULL::heap, $1::jsonb)` which decodes the
hex string back into bytea via standard type input. Round-trip
should be clean. Worth a focused regression test asserting that
`source_identity` matches byte-for-byte after round-trip — without
it, a future jsonb encoding change could silently corrupt identity
bytes.

### P2 — `requested_columns` includes every non-junk attribute

The trigger builds `requested_columns` from
`pg_attribute WHERE attnum > 0 AND NOT attisdropped`. That sends
*all* columns — including `embedding` — through the JSON wire to
the remote. For a wide table with a large embedding, this bloats
the wire payload and the remote `jsonb_populate_record` cost.

Acceptable for v1 single-row INSERTs; worth a follow-up that
narrows to "columns the application actually wrote" once the
trigger has access to that information (PG triggers don't easily
expose "which columns were in the INSERT statement," so this may
require a richer hook).

### P2 — Trigger validates `TG_OP / TG_WHEN / TG_LEVEL` defensively

Good: rejects with `0A000` if anyone re-attaches the function as
`AFTER UPDATE` or similar. Worth confirming `ec_spire_enable_coordinator_insert`
is the only sanctioned way to install the trigger (prevents
operators from manually `CREATE TRIGGER ... USING ...` with wrong
arguments).

### P3 — Embedding cast to `real[]`

The trigger calls `($1).%I::real[]` where `%I` is the embedding
column name (typed `ecvector`). Requires an `ecvector → real[]`
cast to exist in the extension. Confirmed by 30823 fixture using
`ecvector_to_real_array(...)`. Worth confirming the implicit
`::real[]` cast resolves through that function (vs requiring the
trigger to call it explicitly).

### P3 — `ec_spire_enable_coordinator_insert` re-creates the trigger

`DROP TRIGGER IF EXISTS ec_spire_coordinator_insert_forward ON %s`
followed by `CREATE TRIGGER ...` makes the call idempotent. Good.
Worth pinning that operators can revoke by `DROP TRIGGER
ec_spire_coordinator_insert_forward ON tbl;` directly without going
through a `disable_coordinator_insert` helper, which doesn't exist
yet.

### P3 — `requested_columns` includes the source_identity column

Sending `source_identity` to the remote via the requested-columns
list is fine for v1 (the remote table has the same column). Worth
documenting that schema-aligned coordinator and remote tables are
required (which ADR-068 already states).

## Reviewer recommendation

Accept the trigger front door for v1. The P2 type-shape narrowing
and the wide-projection cost are forward-looking — neither blocks
the read-after-insert lane that 30836 + 30837 closes. The
to_jsonb / source_identity round-trip test is the one cheap
regression worth adding before any production schema with non-toy
identity bytes lands.
