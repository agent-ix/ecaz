# Feedback: SPIRE Final Architecture Review

**Reviewer:** outside-review
**Date:** 2026-05-15
**Verdict:** Changes requested before AWS testing

## Scope

I reviewed the SPIRE production architecture across the CustomScan read path,
remote fanout transport, coordinator-routed DML/PK SELECT front door,
placement/descriptor drift checks, local storage/update boundaries, and the
Phase 12 closeout/runbook material. This was a static architecture/code review;
I did not run tests.

## Findings

1. **P0 - Production remote connections do not implement the libpq TLS/security
   contract.**

   The runbook requires SPIRE to preserve resolved libpq security parameters
   such as `sslmode`, `sslrootcert`, `sslcert`, `sslkey`, and
   `target_session_attrs`, and specifically says that `sslmode=verify-full`
   must be used rather than downgraded (`docs/SPIRE_LIBPQ_RUNBOOK.md:14`).
   The production remote paths all parse the conninfo but connect with `NoTls`:
   async probe/candidate/heap/insert/cancel in
   `src/am/ec_spire/coordinator/remote_candidates/dispatch.rs:532`,
   `:679`, `:923`, `:1129`, `:1297`, and `:1324`, plus the shared blocking
   helper used by descriptor preflight, UPDATE/DELETE/PK SELECT, prepared-xact
   resolution, and reaping at
   `src/am/ec_spire/coordinator/remote_candidates/governance.rs:81`.

   That means the AWS-facing connection contract is not implemented yet. A
   secret value that asks for `sslmode=require` or `verify-full` cannot be used
   as intended because there is no TLS connector or certificate verification
   backend attached to either `postgres` or `tokio-postgres`. At best this fails
   to connect to TLS-required remotes; at worst permissive settings can silently
   use plaintext where the runbook says SPIRE must preserve the provider's
   security policy.

   **Recommendation:** centralize sync and async remote connection creation
   behind one SPIRE helper that parses conninfo and attaches a real TLS backend
   for both `postgres` and `tokio-postgres`, with cancel using the same backend.
   Add a TLS-only local fixture or integration test that proves
   `sslmode=require` connects over TLS, `verify-full` verifies hostname/root
   certs, and certificate/auth failures map to sanitized status categories.

2. **P1 - Coordinator PK SELECT skips the read schema-drift guard.**

   Vector-order CustomScan planning applies read shape fingerprint validation
   before dispatch through `remote_search_libpq_read_shape_status`
   (`src/am/ec_spire/coordinator/remote_candidates/libpq_plan.rs:400`), which
   calls `validate_read_shape_fingerprints_before_remote_dispatch`
   (`src/am/ec_spire/coordinator/remote_candidates/write_payload.rs:437`).
   Coordinator-routed writes also validate write shape before remote dispatch
   for INSERT/UPDATE/DELETE (`write_payload.rs:554`, `:721`, `:771`).

   The transparent PK SELECT path is separate: CustomScan calls
   `ec_spire_forward_coordinator_select_tuple_payload`
   (`src/am/ec_spire/custom_scan/dml.rs:657`), which looks up placement and
   forwards to `spire_coordinator_select_remote_tuple_payload`
   (`src/lib.rs:8917`, `:9047`). The implementation resolves the conninfo and
   immediately opens the remote connection/query
   (`src/am/ec_spire/coordinator/remote_candidates/write_payload.rs:845-902`)
   without any read fingerprint validation.

   This leaves one Phase 13 workload outside the Phase 12 schema-drift contract.
   A coordinator-only, remote-only, or both-sides column-shape drift can make PK
   SELECT materialize stale JSON payloads or fail late with an ordinary remote
   SQL/decode error instead of the strict/degraded `schema_drift` behavior the
   production read path now has.

   **Recommendation:** run `validate_read_shape_fingerprints_before_remote_dispatch`
   in `coordinator_select_remote_tuple_payload` after resolving conninfo and
   before building/executing the remote SELECT. Because this primitive has no
   consistency-mode argument, default it to strict failure unless a deliberate
   degraded PK SELECT contract is added. Add PK SELECT schema-drift fixtures
   matching the read-drift matrix from Phase 12c.4.

3. **P2 - The production read path repeats remote connection and identity work
   per node for candidate and heap phases.**

   For a tuple-payload CustomScan read, the executor builds candidate-receive
   requests and heap-receive requests in separate phases, resolving the
   conninfo secret in both
   (`src/am/ec_spire/coordinator/remote_candidates/production_transport.rs:732`
   and `:787`). The adapter then opens a separate remote connection for each
   phase (`dispatch.rs:679` and `:923`) and repeats `to_regclass` plus endpoint
   identity validation (`dispatch.rs:709-740` and `:953-984`).

   The architecture is correct but expensive for the exact environment being
   tested next. On cross-AZ EC2 PostgreSQL, two connection handshakes, two
   secret lookups, and two endpoint identity probes per participating node will
   dominate small and medium `top_k` latency, and it also makes fanout benchmark
   results harder to interpret as SPIRE search overhead.

   **Recommendation:** either combine candidate and tuple-payload heap
   resolution into one remote endpoint for the production CustomScan read, or
   keep a per-dispatch session alive through both phases and cache
   `remote_index_oid`/endpoint identity for the descriptor generation. Include
   connection-open, secret-lookup, and endpoint-identity-query counters in the
   AWS packet manifests so the benchmark can separate network/session overhead
   from search work.

4. **P3 - The SPIRE module topology is still flattened by `include!` boundaries.**

   `src/am/ec_spire/mod.rs:115-124` text-includes lifecycle, types, remote
   candidates, diagnostics, shape, snapshots, maintenance, debug, and tests into
   the root access-method module. `src/am/ec_spire/custom_scan/mod.rs:206-213`
   does the same for planner, cost, DML, executor, tuple-payload, and tests.
   Several included files are 1,500-2,500 lines.

   This keeps the implementation working, but it weakens ownership boundaries:
   helper visibility is effectively root-module visibility, compile errors point
   through large synthetic modules, and future reviewers have to reason about
   coordinator, transport, DML, and diagnostics in one namespace. That matters
   now because the remaining risks are cross-cutting and security-sensitive.

   **Recommendation:** after the AWS gate or alongside a narrow security patch,
   convert the coordinator and CustomScan `include!` trees into real Rust
   modules with explicit `pub(super)` APIs. Start with the remote connection
   helper as the first true boundary, then split production transport, write
   payload/DML primitives, and diagnostics.

## Residual Risks

- Local cancel propagation depends on dynamically resolving PostgreSQL backend
  internals such as `InterruptPending`, `QueryCancelPending`, and
  `get_timeout_indicator` in
  `src/am/ec_spire/coordinator/remote_candidates/dispatch.rs:1359-1384`. If a
  PG18 build does not export those symbols in the AWS image, remote cancel/timeout
  propagation can silently degrade. Consider making symbol unavailability
  visible in diagnostics or using a supported `pg_sys`/interrupt surface where
  possible.
- Same-node multi-row INSERT remains intentionally unsupported by ADR-069 because
  the prepared GID is one per `(index_oid, node_id, served_epoch)`. Before AWS
  data-loading tests, fail this shape before opening remote sockets so the error
  is deterministic and does not look like a remote 2PC failure.

## Overall Assessment

The CustomScan-first architecture is the right production direction: the AM
stays local-only, coordinator tables carry descriptor/placement state, read
fanout has explicit dispatch states, and coordinator-routed writes have a
coherent 2PC story for INSERT/DELETE with documented UPDATE limitations. The
remaining blockers are not algorithmic; they are production-boundary issues.
Fix TLS/libpq security first, close the PK SELECT drift gap second, then AWS
testing will be measuring the intended system rather than known local-readiness
holes.
