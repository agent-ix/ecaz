# Review: SPIRE DML Relation Context Metadata (30850)

## Summary

The bridge from heap relation OID to the metadata
`classify_dml_frontdoor_query(...)` needs:
- selected `ec_spire` index OID for the heap relation;
- `ec_spire_distributed_table` candidate flag (true when heap has
  an ec_spire index and a single bigint PK);
- v1 bigint primary-key column name + type;
- ordinary column attnum→name mapping (for predicate-column
  resolution);
- embedding columns (the ec_spire index's key columns, bounded by
  `idx.indnkeyatts` so INCLUDE columns aren't treated as
  embedding columns).

Status taxonomy: `no_ec_spire_index` / `unsupported_pk_shape` /
`relation_context_ready`. PG18 fixture builds a bigint-PK ecvector
table and asserts the relation context.

The three review-focus questions:
- Relation-context fields sufficient for next planner-hook slice
  to call `classify_dml_frontdoor_query(...)`: yes — combined with
  the query-extraction layer this gives the classifier every
  fact it needs.
- `idx.indnkeyatts` correct boundary for embedding/index key
  columns: yes — closes the 30841 P3 about INCLUDE columns
  cleanly. Prevents an INCLUDE'd source_identity column from
  being treated as an embedding column for the rejection check.
- Candidate status meaning ("has ec_spire index + one bigint PK"):
  yes for v1; remote-placement enforcement remains correctly
  deferred to executor/replacement slice (since a coordinator-
  only deployment with all `node_id = 0` placements is also a
  valid distributed table per ADR-068).

## Findings

### P2 — `coalesce(min(idx.indexrelid::oid), 0::oid)` returns 0 OID for "no index"

The SQL uses `coalesce(min(...), 0::oid)` and Rust treats
`pg_sys::Oid::from(0u32)` as `InvalidOid`. Standard pattern, works.
But: if a heap relation has *multiple* ec_spire indexes, `min(...)`
silently picks one. ADR-069 v1 allows only one ec_spire index per
table (the routing centroids are per-index), so this should be
safe — but worth either:
- Asserting at most one ec_spire index per heap (via
  `count(*) <= 1` check), erroring with a clear message if not.
- Documenting that "the planner hook routes via the lowest-OID
  ec_spire index when multiple exist."

The first is the safer v1 default; the second only makes sense if
ADR-069 explicitly allows multiple ec_spire indexes per table.

### P2 — Multi-column PK rejected by `unsupported_pk_shape`

`dml_frontdoor_primary_key_column` returns `Option<...>` for the
PK; multi-column PKs return None. The status becomes
`unsupported_pk_shape` and the candidate flag flips false. Good —
the v1 hook fails closed on multi-column PKs without trying to
compose a route.

ADR-069 should pin this: v1 routing requires single-column bigint
PK; composite PKs are deferred. Otherwise an operator with a
`PRIMARY KEY (tenant_id, id)` table will see `unsupported_pk_shape`
and not know whether it's a v1 limitation or a config error.

### P2 — Embedding columns from `indkey[0..indnkeyatts]`

The `LIMIT idx.indnkeyatts`-style guard is the correct fix for the
30841 P3 INCLUDE-column concern. Worth a focused fixture that
creates an ec_spire index *with* an INCLUDE'd `source_identity`
column and asserts the relation context's `embedding_columns` does
NOT include `source_identity`. The current fixture probably doesn't
have an INCLUDE clause; without that test, a future regression
in the SQL would silently re-introduce the bug.

### P3 — `ec_spire_distributed_table` candidate flag

`ec_spire_distributed_table = (index_oid != InvalidOid && pk.is_some())`.
That's the right v1 condition. Worth flagging that this doesn't
require any actual remote placements — a freshly-created
coordinator table with no remote descriptors registered will still
be marked as a candidate. The planner hook will route through
the v1 helpers, which will surface
`ec_spire_placement` empty / descriptor missing as their own clear
errors. Acceptable, but worth pinning that the candidate flag is
a *structural* check (the table has the right shape), not a
*readiness* check (remotes are registered and reachable).

### P3 — `column_names: Vec<(AttrNumber, String)>` size

For a 100-column relation the column_names vec is 100 entries.
The planner hook calls this per query. Worth pinning a future
caching layer keyed on `(heap_relation_oid, relcache_invalidation_xid)`
once the rewrite slice lands, so the per-query cost stays bounded.

## Reviewer recommendation

Accept. The relation-context layer is the right shape and the
`indnkeyatts` boundary correctly closes the 30841 P3. The P2
items (multi-index/multi-column-PK handling, INCLUDE-column
fixture) are worth folding before the executor-replacement slice
depends on them.
