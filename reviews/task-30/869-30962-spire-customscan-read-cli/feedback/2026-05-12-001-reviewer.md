# Review: SPIRE CustomScan Read CLI (30962)

## Summary

Closes Phase 12.7 ecaz-owned local one-coord/two-remote
setup/teardown command row. Adds
`ecaz dev spire-multicluster customscan-read-pg18` as a
CLI-owned entry point for the existing CustomScan readiness
fixture script. Existing `transport-overlap-pg18` remains
the two-remote setup/teardown surface; this completes the
"repeated CustomScan readiness" path under the same
`ecaz dev spire-multicluster` parent.

## Findings

### P2 — CLI-owned wrapper aligns with project rule

Per the project CLAUDE.md "Local Operator CLI" guidance —
"Route PG18 socket work through `ecaz` commands ... instead
of direct `psql`, wrapper scripts, or one-off shell
plumbing" — replacing direct script invocation with an
`ecaz dev` subcommand for the recurring CustomScan read
operation is the right move. Future Stage E reruns (the
30961 cadence) now go through one approval rule covering
the operator surface consistently.

### P2 — Parser-level test scope is correct

Pinning the CLI surface with a parser-level test that does
NOT start PostgreSQL is right for this slice — it covers
"we won't accidentally rename or break the CLI surface"
without dragging in sandbox socket-bind issues that the
30961 setup-retry note already calls out as orthogonal to
fixture evidence.

End-to-end exercise of the underlying script happens in
30961's actual rerun artifacts, not in this CLI test. The
two layers are testing different things — keep the split.

### P2 — Operator-knob forwarding

Worth confirming the new CLI forwards the same flags the
underlying script accepts (`--skip-install`, `--run-dir`,
plus any case/scenario knobs). The 30961 setup-retry
artifacts demonstrate `--run-dir` matters operationally —
losing that flag in the wrapper would force operators back
to the script.

If the parser test asserts each forwarded flag, that's
sufficient. If it only asserts the command exists, pin a
follow-up to add per-flag parser assertions so a future
"clean up unused options" change can't silently strip them.

### P2 — Sibling command alignment

Naming aligns with the existing `transport-overlap-pg18`
sibling: same parent (`ecaz dev spire-multicluster`), same
PG-version suffix style. Good — operators with one in
muscle memory can reach the other by tab-completion. Worth
confirming the help text on both commands names what the
*difference* is (transport-overlap = two-remote
setup/teardown, customscan-read = repeated CustomScan
readiness against an existing setup) so an operator
choosing between them can pick correctly.

### P3 — Tracker closure scope

Closure is correctly scoped to the **local setup/teardown
command surface**. It does NOT claim:

- Phase 12.9 benchmark surface (`ecaz bench spire-pipeline`
  is still open);
- final readiness bundle;
- metric capture pipeline.

Honest scoping. The CLI lane has multiple commands in
flight; this one closes the multi-instance rerun command
specifically.

### P3 — Phase 12.7 lane closure

With 30961 (rerun) + 30962 (CLI) landing together, **Phase
12.7 multi-instance/placement/replica lane is fully
closed.** All eight tracker rows in 12.7 have evidence.

That leaves the open Phase 12 work clustered in:

- 12.2 typed transport cutover (5 items);
- 12.3 cost calibration (2 items);
- 12.4 async writes + cancellation (3 items);
- 12.9 bench harness + readiness bundle (4 items).

## Reviewer recommendation

Accept. Right CLI surface alignment, right test scope,
honest closure boundary. Combined with 30961 closes Phase
12.7 cleanly.
