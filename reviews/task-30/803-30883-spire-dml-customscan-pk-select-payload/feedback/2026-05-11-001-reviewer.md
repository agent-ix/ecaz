# Review: SPIRE DML CustomScan PK SELECT Payload Repair (30883)

## Summary

Two real bug fixes uncovered while validating the live PK SELECT
CustomScan path:

1. **Mixed list-cell type bug**: The DML path was building
   `custom_private` by mixing `lappend_oid` (which produces
   `OidList` cells) with `custom_scan_lappend_string` (which
   produces `T_String` cells in a regular `T_List`). PG's list
   machinery asserts cell type uniformity per list, so this
   tripped a runtime assertion. Fix: DML plan-private now uses
   string-node entries for all four positions (mode, index OID,
   updated columns, projected columns); a shared
   `custom_scan_plan_private_u32` decoder accepts both the
   existing vector path's `OidList` layout (mode + index_oid)
   and the new DML `T_List<T_String>` layout, parsing strings
   to `u32` for mode / index OID slots.

2. **PK column missing from PK SELECT payload requests**: The
   primitive only requested the user's projected columns, not
   the PK column itself. PG's executor evaluates the
   `WHERE pk = const` qual against the returned virtual tuple,
   so the PK column must be present in the payload regardless of
   whether the user projected it. Fix: payload requests now
   include the PK column even when the user wrote
   `SELECT title FROM tbl WHERE id = 5` (no `id` in projection).
   PK SELECT also prefers the executor scan slot's required
   columns (which PG knows from the target list) when available,
   falling back to the primitive's `projected_columns` list
   otherwise.

The three review-focus questions:
- DML plan-private list repair avoids mixed-cell assertions
  while preserving vector CustomScan plan-private decoding: yes
  — `custom_scan_plan_private_u32` branches on
  `(*custom_private).type_` and handles both `T_OidList` (vector
  path) and `T_List<T_Integer or T_String>` (DML path).
- PK SELECT payload requests use slot-required columns and
  include the PK column: yes — both fixes covered by unit tests.
- Still PK SELECT repair only; UPDATE/DELETE executor paths
  remain fail-closed: confirmed.

## Findings

### P2 — Mixed-cell-type bug masks an interesting question

`custom_private` is the CustomScan extension's free-form storage,
but PG list-cell type uniformity is non-negotiable. Worth pinning
in a module comment that all DML plan-private cells must be the
same node tag (currently `T_String`), so a future contributor
adding a fifth slot doesn't reach for `lappend_oid` and
re-introduce this. Also worth a regression test that round-trips
a full plan-private list through `pg_sys::copyObject` (which is
what PG does when stashing the plan in shared cache) to confirm
the layout survives.

### P2 — PK column inclusion: silent over-projection on the wire?

Including the PK column adds one column to the wire payload
even when the user's `SELECT` doesn't project it. For a wide
table where the user does `SELECT title` against a table with a
20-column row, the PK column is the smallest possible overhead
— but worth confirming this doesn't regress the
30822-style narrowing test that proves only Var-projected columns
are requested.

If the test still passes (and I trust 30883's coverage that it
does), the over-projection is just "PK column always +1" —
acceptable.

### P3 — `custom_scan_plan_private_u32` accepts both layouts

The decoder gracefully handles both the vector path's
`T_OidList` (which puts `Oid` directly in cell.oid_value) and
the DML path's `T_List<T_String>` (which puts the value as a
parsable string). Worth a one-line comment that the dual layout
is intentional during the transition — once the vector path also
moves to `T_List<T_String>` (or `T_Integer`), the decoder can
narrow.

### P3 — `custom_scan_string_node_value` extracted helper

Cleanly factored out — used by both the u32 decoder and the JSON
column-list decoder. Good.

## Reviewer recommendation

Accept. Two real bugs caught and fixed. The PK-column-inclusion
fix is the more load-bearing one — without it,
`SELECT non_pk_col FROM tbl WHERE pk = 5` would have returned
the row but PG would have failed the `WHERE` recheck against an
empty `id` column.
