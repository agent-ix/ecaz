# Review: SPIRE DML Query Shape Followups (30853)

## Summary

Closes the 30848 P2 items:

1. **CTE rejection pinned in ADR-069**: read-only `WITH` is
   explicitly fail-closed as `unsupported_subquery_shape` in v1,
   not just `hasModifyingCTE`. ADR text makes this an operator-
   visible v1 limitation.
2. **Bigint-PK equality opcode set widened**: the predicate
   detector now accepts `int8eq`, `int84eq`, `int82eq`, `int48eq`,
   `int28eq` so a `bigint = 5` (where `5` is an `int4` or `int2`
   literal) classifies as a v1 PK predicate. The widened detection
   is gated: int2/int4/int8 const/param values are accepted only
   *after* the predicate has bound to the target PK column and
   one of the bigint equality variants.

Plus value-kind extraction now follows `FuncExpr` /
`RelabelType` / `CoerceViaIO` wrappers when their result type is
`int8` and the inner arg is a coercible integer — handles PG's
implicit casts cleanly.

PG18 fixture confirms `WHERE id = 5` (int4 literal) classifies as
`pk_select_by_pk`; CTE-prefixed SELECT classifies as
`unsupported_subquery_shape`.

The three review-focus questions:
- Accepted equality opcode set is the right v1 surface: yes — the
  five bigint-equality cross-type variants are exhaustive for
  int2/int4/int8 RHS values.
- CTE rejection wording in ADR-069 explicit enough: yes.
- Test-only re-export boundary acceptable until the planner hook
  invokes the classifier directly: yes.

## Findings

### P2 — Implicit-cast walking is depth-1 only

`dml_frontdoor_value_kind` walks one level of `FuncExpr` /
`RelabelType` / `CoerceViaIO` wrapping. PG can stack multiple
coercions in some plan shapes (e.g., `int4 → numeric → int8`).
Worth either:
- Walking arbitrarily deep until a `Const` / `Param` is reached
  (recursive); or
- Documenting that v1 only handles single-step coercions and that
  multi-step coerced predicates fall through to
  `unsupported_pk_predicate`.

The first is more robust; the second is acceptable for v1 if the
common-case `WHERE id = 5` works (which it does — that's a
single-step int4 → int8 coercion).

### P3 — `dml_frontdoor_single_list_expr_arg`

Reads `args.elements` to get the first list element's `ptr_value`.
Standard PG list walking. Worth a comment that this assumes the
T_List shape from PG18; future PG versions may use the unified
list representation that already shipped, in which case the
helper still works but the comment should say so.

### P3 — `args.length != 1`

Rejects multi-arg coercion functions, which is right — only
single-arg `FuncExpr`s are valid implicit casts here. Multi-arg
function calls in a PK predicate (e.g., `id = greatest($1, $2)`)
correctly become `Other`.

### P3 — Coverage of `WHERE id = $1::int`

Worth adding a fixture that uses a parameterized query with an
explicit cast (`$1::int`) to confirm both the param + coerce path
classifies cleanly.

## Reviewer recommendation

Accept. The depth-1 coercion walk is the only forward-looking
concern — fold the recursive walk into the next packet that wires
the executor replacement, since prepared statements with stacked
coercions will be common.
