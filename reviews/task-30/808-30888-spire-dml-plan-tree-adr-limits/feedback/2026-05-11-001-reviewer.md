# Review: SPIRE DML Plan-Tree ADR Limits (30888)

## Summary

Pure docs slice. ADR-069 now reflects the chosen v1 architecture
exactly:
- Replaces the prior "ModifyTable/view-hook" placeholder with
  the planner-hook plan-tree replacement contract for both
  transparent UPDATE and transparent DELETE.
- New section "Transparent DML front-door limitations"
  documents the v1 tradeoffs that come with bypassing
  `ModifyTable`:
  - No `RETURNING`.
  - No coordinator table row-level triggers.
  - No statement-level transition tables.

The wording explicitly names these as v1 contract decisions
("part of the v1 contract rather than incidental implementation
gaps"), with a future-ADR escape hatch if SPIRE ever needs
trigger or transition-table semantics for distributed rows.

The two review-focus questions:
- ADR-069 matches the accepted top-level CustomScan plan-tree
  replacement approach: yes.
- v1 limitations clear and don't overpromise trigger /
  RETURNING semantics: yes.

## Findings

### P3 — Limitation list could mention `WITH ... UPDATE ...`

CTEs that perform DML against the distributed table aren't
listed but are covered by the existing classifier rejection
(`hasModifyingCTE`). Worth a one-line note in the limitations
section that DML inside CTEs falls back to the standard PG
plan, which would silently miss remote rows — though the
classifier should never let it reach plan replacement.

### P3 — Future-ADR pointer

The ADR mentions "a future ADR may introduce a richer
ModifyTable-like integration." Worth pinning the conditions
that would trigger such an ADR (e.g., concrete user demand for
trigger semantics, or RETURNING becoming required for ORM
support). Otherwise the future-work pointer dangles.

## Reviewer recommendation

Accept. ADR-069 is now coherent end-to-end with the implemented
design.
