## Findings

### P0 — My 30914 review was wrong about xid recovery

I wrote in 30914 P3: *"if a placement row exists with xid
matching the GID's xid, COMMIT PREPARED; if not, ROLLBACK
PREPARED."*

That's wrong. `ec_spire_placement` PK is `(index_oid, pk_value)`
— there's no `xid` column. The placement directory doesn't
expose the local-write xid as a queryable field.

The coder's correction is the right shape: `top_xid` in the GID
is **correlation evidence** an operator can use against
PostgreSQL logs (`pg_stat_activity`, log_line_prefix `%x`,
WAL records) — not a join key against the placement table.

The actual recovery rule is what the coder pinned:
- The coordinator decides commit vs rollback based on the
  known coordinator transaction outcome (did the local txn
  commit, did it abort) and the placement-row state for the
  affected key.
- The GID's `top_xid` and `(index_oid, node_id, served_epoch)`
  segments help the operator correlate which prepared remote
  belongs to which historical local txn, which is what they
  need to look up the outcome.

This is operationally cleaner than my suggestion (no schema
change to add a placement-row xid column, no extra index, no
storage cost) and matches PG's standard 2PC recovery posture
(operator decides, doesn't auto-resolve).

Accepting the correction; updating my mental model. The
ADR-069 wording now matches reality.

### P3 — Empty-projection case as a meaningful surface

The empty-projection fixture isn't just defensive — it
documents that callers can request typed transport with zero
projected columns and receive aligned empty arrays. Useful
when a caller wants to verify a row exists without paying the
projection cost. Worth a one-line mention in the typed
transport design doc that this is a supported shape.

### P3 — Diagnostic upgrades from name-only to name+type+OID

Stable error format helps both operator monitoring (grep on
type name) and automated tooling (grep on OID). The OID
inclusion is forward-compat — type renames or schema moves
don't break monitoring tied to OIDs.
