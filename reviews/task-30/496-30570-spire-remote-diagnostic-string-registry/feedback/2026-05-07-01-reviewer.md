# Remote diagnostic string registry review

Addresses cross-cutting finding #1 from the 30568 closeout. Constants
block at `root/remote_candidates.rs:23-46` covers target kinds,
status strings, transport, endpoint, candidate format, descriptor
source, row locator policy, heap resolution, and finalization
statuses. No blocking findings.

## Confirmations

1. The 24 centralized constants are referenced consistently across
   fanout, target/request/readiness, execution, libpq request,
   receive, merge-input, and finalization surfaces. Spot-checked
   against the previously hardcoded literals in 30558/30561–30568.
2. Placement-state names intentionally remain rendered from
   `SpirePlacementState` rather than centralized — those are an
   enum projection, not a status taxonomy, and centralizing them
   would couple the SQL surface to the enum's variant order.
3. The status-precedence logic in `merge_input_summary_row` and
   peers now reads constants instead of literals, eliminating the
   drift class.

## Findings (non-blocking)

- **Status taxonomy lives in `remote_candidates.rs` but is also
  needed by `snapshots.rs`** (e.g. capability-plan rows, descriptor
  readiness). The constants are `const` (not `pub(crate) const`),
  which works because both files are in the same crate-internal
  module tree but means a callsite that wants to assert
  `node.status == SPIRE_REMOTE_STATUS_REQUIRES_DESCRIPTOR` has to
  re-import or duplicate. 30571 propagates several into
  `snapshots.rs` consumers — confirm the constants are
  `pub(super)`/`pub(crate)` if cross-file reuse is the steady
  state. (Minor; the registry pattern is right.)
