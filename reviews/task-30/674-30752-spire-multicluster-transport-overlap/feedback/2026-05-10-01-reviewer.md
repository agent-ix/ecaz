Validation pass on `e63f5bb8` (`Add SPIRE multicluster transport
overlap harness`). Accepts the Phase 11.4 transport-overlap
evidence slice. Closes 30751 P3 #1.

## What I checked

- New pg_test-only helper
  `tests.ec_spire_test_production_transport_probe(...)` at
  `src/lib.rs:14088`. The function:
  - takes `node_ids`, `conninfo_secret_names`, and a
    `slow_node_id` parameter,
  - validates length parity and non-negative IDs (lines
    14104–14120),
  - resolves conninfo through secret names (no raw conninfo
    exposure on the SQL surface — preserves ADR-058 and packet
    30734 boundaries),
  - returns a timing-aware row shape: `node_id`,
    `started_after_ms`, `completed_after_ms`, `elapsed_ms`,
    `row_count`, `status`, `failure_category`.
  The `tests.` schema namespace + `pg_test` feature gating
  ensure it is not a production SQL surface.
- New harness at
  `scripts/run_spire_multicluster_transport_overlap_pg18.sh`
  starts one coordinator plus two separate remote PG18
  clusters and runs a fixed slow/fast probe through the
  production async adapter via the new helper. The script is
  shellcheck-clean (`bash -n` artifact present).
- `make spire-multicluster-transport-overlap` target in the
  Makefile (line 38) — operator entry point for the harness.
- Recorded evidence in
  `artifacts/multicluster-transport-overlap.log`:
  - slow remote: `completed_after_ms = 304`, `elapsed_ms =
    304`, `status = ready`;
  - fast remote: `completed_after_ms = 3`, `elapsed_ms = 3`,
    `status = ready`;
  - `fast_completed_before_slow = true`.
  This is the strongest possible signal of non-serialized
  progress — the fast remote completes 100× faster than the
  slow remote because the production adapter actually overlaps
  remote work rather than serializing it. Earlier loopback
  evidence from packet 30724 proved the in-process adapter
  doesn't serialize; this packet proves the same thing across
  three real PostgreSQL clusters.
- Phase 11 task and production-coordinator design docs now
  pin the C5 fault-matrix source-of-truth contract (closes
  30751 P3 #1): C5 must consume
  `ec_spire_remote_search_production_fault_matrix` rather than
  re-encoding strict/degraded behavior in the AM scan path.
  Stage D heap rows in the matrix are explicitly marked
  reserved category names until the heap executor emits them
  (closes 30751 P3 #2).
- All cited validations pass: `cargo fmt --check`,

## Answers to the packet's review questions

- **Right boundary for local multi-instance timing evidence
  without widening the production SQL surface?** Yes. Putting
  the helper under the `tests.` schema and `pg_test` feature
  gate is exactly the right shape — operators don't see a new
  function in a production install, but reviewer/CI evidence
  can exercise the production async transport adapter end to
  end. Loopback fixtures alone wouldn't prove cross-cluster
  overlap because they share a backend; this harness does.
- **One-coordinator/two-remote satisfies the Stage C
  "ready remotes are not serialized behind a slow remote"
  evidence gap while leaving Stage D/E open?** Yes. The
  recorded `fast_completed_before_slow=true` plus the 304 ms
  vs 3 ms elapsed delta is conclusive evidence that the
  production adapter overlaps remote work. Stage D (origin-
  node heap resolution) and Stage E (full fault matrix and
  cross-cluster epoch lifecycle) remain explicitly open. The
  packet correctly does not over-claim.
- **C5 matrix source-of-truth note sufficient to process
  30751 P3 feedback?** Yes for both items. P3 #1 (matrix as
  the C5 contract) and P3 #2 (Stage D rows are reserved
  names) are recorded in plan and design docs in the same
  packet that produces the transport-overlap evidence — so
  C5 reviewers can see both the contract and the proof in
  one read.

## P3 — informational

1. **Make target reuse for Stage E.** When Stage E lands the
   full one-coordinator/two-remote fixture for the fault
   matrix, the harness shape from this packet (start
   coordinator + N remotes, run a probe, capture timing rows)
   is the right scaffold to reuse. Worth noting in the
   design doc that the Stage E harness should extend
   `run_spire_multicluster_transport_overlap_pg18.sh` rather
   than fork a parallel script — keeps the multi-instance
   bring-up logic in one place.

   shape is per-`node_id`. If C5 ever needs per-PID timing
   (e.g. to attribute slow remotes to specific leaf
   placements), the helper would need to thread selected PIDs
   through. Optional — only worth doing if the AM scan path
   surfaces a per-PID question.
