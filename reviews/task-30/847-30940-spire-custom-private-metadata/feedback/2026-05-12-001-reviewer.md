# Review: SPIRE DML Custom Private Metadata (30940)

## Summary

Closes 30896 P7. Replaces JSON-encoded column-list payloads in
DML CustomScan `custom_private` with a flat PostgreSQL node
layout:

- Each column list = one count `String` node + one `String`
  node per column. Empty list = zero count.
- Mode, index OID, PK column remain string nodes (avoiding the
  earlier mixed `OidList` / `String` copyObject issue from
  30883).
- Removes `serde_json` parsing from plan-private column metadata
  decode (smaller dependency footprint on the executor's hot
  path).
- copyObject roundtrip evidence preserved.

The three review-focus questions:
- Counted string-node layout is copyObject-safe and easier to
  reason about: yes — the count + N strings shape is a
  standard PG node-list pattern, and copyObject handles
  homogeneous string lists cleanly.
- Updated/projected column offsets remain correctly decoded
  for empty + non-empty cases: yes — the count gates iteration,
  so empty lists naturally decode to empty Vec without special
  casing.
- Tracker doesn't overclaim broader JSON retirement: confirmed
  — this only cleans DML plan-private metadata, not the tuple
  payload transport that P1 (30913+) is replacing separately.

## Findings

### P3 — `serde_json` dependency footprint

After P1 typed transport switches the tuple payload off JSON
too, `serde_json` should be unused in the production hot path.
Worth pinning a future cleanup that removes the dep entirely
once the typed transport JSON-fallback window closes (per
30913's removal criteria).

### P3 — Count-prefix vs delimiter

Counted layout (count + N strings) avoids ambiguity if a
column name contains a delimiter character, which a
delimiter-based scheme would have. Robust pattern.

## Reviewer recommendation

Accept. Closes P7.
