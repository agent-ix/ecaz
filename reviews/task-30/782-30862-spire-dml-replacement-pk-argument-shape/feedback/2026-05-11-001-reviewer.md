# Review: SPIRE DML Replacement PK Argument Shape (30862)

## Summary

Extends `SpireDmlFrontdoorReplacementDecisionRow` with the PK
predicate argument shape:
- `pk_value_kind` — `"const_bigint"` / `"param_bigint"` /
  `"other"`.
- `pk_value_const: Option<i64>` — decoded canonical bigint value
  for `const_bigint` (covers int2/int4/int8 constant predicates).
- `pk_value_param_id: Option<i32>` — parameter id for
  `param_bigint`.

Reuses the existing PK predicate extraction, so classifier support
and replacement argument shape stay aligned. The
`replacement_sql` diagnostic surfaces the new fields. PG18 fixture
asserts `WHERE id = 5` (int4 literal) classifies as `const_bigint`
with value `5` for both PK SELECT and non-embedding UPDATE.

The three review-focus questions:
- PK argument shape sufficient for next CustomScan executor: yes
  — combined with the existing `pk_column`, the executor has
  everything to build the primitive call modulo runtime parameter
  evaluation (closed by 30868).
- Const/parameter handling aligned with classifier rules: yes —
  uses the same int2/int4/int8 acceptance set.
- Diagnostic SQL shape change acceptable while plan rewrite is
  disabled: yes.

## Findings

### P3 — `pk_value_const` for `param_bigint` is `None`

The decision row uses two `Option` fields (`pk_value_const` for
const, `pk_value_param_id` for param) keyed by `pk_value_kind`.
That's a tagged-union via parallel options; the typed PK argument
builder (30865) cleans this up into `SpireDmlFrontdoorPkValuePlan`.
Worth pinning that the row shape is the SQL surface (must stay
flat for the diagnostic) while the typed enum is the in-Rust
contract.

### P3 — `other` kind for `pk_value_kind`

Unsupported predicates fall to `pk_value_kind = "other"` with both
options `None`. Consistent with shape classifier returning
`unsupported_pk_predicate`.

## Reviewer recommendation

Accept.
