## C1 — Policy structure

`SpireRecursiveNprobePolicy` (`src/am/ec_spire/options.rs:206`) is a
single explicit container:

- `leaf_level_nprobe: u32` — level 1
- `nprobe_per_level: [u32; 32]` plus length — levels 2 upward
- `nprobe_for_parent_level(level)` returns leaf when `level <= 1`,
  configured when `level - 2 < len`, else `1` (the conservative default)
- `configured_nprobe_for_level(level)` returns `Some(...)` only when the
  level was explicitly set, used by diagnostics to distinguish
  `configured_above_level_1` from `one_child_above_level_1`

Level indexing is consistent with the reloption order ("ordered from
recursive level 2 upward"). The fixed `[u32; 32]` array avoids per-scan
allocation; the 32-entry cap matches the reloption parser cap.

`route_top_graph_object_to_leaf_routes` and the two
`route_recursive_routing_objects_to_leaf_routes_with_policy` /
`_with_policy` siblings now thread the policy through the recursive route
descent in one place. The non-`_with_policy` wrappers preserve binary and
test compatibility.

## C2 — Validation

`parse_nprobe_per_level_reloption` rejects:

- empty input or trailing-comma empties (`""`, `"2,"` — both error)
- non-positive integers (`"0"` errors with explicit "must be greater than 0")
- entries above `EC_SPIRE_MAX_NPROBE`
- malformed parses (`"bad"`)
- lists exceeding `EC_SPIRE_MAX_NPROBE_PER_LEVEL_ENTRIES = 32`

Cross-check: `SpireRecursiveNprobePolicy::from_level_values` re-validates
on the parsed Vec (zero check, length check) so even a synthetic policy
constructed in tests cannot bypass the reloption parser's invariants.

`leaf_level_nprobe == 0` with a non-empty `nprobe_per_level` errors —
prevents a "no leaves probed but configured upper levels" misconfiguration.

## C3 — Diagnostics

`ec_spire_index_options_snapshot` reports per-level resolution:

- `effective_nprobe_per_level={nprobe, ...}` — comma-separated list
  starting at level 1 (leaf), then per upper level
- `configured_above_level_1` when `nprobe_per_level_len > 0`
- `one_child_above_level_1` otherwise (the existing conservative label)

The local PG18 preflight in `30629-spire-scale-packet-runbook` shows this
working: `effective_nprobe_per_level={24,2}` and `configured_above_level_1`
for a build with `recursive_fanout=2, nprobe_per_level=2`.

## C4 — Backward compatibility

Both `route_recursive_routing_objects_to_leaf_routes(...)` and
`collect_snapshot_routed_probe_leaf_rows(...)` retain their original
`nprobe: u32` signatures and internally synthesize
`SpireRecursiveNprobePolicy::conservative(nprobe)`. The new
`_with_policy` siblings are private (`fn` not `pub(super) fn`). External
callers that don't care about per-level tuning continue to compile
unchanged.

## F1 — Double parse on hot path

`relation_options` (`src/am/ec_spire/options.rs:944`) currently:

```rust
let nprobe_per_level = unsafe { read_string_reloption(...) }
    .map(|value| {
        parse_nprobe_per_level_reloption(&value).unwrap_or_else(|e| pgrx::error!("{e}"));
        value
    });
```

This validates the string (good — fail-closed at relation read) but
discards the parsed `Vec<u32>` and stores the raw string. Later,
`EcSpireOptions::nprobe_per_level_values()` re-parses the string into
`Vec<u32>` whenever the policy is consumed. `relation_options` is on the
per-aminsert and per-rescan hot path; the parser cost is small (4-8
single-digit u32s in the local fixture) but the redundancy is avoidable.

**Mitigation:** store the parsed `Vec<u32>` in `EcSpireOptions` directly,
or memoize via a `OnceCell`-style guard on the relcache. Also fine to
defer this to a small follow-up commit.

## F2 — Phase 3 follow-up status

`plan/tasks/30-spire-ivf-foundation.md` Phase 3 closeout section still
shows:

```
- [ ] Explicit user-facing per-level fanout configuration. Phase 3 exposes
  ...
```

This item refers to per-level *fanout* (children per parent), not
*nprobe* (probes per parent). They are sibling parameters: `nprobe`
controls how many children a scan visits at each level; `fanout` controls
how many children each parent has at build time. The Phase 8 pull-forward
covers the former, not the latter.

Recommend adding a one-line clarifying note next to that Phase 3 item
distinguishing nprobe vs fanout, so reviewers reading later don't think
this checkbox should have flipped to `[x]` here.
