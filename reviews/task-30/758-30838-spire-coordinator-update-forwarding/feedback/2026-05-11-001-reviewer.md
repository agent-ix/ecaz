# Review: SPIRE Coordinator Update Forwarding Primitive (30838)

## Summary

First non-embedding UPDATE primitive for ADR-069. Two new SQL
functions:

- `ec_spire_remote_update_tuple_payload(index_oid, pk_column,
  pk_value, row_payload, updated_columns)` — remote endpoint that
  validates pk_column + updated_columns against the remote heap,
  rejects PK-column updates explicitly, and runs a parameterized
  UPDATE matching by `int8send(pk)::bytea`.
- `ec_spire_forward_coordinator_update_tuple_payload(...)` —
  coordinator helper that looks up `(node_id, served_epoch)` from
  `ec_spire_placement` by `(index_oid, pk_value)`, then forwards the
  UPDATE to the owning remote without 2PC.

Per ADR-069's contract for non-embedding UPDATEs, no two-phase
commit is required because the coordinator placement directory
doesn't change — `(id → node_id)` mapping is unaffected by a
non-embedding column change.

The three review-focus questions:
- Split is appropriate (forwarding primitive now, transparent
  UPDATE hook later): yes.
- `ec_spire_placement` as source of truth for `pk_value → node_id`:
  yes, with a constraint conflict noted below.
- No 2PC needed for non-embedding UPDATE: yes, by ADR-069's
  contract.

## Findings

### P1 — Helper rejects `node_id <= 0`, conflicting with 30824

`if node_id <= 0 { pgrx::error!("ec_spire coordinator update
placement points at local node 0"); }` rejects the
`SPIRE_LOCAL_NODE_ID = 0` case that 30824 specifically allowed in
the placement catalog.

This means: if a row was INSERTed via the coordinator-routed path
into the coordinator's own local shard (`node_id = 0`), the UPDATE
forwarding helper will fail to update it. Two acceptable shapes:
- Helper detects `node_id == 0` and runs the UPDATE directly on the
  local heap (no remote dispatch needed).
- ADR-069 documents that v1 UPDATE forwarding only supports remote
  shard rows; coordinator-local rows must be UPDATEd via plain SQL
  with a manual `WHERE pk = ...`.

Either is acceptable for v1 but the current "error out" is the
worst of both worlds: the placement catalog says "local node 0 is
valid," but the UPDATE path says "local node 0 is invalid." Pick
one.

### P2 — UPDATE forwarding has at-most-once semantics, not exactly-once

Without 2PC, the forwarding helper sends the UPDATE to the remote
and waits for the response. If the connection drops between the
remote commit and the coordinator receiving the response,
PostgreSQL's standard libpq behavior surfaces an error to the
coordinator — but the row may have been updated on the remote
anyway. The coordinator's caller then sees an error and may retry,
producing an idempotent second UPDATE (same value) or a different
result (if the application re-derives the new value). Acceptable
for v1, but worth pinning in ADR-069 that:
- Non-embedding UPDATE has at-most-once visible semantics from the
  application's perspective.
- Idempotent UPDATEs (e.g., `SET title = $new_value WHERE id = $id`)
  are safe to retry.
- Non-idempotent UPDATEs (e.g., `SET counter = counter + 1`) need
  application-level idempotency.

### P2 — `pk_column` and `pk_value` separately validated

`pk_column` is a free-text identifier that the remote endpoint
quotes via `quote_identifier`. `pk_value` is bytea. Combined with
the `pk_column != updated_column` check this is safe. Worth a
comment that the helper trusts the caller to pass the *trigger's*
or *application's* canonical PK column and bytes — there's no
cross-check that `pk_value` is the bigint encoding of the row the
caller intends.

### P3 — `validation_columns` includes pk_column for catalog validation

Good defense — the validator from 30807 confirms both pk_column
and updated_columns exist on the remote heap before any SQL
construction. Catches schema-drift between coordinator and remote.

### P3 — `served_epoch <= 0` rejection

`if served_epoch <= 0 { error }` matches the placement catalog's
`served_epoch > 0` constraint. Defensive; harmless.

### P3 — Test coverage

The packet ships one fixture exercising the title UPDATE through
the helper. Worth follow-up coverage for:
- UPDATE of multiple non-embedding columns in one call.
- UPDATE where the remote row doesn't exist (placement points at
  the right node but the row is gone — should the remote endpoint
  surface `updated_count = 0` so the caller can detect?).
- UPDATE with a `pk_value` not in the placement catalog (should
  surface "placement row missing" vs silently no-op).

The first looks already supported; the second / third are real
operational concerns that merit fixtures before the transparent
UPDATE hook lands.

## Reviewer recommendation

Accept the primitive shape. The P1 `node_id == 0` conflict is a
hard contradiction with the 30824 catalog policy that the next
slice must resolve before the transparent UPDATE hook calls this
helper. The at-most-once semantics are correct per ADR but worth
documenting before applications depend on them.
