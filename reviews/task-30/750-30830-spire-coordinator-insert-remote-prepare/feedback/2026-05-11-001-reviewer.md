# Review: SPIRE Coordinator Insert Remote Prepare (30830)

## Summary

First mutating 2PC primitive lands. `coordinator_insert_prepare_remote_sql`:
- Consumes the 30829 dispatch readiness plan and returns early on
  blocked status.
- Acquires a libpq governance permit, opens the libpq connection,
  generates a coordinator-scoped GID
  (`ec_spire_insert_<index_oid>_<node_id>_<served_epoch>_<top_xid>_<pid>`),
- Runs `BEGIN; <remote_sql>; PREPARE TRANSACTION '<gid>'`,
- Registers `Commit` and `Abort` xact callbacks that, on the local
  txn outcome, call `COMMIT PREPARED` / `ROLLBACK PREPARED` against
  the remote.
- Returns `remote_insert_prepared` with next step
  `local_placement_directory_write`.

Internal/test-only Rust primitive (no SQL surface), `pg_test` config
sets `max_prepared_transactions = 10` so the PG18 fixture can run.
The 2PC ordering is right: remote prepare succeeds before local
placement staging is allowed; on local commit the remote prepared
xact commits; on local abort it rolls back.

The five review-focus questions:
- 2PC boundary correct: yes; remote PREPARE precedes local
  placement staging, and resolution follows local txn outcome.
- Callback failure swallowed for prepared-xact recovery: yes for
  v1; orphaned prepared xacts surface in `pg_prepared_xacts` for
  manual operator action. See P2 below on GID recoverability.
- Helper internal/test-only: yes — no `#[pg_extern]` exposes raw
  remote SQL.
- `max_prepared_transactions = 10` for pg_test scoped right: see
  P2 below.
- Tracker still leaves transparent DML hook open: confirmed.

## Findings

### P2 — GID includes backend pid; recovery loses pid context

The GID format embeds the **coordinator backend pid**. When a
coordinator backend crashes between local commit and remote COMMIT
PREPARED, the prepared xact orphans on the remote. An operator
running `SELECT * FROM pg_prepared_xacts` on the remote sees
`ec_spire_insert_<oid>_<node>_<epoch>_<xid>_<pid>`, but `pid` is the
*dead* coordinator backend's pid, useless for recovery decisions.

Two acceptable shapes:
- Drop the pid from the GID (the `top_xid` is already locally
  unique within a coordinator instance, and the `(index_oid, node,
  epoch, xid)` tuple is enough).
- Keep the pid but document a recovery runbook: "match
  `ec_spire_insert_*_*_*_<xid>_*` against the coordinator's
  `pg_xact_status` to decide COMMIT vs ROLLBACK PREPARED."

The pid is informative for live debugging but actively harmful for
post-crash recovery. Lower preference for keeping it.

### P2 — `max_prepared_transactions = 10` may be tight for parallel pg_test

The default pgrx test runner runs many pg_test fixtures in parallel
within one PG18 instance. If multiple fixtures call coordinator
INSERT helpers concurrently, the cap of 10 prepared transactions
will trip non-deterministically. Consider raising to 64 or
`max_connections` for the test cluster, with a one-line comment
noting it's a test-only floor.

### P2 — Statement timeout coverage on the BEGIN/INSERT/PREPARE sequence

`remote_search_libpq_connect_with_session_timeouts` sets session
statement_timeout (Stage C). Confirm that timeout *applies* to the
BEGIN / INSERT / PREPARE TRANSACTION batch — if the remote stalls
mid-INSERT, the local backend is blocked on `client.batch_execute`
with no PG-interrupt bridge in this code path. The async cancel
bridge from Stage C handles SELECT side; the synchronous
`batch_execute` here may need a similar bridge or a documented
"prepare may stall up to statement_timeout" contract.

### P3 — `Top transaction id` only set if a real txn exists

`pg_sys::GetTopTransactionId()` allocates a real XID when called,
which forces the local txn into a real transaction state. That's
the desired behavior for 2PC (you can't PREPARE without a
real local txn) but worth documenting. If this primitive is called
inside an autonomous-style read-only context, it implicitly
upgrades to a real write txn, which has WAL implications.

### P3 — Callback closures clone conninfo string twice

`commit_conninfo = conninfo.clone(); commit_gid = prepared_gid.clone();
rollback_gid = prepared_gid.clone();` — three clones for two
callbacks. The Abort callback uses `conninfo` directly (moves it),
so the Commit closure has to clone. Functional; a minor optimization
would put the conninfo in an `Arc` shared by both closures.
Cosmetic.

### P3 — `quote_sql_literal` only doubles single quotes

PG uses single-quote doubling for string literals when
`standard_conforming_strings = on` (default). The function
correctly handles that; backslashes are literal. Worth a comment
that the helper assumes the default (which has been the default
since 9.1) so future PG version bumps don't silently change escape
semantics.

## Reviewer recommendation

Accept the 2PC primitive shape. The P2 GID-pid and prepared-txn-cap
items are operational concerns worth folding into the next mutating
INSERT slice (or the cleanup packet) before this becomes a busy
production code path. The statement-timeout question is the most
load-bearing — confirm or document.
