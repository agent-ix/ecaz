# Front-door Hook Boundary Question

The ADR-069 primitives are now in place for coordinator-routed INSERT, UPDATE,
DELETE, PK-keyed SELECT, and embedding-UPDATE rejection:

- INSERT has a trigger front door because `BEFORE INSERT` sees the incoming row
  before the coordinator heap row is suppressed.
- UPDATE/DELETE/PK SELECT have shared primitives that route by
  `ec_spire_placement`.
- UPDATE/DELETE/PK SELECT also handle `node_id = 0` local placements.

The remaining gap is the transparent front door for:

- `UPDATE documents SET non_embedding_col = ... WHERE id = ...`
- `DELETE FROM documents WHERE id = ...`
- `SELECT cols FROM documents WHERE id = ...`

I do not think a normal row trigger can satisfy this, because remote-owned rows
are not present in the coordinator heap after the INSERT trigger returns
`NULL`; the trigger never sees the target row. A generic updatable view also
looks risky unless it is allowed to call the PK-read primitive only after the PK
qual is known. A naive view over all placement rows would become a cross-shard
non-vector scan, which ADR-069 explicitly defers.

Please confirm the intended v1 front-door integration point before I code the
next slice:

1. Should UPDATE/DELETE be implemented through a planner/executor
   `ModifyTable` hook that recognizes the narrow bigint-PK equality shape,
   calls the existing primitive, and prevents the base heap ModifyTable from
   reporting a false `0` rows affected for remote-owned rows?
2. Should PK-keyed SELECT use a planner hook that installs a dedicated
   SPIRE-owned scan path for the narrow bigint-PK equality shape, or should it
   be an operator-installed proxy view with explicit limitations?
3. For the first implementation, is it acceptable to support only the v1 narrow
   shape already used by INSERT (`bigint` primary key, equality predicate,
   no joins/subqueries in the target DML, no RETURNING for UPDATE/DELETE), with
   all other shapes falling back or erroring explicitly?

My recommendation is to code the next packet as a narrow hook-planning/status
surface plus tests that prove unsupported shapes fail closed, then implement
the hook in a separate packet once the reviewer confirms the hook boundary.
