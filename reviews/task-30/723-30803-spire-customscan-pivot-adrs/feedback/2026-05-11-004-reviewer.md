# Re: DML Front-Door Integration Question (response to coder seq 003)

## Recommendation: Option 2 (planner-hook plan-tree replacement)

The right v1 target is a planner_hook that replaces the entire
`PlannedStmt.planTree` with a `CustomScan` (no `ModifyTable`
above it), keeping `commandType = CMD_UPDATE` / `CMD_DELETE` so
PG generates the right command tag, and incrementing
`estate->es_processed` from inside the CustomScan executor so
`UPDATE N` / `DELETE N` reports correctly.

Specifically: do not put the CustomScan **under** ModifyTable.
Replace ModifyTable entirely.

## Why not the other options

### Option 1 (view + INSTEAD OF trigger): no.

ADR-068's example DDL exposes `documents` as a real table to
applications. A view changes:
- `\d` shows `view`, not `table`.
- `pg_dump` / `pg_restore` semantics shift.
- `ALTER TABLE` no longer applies to the user-visible relation;
  operators have to know which underlying table to alter.
- `CREATE INDEX ON documents` errors against a view.
- ORM tools and migration frameworks treat views differently.

The view approach is what some Citus generations used, and it is
operationally awkward enough that ADR-069 should not adopt it.

### Option 3 (stub rows in coordinator heap): no.

This brings back the failure mode ADR-067 was specifically built
to eliminate. The coordinator heap would grow linearly with
total distributed row count; stub maintenance under concurrent
INSERT/DELETE becomes its own distributed-state problem; vacuum
/ autovacuum on the stub table contends with the actual write
path. The whole point of the CustomScan pivot was getting rid of
this class of solution.

### Option 4 (other PG hook): not really.

The hook surfaces PG exposes for DML interception are:
- `planner_hook` (already installed).
- `set_rel_pathlist_hook` / `set_join_pathlist_hook` (used for
  CustomPath generation, but only at scan level under ModifyTable).
- `ProcessUtility_hook` (only for utility statements, not DML).
- `ExecutorRun_hook` (wraps the whole executor, too coarse to
  rewrite plans).

The "ModifyTable/view-hook" wording in ADR-069 was loose. The
realistic v1 surface is `planner_hook` returning a
plan-tree-replaced `PlannedStmt`. That's what to build.

## Concrete shape for option 2

In the planner hook, when the existing classifier identifies a
supported `UPDATE pk_eq` or `DELETE pk_eq` shape:

1. Run `standard_planner` (or the chained previous hook) to get
   the original `PlannedStmt` so PG's analyzer-set fields like
   `commandType`, `canSetTag`, `rtable`, etc. are populated.
2. **Replace `planTree`** with a `CustomScan` node. The
   `CustomScan.scan.scanrelid = 0` (no real underlying scan)
   and `custom_scan_tlist` describes the result row shape (one
   "rows affected" row, or empty for plain UPDATE/DELETE).
3. Discard the original `ModifyTable` (and its scan child)
   entirely. The new plan tree is just the CustomScan.
4. Keep `PlannedStmt.commandType = CMD_UPDATE` (or `CMD_DELETE`),
   `canSetTag = true`, `hasModifyingCTE = false`,
   `hasReturning = false` (already enforced by the classifier).
5. Drop `relationOids`, `resultRelations` since there is no
   coordinator-side modification (or include them and rely on
   PG's permission check; safer to include them so PG still
   verifies the user has UPDATE/DELETE on the logical relation).

In the CustomScan executor:
- `BeginCustomScan` constructs the `SpireDmlFrontdoorPrimitiveInvocation`
  and dispatches via the existing primitive
  (`ec_spire_forward_coordinator_update_tuple_payload(...)` or
  `ec_spire_prepare_coordinator_delete_tuple_payload(...)`).
- The primitive returns the affected rowcount (1 for v1
  PK-exact-match shapes; the primitive already enforces this).
- `ExecCustomScan`:
  - First call: increment `estate->es_processed` by the
    rowcount, return `NULL` (no tuple to project).
  - Subsequent calls: return `NULL` (end of scan).
- `EndCustomScan` cleans up.

Result: `UPDATE 1` / `DELETE 1` is reported correctly because
PG's `ProcessQuery` reads `estate.es_processed` after executor
completion.

## Required v1 contract pinning in ADR-069

This shape works iff the v1 contract restricts:
- **No RETURNING** (already enforced).
- **No row-level triggers** on the distributed table for UPDATE
  / DELETE — the CustomScan path bypasses ModifyTable, which is
  where row triggers fire. Pin in ADR-069 as a v1 limitation.
- **No statement-level triggers** that depend on transition
  tables — for the same reason.
- **No concurrent FK constraints** that depend on PG's row
  identity (rare for distributed table use cases).

PG-level permission checks (RLS, GRANT/REVOKE) still run because
`PlannedStmt.relationOids` includes the coordinator's logical
relation OID.

## Comparison with PK SELECT (already wired)

PK SELECT works today because:
- `commandType = CMD_SELECT` doesn't enforce ModifyTable.
- `EState.es_processed` increments naturally as the CustomScan
  emits each tuple.
- Row count maps to "rows returned" via the existing PG path.

UPDATE/DELETE need the same shape: replace the plan tree, keep
the commandType, drive `es_processed` via the CustomScan
executor. No ModifyTable involved.

## Citus precedent

This is essentially what Citus does for shard-routed UPDATE /
DELETE — see `MaybePlanCitusModifyTablePath` in their planner
code. The planner hook returns a CustomScan-only PlannedStmt,
the CustomScan calls remote nodes, and the rowcount surfaces via
`es_processed`. That precedent suggests the approach is robust
under PG18.

## Suggested next-packet shape

1. **Plan-replace scaffold**: in the existing planner hook,
   detect supported UPDATE/DELETE shape, run standard_planner,
   replace `planTree` with a CustomScan that carries the
   primitive-plan invocation in `custom_private`. **Initially**
   the CustomScan executor still errors out with "UPDATE/DELETE
   executor not yet wired" — no behavior change.
2. **UPDATE executor wiring**: connect the CustomScan executor
   for UPDATE mode to call
   `ec_spire_forward_coordinator_update_tuple_payload(...)`,
   increment `es_processed`, return NULL.
3. **DELETE executor wiring**: same shape for DELETE.
4. **Multicluster fixtures**: actual `UPDATE distributed_tbl SET
   ... WHERE id = $1` and `DELETE FROM distributed_tbl WHERE id
   = $1` against real two-cluster setup, asserting the row
   count and that the remote heap reflects the change.
5. **ADR-069 docs slice**: pin the trigger / RETURNING /
   transition-table v1 limitations.

## Reviewer recommendation

**Build option 2.** Discard ModifyTable from the plan tree
entirely; do not try to put the CustomScan under it. Drive
rowcount via `es_processed` from the CustomScan executor.
Document the v1 trigger / RETURNING limitations in ADR-069 as
explicit deferrals.

The 30877→30882 scaffold packets are exactly the right
preparation for this — DML mode plumbing, mode-asserting
wrappers, column metadata, metadata validation, primitive
invocation construction. The plan-replace slice is the missing
piece, and once it lands the existing scaffold gives you the
executor-side machinery for free.
