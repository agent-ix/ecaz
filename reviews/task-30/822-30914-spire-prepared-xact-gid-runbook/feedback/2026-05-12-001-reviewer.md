# Review: SPIRE Prepared Xact GID Runbook (30914)

## Summary

Closes 30830 P2 / 30896 H3 — the prepared-transaction recovery
gap that I flagged as one of three highest-priority items before
AWS. Drops the volatile coordinator backend pid from the
prepared-xact GID:

- Old: `ec_spire_insert_<index_oid>_<node_id>_<served_epoch>_<top_xid>_<pid>`
- New: `ec_spire_insert_<index_oid>_<node_id>_<served_epoch>_<top_xid>`

`(index_oid, node_id, served_epoch, top_xid)` is unique within a
coordinator instance and survives backend death — exactly what
recovery needs. Prefix `ec_spire_insert_` retained for
compatibility with both INSERT and DELETE prepare paths (per the
30889 P3 note that the shared 2PC helper kept the historical
prefix).

PG test assertions that the GID has the stable four-field
identity and no pid segment.

Plus runbook docs in ADR-069 and `docs/SPIRE_DIAGNOSTICS.md`
covering the recovery flow.

The three review-focus questions:
- Dropping pid sufficient for the recovery concern: yes — pid
  was the only volatile component; everything else (index OID,
  node ID, served epoch, top XID) is reproducible from the
  coordinator's placement-directory state.
- Runbook decision rules conservative enough: per the request,
  the runbook avoids remote-only bulk resolution when the
  affected key or coordinator outcome is unknown. That's
  correct — silent automated resolution of orphaned 2PC is
  the opposite of safety.
- Fixture change appropriate (creating the remote SPIRE index
  the descriptor names): yes — fixture was previously
  registering a descriptor naming an index that didn't actually
  exist.

## Findings

### P3 — Compatibility prefix `ec_spire_insert_` for DELETE GIDs

The prefix is technically misleading for DELETE prepared
transactions (they're not INSERTs), but renaming the prefix
would break in-flight upgrades carrying orphaned prepared
transactions from before the change. Acceptable — the runbook
should call out that the prefix is operation-agnostic and
operators shouldn't rely on it to distinguish INSERT from
DELETE orphans. Worth confirming this is in
`docs/SPIRE_DIAGNOSTICS.md`.

### P3 — `top_xid` reproducibility

The `top_xid` segment is the coordinator's TopTransactionId at
prepare time. For recovery, the coordinator can correlate this
with the placement directory: the same xid would have written
the placement row in the same transaction. So the GID's xid
matches the local placement's xid → operator can decide commit
vs rollback by checking whether the local row exists.

Worth pinning in the runbook: "if a placement row exists with
xid matching the GID's xid, COMMIT PREPARED; if not, ROLLBACK
PREPARED." That's the algorithmic recovery rule. Already
documented in the docs change?

### P3 — Recovery helper deferred (per the request)

The original 30830 P2 also asked about a coordinator-side helper
`ec_spire_recover_orphaned_prepared_xacts(node_id)`. Not in
this packet — appropriate, the runbook lets operators do
manual recovery via `pg_prepared_xacts` + the placement-
directory check. The helper is a future operator-experience
improvement, not a v1 correctness gap.

## Reviewer recommendation

Accept. Closes the H3 recovery gap cleanly. The pid removal is
the load-bearing change; the runbook docs make manual recovery
deterministic.
