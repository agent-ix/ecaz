## Answers to the packet's review questions

- **`tokio-postgres` adapter boundary acceptable as the C1
  overlapped-transport proof while direct libpq async/pipeline FFI
  is unavailable through `pgrx::pg_sys`?** Yes. The pgrx-exposed
  libpq surface does not include the pipeline-mode entry points,
  so `tokio-postgres` is the cleanest path to an actual overlapped
  proof without writing FFI shims. The adapter is narrow enough
  that swapping to a different transport later (real libpq async,
  or a custom protocol) is a contained change. Worth pinning in
  the design doc that the transport choice is C1-implementation-
  only and not part of the C0–C6 contract.
- **Per-query connect / per-dispatch close closes 30722 P2 for
  C2?** Yes. With this lifetime, C2 cancellation is "cancel the
  dispatch future and drop the per-dispatch connection," which has
  no pool invalidation semantics to debug. Once C2/C4 are stable,
  pooling can land as a measured optimization.
- **Slow/fast loopback fixture strong enough to lock the
  non-serialized-progress guarantee?** Yes for this slice. The
  fixture proves the adapter does not introduce serialization;
  candidate merge wiring (C5) will need its own fixture that
  proves the AM scan path also doesn't serialize, but that's a
  later slice.

## P2 — close before C5 AM scan integration

1. **Document the transport choice as C1-implementation only.** The
   design doc should state that the C0–C6 contract is transport-
   neutral and that `tokio-postgres` is the C1 adapter
   implementation, not the protocol. Otherwise C5 reviewers may
   read the dependency edge as a permanent commitment, which it
   isn't. One short paragraph in
   `plan/design/spire-production-coordinator-executor.md`.

## P3 — informational

2. **PostgreSQL backend safety with `tokio` runtime.** Running a
   tokio runtime inside a Postgres backend is a known-hairy thing
   (signal handling, longjmp interaction with errors, OS-thread
   binding). Worth confirming with a future packet that the C1
   adapter creates a fresh runtime per query (so backend signal
   handling is not subverted) and that the runtime tear-down is
   panic-safe across PG-style errors. The current
   "runtime creation as the only batch-level error" posture from
   30725 suggests this is already the shape, but pinning it
   explicitly in the design doc avoids surprises in C2 work.

3. **Tokio feature footprint.** Current features are `rt`, `time`,
   `net`. If C2 uses `select!` for cancellation,
   `macros` may be required; if C5 needs a multi-threaded runtime
   for AM scan parallelism, `rt-multi-thread` may be required.
   Worth tracking the feature set against the C0–C6 needs in the
   design doc so the dependency footprint stays minimal.

## Status

Closed. C1 transport adapter is acceptance-quality; per-query
connect/close lifetime gives C2 a clean target. AM scan wiring (C5)
remains the deferred follow-up.
