## Answers to the packet's review questions

- **Dry default + explicit live probe is the right production
  operator contract for this stage?** Yes. Phase 8 finding F-PIPELINE-2
  was specifically that an operator inspecting "what does the
  pipeline look like" should not pay the cost of opening remote
  libpq sockets. Splitting them keeps a cheap default for routine
  inspection and an explicit opt-in for actual probing — same
  shape Postgres uses for `pg_stat_*` views vs `pg_stat_reset()`.
- **Dry downstream step statuses (`requires_libpq_executor`,
  zero counts) clear enough for candidates / heap candidates /
  coordinator result?** Yes. The status string is the same across
  all three paths and the recommendation field names the live
  entrypoint, so an operator doesn't have to guess what changed.
- **Operator entrypoint contract makes the cost boundary visible
  before runbook material?** Yes. The dry/live distinction is in
  the contract row itself, not just in narrative documentation,
  so any future runbook generation that consumes the contract
  sees the boundary in machine-readable form.
