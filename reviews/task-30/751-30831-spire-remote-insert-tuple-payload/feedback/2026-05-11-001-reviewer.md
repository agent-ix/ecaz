# Review: SPIRE Remote Insert Tuple Payload (30831)

## Summary

Adds the typed remote-side INSERT endpoint
`ec_spire_remote_insert_tuple_payload(index_oid, row_payload jsonb,
requested_columns text[])`. The endpoint:
- Derives the heap relation OID from the supplied SPIRE index
  (no separate remote table-name contract).
- Validates the column list against `pg_attribute` (reuses
  `ec_spire_validate_tuple_payload_columns` from 30807 — non-system,
  non-dropped, non-empty, no duplicates).
- Quotes column identifiers via the new `ec_spire_quote_identifier`
  helper (`"col"` with `"` doubled).
- Builds dynamic SQL `INSERT INTO {regclass} ({cols}) SELECT {cols}
  FROM jsonb_populate_record(NULL::{regclass}, $1::text::jsonb)`,
  delegating typed conversion to PostgreSQL's standard
  `jsonb_populate_record` machinery.
- Returns `(index_oid, heap_relation_oid, inserted_count,
  payload_column_count, status='ready')`.

The five review-focus questions:
- Right remote-side boundary before the coordinator DML hook: yes,
  this is a stable typed endpoint independent of the caller's table
  shape.
- Heap derived from SPIRE index avoids a separate name contract:
  yes — single source of truth.
- `jsonb_populate_record` for v1 scalar tuple payload: correct
  choice; lets PG own type input including `ecvector` text format.
- Dynamic SQL constrained by catalog-validated column names and
  identifier quoting: yes — column names go through `pg_attribute`
  validation then `quote_ident`-style quoting.
- Tracker leaves tuple-to-JSON construction, hook wiring, and
  CustomScan read-after-insert open: confirmed.

## Findings

### P2 — Single-row endpoint; batch INSERT requires N round-trips

`jsonb_populate_record` produces one record from one JSON object;
the SELECT returns at most one row. For a coordinator INSERT
forwarding a single application row this is correct. For bulk-load
or multi-row INSERT statements, the coordinator would need to call
this endpoint once per row, adding round-trip latency.

Worth pinning a future endpoint variant
`ec_spire_remote_insert_tuple_payloads(index_oid, row_payloads
jsonb[], requested_columns)` that uses
`jsonb_populate_recordset(NULL::{regclass}, $1)` so the bulk-INSERT
path lands cheaply when the coordinator hook supports multi-row
INSERT. Not blocking for v1; the bulk-load tooling track in ADR-069
addresses the real high-throughput path.

### P2 — `requested_columns` list not validated against payload keys

If `requested_columns = ['id','title']` but `row_payload` only
contains `{"id": 1}`, `jsonb_populate_record` returns
`(id=1, title=NULL, ...)`, the SELECT projects `id, title`, and the
INSERT sends `(1, NULL)` to the heap. The heap's `NOT NULL` /
`DEFAULT` semantics then apply — usually causing a clear PG error
on `title NOT NULL`. Acceptable, but the failure surfaces as a
generic NOT NULL violation rather than "payload missing requested
column 'title'".

Suggestion: either explicitly check `row_payload ? column_name` for
each requested column before INSERT, or document that missing keys
will manifest as standard PG insert errors. The first is cheap and
gives clearer operator messages.

### P2 — `regclass` interpolated into SQL via `format!`

Same pattern as 30807 / 30812. The `regclass::text` cast emits
properly-quoted/qualified identifiers, so this is safe in practice.
Same long-standing nit: a future packet should factor a
`quote_qualified_relname(oid)` helper that all remote-SQL builders
use, so the safety property is local to the helper.

### P2 — `Spi::connect_mut` with `client.select` for an INSERT

`Spi::connect_mut(|client| client.select(insert_sql, ...))` runs
the INSERT-CTE and the trailing SELECT. Functional but slightly
unusual; pgrx's `Spi::run_with_args` or `Spi::get_one_with_args`
would be more idiomatic for "INSERT and return count." The CTE
form does work: `INSERT ... RETURNING 1` is wrapped in a CTE that's
counted by the outer SELECT. No correctness concern.

### P3 — `ec_spire_quote_identifier` doubles `"` to `""`

Standard PG identifier quoting. The validator already rejects
column names not matching live `pg_attribute.attname` rows, so
attacker-supplied names can't slip through unless the caller can
also rewrite `pg_attribute`. Belt-and-suspenders quoting is the
right pattern.

### P3 — `#[pg_extern(strict)]` swallows NULL inputs silently

If `index_oid` or `row_payload` is NULL the function returns no
rows. For a remote endpoint called via libpq from the coordinator
that's probably the right shape (NULL is an upstream bug), but
worth documenting that NULL → silent empty result, not an error.

### P3 — Test fixture passes `embedding` as `encode_to_ecvector(...)::text`

The test JSON includes
`'embedding', encode_to_ecvector(ARRAY[1.0, 0.0], 4, 42)::text` —
relies on `ecvector::text` round-tripping cleanly through
`jsonb_populate_record`. This validates the 30823 ecvector finding
end-to-end on the INSERT side, not just the projection side. Good
coverage.

## Reviewer recommendation

Accept. The P2 single-row limitation is the load-bearing one for
ADR-069 bulk-load follow-up; the missing-column validation is a
small operator-experience improvement worth folding before the
coordinator hook starts driving real application INSERTs.
