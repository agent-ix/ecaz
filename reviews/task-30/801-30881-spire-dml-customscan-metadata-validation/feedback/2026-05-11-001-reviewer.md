# Review: SPIRE DML CustomScan Metadata Validation (30881)

## Summary

Adds the operation-specific metadata guard at `BeginCustomScan`
after JSON decode:
- PK SELECT requires non-empty `projected_columns` and empty
  `updated_columns`.
- UPDATE requires non-empty `updated_columns` and empty
  `projected_columns`.
- DELETE requires both empty.

Mirrors the primitive-plan validation from 30866; defense in
depth between the planner-side primitive plan and the
executor-side decoded metadata.

The three review-focus questions:
- Guard matches the primitive-plan contract from
  `dml_frontdoor.rs`: yes — same operation-specific column
  shape rules.
- Guard placed at the right executor boundary
  (`BeginCustomScan`): yes — runs before any primitive
  invocation, so a malformed plan-private payload errors before
  it can reach the remote dispatch.
- Non-behavioral for live UPDATE/DELETE routing: confirmed
  (paths still disabled).

## Findings

None blocking. The two-layer validation (planner-side at
primitive-plan construction, executor-side at metadata decode)
guards against any future divergence between the
plan-construction and plan-deserialization paths.

## Reviewer recommendation

Accept.
