## Answers to the packet's review questions

- **Converting blocked-before-dispatch remote rows into
  degraded skipped dispatches the right executor-state
  model for pre-dispatch Stage E faults?** Yes. The
  alternative — leaving them as `BlockedBeforeDispatch`
  in degraded mode — would mean degraded mode produces a
  result whose status is "blocked" for the same query that
  strict mode would have failed on, which is the opposite
  of degraded-mode semantics. Converting to
  `DegradedSkipped` is the correct strict-vs-degraded
  policy split at the executor-state layer. The new unit
  test locks this transformation against future
  regressions.
- **Version-skew fixture draws the correct boundary by
  stopping at production executor-state summary rather
  than opening sockets?** Yes. The failure mode is
  capability-side (descriptor extension version skew); it
  fires before any socket open per packet 30711's
  pre-dispatch capability gate. Trying to open sockets
  would not add coverage — the gate decided not to. The
  state-summary boundary is exactly where the assertion
  belongs.
- **Should the same pre-dispatch script be extended for
  `epoch_mismatch`, or should epoch freshness get its own
  lifecycle-oriented fixture?** Extend the same script.
  See packet 30780 — coder did exactly that, and it's the
  right call: epoch mismatch is the same pre-dispatch
  capability family (descriptor capability gate),
  different injection target. Sharing the script keeps
  the test-setup logic in one place. Lifecycle-oriented
  cases (DROP / REINDEX / CREATE INDEX CONCURRENTLY from
  packet 30772) are a different family and belong in their
  own fixture.
