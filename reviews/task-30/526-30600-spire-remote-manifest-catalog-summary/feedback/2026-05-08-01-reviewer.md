# Remote manifest catalog summary review

`ec_spire_remote_epoch_manifest_catalog_summary` compares the
current manifest decision against persisted manifest rows for the
active epoch and reports `not_required`, `blocked_upstream`,
`missing`, `stale`, or `ready`. No blocking findings.

## Confirmations

1. **Stale detection covers count drift AND content drift.**
   `persisted_entry_mismatch_count` (lines 2280-2319) compares
   each persisted entry against the current plan's expected
   shape, not just the count. So replaying with new node
   placements after the persist call surfaces as
   `stale_remote_epoch_manifest`, not silently `ready`.
2. **Status precedence is correct:** `not_required` first
   (local-only short-circuit), then upstream blocker
   propagation, then missing (decision says emit, no row), then
   stale (rows exist, content drifted), then ready.
3. PG18 coverage exercises ready, missing, and stale-via-mutation
   paths.

## Findings

- **Active-epoch advance race inherits from 30598.** This
  summary reads the current manifest decision and joins it to
  the persisted catalog. If the active epoch advances between
  the decision computation and the catalog read, the summary
  could observe a `ready` decision against the new epoch but
  catalog rows for the old epoch and report `missing`. Same
  self-healing dynamic as 30598. Track for the eventual
  consistency-pass refactor; not blocking.

- **`persisted_entry_mismatch_count`** is computed via a
  per-entry SQL comparison. With small entry counts (one row
  per remote node) this is fine; if remote-node catalogs ever
  grow large, the per-row LEFT JOIN approach should be
  benchmarked.
