# Review: SPIRE INSERT Trigger Batch Prepare (30973)

## Summary

**Closes Phase 12.4 P9.** Moves coordinator-routed multi-row
INSERT from row-at-a-time remote prepare to statement-level
batch:

- `ec_spire_enable_coordinator_insert(...)` now installs
  BEFORE ROW queue trigger + AFTER STATEMENT flush trigger;
- BEFORE ROW: extracts typed payload, plans owning remote,
  **queues** instead of dispatching;
- AFTER STATEMENT: drains queue in statement order, calls
  `ec_spire_prepare_coordinator_insert_tuple_payload_batch(...)`;
- batch helper builds all remote INSERT SQL and routes
  prepares through the 30972 fanout adapter;
- descriptor refresh staging dedupes
  `(node_id, descriptor_generation)` within the local
  statement so same-node rows don't trip the monotonic
  guard.

P9 marked complete.

## Findings

### P1 — BEFORE ROW + AFTER STATEMENT semantics

The two-trigger pattern is the right shape:

- **BEFORE ROW** queues per-row work in execution order.
  Suppressing the local heap row (returning NULL from the
  trigger to skip the local INSERT) keeps the SPIRE-fronted
  table heap empty, which is the existing v1 contract.
- **AFTER STATEMENT** runs once per statement boundary,
  giving the dispatch a natural batch window without
  needing manual flush tracking.

PostgreSQL's per-statement guarantee that AFTER STATEMENT
fires exactly once per affected statement (multi-row INSERT,
INSERT SELECT, COPY into the table) means the batch boundary
matches the user-visible statement boundary. Correct
boundary.

Worth confirming the AFTER STATEMENT trigger fires under
all the multi-row INSERT shapes — `INSERT ... VALUES (...),
(...)`, `INSERT ... SELECT ...`, and `COPY ... FROM ...`
should all produce a single batch flush. PostgreSQL semantics
say yes by construction; the
`test_ec_spire_trigger_multirow_commits_prepares_sql` fixture
should cover at least the VALUES and SELECT shapes.

### P1 — Statement-order draining

Draining the queue **in statement order** preserves
PostgreSQL's per-row trigger ordering guarantee. Important
because the placement-directory writes happen in this order;
out-of-order draining could produce a placement entry that
disagrees with the order in which the user issued rows.

If the order matters for the user (e.g., serial PK
assignment order), this preservation is load-bearing. Even
if it doesn't strictly need to match user order, statement
order is the only stable order — anything else (hash, batch
partition) introduces non-determinism. Keep.

### P1 — No-orphan contract for batch flush failure

Per 30972, the fanout adapter:

- rolls back any prepared-but-pre-callback remotes on
  partial failure;
- registers callbacks only after all prepares succeed.

This means a batch flush that fails after N of M prepares
succeed rolls back the N successful prepares before
returning failure, before any callback is registered. No
orphan. The trigger flush path inherits this contract
correctly because it routes through 30972's adapter.

Worth confirming the batch flush failure surfaces as a
**statement-level error** that aborts the local
transaction — otherwise the user could see partial-batch
success which would be inconsistent with the no-orphan
guarantee. From the existing prepare error handling this
should fail-closed; worth pinning in a fixture that asserts
"batch with one rejected row aborts the entire local
INSERT statement."

### P2 — Descriptor refresh dedupe

Deduping `(node_id, descriptor_generation)` within the
local statement is the right primitive for batch flush.
Without it, M rows targeting the same `node_id` at the
same generation would each trigger a descriptor refresh
attempt, and the monotonic descriptor guard would reject
all but the first as "no-op generation re-application."

The dedupe means same-node, same-generation rows refresh
the descriptor exactly once per batch. Correct local-
statement behavior.

The dedupe is **per-statement**, not per-transaction. That
matters: if the user issues two multi-row INSERT statements
in one transaction, each statement's batch dedupes
internally but not across the two. That's the right scope
because the second statement could have observed a
descriptor change committed by the first (via the
coordinator).

### P2 — P9 closure justification

With this packet:

- BEFORE ROW + AFTER STATEMENT pair installed by
  `ec_spire_enable_coordinator_insert`;
- statement-level batch routing through 30972 fanout
  adapter;
- descriptor refresh dedupe;
- existing trigger fixtures pass.

This is the load-bearing change for "M remote prepares no
longer serialize for multi-row INSERT." P9 closure
justified.

The remaining ergonomics (e.g., bench harness measuring
the actual speedup) is a separate Phase 12.9 measurement
row, not a P9 gap.

### P2 — Single-row INSERT path

A single-row INSERT still goes through queue-then-flush.
That's a small overhead (queue allocation + statement-
trigger fire) per single-row INSERT. From 30972 the single-
row case routes through the same fanout adapter (batch of
1), so the prepare itself is unaffected.

If the queue-then-flush overhead is measurable, a future
optimization could short-circuit when the queue has
exactly one row at flush time. Optional and likely below
measurement noise; mention only if benchmark evidence
shows pressure.

### P3 — Same-node batching in fanout

The fanout adapter from 30972 opens **one libpq connection
per remote node**. If a batch has M rows targeting one
node, the adapter should issue them as a single transaction
with M INSERTs + one PREPARE TRANSACTION (not M separate
remote transactions). This is what makes batching faster
than the per-row case.

Worth confirming the adapter actually consolidates same-
node rows. If it doesn't, the batch path produces M parallel
remote backends instead of one — still better than M serial
ones, but suboptimal and uses more remote connections than
necessary. From the request's "builds all remote tuple-
payload INSERT SQL and routes the prepares" wording, it
sounds like consolidation happens at the SQL build step
(one INSERT statement covering M rows per node).

If consolidation is per-node, the GID semantics from 30930
(one prepared transaction per `(index_oid, node_id,
served_epoch, top_xid)`) hold cleanly — the batch produces
one prepared txn per node, matching the GID identity.
