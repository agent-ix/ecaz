## Answers to the packet's review questions

- **Correct for the current AM local heap path to reject any
  active remote placement until remote row materialization
  lands?** Yes. The alternative would require the local AM
  cursor to skip remote placements silently, which violates
  the same global-ordering invariant from packet 30757 — a
  user expecting "top-k by score" would see a local-only
  prefix with no signal that remote rows were dropped.
  Failing closed at the loader is correct strict behavior.
- **Gate located at the right boundary, after published
  snapshot validation but before local-store validation?**
  Yes. Published snapshot is the source-of-truth contract
  (we trust the publisher's manifest); local-store
  validation is a coordinator-side resource check (we have
  the local stores the manifest references). Remote
  placement is *between* those: it's a fact about the
  manifest, not a coordinator resource issue. Putting the
  gate here means the operator-visible error names what's
  actually missing rather than a downstream symptom.
- **Should the error remain a plain string containing
  `remote_row_materialization` or be promoted to a shared
  constant/status row before AM cursor wiring?** Promote it.
  Packet 30757 already has
  `SPIRE_REMOTE_EXECUTOR_STEP_REMOTE_ROW_MATERIALIZATION =
  "remote_row_materialization"` as a constant; this gate's
  error message hard-codes the same string literal. Worth a
  small follow-up that imports the constant and references it
  here, so a future rename in either place fails to compile
  in the other rather than silently diverging. P2 below.
