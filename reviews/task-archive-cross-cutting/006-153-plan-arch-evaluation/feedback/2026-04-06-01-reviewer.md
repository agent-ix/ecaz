## 3. GUC Not Registered

FR-009 specifies `tqhnsw.ef_search` as a GUC (`PGC_USERSET`, session-settable via `SET tqhnsw.ef_search = 200`). FR-009-AC-5: "SET tqhnsw.ef_search = 200 SHALL succeed."

The current code only has `ef_search` as an index **reloption** in `options.rs`. No `_PG_init` hook, no `GucRegistry::define_int_guc()`, no GUC anywhere in the codebase (grep confirms zero matches for `GucRegistry`, `define_int_guc`, `_PG_init`, `pg_init`).

These serve different purposes:
- **Reloption**: per-index default, set at `CREATE INDEX ... WITH (ef_search=200)`, stored in pg_class
- **GUC**: per-session override, `SET tqhnsw.ef_search = 200`, takes precedence over reloption

The standard pattern (pgvector `ivfflat.probes`, `hnsw.ef_search`): GUC provides session-level value; scan reads GUC first, falls back to reloption when GUC is at default.

**Action:** Add GUC registration in `_PG_init` as part of A3. Scan reads GUC value, falls back to `relation_options().ef_search`.

## 4. Layer-0-Only vs Multi-Layer

Three documents disagree:

| Document | Position |
|---|---|
| FR-009 Step 2 | Multi-layer greedy descent ("For layer = max_level down to 1") |
| ADR-015 open question | "Layer-0 only is recommended for the initial implementation" |
| `graph.rs` implementation | Returns all `tids[0..count]` without layer awareness |

FR-007 defines the layer-segmented slot layout:
```
Layer 0:  tids[0..2M]
Layer 1:  tids[2M..3M]
Layer L:  tids[(L+1)*M..(L+2)*M]
```

The page format **supports** multi-layer. The `graph.rs` API doesn't expose it.

**Assessment:** Layer-0-only is the right first implementation. Upper-layer greedy descent helps at scale (>500K rows) but is unnecessary for the recall gate at 10K-50K vectors. The entry point at layer 0 is a reasonable starting position for beam search.

**Action:** Resolve ADR-015's open question as "layer-0-only for initial implementation." Add a note to FR-009 Step 2 acknowledging this. Add `load_layer0_neighbors(element_tid, M) -> Vec<ItemPointer>` to `graph.rs` that slices `tids[0..min(2*M, count)]`. This prevents upper-layer neighbor pollution during layer-0 beam search.

## 6. Recall Gate Preconditions

The plan sequences A4 (recall gate) after A3 (wire scan). But recall measurement only makes sense if results come out **distance-ordered**. ADR-015 stages this:

| Stage | Description | Recall measurable? |
|---|---|---|
| Stage 1 | Widen frontier to ef_search, keep linear fallback | No — linear fallback emits in page order |
| Stage 2 | Result buffering, ordered emission | Yes — if linear fallback is exercised infrequently |
| Stage 3 | Remove linear fallback | Yes — clean measurement |
| Stage 4 | Planner integration | Yes — end-to-end |

A4 requires at minimum Stage 2 (result buffering + ordered emission) and ideally Stage 3 (linear fallback removed) for clean measurement. The plan's A3 description implies this ("amgettuple pops from BinaryHeap in distance order") but doesn't explicitly call out that Stages 1-3 are sequential sub-steps of A3.

**Action:** Clarify in task 05 that A4 requires ordered emission (ADR-015 Stages 2+3). Add an explicit checkpoint: "linear fallback removed, amgettuple is purely BinaryHeap drain" as a precondition for the recall benchmark.

## 7. Architecture Assessment

### What's solid

- **Eager search in `amrescan`, drain in `amgettuple`** — this is the right shape per FR-009. The current lazy frontier approach is a staging scaffold toward it.

- **`BeamSearch` as the sole traversal structure** — once A3 completes, the `VisibleCandidateFrontierState` + `bootstrap_expansion` dual structure collapses into one `BeamSearch`. The simplification will be significant: `emitted_result_tids`, `expanded_source_tids`, `VisibleCandidateFrontierState`, `next_block_number`, `next_offset_number`, `scan_exhausted` all go away.

- **ADR-015 staging** — keeping the linear fallback through Stages 1-2 means recall can only go up, not down. Correct engineering discipline.

- **Recall gate before insert/vacuum** — right decision. Graph traversal bugs propagate to insert and vacuum. Validate scan first.

### A3 design tensions to resolve

1. **Closure/borrow interaction.** `BeamSearch::run()` takes `FnMut(&BeamCandidate) -> NeighborIter`. The neighbor function needs to call `load_graph_adjacency` (needs `index_relation: pg_sys::Relation`) and `score_scan_element_result` (needs `&TqScanOpaque` for prepared query and quantizer). But `BeamSearch` is *inside* `TqScanOpaque` (via `bootstrap_expansion`). You can't borrow opaque immutably for scoring while the beam search is borrowed mutably through opaque for expansion.

   Resolution options:
   - Extract the beam search from opaque before `run()`, run it standalone, put results back
   - Make the neighbor function capture `index_relation` + `prepared_query` + quantizer params directly (not `&TqScanOpaque`)
   - Use the eager approach: run beam search fully in `amrescan` with a local `BeamSearch` (not the one in opaque), store results in a result buffer in opaque

   The third option is cleanest and matches FR-009's design.

2. **Result buffer design.** FR-009 says results go in a `BinaryHeap`. But `amgettuple` needs to emit results in ascending score order (lowest score = best similarity). A `BinaryHeap` min-heap works, but the heap needs to hold scored element TIDs *and* their associated heap TIDs for duplicate-coalesced elements. Each result entry needs: `element_tid`, `score`, `heaptids: Vec<ItemPointer>`. Loading heap TIDs eagerly in `amrescan` (by calling `load_graph_element` for each result) keeps `amgettuple` simple but increases amrescan's buffer reads. Loading lazily in `amgettuple` saves reads for LIMIT queries but complicates the drain loop.

   Recommendation: eager loading in `amrescan`. The HNSW result set is bounded by `ef_search` (max 1000), so eagerly loading heap TIDs for ~40-200 results is cheap. This keeps `amgettuple` to a simple pop-and-emit loop.
