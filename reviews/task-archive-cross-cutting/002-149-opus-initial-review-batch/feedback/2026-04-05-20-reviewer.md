# Review: Build Source Scan Snapshot Leak on Error

**File:** `src/am/mod.rs:1547-1634` (`tqhnsw_build_scan_with_source`)
**Severity:** Medium (resource leak on error path)
**Category:** Memory safety / resource management

## Finding

The source column scan manually manages PostgreSQL resources:

```rust
let snapshot = unsafe { pg_sys::RegisterSnapshot(pg_sys::GetLatestSnapshot()) };
unsafe { pg_sys::PushActiveSnapshot(snapshot) };
let scan = unsafe { pg_sys::heap_beginscan(...) };
// ... scanning loop ...
unsafe {
    pg_sys::heap_endscan(scan);
    pg_sys::PopActiveSnapshot();
    pg_sys::UnregisterSnapshot(snapshot);
    pg_sys::ExecDropSingleTupleTableSlot(slot);
}
```

If any `pgrx::error!()` fires during the scan loop (e.g., `build_heap_tuple_with_source` line 1508 "source vector must not be empty"), the cleanup block at lines 1627-1632 is skipped. The snapshot is leaked, the scan is not ended, and the slot is not dropped.

This is mitigated by PostgreSQL's resource cleanup during error handling (transaction abort cleans up snapshots, scans, and memory contexts). However, relying on transaction-level cleanup is fragile and prevents any caller from catching and recovering from the error.

## Recommendation

Consider using RAII wrappers for the snapshot, scan, and slot. Or, restructure to use `PG_TRY/PG_CATCH` semantics (which pgrx provides via its error handling).

A simpler approach: collect the scan results into a vector first, then clean up deterministically:

```rust
let result = std::panic::catch_unwind(std::panic::AssertUnwindSafe(|| {
    // ... scanning loop ...
}));
// cleanup always runs
unsafe {
    pg_sys::heap_endscan(scan);
    pg_sys::PopActiveSnapshot();
    pg_sys::UnregisterSnapshot(snapshot);
    pg_sys::ExecDropSingleTupleTableSlot(slot);
}
match result {
    Ok(count) => count,
    Err(e) => std::panic::resume_unwind(e),
}
```

## Action Required

Consider adding RAII cleanup or catch-unwind to prevent resource leaks on error. Not critical for v0.1 since PostgreSQL's transaction abort handles cleanup, but important for robustness.
