# Review: Linear Scan Buffer Pin/Lock Safety

**File:** `src/am/mod.rs:535-622` (`next_linear_scan_heap_tid`)
**Severity:** Low (correct but worth noting)
**Category:** Correctness / edge cases

## Finding

The linear scan correctly releases buffers before returning heap TIDs (line 611), which is the right pattern for PostgreSQL index scans. However, there is a subtle consideration:

When pending heap TIDs are drained (lines 540-542), the function returns a TID that was decoded from a page that is **no longer pinned**. This is correct because:
1. The TID is a value copy (block_number, offset_number) stored in scan opaque memory
2. The heap TID refers to the heap relation, not the index relation
3. PostgreSQL's executor will separately fetch the heap tuple

**This is correct.** No buffer leak or use-after-free exists.

## One edge case to document

If a concurrent vacuum were to run between `amgettuple` calls and physically remove or rearrange data pages, the `next_block_number` / `next_offset_number` cursor in the opaque state could point to a page that has changed. Currently:
- Vacuum callbacks are no-ops, so this cannot happen yet
- When vacuum is implemented, the scan cursor state must be resilient to concurrent page modifications

## Recommendation

No code change needed. Add a comment in `next_linear_scan_heap_tid` noting that this function relies on the fact that vacuum is currently a no-op, and that concurrent page modification tolerance will be needed when vacuum performs actual deletions.

## Action Required

Documentation only.
