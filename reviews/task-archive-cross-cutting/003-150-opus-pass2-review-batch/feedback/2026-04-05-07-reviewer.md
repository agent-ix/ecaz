# 07 — staged frontier mechanics are coherent

**Severity:** Observation (no action required)

## Summary

The bootstrap frontier scan machinery introduced since the first review is internally coherent and well-tested. Key observations:

### What works correctly

1. **Frontier seeding**: `initialize_scan_entry_candidate` seeds from the metadata entry point and expands via score-ordered multi-hop fill up to `MAX_BOOTSTRAP_FRONTIER_CANDIDATES` (3). The visited set prevents re-seeding.

2. **Consume-and-refill**: `consume_and_refill_bootstrap_frontier` pops the best-scoring candidate and refills from its adjacency. The visited set gates adjacency expansion, and the expanded-source set prevents re-expanding the same source.

3. **Score-ordered expansion policy**: `next_bootstrap_expand_index` correctly selects the lowest-scoring unexpanded candidate with tie-breaking on insertion order.

4. **Frontier head tracking**: `recompute_candidate_frontier_head` is called at every mutation point. The `Option<usize>` representation cleanly handles empty frontiers.

5. **Active candidate bridge**: `maybe_consume_bootstrap_frontier_candidate` correctly gates on all relevant state flags. The single-candidate consumption per scan lifecycle is intentional.

6. **Materialization helper**: `materialize_active_candidate_result` correctly loads the active candidate's element, stores its heap tids in the pending drain machinery, and clears the active slot. Ready for wiring.

7. **Lifecycle cleanup**: `amendscan` frees all heap-allocated scan state in the correct order. `reset_scan_position` clears all mutable state on rescan.

### Regression coverage

The test suite has thorough coverage of:
- Frontier shape after seeding (candidate count, provenance, scores)
- Head selection and consumption ordering
- Consume-refill behavior with adjacency expansion
- Visited and expanded set lifecycles
- Active candidate materialization into pending drain state
- Multi-page, duplicate-heavy scan correctness

### Notes for ordered-scan wiring

When `materialize_active_candidate_result` is wired into `amgettuple`:
- The `maybe_consume` guard will need to allow re-consumption after the active candidate is drained
- The linear scan fallback will need a transition condition (e.g., frontier exhausted → fall back to linear, or eliminate linear scan entirely)
- `xs_orderbyvals` / `xs_orderbynulls` will need to carry the current result score back to the executor for ordered merge
