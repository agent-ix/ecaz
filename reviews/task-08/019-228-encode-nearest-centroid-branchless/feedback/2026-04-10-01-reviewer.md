# Feedback: Branchless `nearest_centroid_index` + Unrolled 16-Way Fast Path

**Reviewer:** Claude (Opus 4.6)
**Date:** 2026-04-10
**Commit reviewed:** `72af185`

## Scope match

Matches the task prompt: `nearest_centroid_index` body rewritten with a
branchless blend, a dedicated `nearest_centroid_index_16(&[f32; 16], f32)`
entry point for the production `(1536, 4)` codebook size, and
`quantize_to_indices` dispatching to the 16-way path via
`<&[f32; 16]>::try_from(codebook)`. The original branching scan is
preserved in `#[cfg(test)]` as `nearest_centroid_index_branching` for
cross-checking. Tie-breaking is preserved (strict `<`).

## Findings

### 1. Correctness is rigorously proven

Three new tests cover the rewrite end-to-end:

- `branchless_matches_branching_over_random_inputs` — 12 000 random
  `(codebook, value)` pairs across codebook sizes {4, 8, 16, 32, 64, 128}
  (every production bit width 2..=7). Deterministic seed.
- `unrolled_16_matches_generic_branchless` — 40 000 random pairs, all
  with 16-element codebooks. Deterministic seed.
- `quantize_to_indices_dispatches_unrolled_for_16_centroids` — 1536-
  element rotated buffer, asserts dispatched output equals the explicit
  generic-path loop.

Together with the preserved
`nearest_centroid_index_prefers_lower_index_on_tie` test, every code
path the optimization touches has a direct assertion. Strong. ✓

### 2. **Microbenchmark skipped — same scope deviation as 10061**

Task L138-153 asks for an `#[ignore]`-gated microbenchmark comparing
three paths (branching / branchless generic / unrolled 16) over 1536
random inputs × 100 000 iterations, with a target of **≥3× speedup**
for the unrolled path.

The packet declines this with the same argument used in packet 227:
"the change is bit-exact; there is no risk of wrong output, only that
it might not actually be faster." That argument proves we have not
introduced a regression *in correctness*. It does not prove we have
avoided a regression *in performance*.

Specifically, on some CPUs `cmov` is slower than a well-predicted
branch. The task explicitly acknowledges this:
> If you only see ~1.5×, the branch predictor was already hiding the
> cost — record the number and decide whether to land for the
> readability/asm-quality improvement alone.

Without running the microbench, we cannot make the decision the task
asked us to make. Pair this microbench with the 10061 one in a single
follow-up so both optimizations' numbers land together and can be read
in context.

### 3. **Asm inspection also skipped — contrary to task Step 2**

Task L60-65 asks to confirm the compiler is actually lowering the
trailing-`if` blend to `cmov` / `select`:
> Inspect the generated assembly with `cargo rustc --release -- --emit
> asm` against a small test crate that calls `nearest_centroid_index` in
> a loop to confirm. If the compiler is *not* generating branchless code,
> fall back to an explicit `f32::to_bits` integer-blend trick — but try
> the straightforward form first.

The packet does not mention asm inspection. Without it, the "branchless"
claim is structural only. If the compiler kept the branch, the rewrite
is a no-op — still correct, just no faster.

**Request:** in the same follow-up that lands the microbench, include
a one-line asm verification. One approach:
```
cargo rustc --release --lib -- --emit asm
# grep for nearest_centroid_index_16 in target/release/deps/*.s
```
and attach a short snippet (5-10 lines covering the inner loop) showing
either `cmov` or the integer-blend fallback. If asm shows a branch,
escalate to the `f32::to_bits` blend per task Step 2's fallback.

### 4. **Dispatch test is weaker than its name suggests — fix before landing**

`quantize_to_indices_dispatches_unrolled_for_16_centroids` asserts output
equality between `quantize_to_indices(codebook, rotated, 1536)` and an
explicit loop calling `nearest_centroid_index(&codebook, v)` per element.
Because test 2 already proves `nearest_centroid_index_16` and
`nearest_centroid_index` produce identical output, this test *cannot
distinguish* "dispatch picked the unrolled path" from "dispatch picked the
generic path." It is a correctness-of-output test, not a
correctness-of-dispatch test — which is a problem, because the whole
point of the branch is the dispatch.

**Required fix before landing** (do both):

1. Rename the test to
   `quantize_to_indices_matches_per_element_scan_for_16_centroids` so
   the name matches what it actually asserts.
2. Add a second test —
   `quantize_to_indices_dispatches_unrolled_for_16_centroids` (keeping
   the old name for the dispatch assertion) — that directly proves the
   `&[f32; 16]` branch is taken. The cleanest approach is a
   thread-local (or `AtomicUsize`) call counter gated behind
   `#[cfg(test)]` inside `nearest_centroid_index_16`, incremented on
   entry, asserted `== 1536` after the dispatch call. A `#[cfg(test)]`
   counter keeps the production path clean and proves the dispatch
   picked the unrolled variant rather than silently falling through.

Without this, the branch ships with zero direct evidence that the
dispatch it exists to perform is actually reaching the unrolled path —
the asm inspection from item 3 is the only other check, and it's a
manual operator step done once rather than a guard against future
regression.

### 5. `needless_range_loop` allow + comment — keep, and do not remove

The targeted `#[allow(clippy::needless_range_loop)]` on
`nearest_centroid_index_16` with an inline comment explaining the
constant-trip-count unroll rationale is exactly the right hygiene: the
compiler needs the indexed form to see the fixed 16-trip count and
unroll. A future refactor that "cleans up" the loop into an iterator
chain would silently defeat the unroll the entire branch exists to
produce.

**Required**: the allow + comment must stay. If the comment is ever
reduced in a future edit, it should be expanded rather than removed —
name the unroll-dependence explicitly so a later reader does not mistake
it for a stylistic carve-out. No code change on this branch; noted here
so the review record makes the load-bearing role of the allow explicit.

### 6. `CodeIndex` type tightening

The return type was tightened from `best_index as CodeIndex` (via
`usize`) to a direct `u16`. `CodeIndex = u16` already, so this is
semantically a no-op and saves one widening hop in the inner loop. Clean
incidental improvement; no action needed.

## Verdict

**Approve with required fixes before landing.** Item 4 (dispatch test
rename + direct call-counter assertion) must land on this branch — the
current test proves output correctness but not that the dispatch
reaches the unrolled path at all. Items 2 and 3 (microbench + asm
verification) land in a single follow-up paired with 10061's
microbench, so both optimizations' numbers can be read in context;
without those, the "performance" half of both tasks is unverified —
we have proven bit-exactness and semantic-equivalence, not actual
speedup.

Item 5 is an explicit "keep and do not remove" note on the
`needless_range_loop` allow. Item 6 is an acknowledgment, no action.

The correctness work on this branch is the strongest of the batch — two
cross-checks × 52K random samples is excellent. The gap is entirely on
the measurement side, where we accepted the claim without evidence.
