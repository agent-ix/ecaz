### 29d-2 — L=64 scan profile (`11107`)

The rerank-budget A/B is a clean profile-by-bisection. Result
table is the right shape: each row trades off recall vs latency
along the only safe scan-time lever, and the curve is monotone.

| `rerank_budget` | recall@10 | mean lat |
|---|---|---|
| 10 | 0.8600 | 3.43 ms |
| 32 | 0.9880 | 5.26 ms |
| 48 | 0.9955 | 6.58 ms |
| 56 | **0.9960** | 7.16 ms |
| 64 (default) | 0.9965 | 7.82 ms |

The first value meeting the ≥0.996 recall floor is 56, which
gives 7.16 ms — misses the 6 ms stop target. Correct call to
document the residual rather than lower the default.

The deeper interpretation (not stated in the packet but worth
recording for future work): heap fetch dominates rerank cost at
this scale. ~64 reranks × ~50 µs warm-cache heap fetch = ~3 ms
of the 7.82 ms total just in heap I/O — that floor is below the
6 ms target, so closing the gap further requires a structural
change (e.g., caching heap source vectors in shared buffers, or
using grouped-PQ as a finer-grain pre-rerank to reduce heap-fetch
count). Both are out of 29d's scope and the right deferral.

One thing the packet didn't note but I want to flag for future
work: the same SIMD pattern that 29d-3 landed for the **build**
distance also applies to `exact_heap_rerank_distance`
(`routine.rs:1817-1828`), which is still a scalar f32 inner
product loop. The expected payoff is small (~1% of L=64 latency
since heap fetch dominates over the IP math), so explicitly **not
worth doing in the landing slice** — but if a reviewer asks "did
you SIMD all the f32 IP hot paths?" the honest answer is "build
yes, rerank deliberately no, because measurement shows heap fetch
dominates rerank cost." File this as a one-line follow-up if
anyone asks.

#### Code review of the SIMD distance

`src/am/ec_diskann/ambuild.rs:516-590`:

- AVX2 + FMA gated via `is_x86_feature_detected!("avx2") && is_x86_feature_detected!("fma")` runtime check, with scalar fallback. Right shape — runtime detection over compile-time means CI binaries work on any modern x86_64.
- `#[target_feature(enable = "avx2,fma")]` on the SIMD function lets the compiler emit those instructions only inside that scope. Correct.
- 4-way unroll (`acc0..acc3`) for hiding FMA latency. 32 floats per iteration = 4× 8-lane vectors. Good cache and pipeline behavior. For 1536-d: 48 iterations exactly + 0 tail (since 1536 % 32 == 0). Optimal alignment to the unroll factor.
- Horizontal sum via `_mm256_storeu_ps` to a stack array + scalar reduction. Slightly more verbose than `_mm256_hadd_ps` chain, but easier to read and the reduction cost is negligible vs the 48-iter inner loop.
- Tail loop uses `get_unchecked` under the bounded `i < len` invariant. Safe.
- Test `source_inner_product_dispatch_matches_scalar` verifies bit-near equivalence (1e-4 tolerance, accounting for FMA reordering). Right tolerance — strict bit equality would fail because FMA reorders fewer rounding steps than the scalar accumulator.
- Non-x86_64 fallback to scalar. Right call (no aarch64 NEON variant landed today, which would be the only other relevant target architecture).

This is textbook SIMD-acceleration shape. No concerns.

### Final readiness sweep (`11109`)

Full release-mode comparison across three engines on the same
isolated real-10k corpus, same warm cache, same build parameters
(`graph_degree=32, build_list_size=100, alpha=1.2`).

**Build + storage:**

| Engine | Build | Index size |
|---|---|---|
| **ec_diskann** | **14.59 s** | **4.94 MB** |
| pgvectorscale | 5.72 s | 5.14 MB |
| ec_hnsw | 5.77 s | 14.42 MB |

ec_diskann is now **2.55×** the strongest reference (pgvectorscale)
on build, comfortably inside the 3× stop condition. Storage is the
**smallest** of the three (-4% vs pgvectorscale, -66% vs HNSW).

**Latency / recall sweep:**

| L | ec_diskann | pgvectorscale | ec_hnsw |
|---|---|---|---|
| 64 | 0.9965 / 7.80 ms | 0.9955 / **3.48 ms** | 0.9695 / **2.91 ms** |
| 128 | 0.9965 / 7.79 ms | 0.9990 / **5.81 ms** | 0.9710 / **4.75 ms** |
| 200 | 0.9970 / **7.98 ms** | **1.0000** / 8.50 ms | 0.9710 / 6.75 ms |
| 400 | 0.9970 / **8.49 ms** | **1.0000** / 17.3 ms | 0.9715 / 13.0 ms |
| 800 | 0.9975 / **9.34 ms** | **1.0000** / 30.1 ms | 0.9715 / 25.5 ms |

(Bold = best of the three on each axis at that row.)

The crossover behavior is interesting and worth being explicit
about for the merge discussion:

- **L=64–128**: pgvectorscale and HNSW are faster (constant factors
  matter most when the rerank window is small and the graph
  traversal is short). ec_diskann's recall is higher than HNSW's
  but it pays for it.
- **L=200**: curves cross. ec_diskann becomes faster than
  pgvectorscale; HNSW is still faster but at materially lower
  recall.
- **L=400 / L=800**: ec_diskann is the fastest of the three at
  near-exact recall. The flatness vs the steep curves is real and
  comes from ec_diskann's fixed `rerank_budget=64` default vs
  pgvectorscale's growing `query_rescore`.

This is the strongest possible landing position for an "initial
tuning lane": competitive on build, smallest on storage, highest
recall, and either fastest or close-to-fastest on latency
depending on operating point.

## Risks and follow-ups

All round-3 risks were addressed in 29d. Newly identified items
(none are landing blockers):

1. **Rerank-distance SIMD is left scalar by deliberate choice.**
   `exact_heap_rerank_distance` could use the same AVX2+FMA
   pattern that ambuild now uses. Expected payoff is ~1% of L=64
   latency because heap fetch dominates rerank cost. File as a
   "match symmetry across IP hot paths" cleanup if anyone asks; do
   not pursue speculatively.
2. **L=64 latency closure requires a structural change** (heap
   source-vector caching, or grouped-PQ as a finer-grain
   pre-rerank to cut heap-fetch count). Out of 29d scope per its
   stop condition; reasonable post-merge investment if low-L
   latency parity becomes a product priority.
3. **Aarch64 NEON SIMD distance is not implemented.** The
   `cfg(target_arch = "x86_64")` path falls to scalar on aarch64.
   For Graviton-class hardware (which the Task 29 charter
   explicitly anticipates as the future product-claim host), this
   would need a NEON variant. Out of scope for the local landing
   slice; track as part of the Graviton benchmarking lane.
